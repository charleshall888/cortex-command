# Research: refine should load parent-epic context

## Research Questions

1. **Is there empirical evidence of epic-context drift in recently-refined tickets with `parent:` set?** → **Weak signal, dominated by selection confounders.** Of 7 sampled child→epic pairs (epics 082, 093, 101): 5 on-epic, 1 implicit-on-epic, 1 mild drift, 0 severe. All 7 child tickets were already tightly scoped to a single decision-record or capability bullet from their epic, so a clean spec is consistent with both "refine pipeline anchored on epic" and "ticket text was self-anchoring." There is no counterfactual (no specs from a refine run that explicitly excluded parent context). Strongest single signal: ticket 091's spec opens with "epic #82" multiple times — but the epic content was the load-bearing input that any author would have read regardless of automation.

2. **What does refine currently load, and where is parent-epic context conspicuously absent?** → Refine loads: backlog item body via `cortex-resolve-backlog-item` `[plugins/cortex-interactive/skills/refine/SKILL.md:26-32]`; requirements files in Clarify `[plugins/cortex-interactive/skills/refine/references/clarify.md:31-37]`; `discovery_source` and `research` frontmatter fields as **background reference only** `[plugins/cortex-interactive/skills/refine/SKILL.md:54-56]`; prior `lifecycle/{slug}/research.md` for the sufficiency check `[plugins/cortex-interactive/skills/refine/SKILL.md:84-96]`. Parent-epic context is **not loaded**. The resolver returns only `{filename, backlog_filename_slug, title, lifecycle_slug}` and does not expose the `parent:` frontmatter field, even though the `BacklogItem` dataclass includes a `parent` field `[cortex_command/overnight/backlog.py:89]`.

3. **What's the minimum-viable epic context shape, and what's a defensible token cap?** → Industry-emerging answer: an **orientation packet** containing parent ID + parent title + 1–3-sentence summary + epic-level "done" criteria, *not* the full epic body. Anthropic's own multi-agent guidance favors curated handoff over transparent share; LangGraph projects only declared schema keys; CrewAI requires explicit `context=[…]` declarations. The corpus survey of 17 epics (see Codebase Analysis) shows existing epic bodies already cluster well below the ≤500 token global cap — 13/17 epics expose a named context section (`## Context from discovery` / `## Context` / `## Framing (post-discovery)`) with median 194 tokens, p90 353 tokens, max 409 tokens; the remaining 4/17 fall back to a first body paragraph at 63–129 tokens. **All 17 epics fit under 500 tokens** by either path. A defensible cap is **≤500 tokens** of injected epic context, with a **soft target of 300 tokens** (covers 11/13 named sections without truncation) and a hard ceiling of 500 (covers 13/13). Well below the threshold where the literature reports 4–6× token inflation and cascading drift (`arxiv 2503.13657`, `arxiv 2510.26585`). Important caveat: this measures fit-under-cap (structural feasibility), not anchoring quality. The 4-epic first-paragraph fallback floor (median 73 tokens) may be too thin to anchor scope on its own — see Q4 / DR-1 trade-offs and Open Questions for the deferred quality-validation plan.

4. **Where in refine should epic context land?** → Strongest leverage in **Clarify** (alignment check explicitly compares ticket intent against epic intent). **Research** is a secondary candidate; the strongest illustrative case is ticket 110, which expanded scope from "two implementations" to "seven" without re-checking the epic's framing of why unification matters — this is a single anecdote, not systemic evidence, and cannot distinguish phase-leverage from idiosyncrasy. **Spec** is also a real candidate on resume paths where Clarify is skipped (see DR-2 below) and as a fallback over legacy un-anchored `research.md`. Rough recommendation: load at Clarify §2 on fresh runs, with a Spec-phase fallback fire when the Clarify hook did not run in this session. Discussed further in DR-2.

5. **How should refine degrade when parent linkage is missing or malformed?** → The current data has **0 dangling parents and 0 nested epics** across 39 child tickets pointing to 22 epics — so the failure surface is small. Safe defaults: (a) child has no `parent:` → silently skip (this is the common case for top-level features), (b) parent ID resolves to no file → warn but proceed (degrades to current behavior), (c) parent ID resolves to a non-epic ticket → treat as "linked context" not "epic context" (still useful), (d) nested epics → load only direct parent, do not recursively walk. Hard-fail is unwarranted — refine must remain runnable on tickets without epic linkage.

6. **Does this drift problem affect adjacent skills (`/cortex-interactive:lifecycle`, `/cortex-interactive:clarify`)?** → `/cortex-interactive:lifecycle` already records `discovery_source` and `research` as background reference context when delegating to refine, but does not load `parent:` either. There is no standalone `clarify` skill; clarify exists only as a reference document inside refine `[NOT_FOUND(query="standalone clarify skill", scope="plugins/cortex-interactive/skills")]`. The fix should be implemented in refine first, with the loader extracted into a `bin/` script reusable by lifecycle if drift evidence justifies expansion. Generalizing prematurely violates the "complexity must earn its place" constraint from `requirements/project.md:19`.

7. **Prior art for parent-context propagation in agentic systems?** → Four observed patterns: schema-projection (LangGraph), explicit declaration (CrewAI), transparent share (AutoGen nested teams — minority pattern), isolated + curated handoff (Anthropic, Claude Code subagents, Cline). Issue trackers (Linear, Jira, GitHub sub-issues) treat parent linkage as **first-class data, opt-in propagation** — the parent edge exists but content is fetched on demand, not auto-embedded. For AI-coding-agent contexts specifically, the consensus is curated orientation-packet over full inheritance. Documented failure modes: token inflation, goal drift on under-specified parent context, stale-context cascades, conflicting-assumption failure across sibling tickets, latent-subagent alignment drift.

**Backlog data model — epic linkage**: Epic↔child relationship uses `parent: <numeric-id>` in child frontmatter; epics have `type: epic` and a "## Child tickets" body section but no inverse `children:` field. Sampled examples: `[backlog/004-prep-hooks-and-apikey-for-sharing.md:7]` → `[backlog/003-shareable-install-epic.md:6]`; `[backlog/020-add-harness-component-pruning-checklist.md:7]` → `[backlog/018-harness-quality-improvements-epic.md:6]`.

**Audit numbers**: 39 child items with `parent:` set; 22 distinct parent IDs; **0 dangling parents**, **0 nested epics** (no epic has `parent:` set). All 22 unique parents resolve to a real file. Zero-padding (e.g., `parent: 049`) is correctly handled. `[premise-unverified: not-searched]` for the historical drift rate of dangling-parent counts — only the current snapshot was audited.

**Resolution mechanics**: `cortex-resolve-backlog-item` exit codes are 0/2/3/64/70 `[plugins/cortex-interactive/skills/refine/SKILL.md:34-40]`; missing parent would surface as exit 3 ("no match"). The resolver's JSON output is a fixed four-field shape — `{filename, backlog_filename_slug, title, lifecycle_slug}` — so adding parent-epic loading requires either (a) extending the resolver's output to include `parent` and the resolved parent's filename/title, or (b) a second resolver invocation after parsing the child's frontmatter directly.

**Epic-body corpus survey (extraction-heuristic input)**: Surveyed all 17 backlog items with `type: epic` set. Two structural patterns emerged:

| Pattern | Count | Example epics | Token range | Notes |
|---------|-------|---------------|-------------|-------|
| `## Context from discovery` named section | 9 | `[backlog/009-requirements-management-overhaul.md:14]`, `[backlog/014-overnight-conflict-visibility-epic.md:14]`, `[backlog/126-eliminate-home-repo-vs-worktree-context-drift-in-overnight-runner.md:14]` | 88–409, median 194 | Most common shape; the "house style" for post-discovery epics |
| `## Context` named section (no "from discovery") | 3 | `[backlog/003-shareable-install-epic.md:14]`, `[backlog/054-harden-settingsjson-permissions-for-public-distribution.md:14]`, `[backlog/063-user-configurable-setup-per-component-opt-in-and-per-repo-permissions-scoping.md:14]` | 133–353 | Older epics or hand-authored |
| `## Framing (post-discovery)` named section | 1 | `[backlog/074-implement-in-autonomous-worktree-overnight-component-reuse.md:13]` | 266 | Single-epic bespoke header |
| No context-style section; first body paragraph | 4 | `[backlog/033-dashboard-ui-quality-baseline-epic.md:13]`, `[backlog/066-suppress-non-decision-output-in-interactive-phases-epic.md:13]`, `[backlog/082-adapt-harness-to-opus-47-prompt-delta-capability-adoption.md:13]`, `[backlog/093-modernize-lifecycle-implement-phase-preflight-options.md:13]` | 63–129, median 73 | Body opens with a paragraph that functions as the epic summary |

Header-keyword choices observed across the 13 named-section epics: every recognized header begins with `## Context` or `## Framing` — no other header keywords function as the summary section. Other body sections present across the corpus include `## Scope`, `## Children` / `## Child tickets`, `## Out of scope`, `## Research`, `## Wave structure`, `## Implementation order`, `## End state`, `## Decompose summary` — none of which are good summary candidates (children list ticket IDs only; scope is structurally different from a 1–3-sentence orientation).

**All 13 named-section bodies fit under the ≤500-token global cap** (max 409 tokens for `[backlog/126-eliminate-home-repo-vs-worktree-context-drift-in-overnight-runner.md:14-26]`); **11/13 fit under a 300-token soft target** (the two over-300 are 063 at 353 and 126 at 409). All 4 first-paragraph fallbacks fit under 130 tokens.

**Derived extraction heuristic** (resolves OQ1):

```
1. If child frontmatter has parent: <id>, resolve <id> to backlog/<id>-*.md
2. If parent file has ## Context from discovery section → use that section body
3. Else if parent file has ## Context section → use that section body
4. Else if parent file has ## Framing (post-discovery) section → use that section body
5. Else use the first non-empty body paragraph (after H1 if present, else first paragraph)
6. Else use just the parent title
7. Truncate to 500 tokens (~2000 chars). Soft-warn at 300 tokens.
```

The priority order is grounded in **purpose-built quality**, not raw frequency. The named context sections (steps 2–4) are explicit "summary for downstream consumers" sections; the first-paragraph fallback (step 5) is a structural extraction whose content varies — sometimes a wave list, sometimes an intro paragraph. Within the named-section group, `## Context from discovery` is the modern post-discovery house style and ranks first; `## Context` is the older bare convention; `## Framing (post-discovery)` is a single-epic bespoke header treated as a near-equivalent of `## Context from discovery`. Frequency happens to align with this quality ordering for steps 2 and 3 (9 and 3 epics) but not for step 4 (`## Framing` at frequency 1) — frequency is incidental, not the principle. Truncation at 500 tokens is the literal global cap stated in Q3; 300 tokens as a soft warning matches the 85th percentile of named-section length in the corpus. No epic in the corpus would currently trigger the 500-token hard cap; two would trigger the 300-token soft warning. **Floor risk**: the 4 fallback epics (033/066/082/093, 63–129 tokens, median 73) are the cases where extraction quality is least validated — content was not inspected, only structural position.

## Web & Documentation Research

**Issue trackers**: Linear, Jira, and GitHub treat parent linkage as a first-class data edge but **do not auto-propagate parent content** to AI agents or workflows. Linear added a ⌘J "send selection as context to agent" affordance in 2025 — opt-in, not ambient (`linear.app/docs/agents-in-linear`). GitHub added the REST endpoint `GET /repos/{o}/{r}/issues/{n}/parent` in 2025 specifically because consumers had to fetch on demand (`github.blog/changelog/2025-09-11`). Jira Automation rules require explicit smart-value expressions like `{{issue.parent.components}}` to inherit — nothing inherits by default (`community.atlassian.com`).

**Agentic frameworks**: Four propagation patterns documented:
- **Schema-projection** — LangGraph passes only keys present in the subgraph's `input_schema`; runtime context does not propagate by default (`docs.langchain.com`, `github.com/langchain-ai/langgraph/issues/5700`).
- **Explicit declaration** — CrewAI requires `context=[task_a, task_b]` on Task definitions; outputs flow only where declared (`docs.crewai.com/en/concepts/tasks`).
- **Transparent share** — AutoGen nested teams pass full ChatMessage history across the boundary (`github.com/microsoft/autogen/pull/5863`); minority pattern.
- **Isolated + curated handoff** — Anthropic's multi-agent system passes "precise, relevant information rather than full conversation history" (`anthropic.com/engineering/multi-agent-research-system`); Claude Code subagents start with **zero parent-conversation context** unless explicitly provided (`code.claude.com/docs/en/sub-agents`); Cline ships per-agent context windows with only the final result flowing back.

**Failure modes** (from `arxiv 2503.13657` Cemri et al. MAST taxonomy, `arxiv 2510.26585`, `arxiv 2505.02709` goal-drift evaluation, plus practitioner blogs):
- Token explosion / 4–6× inflation when full history propagates.
- Goal drift on under-specified parent context.
- Stale-context cascades (parent state mutated after first read).
- Conflicting-assumption failures across sibling tickets that inherit different slices.
- Latent-subagent / alignment drift when unsuppressed parent framing flips child behavior.

## Domain & Prior Art

The emerging consensus across Anthropic guidance, Simon Willison's agentic-engineering patterns, and the foojay.io best-practices write-up is an **orientation packet**: parent identity + 1–3-sentence summary + named done-criteria. Never the full parent body. Never the parent's transcript. Lazy-fetch escape hatch (give the child a parent ID and a way to read more) over preemptive embedding.

For cortex-command's specific shape — a ticket refiner that needs scope-anchoring without context flooding — this maps directly: parse `parent:`, fetch epic title + the named context section if present (with priority `## Context from discovery` → `## Context` → `## Framing (post-discovery)`), fall back to first body paragraph, expose the epic file path so refine can read deeper if it explicitly chooses to. The corpus survey confirms 13/17 epics expose a named section that structurally functions as the orientation-packet summary; the remaining 4 fall back to a first body paragraph that occupies the same slot but whose anchoring quality is unverified (content not inspected during the survey). The named-section pattern is the de facto authoring convention; the first-paragraph fallback is a best-effort extraction. This is a **well-trodden problem with an emerging-but-unstandardized answer** (~18 months old). No RFC, no canonical reference implementation, but the pattern is consistent enough to act on.

## Decision Records

### DR-1: Recommended approach is "orientation packet" — primary at Clarify with Spec-phase fallback

- **Context**: The user observed tickets straying from epic intent during refine — that user-observed report is the load-bearing signal. Empirical sampling on a small confounded sample (6/7 tickets stayed on-epic, 1 had mild drift; tickets were already tightly scoped so the test does not stress the hypothesis) does not corroborate or refute it strongly. The honest A-vs-E comparison: cost of A = a small loader, one more frontmatter parse per refine invocation, ≤500 tokens added context per parent-linked refine, plus the Spec-phase fallback wiring; cost of E = unknown drift rate at unknown rate of occurrence on future tickets that are less tightly scoped than the sample. Both costs are bounded; neither dominates the other on evidence alone. The recommendation to act is conditioned on (a) the user-observed signal being treated as legitimate evidence and (b) the post-deployment measurement plan in Open Questions being adopted, not deferred indefinitely.
- **Options considered**: A (orientation packet, recommended), B (resolver extension as enabler for A), C (full body), D (recursive walker), E (do nothing — defensible alternative if the user observation is reclassified as not-reproducible).
- **Recommendation**: A, with B as the implementation path. Implement at Clarify §2 (Load Requirements Context) where it naturally sits alongside other alignment context, **with a Spec-phase fallback fire** on Clarify-skip resume paths (see DR-2). Cap at ≤500 tokens (hard) / 300 tokens (soft warn). When the child has no `parent:` set, silently skip — current behavior preserved. The extraction heuristic is a fixed priority order against named context sections (`## Context from discovery` → `## Context` → `## Framing (post-discovery)`) with first-paragraph fallback — derived from the observed corpus shape (see Codebase Analysis), not invented. The 17-epic corpus survey establishes structural feasibility (every epic produces a packet under 500 tokens), not anchoring quality — that validation belongs to the post-deployment measurement plan.
- **Trade-offs**: Adds a small loader and one more frontmatter parse per refine invocation. Does NOT solve cases where the epic itself is poorly written (12-line context paragraph still injects 12 lines, though the 500-token cap bounds the worst case). Does NOT auto-detect when epic context is critical vs nice-to-have — every parent-linked ticket gets it. The latter is intentional — selective loading would require a heuristic harder to reason about than "always load if parent exists." **Floor-case risk**: the 4 first-paragraph-fallback epics (033/066/082/093, median 73 tokens) are the population where the heuristic produces the thinnest, least-validated packet. Epic 082 is the active CLAUDE.md policy-anchor epic, so child tickets refining against it would receive the thinnest packet on the most policy-load-bearing parent — the measurement plan should specifically cover this slice.

### DR-2: Phase placement — Clarify primary, Spec fallback for Clarify-skip resume paths

- **Context**: Clarify, Research, and Spec are all candidates. Where does the injection produce the most leverage per token, AND which placements are required to cover all execution paths refine actually takes? SKILL.md Step 2 (`[plugins/cortex-interactive/skills/refine/SKILL.md:42-61]`) defines four resume paths: Path 1 (both artifacts exist → re-run or exit), Path 2 (`spec.md` exists, `research.md` missing → resume at Research, **skip Clarify**), Path 3 (`research.md` exists, `spec.md` missing → resume at Spec, **skip Clarify**), Path 4 (else → start at Clarify). The re-run path on SKILL.md:61 is underspecified — does not state whether re-run re-enters at Clarify or just at Spec.
- **Options considered**: Clarify only; Research only; Spec only; Clarify + Spec fallback (recommended); all three.
- **Recommendation**: **Clarify + Spec fallback**. Primary fire at Clarify §2 on fresh runs (Path 4) and Path 2 (which skips Clarify but goes to Research). Fallback fire at the top of Spec phase (`specify.md §1`) when a session arrives at Spec without having run Clarify in that session — this covers Path 3 (`research.md` exists, `spec.md` missing) and Path 1 re-runs. The fallback is gated on a session-scoped flag set when Clarify ran; if the flag is unset and Spec is entering with an existing `research.md` from a prior run, the loader fires at Spec to anchor the structured-interview surface against the epic. The clarified-intent statement Clarify produces is the scope anchor for Research on fresh runs (per `[plugins/cortex-interactive/skills/refine/SKILL.md:109]`); Research inherits the anchor without a separate injection on Path 4 / Path 2. Spec is downstream of Research on the happy path and would be redundant THERE — but on resume paths where Clarify did not run, the Spec-phase fallback is the only place the loader can fire.
- **Trade-offs**: Two injection sites instead of one — slightly more implementation surface and an extra session-scoped flag. The fallback fires only when Clarify did NOT run in this session, so steady-state token cost is unchanged for fresh-run flows (the dominant path). The re-run interpretation on SKILL.md:61 should be pinned down at spec time — recommended reading: re-run resets to Path 4 (start at Clarify) so the loader fires at the primary site. The Spec-phase scope-reframing surface (`specify.md §4` complexity/value gate, which offers "drop entirely" / "bugs-only" / "minimum viable") can still un-anchor scope after the loader fires at Clarify; this is acceptable because the user's reframing decisions are themselves load-bearing — the loader's job is to anchor the *initial* scope statement, not to lock it.

### DR-3: Failure-mode degradation — silent skip when no parent, warn when dangling

- **Context**: Audit shows 0 dangling parents and 0 nested epics today, but defensive degradation is cheap and prevents future regressions.
- **Options considered**: Hard-fail on missing/dangling; warn-but-proceed; silently skip.
- **Recommendation**: Tiered: (a) no `parent:` field → silently skip (the common case); (b) `parent:` set but file missing → warn to stderr, proceed without epic context; (c) `parent:` resolves to a non-epic ticket → load it anyway as "linked context" (still useful); (d) nested parent chain → load only direct parent, do not recurse.
- **Trade-offs**: The non-epic-parent case (c) silently broadens what "epic context" means. Acceptable because the data model doesn't strictly enforce "parent is always an epic" — keeping the loader permissive matches how the field is actually used.

### DR-4: Implementation should extract a `bin/cortex-load-epic-context` script

- **Context**: Per `requirements/project.md:27` SKILL.md-to-bin parity convention (`bin/cortex-*` scripts must be wired through an in-scope SKILL.md / requirements / docs / hooks / justfile / tests reference), deterministic logic should ship as a `bin/cortex-*` script so the refine SKILL.md can reference it cleanly. The same script is what `/cortex-interactive:lifecycle` would call later if drift evidence justifies expansion (see Q6).
- **Options considered**: Extract to bin/ script vs inline logic in clarify.md vs extend `cortex-resolve-backlog-item`.
- **Recommendation**: New `bin/cortex-load-epic-context <child-backlog-slug>` script that returns JSON `{parent_id, parent_title, parent_summary, parent_path, summary_source}` or empty on no-parent. The `summary_source` field reports which heuristic step produced the summary (`context-from-discovery` / `context` / `framing-post-discovery` / `first-paragraph` / `title-only`) for telemetry and debugging. Bin-script form chosen for testability, telemetry, and parity-gate cleanliness. Parity-gate landing site: the canonical `plugins/cortex-interactive/skills/refine/references/clarify.md` (and a mirrored reference in `references/specify.md` for the Spec-phase fallback fire from DR-2) — both must land in the same commit as the script, or the pre-commit `cortex-check-parity` hook will emit W003 orphan and block the commit. The script name `cortex-load-epic-context` does not collide with the closed `PLUGIN_NAMES` or `RESERVED_NON_BIN_NAMES` sets in `bin/cortex-check-parity`, so no linter source edit is required. Extending `cortex-resolve-backlog-item` is a defensible alternative (grep shows refine SKILL.md, refine clarify.md, the canonical mirrors, and `tests/test_resolve_backlog_item.py` are the only consumers; additive extension is backwards-compatible) — the choice between the two is stylistic, with bin-script form preferred for test isolation and the `summary_source` telemetry field that doesn't fit the resolver's return shape.
- **Trade-offs**: One more script to maintain. Minor — same shape as existing `cortex-resolve-backlog-item`.

## Open Questions

- **Should the orientation packet include sibling-ticket summaries** (other children of the same parent)? The prior-art research did not directly cover this. Probably not in v1 — adds tokens, marginal anchoring value, increases coupling to siblings' state. **Deferred: revisit if drift evidence post-v1 shows sibling-coordination failures.**
- **Should the implementation surface the epic file path in the clarified-intent output**, so Research can opt to read more? Likely yes (matches the lazy-fetch industry pattern), but the exact shape — frontmatter field on research.md, conversation message, or clarified-intent suffix — is a spec-time decision. **Deferred: resolve in Spec.**
- **Is there a measurement plan to validate ROI and anchoring quality post-deployment?** Three measurements should run together: (1) **drift rate audit** — sample N+5 newly-refined tickets with `parent:` set after the loader ships; tag specs against parent epics on the on-epic / drift-mild / drift-severe scale used in this research; compare to the 6/7 on-epic baseline; (2) **floor-case anchoring quality** — specifically sample child tickets refined against the 4 first-paragraph-fallback epics (033/066/082/093) to validate that 73-token packets actually anchor scope, not just fit under cap; (3) **ROI rollback criterion** — define up front what observation would cause the loader to be removed (e.g., zero observed drift reduction at N+10 plus measurable token-budget cost, or new drift modes induced by mis-anchored fallback content). **Deferred: include as a backlog item alongside the implementation ticket; the measurement plan must be authored before the implementation merges, not after.**
