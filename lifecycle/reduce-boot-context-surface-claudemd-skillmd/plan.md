# Plan: reduce-boot-context-surface-claudemd-skillmd

## Overview

Ship boot-context reductions in three ordered commit batches inside one lifecycle PR: (1) **measurement infrastructure** commits first (byte-count utility + baseline), (2) the **fixture-expansion commit** lands as a separate commit before any compression — spec R2's "separate PR" prose is treated as a commit-ordering requirement (the spec's actual acceptance check verifies `git log` order, not PR boundaries), then (3) **trim commits** for CLAUDE.md extraction, description compression, requirements stub, and L2 body extraction. Lever-A description compression runs only after the fixture commit because the fixture must constrain compression rather than be authored alongside it. Body extraction (Lever C, R7) is independent and can land in parallel with description work after the fixture commit. Post-trim measurement and mirror parity verification close the lifecycle.

**Verification-gate honesty note.** The byte-count utility (Task 1) is the sole oracle for baseline.md and post-trim caps. Task 1 includes a calibration spot-check (hand-verified bytes against an independent YAML parser invocation) to detect parser bugs and guards against the lifecycle-prohibited self-sealing verification pattern. The fixture expansion (R2) is treated as a content-preservation gate, not an independent routing-recall test — phrases are sourced from current SKILL.md text to lock in known-routing substrings; the routing-recall risk for utterances *not* currently in any description is mitigated by a manual spot-check of the routing-pressure cluster during description compression, and is flagged for follow-up by a separate backlog ticket ("independently-sourced phrase corpus for skill routing"). Spec R10's `pytest tests/test_plugin_mirror_parity.py` invocation is preserved only as a smoke check — the test as written covers 3 lifecycle reference files (not SKILL.md mirrors), so R10 enforcement falls primarily on the `diff` check in Task 14 with path-anchored filtering.
