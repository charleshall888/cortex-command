# Plan: build-mcp-control-plane-server-with-versioned-runner-ipc-contract

## Overview

Land the MCP control-plane server (`cortex mcp-server`, five stdio tools) on top of an already-largely-implemented runner IPC contract, plus the four adversarial-review hardening fixes (atomic concurrent-start lock, schema-version cap, SIGTERM tree-walk, opaque resumable cursor with truncate-detection), the per-session escalations migration (R17–R22 — clean cutover, no dual-read), the operational-safety gates (`confirm_dangerously_skip_permissions`, server-side max-bytes, `runner-bootstrap.log`), the pre-install in-flight guard (R28), and the end-to-end integration test (R27) that closes the acceptance-test-escapability through-line. Work is sequenced as: foundations → escalations migration → MCP server module → distribution + E2E + pytest budget.

### Task 3: Runner SIGTERM handler walks descendant tree
- **Files**: `cortex_command/overnight/runner.py`, `tests/test_runner_sigterm_propagation.py` (new)
- **What**: Installs a SIGTERM handler in the runner that, on receipt, enumerates descendants via `psutil.Process(os.getpid()).children(recursive=True)`, sends SIGTERM to each, waits **up to 6 seconds** for graceful exit, then SIGKILLs survivors. Closes Adversarial §2d / R12. The walk is at signal-receipt time, not startup, so batch_runner workers and grandchildren are reached regardless of which spawn site set `start_new_session=True`. The 6-second graceful budget is **strictly less than** Task 14's 12-second outer SIGKILL budget so the runner's in-handler survivor-SIGKILL phase always completes before `overnight_cancel`'s outer SIGKILL hits the runner itself (closes the budget-race surfaced in critical review).
- **Depends on**: none
- **Complexity**: complex
- **Context**: Install via `signal.signal(signal.SIGTERM, _handle_sigterm)` near runner startup; the handler iterates descendants, calls `proc.terminate()` (SIGTERM each), waits up to 6 seconds collectively for graceful exit (use `psutil.wait_procs(descendants, timeout=6)`), then calls `proc.kill()` (SIGKILL) on survivors. `psutil` is already a project dependency (per research.md). The recorded-child-PGID approach is rejected (spec R12). **Test fixture must spawn at least one grandchild with `start_new_session=True`** — production topology per Task 19(b) puts grandchildren in their own PGIDs, so the unit test must exercise the PGID-divergent case to actually verify the tree-walk works against grandchildren that `os.killpg` cannot reach. Test fixture: starts a real runner subprocess that spawns one grandchild with `start_new_session=True` (PGID-divergent) and one without (in runner's PGID), sends SIGTERM to runner PID, and asserts BOTH grandchildren are reaped within 8 seconds (6s graceful + 2s overhead). A second test asserts the SIGKILL escalation path: one grandchild spawned with `start_new_session=True` AND with its own SIGTERM-ignoring handler — must be SIGKILL'd, must terminate within the budget.
- **Verification**: `pytest tests/test_runner_sigterm_propagation.py` — pass if exit 0.
- **Status**: [x] complete

### Task 14: Tool — `overnight_cancel` (force flag, signal escalation, self-heal)
- **Files**: `cortex_command/mcp_server/tools.py`, `tests/test_mcp_overnight_cancel.py` (new)
- **What**: Adds the `overnight_cancel` `async def` handler. Required input: `session_id: str`. Optional: `force: bool = False`. Output: `{cancelled: bool, signal_sent: list[str], reason: str, pid_file_unlinked: bool}` where `reason ∈ {"cancelled", "no_runner_pid", "magic_mismatch", "start_time_skew", "signal_not_delivered_within_timeout"}`. Closes R6.
- **Depends on**: [11, 2, 12]
- **Complexity**: complex
- **Context**: Reuse `cortex_command/overnight/cli_handler.py`'s existing self-heal logic at lines 309-402; tool wraps it. `os.killpg`, `os.unlink`, and `psutil.Process(...).is_running()` calls wrapped in `await asyncio.to_thread(...)` per R29. ## Veto Surface

- **Pre-install guard implementation choice (Task 17)**: spec lists three options (`cortex setup` step, `pyproject.toml tool.uv.required-environment`, first-import check). Plan picks first-import check **with carve-outs for pytest, runner-spawned children, and the dashboard** — but the guard still runs on every Python process that imports `cortex_command` (every `cortex` invocation, every IDE introspection, every `python -c "import cortex_command"`). User may prefer the `cortex setup` opt-in path (no `__init__.py` runtime cost, no carve-outs needed, but install-time enforcement is opt-in rather than enforced).
- **Cancel signal-escalation strategy (Task 14)**: plan picks polling-with-12s-cap (`os.killpg(SIGTERM)` + 250ms poll on `is_running()` for up to 12s + `os.killpg(SIGKILL)`) over a fixed-sleep pattern, so the cancel exits early when the runner does. The 12s outer cap is strictly greater than Task 3's 6s in-handler graceful budget — but the gap is 6s, which may be tight under heavy load (large descendant trees, slow psutil enumeration). User may prefer (a) a wider asymmetry (T3=4s, T14=15s) for more headroom, or (b) elimination of the outer-SIGKILL backstop entirely (let the runner's own handler do the SIGKILL escalation; trust T3's tree-walk completing). - **`FeatureConfig` location (Task 7)**: `FeatureConfig` does not exist in the codebase today. Plan defers the choice between (a) creating a new dataclass at `cortex_command/overnight/feature_executor.py` or (b) extending an existing context struct that flows through the same callsites. - **Bundled status + list_sessions in Task 12**: both are read-only-state tools and share scaffolding. Splitting them would add a 21st task without atomicity benefit. User may prefer one task per tool for cleaner failure isolation.
- **Serial-marker convention (Task 20)**: spec says `@pytest.mark.serial` "or equivalent" — the implementer picks. Plan defers; user may want a specific convention pinned (pytest-xdist `xdist_group` is a good default for this project).
