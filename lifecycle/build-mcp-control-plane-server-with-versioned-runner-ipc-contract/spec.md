# Specification: build-mcp-control-plane-server-with-versioned-runner-ipc-contract

> **Epic reference**: This ticket is the DR-1 work item from epic 113. See `research/overnight-layer-distribution/research.md` for cross-ticket context. Scope below is ticket-specific only.

## Problem Statement

The `cortex` overnight runner today must be started from a terminal — the user types `cortex overnight start`, then leaves Claude Code to babysit the launch. This breaks the project's north-star: "send Claude to work with a plan, let it spin up its own teams" should be initiated *from* a Claude Code conversation, not adjacent to one. We will ship a stdio MCP control-plane server (`cortex mcp-server`) that exposes five overnight tools to any Claude Code session, wrapped around a versioned runner IPC contract that closes a concurrent-start race, an unattended-cost foot-gun, a cancel-doesn't-reach-children gap, and other adversarial-review findings — and we will fully migrate `escalations.jsonl` to per-session storage so the MCP `overnight_logs` tool can return escalation records keyed to a single session, matching the contract for `events.log` and `agent-activity.jsonl`.

## Requirements

> **MoSCoW**: All 29 numbered requirements below are **must-have** for this ticket. The "should-have" tier is intentionally empty — the adversarial-review fixes (R8–R16), the full escalations migration (R17–R22), the end-to-end integration test (R27), the pre-install in-flight guard (R28), and the async-correctness requirement (R29) are not optional polish; they close concrete safety/correctness gaps that would otherwise ship hidden. R27 specifically addresses the through-line surfaced in critical review: that individually-targeted acceptance criteria can pass while the named hazard remains open. The "won't-have" tier is enumerated in `## Non-Requirements`. Items that could naturally have been "should-have" — `caffeinate` self-spawn, `audit.log` of `overnight_start_run` calls, `CORTEX_MCP_DISABLE` per-tool opt-out, ±2s cancel-tolerance widening, dual-read backwards-compat for escalations during a release transition — were explicitly declined under Non-Requirements rather than parked as soft "should-have" items, so the boundary stays sharp.

### MCP server module

1. **`cortex mcp-server` subcommand exists**: The stub at `cortex_command/cli.py:262-267` is replaced with a real handler that initializes a FastMCP server over stdio. Acceptance: `cortex mcp-server --help` exits 0 with non-stub output. `python -c "from cortex_command.mcp_server.server import build_server; print(type(build_server()).__name__)"` outputs `FastMCP`.

2. **MCP server uses stderr-only logging**: All Python `logging` output from the server module is configured to stream to `sys.stderr`, not `sys.stdout`. Acceptance: `grep -rn "stream=sys.stderr\|StreamHandler(sys.stderr)" cortex_command/mcp_server/` yields ≥1 match; no `print(` calls in `cortex_command/mcp_server/` outside `if __name__ == "__main__"` guards.

3. **Tool: `overnight_start_run`**: Spawns the runner via `cli_handler.handle_start` semantics in a detached process group. Required input: `confirm_dangerously_skip_permissions: Literal[True]` (Pydantic `Literal[True]`). Optional input: `state_path: str | None`. Output: `{session_id: str, pid: int, started_at: str (ISO 8601)}`. Tool description begins verbatim: "This tool spawns a multi-hour autonomous agent that bypasses permission prompts and consumes Opus tokens. Only call when the user has explicitly asked to start an overnight run." **The handler intercepts Pydantic validation errors on the `confirm_dangerously_skip_permissions` field and returns a Tool Execution Error (`isError: true`) whose body re-includes the verbatim warning sentence** — so a model that called the tool without the parameter receives the warning text in the error response, not a bare JSON-RPC schema-validation failure. Acceptance: `pytest tests/test_mcp_overnight_start_run.py` exits 0; `grep -c "confirm_dangerously_skip_permissions" cortex_command/mcp_server/tools.py` ≥ 1; `grep -c "spawns a multi-hour autonomous agent" cortex_command/mcp_server/tools.py` ≥ 2 (the description and the validation-error path both contain the sentence).

4. **Tool: `overnight_status`**: Reads `lifecycle/sessions/{session_id}/overnight-state.json` (or auto-discovers via active-session pointer when `session_id` omitted) and returns a structured status object. Optional input: `session_id: str | None`. Output: `{session_id, phase, current_round, started_at, updated_at, features: {pending, running, merged, paused, deferred, failed}, integration_branch, paused_reason}`. Acceptance: `pytest tests/test_mcp_overnight_status.py` exits 0; status response **byte-length asserted ≤ 4 500 chars** in test (~1 500 tokens at the 3 chars/token rule of thumb; this is a coarse smoke check, not an Anthropic-tokenizer contract — the 25 K hard cap is enforced by Claude Code at runtime).

5. **Tool: `overnight_logs`**: Returns paginated log lines for events / agent-activity / escalations using opaque cursor tokens. Required input: `session_id: str`. Optional inputs: `files: list[Literal["events", "agent-activity", "escalations"]] = ["events"]`, `cursor: str | None = None`, `limit: int = 100` (server-cap 200), `tail: int | None = None`. Output: `{lines: list[dict], next_cursor: str | None, eof: bool}`. Acceptance: `pytest tests/test_mcp_overnight_logs.py` exits 0; test asserts a 100-line page is **≤ 30 000 chars** (~10 000 tokens at the 3 chars/token rule of thumb — coarse smoke check); test asserts `cursor` is server-internally-base64-decodable; **test asserts nothing about cursor structure leaks into client-visible documentation** (R23).

6. **Tool: `overnight_cancel`**: Reads `runner.pid`, calls `ipc.verify_runner_pid()`, then `os.killpg(pgid, SIGTERM)` followed by `os.killpg(pgid, SIGKILL)` after a 10-second wait. Required input: `session_id: str`. Optional input: `force: bool = False` — when `True` and the SIGTERM-then-SIGKILL escalation does not exit the runner within the 10-second window (e.g. SIGSTOP'd runner), the tool unlinks `runner.pid` so a subsequent `overnight_start_run` can claim the lock; the runner process itself is left in whatever signalled state it ended in (the user is responsible for `kill -9 <pid>` if needed; the manual recovery procedure is documented in `docs/mcp-server.md`). Output: `{cancelled: bool, signal_sent: list[str], reason: str, pid_file_unlinked: bool}` where `reason` is one of: `"cancelled"`, `"no_runner_pid"`, `"magic_mismatch"`, `"start_time_skew"`, `"signal_not_delivered_within_timeout"` (signal was sent successfully via `os.killpg` but the process did not exit within the SIGTERM-then-SIGKILL window — distinct from `"signal_send_failed"` which would imply a syscall-level failure, e.g. EPERM, never observed in practice for our setup but reserved for completeness). Acceptance: `pytest tests/test_mcp_overnight_cancel.py` exits 0; tests cover (i) live-runner cancel, (ii) stale-pid self-heal, (iii) magic-mismatch refusal, (iv) start_time-skew refusal, (v) **SIGSTOP'd runner with `force=False` returns `signal_not_delivered_within_timeout` and leaves runner.pid**, (vi) **SIGSTOP'd runner with `force=True` returns `signal_not_delivered_within_timeout` AND unlinks runner.pid**.

7. **Tool: `overnight_list_sessions`**: Globs `lifecycle/sessions/*/overnight-state.json` and returns active + recent terminated. Optional inputs: `status: list[Literal["planning","executing","paused","complete"]] | None`, `since: str | None` (ISO 8601), `limit: int | None = 10`, `cursor: str | None`. Output: `{active: [SessionSummary], recent: [SessionSummary], total_count: int, next_cursor: str | null}` where `SessionSummary = {session_id, phase, started_at, updated_at, integration_branch}`. Default behavior (no filters) returns all active + 10 most recent terminated. Acceptance: `pytest tests/test_mcp_overnight_list_sessions.py` exits 0; test asserts default shape with 50 fixture sessions returns ≤ 4 000 tokens.

### IPC contract hardening

8. **Atomic concurrent-start lock**: `cortex_command/overnight/ipc.py::write_runner_pid` becomes an atomic `O_CREAT|O_EXCL` claim. On `FileExistsError`: read existing pid, run `verify_runner_pid` logic; if stale → `unlink` and retry once with `O_CREAT|O_EXCL` again. **On the retry's second `FileExistsError`** (a third party — typically a concurrent starter — beat us to the recreated claim): re-read the new pid, run `verify_runner_pid`, and `raise ConcurrentRunnerError(session_id, existing_pid)` regardless of liveness. The retry budget is exactly one — never spin. If the verifier on the retry-conflict reads sees a third stale state, it is treated as an alive race-loser to break the loop deterministically. **The MCP `overnight_start_run` tool surfaces `ConcurrentRunnerError` as `{started: false, reason: "concurrent_runner_alive", existing_session_id: ...}`.** Acceptance: `pytest tests/test_runner_concurrent_start_race.py` exits 0; test fires two parallel `overnight_start_run` calls against a fixture session — both observing a stale lock — and asserts exactly one succeeds, the other returns `concurrent_runner_alive`. Test also covers (a) two concurrent starters with no pre-existing lock (one wins via O_EXCL, other gets `concurrent_runner_alive`), (b) one starter against an alive lock (immediate `concurrent_runner_alive`, no unlink).

9. **`verify_runner_pid` rejects unknown future schema versions**: `cortex_command/overnight/ipc.py:127-165` is changed from `schema_version >= 1` to `1 <= schema_version <= MAX_KNOWN_RUNNER_PID_SCHEMA_VERSION` where `MAX_KNOWN_RUNNER_PID_SCHEMA_VERSION = 1` is a module constant. Acceptance: `pytest tests/test_ipc_verify_runner_pid.py::test_rejects_future_schema_version` exits 0.

10. **Opaque cursor codec with truncate-detection**: New `cortex_command/overnight/cursor.py` exposes `encode(offset: int, file_size_at_emit: int) -> str` (base64-JSON) and `decode(cursor: str) -> dict`. `logs.read_log` is extended to accept the opaque cursor and return `next_cursor` encoding `{offset, file_size_at_emit: post_read_size}`. Cursors are opaque to consumers (per MCP spec). Acceptance: `pytest tests/test_cursor_codec.py` exits 0; test asserts `decode(encode(o, s)) == {"offset": o, "file_size_at_emit": s}` for a representative set of values.

11. **Cursor-invalid signal on shrink**: `read_log` returns a structured `{cursor_invalid: true, current_size: int}` payload and `next_cursor: null` when the cursor's `file_size_at_emit > current_size` (file was truncated). The MCP `overnight_logs` tool surfaces this as `{lines: [], cursor_invalid: true, eof: false, next_cursor: null}`. Acceptance: `pytest tests/test_cursor_truncation.py::test_shrunk_file_returns_cursor_invalid` exits 0.

12. **Runner SIGTERM propagation walks the full descendant tree**: The runner (`cortex_command/overnight/runner.py`) installs a SIGTERM handler that, on receipt, enumerates all descendants via `psutil.Process(os.getpid()).children(recursive=True)` (psutil is already a project dependency, used by `verify_runner_pid` in `ipc.py`), sends SIGTERM to each descendant PID, waits up to 10 seconds for graceful exit, then sends SIGKILL to any survivors. The walk is done at signal-receipt time (not at runner startup), so workers spawned by batch_runner — and their own grandchildren — are reached regardless of which spawn site used `start_new_session=True`. The recorded-child-PGID approach (orchestrator + batch_runner) is **rejected** in favor of the tree walk because batch_runner spawns worker `claude` subprocesses whose PGIDs the runner does not know in advance. Acceptance: `pytest tests/test_runner_sigterm_propagation.py` exits 0; test starts a real runner subprocess, has it spawn two real grandchild subprocess.Popen processes (e.g. `python -c "import time; time.sleep(60)"`), sends SIGTERM to runner's PID, asserts BOTH grandchildren are reaped within 12 seconds (10 s graceful + 2 s overhead). A second test asserts the same with one grandchild ignoring SIGTERM (must be SIGKILL'd, must terminate within the budget).

### Operational safety

13. **`overnight_start_run` confirmation gate** (cross-references R3): The required `confirm_dangerously_skip_permissions: Literal[True]` parameter and the verbatim warning sentence in the tool description are part of the contract. Acceptance: `pytest tests/test_mcp_overnight_start_run.py::test_missing_confirm_arg_returns_validation_error` exits 0; `pytest tests/test_mcp_overnight_start_run.py::test_warning_sentence_present_in_description` exits 0.

14. **Server-side max-bytes-read on file-touching tools**: All file reads in `cortex_command/mcp_server/tools.py` honor a `MAX_TOOL_FILE_READ_BYTES` constant (default 256 MiB; overridable via `CORTEX_MCP_MAX_FILE_READ_BYTES` env var, parsed as integer). The cap defends against (a) accidental whole-file reads (cursor pagination is the primary path; this is belt-and-suspenders) and (b) a single oversized log line that exceeds the cap. **Oversized-line behavior**: if `read_log` encounters a line whose start-offset + length exceeds the budget, it returns `{lines: [{...truncated to ~10K chars with explicit "...(line truncated, original X bytes)" sentinel}], next_cursor: <encodes offset advanced past the oversized line's terminating newline>, truncated: true, oversized_line: true, original_line_bytes: X}`. The `next_cursor` advancing past the line guarantees the client can paginate past a monster line on the next call rather than being stuck. **Whole-file-too-large** (cap exceeded by aggregate of normal-sized lines, not a single line): tool returns whatever lines fit within the budget plus a normal `next_cursor` for resumption — no special flag needed since this is the standard pagination path. Acceptance: `pytest tests/test_mcp_max_bytes_cap.py` exits 0; tests cover (i) oversized single line returns `oversized_line: true` and an advancing cursor, (ii) cursor on the next call returns lines after the monster line, (iii) aggregate-cap path returns normal pagination.

15. **Decline `_meta["anthropic/maxResultSizeChars"]` escape hatch**: No tool in `cortex_command/mcp_server/tools.py` sets the `anthropic/maxResultSizeChars` annotation. Pagination via cursors at the default 25 K token cap is the sole large-output path. Acceptance: `! grep -rn "anthropic/maxResultSizeChars" cortex_command/mcp_server/` (the negation: command must fail because there are zero matches).

16. **MCP-spawned runner — stdin DEVNULL, stdout/stderr to bootstrap log**: The `overnight_start_run` tool's runner spawn uses `subprocess.Popen(..., stdin=subprocess.DEVNULL, stdout=<bootstrap_log_fd>, stderr=<bootstrap_log_fd>, start_new_session=True)` where `bootstrap_log_fd` is an O_CREAT|O_APPEND|O_WRONLY file descriptor on `session_dir / "runner-bootstrap.log"` opened by the MCP tool *before* spawning. This captures pre-`events.log`-init failures (import errors, missing deps, `session_dir` permission errors, early uncaught exceptions) that would otherwise be lost — the runner's own logging-to-stderr convention is preserved while the JSON-RPC stream stays clean (the MCP server's stdin/stdout is never inherited because the runner is detached + redirected to disk). Acceptance: `pytest tests/test_mcp_runner_spawn_devnull.py::test_runner_spawn_uses_devnull_stdin_and_bootstrap_log` exits 0; test fakes `subprocess.Popen` and asserts (a) `stdin=DEVNULL`, (b) `stdout` and `stderr` are file descriptors pointing to a `runner-bootstrap.log` file under the session_dir, (c) `start_new_session=True`. A second test triggers a deliberate `ImportError` in the runner spawn and asserts `runner-bootstrap.log` contains the traceback.

## Non-Requirements

- **No remote / Cloudflare-hosted MCP variant** (DR-3 — rejected by epic on architectural grounds).
- **No SEP-1686 Tasks-based async tools**. SEP-1686 shipped experimental in MCP 2025-11-25 but Claude Code's client-side support is unverified as of 2026-04. Tools remain synchronous-return-immediately; long-running visibility uses `overnight_status` polling, not Tasks.
- **No plugin monitor integration**. Monitors die with the Claude Code session; runner must outlive sessions.
- **No push notifications / streamed tool results**. Pull-based polling only — Claude Code's stdio client does not surface progress notifications visibly today (anthropics/claude-code#4157).
- **No in-process job registry / UUID-based job IDs**. The MCP server is stateless; tools accept `session_id` (filesystem-grounded) and wrap `cli_handler.py` directly. The dylan-gluck "job manager" framing is an architectural reference, not an implementation pattern to copy.
- **No `_meta["anthropic/maxResultSizeChars"]` escape hatch on any tool** (R15).
- **No `CORTEX_MCP_DISABLE` per-tool conditional registration in v1**. All five tools are registered unconditionally; per-tool opt-out can be added later if schema-token cost becomes a measured issue.
- **No `overnight_resume`, `overnight_pause`, `overnight_step` tools in v1**. The five tools enumerated in the ticket body cover the north-star UX. Additional control verbs follow if specific user needs surface.
- **No App Nap / `caffeinate` self-spawning by the runner in this ticket**. The headless-laptop-lid risk is documented in `docs/mcp-server.md` as a known caveat with the workaround `caffeinate -i cortex overnight start`. A follow-up ticket can address self-spawning if it bites in practice.
- **No audit log of `overnight_start_run` invocations in v1**. The required confirm-arg + tool-description warning is the operational gate. An `audit.log` follow-up can be considered after real-world usage.
- **No clock-step resilience widening of the `±2s` cancel verification**. Adversarial §2a is documented as a known caveat; widening requires a `psutil.boot_time()` cross-check that is out of scope.
- **No dual-read / backwards-compatibility migration window for escalations.** R22 is a clean cutover; the in-flight-upgrade hazard is closed by R28's pre-install guard, not by code that reads from both old and new paths during a release. (Considered and explicitly rejected: dual-read doubles the surface area and persists deprecated code paths for a hazard the install guard prevents.)
- **No support for mixed-version cortex installs on the same machine.** The single-cortex-install assumption holds; a hypothetical future v2 schema will require a separate follow-up ticket to define version-coexistence policy.
- **No tokenizer-accurate token-cost contracts.** Acceptance tests use byte/char-length proxies as coarse smoke checks; the 25 K hard cap is enforced by Claude Code at runtime using Anthropic's tokenizer, not by the test suite. (Considered and explicitly rejected: depending on Anthropic's `count_tokens` API in the test suite trades simple smoke checks for an internet-required dependency every test run.)

## Edge Cases

- **Two Claude Code sessions invoke `overnight_start_run` concurrently**: R8's atomic `O_CREAT|O_EXCL` claim guarantees exactly one succeeds. The losing tool call returns `{started: false, reason: "concurrent_runner_alive", existing_session_id: ...}`.
- **MCP client passes a cursor from an old reading after the file shrinks** (truncate-then-grow): R11 returns `{lines: [], cursor_invalid: true, next_cursor: null}` and the client must omit the cursor on retry to re-baseline.
- **`overnight_logs` requested for a session whose runner.pid is stale** (process died uncleanly): The tool reads the log file directly (logs survive runner death); `next_cursor` is emitted normally and `eof: false` if there are more bytes. Stale runner.pid is irrelevant to the read path.
- **`overnight_cancel` invoked after the runner already exited cleanly**: `runner.pid` is absent → tool returns `{cancelled: false, reason: "no_runner_pid"}`. No signal sent.
- **`overnight_cancel` invoked while `verify_runner_pid` rejects on magic mismatch**: Tool returns `{cancelled: false, reason: "magic_mismatch"}` and self-heals by clearing the stale lock — same self-heal already in `cli_handler.py:382-397`.
- **`overnight_status` requested with no `session_id` and no active session**: Tool reads `~/.local/share/overnight-sessions/active-session.json`; if absent, returns `{session_id: null, phase: "no_active_session"}`.
- **`overnight_list_sessions` against a fresh repo with no sessions**: Returns `{active: [], recent: [], total_count: 0, next_cursor: null}`.
- **`overnight_logs` requested for a session_id that does not exist**: Tool returns a structured error `{error: "session_not_found", session_id: ...}` rather than raising. Bad inputs return Tool Execution Errors per MCP SEP-1303 (`isError: true`), not JSON-RPC protocol errors.
- **A second session_dir contains a stale runner.pid from a prior run that crashed**: `verify_runner_pid` recognizes the dead PID via psutil, R8's lock-claim path unlinks it, and the new run proceeds.
- **MCP server hung reading a 1 GB log file**: R14's `MAX_TOOL_FILE_READ_BYTES` cap returns `truncated: true` after reading 256 MiB; Claude Code's tool timeout is not exceeded.
- **The runner is paused (`SIGSTOP`-able state)**: State files are still readable; `overnight_status`, `overnight_logs`, `overnight_list_sessions` continue to work. Cancel via R12's SIGTERM handler depends on the runner's signal-handling — if the runner is hard-stopped, the SIGTERM-then-SIGKILL escalation does not exit it; tool returns `{cancelled: false, reason: "signal_not_delivered_within_timeout", pid_file_unlinked: false}`. **Recovery**: the user calls `overnight_cancel(session_id, force=True)`, which performs the same signal escalation but unlinks `runner.pid` regardless of the outcome — so a fresh `overnight_start_run` can claim the lock. The user is responsible for `kill -9 <pid>` against the SIGSTOP'd process (the `pid` is still in the cancel response). Documented in `docs/mcp-server.md` as the manual recovery procedure.
- **In-flight upgrade attempted**: `cortex` install fails per R28 with a clear error naming the active session and the recovery options (wait for completion, or `overnight_cancel <session_id> --force`). The user can override with `CORTEX_ALLOW_INSTALL_DURING_RUN=1` if they consciously accept the risk that mid-run runners may break under the new IPC and escalations contracts.
- **Mixed-version cortex installs on the same machine** (e.g., a global `uv tool install` + a stale `~/.local/bin/cortex` symlink): formally **out of scope** for this ticket. R9's `MAX_KNOWN_RUNNER_PID_SCHEMA_VERSION = 1` will reject a hypothetical v2-written runner.pid as "unknown future schema", routing R8 through the unlink-and-retry path against a live process; this is a real hazard but only manifests when v2 of the schema actually ships, at which point a follow-up ticket addresses the version-coexistence policy. For this ticket: a single-cortex-install assumption holds (consistent with personal-tooling distribution per `requirements/project.md`).
- **`escalations.jsonl` migration encounters a session that wrote to the old repo-level path before the upgrade**: There is no merge / dual-read path. R22 deletes the legacy file. Past escalations stop being introspectable via `overnight_logs --files=escalations`. This is acceptable because escalations are operational state, not historical record — the orchestrator's cycle-breaking logic uses them only within an active session.

## Technical Constraints

- **File-based state architecture** (requirements/project.md): the MCP server reads from disk and signals the runner via PID/PGID. No in-process mutable state in the server. Same applies to the cursor codec — opacity is a protocol mandate, not just a style choice.
- **Atomicity**: all state writes use `tempfile + durable_fsync + os.replace`, except R8 which adds `O_CREAT|O_EXCL` for the initial runner.pid claim (single-shot, no in-place updates).
- **stdio protocol discipline**: stderr-only logging (R2) is non-negotiable — stdout corruption breaks JSON-RPC.
- **MCP 25 K token output cap is a hard error**: tools enforce pagination via opaque cursors; no `_meta` escape hatch.
- **Per-session single-coroutine dispatch invariant** (referenced in deferral.py:402+ TOCTOU note): the per-session escalations migration preserves this invariant; `_next_escalation_n` remains race-safe under the same dispatch model.
- **macOS detached-runner survival on logout**: empirically works via `start_new_session=True` + reparenting to PID 1 / launchd, but not spec-guaranteed; documented as a known caveat in `docs/mcp-server.md`.
- **Runner uses `--dangerously-skip-permissions`**: the MCP layer cannot relax this; the operational gate is R13's `confirm_dangerously_skip_permissions` arg + warning sentence.
- **Token-budget headroom**: default `overnight_logs` page size is 100 lines (~5 K tokens); `overnight_status` is inline (~1.5 K tokens cap-tested as char-length proxy); `overnight_list_sessions` returns at most ~4 K tokens at default settings. Coarse char-length proxies are used in tests; the 25 K hard cap is enforced by Claude Code at runtime.
- **Async-correct handlers (R29)**: stdio MCP servers serialize on the event loop. `cli_handler.py` is synchronous; tools wrap every blocking call in `asyncio.to_thread`. R8's atomic-claim race fix would otherwise be indistinguishable from "dispatch is serialized" — the integration test in R27 covers both.
- **Cursor opacity is server-internal-only**: the `{offset, file_size_at_emit}` encoding is documented in `cortex_command/overnight/cursor.py` and `research.md`, but **not** in `docs/mcp-server.md` (R23) — clients see only the `cursor_invalid` invalidation behavior. The MCP spec mandates opacity; documentation that leaks the structure invites coupling and defeats the invariant.
- **In-flight upgrade prevention (R28)**: the IPC and escalations contracts are not backwards-compatible during a release transition; the install-time guard prevents mid-run upgrades. This is the project's single-machine-personal-tooling assumption; multi-machine fleets would require a different policy (out of scope per Non-Requirements).
