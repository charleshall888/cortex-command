# Plan: instrument-eventslog-aggregation-for-turns-and-cost-per-tier

## Overview

Extend `claude/pipeline/metrics.py` with stdlib-only functions that discover and pair dispatch events, compute per-`(model, tier)` aggregates (cost, turns, and error counts co-located in one payload), and emit a single new top-level `model_tier_dispatch_aggregates` key in `lifecycle/metrics.json`. Add `--since DATE` and `--report tier-dispatch` CLI flags. No emitter changes — `dispatch_start` already carries `complexity`/`criticality`/`model` and is written immediately before `dispatch_complete` in the same pipeline log. Pairing is file-local (walker is invoked per log, not on a concatenated list) and uses a per-feature FIFO queue of unmatched starts so same-feature concurrent dispatches are handled correctly. All existing `metrics.json` keys remain byte-compatible; the aggregator is additive.

### Task 2: Implement pair_dispatch_events() with per-file, FIFO-queue semantics
- **Files**: `claude/pipeline/metrics.py`, `claude/pipeline/tests/test_metrics.py`
- **What**: Walk a single pipeline log's events in deterministic order; pair each `dispatch_complete` or `dispatch_error` to the OLDEST unmatched preceding `dispatch_start` for the same feature (FIFO, not "most recent"). Emit one paired record per completion/error. Orphan `dispatch_complete`/`dispatch_error` (no preceding unmatched start) → emit record with `tier=None`, `model=None`, `untiered=True` and surface via `warnings.warn`. Daytime-schema `dispatch_complete` (has `mode`, `outcome`, or `pr_url` fields — use positive check, not absence of numeric fields) is skipped entirely, not untiered. `dispatch_progress`, `dispatch_retry`, and any other `dispatch_*` event types are skipped by the walker's explicit `event in {"dispatch_start", "dispatch_complete", "dispatch_error"}` filter.
- **Depends on**: [1]
- **Complexity**: complex
- **Context**: Signature: `pair_dispatch_events(events: list[dict]) -> list[dict]`. Caller passes events from ONE log file at a time — file-local pairing is enforced by the caller (Task 6), not by this function's signature. Each returned record: `{"ts": str, "feature": str, "model": str | None, "tier": str | None, "outcome": "complete" | "error", "cost_usd": float | None, "num_turns": int | None, "error_type": str | None, "untiered": bool}`. `tier` is the `complexity` field from the matched `dispatch_start` (values `"trivial"`, `"simple"`, `"complex"` per `TIER_CONFIG` at `dispatch.py:118-122`). Sort events by `(ts, event_priority)` where `event_priority` is `0` for `dispatch_start` and `1` for `dispatch_complete`/`dispatch_error` — deterministic tie-break when two events share the same `ts` ensures starts are processed before completes at the same timestamp. Data structure: `unmatched_starts: dict[str, collections.deque[dict]]` keyed by feature; `dispatch_start` appends right, completion `popleft()`s. This handles same-feature retry storms (real log `lifecycle/sessions/overnight-2026-04-01-2112/pipeline-events.log` lines 2/4/6/8/10/12 show 6 starts within 33ms for one feature) by preserving count even when dispatches run concurrently. `dispatch_start` at `dispatch.py:442-452` emits `{event, feature, complexity, criticality, model, effort, max_turns, max_budget_usd}`; `dispatch_complete` at `dispatch.py:510-516` emits `{event, feature, cost_usd, duration_ms, num_turns}`; `dispatch_error` at `dispatch.py:528-533 / 553-558 / 571-577` emits `{event, feature, error_type, error_detail}`. Test events constructed inline in test code — no JSONL fixtures needed for this function. Five test cases under `TestPairDispatchEvents`: (a) `test_basic_pair_complete` — one start + one complete; (b) `test_interleaved_different_features` — feature A start, feature B start, feature A complete, feature B complete: each complete pairs to its own feature's start; (c) `test_same_feature_retry_storm` — 3 starts for same feature in sequence, then 2 completes: first complete pairs to start[0], second complete pairs to start[1], start[2] remains unmatched (no orphan emitted for unmatched starts; only unmatched completes become untiered); (d) `test_orphan_dispatch_complete_untiered` — lone complete, no preceding start, assert `untiered=True`, `tier=None`, `model=None`, `warnings.warn` emitted (use `pytest.warns`); (e) `test_daytime_schema_skipped` — complete with `mode`/`outcome`/`pr_url` fields → omitted from output entirely (not untiered) via positive field check; (f) `test_dispatch_progress_noise_ignored` — start, 5 `dispatch_progress` events, complete: one paired record emitted, progress events silently skipped; (g) `test_timestamp_tie_deterministic` — start and complete with identical `ts`, assert pairing succeeds via the `(ts, event_priority)` tie-break.
- **Verification**: `pytest claude/pipeline/tests/test_metrics.py -k pair_dispatch_events` — pass if exit 0
- **Status**: [x] pending

### Task 3: Implement compute_model_tier_dispatch_aggregates() with co-located error counts
- **Files**: `claude/pipeline/metrics.py`, `claude/pipeline/tests/test_metrics.py`, `claude/pipeline/tests/fixtures/dispatch_over_cap.jsonl`
- **What**: Group all paired records (both `outcome="complete"` and `outcome="error"`) by `(model, tier)`; compute per-bucket aggregates. Cost/turn stats are computed from `outcome="complete"` records within each bucket; `error_counts` is a nested dict of `{error_type: count}` computed from `outcome="error"` records within the same bucket. Tuple keys serialize as the string `"model,tier"` for JSON compatibility. Untiered records bucket under the literal key `"untiered,untiered"` with `is_untiered=True`, `budget_cap_usd=None`, `over_cap_rate=None`, `turn_cap_observed_rate=None`.
- **Depends on**: [2]
- **Complexity**: complex
- **Context**: Signature: `compute_model_tier_dispatch_aggregates(paired: list[dict]) -> dict[str, dict]`. Per-bucket field set: `{n_completes: int, n_errors: int, num_turns_mean: float | None, num_turns_median: float | None, num_turns_p95: float | None, max_turns_observed: int | None, p95_suppressed: bool, estimated_cost_usd_mean: float | None, estimated_cost_usd_median: float | None, estimated_cost_usd_max: float | None, budget_cap_usd: float | None, over_cap_rate: float | None, turn_cap_observed_rate: float | None, error_counts: dict[str, int], is_untiered: bool}`. Rules: when `n_completes >= 30`, set `num_turns_p95 = statistics.quantiles(turns, n=100, method='inclusive')[94]`, `p95_suppressed=False`, `max_turns_observed=None`; when `n_completes < 30`, set `num_turns_p95=None`, `p95_suppressed=True`, `max_turns_observed=max(turns)`. When `n_completes == 0`, cost/turn stats are all `None` and `p95_suppressed=False`. `budget_cap_usd` and `turn_cap_observed_rate` lookup `TIER_CONFIG[tier]` (imported from `claude/pipeline/dispatch.py` — verified importable: `TIER_CONFIG` is a module-level dict at `dispatch.py:118-122`). Untiered bucket emits `None` for budget_cap_usd/rates. `over_cap_rate = sum(1 for r in completes if r["cost_usd"] > budget_cap) / n_completes` when `n_completes > 0` and `budget_cap is not None`. Use `statistics.fmean` and `statistics.median`. `error_counts` is populated even when `n_errors == 0` (as empty dict `{}`). Novel `error_type` strings pass through under their raw value without erroring. Tests under `TestModelTierAggregates`: (a) n=3 bucket with inline records → assert `p95_suppressed=True`, `max_turns_observed=<expected>`, `num_turns_p95=None`; (b) programmatically-built n=100 bucket `[{ts:..., outcome:"complete", model:"sonnet", tier:"simple", cost_usd:i*0.5, num_turns:i}]` → assert `num_turns_p95` equals the 95th percentile and `p95_suppressed=False`; (c) `fixtures/dispatch_over_cap.jsonl` with 4 `(opus, complex)` dispatches where 2 exceed the $50 cap → assert `over_cap_rate == 0.5`; (d) bucket with 2 complete + 3 error records (mix of `agent_timeout` and `api_rate_limit`) → assert `n_completes=2`, `n_errors=3`, `error_counts={"agent_timeout": 2, "api_rate_limit": 1}`, `num_turns_mean` computed from the 2 completes; (e) bucket with 0 complete + 2 errors → assert `n_completes=0`, `error_counts={"agent_timeout": 2}`, all cost/turn stats `None`; (f) `(sonnet, simple)` and `(opus, complex)` independence assertion; (g) untiered bucket → `is_untiered=True`, `budget_cap_usd=None`, `over_cap_rate=None`.
- **Verification**: `pytest claude/pipeline/tests/test_metrics.py -k model_tier_aggregates` — pass if exit 0

## Veto Surface

- **Walker tie-break ordering**: I put `dispatch_start` before `dispatch_complete`/`dispatch_error` at identical `ts` to ensure a same-ts start-complete pair resolves correctly. Alternative: lexicographic on `event` field (`dispatch_complete` < `dispatch_error` < `dispatch_start`) would put completes first, which requires a pre-populated starts deque from a prior-ts start — correct for 1:1 flows but fails when a same-ts start+complete is the only pair for that feature. Flag if you want a different tie-break.
- **Untiered in top-level aggregate vs separate**: I keep untiered in `model_tier_dispatch_aggregates` under `"untiered,untiered"` with `is_untiered=True`, and add a top-level `untiered_count` sibling field for programmatic consumers that want the count without iterating buckets. Alternative: drop untiered entirely from the aggregate and surface only via `untiered_count` + CLI banner. Flag if you want the drop-and-count shape.
- **CLI `error_counts_summary` column format**: I chose `"timeout:3,rate_limit:1"` (drop the `agent_` prefix from the canonical names for width). Alternative: full names `"agent_timeout:3,api_rate_limit:1"`. Flag if you want the full names or a different condensation rule.

## Scope Boundaries

Mirrors the spec's Non-Requirements verbatim:
- No emitter changes (`dispatch.py`, `throttle.py` untouched)
- No rate-limit aggregation; `throttled_dispatch` dead-code removal is on ticket `#098`
- No reads of per-feature `lifecycle/{feature}/events.log` from the new aggregator
- No dashboard widget, no model-level token decomposition, no retroactive backfill
- No changes to existing `metrics.json` keys; only additive (single new top-level key `model_tier_dispatch_aggregates`, plus optional `model_tier_dispatch_aggregates_window` and `untiered_count` when conditions trigger)
- No CI hook, no scheduled run, no pre-commit integration
- No threshold-based abort on high untiered rate (spec does not require it; the orphan banner makes the count visible to the operator)
- No fix to recovery-site hardcoded complexity (`merge_recovery.py:336`, `conflict.py:332`, `integration_recovery.py:220`) — documented as Open Decision in spec; aggregator attributes recovery dispatches to the hardcoded tier, which may skew those `(model, tier)` buckets until a separate cleanup ticket threads the parent feature's tier through.
- No filter application to per-feature metrics — `--since` filters only the new `model_tier_dispatch_aggregates` pipeline. The output surface makes this asymmetry explicit via `model_tier_dispatch_aggregates_window` and the CLI window header.
