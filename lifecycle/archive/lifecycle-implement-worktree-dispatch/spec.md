# Specification: lifecycle-implement-worktree-dispatch

## Problem Statement

When the user runs `/lifecycle` interactively (not overnight) and picks "Create feature branch" at the implement-phase pre-flight, the lifecycle skill runs `git checkout` on the main session. The user keeps multiple parallel Claude sessions in the same git checkout of this repo, so any other session that's mid-task suddenly has its commits land on the freshly-switched-to feature branch. This silently corrupts work across the user's parallel sessions. The user wants the implement phase to isolate its branch state so that other sessions in this repo are unaffected — without losing the PR-based workflow that "Create feature branch" was designed for.

7. **R7: In-flight `.dispatching` marker prevents concurrent dispatch via atomic create-or-fail + PID liveness check (must-have).** Before dispatching the agent, the main session writes `lifecycle/{feature}/.dispatching` **atomically** (create-or-fail; if the file already exists, the create fails and the dispatch is refused). The atomic-create mechanism is a bash `set -C` (noclobber) redirect: `(set -C; printf '%s\n%s\n%s\n' "$$" "$LIFECYCLE_SESSION_ID" "$(date -u +%Y-%m-%dT%H:%M:%SZ)" > lifecycle/{feature}/.dispatching) 2>/dev/null`. After the Agent call returns (success or escalation), the main session removes the marker via `rm -f lifecycle/{feature}/.dispatching`.

   - If the marker file does not exist: pass — proceed.
   - If the marker file exists, read the PID (line 1) and run `ps -p $PID > /dev/null 2>&1`. If the PID is alive: the dispatch is in flight. Refuse to enter the implement phase, surface the marker contents to the user, and tell them to wait for the dispatching session to finish or to manually `rm` the marker if they're certain it's orphaned.
   - If the marker file exists but the PID is dead (the parent shell exited without removing the marker): the marker is orphaned. Display the marker contents and prompt via `AskUserQuestion` (default: clean up and proceed). On accept, `rm` the marker and proceed normally.

## Non-Requirements

- This feature does NOT modify `review.md` or `complete.md` reference files. Review and complete run unchanged inside the dispatched worktree agent.
- This feature does NOT change the existing per-task `Agent(isolation: "worktree")` behavior in `implement.md §2` for the "Implement on main" or "Create feature branch" paths. Per-task worktree dispatch remains the default for those paths.
- This feature does NOT introduce a new branch-naming convention. The branch name `worktree/agent-{slug}` is dictated by the existing `cortex-worktree-create.sh` hook (line 30: `BRANCH="worktree/$NAME"`) when the Agent tool is invoked with `name: "agent-{slug}"`.
- This feature does NOT sync the events.log from worktree back to main. Main's events.log captures only its own dispatch + dispatch_complete events; the worktree's events.log (with implementation/review/complete details) lands on main only when the PR merges. This divergence is the accepted cost of keeping main on main.
- This feature does NOT add a budget cap, time cap, or kill switch for the dispatched agent. Interactive Claude Code sessions have no equivalent of overnight `budget_cap`; the user can interrupt the parent session if a runaway dispatch occurs.
- This feature does NOT modify the cleanup hook (`hooks/cortex-cleanup-session.sh`). The hook's existing `worktree/agent-*` filter is reused as-is.
- This feature does NOT promote the new option to default position in a follow-up automatically. Promotion (re-ordering the AskUserQuestion list) is a manual two-line edit if the user wants it later.
- This feature does NOT enforce a tier or criticality restriction. The "Implement in worktree" option is offered for all tier/criticality combinations. The dispatching prompt's model selection (sonnet for low/medium, opus for high/critical) handles model scaling.

## Edge Cases

- **User picks "Implement in worktree" but the current branch is already `worktree/agent-*`**: Pre-flight detects "current branch is not main", skips the prompt entirely (existing implement.md behavior at line 18), and proceeds to §2 Task Dispatch on the current branch. The user is already inside a dispatched agent, so no nested dispatch happens.
- **User picks "Implement in worktree" but the dispatching prompt's `name` parameter would produce a branch that already exists**: `cortex-worktree-create.sh` line 33 fails fast with "worktree path already exists". The Agent call returns an error. Main session surfaces the error and tells the user to `git worktree remove .claude/worktrees/agent-{slug}` and `git branch -D worktree/agent-{slug}` before retrying.
- **`.dispatching` marker is orphaned (main session crashed before removing it)**: On the next `/lifecycle resume {feature}` invocation, SKILL.md Step 2's liveness check reads the PID from line 1 of the marker and runs `ps -p $PID`. The PID is dead → the marker is orphaned. Main displays the marker contents and prompts via `AskUserQuestion` (default: clean up and proceed). On accept, removes the marker and proceeds normally.
- **Two main sessions race to dispatch the same feature simultaneously**: Both sessions reach the atomic-create step. The `set -C` (noclobber) redirect succeeds for exactly one session; the second session's create fails (`>: cannot overwrite existing file`). The losing session surfaces the collision ("another session claimed this dispatch first; aborting") and exits /lifecycle without dispatching.
- **PID reuse after main crash**: Rare but possible — main crashes; the OS reuses its PID for an unrelated process. The marker's `ps -p $PID` liveness check returns alive (false positive), blocking legitimate future dispatches. Documented as a known sharp edge; the user's recourse is to `rm lifecycle/{feature}/.dispatching` manually after inspecting the marker contents and confirming the PID does not match a Claude Code process.
- **Concurrent `/lifecycle resume {feature}` from a third session while a dispatch is in flight**: The third session's SKILL.md Step 2 marker check refuses to proceed and tells the user to wait. No state corruption.
- **Inner agent calls AskUserQuestion despite the prompt instruction**: The agent is technically able to call the tool. The user MAY see the prompt in their main session's terminal (Agent tool may surface inner-agent prompts to parent). Documented as a known sharp edge in §1a — the inner agent's prompt explicitly forbids AskUserQuestion, and Claude is expected to follow.
- **Inner agent's review sub-task fails or hangs**: The review.md `review_verdict` event is not written. The dispatched agent returns with an escalation. Main session surfaces the failure context.
- **Inner agent runs complete.md and tests fail**: complete.md §1 reports failures and does not proceed to §4 Git Workflow. The agent returns with an escalation. The worktree (with the failed tests' state) is left alive for the user to inspect manually via `git worktree list`.
- **PR creation fails inside the inner agent (network, auth, conflict)**: The agent returns with an escalation. The worktree branch exists locally and uncommitted state is preserved for retry.
- **Inner agent hits cycle 2 CHANGES_REQUESTED**: Returns escalation context per R9. User can manually inspect the worktree, address the rework, and re-run /lifecycle from the worktree CWD if they want to continue the work themselves.

## Technical Constraints

- **TC1 — `Agent(isolation: "worktree")` tool isolation is verified**: Empirical probe (Research §OQ-3 deferred resolution) confirmed that Read, Write, and Bash inside an `Agent(isolation: "worktree")` sub-agent all resolve to the worktree CWD. The probe was run during the spec phase of this lifecycle and is documented in the Research artifact's OQ-3 resolution.
- **TC2 — Branch naming is dictated by the hook**: `claude/hooks/cortex-worktree-create.sh:30` hardcodes `BRANCH="worktree/$NAME"`. To get a `worktree/agent-{slug}` branch, the Agent call must pass `name: "agent-{slug}"`. The `agent-` prefix on the slug is what triggers the existing cleanup hook coverage (`hooks/cortex-cleanup-session.sh:46,59`).
- **TC3 — `cortex-worktree-create.sh` is not idempotent**: Line 33 fails fast on existing worktree paths. The dispatch flow must surface this error cleanly when a stale `agent-{slug}` worktree exists from a prior failed dispatch.
- **TC4 — Sequential inline dispatch loses fresh-context isolation AND within-batch concurrency**: Per R5, per-task work runs inline in the inner agent's own context (no sub-`Agent` calls at all), and tasks within a batch run one at a time. Both the "fresh sub-agent per task" property and the within-batch concurrency from implement.md §2b are forfeited. Between-batch dependency ordering is preserved — batch N still waits for batch N-1 commits to land. This is an explicit, documented quality regression, accepted because (a) the alternatives (nested worktrees, non-isolated sub-Agent calls) are either fragile or leak context across a CWD the inner agent cannot control, and (b) the dispatched agent's whole-feature context is bounded by a single PR's scope so context accumulation stays within manageable limits.
- **TC5 — `complete.md §4` already handles `worktree/{slug}` branches**: The binary check `main|master` vs anything else routes the dispatched agent to the push+PR path automatically. No changes to `complete.md` required. (Confirmed in Codebase Analysis.)
- **TC6 — `/pr` and `/commit` are branch-name-agnostic**: Both use `git branch --show-current` and operate on the current branch with no naming assumptions. (Confirmed in Codebase Analysis.)
- **TC7 — Permission inheritance**: The dispatched agent inherits the parent interactive session's permission set (per `docs/sdk.md:153`, interactive Agent sub-tasks do not get `bypassPermissions`). Any tool call that would normally trigger a `PermissionRequest` in the parent session may also trigger one from inside the dispatched agent, surfaced to the parent's terminal. The user must be aware that permission prompts may appear during dispatch. This is documented in §1a as a sharp edge.
- **TC8 — Events.log divergence is accepted**: Main's `events.log` does not see the inner agent's `phase_transition`, `task_complete`, `review_verdict`, or `feature_complete` events until the PR merges and the worktree branch lands on main. Tooling that inspects "recent feature_complete events" from main's checkout may under-count completed features during the window between dispatch and PR merge. Documented in §1a as a known limitation.
