# Specification: Remove dead throttled_dispatch wrapper; keep ConcurrencyManager acquire/release

## Problem Statement

The `throttled_dispatch()` wrapper in `cortex_command/overnight/throttle.py` and its supporting reactive concurrency-shrinkage layer (`ConcurrencyManager.report_rate_limit`/`report_success`, the internal `_total_rate_limits`/timestamp window state, the `throttle_backoff` event, and the `backoff_*` config knobs) is dead code: zero call sites in production, zero `throttle_backoff` events emitted across observed sessions, and a known deadlock at the MAX_5 tier when the dynamic cap collapses to zero. Live dispatch goes through `dispatch_task` directly, with rate-limit handling already routed through the pipeline-layer `api_rate_limit` → `pause_session` path. Removing the dead surface eliminates ongoing maintenance burden, latent deadlock risk, and a documentation lie in `requirements/multi-agent.md:45` that asserts adaptive shrinkage as a current behavior. The `ConcurrencyManager.acquire()`/`release()` semaphore and tier-based concurrency cap (`SubscriptionTier`-bound, 1–3 workers) are load-bearing and must be preserved verbatim.

## Edge Cases

- **`brain.py:236-237` deleted guard fires on `infrastructure_failure`, not 429**: Verified against `cortex_command/pipeline/dispatch.py` `classify_error`. The pre-deletion guard at brain.py was `if result.error_type == "infrastructure_failure" and manager is not None: manager.report_rate_limit()`. `dispatch.py` returns `infrastructure_failure` only for `CLIConnectionError` (the SDK CLI subprocess connection failed) and routes it via `ERROR_RECOVERY` to `pause_human` (operator escalation). 429s are classified separately as `api_rate_limit` and routed to `pause_session`. The deleted call was therefore a misnamed observability hook firing on CLI-down events, not on rate limits, and never participated in the actual 429 path. Removing it is safe; no observability path for 429s is lost (none existed via this guard pre-deletion).
- **Transient 429 during any agent dispatch (brain or feature)**: Already routed via `dispatch_task` → `dispatch.py:320-321` regex match → `error_type=api_rate_limit` → `dispatch.py:246` `ERROR_RECOVERY → pause_session` → `retry.py:379-399` zero-retry session pause. **Pre- and post-deletion behavior is identical for 429 dispatch outcomes.** What changes is removal of dead code that never observed 429s anyway.
- **Morning-report label on rate-limit-induced pauses**: Pre-existing condition: `retry.py:398` hardcodes `error_type="budget_exhausted"` on the `pause_session` branch regardless of upstream classification, so 429-induced pauses surface to the operator as "API budget exhausted." This deletion makes 429 → `pause_session` the dominant path for transient API hiccups, so the latent mislabel becomes operator-load-bearing. **Resolution**: this PR threads the actual `paused_reason` end-to-end via R15-R19 — `retry.py:398` propagates the upstream `error_type` ("budget_exhausted" or "api_rate_limit"), `feature_executor.py:662` and `orchestrator.py` recognize a session-halting error-type set rather than a literal string, and `report.py`/`runner.py` emit distinct user-facing text for each cause. After this PR, an operator reading the morning report sees "Session paused: API rate limit hit" for transient 429s and "Session paused: API budget exhausted" for genuine budget exhaustion, and can decide to resume immediately vs wait for a billing-window reset.
- **MAX_5 tier post-deletion fragility**: At `SubscriptionTier.MAX_5` the cap is 1 worker. Pre-deletion, this tier was the sole plausible deadlock surface for `throttled_dispatch`; the wrapper was never wired to it in production. Post-deletion, MAX_5 single-tenancy means a single transient 429 = immediate `pause_session`. MAX_5 users are **strictly more 429-fragile post-deletion than pre-deletion** (because no in-process layer absorbs even briefly). The deletion is justified because the wrapper had zero callers, not because it was working correctly at MAX_5; the framing "broken at MAX_5" overstates a configuration mismatch. Accepted residual risk: MAX_5 operators should expect more frequent pause-and-resume cycles during Anthropic API hiccups.
- **`mock_manager.stats = {}` in test fixtures**: Verified that no test asserts on the removed stats keys (`total_rate_limits`, `reductions`, `restorations`); existing tests use `mock_manager.stats = {}` (empty dict). After R6 drops the orchestrator's read of `manager.stats`, these fixture writes become harmless dead writes — no behavior change, no fixture updates required.
- **`ThrottleConfig(...)` test constructions**: Verified that zero tests construct `ThrottleConfig` directly; all tests mock `load_throttle_config`. The earlier hedge in research about "test fixtures that pass `backoff_*` overrides" turned out to be unneeded — no such fixtures exist. The `TypeError`-on-construction risk applies only to any future-added test that does construct `ThrottleConfig` with the removed kwargs.
- **Historical `batch-*-results.json` files contain `"throttle_stats": {...}` blocks**: These are read-only artifacts; no consumer in `morning_report` or `dashboard` depends on the key (verified). Forward-compat: future readers iterating historical sessions should treat the field as optional.
- **`test_overnight_state.py` keeps `"throttle_stats"` as a sample fixture key**: The generic `extra_fields` round-trip test passes its own literal payload; unaffected by the deletion. The label remains in the test corpus but is purely a test-data string, not a behavioral coupling.
- **`throttle.py` external imports**: Anything that previously imported `throttled_dispatch` from `cortex_command.overnight` will break with `ImportError`. Verified: no external imports exist (the only references are the export at `__init__.py:58` and the doc reference in `brain.py`).
- **Claude Code CLI's internal 429 retry behavior is unverified**: `claude_agent_sdk` is a subprocess wrapper around the `claude` binary; the codebase contains no retry logic of its own. If the CLI does not retry 429s internally, every transient 429 escalates to `pause_session`. **Risk accepted by user.** Resume-from-pause is recoverable; per the Morning-report edge case above, after R15-R19 land the pause reason text correctly distinguishes "API rate limit" from "API budget exhausted," so an operator can decide between immediate resume and waiting for a reset window.

## Technical Constraints

The cap is hard-limited to 1–3 workers per `requirements/multi-agent.md:75` ("hard limit enforced by `ConcurrencyManager`; not overridable at runtime").
- **Pipeline-level rate-limit handling is the sole rate-limit response post-deletion**: `dispatch.py:246` (`"api_rate_limit": "pause_session"`) and `dispatch.py:320-321` (regex match on `("rate_limit_error", "rate limit", "too many requests")`) classify on the first 429. `retry.py:379-399` triggers immediate session pause without retry. - **`claude_agent_sdk` (subprocess wrapper) does not retry 429s internally**: Verified in `.venv/.../claude_agent_sdk/_internal/{client,query}.py` — zero `retry`/`429`/`max_retries` references. Implementation must search for any `ThrottleConfig(...)` constructions that pass the removed fields and update them.
