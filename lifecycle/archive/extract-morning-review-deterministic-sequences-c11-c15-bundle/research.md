## Web Research

### jq atomic state update

Canonical pattern (strong consensus across multiple tutorials, gists, and the jqlang issue tracker):

```bash
tmp=$(mktemp)
jq '.field = "value"' state.json > "$tmp" && mv "$tmp" state.json
```

- Anti-pattern (textbook): `jq '...' f.json > f.json` — the shell truncates the file before jq reads it. jq has no native in-place edit (`jqlang/jq#105` is still open).
- `mv` within the same filesystem is atomic per POSIX `rename(2)`. The morning-review SKILL.md pattern is correct.
- `mktemp` over hardcoded temp paths — respects `$TMPDIR` (matters for sandbox restrictions).
- `sponge` (moreutils) is a cleaner one-liner but adds a non-default dependency and is not atomic-rename.

### git-worktree GC canonical shape

(Reference: `dviersel/e96650a34d962b4eb17c442081163536` cleanup gist + git-scm docs + `anthropics/claude-code#26725` "Stale worktrees never cleaned up" — Anthropic itself has this open issue.)

- `git worktree remove` first; `git worktree prune` last. Prune is reactive cleanup for orphaned admin entries when directories were deleted out-of-band; not a substitute for `remove`.
- `prune` refuses locked worktrees by default. `remove` requires `--force --force` (twice) for locked worktrees. Path-prefix sweepers should not blanket-`--force`.
- Standard safety guards: skip the main worktree; refuse on `git diff --quiet` / staged changes; refuse on unpushed commits (`git rev-list HEAD..@{upstream}`).
- **Untracked files are NOT considered "dirty" by `git worktree remove` (no `--force`).** A clean worktree containing untracked-but-uncommitted user work is silently removed. Demo command output, scratch notes, manual experiments — all gone if the path matches the regex.
- Path-prefix matching: parse `git worktree list --porcelain` for `worktree <path>` lines and test prefix. Avoid raw `find` over `$TMPDIR` — it can pick up directories that aren't worktrees.

### Parallel CLI invocation

For "close N backlog items" (independent, want "any failed → loud"):

- `xargs -P` is the boring-correct portable choice. Exit semantics: 123 = at least one child exited 1–125; 124 = any child exited 255 (xargs short-circuits); no per-child failure aggregation — you only know "≥1 failed."
- Bash `&` + `wait` aggregating per-pid: `rc=0; for p in "${pids[@]}"; do wait "$p" || rc=1; done` — gives per-item rc.
- GNU parallel (`--joblog`, `--keep-order`, `--retries`) is overkill unless ordered output or per-job audit trail is needed; non-default macOS dependency.
- Output interleaving is the default for parallel runs. Per-item log files or per-line prefixes are required for clean reporting.

### Subcommand vs flag-mode CLI design

Web guidance is directional, not prescriptive (clig.dev; ESR; HN 35789781):

- Flag is correct when the new mode shares ≥80% of the existing tool's machinery and only changes a small part of behavior.
- Subcommand (or sibling tool) when modes diverge enough to need their own flag namespaces, exit semantics, or argument shapes.
- Applied to C14: a `--preflight` flag is structurally cheap (early-return guard clause before the rebase loop runs). The contested question is not technical — it's whether the existing pipeline-critical binary should host two contracts.

### Idempotency in wrapper scripts

Canonical pattern (arslan.io, peterlyons): guard clauses, not retry loops. Return 0 when post-condition is already met (Solaris `patchadd` is the textbook anti-pattern — it returns non-zero on "already installed"). The morning-review re-run is itself the retry; idempotency means re-running is safe.

### Plugin distribution PATH (CRITICAL)

**Anthropic's plugin reference does NOT list `bin/` as a recognized plugin component directory.** Documented components: `skills/`, `commands/`, `agents/`, `hooks/`, `mcp/`. Plugin authors who ship CLIs are expected to handle PATH out-of-band (npm global bin, mise/asdf shims, or shell-rc edits). The plugin Errors tab surfaces "Executable not found in $PATH" as a known failure mode. **Do not depend on `/plugin install` extending the user's PATH.**

## Adversarial Review

### Failure modes the prior agents missed

1. **Atomicity hallucination**: the tradeoffs agent argued for Python form because "the SKILL.md `mv` pattern is not actually atomic." `man 2 rename` says rename is atomic on POSIX. The conclusion (Python form) is correct but the reasoning was wrong; the real argument is **schema/transition bypass** of `state.py:transition()`.
2. **Bundle justification is fiction**: "single acceptance test (one morning-review run)" doesn't exercise all five candidates on a normal morning. Each candidate is conditional on different session state. There is no automated end-to-end morning-review test in `tests/`. Manual fixture replay is the actual acceptance bar.
3. **PATH invariant is false**: `/plugin install` does NOT extend the user's PATH. Plugin docs don't promise it. The author's machine has plugin bin/ on PATH via personal shell rc, not via cortex install. C15's "single PATH-resolved invocation" simplification breaks for users without manual setup.
4. **walkthrough.md drift fix is already done**: ticket body claim of 5 occurrences is stale. The C14 "Repoint as part of C14" line is a no-op. Anyone planning from the ticket text without re-verifying loses cycles.
5. **`~/.local/bin/generate-backlog-index` is not in the current fallback chain**: the ticket promises to remove it but it isn't there.
6. **`bin/cortex-git-sync-rebase` line 2 still has cosmetic comment-line drift** (`# git-sync-rebase.sh`) — not parity-blocking, but worth a one-line fix in any C14-touching ticket.

### Security / data-loss concerns

- **C12 worktree GC silent destruction of untracked work**: `git worktree remove` (no `--force`) refuses dirty worktrees but does NOT consider untracked-but-uncommitted files dirty. A clean worktree containing scratch notes, demo command output, or experimental files matching the regex is silently removed. **Mitigation**: require a `.cortex-demo-marker` file (or similar) before removal; path-prefix is necessary-not-sufficient.
- **C12 cross-session worktree collision**: the regex `demo-overnight-...` is a project convention, not a guarantee. A worktree from a different cortex user (shared laptop, container reuse) WILL match and be removed. Same mitigation: marker file.
- **C11 concurrent writers**: two simultaneous morning-reviews (two Claude sessions) race on pointer-file `rm -f` after state-file write. Not a data loss event but a potential state-machine-confusion event. A Python helper using `flock` or `O_EXCL` closes the gap; bash `mv` cannot.
- **C15 dual-PATH cortex-generate-backlog-index**: editable install + stale plugin install both on PATH — first wins, possibly stale Python. Generic foot-gun for all `cortex-*` shims, not specific to C15.
