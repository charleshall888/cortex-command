# Research: investigate-overnight-morning-review-git-sync-gaps

## Codebase Analysis

### Git Operations Inventory

The overnight runner and morning review perform git operations across three branches: local `main`, the integration branch `overnight/{session_id}`, and per-feature branches `pipeline/{feature}`. The core finding is that **local `main` accumulates commits that are never pushed**, while the integration branch is pushed and merged via PR — creating a divergence.

### Overnight Runner Git Flow (runner.sh, plan.py)

**Session bootstrap** (`claude/overnight/plan.py` lines 358–391):
- `git worktree prune --expire now` — cleanup stale metadata
- `git branch -D overnight/{session_id}` — delete stale branch if exists
- `git worktree add $TMPDIR/overnight-worktrees/{session_id} -b overnight/{session_id}` — creates integration branch from current `main` HEAD

**Pre-runner artifact commit** (`skills/overnight/SKILL.md` Step 4):
- Overnight skill commits batch spec extractions to `main` before launching the runner

**Per-round feature work** (`claude/overnight/batch_runner.py`, `claude/pipeline/worktree.py`, `claude/pipeline/merge.py`):
- Feature worktrees branch off integration branch: `git worktree add ... -b pipeline/{feature} overnight/{session_id}`
- Feature merges go into integration branch: `git merge --no-ff pipeline/{feature} into overnight/{session_id}`
- Feature worktrees and branches are deleted after merge
- **No pushes during the round loop**

**Post-loop artifact commit** (`runner.sh` lines 984–1000):
```bash
(
    cd "$REPO_ROOT"    # ← main repo, checked out to `main`
    git add lifecycle/sessions/${SESSION_ID}/ lifecycle/*/research.md lifecycle/*/spec.md
    git add lifecycle/*/plan.md lifecycle/*/agent-activity.jsonl lifecycle/pipeline-events.log
    git add backlog/index.md backlog/archive/
    git diff --cached --quiet || git commit -m "Overnight session ${SESSION_ID}: record artifacts"
)
```
**This commits to local `main`, not the integration branch.** The subshell `cd "$REPO_ROOT"` switches to the main repo (on `main`), not the worktree.

**PR creation** (`runner.sh` lines 1109–1162):
- `git push -u origin $INTEGRATION_BRANCH` — pushes integration branch only
- `gh pr create --base main --head $INTEGRATION_BRANCH` — creates PR
- **`main` is NOT pushed**

**Morning report commit** (`runner.sh` lines 1206–1211):
- Another `cd "$REPO_ROOT"` subshell commits morning report files to local `main`
- Integration branch pushed again: `git push origin $INTEGRATION_BRANCH`
- **`main` still NOT pushed**

**Cleanup** (`runner.sh` lines 1283–1344):
- `git worktree remove --force $WORKTREE_PATH` — removes integration worktree
- `git branch -D $INTEGRATION_BRANCH` — deletes local integration branch
- Local `main` left with unpushed artifact + morning report commits

### Morning Review Git Flow (skills/morning-review/)

**Step 5 — Pre-commit sync** (`SKILL.md` lines 112–121):
1. `git fetch origin`
2. `git rev-list HEAD..origin/main --count` — check for remote changes
3. If count > 0: `git pull --rebase origin main`
4. `/commit` — commits events.log and backlog changes to local `main`

**Step 6 — PR merge** (`references/walkthrough.md` Section 6):
1. `gh pr merge {number} --merge --delete-branch` — merges PR on GitHub
2. `git worktree remove --force {worktree_path}` — cleanup
3. **No `git pull` after merge** — local `main` is not synced with remote

### The Divergence Mechanism

The divergence follows a specific causal chain:

1. **Runner Phase D** — artifact commit lands on local `main` (line 995)
2. **Runner Phase F** — morning report commit lands on local `main` (line 1210)
3. **Neither is pushed** — only the integration branch is pushed
4. **Morning review Step 5** — fetches and checks for remote changes **before** Step 6 merges the PR. At this point `origin/main` doesn't yet have overnight features, so the rebase is a no-op or handles only unrelated changes
5. **Morning review Step 5** — adds another commit (events.log, backlog) to local `main`
6. **Morning review Step 6** — `gh pr merge` advances remote `main` with overnight feature work
7. **No post-merge sync** — local `main` has 3+ artifact commits; remote `main` has the PR merge commit. They share a common ancestor but have diverged independently.

### Conflict Pattern

When `git pull --rebase` is eventually run, conflicts arise in files that were:
- Committed to local `main` during artifact recording (plan.md files generated by the overnight skill's batch spec extraction)
- Also modified by the overnight feature branches and merged into the integration branch via PR

The specific conflict files are `lifecycle/*/plan.md` — both the local artifact recording and the overnight feature branches generate these files.

### What Gets Pushed vs. Stays Local

| Operation | Branch | Pushed? |
|-----------|--------|---------|
| Feature worktree commits | `pipeline/{feature}` | No (deleted after merge) |
| Feature merges into integration | `overnight/{session_id}` | Yes (line 1111) |
| Artifact commit (Phase D) | `main` (local) | **Never** |
| Morning report commit (Phase F) | `main` (local) | **Never** |
| Morning review `/commit` | `main` (local) | **Never** |
| PR merge | remote `main` only | N/A (GitHub API) |

### Retry Side Effects

Each re-run of runner.sh can produce additional artifact commits on local `main`:
- The `git add`/`git diff --cached`/`git commit` sequence in Phase D has no idempotency check against the session ID
- If files changed on disk between retries, a new commit is created
- Failed attempts that die after Phase D but before Phase E leave artifact commits on local `main` with no corresponding PR

### Alternative Approaches

**Approach A — Push main at session boundaries**: Add `git push origin main` after the artifact commit (Phase D) and morning report commit (Phase F). Pros: simple, keeps remote in sync. Cons: if remote has diverged (e.g., manual pushes during overnight), the push may fail. Needs a fetch+rebase before push, adding complexity.

**Approach B — Commit artifacts to the integration branch instead of main**: Move the artifact recording subshells from `cd "$REPO_ROOT"` to the worktree. Pros: artifacts travel with the PR and merge atomically. Cons: the integration branch worktree may not have the same file layout as main for all globbed paths; needs verification.

**Approach C — Post-PR-merge sync step in morning review**: Add a `git pull --rebase origin main` + `git push origin main` after Step 6 (PR merge). Pros: targeted fix at the exact divergence point. Cons: conflicts still possible if artifact files overlap with feature work.

**Approach D — Combine B + C**: Commit artifacts to integration branch (eliminates the pre-PR local-only commits), and add post-merge sync in morning review (catches any remaining divergence from the morning review commit itself). Pros: addresses both root causes. Cons: most complex, but most complete.

**Approach E — Auto-resolve lifecycle artifacts**: For post-merge rebase conflicts in `lifecycle/*/plan.md` and similar generated files, use `git checkout --theirs` since the overnight version (from the PR) is authoritative. This could be a merge driver or a scripted resolution step.

## Open Questions

- Should artifact recording (runner.sh Phase D) commit to the integration branch instead of main, so artifacts travel with the PR? **Resolved: Yes — move artifacts to integration branch.** This eliminates the root cause of divergence.
- If artifacts stay on main, should runner.sh push main after committing? **Resolved: N/A — artifacts moving to integration branch makes this moot.**
- Should morning review add a post-PR-merge sync step (fetch + rebase + push)? **Resolved: Yes, with selective auto-resolve.** Must be careful about what gets auto-resolved — spec phase will design a strategy that distinguishes known-generated files from user-authored content.
- How should the retry scenario be handled? **Resolved: moot if artifacts move to integration branch.** The integration branch is session-scoped, so retry accumulation on local main goes away. No separate handling needed.
