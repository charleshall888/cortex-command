## Veto Surface

- **Cross-tier coupling in scope (R8)**: widening `cortex init` allowWrite from `lifecycle/sessions/` to `lifecycle/` is a CLI-tier change shipped IN this ticket because the plugin's runner-only hooks fail under interactive sandboxed sessions without it (Adversarial §6). Spec resolved this; flagged for late-stage reconsideration only.
- **No PATH-launcher wrapper (research §5 dropped)**: spec Non-Requirement explicitly excludes `bin/cortex-mcp-server-launcher`. The user's actual workflow is terminal-launched Claude Code, where shell PATH is inherited and `~/.local/bin/cortex` resolves cleanly. User can veto by reinstating the launcher; would add a `bin/cortex-mcp-server-launcher` script + update `.mcp.json` to reference `${CLAUDE_PLUGIN_ROOT}/bin/cortex-mcp-server-launcher` instead of bare `cortex`.
- **Variable rename `sessions_target → lifecycle_target` (Task 7)**: the rename is forced by the spec acceptance grep (`grep -v sessions`). Alternative names (`allow_write_target`, `sandbox_target`) would also satisfy the grep but are less obvious; `lifecycle_target` is the narrow-scope match. User can veto and choose a different name.
- **`--unregister` dual-call removal (Task 7)**: the plan widens `--unregister` to call `settings_merge.unregister` TWICE — once for the legacy narrow path (`lifecycle/sessions/`) and once for the new wide path (`lifecycle/`). This preserves both forward-symmetry (fresh-init-then-unregister cleans up the entry just written) AND migration-symmetry (pre-Task-7-installed users can run `cortex init --unregister` once after upgrade and have BOTH legacy and new entries cleaned). Spec doesn't explicitly require either. User can veto by reverting to single-path unregister; doing so creates a migration trap (legacy users keep an orphaned entry forever) OR a forward orphan (the wide entry is never removed) depending on which path is chosen — pick-one-orphan trade.
- **Task ordering — refactor justfile FIRST (Task 1 before Task 2)**: ordering is load-bearing per research §2. If user prefers authoring plugin.json first, the very next `just build-plugin` invocation populates the plugin with all 14 cortex-interactive skills (recipe pollution). Reversing the order would require post-hoc cleanup. Strongly recommend keeping the refactor-first ordering.
- **Committed stub `.claude-plugin/marketplace.json` (Task 10)**: this ticket commits a minimal marketplace manifest at the repo root listing only `cortex-overnight-integration` so R11/R12/R13 are reproducible from any commit that includes this ticket — including a fresh clone. User can veto by removing this task, accepting that R11/R12/R13 become non-exercisable from a fresh clone until ticket 122 lands and that the spec's Task 12-equivalent interactive verification is effectively deferred to ticket 122. The committed-stub approach is preferred because it preserves spec scope (R11/R12/R13 stay in this ticket) and adds only a normal file-edit at the 122 handoff (not a merge conflict).
