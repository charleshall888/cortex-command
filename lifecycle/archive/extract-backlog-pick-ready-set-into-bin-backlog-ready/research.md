### Current `pick` subcommand semantics (`skills/backlog/SKILL.md:78–94`)

1. Read `backlog/index.json`. If missing, report "Run reindex first" and stop.
2. Filter to actionable items: `status` is `backlog`, `refined`, or `in_progress`, AND `blocked_by` is empty (`[]`) or all blockers resolved. Prefer `status: refined` items (spec-approved, overnight-eligible) — list them first within each priority tier.
3. Sort by priority (critical → low), then ID ascending.
4. If 0 / 1 / 2–4 / 5+ items, branch into different selection UX.

### Current `ready` subcommand semantics (`skills/backlog/SKILL.md:96–102`)

Reads `backlog/index.md` and quotes items from the `## Refined` and `## Backlog` sections. The Backlog section in `index.md` is generated by `backlog/generate_index.py:188` and includes `status in ("backlog", "open", "blocked")` items whose blockers all resolve. **This means today's `/backlog ready` includes `status: blocked` items with all blockers terminal — a behavior the agents' proposed predicate (status ∈ {backlog, refined, in_progress}) silently drops.** See Adversarial #4.

### Filter / blocker predicate — three implementations exist

| Source | Predicate definition | Treatment of non-digit blockers (e.g. `anthropics/claude-code#34243`) |
|--------|----------------------|------------------------------------------------------------------------|
| `backlog/generate_index.py:177–199` | `int(b) not in active_ids` after `if b.isdigit()` filter | Silently skipped → treated as resolved |
| `cortex_command/overnight/backlog.py:498–512` (`filter_ready`) | `status_by_id.get(bid, "backlog") not in TERMINAL_STATUSES` with dual-key lookup including zero-padded + UUID forms | Default `"backlog"` not in TERMINAL_STATUSES → treated as blocking |
| `skills/backlog/SKILL.md:82–84` | English prose, undefined "resolved" | Agent decides at runtime |

These three disagree. The new script makes the inconsistency a concrete contract — Spec must pick one canonical predicate and ideally consolidate the others to use it.

### Conventions

- Naming: `bin/cortex-{verb}-{noun}` kebab-case.
- Output: JSON to stdout, status/warnings to stderr.
- Exit codes: 0 success (incl. empty result), non-zero only when source state is missing/malformed.
- Tests: subprocess invocation against fixture backlog dirs (template: `tests/test_backlog_worktree_routing.py`).
- Logging: every shim wraps with `cortex-log-invocation` (fail-open via `|| true` and internal `trap 'exit 0'`).

- The "high-freedom vs low-freedom" framework: filter+sort+group is **low freedom** (one correct answer); the selection itself is **high freedom** (judgment). The proposed split is textbook.

### Anti-patterns

- *"Solve, don't punt"* — script must emit valid JSON even on error (`{"error": "..."}` to stdout, human message to stderr) rather than letting tracebacks corrupt the output stream.
- "Create verifiable intermediate outputs" — argues for an opt-in `--include-blocked` (or equivalent) so the agent can answer "why isn't ticket X in the list?" without re-reading `index.json`.

### JSON output convention — array-of-groups envelope is portable

| Concern | Object-keyed `{"critical": [...]}` | Array-of-groups `{"groups": [{"priority":"critical","items":[...]}]}` |
|---|---|---|
| Spec ordering | Object members are unordered per JSON spec; CPython preserves insertion since 3.7 but downstream consumers (Go maps, JS round-trip) may not. | Arrays are spec-ordered. Portable. |
| Adding metadata per group | Awkward (sibling object). | Trivial — additional field on the group object. |
| jq idiom | `.critical[]` direct, but `keys` sorts alphabetically. | `.groups[] | select(.priority=="critical") | .items[]` — uniform. |
| Empty groups | Easy to omit; consumers must defend against `KeyError`. | Easy to include with `items: []` — uniform schema. |

Consensus from `gh`, `kubectl`, `aws`: envelope objects with arrays inside (e.g. `{"items": [...]}` or `{"groups": [...]}`). [clig.dev](https://clig.dev) endorses: empty result → exit 0, non-zero only for real errors; JSON to stdout, status to stderr; respect `NO_COLOR` if any color is added.

## Tradeoffs & Alternatives

### Alternative A — New `bin/cortex-backlog-ready` (the ticket's proposal). **Recommended.**

- Read-side semantics live in a read-side script, separate from `cortex-generate-backlog-index` (write-side).
- Wires cleanly through parity linter via SKILL.md mention; no allowlist row.
- `just build-plugin` already mirrors `bin/cortex-*` to `plugins/cortex-interactive/bin/`.
- Token cost: depends on filter ratio — see Adversarial #10. Claim of "3–4 turns saved" is unverified; should be treated as plausible but not load-bearing.

### Alternative B — `--ready` flag on `cortex-generate-backlog-index`. **Rejected.**

- Conflates write-side ("generate index") with read-side ("filter for readiness") under one verb.
- Forces a frontmatter rescan when an O(1) `index.json` read would do.
- Misaligned with DR-2 ("distinct exit codes per failure class") because read-only and write-with-validation have different failure modes.

### Alternative C — Python module via `cortex_command.backlog.ready`. **Defer; ship A first.**

- `cortex_command/backlog/` package does NOT currently exist. Existing peers' Branch (a) always falls through to Branch (b). C is functionally identical to A for SKILL.md purposes (still needs a `bin/cortex-backlog-ready` shim). The decision is internal: invest in the package now, or defer until a packaging-driven ticket forces it. Defer.

### Alternative D — Inline keep-as-is. **Rejected.**

- Abandons the C7 line item from epic 101.
- Each agent re-derives mechanical logic from prose.

### Alternative E — Two scripts (`cortex-backlog-ready` + `cortex-backlog-pick-options`). **Rejected.**

- Both consumers operate on the same query result; splitting doubles maintenance for presentation-layer transforms the agent can do trivially.

## Adversarial Review

### 1. A pre-existing canonical implementation no other agent surfaced — `cortex_command/overnight/backlog.py:filter_ready()` (line 433)

Already partitions backlog items into `eligible` / `ineligible` with reason strings, using a sophisticated dual-key lookup that handles zero-padded and UUID forms of blocker IDs. **Shipping a third re-implementation** (after this and `generate_index.py`) directly violates DRY and ossifies the existing inconsistency. The correct architectural move is for the new script to call `filter_ready()` (relaxing the lifecycle-artifact requirements that are overnight-specific) or extract a shared helper that both call.

### 2. Predicate inconsistencies are real — three definitions disagree today

Documented above ("Filter / blocker predicate — three implementations exist"). Spec must pick one and either consolidate the others or document why the divergence persists. The ticket's "byte-equivalent extraction" framing is false: there is no byte-equivalent target — today's behavior is already inconsistent across `pick`, `ready`, `index.md`, and overnight gates.

### 3. `status: blocked` items leak through today's `/backlog ready` but not the proposed predicate

`generate_index.py:188` includes `status ∈ ("backlog", "open", "blocked")` in `index.md`'s Backlog section if blockers all resolve. Today's `/backlog ready` reads that section verbatim, so `status: blocked` items with all-terminal blockers appear. The proposed filter (`status ∈ {backlog, refined, in_progress}`) silently drops them. This is a user-visible behavior change for `/backlog ready` and must be a deliberate spec decision, not a default.

### 4. `contingent` is a priority and one item already has it

`backlog/index.json` contains a `priority: contingent` item. `_PRIORITY_RANK` maps unknown priorities to rank 9 and sorts last but does not drop. The new script must decide: closed-enum filter (drops contingent) or open-enum sort-last (preserves today's behavior for `pick`). Default to preserve.

### 5. Stale-index regression risk — ticket 038 documents the bug

`backlog/038-add-index-regeneration-to-overnight-pre-flight-and-investigate-staleness-gaps.md` documents that `index.json` can silently lag `backlog/*.md`. The proposed script reads `index.json` blindly; an mtime check warning to stderr (exit 0, fail-open) is cheap defense.

### 6. `/backlog ready` rendering diff is silently introduced

Today: `/backlog ready` reads `index.md`'s Refined and Backlog sections — bullet-list format, fixed columns. After extraction: agent renders JSON. Output won't be byte-equivalent. No snapshot test exists today and none was proposed. Spec should pin the rendered shape (preferably with a test fixture).

### 7. JSON-schema choice — three competing proposals from agents 1, 2, and 4

- Agent 1: `{"critical": [...], "high": [...], ...}` — priority-keyed object.
- Agent 2: `{"schema_version": 1, "groups": [{"priority": "critical", "items": [...]}, ...]}` — array-of-groups envelope.
- Agent 4: `{"by_priority": {"critical": [...]}, "refined": [...], "backlog": [...]}` — duplicates items between sections; agent must dedupe.

None included a `schema_version`-ish field for forward compat; the existing `index.json` itself has no schema_version it could derive from. The array-of-groups envelope (Agent 2) is the most defensible: spec-portable order, room for metadata, jq-uniform. Agent 4's hybrid is weakest (duplication). Agent 1's keyed object is fine for internal use but worse for forward compat.

### 8. Token-savings claim ("3–4 turns saved per `/backlog pick`") is unverified

Today the agent does one Read of `index.json` plus mental filter. After extraction the agent runs the script (one tool call) and reads its JSON. If the filter retains ~80% of items (current ratio: 20/21 are unblocked), byte volume is comparable. The claim is plausible but not load-bearing — frame as "savings expected, magnitude TBD" not "3–4 turns saved."

### 9. Plugin-mirror commit ordering

The pre-commit hook fails (does not auto-stage) when `bin/cortex-*` is staged without a fresh mirror. SKILL.md update + new shim + `just build-plugin` output must all land in the same commit. Standard practice but worth calling out in the spec.

### 10. Debug affordance loss

After extraction, "why isn't ticket X in the list?" requires re-running with debug flags or re-reading `index.json`. Mitigation: add `--include-blocked` to emit the rejection reasons. Skip `--schema` / `--debug-rejections` as YAGNI unless a concrete consumer is identified.

### 11. Concurrency

`cortex_command/common.py:atomic_write` uses tempfile + `os.replace` (POSIX atomic on the same filesystem). Concurrent generate + read is safe — the reader sees old or new, never partial. No mitigation needed.

### 12. Fail-open `cortex-log-invocation`

Reviewed at `bin/cortex-log-invocation`. Fail-open contract is sound (`trap 'exit 0'` + caller `|| true`). Side effect: creates `lifecycle/sessions/<UUID>/` if `LIFECYCLE_SESSION_ID` is set. Not a blocker; flagged for awareness.
