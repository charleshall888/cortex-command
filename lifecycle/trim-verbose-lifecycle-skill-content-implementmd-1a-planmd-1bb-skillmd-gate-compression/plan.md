# Plan: Trim verbose lifecycle skill content (implement.md §1a + plan.md §1b + SKILL.md gate compression)

## Overview

Land three trims in one PR by extracting `daytime-dispatch.json`'s atomic-write pattern into a new helper module (`cortex_command.overnight.daytime_dispatch_writer`), pointer-collapsing `plan.md` §1b synthesizer-internal prose while preserving the dispatcher contract and SEC-1 mitigation inline, and removing the duplicate `SKILL.md` "Complexity Override" section while preserving both Gate 1 and Gate 2 inline behavior. Mechanical edits run sequentially per-file to avoid edit churn; plugin mirror regeneration runs last.

## Changes to Existing Behavior (called out by critical review)

- **Atomic-write durability strengthens on macOS**: the helper module follows the canonical `state.py` pattern using `durable_fsync` (which calls `fcntl.F_FULLFSYNC` on Darwin per `cortex_command/common.py`), where the existing inline recipe uses plain `os.fsync` (no F_FULLFSYNC barrier on Darwin). On macOS hosts, `daytime-dispatch.json` writes that previously could lose payload on power-loss now durably persist before the rename completes. This is a desirable strengthening, but it is a behavior change, not a pure preservation.
- **`start_ts` source-of-truth migrates from shell to Python**: the existing inline recipe generates `start_ts` via `$(date -u +%Y-%m-%dT%H:%M:%SZ)` evaluated in the caller's shell at Bash-call time; the new helper generates `start_ts` inside the Python process at helper-invocation time. Format is identical (ISO-8601 second-precision UTC), delta is sub-millisecond. Behavioral effect is equivalent in practice; the source-of-truth migration is documented here for reviewer audit.

## Spec Deviation: Alt C adopted for Gate 2 (user-approved during plan-phase critical review)

The approved spec (R8, R11) specified Gate 2 (Specify→Plan ≥3-Open-Decisions auto-escalation) removal, with a documented coverage-hole note for the simple+low+≥3-Open-Decisions case. - **What**: Introduce a single-purpose module that owns atomic writes to `lifecycle/{feature}/daytime-dispatch.json`. - **Complexity**: simple
- **Context**:

  - **Dispatch JSON schema** (per `implement.md` §1a Step 2 current text): `{"schema_version": 1, "dispatch_id": "<32-char hex>", "feature": "<slug>", "start_ts": "<ISO 8601>", "pid": null|<int>}`. Import `durable_fsync` from `cortex_command.common` (existing import path; see `state.py:22`).
  - **`update-pid` mode**: read existing dispatch JSON via `with open(path) as f: data = json.load(f)` (context-managed close), mutate `data["pid"] = args.pid`, then atomic-write via the same pattern.

  - **Test file** (`tests/test_daytime_dispatch_writer.py`): use `tmp_path` pytest fixture; tests cover (a) `init` writes JSON with all five expected fields and `pid: null`, (b) `update-pid` mutates only `pid` and preserves other fields, (c) tmp file is cleaned up after success (no `.daytime-dispatch-*.tmp` files remaining), (d) calling `update-pid` against a missing dispatch file raises `FileNotFoundError` (not silently no-op).
