Important honesty caveat: subprocess-startup cost (`fork/exec` + Python interpreter spin-up + `cortex` CLI module imports) dominates each subprocess sample. The reported numbers are therefore a high-bound estimate of what the MCP-runtime path will see — real production calls amortize the same cost per tool dispatch, so the comparison is structurally honest, but a single `cortex` subprocess in steady state will likely reduce per-call overhead via OS file-system caches across iterations.

## Known limitations

- Sample size is 30 iterations per scenario; this is enough to stabilize p50 and produce indicative p95/p99, but tail-latency estimates are noisy. Increase via the `--iterations` knob in future re-runs if a regression hunt requires it.
- Subprocess path includes Python-interpreter and CLI cold-start cost that the in-process path skips. This **inflates** the subprocess numbers vs. a hypothetical long-lived Python helper with a JSON-RPC contract — a future ticket may evaluate the trade-off if the threshold is exceeded post-merge.
- The fixture session is the user's most recent real overnight session under `lifecycle/sessions/`; tests run against live files, so re-runs against a different session may yield different absolute numbers (the *delta* is the load-bearing comparison).
- `overnight_start_run` is exercised at the Pydantic-model-shape level only (skipped real spawn for safety). The other four tools exercise the real subprocess+JSON pipeline end-to-end.
