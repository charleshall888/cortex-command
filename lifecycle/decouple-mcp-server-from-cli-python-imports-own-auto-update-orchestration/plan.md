# Plan: decouple-mcp-server-from-cli-python-imports-own-auto-update-orchestration

## Overview

Refactor the cortex MCP server out of `cortex_command/mcp_server/` into a plugin-bundled PEP 723 single-file (`plugins/cortex-overnight-integration/server.py`) that has no `cortex_command.*` imports — the only contract with the CLI is `subprocess.run(cortex_argv) + versioned JSON`. R18 (sandbox empirical probe) runs first and hard-gates the auto-update orchestration tasks (R10–R14): if PASS, the plugin owns MCP-orchestrated `cortex upgrade` (flock, post-flock re-verify, verification probe, NDJSON error log, synchronous schema-floor gate); if FAIL/PARTIAL, those tasks are replaced by a single discoverability-nudge fallback task (R19). CLI-side work (`--print-root`, `--format json` on `overnight start/logs/cancel`) and decoupling tasks (R1–R9, R15–R17) proceed regardless of R18 verdict.

## Scope Boundaries

Mirroring spec Non-Requirements — the following are explicitly excluded:

- No inline auto-update gate inside `cortex_command/cli.py::main()` (rejected during ticket 145's plan-phase critical review and reaffirmed in spec).
- No removal of CLI's overnight orchestration logic. The CLI keeps owning all heavyweight code; the MCP only delegates.
- No remote MCP transport (SSE/HTTP). Stdio remains the only transport.
- No vendored Python deps for the plugin. PEP 723 `# /// script` + uvx caching is the deps mechanism.
- No multi-version compatibility between MCP and CLI. First implementation pins MCP to a single CLI major schema version.
- No auto-update for bare-shell `cortex` invocations. Bare-shell users explicitly run `cortex upgrade`.
- No second flock at `~/.local/share/uv/tools/cortex-command/.cortex-update.lock` for cross-tool serialization.
- No `CORTEX_REPO_PIN_SHA` env-var opt-in for tamper window.
- No `cortex selftest` new command. Verification probe (R12) reuses `cortex --print-root` and `cortex overnight status --format json`.
