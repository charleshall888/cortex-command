# Plan: adopt-xhigh-effort-default-for-overnight-lifecycle-implement

## Overview

Replace the 1D `EFFORT_MAP` in `cortex_command/pipeline/dispatch.py` with a centralized 2D `_EFFORT_MATRIX` keyed on `(complexity, criticality)` plus a small skill-override gate (`review-fix`/`integration-recovery` on Opus → `max`); upgrade `claude-agent-sdk` to a version exposing `stop_reason` so dispatch event logging can surface truncations from both emitters; and extend the metrics aggregator to bucket by effort so the post-flip cost-regression rollback signal exists before the flip lands.

## Veto Surface

- **SDK upgrade range**: Task 1 picks the lowest version `>=0.1.46` that lock-resolves cleanly. The user may prefer pinning to the exact latest stable (`v0.1.71`) to fold in unrelated bug fixes, or to a more conservative point-release that minimizes the diff between current and new. The plan does not pick a specific version; the implementer evaluates `uv lock` output and the three known hazards (v0.1.55 effort-int arm, v0.1.67 sniffio, v0.1.70 mcp floor) before choosing.
- **Skill-override mechanism (flat dict vs. computed gate)**: spec Req #3 specifies a flat dict + Opus gate. The plan follows that. An alternative — making the gate a computed function — is intentionally not adopted (spec Non-Requirements: "Reserve the 3D refactor for if/when a third skill-specific exception lands").
- **Helper signature ordering**: `resolve_effort(complexity, criticality, skill, model)` matches the order spec Req #3's acceptance test calls assume. Reordering to `(skill, complexity, criticality, model)` would force test rewrites.
- **`xhigh`-on-non-Opus runtime guard — plan deviates from spec §3 wording**: spec §3 literally specifies an `assert` raising `AssertionError`. The plan deviates and uses `raise ValueError(f"...")` instead. Three reasons: (1) `assert` is gated on `__debug__` and stripped under `python -O` / `PYTHONOPTIMIZE=1`, so the spec's "MUST fail loudly" property would silently disappear under any future containerized runtime, CI flag, or LaunchAgent migration; (2) every sibling invariant in `cortex_command/pipeline/dispatch.py` already uses `raise ValueError` (`resolve_model` at lines 191/196, `dispatch_task` at lines 428/434/436), so `assert` would create heterogeneous error paths in a single module; (3) the spec owns the *intent* (fail loudly, dispatch-time, surface via the existing dispatch error path), not the Python primitive — `ValueError` honors the intent and surfaces through the same `except` shape callers already use. The unit test `test_resolve_effort_guards_unsupported_combo` asserts `pytest.raises(ValueError)`, locking the contract.
- **Truncation allow-list locality**: the set `{"max_tokens", "model_context_window_exceeded"}` is duplicated locally in `dispatch.py` and `runner.py` rather than centralized. This is intentional per spec Edge Cases (preserves stop_reason passthrough for unknown future values) but worth flagging — a future ticket could centralize once a third truncation reason emerges.

## Scope Boundaries

Excluded from this feature, per spec §Non-Requirements:

- Per-call `effort_override` plumbing through `retry_task` — the matrix resolves effort centrally; no caller threading needed.
- 3D `(skill, complexity, criticality)` matrix — skill overrides remain a small flat dict applied after matrix lookup.
- Setting `max_tokens` on dispatches — `ClaudeAgentOptions` has no field and the CLI has no flag in v2.1.x; this ticket addresses *visibility* of truncation, not avoidance.
- Bumping Haiku effort — `(trivial, low/medium)` stays at `low`.
- Per-attempt effort variation in the retry loop — effort is resolved per-dispatch from the matrix; same matrix entry on retry without escalation.
- Detecting `stop_reason == "refusal"` or other non-truncation values as separate events — the full value is logged on `dispatch_complete`; only `max_tokens` and `model_context_window_exceeded` get the dedicated `dispatch_truncation` event.
- Tightening `test_effort_value_passthrough` to assert the SDK's typed Literal accepts `xhigh` — deferred to a follow-up ticket once SDK PR #835 merges.
