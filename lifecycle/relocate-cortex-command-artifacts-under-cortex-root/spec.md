# Specification: relocate-cortex-command-artifacts-under-cortex-root

## Problem Statement

The seven cortex-managed paths (`lifecycle/`, `research/`, `retros/`, `backlog/`, `requirements/`, `lifecycle.config.md`, `.cortex-init`) plus `debug/` currently scatter across the repo root, conflating tool-managed state with user project content. End-users who want to gitignore the tool's working area cannot do so cleanly, and the conceptual contract between "tool-managed" and "project-owned" is encoded across 30+ hardcoded sites. This work consolidates everything under a single visible `cortex/` umbrella directory in one atomic commit, switching `cortex init`'s sandbox grant to umbrella `cortex/` (DR-5), tagging a major-version bump (DR-9), and rewriting the few requirements/skill/doc texts that name the old paths inline. Single-user reality (cortex-command's own repo) bounds migration cost. Foundation #201 (upward-walking project-root detection at `cortex_command/common.py:55-103`) is shipped.

## Phases

- **Phase 1: Root-detection & init contract** — Switch `_resolve_user_project_root()` predicate to `cortex/`, collapse `_CONTENT_DECLINE_TARGETS`, swap `cortex init` sandbox-registration to umbrella `cortex/`, update error-message strings.
- **Phase 2: Path-literal migration** — Rebase every code, hook, bin, and plugin-canonical path literal from `lifecycle/`/`research/`/`backlog/`/`requirements/`/`retros/`/`debug/`/`lifecycle.config.md`/`.cortex-init` to the `cortex/` umbrella equivalents.
- **Phase 3: Data migration, doc rebases, and `git mv` storm** — Run encoded-data migration (287 backlog YAML lines + 61 critical-review-residue artifacts + ~6 research prose cross-refs), update operational docs (CLAUDE.md, docs/setup.md, docs/agentic-layer.md, README.md), update skill prose (`skills/lifecycle/SKILL.md`, `skills/refine/SKILL.md`), update `requirements/project.md:28` sandbox-constraint text, then execute the `git mv` storm.
- **Phase 4: Single atomic commit & finalize** — Refresh sandbox preflight against pre-relocation HEAD, `git add -A` and commit, post-commit `cortex init --update`, `cortex/README.md` authoring, `cortex.gitignore` template, CHANGELOG entry, version tag, `/plugin update cortex-core`.

## Edge Cases

- **Relocator self-reference**: The lifecycle directory doing this work (`lifecycle/relocate-cortex-command-artifacts-under-cortex-root/`) is itself in scope for the `git mv` storm. Mitigation: the relocation commit executes from a fresh shell session with `LIFECYCLE_SESSION_ID` unset, and after this lifecycle is marked complete (status:complete on backlog item #202 written to `lifecycle/.../events.log` via the closing `/cortex-core:lifecycle` review pass). Procedural; no code-side guard.
- **Worktree settings inheritance** (DR-8): `cortex_command/overnight/worktree.py:174` copies `~/.claude/settings.local.json` into the worktree at creation. After the commit lands, the user must run `cortex init --update` once before the next overnight session so subsequent worktree spawns inherit the umbrella `cortex/` grant rather than stale `lifecycle/`/`research/` entries.
- **Plugin version skew** (DR-9): Until `/plugin update cortex-core` runs, an existing Claude Code session's loaded plugin still references `lifecycle/<feature>/` in skill prose; spawned agents may write to the old path. Mitigation: major-version-bump tag + CHANGELOG cutover note. No fallback shim — single user, hard cutover.
- **Mid-storm crash recovery**: If the relocation commit fails partway (pre-commit gate rejects, hook crash), the working tree is in a half-relocated state. Recovery: `git stash` or `git checkout -- .` to revert to pre-relocation HEAD, fix the gate failure, retry from a clean working copy. No code-side recovery logic.
- **Backlog YAML migration partial failure**: The encoded-data migration script must be idempotent — re-running it on an already-migrated file is a no-op. Achieved by `sed`-style `path -> cortex/path` substitutions that only fire when the path lacks the `cortex/` prefix. Acceptance: re-running the migration script after the commit produces zero file changes.
- **`_resolve_user_project_root()` upward-walk from an old clone**: A user who has not pulled this commit and tries to run a CLI from inside the old `lifecycle/<feature>/` directory still hits the pre-relocation resolver, which is fine. Post-pull, the resolver looks for `cortex/` — running a CLI from inside `lifecycle/<feature>/` (which now lives at `cortex/lifecycle/<feature>/`) walks upward correctly.
- **`bin/cortex-check-parity` PREFLIGHT_PATH stale-commit-hash on first post-relocation commit**: The preflight regenerated in Requirement 19 must reference pre-relocation HEAD. A subsequent commit that touches sandbox-source files will require its own fresh preflight regeneration with the post-relocation PREFLIGHT_PATH literal — that is normal post-relocation operation, not a relocation-specific failure mode.

## Technical Constraints

- **DR-7 operational preconditions are binding**: single atomic commit; `git add -A` (no partial staging); fresh sandbox preflight against pre-relocation HEAD; no active overnight session during the commit; commit executed from a fresh shell with `LIFECYCLE_SESSION_ID` unset.
- **DR-5 umbrella grant**: `cortex init` registers a single umbrella `cortex/` entry; the dual-registration legacy is collapsed.
- **DR-8 post-commit step**: `cortex init --update` must run before the next overnight session to refresh worktree-inherited sandbox grants.
- **DR-9 major-version cutover**: tag `vN.0.0`, update CHANGELOG, document `/plugin update cortex-core`. No backwards-compat fallback for path lookup.
- **Per `requirements/project.md` "File-based state"**: relocation preserves file-based state; no database introduced.
- **Per `requirements/project.md` "Workflow trimming"**: this is a relocation, not a workflow removal. CHANGELOG entry is a relocation note, not a tombstone.
- **Per `requirements/project.md` "SKILL.md-to-bin parity enforcement"**: `bin/cortex-check-parity` glob updates (Requirement 10) must keep the gate functional across the commit boundary; the staged edits to `bin/cortex-check-parity` and `.githooks/pre-commit` land in the same commit as the renames so the gate's working-copy reads see a coherent state.
- **Per `requirements/project.md` "Sandbox preflight gate"**: edits to `cortex_command/overnight/runner.py` trigger the gate (line 1804 path-literal swap + lines 421/423 backlog-path swaps). Requirement 19 satisfies this trigger.
- **Encoded-data migration must be idempotent**: re-running yields no changes. Script-level invariant.
- **`_CONTENT_DECLINE_TARGETS` collapses to single entry** intentionally — a populated `cortex/` directory is now the sole "this repo is already initialized" signal for `cortex init`'s R19 gate.
- **`retros/` relocates rather than deletes**: per research recommendation §6 — irreversibility + zero code-impact + audit-value preservation.
