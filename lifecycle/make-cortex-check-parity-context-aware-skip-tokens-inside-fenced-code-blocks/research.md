## Web Research

### Prior art and reference implementations

- **Vale** (markup-aware prose linter): block and inline code ignored by default. Recognizes (a) indented blocks, (b) fenced blocks with 3+ backticks, (c) backtick code spans. Built on a markup AST, not regex. Provides `IgnoredScopes` / `SkippedScopes` / `BlockIgnores` / `TokenIgnores` knobs and HTML-comment escape hatches.
- **markdownlint** (DavidAnson/markdownlint, JS): uses **micromark** CommonMark parser; AST-based. Inline disable via `<!-- markdownlint-disable-line MDxxx -->`.
- **PyMarkdown** (PyMarkdownLnt, Python): AST-based, mirrors markdownlint rule names. Useful prior art if you ever want a Python AST linter framework.
- **markdown-link-check** (npm): user-provided ignore patterns + disable comments — cautionary tale that even a dedicated link checker initially failed to distinguish links inside code fences via regex-only.
- **remark + unist-util-visit + remark-retext** (JS): walks MDAST; `inlineCode` nodes are always non-prose by default.

### Python parser libraries (alternatives to stdlib regex)

- **markdown-it-py** (executablebooks): 100% CommonMark compliant. Token types of interest: `fence`, `code_block`, and `code_inline` (within `inline` token children). Documented walking pattern:
  ```python
  md = MarkdownIt("commonmark")
  for tok in md.parse(text):
      if tok.type in ("fence", "code_block"):
          continue
      if tok.type == "inline":
          for child in tok.children or []:
              if child.type == "code_inline":
                  continue
              # scan child.content for cortex-* tokens
  ```
- **mistune** (lepture): fastest pure-Python option in benchmarks; CommonMark-compatible.
- **marko** (frostming): CommonMark 0.31.2 spec compliant.
- **commonmark-py**: reference-style spec parser; slower than mistune.

### CommonMark spec essentials (https://spec.commonmark.org/0.30/)

**Section 4.5 — Fenced code blocks:**
- Fence is **at least 3** consecutive backticks OR at least 3 tildes (cannot mix).
- Opening fence may be indented 0–3 spaces.
- Closing fence must be the **same character** AND **at least as long** as the opening fence.
- **Variable-length backtick fences are explicitly supported** (open with 4+ backticks when the content contains triple backticks).
- **Cannot be nested**; content is literal text.

**Section 4.4 — Indented code blocks**: 4+ spaces (or 1 tab). Spec gives list-item continuation precedence.

**Section 6.1 — Code spans (inline):** open with N backticks, close with exactly N backticks. Variable-length supported (use `` `` `` to embed a backtick).

### Known patterns and anti-patterns

**Naive regex pitfalls** (all relevant to the current `FENCED_CODE_RE`):
1. **Fixed-length-only fence regex** like ` ```...``` ` **breaks on**: 4+ backtick fences, tilde fences, indented fences (1–3 leading spaces), fences in first/last line of a file with no surrounding `\n`.
2. **Greedy/non-backreferenced** patterns can match across two adjacent code blocks, eating prose between them.
3. **Inline code spans** are usually missed entirely by fence-only regex.
4. **Indented (4-space) code blocks** are silently ignored by fence-only regex.
5. **HTML comments and front matter** are typically excluded by AST linters but ignored by hand-rolled regex.

**Better regex** (precedent from Python-Markdown):
```
r'(?ms)^([ \t]{0,3})(`{3,}|~{3,})([^\n]*)\n(.*?)\n\1\2[ \t]*\r?\n?'
```
Captures indent, fence char/length via backreference `\2`, and reuses `\1` so the closing fence has the same indent and length. Still doesn't solve inline spans cleanly.

**Parser-AST vs. regex preprocessing**: AST is correct by construction, handles all CommonMark edge cases for free, and lets you exclude inline code spans without extra logic. Adds a dependency. Performance is fine for "thousands of small markdown files" — markdown-it-py in single-digit ms each.

## Tradeoffs & Alternatives

| Approach | Description | Implementation cost | False-negative risk on candidates | Stdlib? | Maintainability |
|---|---|---|---|---|---|
| **A. Skip fenced blocks AND inline spans entirely (asymmetric)** | Both functions return `set()` on candidate side, keep returning tokens on wiring side | Low (~10 LOC) | High on inline (loses `` `cortex-foo` `` detection); 6 deployed scripts become W003 orphans without parallel wiring rule | Yes | Simple but two-mode |
| **B. Path-segment heuristic** | Skip only tokens inside path-shaped contexts (`/cortex-foo/...`) | Medium (~30 LOC + edge cases) | Low | Yes | Fragile; "is this a path?" needs precise definition |

| **E. Sentinel preprocessing two-pass** | Replace fenced/inline with same-length spaces, run regex on cleaned text | Low (mirrors `_strip_excluded_tokens` at line 367) | Same as A (this is A's natural implementation) | Yes | High (idiomatic with existing code) |

### Recommended approach (pre-adversarial)

**Asymmetric A scoped to fenced blocks ONLY (NOT inline spans), implemented via the sentinel-blanking pattern from E.**

Rationale grounded in the audit:

1. **Fenced-only skip on candidate side is risk-free for wiring**: 0 deployed scripts depend on fenced-only wiring. Path-qualified `bin/cortex-*` mentions inside fenced blocks remain caught by `PATH_QUALIFIED_RE` upstream.
2. **Skipping inline spans on candidate side would break wiring**: 6 deployed scripts are inline-only-wired. Removing inline spans from candidates is fine; on wiring it generates 6 new W003s.
3. **Asymmetric**: change `collect_reference_candidates` only; keep `collect_wiring_signals` symmetric. Mirrors existing R5(d) self-reference precedent.
4. **Inline on candidate side is worth keeping**: author-quoted prose like `` `cortex-pr-rollup` `` is a real drift signal — if a script is renamed, the inline mention should trip E002.
5. **Stdlib-only convention upheld**: no new dependencies.
6. **Self-test additions bounded**: ~3 cases.

## Adversarial Review

### Failure modes and edge cases

1. **The 4-backtick fence is not just "missed" — it produces a *spurious partial match*.** When an author writes a 4-backtick fence (used precisely when content contains literal triple backticks), today's `r"```.*?```"` matches ` ```` ... ``` ` (stopping at the first inner triple-backtick). The first half is treated as fenced; the second half (containing the actual `cortex-*` token) is treated as **prose**. Under the proposed fix this is doubly broken — placeholder tokens in the back half *fail the build* despite being intended as code. Silently divergent from CommonMark.

2. **The `wired ⊇ candidates` invariant breaks.** Today every wiring-detected token is also candidate-eligible (modulo R5 filters). The asymmetry creates a new class: **wired by fenced presence but never candidate-eligible**. Two concrete sub-failures:
   - A future maintainer adding a new R-code may assume `collect_wiring_signals(text) ∩ candidate_universe == itself`. No test or assertion enforces it.
   - The pre-flight audit is a **snapshot**. SKILL.md files (`skills/morning-review/SKILL.md:104`, `skills/lifecycle/SKILL.md:100,106,155,163`, refine, complete) wire `cortex-update-item` and `cortex-generate-backlog-index` partly through fenced bash blocks (in addition to inline). If a refactor strips an inline mention while keeping the fenced example, the script silently relies on fenced-only wiring. Combined with a typo on the candidate side, drift becomes invisible.

3. **The motivating bug may already be resolved and recurrence isn't proven.** `git log --since '2026-04-15' -- docs/setup.md` shows 17 commits without recurrence. The placeholder was rephrased in `f817c11`. The no-clone wheel-install migration commits (`bc07a80`, `f7c5210`, `4574a21`) are all in main and didn't trip E002. This may be **complexity earning its place by solving a problem that no longer exists** — direct conflict with `requirements/project.md:19`.

4. **Inline placeholders DO leak — fix doesn't address the larger class.** Lifecycle/research files mention undeployed worked-illustration tokens (e.g., `cortex-mcp`, `cortex-doctor`, `cortex-deploy`, `cortex-resolve`) inside inline backticks. Each is a latent E002. The fenced-only fix doesn't help this larger class.

5. **Reverse-asymmetry footgun.** If the implementer copy-paste-mistakes the direction (skip fenced on **wiring**, keep on candidates), every fenced bash example of a deployed script becomes a W003 orphan, and fenced-only typos never trip E002. Pre-commit explodes globally on first run. The two functions look identical; nothing structural prevents the mistake.

6. **List-item indented fences.** A 4-space-indented numbered tutorial step containing a fence is valid CommonMark but the fence opener at 4-space indent is treated by today's regex as a regular fence open (the regex doesn't enforce 0–3 spaces of indent), and indented `cortex-foo` *outside* fences (a code block by indentation alone) leaks entirely. `docs/setup.md` uses indented fences in tutorial sections.

7. **Token at fence boundary** — pattern like ` ```...```cortex-foo``` ` (token immediately abutting closing backticks). Non-greedy fenced regex consumes the outer ` ``` `, leaves inner backticks unbalanced, and `INLINE_CODE_RE` may then grab the leftover. Token is double-counted as both fenced AND inline. Edge but real.

8. **Plugin mirror sync race.** Change must propagate to `plugins/cortex-interactive/bin/cortex-check-parity`. If a contributor edits canonical without rebuilding the plugin, and the pre-commit dual-source hook regenerates the mirror in the same staged commit, fixtures must pass against both. If a contributor bypasses the build hook, plugin consumers run an older linter against newer fixtures. Mitigation exists (`just setup-githooks`) but the fix doesn't strengthen it.

9. **Self-test coverage gap is structural, not numeric.** The failure surface is multidimensional: {fenced, inline, indented, tilde, 4-backtick} × {opening-position} × {closing-position} × {token-position}. 2-3 cases cover ~5%. Realistic minimum: ≥10 LOC of code and ≥50 LOC of tests/fixtures.

### Anti-patterns

10. **Regex-driven markdown parsing is the documented anti-pattern** (per Web research). The project's stdlib-only constraint is a **soft convention** (script header comment), not a policy gate. The asymmetry is itself complexity that exists *only because* the underlying scanner is regex-not-AST. If markdown-it-py is ever adopted elsewhere in the repo (e.g., for dashboard previews), this fix becomes a vestige.

11. **Cherry-picking fenced as special** while leaving five other markdown contexts un-special (indented code blocks, GFM math fences `$$`, HTML `<pre>`/`<code>` blocks, reference-link definitions, YAML frontmatter — currently scanned and contains `cortex-*` slugs) creates a "why is fenced exempt but indented isn't?" maintenance question.

12. **The simpler approach C (PLACEHOLDER_NAMES) is dismissed as "doesn't fix the design issue"**, but per failure modes 4, 11, and 12 above the design issue isn't being fixed by approach A either — it's being made strictly worse along the inline-and-indented axes. A 4-line `PLACEHOLDER_NAMES` patch is precedented (mirrors `RESERVED_NON_BIN_NAMES`) and zero-asymmetry.

### Assumptions that may not hold

13. **"Fenced contents are always code-context placeholders."** Counter: SKILL.md routinely uses fenced bash blocks **as the wiring** for `cortex-update-item` etc. Today the linter still detects via inline mentions; the fix bets on that redundancy continuing forever.

14. **"Pre-flight audit is sufficient evidence of zero false-negative risk."** Audit is over the current snapshot. Doesn't cover: brand-new files in the same PR as a typo, files just outside scope that may enter via hook updates, plugin-side mirrors, lifecycle files (author-volatile). A new SKILL.md authored by an overnight session that documents `cortex-foo` only via fenced bash would (under the fix) fail to flag a typo'd reference.

15. **"Fenced wiring stays load-bearing forever."** True today only because of redundancy — every fenced wiring also has inline or path-qualified mentions. No test asserts "every deployed `bin/cortex-*` has at least one non-fenced mention." Redundancy can erode silently.

### Recommended mitigations (if proceeding)

- **Add a pre-commit invariant assertion** — emit a new W004 ("script wired only via fenced code; vulnerable to silent drift if fence is removed") for any deployed `bin/cortex-*` whose only wiring is fenced. Converts the latent footgun into visible policy.
- **Add an explicit comment block** above `collect_reference_candidates` documenting the asymmetry, naming the invariant, and explaining why each scan is which mode. Defends against the reverse-asymmetry footgun.
- **Strengthen `FENCED_CODE_RE`** to handle tilde fences, ≥3-backtick variable-length fences with backreference for the closing, and explicitly *not* match indented code blocks (those need separate handling). OR accept the parser dependency and use markdown-it-py.
- **Add fixture directories**: `valid-fenced-placeholder/`, `valid-tilde-fenced-placeholder/`, `invalid-fenced-only-wiring-undeployed/` — the last fixture's name itself is the documentation that this is the agreed false-negative trade-off.
- **Reconsider doing nothing.** If the false positive recurs 2–3 times over the next 60 days, *then* implement. CLAUDE.md/project.md says complexity must earn its place.
- **Cheaper alternative if you must act now**: extend `RESERVED_NON_BIN_NAMES` (or add a parallel `PLACEHOLDER_NAMES` set with rationale comments) for the 1–3 known doc placeholder tokens. 4-line patch, zero asymmetry, zero new failure modes.
