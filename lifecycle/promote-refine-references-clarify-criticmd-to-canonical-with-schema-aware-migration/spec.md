# Specification: promote-refine-references-clarify-criticmd-to-canonical-with-schema-aware-migration

## Problem Statement

`skills/refine/references/clarify-critic.md` (215 lines) and `skills/lifecycle/references/clarify-critic.md` (167 lines) are duplicate copies of the same skill artifact, but the refine version is a schema-evolved superset (adds `## Parent Epic Loading`, `## Parent Epic Alignment`, `parent_epic_loaded` field, object-shape `findings` with `origin`, soft rubric-dimension cap). The duplication eats token context per feature and creates two writable copies of one logical artifact — a SSOT violation. Hard-deleting the legacy copy is the textbook fix for the duplication. Because refine's version changes the event schema, archived `clarify_critic` events from before this adoption must remain compatible with the existing test consumer (`check_invariant` in `tests/test_clarify_critic_alignment_integration.py`) — and this compatibility currently does not exist (the markdown documents legacy-tolerance fallbacks in prose, but `check_invariant` crashes with `AttributeError: 'str' object has no attribute 'get'` on every real archived v1 event). Phase 3 (a `schema_version` field + a real-archived-event replay test + a Python normalizer + a JSONL-emission spec amendment with a structurally-stronger JSONL example + a producer-side JSONL emission check) is the safety mechanism that makes the Phase 2 migration safe — it converts the prose-only legacy-tolerance into an executable, asserted-by-test contract for the current test consumer (and any future production consumer per #186 onwards). The beneficiaries are: (a) the orchestrator agent reading clarify-critic.md, which gets one canonical home instead of two; (b) the test consumer, which stops crashing on real v1 events; (c) future maintainers, who get an explicit `schema_version` marker to anchor v1↔v2 boundaries; (d) the parser surface (`cortex_command/pipeline/metrics.py:parse_events()`), which regains visibility into post-migration clarify_critic events via the JSONL-example replacement and the post-migration emission check (note: the 43 archived YAML-block events remain parser-invisible per Non-Requirements; this beneficiary is forward-looking only).

   Acceptance: `grep -c '_normalize_clarify_critic_event' tests/test_clarify_critic_alignment_integration.py` ≥ 2 (definition + at least one call site).

The test loads `tests/fixtures/clarify_critic_v1.json`, applies `_normalize_clarify_critic_event`, asserts: (a) `schema_version == 1` post-normalization; (b) `parent_epic_loaded is False` post-normalization; (c) every item in `findings` is a `dict` with keys `text` (str) and `origin` (str); (d) every `origin` value is `"primary"` (no alignment findings in v1); (e) `check_invariant(normalized_evt) is True`. The `## Event Logging` section explicitly states that the orchestrator emits each event as a single-line JSON object (JSONL) to `events.log`. **The example block at lines ~166-193 of clarify-critic.md is replaced with a single-line JSONL example** (matching the structural pattern of `skills/lifecycle/references/plan.md:138`'s `plan_comparison` v2 example, which is single-line JSON). Acceptance (all three exit 0): `grep -q 'minimal v1' skills/refine/references/clarify-critic.md`, `grep -q 'v1+dismissals' skills/refine/references/clarify-critic.md`, `grep -q 'YAML-block' skills/refine/references/clarify-critic.md`.

## Non-Requirements

- **Format reconciliation for the 43 archived YAML-block events**: out of scope. They remain on-disk as text but stay invisible to `parse_events()`. This ticket *codifies* this status — the JSONL-emission requirement and the post-migration check together mean the 43 events are permanently parser-invisible going forward; they are not regressed, but they are also not a future cleanup target unless a separate ticket files. Recommended follow-up backlog ticket title: "Backfill or re-archive the 43 YAML-block clarify_critic events for parse_events() visibility."
- **Migrating the producer to `record_event()`**: out of scope. clarify_critic stays markdown-spec'd-and-orchestrator-emitted, matching the `plan_comparison` v2 pattern. Adding it to `EVENT_TYPES` would be unused.
- **Runtime validator for the cross-field invariant** (`alignment finding → parent_epic_loaded: true`) and for the closed-allowlist warning templates: out of scope; covered by backlog #186. Backlog #186 is also the named promotion trigger for relocating `_normalize_clarify_critic_event` from `tests/` to `cortex_command/` (see R5 + Decisions Resolved Pre-Spec §1).
- **Markdown link-validator pre-commit hook** to audit cross-skill `..` references: out of scope; flag for follow-up. The new reference is at the same risk class as the 5 existing instances in `skills/refine/SKILL.md`; this ticket does not regress the existing risk profile.
- **Production-side normalizer in `cortex_command/`**: out of scope for this ticket. The current consumer set is test-only; production-side affordance is YAGNI for #175 alone. Promote with a one-import refactor when backlog #186 (the runtime cross-field invariant validator) lands — it will be the first production consumer that needs to read archived clarify_critic events and will hit the same `AttributeError` the test currently does.
- **Backporting the `schema_version` field to existing archived events**: out of scope. "Frozen v1 events live forever" per event-sourcing literature; the legacy-tolerance fallback is the contract.
- **OQ3 evidence-gathering for promoting `schema_version` from SHOULD to MUST on the producer side**: out of scope for #175. Phrased as SHOULD now; if a future producer-side regression is observed (e.g., a clarify_critic event lands without `schema_version: 2`), an F-row in events.log + an `effort=high` dispatch attempt will provide the evidence per OQ3 policy. (See Decisions Resolved Pre-Spec §3 for re-evaluation triggers.)

## Edge Cases

- **Hybrid findings list (mix of `str` and `dict`)**: the normalizer handles per-element, not per-list. Each item is independently checked: `dict` passes through; `str` is wrapped. No archived event currently has this hybrid shape, but the normalizer is robust to it.
- **Findings list contains a non-`str`-non-`dict` item** (e.g., `int`, `None`): treat as a malformed event. The normalizer raises `TypeError` with a message identifying the offending item type. Better to fail loudly than to silently produce malformed normalized output.
- **Event missing `findings` field entirely**: `evt.get("findings", [])` defaults to empty list; normalizer returns `findings: []`. `check_invariant` returns True trivially.
- **Mid-edit working-tree race — overnight-execution variant**: the §3a path edit and the file deletion are part of the same migration commit (R13 enforces the same-commit semantics; the `.githooks/pre-commit` drift hook structurally prevents partial-state commits via auto-rebuild + drift-check). The orchestrator MUST NOT invoke a fresh `/cortex-core:lifecycle clarify` or `/cortex-core:refine` pass against the staged-but-uncommitted tree between R1 deletion and the migration commit landing — doing so would 404 on the deleted reference. Acceptable enforcement: implementation phase performs all R1+R3+test edits as a single staged-and-committed sequence, and any clarify-pass invocation that runs concurrently against a partial state must be aborted by the orchestrator.
- **Replay test against future archive churn**: the fixture is pinned to `tests/fixtures/clarify_critic_v1.json` (test-owned), so archive moves/compactions don't break it. The provenance file documents the source for forensic purposes.
- **`tests/fixtures/clarify_critic_v1.json.provenance` accidentally edited**: the fixture itself is the source of truth for the test; the provenance file is documentation. If they drift, the test still passes (it reads only the fixture). This is acceptable.
- **Orchestrator emits a clarify_critic event with `schema_version: 2` BUT bare-string findings** (malformed v2 producer): R14's post-migration JSONL check catches the format-shape regression. The cross-field invariant check (whether the event is internally consistent given its claimed schema_version) is the eventual concern of backlog #186.
- **A future v2 archived event accumulates**: the test name `test_clarify_critic_v1_replay_invariant` carries the version; a v2 corpus eventually gets a parallel `_v2_` test rather than mutating the v1 one. (Documented in spec, not enforced; future maintainers' decision.)
- **R14 post-migration check fires in a fresh-clone CI environment**: in a fresh tree with zero post-cutoff `clarify_critic` events, the test passes trivially (zero-iteration loop). The contract becomes binding the moment any orchestrator writes a post-cutoff event. This is the intended behavior — false-negatives are not possible because the test only fails on observed YAML-block emission post-cutoff.

## Technical Constraints

- **Cross-skill `..` reference is durable but unaudited**: 5 existing instances in `skills/refine/SKILL.md` (lines 39, 66, 87, 157, 164) plus the new instance in `skills/lifecycle/references/clarify.md` §3a. No tooling validates these. If `skills/refine/` is renamed, all silently break. Out of scope to fix; documented for follow-up.
- **`parse_events()` JSONL-only contract**: the JSONL-emission requirement is the spec's anchor. Future producers MUST emit JSONL; the markdown spec is authoritative; R14 is the executable check.
- **OQ3 SHOULD/MUST asymmetry**: producer-side `schema_version` is SHOULD (positive routing per OQ3 grandfathering for new MUST language); reader-side legacy-tolerance is MUST (defensive must — the failure mode is reader crash). The asymmetry is the OQ3-spirit-compliant compromise pending a future evidence-gathering exercise.
- **No runtime cross-field invariant validation**: the markdown asserts `alignment finding → parent_epic_loaded: true` in prose; `check_invariant` enforces it on dispatched dicts; backlog #186 plans the runtime validator. Adding `schema_version` to the same prose layer has the same enforcement gap; this is acceptable until #186 lands.
- **Plugin mirror enforcement**: `.githooks/pre-commit` drift hook fails the commit if `plugins/cortex-core/` is out of sync with canonical sources. **Phase 3 of the hook auto-invokes `just build-plugin` before the drift loop** — the implementer does not need to invoke it manually. **On drift-failure recovery procedure**: `just build-plugin && git add plugins/cortex-core/ && git commit ...`; do NOT use `--no-verify` and do NOT revert R1. This is the documented recovery path; agents that encounter a drift-failure error message at commit time should follow it rather than treat the deletion as wrong.
- **Test runs in default `pytest` config**: no special markers required; tests are standard unit tests reading fixture files via `pathlib.Path(__file__).parent / "fixtures" / "..."`.

## Decisions Resolved Pre-Spec

These are the four Research-phase open questions, the chosen path for each, the rejected alternative, and the re-evaluation trigger that would re-open the decision. They are recorded here at the spec layer (not deferred to research.md) because the spec is the entire communication channel for overnight handoff per project requirements.

1. **Python normalizer placement** — chosen: test-only at `tests/test_clarify_critic_alignment_integration.py`. Rejected: production-side helper in `cortex_command/overnight/events_clarify_critic.py`. Re-evaluation trigger: backlog #186 (runtime cross-field invariant validator) lands and needs to read archived clarify_critic events — at that point, promote the helper from tests to `cortex_command/` via a one-import refactor.

2. **Format reconciliation strategy** — chosen: spec-amend the markdown to require JSONL emission going forward + replace the example block with JSONL (R9). Rejected: (a) defer entirely to a separate ticket (would leave R9 performative); (b) extend `parse_events()` to recognize YAML-block events (would freeze the format ambiguity as a permanent maintenance burden). Re-evaluation trigger: if the post-migration JSONL check (R14) starts failing — i.e., orchestrators emit YAML-block clarify_critic events after the migration cutoff — escalate to either (i) producer-side effort=high dispatch to gather OQ3 evidence for tightening producer guidance to MUST, or (ii) extend `parse_events()` to recognize YAML-block events.

3. **Producer SHOULD vs MUST on `schema_version`** — chosen: producer SHOULD; reader MUST tolerate absence as v1. Rejected: (a) producer MUST + reader MUST + OQ3 evidence-gathering scoped into this ticket (would inflate scope to a dispatch-experiment exercise); (b) symmetric SHOULD on both sides (would make the reader-side defensive contract unenforceable). Re-evaluation trigger: a producer-side regression — a clarify_critic event is observed missing `schema_version` after the migration cutoff. This produces the F-row evidence required by CLAUDE.md OQ3 policy; an `effort=high` dispatch attempt then provides the second piece of evidence; promotion of producer-side guidance to MUST follows.

4. **v1.5 intermediate shape documentation** — chosen: explicitly enumerate three pre-v2 shapes (minimal v1 / v1+dismissals / YAML-block-on-disk) in the legacy-tolerance prose (R10). Rejected: implicit (just rely on the per-field normalizer to handle all combinations). Re-evaluation trigger: none — this is documentation hygiene; no future event would change the historical shape census.

5. **Replay test fixture provenance documentation** — chosen: sibling `clarify_critic_v1.json.provenance` markdown-comment file with one-line source annotation (R7). Rejected: inline in test docstring (couples test code to archive-line citation). Re-evaluation trigger: the chosen archive line is moved or compacted — at which point either re-pin to a different archive line or convert the fixture to a fully-synthetic dict with a clear "no longer derived from archive" note.
