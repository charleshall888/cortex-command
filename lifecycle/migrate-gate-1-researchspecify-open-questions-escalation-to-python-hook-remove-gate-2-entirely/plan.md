# Plan: migrate-gate-1-researchspecify-open-questions-escalation-to-python-hook-remove-gate-2-entirely

## Overview

Build a single self-contained `bin/cortex-complexity-escalator` Python utility (uv-run script, no `.py` extension, telemetry-shim header) that evaluates both complexity-escalation gates via a `--gate` parameter, collapse the two SKILL.md gate-prose sites to one-line invocations with exit-code branching, and ship comprehensive pytest coverage. The hook is monotonic-upward, uses latest-event tier semantics that recognize all three production `complexity_override` payload shapes when guarding, and uses read-after-write verification to surface silent sandbox denials via non-zero exit consumed by the SKILL.md prose contract.

## Veto Surface

- **Hook script as uv-run self-contained vs cortex_command module + thin bin shim**: chose self-contained per `bin/cortex-resolve-backlog-item` precedent — avoids the `cortex_command/__init__.py` install_guard blast radius during overnight runs. Trade-off: bullet-counting logic is not directly importable from anywhere except via importlib hackery in tests. The alternative (logic in `cortex_command/lifecycle/complexity_escalator.py` + thin `bin/cortex-complexity-escalator` shim importing it) would make the logic cleanly importable but couples the bin entry to the install_guard. Tests use the same importlib pattern as `tests/test_resolve_backlog_item.py`, so the trade-off is bounded.
- **Tier-guard semantics: latest-event effective tier, not literal `read_tier()` call**: spec R6 names `read_tier()` but `read_tier()` only inspects the `tier` field and is structurally blind to the standard `{from, to}` payload required by R7. The plan implements semantic-equivalence with extended payload recognition via an in-script `read_effective_tier()` helper. This is a deviation from R6's literal API call but is necessary to also satisfy R7 and the downgrade-then-re-escalate Edge Case at spec line 100. The alternative (extending the global `cortex_command/common.py:read_tier()` to recognize all three shapes) is rejected as out-of-scope per spec Non-Requirement "No canonicalization of pre-v2 event-payload shapes." Pytest cases assert behavioral skip, not the specific function called.
- **Read-after-write race window**: the hook re-reads only the last line of events.log, not full integrity. If a concurrent writer (another phase on the same feature emitting `phase_transition`, `orchestrator_review`, etc.) appends between this hook's write and its re-read, the assertion fails and the hook exits non-zero, which under R13's SKILL.md prose halts the phase transition. This is a real false-positive failure mode, but is bounded by the project's single-active-session-per-feature convention (`.session` files in `lifecycle/<feature>/`): a feature with one active session has a single writer per phase, so concurrent writes on the same feature are not expected outside of edge cases (e.g., the orchestrator emitting an unrelated event while the hook runs synchronously inside the protocol step). Documented recovery: re-run the hook after the concurrent operation completes; second invocation is idempotent (tier-guard returns "complex" via `read_effective_tier`). The alternative (scan-back from EOF for a matching event, or fcntl locking) is rejected as out-of-scope.
- **Algorithm scope: top-level bullets only at indent==0, no semantic resolved/deferred filtering**: spec R3 says "Sub-bullets nested deeper than the section's primary indent level are not counted" — the plan commits to `primary indent level == 0` because every research.md/spec.md template in the current corpus places `## Open Questions` / `## Open Decisions` at zero indent. A future template that nests these sections under a parent bullet would silently drop all bullets to count 0; mitigation is a follow-up ticket if that template ever lands. The hook also trusts refine's Research Exit Gate to filter resolved/deferred upstream (spec Non-Requirements line 75, Edge Cases line 105) — a user editing research.md after refine's gate runs could re-introduce non-genuine open questions and trigger over-escalation; the spec accepts this trade-off explicitly.
- **Numbered-bullet inclusion is a behavior change from legacy SKILL.md prose**: the current SKILL.md Step 3 §5 and §6 prose says "bullet items (`-` or `*` prefixed lines)" — no mention of numbered bullets. Spec R3 mandates `^\d+\.[ \t]` (numbered) as a third counted prefix. The plan inherits the spec choice. If any current research.md uses numbered enumeration under `## Open Questions` (e.g., a model recapping FMs as `1.`, `2.`, `3.`), the new hook will fire where the legacy prose would not — a one-time behavior shift at deployment. No corpus check is bundled into this ticket; mitigation is the rollback signal noted in spec Non-Requirement "rollback story" (watch-list, 4-week review cadence).
- **Bracketed-prefix collision (R4 idiom rule)**: spec Edge Cases line 106 documents that legitimate `[Trade-off A vs B]:` Open Decisions bullets are excluded by `^\[`. This is a known false-positive trade-off chosen by the spec in favor of correctly handling the more-common template-placeholder pattern. Mitigation: future ticket can tighten the regex to `^\[Only when\b` if a real-decision-with-bracketed-prefix bullet appears in practice. Surfaced here so downstream maintainers know it is an intentional trade-off, not an oversight.
- **Combined deployment + wiring + plugin-mirror staging in Task 1**: chose combined for two reasons. (1) Wiring co-location rule (plan.md:185) says deployment + wiring land in the same task. (2) `.githooks/pre-commit` Phase 3 auto-runs `just build-plugin` and Phase 4 fails on unstaged plugin-mirror drift — if T1 staged only canonical paths and tried to commit before T3 staged the mirrors, the pre-commit hook would block the commit. Folding the mirror staging into T1 keeps every commit boundary clean. The alternative (separate plugin-staging task) is rejected because the pre-commit hook enforces canonical↔mirror lockstep at every commit, not at task boundaries.
- **No CHANGELOG entry**: spec Non-Requirements line 80 makes this optional. The new utility is additive; user-visible behavior of both gates is preserved.

Spec text for the D4 work is `lifecycle/artifact-template-cleanups-architectural-pattern-optional-indexmd-body-trim-frontmatter-preserved-d4-open-decisions-optional/spec.md` Requirement 8 (verbatim acceptance grep: `grep -c "Optional — omit when empty" skills/lifecycle/references/specify.md` returns ≥ 1 AND `grep -c "Open Decision Resolution" skills/lifecycle/references/specify.md` matches the `main` baseline).
