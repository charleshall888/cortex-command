# Specification: Add dashboard visual evaluation criteria to DESIGN.md

## Problem Statement

DESIGN.md defines authoring constraints (tokens, forbidden patterns, component rules) but not outcome evaluation criteria. When Claude or a human reviews dashboard UI — whether via Playwright MCP or direct observation — there is no defined standard for what "good" looks like beyond rule compliance. Adding visual evaluation criteria provides a shared reference for judging dashboard quality across the four dimensions identified in the generative-ui-harness research.

## Requirements

1. **Add `## Visual Evaluation Criteria` section to DESIGN.md**: Append after the existing `## Pattern Examples` section (currently the last section, ending at line 93). The section must contain four criteria in a table with columns: Criterion, Weight, What to evaluate.
   - AC: `grep -c '## Visual Evaluation Criteria' claude/dashboard/DESIGN.md` = 1, pass if true.

2. **Four criteria with weights and evaluation descriptions**: The table must include all four criteria adapted from the generative-ui-harness research DR-4:
   - Information clarity (High): status hierarchy visually distinct via color and size differentiation, feature phases scannable across cards, session info prominent in session panel
   - Consistency (High): design tokens used throughout, badge color meanings stable across panels, no forbidden patterns visible, spacing follows token scale
   - Operational usefulness (Medium): alerts prominent and not buried, swim-lane legible with no overlapping labels, no visible layout shift after HTMX refresh, session history presents completed features with outcome and duration
   - Purposefulness (Low): reads as purpose-built monitoring tool, not generic admin panel — relevant primarily for large visual changes
   - AC: `grep -Ec 'Information clarity|Consistency|Operational usefulness|Purposefulness' claude/dashboard/DESIGN.md` = 4, pass if true.

3. **Usage guidance paragraph**: After the criteria table, include a brief paragraph (2-4 sentences) explaining how the criteria are applied: via Playwright MCP (ticket 029) for Claude-driven evaluation using interactive navigation and screenshots, as review criteria for dashboard PRs, or as acceptance criteria in dashboard feature specs. Note that all four criteria can be assessed by Claude via Playwright MCP, though evaluation reliability varies — consistency is the most mechanically verifiable, while purposefulness requires the most judgment.
   - AC: `grep -c 'Playwright' claude/dashboard/DESIGN.md` >= 1, pass if true.

4. **Match DESIGN.md style conventions**: Terse, directive prose. `##` heading level. Table for structured reference data. No verbose explanations.
   - AC: Interactive/session-dependent — requires reading the result in context of the existing file.

## Non-Requirements

- No lifecycle spec template modifications — criteria live in DESIGN.md only
- No separate CONTEXT.md file — DESIGN.md is the single source for dashboard design guidance
- No process gates, rubric scoring infrastructure, or automation
- No changes to existing DESIGN.md sections — additive only
- No evaluator calibration or prompt engineering — that belongs in ticket 029's evaluation workflow

## Edge Cases

- **DESIGN.md regenerated by `/ui-brief`**: The file header says "Edit freely — re-running `/ui-brief` will not overwrite this file." The new section is safe from overwrite.
- **Criteria referenced by tickets that predate this change**: Tickets 029 and 031 reference "quality criteria in DESIGN.md" generically. The section name (`## Visual Evaluation Criteria`) must be stable — do not rename without updating downstream references.

## Technical Constraints

- The new section appends at end — no modification to existing content
- File is markdown with tables — no YAML frontmatter, no executable content
