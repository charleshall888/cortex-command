# Specification: promote-lifecycle-state-out-of-eventslog-full-reads

## Problem Statement

Lifecycle state (`criticality`, `tier`, `tasks_total`, `tasks_checked`, `rework_cycles`) is extracted at read-time from append-only artifacts (`events.log`, `plan.md`, `review.md`) via full-file reads at `cortex_command/common.py:143-272` (`detect_lifecycle_phase`), `:279-313` (`read_criticality`), `:320-354` (`read_tier`), plus `skills/lifecycle/references/complete.md:25-26` (the "weakest consumer-value ratio" case — checkbox + verdict regex over two whole files for two integers). The same scan logic is duplicated as prose in 10+ skill-prompt stanzas, which Claude executes as Read-tool full-file fetches into context (~3,000-9,000 input tokens per stanza-hit on a median 8.7 KB events.log). Two parallel tier readers (`common.py:read_tier` "last-tier-wins" vs `cortex_command/overnight/report.py:746-778` `_read_tier` canonical "`lifecycle_start → complexity_override.to`") silently disagree, and `outcome_router.py:830` consumes the non-canonical one — today's corpus has zero divergent lifecycles but any future event carrying a stray `tier` field on a non-override event triggers the divergence. This spec ships (a) bin-script readers (the audit's recommendation) that replace Read-tool full-fetches with single-line JSON returns, (b) an in-process `lru_cache` for the dashboard's long-lived FastAPI poller, and (c) the semantic alignment of `common.py:read_tier` to the canonical rule with an enforced corpus-audit acceptance gate. Structural promotion to a per-feature state file is not pursued: research established it is premature against current measurements and adds W4-falsifying concurrency surface; the audit's actual recommendation was the bin-reader pattern (`research/lifecycle-discovery-token-audit/research.md:49,55,111,114`).

## Requirements

1. **Add `functools.lru_cache(maxsize=128)` to `read_criticality`, `read_tier`, and `detect_lifecycle_phase` in `cortex_command/common.py`, keyed on `(path, mtime_ns)` of the underlying file(s).** The dashboard's `_poll_state_files` runs `parse_feature_events` every 2 seconds per feature; `events.log` and `plan.md` mtimes change at most ~10×/hour/lifecycle, so cache-hit rate is ~93-99% in steady state. Acceptance: `python -c "from cortex_command.common import read_criticality; print(read_criticality.__wrapped__ is not None)"` exits 0 and prints `True`. A new test in `tests/test_lifecycle_state.py` exercises mtime-invalidation (call → mutate file → call again → assert fresh result) and exits 0.

2. **Align `cortex_command/common.py:read_tier` to the canonical `lifecycle_start → complexity_override.to` rule** that `cortex_command/overnight/report.py:_read_tier` already implements, AND gate the change with an enforced corpus-audit. Acceptance has three parts:
   - **(2a)** `pytest tests/test_read_tier_parity.py -q` exits 0. The parity test covers: (i) `lifecycle_start tier:simple` alone returns `simple`; (ii) `lifecycle_start tier:simple` followed by `complexity_override to:complex` returns `complex`; (iii) a stray non-override event carrying a `tier` field (e.g., `batch_dispatch tier:complex`) emitted after `complexity_override to:simple` still returns `simple`; (iv) `common.py:read_tier` and `report.py:_read_tier` return identical values across the full fixture corpus AND the in-tree `lifecycle/*/events.log` corpus.
   - **(2b)** New script `bin/cortex-audit-tier-divergence` (mirrored to `plugins/cortex-core/bin/`) scans `lifecycle/*/events.log` and exits non-zero if any lifecycle has `read_tier_last_wins ≠ read_tier_canonical`. Acceptance: `cortex-audit-tier-divergence` exits 0 against the current corpus (today N=0; verified ad-hoc 2026-05-11) and exits 1 against a fixture corpus containing a `phase_transition tier:complex` event emitted after `complexity_override to:simple`.
   - **(2c)** The pre-commit hook invokes `cortex-audit-tier-divergence` when staged diffs touch `cortex_command/common.py` or `cortex_command/overnight/report.py`. Acceptance: a staged diff modifying `common.py:read_tier` body without the audit running clean blocks at pre-commit with stderr matching `tier-divergence`.

3. **Add `bin/cortex-lifecycle-state` (bash, not Python)** invoked as `cortex-lifecycle-state --feature <slug> [--field criticality|tier]`. Implementation uses `grep '"event":"lifecycle_start"' | tail -1 | jq` for baseline, `grep '"event":"criticality_override"' | tail -1 | jq` for override, `grep '"event":"complexity_override"' | tail -1 | jq` for tier override. Returns JSON on stdout (e.g., `{"criticality": "high", "tier": "complex"}`). Mirrored into `plugins/cortex-core/bin/` via the existing dual-source pre-commit hook. Acceptance: against a fixture lifecycle with `lifecycle_start tier:complex criticality:high`, `cortex-lifecycle-state --feature <fixture> --field tier` exits 0 and stdout satisfies `jq -e '.tier == "complex"'`. Per-call cost target: ≤15ms wall time on a 15 KB events.log (verifiable via `time cortex-lifecycle-state --feature <fixture>` in CI assertion).

4. **Add `bin/cortex-lifecycle-counters` (bash)** invoked as `cortex-lifecycle-counters --feature <slug>`. Returns JSON `{"tasks_total": <int>, "tasks_checked": <int>, "rework_cycles": <int>}` by parsing `plan.md` checkbox state (`grep -cE '^- \[ \]|^- \[x\]'` for total; `grep -cE '^- \[x\]'` for checked) and `review.md` verdict-count via the same regex `common.py:detect_lifecycle_phase` uses. Mirrored to `plugins/cortex-core/bin/`. Acceptance: against a fixture with 5 checkboxes (3 checked) and 2 verdict lines in review.md, exits 0 and stdout satisfies `jq -e '.tasks_total == 5 and .tasks_checked == 3 and .rework_cycles == 2'`.

5. **Replace `skills/lifecycle/references/complete.md:25-26`** with a single `cortex-lifecycle-counters --feature {feature}` invocation. Acceptance: `grep -c "Count the total checkboxes" skills/lifecycle/references/complete.md` = 0; `grep -c "cortex-lifecycle-counters" skills/lifecycle/references/complete.md` ≥ 1.

6. **Replace the duplicate "scan events.log for the most recent ..." prose stanzas** in `skills/lifecycle/SKILL.md:79,81`, `references/plan.md:21,269`, `references/specify.md:147`, `references/orchestrator-review.md:7`, `references/implement.md:246`, `skills/refine/SKILL.md:159`, `skills/dev/SKILL.md:126`, `skills/morning-review/references/walkthrough.md:237` with a single `cortex-lifecycle-state --feature {feature} --field {criticality|tier}` invocation. Acceptance: `grep -rnE "scan.*for the most recent.*lifecycle_start|read events\.log for.*criticality|read events\.log for.*tier" skills/` returns no matches across the listed files; `grep -rc "cortex-lifecycle-state" skills/lifecycle/ skills/refine/ skills/dev/ skills/morning-review/` returns ≥ 9 hits.

## Non-Requirements

- **No `lifecycle/<feature>/state.json` file.** Per research and the user-approved recommendation, structural promotion is not pursued: research established W4 single-writer-per-lifecycle is empirically false (morning-review is a documented third writer class), the authority-vs-cache contract is incoherent, and the audit's actual cost finding is Tier-3. Structural shift may be revisited if profiling later shows a measured hot path, but no measurement-gated follow-up machinery ships in this ticket — the prior R9/R10 instrumentation+trigger design was found to have arithmetically-unreachable thresholds at the instrumented site (~99% steady-state cache-hit rate vs. <80% trigger threshold).
- **No `schema_version` field, no atomic-write helper integration, no CAS or version-locking, no field-partitioning protocol.** All conditional on the structural shift that is not happening here.
- **No new `tests/` infrastructure for concurrent-writer scenarios** (no `multiprocessing.Process` fixtures). No new writer is introduced.
- **No changes to Python writer code** (`cortex_command/pipeline/review_dispatch.py`, `cortex_command/overnight/runner.py`, `cortex_command/overnight/outcome_router.py`, `cortex_command/overnight/feature_executor.py`, `bin/cortex-complexity-escalator`). Writers continue to emit to `events.log` exactly as today.
- **No changes to `phase` derivation.** `detect_lifecycle_phase`'s 6-step state machine over file presence + checkbox state continues to be the source of truth for `phase`; only its file reads are wrapped in `lru_cache`.
- **No changes to skill-prompt event-emit instructions.** Only reader-stanzas are consolidated.
- **No events-registry pin for "state-source-of-truth" events.** Defensible but separable; may be filed as a follow-up if a future ticket attempts to demote one of `lifecycle_start`, `criticality_override`, `complexity_override`, `phase_transition`, `review_verdict`, or `feature_complete`.
- **No verdict-regex golden-fixture test.** Defensible but separable; may be filed as a follow-up if the reviewer prompt is rewritten.
- **No perf-instrumentation JSONL in `dashboard/data.py`** and no measurement-gated follow-up ticket creation procedure. The instrumentation site would record only the dashboard regime where caching is maximally effective; it cannot produce data that would justify a future structural shift.
- **No migration of in-flight lifecycles.** The lru_cache addition is transparent; first-call cost is unchanged.

## Edge Cases

- **`events.log` mtime updated mid-call**: `lru_cache` keys on `(path, mtime_ns)` captured at function entry. If a writer renames a new `events.log` between mtime capture and read, the reader sees one or the other consistently — the captured key matches the file actually read.
- **Cache miss across processes**: skill-prompt callers spawn fresh bash subprocesses for `cortex-lifecycle-state` / `cortex-lifecycle-counters`. Per-call cost is ~7-15ms (bash + grep + jq) regardless of cache, since the cache is in-process to the Python interpreter and bash callers don't share it. This is acceptable: the token win at the Claude layer (~30 tokens returned vs. ~3,000-9,000 tokens from a Read-tool full-fetch) dominates whether or not the script itself is cached.
- **`detect_lifecycle_phase` cache key spans three files** (events.log + plan.md + review.md): key is the tuple `(events_mtime, plan_mtime, review_mtime)`. Any one file changing invalidates the cache for that lifecycle.
- **read_tier semantic alignment changes behavior at `outcome_router.py:830`** for any lifecycle whose events.log has a stray non-override `tier` field after a `complexity_override`. Mitigation is enforced via Requirement 2b (corpus-audit acceptance gate). Today the in-tree corpus has zero affected lifecycles (verified 2026-05-11 ad-hoc audit; the same audit ships as `bin/cortex-audit-tier-divergence`).
- **Paused-then-resumed overnight session straddling the deploy boundary**: an active session paused under the old code and resumed under the new code can theoretically gate the same feature under two different tier rules. Because today's corpus audit returns N=0 affected lifecycles, no in-flight session is exposed; if a future event-emitter pattern introduces stray `tier` fields, the pre-commit gate (2c) fires before merge. Mitigation surface: enumerate any affected lifecycle in PR description and require draining the overnight runner before deploy.
- **Bin-script vs Python-helper drift**: both readers must agree on the canonical rule. `tests/test_read_tier_parity.py` covers Python-side; new `tests/test_bin_lifecycle_state_parity.py` runs `cortex-lifecycle-state` as a subprocess against fixtures and asserts its output matches `common.py:read_criticality` / `read_tier` output. Acceptance: `pytest tests/test_bin_lifecycle_state_parity.py -q` exits 0.
- **`cortex-log-invocation` shim cost**: per `bin/cortex-log-invocation:1-30`, the shim is bash, fail-open (`trap 'exit 0' EXIT`), and adds ~1-2ms. New bin scripts honor the shim (calling it at script entry per the convention at `bin/cortex-resolve-backlog-item:19`).
- **events.log absent** (in-flight lifecycle that hasn't emitted lifecycle_start yet): bin script returns `{}`; common.py readers retain their existing defaults (`read_criticality` → `medium`, `read_tier` → `simple`).

## Technical Constraints

- **`functools.lru_cache` is stdlib.** No new dependency. Existing precedent: `cortex_command/` uses `functools.lru_cache` in two places today.
- **Bin scripts use bash + grep + jq, not Python.** Empirical measurement: bash + grep + jq round-trip is ~7-8ms on a 15 KB events.log fixture; Python interpreter boot alone is ~20ms. The bash implementation halves per-call latency without changing the caller contract.
- **Dual-source enforcement** (per `CLAUDE.md`): canonical bin scripts at `bin/` auto-mirror to `plugins/cortex-core/bin/` via the pre-commit hook. Edit canonical only.
- **SKILL.md-to-bin parity gate** (`bin/cortex-check-parity`): all three new bin scripts must be referenced from at least one in-scope SKILL.md / references / hooks / justfile / tests file. They are (R5, R6, R2c).
- **No-lock convention preserved** (`requirements/pipeline.md:127,134`): no new writers, constraint trivially satisfied.
- **Test corpus availability**: existing `tests/fixtures/state/*` covers reader test scaffolding; new tests reuse the scaffold pattern.
- **Empirical baseline (recorded 2026-05-11 on this machine)**: Python boot ~20ms, bash boot ~3ms, bash + grep + jq on 15 KB events.log ~7-8ms, in-tree events.log median 8.7 KB / 124 lines, in-tree corpus has 28 events.log files with zero `read_tier` divergence.
