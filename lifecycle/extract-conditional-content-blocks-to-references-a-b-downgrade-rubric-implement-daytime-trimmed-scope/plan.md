## Veto Surface

- **Test-task serialization across tasks 2/3/4/5**: Tasks 2/3/4/5 all add or modify functions in the same file (`tests/test_critical_review_classifier.py`) and are serialized via `Depends on` (Task 2 → 3 → 4 → 5) to avoid merge conflicts during parallel dispatch. The alternative is to combine all four test functions + docstring update into a single task — fewer batches, but a single implementer must author all four fixtures and assertion sets plus the docstring (~150 lines of net additions) in one pass. Serialized chain isolates each fixture pair for review. Flag if the user prefers the combined-task variant.
- **Trigger 3 pair as one task vs two**: Tasks 3a (no-straddle) and 3b (with-straddle) are bundled into Task 3 because the spec REQUIRES they share `finding`/`evidence_quote`/`fix_invalidation_argument` content (Req 4 isolation constraint) — authoring them in separate tasks risks the second author re-paraphrasing the shared envelope and breaking the isolation property. Flag if the user wants them split anyway.
- **Calibration step shipped in test docstring vs separate artifact**: each test task includes a one-time calibration step recorded in the test's docstring as the discrimination-evidence artifact. The alternative is to ship calibration transcripts as separate `tests/fixtures/calibration-trigger-N.md` files. Docstring-embedded is simpler (one diff per task) but the transcripts can be quite long. Flag if the user prefers the separate-artifact form.
- **Trim aggressiveness in §1a.vii**: the largest single block of recipe-style prose available for collapse sits in §1a.vii (lines 128–169) — the tier-1/3 verbatim-message catalog plus the tier-1 `outcome` field map. The plan tightens the catalog into a pointer to `daytime_result_reader`'s output schema while keeping the `outcome` field map inline (it is skill-side classification, not the helper's responsibility, and is required by `test_skill_contracts` lines 399–402). The alternative is to collapse the `outcome` map too, but the test would then fail. Flag if the user wants a more aggressive collapse than the plan describes.
- **Inline retry loops vs refactoring `_run_n_times`**: each new test inlines a small 2-of-3 retry loop wrapping the synthesizer-only subprocess (the minimum-blast-radius option). The alternatives are (a) refactor `_run_n_times` to accept a dispatch callable so the new tests can share retry infrastructure, or (b) introduce a new helper `_run_synthesizer_n_times` parallel to `_run_n_times`. Both alternatives create a shared helper used by 4 tests; inline duplication is small (~5 lines per test) and avoids touching infrastructure used by `test_weak_argument_downgrade`. Flag if the user wants the shared-helper path.
