# Specification: Instrument skill-name on dispatch_start for per-skill pipeline aggregates

> Background reference (epic context, do not reproduce): `research/extract-scripts-from-agent-tool-sequences/research.md` â€” observability floor section. This spec is scoped to ticket 104 only.

## Problem Statement

Today, `dispatch_start` events in `lifecycle/pipeline-events.log` (emitted at `cortex_command/pipeline/dispatch.py:446`) record `model`, `complexity`, `criticality`, `feature`, and budget metadata â€” but not which call-site (skill) initiated each sub-agent dispatch. As a result, per-skill cost aggregates are not computable, which blocks data-driven ranking of pipeline-side script-extraction candidates (parent epic 101, candidates C8 and C9) and makes post-ship ROI validation impossible for anything the pipeline dispatches. This spec adds a `skill` parameter (plus `attempt`, `escalated` (sticky), `escalation_event` (one-shot), and `cycle` for retry-aware and review-cycle-aware cost analysis) to `dispatch_task`, threads it through all 7 caller files, and extends `cortex_command/pipeline/metrics.py` with a conditional `(skill, tier)` / `(skill, tier, cycle)` aggregator and a new `--report skill-tier-dispatch` CLI mode that consumes `pipeline-events.log`.

## Non-Requirements

- **No retrofitting of historical event logs.** Historical `dispatch_start` events without `skill` are bucketed as `"legacy"` at read-time; no rewrite of past JSONL.
- **No interactive tool-call instrumentation.** That surface is owned by ticket 103; this ticket touches only pipeline sub-agent dispatch.
- **No dashboard changes.** Dashboard cost tracking reads `agent-activity.jsonl`; this ticket does not affect that pipeline (`requirements/observability.md:30â€“34`).
- **No central skill registry beyond the `Skill` Literal.** No separate `skills.py` module, no JSON config, no validation against external source. mypy + the runtime guard are the enforcement boundary.
- **No fix for the `pair_dispatch_events()` orphan blind spot.** Crashed dispatches still drop silently. A new backlog item is filed during implementation; this ticket only documents the gap in the report header.
- **No "other" or `null` escape hatch in the Literal.** Closed vocabulary; new dispatch sites must extend the Literal explicitly. mypy will reject unregistered strings; the runtime guard is defense-in-depth.
- **No composable `--group-by` CLI syntax.** Named-mode `--report skill-tier-dispatch` per user direction. Revisit if `--report` choices grow past 5.
- **No graceful degradation for the required `skill` kwarg.** Calling `dispatch_task` without `skill` raises `TypeError` (Python's standard behavior for missing required kwarg) â€” intentional fail-loud.

## Edge Cases

- **Caller passes a typo'd skill string**: e.g., `dispatch_task(skill="implemnt", ...)`. Runtime guard at dispatch.py:446 raises `ValueError("unregistered skill: 'implemnt'")`. (mypy would catch this at type-check time if CI runs mypy; runtime guard is the safety net.)
- **Historical event log entry without `skill` field**: aggregator buckets as `skill="legacy"`. The legacy bucket appears as a row in the `--report skill-tier-dispatch` output for traceability rather than being silently dropped.
- **Caller passes `cycle` for non-`review-fix` skill**: not prevented by the type system (cycle is `int | None` for all dispatches), but R14's symmetric runtime guard at dispatch.py:446 raises `ValueError("cycle is only valid for skill='review-fix'")`. mypy alone cannot catch this (the type allows it for any dispatch); runtime is the safety net.
- **Retry escalation Sonnetâ†’Opus**: produces `attempt=N, escalated=True, escalation_event=True` on the FIRST attempt at the new tier. Subsequent retries at the escalated tier (after a retry-class failure at Opus) produce `attempt=N+1, escalated=True, escalation_event=False` â€” `escalated` stays sticky, `escalation_event` is one-shot. Aggregator can answer "how often did we escalate" (count `escalation_event=True`) AND "how much cost lives at escalated tiers" (filter `escalated=True`) without conflating the two questions.
- **Organic Opus dispatch (initial model is already Opus)**: produces `attempt=N, escalated=False, escalation_event=False` on every attempt â€” the ladder dead-ends at Opus and never re-fires escalation. Filtering `escalated=False` correctly excludes only organic-tier dispatches.
- **Idempotency-skip path** (`feature_executor.py:574â€“585`): emits `task_idempotency_skip` with no `dispatch_start`. The per-skill aggregator under-counts work-attributable-to-implement vs `task_*` events. Disclosed in the report header.
- **Crashed dispatch with no terminal event**: existing `pair_dispatch_events()` silently drops. Disclosed in the report header; separate backlog item filed.
- **`integration_recovery.py:216` is a standalone CLI not invoked from the overnight orchestrator**: still uses dispatch_task directly with `skill="integration-recovery"`. Aggregator picks it up because the CLI also writes to a `pipeline-events.log` (verify file path during implementation).
- **In-flight overnight session at PR merge time**: per user direction (Q1: required from day 1), if an in-flight session running OLD code dispatches into NEW refactored modules mid-feature, it crashes with `TypeError: dispatch_task() missing 1 required keyword-only argument: 'skill'`. The crash surface is compound: the same TypeError fires at `retry.py:240` (where `retry_task` invokes `dispatch_task` and now must also forward `skill`), at `dispatch.py:446` for the initial dispatch, and across all 6 other caller modules that loaded as imports before the merge. A mid-feature retry escalation in flight at merge time crashes at the retry boundary before reaching dispatch_task. Mitigation: avoid landing this PR while an overnight session is running (single-operator project; coordination is self-discipline, not multi-party). Documented as a known constraint of the rollout choice; the user explicitly chose fail-loud over `Skill | None = None` in Spec Â§2 Q1 to prevent silent miss-bucketing of dispatches by future contributors who forget the kwarg.

## Changes to Existing Behavior

- **MODIFIED**: `dispatch_task` signature gains 5 new keyword-only parameters (`skill: Skill` required; `attempt: int = 1`, `escalated: bool = False`, `escalation_event: bool = False`, `cycle: int | None = None` with defaults). All call sites updated atomically in one PR.
- **MODIFIED**: `retry_task` signature gains `skill: Skill` required keyword-only parameter, threads to dispatch_task; computes `attempt`, `escalated`, and `escalation_event` internally. To support `escalated`, retry.py must snapshot `initial_model = current_model` immediately after the existing `current_model = resolve_model(...)` at retry.py:212, holding `initial_model` constant across the retry loop while `current_model` continues to mutate on `escalate` recovery paths (line 412). To support `escalation_event`, retry.py must additionally maintain a `previous_attempt_model` local (initialised to `None`, updated to `current_model` at the end of each loop iteration) so the next dispatch can compute `escalation_event = (current_model != previous_attempt_model)` â€” true only on the first attempt at a newly-escalated tier. Both `initial_model` and `previous_attempt_model` are new locals introduced by this ticket.
- **MODIFIED**: `dispatch_start` event JSONL shape gains `skill` (always present), `attempt` (always present, defaults to 1), `escalated` (always present, sticky bool, defaults to false), `escalation_event` (always present, one-shot bool, defaults to false), and `cycle` (present only for `review-fix` skill). Event-log readers must handle missing fields gracefully (forward-compatible).
- **MODIFIED**: `cortex_command/pipeline/metrics.py` `--report` argparse choice grows from `["tier-dispatch"]` to `["tier-dispatch", "skill-tier-dispatch"]`.
- **ADDED**: `compute_skill_tier_dispatch_aggregates()` function in metrics.py; `_format_skill_tier_dispatch_report()` formatter. Bucket key shape is conditional: `"<skill>,<tier>"` for non-review-fix skills, `"<skill>,<tier>,<cycle>"` for review-fix (preserves the cycle disentanglement promise from Spec Â§2 Q2 / critical-review Â§4 Q2).
- **ADDED**: `lifecycle/metrics.json` output gains `skill_tier_dispatch_aggregates` top-level key alongside the existing `model_tier_dispatch_aggregates`.
- **RENAMED in vocabulary** (compared to research's initial proposal): `merge-repair` â†’ `conflict-repair`; `test-repair` â†’ `merge-test-repair`. The rename happens before any dispatch_start event is written, so no historical data uses the old names.

## Technical Constraints

- **Append-only JSONL** (`requirements/pipeline.md:129`): new fields added forward-compatibly; never modify historical events.
- **Closed Literal vocabulary**: `Skill` is the canonical list. Adding a new dispatch site requires extending the Literal explicitly. No central registry beyond this type alias.
- **mypy at type-check + runtime guard at dispatch.py:446**: defense-in-depth. If CI mypy is bypassed, the runtime guard catches typos at dispatch time (fail loud, not at aggregator read time).
- **Python dict insertion order = JSONL key order**: insert new keys (`skill`, `attempt`, `escalated`, `escalation_event`, `cycle`) at the correct positions in the dict literal at dispatch.py:445â€“454. Downstream consumers parse by name (verified in research), but readability matters.
- **Test fixture compatibility**: `TestPairDispatchEvents` synthetic helpers (`_start()`, `_complete()`) at `test_metrics.py:54â€“287` need a `skill` parameter (default `"implement"`). Existing tier-dispatch tests should continue to pass after the helper signature change.
- **No regression on existing tier-dispatch aggregator**: `compute_model_tier_dispatch_aggregates()` and `_format_tier_dispatch_report()` must remain bytewise-equivalent in their output for unchanged inputs. Adding the new aggregator must not perturb the old one.
- **Caller-side discipline for `cycle`**: Only `review-fix` callers (review_dispatch.py:383 and :496) pass `cycle`. dispatch.py docstring documents this convention; R14's symmetric runtime guard enforces it (raises `ValueError` if `cycle is not None and skill != "review-fix"`).
