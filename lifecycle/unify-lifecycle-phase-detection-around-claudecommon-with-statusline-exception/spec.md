# Specification: Unify lifecycle phase detection around `cortex_command.common` with statusline exception

## Problem Statement

Lifecycle phase detection currently lives in **seven** places, not the four originally claimed: `cortex_command.common.detect_lifecycle_phase()` (canonical Python, returns bare phase string), the bash hook `hooks/cortex-scan-lifecycle.sh:170-207` (richer output with `implement:N/M` and `implement-rework:cycle`), the bash statusline `claude/statusline.sh:377-402`, the prose ladder in `skills/lifecycle/SKILL.md:43-66` (LLM-interpreted), the dashboard's `parse_plan_progress` at `cortex_command/dashboard/data.py:340-362` (re-derives plan progress with a different regex than the canonical detector), a sibling Python implementation at `tests/lifecycle_phase.py:28-91` (verbatim copy of the canonical's logic, exercised by `tests/test_lifecycle_state.py`), and a literal-string consumer at `cortex_command/dashboard/data.py:1196` (`compute_slow_flags`) that branches stall-detection on the exact value `"implement"`. The implementations are not currently equivalent — Python returns just `"implement"` for both mid-implement and CHANGES_REQUESTED-rework, while bash distinguishes them as `"implement-rework:$cycle"`. The drift surface is real and the comment-only mitigation already in place at `hooks/cortex-scan-lifecycle.sh:168` ("Mirrors claude.common.detect_lifecycle_phase — keep in sync") has empirically failed. Reducing the drift surface, reconciling semantics, and turning literal-string production branches into vocabulary-aware code shrinks the cognitive cost of maintaining the lifecycle phase model and prevents silent regressions in the rework path.

## Requirements

1. **Canonical Python detector returns structured output and full phase vocabulary**: `cortex_command.common.detect_lifecycle_phase(feature_dir: Path)` returns a `dict[str, str | int]` with keys `phase`, `checked`, `total`, `cycle`. The `phase` field admits the values `{"research", "specify", "plan", "implement", "implement-rework", "review", "complete", "escalated"}`. `checked` and `total` are integer plan-task counts (0 when no `plan.md`); `cycle` is the integer review cycle number (1 when no `review.md`, otherwise the number of `verdict` matches in `review.md`). **Acceptance**: `python3 -c "from cortex_command.common import detect_lifecycle_phase; from pathlib import Path; r = detect_lifecycle_phase(Path('lifecycle/unify-lifecycle-phase-detection-around-claudecommon-with-statusline-exception')); assert isinstance(r, dict) and set(r.keys()) == {'phase', 'checked', 'total', 'cycle'}, r"` exits 0.

2. **CLI default output is JSON; bare-string output retired**: `python3 -m cortex_command.common detect-phase <dir>` emits a single line of minified JSON to stdout (`{"phase":"...","checked":N,"total":M,"cycle":K}\n`) and exit 0. Errors emit plain text to stderr and exit non-zero. **Acceptance**: `python3 -m cortex_command.common detect-phase lifecycle/unify-lifecycle-phase-detection-around-claudecommon-with-statusline-exception | python3 -c "import json,sys; d=json.loads(sys.stdin.read()); assert set(d.keys())=={'phase','checked','total','cycle'}, d"` exits 0.

3. **Hook subprocesses to canonical CLI; bash glue layer is normatively specified**: `hooks/cortex-scan-lifecycle.sh:170-207` `determine_phase()` body is replaced with a subprocess call to `python3 -m cortex_command.common detect-phase <dir>`. The hook then re-emits the wire format consumed by statusline using this **normative encoding**:
   - `phase=="implement"` AND `total>0` → emit `"implement:$checked/$total"`
   - `phase=="implement"` AND `total==0` → emit `"implement:0/0"` (preserves prior behavior at `claude/statusline.sh:540`)
   - `phase=="implement-rework"` → emit `"implement-rework:$cycle"`
   - any other phase → emit the bare phase string verbatim (`"complete"`, `"research"`, etc.)
   The bash ladder regexes at L170-207 are removed. **Acceptance**: `grep -cE 'sed -n.*verdict|grep -cE.*Status.*\[ x\]' hooks/cortex-scan-lifecycle.sh` returns 0; `grep -c 'python3 -m cortex_command.common detect-phase' hooks/cortex-scan-lifecycle.sh` returns ≥1.

4. **Hook hard-fails on missing Python or cortex_command, but only after confirming a cortex repo**: The precondition check (`command -v python3 >/dev/null && python3 -c "import cortex_command.common" 2>/dev/null`) runs **after** the existing `[[ -d "$LIFECYCLE_DIR" ]] || exit 0` guard at `hooks/cortex-scan-lifecycle.sh:21` and **before** the feature-dir iteration. Non-cortex repos exit 0 unchanged at L21. Cortex repos with missing Python or cortex_command emit a stderr diagnostic (`cortex_command not available; cortex-scan-lifecycle hook requires the cortex CLI — install via 'uv tool install -e .' from the cortex-command repo`) and exit non-zero. **Acceptance**: in a fresh tmpdir, `cd "$(mktemp -d)" && mkdir lifecycle && PATH=/usr/bin bash <repo>/hooks/cortex-scan-lifecycle.sh </dev/null; echo $?` returns non-zero with the diagnostic on stderr; in the same tmpdir without `mkdir lifecycle`, the same command exits 0 silently.

5. **Skill prose ladder replaced with CLI invocation**: `skills/lifecycle/SKILL.md` Step 2 "Artifact-Based Phase Detection" (currently L41-66) is rewritten to instruct the agent to invoke `python3 -m cortex_command.common detect-phase <dir>` and route on the returned `phase` field. A brief reference table mapping each `phase` value to its semantic meaning is retained for narrative clarity (one line per phase). **Acceptance**: `grep -c 'python3 -m cortex_command.common detect-phase' skills/lifecycle/SKILL.md` returns ≥1; `grep -cE 'plan\.md exists with all \[x\]' skills/lifecycle/SKILL.md` returns 0.

6. **Dashboard `parse_plan_progress` folded into canonical detector**: The `parse_plan_progress` function at `cortex_command/dashboard/data.py:340-362` is removed. Its single caller is updated to consume `checked`/`total` from the canonical detector's dict return (passed in via `parse_feature_events`). **Acceptance**: `grep -c 'def parse_plan_progress' cortex_command/dashboard/data.py` returns 0; `grep -rc 'parse_plan_progress' cortex_command/dashboard/` returns 0.

Its single consumer `tests/test_lifecycle_state.py` is updated to import `detect_lifecycle_phase` from `cortex_command.common` instead, and its assertions are updated to read the `phase` field from the dict return. **Acceptance**: `test -f tests/lifecycle_phase.py; echo $?` returns 1 (file does not exist); `grep -rc 'tests.lifecycle_phase\|from .lifecycle_phase' tests/` returns 0; `pytest tests/test_lifecycle_state.py -x` exits 0.

8. **`compute_slow_flags` updated to handle `implement-rework`**: `cortex_command/dashboard/data.py:1196` literal-string branch on `current_phase == "implement"` is updated to handle both `"implement"` AND `"implement-rework"` for stall-detection purposes (both phases can stall and both should be eligible for the slow flag). **Acceptance**: `grep -E 'current_phase == "implement-rework"|current_phase in \(.*"implement-rework"' cortex_command/dashboard/data.py` returns at least one match in the `compute_slow_flags` function; a new test case in `cortex_command/dashboard/tests/test_data.py` exercises the rework path and asserts the slow flag fires when the rework phase has been active beyond the threshold.

## Non-Requirements

- **NOT migrating `backlog/index.json` schema or dashboard `current_phase` to a structured object.** Both remain scalar strings; the dict is the function's in-process API only. Boundary projection at consumers handles this (R10).
- **NOT adding a `--json` flag or `detect-phase-detailed` sibling subcommand.** The CLI default switches to JSON in this change. Bare-string output is retired.
- **NOT adding a bash fallback ladder in the hook for the case where Python is unavailable.** Hook hard-fails per Requirement 4 (after the lifecycle-dir guard).
- **NOT migrating the statusline to Python or codegen.** The < 500ms latency budget is structural; the bash ladder stays. Equivalence is enforced by the three-layer parity tests in R12.
- **NOT adding `cortex detect-phase` as a top-level `cortex` CLI subcommand.** Following the existing precedent at `hooks/cortex-scan-lifecycle.sh:379` (`python3 -m cortex_command.pipeline.metrics`), the hook uses `python3 -m cortex_command.common detect-phase`.
- **NOT modifying the plugin manifest at `plugins/cortex-overnight-integration/.claude-plugin/plugin.json` to declare a Python dependency.** Runtime check (R4) handles dependency enforcement; manifest schema change is out of scope.
- **NOT updating `cortex_command/overnight/backlog.py` `TERMINAL_STATUSES` (the duplicate noted at `cortex_command/common.py:41-42`).** Separate follow-up.
- **NOT enforcing cycle parity between statusline and canonical Python.** The statusline ladder does not extract `cycle` from `review.md` (verified at `claude/statusline.sh:382-388`). R12b explicitly excludes the cycle dimension from the statusline parity comparison; cycle correctness is enforced by R12a (glue unit test) and R12c (end-to-end hook test) instead.

## Edge Cases

- **Empty lifecycle directory** (no artifacts at all): canonical detector returns `{"phase": "research", "checked": 0, "total": 0, "cycle": 1}`. Glue emits `"research"`. Statusline emits `"research"`. R12 fixtures cover.
- **`plan.md` with zero `**Status**: [x]` patterns** (no checkboxes whatsoever): canonical returns `{"phase": "implement", "checked": 0, "total": 0, "cycle": 1}`. Glue emits `"implement:0/0"` per R3 normative encoding. Statusline at L540 sees `_lc_total=0`, leaves `_lc_p_pct=0`, displays `"Implement 0/0"`. Consistent.
- **`review.md` with multiple verdicts** (cycle 2+): canonical reads the **last** `verdict` regex match and counts total matches as `cycle`. Glue emits `"implement-rework:$cycle"` for CHANGES_REQUESTED. Hook end-to-end test (R12c) covers cycle 1 and cycle 2; statusline ladder does not extract cycle (acknowledged in Non-Requirements + R12b).
- **`events.log` contains `feature_complete` AND `review.md` exists**: events.log wins; canonical returns `{"phase": "complete", ...}`. Cycle field is whatever `review.md` indicates (or 1 if absent).
- **Hook invoked when `python3` is on PATH but `cortex_command` is not importable** (wrong venv): if a `lifecycle/` dir exists in CWD, R4 precondition fails after L21 guard; hook exits non-zero with diagnostic. If no `lifecycle/` exists in CWD, hook exits 0 at L21 silently. The precondition check is a no-op for non-cortex repos.
- **Hook invoked outside a cortex repo** (no `lifecycle/` directory): existing L21 guard handles this. R4 precondition does not run. No change in behavior. Acceptance test in R4 includes the negative case (no `lifecycle/` → exit 0).
- **`detect_lifecycle_phase` called with a non-existent `feature_dir`**: returns `{"phase": "research", "checked": 0, "total": 0, "cycle": 1}` (artifact-presence default).
- **Multiple feature directories at `lifecycle/*/`**: hook subprocesses N times, paying ~30–80ms cold start per invocation. For N≥10 this approaches a noticeable SessionStart delay. **Mitigation**: profile during implementation; if cumulative time exceeds 1s on a representative project, fold the loop into a single Python invocation that processes all dirs and emits an array of results.
- **`current_phase == "implement-rework"` for stall detection**: R8 ensures `compute_slow_flags` includes `implement-rework` as a slow-eligible phase. Without R8, rework features would silently bypass stall detection — a regression caught by reviewer 4.
- **Backlog item with rework state in `index.json`**: after this change, `backlog/index.json` may contain `"lifecycle_phase": "implement-rework"`. R10 documents the vocabulary expansion. Existing consumers that filter by exact-string match on `"implement"` will not see rework-state features under that filter; this is the intended behavior (rework is now a distinguishable phase).

## Technical Constraints

- **Statusline latency budget**: `< 500ms` per invocation (`requirements/observability.md:23, 91`). Subprocess to Python adds ~30–80ms per call. Statusline must remain pure bash.
- **Plugin distribution byte-identity**: top-level `hooks/cortex-scan-lifecycle.sh` is canonical; `plugins/cortex-overnight-integration/hooks/cortex-scan-lifecycle.sh` is a byte-identical mirror regenerated by `just build-plugin` (`justfile:475-507`); enforced by `.githooks/pre-commit:149-176`.
- **Hook subprocess cost scales with N feature directories**: SessionStart iterates `lifecycle/*/`. For N feature directories, total cold-start cost is N × ~30–80ms. Acceptable for typical projects (N < 10); profile during implementation for larger projects.
- **`cortex_command/common.py` import graph stays stdlib-only**: imports `json`, `os`, `re`, `sys`, `tempfile`, `pathlib`. No new imports.
- **File-based state with no locks**: lifecycle artifacts are read concurrently; atomic writes (`os.replace`) and forward-only state transitions make concurrent reads safe.
- **`bin/cortex-*` parity gate**: applies to `bin/cortex-*` only, not to `python3 -m cortex_command.X` CLIs.
- **`requires-python = ">=3.12"`** (`pyproject.toml:8`): type hints in the new dict-returning detector use modern syntax.
- **Bash compatibility**: hook and statusline must remain compatible with macOS bash 3.2+. JSON parsing in the hook uses `jq` if available with sed/awk fallback.
- **Statusline cycle-blindness is structural, not a bug**: the bash ladder at `claude/statusline.sh:382-388` reads only `verdict`, never `cycle`. Adding cycle extraction would either require modifying the ladder (forbidden by R11) or adding a sed pass that erodes the latency budget. The non-requirement at the bottom of Non-Requirements documents this; R12b excludes cycle from the statusline parity dimension.
