### S1/S5 server analysis

`_CORTEX_ROOT_CACHE` is a module-level global (server.py:191), populated lazily in `_get_cortex_root_payload()` on first call, never invalidated for the MCP server's lifetime. The function's own try-block (lines 217-243) catches `RuntimeError` (non-zero exit) and `json.JSONDecodeError` (parse failure) — but **does NOT catch `FileNotFoundError`** from `subprocess.run`. The OSError catch is at *call sites* (e.g., line 467 in `_maybe_check_upstream`), and several call sites (lines 1485, 974, 1149, plus delegations) invoke `_get_cortex_root_payload()` with no try/except — meaning a missing `cortex` binary crashes those dispatches.

Subprocess error pattern at most leaf delegation sites: `try/except (subprocess.TimeoutExpired, OSError)`, no `check=True`, NDJSON-log to stderr, return None to signal graceful degradation. The pattern is consistent at most call sites but **not** at the discovery-cache bootstrap.

Plugin invariant (server.py:1-30): zero `cortex_command.*` imports; sole contract is `subprocess.run` + versioned JSON. Major schema mismatch rejected; minor-greater accepted; unknown fields ignored via Pydantic `extra="ignore"`.

### Conventions to follow

- Atomic file writes (`os.replace` after tempfile) — applies to N6 if a manifest is added.
- `feature_complete` JSON event in events.log is the binary completion marker.
- Subprocess error handling: catch `(TimeoutExpired, OSError)`, no `check=True`, NDJSON-log, return structured error.
- Schema versioning + extra="ignore" for forward compatibility on JSON contracts.
- Dual-source: edit canonical only; pre-commit regenerates mirrors and verifies drift.

### N8 — replacement source for the progressive-disclosure citation

Recommended primary source: **`https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview`** (Anthropic Agent Skills overview). Has an explicit section *"Three types of Skill content, three levels of loading"* (Level 1 Metadata, Level 2 Instructions, Level 3 Resources/code) with token-cost framing. **Caveat (from adversarial review)**: `docs.claude.com` 302-redirects to `platform.claude.com`. Anthropic's docs site has no published commitment to URL stability — citing a versioned external URL accumulates the same kind of debt N8 itself documents. Mitigation: pair the URL with a brief inline self-contained description so future doc churn doesn't re-stale this fix.

### S1 prior art — cache invalidation for cached external-tool paths

- **pipx parallel** (`pypa/pipx` issue #449): caches venv paths, *crashes* when target disappears post-upgrade, recommends manual `pipx reinstall-all`. Worse UX than auto-recovery; not appropriate for long-lived MCP servers.
- **Anthropic MCP Python reference** (`anthropics/skills/skills/mcp-builder/reference/python_mcp_server.md`): does NOT cover subprocess-path caching. Covers HTTP/Pydantic patterns only.
- **MCP `tools/list_changed`**: server→client cache invalidation for tool listings, not internal subprocess paths.
- **Standard Python idioms**: `cachetools.TTLCache`, exception-driven re-resolution. The exception-driven pattern (catch `FileNotFoundError`, re-resolve, retry once) targets the actual failure mode (paths go from valid to FNF instantly at upgrade boundaries; TTL adds a stale-but-served window without solving the failure).

### S5 prior art — MCP graceful-degrade when external binary is missing

- **Anthropic MCP Python reference** prescribes errors as **string return values from tool functions**, not raising exceptions to the protocol layer. Canonical shape: `"Error: <human-readable message with remediation hint>"`.
- **MCP spec**: no formal "missing-dependency" hook; degradation is a server-implementation concern.
- **Real-world MCP failures** (AWS OpenAPI, Playwright, Firebase): hard crashes are the prevalent anti-pattern; lesson is "don't kill the server process for a single missing dependency."
- **Claude Code Read-tool issue #44735**: explicit precedent for graceful degradation — return useful metadata instead of failing the whole turn.

### Known patterns / anti-patterns

| | |
|---|---|
| Pattern | Lazy re-discovery on `FileNotFoundError`, single retry, no TTL (S1) |
| Pattern | Per-tool-call dependency validation; structured error strings with remediation (S5) |
| Pattern | Errors as `str` return values, prefixed `"Error: ..."`, with one actionable next step |
| Anti-pattern | Hard crash / sys.exit at startup on missing dependency (breaks `initialize`) |
| Anti-pattern | TTL caches for "external tool may be upgraded" — wrong shape; stale-but-served window |
| Anti-pattern | Citing in-repo files that may be retired (the N8 root cause itself) |

## Tradeoffs & Alternatives

### S1 — MCP cache invalidation

| Option | Pros | Cons |
|---|---|---|
| **A1** path-existence check before reuse | simple, lightweight | misses stale symlinks/version mismatches; per-call file I/O |
| **A2** invalidate-on-FileNotFoundError | reactive, minimal surface, aligns with existing pattern | needs scope: which call site; retry-once logic |
| **A3** TTL expiry (5 min) | catches drift without exception | wrong failure shape (paths go from valid→FNF instantly); thrashes |
| **A4** eager re-detect each call | no stale risk | 30s timeout overhead per call; defeats caching |
| **A5** combined A1+A2 | defense in depth | duplicated handlers; unclear semantics |

**Recommended: A2 (Invalidate-on-FileNotFoundError) with adversarial-fix corrections** — wrap `_get_cortex_root_payload`'s subprocess call (server.py:217-222) in a `FileNotFoundError` handler that raises a typed `CortexCliMissing` exception. Call sites either (a) clear `_CORTEX_ROOT_CACHE`, retry exactly once with a re-discovered payload (with an attempt counter to prevent infinite recursion), or (b) return a structured error to the tool caller. **Do not generic-retry-on-OSError** — that catches network/permission errors that are not stale-path symptoms.

### S5 — MCP graceful-degrade for missing CLI

| Option | Pros | Cons |
|---|---|---|
| **B1** startup check (fatal) | early detection | breaks plugin install/initialize for marketplace users |
| **B1'** startup check (non-fatal stderr log) | early detection without breaking initialize | startup message may be missed in interactive UX |
| **B2** lazy first-call detection | contextual feedback; respects marketplace install contract | invisible to autonomous-overnight users until morning report |
| **B3** periodic re-check | dynamic | overkill; threading complexity |
| **B4** install-pointer in tool list | discoverable | adds fake tool; doesn't prevent real failures |
| **B5** combined B1' + B2 | dual notification (startup-stderr + per-call structured error) | code at two sites |

**Recommended: B5 (hybrid B1' + B2)** — adversarial review correction over the agents' B2-only recommendation. For autonomous-overnight first-use, a structured error in tool response is invisible to humans until morning; a startup-stderr log lands in MCP debug output / morning postmortems. Per-call structured error gives interactive users contextual feedback. Both are needed.

**Install-pointer text**: agents originally suggested `uv tool install -e .` — wrong for marketplace users (requires local clone). Use `https://github.com/charleshall888/cortex-command#install` (anchor-only, stable) and mention both `install.sh` (marketplace path) and `uv tool install -e .` (dev-mode path).

### N6 — lifecycle-archive sample strategy

| Option | Pros | Cons |
|---|---|---|
| **C1** 5-dir sample then full | low risk; manually inspectable | tiny sample misses edge cases (e.g., cross-references) |
| **C2** ~20-dir medium sample | better coverage | spot-check vs full validation tradeoff |
| **C3** dry-run first | full coverage, no risk | recipe doesn't currently support `--dry-run` |
| **C4** skip sample, full run | fastest | no staged validation |

**Recommended: C3 with recipe modification** — adversarial review surfaced that bare `mv` will break ~730 cross-lifecycle path references in research/spec/plan files (verified via `grep -rn "lifecycle/[a-z][a-z-]*/" lifecycle/*/research.md lifecycle/*/spec.md lifecycle/*/plan.md`). Plus first-5-by-listing is unrepresentative. Recipe needs:
- `--dry-run` mode (output would-be-archived list without moving).
- Pre-flight cross-reference grep against the dry-run list (warn or block on broken refs).
- Manifest write before each `mv` so partial failures can be rolled back.
- Either: prefix-rewriting of `lifecycle/{slug}/` → `lifecycle/archive/{slug}/` references in moved files AND in remaining lifecycle/backlog/research files that cite them, OR explicit acceptance that archived paths break and consumers (morning report, lifecycle navigation) handle this.

If recipe-repair scope grows beyond a small fix, defer the full archive run to a separate ticket and limit this ticket to recipe enhancements + a documentation-only sample-of-three demonstration. Recipe-repair is in-scope per the user direction during clarify.

## Adversarial Review

### Failure modes and edge cases

- **N3/N4 over/under-broadening**: The "skills/, hooks/, plugins/*/skills/" recommendation under-broadens by missing `claude/hooks/` (8 production hook scripts including `cortex-skill-edit-advisor.sh`, `cortex-output-filter.sh`, `cortex-sync-permissions.py`, `cortex-worktree-create.sh`, `cortex-worktree-remove.sh`), `bin/cortex-*` utilities (lifecycle-relevant per CLAUDE.md), and `claude/common.py` (lifecycle slug helper, referenced by `skills/lifecycle/SKILL.md:35`). It over-broadens by including `plugins/*/skills/` as a generic trigger when `plugins/cortex-interactive/skills/` is build-output (the user shouldn't edit it; gate firing tells them to start a lifecycle when what they need is "edit the canonical instead").

- **N6 will break ~730 cross-lifecycle references**: `grep -rn "lifecycle/[a-z][a-z-]*/" lifecycle/*/research.md lifecycle/*/spec.md lifecycle/*/plan.md` returns ~730 hits, plus ~140 from `backlog/` and `research/`. The recipe's bare `mv` does no rewriting. C1 (tiny sample) doesn't catch this because 5 dirs is too small to surface the ref-graph.

- **N6 `feature_complete` not sufficient quality marker**: The string can be written by out-of-band closure paths (`/cortex-interactive:lifecycle "Close lifecycle"` per SKILL.md:96-107) where the directory has only events.log + index.md, no research/spec/plan. Archiving these is fine, but the *real* test is "is anything else citing it" — which the recipe never asks.

- **N6 stale-symlink fragility**: Recipe uses `realpath "$dir"` for worktree comparison; stale symlinks (left by removed worktrees) either error out (set -e kills the recipe) or resolve incorrectly.

- **S1 retry-once trap**: `_get_cortex_root_payload` itself raises `FileNotFoundError` if cortex is uninstalled. Naive retry-by-recall loops infinitely without an attempt counter. Plus `FileNotFoundError` can come from a child subprocess inside `cortex` (e.g., cortex invokes `git`) — false-positive invalidation.

- **S5 lazy-only fails autonomous-first-use**: For overnight first-use with missing CLI, structured error in tool response is invisible until morning report. Hybrid B5 needed (startup stderr log + per-call structured error).

- **S5 wrong install command**: `uv tool install -e .` requires local clone; marketplace users don't have one. Correct pointer: `install.sh` (curl pipe to sh) per `docs/setup.md:27`, or `https://github.com/charleshall888/cortex-command#install`.

- **S5 doesn't catch missing `CORTEX_COMMAND_ROOT`**: per `docs/setup.md:53`, the overnight plugin requires this env var. Even with `cortex` on PATH, tool calls may fail. S5 as scoped doesn't address this distinct missing-config case.

- **N8 missed runtime reference**: Initial scope (README + docs/overnight-operations.md) misses the most agent-impactful instance: `claude/hooks/cortex-skill-edit-advisor.sh:30,37,55,63` prints `~/.claude/reference/context-file-authoring.md` to Claude on every SKILL.md edit. This live agent-facing output is the dominant N8 failure mode.

- **N9 removing line strands users**: `setup-github-pat.sh` lives only in `claude/hooks/`, NOT in any plugin. Per `docs/agentic-layer.md:206` it's a "manual-wire helper." Deleting the symlink instruction without replacement leaves users with no install path — including the maintainer setting up new machines.

- **N3/N4 dual-source rebuild on edit**: Editing canonical SKILL.md auto-regenerates the plugin mirror via pre-commit. This is correct behavior, but worth confirming the lifecycle gate semantics (does the gate fire on auto-generated outputs or only on user-initiated edits?).

### Security concerns / anti-patterns

- **S5 install-docs URL in persisted error log**: Structured errors land in NDJSON `last-error.log`. URL must be auth-free and stable; avoid versioned/tokened URLs.
- **N6 partial failure has no rollback**: `set -euo pipefail` aborts mid-run; some dirs already moved with no manifest of what was moved. Need explicit manifest-before-mv.
- **S1 cross-process cache concerns**: The MCP server is single-process (so module globals are fine for one MCP), but if multiple MCP processes ever coexist (unlikely but worth noting), each holds independent stale caches. Out of scope for this ticket per server-process model, but worth documenting.

### Sequencing / interaction

- **S1+S5 interact at FNF**: If both implement FNF handlers, ensure exactly one returns the user-facing error. Recommended order: implement S5 first (establishes structured-error response shape), then S1 (which can return that same shape when re-discovery fails).
