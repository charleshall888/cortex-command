# Specification: Disambiguate orchestrator prompt tokens to stop lexical-priming escape

> Epic reference: `research/orchestrator-worktree-escape/research.md` (epic 126). This ticket implements the prompt-disambiguation repair item named in the epic's feasibility table as the cheapest high-leverage fix for the session-1708 failure mechanism. Other repair items from the epic (pre-commit hook, morning-report un-silence, followup-item persistence, frontmatter rollback, PR-gating, worktree GC) are out of scope here.

## Problem Statement

Session `overnight-2026-04-21-1708` failed at `feature_start` on all three assigned features because the orchestrator agent mis-substituted per-feature paths — producing absolute paths with literal `{slug}` text, or omitting substitution entirely. Root cause is a token-name collision plus lexical priming in `claude/overnight/prompts/orchestrator-round.md`: session-level tokens (pre-filled by `fill_prompt()` in `runner.sh:386-391` as absolute home-repo paths) and per-feature tokens (expected to be substituted by the orchestrator at runtime from `state.features[<slug>]`) share identical `{token}` syntax, and `{plan_path}` is used for both semantics. The prompt is full of `Path("{session-token}")` literals that prime the agent to treat every `{token}` as an already-substituted absolute path. This spec eliminates both the name collision and the shape-level priming, and establishes the orchestrator's substitution responsibility as an explicit contract.

Occurrences of `{feature}` elsewhere in the file (inside Step 0 Python examples like `entry["feature"]` at lines 89, 102, 108, 109, 115, 136, 148 — which are live Python-dict-access expressions, not template tokens) are **intentionally unchanged** and remain.

8. **Deploy atomicity**: the prompt edits (`orchestrator-round.md`) and the shell edit (`runner.sh:387`) must land **in a single git commit**, AND the merge must occur outside any active overnight-runner process lifetime. Acceptance — three sub-checks:
   - (8a) The implementing commit's diff (`git show <sha>`) touches both `claude/overnight/prompts/orchestrator-round.md` AND `claude/overnight/runner.sh` in one commit and MUST NOT be split across two commits.
   - (8b) Before merging to main, the implementer MUST verify that no overnight runner process is active. Concrete check: `ls ~/.local/share/overnight-sessions/active-session.json 2>/dev/null` returns nothing, OR the file's recorded PID is not alive (`ps -p $(jq -r .pid ~/.local/share/overnight-sessions/active-session.json)` exits non-zero). This check is operational — the implementer performs it pre-merge, not CI.
   - (8c) **Critical caveat**: single-commit atomicity protects only against a NEW session starting with mismatched files on disk. It does NOT protect against an ALREADY-RUNNING session. `runner.sh` is sourced once by bash at session start and its `fill_prompt()` body is held in memory for the session's full lifetime. If a `git pull` (or merge) updates the prompt file on disk mid-session, the in-memory `fill_prompt()` still substitutes `{plan_path}` (no longer present in the new prompt), leaking the `{session_plan_path}` literal into the filled prompt. The mitigation is (8b) — do not deploy while a session is running. There is no post-hoc defense for this race; the deploy window MUST be coordinated manually.

   **Future-work pointer**: automating (8b) via a pre-commit or pre-push hook that consults `~/.local/share/overnight-sessions/active-session.json` is out of scope for this ticket but is a reasonable follow-up. Track separately if operator discipline proves insufficient.

## Non-Requirements

- **Not fixing the `BacklogItem.plan` override across the downstream pipeline**. The state-schema field `state.features[<slug>].plan_path` is written by `claude/overnight/plan.py:404` (`plan_path=item.plan if item.plan else f"lifecycle/{slug}/plan.md"`), but is **only read by the orchestrator prompt** (lines 244 and — post-this-spec — the `{{feature_plan_path}}` token substituted at dispatch). Every other downstream consumer hardcodes `lifecycle/{feature}/plan.md`: `claude/overnight/feature_executor.py` at lines 238, 518, 686, 755; `claude/overnight/report.py:734`; `claude/overnight/daytime_pipeline.py` at lines 208, 325, 328, 331; `claude/dashboard/seed.py:101`. Consequently, a backlog item with a custom `plan:` frontmatter override will have its plan-gen sub-agent write to the override path (state-driven via `{{feature_plan_path}}`), but the downstream executor/report/dashboard will read from the default path `lifecycle/{slug}/plan.md` and silently miss the override. **This is a pre-existing pipeline-wide bug, not introduced by this ticket.** Unifying all 7+ sites on a single source of truth (state-driven OR hardcoded-literal, chosen consistently) is a separate unification ticket; this spec preserves the current semantics. The orchestrator prompt is made internally consistent (state-driven throughout) but is not aligned with the downstream pipeline.
- **Not modifying `claude/overnight/feature_executor.py:518`** (or the sibling hardcodes at lines 238, 686, 755 in the same file, or in `report.py`, `daytime_pipeline.py`, `dashboard/seed.py`). See the preceding bullet for the reasoning — a state-unification change across 7+ sites is outside this ticket's prompt-disambiguation scope.
- **Not validating `state.features[<slug>]` field contents** for injection/escaping. The state-field → sub-agent-prompt flow is unchecked; this ticket preserves that behavior (neither regressing nor improving it). Injection hardening, if warranted, is a separate security-focused ticket.
- **Not migrating other prompts** (`batch-brain.md`, `repair-agent.md`, pipeline prompts under `claude/pipeline/prompts/`) to the double-brace convention. Those prompts are single-layer (all tokens pre-filled by a Python dispatcher via `_render_template` or `.replace()` chains) and do not exhibit the priming vulnerability. Convention codification for future dual-layer prompts is deferred.
- **Not renaming the `PLAN_PATH` shell env var**, the `DEFAULT_PLAN_PATH` Python constant, the `BATCH_PLAN_PATH` batch-round filename, or the `state.features[<slug>].plan_path` state-schema field. Only the prompt-facing token string changes.
- **Not adding live/integration test harness** for orchestrator behavior. Acceptance-criterion 3 from the original ticket body ("a re-run of a failed-plan-parse style scenario produces correctly-substituted per-feature paths") is **dropped** as an automated requirement per interview decision; the unit test (Requirement 7) is the CI-gated validation and it validates shell-layer only. Live observation of the first post-deploy overnight session is the informal agent-layer check, not a release gate.
- **Not adding a prompt-linter or pre-commit hook** to detect future dual-layer token collisions. Convention enforcement is deferred.
- **Not changing `orchestrator-round.md` line numbers** reported in the ticket body (258-285) to match current on-main drift (260-285). Line numbers in this spec are approximate; grep-based acceptance criteria are the source of truth.

## Edge Cases

- **Cross-repo `PLAN_PATH` override at `runner.sh:278`**: Expected behavior — `{session_plan_path}` resolves to a home-repo path when the session is cross-repo (mirror of current `{plan_path}` behavior). Name alone does not imply location; the runner's existing cross-repo semantics are preserved and are documented in runner.sh.
- **`fill_prompt()` reaching double-brace tokens**: Expected behavior — `fill_prompt()` uses `str.replace('{session_plan_path}', ...)` and other single-brace-exact-name replacements. The Python string `'{session_plan_path}'` does NOT match against `'{{feature_plan_path}}'` (different characters at every position, no exact substring match on the shorter token within the longer one because the names differ). Session-level token names are disjoint from per-feature token names by design (session names use `session_` prefix; per-feature names use `feature_` prefix), so no accidental substitution can occur. Unit test R7e validates the double-brace tokens reach the filled output untouched.
- **Orchestrator mis-substitutes double-brace tokens**: Expected behavior — Anthropic's `{{double_brackets}}` is the documented template-variable convention Claude is trained on. The orchestrator should interpret `{{feature_slug}}` as a substitutable variable. If the first post-deploy overnight session nevertheless surfaces mis-substitution (agent fails to substitute, or substitutes to wrong value), escalate to C (build-time inline) or D (tool-call dispatch) alternative in a follow-up ticket. **Failure shape**: an un-substituted `{{feature_plan_path}}` in the sub-agent's plan-gen prompt would cause the sub-agent to write to a literal filename `{{feature_plan_path}}` on disk (or fail outright), which existence-check at line 244 catches on the next round. Detection is the same as pre-fix — on the first round's feature_start or plan-gen attempt.
- **Orchestrator treats `<substitution_contract>` block as data vs instruction**: Expected behavior — the XML-tagged block follows Anthropic's prompt-structuring guidance for role/instruction demarcation. The imperative markers required by R6d (`MUST NOT`, `YOU substitute`) and R6f (`CRITICAL:` / `IMPORTANT:` / `YOU MUST:` opener) are designed to break out of preamble-reading mode. If the block is ignored in practice (evidence: first-overnight mis-substitution happens despite the block), the fallback is the same as the preceding edge case.
- **Mid-session prompt/runner skew during deploy**: Prevented by R8's single-commit requirement plus R8b's pre-merge check that no active runner is running. R8c explicitly documents the residual risk: a mid-session `git pull` leaves `runner.sh`'s already-loaded bash function body pointing at the old token name (`{plan_path}`) while the on-disk prompt has the new name (`{session_plan_path}`); that session's substitution silently breaks. There is no automated defense; R8b is the only gate. Operator discipline is required.
- **Tests that pattern-match `orchestrator-round.md`**: `tests/test_events_contract.py:20` matches `log_event(` occurrences (unaffected by token rename). `tests/test_runner_signal.py:86` uses a minimal stubbed template that doesn't reference `{plan_path}` (unaffected). No test file asserts on the token names being changed.
- **`BacklogItem.plan` frontmatter override**: Expected behavior — for a feature whose backlog item sets `plan:` to a custom path, `state.features[<slug>].plan_path` reflects the override. The orchestrator (post-spec) substitutes `{{feature_plan_path}}` with the override path; the plan-gen sub-agent writes there. The downstream executor/report/daytime_pipeline/dashboard reads from the hardcoded default `lifecycle/{feature}/plan.md` and misses the override. This is a **pre-existing pipeline-wide inconsistency** (documented in Non-Requirements), not introduced by this ticket. No behavior change vs pre-spec — the orchestrator was already state-driven at line 244 and the downstream consumers were already hardcoded.

## Technical Constraints

- **Delimiter choice rationale**: Per-feature tokens use double-brace `{{feature_X}}` syntax matching Anthropic's documented `{{double_brackets}}` template convention — Claude is trained on this shape as a substitution target. Session-level tokens remain single-brace `{X}`. The two tiers are visually distinct (double vs single brace) AND semantically grouped (per-feature tokens carry `feature_` prefix, session tokens carry `session_` prefix where applicable), providing two layers of disambiguation. An alternative angle-bracket convention (`<feature.X>`) was considered and rejected during critical review because it collides with Claude's structural-XML trained prior and risks the orchestrator parsing the tokens as empty XML elements to preserve literally.
- **Deploy atomicity**: `fill_prompt()` re-reads the prompt file each round, but `runner.sh` is loaded once by bash at session start. Splitting the two edits across two commits creates a window where an in-flight session sees mismatched substitution (new prompt, old substitution table). Requirement 8 mandates a single commit and a pre-merge no-active-runner check. Mid-session deploys are NOT defended against beyond that operational check.
- **Single-layer prompts remain single-layer**: `batch-brain.md`, `repair-agent.md`, and `claude/pipeline/prompts/*.md` are fully pre-filled by Python dispatchers (`brain.py:214` via `_render_template`, `conflict.py:290-298` via chained `.replace()`, `feature_executor.py:560` via `_render_template`) and must continue to use single-brace `{token}` syntax. The double-brace convention introduced in this spec is scoped to `orchestrator-round.md`'s dual-layer dispatch block only.
- **`fill_prompt()` substitution contract**: The `str.replace` chain at `runner.sh:386-391` is whitelist-based (six tokens: `{state_path}`, `{plan_path}`, `{events_path}`, `{session_dir}`, `{round_number}`, `{tier}`). This spec changes one entry (`{plan_path}` → `{session_plan_path}`); no other tokens in that chain are touched. Python `str.replace('{session_plan_path}', VALUE)` does not match the distinct string `{{feature_plan_path}}` (no substring match since the per-feature name differs), so the double-brace tokens pass through `fill_prompt()` untouched into the orchestrator's context. The orchestrator agent performs the per-feature substitution at sub-agent dispatch time, reading from `state.features[<slug>]`.
- **Agent-layer validation limitation**: The unit test in R7 validates shell-layer `fill_prompt()` behavior only. The agent's runtime substitution of `{{feature_X}}` tokens has no automated validation — the spec accepts this limitation per interview decision. The first post-deploy overnight session's `feature_start` outcomes serve as informal agent-layer evidence; a silent regression of the session-1708 bug is possible and would cost one overnight cycle to detect. R6's strengthened imperative markers (MUST NOT, YOU substitute, CRITICAL/IMPORTANT/YOU MUST opener) are the primary mitigation for agent-layer risk.
- **XML tag naming**: Use `<substitution_contract>` as the tag. No other XML tags of this name exist in the repo (`grep -r substitution_contract` returns zero matches pre-fix); post-fix it serves as a recognizable marker for future prompt authors. The XML tag is structural (Anthropic-conventional role-demarcation) and distinct from the per-feature `{{feature_X}}` template tokens — two different shapes, two different intents.
- **No dependency changes**: The spec introduces no new Python packages, no new shell dependencies, and no new CI steps beyond the unit test file.
- **Test invocation**: The new `tests/test_fill_prompt.py` must integrate with pytest. The test MUST invoke the real `fill_prompt()` from `claude/overnight/runner.sh` via `subprocess.run` (see R7a) — extracting the substitution logic into a Python helper and testing that copy is explicitly prohibited to avoid silent divergence between test subject and production code.
