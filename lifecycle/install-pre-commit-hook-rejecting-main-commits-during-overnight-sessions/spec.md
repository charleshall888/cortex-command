# Specification: Install pre-commit hook rejecting main commits during runner-spawned children

## Problem Statement

Overnight runner agents can escape worktree isolation by `cd`-ing into the home repo and running `git commit`, landing artifact commits on local `main` instead of the integration branch. Session `overnight-2026-04-21-1708` did exactly this and caused permanent data loss (followup items 101/102/103 were silently overwritten by a later `/discovery` decompose). Worktree isolation at the Bash-tool layer is cosmetic — agents can always `cd` elsewhere — so the invariant must be enforced at the git layer. This spec installs a `pre-commit` policy that rejects commits to `main` from runner-spawned child processes (gated on `CORTEX_RUNNER_CHILD=1`), plus a runner-startup verification that refuses to launch overnight without the hook in place. As a sibling change, this spec also closes a load-bearing gap left by ticket 129's incomplete migration: the runner-side morning-report commit step was retired from `runner.sh` but never ported to `runner.py`, so `lifecycle/morning-report.md` is written to a tracked path but never auto-committed. This spec ports that step. Beneficiaries: cortex-command developers running overnight sessions, who lose work today and recover only via stub reconstruction; and the morning-review audit trail, which has been silently broken since the Python migration.

## Architectural Insight (load-bearing for the design)

The runner process itself (the `cortex overnight start` invocation) does NOT have `CORTEX_RUNNER_CHILD=1` set in its own `os.environ`. The variable is set only for child subprocesses spawned at `runner.py:714` (orchestrator) and `runner.py:901` (batch runner). When the runner directly invokes `subprocess.run(["git", "commit", ...])` (without an explicit `env=`), the subprocess inherits the runner's env, which contains no `CORTEX_RUNNER_CHILD`. Phase 0 therefore does not fire on runner-direct commits. This is what makes Phase 0 + the morning-report commit (Req 9) coexist cleanly: the legitimate commit is issued from above the gate; only orchestrator/batch-spawned commits are below the gate. `_commit_followup_in_worktree` already exhibits this pattern (it commits from the runner directly). No carve-out, second env var, or path-allowlist is needed.

## Requirements

> **MoSCoW classification**: All 9 requirements are **Must-have**. Rationale: this is a `criticality: critical` ticket addressing documented data loss and an incomplete sibling-ticket migration. The hook (Reqs 1-3) is the load-bearing escape enforcement, the runner-startup verification (Req 4) ensures the hook is active when a session launches, the logging change (Req 5) makes hook rejections debuggable, the requirements-doc update (Req 6) brings normative documentation into alignment with both this ticket's design and ticket 129's, the threat-model documentation (Req 7) bounds operator expectations honestly, the non-regression check (Req 8) protects the existing 5-phase pipeline, and the morning-report commit port (Req 9) closes a Critical-tier audit-trail gap left open by ticket 129's incomplete Python migration. No requirement is should-have or won't-do.

1. [Must] **Phase 0 in `.githooks/pre-commit`**: A new `# ----- Phase 0 — overnight main-branch guard -----` block is inserted at the top of `.githooks/pre-commit`, before the existing Phase 1. The block reads `${CORTEX_RUNNER_CHILD:-}` via strict equality (`[ "${CORTEX_RUNNER_CHILD:-}" = "1" ]`) and the current branch via `git symbolic-ref --quiet HEAD`. If `CORTEX_RUNNER_CHILD == "1"` AND the resolved branch is `refs/heads/main`, the hook exits non-zero with a `pre-commit:` stderr message that names ALL THREE: the rule (`Phase 0 — overnight main-branch guard`), the resolved branch (`refs/heads/main`), and the gate signal observed (`CORTEX_RUNNER_CHILD=1`), plus the `--no-verify` bypass note for emergencies. In all other cases (including detached HEAD, where `git symbolic-ref` exits non-zero), Phase 0 exits 0 and falls through to Phase 1. **Acceptance**: `bash tests/test_overnight_main_commit_block.sh` exits 0; pass if exit code = 0.

2. [Must] **Phase 0 contains the verification sentinel**: The Phase 0 block must contain the literal string `Phase 0 — overnight main-branch guard` (in a comment) so the runner-startup verification (Req 4) can detect it. **Acceptance**: `grep -c 'Phase 0 — overnight main-branch guard' .githooks/pre-commit` = 1.

3. [Must] **Hook fires from worktree-rooted commits — test with two scaffolds**: A regression test at `tests/test_overnight_main_commit_block.sh` constructs **two** ephemeral repos to cover both commit-source contexts without violating git's branch-checkout exclusivity:
   - **Scaffold A (worktree-on-main)**: temp repo on `--initial-branch=trunk`, `git branch main trunk` (creates main as a ref), configure `core.hooksPath`, copy `.githooks/pre-commit` in, `git worktree add wt main` (worktree checks out main, home repo stays on trunk). Tests run from inside the worktree.
   - **Scaffold B (`$REPO_ROOT`-on-main, the actual session-1708 vector)**: temp repo on `--initial-branch=main`, configure `core.hooksPath`, copy `.githooks/pre-commit` in, `git worktree add wt -b integration` (worktree on a separate branch). Tests run from inside the home repo.

   The four canonical assertions:
   - **(a) Scaffold A, branch=main (worktree), `CORTEX_RUNNER_CHILD=1`, commit issued from worktree** → exit non-zero AND stderr matches **all three** of `"Phase 0"`, `"refs/heads/main"`, `"CORTEX_RUNNER_CHILD"`.
   - **(b) Scaffold B, branch=main (home repo), `CORTEX_RUNNER_CHILD=1`, commit issued from `$REPO_ROOT`** → exit non-zero AND stderr matches all three substrings as in (a).
   - **(c) Scaffold A, branch=main (worktree), no `CORTEX_RUNNER_CHILD`** → exit 0 (interactive case allowed).
   - **(d) Scaffold A, branch=integration (worktree on a separate ref via `git worktree add wt -b integration`), `CORTEX_RUNNER_CHILD=1`, commit issued from worktree** → exit 0 (legitimate runner commit on integration branch allowed).

   Additionally, the test asserts the gitdir-sharing mechanism for Scaffold A: `git -C $WORKTREE rev-parse --git-common-dir` resolves to the home repo's `.git/`, AND `git config --get extensions.worktreeConfig` is empty/false in the test repo (verifies the indirection is the actual hook-resolution path, not a worktree-local copy). **Acceptance**: `bash tests/test_overnight_main_commit_block.sh` exits 0; pass if exit code = 0. The script must run under `just test-hooks` (i.e., be picked up by `tests/test_hooks.sh`).

4. [Must] **Runner-startup hook verification, gated by lifecycle.config.md field**: When invoked, `cortex overnight start` reads the target repo's `lifecycle.config.md` (resolved via `git rev-parse --show-toplevel`; falls back to `Path.cwd().resolve()` when not in a git tree). If the file contains a top-level YAML frontmatter or body field `overnight_hook_required: true`, the runner verifies all three: (i) `git config --get core.hooksPath` returns a non-empty value, (ii) the `pre-commit` file at that path is present and executable, (iii) the file contains the sentinel string `Phase 0 — overnight main-branch guard`. If any check fails, the runner refuses to launch (exit non-zero) with stderr: `cortex overnight: hook guard not installed but lifecycle.config.md requires it. Run 'just setup-githooks' to enable the pre-commit hook before launching overnight.` If `overnight_hook_required` is false, missing, or the file does not exist, the verification is skipped. cortex-command's `lifecycle.config.md` MUST be updated by this ticket to set `overnight_hook_required: true`. No env-var override is provided; failure mode is fail-closed when the field is true. **Acceptance**: `grep -c 'overnight_hook_required: true' lifecycle.config.md` = 1; `cortex overnight start --dry-run` (in cortex-command, with the field set true) and `core.hooksPath` unset exits non-zero and stderr contains "hook guard not installed"; with hook properly installed, the dry-run proceeds normally; running `cortex overnight start --dry-run` in a tmp repo without the field present skips verification (exits 0 on the verification stage). Specific test path: `tests/test_runner_hook_guard.py`.

5. [Must] **`_commit_followup_in_worktree` logs commit failures**: At `cortex_command/overnight/runner.py:435-446`, when the `git commit` subprocess returns non-zero (currently swallowed silently by `check=False`), the runner writes (a) a stderr line `runner: followup commit failed for session=<id> branch=<resolved-branch> rc=<returncode>` AND (b) a structured `events.log_event` entry of type `followup_commit_failed` with fields `session_id`, `worktree_path`, `branch`, `returncode`. The `git commit` subprocess invocation is amended to capture stderr (replace `subprocess.DEVNULL` with `subprocess.PIPE`) so the rejected-by-hook reason is included in the event payload. **Acceptance**: `grep -n 'followup_commit_failed' cortex_command/overnight/runner.py` returns at least 1 line; unit test at `tests/test_runner_followup_commit.py` extended to assert that simulated commit failure produces both the stderr line and the events.log entry.

6. [Must] **`requirements/pipeline.md:24` clause rewritten to reflect post-129 design**: The Session Orchestration AC line "The morning report commit is the only runner commit that stays on local `main` (needed before PR merge for morning review to read)" is replaced with text that accurately documents post-129 + this-ticket steady state: morning report writes to `lifecycle/sessions/{session_id}/morning-report.md` (gitignored, per-session archive) AND `lifecycle/morning-report.md` (tracked, latest copy); the latter is committed to local `main` from the runner process (NOT from a runner-spawned child), so Phase 0 does not block it. **Acceptance**: `grep -c 'morning report commit is the only runner commit that stays on local' requirements/pipeline.md` = 0; `grep -c 'lifecycle/morning-report.md' requirements/pipeline.md` ≥ 1; `grep -c 'runner process (NOT from a runner-spawned child)' requirements/pipeline.md` ≥ 1.

7. [Must] **Threat-model section in hook**: The `.githooks/pre-commit` header comment includes a paragraph documenting the threat model boundary: "Phase 0 catches accidental and tooling-bug escapes only. Adversarial agents can bypass via `git commit --no-verify`, `git -c core.hooksPath=/dev/null commit`, `GIT_DIR=...` overrides, plumbing-level commits (`git update-ref`, `commit-tree`), or self-modifying this file. The canonical adversarial defense is GitHub branch protection on `main` (require PR, disallow direct push)." **Acceptance**: `grep -c 'adversarial' .githooks/pre-commit` ≥ 1.

8. [Must] **All existing hook phases continue to pass**: Phase 0 is additive; Phases 1, 1.5, 1.6, 1.7, 2, 3, 4 must remain functional after the edit. **Acceptance**: `bash tests/test_drift_enforcement.sh` exits 0; `just test-hooks` exits 0.

9. [Must] **Port morning-report runner-side commit step from `runner.sh` to `runner.py`**: Add `_commit_morning_report_in_repo(project_root: Path, session_id: str, events_path: Path)` to `cortex_command/overnight/runner.py`. The function runs in the runner process (not in any spawned child), so the subprocess inherits the runner's `os.environ` which contains no `CORTEX_RUNNER_CHILD` — Phase 0 does not fire. The function:
   - Stages the per-session report (`git add lifecycle/sessions/<session_id>/morning-report.md`) — uses `git add -f` or honors gitignore, decision deferred to implementation but the per-session path is currently gitignored at `.gitignore:41`, so plain `git add` is a no-op there; document the decision in the function docstring.

## Non-Requirements

- **Does NOT defend against adversarial agents**: bypasses (`--no-verify`, per-command `core.hooksPath` overrides, `GIT_DIR=`, plumbing commits, self-modifying the hook) are out of scope. Server-side branch protection on `main` is the canonical adversarial defense and is not implemented in this ticket.
- **Does NOT install or enforce hooks in downstream user repos by default**. Downstream repos opt in by setting `overnight_hook_required: true` in their own `lifecycle.config.md` and installing a compatible hook. cortex-command opts itself in via Req 4. Helping downstream users install the hook is a separate concern.
- **Does NOT auto-mutate `core.hooksPath`** from `python-setup`, `cortex init`, or any setup recipe. The setup remains the existing manual `just setup-githooks` step. The runner-startup verification (Req 4) is the enforcement mechanism, not silent config mutation.
- **Does NOT provide an emergency env-var override** for the runner-startup verification. Fail-closed is the agreed posture.
- **Does NOT replace `LIFECYCLE_SESSION_ID` as a telemetry signal**. The variable continues to be used by `bin/cortex-log-invocation` (per `requirements/observability.md`); only the hook's gate predicate uses `CORTEX_RUNNER_CHILD` instead.
- **Does NOT rewrite `cortex-scan-lifecycle.sh`** to stop overwriting `LIFECYCLE_SESSION_ID`. The overwrite is preserved (it serves the SessionStart context-injection purpose); the hook simply does not depend on the variable.
- **Does NOT introduce a `pre-commit.d/` chain or any new abstraction** for hook composition. Phase 0 is added inline as a numbered phase block matching the existing 7-phase structure.
- **Does NOT add a path-allowlist or second env-var carve-out** to Phase 0. The architectural insight (runner-direct commits live above the gate) makes such carve-outs unnecessary; adding one would weaken the gate against accidental escapes that happen to touch allowlisted paths.

## Edge Cases

- **Detached HEAD**: `git symbolic-ref --quiet HEAD` exits non-zero. Phase 0 treats this as "not on `main`" and exits 0 (fail-open). Matches `pre-commit-hooks/no-commit-to-branch` convention. Verified during rebase, cherry-pick, and bisect states.
- **Initial commit on unborn branch**: `git symbolic-ref --quiet HEAD` returns `refs/heads/main` (or whatever the unborn branch is). Phase 0 fires correctly: an unborn-branch commit on `main` from a runner child is rejected.
- **`CORTEX_RUNNER_CHILD` set to anything other than `"1"`**: Strict equality check (`[ "${CORTEX_RUNNER_CHILD:-}" = "1" ]`). Empty, unset, or any other value (e.g., `"0"`, `"true"`, `" 1 "`) → no Phase 0 rejection.
- **Worktree on `main` due to misconfiguration**: A worktree accidentally checked out on `main` while a runner-spawned child is committing → Phase 0 rejects (correct behavior — the commit was about to land on local main from a runner-spawned process). The new logging in Req 5 surfaces the rejection to morning review.
- **Morning-report commit (Req 9)**: issued from the runner process directly, where `CORTEX_RUNNER_CHILD` is unset. Phase 0 sees no signal, allows the commit. Verified by Req 9's test at `tests/test_runner_morning_report_commit.py`.
- **`extensions.worktreeConfig=true` with per-worktree `core.hooksPath` override**: Out of scope (documented as adversarial bypass in Req 7). Phase 0 cannot detect this. The Req 3 test asserts `extensions.worktreeConfig` is unset in the test repo.
- **Hook file present but not executable**: Runner-startup verification checks the executable bit. If absent → refuse to launch with the standard message.
- **`cortex overnight start` invoked outside a git working tree**: `git rev-parse --show-toplevel` fails; falls back to `Path.cwd().resolve()`. If `lifecycle.config.md` is not present at that path, verification is skipped. The runner has other guards for non-git invocations.
- **`lifecycle.config.md` present but malformed YAML**: parser errors are caught; verification is skipped with a stderr warning `cortex overnight: lifecycle.config.md present but unparseable; hook verification skipped`.
- **`overnight_hook_required: true` but the field changes during a session**: the verification runs only at runner startup; mid-session config changes have no effect.
- **Mismatched hook (sentinel string absent)**: Runner-startup verification refuses to launch — protects against partially-updated checkouts where the hook file exists but predates Phase 0.
- **`bash` 3.2 compatibility**: `${CORTEX_RUNNER_CHILD:-}` is bash-3.2 safe. No `mapfile`, no associative arrays, no `[[ =~ ]]` regex matching required.
- **Tests running in CI**: CI environments that run the test suite without `core.hooksPath` configured must not be affected. Tests construct their own ephemeral repos (per `tests/test_runner_followup_commit.py:54-100` pattern); the runner-startup verification only fires for `cortex overnight start` invocations against repos that set `overnight_hook_required: true`.
- **Detached HEAD + `git update-ref refs/heads/main HEAD` adversarial sequence**: explicitly out of scope (covered by Req 7's threat-model paragraph naming `git update-ref` and `commit-tree`).

- **MODIFIED `cortex overnight start` startup path**: reads `lifecycle.config.md`'s `overnight_hook_required` field; performs hook verification when true; refuses to launch on missing hook.
- **MODIFIED `requirements/pipeline.md`**: rewrites the Session Orchestration AC to reflect post-129 + this-ticket steady state (morning report committed to local main from the runner process).
- **MODIFIED `lifecycle.config.md`**: adds `overnight_hook_required: true`.
- **MODIFIED `cortex_command/overnight/runner.py`** (post-round-loop): adds `_commit_morning_report_in_repo` and wires the call into `_run_post_round_loop` after `_generate_morning_report`.
- **ADDED `tests/test_overnight_main_commit_block.sh`**: four-case regression test for Phase 0 with two scaffolds and gitdir-sharing assertions.
- **ADDED test cases in `tests/test_runner_followup_commit.py`**: asserts logging on commit failure.

## Technical Constraints

- **Bash 3.2 compatibility** (macOS system bash): Phase 0 must avoid `mapfile`, associative arrays, and process substitution where a portable alternative exists. Literal `$VAR` checks and `git symbolic-ref` are sufficient.
- **`set -euo pipefail` inheritance**: Phase 0 inherits this from the existing hook header. Default values via `${VAR:-}` are required for any potentially-unset variable.
- **Single-file dispatcher convention**: The repo uses one `.githooks/pre-commit` for all policies. Phase 0 must coexist as a numbered phase, not a new file.
- **`pre-commit:` stderr prefix**: hook-level error messages follow this convention (existing at `.githooks/pre-commit:49,68`).
- **Sub-millisecond perf budget for Phase 0**: A single env-var read + `git symbolic-ref HEAD`. Cost is negligible relative to the existing hook's `just _list-build-output-plugins` phase (~30-80ms each).
- **`CORTEX_RUNNER_CHILD=1` is set unconditionally** at `runner.py:714` (orchestrator spawn) and `runner.py:901` (batch runner spawn). The gate signal is load-bearing on these two sites being unchanged. Req 4 verifies hook installation but does NOT verify the gate signal at the spawn sites — a future refactor that drops the env var passes Req 4's checks while silently disabling Phase 0. This is a known landmine; mitigation (an integration test exercising the actual spawn paths with a smoke commit) is out of scope here. Documented in Open Decisions.
- **Runner-direct commits live above Phase 0**: Req 9's `_commit_morning_report_in_repo` and the existing `_commit_followup_in_worktree` both run in the runner process where `CORTEX_RUNNER_CHILD` is unset. This is the load-bearing architectural property that makes Phase 0 + legitimate runner commits coexist without carve-outs.
- **`git symbolic-ref` is the canonical idiom** for branch detection; `--quiet` suppresses the "not a symbolic ref" message; non-zero exit on detached HEAD is caught and treated as not-on-main.
- **`requirements/pipeline.md` is normative**: the doc update in Req 6 reflects post-129 + this-ticket steady state.
- **`lifecycle.config.md` parsing**: Req 4's verification reads the field via the existing `cortex_command.common` parsing helpers; if the field is absent or unparseable, treat as opt-out (skip verification).

## Open Decisions

- **Spawn-site CORTEX_RUNNER_CHILD assertion**: known landmine. A future refactor that drops `env={**os.environ, "CORTEX_RUNNER_CHILD": "1"}` at `runner.py:714` or `runner.py:901` silently disables Phase 0 enforcement. Req 4 verifies hook installation but not gate-signal presence at spawn. A mitigation (e.g., a smoke test that asserts `subprocess.Popen([...], env=...)` calls in runner.py contain `CORTEX_RUNNER_CHILD` for both spawn sites, or an integration test that issues an actual commit through the spawn chain) is owed but deferred to a follow-up ticket because integration testing of the runner spawn path requires test infrastructure not present today. **Reason this is open and not resolved at spec time**: implementing the integration test requires writing a harness that exercises `runner.py`'s spawn paths end-to-end, which is itself non-trivial (likely 50+ LOC of test scaffolding) and decomposable into a separate ticket; the right call is a follow-up rather than an inline expansion of Req 8.
