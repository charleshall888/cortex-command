## Veto Surface

- **PR-A scope size**: Tasks 1-9 all land in a single PR against cortex-command. Alternative is splitting (e.g., vendor-only PR + classification + workflow PR), but spec Technical Constraints require justfile classification (Task 1) to land in or before the same commit that introduces plugin.json files (Tasks 2 and 5), and the vendor-without-marketplace-entry intermediate state is a half-finished sunset. Single PR is operationally simplest and matches the merge-train sequencing in spec lines 109-115. Re-litigate if the PR review burden is unacceptable.
- **Task 6 bilateral edit as one task vs two**: The devils-advocate description and critical-review description are different files but the spec mandates bilateral consistency (R2 Technical Constraint). One task makes the consistency invariant atomic; two tasks risks landing only one half. I chose one. Alternative is splitting if parallel implementer dispatch is preferred — but the build-plugin step couples them anyway.
- **Task 9 (workflow port) included in PR-A**: Could be deferred to a follow-up PR. I chose to include it because spec R11 explicitly requires the workflow exists post-vendor; deferring leaves cortex-command without CI for the vendored plugins between PR-A merge and a hypothetical follow-up PR. If implementation discovers workflow CI bring-up is unexpectedly complex, this is the natural cut point.
- **Task 9's call-graph step strategy**: Plan chose to keep the multi-positional-arg invocation and update the path list to all six current plugins (option a in research). Alternatives: (b) modify `validate-callgraph.py` to accept a glob — out of scope, expands ticket; (c) shell expansion in `run:` block — fragile across runners. Re-litigate only if the explicit-list maintenance burden is judged worse than (b).
- **Task 13 tier escalation to `complex`**: Originally `simple`; escalated because the parity-verification artifact requires structured fenced-block sections, captured exit codes, real interactive `/plugin install` transcripts, and gh-CLI calls — beyond the 5-15 minute one-file-edit envelope. `complex` is more honest about the implementer cost.
- **Skill invocation choices in Task 13 (R12)**: Spec R12 says "manually invoke one skill from each (e.g., `/r8-analyzer` and `/devils-advocate`)". `/r8-analyzer` is the recommended choice for android-dev-extras since it has the most reference structure and exercises the deepest path. If `/r8-analyzer` is awkward to invoke without an actual Android project, fall back to `/edge-to-edge` or `/android-cli`; the verification AC accepts any of the three.
- **Cross-repo Merge Gates G1/G2 as in-AC checks vs. external workflow gating**: Plan encodes G1/G2 as preconditions inside the verification commands of Tasks 10, 12, 13, 14 (the orchestrator checks them via `git fetch origin main` + `git show`). Alternative was an external orchestrator state machine that pauses dispatch until merge events are observed. Inline-AC approach is simpler, requires no orchestrator changes, and degrades cleanly (a manually-running implementer reads the verification block, runs the gate check, dispatches the task only if it passes). Trade-off: the implementer (or overnight orchestrator) MUST honor the precondition order — if they run AC commands from Task 14 before G1/G2 hold, the AC fails and the task remains pending; the gate is fail-closed but not fail-fast.
- **Task 15 rollback as documentation vs. automation**: Plan's Task 15 is a documented playbook with conditional execution, not a runnable automation. The spec's rollback procedures are inherently judgment-laden (e.g., "review the diff first to preserve unrelated edits in chickfila-android settings.local.json"). Hard-automating them would require deeper coordination logic than the rest of the plan. Re-litigate only if rollback frequency justifies it.
- **Vendoring style (research Alt 5b: plain copy without history)**: Already decided in spec; called out here because if the user changes their mind, Tasks 2 and 5 would need to be replaced with `git subtree add` invocations and a fundamentally different PR shape. Sibling archive preserves provenance retrievably, so this trade-off is settled per spec.
