# Specification: Vertical-planning adoption as REPLACEMENT (#182)

## Problem Statement

Cortex's lifecycle artifact templates carry three sections in `skills/lifecycle/references/plan.md` that the epic-172 audit flagged as low-utility: `## Scope Boundaries` (zero programmatic consumer; 53% populated; often duplicates spec.md's `## Non-Requirements`), `## Verification Strategy` (compressible to ≤5 lines), and `## Veto Surface` (most-cited section in retros — 13 mentions — but vocabulary-collided with the new "phase" language). The audit's DR-1 recommends REPLACEMENT, not addition: a single `## Outline` section absorbs the responsibilities of the first two; `## Veto Surface` renames to `## Risks` to align with new vocabulary; a tier-conditional `## Acceptance` section preserves the whole-feature acceptance contract for complex-tier plans where the per-phase Checkpoint chain isn't semantically sufficient. The spec.md template gains a matching `## Phases` section + per-requirement Phase tags to make vertical-slice boundaries visible at the spec layer. Three new orchestrator-review gates (P9 for plan outline, S7 for spec phases, P10 for complex-tier `## Acceptance`) enforce the structure. Bundled in this ticket per the #178 R5 amendment trail: `cortex_command/pipeline/metrics.py:221` gains named-rename tolerance for the three documented verdict-JSON field-name variants (FM-7 in-flight rename tolerance — NOT an unbounded-drift defense; the structurally-strong defense is write-time schema validation at `cortex_command/overnight/events.py`, which is out of scope for this ticket and recorded as future follow-up); a plan-parser regression test locks in current parsing behavior; and `cortex_command/overnight/report.py` gains the tier-conditional verification-source reader the morning report needs once `## Verification Strategy` is gone. Lands AFTER cross-skill collapse (174–176, verified complete).

> **Terminology note**: throughout this spec, "tier" refers to the persisted complexity value (`tier: simple|complex`) written to `lifecycle/{feature}/events.log` per `skills/lifecycle/SKILL.md:250`. This is the same value the Clarify phase produces as "complexity" — the events.log field name is `tier`, the user-facing assessment is "complexity," and the two are synonymous. The spec uses "tier" where it refers to the persistence layer (R13's reader; gate execute-time logic) and "complexity" where it refers to the user-facing assessment dimension. Critically, R13(a)'s reader must extract `event["tier"]`, NOT `event["complexity"]` — the latter field does not exist in events.log and would return `None` on every well-formed event.

## Non-Requirements

- The optional `cortex plan-outline` topological-sort renderer (Mermaid or text-based) is NOT in scope — deferred per ticket DR-3.
- Mermaid diagrams inside `## Outline` are NOT mandated — allowed where they add value per ticket DR-3, but the template instructs text-outline as the required shape and Mermaid as optional.
- Support for `## Phase N:` H2 headings nested INSIDE `## Tasks` is NOT in scope — the H3 convention (R1) structurally avoids the parser-truncation failure mode; the "hard-break behavior on nested H2" regression test from the ticket is intentionally omitted because the H3 decision precludes the failure path.
- Propagation of the provenance note into each authored `lifecycle/{feature}/plan.md` is NOT in scope — the note lives in the canonical template prose only (per AP-1).
- Shipping a specific "X-line per-feature reduction" claim in the canonical plan template or in the morning report is NOT in scope — the audit's 40–90-line estimate is research-internal context; measurement happens during implementation, not as a shipped claim (per FM-8).
- Creating a separate `research/epic-172-audit/critical-review-residue.json` artifact for the Q-C critical-review is NOT in scope — Q-C citations rewrite to point to `research/epic-172-audit/research.md:266,274,277,278` directly (per FM-9).
- Soft-deletion / deprecation markers for `## Scope Boundaries` and `## Verification Strategy` are NOT in scope — hard deletion per `requirements/project.md` Workflow trimming philosophy.
- Schema-validation for `review_verdict` events at write-time (`cortex_command/overnight/events.py`) is NOT in scope — `metrics.py:221`'s alias-lookup is defense-in-depth only; the upstream strict-schema work is a separate concern.
- Migration of in-flight `lifecycle/{feature}/plan.md` files authored under the old template is NOT in scope — the `report.py` compat shim (R13e) handles them transparently.
- Refactoring or generalization of the alias-lookup pattern across other `metrics.py` field accesses is NOT in scope — the `_VERDICT_FIELD_ALIASES` constant is closed; new fields require explicit additions (per FM-6 scoping).

## Edge Cases

- **In-flight plans authored under the OLD template** (still using `## Verification Strategy`, no `## Outline`/`## Acceptance`): the `report.py` fallback chain (R13d) falls through to `_read_verification_strategy()`. The morning report continues to render verification text for these plans unchanged. No in-flight plan.md migration required.
- **Existing archived plans referencing `Veto Surface` by name** (~108 files in `lifecycle/`): unchanged. The R2 retro-anchor prose note ("Renamed from `## Veto Surface` 2026-05-11") sits in the canonical template at `skills/lifecycle/references/plan.md §3` and is a reviewer-facing anchor — NOT a search-path-located breadcrumb. Operators retro-searching for "Veto Surface" grep `lifecycle/` and land on the archived artifacts directly (the legacy name lives in the corpus itself). The anchor's value is editorial clarity for someone reading the new template who wonders "what happened to Veto Surface?", not search-path preservation. R2's grep acceptance is honest about this — a fact-of-presence test, not a consumer test.
- **Refining agents going off-template and writing `## Phase N:` H2 inside `## Tasks`**: structurally avoided by the H3 convention (R1) — agents copying the canonical template see `### Phase N:` and follow suit. If an agent deviates, the parser regression test (R12) does NOT catch it (it tests the in-template H3 case); orchestrator-review's P9 only checks Outline shape, not Tasks body shape. Document as a known agent-discipline boundary; not auto-detected.
- **Complex-tier plan ships without `## Acceptance`** (P10 pending fix-dispatch): the morning report's R13d fallback chain falls through from `_read_acceptance(feature)` (empty) to `_read_last_phase_checkpoint(feature)`, rendering the last-phase Checkpoint from `## Outline`. This is the richer-than-generic fallback for the most common during-fix-dispatch state. If the plan ALSO has no `## Outline` (true degenerate state), the chain falls through to `_read_verification_strategy(feature)` (absent under R5), and only then to the generic "See feature plan for verification steps" message.
- **HYBRID plan with BOTH `## Outline` AND `## Verification Strategy`** (during-migration shape when refining agents append `## Outline` to legacy plans without removing old sections): R13e's HYBRID-plan policy silently prefers the new-shape reader. This is intentional — the new shape is the going-forward source of truth. Legacy `## Verification Strategy` content in HYBRID plans is invisible to the morning report. Operators wanting the legacy section's content rendered must complete the migration (delete `## Verification Strategy`).
- **Simple-tier plan with a single-phase `## Outline`**: P9 passes (R1 + R8 tolerance). `_read_last_phase_checkpoint(feature)` returns the first-and-only phase's Checkpoint. The morning report renders that Checkpoint as the verification text.
- **Low-criticality + simple-tier plan with no `## Outline`** (P9 skipped by orchestrator-review's `low+simple` skip rule; refining agent forgets the Outline): `_read_last_phase_checkpoint(feature)` returns empty → falls through to `_read_verification_strategy(feature)` (absent under R5) → morning report shows generic fallback "See feature plan for verification steps." Acceptable for the `low+simple` risk profile — this is the cell the skip rule intentionally exempts from quality enforcement.
- **Critical-criticality + simple-tier plan** (severe-risk small-surface change, e.g., a one-line fix to a security predicate): the skip rule does NOT skip; P9 fires; P10 does NOT fire (P10's inner condition is `tier=complex`); the morning report renders last-phase Checkpoint via R13d. This is a known gating-policy limitation — the whole-feature acceptance contract is NOT enforced at this cell. Escape hatch: operators can escalate via `complexity_override` event from `simple` → `complex`, which triggers P10 enforcement and adds `## Acceptance` to the plan via fix-dispatch. The Clarify §4 user resolution chose `tier=complex` (NOT criticality-based) as the P10 gating dimension; the criticality+simple gap is the accepted trade-off for keeping the gating model single-axis.
- **Plugin-mirror drift**: caught by R14 (parity test) at test-suite invocation AND by the pre-commit hook at commit time. Defense-in-depth — if a developer skips `just setup-githooks`, R14 still catches the drift in CI.
- **Dispatch.py / test_dispatch.py docstrings outside the rename scope**: R2 specifies the 2 known live consumers (`dispatch.py:271`, `test_dispatch.py:943`). If a future `grep` after the commit surfaces additional `Veto Surface` strings in active code (not in `lifecycle/` archived artifacts), those become a follow-up correctness fix, not a regression of this ticket.
- **`metrics.py:221` event with no recognized alias field**: `_extract_verdict(event)` returns `None`; the existing `[v for v in [...] if v is not None]` filter or equivalent guard (whichever the implementation uses) skips the event safely. No silent count corruption.
- **`metrics.py:221` event with a NEW alias not in `_VERDICT_FIELD_ALIASES`** (unbounded future drift): the closed-enum lookup returns `None`; the event is silently dropped from the metrics aggregation. R11's named-rename tolerance does NOT defend against this — it covers ONLY the three pre-declared aliases. The unbounded-drift defense lives at the upstream schema-validation surface (`cortex_command/overnight/events.py`), which is recorded in Non-Requirements as out-of-scope for this ticket and listed as a future follow-up. Operators detecting a `review_verdicts` count drop should suspect a new alias and amend `_VERDICT_FIELD_ALIASES` with an accompanying unit test (per Technical Constraints).
- **Multiple alias fields present in one event** (e.g., both `"verdict"` and `"review_verdict"`): the tuple-ordering of `_VERDICT_FIELD_ALIASES` resolves precedence — canonical `"verdict"` wins. This deterministic precedence is part of the contract and is captured in the R11 unit test.
