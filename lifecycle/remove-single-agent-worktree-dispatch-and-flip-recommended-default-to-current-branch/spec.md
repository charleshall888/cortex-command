# Specification: Remove single-agent worktree dispatch and flip recommended default to current branch

> Epic reference: `research/implement-in-autonomous-worktree-overnight-component-reuse/` (DR-2 reversed). Ticket-specific research: `lifecycle/remove-single-agent-worktree-dispatch-and-flip-recommended-default-to-current-branch/research.md`.

## Problem Statement

The lifecycle implement-phase pre-flight at `skills/lifecycle/references/implement.md` §1 offers four branch-strategy options; option 1 ("Implement in worktree", via single-agent `Agent(isolation: "worktree")` dispatch) has thin usage evidence (one observed successful run in event logs) and its scaffolding incurs ongoing maintenance cost — §1a's verbatim prompt (~77 lines), `.dispatching` marker semantics, TC8 events.log divergence, SKILL.md Step 2 "Dispatching Marker Check" and "Worktree-Aware Phase Detection" blocks, worktree-agent skip conditions in Register Session and Backlog Write-Back, a worktree-agent context guard in implement.md §1, and a worktree-prune block in `hooks/cortex-cleanup-session.sh`. Additionally, `anthropics/claude-code` issue #39886 suggests `Agent(isolation: "worktree")` can silently fail to create isolation, meaning the observed success may have run without isolation anyway. This ticket removes option 1 and all associated scaffolding in a single atomic PR, promotes "Implement on current branch" (renamed from "Implement on main") to the recommended default behind #096's uncommitted-changes guard, and records the reversal of epic #074's DR-2 in its canonical decomposition record. It also backfills one historical feature's events.log to preserve correct phase-detection after the Worktree-Aware Phase Detection override is deleted.

All R1–R19 below are **must-have** for the feature to be considered complete. There are no should-haves; scope boundaries are enforced via Non-Requirements.

## Edge Cases

- **In-flight session during rollout (PR-level atomicity)**: the rollout is a single PR. Within the PR, there is no intermediate state where routing is gone but §1a body remains; there is no window in which SKILL.md's Worktree-Aware Phase Detection could still fire against stale `dispatch_complete` events while the routing path has already been removed.
- **Mid-session `git pull` (user-filesystem atomicity)**: this repo symlinks `skills/*`, `hooks/*`, and `claude/*` into `~/.claude/*`. A `git pull` updates all 17+ files simultaneously at the filesystem level — the user-visible SKILL.md / implement.md / cleanup-session.sh switch together. Skills re-read their files on each invocation, so the next lifecycle-related skill invocation after the pull sees the fully-updated state. Historical events.log entries with `mode: "worktree"` remain on disk but R19 backfills the one known case that would otherwise misclassify.
- **Prior `worktree/agent-*` branches exist at merge time**: `git worktree list` and `git branch --list 'worktree/agent-*'` both return empty on this branch. If a user manually created a `worktree/agent-*` branch between spec approval and merge, the post-merge cleanup-hook deletion (R13) means that branch will not be cleaned by session end; the user must manually `git branch -D` it. This is acceptable because no live path creates such branches.
- **Orphaned `.dispatching` files on disk at merge time**: if a crashed §1a session left a `lifecycle/{feature}/.dispatching` marker, post-R9 no code reads it, so it becomes harmless orphaned state. Users can optionally delete via `rm lifecycle/*/.dispatching`; no ticket-owned sweep.
- **Hook syntax after R13 deletion**: `bash -n hooks/cortex-cleanup-session.sh` must still exit 0. The deleted block is a standalone subshell `( … ) || true`; removing it cleanly leaves the preceding `.session` cleanup loop and the trailing `exit 0` intact.
- **Rename string-match regression**: if R2's rename updates the option text but misses the routing-match substring (or vice-versa), selection dispatch falls through. The lock-step coupling is required by R2 and R6 together; R6's routing-block acceptance criterion catches the coupling.
- **Downstream tickets (#110, #123) not yet refined**: R15 and R16 append dated blockquote annotations inside the `## Out of scope` section of each ticket. Both tickets are `status: backlog`; their future /refine cycle reads the ticket body and will encounter the annotations. If either ticket enters /refine concurrently with #097's merge (race condition), the refined spec may not reflect the annotations; this is acceptable because #097 is not blocking either ticket — re-running /refine on the affected ticket after noting the annotation resolves the drift.
- **Issue #39886 context in annotation**: R14's two-sentence framing names issue #39886 as a reframing lens — the "one successful §1a run" may have been a silent non-isolation. This does not change R4 (deletion); it only changes the drift-note narrative so future readers do not misread the removal as "working feature deleted."
- **Historical feature misclassification after R10**: without R19's backfill, `lifecycle/devils-advocate-smart-feedback-application/` would be misclassified as `phase: implement, 0/4` by `claude/common.py::detect_lifecycle_phase()` (and its dashboard/statusline/hook consumers) because its terminal event is `dispatch_complete (mode: "worktree")` without a subsequent `feature_complete`. R19 appends the missing event. If a grep-scan before merge reveals additional historical cases beyond this one, they must also be backfilled before merge (see R19 scope note).

## Technical Constraints

- **Single-PR atomic rollout (PR-level)**: all 19 requirements land in one PR. Two-PR staging (routing change → destructive cleanup) is explicitly rejected because the intermediate state — routing gone, §1a body still present, SKILL.md Worktree-Aware Phase Detection still live — could trigger a `continue-in-worktree` / `dispatch-fresh` / `exit` AskUserQuestion prompt for a feature whose dispatch path no longer exists. Single-PR avoids that reachable "broken but present" state.
- **Single-PR atomicity is PR-level, not user-filesystem-level**: because this repo symlinks `skills/* → ~/.claude/skills/*` etc., merging to `main` does not update a user's live system. The user's next `git pull` updates all files at once (filesystem-level atomicity via symlinks). Between merge and pull, the user's system runs the pre-PR code, which is by definition consistent. Mid-session pull is addressed under Edge Cases (skills re-read per invocation; events.log entries covered by R19).
- **Rollback asymmetry**: a `git revert` of the single PR restores all deleted code but also reverts R14's `decomposed.md` annotation and R15/R16's scope-amendment notes on #110/#123. The code/narrative alignment post-revert would require manual re-verification. This asymmetry is acceptable for a destructive + narrative change; revert workflow should hand-check the annotations are consistent with the intended state.
- **Option-label rename coupling**: the AskUserQuestion option text at (former) implement.md:15 and the routing-match substring at (former) implement.md:25 must update together. R2 and R6's acceptance criteria both verify post-rename state so either-sided miss fails.
- **Drift-note location**: `decomposed.md` is the canonical home for DR-2 (the bullet physically lives there, lines 65–67). The sibling `research.md` is referenced by 7+ downstream tickets (074, 075, 076, 077, 078, 079, 080) whose modularization decomposition is unaffected by this reversal; adding a top-level "Superseded-by" banner to `research.md` would incorrectly imply the whole epic is reversed. The R14 annotation goes only in `decomposed.md`.
- **Amendment-note shape pinned across R14/R15/R16**: all three annotations use the same dated-blockquote shape (`> **2026-04-22 (ticket #097) — …`) for consistency and to make them greppable / identifiable in future tooling. This is newly-pinned by this ticket; future amendment-note authors should follow the same shape.
- **Repo precedent over external convention**: this repo does not use formal ADR files; design decisions live as inline bullets in research artifacts. Web-research surfaced the "new ADR + Superseded-by" canonical pattern; it is explicitly not adopted here.
- **R19 backfill is single-file in current scope**: one grep over `lifecycle/*/events.log` before merge (`grep -rln 'dispatch_complete.*"mode": *"worktree"' lifecycle/*/events.log`) confirms the only case requiring backfill is `devils-advocate-smart-feedback-application`. If additional cases surface between spec approval and merge, R19's scope expands to include them.
- **Commit-artifacts convention**: on spec approval, the spec+research artifacts are committed via `/commit` per project convention (specify.md §5).
- **Test-suite change scope**: R18 is a single regex + comment update — no new tests, no existing test semantics change. The test's purpose (verifying §1a/§1b structural anchor parses) is preserved; only the anchor string changes in lock-step with R5.
