# Plan: resolve-cortex-commandbacklog-packaged-dispatch-dead-branch-in-bin-cortex-wrappers

## Overview

Package the four `backlog/*.py` modules with bash wrappers (`update_item`, `create_item`, `generate_index`, `build_epic_map`) into a real `cortex_command.backlog` subpackage, replace the bash wrappers with `[project.scripts]` entry points, and relocate `install_guard.check_in_flight_install()` from `cortex_command/__init__.py` (fires on every package import) into `cortex_command.cli._dispatch_upgrade()` (the only true install-mutation path). All 22 spec requirements land in a single commit per Requirement 21; ordering inside the commit is structured here as 13 tasks plus a final integration task that performs the manual procedural steps (`uv tool install -e . --reinstall` then `just build-plugin` then atomic `git add`) before invoking `/cortex-interactive:commit`.

### Task 9: Add `cortex_command/backlog/_telemetry.py` and wire `log_invocation()` into all four `main()` functions
- **Files**: `cortex_command/backlog/_telemetry.py` (new), `cortex_command/backlog/update_item.py`, `cortex_command/backlog/create_item.py`, `cortex_command/backlog/generate_index.py`, `cortex_command/backlog/build_epic_map.py`
- **What**: Create a `log_invocation(script_name: str) -> None` helper that emulates `bin/cortex-log-invocation`'s fail-open contract: resolve `LIFECYCLE_SESSION_ID` from env; on any of the five fail-open conditions enumerated in the bash shim (no `LIFECYCLE_SESSION_ID`, no `CORTEX_COMMAND_ROOT`/repo root, session dir missing, write-denied, other), write a breadcrumb with the matching category string to `~/.cache/cortex/log-invocation-errors.log` and return silently; otherwise write one JSONL record to `lifecycle/sessions/<id>/bin-invocations.jsonl`. Each of the four `main()` functions adds `log_invocation("cortex-<command-name>")` as its very first statement before any argv parsing or other work. Wrap the entire helper body in a single `try/except Exception` that swallows all exceptions (matching the bash shim's `trap 'exit 0' EXIT`).
- **Depends on**: [1]
- **Complexity**: complex
- **Context**: Reference shim is `bin/cortex-log-invocation:60-61`, which uses `printf` to emit a single line: `{"ts":"%s","script":"%s","argv_count":%d,"session_id":"%s"}\n` followed by a newline. The actual record schema is **four** fields in **this order**: `ts` (ISO 8601 string), `script` (str — the user-visible command name like `cortex-update-item`), `argv_count` (int — count of argv entries, NOT the argv list itself), `session_id` (str). There is no `pid`, no `cwd`, and no `argv` field. To match byte-for-byte: use `json.dumps(record, separators=(",", ":"), ensure_ascii=False)` plus an explicit Python ordered dict (or list of `(key, value)` pairs serialized via `json.dumps`) to preserve insertion order; append `"\n"` for the trailing newline. The aggregator `bin/cortex-invocation-report:62` does literal-substring matching on `"\"script\":\"" + name + "\""`, which only requires the `"script":"<name>"` substring to appear unbroken — but the byte-equivalence claim is broader: every field name, value-encoding choice, and separator must match. The breadcrumb-category enumeration must mirror the bash shim's five categories: `no_session_id`, `no_repo_root`, `session_dir_missing`, `write_denied`, `other`. The `script_name` argument is the user-visible command name (`cortex-update-item`, etc.), NOT the Python module name. Spec Non-Requirement #7 explicitly forbids physically unifying the two implementations under one shared serializer; the plan's veto-surface acknowledges the contradiction with R12's "shared serializer" wording — implementer must hand-roll byte-matching Python output rather than refactor the bash shim.
- **Verification**: `python3 -c "from cortex_command.backlog import _telemetry; assert callable(_telemetry.log_invocation)"` — pass if exit 0; AND `for f in cortex_command/backlog/{update_item,create_item,generate_index,build_epic_map}.py; do head -100 "$f" | grep -q "_telemetry.log_invocation" || { echo "missing in $f"; exit 1; }; done` — pass if exit 0; AND **byte-equivalence check** (defined in detail in Task 12's `test_telemetry_byte_equivalence.py`): `pytest cortex_command/backlog/tests/test_telemetry_byte_equivalence.py -v` — pass if exit 0. ## Veto Surface

The user may want to revisit before implementation begins:

- **Telemetry helper duplication**: The plan keeps `cortex_command/backlog/_telemetry.py` separate from `bin/cortex-log-invocation` (per spec Non-Requirement #7). Byte-equivalence is asserted but the two implementations remain physically separate. Spec R12's "Use a shared serializer" wording contradicts Non-Requirement #7's "remain physically separate" — the plan resolves the contradiction in favor of physical separation (per Non-Requirement #7) and adds a real byte-equivalence test in Task 12 to compensate. If the user prefers a unified helper now (e.g., a `cortex_command/_telemetry.py` invoked from both bash and Python paths), this expands scope but eliminates the contradiction at its source.
- **Parity-linter shadow detection (deferred)**: After T8's set-union, a future maintainer could add a `bin/cortex-X` script and a `[project.scripts]` `cortex-X = ...` entry pointing at the same name; set-union deduplicates, so neither E001/E002/W003 nor W005 fires. This plan does NOT add a new diagnostic (e.g., E004 source-ambiguity) for the overlap case. If the user wants the new diagnostic now, T8's scope expands; otherwise the gap is a known-but-deferred coverage hole.
- **Documentation home for the audit classification (Task 7)**: The spec offers two locations (`requirements/observability.md` under a new subsection, OR a module-level comment block in `cortex_command/install_guard.py`). This plan picks `requirements/observability.md` as the load-bearing doc. If the user prefers the install_guard module docstring as the home (keeps install-guard semantics co-located), Task 7's Files swap to `cortex_command/install_guard.py`.
- **Pre-commit gate placement (Task 10)**: Adding `--check-entry-points` as a sibling to `--check-shims` keeps the two modes parallel; alternatively, `--check-shims` could be extended to also walk entry-point modules, collapsing them into one mode. Plan picks the sibling approach for clarity; user may prefer the unified mode.
- **Subprocess call site in `update_item.py` (Task 3)**: The `_resolve_generate_index` helper becomes vestigial after the `python -m` swap (its only caller is the call site being changed). This plan deletes the helper. If the user wants to keep it as a stable public API for any external caller, retain it but have it return a `["python", "-m", ...]` list instead of a `Path`. No external callers were found in the audit, so deletion is the cleaner default.
