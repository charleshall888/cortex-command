# Specification: Package cortex_command.backlog and replace bash wrappers with entry points

## Problem Statement

Four `bin/cortex-*` wrappers (`cortex-update-item`, `cortex-create-backlog-item`, `cortex-generate-backlog-index`, `cortex-build-epic-map`) carry a dead `Branch (a): packaged form` probe for `cortex_command.backlog.<module>` that has never been live — `cortex_command.backlog` is not declared as a package. Three of the four `backlog/*.py` scripts already `from cortex_command.common import ...`, so they are *de facto* package members shelved in the wrong directory. This ticket closes out the deferred packaging work captured in `lifecycle/archive/publish-cortex-interactive-plugin-non-runner-skills-hooks-bin-utilities/spec.md:90`, replaces the bash wrappers with `[project.scripts]` entry points, and relocates `install_guard.check_in_flight_install()` out of `cortex_command/__init__.py` (which fires on every package import) into the actual install-mutation path (`cortex upgrade` → `_dispatch_upgrade`). The relocation also fixes a latent bug: today `cortex overnight status` during an active session fires the guard and aborts because no carve-out covers read-only inspection — narrowing placement to `_dispatch_upgrade` eliminates that false positive. Beneficiaries: future submodules added to `cortex_command/`; users invoking backlog tooling or read-only inspection commands during active overnight sessions.

## Implementation Procedure (Required Manual Sequence)

The 22 requirements above land in **one git commit**, but the implementer must follow this manual sequence to avoid pre-commit drift failures:

1. Move files, edit imports, edit `pyproject.toml`, etc. — make all source-tree edits per Requirements 1-18.
2. Run `uv tool install -e . --reinstall` to regenerate the four new entry-point binaries (so Requirement 5's manual `which`-based check works).
3. Run `just build-plugin` to delete plugin mirrors via rsync `--delete`.
4. Run `git add bin/ plugins/cortex-interactive/bin/ pyproject.toml cortex_command/ tests/ skills/ requirements/ <other paths>` — stage everything in one shot.
5. Run `just test` — must pass.
6. Run `git commit` (or `/cortex-interactive:commit`) — pre-commit hooks must pass.

Skipping step 3 (`just build-plugin` before staging) causes Phase 4 dual-source drift to block the commit. Skipping step 2 causes Requirement 5's `which` check to fail. Skipping step 4's "stage everything together" produces intermediate states where Phase 1.5 fires E002.

- **Not publishing `cortex-command` to PyPI or any other package registry**. Per `requirements/project.md:54`, the project remains editable-install-only.
- **Not providing migration tooling for users with active backlog ticket files**. Backlog YAML files are user data and are not affected.
- **Not maintaining a backwards-compatible `from backlog.X import Y` shim**. Production callers (5 sites) are updated atomically in the same commit.
- **Not adding new install_guard carve-outs**. The relocation strategy eliminates the need for the carve-outs being removed (pytest, dashboard, runner-child).
- **Not extending `bin/.parity-exceptions.md` schema to support entry-point allowlisting**. W005 is scoped to `bin/`-discovered commands only in this ticket. - **Not unifying `cortex-log-invocation` and `cortex_command/backlog/_telemetry.py`** under one shared serializer in this ticket. Byte-equivalence is asserted by Requirement 12's acceptance criteria but the two implementations remain physically separate. A future ticket may consolidate.

## Edge Cases

- **Stale plugin-cache install on user's machine**: A user with an older `~/.claude/plugins/cache/cortex-command/cortex-interactive/<sha>/bin/cortex-update-item` (the bash wrapper from before this ticket) until they run `/plugin install` to refresh. The stale wrapper tries Branch (a) → succeeds (cortex_command.backlog is now importable post-CLI-install) → executes `python -m cortex_command.backlog.update_item`. Behavior matches the new entry point. After `/plugin install`, the regenerated plugin no longer ships these four wrappers.

- **User without `uv tool install -e .`**: A plugin-only install (no CLI install). The stale plugin-cache wrapper tries Branch (a) (fails — no `cortex_command` package importable), Branch (b) (requires `CORTEX_COMMAND_ROOT`; fails if unset), Branch (c) (clear error). Pre-existing constraint of the editable-install-only model per `requirements/project.md:54`.

- **Active overnight + user-driven backlog edit**: Today this works silently via `python3 backlog/update_item.py` (Branch b). Post-this-ticket, it works via the new entry point — the guard no longer fires from `import cortex_command`. Verified by Requirement 16.

- **Active overnight + read-only inspection (`cortex overnight status`)**: Today this **fails** (`InstallInFlightError`) because no carve-out covers it — a latent bug. Post-this-ticket, the guard moves to `_dispatch_upgrade` only, so `cortex overnight status` no longer fires it. Verified by Requirement 17's second test.

- **Active overnight + `cortex upgrade`**: Today fires the guard (correct). Post-this-ticket, still fires the guard (correct) — same user-visible behavior. Verified by Requirement 17's first test.

- **A future submodule (`cortex_command/foo/`) added without an explicit guard call**: Acceptable — by design, the guard is opt-in for install-mutation paths only. A new submodule that doesn't mutate the install on disk does not need to fire the guard.

- **Stale-pointer self-heal warning narrowed**: Today the warning emits on every `import cortex_command`; post-this-ticket it emits only on `cortex upgrade`. A user invoking morning-review tooling or `python -m cortex_command.overnight.report` after a runner crash will no longer see the recovery hint. Mitigation: the audit document in Requirement 9 should note this narrowing; future telemetry/UX work may re-surface the warning at appropriate boundaries (out of scope for this ticket).

- **`_telemetry.py` write failure**: The helper's fail-open guarantees the entry-point command still succeeds even if the telemetry write fails (file permission denied, disk full, etc.). Acceptance via Requirement 12(b)(c).

- **Parity linter (E002 drift)** triggers when "in-scope file mentions cortex-X but bin/cortex-X doesn't exist."
