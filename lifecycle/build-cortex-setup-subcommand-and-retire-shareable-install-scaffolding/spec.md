# Specification: Retire shareable-install scaffolding (no `cortex setup` subcommand)

> **Epic reference**: This ticket implements DR-5 of [the overnight-layer-distribution epic](../../research/overnight-layer-distribution/research.md). It is the **first** merge in the plugin-migration chain — tickets 120 (`cortex-interactive`), 121 (`cortex-overnight-integration`), and 122 (marketplace manifest) are all `blocked-by: 117` and ship after this.
>
> **Scope inversion note**: The backlog item's title reads "Build cortex setup subcommand and retire shareable-install scaffolding." Through spec refinement, the scope collapsed: a `cortex setup` subcommand is not needed. Cortex-command becomes pure CLI-plus-plugins — no global-rules injection, no reference-doc symlinks, no settings.json ownership, no post-install command to run. This ticket ships pure retirement; there is no replacement subcommand. The title should be read as "retire shareable-install scaffolding (cortex setup was the proposed replacement; it turned out not to be needed)."

## Problem Statement

Today `just setup` + the project-local `/setup-merge` skill deploy ~40 host touchpoints (skills, hooks, rules, reference docs, notify.sh, statusline.sh, settings.json, `~/.local/bin/*`) into the user's `~/.claude/` and `~/.local/bin/`. Tickets 120/121/122 ship the skills + hooks + `bin/` utilities as Claude Code plugins. Machine-config absorbs `~/.claude/settings.json` and `~/.claude/notify.sh` as personal machine setup. The audit of rules + reference content (session 2026-04-23) found all of it either (a) redundant meta-documentation best kept in `docs/`, (b) skill-scoped content that should live in its skill's SKILL.md, (c) plugin-scoped reference material that ships at `${CLAUDE_PLUGIN_ROOT}/references/` when plugins land in 120/121. Nothing in cortex-command needs globally-injected rules or symlinks to `~/.claude/` once plugins exist. This ticket retires the entire install surface — `just setup`, `just deploy-*`, `/setup-merge`, `claude/Agents.md`, `claude/rules/`, `claude/reference/`, `claude/settings.json`, `hooks/cortex-notify.sh`, and the `setup` subparser stub in `cortex_command/cli.py` — without building a replacement subcommand. Machine-level install becomes "`uv tool install -e .` + `/plugin install` in Claude." Per-repo setup belongs to `cortex init` (ticket 119). Rule content that was load-bearing (commit format, never `git -C`, no heredocs, avoid compound commands) migrates into the relevant skills when tickets 120/121 move those skills into plugins.

## Non-Requirements

- **No deployment to `~/.claude/`**: not rules, not reference docs, not settings.json, not notify.sh, not statusline.sh. Cortex-command writes nothing under `~/.claude/` after this ticket.
- **No deployment to `~/.local/bin/`**: the cortex CLI entry point is placed by `uv tool install -e .`. Other utilities move to the cortex-interactive plugin's `bin/` (ticket 120) — not deployed by this ticket.
- **No `--merge-settings`, `--verify-symlinks`, `--with-extras`, `--dry-run`, `--with-statusline`, `--prune-orphans` flags**: no `cortex setup` command exists to flag.
- **No pre-migration of rule content into skills**: rule content (commit format, never `git -C`, avoid compound commands, no heredoc) does NOT get inlined into any `skills/*/SKILL.md` in this ticket. Inlining happens in tickets 120/121 as part of moving the skills into plugin directories.
- **No pre-migration of reference docs into plugin**: `claude/reference/*.md` are deleted, not moved. Tickets 120/121 re-author or re-import them at `plugins/*/references/` as part of plugin creation.
- **No deprecation banner on `just setup`**: deleted cleanly; maintainer accepts the short transitional gap per prior scoping.
- **No removal or cleanup of pre-existing legacy symlinks** in the user's `~/.claude/` from prior `just setup` runs: if a user has old `~/.claude/{hooks,skills,rules,reference}/*` symlinks or `~/.claude/{CLAUDE.md,notify.sh,statusline.sh}`, they remain until the user cleans up or fresh-installs. Migration scripts are ticket 124's scope.
- **No management of `~/.claude/CLAUDE.md`**: cortex-command does not install or touch this file. Users' global CLAUDE.md is their own concern.
- **No per-repo sandbox allowWrite setup**: today's `just setup` writes `$(pwd)/lifecycle/sessions/` into `~/.claude/settings.local.json` (justfile:100-117, 390-408). This moves to ticket 119 (`cortex init`) as per-repo overnight enablement.

After 117 deletes `claude/reference/`, pre-existing skill symlinks on old installs still work (they point into cortex-command's clone, which no longer has reference/ — the references break). New installs (via `uv tool install -e .`) don't symlink skills into `~/.claude/skills/` at all; they only work once plugins land in 120/121. The maintainer accepts both gap modes per earlier scoping ("we'll work directly on those next tickets").
- **No preservation of `.claude/skills/setup-merge/` as archive material**: deleted outright, not moved to an `archive/` directory. Git history is the archive.

## Edge Cases

- **Existing install with `~/.claude/settings.json → claude/settings.json` symlink**: R6 deletes the target file; the symlink dangles. Claude Code will fail to load cortex-specific settings (permissions, hook wiring, statusline) on the next session. The maintainer's mitigation: machine-config takes over `~/.claude/settings.json` before pulling 117 (manual coordination), OR accepts the broken state until machine-config is updated. Cortex does NOT ship a migration step.
- **Existing install with `~/.claude/CLAUDE.md → claude/Agents.md` symlink**: R3 deletes the target; symlink dangles. Claude Code fails to load cortex-specific global instructions. Same mitigation pattern as above; no cortex-side action.
- **Existing install with `~/.claude/rules/cortex-*.md → claude/rules/*.md` symlinks**: R4 deletes the targets; symlinks dangle. No functional impact beyond losing those two specific rules (git commit format, sandbox behaviors); rules in-plugin (via 120/121) re-provide the equivalent guidance when plugins install.
- **Existing install with `~/.claude/reference/*.md → claude/reference/*.md` symlinks**: R5 deletes targets; symlinks dangle. Skills that reference these paths (pre-plugin `skills/*/SKILL.md` files) fail their conditional-loading reads. Skills continue to function but without reference-doc augmentation until plugins land.
- **Existing install with `~/.claude/notify.sh → hooks/cortex-notify.sh` symlink**: R7 deletes target; symlink dangles. `settings.json` hook wiring that references `~/.claude/notify.sh` (if still active on old installs) silently fails to fire notifications. Non-critical; user-visible as "no sound/banner on session complete" until machine-config provides a replacement.
- **`just --list` after 117 lands**: shows remaining recipes only (overnight-*, dashboard, test*, backlog-index, validate-spec, setup-tmux-socket, setup-github-pat*, lifecycle-archive, python-setup, etc.). The user running `just setup` sees an error from `just` ("error: recipe `setup` not found"). Acceptable; the deprecation is loud, not silent.
- **`cortex setup` invocation on new install**: argparse rejects the unknown subcommand with its standard error ("invalid choice: 'setup'"). Users following stale docs that mention `cortex setup` hit an obvious error pointing at the argparse help output. R11 and R12 rewrite those docs.
- **Backlog item 118's `cortex upgrade` references `cortex setup --verify-symlinks`**: R11 audits backlog/ — 118 surfaces as a non-complete file with a `cortex setup` reference. 118's author updates the definition when they pick up 118 (likely to `git -C ~/.cortex pull` without a setup call, or `uv tool install -e ~/.cortex --force` to rebuild the tool entry point). 117 flags the drift but does not mandate the replacement text.
- **Ticket 119's `cortex init` inherits the sandbox allowWrite responsibility**: this is Non-Requirement territory, but if 119 isn't written to include it, overnight runs will fail with sandbox-blocked writes. 119's author responsibility; R11's audit catches any doc references that assume `just setup` provided the allowWrite entry.

## Technical Constraints

- **argparse subparser removal**: remove the `subparsers.add_parser("setup", ...)` call and associated `_make_stub("setup")` handler in `cortex_command/cli.py`. If the EPILOG text references `setup` specifically, update or remove the mention. No other CLI surface changes in this ticket.
- **Exit code contract** (for the one remaining behavior — running `cortex` with `setup` as argument): argparse's default rejection of an unknown subcommand produces exit 2 with a stderr message listing valid subcommands. Acceptable.
- **No new Python dependencies**: this ticket only removes code and files.
- **No changes to `[project.scripts]`** in `pyproject.toml`: users do NOT need `uv tool install -e . --force` to pick up these changes — the editable install reads `cortex_command/` live.
- **Docs ship in the same PR as the deletions**: R11's documentation updates land with R1-R10 in a single PR. Do not split — stale docs pointing at deleted recipes are a broken user experience.
- **Ordering within the PR**: order the commits so that at no mid-PR commit does a doc file reference something the same commit has just deleted (avoid broken intermediate states that complicate bisect). Concretely: commit documentation updates before or alongside the deletions.
- **Repo root contract**: The cortex-command repo root is determined by `uv tool install -e <path>` at install time (the repo path passed to `uv tool`) or by `$CORTEX_COMMAND_ROOT` in the user's shell profile. Nothing in this ticket depends on repo-root resolution — there is no runtime path-computation code to write.
- **Test environment**: no new tests. Running the existing test suite (`just test`, or `pytest` against `tests/` + `cortex_command/tests/`) after the deletions must exit 0. Any tests that reference `merge_settings.py`, `setup-merge`, or the deleted justfile recipes get removed as part of their test file's edits (or the test file itself if it was only about these).
- **Backlog audit is advisory, not mandatory**: R11's grep audit targets the active (non-complete) backlog items for broken references. Since 118/120/121/122/124 are all currently in the backlog and will be updated by their respective authors as those tickets are worked, R11 flags stale text but does not mandate its replacement — that's each ticket author's responsibility when they pick up their ticket. 117 is responsible for surfacing the drift, not for resolving it in every sibling ticket.
