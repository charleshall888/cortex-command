# Multi-step Complete phase and interactive worktree lifecycle

Cortex's lifecycle Complete phase becomes a single multi-step phase rather than splitting into a new Submit phase plus Complete. The phase progresses through sequential steps — running tests, committing lifecycle artifacts, creating a PR, writing `pr.json`, emitting `pr_opened`, exiting with a merge-wait handoff message, then on re-invocation routing through a state-aware evaluation-order matrix to perform post-merge worktree cleanup, backlog write-back, and `feature_complete` event emission. This shape treats merge as the terminal event for a lifecycle feature, aligning with GitHub, Linear, Jira, and GitLab's convention that "Done" means merged, not PR-opened.

Two related decisions are locked alongside this phase restructuring:

**WorktreeCreate-hook bypass is intentional and permanent.** Lifecycle skills create `interactive/{slug}` worktrees via a direct `git worktree add` call inside `create_worktree()`, bypassing Claude Code's native `--worktree` flag and its associated `WorktreeCreate` hook. This is by design: the `WorktreeCreate` hook fires only for Claude Code's own `--worktree` launch path; lifecycle's programmatic creation intentionally does not trigger it. The hook's responsibilities (path resolution, `.venv` symlink, settings copy) are handled inside `create_worktree()` directly, which is the single chokepoint for worktree creation. This bypass is permanent: a future refactor that re-routes lifecycle worktree creation through the hook would require updating all hook responsibilities and all callers, and would introduce a dependency on Claude Code's internal `--worktree` launch flow that the overnight runner cannot use.

**PATH bootstrap migrated to a `SessionStart` hook with cortex-shape gate.** The PATH bootstrap (adding `.local/bin` and `opt/homebrew/bin`) that was previously inlined in the `WorktreeCreate` hook at `claude/hooks/cortex-worktree-create.sh` is migrated to a new `SessionStart` hook (`claude/hooks/cortex-session-start-path-bootstrap.sh`) registered in `plugins/cortex-core/hooks/hooks.json`. The hook exits 0 silently when `cortex/lifecycle/` does not exist in the resolved repo, preventing PATH mutation outside cortex contexts. This covers the Variant B flow (fresh `claude --worktree` session starting inside a worktree) without requiring the inline bootstrap in `WorktreeCreate`. For Variant A (parent session invokes `claude --worktree` as a subprocess), the parent's `SessionStart` already ran before `WorktreeCreate` fired, so PATH was bootstrapped earlier in the parent's lifecycle; the two cases compose without the inline hook.

Three-criteria gate: **hard to reverse** — the phase restructuring changes when `feature_complete` fires and what "Done" means; reversing it would require coordinating changes across `complete.md`, `review_dispatch.py`, `metrics.py`, `statusline.sh`, `cortex-scan-lifecycle.sh`, `SKILL.md` kept-pauses inventory, and the parity test; **surprising without context** — a contributor encountering a re-runnable `complete` invocation that exits mid-phase with "merge on GitHub, then re-invoke" would not expect this from surrounding lifecycle conventions; **real trade-off** — gives up phase-boundary observability (a named Submit phase would appear as a distinct phase in statusline and scan output) in exchange for approximately 25 files less churn.

## Considered Options

- **R1: New Submit phase before Complete** (rejected alternative): introduce a new lifecycle phase named Submit that owns PR creation and the merge-wait pause, leaving Complete as a clean post-merge finalization phase. Rejected because a mid-phase pause in the restructured Complete already affords the same user redirect point without naming a new phase. A new phase earns approximately 25 files of churn (SKILL.md, statusline, scan-lifecycle, events-registry, parity tests, morning-review, backlog status schema) for a named boundary that the multi-step single-phase model provides structurally without the cost. Phase-boundary observability is the only gain, and it is recoverable via the "Complete (awaiting merge)" statusline sub-state.

- **R2: Bimodal Complete** (rejected alternative): keep Complete as a single-pass phase for features without interactive worktrees (no merge-wait pause), and introduce the merge-wait pause only for interactive-worktree features. Rejected because bimodal behavior is harder to document and reason about — the same lifecycle command has different observable semantics depending on which implement option was used. This weakens the project's structural-enforcement direction (CLAUDE.md: prefer structural separation over prose-only enforcement). A uniform multi-step protocol applied to all features is strictly clearer, and the non-interactive path simply skips the worktree cleanup step silently.

- **R3: PR creation in Review's tail** (rejected alternative): move PR creation from Complete into the tail of the Review phase, making Review end with a pushed branch and an open PR. Rejected because Review has an explicit read-only contract — it runs checks, gathers feedback, and produces a recommendation, but does not write to GitHub. Moving PR creation into Review breaks that contract and couples the Review phase to GitHub auth and network availability. The Complete phase is the correct owner of GitHub-side state changes.

- **R4: Status transitions only** (rejected alternative): instead of a multi-step Complete with state-awareness, record the pre-merge sub-state purely as a backlog frontmatter status value (e.g., `status: submitted` or `status: pending-merge`). Rejected because this splits lifecycle state across two stores — the events log and the backlog frontmatter — with no single authoritative source for "what phase am I in." The events log is the canonical runtime state store for lifecycle phases per the existing architecture; the backlog frontmatter is a user-facing summary field synchronized at phase boundaries. Adding a transient in-flight status to backlog frontmatter would create drift risk (the backlog item could be manually edited out of sync) and would require adding a new `status` vocabulary value, widening the status schema for every downstream reader.
