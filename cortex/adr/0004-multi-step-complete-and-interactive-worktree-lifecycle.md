# Multi-step Complete phase and interactive worktree lifecycle

Cortex's lifecycle Complete phase becomes a single multi-step phase rather than splitting into a new Submit phase plus Complete. The phase progresses through sequential steps — running tests, committing lifecycle artifacts, creating a PR, writing `pr.json`, emitting `pr_opened`, exiting with a merge-wait handoff message, then on re-invocation routing through a state-aware evaluation-order matrix to perform post-merge worktree cleanup, backlog write-back, and `feature_complete` event emission. This shape treats merge as the terminal event for a lifecycle feature, aligning with GitHub, Linear, Jira, and GitLab's convention that "Done" means merged, not PR-opened.

Two related decisions are locked alongside this phase restructuring:

**WorktreeCreate-hook bypass is intentional and permanent.** Lifecycle skills create `interactive/{slug}` worktrees via a direct `git worktree add` call inside `create_worktree()`, bypassing Claude Code's native `--worktree` flag and its associated `WorktreeCreate` hook. This is by design: the `WorktreeCreate` hook fires only for Claude Code's own `--worktree` launch path; lifecycle's programmatic creation intentionally does not trigger it. The hook's responsibilities (path resolution, `.venv` symlink, settings copy) are handled inside `create_worktree()` directly, which is the single chokepoint for worktree creation. This bypass is permanent: a future refactor that re-routes lifecycle worktree creation through the hook would require updating all hook responsibilities and all callers, and would introduce a dependency on Claude Code's internal `--worktree` launch flow that the overnight runner cannot use.

**PATH bootstrap migrated to a `SessionStart` hook with cortex-shape gate.** The PATH bootstrap (adding `.local/bin` and `opt/homebrew/bin`) that was previously inlined in the `WorktreeCreate` hook at `claude/hooks/cortex-worktree-create.sh` is migrated to a new `SessionStart` hook (`claude/hooks/cortex-session-start-path-bootstrap.sh`) registered in `plugins/cortex-core/hooks/hooks.json`. The hook exits 0 silently when `cortex/lifecycle/` does not exist in the resolved repo, preventing PATH mutation outside cortex contexts. This covers the Variant B flow (fresh `claude --worktree` session starting inside a worktree) without requiring the inline bootstrap in `WorktreeCreate`. For Variant A (parent session invokes `claude --worktree` as a subprocess), the parent's `SessionStart` already ran before `WorktreeCreate` fired, so PATH was bootstrapped earlier in the parent's lifecycle; the two cases compose without the inline hook.

Three-criteria gate: **hard to reverse** — the phase restructuring changes when `feature_complete` fires and what "Done" means; reversing it would require coordinating changes across `complete.md`, `review_dispatch.py`, `metrics.py`, `statusline.sh`, `cortex-scan-lifecycle.sh`, `SKILL.md` kept-pauses inventory, and the parity test; **surprising without context** — a contributor encountering a re-runnable `complete` invocation that exits mid-phase with "merge on GitHub, then re-invoke" would not expect this from surrounding lifecycle conventions; **real trade-off** — gives up phase-boundary observability (a named Submit phase would appear as a distinct phase in statusline and scan output) in exchange for approximately 25 files less churn.

## Considered Options

- **R1: New Submit phase before Complete** (rejected alternative): introduce a new lifecycle phase named Submit that owns PR creation and the merge-wait pause, leaving Complete as a clean post-merge finalization phase. Rejected because a mid-phase pause in the restructured Complete already affords the same user redirect point without naming a new phase. A new phase earns approximately 25 files of churn (SKILL.md, statusline, scan-lifecycle, events-registry, parity tests, morning-review, backlog status schema) for a named boundary that the multi-step single-phase model provides structurally without the cost. Phase-boundary observability is the only gain, and it is recoverable via the "Complete (awaiting merge)" statusline sub-state.

- **R2: Bimodal Complete** (rejected alternative): keep Complete as a single-pass phase for features without interactive worktrees (no merge-wait pause), and introduce the merge-wait pause only for interactive-worktree features. Rejected because bimodal behavior is harder to document and reason about — the same lifecycle command has different observable semantics depending on which implement option was used. This weakens the project's structural-enforcement direction (CLAUDE.md: prefer structural separation over prose-only enforcement). A uniform multi-step protocol applied to all features is strictly clearer, and the non-interactive path simply skips the worktree cleanup step silently.

- **R3: PR creation in Review's tail** (rejected alternative): move PR creation from Complete into the tail of the Review phase, making Review end with a pushed branch and an open PR. Rejected because Review has an explicit read-only contract — it runs checks, gathers feedback, and produces a recommendation, but does not write to GitHub. Moving PR creation into Review breaks that contract and couples the Review phase to GitHub auth and network availability. The Complete phase is the correct owner of GitHub-side state changes.

- **R4: Status transitions only** (rejected alternative): instead of a multi-step Complete with state-awareness, record the pre-merge sub-state purely as a backlog frontmatter status value (e.g., `status: submitted` or `status: pending-merge`). Rejected because this splits lifecycle state across two stores — the events log and the backlog frontmatter — with no single authoritative source for "what phase am I in." The events log is the canonical runtime state store for lifecycle phases per the existing architecture; the backlog frontmatter is a user-facing summary field synchronized at phase boundaries. Adding a transient in-flight status to backlog frontmatter would create drift risk (the backlog item could be manually edited out of sync) and would require adding a new `status` vocabulary value, widening the status schema for every downstream reader.

## branch-mode default (Approach C) + Approach A deferred design surface

This section amends ADR-0004 to record the C/A split framing introduced by the `lifecycle-implement-auto-enter-worktree-drop` lifecycle. The Status field above remains Proposed; this amendment is institutional memory for the multi-phase continuation, not a promotion to Accepted.

- **(a) Variant A/B was the deliberately-scoped first cut per epic #240.** The original interactive-worktree work shipped under epic #240 as "T10 ships only the create + handoff step" — a deliberate scoping choice that landed worktree creation and the merge-wait handoff without yet committing to a per-repo default or to mid-session auto-enter. This C/A split is a routine multi-phase continuation of that scoped first cut, not a course correction.

- **(b) This lifecycle adds the per-repo `branch-mode` default (Approach C).** A new optional `branch-mode` field in `lifecycle.config.md` lets a repo declare its default branching strategy (e.g., interactive worktree vs. in-place branch) so the implement phase can resolve the default without re-prompting on every feature. A minimal `read_branch_mode()` parser primitive reads the raw frontmatter value; caller-side validation against the closed set keeps the primitive narrow and forward-compatible. This is Approach C: pick the lane via per-repo config, leave session entry semantics unchanged.

- **(c) Approach A's deferred design surface comprises one platform-side constraint plus cortex-side scoping decisions.** Approach A is the mid-session `EnterWorktree(path=...)` auto-enter shape that this lifecycle does NOT ship. Its deferral is bounded by:
  - **Platform-side constraint (one):** Anthropic's `ExitWorktree` is documented as a no-op when invoked cross-session against worktrees created outside the current session. The multi-step Complete phase's same-session Complete re-invocation pattern (re-entering Complete after merge from within the worktree-entering session) needs an out-of-worktree CWD before Step 8's hard guard runs; `ExitWorktree` cannot provide that programmatically in this case. This is a genuine platform-side blocker, not a cortex-side choice.
  - **Cortex-side scoping decisions (two):** (i) consumer-repo authorization for `EnterWorktree` — originally framed as a `cortex init` opt-in clause that consumer repos sign onto before the harness will auto-enter their worktrees (this deferred surface was resolved by ADR-0006 and is **now superseded by ADR-0008**: cortex writes no consumer-`CLAUDE.md` clause — the user's live selection of a `worktree`-labeled picker option authorizes `EnterWorktree`, and the suppressed-picker path routes structurally to the cd-shim); (ii) interaction with the WorktreeCreate-hook bypass already documented earlier in this ADR — lifecycle's programmatic worktree creation deliberately bypasses Claude Code's `--worktree` launch path, and any Approach A design must reconcile its session-entry shape with that bypass.

- **(d) Forward reference.** The follow-up backlog ticket `lifecycle-implement-auto-enter-worktree-deferred` (filed by Task 10 of this lifecycle) decomposes Approach A's deferred design surface into concrete sub-work — the platform constraint plus the two cortex-side scoping decisions — and is the canonical place to track A's path to shipping. The framing for that ticket is "decompose A's design surface and ship," not "wait for upstream."

Originating work: lifecycle slug `lifecycle-implement-auto-enter-worktree-drop`.

## Approach A resolved decisions

This section records the three decisions resolved by the `lifecycle-implement-auto-enter-worktree-via` lifecycle that completes Approach A's deferred design surface. Per the ADR-README promotion gate, this amendment co-promotes the ADR's Status field from `proposed` to `accepted` in the same commit.

**Decision (1) — Cross-session-exit interaction model: Complete-phase teardown stays as-is; the user exits the worktree before re-invoking Complete.** The load-bearing constraint is the live `EnterWorktree` / `ExitWorktree` tool schema clause that forbids skill-side proactive `ExitWorktree` invocation (*"Do NOT call this proactively — only when the user asks"*), which makes a programmatic same-session teardown unavailable to the lifecycle skill. The resolution is codified by spec requirement R13 (`skills/lifecycle/references/complete.md` Step 8 hard guard prose updated to teach both exit paths — `ExitWorktree action="keep"` when EnterWorktree session state is live, and `cd $(git rev-parse --show-toplevel)` for the cross-session fallback) and the corresponding Non-Requirement clause that the same-session Complete UX stutter is an accepted long-tail edge rather than a regression.

**Decision (2a) — Authorization shape: dual-path authorization covering picker-fires and suppressed-picker entries.** **[Superseded in part by ADR-0008 — see the supersession note at the end of this decision.]** The load-bearing schema clause is the `EnterWorktree` requirement that *"'worktree' is explicitly mentioned by the user or in CLAUDE.md / memory instructions"* before the tool is callable. The resolution is two complementary surfaces: the picker-fires path is covered by the user's selection of an AskUserQuestion option whose label contains the literal word `worktree` (spec requirement R19 pins this via a parity test on `skills/lifecycle/references/implement.md` §1), and the suppressed-picker path (per-repo `branch-mode` default) is covered by a cortex-managed fenced authorization clause that `cortex init` writes into the consumer's `CLAUDE.md` (spec requirements R5, R6 for the writer with the literal-word-twice invariant, plus R20 for the `cortex init --verify-worktree-auth` runtime probe that guards every §1a entry against drift). The clause body, fence shape, lifecycle (write, refresh on canonical-version bump, revoke), and the rejected `.claude/cortex-authorizations.md` sibling-location alternative are recorded in their own load-bearing artifact, ADR-0006. **Supersession note:** ADR-0006 is now superseded by ADR-0008, which removed the consumer-`CLAUDE.md` fence write entirely. The picker-fires half of this dual-path decision survives unchanged; the suppressed-picker half no longer writes or verifies a fence — it routes structurally to the cd-shim. Decision (2a)'s consumer-`CLAUDE.md` fence write is therefore historical, not a live decision.

**Decision (2b) — WorktreeCreate-hook bypass requires no expansion: verification at the `create_worktree` call site, not at the hook registry.** The load-bearing invariant is that `cortex_command/pipeline/worktree.py::create_worktree` invokes `git worktree add` directly via `subprocess` rather than routing through Claude Code's `--worktree` launch path — meaning the `WorktreeCreate` hook never fires for lifecycle-created worktrees and therefore needs no expansion to cover the Approach A auto-enter shape. The resolution is codified by spec requirement R18 (`tests/test_create_worktree_bypass.py` reads `create_worktree`'s source and asserts the subprocess invocation contains `'git', 'worktree', 'add'` and does NOT contain `'claude', '--worktree'`). This is the structural assertion that protects the bypass invariant the existing ADR-0004 text already declares; the prior R18 framing (a `hooks.json` grep) verified the wrong invariant and was reworked per the spec's critical-review trail.
