# Specification: add-bidirectional-concurrency-guards-for-interactive

## Problem Statement

The new interactive worktree mode (epic #237, replacing daytime autonomous via #246) places long-lived `claude` CLI sessions inside per-feature worktrees. Without concurrency guards, two interactive sessions can drive `implement` on the same feature simultaneously (corrupting the shared `interactive/{slug}` branch), an interactive session can collide with a live overnight runner on the same repo (the shared-git-index race that wontfix #135 named), and overnight can dispatch into a feature that already has a live interactive owner (clobbering uncommitted work). The benefit accrues to anyone running worktree-interactive: a single coherent contract — at most one live interactive owner and one overnight runner per feature, with stale state never destroying uncommitted work.

- **#239 (manage-interactive-feature-worktree-lifecycle-creation, currently in planning)** owns the worktree-creation primitive in `implement.md` option 2 body (`create_worktree(feature, base_branch)`) and the multi-step `complete.md` cleanup protocol. **#241 owns**: lock-file format, liveness semantics, preflight rejection logic, overnight-side per-round scan. **Specific reconciliation**: (a) #239's R9 prose mentions "Pre-flight: checks ... for liveness (via `kill -0`)"; the implementation that satisfies #239's R9 acceptance gates (which only grep for `create_worktree` and `interactive/`) AND #241's R13 acceptance (no new inline `kill -0`) is a single bash invocation of `cortex-interactive-lock acquire {slug}` in implement.md option 2 — the helper does the kill -0 internally. (b) `release_lock` (per #241 R1.d) is invoked at #239's Complete step 8 (before `cleanup_worktree`), making lock-release and worktree-cleanup naturally adjacent. (c) Lock-file path is `cortex/lifecycle/{slug}/interactive.pid` (per-feature directory, matching daytime.pid precedent at `daytime_pipeline.py:71`) — #239's R9 prose reference to `cortex/lifecycle/sessions/{slug}.interactive.pid` is a path-naming error that #239's plan should correct in lockstep with #241's landing.
- **#240 (Variant A interaction model + PR-creation hook)** owns the `cd` semantics. #241's helper writes the lock from whatever CWD it's invoked in because it resolves the path via `_resolve_user_project_root()` (main-repo-anchored).
- **#246 (daytime-autonomous removal)**: sequence-independent — distinct artifacts (interactive.pid vs daytime.pid); #241 can ship before, after, or in parallel.

## Phases

- **Phase 1: Interactive-path guards** — Python helper module + per-feature `interactive.pid` lock acquisition (env-var-primary identity with auxiliary PID liveness hint) + overnight-active rejection mirror in the implement preflight + skill-prose integration.
- **Phase 2: Overnight-path guard** — per-round inverse scan in `cortex_command/overnight/orchestrator.py` that excludes live-interactive-owned features from feature dispatch on each round entry (no `state.features` mutation, no persistence), with `feature_skipped_interactive_active` events emitted at exclusion time.

3. **Lock file JSON schema with env-var-primary identity**: The lock file contents are JSON: `{"schema_version": 1, "magic": "cortex-interactive-lock", "session_id": <string from CLAUDE_CODE_SESSION_ID>, "pid": <int from os.getppid()>, "start_time": <float | null>, "acquired_at": <ISO 8601 timestamp>}`. `session_id` is the authoritative identity (from env). `pid` is best-effort liveness hint — `os.getppid()` (the helper's immediate parent, may be zsh/uv/bash depending on invocation chain). `start_time` is `psutil.Process(pid).create_time()` rounded to milliseconds when accessible, else `null` (AccessDenied or NoSuchProcess → null, do not raise). Atomic-written via tempfile + `os.replace`, mode 0o600. Acceptance: `python3 -c "import json,subprocess; subprocess.run(['cortex-interactive-lock','acquire','probe'],check=True); d=json.load(open('cortex/lifecycle/probe/interactive.pid')); assert set(d) >= {'schema_version','magic','session_id','pid','start_time','acquired_at'}; assert d['magic']=='cortex-interactive-lock'"` exits 0 in a test fixture. **Phase**: Phase 1.

4. **Liveness predicate — env-var authoritative, auxiliary PID is hint, all branches enumerated**: `verify_live_owner(lock)` returns one of `LIVE | STALE` using this strictly-ordered branch table. The function MUST cover all eight enumerated cases; no implicit fall-through is permitted.

   | # | env-var match? | os.kill verdict | start_time | psutil.create_time | Verdict | Recovery reason (if STALE) |
   |---|---|---|---|---|---|---|
   | 1 | YES | (not consulted) | (not consulted) | (not consulted) | **LIVE** | — |
   | 2 | NO / absent | raises ESRCH | (not consulted) | (not consulted) | **STALE** | `"esrch"` |
   | 3 | NO / absent | raises EPERM | (not consulted) | (not consulted) | **LIVE** | — (conservative; cross-sandbox or cross-uid kill EPERM implies process exists) |
   | 4 | NO / absent | succeeds | null (lock had no start_time at acquire) | (not consulted) | **LIVE** | — (conservative; PID alive is best signal we have when start_time was unrecordable) |
   | 5 | NO / absent | succeeds | non-null | matches within ±2s | **LIVE** | — |
   | 6 | NO / absent | succeeds | non-null | mismatches | **STALE** | `"start_time_mismatch"` |
   | 7 | NO / absent | succeeds | non-null | raises psutil.NoSuchProcess | **STALE** | `"nosuchprocess"` |
   | 8 | NO / absent | succeeds | non-null | raises any other psutil exception (AccessDenied, ZombieProcess, etc.) | **LIVE** | — (conservative) |

   Recovery reasons in column 7 form an exhaustive enum: `{"esrch", "start_time_mismatch", "nosuchprocess"}`. (Cases 4 and 5 don't produce STALE so don't need a reason.) Acceptance: unit tests in `tests/test_interactive_lock.py` cover all 8 rows of the table with one test per row (`test_verify_live_owner_row_1` through `_row_8`); `pytest tests/test_interactive_lock.py -k verify_live_owner_row` exits 0 with all 8 tests passing. **Phase**: Phase 1.

## Non-Requirements

- **Shared `pid_lock.py` helper extraction** is OUT of scope. The new `cortex_command/interactive_lock.py` is purpose-built for this ticket; abstracting across runner.pid/daytime.pid (the latter being deleted by #246) is premature per CLAUDE.md.
- **Statusline / dashboard live-owner indicator** is OUT of scope. Today the statusline doesn't display per-feature interactive-owner state; adding it (with 500ms budget implications) is a separate follow-up.
- **Cross-checkout concurrency coverage** is OUT of scope (per user direction). The guard is per-checkout; two clones can each have a live owner on the same feature; collision surfaces at git push time.
- **Sub-agent participation in the lock contract** is OUT of scope. Sub-agents dispatched via `cortex_command/pipeline/dispatch.py` have `CLAUDECODE` cleared and `CLAUDE_CODE_SESSION_ID` not explicitly forwarded. Sub-agents inherit the parent session's lock claim implicitly (the parent session is the lock owner; the sub-agent is acting on behalf of the parent and operates within the parent's worktree). Spec contract: sub-agents do NOT call `cortex-interactive-lock acquire`; if they do (test fixture, etc.), behavior is undefined.
- **Hardening the `active-session.json` "is runner really executing" check** (backlog #134) is OUT of scope. The interactive mirror inherits whatever active-session contract guarantees today.
- **Per-feature gate inside `feature_executor.execute_feature` (defense-in-depth)** is OUT of scope. The TOCTOU window between runner.py startup scan and feature_executor dispatch is accepted (a late-arriving interactive owner colliding with overnight surfaces as a `git worktree add` failure during create_worktree, which the orchestrator handles via existing error paths). Add only if observed in production.
- **`/clear` mid-session as a stale-recovery trigger** is OUT of scope. The liveness predicate (R4) does NOT verify session_id from env against the stored lock — it only verifies env-var match as an OPTIMIZATION (env-var match = authoritative LIVE). On `/clear`, env-var rotates → primary check falls through → auxiliary PID+start_time check applies → if the same claude process is still alive, PID matches and verdict is LIVE. `/clear` is harmless.
- **Atomic O_EXCL lock creation** is OUT of scope. Human-paced interactive launch makes write-race-on-claim a non-concern; the read-then-write sequence with stale recovery handles all practical cases. PID-reuse-on-read is closed by R4's start_time check.
- **`fcntl.flock` / `lockf` with fd held for session lifetime** is OUT of scope. Adds fd-management complexity without closing the failure modes R4 already addresses.
- **Worktree creation in implement.md option 2** is OUT of scope (owned by #239). This ticket adds preflight guards around #239's worktree-creation step but does not create worktrees itself.
- **Multi-step Complete protocol and `release_lock` invocation timing** is OUT of scope (owned by #239). #241 provides `release_lock` as a primitive; #239's Complete step 8 invokes it before `cleanup_worktree`.

## Edge Cases

- **Mac standby + Claude Code crash**: User's laptop sleeps; `claude` is killed by force-quit/OOM. Lock file remains pointing at the now-dead session_id and PID. **Expected**: next preflight (in a NEW claude session with a new CLAUDE_CODE_SESSION_ID) — env-var primary check mismatches → fallback to PID+start_time → `os.kill` returns ESRCH → STALE → emits `interactive_lock_stale_recovered` with `recovery_reason: "esrch"`, unlinks, re-acquires. Worktree contents preserved.

- **`claude --resume` on a crashed session**: Resume mints a new process with a new PID. If `--resume` preserves the original `CLAUDE_CODE_SESSION_ID` (per Anthropic docs), env-var primary check MATCHES and verdict is LIVE — but the stored PID points at the dead PID. **Expected**: env-var match wins (authoritative LIVE). The auxiliary PID is stale but ignored. The resumed session can proceed (it IS the owner per session_id). Optional: `release_lock`-on-clean-exit at end of resumed session writes a fresh PID+start_time to keep the auxiliary hint accurate, but this is not strictly required for correctness.

- **`/clear` mid-feature**: Env-var rotates → primary check fails. Auxiliary PID+start_time check: same claude process still alive, so `os.kill` and `psutil.create_time()` both succeed against the stored PID → verdict is LIVE. The lock holds across `/clear`. (Per Non-Requirements: `/clear` is explicitly not a stale-recovery trigger.)

- **Two clones of the repo, same feature**: Per-checkout lock; cross-checkout out-of-scope. Both interactive sessions succeed locally; collision surfaces at `git push` time as a fast-forward rejection.

- **Read-while-write race on interactive.pid**: Atomic `tempfile + os.replace` (R3) means readers observe either old or new content — never partial. On JSON parse failure or schema mismatch, defensive parse returns None and the lock is treated as absent (no owner; conservative — prefer false-negative over false-positive blocking).

- **PID reuse over multi-day interactive session**: The recorded `pid` is `os.getppid()` from the helper — likely uv or zsh, both potentially long-lived but could exit. If reused, env-var primary check matches the still-live claude (via `CLAUDE_CODE_SESSION_ID`) → verdict is LIVE; auxiliary PID staleness is irrelevant. If claude itself exits and a new claude session starts, the new CLAUDE_CODE_SESSION_ID won't match → fallback to PID check → may produce false-LIVE if reused PID has matching create_time (rare, but PID+start_time check is best-effort). The session_id is the primary defense; PID is auxiliary.

- **Runner.py / orchestrator.py crashes mid-overnight**: R9's scan does NOT mutate or persist any state (per R9's revised design — per-round filter only). No half-persisted filter to recover from. Next runner startup re-runs the scan on round 1 entry; current live-lock state on disk drives the filter; no stale persisted intermediate.

- **Owner exits during overnight run (between round N and round N+1)**: Round N filtered the feature (owner alive); owner exits; round N+1's scan re-derives, sees no live lock, the feature is in `eligible` again and dispatches in round N+1. **No multi-overnight delay** — convergence is per-round, not per-overnight-run.

- **Owner acquires AFTER round-N scan but BEFORE round-N feature dispatch (the principal TOCTOU window)**: Round N's scan computed `eligible` without the just-acquired lock; the feature dispatches; `git worktree add` fails or `create_worktree` finds an existing worktree (owner has already created it). The orchestrator's existing error path handles the collision; the feature is marked failed-for-this-round and retried in round N+1 when the next scan picks up the live lock and filters it out. Per Non-Requirements: per-feature gate in `feature_executor.execute_feature` not added (would close this window but is currently judged not worth the complexity unless production failures emerge).

- **Bash-tool subprocess parent isn't claude (e.g., wrapper, sub-agent context)**: Per R3, `pid` is `os.getppid()` from the helper — whatever the helper's immediate parent is (uv, zsh, sandbox-exec, etc.). The session_id from `CLAUDE_CODE_SESSION_ID` env is authoritative; PID is hint. Wrapper scenarios are handled by env-var primacy without special-casing.

- **`CLAUDE_CODE_SESSION_ID` absent (very old Claude Code or future rename)**: Helper records `session_id: null`. Verification falls back entirely to PID+start_time (R4 fallback branch). Lock still works — degraded mode without env-var defense. Log a one-time warning to events.log: `interactive_lock_acquired` event has `session_id_source: "env_absent_pid_only"` to flag the degraded state for operator visibility.

- **`runner.pid` JSON parse fails in overnight-active rejection mirror**: Defensive parse — treat as absent, fall through to "no runner active, proceed." Mirrors existing `install_guard.py:223` self-heal pattern.

## Technical Constraints

- **Atomic writes** (`pipeline.md:21,126`): `interactive.pid` writes use tempfile + `os.replace`. Mode 0o600.
- **State-file-locking architectural constraint** (`pipeline.md:127,134`): readers may observe state mid-mutation; defensive parse on read.
- **PID-reuse defense via psutil + start_time** (`pipeline.md:28`): ±2s tolerance mirrors `verify_runner_pid` (`ipc.py:431`); explicit best-effort posture in R4.
- **Path resolution to main repo root** (`common.py:55-103`): `_resolve_user_project_root()` terminates on `.git` (file or dir), so worktree CWD resolves up to the main repo. Lock path is main-repo-absolute, NEVER CWD-relative.
- **Sandbox allowWrite per-repo registration** (ADR-0003 / `cortex init`): main repo's `cortex/` umbrella is registered. R14's sandbox-write probe verifies that worktree-CWD-to-main-repo-path writes succeed under this registration.
- **File-based state** (ADR-0001): plain JSON files; no daemons.
- **Skill-prose Bash discipline**: skill prose calls `cortex-interactive-lock acquire/release` as single bash invocations; complex logic lives in the Python helper.
- **Events-registry conformance**: `cortex-check-events-registry --staged` and `--audit` both pass.
- **`pipeline.md:130` rationale field**: `feature_skipped_interactive_active` MUST include non-empty `rationale`.
- **No `state.features` mutation, no save_state from R9's path**: per the revised R9, the inverse scan is per-round in `orchestrator.py` and operates on the local `feature_names` variable. `state.features` (dict[str, OvernightFeatureStatus]) is untouched — runtime fields (status, recovery_attempts, worktree_path, spec_path, plan_path, started_at) are preserved across rounds because the filter is computed at dispatch time, not persisted.
