# Research: Resolve cortex CLI interpreter for hook consumers

Goal: make `hooks/cortex-scan-lifecycle.sh` resilient to install topology by adding `cortex --print-python` (argparse short-circuit, minimal imports) to the CLI and updating the hook to resolve the interpreter via `CORTEX_PY=$(cortex --print-python ... || ... python3)` at every bare-`python3` site. Preserve the existing loud-error discipline.

## Codebase Analysis

**Files that will change**

- `cortex_command/cli.py` — add `--print-python` flag and dispatcher. Existing precedent at `cli.py:354-362` (flag definition) and `cli.py:776-777` (interception in `main()`); existing dispatcher `_dispatch_print_root()` at `cli.py:181-259` lazy-imports `cortex_command` modules *inside* the handler (good model), but a true zero-import flag needs more — see Adversarial γ.
- `hooks/cortex-scan-lifecycle.sh` — substitute `$CORTEX_PY` at **three** bare-`python3` sites, not two: (1) preflight import-check at `~line 26-29`, (2) inline-batch phase-detection at `~line 244-256`, (3) metrics regeneration at `~line 417`. The user's original framing said "two sites" — that was an undercount.
- `plugins/cortex-overnight/hooks/cortex-scan-lifecycle.sh` — auto-regenerated by `just build-plugin`. Pre-commit at `.githooks/pre-commit` phase 2-4 detects drift (scans `hooks/cortex-*`, runs build, `git diff --quiet -- plugins/...` blocks on drift).
- Test surface: `tests/test_hooks.sh:94-143` has fixture scaffolding (TMPDIR + sed-substituted JSON fixtures + stdin redirect + jq-parsed output). Backlog #170 documents this scaffolding as silently-failing in uv-tool topology — i.e., the bug we're fixing also breaks the test that would catch it.

**Existing patterns**

- `--print-root` (`cli.py:354-362`, `cli.py:776-777`, dispatcher at `cli.py:181-259`) is the structural precedent: top-level flag intercepted in `main()` before subcommand dispatch.
- `--print-root --format json` exists (suppressed in `--help` via `argparse.SUPPRESS` at `cli.py:366-372`). It returns a versioned JSON envelope per ADR-0002.
- The hook's existing "fail loud with remediation" discipline at `cortex-scan-lifecycle.sh:31-37` is a grandfathered MUST per CLAUDE.md's MUST-escalation policy. Preserve.

**Other bare-`python3` cortex_command consumers (outside the hook)**

- `bin/cortex-backlog-ready:6-7` — same pattern: `python3 -c "import cortex_command..."` + `python3 -m cortex_command...`. Two-branch wrapper.
- `bin/cortex-morning-review-complete-session:6-7` — same.
- Embedded `python3 -c` snippets in `skills/critical-review/references/residue-write.md`, `skills/lifecycle/references/implement.md`. Same bug, different contexts.
- These are tracked by an R14 "wrapper-cleanup" track and a `bin/.audit-bare-python-m-allowlist.md` allowlist — i.e., the project already recognizes this anti-pattern but the cleanup has not landed.

## Web Research

- **uv tool install layout**: tool venv at `~/.local/share/uv/tools/<pkg>/`; console-scripts installed in `~/.local/bin` as **symlinks on Unix, copies on Windows**. No `uv tool python <name>` flag exists today — this is the gap `--print-python` fills.
- **pipx parallels**: pipx exposes `pipx environment` and `pipx list --json`, but no `pipx python <name>`. Cortex's approach is more ergonomic than the established pipx pattern.
- **CLI introspection idiom**: bare-string single-line output (`git --exec-path` precedent) is the cleanest shell `$(...)` contract for single-value flags. Structured JSON is right for multi-field endpoints (which `--print-root` is).
- **Hook-from-shell → tool-python patterns**: pre-commit (the framework) has long-standing community asks for exactly this surface (`pre-commit` #1522 / #2133 / #2341). Cortex is filling a known gap.
- **`python3` PATH risk**: documented anti-patterns include macOS Xcode-CLT-stub `/usr/bin/python3` (opens an interactive "Install Developer Tools" prompt — disastrous from a silent hook), RHEL python2-as-python3, and PEP 668 externally-managed-environment refusal.
- **Pre-argparse short-circuit**: Click `is_eager` does NOT beat heavy imports. Idiomatic technique is a `sys.argv` peek BEFORE any `from cortex_command import ...`. But see Adversarial γ — the existing `cli.py:17` blocks this.

## Tradeoffs & Alternatives

Five candidate approaches were evaluated.

- **A. New `--print-python` flag** (user's choice). Pros: ~10 LOC, mirrors `--print-root`, single source of truth (`sys.executable`), survives every install topology the CLI itself survives, trivial shell consumption. Cons: adds a subprocess on the SessionStart hot path (see Adversarial α — measured ~400ms cold-start cost on the test host).
- **B. Extend `--print-root` JSON with a `python` field**. **Disqualified by measurement**: `_dispatch_print_root()` (`cli.py:234-243`) shells out to `git remote get-url` and `git rev-parse HEAD` with 5s timeouts each to populate `remote_url`/`head_sha`. That is the heaviest introspection call the CLI exposes today — using it from a hot-path hook would be a much larger regression than A.
- **C. Read the cortex shim's shebang directly** (no CLI change). Disqualified: uv installs symlinks (which dereference) on Unix and copies on Windows; uv's roadmap discusses native launchers; shebang-length limits (~127 bytes on Linux) bite deep CI/sandbox paths. Works today, brittle to install-toolchain evolution.
- **D. Defer to the subcommand refactor** (`cortex hooks scan-lifecycle`). Complementary, not competing: A is a 10-LOC primitive; D is a ~470-LOC port of the hook's bash logic. A serves at least three OTHER consumer classes (bin/cortex-* scripts, embedded skill snippets, third-party hooks) that D would not subsume.
- **E. Pip-install `cortex_command` into system python** (operational workaround). Contradicts project.md's non-editable-wheel distribution stance. Creates a drifting second install. Not a fix.

**Recommendation: A**, with caveats from Adversarial α/β/γ requiring spec resolution.

## Adversarial Review

Adversarial review challenged the "A dominates" conclusion on multiple axes. The high-signal findings:

- **α — Cold-start latency on the hot path**: empirical measurement shows `cortex --print-root` averages ~400ms wall-clock on the test host. A new `cortex --print-python` flag — even with the pre-argparse `sys.argv` peek — pays the same Python interpreter cold-start. Argparse parses in microseconds; the interpreter cold-start swamps it. Hook fires on every SessionStart and every `/clear`. Mitigation: cache the resolved interpreter in `$CLAUDE_ENV_FILE` so subsequent invocations within the same session pay the cost once, not per-event. SessionStart itself still pays it.
- **β — Stderr-swallowing hides legitimate failures**: `2>/dev/null || echo python3` is too permissive. It silently falls through to bare `python3` in three failure cases that should surface: (1) uv tool venv corruption (cortex segfaults), (2) PYTHONHOME/PYTHONPATH pollution crashing the CLI on site init, (3) macOS Xcode-CLT stub `python3` opening an interactive "Install Developer Tools" GUI from a SessionStart context. Mitigation: only fall back when `command -v cortex` itself fails (genuine pre-flag-existing CLI). If `cortex` exists but errors, surface — preserves the existing fail-loud discipline.
- **γ — The "no cortex_command imports" promise is unreachable as stated**: `cli.py:17` does `from cortex_command.init.handler import main as init_main` at module scope, which transitively imports `cortex_command.init.scaffold` and `cortex_command.init.settings_merge`. By the time `main()` runs and could peek at `sys.argv`, the init subtree is already loaded. To honor the promise, either: (a) hoist the peek ABOVE line 17 (ugly), or (b) convert `init_main` to lazy import (the pattern already used for overnight dispatchers at `cli.py:48-66`). Option (b) is the principled fix.
- **δ — Cross-checkout dev-loop staleness** (non-regression call-out): if the user is editing `cortex_command.common.detect_lifecycle_phase` in a dev checkout but hasn't reinstalled, the hook silently uses the stale installed version. This is true today with bare `python3` too — not introduced by the change. Spec should explicitly note non-regression rather than ignore.
- **ε — Test scaffolding is already known-fragile (backlog #170)**: tests must verify both branches of the `|| echo python3` fallback are exercised under fixtures. A test fixture that does NOT stage a `cortex --print-python` on `$PATH` will fall through to bare `python3` and silently test the OLD code path while appearing green.
- **ζ — bin/cortex-* deferral conflicts with Solution Horizon**: CLAUDE.md says "the same patch would apply in multiple known places you can name." Adversarial named four. Solution Horizon argues to include or to name a concrete blocker. Spec must address.
- **η — Behavior-change surface (newly-reachable hook code)**: today in uv-tool topology, the hook exits 1 at the preflight gate. After the fix, the hook NOW reaches session-migration logic, pipeline-state detection, batch phase-detection, and metrics regeneration — all of which have their OWN latent failure modes in that topology that have been masked by the early-exit. Spec should require a smoke-test that exercises the full hook in uv-tool topology, or explicitly acknowledge the increased surface.
- **θ — Plugin-mirror scope**: `just build-plugin` mirrors `hooks/cortex-scan-lifecycle.sh` into `plugins/cortex-overnight/hooks/`. The `.githooks/pre-commit` phase 2-4 enforcement may scan a wider plugin set than the agents verified. Spec must confirm the closed set of mirrored locations before landing.
- **ι — Contract-shape inconsistency with ADR-0002**: bare-string single-line output (matches `git --exec-path` precedent, simpler shell consumption) versus JSON envelope (consistent with `--print-root` and ADR-0002's documented introspection-flag convention). Picking bare-string sets a precedent that future flags will inherit. Spec must pick explicitly and justify against ADR-0002.

## Pivot (post-clarify): `--print-python` retired in favor of subcommand refactor

User-driven pivot during Research Exit Gate. The `cortex --print-python` proposal was retired after measurement and reframing surfaced that:

1. The hook currently spawns Python **three times** per SessionStart (preflight import-check, batch phase-detection, metrics regen). Adding `cortex --print-python` would add a **fourth** spawn (~400ms cold-start), nearly doubling SessionStart cost to ~700ms in the success case.
2. The simplest fix paying zero added cost — graceful-degrade — sacrifices lifecycle context in the broken topology, which the user does not want.
3. The shape that actually wins on latency is the subcommand refactor we originally deferred: collapse the three Python spawns into one `cortex hooks scan-lifecycle` invocation. **Faster than today** (~150-200ms vs ~300ms current), obviates the install-topology bug entirely (cortex shim runs in the right venv by construction; no interpreter discovery needed), and shrinks the 460-line bash hook to a ~5-10 line shell wrapper.

The original deferral was based on incomplete analysis. The subcommand refactor is structurally better AND faster — not a "cleanliness win for later" but the correct fix.

## Subcommand refactor — viability checks

Three pre-commit checks were run against the candidate shape `bash one-liner → cortex hooks scan-lifecycle`. All clean enough to proceed.

1. **First-run cortex setup**: `cortex_command/__init__.py:1-9` documents that the install-in-flight guard (R28) was demoted to the upgrade dispatch path only. `import cortex_command` and read-only subcommands are safe to execute mid-session. No surprise first-run cost.

2. **Hook output consumers**: one important structural consumer at `tests/test_lifecycle_phase_parity.py:540-541` reads `hookSpecificOutput.additionalContext` and compares output equivalence against the statusline's parallel phase-detection. The model consumes the additionalContext text via prompt injection but doesn't pattern-match phrasings. Golden-file tests for the Python port are required to satisfy the parity test.

3. **CLI version pin**: `plugins/cortex-overnight/server.py:103-106` defines `CLI_PIN = ("v2.1.2", "2.0")` and a `MCP_REQUIRED_CLI_VERSION` floor-check. The mechanism gates the MCP server path with structured error messaging, but does NOT gate the SessionStart hook. Therefore the bash one-liner must include its own fallback for the version-skew window where the plugin has updated but the user hasn't bumped the CLI.

Held for Spec (substantive design choices, not user-decision-gated):

- **Q-A — bash one-liner fallback on unknown-subcommand**: when the plugin updates but the CLI hasn't, `cortex hooks scan-lifecycle` returns argparse exit 2 ("unknown subcommand"). Spec should pin the bash shape so this case silent-degrades:
  ```bash
  command -v cortex >/dev/null || exit 0
  input=$(cat)
  cwd=$(printf '%s' "$input" | jq -r '.cwd // empty')
  [[ -d "$cwd/cortex/lifecycle" ]] || exit 0
  printf '%s' "$input" | cortex hooks scan-lifecycle 2>/dev/null
  rc=$?
  # argparse exit 2 = unknown-subcommand on old CLI; silent-degrade
  [[ $rc -eq 2 ]] && exit 0
  exit $rc
  ```
  Other rc codes propagate. Spec to confirm exact bash shape and the argparse exit-code semantic across cortex CLI versions.

- **Q-B — bin/cortex-* consumer scope**: Solution Horizon names four other callsites with the same bare-`python3 -c "import cortex_command"` pattern (bin/cortex-backlog-ready, bin/cortex-morning-review-complete-session, embedded snippets in skills/critical-review/references/residue-write.md and skills/lifecycle/references/implement.md). Once the hook moves to a Python subcommand, those callsites still exist and still suffer the install-topology bug for their own reasons. **Spec should evaluate**: include conversions in this ticket (small mechanical patch surface), or defer with R14 wrapper-cleanup track as the named blocker. Recommend deferral with named blocker — the hook is the SessionStart hot path users hit on every session; the other callsites are command-line tools users invoke explicitly, where the install-topology error message is more discoverable and less surprising.

- **Q-C — newly-reachable code paths in uv-tool topology**: today the hook exits at the preflight gate in uv-tool topology. After the refactor, the full hook logic (session-migration, batch phase-detection, metrics regen) runs in that topology for the first time. Spec should require a uv-tool-topology smoke-test acceptance criterion — manual or scripted run of the new hook against a real uv-tool install, exercising all branches.

- **Q-D — CLI_PIN bump**: this work adds a `hooks scan-lifecycle` subcommand to the CLI; plugins/cortex-overnight requires that subcommand. Spec should require `CLI_PIN` bump in `plugins/cortex-overnight/server.py:106` as part of this work, with the new CLI tag pinned to the version that ships the subcommand. Plus a regression test that the pin-check fires when the floor is violated.

- **Q-E — golden-file equivalence tests**: parity test at `tests/test_lifecycle_phase_parity.py:540-541` is a structural consumer. Spec must require golden-file tests comparing bash-hook output (pre-refactor, captured) vs Python-subcommand output (post-refactor) on a matrix of fixture inputs covering all branches (no lifecycle dir, single feature, multiple features, /clear migration, Morning Review active, etc.). Without these, the refactor can land green while silently regressing the parity test.

- **Q-F — session-migration test cases**: the bash hook's `.session` / `.session-owner` chain at lines 38-65 has subtle file-mtime semantics. Spec must require table-driven tests for the Python port covering each migration branch (fresh session, post-clear with no migration done, post-clear with migration done, multi-clear chain), independent of the bash test scaffolding (which is the known-fragile backlog #170 issue).
