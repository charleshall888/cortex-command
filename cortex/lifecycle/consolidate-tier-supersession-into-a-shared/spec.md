# Specification: Consolidate tier supersession into a shared pure-events reducer

## Problem Statement
`extract_feature_metrics` (`cortex_command/pipeline/metrics.py`) folds tier supersession with a hand-written loop that re-implements the canonical rule in `reduce_lifecycle_state` (`cortex_command/common.py`). The two were deliberately aligned in commit `3deac51b` and agree on 100% of real historical data (audit: 0 out-of-vocab values across 279 logs / 340 vocab-bearing rows), but agreement is held only by-test — and the metrics fold is, in fact, **absent from the systematic parity harness**, so there is no automated guard against future drift. This work single-sources the **supersession rule** (lifecycle_start seeds; complexity_override supersedes via `.to`; vocab gating) into one pure reducer both the file-based readers and the in-memory metrics path project from — so the *rule* agrees by construction.

**Scope precision (what consolidation does and does NOT unify):** the shared core unifies the supersession rule and its vocabulary gate only. Two concerns deliberately remain caller-local and are NOT claimed to be unified: (1) `initial_tier` first-seed capture (a single-consumer forensic projection — see R6/Non-Requirements), and (2) the *intake* (file decode + JSONL parse). R8 hardens the metrics intake against a non-UTF-8 crash not to "unify the parse" but because research showed the intake divergence (read_tier degrades gracefully, parse_events raises an uncaught `UnicodeDecodeError` that crashes `extract_all_feature_metrics` for every feature) is **more severe than any fold divergence** — leaving it would let the two paths still disagree (graceful vs crash) on byte-corrupt input, hollowing the consolidation's value. Beneficiaries: maintainers (one rule + one vocab gate to change, structural drift-resistance) and the overnight metrics pipeline (crash-resistance on corrupt logs). Standalone value is low (the rule has near-zero churn); the durable value is by-construction unity of the supersession rule for every tier reader including the in-memory case.

## Phases
- **Phase 1: Shared pure reducer** — extract a pure `reduce_lifecycle_events` from `reduce_lifecycle_state`, have the Path reader delegate, preserving every existing consumer's behavior byte-for-byte.
- **Phase 2: Metrics delegation + intake hardening + parity coverage** — point `extract_feature_metrics` at the shared core, keep `initial_tier` local, harden the `parse_events` intake, and add the metrics fold to the systematic parity coverage.

## Requirements

1. **Pure event-fold reducer exists.** A pure `reduce_lifecycle_events(records)` is added to `cortex_command/common.py`. It performs the supersession fold + per-value vocabulary gating over already-parsed records and does **no** file I/O, no `json.loads`, and no line-number tracking. It returns the bare accumulated `state` (at most `{"criticality","tier"}`, criticality-then-tier insertion order) plus the positions of records it rejected for an out-of-vocab value — it does **not** return or construct a `LifecycleStateReduction`. Acceptance: `grep -c "def reduce_lifecycle_events" cortex_command/common.py` = 1; `python3 -m pytest tests/test_reduce_lifecycle_state.py -q` exits 0. **Phase**: Shared pure reducer

2. **Path reader delegates with byte-for-byte preserved behavior.** `reduce_lifecycle_state(events_path: Path) -> LifecycleStateReduction` keeps its exact signature and tolerant-read contract (never raises, never `None`, `errors="replace"` decode), parses JSONL + tracks 1-based line numbers itself, delegates the fold to `reduce_lifecycle_events`, and produces `skipped_lines` as the **ascending-sorted union** of parse-failure line numbers and vocab-rejection line numbers — byte-for-byte identical to today's single-pass output, including ordering when the two kinds interleave. `corrupted` and the `LifecycleStateReduction` wrapper remain Path-shell concerns. Acceptance: every existing test in `tests/test_reduce_lifecycle_state.py` passes unchanged — torn-line lineno (`skipped_lines == (2,)`, :57/:66), per-value mixed-line rejection (:84), corruption suite (:133–199), missing-file equality (:100); PLUS a **new interleaving test** (R4) pins exact ordering. `python3 -m pytest tests/test_reduce_lifecycle_state.py -q` exits 0. **Phase**: Shared pure reducer

3. **`LifecycleStateReduction` is NOT grown.** The NamedTuple retains exactly its two declared fields (`state`, `skipped_lines`); no `initial_tier` (or other) field is added. Acceptance: a structural field-set assertion `LifecycleStateReduction._fields == ("state", "skipped_lines")` (this detects a *defaulted* added field, which the `:100` equality assertion does NOT — verified: `Three(state={}, skipped_lines=()) == Three(state={}, skipped_lines=())` is `True` when the third field defaults, so equality alone is not a growth guard); `python3 -m pytest tests/test_reduce_lifecycle_state.py -q` exits 0. **Phase**: Shared pure reducer

4. **Pure-core unit coverage added.** `tests/test_reduce_lifecycle_state.py` gains thin tests exercising `reduce_lifecycle_events` directly: empty input → empty state + no rejections; seed + complexity_override + criticality_override → both axes in insertion order; out-of-vocab value → dropped from state + its position reported; non-dict record → silent no-op (not reported); a second `lifecycle_start` re-seeds `state["tier"]`; and a **mixed parse-failure + vocab-rejection interleaving** case through the Path reader — e.g. a vocab-rejected value on line 1 and a torn-JSON line 2 — asserting `skipped_lines == (1, 2)` (ascending order, not category-grouped). ## Priority (MoSCoW)
The requirements are tiered against the two viable approaches research surfaced — **B** (consolidate by construction, the chosen scope) and **C** (the test-only floor that leaves `common.py` untouched). This makes the §4 downsize concrete.
- **Must (Approach B core, "agree by construction"):** R1, R2, R3, R5, R6, R10 — the pure-reducer extraction, delegation, and metrics consumption. Dropping any of these abandons the consolidation.
- **Must (locks behavior the consolidation introduces):** R7 (vocab-drop lock) and R8 (intake hardening — user explicitly folded in). R4 (pure-core unit coverage) is must-have for the new entry point.
- **Should (highest standalone value; also the Approach-C floor):** R9 + R9a — closing the parity-harness gap with an independent oracle and dispositioning the degraded `:1749` assertion. This is the one deliverable with genuine value even if the consolidation (R1–R6) were dropped entirely; under B it is the regression guard.
- **Downsize to Approach C** would keep only R7 + R9 + R9a (the independent-oracle parity coverage and intake-divergence case), drop R1–R6, and leave `common.py` untouched — sacrificing by-construction unity for a test-only guard, justified by the audit's zero-real-data-risk finding.

## Non-Requirements
- Does **NOT** grow `LifecycleStateReduction` with an `initial_tier` field (A-Option-2 explicitly rejected: one consumer, taxes three readers that don't want it, makes the `:100` equality assertion a `TypeError` footgun, and conflates a first-seed-latch fold with the last-writer accumulator).
- Does **NOT** move `initial_tier` into the shared core — it is a single-consumer forensic projection and stays local to metrics.
- Does **NOT** unify the criticality axis into metrics (`extract_feature_metrics` does not read criticality).
- Does **NOT** change `reduce_lifecycle_state`'s tolerant-read contract, its `skipped_lines`/`corrupted` semantics, the criticality-then-tier insertion order, the `.to`-only supersession (no `.tier`/`.criticality` fallback), the non-dict-silent-skip rule, or the `lru_cache`/`__wrapped__` hooks for its existing consumers.
- Does **NOT** touch the `_DAYTIME_DISPATCH_FIELDS` historical-compat shim or the dispatch-pairing tier path (`start.get("complexity")` at metrics.py:450/485).
- Does **NOT** address the two `criticality_override` rows that carry their value in a non-standard `criticality` field instead of `to` (pre-existing latent drop on the criticality axis; both readers already read `.to` only, so no regression — out of scope, not on the tier axis).

## Edge Cases
- **Empty input**: `reduce_lifecycle_events([])` → empty state, no rejected positions; `reduce_lifecycle_state` on a missing file → `LifecycleStateReduction(state={}, skipped_lines=())`, no raise.
- **`lifecycle_start` re-seed**: a second `lifecycle_start` re-seeds `state["tier"]` (last writer wins); metrics' local `initial_tier` keeps the **first** in-vocab seed and is NOT overwritten.
- **Mixed valid + invalid on one line**: the valid value accumulates into state; the line is flagged once by the Path shell (`skipped_lines`) and the record's position reported by the core.
- **Non-dict JSON** (e.g. `42`, `"x"`, `[...]`): silent no-op in the core — not a rejection, not a skip, not a corruption signal.
- **Criticality-only `lifecycle_start`** (no `tier`): does not set `initial_tier`; a later `complexity_override` does NOT set `initial_tier` (only `lifecycle_start` seeds it).
- **Out-of-vocab tier** (e.g. `"trivial"`): dropped from both final `tier` and `initial_tier` (vocab-consistent); a no-op on 100% of real data per the audit.
- **Non-UTF-8 byte in a log**: `parse_events` decodes with `errors="replace"`; `extract_all_feature_metrics` completes for all features instead of crashing.

## Changes to Existing Behavior
- **MODIFIED**: `extract_feature_metrics` `tier`/`initial_tier` now reject out-of-vocab values (previously accepted any non-`None` string). No-op on all real data (audit); locked by R7. This makes the forensic metrics reader vocab-consistent with the gate readers.
- **MODIFIED**: `parse_events` decodes with `errors="replace"` (previously strict UTF-8, which raised `UnicodeDecodeError` and crashed `extract_all_feature_metrics` on a byte-corrupt log).
- **ADDED**: pure `reduce_lifecycle_events` entry point in `common.py`; the supersession rule is now single-sourced for the file-based readers and the in-memory metrics path.
- **ADDED**: `extract_feature_metrics` enters the systematic parity coverage.

## Technical Constraints
- **Core/shell boundary**: the pure core owns the fold + vocabulary gating only; the Path shell owns file I/O, JSONL parsing, 1-based line-number tracking, the `skipped_lines` merge, and the `LifecycleStateReduction`/`corrupted` wrapper; metrics calls the bare core and ignores the rejected-position channel. (Exact signature — `Iterable[dict]` returning rejected indices vs `Iterable[(lineno, dict)]` returning linenos — is an implementation choice, provided the Path shell reproduces today's exact `skipped_lines`.)
- **`common.py` is lifecycle-protected**; this work runs under the lifecycle. `metrics.py` is not protected.
- **Sequential dispatch, not worktree**: `just test` runs the editable install, so a worktree would verify stale code. Phase 1's `common.py` edits must be complete before Phase 2 verification.
- **Commit hygiene**: on `main` with possible concurrent sessions, commit with explicit `git commit -- <pathspec>`.
- **No dual-source co-commit**: `cortex_command/` is not mirrored to `plugins/cortex-core/` (only `bin/hooks/skills` are).
- **No new event** is minted; no `bin/.events-registry.md` row is required.
- Preserve `#287` invariants: never raise / never `None`, per-value rejection, insertion order, `.to`-only supersession.
