# Research: Route the model-less agentic-search/gather fan-out dispatches to a Sonnet grunt tier

**Clarified intent:** Bring the dispatch sites that currently pass *no* model parameter — and therefore inherit the parent (Opus in interactive sessions) model — under the `cortex-resolve-model` matrix via a new role resolving to the `sonnet` tier, so interactive agentic-search/gather runs on Sonnet while judgment work stays on the stronger model. Tier `complex`, criticality `medium`.

**Prior art in-repo:** This topic was researched once before — `cortex/research/archive/subagent-model-routing/research.md` (DR-1, "Option D") chose to scatter literal `model: "sonnet"` strings into each skill's `Agent()` call and flagged its own risk as *"every new skill needs to remember model selection. Risk of drift."* Routing through `cortex-resolve-model` is the durable version of that rejected approach — it keeps the model name owned in one place, which is the entire point of the single-matrix-owner architecture.

## Web Research

**The "expensive orchestrator + cheap worker" pattern is Anthropic's own, validated design.** [Anthropic's multi-agent research system](https://www.anthropic.com/engineering/multi-agent-research-system) is precisely this: **Opus lead orchestrator + Sonnet parallel sub-agents**, which beat single-agent Opus by **90.2%** on their internal eval — the gain attributed to parallel breadth across separate context windows, not a smarter single brain. Their stated rationale matches `sdk.md:149` exactly: subagents get self-contained bounded tasks, search independently, return condensed summaries; *individual-agent quality need not be maximal because the orchestrator synthesizes*. Cost framing: multi-agent ≈ 15× chat tokens, so they gate it to high-value parallelizable work (research = good fit; coding = poor fit).

**Claude Code natively supports per-subagent model selection** ([model-config docs](https://code.claude.com/docs/en/model-config)): the Agent/Task tool takes a `model` parameter (`opus`/`sonnet`/`haiku`/`fable`/`inherit`); omitting it inherits the parent. Anthropic's own **Explore agent runs on Haiku** for read-only searches, while Plan/general-purpose **inherit** — confirming "cheap model for search/gather" is already shipped behavior. A blocked/unavailable override **falls back to the inherited model rather than failing** — a relevant safety property for the fail-handling decision.

**Sonnet-as-worker is well-supported for breadth-first gather, with one caution.** Sonnet matches/beats Opus on production-shaped work, but Opus retains a real ~18-point edge on *deep, multi-hop* agentic search (BrowseComp: Opus ≫ Sonnet). The research fan-out is breadth-first / shallow-per-agent / orchestrator-synthesizes — squarely in Sonnet's strong regime. **Documented pitfalls to design around:** (1) the *quality cliff* from down-routing a step that looked like "grunt work" but was secretly deep synthesis — no clean threshold exists; mitigate by keeping the orchestrator strong (it repairs weak worker output) and giving workers narrow well-specified tasks; (2) *prompt-cache fragmentation* — caches are per-model, and dynamic multi-model handoffs can forfeit the cache discount.

## Requirements & Constraints

- **Single-matrix-owner is the load-bearing constraint** — three independent assertions pin `cortex-resolve-model` as the sole owner of lifecycle model choice: `docs/internals/sdk.md:144` (*"do not restate its table here; the golden-anchor test is the parity anchor"*), `skills/lifecycle/references/criticality-matrix.md:24` (*"run it at each dispatch site rather than reading a model from this table"*), and `resolve_model_cli.py:5` (*"single executable source"*). **Consequence: the grunt tier must be a role inside the verb, not a hardcoded `"sonnet"` in a skill.**
- **The rationale already exists in the owning doc.** `sdk.md:149` *"Parallel agents → sonnet… the orchestrator synthesizes the outputs, so individual-agent quality need not be maximal."* This change instantiates an already-documented principle; `sdk.md:152` *"Reviews follow criticality"* covers the judgment carve-out. The two lifecycle/pipeline matrices are *intentionally separate and unmergeable* (`sdk.md:142`; `resolve_model_cli.py:6-10` — the verb does not import `dispatch.py`). The grunt role belongs to **Path A (lifecycle), not Path B (pipeline)**.
- **L1 surface ratchet measures only `description`+`when_to_use` frontmatter** (`tests/test_l1_surface_ratchet.py`), and `research` sits at **379B == its 379B baseline (zero headroom)**. Body prose costs nothing; **frontmatter must not be touched.**
- **ADR-0009 propagation** (`CLAUDE.md:73-75`): the `cortex-resolve-model` call must run in the SKILL.md *body* and the *resolved string* propagate into the Agent dispatch; raw `${CLAUDE_SKILL_DIR}`/bare paths must not enter subagent prompt blocks (lint `cortex-check-skill-path`).
- **MUST-escalation** (`CLAUDE.md:77-85`): new wiring prose must mirror the existing *soft* phrasing (*"halt and escalate rather than guessing"*); a MUST would require an evidence artifact + effort attempt.
- No dedicated ADR governs `cortex-resolve-model` today; ADR-0019 sets the precedent that editing a promoted verb is the sanctioned home for dispatch-policy mechanism. A new/amended ADR for the role is spec/plan territory.

## Tradeoffs & Alternatives

**Decision 1 — routing unit: per-angle (binary split), not per-skill.** Route the gather angles (Codebase, Web, Requirements, + gather-shaped chosen angles) to the new Sonnet role; the **Adversarial** angle omits `model` and inherits the Opus parent. Rationale: adversarial is the fan-out's *error-correction layer* (`fanout.md:24` — dispatched last over a summary "to cut error amplification… preserves the centralized-synthesis error-correction layer"); downgrading the one agent whose job is to catch what the cheap gatherers missed is a false economy, and it's nearly free to leave on Opus (runs only at high/crit, works over a brief summary, inherits the already-Opus parent → no role/verb call needed).

**Decision 2 — role shape: a new criticality-independent constant, modeled on `synthesizer`.** Add one entry to `_CRITICALITY_INDEPENDENT` (`resolve_model_cli.py:72`) resolving to `"sonnet"`. The module's own comment argues constants over a 4-identical-cell row (`:65-72`: a constant "cannot drift", "removes the empty low/medium/high cells a future editor might be tempted to 'fill in'", and "lets the standalone… dispatch sites resolve with no lifecycle-state read"). All three apply: gather is Sonnet at every criticality (criticality scales *angle count* and triggers adversarial, not the per-gatherer model), and research has a standalone mode with no lifecycle session. Reusing `competing-plan` is wrong (it's critical-gated). **Lowest test churn too** (see below).

**Decision 3 — failure handling: recommend halt-on-nonzero for consistency; degrade-loud is the documented alternative.** See OQ3 below — this is the one genuine fork, surfaced for spec/critical-review.

**Decision 4 — opt-out: none.** An opt-out re-imports the exact cost the feature removes (lets a user push gather back onto Opus), adds standing config surface against "prescribe What/Why not How", and fails the solution-horizon YAGNI test (no named follow-up, no second site, no constraint sidestepped). The existing escape hatches already cover the cases: a *global* change is a one-line edit to the owned constant; *per-run depth* is governed by criticality; *robustness* is covered by the Decision-3 fallback. Revisit only on a real events.log/transcript artifact per the escalation discipline.

## Dispatch Mechanics & Failure Modes

- **Wiring is the standard two-ingredient pattern**: a shell line `model=$(cortex-resolve-model --role <gather-role>)` plus prose binding `$model` to the Agent dispatch's `model:` param. No skill writes a literal `Agent(model:)` example — all sites express it in prose (`implement.md:174-178` etc.). The Agent tool's `model` enum (`sonnet`/`opus`/`haiku`/`fable`) consumes the bare alias directly — no translation step.
- **Failure philosophies differ by path, by design.** Interactive verb = fail-loud + "halt and escalate" (every sibling site). Pipeline `resolve_model()` = raises, but wrapped in an escalation ladder (`dispatch.py:209-215,661`; `retry.py:286`) with classify→recover (retry/escalate/pause). The research fan-out is Path A (interactive) — there is no retry/escalation infra to lean on.
- **The constant role nearly eliminates the failure surface.** A criticality-independent constant resolves unconditionally given the correct role name; the only failures are a typo'd role (static — caught by the golden test + contract checker pre-commit) or a missing/broken `cortex-resolve-model` binary (a global install fault, not specific to research). So "fail-loud vs fallback" is a near-dead path either way.
- **Pin-vs-inherit observability (OQ4c) is benign.** *Effort:* the interactive Agent tool has **no effort param** (effort is Path-B only, `dispatch.py:798`) — zero effort interaction, one whole branch of the worry gone. *Version:* identical bare-alias resolution either way (no skew). *Cost:* the only first-order observable — pinning `sonnet` when the parent is Opus is the entire point; when the parent is already Sonnet it's cost-neutral. *Cache:* not encoded in-repo (platform behavior); research agents are fresh independent sub-agents with their own context, so there is little parent-cache to reuse regardless — the delta is second-order.
- **Per-run override precedent:** strong on Path B (`dispatch_task(model_override=…)`, `dispatch.py:569,599-602`; callers in retry/recovery), **none** on Path A. No `--model` CLI flag / env var exists. An interactive opt-out would be net-new surface — reinforcing Decision 4.

## Open Questions

1. **[RESOLVED — confirm at spec/critical-review] Where does the `Tradeoffs` research-angle route?** The Codebase agent classed it as *judgment* (keep on the strong model); the Tradeoffs agent argues it's safe on Sonnet (it's folded into the Opus synthesis and re-challenged by the adversarial pass). **Recommendation:** operative rule = *the Adversarial angle inherits Opus; every other angle (core gather + all chosen angles, including Tradeoffs) routes to the Sonnet role.* This keeps the wiring a clean binary split rather than a per-angle model map, and Tradeoffs is double-checked downstream. Flagged for the critical-review gate to pressure-test.

2. **[RESOLVED] Flat `sonnet` constant vs tier-keyed `haiku→sonnet` row?** `sdk.md:149` ("parallel agents → sonnet") and `sdk.md:150` ("exploration → haiku unless high/critical") are not actually in conflict for this feature: `:149` governs **parallel multi-agent fan-out** (this change) → sonnet; `:150` governs **single-agent exploration** (the pipeline's trivial cell, Claude Code's Explore-on-Haiku) → a different context. The user's framing was explicitly "Sonnet grunt." → **flat `sonnet` constant.** *Deferred as out-of-scope future refinement:* routing the pure read-only Codebase-search core agent to Haiku for an even cheaper tier.

3. **[RESOLVED with a recommendation — the spec's central fork] Halt-on-nonzero vs degrade-loud-to-inherit on resolver failure.** Because the role is a constant, runtime failure is near-impossible (only a static mis-wire caught by tests, or a broken global install), so the practical behavior difference is negligible. **Recommendation: halt-on-nonzero, matching every sibling resolve site** — simplest, most consistent, and avoids inventing a "loud degradation note" mechanism in an interactive skill that may have no events.log (standalone mode). **Documented alternative (rejected, but recorded for the gate):** degrade-loud-to-inherit-parent + emit a degradation event (mirroring `dispatch.py:931-948` `dispatch_effort_ignored`), on the argument that the fallback (Opus) is a quality *upgrade*, so breaking interactive `/research` over an infra glitch is all cost and no correctness benefit. The spec §4 / critical-review should make the final call; if degrade-loud is chosen, the "how to be loud without a guaranteed events.log" mechanism becomes a sub-decision.

4. **[RESOLVED] Opt-out / override?** None — see Tradeoffs Decision 4 (re-imports the removed cost; fails the solution-horizon YAGNI test; existing escape hatches suffice).

5. **[RESOLVED] Do the currently-inheriting judgment dispatches (research-Adversarial, critical-review reviewers `:58`/`:66`, clarify-critic) get explicitly pinned?** No — they **stay inheriting the parent** (the conservative, in-scope reading). Explicitly pinning critical-review's reviewer wave to a criticality role is a separate, larger change and is out of scope here.
