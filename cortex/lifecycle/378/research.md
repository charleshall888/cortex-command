# Research: 378 — 374 follow-ups (lifecycle_slug frontmatter coercion, refine spec-approve routing residue, CLI_PIN dedup)

Three 374-adjacent residues. This research reconciled several contradictions between agents against the live tree; **verified** claims below were checked empirically, not asserted.

## Codebase Analysis

### Item 1 — numeric `lifecycle_slug` / stale `lifecycle_phase`

**Writers (all emit unquoted; none uses `yaml.safe_dump`):**
- `cortex_command/backlog/update_item.py:55-86` `_set_frontmatter_value` — a per-key line editor, `f"{key}: {value}\n"` with a bare `str(value)`, for **every** scalar field (`status`, `priority`, `lifecycle_slug`, `parent`, `spec`, …). This is the writer that produced `lifecycle_slug: 378` this session (via `enter.py:175` → `start_sync.py:111` → `cortex-update-item --lifecycle-slug`).
- `cortex_command/backlog/create_index.py::_render` — f-string `feature: {feature}` into `cortex/lifecycle/{feature}/index.md` (a **different key**, `feature:`, from `lifecycle_slug:`). Deliberately non-YAML (docstring: PyYAML can't reproduce the ordered-block/inline-array/bare-null shape).
- `cortex_command/overnight/report.py:460` — `f"lifecycle_slug: {x}"` in the overnight follow-up-ticket writer.

**Existing quote-on-write precedent (house pattern):** `cortex_command/backlog/create_item.py` already double-quotes numeric-look-alike scalars at write time — `schema_version: "1"`, `parent: "{parent}"` — and has a dedicated `_yaml_safe_title_value()` helper for `title`. This precedent is **title/field-scoped**, not universal.

**Readers — only `yaml.safe_load`-based ones are exposed:**
- `cortex_command/backlog/resolve_item.py` — `_parse_frontmatter` (yaml) → int; `_resolve_lifecycle_slug` (line 133-136) returns the raw int (type-hinted `-> str`, unenforced); `_resolve_lifecycle_slug_frontmatter` (line 218-233) guards with `isinstance(slug_val, str)` so it *silently fails to match* rather than crashing — an inconsistency between the two sites.
- `cortex_command/lifecycle/resolve.py:181-182` — `_resolve_backlog()` calls `resolve_item.resolve()` **in-process**, so `backlog.get("lifecycle_slug")` may be an int; `(lifecycle_base / slug).is_dir()` then raises **`TypeError`** (Path / int). This is a **live crash in the served next/advance resolve path** (#374's own loop), reachable when a numeric-ID token resolves against an item whose `lifecycle_slug` is an unquoted int and the lifecycle dir does not yet exist.
- **`generate_index.py:164` does NOT crash** (VERIFIED — ran clean, exit 0, with the non-terminal `378` present): its `_parse_frontmatter` (line 46-60) is a hand-rolled string-only splitter (`dict[str,str]`), so `.strip()` always operates on a string. `overnight/backlog.py`'s `_parse_frontmatter` is likewise string-only and safe. The ticket's "crash at generate_index.py:164" is **misdiagnosed**.

**Semantic root cause (upstream of the quoting):** `cortex_command/lifecycle/parse_args.py:56` `_VALID_SLUG = re.compile(r"^[a-z0-9]+(-[a-z0-9]+)*$")` accepts a bare all-digit token (`"378"`) as a valid slug; `resolve.py`'s numeric→kebab remap fires **only when the lifecycle dir already exists** (resume), so a brand-new feature's raw numeric ID propagates unremapped into the writers. `skills/lifecycle/SKILL.md` documents `feature` as a *kebab-case slug* — a bare digit is never semantically valid.

## Web Research

- **The Norway problem / YAML 1.1 implicit typing** (PyYAML): unquoted plain scalars are pattern-matched to int/float/bool/null before falling back to string — so `lifecycle_slug: 378` becomes an `int`. Canonical writeups: StrictYAML "why implicit typing removed", tinita "To Quote or not to Quote" (rule: quote any plain scalar that would resolve to bool/int/float/null).
- **Write-time quoting** is the durable boundary fix. PyYAML has no per-value quoting API; the standard pattern is a `str`-subclass + representer, or (ruamel) `SingleQuotedScalarString`. For a per-key line editor (this repo's shape), the equivalent is quoting the single scalar with proper escaping — **not** a `safe_dump` block round-trip (which reorders keys and drops comments).
- **"Parse, don't validate"** (Alexis King / DevIQ): coerce to a well-typed value once at the boundary; re-validating at every consumer is the anti-pattern and is exactly how the parser asymmetry arose. Applied here → fix the writer (single producer for all future reads); optionally add one defensive coercion at the canonical read accessor as belt-and-suspenders for files already on disk.
- **Single-sourcing a duplicated version constant** (Python Packaging guide): leaf zero-dependency file, regex-read without importing, `importlib.metadata`, or VCS-tag-derived. Explicit caveat: import-based single-sourcing "will fail if `__init__.py` imports install_requires deps not yet installed" — the exact bootstrap constraint here. When a shared import is genuinely blocked, a **parity/drift test** is the accepted fallback.

## Write-Path & Backfill Analysis

- **Backfill set = exactly 4 files (VERIFIED by grep of the whole tree):** `cortex/backlog/374-…​.md:15` (`lifecycle_slug: 374`), `cortex/backlog/378-…​.md:13` (`lifecycle_slug: 378`), `cortex/lifecycle/374/index.md:2` (`feature: 374`), `cortex/lifecycle/378/index.md:2` (`feature: 378`). Of 160 backlog files carrying `lifecycle_slug:`, only these two are numeric; the rest are kebab strings or `null`.
- **Quoting `"374"`/`"378"` is the correct backfill target — not kebab-remap, not null.** The on-disk dirs are literally `cortex/lifecycle/374/` and `.../378/`; `spec:` points inside them; the failing test resolves the slug against **directory existence** (`(…/lifecycle/374).is_dir()`). Remapping to a kebab slug without physically moving the dir + rewriting `spec:` would flip the entry from `resolved` to `broken` and fail the test *harder*; null would drop the binding. Quoting is minimal and correct. (Caveat: it greens the test but leaves a semantically non-kebab, now-typed slug — the semantic fix is the `_VALID_SLUG`/remap layer, tracked separately.)
- **Stale `lifecycle_phase` is a DIFFERENT bug class** — a write-omission, not a quoting bug (values like `research`/`specify` are always alphabetic → never mistyped). Only `start_sync.py:105` (sets `research` at entry) and `wontfix_cli.py:143` (`wontfix`) ever write it; `spec_approve.py`, `finalize.py`, `close_tickets.py` (which set `status: complete`) never touch it. **VERIFIED: 60 `status: complete` items still carry `lifecycle_phase: research`.** Systemic and pre-existing; no numeric/malformed values, so it is *not* part of the item-1 backfill.

## Tradeoffs & Alternatives

### Decision A — item 1 contract
- **A1 quote-on-write + backfill** — fix the writer(s) to emit YAML-safe scalars, then backfill the 4 files. Closes the defect at its single origin; protects `parent`/`spec`/any future scalar; backfill is cheap.
- **A2 coerce-at-consumers** — rejected as the *sole* fix: it is 1-of-3 parsers (not "two consumers"), needs two internal sites within `resolve_item.py` alone, leaves the on-disk corpus permanently ambiguous to any correct YAML reader, and (VERIFIED) does **not** green the test — the test parses committed files' raw YAML directly.
- **A3 shared writer-side helper** — generalize A1 into one field/type-aware YAML-safe scalar helper reused by `update_item.py`, `create_index.py` (the `feature:` key), and `overnight/report.py`, modeled on `_yaml_safe_title_value()`.
- **Recommended: A1 as A3's shape (shared helper) + a defensive `str()`/`isinstance` coercion at the yaml readers** (`resolve_item.py:136/218`, `resolve.py:181`) as defense-in-depth. Both the writer fix **and** the 4-file backfill are required (independent: `update_item` line-edits per-key and never rewrites `lifecycle_slug`).

### Decision B — item 3
- **B1 consolidate to a shared import** — rejected: blocked by the independent-plugin-install constraint (neither plugin can rely on the other's dir) *and* the stdlib-only/wheel-not-importable constraint. A shared module has nowhere to live that both installs can import.
- **B2 extend the existing overnight-only machinery to also cover cortex-core** — (a) immediate: bump `install_core.py` to `v2.35.0`; (b) durable: a second `cortex-rewrite-cli-pin` target + `git add` in `auto-release.yml`, plus extending `release.yml`'s drift lint and `tests/test_release_artifact_invariants.py`, plus a pre-commit parity check (extend Phase 1.97 or a new `bin/cortex-check-cli-pin-parity` in the `--staged/--audit` two-mode shape).
- **Recommended: B2.** The install-state parity precedent is structurally similar but not fully analogous (CLI_PIN is a mutable release-bumped value, cannot be wheel-canonical). Its *shape* (duplicate-by-necessity + automated parity) transfers.

## Adversarial Review

- **Item 3 sequencing hazard (VERIFIED):** both plugins run `uv tool install --reinstall …@<their-pin>` at session start → with divergent pins the CLI **ping-pongs** between v2.34.6 and v2.35.0 each session (not cosmetic). v2.35.0 (2026-07-10) and v2.34.6 (2026-07-07) are both real tags; **v2.35.0 is the correct target**. **HEAD is `v2.35.0-76-g…` — 76 commits past the last tag with no v2.36** → the auto-release pipeline appears **dormant** (possible PAT expiry; a weekly PAT-expiry probe exists). Two consequences: (1) B2(b) cannot be validated end-to-end until the pipeline runs; (2) if the pipeline *is* healthy, B2(a) alone is self-defeating — pushing the fix cuts v2.36.0 and rewrites overnight's pin, re-stranding core — so **B2(a)+B2(b) must land in the same push**.
- **Item 2 reframe:** ADR-0024 keeps the typed subcommands "not retired" and callable, so `specify.md` commanding `spec-approve` is an *incomplete R17*, not a live bug. This opens a smaller option: carve the call as an explicit documented R17 exception, changing nothing in advance.py. And the **risk ranking is inverted** — advance.py is *not* gated, whereas the specify.md/SKILL.md reroute *is* gated + dual-source-mirrored (so sequence it away from any concurrent skill-editing session — see the "dual-source drift blocks all commits" hazard).
- **Item 1 — blanket quoting is dangerous (VERIFIED against `374-…​.md`):** frontmatter carries legitimately non-string scalars — `created`/`updated` → `date`, `blocked-by` → `int`, and 15+ `lifecycle_slug: null` → `None`. A "quote every scalar" helper would (1) turn `null` into the string `"null"`, breaking the test's None sentinel and the resolver's null-fallback; (2) flip dates to strings; (3) churn-quote `updated` corpus-wide (`update_item` rewrites it every call). The helper **must be field/type-aware** (quote only intended-string values that would mistype) and **escape** embedded YAML metacharacters, not merely wrap. Keep it a single-scalar line editor — never a `safe_dump` block round-trip.
