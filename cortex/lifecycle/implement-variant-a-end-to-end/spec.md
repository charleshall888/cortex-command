# Specification: implement-variant-a-end-to-end

## Problem Statement

The lifecycle implement-phase preflight option 2 ("Implement on feature branch with worktree") creates an `interactive/{slug}` worktree (shipped in #239) and acquires a per-feature lock (shipped in #241), but does not wire the active Claude Code session into the worktree, does not route orchestrator-session events.log writes to the worktree, and does not open a worktree-aware PR at the Complete phase. Without this work, lifecycle metadata that the orchestrator session writes (events.log entries, backlog frontmatter updates the orchestrator drives) lands in the home repo as untracked files — a metadata-layer analog of the code-layer drift fixed by prior tickets #126 and #130 (the precedent is the routing pattern, not the layer). Beneficiaries: lifecycle users who select option 2 see a coherent worktree state at completion time (their CWD, statusline, the merged sub-agent feature commits, the orchestrator-written events.log entries, and the PR all resolve to `interactive/{slug}`), and the post-merge re-invocation flow has the artifacts it needs in the right tree. Note: sub-agent build-time state is transient and orthogonal — see R7's note on §2(e) merge-back.

## Phases

- **Phase 1: Writer-site inventory & characterization** — enumerate the orchestrator-session-reachable writer surface by call-graph from `skills/lifecycle/references/implement.md` and `complete.md`, and add characterization tests that pin current (pre-refactor) behavior so Phase 3's refactor can be verified non-regressive.
- **Phase 2: Sandbox infrastructure** — extend `cortex init` so the worktree base path is registered in both `sandbox.filesystem.allowWrite` and `additionalDirectories` of `~/.claude/settings.local.json`, so a Variant-A session's first Write tool call inside the worktree does not sandbox-deny.
- **Phase 3: Path-resolution refactor** — add a new `_resolve_user_project_root_from_cwd()` helper and per-callsite `worktree_root: Path | None = None` parameter for the narrow set of orchestrator-session writer sites identified in Phase 1. Leave the existing `_resolve_user_project_root()` semantics untouched.
- **Phase 4: Lifecycle integration** — wire the `cd` handoff into `implement.md` §1a and the `cd-in-then-out` pattern around `/cortex-core:pr` in `complete.md` Step 3, with the explicit scope statement (Variant A's `cd` affects only orchestrator-session Bash; sub-agent dispatch is unaffected).

## Requirements

> **MoSCoW classification**: R1-R9 are must-have — each is load-bearing for the feature working at all (R3 depends on R2's helper; R7-R8 lifecycle integration depends on R6 sandbox infra and R3 refactor; R9 detection is the precondition for R8's routing logic). R10 (Slice 7 commit shape) is should-have — the work could land as a single bundled PR, but the four-PR sequence is preferred for reviewability, bisect-ability, and independent revertability. There are no won't-do items inside this spec — won't-do is captured in `## Non-Requirements` below.

1. **(Must)** **No regression to `_resolve_user_project_root()` semantics**: The existing function at `cortex_command/common.py:55-103` retains its env-first contract — env value is returned verbatim when set; walk-up happens only when env is unset. Acceptance: `cortex_command/tests/test_common.py:55-56` and `tests/test_common_utils.py:486` remain green without modification; `just test` exits 0 after Phase 3 lands. **Phase**: Phase 3.

2. **(Must)** **New `_resolve_user_project_root_from_cwd()` helper**: a new module-level function in `cortex_command/common.py` performs walk-up from `Path.cwd().resolve()` ignoring `CORTEX_REPO_ROOT` entirely, with the same stop conditions as the existing function (`.git` file or directory, or presence of a `cortex/` directory). Raises `CortexProjectRootError` when no ancestor matches. Acceptance: `grep -c "def _resolve_user_project_root_from_cwd" cortex_command/common.py` = 1; new tests in `cortex_command/tests/test_common.py` verify (a) returns the worktree root when CWD is inside a worktree even with `CORTEX_REPO_ROOT` set to the main repo, (b) raises `CortexProjectRootError` from a non-cortex directory. **Phase**: Phase 3.

3. **(Must)** **Per-callsite `worktree_root` parameter**: each writer-site identified in Phase 1's inventory as orchestrator-session-reachable and currently CWD-pinned (e.g., `cortex_command/refine.py:117`, `cortex_command/critical_review.py:340-349,375-416,318-322`, `cortex_command/discovery.py:189-197`, `bin/cortex-complexity-escalator:265,296`) accepts a `worktree_root: Path | None = None` parameter; when present, absolute paths are resolved as `worktree_root / "cortex/lifecycle" / ...`; when absent, callers receive existing behavior. This mirrors the existing `lifecycle_base: Path = Path("cortex/lifecycle")` shape at `cortex_command/common.py:453,526` and the post-#130 `backlog_dir` parameter pattern. Acceptance: each refactored writer site has at least one test passing a non-None `worktree_root` that lands the write in that path; `just test` exits 0 after Phase 3 lands. **Phase**: Phase 3.

4. **(Must)** **Phase 1 inventory artifact**: Phase 1's first commit produces `cortex/lifecycle/implement-variant-a-end-to-end/writer-sites.md` enumerating each orchestrator-session-reachable writer site by file:line, with reachability rationale (which `implement.md`/`complete.md` step routes to it). Sites in `cortex_command/overnight/daytime_pipeline.py` are excluded (slated for #246 removal). Acceptance: file exists and lists at minimum the eight ticket-named sites plus any additional sites surfaced by call-graph audit; `wc -l cortex/lifecycle/implement-variant-a-end-to-end/writer-sites.md` > 20. **Phase**: Phase 1.

5. **(Must)** **Characterization tests for current behavior**: Phase 1's first commit adds `tests/test_variant_a_writer_sites_baseline.py` (or per-module additions to existing test files) that pin current pre-refactor behavior for each inventoried site. These tests pass on `main` *and* after Phase 3 lands (regression detection). Acceptance: `just test` exits 0 after Phase 1 lands; subsequent Phase 3 must not require modifying these baseline assertions, only adding new ones for the `worktree_root` parameter behavior. **Phase**: Phase 1.

6. **(Must)** **`cortex init` registers worktree base in sandbox**: `cortex init` Step 7 (currently registers `<repo>/cortex/` umbrella in `allowWrite`, per `cortex_command/init/handler.py:194-198`) is extended to additively register the worktree base path (`$TMPDIR/cortex-worktrees/` by default, or the value of `CORTEX_WORKTREE_ROOT` if set) in *both* `sandbox.filesystem.allowWrite` *and* `additionalDirectories` of `~/.claude/settings.local.json`. The registration is additive (no removal of existing entries), idempotent, and uses the existing `fcntl.flock`-guarded merge per ADR-0003. Acceptance: after `cortex init` runs in an isolated test fixture, the resulting `settings.local.json` contains the worktree base path in both arrays (verified by JSON-load + key inspection in `cortex_command/init/tests/`); re-running `cortex init` does not duplicate the entry. **Phase**: Phase 2.

7. **(Must)** **Implement-phase `cd` handoff**: `skills/lifecycle/references/implement.md` §1a (which currently creates the worktree and acquires the lock) is extended after worktree creation and lock acquisition to (a) capture `_origin_pwd=$(pwd)` for later restore, (b) issue `cd $(cortex-worktree-resolve interactive/{slug})` via a Bash tool call, (c) record the cd as an `interactive_worktree_entered` event in `cortex/lifecycle/{slug}/events.log` written via the new helper (path resolves to the worktree post-cd). The prose for §1a explicitly states that Variant A's `cd` affects only the orchestrator session's own Bash tool calls; sub-agent dispatch via `Agent(isolation: "worktree")` in §2 is unaffected (sub-agents remain rooted at `$TMPDIR/cortex-worktrees/{task-name}/`). **Sub-agent commit reconciliation**: §1a prose includes an explicit cross-reference to the existing §2(e) Worktree Integration step (`implement.md:151-161`), which after each task completion runs `git merge worktree/{task-name}` from the feature-branch CWD (which under Variant A is `interactive/{slug}` post-cd) and then `git worktree remove` for the sub-agent worktree. Under Variant A this means sub-agent commits land in `interactive/{slug}` as part of the existing dispatch flow — Variant A inherits this merge-back behavior unchanged. The §1a paragraph that begins "Variant A's cd affects only orchestrator-session Bash" ends with the cross-reference to §2(e). Acceptance: §1a contains a `cd $(cortex-worktree-resolve` Bash directive, a paragraph beginning "Variant A's cd affects only orchestrator-session Bash" whose final sentence references §2(e) Worktree Integration; `grep -c "interactive_worktree_entered" bin/.events-registry.md` = 1; the parity tests at `tests/test_lifecycle_kept_pauses_parity.py` continue to pass. **Phase**: Phase 4.

8. **(Must)** **Complete-phase `cd-in-then-out` around `/cortex-core:pr`**: `skills/lifecycle/references/complete.md` Step 3 (PR creation) is updated so that when the lifecycle detects an `interactive/{slug}` worktree (via reading `interactive.pid` AND corroborating with `git rev-parse --show-toplevel` against `pwd` — see Edge Cases), it (a) saves `_origin_pwd=$(pwd)`, (b) `cd`s into the worktree if not already there, (c) invokes `/cortex-core:pr`, (d) restores `cd "$_origin_pwd"` after the PR skill returns. The existing Step 8 cd-out hard guard (`complete.md:159-163`) composes correctly because of (d). Acceptance: Step 3 prose contains the four-step sequence verbatim or paraphrased; the parity tests at `tests/test_lifecycle_kept_pauses_parity.py` continue to pass; `tests/test_lifecycle_kept_pauses_parity.py` or a new sibling test verifies the cd-out hard guard is reachable when cleanup is invoked after a successful PR-open. **Phase**: Phase 4.

9. **(Must)** **Advisory worktree detection**: detection of "this lifecycle is running in an interactive worktree" reads `interactive.pid` via `cortex_command/interactive_lock.py:read_lock(feature_slug)` AND corroborates by comparing `git rev-parse --show-toplevel` against `pwd`. If either signal is absent or contradictory (e.g., stale PID after `/clear` and pwd not the worktree), the detection treats the session as NOT in Variant A and skips the cd-in-then-out branch in Step 3 (proceeds with `/cortex-core:pr` from current cwd). Acceptance: `complete.md` Step 3 contains both signal reads; a test in `tests/test_complete_pr_routing.py` (new file) verifies the four detection-outcome cases (both signals positive → cd-in-then-out path; PID stale + pwd in worktree → cd-in-then-out path; PID present + pwd NOT in worktree → non-worktree path; both absent → non-worktree path). **Phase**: Phase 4.

10. **(Should)** **Slice 7 commit shape**: the work lands as four PRs in sequence — PR-1 (Phase 1 inventory + characterization tests; no behavior change), PR-2 (Phase 2 cortex init extension; isolated sandbox infra change), PR-3 (Phase 3 new helper + per-callsite parameter), PR-4 (Phase 4 lifecycle prose). Each PR is independently revertable. PR-3 depends on PR-1's tests (otherwise refactor regressions are undetectable); PR-4 depends on PR-2 + PR-3 (otherwise sandbox-deny + missing helper). PR-2 may land in parallel with PR-1 (no dependency). Acceptance: `git log --oneline main..` after all four PRs land shows four distinct merge commits with `#240` or feature-branch tags; `git revert <PR-4-merge>` cleanly removes the lifecycle prose without touching Phase 1-3 work. **Phase**: cross-cutting (all four phases).

## Non-Requirements

- **CwdChanged hook registration**: not in scope. Documented in research as a future backstop. Revisited only if a real failure mode (e.g., orchestrator-session writes still landing in the wrong tree after Phase 3) emerges post-ship.
- **Automation of post-merge worktree cleanup**: not in scope. Today's path (user re-invokes `/cortex-core:lifecycle complete <slug>` after merging; Step 8 calls `cleanup_worktree`) is the in-scope answer per upstream cli/cli#3442 constraint (`gh pr merge --delete-branch` cannot clean local worktrees).
- **Sub-agent dispatch CWD modification**: not in scope. `Agent(isolation: "worktree")` sub-agents in `implement.md` §2 are independently rooted at `$TMPDIR/cortex-worktrees/{task-name}/`; Variant A does not change this.
- **Daytime-pipeline writer-site cleanup**: not in scope. `cortex_command/overnight/daytime_pipeline.py:216-401` writer sites are slated for #246 removal; refactoring them here would be duplicate work.
- **Variant B (fresh `claude --worktree` session)**: rejected at the decompose-gate of epic #237 / discovery `swap-daytime-autonomous-for-worktree-interactive`; user committed to Variant A.
- **Resolver-semantics demotion (A3a)**: explicitly rejected (research FM1-FM3 regress #241, dispatch.py, runner.py). Spec preserves `_resolve_user_project_root()` semantics unchanged.
- **`interactive.pid` session_id migration on `/clear`**: see `## Open Decisions` — surfaced for spec-time resolution.
- **Automated pre-create migration check for unmigrated `settings.local.json`**: not in scope. Users who installed before Phase 2 ships need to re-run `cortex init` manually after Phase 2 lands; the error path (sandbox-deny on first Write inside the worktree) is non-destructive and recoverable. A deterministic pre-create check requires both a sandbox read-path to `~/.claude/settings.local.json` from inside §1a Bash and a code-layer helper (skill prose is not a deterministic gate per CLAUDE.md). Deferred as a follow-up if real users complain.
- **New events beyond `interactive_worktree_entered`**: no other new events. Existing `pr_opened`, `feature_complete`, `interactive_lock_*` family carry all other lifecycle telemetry.

## Edge Cases

- **Stale `interactive.pid` after process crash + restart at same PID**: detection (Requirement 9) corroborates via `git rev-parse --show-toplevel` against `pwd`. If pwd is the main repo, the detection skips Variant A handling regardless of the stale PID.
- **`interactive.pid` session_id rotated by `/clear`**: same corroboration. The PID liveness check in `interactive_lock.py:23-31`'s 8-row table may report LIVE (Row 4/8 default-LIVE on ambiguity), but the corroboration with `git rev-parse` against `pwd` is the decisive signal.
- **User runs `cortex init` without the new step**: Phase 2's registration is opt-in via re-running `cortex init`. Users on an unmigrated `settings.local.json` who select option 2 will see sandbox-denied Write on first attempt inside the worktree. Recovery: re-run `cortex init` once to additively register the worktree base path, then retry the lifecycle invocation. **This migration gap is unmitigated by this spec** — see Non-Requirements ("Automated pre-create migration check for unmigrated settings.local.json"). An automated pre-create check that reads `~/.claude/settings.local.json` from inside §1a Bash and prompts the user is deferred as a follow-up; it requires both a sandbox read-path for the user-settings file and a deterministic gate (skill prose alone is not a deterministic gate per CLAUDE.md's "structural separation over prose-only enforcement" principle).
- **Compound `cd <worktree> && cortex-update-item ...`**: per research FM7, CwdChanged latency is not in this design's failure mode because CwdChanged is not registered. The orchestrator-session Bash cd carries over natively (per Claude Code tools-reference docs); the compound case works because no async env-refresh is involved.
- **User exits the lifecycle mid-implement and re-invokes**: the orchestrator session may start in the home repo (SessionStart hook pins `CORTEX_REPO_ROOT` to home). Re-invocation routing detects the live `interactive.pid` and re-issues the cd handoff. The events.log written during the prior session is intact in the worktree.
- **Worktree path outside both `additionalDirectories` and project dir**: Bash `cd` carry-over fails per Claude Code semantics; the user sees `Shell cwd was reset to <dir>`. Phase 2's `additionalDirectories` registration prevents this on the default path.
- **`gh pr create` invoked while CWD is not the worktree but `interactive.pid` exists for the slug**: detection (Requirement 9) takes the cd-in-then-out branch and the cd-in step lands the pr from the correct tree.
- **Multiple concurrent lifecycles** (different slugs, each in its own worktree): the per-feature `interactive.pid` (shipped #241) prevents same-slug collision. Different-slug concurrent worktrees are independent — Variant A's `cd` is per-session.
- **Phase 1 inventory audit discovers a writer-site reachable from Variant A that is NOT in the ticket's eight named sites**: Phase 1's artifact (`writer-sites.md`, Requirement 4) is the source of truth; Phase 3 refactors every site listed. If the count materially exceeds the ticket's framing, surface back to the user before continuing to Phase 3.

## Changes to Existing Behavior

- **MODIFIED**: `skills/lifecycle/references/implement.md` §1a — after worktree creation and lock acquisition, the lifecycle skill now issues `cd $(cortex-worktree-resolve interactive/{slug})` and emits `interactive_worktree_entered` event. Previously: §1a created the worktree, acquired the lock, and exited with a handoff note expecting the user to manually `cd`.
- **MODIFIED**: `skills/lifecycle/references/complete.md` Step 3 — PR creation now follows the `cd-in-then-out` pattern around `/cortex-core:pr` when the session is detected to be in a Variant-A worktree. Previously: PR creation assumed current cwd was the checkout.
- **ADDED**: new helper `_resolve_user_project_root_from_cwd()` in `cortex_command/common.py` for the narrow set of writers that must follow CWD.
- **ADDED**: `cortex init` registers worktree base path in `sandbox.filesystem.allowWrite` and `additionalDirectories`.
- **ADDED**: new event `interactive_worktree_entered` registered in `bin/.events-registry.md`. Schema: `{schema_version: 1, ts, event, feature, worktree_path}`.
- **UNCHANGED**: `_resolve_user_project_root()` semantics; `cortex_command/interactive_lock.py` (writes still land in main repo's `cortex/`); `cortex_command/pipeline/dispatch.py:555-560` (sub-agent CORTEX_REPO_ROOT pin); `cortex_command/overnight/runner.py:2020-2024` (telemetry routing); `skills/pr/SKILL.md` (no `-C` or `--worktree` flag added).

## Technical Constraints

- **ADR-0001 (file-based state, accepted)**: all lifecycle state lands under `cortex/`. Writer-site refactor preserves this — writes land in either the main repo's `cortex/` (lock files, telemetry) or the worktree's `cortex/` (orchestrator-session events.log).
- **ADR-0003 (per-repo sandbox registration, accepted)**: `fcntl.flock`-guarded additive merge into `~/.claude/settings.local.json`. Phase 2's extension respects this — adds to `allowWrite` and `additionalDirectories` arrays without removing existing entries.
- **ADR-0004 (multi-step Complete + interactive worktree lifecycle, PROPOSED)**: surfaced at decision points. Spec follows ADR-0004's pattern: PR-creation in Complete (not Review tail), uniform multi-step protocol (not bimodal), WorktreeCreate-hook bypass intentional. Spec does not assume ADR-0004 is accepted — `complete.md` modifications respect the existing prose contract.
- **CLAUDE.md "Prescribe What and Why, not How"**: Spec prescribes outcomes (worktree base registered in both `allowWrite` and `additionalDirectories`; new helper exists; cd-in-then-out pattern is in `complete.md` Step 3 prose) and leaves line-level mechanics to the plan phase.
- **MUST-escalation policy**: no new MUST/CRITICAL/REQUIRED escalations introduced. Soft positive-routing phrasing throughout the spec language.
- **`bin/.events-registry.md` gate**: `interactive_worktree_entered` registered in Phase 4 before backlog `grep -c` gates can reference it. Producer: `skills/lifecycle/references/implement.md` §1a. Consumer: statusline and downstream re-invocation routing (TBD).
- **`tests/test_lifecycle_kept_pauses_parity.py` parity**: Phase 4 changes to `implement.md` and `complete.md` must not alter the kept-user-pauses inventory; if they do, the SKILL.md inventory updates land in the same commit.
- **Existing test contracts**: `cortex_command/tests/test_common.py:55-56`, `tests/test_common_utils.py:486` (both assert env-first semantics) stay green — no resolver-semantics change.
- **Sub-agent CWD isolation**: `Agent(isolation: "worktree")` in `implement.md:146,160` is intentional; spec does NOT modify sub-agent dispatch.
- **Multi-PR sequencing**: Slice 7 — PR-1 → (PR-2 ∥ PR-3) → PR-4. Each PR independently revertable.

## Open Decisions

- **`interactive.pid` session_id migration on `/clear`**: extend `hooks/cortex-scan-lifecycle.sh:46-72`'s `.session` migration to also rewrite `interactive.pid`'s session_id field when the session-id rotates, OR carve out as a follow-up ticket. Reason this cannot be settled at spec time: the choice depends on how often `/clear` is used mid-implement in practice (operational data only available post-ship) — a low-frequency edge case argues for follow-up; high-frequency argues for in-scope. Surface to user during §4 approval or close at first review-pass once Phase 4 lands.

## Proposed ADR

None considered. ADR-0004 (multi-step Complete + interactive worktree lifecycle, PROPOSED) is the relevant existing ADR; this spec consumes it rather than proposing a new one. If the `interactive.pid` migration question in `## Open Decisions` is resolved during implementation in a way that locks in a durable cross-cutting pattern, an ADR proposal may be drafted at that point.
