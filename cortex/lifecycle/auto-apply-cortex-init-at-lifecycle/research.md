# Research: auto-apply cortex init at /lifecycle entry via `cortex init --ensure`

Synthesis of five parallel research angles (codebase, web, requirements, tradeoffs, adversarial) on the proposed `cortex init --ensure` flag, its wiring into `/cortex-core:lifecycle` at entry, and the co-traveler fix to the no-flock race in `ensure_claude_md_authorization`. The discovery-phase research at `cortex/research/auto-init-and-update/research.md` already eliminated SessionStart-based triggers and the manually-bumped `_INIT_SCHEMA_VERSION` integer; this research validates within the narrow "--ensure wired into /lifecycle only" design space and surfaces concrete failure modes the discovery did not engage with.

## Codebase Analysis

### Patterns and conventions to follow

- **Hashing**: `hashlib.sha256` is the established idiom (`overnight/plan.py:104`, `critical_review/__init__.py`, `feature_executor.py`).
- **Atomic write**: `cortex_command.common.atomic_write` (`common.py:689–729`). All cortex-managed state writes use this. Note: atomic_write swaps inodes via `os.replace`, which is why the lock must be on a **sibling** file, not the target file itself.
- **`.cortex-init` JSON shape today**: `{"cortex_version": "<PEP-440>", "initialized_at": "<ISO-8601-UTC>"}`. The new field is `init_artifacts_hash: "<64-char-hex>"` (optionally prefixed with a schema version per Q4 below).
- **Argparse flag naming**: lowercase verbs, mutually exclusive with `--update`, `--unregister`, `--revoke-worktree-auth`, `--verify-worktree-auth`.
- **Events registry**: per `cortex/requirements/project.md`, new events register in `bin/.events-registry.md`. Current decision: `--ensure` is silent operational (like `--update`/`--force`) and emits no named event. If a debug NDJSON audit is added per Adversarial mitigation #5, it does NOT need to be a registered event — audits are not events.

### Integration points

- **Layer 3 install-in-progress marker**: written by `plugins/cortex-overnight/install_core.py` (stale-tolerance constant `_INSTALL_MARKER_STALE_SECONDS = 600.0`). Already consumed by `cortex-cli-version-sync.sh`. The `--ensure` interaction with this marker is an Open Question (Q3).
- **`cortex --print-root --format json` envelope** (`cortex_command/cli.py:264–271`): publishing `init_artifacts_hash` here is the named follow-up in the ticket's Edges; not in-scope for this lifecycle but the envelope shape is the natural consumer surface for future tooling.
- **Lifecycle SKILL.md entry structure**: Step 1 (Identify the Feature, ~lines 43–70), Step 2 (Check for Existing State, ~lines 71–124), Step 3 (Execute Current Phase, ~lines 126–182). The ticket says "phase 1 entry" but the skill does not have numbered phases — the phase enum is `research|specify|plan|implement|review|complete`. "Before phase 1 work begins" most naturally maps to "before Step 3 dispatches a phase", which corresponds to "after Steps 1–2 resolve the feature and check state, before any phase work runs."

## Web Research

### Hash-driven idempotent dispatch — directly applicable prior art

- **Terraform** uses `.terraform.lock.hcl` (sibling-lockfile beside manifest) and the canonical three-state exit contract via `-detailed-exitcode`: `0`=no-op, `1`=error, `2`=changes pending/applied. `terraform init -lockfile=readonly` is the explicit drift-detection mode. ([Terraform Dependency Lock File](https://developer.hashicorp.com/terraform/language/files/dependency-lock))
- **Bun install** (2025) fixed a bug where `--frozen-lockfile` "incorrectly reported that the lockfile changed when it didn't" by switching from filesystem-input hashing to a hash of canonical content (sorted package names+versions). Direct lesson: hash the **canonical desired-state structure**, not raw file bytes.
- **Helm** has documented non-determinism bugs (helm#13245, helm#7008): the lock digest based on `requirements.yaml` text rather than the resolved dependency state both misses real drift and triggers false drift. Lesson: hash the resolved output, not the input recipe.
- **pnpm `--frozen-lockfile`** (pnpm#9995) shows that drift-check and apply paths must share their definition of "in sync" or you ship inconsistencies.
- **pre-commit** has years of unresolved user friction over exit-code semantics (autoupdate exits non-zero on any modification — pre-commit#2240, #3182). The "exit non-zero on any change vs exit 0 on successful auto-apply" tension is unresolved across the ecosystem; the design must take an explicit position.

### File locking in Python CLI tools

- **`filelock` library** is the most-recommended cross-platform choice over raw `fcntl` or `portalocker`. Sibling `.lock` pattern, `with FileLock(...)` context-manager, `timeout=-1` default (indefinite) overridable via `blocking=False` to fail-fast.
- **Canonical `lock → read → splice → atomic-rename` pattern**: `with FileLock(path + ".lock"): ...` then `tempfile.NamedTemporaryFile(dir=os.path.dirname(path))` + `os.replace`.
- **Contention-timeout convention**: SuperFastPython's backoff guide recommends `LOCK_NB` + exponential backoff with jitter for CLI tools where indefinite blocking is user-hostile.
- **uv** (uv#13883, #15335, #15704) is actively shoring up concurrency: "file-based lock to the target virtual environment when installing." Documents TOCTOU concerns.

### CLAUDE.md / `.claude.json` concurrent-write incidents — direct prior art

- **claude-code#28847**: "Running multiple Claude Code instances simultaneously causes `.claude.json` corruption due to **unsynchronized concurrent writes**. … Without file locking or atomic write operations, writes overlap and truncate." 305 `.claude.json.corrupted.<ts>` backup files in a single day. Hotfix in 2.1.61.
- **claude-code#40226**: Same root cause + a **recovery-loop-with-no-backoff** that requires manual `^C`. Lesson: lock-contention failure paths must have bounded retry + clean exit, not infinite loop.
- **claude-code#29162**: Debug logs show "Failed to write file atomically" / "Falling back to non-atomic write" — atomic-write attempts exist but fail under EPERM and silently degrade.
- **Direct implication**: the no-flock race in `ensure_claude_md_authorization` is an instance of the exact failure class that has produced documented user-visible corruption in the parent product.

### Splice-into-shared-files race

- **Ansible blockinfile / lineinfile** (ansible#68919): "running tasks from different playbooks simultaneously, sometimes only one update is in the file, but both tasks finish successfully" — silent data loss. The ecosystem leader does not solve concurrent-splice; the application layer must.
- **chezmoi / home-manager** use full-file-replacement-from-template models that sidestep the concurrent-splice problem entirely. The "splice fences into shared files" model is inherently more fragile.

### Agent-entry-driven idempotent precondition

- **`git lfs install`** is the closest analogue: invoked every clone, idempotent, mutates pre-push hook. Failure mode (git-lfs#2865): naive install fails when a custom pre-push hook already exists; mitigation is `git lfs update --force` or `--manual`.

### 2025 supply-chain context

- **npm Shai-Hulud** (796 packages, 132M monthly downloads via preinstall scripts), debug/chalk/axios compromises crystallized industry consensus that **silently running upstream-supplied code on entry is the threat model to design against**. Industry response: pnpm v11 defaults `strictDepBuilds=true`; Dependabot added native `minimum-release-age` (July 2025); Yarn Berry has `enableScripts: false` since v2.
- **Direct implication**: `cortex init --ensure` runs Cortex-controlled code (not third-party), so the specific threat model doesn't transfer, but the **principle** does — be explicit about (a) operation is local-only, (b) what it can modify, (c) opt-out lever exists. See Open Question Q5 on `CORTEX_AUTO_ENSURE=0`.

## Requirements & Constraints

### ADRs that bind

- **ADR-0001 (file-based state)**: `--ensure` and the hash-compare must operate on file-based state only — no external cache, no daemon.
- **ADR-0003 (per-repo sandbox registration)**: `~/.claude/settings.local.json::sandbox.filesystem.allowWrite` is "the only write cortex-command makes outside its own tree." Uses `fcntl.flock` already to serialize concurrent inits. The CLAUDE.md write in ADR-0006 widened this; `--ensure` further widens the apply-frequency of that write.
- **ADR-0006 (consumer CLAUDE.md authorization surface)**: specifies the fence shape, the three subcommands (`cortex init` writes/replaces by version, `--revoke-worktree-auth` removes, `--verify-worktree-auth` probes), and the on-disk lifecycle. **Explicitly does NOT specify WHEN the write happens** — assumes manual `cortex init`. Auto-application at skill entry is a legitimate extension that does not contradict ADR-0006's on-disk shape, but it implicitly answers a question the ADR explicitly left open. See Open Question Q6 on whether an ADR amendment is warranted.
- **No new ADR likely required** per the three-criteria gate in `cortex/adr/README.md` (hard to reverse / surprising without context / real trade-off — `--ensure` extends an existing surface rather than introducing a new policy). The decision and rationale should be documented in the spec or PR description.

### Project rules that bind

- **MUST-escalation policy (post-Opus 4.7)**: default to soft positive-routing phrasing; MUST escalations require F-row evidence from `events.log` or a transcript excerpt showing Claude skipped the soft form. For a brand-new wiring (no prior F-row evidence), the spec must use soft routing for v1.
- **"Prescribe What and Why, not How"**: the lifecycle skill prose should describe the gate (init state is bootstrapped-or-refreshed) and its intent — not the step-by-step CLI invocation.
- **Structural separation over prose-only enforcement** for sequential gates. The entry-time directive IS a sequential gate (must fire before phase 1 work). See Open Question Q1 on wiring location — this principle pushes toward a skill-helper module rather than prose alone.
- **Kept user pauses** inventory in `skills/lifecycle/SKILL.md` + `tests/test_lifecycle_kept_pauses_parity.py` parity test. Any new `AskUserQuestion` pause (e.g., a first-init confirmation per Q2) must be added to both.
- **Skill size cap 500 lines**. Lifecycle SKILL.md is ~236 lines today; room for the directive.
- **SKILL.md-to-bin parity** (`bin/cortex-check-parity`): `--ensure` is a new flag on an existing `cortex` console-script, not a new `cortex-*` bin script. Parity is satisfied if the new flag is referenced in the lifecycle SKILL.md (the in-scope reference surface). No exception entry needed.
- **Conditional Loading**: tags `[auto-init-and-update, install, lifecycle]` match no area-doc phrases. Only `project.md` and `glossary.md` (absent: skipped) load.

### Layer 1/2/3 auto-update architecture is orthogonal

- Layers 1–3 (`docs/internals/auto-update.md`) concern CLI version pairing via `CLI_PIN`. The `--ensure` flag concerns artifact-state scaffolding. They are independently-justified.
- **`CORTEX_AUTO_INSTALL=0` opt-out applies to Layers 2 & 3 only.** It does NOT gate `--ensure`. A separate opt-out (e.g., `CORTEX_AUTO_ENSURE=0`) is needed if the design wants the same defense-in-depth lever. See Open Question Q5.

## Tradeoffs & Alternatives

The seven alternative dimensions investigated and the agent's recommendations, with where the adversarial review contradicted them. The composite that survives both reviews is captured below in Open Questions where genuine user input is required, and in the recommendations otherwise.

### 1. Hash representation

- **Alt A** (single SHA-256 over canonical bundle) — simplest, ~15 LOC, matches Decision 5 of the discovery research.
- **Alt B** (multi-part hash by input class) — debuggable but adds schema-stability burden.
- **Alt C** (version-string only) — explicitly rejected by Decision 5 (silent rot when developer forgets to bump).

**Recommendation**: Alt A with a `v1:` schema prefix (so the comparator can detect schema-version skew vs content drift) and a `CORTEX_DEBUG=1` per-input-class breakdown. **Adversarial caveat** (F1–F4): the canonicalization contract MUST be explicit and unit-tested — POSIX-string sort key (not `pathlib.Path`), tuple ordering preserved (not `set()`/`frozenset()`), `\r\n→\n` normalization on every byte input, and `importlib.metadata.version()` MUST NOT be in the hash inputs (else every patch bump churns the hash).

### 2. Dispatch mechanism

- **Alt A** (branch on marker-presence, then hash compare) — mirrors existing handler shape.
- **Alt B** (always normalized merge) — requires refactoring 4 helpers that have non-uniform idempotency.
- **Alt C** (Alt A + `--force-refresh` debug flag) — cheap orthogonal addition.

**Recommendation**: Alt C. Marker-presence dispatch preserves the first-init vs `--update` distinction; the override flag gives diagnose / critical-review a clean lever.

### 3. Skill wiring

- **Alt A** (Bash from prose) — the ticket's stated approach; lowest cost; matches "lifecycle invocation IS consent" framing because the user sees the Bash call in transcript.
- **Alt B** (skill-helper module `cortex_command/lifecycle.py` calling `--ensure` internally) — matches the project's "Skill-helper modules" pattern and "structural separation over prose-only enforcement" principle.
- **Alt C** (PreToolUse hook) — reintroduces hook-managed writes that the discovery rejected for SessionStart.

**Recommendation in conflict**: Agent 4 favored Alt A with a documented promotion trigger; Agent 5 favored Alt B citing the CLAUDE.md structural-separation principle directly. This is **Open Question Q1** below.

### 4. Layer-3 ordering

- **Alt A** (poll install-in-progress marker, SKIP-and-diagnostic if mid-flight) — Agent 4's pick.
- **Alt B** (poll-and-WAIT with bounded budget) — best correctness, worst latency.
- **Alt C** (fold `--ensure` into Layer 3 as post-reinstall continuation) — best correctness, cross-module coupling cost.
- **Alt D** (accept timing-implicit avoidance) — silent failure mode.

**Recommendation in conflict**: Agent 4 favored Alt A; Agent 5 surfaced F8 (skip → false-positive "ensured" can disagree with the implement-phase §1a probe firing later) and recommended fail-closed instead. This is **Open Question Q3** below.

### 5. R19 consent-gate inheritance on first-init dispatch

- **Alt A** (`--ensure` bypasses R19 — `/lifecycle` IS consent).
- **Alt B** (`--ensure` keeps R19 — defense in depth for foreign-repo accident scenario).
- **Alt C** (`--ensure` adds its own interactive prompt for first-init).

**Recommendation in conflict**: Agent 4 favored Alt B (keep R19) explicitly flagging it for user confirmation; Agent 5 favored Alt A (`/lifecycle` IS consent — treat marker-absent + cortex/-has-content as drift, not as suspicious foreign-repo) because keeping R19 defeats the ticket's purpose. This is **Open Question Q2** below.

### 6. CLAUDE.md flock primitive

- **Alt A** (verbatim port of `settings_merge.py` indefinite-blocking `LOCK_EX`).
- **Alt B** (`filelock` library — adds a runtime dep).
- **Alt C** (`LOCK_EX | LOCK_NB` + bounded budget mirroring `ipc.py`'s takeover-lock pattern).
- **Alt D** (defer the flock fix entirely).

**Recommendation in conflict**: Agent 4 favored Alt A (verbatim port); Agent 5 favored Alt C (mirror the newer `ipc.py` pattern) and additionally recommended **also** updating `settings_merge.py` to match in the same PR — because `settings_merge.py`'s indefinite-block is itself a latent issue that auto-apply will surface (A1, S2). This is **Open Question Q7** below.

### 7. Failure visibility on `--ensure` errors

- **Alt A** (fail-fast — exit non-zero, lifecycle halts).
- **Alt B** (warn-and-proceed — softens a structural gate into prose-monitored warning).
- **Alt C** (silent-skip with NDJSON audit, Layer-3 pattern).

**Recommendation**: Alt A with Alt C-style NDJSON audit appended. Adversarial A3 raised that fail-fast in a 3-hour-deep autonomous session is hostile if the failure is transient — but the counter-argument is that `/lifecycle` is the moment of commitment to multi-hour work; failing fast at the gate is cheaper than discovering at hour 2 that artifacts are stale. **Q8** captures this for user confirmation.

## Adversarial Review

Critical issues that survived the synthesis and that the spec must address explicitly (not "consider further"):

### Failure modes

- **F1–F4 (hash determinism)**: `_GITIGNORE_TARGETS` is actually a tuple, not a set (the discovery prose was loose); sort key must be POSIX-string-relpath not `pathlib.Path`; CRLF normalization must apply to ALL byte inputs; `importlib.metadata.version()` MUST NOT be a hash input (else churn on every patch bump). **Required mitigation**: explicit canonical-serialization contract + a pinned-hash unit test that fails on any input-shape drift.
- **F5 (R19 + missing marker)**: `--ensure` cannot route through the existing first-init path unmodified — the path raises `ScaffoldError` via R19 when `cortex/` has any content. A user whose previous `cortex init` crashed before writing `.cortex-init` is permanently stuck at the gate with no recourse from `/lifecycle`. **Required mitigation**: route `--ensure` to the `--update` (additive, no overwrite) code path even when the marker is absent.
- **F6 (`--update` side-effect)**: `--update` calls `unregister_matching_in_place("cortex-worktrees")` as a settings migration. Silent auto-firing inherits this side-effect. Substring match against user-added entries is the failure mode. **Mitigation**: the spec should explicitly evaluate whether this unregister is correct under auto-apply, or whether it should be gated on `--update` invoked explicitly by the user.
- **F7 (fence-version bump → silent commit-history contamination)**: `_CLAUDE_MD_AUTH_VERSION` bump → next `/lifecycle` → auto-rewrite of CLAUDE.md → user `git commit -a` includes the rewrite. **Mitigation**: the spec should consider whether `--ensure` should print a single-line stderr notice ("cortex applied init updates: CLAUDE.md fence v1→v2") so the user knows the diff isn't theirs.
- **F8 (Layer-3 race with §1a)**: poll-and-skip risks entry-time `--ensure` saying OK and implement-time §1a probe saying stale. **Mitigation**: fail-closed on install-in-progress (5s budget then exit 2) per Q3.
- **F9 (worktree data-loss)**: `git rev-parse --show-toplevel` in a `git worktree add` worktree returns the worktree root, not the primary. First-init via `--ensure` writes into the feature worktree; worktree removal silently destroys the cortex/ data. **Required mitigation**: detect worktree-attached via `git rev-parse --is-inside-work-tree` + `--git-common-dir` mismatch; refuse `--ensure` with a diagnostic pointing to the primary worktree.
- **F10 (additionalDirectories not in hash inputs)**: a known omission — the Edges section's "every input that affects user-visible init outputs must be in the hash" is already violated. **Mitigation**: enumerate hash inputs explicitly in the spec; add an input-contract test.
- **F11 (parity test breakage)**: prose-only directive is safe. A first-init `AskUserQuestion` confirmation pause (per Q2) would require updates to both `skills/lifecycle/SKILL.md` "Kept user pauses" and `tests/test_lifecycle_kept_pauses_parity.py` per CLAUDE.md.

### Security concerns

- **S1 (no per-user opt-out for auto-apply)**: `CORTEX_AUTO_INSTALL=0` does not gate `--ensure`. A force-pushed malicious release that bumps `_CLAUDE_MD_AUTH_VERSION` + ships a hostile fence body would silently rewrite consumer CLAUDE.md on next `/lifecycle`. **Mitigation**: add `CORTEX_AUTO_ENSURE=0` opt-out (see Q5).
- **S2 (flock authenticates nothing)**: any process can hold the sibling-lockfile indefinitely. With indefinite-blocking `LOCK_EX` (Alt A in dimension 6), this is a trivial DoS against `/lifecycle` entry. **Mitigation**: use `LOCK_EX | LOCK_NB` + bounded budget (Alt C in dimension 6).
- **S3 (R19 + autoapply is wrong default)**: R19 fires on any `cortex/`-has-content repo including the cortex-command repo itself or any repo where init crashed mid-flight. Auto-apply makes R19's diagnostic less discoverable (one Bash transcript line); user is told to take manual action that defeats the automation. **Mitigation**: route `--ensure` to `--update` code path on marker-absent (per F5).

### Assumptions worth flagging

- **A1 (auto-apply pressure is rare)**: contradicted by Claude Code #28847 (305 corrupted backups/day from concurrent writes — multi-window is the normal mode for power users). Indefinite-blocking flocks are not acceptable under auto-apply.
- **A2 (prose-only wiring is sufficient)**: contradicted by CLAUDE.md's own "structural separation over prose-only enforcement" principle for sequential gates. The MUST-escalation policy explicitly tolerates soft-routing skips — guaranteeing some `/lifecycle` invocations proceed without `--ensure` firing.
- **A3 (`~30–50ms cold start` is the worst case)**: that's CLI startup only. First-init on cold-cache disk is 500ms–2s. Multi-second pauses with no explanation visible to the user other than the Bash transcript line.
- **A4 (`/lifecycle` invocation IS consent)**: assumes the user knows `/lifecycle` implies cortex installation. The current skill description does not say "this also bootstraps cortex artifacts in your repo." Making `/lifecycle` a consent surface is a higher bar than its current "build a feature" framing.
- **A5 (marker-corruption recovery)**: hand-edited or partially-written `.cortex-init` produces a "missing marker" indistinguishable from a never-initialized repo. **Mitigation**: emit a stderr warning naming the file and treat as drift (route to `--update`), not as missing (which triggers R19).

## Open Questions

These are decisions the Spec phase must resolve — most are user-input questions, not research-resolvable ambiguities:

**Q1. Wiring location (dimension 3 conflict)**. Skill prose ("Before phase 1, run `cortex init --ensure`") vs skill-helper module (`cortex_command/lifecycle.py` with a `[project.scripts]` entry that internally invokes `--ensure`). The structural-separation principle in CLAUDE.md pushes toward the module; the ticket spec assumes prose. The skill-helper module is more expensive now but is also the durable form per the Solution-horizon principle. Decision affects test surface (`tests/` shape) and the SKILL.md prose budget. **Resolved (user): skill-helper module.** Encode the `--ensure` call structurally in a Python entry point invoked from the lifecycle skill, not in prose. Matches the structural-separation principle.

**Q2. R19 inheritance on first-init dispatch (dimension 5 conflict)**. Bypass R19 (treat `/lifecycle` as consent — Adversarial S3, F5 favor this) vs keep R19 (defense in depth — Agent 4 favors this). This materially affects whether `--ensure` actually removes the manual `cortex init` verb from the user's mental model. **Resolved (user): bypass R19.** `/lifecycle` invocation is the consent signal; route marker-absent + `cortex/`-has-content through the `--update` (additive, no overwrite) code path, not first-init's R19 gate.

**Q3. Layer-3 ordering mitigation (dimension 4 conflict)**. Poll-and-skip (Agent 4) vs fail-closed with bounded budget (Adversarial F8) vs fold into Layer 3 (Alt C, more correct but cross-module coupling). The F8 contradiction with the implement-phase §1a probe is concrete and should drive the choice. **Resolved (user): fail-closed bounded wait.** 5-second budget then exit 2 with a diagnostic naming the install-in-progress holder. Avoids the F8 entry-vs-implement gate disagreement.

**Q4. Hash schema versioning and migration on schema bump**. Should `.cortex-init` carry an `init_artifacts_hash_schema` field independent of the hash itself, so a future schema bump (`v1:` → `v2:`) runs `--update` once without comparing hashes across schema boundaries? Avoids the "mass `--update` storm on schema bump" failure mode. *Deferred: resolved in Spec — mechanical schema decision, surfaced in the spec interview alongside the `.cortex-init` field-shape decision.*

**Q5. Per-user opt-out lever**. Add a separate `CORTEX_AUTO_ENSURE=0` env var (mirroring `CORTEX_AUTO_INSTALL=0`)? With or without a single-shot stderr notice on first apply per session? The supply-chain framing in 2025 npm Shai-Hulud makes this a defense-in-depth question, not a UX-purity one. *Deferred: resolved in Spec — strong recommendation is yes (defense-in-depth at no extra implementation cost); the spec interview confirms shape and whether the stderr notice fires.*

**Q6. ADR-0006 amendment**. ADR-0006 explicitly does not specify when the fence write happens. Auto-application at skill entry is a legitimate extension that does not contradict the on-disk shape, but it implicitly answers a question the ADR left open. Is a paragraph appended to ADR-0006, a new ADR-0007, or a paragraph in the spec / PR description the right durable home for the decision? *Deferred: resolved in Spec — documentation placement decision, not a design decision; the spec captures the decision and the PR review chooses the durable home.*

**Q7. CLAUDE.md flock primitive (dimension 6 conflict)**. Verbatim port of `settings_merge.py`'s indefinite-blocking `LOCK_EX` (Agent 4) vs mirror `ipc.py`'s `LOCK_EX | LOCK_NB` + bounded budget (Adversarial S2, A1).
