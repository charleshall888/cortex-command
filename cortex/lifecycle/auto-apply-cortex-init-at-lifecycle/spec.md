# Specification: auto-apply cortex init at /lifecycle entry via `cortex init --ensure`

## Problem Statement

Users on previously-initialized cortex repos stay drifted indefinitely after the CLI bumps its template inventory, gitignore targets, CLAUDE.md fence version, or allowWrite shape, because the existing `cortex init --update` verb requires the user to remember to invoke it manually. Historical churn (~one schema-level change every two months over the past 16 months) shows this is a recurring drift surface, not a one-off. This work folds drift-recovery into the user's existing `/cortex-core:lifecycle` invocation so previously-initialized repos self-refresh whenever the user commits to multi-hour lifecycle work on them. Brand-new clean repos (no `cortex/` directory) also bootstrap automatically on first `/lifecycle` invocation. Repos that look like foreign content (a `cortex/` directory present but no cortex-authored marker) continue to require manual `cortex init` confirmation — the R19 content-aware decline gate stays in place for that case. Net effect: the manual init verbs leave the user's mental model for the common steady-state path (drift on initialized repos and clean first-init on fresh repos) while preserving the foreign-repo protection for the unanticipated-misfire case.

## Phases

- **Phase 1: Hash signal + marker schema** — derive the `init_artifacts_hash`, persist it in `.cortex-init`, enforce the hash-input contract via a pinned-list test, and define the install-in-progress marker path as a stdlib-only shared constant. On-disk schema change ships in this phase (every `cortex init` / `--update` after Phase 1 writes the new field).
- **Phase 2: `cortex init --ensure` flag** — single commit landing the hash-compare dispatch, the narrow R19-bypass, the install-in-progress fail-closed wait, the marker-corruption-recovery discrimination, the `CORTEX_AUTO_ENSURE=0` opt-out, and the fail-fast contract. R4–R8 are sequenced so a partial Phase-2 landing cannot expose the F8 race (see Phase 2 sequencing constraint below).
- **Phase 3: Lifecycle skill-helper module + wiring** — `cortex_command/lifecycle/init_ensure.py` console-script invoked structurally from `skills/lifecycle/SKILL.md` before phase dispatch, with worktree-attached refusal. Closes the loop end-to-end.

**Phase 2 sequencing constraint**: R4–R8 within Phase 2 MUST land in a single commit (or in a sequence where R6's install-in-progress check precedes R4's hash-compare path being reachable from `/lifecycle`). The acceptance criteria for R4 below explicitly require R6's lock-check to be in place — landing R4 alone is treated as an incomplete Phase 2.

## Requirements

1. **Hash derivation produces a deterministic content-hash over an EXPLICIT static input list**: A new `_compute_init_artifacts_hash()` helper in `cortex_command/init/scaffold.py` returns a string of shape `"v1:<64-hex>"`. The inputs are enumerated by an explicit module-level constant `_HASH_INPUT_TEMPLATES: tuple[str, ...]` containing every template-file POSIX-relative path under `cortex_command/init/templates/cortex/` (and `claude_md_authorization.md` since the fence is hashed even though it is not scaffolded). The helper iterates the list verbatim (NO `Traversable.iterdir()` traversal) so the wheel-vs-editable backend cannot diverge on the input set. Each file's bytes are normalized via (a) `replace(b"\r\n", b"\n")`, (b) UTF-8 BOM strip (drop a leading `b"\xef\xbb\xbf"` if present), (c) ensure exactly one trailing `b"\n"` (strip trailing whitespace then append `b"\n"`). After templates, the hasher appends serialized literals: (d) `_GITIGNORE_TARGETS` as the declaration-ordered tuple's `repr()` bytes (the tuple type and order are part of the hash; NOT a `set()`/`frozenset()`); (e) `_CLAUDE_MD_AUTH_VERSION` as `str(int).encode()`; (f) the allowWrite entry shape literal `b"cortex/"`. **MUST NOT** include `importlib.metadata.version("cortex-command")` in the inputs. Acceptance: `python3 -c "from cortex_command.init.scaffold import _compute_init_artifacts_hash; print(_compute_init_artifacts_hash())"` returns the same string across two separate process invocations on the same source tree; a unit test with `tempfile.NamedTemporaryFile` injecting a leading BOM into a template (via a fixture overriding `_HASH_INPUT_TEMPLATES`'s read path) produces the same hash as the same fixture without the BOM. **Phase**: Phase 1: Hash signal + marker schema.

2. **The `.cortex-init` marker persists `init_artifacts_hash` alongside the existing `cortex_version` field**: `cortex_command/init/scaffold.py:402-425` (`write_marker`) is extended to compute the hash at write time and include it in the JSON payload under the key `init_artifacts_hash`. Existing fields (`cortex_version`, `initialized_at`) remain unchanged. The `cortex_version` field is the cortex-provenance signal used by R8's marker-corruption-recovery to distinguish a legitimate cortex marker from an unrelated file. Acceptance: `cortex init --force` in a scratch repo writes a `cortex/.cortex-init` whose `jq -r .init_artifacts_hash` returns a `v1:` prefixed 67-character string AND whose `jq -r .cortex_version` returns a non-empty PEP 440 version string. **Phase**: Phase 1: Hash signal + marker schema.

3. **A contract test enforces the explicit hash-input list against the templates directory**: `tests/test_init_artifacts_hash_inputs.py` (new file) (a) asserts the helper returns a deterministic output across two consecutive in-process invocations, (b) asserts `_HASH_INPUT_TEMPLATES` enumerates EVERY file under `cortex_command/init/templates/cortex/**` plus the `claude_md_authorization.md` fence template — by `os.walk`-ing the templates directory and comparing the result against `_HASH_INPUT_TEMPLATES` — failing when a contributor adds a new template file without updating the list, (c) asserts the BOM-strip and trailing-newline-normalize transforms produce stable output across BOM-present / BOM-absent fixture variants. The test is the structural enforcement of the Edges principle "every input that affects user-visible init outputs must be in the hash inputs." Acceptance: `pytest tests/test_init_artifacts_hash_inputs.py -q` exits 0; adding (simulated) a new file under `templates/cortex/` without updating `_HASH_INPUT_TEMPLATES` causes the test to fail. **Phase**: Phase 1: Hash signal + marker schema.

4. **`cortex init --ensure` flag exists and hash-dispatches with the R19 narrow-bypass**: A new `--ensure` argparse flag added to `cortex_command/init/handler.py:228-308`, mutually exclusive with `--update`, `--force`, `--unregister`, `--revoke-worktree-auth`, `--verify-worktree-auth`. Behavior (in order): (a) honor `CORTEX_AUTO_ENSURE=0` per R7 — if set, exit 0 silently; (b) run R6's install-in-progress lock-check (this dependency means R6 MUST be present before R4 is reachable from `/lifecycle`); (c) compute the installed hash via R1; (d) read `<repo>/cortex/.cortex-init` if present and extract `init_artifacts_hash` AND `cortex_version`; (e) dispatch on the four cases below. The dispatch table is the load-bearing security boundary — see R5 for the rationale:
   - **marker present + hash matches**: exit 0, no writes.
   - **marker present + hash mismatch**: dispatch through the `--update` code path at handler.py:248-252 (additive scaffold, NO R19); write the new marker.
   - **marker absent + `cortex/` absent OR empty**: dispatch through clean first-init (handler.py path for `not marker_present and not args.update and not args.force` BUT skipping `check_content_decline` because `cortex/` is empty/absent so R19 would no-op anyway); write the marker.
   - **marker absent + `cortex/` has content (R5 boundary case)**: KEEP R19 — exit 2 with the existing R19 decline diagnostic, pointing the user to `cortex init` (manual). Do NOT auto-dispatch.
   Acceptance: (1) `cortex init --ensure` in a clean scratch repo (no `cortex/`) exits 0 and writes `cortex/.cortex-init` with the current hash; (2) second invocation exits 0 with no file mtime change; (3) mutating the installed hash (e.g., simulating a `_CLAUDE_MD_AUTH_VERSION` bump via monkeypatch) and re-invoking exits 0 with marker mtime updated; (4) in a scratch repo where `cortex/` exists with arbitrary non-cortex content AND no marker, `cortex init --ensure` exits 2 with the R19 decline message (foreign-content protection preserved); (5) `cortex init --ensure` rejects combinations like `--ensure --update` with the argparse-mutex error. **Phase**: Phase 2: `cortex init --ensure` flag.

5. **R19 stays in force for the marker-absent + cortex/-has-content case**: The narrow bypass from R4 routes ONLY the marker-present hash-mismatch case around R19. The marker-absent + `cortex/`-has-content case continues to fire R19's `check_content_decline` gate, preserving the foreign-repo / wrong-window protection that the R19 gate was originally built for. The `/cortex-core:lifecycle` invocation is consent to refresh a previously-initialized cortex repo OR to bootstrap a clean one; it is NOT consent to mutate arbitrary `cortex/` content the user did not author. Acceptance: in a scratch repo where `cortex/backlog/SOMETHING.md` exists but no `.cortex-init` marker, `cortex init --ensure` exits 2 with the R19 decline message; the user is told to run `cortex init` manually. **Phase**: Phase 2: `cortex init --ensure` flag.

6. **`--ensure` fails closed on Layer-3 install-in-progress contention WITHOUT importing from the plugin**: A new stdlib-only helper `cortex_command/init/install_state.py` defines `install_in_progress_marker_path() -> Path` returning `Path(os.environ.get("XDG_STATE_HOME", str(Path.home() / ".local/state"))) / "cortex-command" / "install.in-progress"`. The marker path is documented as a stable cross-shipping-unit contract. `plugins/cortex-overnight/install_core.py` (the marker writer) is updated to compute the path from this same logic — either by importing from `cortex_command.init.install_state` (the plugin already runs against an installed cortex-command wheel) OR by vendoring the helper following the existing `cortex_command/install_guard.py` precedent (kept in lockstep by `just sync-install-guard`-equivalent + a pre-commit byte-identity guard). The implementation MUST NOT have `cortex_command/init/` import from `plugins/cortex-overnight/install_core.py` (which `sys.exit(1)`s at import time when `CLAUDE_PLUGIN_ROOT` is unset). Before computing the hash, `--ensure` reads `install_in_progress_marker_path()` and checks `os.path.exists` + `mtime` freshness against the documented 600-second tolerance (also defined as a module-level constant in `install_state.py`). If the marker exists and is fresh, `--ensure` polls up to a bounded 5-second budget at 50ms intervals; if still present after 5s, exit 2 with a diagnostic: `cortex init --ensure: Layer-3 install in progress (marker mtime <ts>); rerun after install completes`. Acceptance: (1) with the install-in-progress marker manually present and fresh, `cortex init --ensure` exits 2 within 6 seconds with the named diagnostic on stderr; (2) remove the marker and re-invoke exits 0; (3) `install_in_progress_marker_path()` returns the SAME `Path` value when called from `cortex_command/init/install_state` AND from the plugin's writer code path — verified by a parity test `tests/test_install_state_path_parity.py` that imports both and asserts equality; (4) the cortex CLI does NOT crash with "plugin path mismatch" when `CLAUDE_PLUGIN_ROOT` is unset (i.e., normal user-shell invocation). **Phase**: Phase 2: `cortex init --ensure` flag.

7. **`CORTEX_AUTO_ENSURE=0` per-user opt-out exists**: When the environment variable `CORTEX_AUTO_ENSURE=0` is set, `cortex init --ensure` exits 0 silently without computing the hash, reading the marker, or writing any file. This mirrors the `CORTEX_AUTO_INSTALL=0` opt-out shape from Layers 2/3 (`plugins/cortex-overnight/install_core.py:175-183`). **Documented limitation**: this is an opt-out-to-decline lever, not an opt-in-to-proceed gate — it defends against repeated auto-apply on repos the user knows they want to keep manual, but does NOT defend against unanticipated misfires (the user would have to set the variable before the misfire). The foreign-content protection for that case is R5's narrow-bypass design (R19 still fires on marker-absent + cortex/-has-content), not R7. Acceptance: `CORTEX_AUTO_ENSURE=0 cortex init --ensure` in a drifted repo exits 0 without modifying any file (`git status --porcelain` shows no changes); unset the variable and re-invoke applies normally. **Phase**: Phase 2: `cortex init --ensure` flag.

8. **`--ensure` fails fast on errors with cortex-provenance-discriminated marker recovery**: When `--ensure` encounters an unwritable target file or any operational failure, it exits non-zero with a single-line stderr diagnostic naming the failure mode. Exit codes follow the existing handler contract (handler.py:24-38): 2 for user-correctable gates, 1 for unexpected runtime failures. **Marker recovery is provenance-discriminated**: if `.cortex-init` exists but the `init_artifacts_hash` field is missing or unparseable, the helper checks whether the `cortex_version` field is present and is a parseable PEP 440 version string. If yes → treat as legitimate pre-Phase-1 cortex marker, emit a stderr warning (`cortex init --ensure: .cortex-init missing init_artifacts_hash field; refreshing`), and route to the R4 dispatch (which itself respects R5's boundary). If `cortex_version` is also missing or malformed → treat as a foreign artifact, exit 2 with the diagnostic `cortex init --ensure: .cortex-init exists but lacks cortex_version field; this does not look like a cortex marker. Run \`cortex init\` manually to confirm scaffolding intent`. This prevents the failure mode where any tool writing a `.cortex-init` file in a non-cortex repo silently triggers a cortex update. Acceptance: (1) with a hand-truncated `.cortex-init` that retains `cortex_version` but drops `init_artifacts_hash`, `cortex init --ensure` emits the named warning and exits 0 after writing a fresh marker; (2) with a `.cortex-init` lacking `cortex_version` (e.g., a foreign artifact), `cortex init --ensure` exits 2 with the named diagnostic; (3) with `cortex/` unwritable (chmod 555), it exits non-zero with a diagnostic naming the path. **Phase**: Phase 2: `cortex init --ensure` flag.

9. **Skill-helper module `cortex_command/lifecycle/init_ensure.py` exists**: A new module under the existing `cortex_command/lifecycle/` package with a `[project.scripts]` console-script entry (e.g., `cortex-lifecycle-init-ensure`). The module's entry function invokes the in-process `cortex init --ensure` logic (NOT a Bash subprocess to `cortex`) so the call is testable end-to-end via direct import. The module is the structural-separation form of the wiring per CLAUDE.md's "structural separation over prose-only enforcement for sequential gates" principle. Acceptance: `cortex-lifecycle-init-ensure` (console-script) and `python3 -m cortex_command.lifecycle.init_ensure` both invoke the same code path; both return the same exit codes as `cortex init --ensure`; both honor `CORTEX_AUTO_ENSURE=0`. **Phase**: Phase 3: Lifecycle skill-helper module + wiring.

10. **`/cortex-core:lifecycle` invokes the helper before phase dispatch with dual-source propagation verified**: `skills/lifecycle/SKILL.md` adds one directive at the entry point (after Step 2 "Check for Existing State" resolves and before Step 3 "Execute Current Phase" dispatches): instruct the skill to run `cortex-lifecycle-init-ensure` (the console-script from R9) and halt on non-zero exit. The directive is prose pointing at the structural helper, NOT prose that the model must interpret to compose CLI invocations. The wiring is mirrored to `plugins/cortex-core/skills/lifecycle/SKILL.md` by the existing dual-source pre-commit hook. Acceptance: (1) `grep -c "cortex-lifecycle-init-ensure" skills/lifecycle/SKILL.md` ≥ 1; (2) `just check-dual-source` (or the equivalent existing dual-source enforcement recipe) exits 0 with no pending mirror diffs; (3) the existing dual-source-drift test `tests/test_dual_source_drift.py` (or its current name) exits 0 — failing it requires updating the SKILL.md change and re-running the mirror generator. This acceptance set ensures the mirror state is a consequence of THIS Phase 3 change rather than ambient state, by gating on the parity check that the dual-source pre-commit hook enforces. **Phase**: Phase 3: Lifecycle skill-helper module + wiring.

11. **The helper refuses invocation inside a `git worktree add` worktree**: Before any other check, the helper module runs `git rev-parse --is-inside-work-tree` AND `git rev-parse --git-common-dir` and detects the attached-worktree case (where `--git-common-dir` resolves to a path outside the worktree's `.git/`). On detection, exit 2 with a diagnostic: `cortex-lifecycle-init-ensure: invoked inside a git worktree (<worktree-root>); run from the primary worktree (<primary-root>) to bootstrap or refresh cortex/`. This prevents the F9 data-loss path where first-init scaffolds `cortex/`, CLAUDE.md, `.gitignore`, and an allowWrite entry into a feature worktree that is later silently destroyed by `git worktree remove`. Acceptance: `git worktree add /tmp/wt-test HEAD && cd /tmp/wt-test && cortex-lifecycle-init-ensure` exits 2 with the diagnostic; rerunning from the primary worktree exits 0. **Phase**: Phase 3: Lifecycle skill-helper module + wiring.

## Non-Requirements

- **No CLAUDE.md flock fix.** The latent no-flock race in `ensure_claude_md_authorization` (`cortex_command/init/scaffold.py:554-633`) is explicitly out of scope. Rationale (per user decision Q4): the hash-content design means CLAUDE.md is only rewritten when `_CLAUDE_MD_AUTH_VERSION` actually bumps; that constant has been `1` since ADR-0006 landed (zero bumps in 16 months), so `--ensure`'s steady-state write frequency to CLAUDE.md equals manual `cortex init`'s. The race remains latent, recoverable (user can re-run `cortex init --force`), and below the fix-threshold. No follow-up ticket is filed.
- **No `--print-root --format json` envelope publication of `init_artifacts_hash`.** Deferred to a separate ticket when future external consumers (a `cortex doctor` drift check, a status-line indicator, a CI probe) actually exist.
- **No `--print-root --format json` envelope publication of the install-in-progress marker path.** R6's shared-constant approach via `cortex_command/init/install_state.py` is the v1 solution. Publishing the path in the envelope is a more general public-contract surface that the same future-consumers ticket can pick up.
- **No wiring into skills other than `/cortex-core:lifecycle`.** `/discovery`, `/refine`, `/backlog`, `/requirements-gather`, `/requirements-write`, `/critical-review`, `/research` continue to expect manual `cortex init --update`.
- **No SessionStart-based trigger.** Rejected by discovery Decision 1.
- **No manually-bumped `_INIT_SCHEMA_VERSION` integer.** Rejected by discovery Decision 5.
- **No NDJSON audit log of `--ensure` invocations or outcomes.** Add when actual incidents accumulate that an audit log would have helped diagnose.
- **No interactive consent prompt on first-init.** R5's narrow-bypass design preserves R19's foreign-content protection structurally; an `AskUserQuestion`-style prompt is the wrong tool because Claude is the invoker, not a human at a terminal. This also keeps the `tests/test_lifecycle_kept_pauses_parity.py` inventory untouched.
- **No `--force-refresh` debug flag.** YAGNI until a debug case actually requires it.
- **No update to `settings_merge.py`'s indefinite-blocking flock pattern.** Out of scope by parity with the CLAUDE.md flock scope decision.

## Edge Cases

- **Marker corruption with cortex_version present**: covered by R8 — emit warning, route to drift path.
- **Marker corruption with cortex_version absent or malformed**: covered by R8 — exit 2 with foreign-artifact diagnostic, no silent dispatch.
- **Worktree-attached invocation**: covered by R11 — refuse with diagnostic.
- **Foreign repo with stray `cortex/` directory but no marker**: covered by R5 — R19 fires, declines, points to manual `cortex init`.
- **Layer-3 install racing with `--ensure`**: covered by R6 — fail-closed bounded wait via shared marker-path constant.
- **Fence-version bump triggers auto-rewrite while user has CLAUDE.md staged**: out of scope to actively mitigate; the user sees the diff in their commit. If this becomes painful, a one-line stderr notice on actual write is the proportional next step — captured for follow-up, not in v1.
- **`additionalDirectories` settings field added in a future cortex release without updating hash inputs**: R3's contract test fires on this — adding any new template/constant requires updating `_HASH_INPUT_TEMPLATES` and the helper input list.
- **Mass `--update` storm at Phase 3 land**: every existing user's first `/cortex-core:lifecycle` invocation post-Phase-3 hits a `.cortex-init` lacking `init_artifacts_hash`; R8's provenance check sees `cortex_version` present (the pre-Phase-1 marker shape), routes through `--update`, and writes the new field. This is a one-time storm at Phase 3 release, semantically equivalent to the `v1: → v2:` storm — accepted cost, no active mitigation. Release notes should mention it.
- **Mass `--update` storm on hash-schema bump (`v1:` → `v2:`)**: minimized by the `v1:` prefix design (the comparator detects schema-version skew vs content drift), but a `v2:` introduction still means every repo's `--ensure` triggers a one-time `--update` on first invocation after the bump. Accepted cost; future schema bumps should ship with release notes.
- **PYTHONHASHSEED variability across Python processes**: prevented by R1's explicit static template list (no `iterdir()` traversal whose order could depend on hash randomization) and `tuple` (not `set`/`frozenset`) serialization of `_GITIGNORE_TARGETS`. R3's contract test verifies determinism across separate process invocations.
- **Wheel-installed vs editable-installed (`uv pip install -e .`) divergence**: prevented by R1's explicit static template list (no `Traversable.iterdir()` whose child-set may differ between zipimport-backed wheel and filesystem-backed editable) and the BOM-strip / trailing-newline normalization byte transforms.
- **Multiple `/lifecycle` invocations in concurrent Claude windows on the same repo, both detecting drift simultaneously**: both invoke the helper; `settings_merge.py`'s existing flock serializes the allowWrite write. The CLAUDE.md splice is unprotected (per the explicit Non-Requirement above) — last-writer-wins is the accepted behavior.
- **`--update` path inherits the `unregister_matching_in_place("cortex-worktrees")` settings migration** (handler.py:299-306): routing `--ensure` through the `--update` code path inherits the substring expunge. Steady-state no-op; edge case is a user-added entry containing the substring. Accepted as-is.
- **cortex-overnight plugin not installed**: `--ensure` does NOT depend on plugin import (per R6's shared-constant design). When the plugin is absent, the install-in-progress marker simply never exists, and the lock-check no-ops.

## Changes to Existing Behavior

- **ADDED**: `cortex init --ensure` flag and the `cortex-lifecycle-init-ensure` console-script. New surfaces; do not modify existing flag behavior.
- **ADDED**: stdlib-only `cortex_command/init/install_state.py` module exposing `install_in_progress_marker_path()` and the `INSTALL_MARKER_STALE_SECONDS` constant — the shared cross-shipping-unit contract for the Layer-3 marker path.
- **MODIFIED**: `cortex_command/init/scaffold.py:402-425` (`write_marker`) — adds `init_artifacts_hash` to the JSON payload. Pre-Phase-1 markers without the field are detected by R8's provenance check (cortex_version present) and refreshed on next `--ensure`.
- **MODIFIED**: `plugins/cortex-overnight/install_core.py` — replaces its current marker-path resolution with a call into `cortex_command.init.install_state.install_in_progress_marker_path()` (or vendors the helper per `install_guard.py` precedent). Behavior is unchanged; the change is structural to eliminate the cross-package coupling R6 would otherwise inherit.
- **MODIFIED**: `skills/lifecycle/SKILL.md` — adds one directive between Step 2 and Step 3 instructing the skill to invoke the helper before phase dispatch. Auto-mirrored to `plugins/cortex-core/skills/lifecycle/SKILL.md` by the existing dual-source pre-commit hook.

## Technical Constraints

- **Hash determinism**: per R1, the canonicalization contract is non-negotiable — explicit static template list (NO `iterdir()`), CRLF + BOM + trailing-newline normalization on every byte input, tuple ordering preserved, no `importlib.metadata.version()` in inputs. Enforced by R3 contract test.
- **Cross-shipping-unit marker contract**: per R6, the install-in-progress marker path is a stdlib-only documented contract shared between the wheel (reader) and the plugin (writer). Enforced by a parity test (R6 acceptance #3). The wheel MUST NOT import from `plugins/cortex-overnight/install_core.py`.
- **Exit-code contract**: per existing `cortex_command/init/handler.py:24-38` — 0 success/no-op, 2 user-correctable gate failure, 1 unexpected runtime failure. `--ensure` honors this contract identically.
- **Hash algorithm**: `hashlib.sha256` per existing codebase convention (`overnight/plan.py:104`, `critical_review/__init__.py`, `feature_executor.py`).
- **Skill-helper module pattern**: per `cortex/requirements/project.md`'s "Skill-helper modules" architectural constraint — the helper exposes a `[project.scripts]` console-script entry and is also invocable via `python3 -m cortex_command.lifecycle.init_ensure` for working-tree dogfooding.
- **No new event registration**: `--ensure` is silent operational (like `--update`/`--force`) and emits no named event.
- **Skill size cap**: `skills/lifecycle/SKILL.md` is ~236 lines today; adding one directive fits well under the 500-line cap.
- **Parity test**: `bin/cortex-check-parity` is satisfied if the new helper's CLI/module name is referenced in the lifecycle SKILL.md. No `bin/.parity-exceptions.md` entry needed.
- **Dual-source enforcement**: R10's acceptance gates on the existing dual-source pre-commit hook + drift test — the same machinery that already keeps `plugins/cortex-core/skills/` in lockstep with `skills/`.
- **MUST-escalation policy**: the wiring directive in `skills/lifecycle/SKILL.md` uses soft positive-routing phrasing. No MUST escalation — the structural enforcement is in the helper module's existence and the dual-source hook, not in the prose.
- **Solution horizon**: the helper module and the shared install-state module are durable shapes; future entry-time concerns (a doctor check, a status-line refresh) can land in the same modules without revisiting the wiring.
