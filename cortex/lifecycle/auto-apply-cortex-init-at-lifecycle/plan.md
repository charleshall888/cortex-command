# Plan: auto-apply-cortex-init-at-lifecycle

## Overview
Land `cortex init --ensure` as a structural drift-recovery helper composed of (a) a deterministic content-hash over an explicit static template list persisted in `.cortex-init`, (b) a stdlib-only shared install-in-progress marker-path module the wheel and the plugin both consume, and (c) a `cortex_command/lifecycle/init_ensure.py` skill-helper module the lifecycle SKILL.md invokes structurally between Step 2 and Step 3. The work decomposes into three sequential phases (hash signal + marker schema → flag + dispatch → skill-helper + wiring); Phase 2's `--ensure` flag and its install-in-progress lock-check land before Phase 3 wires the helper into `/cortex-core:lifecycle`, satisfying the spec's "R6 precedes R4-reachable-from-lifecycle" sequencing constraint.

## Outline

### Phase 1: Hash signal + marker schema (tasks: 1, 2, 3)
**Goal**: Persist a deterministic `init_artifacts_hash` in `.cortex-init` and enforce the hash-input contract by a structural test.
**Checkpoint**: `cortex init --force` in a scratch repo writes a marker whose `jq -r .init_artifacts_hash` returns a `v1:`-prefixed 67-character string AND `pytest tests/test_init_artifacts_hash_inputs.py -q` exits 0.

### Phase 2: `cortex init --ensure` flag (tasks: 4, 5, 6)
**Goal**: Land the `--ensure` flag with hash-compare dispatch, R19 narrow-bypass, install-in-progress fail-closed wait, marker-corruption recovery, and `CORTEX_AUTO_ENSURE=0` opt-out — all reachable from the CLI but not yet wired into the lifecycle skill.
**Checkpoint**: `cortex init --ensure` (a) exits 0 on a clean repo and writes the marker; (b) exits 0 with no mtime change on second invocation; (c) exits 2 with the R19 decline diagnostic on marker-absent + cortex/-has-content; (d) exits 2 within 6 seconds when the install-in-progress marker is fresh; (e) exits 0 silently under `CORTEX_AUTO_ENSURE=0`.

### Phase 3: Lifecycle skill-helper module + wiring (tasks: 7, 8, 9)
**Goal**: Surface `--ensure` as a console-script the lifecycle skill invokes structurally before phase dispatch, with worktree-attached refusal as the F9 data-loss guard.
**Checkpoint**: `/cortex-core:lifecycle <slug>` in a drifted previously-initialized repo refreshes the marker before phase dispatch, AND `cortex-lifecycle-init-ensure` invoked inside a `git worktree add` worktree exits 2 with the named diagnostic.

## Risks

- **Phase-2 single-commit constraint vs per-task commits**: spec §Phase 2 sequencing requires R4–R8 land in a single commit OR in a sequence where R6 precedes R4-reachable-from-lifecycle. Plan satisfies the second clause: Tasks 4–6 land before Task 8 wires `--ensure` into the lifecycle skill, so a partial Phase-2 landing cannot expose the F8 race. The implement phase's per-task commit cadence is therefore safe — the constraint is structural, not literal-single-commit.
- **Marker-path module: import vs vendored (Task 4)**: spec offers two options. Plan picks import (plugin → wheel), since the plugin already runs against an installed cortex-command wheel per spec assertion. If the plugin's PEP 723 venv turns out to lack `cortex_command` (contradicting the spec assertion), Task 4 must switch to byte-identical vendoring under a `just sync-install-state-path` recipe modeled on `just sync-install-guard`. Surface to operator before flipping the choice.
- **PEP 440 parser dependency in R8 (Task 5)**: R8's marker-recovery requires checking whether `.cortex-init`'s `cortex_version` field is a parseable PEP 440 string. `packaging.version.Version` is an indirect dep via setuptools but is not a direct cortex-command dep today. If adding `packaging` as a direct dep is undesirable, fall back to a regex check `^\d+\.\d+` — the discrimination is "looks like a version number" not "is canonical PEP 440," and the regex captures the failure mode.
- **`cortex_version`-field-presence is a weak provenance signal (R8 design ceiling)**: spec R8 chose this discrimination knowingly — distinguishing corruption from foreign artifact via a single user-editable JSON field. A hand-edited or coincidentally-named `.cortex-init` containing `cortex_version: "1.2.3"` will route through R8's recovery-via-refresh path. The plan's Task 5 case (v) tightens this by re-firing `check_content_decline` whenever `cortex/` contains anything beyond the marker file itself, so the failure mode is narrowed to "marker hand-edited AND `cortex/` is empty otherwise" — a low-stakes case where the cost of false-refresh is overwrite-with-current-templates. If the user wants stronger provenance (HMAC-signed marker, directory-content fingerprint), that is a follow-up ticket — not in this lifecycle.
