# Specification: Reframe discovery to principal-architect posture (#195)

## Problem Statement

The current `/cortex-core:discovery` skill produces over-decomposed, undersized, prescriptive tickets that pigeonhole the downstream `/cortex-core:refine` phase. Each finding becomes one ticket; tickets quote implementation mechanism in their bodies; the downstream phases consume narrowly-scoped pieces that pre-determine what gets built. This change reframes discovery's decompose phase from finding-mirroring (one-finding-to-one-ticket) to architectural distillation (name the pieces by role, integration, and edges; tickets follow from the architecture). The reframe lands DR-1 from `research/discovery-architectural-posture-rewrite/research.md` plus 9 DR-G carryovers per commit 41ff883. Beneficiaries: the user (better-shaped backlog), the refine phase (less prescriptive starting material), and the lifecycle phase (cleaner spec entry).

**Defect-vs-novel binary (referenced in R5):** The prior decompose protocol classified each ticket as either a "defect" (fix to an existing surface) or "novel" (new behavior), and applied different prescriptive-prose rules to each. R5 eliminates this binary — the same uniform template applies to all pieces regardless of defect/novel character.

6. **R6 — Section-partitioned prescriptive-prose check with LEX-1 regex baked into the spec.** Replace the lexical-headers ban currently at `skills/discovery/references/decompose.md:147` with a section-partitioned check.

   **LEX-1 regex specification** (the check's exact behavior, not deferred to implementation):
   - **Pattern 1 (path:line)**: `\b[\w./\-]+\.(md|py|sh|json|toml|yml|yaml):\d+(?:-\d+)?\b` — matches `decompose.md:147`, `bin/foo.py:42-58`, etc. Bare paths without `:line` do NOT match (those are narrative references, not prescriptions).
   - **Pattern 2 (section-index)**: `(?:§|R)\d+(?:[a-z]\)?|\([a-z]\))?\b` — matches `§5`, `§3a`, `§3(a)`, `R2`, `R2(b)`. Single-letter `§` references without a digit do NOT match.
   - **Pattern 3 (quoted-prose-patch)**: a fenced code block (` ``` ` or `~~~`) of ≥2 non-empty lines appearing inside a forbidden section. Single-line fenced blocks and inline backticks do NOT match (narrative code references like `` `cortex-update-item` `` are fine).
   
   **Forbidden sections (per ticket body)**: `## Role`, `## Integration`, `## Edges`. **Permitted section**: `## Touch points`.
   
   **Section-boundary detection**: a section begins at a line matching `^## (Role|Integration|Edges|Touch points)$` and ends at the next line matching `^## ` (any sibling-level heading) or end-of-file. Third-level subsections (`### foo`) inside a section are part of that section. Fenced code blocks are tolerated as ranges (do not split sections).
   
   **Worked examples**:
   - **PASSES** (no flag): `"## Edges\n\nThis piece breaks if the phase-transition contract changes."` (named contract, no path:line)
   - **PASSES** (no flag): `` "## Role\n\nThe role is to track lifecycle state. See `cortex-update-item` for the helper." `` (inline backtick narrative)
   - **FLAGS**: `"## Edges\n\nThis piece must update decompose.md:147 to replace the ban."` (path:line in forbidden section)
   - **FLAGS**: `"## Integration\n\nFollows the pattern in §3a."` (section-index in forbidden section)
   - **NOT flagged** (anti-pattern: false positive on legitimate narrative): `"## Edges\n\nThe phase-transition contract is documented in skills/lifecycle/SKILL.md."` (bare path, no `:line`, no `§`)
   
   The check runs at decompose ticket-write time. Defense-in-depth at architecture-write time (carryover #2) is deferred per LEX-2: single-gate at decompose-§5 with pre-commit hook re-run as the second-actor surface. **LEX-2 stopping rule** (resolves the R6 reopening contingency): LEX-2 may reopen at most ONCE per implementation cycle. A second reopening triggers a scope-rethink escalation (the same path as R12's three-failed-re-walks).
   - **Acceptance**: `grep -c "section-partitioned\|path:line\|section-index\|quoted-prose-patch" skills/discovery/references/decompose.md` ≥ 3 AND the three regex patterns + section-boundary rule + worked examples + 2 anti-patterns are documented inline (verifiable by `grep -c "PASSES\|FLAGS\|NOT flagged" skills/discovery/references/decompose.md` ≥ 4).

## Non-Requirements

- **No nested-skill recursion for promote-sub-topic.** R4's 4th approval option creates a `needs-discovery` backlog ticket linked via a body-section reference (`## Promoted from`), NOT a recursive `/cortex-core:discovery` invocation. Recursive semantics (parent-state-during-sub, return-to-parent flow, slug hierarchy) are deferred to a follow-up ticket if the minimal-stub form proves insufficient.
- **No `parent_discovery:` frontmatter field.** The promote-sub-topic linkage is body-section-only because no consumer reads `parent_discovery:` (verified zero matches across the codebase). Re-introducing a frontmatter field requires defining backlog-schema docs, refine-side reader, and lifecycle integration first — out of scope here.
- **No defense-in-depth prescriptive-prose check at architecture-write time** (in this iteration). The check runs once at decompose ticket-write time. Pre-commit hook is the second-actor surface. Carryover #2 is deferred per LEX-2 with a stopping rule (one reopen per cycle). Spec records the rationale.
- **No automated heterogeneous-backlog migration.** Existing tickets with Value/What/Why/Findings sections stay as-is; only tickets produced after the rewrite lands use Role/Integration/Edges/Touch-points. Human-readable search across the heterogeneous backlog is lossy — accepted cost, **bounded to open tickets at ship time**. Closed-then-reanimated tickets (R13 re-run path) re-emit under the new template; the prior `## Dropped Items` audit trail from the old `decomposed.md` shape is preserved in the closed artifact but not regenerated.
- **No multi-epic discovery output.** Single-epic-per-discovery constraint stays. If pieces span >1 logical grouping, the discovery splits at clarify time before research, not at decompose. The user must know the grouping shape before research runs.
- **No CHANGELOG entry for R7-era event-name removal from prose.** The events themselves remain in `bin/.events-registry.md` as `deprecated-pending-removal` with `2026-06-10` sunset — the registry IS the deprecation record.
- **No Vale or external-tooling integration.** The lexical scanner is stdlib-only Python matching existing `bin/cortex-check-*` patterns.
- **No changes to backlog/195 fields beyond `## Touch points` relocation.** R14 is a body-section relocation; frontmatter, title, and overall structure stay.
- **No new `re_walk_attempt` event type.** R12's attempt counter reuses the existing `architecture_section_written` event with an extra `re_walk_attempt: <int>` field, keeping the event registry to three new rows.

## Edge Cases

- **Architecture section with zero pieces.** Discovery exits to decompose with the zero-piece branch from R10. Decomposed.md still written as audit trail with explicit `decomposition_verdict: zero-piece` and a one-sentence rationale.
- **Architecture section with exactly one piece.** Single-piece branch in R10's §4 augmentation: create one ticket directly, no epic. No "Why N pieces" gate fires (piece_count = 1 < threshold).
- **Topic where the architecture section is genuinely not applicable** (e.g., a "find-and-document" investigation with no pieces to construct). Per R1, non-constructive topic shapes use the permissive paragraph framing OR the zero-piece exit. Spec phase walks one such case during R12 re-walk to validate.
- **User picks "revise" on a 9-piece architecture section.** Per R4 GATE-2 resolution: agent opens a free-text prompt scoped to the architecture section; user can name specific pieces, suggest splits/merges, or provide directional feedback. Agent re-walks the architecture section under the new direction; re-presents. Loop continues until approve or drop.
- **User picks "revise-piece <N>" on the post-decompose batch-review gate (R15).** Agent opens a free-text revision prompt scoped to ticket N's body. Agent re-walks ticket N's Role/Integration/Edges/Touch-points under the user's direction; re-presents the full batch (not just ticket N). Loop continues until approve-all or all pieces are dropped.
- **Re-run of `/cortex-core:discovery` on a topic with existing research.** Per R13: new slug `{topic}-2`, prior artifact untouched, `superseded:` frontmatter linkage on the new artifact. Events emit to `research/{topic}-2/events.log` via R9's `resolve-events-log-path` honoring the `-N` suffix. User chooses which slug downstream consumers (refine, lifecycle) point at via `discovery_source:` frontmatter on new backlog tickets.
- **R3's structural test ambiguity.** "Share ≥1 named contract surface" requires the pieces to have named contract surfaces in their Role/Integration content. If a piece's Role names no contracts (vague Role authoring), R3 cannot evaluate — surfaces back to the architecture-authoring step in R1 to re-author the Role with named contracts. This converts the gate-failure into a re-author signal rather than a stuck state.
- **R1 template walk-back loop.** If R3 instructs a merge and Pieces is re-emitted, Integration shape and Seam-level edges must be re-walked. The walk-back protocol re-runs §6 from `### Pieces` forward, not the full Architecture section. If walk-back triggers another merge (cascading), the protocol terminates the loop after 2 walk-backs and escalates to user revision via R4.
- **Test failure during R11 audit reveals a rule the trim deletes is still load-bearing.** Spec phase pauses; user is asked whether to retain the rule (modifying the trim plan) or accept the test-rewrite cost.
- **Prescriptive-prose scanner false-positive on legitimate narrative reference.** Per R6 worked examples and anti-patterns: bare paths without `:line` do NOT match; inline backtick code references do NOT match. If a R12 re-walk surfaces a legitimate use case that false-positives anyway, the regex refinement reopens via LEX-2 (one use only per stopping rule).
- **R12 re-walk fails one of the three pass criteria.** Implementation halts. The criterion-to-amendment-surface mapping is explicit: (i) → falsification rule or target value; (ii-a) → LEX-1 regex; (iii) → Edges rubric or R1 authoring guidance. Three failed re-walks across either corpus (summed) escalate to scope-rethink — spec is invalidated, ticket #195 returns to discovery for direction reconsideration.
- **Standalone discovery emits events to `research/{topic}/events.log`; CLAUDE.md MUST-escalation policy expects evidence at `lifecycle/<feature>/events.log`.** This is EVT-2 — the spec acknowledges the limitation: standalone-discovery F-rows are weaker MUST-escalation evidence than lifecycle-attached F-rows. The R9 `resolve-events-log-path` resolver emits to lifecycle-events-log when run under an active lifecycle (the typical case for /cortex-core:refine-invoked discovery), so the MUST-evidence path works for the common path. Standalone-discovery F-rows surface in `research/{topic}/events.log` and can be aggregated manually if MUST escalation is ever needed — accepted limitation.

## Technical Constraints

- **CLAUDE.md MUST-escalation policy**: new soft gates (R3 falsification, R4 approval, R6 prescriptive check, R15 batch-review) ship in positive-routing form. No MUST language without effort=high evidence — verified before commit per the policy's evidence-artifact link requirement. **EVT-2 reconciliation**: when discovery runs under a lifecycle, R9's resolver routes events to `lifecycle/{slug}/events.log` — the standard MUST-evidence path. Standalone-discovery events go to `research/{topic}/events.log` and are weaker MUST-evidence (accepted limitation per Edge Cases).
- **Project.md SKILL.md-to-bin parity**: `bin/cortex-check-prescriptive-prose` requires ≥ 1 in-scope reference. R7's justfile recipe + R6 decompose.md prose reference satisfy this. Unit B's atomic-commit constraint enforces this at commit time.
- **Project.md SKILL.md size cap (500 lines)**: not at risk. SKILL.md is 73 lines; reference files are not capped.
- **Project.md Skill-helper-modules pattern**: `cortex_command/discovery.py` follows the precedent of `cortex_command/critical_review.py` — atomic subcommands fusing validation + mutation + telemetry. Event names register in `.events-registry.md` even though SKILL.md does not contain literal `"event": "<name>"` strings (per project.md L33 caveat).
- **Project.md workflow-trimming bias**: R10 trim is hard-deletion; CHANGELOG entry is required if any retired surface had real consumers (verified zero per the audit in R11). R7-era events are NOT in CHANGELOG scope (already covered by the registry sunset row).
- **Dual-source mirror**: `bin/cortex-check-prescriptive-prose` is canonical; `plugins/cortex-core/bin/cortex-check-prescriptive-prose` is mirrored automatically by `.githooks/pre-commit`.
- **Events registry referential integrity**: `bin/cortex-check-events-registry --staged` validates every `"event": "<name>"` literal in skill prose against the registry. Unit A's atomic-commit constraint (parser update + registry rows + extended enum, all in one commit) prevents the gate from blocking on partial states.
- **Tone/voice policy (docs/policies.md)**: new approval-gate prompts (R4, R15) contain no tone-shaping directives. The user-facing surface is informational, not conciliatory.
- **OQ3 escalation scope**: all behavior failures except tone are MUST-escalation eligible. If post-ship observation shows the falsification gate, approval checkpoint, or batch-review gate is routinely skipped, the cumulative `architecture_section_written` / `approval_checkpoint_responded` events provide the F-row evidence for MUST escalation.
- **Atomicity enforcement**: Units A/B/C are checked at PR-review time, not by a pre-commit hook. The Implementation Topology section above is the contract; reviewers verify it. PR descriptions should name the Unit being landed.
