# Specification: overnight-review-gate-crashes-to-cycle

## Problem Statement

In the overnight run `overnight-2026-06-09-0222`, the post-merge review gate crashed systemically: the review sub-agent exited 1 before writing `review.md`, `parse_verdict` returned the `{"verdict":"ERROR","cycle":0}` sentinel, and the feature was deferred — but its already-merged code was **never reverted**, so unreviewed (and in one case runtime-broken) code shipped on the integration branch under a green-looking PR. Research and an adversarial critical review confirmed this is not one bug but several, all undermining a single invariant — *no unreviewed code reaches `merged` on the integration branch* — plus a diagnosability gap that makes the crash itself invisible. This fix makes the review gate **fail-safe** (on a crash/error/defer the feature's merge is reverted and the feature is surfaced for triage, so unreviewed code does not silently ship), closes additional unreviewed-merge paths, and makes the next crash a one-line diagnosis. It is *containment + diagnosability*, not a root-cause repair: the specific exit-1 cause is unconfirmable from current evidence (the child stderr is discarded), so the crash may recur on the next run — but it can no longer ship unreviewed code, and the next occurrence will carry a diagnosable stderr. Beneficiary: anyone relying on the overnight runner to ship only reviewed work.

## Phases
- **Phase 1: Make the crash diagnosable** — capture the review subprocess's real stderr + exit code so the next occurrence self-diagnoses.
- **Phase 2: Fail-safe rollback on review failure** — revert the feature's *live* merge commit(s) (SHA-anchored, under lock) on every non-APPROVED outcome.
- **Phase 3: Correct handling + surfacing** — a could-not-run review writes a real deferral file, surfaces accurately (reconciled with the revert), and is distinguished from a review that ran and said no.
- **Phase 4: Close every unreviewed-merge path + systemic detection** — route post-test-recovery merges, the repair_completed ff-merge, and the sync `completed` branch through review (no silent skip at any merge-to-`merged` site), and make systemic review crashes trip the existing circuit breaker coherently.

## Requirements

**Priority (MoSCoW)**: Must-have — R1, R2, R3, R4, R6, R8, R10, R11, R12 (the diagnosability deliverable + the "no unreviewed code reaches merged" invariant across *every* merge site + correct surfacing + systemic detection). Should-have — R5 (dead-code removal: correctness-neutral cleanup), R7 (resume-idempotent deferral file: prevents a minor duplicate-file bug), R9 (could-not-run distinguishing marker: triage quality-of-life). No requirement is nice-to-have; everything below was scoped in by the full-trust-fix decision, but R5/R7/R9 do not block the core invariant if deferred.

1. **Capture child stderr in the dispatch_error event**: `dispatch_task`'s error handler records the captured child stderr (`_stderr_lines`, already collected and redacted/capped) and the subprocess `exit_code` into the `dispatch_error` event payload, instead of only the placeholder `error_detail`. Acceptance: a unit test that drives `dispatch_task` into the `ProcessError` branch with seeded stderr lines asserts the emitted `dispatch_error` event payload contains those stderr lines (or their redacted form) and a non-null `exit_code`; `just test` exits 0. File: `cortex_command/pipeline/dispatch.py:~796-813`. **Phase**: Make the crash diagnosable

2. **Every roll-back-able merge site returns its merge SHA**: A feature's merge can be created at more than one site — the primary merge (`outcome_router.py:984`), the cycle-1 rework re-merge (`review_dispatch.py:445`), and the post-test-recovery re-merge (inside `recover_test_failure`, `cortex_command/pipeline/merge_recovery.py`). Each site that produces a merge that may later need reverting captures `git rev-parse HEAD` immediately after its successful `--no-ff` merge and surfaces that SHA to the rollback caller: `MergeResult.merge_sha` (primary), a `merge_sha` carried on `ReviewResult` (rework re-merge), and the recovery result's `merge_sha` (or the post-recovery HEAD captured at the call site under the re-acquired lock). The capture is `None` only when no merge landed. Acceptance: tests assert each of the three result types exposes the live merge SHA after a successful merge at that site, and `None` when the merge did not land; `just test` exits 0. Files: `cortex_command/pipeline/merge.py:30-39,~289`, `cortex_command/pipeline/review_dispatch.py:40-45,~445`, `cortex_command/pipeline/merge_recovery.py`. **Phase**: Fail-safe rollback on review failure

3. **Revert the LIVE merge on every non-APPROVED review outcome, SHA-anchored and under the lock**: On the review-deferred path (`outcome_router.py:1019-1040`) AND the review-crash `except` path (`:1041-1080`), the feature's *current* merge commit (the live SHA from R2 for whichever site last merged it) is reverted via `git revert -m 1 --no-edit <live_merge_sha>` before worktree cleanup, while `ctx.lock` is held. This uses a rewritten SHA-anchored `revert_merge(merge_sha, repo_path=...)` (the current `revert_merge` is dead code that reverts `HEAD` with `check=True` — it is replaced, not merely called); the new form captures the revert result (no `check=True`), and on a non-zero revert runs `git revert --abort` and escalates to a blocking deferral. Acceptance — two complementary checks: (a) a mock-spy unit test asserts `revert_merge` is invoked with the feature's live captured SHA on both the deferred and `except` paths; AND (b) a real-git harness test (extending the existing `RealGitWorktree` test fixture) asserts that after a deferred review the integration branch's tree no longer contains the feature's changes and a revert commit exists. Mock-only tree assertions do not satisfy (b). `just test` exits 0. Files: `cortex_command/overnight/outcome_router.py:984,1019-1080`, `cortex_command/pipeline/merge.py:335-367`. **Phase**: Fail-safe rollback on review failure

4. **Rework-path rollback reverts the live re-merge SHA, not a stale one**: When review defers after a cycle-1 re-merge, the rollback reverts the re-merge's live SHA (threaded via R2 onto `ReviewResult`), not the original first-merge SHA. It accounts for the fact that `merge_feature` may have already reverted the re-merge on a test failure (it logs `merge_revert_error` and returns `success=False` without `check=True`), so it queries actual branch state and does not attempt to revert an already-reverted commit. Acceptance: a real-git harness test driving a cycle-1 re-merge followed by a cycle-2 defer asserts no feature code remains on the integration branch and no failed double-revert occurs; `just test` exits 0. Files: `cortex_command/pipeline/review_dispatch.py:444-465`, `cortex_command/pipeline/merge.py:301-324`. **Phase**: Fail-safe rollback on review failure

5. **Remove the dead sync review-defer branch**: The `_apply_feature_result` review-defer branch (`outcome_router.py:657-680`) is unreachable — `review_result` is initialized to `None` at `:498` and never reassigned. Remove it (or, if retained for a documented reason, annotate it `# unreachable: review runs only in async apply_feature_result`). Acceptance: `grep -c "review_result = " cortex_command/overnight/outcome_router.py` returns 0 (no live assignment) AND either the `:657` branch is gone or carries the unreachable annotation; `just test` exits 0. File: `cortex_command/overnight/outcome_router.py:498,657-680`. **Phase**: Fail-safe rollback on review failure

6. **A could-not-run review writes a blocking deferral file, with surfacing reconciled to the revert**: The live ERROR/deferred path writes a `DeferralQuestion` via `write_deferral` (it currently does not — only the dormant `except` and CI/REJECTED paths do), at `SEVERITY_BLOCKING`, carrying the review verdict. Because Phase 2 now **reverts** the merge, the deferral/report message is reconciled with the post-revert state: when the revert succeeded the feature is no longer on the integration branch, so the surface says "merge reverted — safe to re-review/re-run" rather than the legacy "Feature is on the integration branch — do NOT re-run"; the legacy "do NOT re-run" annotation is reserved for the dependent-conflict abort case (R-edge) where the merge genuinely remains. Acceptance: a test simulating a review-ERROR feature asserts (a) a deferral file is written to the deferred dir, (b) the morning report renders the feature in the Deferred Questions section, and (c) for a successfully-reverted feature the rendered surface does NOT carry the "on the integration branch — do NOT re-run" annotation (which would be false post-revert); `just test` exits 0. Files: `cortex_command/pipeline/review_dispatch.py:300-314` and/or `cortex_command/overnight/outcome_router.py:1019-1040`, `cortex_command/overnight/report.py:~996-1015,978-1025`. **Phase**: Correct handling + surfacing

7. **Deferral-file write is resume-idempotent with a single ID source**: Re-running the defer path for the same feature (e.g., on session resume) does not mint a duplicate `-q00N.md` file, and the review-defer and `except`-crash paths use a single reconciled question-id source (not `question_id=0` scan in one place and `_next_escalation_n` in the other). Acceptance: a test invoking the defer path twice for the same feature asserts exactly one deferral file exists for it; `just test` exits 0. Files: `cortex_command/overnight/deferral.py:177-195`, `cortex_command/pipeline/review_dispatch.py:610`, `cortex_command/overnight/outcome_router.py:1052`. **Phase**: Correct handling + surfacing

8. **Review-deferred backlog write-back uses status `deferred`**: The review-deferred path writes backlog status `deferred` (not `in_progress`, which is not even a valid OvernightState status and reads as ordinary active work). Acceptance: a test asserts the backlog write-back on the review-deferred path is called with `"deferred"`; `just test` exits 0. File: `cortex_command/overnight/outcome_router.py:1031-1035`. **Phase**: Correct handling + surfacing

9. **Distinguish could-not-run from review-ran-and-said-no**: The `FEATURE_DEFERRED` event for a review that could not run (verdict `ERROR`) carries a distinguishing marker (e.g., `review_dispatch_crashed: true` / `could_not_run: true`) so morning triage can separate an infra crash from a substantive REJECTED/CHANGES_REQUESTED. Detection keys off the verdict **string** (`ERROR`), not `cycle` (the `cycle:0` value is cosmetic — every consumer keys off the verdict string). Acceptance: a test asserts the `FEATURE_DEFERRED` event for an ERROR outcome carries the distinguishing flag and a REJECTED outcome does not; `just test` exits 0. Files: `cortex_command/pipeline/review_dispatch.py:48,300-314`, `cortex_command/overnight/outcome_router.py:1024-1030`. **Phase**: Correct handling + surfacing

10. **Test-recovery success routes a review-qualifying feature through review — no silent skip**: The post-merge test-recovery success branches (`outcome_router.py:~1234` flaky, `~1253` recovered) invoke `dispatch_review` for every feature where `requires_review(tier, criticality)` is true, before marking `merged`. A review-qualifying feature is **never** marked `merged` on this path without either an APPROVED review or a revert+defer; the only path that skips review is a feature for which `requires_review` is already false (a legitimate non-review, logged as such — not a blanket escape). If review on this path crashes/defers, the recovery re-merge is reverted (R2/R3) and the feature is deferred. Acceptance: a test with a complex/high feature that passes only after test-recovery asserts `dispatch_review` is invoked on the recovery success path, and a test asserting that a review-qualifying feature on this path is *not* marked `merged` without an APPROVED verdict (a skip-with-rationale is accepted only when `requires_review` is false); `just test` exits 0. File: `cortex_command/overnight/outcome_router.py:~1217-1260`. **Phase**: Close every unreviewed-merge path + systemic detection

11. **Systemic review crashes trip the existing circuit breaker coherently**: A review-dispatch crash deferral increments the systemic-failure counter (`cb_state.systemic_pauses_in_batch`) AND is recorded in the structure the systemic-failure block reads to compute its trailing `cause_class` (the block at `outcome_router.py:893-908` currently derives `cause_class` from `features_paused[-N:]`; review crashes route to `features_deferred`, so the wiring must ensure the emitted event is coherent — a non-empty `cause_class`, not just a flipped flag). When `SYSTEMIC_FAILURE_THRESHOLD` identical review crashes occur in a batch, `global_abort_signal` is set `True` and a `PIPELINE_SYSTEMIC_FAILURE` event is emitted carrying a non-empty cause. Acceptance: a test simulating `SYSTEMIC_FAILURE_THRESHOLD` systemic review crashes asserts `global_abort_signal` becomes `True` AND the emitted `PIPELINE_SYSTEMIC_FAILURE` event carries a non-empty `cause_class`; `just test` exits 0. Files: `cortex_command/overnight/outcome_router.py:576-607,893-908`. **Phase**: Close every unreviewed-merge path + systemic detection

12. **Close the remaining unreviewed-merge paths (uniform review-or-revert at every merge-to-`merged` site)**: The post-merge review gate currently lives only on the async primary-merge path (`outcome_router.py:1006`). Two other sites transition a feature to `merged` with no review: the `repair_completed` ff-merge of a resolved-conflict branch (`outcome_router.py:524-560`, `git merge --ff-only` → `features_merged` → write-back `"merged"`), and the sync `_apply_feature_result` `completed` branch (`:592` → `merge_feature` at `:646` → `features_merged` at `:681`). For any feature where `requires_review(tier, criticality)` is true that reaches `merged` via either site, dispatch review before marking `merged`, and on a non-APPROVED/crash outcome revert (R2/R3) + defer — the same gate as R10. If implementation determines a site is provably unreachable for review-qualifying features (as the dead branch in R5 is), document that with an `# unreachable`-style annotation instead of gating it; the requirement is that no review-qualifying feature reaches `merged` un-reviewed via any path, not that dead code be decorated. Acceptance: a test asserts a review-qualifying feature reaching `merged` via the `repair_completed` ff-merge path is routed through `dispatch_review` (or reverted+deferred on non-APPROVED), and the same for the sync `completed` path if reachable; OR, for any site proven unreachable, the code carries an unreachable annotation and a test pins that the async primary path is the sole live review-qualifying merge site; `just test` exits 0. Files: `cortex_command/overnight/outcome_router.py:524-560,592-698`. **Phase**: Close every unreviewed-merge path + systemic detection

## Non-Requirements

- **Confirming the specific exit-1 root cause in this lifecycle.** The cause is not confirmable without the captured stderr (R1 delivers it) or a live detached repro; the modern CLI tolerates a bad `--effort` value (warns, exits 0), and the auth hypothesis is weakened. Final confirmation is deferred to the next overnight run's captured stderr. The crash may recur on the next run; what changes is that it can no longer ship unreviewed code and will be diagnosable. (User decision: don't overengineer.)
- **No startup auth/CLI canary and no `--effort` validation/patch.** Speculative defensive layers for an unconfirmed cause are explicitly out of scope (user decision: "don't overengineer defensive programming").
- **No retry-before-defer.** Disclosed cost: with no retry, a *single transient* review-dispatch flake (not just a systemic one) now triggers a full merge→revert→blocking-deferral of an otherwise-good feature — a wasted merge cycle and a false `SEVERITY_BLOCKING` deferral a human triages. This is accepted because the observed failure was systemic (byte-identical across features), not a transient flake, and the user chose not to add retry machinery; the circuit breaker (R11) handles the systemic case, and the per-flake cost is a rare, safe (never-ships-unreviewed) over-defer.
- **No move-merge-after-APPROVED restructure.** Switching to gate-before-merge collides with the rework loop's mid-review re-merge and the post-merge test-gate timing — an architecture change disproportionate to this fix. File as a follow-up if the merge-then-review model proves insufficient.
- **`report.py` whole-lifecycle-tree globbing** is out of scope — tracked separately as ticket #294.
- **No new terminal-status vocabulary.** `deferred` already covers the outcome; `common.py:TERMINAL_STATUSES` is unchanged (avoids the lifecycle-gated path).

## Edge Cases

- **Dependent-feature revert conflict** (R-edge): reverting feature X's merge when a later feature Y merged code that textually depends on X produces a revert conflict. Behavior: `git revert --abort`, then write a blocking deferral that names Y as now referencing reverted code. In this case the merge genuinely **remains** on the branch, so the surface uses the legacy "still on the integration branch — do NOT re-run, manual rollback needed" annotation (the one case where it is accurate). This is the one acknowledged path where a defer leaves code merged — it is surfaced as a blocking deferral, not shipped silently.
- **Pre-spawn raise routes to the `except` path**: a non-canonical tier (`KeyError` in the model/effort matrix) or `resolve_effort` `ValueError` propagates out of `dispatch_review` (no try/except around the matrix lookups) to `outcome_router.py:1041`. Both the `rr.deferred` and `except` paths revert + surface (R3, R6) — neither leaves code merged.
- **Resume after a partial review-defer**: the deferral-file write is idempotent (R7); the rollback is a forward revert commit that nets out the bad merge and does not double-apply on re-run.
- **Rework re-merge already reverted by merge_feature on test failure**: the rollback queries live branch state and does not attempt to revert an already-reverted commit (R4).
- **Systemic crash short of threshold**: fewer than `SYSTEMIC_FAILURE_THRESHOLD` review crashes do not trip the breaker; each is individually rolled back, deferred, and surfaced (fail-forward), and appears in the morning report.

## Changes to Existing Behavior

- MODIFIED: a review-deferred/crashed/errored feature now **reverts its live merge commit** (was: code left on the integration branch).
- MODIFIED: a could-not-run (ERROR) review now **writes a blocking deferral file** and surfaces in the report + PR (was: a bare `FEATURE_DEFERRED` event with no file, invisible in the report and on a green PR).
- MODIFIED: the deferral/report surface for a reverted feature now says **"reverted — safe to re-review"** instead of the legacy "on the branch — do NOT re-run" (which becomes false after the revert; reserved for the dependent-conflict abort case).
- MODIFIED: review-deferred backlog write-back uses status **`deferred`** (was: `in_progress`).
- MODIFIED: the `dispatch_error` event now carries the **child stderr + exit code** (was: a placeholder string only).
- MODIFIED: post-test-recovery merged features now go **through review** when qualifying, with no silent skip (was: review skipped entirely → unreviewed code shipped).
- MODIFIED: the `repair_completed` ff-merge and the sync `completed` branch now route review-qualifying features **through review-or-revert** (was: marked `merged` with no review) — closing every merge-to-`merged` path uniformly.
- MODIFIED: systemic review crashes now **trip the circuit breaker** with a coherent cause and pause the session (was: invisible to the breaker; whole batch burned).
- REMOVED: the dead sync `_apply_feature_result` review-defer branch (or annotated unreachable).
- REPLACED: `revert_merge` is rewritten from HEAD-anchored dead code to a SHA-anchored, capture-and-handle helper.

## Technical Constraints

- **All git reverts — and the recovery re-merge that R10 review-gates — run while `ctx.lock` is held.** The home integration worktree is a single physical checkout shared across concurrently-running features; concurrent `git checkout`/`merge`/`revert` would corrupt the index. The lock is currently *released* across the `recover_test_failure` call (`outcome_router.py:~928`); R10's review + R2's recovery-SHA capture + R3's revert on that path must execute under the re-acquired lock. SHA-anchoring (not `revert -m 1 HEAD`) is required because concurrent features advance `HEAD` to another feature's merge.
- **`revert_merge()` is rewritten, not reused.** The existing `merge.py:335` form reverts `HEAD` with `check=True` and has zero callers; the new form takes a SHA, reverts that specific commit, and captures-and-handles (abort + escalate on conflict). There is no live "fourth revert site" to consolidate — this is a behavior change of dead code.
- **Reuse existing registered events** `merge_reverted` / `merge_revert_error` for the revert; if a distinguishing event/field is added, register it in `cortex_command/overnight/events.py` + `bin/.events-registry.md` in the same commit. The events-registry drift gate hard-codes producer line numbers for `review_verdict` (`review_dispatch.py`) and `dispatch_error` (`dispatch.py`); inserting code shifts those lines, so the registry rows must be updated in the same commit or the pre-commit gate fails.
- **Preserve the defer-on-failure must-have** (`pipeline.md`): the fix *adds* rollback and *keeps* defer + deferral file; it does not replace defer with a silent halt. The repair-attempt-cap is unchanged.
- **Test fidelity**: the rollback acceptance criteria (R3b, R4) require a real-git harness (extend the existing `RealGitWorktree` fixture) — mock-only tests of `merge_feature`/`subprocess.run` cannot observe branch-tree state and would be self-sealing. Mock-spy tests (R3a) verify the call contract; real-git tests verify the tree outcome.
- **Sequencing**: Phase 1 (stderr capture) lands first and independently (low-risk, prerequisite). Phase 2 (rollback) must land with the lock-scoping and the R2 SHA-threading across all three merge sites; it must not introduce a fresh corruption race. Phases 2 and 4 share the "where does code reach `merged`" surface — verify the invariant holds on the defer path and the recovery path before declaring the gate trustworthy.
- **Requirement-vs-code drift to reconcile during implementation**: `pipeline.md` describes a synthetic "morning-review skip `APPROVED, cycle:0`" event that is not emitted in code; do not build any handling that "preserves" it. Handling keys off the verdict string only.
