# Plan: overnight-review-gate-crashes-to-cycle

## Overview
Make the overnight post-merge review gate fail-safe and diagnosable: capture the review subprocess's real stderr (Phase 1), thread each merge site's live commit SHA and revert that SHA under the lock on every non-APPROVED outcome (Phase 2), write a real reconciled blocking deferral on a could-not-run review (Phase 3), and close every remaining merge-to-`merged` path so no review-qualifying feature ships unreviewed, with systemic review crashes tripping the circuit breaker (Phase 4). This is containment + diagnosability over a merge-then-review (TAP-model) pipeline, **introducing** a SHA-anchored revert (replacing the dead, HEAD-anchored `revert_merge`) rather than restructuring to merge-after-approval. All git mutations on the shared integration worktree — including the recovery re-merge — run under `ctx.lock`.

**Architectural Pattern**: pipeline

## Outline

### Phase 1: Make the crash diagnosable (tasks: 1)
**Goal**: The next review-dispatch crash self-diagnoses from its own event payload.
**Checkpoint**: A `dispatch_error` event carries the child's real stderr + exit code; `just test` green.

### Phase 2: Fail-safe rollback on review failure (tasks: 2, 3, 4, 5)
**Goal**: Every non-APPROVED review outcome reverts the feature's live merge under the lock; dead revert code is replaced, dead branch removed.
**Checkpoint**: Real-git harness tests prove a deferred review leaves no feature code on the integration branch via both the primary and rework re-merge paths.

### Phase 3: Correct handling + surfacing (tasks: 6, 7)
**Goal**: A could-not-run review writes a blocking deferral file, surfaces accurately reconciled to the revert, uses `deferred` backlog status, and is distinguished from review-ran-and-said-no — idempotently on resume.
**Checkpoint**: A review-ERROR feature renders in the morning report's Deferred Questions without a false "do NOT re-run" annotation, with exactly one deferral file on re-run.

### Phase 4: Close every unreviewed-merge path + systemic detection (tasks: 8, 9, 10)
**Goal**: Every merge-to-`merged` site routes review-qualifying features through review-or-revert; systemic review crashes trip the breaker coherently.
**Checkpoint**: Tests pin that the test-recovery path, the `repair_completed` ff-merge, and the sync `completed` branch never mark a review-qualifying feature `merged` without an APPROVED verdict, and `SYSTEMIC_FAILURE_THRESHOLD` review crashes set `global_abort_signal` with a non-empty `cause_class`.

## Risks

- **`outcome_router.py` is the contention hotspot, serialized structurally via `Depends on` edges (not prose)** — Tasks 3, 4, 5, 6, 7, 8, 9, 10 all edit `outcome_router.py`. The implement phase computes parallel-dispatch batches from `Depends on` alone (topological level grouping) and merges worktrees back with no rollback on conflict, so any two same-level tasks editing this file would corrupt each other. To prevent that, the `Depends on` graph encodes a strict linear spine through every `outcome_router.py`-editing task — 3 → 4 → 5 → 6 → 7 → 8 → 9 → 10 — by adding **serialization edges** beyond the logical ones: Task 5 ← [4], Task 6 ← [3,**5**], Task 8 ← [2,3,6,**7**] (the bolded deps are serialization-only). This places each of these tasks in its own batch level, so none is ever parallel-dispatched into a worktree concurrent with another `outcome_router.py` editor. Tasks 1 and 2 (which do not touch `outcome_router.py`) remain free to run in parallel at level 0. This replaces the earlier prose-only "must run sequentially" note, which the batcher could not act on — encoding the constraint where the dispatcher reads it (per CLAUDE.md "structural over prose-only enforcement").
- **No retry-before-defer** (spec Non-Requirement) — a single transient review-dispatch flake now triggers a full merge→revert→blocking-deferral of an otherwise-good feature. Accepted: the observed failure was systemic, the circuit breaker (Task 10) handles the systemic case, and the per-flake cost is a rare, safe over-defer. If the team later wants retry, it is a follow-up, not a re-plan of this work.
- **Root cause is not confirmed in this lifecycle** — Phase 1 delivers diagnosability (captured stderr) but does NOT fix the exit-1 cause (top candidate: `--effort xhigh` rejected by an outdated CLI; auth-readability secondary).
