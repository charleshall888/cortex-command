## Problem Statement

#360 is the critical-review child of #357's provisional-tail sweep of the skill-value audit. It trims 26 scored-but-never-adversarially-verified candidates from `skills/critical-review/` (~4.9k weighted tokens) to reduce the per-run token cost of a shared, lifecycle-invoked skill, and completes one carried cross-site parity residual in `critical-review/SKILL.md`. The value is maintainability-through-simplicity on a hot path (reviewer prompts render 3–4× per critical-review run); the risk is that this skill gates lifecycle spec/plan approval, so an over-trim has lifecycle-wide blast radius. Every trim is gated by verification before it applies; candidates that fail verification are recorded refuted so they are not re-proposed.

## Phases

- **Phase 1: Safe body & orchestrator-only trims** — verify-and-apply the low-coupling candidates (SKILL.md body, angle-menu.md, residue-write.md) plus the cortex-resolve-model parity trim.
- **Phase 2: Dispatched prompt-template trims** — verify-and-apply the high-coupling candidates (reviewer-prompt.md, synthesizer-prompt.md, fallback-reviewer-prompt.md) against the synthesizer parse contract, with behavioral (run-and-observe) verification because no automated test exercises these reference files.
- **Phase 3: Landmine dispositions, correction & handoff** — refute/correct the landmine candidates, run the batch integration gate, write the verdict record, and wire #357 to consume it.

## Requirements

**Priority (MoSCoW)**: All requirements are **Must** — each is a verify-then-apply-or-refute of an in-scope candidate, a landmine disposition, or the batch's integrity gate; the batch is not coherent with any dropped. There is deliberately **no Should/Could tier**: a candidate that fails verification becomes a recorded refute (still Must), not a silent skip. The **Won't** tier is `## Non-Requirements`.

**Verification bar (applies to every "apply" requirement)** — a trim applies only when, in one pass: (1) every keep-list token still greps present in the edited file; (2) the removed phrase's count actually dropped / the file net-shrinks (two-sided check); (3) the candidate's *file-appropriate* pinned pytest passes (a test that actually reads the edited file — see the per-requirement criteria; SKILL.md-reading tests do NOT gate reference-file trims); and (4) after `just build-plugin`, `diff -rq skills/critical-review/ plugins/cortex-core/skills/critical-review/` is empty. **The bar proves token survival, not behavioral equivalence** (Edge Cases); for Phase-2 candidates it is therefore supplemented by a run-and-observe check (Reqs 7–8).

**Keep-list construction & its limit**: the keep-list is built per candidate from its `claim` keep-language + every `pins` literal + every `mech_pins` token in the span, and recorded in the verdict record (Req 13). For candidates overlapping a prior lifecycle's adjudicated span, it is **seeded from the prior `verifier_reason`/`downgrade_to`** in `cortex/lifecycle/adversarially-verified-trim-of-critical-review/trim-map.md`. Because `claim` is authored by the same audit that proposed the trim, the keep-list cannot catch a load-bearing surface the audit failed to recognize — this residual risk is why Phase-2 adds run-and-observe and why the highest-coupling trims are isolated and verified first.

## Edge Cases

- **A candidate's declared span contains a pinned literal it doesn't intend to cut** (e.g. s2's span 9–28 contains the pinned exit-2 reaction): trim only the claim-scoped sub-lines, never the whole declared span; the pinned pytest is the backstop.
- **Over-cut into an adjacent excluded/guard span in a shared file**: there is **no automated line-range gate** (the pins test does not byte-lock the guard prose at lines 37/49/55/80/86). Enforcement is execution discipline: edit only the claim-scoped lines, and after each verification-gates.md edit run `git diff skills/critical-review/references/verification-gates.md` and confirm every changed hunk falls within the authorized region (with s2 refuted by default, the only authorized vgates edit is the line-33 correction, Req 11) — abort and revert if any hunk strays. This is a manual self-check, not a structural gate; it is called out here precisely because the structural gate does not exist.
- **Pin greps present but surrounding prose degraded** (token-survival ≠ behavioral equivalence): the general condition of Phase-2 trims, not a one-off — mitigated by the run-and-observe checks (Reqs 7–8), keep-list-from-prior-verdict seeding for adjudicated spans, and isolating the highest-coupling trims first. s3(vgates) is the proof case (a factually wrong line survived a prior two-pass review) and is handled as a correction, not a trim.
- **A candidate fails verification mid-batch**: record it `verified_refuted` with rationale and move on — refuted candidates are not re-proposed.
- **Commit blocked by whole-tree drift**: because `just build-plugin` regenerates the entire cortex-core mirror tree and the pre-commit Phase-4 drift check is whole-tree, a pathspec-scoped commit blocks if other canonical critical-review files are edited-but-unstaged in the working tree — see the commit-ordering constraint below.

## Technical Constraints

- Hard pins (`tests/test_critical_review_reference_pins.py`): total-failure literal, `## Step 2a.5/2c.5/2d.5:` designators, exit-2 reaction string, per-section `- **Exit 0/3/4**` markers — preserved byte-exact.
- L1 surface ratchet: critical-review frontmatter at 795B / 795 budget (zero headroom) — no frontmatter edits.
- **Test coverage is uneven**: `test_dispatch_template_placeholders.py` and `test_critical_review_classifier.py` read **SKILL.md**, not the reference prompt files; the classifier tests are `@pytest.mark.slow` and skipped unless `--run-slow`. Therefore reference-file trims (Reqs 7–9) get no unit-test gate and rely on byte-exact grep + run-and-observe; SKILL.md-template trims (Reqs 3, 8-adjacent) must invoke `--run-slow` to actually assert.
- Mirror parity: canonical-only edits; `just build-plugin` regenerates `plugins/cortex-core/`; commit both together (`tests/test_plugin_mirror_parity.py`, `tests/test_dual_source_reference_parity.py`).
- **Commit ordering (required)**: because the pre-commit hook runs `just build-plugin` (whole-tree rsync) then a whole-tree drift check, work **one file at a time**: edit the canonical file → `just build-plugin` → stage the canonical file *and* its regenerated mirror → `/cortex-core:commit` → next file. Do not leave multiple canonical files edited-but-unstaged, or the whole-tree drift loop blocks the pathspec-scoped commit. Use explicit pathspecs to avoid leaking concurrent-session staged files.
- `bin/.contract-lint-exceptions.md:32` carries an intentional-omission exception for critical-review/SKILL.md subcommand names — preserve or re-justify if a trim touches that passage.
- Reviewer prompts render 3–4×/run; synthesizer 1× (Opus); the `<!--findings-json-->` envelope is orchestrator-LLM-parsed (no Python parser) — prompt-template trims verified against the sentinel regexes in `cortex_command/critical_review/__init__.py` and, behaviorally, via run-and-observe.
- Commit only via `/cortex-core:commit`.
