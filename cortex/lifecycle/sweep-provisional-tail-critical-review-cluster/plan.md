## Risks
- **Commit-ordering serialization kills parallelism.** Tasks 1–7 are serialized because the *implement runner's shared-working-tree batch dispatcher* collides with the whole-tree `just build-plugin` + drift check: a second canonical critical-review file edited-but-unstaged in the shared tree blocks the pathspec-scoped commit. The constraint is on the commit boundary under a shared tree — **not** categorically "inherent to the mirror tree." A theoretical alternative exists (per-task `Agent isolation:"worktree"`, since the seven tasks touch disjoint files/mirrors and would merge conflict-free), but it is not wired into per-task implement dispatch and would *multiply* the partial-edit poisoning hazard across N trees; the safe direction here is serialization + intra-task checkpointing (Task 1 recovery-reset, Tasks 2–7 clean-tree precondition), not cross-task parallelism. The `Depends on` edges therefore encode required serialization (mutual exclusion on the shared tree), not data flow.
- **Verification proves token survival, not behavioral equivalence.** Pin/grep survival can pass while surrounding prose degrades. `SKILL.md` (Task 1) is the orchestrator prompt rendered every run — its two no-pin cuts (s8/s16) are load-bearing orchestrator steps, so Task 1 carries a (c) semantic-read confirm that Step 2b/Step 4 keep their operative control flow. The two dispatched prompt files (Tasks 4–5) get run-and-observe. The remaining orchestrator-read files (Tasks 2–3, 7) rely on grep + pinned pytest + keep-token presence because they never render into a dispatched agent and their operative tokens are pinned. **Caveat:** the run-and-observe checks (Tasks 4–5) and the semantic-read confirms (Tasks 1, 9) are Interactive/session-dependent — under autonomous overnight execution they may be deferred. These tasks must **halt rather than mark complete on grep alone** if the executing context cannot perform the behavioral/read confirm; grep+pytest is not a sufficient substitute for the behavioral gate on these specific candidates.
- **verification-gates.md has no automated line-range gate.** Over-cutting into the unpinned guard prose (37/49/55/80/86) or the 3 excluded `reproposal_of` spans stays grep-green against the pins test alone. Task 7 now adds a **structural backstop** — capture the guard-prose and reproposal_of-span literals pre-edit and assert each still greps present byte-exact post-edit (a positive presence check), supplementing the manual per-hunk diff read. This is stronger than prose-discipline but still not a byte-lock of the exact line ranges; the residual risk is a cut that both removes an excluded literal and re-adds an identical string elsewhere, judged negligible for a single-line trim.
- **Keep-list conflict of interest.** The keep-list is built by the same pass proposing each trim, so it cannot catch a load-bearing surface the audit failed to recognize. Mitigated by seeding adjudicated-span keep-lists from the prior lifecycle's `verifier_reason`/`downgrade_to` and by isolating/verifying the highest-coupling trims in their own phase.
