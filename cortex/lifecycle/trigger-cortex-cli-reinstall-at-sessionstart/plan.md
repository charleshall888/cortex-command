# Plan: trigger-cortex-cli-reinstall-at-sessionstart

## Overview

Ship a SessionStart drift-detector hook (bash trampoline + embedded Python heredoc) that regex-parses `plugins/cortex-overnight/server.py` for `CLI_PIN`, probes the installed CLI, and emits `additionalContext` on drift — visibility-only, no reinstall. Bundle a latent `--refresh-package cortex-command` flag fix into `_run_install_and_verify` so force-pushed release tags actually reinstall cleanly, propagate the flag to all user-facing remediation strings, and update docs.

**Critical-review-revised**: the original plan extracted `CLI_PIN` to a `cli_pin.py` sibling; critical-review found this would break 4+ regex-based callers (`release.yml` cli-pin-lint, `auto-release.yml:138` git-add, `test_release_artifact_invariants.py`, `bin/cortex-rewrite-cli-pin`). Since the hook can regex-parse `server.py` for `CLI_PIN` at sub-millisecond cost (consistent with existing readers), extraction is dropped. The original plan also named `--refresh` (refreshes ALL cached data per uv docs); revised to `--refresh-package cortex-command` (narrower, matches stated intent without invalidating transitive PyPI caches).

## Outline

**Goal**: Ship a SessionStart hook that probes the installed CLI against `CLI_PIN[0]` (regex-parsed from `server.py` at hook-time) and emits `additionalContext` to Claude on drift. Honors dev-mode skip predicates, a 30-minute freshness throttle (sentinel-check BEFORE stdin read for minimal throttle-hit cost), schema-floor parity (wheel-only), and defensive `exit 0` on every error path. Wiring lands together with the script per W003.
**Checkpoint**: `tests/test_cli_version_sync_hook.sh` exits 0 covering all six required scenarios; `jq '.hooks.SessionStart | length' plugins/cortex-overnight/hooks/hooks.json` = 2; `just build-plugin cortex-overnight` produces a byte-identical mirror.

**Goal**: Close the latent stale-cache bug in the existing Layer-2 reinstall path so force-pushed release tags actually pull the new wheel. Use `--refresh-package cortex-command` (narrow scope per uv docs) rather than `--refresh` (which would refresh ALL cached data including transitive PyPI deps). Propagate the flag to all user-facing remediation strings so the manual recovery path matches the auto-recovery argv.
**Checkpoint**: `grep -E "uv.*tool.*install.*--reinstall.*--refresh-package" plugins/cortex-overnight/server.py` returns ≥1 match in `_run_install_and_verify`; `tests/test_no_clone_install.py` exits 0 with new positive assertion that `--refresh-package` is in the install argv; all four user-facing remediation strings (server.py:184, server.py:840, server.py:1864, cli.py:327) include the new flag.

## Risks

- **Hook regex-parses server.py for CLI_PIN**: the hook reads ~2000 lines of `server.py` on every throttle-miss (sub-millisecond cost) and extracts CLI_PIN via stdlib regex. This is consistent with three existing readers (`bin/cortex-rewrite-cli-pin`, `release.yml` cli-pin-lint, `tests/test_release_artifact_invariants.py`) that all anchor on the same `^CLI_PIN\s*:.*=\s*\(` shape. If anyone refactors that line (adds a docstring, splits across multiple lines, or moves it to a typed assignment), four readers break in lockstep — including the hook. Mitigation: the regex shape is enforced by `bin/cortex-rewrite-cli-pin`'s pre-commit pass on every release bump; deviation is detected and blocked at commit time before any reader breaks.
- **Bash heredoc carries ~150 lines of Python**: Task 3 embeds the Python helper as a heredoc inside the bash trampoline (~230-line file total). shellcheck can't lint the Python; pyflakes can't lint the bash. Editor syntax highlighting doesn't switch languages inside heredocs. Mitigation: Task 3's verification includes a `python3 -c` compile check of the extracted heredoc body, so syntactic errors in the Python are caught at gate-time (not just `bash -n`). The architectural tradeoff is acknowledged here, not in a future-deferred note.
- **30-minute throttle window is opinionated**: the user picked 30 min during refine. Real-world drift detection may lag up to 30 min after marketplace update. The sentinel mtime is user-deletable via `rm ${XDG_STATE_HOME}/cortex-command/last-version-check` for immediate next-session re-probe.
- **Throttle-miss latency**: ≤200ms warm cache, ≤500ms cold cache. SessionStart hooks block Claude's first-turn input; this latency is paid every 30 min per session. Acceptable per Anthropic's hook-cost guidance ("anything over 100ms adds noticeable latency" — we're 2-5× above that ceiling on the throttle-miss path, but only every 30 min, not every session).
- **Hook fires before plugin marketplace fast-forward in some Claude Code versions** (#19491 race): research surfaced this. If the hook reads the pre-update plugin's `CLI_PIN`, it sees no drift even when drift exists. Acceptable: the hook is a visibility-only mechanism; the existing `_ensure_cortex_installed` on the next MCP call still catches and fixes drift. The hook's value degrades gracefully under this race.
- **`--refresh-package` semantics depend on uv version**: the flag must be supported in the user's installed `uv`. The hook does not pin a minimum uv version. If a user has a very old `uv` that lacks `--refresh-package`, the install will fail with a flag-not-recognized error and surface in the existing NDJSON audit. Acceptable: uv version drift is a separate operational concern, surfaced via stderr.
