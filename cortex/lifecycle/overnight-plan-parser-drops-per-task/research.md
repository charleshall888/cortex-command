# Research: Overnight plan parser drops per-task Files and Depends-on metadata, collapsing dependency ordering

> Lifecycle: `overnight-plan-parser-drops-per-task` · tier: complex · criticality: high
> Researched: 2026-06-09 · 8 agents (3 core + 4 chosen + adversarial)

## Executive summary (orchestrator synthesis)

The ticket frames this as "the parser is too strict for a bold-paragraph dialect that auto-generated plans emit; relax the parser." Research **partly refutes that framing** and sharpens the fix:

- **The fault axis is not "bold-paragraph."** The parser's per-task field regexes require `[-*]\s+\*\*Label\*\*:` — a **leading bullet** with the **colon outside the bold**. Any deviation silently returns `[]`: no leading bullet (`**Files**: …`), colon inside the bold (`**Files:**`), or both. The discriminator is "does the strict regex match," not one named dialect.
- **The bug is live and in-repo, and generator-source-agnostic.** `cortex/lifecycle/scale-research-fanout-by-complexity/plan.md` (interactively authored) declares real dependencies (`**Depends on**: 1`, `1, 2`, …) with no leading bullet across Tasks 1–11. I verified the parser drops all of them to `[]` today. So this is not a cross-repo one-off — though it is rare (1 active non-archive plan).
- **Root cause on the generation side is an under-specified prompt.** Overnight per-feature plans are LLM-authored from `cortex_command/overnight/prompts/orchestrator-round.md` §3b (lines ~404–428), which *paraphrases* the field names but never pins the literal `- **Files**:` syntax and never points at the canonical template `skills/lifecycle/references/plan.md`. The interactive path shows the literal block; overnight does not. The overnight path also does not run the interactive `orchestrator-review` P2 dependency-graph gate.
- **The dangerous mechanism is silent-`[]` → vacuous-true collapse.** `compute_dependency_batches` (`common.py:670-706`, line 695 `all(d in assigned for d in t.depends_on)`) treats empty `depends_on` as "no prerequisites," collapsing every task into one concurrent batch dispatched via `asyncio.gather` (`overnight/feature_executor.py:728`).
- **The durable fix is a dialect-independent fail-loud invariant**, not (only) relaxing the regex. Relaxing for today's variants leaves the next unrecognized form to fail silently again; fixing the generator prompt reduces frequency but (per the project's MUST-escalation policy) prose instruction can never be the integrity guarantee. The adversarial pass converged on: **if a task body contains a recognizable Files/Depends-on label but the parser extracts no value, raise — never return `[]` silently.** This is cheap operationally (one feature fails, circuit-breaker-exempt, nothing merged) and catches every present and future drop including partial-drop and colon-inside.
- **The ticket's suggested fix #3 (warn when a multi-task plan yields one zero-dependency batch) is the weakest option** and should be demoted: it has a guaranteed false-positive (legitimately fully-parallel plans produce the identical single-batch shape) and a large false-negative (partial drops produce multiple batches and slip past it). It also warns *after* the broken-ordering plan already ran and merged.

## Codebase Analysis

**Runtime consumption call chain (plan.md text → parse → batch → dispatch):**
1. `parse_feature_plan(path)` — `cortex_command/pipeline/parser.py:246`
2. `_parse_tasks(text, path)` — `parser.py:299`; normalizes heading separators via `_normalize_task_separators` (`parser.py:284-296`), matches `### Task N` headings (`parser.py:314-317`), slices each task body.
3. Per-task field extraction — `parser.py:338-349`: `_parse_field_files` (`parser.py:363`), `_parse_field_depends_on` (`parser.py:384`), `_parse_field_string` (`parser.py:408`).
4. `FeatureTask` dataclass — `parser.py:54-64`; `files`/`depends_on` default to `[]` (so no-match → empty).
5. Consumer: `overnight/feature_executor.py:567` parses; `:596` calls `compute_dependency_batches`; `:607` iterates batches; `:728-731` `asyncio.gather`s each batch concurrently. (`feature_executor.py` lives under `cortex_command/overnight/`, **not** `pipeline/` — ticket path was slightly off.)

**The regexes, verbatim:**
- `_parse_field_files` (`parser.py:365`): `r"[-*]\s+\*\*Files\*\*:\s*(.+?)(?:\n[-*]\s+\*\*|\n###|\n##|\Z)"` with `re.DOTALL` — multi-line tolerant. No-match → `[]` (never raises).
- `_parse_field_depends_on` (`parser.py:388`): `r"[-*]\s+\*\*Depends\s+on\*\*:\s*(.+)"` — single-line (no DOTALL). No-match → `[]`. **But** if a bullet matches and the value has no digits and isn't "none" → **raises `ValueError`** (`parser.py:400-403`). Digits scraped via `re.findall(r"\d+", raw)`.
- `_parse_field_string` (Complexity) — same single-line shape; no-match → `None`.

**Load-bearing asymmetry:** `_parse_field_files` is multi-line tolerant (DOTALL); `_parse_field_depends_on`/`_parse_field_string` are single-line. And `_parse_field_depends_on` already fails loud on *present-but-unparseable* but stays silent on *no-match* — the no-match path is the bug.

**The collapse, quoted** (`common.py:695`): `batch = [t for t in pending if all(d in assigned for d in t.depends_on)]`. With every `depends_on == []`, `all(... for d in [])` is vacuously `True`, so the first iteration selects all tasks → one batch → full concurrency, no ordering, **no error** (the collapse is silent; only a genuine cycle raises).

**Conventions to follow:** add a `FeatureTask` field via default-factory; write a `_parse_field_*` helper; the module does no logging — it records coercions in a side channel (`normalized_complexities`, `parser.py:344-348`) that the caller surfaces as events (`feature_executor.py:582-589`). Consumer catches `(FileNotFoundError, ValueError)` → `status="failed"` (`feature_executor.py:567-574`), so a new `raise` becomes a single-feature failure, not a crash.

## Web Research

- **Postel's robustness principle is the wrong default here.** The IETF "Harmful Consequences of the Robustness Principle" (draft-thomson-postel-was-wrong / draft-iab-protocol-maintenance) argues leniency is defensible only for *frozen specs you cannot fix*. When you **control the generator** (an LLM you prompt), a tolerant parser *entrenches* drift: the producer never learns it's malformed, the parser masks it, downstream loses data. Recommended: active maintenance + **strict validation with visible (fatal) failures** as the feedback signal.
- **Reliable LLM-structured-output hierarchy:** constrained decoding (strongest) → schema-as-contract + post-gen validation that *rejects* malformed output + bounded repair loop + drift-rate monitoring → normalize only **enumerable, known-equivalent** variance. Treat schema drift as a generator/contract problem to catch-and-repair, not silently absorb.
- **Fail-silent on malformed input is a textbook anti-pattern** ("log-and-skip → nonsense downstream"). Prescribed fix: validate at the boundary; distinguish *legitimately absent* from *present-but-unparsed* — the latter must be loud. ("Parse, don't validate" / make illegal states unrepresentable.)
- **Markdown bold-label extraction is genuinely dialect-sensitive** (CommonMark flanking rules): `**Field:**` vs `**Field**:` render via different delimiter-flanking, and a regex keyed on one shape silently misses the other. Robust pattern: encode the enumerable dialect set explicitly (colon inside or outside, optional bullet), strip code fences first; prefer a token parser over one rigid regex.
- **Takeaway:** split the variance — *normalize* the known-equivalent dialect family (colon/bullet placement is a real CommonMark artifact, safe to canonicalize) and *fail loud* on everything else. Don't choose "lenient vs strict" globally.

Sources: [draft-thomson-postel-was-wrong-03](https://datatracker.ietf.org/doc/html/draft-thomson-postel-was-wrong-03), [IETF protocol-maintenance](https://www.ietf.org/archive/id/draft-iab-protocol-maintenance-05.html), [Parse don't validate](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/), [Loud failures > silent (InfoWorld)](https://infoworld.com/article/2074667/loud-failures-are-better-than-silent--faulty-behavior.html), [CommonMark 0.31.2 emphasis/flanking](https://spec.commonmark.org/0.31.2/), Pydantic/structured-output guidance (BentoML, MLMastery, Xebia).

## Requirements & Constraints

- **Competing principles, unresolved by the docs:** "Surface failures in the morning report; keep working unless blocked" + "fail-forward model" + "Graceful partial failure" (`project.md:15,50`, `pipeline.md:9,41,130`) **vs** "Prefer structural separation over prose-only enforcement" + "Solution horizon / durable-fix" (`CLAUDE.md`, `project.md:21`). **Crucial scoping note from the agent:** the graceful-partial-failure pole is explicitly about whole-task/whole-feature units failing — *none of it sanctions silently mis-dispatching tasks*. The current silent-`[]` is not "graceful degradation"; it produces a runtime-broken merge with no failure surfaced. Blocking one mis-formatted feature *is* graceful partial failure; silently corrupting ordering is not.
- **Solution-horizon rule directly applies:** the ticket states "not feature-specific — any plan in the [dropping] dialect loses all dependency ordering" → the patch applies to a known class of cases → the rule favors the durable version over the narrowest patch.
- **Intra-feature dependency ordering is documented for the interactive path** (`skills/lifecycle/references/plan.md:224-225`: "omitting or misformatting the field blocks parallel execution"; `orchestrator-review.md` P2 gate checks the dependency graph) **but is an undocumented gap on the overnight-pipeline side** — `pipeline.md` never mentions `compute_dependency_batches` or intra-feature ordering. The overnight plan-gen path does **not** run the P2 gate. → A requirements-doc update (pipeline.md) is warranted alongside the code fix.
- **Authoritative plan-format contract:** `skills/lifecycle/references/plan.md` (template lines 167-182; field list line 77; dependency-annotation rule 224-225). The contract is *implicit* (the template shows syntax; no machine spec), and there is **no SKILL↔parser parity test** tying it to `parser.py` — they can drift freely.
- **No ADR governs parsing/plan-format/fail-loud.** `cortex/adr/README.md`'s three-criteria gate would decide whether a load-bearing parser-contract decision warrants a new ADR. ADR-0007 is a precedent-by-analogy (moved *away* from a silent implicit contract toward a surfaced decision).
- **MUST-escalation policy:** hardening the generator *prompt* with imperative MUST language is the disfavored lever and needs an events.log F-row artifact + an `effort=high`/`xhigh` attempt first. Structural code enforcement (relax + fail-loud at the parse boundary) sidesteps this and is favored by "structural over prose-only."

## Empirical Plan-Dialect Forensics

- **In-repo dialect tally** (forensics agent, keyed on colon-*inside* `**Files:**`): 236/239 field-bearing plans canonical; 3 colon-inside, all interactive dashboard-area **archives**. **This tally undercounted the fault** — it did not search the no-bullet colon-*outside* variant. (See contradiction note below.)
- **Orchestrator-verified correction:** exactly **1 active (non-archive) in-repo plan** uses the dropping no-bullet form — `cortex/lifecycle/scale-research-fanout-by-complexity/plan.md` — and I confirmed by running the actual regexes that its `**Depends on**: 1` / `**Files**: …` lines (Tasks 1–11) return `None`/`[]`. It was **interactively authored**. So: the bug is **live, present, generator-source-agnostic, but rare** in this repo.
- **The cited incident is cross-repo:** run `overnight-2026-06-09-0222` and feature #202 "climb" (Godot `.gd` files) are **not in this tree** — a wild-light run against a separate Godot target. The smoking-gun plan lives there. (The `#202` in this repo's git log is an unrelated artifact-relocation epic.)
- **Generation path:** per-feature `plan.md` is authored by an LLM sub-agent dispatched from `cortex_command/overnight/prompts/orchestrator-round.md` §3b (~404-428). Its instruction *names* the fields in prose but **does not pin the literal `- **Files**:` syntax and does not cite `skills/lifecycle/references/plan.md`.** The session plan (`overnight/plan.py:render_session_plan`) and batch master plan (`overnight/batch_plan.py`) are deterministic Python tables and irrelevant to the dialect bug.
- **Classification:** drift is real but **LLM-variance / non-deterministic / occasional**, and the underlying parser fragility affects *any* author (LLM or human) that omits the bullet or moves the colon.

## Tradeoffs & Alternatives

Four candidate approaches, scored on durability · drift-detection · overnight-autonomy · complexity:

- **A — Relax the parser (Postel/lenient):** accept bullet-optional + colon-inside-or-outside. *Pros:* smallest change, mirrors the existing `_normalize_task_separators` dash-tolerance precedent, recovers ordering silently at 2am. *Cons:* alone, it **codifies drift** and preserves the silent-on-miss anti-pattern — the next unrecognized form fails silently again. Durability High (known forms) / Low (next unknown); detection Poor; autonomy High; complexity Low.
- **B — Fix/tighten the generator:** stronger prompt, or a deterministic post-gen normalizer. *Pros:* fixes at source. *Cons:* prompt route is **unenforceable** (LLM can always drift; MUST-escalation disfavors prose); the normalizer route **is itself a parser** (must recognize the bad dialect to rewrite it) and does nothing for interactively-authored or foreign plans. Durability Medium→High but only for that generator; complexity High.
- **C — Fail loud:** raise when a Files/Depends-on label is present but uncapturable. *Pros:* strongest detection; the silent-`[]`→collapse chain becomes impossible; bounded blast radius (one feature `failed`, circuit-breaker-exempt, nothing merged). *Cons:* blocks an otherwise-good feature on a formatting drift; needs a careful label-probe regex to avoid false positives on prose. Durability High; detection Excellent; autonomy Mixed (safe but can be wasteful); complexity Medium.
- **D — Defense-in-depth combo (tradeoffs agent's original recommendation):** tolerant parse (A) + a post-parse warning when a multi-task plan yields a single zero-dependency batch, reusing the `COMPLEXITY_NORMALIZED` event+renderer plumbing. **The adversarial pass rejected the batch-shape guard** (false-positive + false-negative, below) and demoted warn-after-merge for a high-criticality bug.

**Synthesized recommendation (post-adversarial):** **A (safe normalization of the enumerable bullet/colon dialect family) + C (dialect-independent fail-loud invariant) as the core**, with **B's generator-prompt tightening + a SKILL↔parser parity test as defense-in-depth**, and the batch-shape warning **dropped** in favor of the parse-boundary invariant. Scope of the defense-in-depth layers is the main open design decision (see Open Questions).

## Operational Failure-Mode Under Autonomy

- **A parser `ValueError` is operationally cheap and safe.** Caught at `feature_executor.py:566-574` → `status="failed", parse_error=True`. `parse_error=True` is **excluded from the circuit-breaker count** (`outcome_router.py:836`), so even several malformed plans in a round never trip the 3-pause breaker. The feature surfaces in the morning report's **"Failed Features"** section; **nothing is built or merged.** This *is* graceful partial failure.
- **Asymmetry to respect:** the dependency-*cycle* `ValueError` from `compute_dependency_batches` returns `failed` **without** `parse_error=True` (`feature_executor.py:596-602`), so that path **does** feed the circuit breaker. Any new raise must be routed deliberately (parse path = breaker-exempt; cycle path = breaker-feeding).
- **Per-option 2am consequence:** A (lenient) = **silent-corrupt** on the next unknown form, no surfacing, recurs as the original bug. C (fail-loud) = **loud single-feature block**, clear surfacing, zero corrupt code on the branch. The batch-shape warning = **observational after-the-fact** — the collapsed-ordering plan still runs and merges this session.
- **Surfacing precedent for any non-blocking signal:** `normalized_complexities` → `complexity_normalized` event (`feature_executor.py:582-589`) → `report.py:render_complexity_normalized` dedup section (omitted when empty). Reuse this mechanism *only* for genuinely-ambiguous residual signals, not for the unambiguous drop case.
- **Deferral is not the right channel** (reserved for ambiguous-intent/CI/non-APPROVED-review); a parse failure is unambiguous → `failed`.

The **two-mode `--staged`/`--audit` gate pattern** (`project.md:93`) is the natural home for a new plan-format gate (`--staged` over changed `plan.md`, `--audit` over all `cortex/lifecycle/*/plan.md`).

- **No SKILL↔parser parity test** ties `skills/lifecycle/references/plan.md`'s template to the parser's accepted forms — the gap this bug exploits.
- **Natural homes for the fix's verification:** (a) regression fixtures for the dropping dialects (no-bullet, colon-inside, partial-drop) → `test_parser.py` (extend `_task_block` to emit alternates); (b) first direct tests for `compute_dependency_batches` (`common.py:670`); (c) a `*_parity.py`-style test parsing the literal template block from `references/plan.md` and asserting the parser extracts it; (d) the smoke test (`overnight/smoke_test.py:57-82`) currently emits canonical single-task plans only — no dependency-ordering coverage.

## Adversarial Review

- **Scoping:** the parser fix is **not over-build, it is under-spec'd**. The bug is live in-repo (verified: `scale-research-fanout-by-complexity/plan.md`) and **author-agnostic**; the *cited* incident is the one-off. Over-build risk lives in piling on guards/repair-loops to chase a low dialect-rate **if the parser remains the only defense** — the proportionate scope is the small dialect-independent invariant.
- **The batch-shape drop-signal guard is the weakest mechanism and should be rejected as primary:**
  - **Guaranteed false-positive:** a legitimately fully-parallel N-task plan produces `[[1..N]]` — byte-identical to total-drop. Single-task plans (`[[1]]`) too. Firing on legal input trains operators to ignore it.
  - **Large false-negative:** a *partial* drop (`[T1[], T2→1, T3-dropped]`) yields multiple batches (`[[1,3],[2]]`) — task 3 runs before its real dependency, ordering broken, **guard silent.** Mixed-dialect plans are exactly this shape, and it's the more insidious case (looks ordered).
- **Second silent-corruption path found: phantom dependencies.** `re.findall(r"\d+")` scrapes any digit, so `**Depends on**: Phase 1 complete` → phantom `[1]`. Relaxing the regex can *surface more prose* to the digit-scraper and make this worse. (Note: the adversarial agent's cited line `scale-research-fanout/plan.md:21` is in the Overview, not a task body, and is bullet-less so it isn't parsed today — but the *mechanism* is a real latent risk for any matched task-body value that names a non-task number like "Phase 1".)
- **Symptom vs root cause:** neither relax-only nor fix-generator-only is sufficient. Relaxing chases dialects forever; the generator can't be guaranteed by prose and a normalizer is just another recognizer. The right layer is **the boundary between "label present" and "value parsed."**
- **The dialect-independent invariant (the key deliverable):** *If a task body contains a Files or Depends-on label in any recognizable form but the parser extracts no value, that is a hard parse error — never a silent empty list.* Implementation: after the strict canonical regex fails, run a **loose label detector** (`\*\*\s*Files\s*\*\*` / `\*\*\s*Depends`, colon any position); if it finds a label the strict parser missed → `raise ValueError` → `parse_error=True` → one feature fails, surfaces, nothing merges. Dialect-independent; catches colon-inside, no-bullet, and partial-drop; no false-positive problem. **`compute_dependency_batches` is NOT the right layer** — by then the metadata is already lost; it can't distinguish dropped-deps from no-deps.
- **Warn-vs-defer:** for a `criticality: high`, "runtime-broken code got merged" bug, a non-blocking warning that fires after the merge is the **wrong default**. Fail-loud is correct and its autonomy cost is bounded and already designed-for (one feature, breaker-exempt). Warn is defensible only as a *secondary* signal for the genuinely-ambiguous residual (a clean parse to zero deps) — and even there the right move is usually *not* to warn (it's legal).
- **Assumptions corrected:** "dialect is bold-paragraph" (false — discriminator is bullet/colon-position); "overnight-only / cross-repo" (false — live interactive in-repo instance); "`COMPLEXITY_NORMALIZED` is a clean precedent" (partial — that's *normalize-and-continue* for a **safe over-provision**; dropped ordering has **no conservative default**, so reusing warn-and-continue imports an unjustified tolerance posture).

## Open Questions

These are surfaced for the Spec phase to resolve with the user. **Items 1–4 are explicitly deferred to Spec** (they are design/scope decisions for the requirements interview, not code-resolvable gaps); items 5–6 are resolved inline and recorded for completeness.

1. **Deferred to Spec — primary mechanism & autonomy posture.** Research strongly favors the **dialect-independent fail-loud invariant** (raise on label-present-but-value-empty; one feature fails, circuit-breaker-exempt, nothing merged) over the ticket's "relax the parser" framing and over the batch-shape warning. Confirm the posture: fail-loud-as-primary (recommended) vs. relax-and-warn. This is a genuine product decision about overnight robustness and should be presented at spec approval.
2. **Deferred to Spec — scope of defense-in-depth.** Which layers are in scope: (a) safe normalization of the bullet/colon dialect family; (b) the fail-loud invariant; (c) phantom-dependency guard in `_parse_field_depends_on`; (d) generator-prompt tightening in `orchestrator-round.md` (+ citing `references/plan.md`); (e) a SKILL↔parser parity test; (f) a `pipeline.md` requirements update documenting intra-feature ordering; (g) first direct tests for `compute_dependency_batches`. Minimal durable set vs. full hardening — the spec's complexity/value gate applies.
3. **Deferred to Spec — generator fix lever.** Prompt tightening (disfavored by MUST-escalation; needs evidence) vs. a deterministic post-generation normalizer vs. treating generation as defense-in-depth behind the parser invariant. Note: per the MUST-escalation policy, any imperative-MUST prompt hardening requires an events.log F-row artifact + an `effort=high`/`xhigh` attempt first — likely out of scope as a primary lever.
4. **Deferred to Spec — circuit-breaker routing.** Confirm new raises route through the `parse_error=True` (breaker-exempt) path, not the cycle path that feeds the breaker — and whether that asymmetry should itself be reconciled.
5. **Resolved — inter-agent contradiction (forensics "not reproducible in-repo" vs adversarial "live in-repo").** Resolved by orchestrator verification: the bug **is** live in-repo (`scale-research-fanout-by-complexity/plan.md`, no-bullet form, parser drops it), but rare (1 active plan); the forensics tally undercounted because it searched only the colon-inside variant.
6. **Resolved — is the batch-shape warning (ticket fix #3) worth keeping?** No, not as a primary control — guaranteed false-positive on fully-parallel plans and false-negative on partial drops. The parse-boundary invariant supersedes it.
