# Plan: overnight-plan-parser-drops-per-task

## Overview
Rework the per-task field parsers in `cortex_command/pipeline/parser.py` so they (a) normalize the enumerable bullet/colon dialect family (optional leading bullet, colon inside *or* outside the bold) across all four field parsers, and (b) fail loud — raise `ValueError` — when a colon-adjacent `Files`/`Depends on` field label is present but yields no usable/list-conformant value, while leaving legitimately-absent values and the known value-bearing dialects parsing cleanly. Normalization (R1) recovers the value-bearing dialect family so the only inputs that raise are genuinely unrecoverable drift; a raise routes through the existing `parse_feature_plan` → `cortex_command/overnight/feature_executor.py:566-574` boundary to `status="failed", parse_error=True` (circuit-breaker-exempt), failing exactly one feature with nothing merged. Phase 2 hardens the source (generator prompt), documents the now-load-bearing ordering contract, and adds the first direct tests for the collapse mechanism plus a rename-drift guard.
**Architectural Pattern**: pipeline

## Outline

### Phase 1: Parser correctness (tasks: 1, 2)
**Goal**: The parser recovers the known-safe dialect family (including the multi-line `Files` and trailing-parenthetical `Depends on` forms the live corpus uses), never silently loses or fabricates ordering-critical metadata, and the raise-routing to `parse_error=True` is pinned. Task 1 ships parser code AND its unit tests together so the new behavior gates the task.
**Checkpoint**: `just test` green with new parser tests covering normalization (R1), fail-loud (R2/R4), no-false-positive (R3) including the corpus dialects, and the `ValueError → parse_error=True` routing (R5); the live-corpus audit confirms no active plan newly raises.

### Phase 2: Source & downstream hardening (tasks: 3, 4, 5, 6)
**Goal**: Reduce drift at the generator, document intra-feature ordering, cover the collapse mechanism directly, and guard against field-rename drift.
**Checkpoint**: `just test` green with `compute_dependency_batches` direct tests (R6) and the rename-drift guard (R9); the generator prompt pins the canonical syntax (R7); `pipeline.md` documents the ordering contract (R8).

## Risks
- **The label detector's line-leading anchor is the load-bearing discriminator (R2/R3).** The raise/benign-absent decision hinges on `_field_label_present` matching a top-level field bullet but NOT a nested Context sub-bullet or a colon-free prose mention. Task 1's R3 non-trigger suite now includes a nested colon-adjacent bullet and a colon-inside prose span specifically to exercise the anchor (not just colon-adjacency). If those tests prove the anchor insufficient, tighten it before the task is done.
- **List-conformance tolerates trailing parentheticals by stripping `(...)` spans before validating.** This was forced by the live corpus (`[1] (rationale)`, `none (rationale)`); it both prevents the R4 regression and fixes an existing phantom-dep bug. The residual risk is a value whose real dependency lives inside a parenthetical — but the canonical template never emits that, and the spec's accepted forms put dependencies before any annotation.
- **Task 1 is intentionally the largest task** (parser code for four functions + co-authored tests + corpus audit, two files). Code and tests are merged to close the vacuous-green gap the review found; splitting them would re-open it. The live-corpus audit in its verification is the backstop against an unforeseen regression on the 104 active plans.
- **Scope is deliberately Phase 1 (must) + Phase 2 (should).** The batch-shape warning (ticket fix #3), a full SKILL↔parser parity harness, non-bold field labels, and constrained-decoding plan generation remain explicit Non-Requirements (spec). The Status-dialect relaxation and the corpus-tolerant list-conformance were added because the review showed the original three-function/strict-list scope would silently regress live plans — that is the bug-fix, not scope expansion.

## Acceptance
A feature `plan.md` written in the no-bullet / colon-inside dialect family — including the multi-line `Files` form and trailing-parenthetical `Depends on` values the live corpus uses — parses to the same `files`/`depends_on` as the canonical form; a plan whose task carries a line-leading colon-adjacent `Files`/`Depends on` label that yields no usable or list-conformant value causes `parse_feature_plan` to raise `ValueError`, which `execute_feature` maps to `FeatureResult(status="failed", parse_error=True)` (circuit-breaker-exempt — nothing built or merged); a plan with legitimately-absent dependencies (`none`, `none (annotation)`, or no label) parses to `[]` with no raise and no phantom; and parsing every active `cortex/lifecycle/*/plan.md` produces no unexpected raise. `just test` exits 0 with all new and pre-existing tests passing.
