# Specification: overnight-plan-parser-drops-per-task

## Problem Statement

The overnight runner silently loses intra-feature task dependency ordering. The plan parser's per-task field regexes (`cortex_command/pipeline/parser.py`) require a strict canonical form — a leading bullet with the colon outside the bold (`- **Files**:` / `- **Depends on**:`). Any deviation (no leading bullet, colon inside the bold, or both) is silently dropped to `[]`. With every `depends_on == []`, `compute_dependency_batches` (`common.py:695`, `all(d in assigned for d in [])` is vacuously true) collapses all tasks into one concurrent batch dispatched via `asyncio.gather` (`overnight/feature_executor.py:728`) — destroying phase ordering with no error. A prerequisite task then runs concurrently with its dependent, and runtime-broken code can merge green. The fault is author-agnostic and live in this repo today (`cortex/lifecycle/scale-research-fanout-by-complexity/plan.md`, interactively authored, drops all of its `**Depends on**: N` declarations). Fixing it protects every overnight feature whose plan drifts even slightly from the canonical dialect.

## Phases

- **Phase 1: Parser correctness** (must-have) — the parser recovers known-safe dialect variants and never silently loses or fabricates ordering-critical metadata. This phase alone satisfies the feature's primary outcome (silent ordering-loss/corruption can no longer occur); R1–R5 are must-have.
- **Phase 2: Source & downstream hardening** (should-have) — reduce drift at the generator, document the now-load-bearing ordering contract, cover the collapse mechanism, and guard against future field-rename drift. R6–R9 are should-have: each was judged worth-it (cheap, attacks the root cause, closes a real gap, or guards a demonstrated recurrence class), but the primary outcome does not depend on them. Won't-do items are enumerated in Non-Requirements.

## Requirements

1. **Normalize the bullet/colon dialect family (recover value-bearing variants).** Relax the field regexes in `_parse_field_files` (`parser.py:363`), `_parse_field_depends_on` (`parser.py:384`), and `_parse_field_string` (`parser.py:408`, used for Complexity/Status) to accept: an optional leading bullet (`-`/`*`) and the colon inside *or* outside the bold (`**Files**:` and `**Files:**`). The canonical form continues to parse unchanged, and `_parse_field_files`'s multi-line (`re.DOTALL`) capability is preserved. Acceptance: new cases in `cortex_command/pipeline/tests/test_parser.py` feeding the no-bullet form (`**Depends on**: 1`), the bulleted colon-inside form (`- **Files:** x`), and the no-bullet colon-inside form return the same parsed values as the canonical form; `just test` exits 0. **Phase**: Parser correctness

2. **Fail-loud invariant: label present but no usable value → raise.** When a `Files` or `Depends on` field label is present — defined as a **bold label token immediately colon-adjacent** (`**Files**:` or `**Files:**`) at **top-level (non-nested) task-body indentation** — but no usable value is extracted, raise `ValueError` rather than returning `[]` or scraping incidental digits. "No usable value" means: no value after the colon, an empty/whitespace-only value (covers Files *and* Depends-on symmetrically — `_parse_field_files`'s current `if not raw: return []` empty-value branch must instead raise when the label is present), OR — for Depends-on — a value that is not list-conformant. The raise propagates through `parse_feature_plan` to the existing handler at `feature_executor.py:567-574`, becoming `status="failed", parse_error=True`. Acceptance: `test_parser.py` cases assert `parse_feature_plan` raises `ValueError` for an empty-value Files field (`- **Files**: ` with nothing after the colon) and for a non-list-conformant Depends-on value (`**Depends on**: after Phase 1 is green`); `just test` exits 0. **Phase**: Parser correctness

3. **Legitimately-absent dependencies and prose-free values must NOT raise (no false positives).** `- **Depends on**: none`, the no-bullet `**Depends on**: none`, a task with no Depends-on label at all, and a multi-task plan in which every task legitimately declares `none` all parse to `[]` (or their list value) with no raise. A task body that *mentions* the bold token in prose without an adjacent colon (e.g. a nested Context bullet `  - **Files** are frozen per Task 1`) must NOT trigger R2. Acceptance: `test_parser.py` cases assert no exception and the expected `depends_on`/`files` for each of these inputs, including the prose-mention non-trigger; `just test` exits 0. **Phase**: Parser correctness

4. **Depends-on value must be list-conformant (no phantom dependencies).** `_parse_field_depends_on` must extract task numbers only from a list-conformant value — the forms the canonical template produces: `[N]`, `[N, M]`, bare `N`, `N, M`, or `none`. A value that is not list-conformant (free prose that merely contains an incidental digit, e.g. `after Phase 1 is green`) must raise (per R2's fail-loud path) rather than silently scraping the digit into a phantom dependency. The list-conformance check must accept every form the canonical template emits and reject free prose; its precise shape is an implementation detail. Acceptance: a `test_parser.py` case asserts `**Depends on**: after Phase 1 is green` raises `ValueError` (not `depends_on == [1]`), and the canonical list forms still parse to their integer lists; `just test` exits 0. **Phase**: Parser correctness

5. **Raise-routing is verified: parser `ValueError` → `parse_error=True`.** Add a test asserting that a parser `ValueError` (from R2/R4) is mapped by the feature-execution path to `FeatureResult(status="failed", parse_error=True)` — pinning the currently-untested safety-critical routing that makes a malformed plan fail exactly one feature and stay circuit-breaker-exempt (`outcome_router.py:836`). Acceptance: a test exercising the `feature_executor.py:566-574` boundary asserts a parse `ValueError` yields `status="failed"` AND `parse_error is True`; `just test` exits 0. **Phase**: Parser correctness

6. **Direct tests for `compute_dependency_batches`.** Add the first direct tests for `compute_dependency_batches` (`common.py:670`): correct topological batching for a dependency chain (e.g. `[1]→[2]→[3]` produces ordered batches), the all-empty-`depends_on` collapse case (documents that empty deps collapse to one batch — now unreachable for label-present plans because R2/R4 raise upstream), and cycle detection raises. Acceptance: a new test module asserting these passes; `just test` exits 0. **Phase**: Source & downstream hardening

7. **Pin the plan format in the overnight generator prompt.** Edit `cortex_command/overnight/prompts/orchestrator-round.md` §3b (lines ~411-414) so it shows the literal canonical field syntax (`- **Files**: …`, `- **Depends on**: [N]` — leading bullet, colon outside the bold) and cites `skills/lifecycle/references/plan.md` as the canonical format, replacing the bare field-name paraphrase. Acceptance: `grep -c -- '- \*\*Depends on\*\*:' cortex_command/overnight/prompts/orchestrator-round.md` ≥ 1 AND the file references `references/plan.md`. **Phase**: Source & downstream hardening

8. **Document intra-feature ordering in pipeline requirements.** Add a short statement to `cortex/requirements/pipeline.md` (Feature Execution and Failure Handling) that the overnight runner preserves intra-feature task ordering via `compute_dependency_batches` from per-task `Depends on` metadata, and that unparseable ordering metadata fails the feature loudly (parse_error) rather than degrading silently. Acceptance: `grep -c 'compute_dependency_batches' cortex/requirements/pipeline.md` ≥ 1. **Phase**: Source & downstream hardening

9. **Minimal field-name guard against template/parser rename drift.** Add one focused regression test asserting the parser extracts the ordering-critical fields the canonical template declares — so that renaming an ordering-critical field in `skills/lifecycle/references/plan.md` (e.g. `Depends on` → `Prerequisites`) fails this test at commit time rather than silently dropping to `[]` (R2 cannot catch a rename because it keys on the literal `Files`/`Depends on` tokens). Scope: the ordering-critical field names only (Files, Depends on), not a full SKILL↔parser parity harness. Acceptance: a test feeds a minimal plan rendered with the template's literal `Files`/`Depends on` field labels and asserts the parser returns the expected non-empty `files`/`depends_on`; `just test` exits 0. **Phase**: Source & downstream hardening

## Non-Requirements

- **Full SKILL↔parser parity harness** — a `--staged`/`--audit` commit-time gate tying `references/plan.md`'s *entire* field set to the parser's accepted forms. Out of scope: R9 covers the ordering-critical rename-drift class with one focused test; a full harness over all seven template fields (three of which — What/Context/Verification — are intentionally consumed by the builder agent, not parsed) is more than this fix warrants.
- **Non-bold field labels** (`- Depends on: 1` without the `**` bold markers) — not recognized and out of the recovered dialect family. The canonical template always bolds field labels; extending recognition to non-bold forms would raise prose false-positive risk (`this depends on: the migration`). Named here as an explicit accepted boundary, not a silent omission.
- **Batch-shape warning** (ticket suggested fix #3 — warn when a multi-task plan yields a single zero-dependency batch) — rejected: guaranteed false-positive on legitimately fully-parallel plans (identical batch shape) and false-negative on partial drops. R2/R4 cover the *label-present* partial-drop case (each unextractable task raises); the *no-label / renamed-field* partial-drop residual is guarded by R9 for ordering-critical fields. The batch-shape signal adds nothing the parse-boundary invariant doesn't already cover better.
- **The cross-repo Godot run (#202 "climb")** — out of this repo; not fixable here.
- **Constrained-decoding / structured-output plan generation or a generator repair loop** — out of scope; R7 (prompt) is defense-in-depth, not a generation-integrity guarantee.

## Changes to Existing Behavior

- **MODIFIED**: parser silently returned `[]` for any non-canonical Files/Depends-on field → now normalizes the bullet/colon dialect family (R1) and raises (`parse_error`) when a colon-adjacent label is present but yields no usable value, including empty values (both fields) and non-list-conformant Depends-on prose (R2, R4).
- **MODIFIED**: `_parse_field_depends_on` scraped any digit via `re.findall(r"\d+")`, fabricating phantom deps from prose → now requires a list-conformant value and raises on prose (R4).
- **MODIFIED**: `_parse_field_files` returned `[]` on an empty value → now raises when the label is present but the value is empty (R2).
- **MODIFIED**: overnight plan-gen prompt paraphrased the field names → now shows the literal canonical syntax and cites the canonical template (R7).
- **ADDED**: `cortex/requirements/pipeline.md` documents intra-feature task ordering (R8); first direct test coverage for `compute_dependency_batches` (R6); raise-routing test (R5); field-name rename-drift guard (R9).

## Edge Cases

- **Fully-parallel legitimate plan** (every task `Depends on: none`): parses to one batch with no raise — R2/R4 fire only on a colon-adjacent label that yields no usable/list-conformant value, never on legitimately-absent deps. (R3)
- **Prose mention of a field token** (e.g. a Context bullet `- **Files** are frozen per Task 1`, or nested ≥2-space Context sub-bullets): the bold token is not colon-adjacent and/or not at top-level field indentation → R2 does not trigger; the feature is not failed. (R3)
- **Empty-value field** (`- **Files**: ` / `**Depends on:**` with nothing after the colon): label present, no value → raise (R2), for both fields symmetrically.
- **Prose-with-incidental-digit Depends-on** (`**Depends on**: after Phase 1 is green`): not list-conformant → raise (R4), not a phantom `[1]`.
- **Partial-drop mixed-dialect plan** (some tasks canonical, some with an unextractable colon-adjacent label): each unextractable task raises; the feature fails loudly. The *no-label / renamed-field* partial-drop is the residual guarded by R9 for ordering-critical fields. (R2, R4, R9)
- **Single-task plan**: one batch — fine, no special handling.
- **Zero-task plan**: already raises `ValueError("no task sections")` (`parser.py:319-320`), routed through the same parse_error path — unchanged.
- **`Depends on: none` / no label**: parses to `[]`, no raise. (R3)

## Technical Constraints

- **Label-detector scope** (resolves the false-positive risk): R2's "label present" detector matches a **bold field-label token immediately followed by a colon** (`**Files**:` or `**Files:**`), at **top-level task-body indentation** (not nested ≥2-space Context sub-bullets), in any bullet/colon arrangement. The colon-adjacency requirement is what distinguishes a field label from prose mentioning the bold token (`**Files** are frozen` has no adjacent colon → not a field → no trigger). This is the discriminator the earlier "line-leading" framing lacked.
- **Depends-on list-conformance** (resolves phantom deps): digits are extracted only from a list-conformant value (`[N]`, `[N, M]`, `N`, `N, M`, `none`); non-conformant prose raises rather than scraping incidental digits. The check must accept every form the canonical template emits.
- **Raise routing**: R2/R4's `ValueError` MUST surface from `parse_feature_plan` so its caller at `feature_executor.py:567-574` converts it to `status="failed", parse_error=True`, which is **exempt from the circuit-breaker count** (`outcome_router.py:836`, `if not result.parse_error:`). It must NOT route through the `compute_dependency_batches` dependency-cycle path (`feature_executor.py:596-602`), whose `failed` result is *not* parse_error-exempt. R5 verifies this routing.
- **Graceful-partial-failure reconciliation**: failing one mis-formatted feature loudly is consistent with the project's "graceful partial failure / keep working unless blocked" principle — that principle governs whole-feature/whole-task units failing, and explicitly does not sanction silently mis-dispatching tasks within a feature. Fail-loud (R2/R4) resolves the documented tension in favor of surfacing drift for ordering-critical metadata, while normalization (R1) avoids needless failures for safe known variants.
- **Parser does no logging** (`parser.py` module docstring): R1 normalization recovers silently; do not add logging inside the parser. Surfacing of *recovered* (normalized) drift is intentionally out of scope — drift is attacked at the source by R7 and made loud when unrecoverable by R2/R4.
- **`re.DOTALL` asymmetry**: `_parse_field_files` is multi-line tolerant (DOTALL + terminator lookahead); `_parse_field_depends_on`/`_parse_field_string` are single-line. R1 must preserve `_parse_field_files`'s multi-line capability while relaxing the bullet/colon shape.
- **Existing precedent**: `_parse_field_depends_on` already raises on a present-but-unparseable value (`parser.py:400-403`, when no digits found); R2/R4 extend the same fail-loud posture to the empty-value and non-list-conformant cases for both Files and Depends-on.

## Open Decisions

None — the label-detector discriminator (colon-adjacent, top-level) and the list-conformance check are resolved as Technical Constraints and are implementable from current knowledge.

## Proposed ADR

None considered. The fail-loud-at-parse-boundary posture (R2/R4) is a real trade-off, but it is a reversible code-level behavior; its rationale (the graceful-partial-failure reconciliation) is recorded inline in Technical Constraints rather than promoted to an ADR. Promote later only if the posture proves contentious in review.
