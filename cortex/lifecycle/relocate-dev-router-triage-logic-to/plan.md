## Risks
- **Optimizing an unenforced metric.** No test measures body byte count (L1 is frontmatter-only; the size cap is lines and far off), so the win rests on the epic's Web-backed but untestable attention-dilution thesis — not a failing gate. Reversible to the criticality-only scope if the Step-3c hot-path round-trip is judged not worth it (user resolved this to Full Option B at the exit gate; re-raisable, not re-asked absent a signal).
- **Hot-path Read round-trip on Branch 1 (triage).** Relocating Step 3c inserts an assistant→tool→result Read turn on the busiest invocation, and moves the all-refined/any-unrefined/blocked-note decision tree one hop from the child-map it operates on — the "read-but-not-applied" correctness risk. Mitigated by imperative pointer phrasing; residual runtime risk is untestable and the wiring guard covers static wiring only (documented in the test docstring, not claimed as covered).
- **First `references/` dir for dev + untracked-mirror blind spot.** The pre-commit drift hook and `git diff --quiet` are both blind to brand-new untracked mirror files, so an omitted mirror ref would NOT block the commit. The real protection is R9's `git ls-files --error-unmatch` tracked-state check (Task 5) and the explicit `git add` in Tasks 2/3 — flagged because it is the specific failure mode for dev's first reference directory.
- **Task 2/3 operation density and atomicity.** Each bundles ~7 operations (content-move, body-delete, Step-2 reorder / 3c-stub, referent-rewrite, `build-plugin`, `git add`, commit). Bundling is mandatory — the drift hook requires canonical + mirror to land in one commit — so this is a coupling requirement, not a decomposition failure. The real hazard is turn-exhaustion or a builder stalling mid-task, leaving a pathological partial state (content deleted from the body but the ref uncommitted, or the canonical ref created but the untracked mirror unstaged, which the drift hook does not catch). Mitigation: dispatch these tasks **sequentially on trunk** — they edit the same `skills/dev/SKILL.md` and its mirror, so worktree isolation would race and would also strand the Task-1 baseline; on a mid-task stall, resume the same builder and re-verify the four-path commit landed before advancing.
- **Baseline trust.** The R11 gate is only as strong as the Task-1 baseline: it must be captured on a clean/HEAD tree (the tree is routinely dirty from concurrent sessions) and carry a provenance stamp, node-ID membership is checked WITH a cause note rather than alone, and Task 5 falls back to requiring exit 0 rather than silently degrading if the baseline file is missing.
