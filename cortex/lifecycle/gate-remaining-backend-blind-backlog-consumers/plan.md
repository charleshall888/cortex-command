# Plan: gate-remaining-backend-blind-backlog-consumers

## Overview
Apply #315's skill-layer backend-routing principle to five operations the epic left backend-blind. Phase 1 gates the dashboard backlog panel (wheel code) on the in-process resolver, resolved both synchronously at lifespan startup and per-poll, rendering a placeholder instead of stale local counts under a non-`cortex-backlog` backend. Phase 2 gates four prose consumers (discovery clarify §3, discovery decompose §7, morning-review §4, lifecycle complete.md Step 10) on the `cortex-read-backlog-backend` binstub, mirroring the already-gated siblings. No new shared machinery — both resolution channels already exist (ADR-0016).

## Outline

### Phase 1: Dashboard panel gate (tasks: 1, 2, 3, 4, 5)
**Goal**: The dashboard backlog panel stands down to a placeholder under a non-`cortex-backlog` backend, from the very first served render, and stays byte-identical under `cortex-backlog`.
**Checkpoint**: Dashboard test suite green (`test_poller.py`, `test_data.py`, `test_templates.py`, `test_routes_smoke.py`); a discriminating two-arm test proves the non-local arm skips local reads with leftover items present. Phase 1 is wheel code with no mirror, so each task commits normally (no dual-source coupling); the phase is independently shippable, not coupled to Phase 2.

### Phase 2: Skill-layer prose gates (tasks: 6, 7, 8, 9)
**Goal**: The four backend-blind prose consumers resolve the backend via `cortex-read-backlog-backend` and stand down (skip + advisory) under `none`/external, mirroring their already-gated siblings; each canonical edit + its regenerated mirror + its discriminating test land in one self-contained commit.
**Checkpoint**: Each prose gate has a **discriminating** structural test (positive assertion on the real section AND a negative control over the pre-edit/ungated section, so the test cannot pass green-by-construction); each prose task runs `just build-plugin` and commits canonical+mirror+test together (no canonical-only commit reaches the drift hook); the pre-commit drift + contract-lint gates pass on every commit. Tasks are **serialized** (6→7→8→9) so concurrent `build-plugin`/commit cannot race the shared index.

## Risks

- **Phase 2 commit cadence is self-contained per task (revised after critical review).** Each prose task (6–9) edits its canonical `skills/` file, regenerates its mirror via `just build-plugin`, and commits canonical+mirror+test together — so no canonical-only commit ever reaches the pre-commit drift hook, and every task produces a commit (satisfying the implement phase's per-task-commit checkpoint). Tasks are **serialized** (6→7→8→9) so the implement phase cannot co-batch them into a parallel dispatch where concurrent `build-plugin`/commit would race the shared git index. This replaces the original "defer all Phase 2 to one Task-10 commit" shape, which collided with the implement phase's verbatim builder-commit + zero-commit-fails-checkpoint enforcement.
- **Dispatch must be sequential on trunk, not a worktree.** Phase 1 edits the `cortex_command` wheel; `just test` runs the editable install, so a worktree would verify stale `poller.py`/`app.py`/`data.py` code, and console scripts aren't on PATH until editable reinstall. Phase 2's `just build-plugin` + drift hook also assume the live checkout. Recommend `trunk` at the §4 branch/dispatch surface.
- **Concurrent-session index hygiene.** A `.lock`/`.session` exists; if another live session shares the main index, stage each commit with explicit pathspec (the task's own Files list) so unrelated staged files don't leak in. A concurrent session's in-flight canonical edit could also dirty `plugins/` between build and a task's `git diff --quiet plugins/` check — if that fires, confirm the dirty mirror belongs to another feature before proceeding.
- **Phase 2 tests are discriminating by construction.** Each prose-gate test pairs a positive assertion on the real section with a negative control over the embedded pre-edit section text, so a toothless/ungated edit fails the test. The decompose §7 test specifically asserts the skip-arm advisory (not mere read-before-generate token order), because §7's two-token slice cannot otherwise witness conditionality.
- **Startup-path test fixture friction.** The R1 lifespan-startup test enters the `app.py` `lifespan` with a fixture root that has a `.claude/` dir (the lifespan raises `RuntimeError` if absent) and must monkeypatch `run_polling`/`_pid_file` to avoid leaking spawned pollers and a PID file into the suite (Task 3). This is the only non-poll test path and the easiest to get subtly wrong (it must assert before any poll cycle, not after).

## Acceptance
Under a `none` or external backend, the dashboard backlog panel renders the backend-naming placeholder (never stale local counts) from the first served render onward, and the four prose consumers (clarify §3 coverage scan, decompose §7 index regen, morning-review §4 failed-feature create, complete.md Step 10 index sync) each stand down with a one-line advisory — no writes land in `cortex/backlog/`. Under the non-local arm the two sibling templates that read `state.backlog_titles` (`feature_cards.html`, `escalations_panel.html`) fall back to raw slugs without error — a deliberate, tested consequence (spec.md:69), not a regression. Under `cortex-backlog` (including absent `lifecycle.config.md`), every path is byte-identical to today: the dashboard renders the known counts, the existing `test_data.py`/`test_routes_smoke.py` suites stay green unchanged, and all four prose consumers run as before. `just build-plugin` leaves `plugins/` clean and `just validate-commit` passes on every commit.
