# Specification: gate-remaining-backend-blind-backlog-consumers

> Epic context: child of #315 / follow-on to #317 (config-driven backlog backend). Authoritative scope: `cortex/requirements/backlog.md` + `cortex/adr/0016-configurable-backlog-backend-and-llm-as-adapter.md`. See `research.md` for grounding. This spec applies #315's skill-layer routing principle to five operations the epic left backend-blind (three from the originating audit, two surfaced by this lifecycle's critical review); it adds no new machinery.

## Problem Statement

Epic #315/#317 made the backlog backend config-driven (`cortex-backlog` | `none` | external). Five local-backlog operations were left backend-blind — the three the originating three-agent audit found (dashboard panel, discovery clarify §3, discovery decompose §7) plus two more this lifecycle's critical review surfaced (morning-review §4 failed-feature create, lifecycle complete.md Step 10 index sync). They read or write `cortex/backlog/` unconditionally with no backend check. None hard-halt, so this is #318-class hardening, but under a non-`cortex-backlog` backend they surface stale local data as authoritative (dashboard), do wasteful local work (discovery, complete.md), or write a local file (morning-review §4 — violating the `none` no-writes guarantee), instead of standing down. Gating them makes the "select a different backend" experience consistent with the already-correct consumers (dev Step 3, discovery decompose §5, refine, morning-review §6b).

## Phases

- **Phase 1: Dashboard panel gate** — gate the dashboard backlog panel (Python, wheel code) on the resolved backend; render a placeholder instead of stale local counts under a non-`cortex-backlog` backend. Independently shippable (not lifecycle-gated).
- **Phase 2: Skill-layer prose gates** — gate the four prose-layer backend-blind consumers on the resolved backend (via `cortex-read-backlog-backend`), mirroring the existing gated siblings: discovery clarify §3 (coverage scan), discovery decompose §7 (index regen), morning-review §4 (failed-feature create), and lifecycle complete.md Step 10 (index sync). Canonical prose edits + structural tests + dual-source mirror.

## Requirements

### Phase 1 — Dashboard panel gate

1. **Backend resolved before backlog reads — at poll time AND at startup (closes the first-render window).** `_poll_slow` (`cortex_command/dashboard/poller.py:351–371`) calls `resolve_backlog_backend(root)` once per poll cycle, in-process (mirroring `cli_handler.py:1975`), before the `parse_backlog_counts`/`parse_backlog_titles` calls; routes on `backend == "cortex-backlog"`. **Additionally**, because the dashboard serves a module-level `DashboardState()` singleton and the slow poller is launched fire-and-forget (`asyncio.create_task(run_polling(...))` then immediate `yield`, `app.py:247`), the backend MUST also be resolved **synchronously at lifespan startup** (before the server accepts requests / before `yield`) and written to `state.backlog_backend` — otherwise every request in the window before the first 30s poll renders the default (`cortex-backlog`) arm and shows "no backlog items found" on a non-local repo. **Acceptance**: `grep -c "resolve_backlog_backend" cortex_command/dashboard/poller.py` ≥ 1; a test asserts `state.backlog_backend` reflects the resolved backend on a non-local fixture **before any poll cycle runs** (constructed via the lifespan/startup path, not a poll iteration). **Phase**: Dashboard panel gate

2. **`data.py` pure readers untouched (purity + regression anchor).** `parse_backlog_counts` (data.py:969) and `parse_backlog_titles` (data.py:1026) keep their `backlog_dir`-only signature and bodies — the gate lives in the poller, not in these functions. **Acceptance**: `git diff cortex_command/dashboard/data.py` shows no change to either function's signature or body; all 8 `TestParseBacklogCounts` cases in `cortex_command/dashboard/tests/test_data.py` pass unchanged. **Phase**: Dashboard panel gate

3. **New `DashboardState.backlog_backend` field.** Add `backlog_backend: str = "cortex-backlog"` to the `DashboardState` dataclass (poller.py, near lines 89–90); default = local backend so existing instantiations stay byte-identical. The poller sets it from the resolved backend every cycle. **Acceptance**: `grep -c 'backlog_backend: str = "cortex-backlog"' cortex_command/dashboard/poller.py` = 1; `TestDashboardStateDefaults.test_defaults` (test_poller.py:23) asserts the default is `"cortex-backlog"`. **Phase**: Dashboard panel gate

4. **Non-local backend → local reads skipped even with stale leftover items, proven by a DISCRIMINATING two-arm test.** When `backend != "cortex-backlog"`, the poller sets `state.backlog_counts = {}` and `state.backlog_titles = {}` (skips the parse calls) and sets `state.backlog_backend` to the resolved value. **Acceptance** (the test must actually run a poll iteration — call `_poll_slow` once directly or drive `run_polling` and await one cycle; asserting on a freshly-constructed `DashboardState()` is INSUFFICIENT because its fields default to `{}` and would pass for a never-run gate): build a `tmp_path` repo with `cortex/lifecycle.config.md` declaring `backend: none` (and a parametrized external case, e.g. `github-issues`) PLUS leftover `cortex/backlog/NNN-*.md` items. Assert two discriminating things: (a) `state.backlog_backend == "none"` (this only passes if the poller actually ran AND wrote the resolved value — it fails for a never-run poller, whose default is `"cortex-backlog"`, and for a poller that resolves but forgets to assign the field); (b) `state.backlog_counts == {}` and `state.backlog_titles == {}` with the leftover items present. A spy on `parse_backlog_counts` asserts it is not called on the non-local arm. The positive control in R6 (same fixture, `cortex-backlog` backend, populated counts) is what makes (b) discriminating rather than coincident with the default. **Phase**: Dashboard panel gate

5. **Template renders a distinct placeholder naming the backend; never stale counts.** `backlog_panel.html` becomes a 3-way: `backend != cortex-backlog` → placeholder ("backlog tracked externally via `<backend>`" for an external name, "backlog tracking disabled" for `none`, using `cortex-backlog` terminology, never `local`); else existing populated/empty branches. The header `phase-tag` (currently `{% if state.backlog_counts %}…{% else %}empty{% endif %}`) is updated consistently so it does not read "empty"/counts on a non-local backend. **Acceptance**: a `test_templates.py` direct-render test renders `backlog_panel.html` with `state.backlog_backend = "none"` (and an external value) and asserts the placeholder text is present and no item counts / stack-bar render; rendering with `state.backlog_backend = "cortex-backlog"` and populated counts is unchanged. **Phase**: Dashboard panel gate

6. **`cortex-backlog` arm byte-identical (regression anchor) — proven by a known-value positive control, not a tautology.** With `backend == "cortex-backlog"` (including absent `lifecycle.config.md`, which resolves to `cortex-backlog`), behavior equals today's. **Acceptance** (NOT a self-comparison — asserting the gated output equals `parse_backlog_counts(dir)` is circular because the gate site assigns that call verbatim): (a) the R4 test's `cortex-backlog` arm, driven through the gated poller on the SAME populated leftover fixture, asserts `state.backlog_counts` equals the **expected known non-empty dict** (e.g. `{"backlog": 2, "complete": 1}`) — this is the regression anchor and the discriminating positive control for R4(b); (b) the existing 8 `TestParseBacklogCounts` cases in `test_data.py` stay green unchanged (the pure reader is untouched); (c) a `test_templates.py` direct render of `backlog_panel.html` with `state.backlog_backend = "cortex-backlog"` and populated counts asserts the existing header text (e.g. "N items tracked") is unchanged — guarding the `phase-tag` restructure from regressing the default arm's rendered output; (d) `test_routes_smoke.py` `GET /partials/backlog` still returns 200 and `test_all_ten_partial_routes_covered` stays at 10 (no new route). **Phase**: Dashboard panel gate

- **(Scope reversal, recorded)**: morning-review §4 was initially dropped (Clarify §4) as "transitively unreachable," but this lifecycle's critical review disproved that premise (morning-review is interactive, not the overnight runner; its §6b is already backend-gated) — it is now gated (R11). complete.md Step 10 was likewise added (R12). Both are same-class fixes the originating three-agent audit missed.
- **`review.md:12` requirements guard** — OUT OF SCOPE (unrelated `cortex/requirements/` robustness nit; file separately if wanted).
- **External create / write-back arm** (#318) — `wontfix`; not built here. On the gated read/index/coverage paths the external arm stands down (skip), it does NOT query the tracker.
- **No new shared backend-gate helper** — reuse the two existing channels (in-process `resolve_backlog_backend` for the dashboard; `cortex-read-backlog-backend` binstub for prose). Do not DRY-merge the fail-open binstub with the fail-closed overnight guard (`cli_handler.py:1961–1973`).
- **Backlog engine stays in the wheel** — `cortex_command/backlog/*` is not removed; #321 routes around it.
- **Pre-existing `backlog_titles` docstring gap** in `DashboardState` is not in scope.

## Edge Cases

- **Stale leftover local items** (repo migrated off `cortex-backlog` without cleanup): the dashboard must show the placeholder and discovery must skip — NOT read the leftover items. This is the load-bearing test case (a clean-empty fixture would pass even a broken gate).
- **`none` backend**: dashboard placeholder shown; discovery §3 scan skipped; decompose §7 index regen skipped — **no writes land in `cortex/backlog/`**.
- **External backend** (e.g. `github-issues`): same stand-down as `none` on all three paths (skip + advisory / placeholder); the placeholder may name the backend.
- **Absent `lifecycle.config.md`**: resolves to `cortex-backlog` → today's behavior exactly (regression anchor).
- **Resolver/binstub failure (degenerate input)**: both channels fail **open** → `cortex-backlog` → degrade to local behavior (correct direction for these consumers).
- **Dashboard first-render window (server start before first poll)**: the served `DashboardState` singleton defaults `backlog_backend="cortex-backlog"`, and the poller is launched fire-and-forget (lifespan yields before the first poll). Without a startup resolution, a non-local repo would render "no backlog items found" until the first 30s poll. Closed by R1's synchronous lifespan-startup resolution — the first served render already reflects the real backend.
- **Mid-session config edit**: a config change is picked up within one 30s poll cycle (backend re-resolved every cycle, not cached). This is distinct from the first-render window above (bounded by one poll cycle, not by server start).
- **decompose zero-piece branch under a non-local backend**: the zero-piece branch reaches §7 without entering §5's create-routing, so §7 resolves the backend independently and skips the index regen (per R8) — it is not left ungated.

## Technical Constraints

- **Reuse existing resolution channels**: Python → `resolve_backlog_backend(root)` (`cortex_command/lifecycle_config.py:97`, returns a string, never None/raises, every degenerate input → `"cortex-backlog"`); prose → `cortex-read-backlog-backend` binstub. Both fail open. → ADR-0016.
- **Dual-source mirror**: the four prose-gate edits are canonical top-level `skills/` files that auto-mirror via the pre-commit drift hook — discovery clarify/decompose → `plugins/cortex-core/skills/discovery/`, complete.md → `plugins/cortex-core/skills/lifecycle/`, morning-review → `plugins/cortex-overnight/skills/morning-review/` (justfile `SKILLS=(overnight morning-review)`). Edit canonical (`skills/`) only and commit canonical + mirror together (`just build-plugin`). The dashboard path is wheel code with no mirror.
- **Editing the `cortex_command` package → use SEQUENTIAL dispatch, not a worktree** — `just test` runs the editable install, so worktree verification of `poller.py`/`data.py`/`lifecycle_config.py` edits would test stale code. Console scripts are not on PATH until editable reinstall → test backend resolution in-process.
- **Per-request/per-poll resolution (R3c)**: `app.py` forbids module-level root capture; backend resolution must be per-poll (poller) — consistent with the chosen gate site.
- **Tests must be hermetic** (`tmp_path`, no network). Config-writer idiom: write `cortex/lifecycle.config.md` with `---\nbacklog:\n  backend: {value}\n---\n`. `monkeypatch.delenv("CORTEX_COMMAND_ROOT"/"CORTEX_REPO_ROOT", raising=False)` in routing tests (footgun masks dev-machine routing).
- **Known external `just test` failures** are pre-existing, not regressions: a concurrent-fixture flake (`tests/fixtures/predicate_a_baseline.json` already modified in-tree) and a sandbox-network skip (pypi/MCP DNS blocked).
- **Lint gates**: L201 (use console scripts), cortex-check-contract (double-backtick prose form), SP001/SP002 (no `${CLAUDE_SKILL_DIR}` introduced), L1 surface ratchet (frontmatter-only — reference edits don't touch it).
