# Specification: auto-progress-lifecycle-phases-when-no-blockers

## Problem Statement

Users running `/cortex-core:lifecycle` observe the orchestrator pausing for confirmation at phase boundaries (notably Implementâ†’Review and Reviewâ†’Complete) where no decision needs to be made. The Phase Transition contract in `skills/lifecycle/SKILL.md:161â€“172` already says "proceed automatically," but per-phase prose ambiguity, a known cycle-counter bug in `cortex_command/common.py:185â€“196`, and ceremonial-framing of substantive approval gates cause the model to interpret transitions as user-blocking when they should be autonomous. This wastes daytime time, trains rubber-stamp habits (consent-fatigue), and contradicts Anthropic's monitor-and-intervene posture for capable models.

## Phases

- **Phase 1: Fix the load-bearing primitives** â€” fix the review-cycle counter (R1), register approval event types (R2), and update the phase detector to consult approval events with a migration sentinel for in-flight lifecycles (R3). Phase 1 lands first because R3's detector change must precede R5/R6's emit-side prose â€” otherwise users approving a spec under R5's new prose would emit `spec_approved` events that no consumer reads, and the user-visible behavior wouldn't change. R1 is independent of R5/R6 but is grouped here because all three are common.py / events-registry edits with shared review surface.
- **Phase 2: Rewrite phase-boundary prose** â€” disambiguate the Phase Transition contract (R4), rewrite specify.md Â§4 and plan.md Â§4 to remove ceremonial framing and emit approval events (R5, R6), enrich the resume-staleness prompt (R7), auto-skip refine when artifacts exist (R8), unify the per-phase auto-advance sentence (R9), enumerate kept-pauses inventory (R10).
- **Phase 3: Test surface** â€” phase-detector regression tests (R11) and kept-pauses parity test (R12).

## Non-Requirements

- **Do NOT remove spec or plan approval as a user-facing pause.** The approval gate stays. Only the ceremonial framing around it is removed. Spec/Plan approval IS a real user decision per `requirements/project.md:13` (handoff contract) and `docs/interactive-phases.md:49â€“78`.
- **Do NOT extract phase-transition logic into a Python helper module** (`cortex_command/lifecycle.py`). Once the cycle-counter fix lands in `common.py`, the remaining routing is correctly described by prose. A Python helper is premature optimization.
- **Do NOT remove within-phase AskUserQuestion sites.** Phase-transition auto-advance does not eliminate prompts inside a phase (e.g., implement.md Â§1 branch selection, batch-failure retry/skip/abort).
- **Do NOT remove critical-review synthesis presentation** at specify.md:149 / plan.md:271. Apply/Dismiss disposition of findings is a real user surface.
- **Do NOT remove the `## Requirements Drift` re-dispatch escalation** at review.md:147.
- **Do NOT change the soft positive-routing tone.** No new MUST/CRITICAL/REQUIRED prose; rewrites use the existing canonical "Proceed automatically â€” do not ask the user for confirmation" phrasing.

## Edge Cases

- **In-flight lifecycle paused at spec approval gate**: Existing lifecycles with `spec.md` present but no `spec_approved` event would auto-advance under naive new prose. Mitigation: Requirement 3 makes the phase detector consult `spec_approved`/`plan_approved` events before advancing past Specify/Plan. Existing in-flight features without the event remain in their current phase.
- **Reviewer overwrites `review.md` on cycle 2**: Current cycle-counter regex against `review.md` returns 1 forever. Mitigation: Requirement 1 â€” count `review_verdict` events in `events.log` (append-only by design).
- **Critical-review synthesis with all findings dispositioned `Apply` automatically**: not in scope; user still sees synthesis and decides per existing alignment-considerations propagation pattern.
- **Resume after long pause with stale spec/plan**: Requirement 7 â€” the resume prompt surfaces staleness signals so the user can choose to re-run a phase.
- **Refine invoked on a topic with both artifacts present from a prior run**: Requirement 8 â€” auto-skip with announcement; explicit `--rerun` or user request triggers overwrite.
- **`spec_approved` event already present from prior cycle when user requests changes**: Requirement 5's flow re-emits `spec_approved` only on the new `Approve` selection; intermediate revision loops do not write the event.

## Technical Constraints

- **Soft positive-routing prose only** (`CLAUDE.md` MUST-escalation policy). No new MUST/CRITICAL/REQUIRED language.
- **events.log is append-only** by design; readers must tolerate mid-mutation reads (per `requirements/pipeline.md:132â€“134`). Approval events follow the existing JSONL pattern.
- **Phase detector returns must remain JSON-parseable** with stable field names (`phase`, `checked`, `total`, `cycle`). Adding new event-lookup logic must not change the return shape.
- **Forward-only phase transitions** (`requirements/pipeline.md:19â€“42`) must be preserved; auto-advance never moves backward.

- **MUST-escalation grandfather rule**: pre-existing MUST language stays; rewrites that *remove* MUSTs (none planned but possible if discovered) document the removal in the PR per epic #82's grandfathering carve-out.
