## Problem Statement

`skills/lifecycle/references/implement.md` is the skill-value audit's highest-cost single file: its content rides into every dispatched builder prompt, so each resident token multiplies. Eleven verified trim verdicts (~13k weighted tokens) remain unapplied. Separately, section §1a-i's "interactive worktree liveness check" — and the coupled picker fire-condition (iv) in `cortex_command/lifecycle_implement.py` — probe `cortex/lifecycle/sessions/{slug}.interactive.pid`, a path **nothing writes** (a documented "path-naming error"); the real lock lives at `cortex/lifecycle/{slug}/interactive.pid` via `cortex-interactive-lock`. This leaves the suppressed branch-mode path (which skips §1 Step A/B) with zero real same-slug concurrency protection. This feature applies the eleven trims and replaces the dead check with a real, correctly-ordered lock probe.

## Edge Cases

- **Concurrent same-slug suppressed session**: a second suppressed-path session for the same `{slug}` hits R1's `cortex-interactive-lock acquire`, finds the first session's live lock, exits non-zero → surfaces the rejection and does not create a worktree. This is the intended new guard (today: no guard).
- **Overnight becomes live before acquire**: with R1's corrected ordering, the §1a-ii overnight guard runs *before* the acquire, so an overnight-live rejection exits §1a **without** having acquired a lock — no orphan. (The symmetric hazard on the selected path, where Step B acquires before §1a-ii, is pre-existing and carved out — see Non-Requirements.)
- **Stale lock (dead owner PID)**: `scan_live_locks` / `acquire` classify a dead-PID lock as STALE and recover it (do not block) — a crashed prior session does not permanently wedge the slug (`interactive_lock.py` adds only `is_live` slugs to the scan set).
- **`/clear` mid-session (alive-PID lock)**: a `/clear` keeps the **same OS process** alive, so a lock written before the `/clear` reads LIVE (live PID → Row-4 conservative-LIVE on null `start_time`, or Row-1 on unchanged session_id). After R2, fire-condition (iv) will therefore fire the picker on that lock where the old dead-phantom path never did. This is a new (benign — there genuinely is a live owner) picker-fire on the same slug; it is not a false wedge (the process is alive). Recorded so the behavior is not surprising; no code change.
- **Same-session self-detection**: after R1 acquires on the suppressed path, the session holds its own lock for its lifetime (no in-flow release). If that session later returns to `main`/`master` and re-runs the implement picker, fire-condition (iv) matches its own lock. The picker only renders on `main`/`master` — but note this gate is **prose** in `implement.md` (the "not on main/master → skip" clause), **not** a predicate inside `should_fire_picker`; the normal post-`EnterWorktree` flow is on a feature branch, so this bites only an abnormal return-to-main re-preflight.

- **a3 fixture** uses `"magic": "cortex-interactive-lock"` + a live self-PID (a dead PID classifies STALE → test fails). Row-4 (null session_id) is a valid generic-LIVE lock for exercising `scan_live_locks`; it need not replicate the Row-1 self-session lock the live `acquire` writes.
- **s12** cut must retain `create_worktree`, `--feature interactive-`, and the `**iii.` marker (an unrecorded dependency of `test_lifecycle_enterworktree_callsites` + `test_gate_and_gated_path_use_same_binary`).
- **s13 + s14** must leave ≥1 `**`-prefixed line after the step-v body (block-extractor terminator).
- **New `merge-back.md`** must be git-tracked and committed in the same commit (the `test_lifecycle_references_resolve` `git ls-files` walk; `git add` explicitly — pathspec commits drop untracked new files).
- **SP001/SP002**: implement.md (a reference file) cannot resolve `${CLAUDE_SKILL_DIR}` itself; the merge-back read directive carries a body-resolved absolute path, enforced by `cortex-check-skill-path`.
- implement.md is a **reference file, exempt** from SKILL.md size/L1 ratchets.
- **Assumption (record, not fix)**: the suppressed-path guard's cross-session value depends on the lock's `pid=os.getppid()` staying alive (persistent Bash-tool shell). If the harness spawns per-call shells, the guard degrades to same-session-only. Pre-existing #241 behavior. This repo's own `branch-mode` is `prompt` (picker renders), so the suppressed path is exercised mainly by consumer repos / when `worktree-interactive` is configured.
