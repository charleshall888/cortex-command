# Specification: Move overnight-suitability judgment from /refine into /overnight curation

## Problem Statement

Today `/refine` decides whether a freshly-specced ticket is a good unattended-overnight candidate and warns about it — inferring its run-mode by counting `phase_transition` rows (a brittle proxy), and firing at spec-writing time rather than when the operator is actually choosing what to run overnight. That bloats `/refine` with an execution concern that isn't its job and lands the warning too early to act on. This feature relocates the judgment to overnight's interactive curation gate — the one place with both a human in the loop and the full execution picture — so poor-fit specs are set aside (with reasons, bias toward exclusion), reviewable and re-addable before the run list is frozen, and the unattended harness runs only that frozen list. Research uncovered that the relocation is impossible without first closing a latent gap: the operator's Step 6 curation never reaches what executes (`launch` re-selects from scratch), so this feature also adds the curation→launch handoff that makes any curation — including the pre-existing `[R]emove` — actually stick. **Honest scope note:** for the population that refines a ticket and then builds it interactively (or never queues it overnight), the relocated signal is never reached — for them the refine-time heads-up is *removed, not relocated*. This is an accepted trade (the early signal "lands too early to act on" for the overnight decision), but it is a real loss of signal for that path, not a delay.

## Phases
- **Phase 1: Curation→launch frozen-list handoff** — make `cortex overnight launch` execute the operator's approved curated set instead of re-deriving it, closing the latent "[R]emove doesn't stick" gap.
- **Phase 2: Overnight curation suitability set-aside** — the `/overnight` Step 6 gate sets aside poor-overnight-fit specs with reasons (bias to exclude), surfaced and re-addable, then passes the frozen set to launch via Phase 1.
- **Phase 3: Remove the /refine advisory** — excise the Step 6 overnight-candidate advisory and its run-mode detection, documenting that its protection now lives at overnight curation.

**Phase ordering is a correctness constraint, not just sequencing:** Phase 3 (removing the only existing suitability signal) must land only after Phase 2 (the replacement surface) is live — otherwise a window opens with no suitability signal anywhere. Ship Phases 1–3 as a single PR, or gate the Phase-3 removal commit behind Phase-2 completion. Do not land the "byte-neutral, touches no test" Phase-3 deletion first.

## Requirements

**Priority (MoSCoW)**: Requirements 1–6, 9, 10, and 12 are **must-have** — the feature is either incoherent or unshippable without them. Requirements 7, 8, and 11 are also **must-have**, on correctness/safety/policy grounds specifically: re-adding a hard-ineligible item would crash bootstrap (7); set-aside stability across re-renders is the structural floor of the no-silent-drop guarantee, not UX polish (8 — see R6); and the affordance-preservation note is a CLAUDE.md authoring obligation (11). Won't-do is enumerated under Non-Requirements. There is no should-have tier in this spec — every requirement is load-bearing for either coherence or the safety guarantee.

1. **Launch accepts an explicit curated selection set**: `cortex overnight launch` gains an input by which the caller supplies the exact set of feature slugs to execute (recommended surface: `--only <comma-separated-slugs>`). Acceptance: `cortex overnight launch --help` lists the new option (binary: option name appears in `--help` output / `grep` of `cortex_command/cli.py` launch subparser = 1). **Phase**: Curation→launch frozen-list handoff.

2. **handle_launch executes exactly the supplied set; absent the input, behavior is unchanged**: When a curated set is supplied, the bootstrapped `overnight-state.json` contains exactly those features (scored, batched, repo-validated, and spec-extracted through the existing machinery); slugs not in the set are excluded, and a supplied slug that is not eligible is surfaced as an error rather than silently dropped. When the input is **absent** (distinct from empty — `getattr(args, "only", None) is None`), `handle_launch` behaves as today (full re-selection); when **empty**, it returns the existing `nothing_ready`-style refusal. Acceptance: new test in `cortex_command/overnight/tests/test_cli_launch_prepare.py` — (a) launch with a curated subset → state features == that subset; (b) launch with no curated input → unchanged (existing assertions still pass); (c) launch with empty set → `nothing_ready`. Binary: `just test` exits 0. **Phase**: Curation→launch frozen-list handoff.

3. **The frozen curated set must be dependency-closed; launch refuses a non-closed set fail-loud**: When the supplied set keeps (or re-adds) a feature whose in-session blocker (`intra_session_blocked_by`) is not also in the set, `launch` refuses with an error naming the missing blocker(s) rather than (a) silently refusing the dependent or (b) running it with an unmet precondition. The `/overnight` Step 6 set-aside rubric is correspondingly blocker-aware: it does not set aside a feature that is the in-session blocker of a kept feature without surfacing that the dependent must also be set aside (or the blocker re-added). Acceptance: new test — a curated set that drops an in-session blocker but keeps its dependent → `launch` returns a `dependency_not_closed`-style error naming the blocker, and never bootstraps the dependent with a dangling `intra_session_blocked_by`. Binary: `just test` exits 0. **Phase**: Curation→launch frozen-list handoff.

4. **The `/overnight` skill passes the post-curation frozen set to launch; what is approved is what executes**: At Step 7, after Step 6 curation, the skill invokes `cortex overnight launch` with exactly the active pool displayed at the moment of approval. 5. **`[R]emove` now sticks at execution (latent-bug fix)**: A feature removed at the Step 6 gate does not appear in the launched/bootstrapped set. Acceptance: covered by Requirement 2's test (curated set excludes the removed slug → state features exclude it). **Phase**: Curation→launch frozen-list handoff.

6. **Step 6 sets aside poor-overnight-fit specs with per-item reasons, biased toward exclusion**: At the approval gate, for each candidate whose `spec.md` is already displayed inline, the skill applies a suitability judgment and moves poor unattended candidates into a separate, numbered "Set Aside" pool — each with a one-line reason — defaulted to excluded. The judgment rubric (the *What/Why*, seeded from the removed `/refine` advisory; the LLM determines application) names both **mechanical** signals (an acceptance criterion marked Interactive/session-dependent; unresolved items under the spec's `## Open Decisions`) and **soft** signals (needs network/credentials the sandbox lacks; leans on human-visual or human-judgment verification; exploratory/under-specified). Acceptance: `grep` of `skills/overnight/SKILL.md` or `references/new-session-flow.md` for the set-aside rubric/section = 1; behaviorally Interactive/session-dependent. **Phase**: Overnight curation suitability set-aside.

7. **No silent drop; the guarantee is structural at the approval→launch boundary**: Every set-aside item is shown with its reason at the gate, and the operator can re-add any of them before approval via a new conditional menu verb `[I] Include a set-aside item` (rendered only when the set-aside pool is non-empty). The no-silent-drop guarantee does not rest on the LLM remembering prior decisions: immediately before *every* approval prompt the skill re-displays the complete current state — the active pool, the set-aside pool (with reasons), and the hard-ineligible pool — and the set passed to `launch` (R4) is exactly the active pool the operator approves. Because what is displayed-and-approved is what executes (R2/R4), a silently-reversed re-add is visible to the operator in the final display before they approve. ## Non-Requirements

- **No persistent/cross-session re-add memory.** A re-add is a one-shot, session-scoped override; it holds through this session's re-renders and into the frozen list, but the system does not remember it to suppress the same set-aside in a future curation. (Avoids a new persistent store; the within-session floor is structural via R7's re-display + R4's approval==execution tie, not via a store.)
- **No new `set_aside` field in the selection envelope.** The suitability set-aside lives in the skill's Step 6 presentation and the Phase-1 frozen-set handoff — `select_overnight_batch` / `_selection_summary_payload` are not extended with a suitability channel.
- **No suitability logic in `filter_ready` / `select_overnight_batch` / the runner.** Suitability is never a hard eligibility gate and never re-evaluated by the unattended harness — only the human-frozen list executes. (This is "never a hard gate," deliberately; the structural-separation reconciliation is in Technical Constraints.)
- **No backend-awareness in the relocated logic.** Overnight is already fail-closed to the `cortex-backlog` backend upstream of selection; the new logic adds no backend branching and does not route through the fail-open `cortex-read-backlog-backend` reader.
- **No change to `/refine`'s `phase_transition` event-logging boundary** (the §5 skip-note); this feature does not "repair" it.
- **No session-size cap introduced.**

## Edge Cases

- **Curation drops an in-session blocker but keeps its dependent** (the dominant non-dependency-closed case, and one the bias-to-exclude rubric can actively create): `launch` refuses fail-loud, naming the missing blocker so the operator re-adds it (or removes the dependent too). It must NOT silently refuse the dependent (pre-group filter behavior) or run it with an unmet precondition (post-group filter behavior — the orchestrator round gate treats an absent blocker as non-blocking). (Requirement 3.)
- **Operator re-adds a hard-ineligible item**: refused with the item's reason (missing artifact / blocked / epic) — it never enters the active set. (Requirement 8.)
- **`--only` includes a slug that is not eligible at launch time** — three distinct causes the error message must disambiguate: a typo, external drift (became blocked/merged since prepare), or curation-induced (the operator/set-aside removed an in-session blocker). Launch surfaces it as an error rather than silently dropping it; the dependency-closed case routes to Requirement 3's blocker-named message.
- **prepare-time vs launch-time eligibility drift**: the operator curates against `prepare`'s prepare-time universe, but `launch` re-derives eligibility at launch time; a slug that was eligible (or intra-session-blocked) at prepare can change class by launch. This is fail-loud per the above, not silent.
- **`--only` set is empty** (operator removed everything): launch returns the existing `nothing_ready`-style refusal rather than bootstrapping an empty session.
- **All eligible candidates are set aside** (aggressive bias-to-exclude): the active pool is empty until the operator re-adds; the gate makes the empty-active state obvious so the operator re-adds rather than approving an empty run.
- **Under-flagging (the residual risk bias-to-exclude does NOT cover)**: a genuinely poor-fit spec the prose judge misses stays in the active pool and runs unattended. The backstop is the operator reviewing the full plan + inline specs at the gate (which they already see) and the recoverability of an overnight failure (surfaced in the morning report) — not a structural detector. This is an accepted residual risk of the prose-judgment choice; see Technical Constraints.
- **A spec with `## Open Decisions: - None.`** (idiom for "no open decisions"): not a set-aside trigger — only genuinely unresolved decisions count (matches the `complexity_escalator` idiom-exclusion convention).
- **`cortex overnight launch` invoked without the curated input** (direct CLI use, no skill): unchanged full-selection behavior — back-compatible.

## Changes to Existing Behavior

- **MODIFIED**: `cortex overnight launch` re-selected the batch from scratch on every invocation → now executes the caller-supplied curated set when provided (full re-selection remains the default when no set is supplied).
- **MODIFIED**: Step 6 `[R]emove` was display-only with respect to execution (launch re-selected, re-including the removed feature) → removals now reach the executed set.
- **ADDED**: Step 6 gains a "Set Aside" pool (poor-fit specs, excluded by default, with reasons) and an `[I] Include` re-add verb; suitability judgment now happens at overnight curation.
- **REMOVED**: `/refine`'s Step 6 overnight-candidate advisory and its `phase_transition`-count run-mode detection.
- **UNAFFECTED (explicitly verified, not changed)**: `cortex overnight start` (run-now and `--dry-run`), `cortex overnight schedule`, and the MCP `overnight_start_run`/`overnight_schedule_run` tools consume the already-bootstrapped `overnight-state.json` via `--state`, not `launch`'s live selection — they are insulated from the `--only` change. The `--dry-run` reference snapshot (`tests/fixtures/dry_run_reference.txt`) is produced from a pinned state fixture and is not perturbed.

## Technical Constraints

- `handle_launch` (`cortex_command/overnight/cli_handler.py:2100`) currently calls `select_overnight_batch` (`backlog.py:1020`) fresh. The curated-set input must restrict the executed set to the operator's frozen slugs while flowing them through scoring/batching/`validate_target_repos`/`extract_batch_specs`, AND enforce dependency-closure (Requirement 3) before bootstrap.
- **Intra-session dependency machinery is the load-bearing hazard.** `select_overnight_batch` derives each intra-session dependent's round and `intra_session_deps` from the *full* item universe (`backlog.py:1051-1094`); `plan.py:427` copies `intra_session_deps` onto each feature's `intra_session_blocked_by`. The downstream consumers — the orchestrator round gate (`orchestrator-round.md:189-190`) and `sweep_blocker_failed_dependents` (`state.py:709`) — treat a blocker absent from `state.features` as non-blocking. Therefore a filtered subset must never leave a kept dependent with a blocker slug absent from the executed set; Requirement 3's dependency-closure refusal is what prevents a dangling `intra_session_blocked_by` from reaching state.
- `filter_ready` stays a pure eligibility gate (existence/status/blockers/epic/branch-merge) — do not add spec-body reads there.
- **`prepare` re-render must re-apply curation.** Step 6's display is rendered from `cortex overnight prepare` (read-only, re-selects the full eligible set on each `[T]`/`[R]` re-render). The skill must overlay its maintained curation (active/set-aside pools, operator re-adds) on `prepare`'s raw output rather than presenting `prepare`'s uncurated selection — otherwise removed/set-aside items reappear on re-render (Requirement 9). `prepare` itself is not given the `--only` input; the overlay is the skill's responsibility.
- **Structural-separation reconciliation (CLAUDE.md).** CLAUDE.md prefers structural separation over prose-only enforcement for sequential gates, permitting prose-only "where the cost of occasional deviation is low." Suitability here is intentionally prose-only (Non-Requirements) because: (a) a human approval gate sees every active item and every set-aside reason before anything runs; (b) a missed flag's cost is a recoverable overnight failure surfaced in the morning report, not data loss or an irreversible action; (c) the deterministically-checkable signals (`## Open Decisions`) couple to fragile spec-template markers (`Interactive/session-dependent` is comment-only and unpinned), so a deterministic matcher would be brittle. This is a deliberate, documented acceptance of the structural-vs-prose trade, not an oversight — the structural floor is applied where it matters (the approval→execution boundary, R4/R7), not on the judgment itself.
- The Step 6 suitability rubric is authored as soft positive-routing prose (MUST-escalation policy: no new MUST without the evidence-artifact + effort-first ritual), and minimally — directing the LLM only where its handling would otherwise be unclear (judge-once + re-apply-curation across re-renders; three-pool re-add semantics; dependency-closure awareness), trusting its judgment for the signal application.
- Mirrors: canonical `skills/refine/SKILL.md` → `plugins/cortex-core/skills/refine/`; canonical `skills/overnight/SKILL.md` + `references/` → `plugins/cortex-overnight/skills/overnight/`. Edit canonical only; `just build-plugin` regenerates; commit canonical + mirror together.
- Removing the refine advisory is byte-neutral on the L1 ratchet (frontmatter-only) and touches no test, including `test_lifecycle_kept_pauses_parity.py` (the advisory is prose, not an `AskUserQuestion`).

## Open Decisions

- **Mechanism for threading the dependency-closed curated set through `select_overnight_batch`/`group_into_batches`**: restrict the eligible universe to the closed set before grouping (so batch/round assignment recomputes cleanly over the subset) vs. select-then-filter-then-regroup. This is now a pure implementation choice — NOT a correctness fork — because Requirement 3 guarantees the set is dependency-closed before either path runs, so neither orphans a dependent. Deferred because the cleanest threading is only visible once working in `group_into_batches` against a closed subset.

## Proposed ADR

### Proposed ADR: 0021-curation-owns-the-frozen-run-list-suitability-is-not-a-selection-gate

Context: overnight `launch` historically re-ran `select_overnight_batch` on every invocation, so the operator's interactive Step 6 curation never reached execution (a latent `[R]emove`-doesn't-stick gap), and overnight-suitability was judged early in `/refine`. Decision: the interactive curation gate becomes the single owner of "what runs unattended" — `launch` executes the operator's frozen curated set (full re-selection only as the no-curation default), the active list approved at the gate is exactly the set bootstrapped and run, suitability is judged by the LLM at curation time (where the spec bodies are in hand and a human is present), surfaced with reasons and re-addable, and the unattended harness makes no suitability calls and never re-selects. The frozen set must be dependency-closed; `launch` refuses a non-closed set fail-loud. Trade-off: `launch` is no longer a pure idempotent re-select. Concretely, the only live consumer of `launch`'s selection is the `/overnight` skill (and direct CLI use); `cortex overnight start`/`schedule` and the MCP start/schedule tools consume the already-bootstrapped state via `--state` and are unaffected — verified, not assumed. Suitability is judged by prose-guided LLM rather than a deterministic Python gate, accepted because every set-aside is surfaced and re-addable at a human gate and a missed flag is a recoverable overnight failure, not data loss; the structural guarantee is applied at the approval→execution boundary (what is approved is what runs) rather than on the judgment. This is hard to reverse (changes `launch`'s contract), surprising without context (why doesn't `launch` just re-select?), and a real trade-off (honor-curation vs. always-fresh-select; prose-judgment-with-human-backstop vs. structural gate).
