# Specification: scale-research-fanout-by-complexity

## Problem Statement

The parallel research fan-out is modest and structurally flat. `/cortex-core:research` (which `/refine` and `/lifecycle` delegate to) sizes its agent count with `agent_count = max(tier_count, criticality_count)` (`skills/research/SKILL.md:47-57`), which tops out at 5 and — because it's a `max()` of two scalars — gives the complex+critical corner the *same* count (5) as simple+critical or complex+high. The two factors never compound, so the hardest, riskiest work gets no more investigation than work elevated on a single axis. Meanwhile `/cortex-core:discovery` doesn't fan out at all: its research phase is a sequential single-orchestrator pass. This feature makes research depth scale with *both* complexity and criticality combined — roughly doubling for complex and/or critical work, peaking at the complex+critical corner — and brings discovery onto the same scaled engine.

## Phases

- **Phase 1: Shared fan-out engine** — Replace the `max()` model with a 2D tier×criticality count matrix (corner = 10), introduce hybrid angle selection (mandatory core + orchestrator-chosen angles), extract the matrix + dispatch protocol into a shared reference consumed by `/research`, add a regression test, and reconcile the stale single-vs-parallel docs. `/refine` and `/lifecycle` inherit automatically via their existing delegation to `/research`.
- **Phase 2: Discovery integration** — Add a research-sizing complexity/criticality assessment to discovery's Clarify phase, and rewrite discovery's Research phase to consume the shared fan-out engine while preserving discovery's own `## Architecture`/`### Pieces` synthesis schema (which the Decompose phase depends on).

## Requirements

1. **2D count matrix replaces `max()`**: `skills/research/SKILL.md` Step 2 expresses agent count as an explicit 2-row × 4-column lookup table (tier × criticality), not a `max()` of two scalars. Both axes increase the count monotonically; the complex+critical cell is the strict peak. Proposed grid (adjustable at approval):

   | tier \ criticality | low | medium | high | critical |
   |--------------------|-----|--------|------|----------|
   | **simple**         | 3   | 4      | 5    | 6        |
   | **complex**        | 5   | 6      | 8    | 10       |

   This is the **corner-anchored** ramp (user-selected at approval): the complex+critical corner doubles (5→10), single-axis cells rise modestly, and the cap stays at 10 (the research-supported concurrency/diminishing-returns ceiling). The grid is the *resolved* value, not a proposal. **Phase**: Shared fan-out engine. **Acceptance**: `grep` of the canonical fan-out reference (R6) shows an 8-cell table; the floor cell (simple+low) is 3 and the corner cell (complex+critical) is 10; no occurrence of `max(tier_count` remains in `skills/research/SKILL.md` (`grep -c 'max(tier_count' skills/research/SKILL.md` = 0).

2. **Monotonic + peak invariant is testable**: A regression test asserts, for the resolved grid, that (a) every cell is ≥ its left neighbor and ≥ its upper neighbor (monotonic on both axes), (b) the floor = 3, (c) the corner = 10, and (d) no cell exceeds the corner. **Phase**: Shared fan-out engine. **Acceptance**: a new `tests/test_research_fanout_matrix.py` (or equivalently-named test) parses the grid from the canonical reference and asserts (a)–(d); `just test` exits 0.

3. **Hybrid angle selection**: The dispatch protocol defines a *mandatory core* that always runs — Codebase, Web, Requirements & Constraints — plus an Adversarial/critique agent that is **always present for high and critical** work and dispatched **last**, over a summary of the other agents' findings (preserving the centralized-synthesis error-correction layer). The remaining slots the matrix buys are **chosen by the orchestrator per task** and must be distinct, non-redundant angles (the *what*); how to keep them distinct — including subdividing an existing angle by scope when truly distinct angles run out — is left to orchestrator judgment per the What/Why-not-How principle (one illustrative example, e.g. codebase-by-subsystem, suffices in the reference). The angle roster is **not** a hardcoded ordered list beyond the mandatory core, and there is **no** topic→angle keyword router. **Phase**: Shared fan-out engine. **Acceptance**: the canonical fan-out reference names the mandatory-core set and the always-last adversarial rule for high/critical, and states the orchestrator-chooses-remaining + distinct-non-redundant rule in prose (verifiable by `grep` for "mandatory core" anchor); `just test` exits 0.

4. **Output schema follows actual angles**: Because the angle set is now partly dynamic, `/research`'s synthesis (`skills/research/SKILL.md` Step 4) emits one section per dispatched angle (mandatory core always; chosen angles labeled by the angle the orchestrator selected; Adversarial when present), rather than a fixed five-heading template. Empty/failed-agent handling and the `## Open Questions` contract are preserved. The stale self-documentation at `skills/research/SKILL.md:194` ("Step 4's `### Output structure` block is the canonical schema source") is reworded to reflect that the schema is now angle-driven (only `## Open Questions` remains a fixed contract heading). This is safe because no machine consumer parses `/research`'s lifecycle `research.md` by section name except `## Open Questions` (`cortex_command/lifecycle/complexity_escalator.py`). **Phase**: Shared fan-out engine. **Acceptance**: Step 4 describes per-angle section emission keyed to dispatched angles; the line-194 self-doc no longer claims a fixed canonical schema; the `## Open Questions` section semantics (consumed by `cortex-complexity-escalator`) are unchanged (`grep -c 'Open Questions' skills/research/SKILL.md` ≥ 1).

5. **Stale single-vs-parallel docs reconciled**: The pre-existing binary "Single research / Parallel research" framing — already inconsistent with the graduated formula — is updated to reference the matrix. Sites: `skills/lifecycle/references/criticality-matrix.md` (Scaled-behaviors column), `docs/agentic-layer.md` (criticality table), `skills/lifecycle/assets/model-selection.md` (parallel-research rows), plus the "3–5" range strings in `skills/research/SKILL.md` frontmatter and `docs/skills-reference.md`. Additionally, the research-sufficiency signals at `skills/lifecycle/references/clarify.md:99-100` that reference "research.md's codebase analysis" are reworded to not assume a fixed `## Codebase Analysis` heading (since R4 makes the heading set angle-driven) — phrase them in terms of "codebase findings" present anywhere in research.md. **Phase**: Shared fan-out engine. **Acceptance**: none of these files assert that research is "single" for high criticality or that parallel research is critical-only; the clarify.md sufficiency signals no longer name a fixed heading; the "3–5" literal range strings are updated to the new range (`grep -rn '3–5\|3-5' skills/research/SKILL.md docs/skills-reference.md` returns no agent-count range claim).

6. **Shared fan-out reference**: The count matrix + hybrid angle-selection + dispatch protocol live in one canonical reference at **`skills/lifecycle/references/fanout.md`** (chosen because it is consumed by both `/research` and discovery — matching the existing shared-reference pattern of `load-requirements.md` and `orchestrator-review.md` under `skills/lifecycle/references/`), cited by both entry points so they cannot drift. `skills/research/SKILL.md` consumes it rather than inlining the matrix. **Phase**: Shared fan-out engine. **Acceptance**: `skills/lifecycle/references/fanout.md` exists and contains the matrix; `skills/research/SKILL.md` cites it; discovery's research reference (R8) cites the same file; dual-source mirror parity (`tests/test_dual_source_reference_parity.py`) passes.

7. **Discovery Clarify assesses research-sizing complexity + criticality, biased upward**: Discovery's Clarify phase (`skills/discovery/references/clarify.md`) produces a complexity (`simple|complex`) and criticality (`low|medium|high|critical`) assessment whose **sole purpose is sizing the research fan-out** — explicitly distinct from the implementation-complexity that `/refine`/`/lifecycle` assess when a ticket is ready to build. This **deliberately reverses** the existing documented design decision at `skills/discovery/references/clarify.md:62-66` (the Thought/Reality row "Discovery Clarify does not assess implementation complexity — there is nothing to implement yet"); that row is reworded to scope the research-sizing carve-out, and any test asserting the old constraint is updated.

   **Upward bias (user directive)**: Because discovery sets the *initial direction* of an epic — a wrong direction here propagates across every ticket the discovery spawns — discovery's research-sizing assessment skews higher than the same topic would rate under refine/lifecycle. The clarify guidance encodes this as: criticality floors at `medium` (never `low`) and rises to `high`/`critical` when the topic seeds a whole epic or sets multi-ticket direction; complexity skews `complex` for any multi-faceted or epic-seeding topic. The intent (the *why*: discovery is high-leverage and divergence is expensive) is stated so the model applies judgment rather than following a rigid lookup. **Phase**: Discovery integration. **Acceptance**: discovery clarify.md emits the two assessments as named clarify outputs, states the upward-bias intent and the `medium` criticality floor, and the stale row is reworded (`grep -c 'research-sizing\|research sizing' skills/discovery/references/clarify.md` ≥ 1; `grep -ci 'epic' skills/discovery/references/clarify.md` ≥ 1); `just test` exits 0.

8. **Discovery Research uses the shared fan-out engine, keeps its schema**: Discovery's Research phase (`skills/discovery/references/research.md`) dispatches parallel agents sized by the shared matrix (R6) using the persisted clarify assessment (R9), replacing the sequential single-orchestrator §2–§5 investigation. Discovery still synthesizes findings into its existing artifact schema — `## Architecture` with `### Pieces` and `### How they connect`, plus Research Questions / Feasibility — because that schema is **machine-parsed**, not merely read: `cortex_command/discovery.py:_extract_architecture` (the `cortex-discovery generate-brief` path) hard-parses `## Architecture` → `### Pieces` / `### How they connect` via regex and raises `BriefGenerationError("missing-architecture-section")` if absent, and decompose.md treats `### Pieces` as "the decomposition source of record." **Phase**: Discovery integration. **Acceptance**: discovery research.md cites the shared fan-out reference and retains the `## Architecture`/`### Pieces`/`### How they connect` output schema; `cortex-discovery generate-brief` still succeeds against a discovery research.md (i.e. `tests/test_discovery_brief.py` passes); `just test` exits 0.

9. **Discovery research-sizing assessment persists across phase-resume**: Because discovery supports independent phase entry (`/cortex-core:discovery research`), the R7 complexity/criticality assessment must be persisted at Clarify time and read back at Research time — conversation context alone is insufficient across a resume boundary. The assessment is written to a durable location (discovery's `events.log` via the `cortex-discovery` helper, mirroring the lifecycle's `lifecycle_start`/`complexity_override` pattern, or research.md frontmatter) and read at Research entry. If Research is entered with no persisted assessment (legacy discovery dirs or direct phase-jump before Clarify ran), the matrix inputs default to the floor (simple/low) or the user is re-prompted — never an unhandled-missing-input failure. **Phase**: Discovery integration. **Acceptance**: a persistence write exists at Clarify and a read exists at Research entry (verifiable by `grep` in the discovery references and/or `cortex_command/discovery.py`); a test covers the resume-without-assessment default path; `just test` exits 0.

## Non-Requirements

- **No interactive cost circuit-breaker.** Doubling agents doubles token spend; the matrix cap (corner = 10) is the only cost guard. A per-dispatch budget breaker for the interactive path is out of scope (it doesn't exist today for research and is a separate concern).
- **No change to the overnight pipeline's 1–3 feature-level concurrency limit.** That governs concurrent *features*, not research sub-agents within one invocation; untouched.
- **No model-selection change.** Which model research agents run on is unchanged; this feature only changes how many agents and which angles.
- **No topic→specialist-angle keyword router.** Angle selection beyond the mandatory core is orchestrator judgment, not a hardcoded mapping.
- **No new MUST/CRITICAL escalations.** New dispatch prose uses soft positive-routing per the CLAUDE.md MUST-escalation policy.

## Edge Cases

- **Count exceeds distinct available angles** (e.g., corner = 10 but a small/narrow task has ~6 genuinely distinct angles): the orchestrator subdivides an existing angle by scope rather than inventing redundant duplicates, and may note in `## Open Questions` when subdivision was reached. The matrix sets an *upper bound* on investigation, not a quota that must be filled with redundant agents.
- **Discovery topic with thin implementation signal**: discovery's research-sizing criticality may legitimately be `low`/`medium`; the matrix still applies and yields the floor-ish counts. Discovery is never forced into a large fan-out for a narrow topic.
- **Adversarial agent at low criticality**: the always-present-adversarial rule binds only for high/critical; at low/medium the adversarial angle is optional (orchestrator may still choose it if the matrix count allows and the task warrants).
- **`/research` standalone mode** (no `lifecycle-slug`): the matrix and hybrid selection apply identically; only output routing (conversation vs `research.md`) differs, as today.

## Changes to Existing Behavior

- **MODIFIED**: `/research` agent count `max(tier_count, criticality_count)` (range 3–5) → 2D matrix (range 3–10, corner = complex+critical).
- **MODIFIED**: `/research` fixed 5-angle ordered roster → mandatory core + orchestrator-chosen distinct angles (+ always-last adversarial for high/critical).
- **MODIFIED**: discovery Clarify — now assesses research-sizing complexity/criticality (previously explicitly did not assess complexity).
- **MODIFIED**: discovery Research — sequential single-orchestrator investigation → parallel fan-out via the shared engine; synthesis schema preserved.
- **MODIFIED**: stale single-vs-parallel framing in criticality-matrix.md, agentic-layer.md, model-selection.md; "3–5" range strings in research frontmatter + skills-reference.md.
- **ADDED**: shared fan-out reference file (`skills/lifecycle/references/fanout.md`); regression test for the matrix invariants; discovery research-sizing assessment + its persistence path.

## Phase boundary

Phase 1 (shared fan-out engine) must be fully landed and green (`just test` exits 0) before Phase 2 (discovery integration) begins — Phase 2 depends on R6's shared reference existing, and Phase 1 carries little integration risk while Phase 2 carries most of it. Treat the Phase 1→2 boundary as a hard gate, not a soft transition.

## Technical Constraints

- **Decompose schema dependency**: discovery's Decompose phase reads `## Architecture` → `### Pieces` / `### How they connect` from `research.md` (`skills/discovery/references/decompose.md` calls `### Pieces` "the decomposition source of record"), AND `cortex_command/discovery.py` machine-parses the same headings (plus `## Headline Finding`) for the R4 gate's brief and the `score-corpus` path. The discovery research rewrite must preserve these sections — this is why discovery shares the fan-out *engine* but not `/research`'s output schema. Note: `## Headline Finding` is parsed by `cortex_command/discovery.py` but is a pre-existing inconsistency (not currently in the discovery research template) — the R8 rewrite must not drop it if present and should not be derailed by it; reconciling that gap is out of scope for this feature. `/research`'s own schema (R4) is safe to make dynamic because no machine consumer parses it by section name except `## Open Questions` (consumed by `cortex_command/lifecycle/complexity_escalator.py`).
- **Dual-source mirrors**: edit `skills/` only; `plugins/cortex-core/` regenerates via pre-commit hook (`tests/test_dual_source_reference_parity.py`).
- **SKILL.md 500-line cap** (`tests/test_skill_size_budget.py`): `skills/research/SKILL.md` is ~258 lines; extracting the matrix + dispatch to a shared reference keeps it well under budget.
- **Authoring policy**: prescribe What/Why not How; soft routing (no new MUST without evidence artifact) per CLAUDE.md.
- **No code computes the count**: it remains prose-applied by the model; the regression test asserts the *grid's* invariants (parsed from the reference), not a Python dispatch function.
- **`## Open Questions` contract**: consumed by `cortex-complexity-escalator`; must survive the Step 4 schema change.
