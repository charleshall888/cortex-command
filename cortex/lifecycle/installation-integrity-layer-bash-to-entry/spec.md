# Specification: installation-integrity-layer-bash-to-entry

## Problem Statement

Installed users hit "binary not on PATH" friction whenever skill prose invokes a `bin/cortex-*` script that has no corresponding `[project.scripts]` entry point. The repo's `bin/` directory is not on an installed user's PATH at any layer â€” only wheel binstubs in `~/.local/bin/` are â€” so any skill that shells out to `cortex-lifecycle-state`, `cortex-resolve-backlog-item`, `cortex-commit-preflight`, etc. fails with `command not found` on a non-dogfooder install. The discovery surfaced 8 distinct drift signatures across â‰Ą9 skill files; the pain is observed, not hypothetical. This work closes the friction class by promoting the 13 skill-prose-referenced scripts to wheel-tier Python entry points (matching the 22 entries already in `pyproject.toml`), and adds a SessionStart PATH self-test in `plugins/cortex-core/hooks/cortex-session-start-path-bootstrap.sh` that emits advisory `additionalContext` listing any expected entry point absent from PATH. The install-version pin probe is wholly delegated to ticket 235 (already refined, drift-detector + `--refresh` flag fix). Beneficiaries: installed users (the immediate "command not found" friction class disappears) and the broader wheel-first direction codified in ADR-0002.

## Phases

- **Phase 1: Foundation** â€” Promote `cortex-log-invocation` (the sibling-lookup keystone); ship the corrected dual-channel wrapper template that honors `CORTEX_COMMAND_FORCE_SOURCE=1`; update all remaining `bin/cortex-*` scripts' `"$(dirname "$0")/cortex-log-invocation"` line to a pattern that distinguishes "missing" from "present-but-broken"; ship the Phase 1 smoke test that exercises every rewritten script's invocation pathway; establish the golden-replay parity-test scaffold for `cortex-log-invocation`.
- **Phase 2: DR4 milestone** â€” Promote `cortex-resolve-backlog-item` per DR4; remove the now-stale install_guard comment block at `bin/cortex-resolve-backlog-item:31-36`.
- **Phase 3: Bulk migration** â€” Promote the remaining 11 skill-prose-referenced scripts (mix of PEP 723, `#!/usr/bin/env python3`, and bash); land their parity-test fixtures and tests.
- **Phase 4: PATH self-test** â€” Extend `cortex-session-start-path-bootstrap.sh` with the self-test downstream of the existing PATH augmentation; emit advisory `additionalContext` only (no sentinel file); skip when dogfooder predicates fire.
- **Phase 5: Docs + prose sweep** â€” Add a paragraph to `cortex/requirements/project.md`'s "Wheel-binstub vs working-tree invocation" section documenting `CORTEX_COMMAND_FORCE_SOURCE=1`; sweep skill prose for path-qualified `bin/cortex-<name>` references to promoted scripts and rewrite to bare-name.

## Requirements

1. **`cortex-log-invocation` is a wheel-tier Python entry point**: `pyproject.toml [project.scripts]` contains `cortex-log-invocation = "cortex_command.log_invocation:main"` (final module path TBD during plan; the exact module path may differ but the entry name is fixed). A new module exists with a `main(argv: List[str] | None = None) -> int` signature. The module preserves the existing bash script's stdout/stderr/exit-code contract. Acceptance: after `uv tool install --reinstall --refresh git+<...>@<branch>`, `command -v cortex-log-invocation` resolves to a path under `~/.local/bin/`. **Phase**: Phase 1: Foundation.

2. **Dual-channel wrapper template honors `CORTEX_COMMAND_FORCE_SOURCE=1`**: A canonical wrapper template ships at `bin/cortex-log-invocation` (and is the template for all subsequent promotions). The wrapper's branch order is: (a) if `CORTEX_COMMAND_FORCE_SOURCE=1` is set, exec the working-tree Python module via `python3 -m cortex_command.<module>` (using `CORTEX_COMMAND_ROOT` if set, else `dirname "$0"/..`); (b) else try wheel-import (`python3 -c "import cortex_command.<module>"` to probe, then exec via the module's import path); (c) else fall back to working-tree mode; (d) else error with a helpful message. Acceptance: with `CORTEX_COMMAND_FORCE_SOURCE=1` set, `bin/cortex-log-invocation --help` exits 0 against the working tree even when an installed wheel exists; with the env var unset and a wheel installed, the same invocation exits 0 against the wheel. **Phase**: Phase 1: Foundation.

3. **All `bin/cortex-*` scripts distinguish missing from broken in the sibling lookup**: Every remaining `bin/cortex-*` script's line-1/2 sibling-lookup pattern (`"$(dirname "$0")/cortex-log-invocation" "$0" "$@"` or equivalent) is rewritten to a pattern that (a) silently skips the call when `cortex-log-invocation` is not on PATH (`command -v` fails) â€” preserving the "log-invocation may not exist" graceful path â€” but (b) emits a single-line `stderr` warning when `cortex-log-invocation` IS on PATH and exits nonzero. The canonical rewrite shape:
   ```bash
   if command -v cortex-log-invocation >/dev/null; then
     cortex-log-invocation "$0" "$@" || echo "cortex-log-invocation failed: $?" >&2
   fi
   ```
   Acceptance: (a) `grep -c '"\$(dirname "\$0")/cortex-log-invocation"' bin/cortex-*` = 0; (b) `grep -lE 'command -v cortex-log-invocation' bin/cortex-* | wc -l` â‰Ą (count of `bin/cortex-*` scripts that previously invoked sibling log-invocation); (c) `grep -lE 'cortex-log-invocation failed:.*>&2' bin/cortex-* | wc -l` â‰Ą same count (loud-on-broken contract present in every rewrite). **Phase**: Phase 1: Foundation.

4. **Phase 1 smoke test exercises every rewritten script**: A new test file `tests/test_phase1_sibling_rewrite_smoke.py` (or shell equivalent) invokes each rewritten `bin/cortex-*` script with a trivial argv (e.g., `--help` if accepted, else with `--nonexistent-flag` and asserting nonzero exit from the script's own argparse but no spurious stderr from the sibling-lookup rewrite). The smoke test fails if any rewritten script emits the `cortex-log-invocation failed:` warning under normal conditions OR fails to invoke its own primary work. This catches quoting/escaping regressions from the bulk rewrite without requiring full per-script parity coverage in Phase 1. Acceptance: `python3 -m pytest tests/test_phase1_sibling_rewrite_smoke.py` exits 0. **Phase**: Phase 1: Foundation.

12. **Self-test emits advisory `additionalContext` only â€” no sentinel file**: When one or more expected entry points are missing from the augmented PATH, the self-test emits `{"hookSpecificOutput": {"hookEventName": "SessionStart", "additionalContext": "<message>"}}` on stdout with a factual statement (e.g., `The following cortex entry points are not on PATH: cortex-foo, cortex-bar. 13. **Self-test skips on dogfooder / dev-mode predicates with explicit rationale**: The self-test exits 0 silently (no probe) when ANY of: (a) `CORTEX_DEV_MODE=1` is set; (b) `$CWD/pyproject.toml` exists and contains a line matching `^name\s*=\s*"cortex-command"` (i.e., the user is inside the cortex-command source tree); (c) the hook fails to invoke `python3` or `importlib.metadata` raises. The skip is silent â€” no additionalContext. **Rationale for predicate (b)**: dogfooders working in the source tree see drift between their working-tree `[project.scripts]` and their installed wheel that the self-test cannot detect (the self-test reads the installed wheel's metadata, not the working-tree pyproject). The drift dogfooders introduce is detected by ticket 235's CLI_PIN drift comparison, not by this PATH self-test. Skipping for this population avoids redundant noise on a signal 235 already owns. Acceptance: with `CORTEX_DEV_MODE=1` set, the hook runs to completion and emits no `additionalContext` on stdout. **Phase**: Phase 4: PATH self-test.

14. **Self-test is defensive â€” every error path exits 0 silently**: Mirroring the defensive posture (not the predicate set) of 235's hook, the self-test exits 0 silently on: probe subprocess timeout; `importlib.metadata` raising `PackageNotFoundError`; the parity-exceptions file missing or malformed; any unhandled Python exception. The hook does not brick Claude Code launch under any failure mode. The error-path predicate set differs from 235's because the probe surface is different (importlib.metadata + parity-exceptions parse, vs 235's `cortex --print-root` JSON parse); the defensive posture is shared but not the specific predicates. Acceptance: invoking the hook with `PATH=/nonexistent` (no python3 reachable) exits 0 with empty stdout. **Phase**: Phase 4: PATH self-test.

16. **Parity contract: byte-identical by default; structurally-equivalent only for JSON whose key order is jq-determined**: The parity tests for every promoted script assert byte-identical stdout, stderr, and exit-code matching the captured fixtures BY DEFAULT. The "structurally-equivalent" escape applies only when ALL of the following hold for a specific stream: (a) the stream is a JSON document (object or top-level array), (b) the diff between Python output and bash output is entirely intra-object key reordering (no value changes, no whitespace changes outside JSON's allowed serialization, no number-format changes), AND (c) the test file has an explicit `@pytest.mark.structural_equivalence(stream="stdout"|"stderr")` decorator naming which stream is structurally compared. Anything else â€” locale-dependent strings, trailing whitespace, number formatting drift, line-ending differences, env-var leakage, timestamps â€” is a parity failure. Acceptance: a unit test `tests/test_parity_contract.py` exercises the structural-equivalence helper against intentionally-key-reordered JSON (passes) and against intentionally-whitespace-mutated JSON (fails); the test exits 0. **Phase**: Phase 1: Foundation (the contract ships in Phase 1 alongside the cortex-log-invocation parity scaffold; Phases 2â€“3 consume the established contract).

17. **Fixture capture determinism harness**: Each script's fixture-capture procedure is documented in `tests/fixtures/<script>/README.md` and specifies: (a) the jq version pinned for capture (output of `jq --version`); (b) `LC_ALL=C` is set during capture; (c) `TZ=UTC` is set during capture; (d) for `cortex-log-invocation` specifically, the captured stdout must not contain wall-clock timestamps, PIDs, hostnames, or other non-deterministic content â€” if the bash script emits any such content under normal operation, fixture capture uses a wrapper that filters or freezes them (e.g., piping through `sed 's/[0-9]\{4\}-[0-9]\{2\}-[0-9]\{2\}T[0-9]\{2\}:[0-9]\{2\}:[0-9]\{2\}/<FROZEN_TS>/g'`). The Python re-implementation matches the frozen/filtered form. Acceptance: every `tests/fixtures/<script>/README.md` exists and contains the four required lines (jq version, LC_ALL, TZ, determinism notes). **Phase**: Phase 1: Foundation.

## Non-Requirements

- **The pin probe is NOT in 252's scope.** Ticket 235 owns the SessionStart drift-detector hook, the `--refresh` flag fix in `_ensure_cortex_installed`, and the docs update for the Bash-tool carve-out section. 252 does not duplicate or compete with that hook surface.
- **`cortex-jcc` is NOT promoted.** It is a `just -f $CORTEX_COMMAND_ROOT/justfile` wrapper that requires a working-tree justfile to function; promotion would require shipping the project justfile as wheel data via `importlib.resources` and re-resolving â€” out of scope. It remains bash, in-tree, dogfooder-only.
- **The 12 bin/cortex-* scripts not referenced from skill prose are NOT promoted.** `cortex-archive-rewrite-paths`, `cortex-archive-sample-select`, `cortex-audit-doc`, `cortex-check-events-registry`, `cortex-check-path-hardcoding`, `cortex-count-tokens`, `cortex-invocation-report`, `cortex-jcc`, `cortex-measure-l1-surface`, `cortex-requirements-parity-audit`, `cortex-rewrite-cli-pin`, and `cortex-invocation-report` are justfile/pre-commit utilities that never run from an installed user's PATH chain. Promoting them fails the "earn its place" test.
- **No `bin/.parity-exceptions.md` schema extension.** The existing five-column table (`script | category | rationale | lifecycle_id | added_date`) is reused as-is; the self-test consumes the `script` column and the existing category enum, treating all three categories (`maintainer-only-tool`, `library-internal`, `deprecated-pending-removal`) as doctor-irrelevant.
- **No sentinel file under `cortex/.cache/`.** Earlier drafts proposed `cortex/.cache/path-selftest.json` as a belt-and-suspenders audit trail surviving claude-code#13650/#16538. Dropped because no consumer (skill, hook, or CLI subcommand) is in scope to read it; an unread audit trail is dead code. The known additionalContext-drop risk is accepted as residual â€” see Technical Constraints.
- **No new `cortex doctor` CLI subcommand.** The self-test lives inside the SessionStart hook, not as a separately invokable subcommand. A future ticket may expose it as `cortex doctor --path-self-test` if a manual-invocation use case emerges.
- **No CI shadow mode for parity tests.** Parity verification is done via committed golden fixtures, not by running both bash and Python versions in parallel.
- **No cross-plugin coupling.** A new SessionStart hook in cortex-core is NOT added; the existing `cortex-session-start-path-bootstrap.sh` is extended in place. 235's `cortex-overnight` hook is not modified by 252. Each hook handles its own PATH augmentation independently; no inter-hook coordination protocol is introduced.
- **No measurement of the O(PATH)-walk cost from `command -v cortex-log-invocation`.** The shell built-in cost is sub-millisecond on benign PATHs; an acceptance test that benchmarks it would over-engineer a non-issue. If overnight runs surface measurable latency growth, a follow-up addresses it.
- **No "structurally-aligned" carve-out for already-Python scripts.** All 13 promoted scripts (regardless of pre-promotion shebang) get â‰Ą3 captured fixtures and a parity test. The earlier draft's exception for already-Python scripts was undefined and self-attested; closing it removes ambiguity.

## Edge Cases

- **`cortex-command` package not importable** (broken install): `importlib.metadata.entry_points(group='console_scripts')` may raise `PackageNotFoundError`. The self-test exits 0 silently per requirement 14.
- **`bin/.parity-exceptions.md` malformed**: a missing-table or malformed-table state breaks the linter independently of 252. The self-test treats parse failure as "no exceptions" (over-enumerate is preferable to silently skipping checks) and proceeds with the full importlib.metadata set. If that produces noisy advisories, the parity-exceptions file's drift is the root cause and gets surfaced upstream.
- **User has a fresh `uv tool install` that completed seconds before SessionStart**: the augmented PATH includes `~/.local/bin/`; all entry points should resolve. If the wheel's `[project.scripts]` table has an entry that uv hasn't yet symlinked (extremely narrow race window), the self-test reports a false positive once. The next session corrects it.
- **Dogfooder running in working-tree with a stale wheel installed**: predicate (b) in requirement 13 detects this (`$CWD/pyproject.toml` names cortex-command) and silently skips. The dogfooder's CLI version drift is detected by 235's CLI_PIN comparison; 252's PATH self-test does not duplicate that signal.
- **Both 235 and 252 fire `additionalContext` in the same SessionStart**: post-marketplace-fast-forward the user may see two advisories â€” 235's "CLI 2.0.0 â†’ 2.1.0" drift message AND 252's "entry points missing from PATH" advisory. This double-notification is acceptable because each addresses a structurally distinct dimension of drift (version drift vs PATH coverage drift); the remediation for both is the same `uv tool install --reinstall --refresh` invocation, which Claude can dedupe in its response.
- **`CORTEX_COMMAND_FORCE_SOURCE=1` set but `cortex_command` is not importable from the working tree** (e.g., user is outside the repo with the env var set): the wrapper exec fails with a Python ImportError message on stderr. When this fires from a sibling-script's line-1 invocation of `cortex-log-invocation`, the rewritten pattern in requirement 3 emits a single-line `cortex-log-invocation failed: 1` warning to stderr (loud, not silent). When this fires from a direct user invocation of `cortex-foo`, the ImportError surfaces directly.
- **`cortex-log-invocation` itself fails or is uninstalled mid-session**: every promoted script's `command -v cortex-log-invocation >/dev/null && cortex-log-invocation ... || echo "cortex-log-invocation failed: $?" >&2` degrades to no-logging with a single stderr line if log-invocation is broken on PATH, or silently if it's absent. The script's primary work proceeds. Telemetry gap is acceptable; the stderr warning ensures the gap is observable rather than invisible.
- **`additionalContext` silently dropped by claude-code#13650 or #16538**: the user sees nothing this session. The next session re-runs the self-test; if the gap persists, the bash invocation will fail directly and Claude can diagnose from the `command not found` error. This is the accepted residual risk (see Technical Constraints).
- **`cortex/.cache/` exists from an earlier draft that wrote sentinels**: post-migration, the path-selftest.json file may exist as a stale artifact in some users' trees. No cleanup requirement â€” the file is gitignored and harmless. A future ticket may add a one-time-cleanup hook if user reports surface.
- **PATH resolution preference between working-tree `bin/cortex-log-invocation` and wheel binstub `~/.local/bin/cortex-log-invocation`**: the bootstrap hook prepends `$HOME/.local/bin` to PATH (line 29), so the wheel binstub wins by default on dogfooders. If the dogfooder additionally adds their working tree to PATH before invoking Claude Code, the working-tree wrapper may resolve first; the dual-channel wrappers behave identically in both cases (both honor `CORTEX_COMMAND_FORCE_SOURCE=1` first, fall through to wheel-import second, working-tree third).
- **Promoted module's argparse breaks an existing JSON contract**: the golden-replay parity tests catch this via requirement 16's byte-identical default. Phase 3's bulk migration must not merge without all parity tests green.
- **`cortex-lifecycle-state`'s torn-line behavior is undocumented at spec-write time**: requirement 8's pre-capture verification step makes the implementer record the current behavior before writing Python. The Python matches what bash does. If bash currently fails on torn lines, the Python fails too â€” parity is preserved, but the original "jq-tolerant" framing is replaced by "behavior-matches-current-bash."
- **Rollback of Phase 1**: if Phase 1's combined PR (cortex-log-invocation promotion + sibling-pattern rewrite across all bin/cortex-*) breaks the bash invocation chain post-merge, the rollback procedure is: (a) `git revert` the merge commit to restore the previous bin/ state; (b) re-run `uv tool install --reinstall --refresh` to restore the wheel binstub set; (c) the parity-test fixtures committed in Phase 1 remain in tree but become inert (harmless). The Phase 1 smoke test (requirement 4) gates against regressions before merge.

## Technical Constraints

- **Wheel-binstub vs working-tree invocation** (`cortex/requirements/project.md:38-39`): the dual-channel wrapper's `CORTEX_COMMAND_FORCE_SOURCE=1` escape hatch is the project-sanctioned override for the constraint's "use `python3 -m` invocation to run against the working tree when wheel reinstall between phases is not feasible" rule.
- **Plugin-imports-zero-cortex-modules contract** (`docs/internals/mcp-contract.md:3-4`): the PATH self-test runs from `cortex-session-start-path-bootstrap.sh` which is a `cortex-core` plugin hook, NOT cortex-overnight. cortex-core's plugin contract permits importing `cortex_command.*` (only cortex-overnight's plugin is bound by the zero-imports rule). The self-test invokes the Python module via `python3 -m`, not by importing into the hook process.
- **Sandbox writability** (ADR-0003): the self-test does NOT write any file; no sandbox-writable path is consumed. (Earlier draft proposed `cortex/.cache/path-selftest.json` under the sandbox-allowlisted `cortex/` umbrella; sentinel dropped per Non-Requirements.)
- **`uv tool install` cache behavior**: new `[project.scripts]` entries do not auto-take-effect on already-installed tools; require `uv tool install --reinstall --refresh`. The 252 migration adds entries the user's installed wheel may not have until they reinstall; the PATH self-test surfaces the gap so they know to reinstall.
- **`cortex --print-root` envelope is forever-public-API** (`docs/internals/mcp-contract.md:22-28`): the self-test does NOT depend on this envelope; it uses `importlib.metadata.entry_points` directly. No envelope-shape coupling is introduced.
- **`bin/cortex-check-parity` exception schema** (`bin/.parity-exceptions.md` lines 5â€“14): the self-test consumes the `script` column and treats all three category enum values as doctor-irrelevant. No schema modification.
- **Claude Code SessionStart `additionalContext` reliability â€” RESIDUAL RISK**: claude-code#13650 and claude-code#16538 may silently drop `additionalContext` output. Mitigation: **none in 252's scope**. The self-test's advisory is best-effort visibility; when dropped, the user sees nothing this session, and the bash invocation will fail directly on next attempt (Claude diagnoses from the `command not found` error). This residual risk is accepted because the alternative (sentinel-file fallback) was dropped for lack of an in-scope consumer (Non-Requirements). A future ticket may revisit if the Claude Code bugs persist materially.
- **`cortex_command/__init__.py` install_guard state** (verified by adversarial review): no longer triggers install_guard at import; promoted modules may import `cortex_command.*` freely at module load. The pre-existing "defer to function body" discipline is unnecessary for newly-promoted scripts.
- **`cortex-log-invocation` is load-bearing**: every existing `bin/cortex-*` script invokes it at line 1â€“2. The Phase 1 sibling-pattern rewrite must ship in the same PR as the log-invocation promotion to keep CI green. **Rollback contract**: if Phase 1 breaks the bash chain post-merge, revert the merge commit and re-run `uv tool install --reinstall --refresh` (see Edge Cases for the full procedure).
- **Golden-replay fixtures are committed and deterministically captured**: fixture capture follows requirement 17's determinism harness (jq version pinned, `LC_ALL=C`, `TZ=UTC`, timestamp filtering). Fixtures are NOT regenerated by tests at runtime. Documented in `tests/fixtures/<script>/README.md`.
- **252 and 235 independently handle PATH augmentation**: each plugin's SessionStart hook bootstraps its own PATH context internally. Claude Code does not guarantee SessionStart hook ordering or shared environment between hooks; each hook is expected to be self-contained. No inter-hook coordination is required or attempted.
