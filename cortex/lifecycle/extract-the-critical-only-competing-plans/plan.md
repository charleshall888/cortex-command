# Plan: extract-the-critical-only-competing-plans

## Overview

Physically relocate `plan.md`'s §1b "Competing Plans (Critical Only)" block (≈104 lines, ≈43% of the file) into a new sibling reference `competing-plans.md` that is Read only on the ≈2% `critical` planning path, leaving a load-bearing stub heading + one-line pointer in `plan.md` and gating the Read inside §1a's `critical` branch through the SKILL.md propagation manifest. Pure physical relocation behind the existing criticality branch — the critical arm runs byte-identical logic, just Read from a new file; non-critical plans stop loading the block. Dispatch is **sequential on trunk (`main`)**, not worktree-parallel: the mirror-coupling forces ordered commits, the kept-pauses re-anchor must follow the `plan.md` cut, and `just test` / `just build-plugin` operate on the shared checkout.

## Risks

- **Dispatch mode is load-bearing, not a default.** This feature must run **sequential on trunk (`main`)**, not worktree-parallel. Three reasons compound: (1) the dual-source mirror coupling requires ordered `build-plugin` + commit steps that parallel worktrees would race; (2) Task 5's kept-pauses re-anchor must be computed against the post-cut `plan.md` (Task 2), a strict ordering; (3) `just test` / `just build-plugin` act on the shared checkout, and the Req-6 edit touches `cortex_command/overnight/**` — worktree verification could test stale editable-install state. Recommend `Implement on current branch (trunk)` at the §4 surface.
- **Mirror-coupling forbids deferring the mirror to a later commit.** On `main`, canonical edits and their regenerated mirror must land in the **same** commit (the drift pre-commit hook blocks a canonical-only commit). Tasks 1–3 edit the working tree; Task 4 is the single Phase-1 commit point; Tasks 5–7 edit the working tree; Task 8 is the single Phase-2 commit point. Do not commit a canonical file before its mirror is regenerated.
- **Cold-read under-triggering is the accepted residual runtime risk.** After extraction the critical arm performs a *separate* Read of `competing-plans.md`; if the model skips the §1a pointer the critical-tier flow silently no-ops with no test catching it. Mitigated by placement (directive on the §1a routing line, one level deep) — which makes over-fetch avoidance structural but leaves the Read itself prose the model must obey, not structurally enforced — and its static presence is guarded by Task 7's §1a-directive-line assertion; but the runtime miss stays untestable and is accepted, not closed. This only affects the ≈2% `critical` path and matches the pre-existing `critical-review-gate` lazy-read pattern.
- **No new ADR.** The approach rides the existing ADR-0009 body-propagation rule and the established `critical-review-gate` lazy-sibling pattern; it introduces no hard-to-reverse or trade-off-bearing decision (pre-committed by discovery R3 / epic #340).
