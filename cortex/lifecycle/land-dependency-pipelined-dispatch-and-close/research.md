# Research: Land dependency-pipelined dispatch and close the plan-authoring gap that forces write-serialization edges

Clarified intent: land #401 item 2 (dependency-pipelined dispatch in implement.md §2) and give plan
authors a way to express "these tasks conflict on a file but are not dependent," so non-logical
write-serialization edges stop being the only available tool. Eight angles dispatched (complex ×
high); the adversarial pass materially reversed two of the ticket's premises — see Open Questions
for the consolidated resolutions.

**Headline synthesis**: the evidence supports the *plan-authoring and picker-surface* half of the
ticket and does not support the *dispatch-mechanics* half. The 27% figure attributed to a single
false edge is a graph-shape effect computed largely from placeholder durations; the pipelining
mechanism itself was priced at ~5% by the same (weak) simulation. ADR-0030's deferral bar —
"a measured case survives token accounting" — is unmet on every count. The durable fix is:
ordering-only edge annotation (not deletion, not mutex-substitution), a corrected plan.md §4 picker
protocol (the actual #358 defect site), tradeoff copy at both picker surfaces, hub-file threshold
≥2 with a seam-resistant caveat, and a width checklist item — with an ADR-0031 that re-affirms the
batch barrier and names the preconditions any future pipelining must clear.

## Web Research

- **Universal prior-art pattern**: mature build/workflow systems all separate "must happen before"
  (dependency edge) from "must not run concurrently" (a second, orthogonal constraint type):
  Ninja `pool`s, GNU Make order-only prerequisites / `.WAIT` / targeted `.NOTPARALLEL`, Gradle
  `mustRunAfter`/`shouldRunAfter` (ordering that applies only if both tasks are scheduled, creates
  no execution dependency, dropped if it would block parallelism — the closest analogue to a
  "dissolvable" edge), Gradle Build Services `maxParallelUsages`, SCons `SideEffect()`, Airflow
  pools, Prefect tag concurrency limits. None overload the dependency edge to mean mutual
  exclusion — which is exactly what plan.md's current remedy does.
- **CPU analogy** (bounded): write-serialization edges are WAW *name* dependencies; worktree
  isolation is the register-renaming step that dissolves them. The analogy fails where both tasks
  structurally rewrite the same regions — git textual merge of two rewrites is a guaranteed
  conflict, so an ordering edge does real work even under isolation (Adversarial §2).
- **Formal container**: Mutual Exclusion Scheduling (Baker & Coffman 1996, TCS 162(2):225–243) —
  DAG + conflict graph whose same-time tasks must form an independent set; NP-hard for ≥3 workers.
  Current behavior = BSP supersteps (level batches with barriers); the target = list scheduling.
  Brent's theorem (T_P ≤ W/P + S) and Dilworth width give citable graph-shape checks.
- **Admission alternatives**: pessimistic pre-flight write-set check (proposed here) vs
  OCC-style optimistic dispatch + validate + abort/retry; retry cost of an aborted agent task makes
  pessimistic admission the right default for this domain.
- **Practitioner literature** (Augment Code guides, 2026): worktree-per-agent for isolation,
  decompose by file-ownership boundary, coordinator agent for hub files — nobody has an automated
  Files-disjointness scheduler; a 2026 preprint (arXiv:2607.00041 "ATM", unreviewed) proposes
  broker-mediated pre-write admission over write-set overlap — validation that the gap is real,
  not established art to copy.

## Requirements & Constraints

- **ADR-0030** (`cortex/adr/0030-mode-agnostic-interactive-dispatch.md`, status **proposed**,
  landed 2026-07-20 from #401): "Dependency-pipelined dispatch — deferred until the upstream
  dispatch-mode substrate is stable and a measured case survives token accounting... The **batch
  barrier stays**." Companion spec Non-Requirements name the blockers: unpriced orchestrator-turn
  cost, an unscoped per-task dispatch event, a `metrics.py` batch-semantics decision, a

  amended/superseded/accepted — not silently ignored.
- **Token economy** (`cortex/requirements/project.md:7,21`): runtime cost is turns × context;
  levers are session length, turn count, fan-out width. **No requirements text values wall-clock
  makespan** — the ticket's justification metric is one the stated quality bar does not price.
- **#39886** (`cortex/requirements/multi-agent.md:94`): `Agent(isolation: "worktree")` may silently
  run against the parent CWD; `implement.md` §2b is a named susceptible caller; no mitigation
  shipped (#208's research proposed detection assertions; none landed). Any change predicating
  safety on "isolation actually in effect" inherits this unmitigated.
- **Scope**: multi-agent.md's dispatch requirements (ConcurrencyManager, model matrix) govern the
  overnight runner, not interactive §2; its #39886 edge-case note is the one cross-reference into
  the interactive lifecycle.
- **ADR-0008** (accepted): picker selection is the `EnterWorktree` authorization surface — any
  change to when/whether the worktree option renders interacts with authorization.
- **ADR-0012** (accepted): the trunk/worktree `dispatch_choice` is resolved at **plan approval**
  (Plan §4) and propagates via the `plan_approved` event; implement §1's picker is the fallback.
  Most dispatch modes are actually chosen at §4 — the surface the ticket's item 4 must reach.
- **Authoring policy**: structural-separation preference (prose-only OK only where deviation is
  cheap), verb-first (behavior in verbs, control flow in prose), MUST-escalation (new MUST language
  needs a linked evidence artifact), What/Why-not-How, zero-sweep (`ZERO_SWEEP_FILES` — implement/
  plan/review.md gain no raw event-emission surface), kept-pauses parity.

## Dispatch Substrate

- **What #401 actually landed**: item 1 shrank to a tightened final-message report contract
  (task name, status, files, verification, commit hash, deviations) — no SendMessage, no
  task-reports files. §2c is deliberately mode-agnostic: "the report is the builder's final message
  in whatever shape the runtime delivers it... completion is always derived from the §2d git
  checkpoint, never from return-delivery shape."
- **Runtime contract** (recorded in ADR-0030 and #401's research; nowhere else in-repo): subagents
  run background-by-default since Claude Code v2.1.198; completion notifications reliably carry the
  final report since v2.1.211; `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1` forces synchronous —
  **only the synchronous direction is pinnable**. `docs/internals/sdk.md:241` still asserts the
  stale "synchronously coupled" premise and should be corrected.
- **Feasibility**: pipelining is mechanically expressible under the current background runtime, but
  not safely expressible as durable mode-agnostic prose — a forced-synchronous session would need a
  different dispatch-issuing pattern (one-message-multi-call batch-returns), i.e. mode-adaptive
  prose, which ADR-0030 calls the worst outcome. Completion observation (git checkpoint) would
  carry over; what changes is the whole loop shape: issue dispatches as tasks become ready across
  turns, react to earliest completion, verify per-completion instead of per-batch.

## Picker & Trunk-Mode Policy

- **Mechanism** (`cortex_command/lifecycle_implement.py:79-115`;
  `cortex_command/lifecycle/branch_decision.py:83-132`): `should_fire_picker` first-match order —
  branch_mode unset/invalid → `"prompt"` mode → **dirty tree (unconditional)** → live interactive
  lock; else suppressed. Firing means the picker *prompts a human* instead of silently auto-entering
  a configured mode — a deliberate fail-safe (pinned by
  `tests/test_lifecycle_implement_branch_mode.py::test_a2_trunk_dirty_tree_fires`).
  `worktree_option_available` = `shutil.which("cortex-lifecycle-prepare-worktree")` — CLI presence
  only. `uncommitted_changes` demotes only the current-branch option. A dirty tree cannot
  mechanically block worktree creation (`git worktree add` checks out a fresh base).
- **Two verbs, two call sites**: implement §1 uses composed `cortex-lifecycle-branch-decision`
  (carries rendering guards); Plan §4 uses raw `cortex-lifecycle-picker-decision` (bare
  `{fire, reason}`) while claiming "Implement §1's preflight owns the picker guards" — §4 asks the
  model to apply guard fields its own command block cannot produce. **§4 is additionally
  self-dirtying**: plan.md is written in §3 and committed in §5, so `git status --porcelain` is
  near-always non-empty at §4 — `dirty_tree` fires *structurally* at plan approval in every
  commit-artifacts repo, noise masquerading as signal. The composed verb works pre-approval (no
  `dispatch_choice` yet → falls through to config+gate → `prompt` with guards) but its `resolved`
  state needs a §4-specific rendering rule.
- **Dirt attribution**: `_is_dirty_tree` is pathspec-less; scoped-porcelain precedent exists
  (`cortex_command/lifecycle/complete_route.py:340-364`). Feature-scoped attribution (own Files vs
  foreign paths) is feasible; true session-ownership attribution has no existing bookkeeping.
- **Change shapes for item 4**: (a) prose-only tradeoff annotation at the picker surfaces —
  cheapest, no wheel change, targets the actual gap (no text explains trunk's downstream
  serialization cost); (b) reorder/gate `dirty_tree` in the wheel — breaks 4+ pinning tests on a
  closed `REASONS` frozenset; (c) scoped dirt attribution — public-API change on a deliberately
  closed surface (spec R3/R4 of `lifecycle-implement-auto-enter-worktree-drop`), own spec cycle.
  The forensics finding redirects (a): the defect site is **plan.md §4's protocol**, not
  implement §1's rendering — fixing §4 (composed verb + reason→render mapping + self-dirtying
  acknowledgment) is the change that would have prevented #358's trunk outcome.

## Tradeoffs & Alternatives

- **A — marked dissolvable edges (ticket's proposal)**: overloads `Depends on` with a second,
  mode-conditional meaning; reviewer must re-derive which edges are real; under trunk it degrades
  to today. Redundant *as mutex* once any dispatch-time disjointness gate exists — but see the
  adversarial reversal: as *ordering-only* annotation it is the right shape (mustRunAfter
  semantics), because ordering is what the gate cannot provide.
- **B — delete the edge advice; the in-flight Files-disjointness gate IS the serialization**:
  cleanest on paper (one field, one meaning; converts an honor-system rule into a dispatch-time
  gate). **Rejected by the adversarial pass on four grounds**: (1) the overnight executor consumes
  the same plans with no gate and a shared batch worktree — deleting authored edges co-schedules
  racing pairs there; (2) mutex ≠ ordering — deleting `16 ← 12` makes 16 eligible *before* 11/12
  run at all, inverting authored order, and the edge's transitive serializations (16↔11 on
  night_rig.gd, 16↔8 on main.tscn) vanish invisibly; (3) a prose-maintained in-flight set is
  conversation state — TOCTOU by construction, lost on compaction/restart, string-compared Files
  on a case-insensitive filesystem — violating the structural-separation convention for a gate
  whose deviation cost is silent interleaved writes on main; (4) it inverts the safety default:
  today a forgotten edge usually still serializes by level accident; under eligibility dispatch an
  incomplete Files list *actively* co-schedules at the earliest opportunity.
- **C — separate Conflicts/pool field**: right shape for non-file shared resources; hold in
  reserve; not needed for #358's plain file overlap.
- **D — structural dissolution**: worktree-by-default per task is a net regression for the
  motivating hub-file case (concurrent same-insertion-point appends are guaranteed merge
  conflicts; churn cost); threshold-only half (hub-file seam ≥2) is cheap and fixes #358's literal
  2-writer case — with the caveat that #358's hubs (delete/extend/re-point of night_rig.gd) are
  seam-resistant: the rule must name serialization as the honest remedy when a seam can't apply.
- **The 5%/27% decomposition**: 135→99 is a graph-shape effect (edge deletion cuts DAG depth
  11→8; unchanged level-batching already parallelizes the moved tasks); only 142→135 prices the
  pipelining mechanism. But the decomposition's magnitudes are unrecoverable — the placeholder
  durations are concentrated in exactly the moved tasks (Forensics). The duration-independent
  structural fact (depth 11→8 from one edge) is what survives.
- **Width guidance**: checklist P-item now (matches P11's shape, ships with the threshold change),
  computed lint verb as a future upgrade — mirroring how P11's own ≥3 threshold was corrected only
  after a real miss.

## Adversarial Review

Beyond the reversals folded in above, the standing findings any spec must address:

- **Stale-base window**: "`Depends on` set is `[x]`" is the wrong eligibility condition under
  pipelining — `[x]` flips at checkpoint (§2d) but merge-back is §2e, so a dependent dispatched in
  the window branches from a HEAD without its dep's commits. The condition must be "deps merged
  back," fusing §2d+§2e into a per-task step — a real protocol restructuring nobody scoped.
- **Batch-shaped downstream**: sequential checkpoint (`git log --oneline -N`, N = batch size)
  breaks under interleaved commits; `metrics.py` batch derivations corrupt under pseudo-batches;
  the `implement-batch-failure` pause must define admit-vs-freeze while pending. All three are
  ADR-0030 blockers still live.
- **Commit-disjointness ≠ Files-disjointness**: concurrent trunk builders collide on
  `.git/index.lock`, and this repo's dual-source pre-commit hook gates every commit on a clean
  mirror tree — concurrent in-flight commits fail nondeterministically.
- **P11 contradiction**: the checklist blesses the serializing-edge remedy B would delete.
- **Sibling carve-out**: plan.md:86's sibling rule covers the one configuration (co-scheduled
  siblings in ONE worktree) where neither isolation nor an in-flight gate applies — it must survive
  any rewording verbatim in effect.
- **Version/dialect skew**: plans outlive skill snapshots (#358 ran a pre-hub-rule snapshot);
  marked-edge safety is asymmetric — an executor treating a marked edge as real is merely
  conservative; an executor missing a gate that replaced deleted edges is unsafe. Annotation must
  stay within the parser-tolerated dialect (parenthetical / em-dash; single-hyphen fails R4).
- **#39886 blast radius**: pipelining widens exposure — a silently mis-parented builder writes the
  parent tree while others are in flight, voiding the gate's isolation premise.
- **Housekeeping**: kept-pause markers survive any §2 edit; `docs/internals/sdk.md:241` stale line
  corrected in the same change; #401's ≤10-net-lines precedent for implement.md is a scope signal
  against in-prose pipelining machinery.

## Open Questions

- **Does the #358 evidence justify reopening ADR-0030's pipelining deferral?** Resolved: **no**.
  The 142/135/99 simulation is majority-placeholder in exactly the tasks that matter, the run was
  54% complete, the citing artifact is off-disk, no token accounting exists, and the metric
  (wall-clock) is one no requirements text prices. Every ADR-0030 blocker (per-task event, metrics
  semantics, commit serialization, batch-failure pause, mode stability, stale-base window) is
  live. Recommendation: item 1 lands as an **ADR-0031 that re-affirms the barrier** and names the
  preconditions for any future pipelining attempt; the dispatch prose itself does not change.
  (Scope delta vs the ticket's Role — surfaced for user sign-off at spec approval.)
- **Marked edges vs edge deletion vs a separate field — which closes the authoring gap?**
  Resolved: **ordering-only annotation** on the existing `Depends on` field, in the
  parser-tolerated parenthetical dialect (e.g. `[12] (write-serialization: night_rig.gd)`),
  documented with mustRunAfter semantics: executors running per-task isolation may relax it to
  not-before; no executor may delete it; overnight treats it as a real edge (conservative-safe).
  Alternative B (delete the advice, rely on an in-flight disjointness gate) is rejected on the
  overnight-executor, mutex≠ordering, and prose-TOCTOU grounds above. A `Conflicts:`/pool field
  (C) stays in reserve for non-file resources.
- **What is item 4, really?** Resolved: not a dirty-tree suppression (none exists — the code
  fail-safes into a prompt; worktree is hidden only when its CLI is absent). The real defects:
  (1) Plan §4 instructs the raw picker verb whose output cannot populate the rendering guards §4
  claims are owned elsewhere — replace with the composed `cortex-lifecycle-branch-decision` plus a
  reason→render mapping; (2) §4 is structurally self-dirtying (plan.md uncommitted at approval) —
  the protocol must treat `dirty_tree` at §4 as expected, not as a worktree blocker; (3) no picker
  surface explains trunk's downstream cost — add the one-line tradeoff ("no isolation → same-file
  tasks serialize; plan authors must add write-serialization edges") at **both** surfaces (Plan §4
  per ADR-0012, implement §1 fallback). Wheel changes (shapes b/c) are not required and are
  deliberately avoided (closed API, 4+ pinning tests).
- **Hub-file seam threshold**: Resolved: restate as "any file two tasks would both edit" (≥2),
  keeping it authoring guidance (never a gate), with the seam-resistant caveat: when a seam can't
  apply (structural rework, deletions), the honest remedy is an annotated write-serialization
  edge, not a seam.
- **Graph-width guidance placement**: Resolved: author-facing width rule in plan.md beside
  Straggler isolation + a new checklist row (P13: flag when critical-path length is a large
  fraction of task count, e.g. depth > ~N/2, and any single-task level between multi-task levels),
  flag-only like P1/P11. A computed lint verb is a named future upgrade, not in scope.
- **The fourth counterfactual row ("level-batched, minus the edge")**: Deferred — unmeasurable;
  the pending-tail durations were never recorded and the run never finished. The
  duration-independent structural fact (depth 11→8 from one edge deletion) carries the authoring
  argument without it.
- **Pipelining preconditions** (if ever reopened): Deferred to ADR-0031's list — durable per-task
  dispatch/completion events, metrics batch-semantics decision, fused per-task checkpoint+merge
  step (closing the stale-base window), admission policy during the batch-failure pause, a
  commit-serialization story, substrate stability/pinnability, and #39886 mitigation. Not this
  ticket's scope.
- **#39886 silent isolation failure**: Deferred — pre-existing, unmitigated, tracked outside this
  ticket (multi-agent.md names it; mitigation ideas in #208's research). This ticket must simply
  not increase reliance on "isolation is in effect" — the ordering-only-annotation resolution
  satisfies that (relaxation is an optimization, never a safety dependency).
- **Stale `docs/internals/sdk.md:241`**: Resolved: correct the stale synchronous-coupling line as
  a rider in this change.
