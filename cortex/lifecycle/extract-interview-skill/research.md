# Research: Extract the interview "grilling" loop into a standalone /interview skill

Topic: extract the one-at-a-time interview loop from `requirements-gather` into a standalone, user-invocable grilling skill, that may also become the single source of truth for the loop rule currently duplicated across the requirements/spec interview surfaces. Two knobs were proposed in Clarify: (1) who decides "done" (user vs model-judges-coverage); (2) what happens to answers (context-only vs synthesized artifact).

## Web Research

**Matt Pocock grill-me** ([mattpocock/skills](https://github.com/mattpocock/skills)) is a ~5-instruction kernel: interview relentlessly until shared understanding; walk the decision tree resolving dependencies one-by-one; recommend an answer per question; ask one at a time; explore the codebase instead of asking when possible. It produces **no artifact** (primes context only) and stops on a soft "shared understanding" condition — Pocock reports 45-minute / 50-question sessions.

**Critical findings (the user asked us to judge value, not copy):**
- The transferable kernel (one-at-a-time, recommend-a-default, inspect-codebase-first, resolve-dependencies-in-order) is genuinely valuable — adopt it.
- **Field reviews favor the artifact-producing variants** (`grill-with-docs` → `CONTEXT.md`, `to-prd`), not pure priming. Pure priming evaporates on `/clear` or context compaction.
- The open-ended "relentless" stop condition is the weak spot → adopt **coverage/saturation-based stopping** (GitHub spec-kit `/speckit.clarify`: "structured, sequential, coverage-based questioning"; qualitative-research saturation: stop when new questions stop yielding new information) with a soft cap to prevent fatigue.
- **Funnel ordering** (NN/g): broad/open first, narrow/closed last.
- **Anchoring tension:** recommend-before-asking is an elicitation anti-pattern *when measuring true unbiased preference*, but defensible here because the user is a decision-maker offered a sane default to accept/reject — provided recommendations are easy to reject and **suppressed on taste/preference questions**.
- **Modern-model autonomy guidance** (GPT-5 prompting guide) pushes the *opposite* of relentless interviewing: reduce clarifying questions, proceed on reasonable assumptions. Reconcile by scoping the interview to high-ambiguity / high-rework-cost moments, not always-on.

## Requirements & Constraints

- **Solution horizon** (`project.md:21`): the durable version is warranted when "the patch applies in multiple known places you can name" — but a scoped phase of a multi-phase plan is not a stop-gap. The prior lifecycle pre-scoped the durable form and gated it on drift; that gate governs the DRY work specifically.
- **Complexity must earn its place; when in doubt simpler wins** (`project.md:19`). **Maintainability through simplicity** — "complexity managed by iteratively trimming" (`project.md:50`).
- **Shared sub-skill is in scope** — `backlog-author` is explicitly named as the precedent for a shared sub-skill consumed by multiple callers (`project.md:59`).
- **MUST-escalation policy** (`CLAUDE.md`): new authoring uses soft positive-routing phrasing; zero MUST/CRITICAL/REQUIRED tokens absent a linked evidence artifact. The prior grilling work enforced this across the edited files; Pocock's own skills use zero escalation language.
- **Prescribe What and Why, not How**; prefer structural separation over prose-only enforcement for gates.
- **Kept-pauses parity obligation:** if `specify.md` pause line-anchors move, the `skills/lifecycle/SKILL.md` inventory **and** `tests/test_lifecycle_kept_pauses_parity.py` must be updated in the same change.
- **Frontmatter contract:** required `name` + `description`; optional `when_to_use:` (concatenated to description for routing), `argument-hint:`, `inputs:`/`outputs:`. Skills ship via plugins, not the wheel (ADR-0002). Commit via `/cortex-core:commit`.
- **No `cortex/requirements/glossary.md` exists on disk** despite being listed in `project.md` Global Context — the loader treats it as "skipped: file absent." requirements-gather's glossary apparatus has therefore never run against a real file (informational; not blocking).

## Tradeoffs & Alternatives

Four "sharing" shapes were evaluated (this repo's skills are prose, so "share" is ambiguous):
- **A. Shared reference file** — `/interview` (or `interview-cadence.md`) owns the loop; callers replace duplicated prose with a one-line pointer (the `load-requirements.md` idiom). Lowest-disruption, matches the repo's one established prose-sharing convention, preserves specify.md's parity gates. **Recommended for the DRY part — if the DRY part is done at all.**
- **B. Skill-tool delegation** — callers invoke `/interview` at runtime, passing knobs as args. Highest complexity; a context round-trip; for specify.md it would relocate the parity-anchored pause at line 36. Over-engineering for thin shared content.
- **C. Documented-pattern-only** — accept the duplication; cite one canonical pattern. Wins **if** the shared surface really is just the one cadence sentence (which Agent 1 confirms it largely is).
- **D. Hybrid** (gather delegates at runtime; specify reads a reference) — worst long-term: two consumption contracts for one primitive; asymmetric treatment of near-identical consumers.

**Knob expression:** keep **answer-disposition out of the primitive** (it is the caller's job — mirrors the requirements-gather/requirements-write split); express **termination** as a declared mode the caller selects (`done-by: user | coverage`), not branching prose. Argument-hint params beat distinct modes because the loop body is identical — only the exit condition differs.

## Adversarial Review

- **The DRY predicate has not fired.** The prior lifecycle gated reference-file extraction on *observed drift*; review found the copies still equivalent. Building shared infrastructure now to DRY a single sentence runs against "complexity must earn its place." Agent 4's own disqualifier ("C wins if the shared surface is just the cadence sentence") applies, because it is.
- **Unbundle Want A from Want B.** *Want A* = "let me invoke the grilling standalone" (a thin self-contained skill, touches nothing load-bearing). *Want B* = "single source of truth for the loop rule" (the deferred, drift-gated extraction). Fusing them manufactures the cost; they should be decided separately.
- **The dual-mode trap is real.** A standalone mode (user-stops, no artifact, raw context) plus two callers that consume the loop incompatibly (model-stops, structured Q&A block / spec.md, downstream synthesis) is a primitive consumed three different ways — the semantic coupling is the leak, regardless of static-file vs runtime delivery.
- **`specify.md` convergence is net-negative.** Reward = delete two pointer notes; risk = churn on a parity-gated (lines 36/67/155, ±35 tol), byte-mirrored, load-bearing file, plus mid-interview read-indirection for a one-sentence rule. Leaving specify.md alone is strictly safer for equal value.
- **Chosen defaults are field-disfavored.** Context-only + user-decides-done inherits grill-me's over-interrogation (no real stop condition) and context-evaporation (one `/clear` and the priming is gone, with no artifact to recover it).
- **Anchoring is *worse* without an artifact** — in requirements-gather a wrong recommendation is corrected into a structured block; in raw context the model's suggestion and a terse "sure" sit as equally-weighted tokens with no synthesis firewall. The glossary skill's "a recommendation alone is not consent to persist" guard has no standalone-mode equivalent.
- **Routing collision (missed by agents 1-4):** `backlog-author`'s description advertises `"interview"` as a trigger word and it has an `interview` subcommand ("guided human-facing authoring session"). A `/interview` skill competes for the exact dispatch token — a testable routing-accuracy regression.
- **`clarify.md:57` intentionally batches ≤5 questions** (an inventoried kept-pause) — a deliberate, tested exception to one-at-a-time. A "single source of truth" the most-used clarify gate ignores is a third dialect, undercutting the single-canonical-loop premise.
