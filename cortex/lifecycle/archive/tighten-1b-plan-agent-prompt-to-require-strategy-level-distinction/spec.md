# Specification: Tighten §1b plan-agent prompt to require strategy-level distinction

## Problem Statement

The §1b plan-agent prompt template at `skills/lifecycle/references/plan.md` (the line containing *"explore a different architectural strategy, decomposition, or ordering than the obvious default"*) currently admits ordering-only differentiation between plan variants. The historical corpus of four `plan_comparison` events shows variants always differ on task ordering and granularity (16/14/13 tasks; bottom-up vs registry-first vs runner-first), not on architectural strategy. **Importantly, this is also true for the typical cortex feature**: the n=4 corpus is fix/refactor work where only one architectural pattern fits, and that pattern is fixed by the existing system. The realistic value of forcing each variant to commit to a named architectural-pattern category is therefore not "produce strategy-distinct variants" (which the corpus shows is rarely possible) but **provide the downstream synthesizer (#160) with a structured category-collision signal** — when all variants name the same category, the synthesizer applies position-bias mitigations (DR-4: strip labels, randomize order, swap-and-require-agreement); when variants name different categories (the rarer architectural-decision-rich case), position-bias risk is lower and the synthesizer can rank with more confidence. Tightening this prompt is the documented primary fix per `research/competing-plan-synthesis/research.md` DR-3 Option 4. The empirical literature on multi-agent diversity (Spark Effect arXiv:2510.15568; Dipper arXiv:2412.15238) supports closed enumerated vocabularies as the prompt-level intervention with the strongest measured leverage; in cortex's case, that leverage is principally signal for the synthesizer rather than agent-side diversity.

## Edge Cases

- **Modal cortex feature fits exactly one category**: For sub-1000-line edits where existing system architecture forces all variants into the same category (n=4 historical corpus), all 2-3 plan agents will name the same category. R2's self-check (per the corrected wording) asks how this approach differs from OTHER variants in this plan_comparison — so plan agents legitimately express decomposition/ordering distinctions there, with no contradiction with R1. The synthesizer (#160) sees the same-category signal and applies DR-4 mitigations. **This is the primary value path of the intervention, not an edge case.**
- **Variants land in different categories (rare)**: For architectural-decision-rich features, variants may name different categories. The synthesizer sees stronger distinguishability and can rank with more confidence (lower position-bias risk per Shi et al. 2024).
- **Plan agent ignores the constraint** (writes a plan without the field, or names a category outside the closed list): Detection is via P8 in orchestrator-review (structural check); remediation is via the existing fix-dispatch path (orchestrator-review Cycle 1 → fix subagent → Cycle 2). This is the same enforcement model used for other Plan Format fields (P1-P7).
- **#160 rewrites §1b before #159 ships**: The R8 resumption protocol mandates re-validation of the spec's anchors before plan-phase entry. If `"or ordering than the obvious default"` is no longer present in the §1b region (because #160's rewrite eliminated it), the spec is **re-specced**, not blindly applied.
- **Pytest false-positive on intentional plugin-tree edits**: `plugins/cortex-interactive/skills/...` is regenerated by `just build-plugin`. If an implementer edits the plugin tree directly, the pytest in R7 fails. This is correct behavior — the canonical is `skills/...`. Recovery: copy the intended changes back to `skills/...`, re-run `just build-plugin`, and commit the regenerated tree.

## Technical Constraints

- **Dual-source enforcement** (`justfile:499`, `.githooks/pre-commit` Phase 4): `skills/<name>/...` is canonical; `plugins/cortex-interactive/skills/<name>/...` is regenerated build output produced by `just build-plugin`. Pre-commit Phase 4 runs `just build-plugin` then `git diff --quiet -- plugins/$p/` to detect drift. Phase 4 is bypassable via `--no-verify` — Requirement 7's pytest is the defense-in-depth check. (Note: `bin/cortex-check-parity` is a SKILL.md-to-bin token-graph linter, not a file-content equality enforcer; do not name it as the bypassable enforcement layer.)
- **OQ3 MUST-escalation policy (CLAUDE.md)**: New prompt language uses lowercase "must" only — no uppercase MUST/CRITICAL/REQUIRED. R1's acceptance criterion includes a baseline-difference check on uppercase markers to enforce this.
- **Verbatim copy semantics (`skills/lifecycle/references/plan.md` §1b dispatch instruction)**: The §1b plan-agent prompt template is copied verbatim per dispatch — agents are explicitly told not to omit, reorder, or paraphrase. New prompt language must be inside the §1b verbatim template block to actually reach the plan agents.
- **Soft-positive-routing default (Opus 4.7)**: Per the post-4.7 harness adaptation in CLAUDE.md, prompt language uses positive routing ("name your category from this list", "state how it differs from the other variants") rather than negative MUST-not framings.
- **Plan agent model is sonnet**: New prompt language must work on sonnet (the dispatched plan-agent model). The closed taxonomy uses LLM-friendly category names heavily represented in sonnet's training data.
- **Position-bias literature (Shi et al. 2024, arXiv:2406.07791)**: The downstream synthesizer's ranking accuracy degrades parabolically as candidates approach quality parity. The closed taxonomy primarily provides the synthesizer with a category-collision signal so it knows when to apply DR-4 mitigations — not so it eliminates similarity (which the corpus shows is rarely possible for cortex's typical work).
- **Resumption coupling with #160**: #160 may rewrite the §1b region (`plan.md:21-119` per its scope). R8's re-validation protocol is mandatory before plan-phase entry. Spec anchors are quoted text strings, not line numbers, to survive #160's edits — but the structural shape of §1b may also change, in which case the spec is re-specced rather than mechanically applied.
