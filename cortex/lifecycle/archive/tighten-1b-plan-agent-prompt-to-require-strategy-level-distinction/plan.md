### Task 4: Add dual-source parity pytest with sentinel-mutation gate (R7)
- **Files**: `tests/test_dual_source_reference_parity.py` (new)
- **What**: Write a parameterized pytest that discovers dual-source skill files under `skills/*/SKILL.md`, `skills/*/references/*.md`, and `skills/*/assets/*.md` and parameterizes over **both BUILD_OUTPUT_PLUGINS** with their respective SKILLS arrays — `cortex-interactive` (commit, pr, lifecycle, backlog, requirements, research, discovery, refine, retro, dev, fresh, diagnose, evolve, critical-review — 14 skills, 36 files) and `cortex-overnight-integration` (overnight, morning-review — 2 skills, 3 files). For each (plugin, canonical_path) pair, compute the mirror path as `plugins/{plugin}/{canonical_path}` and assert byte-identical via a pure helper `assert_byte_parity(canonical_bytes: bytes, mirror_bytes: bytes) -> None` that raises `AssertionError` on mismatch. The file-pair test reads both files via `pathlib.Path.read_bytes()` and calls `assert_byte_parity`. Add a separately-collected sentinel test `test_assert_pytest_fails_on_mutation` that picks the FIRST canonical/mirror pair from the sorted parameterized set, reads both into memory, mutates the mirror bytes by appending one byte IN MEMORY (no filesystem write), calls `assert_byte_parity` against the mutated bytes, and asserts an `AssertionError` is raised. Both BUILD_OUTPUT_PLUGINS are covered because spec R7's stated purpose (M5 defense-in-depth against `--no-verify` drift bypassing pre-commit Phase 4) requires symmetry with Phase 4's own coverage, which iterates both plugins. (Spec R7's literal mirror-path text mentions only `plugins/cortex-interactive/...` — this plan extends to both BUILD_OUTPUT_PLUGINS as a strengthening of M5's stated intent; the deviation is documented in Veto Surface.) Parameterized set must produce ≥20 test IDs at collection time (currently 36 + 3 = 39 file-pair tests + 1 sentinel = 40 in the current tree).
- **Depends on**: [1, 2, 3]
- **Complexity**: simple
- **Context**:
  - File location: `tests/test_dual_source_reference_parity.py` alongside existing tests (`tests/test_check_parity.py` is a sibling for naming reference but tests a different concern — token-graph linting, not byte parity). There is no existing canonical→mirror byte-equality pytest convention in cortex; this test introduces the pattern.
  - Discovery + per-plugin filter:
    1. Define a top-of-file `PLUGINS` constant — a `dict[str, tuple[str, ...]]` mapping plugin name to its SKILLS tuple. Hard-code as: `PLUGINS = {"cortex-interactive": ("commit", "pr", "lifecycle", "backlog", "requirements", "research", "discovery", "refine", "retro", "dev", "fresh", "diagnose", "evolve", "critical-review"), "cortex-overnight-integration": ("overnight", "morning-review")}`. Top-of-file comment must instruct the next maintainer to keep these aligned with `justfile` `build-plugin` recipe (resolve via `grep -n "SKILLS=(" justfile`); the test fails on count mismatch if they drift apart.
    2. Glob the union of `skills/*/SKILL.md`, `skills/*/references/*.md`, `skills/*/assets/*.md` from repo root (current tree: 39 files).
    3. For each globbed canonical path, identify which plugin it maps to by checking which plugin's SKILLS tuple contains the second path component (skill name). Skills not in any plugin's SKILLS produce no test (defensive — currently every skill in the glob output IS in one of the two SKILLS arrays, so no orphans exist).
    4. Per-plugin counts: `cortex-interactive` = 36 files, `cortex-overnight-integration` = 3 files (overnight/SKILL.md, morning-review/SKILL.md, morning-review/references/walkthrough.md). Total = 39 file-pair tests + 1 sentinel = 40 collected items.
  - Mirror-path computation: `Path(f'plugins/{plugin_name}') / canonical_path` — i.e., for canonical `skills/commit/SKILL.md` (cortex-interactive), mirror is `plugins/cortex-interactive/skills/commit/SKILL.md`; for canonical `skills/overnight/SKILL.md` (cortex-overnight-integration), mirror is `plugins/cortex-overnight-integration/skills/overnight/SKILL.md`.
  - Pure parity helper: `assert_byte_parity(canonical_bytes: bytes, mirror_bytes: bytes) -> None` is a module-level function with no I/O. The file-pair test reads files via `Path.read_bytes()` and calls the helper; the sentinel test mutates bytes in memory and calls the same helper. This factoring is what makes the sentinel crash-safe — no filesystem mutation, no `try/finally` needed, no SIGKILL/segfault concern.
  - Test parameterization: use `pytest.mark.parametrize` over the **sorted** (canonical, mirror) tuple list. Sorting makes the sentinel target deterministic — picking `parameterized_set[0]` always selects the lexicographically first pair regardless of glob order. (This avoids hardcoding `commit/SKILL.md` as the sentinel target.)
  - Sentinel-mutation gate (implements spec R7c): `test_assert_pytest_fails_on_mutation` reads canonical+mirror bytes from the first sorted pair, appends `b'\x00'` to the mirror_bytes copy in memory, asserts that calling `assert_byte_parity(canonical_bytes, mutated_mirror_bytes)` raises `AssertionError`. No filesystem state changes; no `try/finally` needed. The test self-verifies on each run.
  - conftest pattern: see `tests/conftest.py` for `--run-slow` marker — not needed for this test but illustrates the conftest registration pattern.
  - Pytest collection: `pytest --collect-only tests/test_dual_source_reference_parity.py` reports 40 collected items in the current tree (39 parameterized + 1 sentinel). The double-floor verification (≥20 AND ≥39) catches both the spec R7b minimum and a SKILLS-array-extension drift where new mirrored files are not picked up by the test.
- **Verification**: Run `bash -c 'set -e; pytest tests/test_dual_source_reference_parity.py -v && COUNT="$(pytest --collect-only -q tests/test_dual_source_reference_parity.py 2>/dev/null | grep -cE "::test_")" && [ "$COUNT" -ge 20 ] && [ "$COUNT" -ge 39 ]'` — pass if exit 0 (pytest passes AND collected items ≥ 20 AND ≥ 39).
