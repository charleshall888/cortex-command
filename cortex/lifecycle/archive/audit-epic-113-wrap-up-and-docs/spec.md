# Specification: Audit epic 113 wrap-up and post-113 distribution-doc cleanup

> **Epic Reference**: This audit follows the close of epic 113 (CLI + plugin distribution rebuild). Closure of 113's 11 children verified as a precondition during research; see `research.md` Codebase Analysis. The audit is a fresh, self-contained docs-cleanup task — not an additional 113 child.

## Problem Statement

Epic 113 reshaped cortex-command's distribution layer (CLI installed via `uv tool install -e .`, plugins via `/plugin install`, four plugins in `.claude-plugin/marketplace.json`, dual-source enforcement via `.githooks/pre-commit`). Several user-facing docs, manifest descriptions, and review-criteria configs still describe the pre-113 state — symlink-based deployment, ticket-N future-tense scaffolding, transitional product descriptions. New readers (people cloning or forking the repo) currently encounter a mix of accurate steady-state docs and stale 113-transition language. The maintainer pays continuous archaeology cost reconciling the two. Fixing this now also blocks future stales from accreting on top of current ones.

## Non-Requirements

- **No CI grep linter.** Verification is a one-time end-of-cleanup grep (R10), not an enduring CI gate. Forbidden-phrase lists rot fast at this scale; revisit only if the same phrases regress within 90 days.
- **No edits to `retros/`.** Project convention: retros are dated problem-only logs, append-only.
- **No pre-113 migration aside.** Ticket 124 (migration guide) was deliberately `wontfix`; the audit does not re-open that decision.
- **No restructuring of existing docs.** Surgical edits for everything except `docs/plugin-development.md`. No section renames, no doc-ownership transfers.
- **No code, hook, script, or behavior changes.** Docs and config-text only. The `.githooks/pre-commit` dual-source check is preserved untouched.
- **No fix to `docs/setup.md` "broken `remote/SETUP.md` link".** During pre-write verification this link was confirmed already absent — fixed in prior lifecycle `lifecycle/remove-dangling-remote-setup-references/`. The user's "Drop the broken link" answer applies to a state that no longer exists. The companion stale entry in `requirements/remote-access.md:60` IS in scope (R10) since it is itself a piece of stale documentation referencing the same already-resolved issue.
- **No creation of `remote/SETUP.md` in this repo.** Per `requirements/project.md`, machine-setup automation is owned by the user's machine-config repo; cortex-command should not host that doc.
- **No tightening of `CLAUDE.md:48` setup-githooks framing.** Deferred per Open Q&A: the line is correct for contributors (including the maintainer); rewording is low-value. (Note: `CLAUDE.md:18` and `:47` are in scope per R11 — different lines, different concerns.)
- **No rewrite of `docs/agentic-layer.md:311`.** The "symlink" reference is the legitimate `lifecycle/sessions/latest-overnight` runner-state symlink, unrelated to install distribution.
- **No global rename of "symlink" across the repo.** The word remains valid in non-install contexts (runner state, Python `__file__` resolution). Forbidden-phrase grep targets specific install-flow phrasings, not the bare word.
- **Scope rule for "adjacent stale" entries.** R10 cleans up `requirements/remote-access.md:60` because that entry has the same root cause as content R1–R9 fix (a pre-113-era reference to an already-resolved condition). Stale residues from *unrelated* origins — e.g., other lifecycle "open issues" sections in `requirements/`, retros, or backlog items not connected to 113 — are out of scope. The principle is "same-cause adjacent residue is in scope; unrelated stale residue is not."

## Edge Cases

- **`$schema` URL availability.** `https://json.schemastore.org/claude-code-marketplace.json` is the URL Anthropic uses in their own `.claude-plugin/marketplace.json`. If the URL is temporarily unavailable to a reader's editor, the field is still valid JSON and harmless — schemastore is the canonical pattern. No fallback needed.
- **`category` vocabulary fit.** All four plugins are framed as `development` per Anthropic's vocabulary. If a plugin clearly belongs to another category in the future (e.g., `cortex-ui-extras` could plausibly be `design`), revisit then. For the audit, picking `development` uniformly is defensible and avoids debate during review.
- **`docs/plugin-development.md` rewrite scope drift.** The user's directive is "rewrite as post-122 dogfood guide" — bounded to maintainer use. If the rewrite drifts into "how to author a plugin from scratch," cut that content; it is a separate doc that may or may not be needed in the future.
- **Pre-commit hook behavior on docs-only commits.** Verified: `.githooks/pre-commit` only triggers `just build-plugin` when staged paths include `skills/`, `bin/cortex-*`, or `hooks/cortex-validate-commit.sh`. Docs-only commits do not trigger plugin rebuild. No special handling needed.
