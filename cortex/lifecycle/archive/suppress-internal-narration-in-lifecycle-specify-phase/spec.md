# Specification: Suppress internal narration in lifecycle specify phase

> **Epic reference**: decomposed from [research/audit-interactive-phase-output-for-decision-signal/research.md](../../research/audit-interactive-phase-output-for-decision-signal/research.md). This ticket owns the specify-phase narration surfaces (§2a, §2b, §3a) and the paired Fix Agent Prompt Template edit in `orchestrator-review.md`. `orchestrator-review.md §4`'s one-line on-pass assessment, `clarify.md`, `plan.md`, and `/critical-review` remain out of scope (owned by sibling tickets).

## Problem Statement

The specify phase emits internal orchestrator narration in four locations that add noise without adding decision signal, pushing user-facing output further from the user's actual decision points (question answers, spec approval). This violates the project's "signal over noise" posture and makes long specify runs harder to follow. Removing or restructuring these surfaces tightens the interactive experience and aligns the phase with DR-4's mechanism principle (remove the output *requirement* rather than instruct "be compact").

## Requirements

**MoSCoW classification**: All seven requirements below are **Must-have**. Each one is either a direct edit to one of the four narration surfaces named in the backlog body (R1–R4 cover §2a clean-pass, §2a failure-path cycle 1 and cycle≥2, and §2b; R5 covers the §3a Fix Agent Prompt Template; R6 covers the §3a orchestrator-side disposition of the fix-agent return) or a technical-guard requirement explicitly chosen during the spec interview (R7 preserves the per-cycle `orchestrator_review` event across silent re-runs — flagged by adversarial review F6 and selected by the user). Should-have and won't-do scope appears in **Non-Requirements** below (e.g., the re-read + validity guard before silent re-run, the `pre_write_check` event, the `confidence_check` "pass" event), not as deferred or optional requirements here. No item in this list is a "nice-to-have" — dropping any one of R1–R7 would leave a named narration surface unsuppressed or a flagged adversarial-review failure mode unmitigated.

## Non-Requirements

- **Re-read + validity guard before silent re-run is NOT in scope.** The adversarial review raised that a fix-agent could return `verdict: revised` against a malformed or empty artifact, and the orchestrator's silent re-run would pass it. The user chose to accept this risk for this ticket rather than add a re-read-and-validate guard. **Note on accepted failure mode**: contrary to the original draft of this bullet, §4 (User Approval in `specify.md` lines 154–162) does NOT re-read the spec from disk — it presents a Produced/Value/Trade-offs paraphrase via `AskUserQuestion` generated from the orchestrator's in-context understanding. Therefore a malformed rewrite (empty file, truncated markdown, wrong-path write) will NOT automatically surface at §4. The user sees a normal approval prompt generated from what the orchestrator *believes* it wrote. This is an accepted hidden failure surface for this ticket; a future ticket may add the re-read guard or a §4 disk-read requirement.
- **`orchestrator-review.md §4` on-pass one-line assessment is NOT modified.** One line is an acceptable user-visible surface; out of scope per the backlog body.
- **`clarify.md`, `plan.md`, `/critical-review` narration surfaces are NOT modified.** Owned by sibling tickets in epic 66. See Technical Constraints on scope of mechanism choices.
- **`pre_write_check` event is NOT added.** The user chose pure silence for §2b; no new event schema.
- **`confidence_check` event with `action: pass` is NOT added.** Preserves the current invariant; cycle counting (specify.md line 57) does not change. Noted qualifier from interview: the user may revisit this later if audit visibility becomes valuable.
- **Events-log asymmetry across specify-phase gates is accepted.** After this ticket lands, a clean specify run logs nothing on §2a pass, nothing on §2b pass, and `orchestrator_review` on every §3a cycle. `/lifecycle resume` consumers (and any other events-log reader) cannot distinguish "§2a ran and passed" from "§2a was never reached (e.g., crash, skip)." The user accepted this asymmetry in the Q1/Q2 interview responses. A future ticket may add pass events if this cost outweighs the current simplicity.
- **No tests are added.** No existing tests assert the current narration behavior; behavioral tests of skill-reference-file instructions are not a current pattern in this codebase. Verification of the edits is via `grep` acceptance criteria against the static reference files — runtime behavior (whether fix-agents actually emit parseable envelopes, whether the orchestrator actually suppresses them, whether malformed envelopes are detected) is NOT verified by this ticket.
- **The user-facing announcement language for §2a failure-path is NOT a templated markdown block inside the skill file.** The instruction prescribes shape (bulleted list, ≤15-word bullets, bind-by-reference) with one inline example, but the live announcement is produced by the orchestrator at runtime, not copied verbatim from the reference file.

## Edge Cases

- **A fix-agent returns `verdict: failed`**: Orchestrator treats this as the flagged issues remaining and proceeds to the cycle-cap check (Step 4 "On flag") per the existing Step 4 semantics. The `orchestrator_review` event for the re-run cycle is still written; its `verdict` field reflects the re-review's verdict on the artifact (not the fix-agent's envelope verdict).
- **A fix-agent returns a malformed envelope** (missing field, wrong type, extra prose around the schema): Detection is LLM-mediated — the orchestrator reads the return text and classifies it. The instruction treats a malformed envelope as a fix failure for the cycle (equivalent to `verdict: failed`). The event log records the re-review `orchestrator_review` event. No user-visible relay of the malformed envelope. (Accepted failure mode; reliability here rests on the same instruction-following surface DR-4 calls out as imperfect — see Technical Constraints for the honest framing.)
- **A fix-agent writes a revised artifact that is malformed on disk (empty file, corrupted markdown) — accepted hidden failure mode**: No explicit guard is added (see Non-Requirements). The re-review step (`orchestrator-review.md` line 35: "the artifact is already in context") evaluates from the orchestrator's in-context copy of the revised spec, not from a fresh disk read. The Post-Specify Checklist S1–S6 (`orchestrator-review.md` lines 138–145) measures content coverage and criteria form — none of S1–S6 checks "file exists," "file non-empty," "YAML frontmatter parses," or "markdown well-formed." Consequently a truncated or wrong-path rewrite may pass cycle 2 silently and reach §4 approval with the artifact in a broken state on disk while the in-context state is coherent. This risk is knowingly accepted in exchange for not adding the re-read guard this ticket; Non-Requirements documents the same accepted failure surface.
- **A fix-agent returns `files_changed` naming a path that does not match the artifact on disk**: Not cross-validated. Part of the same accepted hidden failure mode.
- **Signal list at specify.md lines 47–51 is extended by appending new signals (e.g., C4)**: R2/R3's structural-anchor phrasing ("signals flagged in §2a's Research Confidence Check") continues to work without further changes because it binds to the section heading, not to the signal identifier list. It does NOT automatically survive renaming the §2a heading, splitting §2a into sub-sections, or moving the signal list into a different phase — those edits would require R2/R3 to be updated in a follow-up ticket.
- **§2a signals are evaluated in cycle ≥ 2 and the user declines loop-back**: Existing `confidence_check` event with `"action": "declined"` is written per lines 72–75. The spec-body advisory callout at lines 102–109 still prepends to the spec. The R3 bulleted-list directive applies to how the cycle≥2 signals are presented before the user's choice.
- **The orchestrator is resumed after a crash mid-fix-dispatch**: No change from current behavior. The `orchestrator_dispatch_fix` event at lines 69–72 already exists; the envelope change does not alter resume semantics.

## Technical Constraints

- **DR-4 mechanism principle**: For in-context narration (clean-pass path, pre-write checks), remove the output *requirement* rather than adding "be compact" instructions. Structural removal > suppression directive.
- **R5 envelope is an instruction-level convention, not a runtime-enforced contract.** The fix-agent is dispatched via the Task tool, which returns free-form text. No JSON schema is validated by the runtime, no parser or hook enforces the envelope shape, and the orchestrator that "parses" the envelope per R6 is the same LLM that produced it. The reliability gain over B1 (plain behavioral instruction) is real but bounded: the envelope gives the fix-agent a concrete output shape to target (shorter rationales, fewer escape-hatches into prose), and gives the orchestrator a concrete shape to consume (easier to mechanically recognize "malformed" and fall back to `verdict: failed`). It does NOT give deterministic enforcement. Cited precedents like Google ADK `disallow_transfer_to_parent=True` and OpenAI Agents SDK JSON-schema envelopes enforce at runtime; this design does not. The reliability claim is accordingly scoped — better than B1 for this surface, not equivalent to a real contract. Non-Requirements line on "no tests" confirms runtime behavior is unverified by this ticket.
- **B2 envelope, ≤15-word cap, and "do not relay" are scoped to the §3a fix-agent re-run surface — NOT epic-level conventions for sibling tickets.** Epic 66 contains sibling tickets for `clarify.md` / `clarify-critic.md` (#068), `/critical-review` (#067), and `plan.md`. The structural-envelope choice here is justified by the §3a boundary's specific shape: the fix-agent returns a single terse disposition against a single artifact, and the re-run loop requires the orchestrator to classify pass/fail without surfacing prose. Sibling surfaces have different shapes: clarify-critic returns a `findings` *array* of prose objections that `clarify-critic.md` line 141 explicitly requires the orchestrator classify downstream via the Apply/Dismiss/Ask framework; `/critical-review` Step 4 must surface an Applied-changes bullet list by design (per `backlog/067-...md`); plan.md has not yet been spec'd. The epic research (`research/audit-interactive-phase-output-for-decision-signal/research.md` line 62) explicitly defers structured-output approaches for clarify-critic in favor of trying the behavioral fix first. **This spec does not set that precedent.** Sibling tickets must pick their own mechanism and output-cap based on their boundary characteristics; the ≤15-word cap in R2/R3/R5 is calibrated to the specific output shapes of this ticket (terse gate-signal bullets and one-scalar rationale) and is NOT a uniform epic-wide rule.
- **Preservation anchors** (do NOT modify, and do NOT extend the R6 "do not relay" pattern to these surfaces): `orchestrator-review.md §4` one-line assessment; `AskUserQuestion` channel directive in clarify §4; loop-back control-flow gate in specify §2a; anti-warmth instructions in `/critical-review` reviewer prompt; `/critical-review` Step 4 Applied-changes bullet list (decision-relevant user output by design); `clarify-critic.md` `findings`-array prose contract.
- **Event-schema stability**: The `orchestrator_review` and `orchestrator_dispatch_fix` events already exist (`orchestrator-review.md` lines 39–42, 69–72). No event schemas are added. Cycle-cap logic (lines 161–168) and cycle counting at `specify.md` line 57 are preserved unchanged.
- **No tests currently assert narration behavior**; verification relies on grep-based acceptance criteria against the edited reference files. Runtime behavior (fix-agent envelope emission, orchestrator suppression, malformed-envelope detection, on-disk artifact integrity post-fix) is NOT verified by this ticket — this is a knowingly accepted verification gap, documented in Non-Requirements.
