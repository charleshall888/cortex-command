# Specification: Add uncommitted-changes guard to lifecycle implement-phase pre-flight

## Problem Statement

The lifecycle implement-phase pre-flight (`skills/lifecycle/references/implement.md §1`) presents the user with four branch-selection options when running `/lifecycle implement` on `main`/`master`. Today none of these options are guarded against a dirty working tree. Epic #093 will flip the recommended default to "Implement on current branch" (#097) — once that flip lands, trunk-based implementation becomes the ergonomic default, and a dirty working tree silently mixes unrelated work into the commits an implement run lands on `main`. This ticket adds a minimal, non-blocking guard that demotes the current-branch option when the working tree is dirty, so the default-flip in #097 does not newly expose the trunk-safety surface without a signal. The beneficiary is the developer running `/lifecycle implement` on `main` with in-progress local work who would otherwise be pushed by #097's default toward sweeping that work into a main-branch commit.

## Requirements

All 9 requirements below are must-have. No should-haves apply to this feature. Won't-dos are captured in the Non-Requirements section.

1. **Porcelain detection runs before the pre-flight AskUserQuestion.** Immediately before the `AskUserQuestion` call in `skills/lifecycle/references/implement.md §1`, run `git status --porcelain` (no path filter, no additional flags). Acceptance: `grep -c 'git status --porcelain' skills/lifecycle/references/implement.md` ≥ 1 (verifies the command is present); Interactive/session-dependent for ordering: confirming the `git status --porcelain` call appears before the `AskUserQuestion` call in §1 requires reading the §1 prose; grep cannot assert relative line-ordering of two distinct phrases without a positional extractor.

2. **Non-empty porcelain output demotes the "Implement on current branch" option.** When `git status --porcelain` returns non-empty output, the option whose label is `**Implement on main**` (today's name; renamed to `**Implement on current branch**` by #097) is demoted: (a) its description has a one-line warning prefix prepended, and (b) its `(recommended)` suffix is stripped from the label if present. The option stays selectable and stays at its existing position. Acceptance: `grep -c 'If non-empty' skills/lifecycle/references/implement.md` ≥ 1 AND `grep -c 'prefix' skills/lifecycle/references/implement.md` ≥ 1 AND `grep -c '(recommended)' skills/lifecycle/references/implement.md` ≥ 1 — these three substrings together witness that the conditional trigger, the prefix-prepend step, and the suffix-strip step are all documented in §1.

3. **Warning text is fixed.** When the guard fires, the prefix prepended to the option's description is exactly: **`Warning: uncommitted changes in working tree — this will mix them into the commit on main.`** — no bikeshed variants. Acceptance: `grep -c 'Warning: uncommitted changes in working tree' skills/lifecycle/references/implement.md` = 1.

4. **Autonomous worktree option gets a dirt-behavior caveat.** The description of the "Implement in autonomous worktree" option (option 2) gains a one-clause prose note: **`uncommitted changes remain on main and do not travel to the worktree`**. This note is static — present always, regardless of the porcelain result — because option 2's dirt-stranding behavior is a permanent property, not a conditional. Acceptance: `grep -c 'uncommitted changes remain on main' skills/lifecycle/references/implement.md` = 1.

5. **Forward-compatible phrasing.** The guard prose refers to the demoted option abstractly (e.g., "the option that keeps the user on the current branch" or by its canonical label) such that #097's renaming of "Implement on main" → "Implement on current branch" and the flip of `(recommended)` from option 1 (today's worktree) to this option does not require re-editing the guard text. Acceptance: Interactive/session-dependent — reviewer reads §1 prose and confirms the guard does not bake in language that #097's rename would invalidate. Rationale: a command-level assertion cannot test for absence of fragile future-coupled wording.

6. **Selection remains unblocked.** The guard does not remove the option from the `AskUserQuestion` list or add a gating pre-question. Users who select the demoted option proceed to §2 Task Dispatch as today. Acceptance: Interactive/session-dependent: confirming the absence of any early-exit, abort, or pre-question flow around the dirty-tree condition requires structural reading of §1; grep cannot efficiently prove absence of a control pattern.

7. **Porcelain-failure fallback.** If `git status --porcelain` exits non-zero (e.g., `.git` missing because the user ran the skill from outside a repo, corrupt index, bisect/rebase state), the guard does not fire — neither demotion nor warning prefix are applied — and a single-line diagnostic (`uncommitted-changes guard skipped: git status failed`) is surfaced alongside the prompt. The pre-flight continues normally. Acceptance: `grep -c 'uncommitted-changes guard skipped: git status failed' skills/lifecycle/references/implement.md` = 1 AND `grep -c 'fallback' skills/lifecycle/references/implement.md` ≥ 1 — these two substrings together witness that both the diagnostic text and the fallback continuation prose are documented in §1.

8. **Option routing is not disturbed.** The demotion modifies only the description and the label's `(recommended)` suffix — not the canonical label substring used for routing (`"Implement on main"` today, `"Implement on current branch"` after #097). Acceptance: `grep -cE '"Implement on main"|"Implement on current branch"' skills/lifecycle/references/implement.md` ≥ 1 (verifies routing labels are present in §1); Interactive/session-dependent for routing-block integrity: confirming the routing match strings at lines 20–24 are unchanged and that the demotion prose touches only description and suffix requires reading the §1 routing block; grep cannot prove lines are unmodified relative to a baseline.

9. **Guard is documented alongside the existing Worktree-agent context guard.** The new guard's prose sits in §1 adjacent to the existing `Worktree-agent context guard` paragraph (currently at `implement.md:18`) and follows the same structural pattern (bold label, single paragraph, check-and-note-alongside-prompt). Acceptance: `grep -c 'Uncommitted-changes guard' skills/lifecycle/references/implement.md` = 1 (verifies the guard's heading label is present); Interactive/session-dependent for structural adjacency: confirming the new guard paragraph appears immediately adjacent to the Worktree-agent context guard within §1 requires reading the §1 layout; grep cannot assert proximity of two distinct paragraphs without a positional extractor.

## Non-Requirements

- The guard does not block selection or force the user to commit first. Users who pick the demoted option knowingly proceed as today.
- The guard does not re-check porcelain state after the AskUserQuestion resolves. The check is prompt-time, not dispatch-time. Race conditions between the check and selection are accepted as out-of-scope.
- The guard does not auto-commit, auto-stash, or otherwise modify the working tree. No code is executed against the user's tree beyond a read-only `git status --porcelain`.
- The guard does not fire on option 2 (autonomous worktree) or option 4 (create feature branch). Only option 3 is demoted. Option 2 gets a static caveat per Requirement 4; option 4 is unchanged.
- The guard does not attempt to prevent "option 2 silently strands uncommitted work on main" or "option 4 carries dirt across via `git checkout -b`." These are known gaps in the other options' dirt-handling behavior; they are left for future tickets and are explicitly out of scope here.
- The guard does not filter by uncommitted-state category. Any non-empty `git status --porcelain` output triggers demotion — untracked, unstaged, staged, unmerged all count uniformly. Stashes and ignored files are excluded by porcelain's defaults; no additional flags are passed.
- The guard does not path-filter `git status --porcelain`. It runs against the whole tree. Lifecycle-artifact dirtiness (e.g., an uncommitted `lifecycle/{feature}/plan.md` from the previous phase) counts toward demotion. Rationale: phase transitions should commit their artifacts; if they haven't, a warning before the artifact is swept into a main-branch commit is the intended behavior.
- The guard does not operate on non-interactive paths. Inside `Agent(isolation: "worktree")` dispatched by §1a and inside the daytime pipeline (§1b), `AskUserQuestion` is forbidden — there is no prompt to demote, so the guard is inert. This is accepted scope.
- The guard does not introduce any structured `recommended: true` option field; `(recommended)` remains a label-suffix prose convention.
- No modifications to `skills/pr/SKILL.md`, `skills/overnight/SKILL.md`, or any other skill that runs its own porcelain check.
- No modifications to the backlog item or the broader epic's end-state; this ticket ships strictly what `backlog/096-*.md` acceptance criteria require plus the narrow option-2 caveat (Requirement 4).

## Edge Cases

- **User on a feature branch, not main/master**: the `implement.md §1` branch-selection prompt is skipped entirely per the existing line 11 condition. The guard never fires in this flow. Expected behavior: pre-flight continues on the current branch without any guard output.
- **Clean working tree on main**: `git status --porcelain` is empty; the guard does not fire; option 3's description and label are presented unchanged. Expected behavior: identical to today's pre-flight.
- **Mid-merge / mid-bisect / mid-rebase**: `git status --porcelain` exits with non-zero status OR returns non-empty output depending on the state. If exit non-zero, Requirement 7's fallback applies. If exit zero with output, the guard fires normally (which is correct — dirty is dirty). Expected behavior: guard either fallbacks to diagnostic or fires demotion; prompt continues.
- **`.git` missing / repo corruption**: Requirement 7's fallback applies — guard skipped, diagnostic printed, pre-flight continues. Expected behavior: no crash, no halt.
- **`lifecycle/{feature}/plan.md` uncommitted from previous phase**: this counts as dirt by design (see Non-Requirements). Guard fires. User sees the warning and can commit the plan.md separately via `/commit` before re-running `/lifecycle implement`.
- **User picks the demoted option anyway**: the pre-flight proceeds to §2 Task Dispatch as today. The guard is informational only; no downstream behavior is changed based on the demotion.
- **Multiple Claude sessions in the same repo at once**: each session runs its own pre-flight independently. The guard is a read-only per-session check; no inter-session coordination is required or implemented.
- **`(recommended)` suffix absent on the current-branch option (pre-#097 world)**: the suffix-strip step is a no-op. Only the description prefix is applied. Expected behavior: warning prefix present, label unchanged.

## Changes to Existing Behavior

- **MODIFIED: `skills/lifecycle/references/implement.md §1` pre-flight**: Gains a new guard paragraph ("Uncommitted-changes guard") adjacent to the existing Worktree-agent context guard. Prose-only change; no new code path and no script.
- **MODIFIED: option 3's description (today "Implement on main", post-#097 "Implement on current branch")**: Under the guard's fire condition, a one-line warning prefix is prepended and the `(recommended)` suffix is stripped if present. Under clean-tree condition, unchanged.
- **MODIFIED: option 2's description ("Implement in autonomous worktree")**: Gains a static one-clause caveat noting uncommitted changes remain on main. Unconditional — present in clean and dirty states alike.
- **ADDED: a single-line diagnostic surfaced on `git status --porcelain` failure**: `uncommitted-changes guard skipped: git status failed`. New prose only.

## Technical Constraints

- **File scope**: all changes are confined to `skills/lifecycle/references/implement.md §1`. No other files are modified by this ticket. Routing block at lines 20–24 is not edited (label match strings remain canonical).
- **Pattern reuse**: the detection idiom `git status --porcelain` with non-empty stdout is the convention established by `skills/pr/SKILL.md:9`, `skills/overnight/SKILL.md:168`, and `claude/statusline.sh:156`. The disposition (demote, not block) diverges from those precedents' blocking disposition deliberately, per the ticket's non-blocking stance.
- **Guard structural pattern**: mirrors the existing `Worktree-agent context guard` at `implement.md:18` — a single paragraph positioned immediately before the `AskUserQuestion` call, with a note alongside the prompt rather than a hard block.
- **No submodules in this repo**: `.gitmodules` does not exist at repo root. Submodule state cannot cause false-positive porcelain output here. This assumption is documented so a future clone that adds submodules knows the guard will fire on their state transitions.
- **No interaction with epic-sibling tickets at the code level**: #094 operates on `claude/pipeline/worktree.py` (daytime pipeline internals) and does not touch `implement.md`. #095 operates on `daytime_pipeline.py` and `implement.md §1b vii`, which is not in §1's pre-flight. #097 operates on §1 but lands after this ticket and is explicitly its downstream dependent. Merge order is #096 → #097; #094 and #095 land in parallel without conflict.
- **No tests to update**: no test suite exercises `implement.md §1`'s pre-flight prompt directly. The existing test suite covers pipeline modules and is not affected.
- **Symlink propagation**: `skills/lifecycle/references/implement.md` is symlinked to `~/.claude/skills/lifecycle/references/implement.md` via the repo's deploy pattern. Edits to the source propagate automatically on the next `/lifecycle` invocation.
