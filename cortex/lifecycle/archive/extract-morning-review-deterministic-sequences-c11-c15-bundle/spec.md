# Specification: Extract morning-review deterministic sequences (C11 + C12)

## Problem Statement

The `/morning-review` skill currently runs two deterministic sequences as agent-orchestrated shell tool calls in `skills/morning-review/SKILL.md`. **C11** (Step 0, lines 23–48) marks the overnight session complete via raw `jq '.phase = "complete"'` writes to `overnight-state.json`. The current path pre-gates on `phase == "executing"` (which is functionally equivalent to what the canonical `cortex_command.overnight.state.transition()` validator enforces on this edge), so the change here is **consolidation**, not a bug fix: routing the write through `transition()` adds a single concrete improvement — a malformed prior phase value (e.g., a typo like `"completed"` or a phase outside `PHASES`) now raises `ValueError` instead of silently overwriting. **C12** (Step 0, lines 50–75) sweeps stale `demo-overnight-*` worktrees under `$TMPDIR` via inline POSIX-ERE regex prose. The inline form is fragile (the SKILL.md already documents ERE-vs-BRE pitfalls), un-fixture-tested, and lacks a precondition that would prevent destruction of a worktree containing untracked user work — `git worktree remove` (no `--force`) refuses dirty *tracked* state but does not consider untracked files dirty.

This ticket extracts both sequences into testable scripts under the project's existing conventions. C11 becomes a Python helper (under `cortex_command.overnight.complete_morning_review_session`) plus a bash shim that mirrors `bin/cortex-update-item`. C12 becomes a narrow bash leaf script with hardcoded prefix and an `git status --porcelain --ignored=traditional` check that refuses removal of any worktree containing uncommitted state. The C11 helper does not change the concurrent-writer race semantics — the cross-file ordering between the state-file and pointer-file writes remains as it was, and broader IPC review is out of scope.

Three other candidates from the original ticket are out of scope per the research-recommended scope reduction: C13 (already adopted in walkthrough.md §5), C14 (a 2-line inline `git fetch && git rev-list --count` is simpler than any wrapper, and a wrapper would add new install-state failure modes), and C15 (the "PATH invariant" the original ticket assumed is false in the general user case — `/plugin install` does not extend PATH, so the fallback chain in `complete.md` is load-bearing).

## Edge Cases

### C11 (session completion)

- **Pointer file absent at Step 0 entry**: helper is invoked WITHOUT `--pointer` (the SKILL.md prose's path-resolution branch picked the `lifecycle/sessions/latest-overnight/overnight-state.json` fallback). Helper performs the state write but does NOT attempt to unlink any pointer. Exit 0. Preserves the existing SKILL.md "Skip if Step 0 used the fallback path (no pointer file)" semantic.
- **Neither pointer nor fallback resolves to a readable file**: SKILL.md's prose continues to be responsible for skipping Step 0 entirely (the agent's path-resolution branch decides not to invoke the helper). Helper itself, when invoked with a non-existent state path, exits 0 silently. Preserves SKILL.md line 30 ("If neither path resolves to a readable file, skip Step 0 entirely — do not error").
- **`phase` is `"complete"`, `"paused"`, or `"planning"`**: helper exits 0 without invoking `transition()` (which would `ValueError`) and without unlinking the pointer.
- **`phase` is malformed (not in `PHASES`)**: helper exits non-zero with a stderr error message naming the invalid phase value. Does NOT silently overwrite. This is the actual schema improvement this ticket lands.
- **State file is unparseable JSON**: helper exits non-zero with stderr error. Does NOT overwrite a corrupt file.
- **Concurrent morning-review invocations** (two simultaneous `/morning-review` runs against the same project): the validator (`cortex_command.overnight.state.transition`) is stateless w.r.t. disk and holds no lock — it does grammar checking only. Both runners can read while `phase=executing`, both can `transition()` successfully, both can `save_state()`. The atomic per-call writes preserve the file but last-writer-wins; idempotent in the case where both writes carry `phase=complete`. The pre-extraction inline path has identical race semantics — this spec does not change them. **The narrative does NOT promise "the second runner sees `phase: 'complete'` and skips silently"** — that's only one of multiple legal interleavings. Out of scope per IPC-review carve-out.
- **Cross-file consistency between state-file write and pointer unlink under concurrent writers**: the helper performs `save_state()` then `unlink(missing_ok=True)` as two independent atomic operations. Concurrent invocations can produce cross-file inconsistency (e.g., runner A reads pointer → runner B unlinks pointer → runner A writes state via the path it cached before B's unlink). `unlink(missing_ok=True)` idempotency does NOT address this read-then-act window. Out of scope per IPC-review carve-out.

### C12 (worktree GC sweep)

- **`$TMPDIR` is unset or `realpath` fails**: script exits 0 silently. Preserves SKILL.md line 66 ("Skip this sweep sub-step silently if `$TMPDIR` is unset or `realpath` fails").
- **`git worktree list --porcelain` fails (not in a git repo, repo is corrupt)**: script exits 0 silently after logging to stderr. Sweep is best-effort cleanup; failure must not block the rest of Step 0.
- **Worktree path matches the `demo-overnight-` prefix and is the active session**: the active-session-ID positional-arg exclusion fires BEFORE the uncommitted-state check. Active session is never removed, even if clean.
- **Worktree path matches the prefix and contains untracked files**: SKIP with stderr log `[gc-demo-worktrees] skipping <path>: uncommitted state`. Do not remove. This is the load-bearing safety guard against destruction of in-flight user work in demo worktrees.
- **Worktree path matches the prefix and has uncommitted tracked changes**: SKIP either via the uncommitted-state precondition (R9) OR via `git worktree remove` itself refusing the operation; both behaviors are acceptable. The pre-check provides the cleaner stderr message.
- **Worktree path matches the prefix, is clean, is not the active session**: REMOVE via `git worktree remove` (no `--force`). On `git worktree remove` failure for any reason (locked, network, permissions), log stderr and continue with the next path. Preserves SKILL.md line 73 ("On failure, print the stderr and continue with the next path — do not abort the sweep").
- **No candidates match**: script runs `git worktree prune` once at the end (preserves the existing belt-and-suspenders behavior), exit 0.
- **`git worktree prune` itself fails**: log stderr, exit 0. Preserves SKILL.md line 74 ("Errors are non-fatal — log and continue").

## Technical Constraints

- **Canonical state-machine API surface (verified at spec time against `cortex_command/overnight/state.py`)**:
  - `def load_state(state_path: Path = DEFAULT_STATE_PATH) -> OvernightState` (line 334) — module-level function; not a classmethod.
  - `def save_state(state, path) -> None` (line 404) — module-level function; not an instance method.
  - `def transition(state: OvernightState, new_phase: str) -> OvernightState` (line 543) — module-level function; takes the state object as first arg; raises `ValueError` on invalid transitions per the grammar at lines 561–599.
  - The C11 module must use exactly these entry points. **Do NOT** write `state.transition(...)`, `state.save()`, or `OvernightState.from_disk(...)` — those methods do not exist on the dataclass.
- **Atomicity**: `os.replace()` (Python) and POSIX `rename(2)` (bash `mv` on the same filesystem) are both atomic. The bash `mv tmp target` pattern in current SKILL.md is structurally correct; the bug being fixed is NOT atomicity. `save_state()` already uses `os.replace()` per `requirements/pipeline.md:134`.
- **Concurrency**: `transition()` is stateless w.r.t. the disk file and holds no lock. Concurrent callers all pass the grammar check independently; last-writer-wins on the file system. This spec preserves prior race semantics — concurrency-safe extraction is a separate IPC concern.
- **`cortex-log-invocation` shim line is mandatory**: per project convention, the first non-comment line after the shebang must invoke `"$(dirname "$0")/cortex-log-invocation" "$0" "$@" || true`. DR-7 telemetry depends on it. Both new scripts must include this line.
- **POSIX ERE regex syntax for C12**: matched via `[[ "$path" =~ ... ]]` (Bash) or `grep -E` / `awk`. Plain `grep` (BRE) and `case` glob both fail to express `[0-9]{4}` quantifiers correctly. SKILL.md lines 56–60 document this — preserve the constraint inside the script.
- **Pytest fixture-repo + `subprocess.run()` pattern**: per `tests/test_git_sync_rebase.py` precedent. Each test function creates a clean tmpdir, sets up fake git state, invokes the real script via `subprocess.run([str(SCRIPT_PATH), ...], cwd=...)`, and asserts on filesystem state and exit code.
- **Dual-source byte-identity**: `bin/cortex-*` and `plugins/cortex-interactive/bin/cortex-*` must be byte-identical at commit time; the `just setup-githooks` pre-commit hook enforces this.
- **`just build-plugin` filter**: `rsync -a --delete --include='cortex-*' --exclude='*'` — both new scripts are auto-included.
- **`bin/cortex-check-parity` scope**: scans `skills/**/*.md`, `CLAUDE.md`, `requirements/`, `tests/`, `justfile`, `claude/hooks/`, `hooks/`. The SKILL.md inline-code references in Requirement 12 satisfy the wiring signal.
- **`requirements/pipeline.md:134` atomicity invariant**: "All session state writes use tempfile + `os.replace()` — no partial-write corruption." Met automatically by `save_state()`.
- **`requirements/project.md:27` parity invariant**: "Drift between deployed scripts and references is a pre-commit-blocking failure mode." Met by Requirement 14.
