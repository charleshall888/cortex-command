# Specification: Sunset cortex-command-plugins and vendor residual plugins into cortex-command

> Epic context: This ticket implements DR-1 (Option C) from [`research/post-113-repo-state/research.md`](../../research/post-113-repo-state/research.md). The post-113 audit re-examined DR-9's premise (kept marketplaces split to avoid forcing global install of orthogonal skills) and found the premise no longer carries under per-plugin install semantics. See [`lifecycle/archive/sunset-cortex-command-plugins-and-vendor-residual-plugins-into-cortex-command/research.md`](research.md) for ticket-specific implementation research.

## Problem Statement

The cortex-command-plugins sibling repo's premise (per-marketplace install would force global install of orthogonal skills) was invalidated by Claude Code's per-plugin install model. Maintaining two marketplaces incurs recurring orphan-state cleanup (N1, N2, N5 in the post-113 audit) on every plugin vendor pass. Sunsetting the sibling repo by vendoring its two residual plugins (`android-dev-extras` and `cortex-dev-extras`) into cortex-command, archiving the sibling, and consolidating into a single marketplace replaces forever-recurring sync cost with a one-time migration. Beneficiaries: the maintainer (primary) and one known downstream consumer (chickfila-android). Why now: each future vendor pass repeats the orphan-state cleanup; sunset is the only durable fix.

## Edge Cases

- **Both marketplaces enabled simultaneously during migration window**: two copies of skills load; slash-command resolution behavior is unspecified per Claude Code docs. Mitigation: redirect README (R8) directs users to `/plugin uninstall <name>@cortex-command-plugins` BEFORE `/plugin install <name>@cortex-command`. chickfila-android settings.local.json migration (R10) executes the swap atomically for the known consumer.
- **Pre-commit Phase 1 ordering within the vendor PR**: justfile:404 update must be in or before the commit that introduces `plugins/X/.claude-plugin/plugin.json` files. Pre-commit catches unclassified plugins at commit time during development; the commit fails with "plugin 'X' not classified" before it can be created. After merge — especially after squash-merge — this signal is no longer recoverable, and post-merge AC commands cannot detect ordering violations because failed intermediate commits never existed in history. The load-bearing safeguard is reviewer attention to commit shape during PR review, not a post-merge automated check.

R11 includes the universal guards and drops the UI-specific one.
