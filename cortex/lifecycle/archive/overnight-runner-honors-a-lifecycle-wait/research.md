# Research: Overnight runner honors a lifecycle "wait"/feature_paused instead of re-executing it

> **OUTCOME — CLOSED WON'T-FIX (2026-06-30, #310).** Refine concluded the ticket's premise is inverted. "Approve plan but wait" is a *soft session-boundary deferral* — the operator wants the next step in a fresh session/later, or is avoiding concurrent work — not an official hold. Overnight is a legitimate fresh-context pickup path, and it **reuses an operator-approved `plan.md`**: `cortex_command/overnight/prompts/orchestrator-round.md:210` synthesizes plans only for features whose `plan.md` is *missing*, so the approved plan is honored, not discarded. Building this eligibility gate would *break* the intended handoff. "paused" stays reserved for mid-step interruption (auto-retries); dependency waits are `blocked_by`. ADR-0012's "deferred limitation / not a claim of safety" note is corrected to match. The research below is retained as the decision record (the held-vs-idle / signal-discrimination / resume tangle only existed because the ticket forced a soft deferral to be read as a hard hold).

**Backlog item:** 310 · **Lifecycle slug:** `overnight-runner-honors-a-lifecycle-wait` · **tier:** complex · **criticality:** high

**Clarified intent:** Make the overnight eligibility path (`filter_ready` in `cortex_command/overnight/backlog.py`) skip a feature whose **interactive** lifecycle is in the `feature_paused` state (operator chose "Approve plan but wait"), so the deferral holds until the operator resumes — closing the gap ADR-0012 recorded as a deliberate deferred limitation. The read path must touch only the per-feature `cortex/lifecycle/{slug}/events.log`, never `overnight-events.log`.

> Research dispatched 7 core-wave angles + 1 adversarial pass. The adversarial pass overturned the Clarify-phase assumption that "resume = the next `phase_transition` clears `paused`," and verified two design-shaping hazards against real on-disk data (legacy overnight-schema pauses in per-feature logs; multi-phase pause breadth). The safety of the core gate is solid; the genuinely-open decisions are the resume semantics, pause-intent discrimination, and the mid-session race — surfaced under Open Questions.

## Codebase Analysis

**Primary change — `cortex_command/overnight/backlog.py:filter_ready` (lines 448–632).** The new pause gate slots in after the artifact (`research.md`/`spec.md`) checks, in **both** code paths that perform per-item gating:
- **Phase-1 loop** (~line 536, after the spec check at 532; local slug var is `slug`, bound at 519). Reject pattern: `result.ineligible.append(IneligibleItem(item, <reason>)); continue` — mirrors the epic/artifact rejects.
- **Phase-2 BFS promotion block** (~line 609, after the spec check at 604; local slug var is `item_slug`, bound at 593). Reject pattern: append `IneligibleItem` + `promoted = True; continue` — mirrors lines 588-589/601-602.

The check belongs **inside `filter_ready`**, peer to the artifact-existence checks — **not** in `cortex_command/backlog/readiness.py:is_item_ready`, which is contractually I/O-free (shared by `generate_index.py` and `bin/cortex-backlog-ready`; a lifecycle-events read would contaminate them).

**The reusable signal already exists — `cortex_command/common.py:detect_lifecycle_phase` (240–336).** `_detect_lifecycle_phase_inner` scans the per-feature `events.log` in line order, tracks `last_significant_event` over `{phase_transition, feature_complete, feature_wontfix, feature_paused}` (305-311), sets `paused = last_significant_event == "feature_paused"` (315), and `_result()` exposes `paused` as a boolean that is True only for non-terminal phases (327). It is `lru_cache`d on `_stat_key` tuples of all five artifacts and exposes `__wrapped__` (441). `feature_dir` derives from the same `project_root / "cortex" / "lifecycle" / slug` path `filter_ready` already uses. No import cycle (`common.py` imports nothing from `overnight.backlog`; backlog.py already imports from `common`).

**Reason-string convention:** `IneligibleItem(item, reason)` is surfaced by `select_overnight_batch` as `"Not ready: {title} ({reason})"` (1114-1120). Existing reasons are lowercase parentheticals (`"epic is non-implementable"`, `"spec file not found: ..."`, `"pipeline branch already merged into main"`). A paused reason should match — e.g. `"paused by operator (Approve plan but wait)"`.

**Callers and the #323 curation path inherit the gate automatically.** `select_overnight_batch` (1043) calls `filter_ready`; `handle_prepare` (cli_handler.py:2062) and `handle_launch` (2143) call `select_overnight_batch`. `filter_selection_to_curated_set` (1132-1250, the #323 `--only` path) post-filters the already-computed selection — a paused feature never lands in `batches`, so a curated paused slug falls into the fail-loud `ineligible_slug` branch (1177-1189). No separate changes needed in any caller, `events.py`, `outcome_router.py`, or `lifecycle_implement.py`.

**Files that change:** `cortex_command/overnight/backlog.py` (gate); `cortex_command/overnight/tests/test_backlog.py` (new `TestFilterReadyPausedGate`); optionally `cortex_command/common.py` (only if a focused interactive-pause reader is chosen over reusing `detect_lifecycle_phase`); optionally `bin/.events-registry.md` (add `overnight/backlog.py` to the `feature_paused` consumers cell — registry hygiene, non-blocking).

## Web Research

The canonical two-state model — verified across multiple mature schedulers — is **HTCondor's `held` vs `idle`**:
- **held** = a deliberate signal that manual intervention is needed; **never auto-rerun**; stays in queue, visible, ineligible; cleared only by explicit `condor_release` (or a `periodic_release` expression).
- **idle** = transiently waiting; **may auto-rerun** under retry policy.

This is exactly the split this ticket must preserve: the **interactive `feature_paused`** is a *held* (deliberate, must not auto-run), while the **overnight status `paused`** (pipeline.md) is *idle*-like (recoverable, auto-retries on resume). Conflating them is the central anti-pattern. Corroborating prior art: Airflow DAG pause (scheduler creates no new runs until resumed), Argo `suspend` (and its explicit *indefinite-manual-resume vs `duration:`-auto-resume* distinction — an acknowledgment that "wait for a human" ≠ "wait then continue"), Temporal activity Pause/Unpause (stops scheduling new retries — an operator hold layered separately from the retry machine), GitHub Actions environment protection, GitLab `when: manual`, Jenkins `input`, Slurm/Maui held states (Maui distinguishes user/system/batch/defer hold *types*).

**Event-sourcing reducer guidance** (Fowler; event-driven.io; CockroachDB):
- A "last-significant-event-wins" fold **requires guaranteed append order** and **filtering to significant event types** — a stale/non-significant event treated as a state reset is the classic accidental-clear bug.
- Idempotent handlers (at-least-once delivery) and unique event ids.
- **Same event name meaning two things across subsystems** is a named hazard — namespacing/discrimination is the mitigation (directly relevant to OQ2).

**Anti-patterns flagged:** (a) silently overriding a deliberate human deferral by auto-executing; (b) a gate with no defined un-block path → stuck forever (note: a deliberate hold should *escalate/notify* on timeout, never silently auto-resume); (c) "blocks vs optional" ambiguity must be encoded explicitly.

## Requirements & Constraints

- **`feature_paused` is already a registered, dual-emitted event** (`bin/.events-registry.md:31`, target `per-feature-events-log | overnight-events-log`). The interactive "wait" arm writes the per-feature log via `cortex-lifecycle-event log --event feature_paused` (added #330); overnight `outcome_router.py` writes the overnight log. **This change is a pure READ — no new event minted, no events-registry/parity/size-budget gate tripped.** Optional hygiene: add `filter_ready` to the `feature_paused` consumers cell (currently lists only `dashboard/data.py`).
- **ADR-0012 (status: accepted → binding)** introduced "Approve plan but wait" (emits `plan_approved{dispatch_choice:"wait"}` + `feature_paused`) and **explicitly records this overnight-honoring gap as a deliberate deferred limitation that is "not a claim of safety"** — the eligibility gate "keys on backlog status + research/spec presence and does not read `dispatch_choice` or `feature_paused`." Per project.md **Solution horizon**, this is a pre-named planned follow-up that sidesteps a known constraint → the *durable* fix (actual enforcement) is warranted.
- **Semantic-collision constraint:** pipeline.md "Feature Execution and Failure Handling" — overnight feature status `paused` "means a recoverable error … paused features **auto-retry** when the session resumes." This is the **opposite** of the interactive `feature_paused`. The read-path restriction (per-feature log only) is the structural guard; OQ2 (intent discrimination) is the residual guard for legacy data.
- **detect_lifecycle_phase read precedent:** scans the per-feature `events.log` in **line order, explicitly NOT ts-sorted** (malformed/missing ts would mis-sort). The new gate should mirror this (reuse it, or share its 4-event significant set rather than re-listing).
- **`filter_ready` reads no `events.log` today** — only frontmatter status, blockers (`is_item_ready`), epic type, artifact presence, git. This change introduces the first such read; no requirement prohibits it. Only stated constraints: per-feature-only read target (topic) and "without coupling … in a brittle way" (ticket Edge).
- **Backlog status vocabulary divergence** (project.md Architectural Constraints): the `overnight/backlog.py` frozenset/`ELIGIBLE_STATUSES` is a "known divergence tracked for a separate follow-up." The pause-skip is an *additional gate*, not a terminal-status-vocabulary change, so it does not trip the two-location update rule — but should not entangle with that divergence.
- **SKILL.md-to-bin parity does NOT apply** — `overnight/backlog.py` is an internal module, not a `bin/cortex-*` console script; it is not on the mandatory-lifecycle file list. `grep -c "feature_paused"` resolves as a legitimate Done-When token (registry row + codebase literal).
- **Existing precedent for overnight-skips-on-interactive-state:** `orchestrator.py:~100-140` already emits `feature_skipped_interactive_active`, keyed on **live `interactive.pid` locks** (not events.log), inside the **per-round** loop (not `filter_ready`), to the overnight log. This is the architectural hook the adversarial pass argues a mid-session pause check belongs in (OQ4), and the rationale-field convention (skip decisions carry a `rationale`).

## Tradeoffs & Alternatives

**Decision 1 — which signal the gate keys on:**

- **Option A — reuse `detect_lifecycle_phase(feature_dir)["paused"]`.** Lowest net-new code (one call), single source of truth (joins `scan_lifecycle`, `dashboard`, `generate_index` as a consumer of the same detector), respects the `generate_index.py:177-181` decision that paused-ness is read *live* from events.log (never cached in the index). The phase-gating is a feature: a paused-then-completed feature already reports `paused=False`. **Cost:** per otherwise-eligible item adds ~5 `os.stat` + reads of `events.log`/`plan.md`/`review.md` — `lru_cache`-backed, sub-ms, negligible at selection scale (order the check *after* the artifact checks so it runs only for otherwise-eligible items). **Hazard (see Adversarial + OQ2):** schema-blind — would treat a legacy overnight-error pause as a hold.
- **Option B — `read_dispatch_choice(events_path) == "wait"`.** Narrower read, but **broken for resume**: the picker-driven resume never emits a new `plan_approved`, so `dispatch_choice` stays `"wait"` until Plan §4 re-runs; it also can't see `feature_complete`/`escalated` and returns `None` on legacy logs. A second, narrower notion of "paused" spread across the codebase.
- **Option C — new backlog frontmatter `status: paused/deferred`.** Highest complexity (touches `STATUSES`/`ELIGIBLE_STATUSES`/`normalize_status`, the divergent frozensets), needs a writer on the interactive side (brushes the non-goal) **and** a resetter on resume, and re-caches events.log state into a stale-prone `index.json` — directly reversing the deliberate `generate_index.py` strip-paused decision. Rejected.

**A focused-reader variant of A** (a new `events.log`-only helper in `common.py` that returns the interactive-pause verdict) is the literal fit for "touch only events.log" and the natural home for OQ2's intent discrimination — at the cost of a second reducer to keep in sync with the 4-event set. (Note the precision point: `detect_lifecycle_phase` honors file-separation — it never reads `overnight-events.log` — but it does read `review.md` and stat the other artifacts, so it touches more than *only* `events.log`. If the constraint is read literally, the focused reader is required; if the constraint is really "never `overnight-events.log`," Option A satisfies it.)

**Decision 2 — where the check lives:** inside `filter_ready`, **mirrored into Phase-1 and Phase-2**, after the artifact checks, emitting a structured `IneligibleItem`. Durability note: the post-status gates are *already* inconsistently duplicated (research/spec in both phases; pipeline-merge Phase-1 only). Adding paused as a third inline-duplicated check increases drift in two namable places — the durable move is a shared `_passes_post_status_gates(item, project_root)` helper called from both phases; the minimal move is to mirror paused as research/spec are mirrored and file the helper-extraction as a follow-up.

**Recommended:** Option A **with intent discrimination** (OQ2), check inside `filter_ready` mirrored into both phases. Validating the ticket's suggested Option A held up — it is the only option that aligns with the live-read decision and avoids stale caching — but the naive form is not safe; discrimination is required.

## Signal Semantics & Resume Correctness

**Events that clear `paused`:** `phase_transition` (any from/to), `feature_complete`, or `feature_wontfix` appended *after* `feature_paused` (the latter two also short-circuit to terminal `complete`). Line-position-last wins.

**The Clarify assumption is wrong — "resume = next `phase_transition`" does NOT match emission:**
- The wait arm (`plan.md:293-296`) appends `plan_approved{wait}` + `feature_paused` and halts; it emits `phase_transition plan→implement` **only on the branch-mode arm, explicitly NOT on "wait"** (plan.md:302). So the paused log ends `…feature_paused` with no transition.
- Resume re-enters Implement §1 (plan already approved), emitting `interactive_worktree_entered` + `batch_dispatch` — **neither is a significant event**. The only `phase_transition` in implement.md fires at the *end* (`implement→review`, implement.md:274; `review→implement-rework`, 247).
- **Consequence:** `paused` does **not** clear at operator-resume-entry; it clears only when the interactive implement work finishes (`implement→review`) or rework/terminal. There is **no `feature_resumed` event anywhere in the repo.**

**Soundness as a safety gate: solid.** No accidental-unpause vector survives for a feature that is paused *at the selection snapshot*:
- **File separation PROVEN for current code:** every `outcome_router.py` `FEATURE_PAUSED` emit (545, 620, 649, 674, 828, 889, …) targets `log_path=ctx.config.overnight_events_path`; there is no per-feature write target in the file. `overnight_events_path` and `pipeline_events_path` are flat files in `cortex/lifecycle/`, distinct from per-feature dirs. The gate reads the per-feature log.
- **`review_dispatch.py` is the sole automated writer of `phase_transition` into the per-feature log** (214/218/318/607; `feature_complete` at 324/613) — but only while overnight is *actively executing* that feature. For a feature paused at selection (correctly gated out), it never runs. **However** this safety is self-referential — "closed iff the gate is correct" — and has a real hole for the mid-session race (see Adversarial / OQ4).
- Line-order sound: both per-feature writers append at EOF (interactive `flock`+`O_APPEND`; `review_dispatch` via `state.py:log_event` `open(…,"a")` = `O_APPEND`). Torn final line fails `json.loads` → skipped → fails *open* (a just-written pause missed for one read).
- Terminal events supersede paused — no gap, no double-jeopardy; a paused-then-completed feature reads `complete` and is excluded once by the terminal/status gate.

**Two candidate resume intents (the core product fork → OQ1):**
- **Intent A (conservative, matches current emission):** a paused feature is overnight-ineligible until a human pushes it past the paused phase. Overnight never re-picks it on its own. Safe; zero new signal. But a resumed-then-abandoned feature stays ineligible until it reaches review/terminal — a "roach motel" if the intent was "hand back to overnight."
- **Intent B (hand back to overnight):** the operator can un-pause so a *future overnight run* implements it without interactive work — **unmet today**; needs an explicit resume signal (a `feature_resumed` event, emit `plan→implement` on wait-resume to restore the symmetry plan.md skips, or a CLI un-pause verb).

## Edge Cases & Cross-System State Interactions

1. **Paused feature as a blocker of an eligible dependent — ruled out, with a load-bearing rider.** "paused" is a phase concept, not a status; a paused blocker keeps a non-terminal `ELIGIBLE_STATUSES` value, so it is non-terminal in the BFS. Because the gate keeps it out of `result.eligible`, its dependent stays in `pending_blocked` → falls through to `"blocked by 0NN (not in session)"` — correctly held, not wrongly promoted. **Rider:** the gate must mark the blocker **ineligible but keep it in `all_items`** (`status_by_id`/`slug_by_id` are built from `all_items` at 478-493); pre-filtering it out of `all_items` would mislabel the dependent as `"external blocker"`. The standard reject pattern (append `IneligibleItem` + `continue`) preserves this — but no existing test pins it.
2. **Mid-session interactive pause — selection-time-only; gate must not try to solve it.** `filter_ready`/`select_overnight_batch` run only at prepare/launch; the round loop never re-selects. A pause after launch is not honored for the in-flight feature. **The adversarial pass sharpens this (OQ4): the pause marker is not just ignored — it is erased** by `review_dispatch`'s `phase_transition`.
3. **Visibility — a pre-session operator-paused skip is invisible to all post-session surfaces** (dashboard/report consume the overnight pause stream + `fs.status=="paused"`). It appears only in the prep-time selection summary, where folding it into the generic ineligible bucket makes the header advise **"run /discovery first"** — wrong remediation for a paused item (needs un-pausing). **Spec must give a distinct reason and segregate it** from genuine not-ready items. No post-session surfacing needed (unlike `recoverable_branch`, no surprising work-state exists).
4. **#323 curation of a paused slug — ruled out.** Fails loud via `ineligible_slug`. Optional polish: add "or operator-paused" to the cause list in that error message (currently lists typo/drift/blocker-removed).
5. **Status orthogonality — gate is non-redundant, no exception status.** Pause is orthogonal to the status gate; no eligible status implies pause, and there is no status under which an operator pause should be overridden.
6. **Resume re-eligibility — clean in the common case, gap in the corner.** A later `phase_transition` re-admits with all other gates re-applying. But with no `feature_resumed` event, a same-phase resume (no new significant event) leaves `paused=True` → permanent exclusion (→ OQ1).

## Adversarial Review

The proposal's primitive (reuse `paused` inside `filter_ready`) is right, but the safety framing had concrete holes, several **verified against real on-disk data**:

1. **Schema conflation — VERIFIED (drives OQ2).** Real per-feature `events.log` files carry overnight-schema `feature_paused` rows: `lead-refine-4-complexity-value-gate:19` (`{"v":1,"session_id":"manual","details":{"error":"branch appears stale…","no_commit_guard":true}}`) and `archive/rewrite-verification…:60,71` (`details.error:"Task N failed after K attempts"`). `detect_lifecycle_phase` matches `feature_paused` with **zero schema discrimination** (ignores `v`/`session_id`/`details`). Today these are followed by clearing events (not last-significant) so they don't bite, but the gate has **no defense**: the first overnight error-pause landing as the last significant line of a still-eligible item silently converts a transient retriable failure into a permanent eligibility block. **Discriminator available:** overnight error-pauses carry `details.error` (+`v`/`session_id`/`round`); interactive pauses carry operator `reason`/`rationale` (or the bare `{ts,event,feature[,phase]}` shape) and no `details.error`.
2. **The pause-erasure race (sharpens edge #2) → OQ4.** `filter_ready` runs once; the orchestrator executes a frozen plan. A feature eligible-at-selection then paused mid-session is executed, and `review_dispatch` writes `phase_transition implement→review` as its first action into the per-feature log → `paused=False` **permanently**. The deliberate "wait" is overridden *and the marker annihilated* — the next night's gate can't see it either. "Document the boundary" understates the data loss.
3. **Self-referential safety is not closed** for the mid-session race (an ordering where `phase_transition` lands in a paused feature's log with no operator resume).
4. **Phase breadth — VERIFIED (→ OQ3).** `paused` trips on `feature_paused` at *any* phase. Real specify→plan pauses exist: `overnight-dispatch-sends-opus-only-xhigh:10` (`phase:plan, rationale:operator approved spec; paused before Plan`) and `review-gate-flag-coherence:9` (`reason:operator chose approve-spec-but-wait-to-plan`). `filter_ready` gates only on research+spec presence, so these pass artifact checks and the gate suppresses them — arguably correct, but unstated, and these research+spec-complete items must not route into the "run /discovery first" bucket.
5. **Resume stranding under Intent A is real** (→ OQ1): no `feature_resumed`; a wait-then-resume-then-re-wait feature is stranded; the prep-summary mitigation only helps an operator who runs `prepare` and reads it — it does nothing for the silent overnight-selection drop.
6. **No test pins "keep in `all_items`"** (edge #1 rider); torn-read handling is inconsistent between `detect_lifecycle_phase` (swallows) and `outcome_router`/`reduce_lifecycle_state` (treats as corruption) reading the same file — low impact for a one-shot selection read.

**Adversarial-recommended mitigations** (spec-phase decisions, several beyond the minimal ticket scope): discriminate pause intent (1); close the erasure hole via a per-round check or by making `review_dispatch` refuse to overwrite a pause it didn't author (2/4); define a resume signal (3/OQ1); segregate the ineligible reason (4); pin the blocker-graph invariant with a test (6); decide phase breadth explicitly (4).

## Open Questions

These require resolution at the Research Exit Gate / Spec phase. OQ1, OQ2, OQ4 are genuine product/scope decisions; OQ3, OQ5 are spec-coverage items with a clear recommended default.

- **OQ1 — Resume semantics (the core product fork). What does "the operator resumes" mean?** *Intent A* (conservative, matches current emission, gate-only): a paused feature stays overnight-ineligible until a human pushes it past the paused phase; overnight never re-picks it on its own. *Intent B* (hand back to overnight): the operator can un-pause so a future overnight run implements it without interactive work — requires a new resume signal (`feature_resumed` event, emit `plan→implement` on wait-resume, or a CLI un-pause verb), and brushes the "don't change the interactive surface" non-goal. **Recommended default: Intent A** (in scope; delivers the ticket's primary outcome; B is a scope expansion). The Clarify-phase assumption that resume happens automatically via a phase_transition is **incorrect** and must not be relied on.
- **OQ2 — Pause-intent discrimination (correctness).** Should the gate distinguish interactive operator pauses from overnight transient-error pauses (both can appear as `feature_paused` in the per-feature log; legacy data verified)? **Recommended: yes** — gate only on the interactive-pause shape (operator `reason`/`rationale`, or absence of overnight `details.error`/`session_id`), so a stale/legacy overnight error-pause as the last significant line does not permanently suppress an otherwise-eligible feature. This is cheap and prevents the mirror-image failure (should-run feature silently blocked). Naive `detect_lifecycle_phase["paused"]` does **not** discriminate.
- **OQ3 — Phase breadth.** Confirm the gate is intended to suppress operator pauses at **any** phase (`approve-spec-but-wait-to-plan` and `approve-plan-but-wait-to-implement` both exist), not just plan→implement. **Recommended: yes, all-phase** — any deliberate operator pause should hold. The ineligible reason for these items must be segregated from the "run /discovery first" bucket (they already have research+spec).
- **OQ4 — Mid-session race / pause erasure.** `filter_ready` is selection-time-only; a feature paused after launch is executed anyway and its pause marker is erased by `review_dispatch`'s `phase_transition`. Accept selection-time-only and **document the boundary + the erasure** (in scope), or add a per-round pause check alongside the existing `interactive.pid` skip in `orchestrator.py` (scope expansion that closes the race). **Recommended default: document + accept** for this ticket; file the per-round check as a follow-up unless the user wants it folded in.
- **OQ5 — Reason segregation.** Emit a distinct ineligible reason (e.g. `"paused by operator (Approve plan but wait)"`) and exclude paused items from the prep-summary "N ineligible — run /discovery first" line (route to a "held by you" sub-bucket). Add "or operator-paused" to the `ineligible_slug` curation error cause list. **Recommended: yes.**
- **OQ6 — Durability of placement.** Mirror the paused check into Phase-1 and Phase-2 inline (minimal, in scope) vs. extract a shared `_passes_post_status_gates` helper now (durable; also de-duplicates the existing research/spec mirror). **Recommended: minimal mirror + follow-up ticket**, unless the user wants the cleanup folded in.

## Note on the contradiction resolved during research

The Signal-Semantics angle proved "file separation" for the *current* `outcome_router` code (overnight `feature_paused` → overnight log only); the Adversarial angle verified that *legacy* per-feature logs nonetheless contain overnight-schema `feature_paused` rows, and that `detect_lifecycle_phase` is schema-blind. Both are correct for different time windows. The reconciliation is OQ2: the read-path restriction is necessary but **not sufficient** — the gate also needs pause-intent discrimination to be robust against legacy/edge data.
