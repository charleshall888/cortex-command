# Specification: Runtime adoption telemetry for bin/cortex-* scripts (DR-7)

> **Epic reference**: This ticket is a sub-ticket of epic 113 (now complete). Epic-level research at `research/extract-scripts-from-agent-tool-sequences/research.md` (DR-7 in particular) established the runtime-adoption-failure detection gap that this ticket closes. Lifecycle-scoped research at `lifecycle/archive/add-runtime-adoption-telemetry-via-pretooluse-bash-hook-matcher-dr-7/research.md` evaluated five alternative mechanisms; the spec below adopts Alt 5 (per-script invocation shim) composed with DR-5 (ticket 102 static parity lint), revised in response to critical review.

## Problem Statement

The cortex-command project currently cannot detect when a `bin/cortex-*` script is wired into the codebase but never actually invoked at runtime — the DR-7 third failure mode that DR-5's static parity lint cannot catch (research: §Codebase Analysis, §Tradeoffs). Without this signal, scripts ship and rot silently. The chosen mechanism is a per-script invocation-logging shim (`bin/cortex-log-invocation`) called as the first line of every `bin/cortex-*` script, writing one JSONL record per invocation to a per-session log. An aggregator CLI (`bin/cortex-invocation-report`) joins the logs against the live `bin/cortex-*` inventory to surface a **list of candidates for review** — scripts with zero invocations in the recent window. The aggregator does not by itself prove "the agent had occasion and chose Read+Grep instead" — that requires human interpretation against the script's expected use frequency and audience. Composed with DR-5 (which catches "wired-but-not-referenced") and a new `--check-shims` pre-commit gate (which catches "shim line missing from a new script"), this provides the strongest practical runtime-adoption signal achievable without coupling to undocumented Claude Code internals or adding a per-Bash-call hook surface. The shim depends on prior `cortex init` having registered the per-session log path in the user's sandbox `allowWrite` list; without that registration the helper fails open silently in default permission mode and the aggregator's `--self-test` mode is the diagnostic path.

## Requirements

1. **Shim helper script `bin/cortex-log-invocation` exists and is executable**: `test -x bin/cortex-log-invocation && echo OK` returns "OK".

2. **Helper writes one JSONL record per invocation to the per-session log**: After running `LIFECYCLE_SESSION_ID=test-sid bin/cortex-log-invocation /path/to/some-script a b c` from project root, `wc -l < lifecycle/sessions/test-sid/bin-invocations.jsonl` outputs `1`, and the line parses as JSON containing keys `ts`, `script`, `argv_count`, `session_id`.

3. **Helper fails open when LIFECYCLE_SESSION_ID is unset**: `LIFECYCLE_SESSION_ID= bin/cortex-log-invocation /tmp/x; echo $?` outputs `0` and writes nothing.

4. **Helper fails open when log target is unwritable** (sandbox denial, missing directory, permission error, no project root resolvable): with the failure scenario simulated, `bin/cortex-log-invocation /tmp/x; echo $?` outputs `0` and the calling script continues normally. Helper writes a one-line diagnostic to `~/.cache/cortex/log-invocation-errors.log` (best-effort, also fail-open) recording the failure class — this is the breadcrumb the `--self-test` mode reads to detect silent-failure populations.

5. **Helper performance budget**: `time bin/cortex-log-invocation /tmp/x` reports `real` < 0.05s on the dev machine. Implementation must use bash + POSIX utilities (no `jq`, no Python interpreter spawn). Note: for the four bash-wrapper scripts (`cortex-create-backlog-item`, `cortex-generate-backlog-index`, `cortex-jcc`, `cortex-update-item`), the shim invocation doubles the per-call process-spawn count (one for the shim + one for the underlying `python3 -m`). Acceptance is "negligible additive overhead," not "zero overhead."

6. **All canonical `bin/cortex-*` scripts (excluding the helper itself) reference `cortex-log-invocation` near the top of the file**: For every `f` in `bin/cortex-*` other than `bin/cortex-log-invocation`, `head -50 "$f"` contains the literal string `cortex-log-invocation`. Acceptance loop: `for f in bin/cortex-*; do [ "$(basename "$f")" = "cortex-log-invocation" ] && continue; head -50 "$f" | grep -q "cortex-log-invocation" || echo "MISSING: $f"; done` produces no output. (The grep is intentionally permissive to accept all valid invocation forms — bash, Python with `subprocess.run`, with or without `|| true` / `check=False`. The presence of the literal name in the first 50 lines is the gate; correct fail-open semantics are spec-mandated by the prescribed invocation patterns in Changes to Existing Behavior, not enforced by R6.)

7. **Aggregator CLI `bin/cortex-invocation-report` exists and is executable**: `test -x bin/cortex-invocation-report && echo OK` returns "OK".

8. **Aggregator default output reports per-script invocation counts**: After populating one log entry under `cortex-update-item`, `bin/cortex-invocation-report` output contains a line where the script name `cortex-update-item` appears alongside its invocation count.

9. **Aggregator default output includes a `CANDIDATES FOR REVIEW` section listing inventory items with zero counts in the recent window**: After populating one log entry under `cortex-update-item` only, `bin/cortex-invocation-report | grep -A 100 "CANDIDATES FOR REVIEW"` contains at least one of the other inventory entries (e.g., `cortex-audit-doc`). Section heading is `CANDIDATES FOR REVIEW` (not `NEVER INVOKED`) to honestly represent that the listing is a starting point for human triage, not a definitive runtime-non-adoption verdict.

10. **Aggregator `--json` flag emits structured JSON to stdout**: `bin/cortex-invocation-report --json | jq -e '.scripts | type == "array"'` exits 0; the JSON contains `inventory`, `scripts`, and `candidates_for_review` arrays plus a `metadata` object reporting log volume and any errors logged in `~/.cache/cortex/log-invocation-errors.log`.

11. **Aggregator `--check-shims` flag verifies each `bin/cortex-*` references the helper**: With all shims in place, `bin/cortex-invocation-report --check-shims; echo $?` outputs `0`. After removing the shim line from one script, the same command exits non-zero and writes the missing script's name to stderr. The check uses the same logic as R6's grep.

12. **Aggregator handles no-data case gracefully**: With no `lifecycle/sessions/*/bin-invocations.jsonl` files present, `bin/cortex-invocation-report; echo $?` outputs `0` and the report includes "No invocations logged" plus the full `CANDIDATES FOR REVIEW` section listing every inventory item.

13. **Aggregator `--self-test` flag verifies end-to-end telemetry function**: `bin/cortex-invocation-report --self-test; echo $?` exits `0` if and only if (a) the helper resolves the per-session log path successfully under the current `LIFECYCLE_SESSION_ID`, (b) a probe write succeeds, (c) the probe record is read back. Exit non-zero if any step fails; print to stderr which step failed and a remediation hint (e.g., "run `cortex init` to register the log path in sandbox.filesystem.allowWrite"). This is the user's diagnostic for "is my telemetry actually working?" — addresses the silent-failure mode in default permission mode without `cortex init` registration.

14. **Pre-commit gate runs `--check-shims` automatically when `bin/cortex-*` files are staged**: `.githooks/pre-commit` is updated such that staging a new or modified `bin/cortex-*` file triggers `bin/cortex-invocation-report --check-shims`. If the check fails, the commit is rejected with a message naming the script(s) missing the shim line. Acceptance: with `.githooks/pre-commit` installed (`just setup-githooks` already run), `git add bin/cortex-some-new-script` followed by `git commit -m "test"` exits non-zero and stderr names the missing-shim script.

15. **Plugin distribution via `just build-plugin`**: After `just build-plugin`, `diff bin/cortex-log-invocation plugins/cortex-interactive/bin/cortex-log-invocation` and `diff bin/cortex-invocation-report plugins/cortex-interactive/bin/cortex-invocation-report` both exit 0.

16. **Ticket 103 amended in place to reflect the new mechanism**: `backlog/103-add-runtime-adoption-telemetry-via-pretooluse-bash-hook-matcher-dr-7.md` has its `title:` frontmatter and `# Heading` rewritten to name the per-script invocation shim mechanism (suggested: "Add runtime adoption telemetry via per-script invocation shim (DR-7)"), and the `## Scope`, `## Out of scope`, and the 2026-04-27 scope-amendment sections rewritten to describe the shim approach. Acceptance: `grep -c "PreToolUse Bash hook matcher" backlog/103-*.md` outputs `0` AND `grep -c "invocation shim" backlog/103-*.md` outputs ≥ 2.

17. **`requirements/observability.md` adds a sixth subsystem section for runtime adoption telemetry**: A new subsection under `## Functional Requirements` documents the shim + aggregator subsystem with Description, Inputs, Outputs, Acceptance criteria, Priority. Acceptance: `grep -c "Runtime Adoption Telemetry" requirements/observability.md` ≥ 1, AND `awk '/### Notifications/,/### In-Session Status CLI/' requirements/observability.md | grep -c "Runtime Adoption Telemetry"` ≥ 1.

18. **`just test` continues to pass**: `just test` exits 0.

## Non-Requirements

- The shim does NOT capture the agent's reasoning for choosing or not choosing a script — only whether the script ran. Agent intent classification (Alt 3's "Read+Grep instead of script" heuristic) is explicitly out of scope; high false-positive rate per Adversarial F10.
- The shim does NOT integrate with pipeline `agent-activity.jsonl` or `pipeline-events.log` — those remain pipeline-scoped per ticket 103's original out-of-scope item.
- The shim does NOT capture non-`bin/cortex-*` scripts or arbitrary Bash invocations — this is an intrinsic property of the per-script shim approach.
- No PreToolUse hook is added — this is the explicit scope amendment from ticket 103's original proposal.
- The aggregator does NOT classify scripts by expected frequency or agent-facing-vs-utility taxonomy in v1. The `CANDIDATES FOR REVIEW` list is a triage starting point: the consumer (a human reviewing the report) is expected to know that, e.g., `cortex-git-sync-rebase` is a developer utility while `cortex-update-item` is heavily agent-facing, and to interpret zero-count entries accordingly. A future v2 may add per-script tags via inline metadata if false-positive volume warrants it.
- The helper does NOT distinguish "sandbox-denied write" from "successful no-op" in v1. Both fail open silently. Users who have not run `cortex init` (or whose `cortex init` registered a path that does not match the helper's resolved log path) will produce no telemetry in default permission mode. The `cortex-invocation-report --self-test` command is the diagnostic for this case. A `~/.cache/cortex/log-invocation-errors.log` breadcrumb (R4) accumulates per-failure-class entries for postmortem.
- Telemetry from agent worktrees (overnight isolation under `/private/tmp/claude-503-overnight-worktrees-*`) is NOT captured separately. The helper writes to `lifecycle/sessions/<id>/bin-invocations.jsonl` resolved relative to the worktree's `git rev-parse --show-toplevel`; that path may not exist inside a fresh worktree, in which case the helper fails open. v1 telemetry is therefore biased toward main-tree invocations; the bias is acknowledged here, not corrected in v1.
- `cortex-jcc`'s adoption is captured at the wrapper level only. The underlying `just` recipes that `cortex-jcc` dispatches are NOT separately tracked; the only signal is "did the dispatcher run." For DR-7's purpose of catching wired-but-unused `bin/` scripts this is sufficient; recipe-level telemetry is a separate (out-of-scope) concern.
- No log rotation policy is shipped — per-session granularity inherits lifecycle session-archive cleanup. With volume estimated at ~70 KB/day across all session dirs, rotation is deferrable for years.
- No backfill of historical adoption data — Alt 3's day-one backfill from existing Claude Code session JSONL is rejected per Adversarial F1, F6 (Anthropic auto-update data-loss bugs #41591, #53417 active in this user's version range).
- The aggregator does NOT trigger automatically — it is invoked on-demand (manual run or weekly), per ticket 103's original "weekly or on-demand aggregator" framing.

## Edge Cases

- **Script run outside a Claude session** (user invokes from a regular shell): `LIFECYCLE_SESSION_ID` unset → helper exits 0 silently → script proceeds normally → no log entry. (R3.)
- **Concurrent script invocations from parallel agent worktrees writing to the same session log**: POSIX `O_APPEND` atomicity at sub-4KB records makes this safe on local APFS/ext4 (research-verified). Each helper invocation writes one line < 1KB.
- **Session directory archived during aggregator run**: aggregator iterates `lifecycle/sessions/*/bin-invocations.jsonl` with `[ -f ... ]` guards; missing files skipped; aggregator continues.
- **Helper errors mid-write (disk full, permission denied, sandbox denial)**: shim invocation is fail-open; calling script continues regardless of helper exit code. Failure class is recorded as a one-line entry in `~/.cache/cortex/log-invocation-errors.log` (best-effort; itself fail-open if the cache path is unwritable). The aggregator's `--self-test` and `--json` modes surface error counts from this breadcrumb.
- **Plugin install without `cortex init`**: helper writes silently fail in default permission mode; user's reports show `CANDIDATES FOR REVIEW` listing every script. `cortex-invocation-report --self-test` (R13) is the recommended first-run diagnostic — it surfaces "log path not in allowWrite" as a clear, remediation-actionable error rather than letting the user mistake silent-failure for non-adoption.
- **Worktree-rooted helper invocations** (overnight pipeline, agent isolation): `git rev-parse --show-toplevel` from inside a worktree returns the worktree path, not the project root. The `lifecycle/sessions/<id>/` directory typically does not exist in the worktree → helper fails open. v1 does not bridge this gap. Reports are biased toward main-tree invocations (acknowledged in Non-Requirements).
- **New `bin/cortex-*` script added without the shim line**: pre-commit gate (R14) catches this on `git commit`. Manual `cortex-invocation-report --check-shims` is the offline check.
- **uv-script PEP 723 files** (`cortex-audit-doc`, `cortex-count-tokens`): the shim invocation goes immediately AFTER the `# ///` close marker, before the script's first import. uv parses metadata first, runs the script body second — so the shim runs after dependency resolution (which can take 100ms–1s). The "negligible overhead" framing in R5 weakens here; the shim still records the invocation but contributes single-digit-ms overhead on top of uv's already-paid setup cost.
- **Script-shape variation across the 9-item inventory**: the inventory is heterogeneous — 1 bash leaf (`cortex-git-sync-rebase`), 4 bash wrappers (`cortex-create-backlog-item`, `cortex-generate-backlog-index`, `cortex-jcc`, `cortex-update-item`), 2 plain Python (`cortex-archive-rewrite-paths`, `cortex-archive-sample-select`), 2 PEP 723 uv-script Python (`cortex-audit-doc`, `cortex-count-tokens`). The Changes to Existing Behavior section enumerates the prescribed shim form per class.
- **Helper invoked with no argv beyond script name** (`bin/cortex-log-invocation /path/to/foo`): writes a record with `argv_count: 0`. Valid; not an error.
- **JSONL log file has malformed lines** (e.g., partially-written record from an interrupted process): aggregator skips bad lines and reports a count of skipped records in `--json` output's metadata block.

## Changes to Existing Behavior

- **ADDED**: `bin/cortex-log-invocation` (new shim helper script — bash, POSIX utilities only).
- **ADDED**: `bin/cortex-invocation-report` (new aggregator CLI — bash + POSIX utilities; `jq` optional for `--json` output).
- **ADDED**: `~/.cache/cortex/log-invocation-errors.log` (helper-side error breadcrumb; created on demand; not committed; not rotated).
- **ADDED**: New observability subsystem section in `requirements/observability.md` (sixth subsystem: "Runtime Adoption Telemetry").
- **ADDED**: Pre-commit hook entry in `.githooks/pre-commit` invoking `bin/cortex-invocation-report --check-shims` when `bin/cortex-*` files are staged.
- **MODIFIED**: Each canonical `bin/cortex-*` script (9 files) gains one new shim invocation per its script class:
  - **Bash leaf and bash wrappers** (5 files: `cortex-archive-sample-select` is bash, `cortex-create-backlog-item`, `cortex-generate-backlog-index`, `cortex-git-sync-rebase`, `cortex-jcc`, `cortex-update-item`): first line after the shebang and any leading comments — `"$(dirname "$0")/cortex-log-invocation" "$0" "$@" || true`. Absolute path via `dirname "$0"` avoids reliance on `bin/` being on PATH.
  - **Plain Python scripts** (2 files: `cortex-archive-rewrite-paths`, `cortex-count-tokens` if it is plain Python — verify before edit): immediately after the shebang and any leading docstring — `import os, subprocess, sys; subprocess.run([os.path.join(os.path.dirname(os.path.abspath(__file__)), "cortex-log-invocation"), sys.argv[0], *sys.argv[1:]], check=False)`. Absolute path via `__file__`. `subprocess` and `os` explicitly imported on the same statement to avoid NameError.
  - **PEP 723 uv-script Python** (`cortex-audit-doc`, `cortex-count-tokens` if uv-script — verify before edit): the same Python form as above, placed immediately AFTER the `# ///` close marker, before the script's first `import` statement. uv parses the metadata block first; the shim runs after dependency resolve.
  - The exact set of files in each class must be verified by reading the script shebangs at implementation time; the lists above are research-derived best estimates.
- **MODIFIED**: `backlog/103-add-runtime-adoption-telemetry-via-pretooluse-bash-hook-matcher-dr-7.md` — title and body rewritten to reflect the shim mechanism in place of the originally-proposed PreToolUse hook.
- **MODIFIED (auto-mirrored)**: `plugins/cortex-interactive/bin/` gains `cortex-log-invocation` and `cortex-invocation-report` via `just build-plugin` running through the existing `bin/cortex-*` glob distribution path.

## Technical Constraints

- **Log target**: `lifecycle/sessions/<LIFECYCLE_SESSION_ID>/bin-invocations.jsonl`, where the path resolves relative to `git rev-parse --show-toplevel` (the helper's notion of "project root"). `LIFECYCLE_SESSION_ID` is the env var set by the lifecycle SessionStart hook. The path is in `~/.claude/settings.local.json` `sandbox.filesystem.allowWrite` if-and-only-if `cortex init` has been run on the user's machine. The spec depends on this prior registration; fresh-install users who haven't run `cortex init` will produce no telemetry in default permission mode (see Non-Requirements + R13 self-test).
- **Record format**: One JSON object per line, kept ≤ 4KB to preserve POSIX `O_APPEND` atomicity. Schema: `{"ts": <ISO 8601 UTC>, "script": <basename of $0>, "argv_count": <int>, "session_id": <string>}`. Argv values themselves are NOT recorded (privacy minimization — only the count).
- **Helper performance**: < 50ms per invocation (R5). Implementation in pure bash with POSIX utilities. Wrapper scripts pay ~2× process spawn relative to pre-shim baseline; the spec's stance is that this is "negligible additive overhead" for the wrappers' use case (interactive backlog operations and lifecycle commands), not a hot per-Bash-call path.
- **Helper language**: bash. The helper is a leaf utility, mirroring `bin/cortex-git-sync-rebase` (existing bash precedent).
- **Aggregator language**: bash + POSIX utilities by default; `jq` optional for `--json` output. Aggregator must work without `jq` for the default human-readable mode.
- **Inventory source**: aggregator runtime-globs `bin/cortex-*` (relative to `git rev-parse --show-toplevel`). No static inventory file is checked in; the inventory is whatever exists in `bin/` at aggregator-run time. This handles inventory drift but does NOT classify scripts (see Non-Requirements).
- **Plugin distribution**: existing `BIN=(cortex-)` glob in `justfile:481` and `rsync --include='cortex-*'` at `justfile:501` automatically distribute both new scripts. No HOOKS array, pre-commit regex, or `hooks.json` edits required for the helper / aggregator scripts themselves. The pre-commit gate in R14 is added to `.githooks/pre-commit` directly (not via the dual-source mechanism, which is for `bin/` and `hooks/` artifacts, not for the pre-commit script itself).
- **Dual-source enforcement**: pre-commit dual-source drift hook (`.githooks/pre-commit`) covers `bin/cortex-*` per its existing regex. New `bin/cortex-*` scripts (including the new helper and aggregator) are protected automatically.
- **Composition with DR-5 (ticket 102)**: DR-5 catches "SKILL.md doesn't reference the script" (day-one + drift). The shim catches "script ran at least once." R14 (pre-commit `--check-shims`) catches "script committed without shim line." Together these provide layered detection, but none of them detects "agent had occasion to use the script and chose Read+Grep instead" — that requires the human consumer interpreting the `CANDIDATES FOR REVIEW` list against expected use frequency (see Problem Statement and Non-Requirements).
- **Volume estimate**: 9 scripts × ~5 invocations per script per active session × ~10 sessions per day ≈ ~450 records/day under the assumption that all invocation paths successfully write to the log. The actual volume in v1 is biased toward overnight runs (which use `--dangerously-skip-permissions` and bypass `allowWrite` enforcement entirely) and away from daytime default-mode runs without `cortex init` (which silently fail). Reports should be interpreted with this bias in mind until the population is measured via R13's self-test.
