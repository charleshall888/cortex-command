There are **three sites** where `BacklogItem` is constructed from source data — all three must be updated:

1. **`parse_backlog_dir()` (line ~288)**: Reads markdown files with manual regex-based frontmatter parsing. Must add:

2. **`load_from_index()` (line ~331)**: Reads from `backlog/index.json`. Must add:

3. **`backlog/index.json` schema**: The index is generated by a separate tool. When `areas` field is added to backlog YAML, the index regeneration tool must also include it. If the index is stale, `load_from_index()` will silently return empty `areas` lists.

### `group_into_batches()` Algorithm (`claude/overnight/backlog.py:869–966`)

Three-phase algorithm:

**Phase 1 (lines 900–912)**: Extracts up to 2 "quick win" items (type=`bug` or `chore`, constant `_MAX_QUICK_WINS=2`) into a standalone first batch. These items **bypass Phase 2 entirely** — they are not subject to area-separation.

**Phase 2 (lines 915–940)**: Greedy tag-overlap assignment for remaining items. Internal structure: `batches: list[tuple[list[BacklogItem], set[str]]]` where the tuple is `(batch_items, batch_tags)`. For each item:
- Finds the batch with most tag overlap (set intersection count)
- Creates a new batch if no tag overlap exists (`best_overlap > 0` check at line 934)
- **Area-separation would insert a pre-filter here**: before computing tag overlap, skip any candidate batch that already contains an item with overlapping areas

**Phase 3 (lines 942–949)**: Splits oversized batches (> `batch_size_cap`, default 5) via `_split_oversized_batch()`. This function groups items by dominant tag and splits by size. **It does not account for areas** — oversized-batch splitting can recreate area collisions that Phase 2 prevented.

### Area-Separation Pre-Filter (Insertion Point)

Within Phase 2's batch-selection loop (after line 921, before line 923), add:

```python
item_areas = set(item.areas) if item.areas else set()
# Pre-filter: skip any candidate batch that has area overlap with this item
if item_areas:
    batch_areas = {a for b_item in batch_items for a in (b_item.areas or [])}
    if item_areas & batch_areas:
        continue  # Skip this candidate batch
```

The algorithm then falls through to `best_overlap == -1` (no valid batch found), opening a new batch. This makes area-separation a **hard constraint** — area overlap always forces a new batch, even when tag overlap exists.

### `_detect_risks()` (`claude/overnight/plan.py:47–95`)

Current function checks:
- **Lines 61–75**: Shared parent epics **across** batches → warning
- **Lines 77–93**: Overlapping tags **across** batches → warning

The tag-across-batches check is backwards: `group_into_batches()` ensures tag overlap **within** batches, so this check fires on normal output. With area-separation, the appropriate post-grouping check is to validate that no batch contains items with area overlap (confirming the constraint was honored).

**Parent-epic check must be preserved.** The two checks serve different purposes:
- Tag check: backwards/confusing — should be replaced
- Parent-epic check: valid — warns about hierarchical dependencies split across batches

## Web Research

### Conflict-Aware Scheduling Maps to Bin Packing with Conflicts (BPPC)

The "items that must not share a batch" problem is a well-studied variant called **Bin Packing with Conflicts (BPPC)**. Tasks form a conflict graph where edges connect pairs that must not be co-scheduled. Batch assignment becomes a graph coloring problem — each color class (batch) must be an independent set.

Two practical greedy algorithms:
- **DSATUR** (Brélaz 1979): picks the vertex with the highest saturation degree (most distinct colors already assigned to its neighbors); O(n log n + e). Highly-constrained nodes get placed first.
- **RLF** (Leighton 1979): builds one color class at a time to completion; O(n²) for sparse graphs. Fewer colors (batches) in practice.

The proposed pre-filter approach is essentially DSATUR without a priority queue — O(n × b × |areas|), trivially fast for the overnight runner's typical n of 5–20 tasks.

**BPCC variant** (Bin Packing with Compatible Categories) is a close match: items belong to categories (areas), incompatible category pairs cannot co-exist in the same bin. The conflict graph edges are derived from category overlap.

### Declaration-Based vs. Automatic Inference

Nx/Turborepo use **inferred** dependency graphs (explicit imports in package.json/tsconfig). CODEOWNERS maps file path patterns to owner teams for review assignments — declaration-based, but applied per-PR, not across concurrent PRs.

**ML-based conflict prediction** (arxiv:1907.06274) achieved F1 scores of 0.57–0.68 using only VCS history features — no code analysis required. **Critical limitation**: requires historical data. Net-new projects have no history, which is the exact scenario where conflicts are most likely. Declaration-based areas are the practical stand-in when history is sparse.

**Kubernetes anti-affinity** is the closest operational analogy: hard (`requiredDuring...`) and soft (`preferredDuring...`) anti-affinity rules, with `topologyKey` mapping to the area concept.

### Event Log Join: Canonical Slug Pattern

Stripe's "canonical log lines" pattern: use a stable, machine-generated identifier (`request_id`) as the join key across write sites — not display names. Display names are subject to capitalization differences, whitespace inconsistencies, and reformatting.

For append-only NDJSON logs: normalize at read time using a shared deterministic function when retroactive fixes are not possible. Going forward, write the canonical key (`lifecycle_slug`) at every write site.

## Tradeoffs & Alternatives

### Where to Populate the `areas` Field

Three options:
- **(a) `/refine` at spec time** — human-reviewed, populated before overnight planning
- **(b) `/lifecycle` plan phase** — more concrete file paths available, but runs later
- **(c) Manual / no enforcement** — fully opt-in

**Recommendation**: `/refine` (option a). Spec time is the earliest point where scope and affected areas are known. The human can validate the declaration before committing to overnight execution. No need to parse plan markdown for file paths. If unpopulated, the constraint is silently skipped — this is the defined fallback behavior.

### Algorithm Constraint Placement

Three options:
- **(a) Pre-filter (hard constraint)**: skip candidate batch if area overlap exists
- **(b) Post-sort**: group first, then redistribute
- **(c) Penalty scoring**: reduce tag-similarity score when area overlap exists

**Recommendation**: pre-filter (option a). Semantically clearest: area overlap is a hard blocker. Behavior is deterministic and predictable. Net-new projects where all items share areas will produce single-item batches (serialized execution) — this is the intended outcome, not a bug, and should be documented.

### `_detect_risks()` Refactoring

Three options:
- **(a) Full replacement** with within-batch area-overlap validation
- **(b) Supplement** — add area check alongside existing checks
- **(c) Remove entirely**

**Recommendation**: option (a) partial replacement. Replace the tag-across-batches check (which is backwards/confusing) with a within-batch area-overlap check. **Keep the parent-epic check** (lines 61–75) — it serves a different purpose (hierarchical dependency warnings) and should not be lost.

### `_split_oversized_batch()` Interaction

`_split_oversized_batch()` groups by dominant tag and splits by size. It does not currently account for areas. After Phase 2 area-separation, an oversized batch could be split in a way that places area-overlapping items into the same sub-batch.

Options:
- **(a) Update `_split_oversized_batch()`** to also preserve area locality
- **(b) Post-split validation pass** — run area-overlap check after splitting and re-separate if needed
- **(c) Accept the limitation** — document that area separation is best-effort when oversized batches split

Given that Phase 2 already prevents area overlap within batches, oversized batches by definition contain items that cleared the area check. The split might combine items from sub-groups that, when paired, share areas — but this requires the oversized batch to have items with partially overlapping area sets. This is an edge case. **Recommendation: option (c) initially**, with a note that `_split_oversized_batch()` should be updated in a follow-on if area-collision-from-split is observed in practice. Document the limitation explicitly.

## Adversarial Review

### Phase 1 Quick Wins Bypass Area Separation

Items typed `bug` or `chore` (up to 2) are placed in a standalone first batch in Phase 1, before Phase 2's area-separation logic runs. They are not subject to the area-overlap constraint.

**Assessment**: This may be acceptable — quick wins are fast items that don't need deep context. But if two quick-win items modify the same files (area overlap), they will conflict despite the area-separation infrastructure. **Design decision needed at spec time**: should quick wins participate in area-separation, or is the exemption intentional?

### Index Staleness Risk

`backlog/index.json` is a generated secondary cache. When a human adds `areas: [foo, bar]` to a backlog markdown file, `load_from_index()` will return empty `areas` until the index is regenerated. `parse_backlog_dir()` reads markdown directly and would pick up the new field immediately.

**The overnight runner must use `parse_backlog_dir()` or always regenerate the index before loading.** This is an operational constraint that the spec must address.

### Three Parsing Sites, Not One

Agent 4's recommendation ("add `areas` to BacklogItem") described only the dataclass change. The actual implementation requires changes to three sites: the dataclass definition, `parse_backlog_dir()`, and `load_from_index()`. Missing any one site produces silently-empty areas lists.

### Event Join in 015/016 (Background Concern)

The adversarial agent raised a concern that the 015/016 implementations may lack a test verifying that `merge_conflict_classified` event feature names match state dict keys. Both 015 and 016 have `feature_complete` events as of 2026-04-03 — they are considered complete. This concern is noted for reference but is not a blocker for this ticket.
