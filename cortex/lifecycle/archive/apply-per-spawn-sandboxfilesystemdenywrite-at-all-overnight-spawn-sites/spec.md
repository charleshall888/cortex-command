# Specification: apply-per-spawn-sandboxfilesystemdenywrite-at-all-overnight-spawn-sites

## Problem Statement

Session `overnight-2026-04-21-1708` demonstrated that the overnight orchestrator's Bash tool executed `cd $REPO_ROOT && git commit`, landing plan commits on local `main`. This was possible because the orchestrator spawn at `cortex_command/overnight/runner.py:946` has no per-spawn sandbox — worktree isolation at the Bash-tool layer is cosmetic without OS-level kernel enforcement. Compounding this: per-feature dispatch's existing `sandbox.write.allowOnly` shape at `cortex_command/pipeline/dispatch.py:546` is structurally a silent no-op (sandbox-runtime consumes only `sandbox.filesystem.{allowWrite,denyWrite}`); and `cortex_command/overnight/feature_executor.py:603` admits the home repo into cross-repo dispatch allowlists by passing `Path.cwd()` unconditionally as `integration_base_path`. `requirements/project.md:34` (defense-in-depth-for-permissions) names sandbox configuration as the critical security surface for autonomous execution; this work hardens that surface across all three spawn sites and replaces the reverted ticket-128 git-pre-commit-hook approach for the Bash-tool-routed subset of its threat-model coverage.

## Requirements

1. **Orchestrator spawn passes `--settings <tempfile>`**: `cortex_command/overnight/runner.py:946` `_spawn_orchestrator` constructs a per-spawn JSON settings file via `tempfile.mkstemp(prefix="cortex-sandbox-", suffix=".json", dir=<session_dir>/sandbox-settings/)` (mode 0o600), writes JSON via `cortex_command/common.py:atomic_write`, and includes `--settings <tempfile-path>` in the `subprocess.Popen` argv.
   - Acceptance: `grep -c '"--settings"' cortex_command/overnight/runner.py` ≥ 1 inside the `_spawn_orchestrator` function body.
   - Acceptance: `tests/test_runner_sandbox.py::test_orchestrator_spawn_includes_settings_flag` mocks `subprocess.Popen` and asserts the captured argv contains `"--settings"` followed by an existing file path.

2. **Per-spawn JSON shape**: The settings JSON contains exactly:
   - `sandbox.enabled: true`
   - `sandbox.failIfUnavailable: true` (subject to Req 4 kill-switch)
   - `sandbox.allowUnsandboxedCommands: false`
   - `sandbox.enableWeakerNestedSandbox: false`
   - `sandbox.enableWeakerNetworkIsolation: false`
   - `sandbox.filesystem.denyWrite`: array of specific git-state-mutation paths per repo (per Req 3)
   - `sandbox.filesystem.allowWrite`: empty (cortex-init's user-scope allowWrite for `lifecycle/sessions/` merges in via documented multi-scope merge precedence; the per-repo deny-set carves out only the specific git-ref paths, NOT the broader `lifecycle/sessions/` allow)
   - Acceptance: `tests/test_runner_sandbox.py::test_orchestrator_settings_json_shape` invokes the settings-builder with a fixture state and asserts `json.loads(result)` equals an expected dict with exactly these keys + types.

3. **Orchestrator deny-set enumerates specific git-state paths per repo**: For each repo (home repo + each cross-repo via `state.integration_worktrees.keys()`), the deny-set adds four entries:
   - `<repo>/.git/refs/heads/main`
   - `<repo>/.git/refs/heads/master`
   - `<repo>/.git/HEAD`
   - `<repo>/.git/packed-refs`
   This is static per-repo enumeration — no dynamic `git symbolic-ref` resolution is required. The four-path enumeration covers the session-1708 vector (commit-on-main via Bash) for repos using `main` OR `master` as default branch. Custom default branches (`develop`, `trunk`, etc.) are NOT covered by V1; documented as limitation. **Rationale for not denying entire `<repo>`**: under documented `denyWrite > allowWrite` precedence, denying `<repo>` would override cortex-init's user-scope `allowWrite` for `<repo>/lifecycle/sessions/` and block the orchestrator from writing its own `events.log` — the runner would crash on first event-log write. Enumerating specific git-paths avoids this collision while still covering the originating threat vector.
   - Acceptance: `tests/test_runner_sandbox.py::test_denyset_specific_git_paths` populates `state.integration_worktrees` with two cross-repo paths, calls the deny-set builder, asserts every entry matches one of `*/.git/refs/heads/main`, `*/.git/refs/heads/master`, `*/.git/HEAD`, `*/.git/packed-refs` AND no entry is a bare repo root path.

4. **`CORTEX_SANDBOX_SOFT_FAIL` kill-switch**: When the env var `CORTEX_SANDBOX_SOFT_FAIL=1` is set at any settings-builder invocation (orchestrator-spawn time AND each per-dispatch invocation per Req 8 — both call sites read `os.environ` at construction), the per-spawn JSON sets `sandbox.failIfUnavailable: false` instead of `true`. Documented in `docs/overnight-operations.md` as the user-facing recovery path for Anthropic open regressions [#53085](https://github.com/anthropics/claude-code/issues/53085) and [#53683](https://github.com/anthropics/claude-code/issues/53683). Mid-session toggling: if the user changes the env var between dispatches, only NEW spawns reflect the change; the orchestrator's own per-spawn settings JSON (built once at orchestrator-spawn) is NOT re-built for the lifetime of that orchestrator process.
   - Acceptance: `tests/test_runner_sandbox.py::test_soft_fail_killswitch_set` runs the builder with `CORTEX_SANDBOX_SOFT_FAIL=1` and asserts `result["sandbox"]["failIfUnavailable"] is False`.
   - Acceptance: `tests/test_runner_sandbox.py::test_soft_fail_killswitch_unset` runs the builder without the env var and asserts `result["sandbox"]["failIfUnavailable"] is True`.
   - Acceptance: `tests/test_runner_sandbox.py::test_soft_fail_per_dispatch_re_read` calls the per-dispatch builder twice with env var unset → set, asserts the second result has `failIfUnavailable: false` and the first has `true`.

The originally-specced typed-field migration was structurally impossible. The `--settings <tempfile>` mechanism mirrors the orchestrator path (Req 1) so both spawn sites use one canonical mechanism. The per-feature `allowWrite` list contains: the worktree path, plus risk-targeted out-of-worktree writers (Req 10). The dispatch path also sets `TMPDIR=$TMPDIR` explicitly into the dispatched-agent env (passed via SDK `env=` parameter) to prevent the unset-TMPDIR fallback to `/tmp/`.
   - Acceptance: `tests/test_dispatch.py::test_settings_tempfile_used` mocks the SDK call, dispatches a feature, asserts the captured `ClaudeAgentOptions.settings` is a filepath that exists, and `json.loads(open(path).read())` contains `sandbox.filesystem.denyWrite`, `sandbox.filesystem.allowWrite`, `sandbox.enabled: true`, `sandbox.failIfUnavailable`, `sandbox.allowUnsandboxedCommands: false`, `sandbox.enableWeakerNestedSandbox: false`, `sandbox.enableWeakerNetworkIsolation: false`.
   - Acceptance: `tests/test_dispatch.py::test_dispatched_env_locks_tmpdir` mocks the SDK call, asserts the captured `env` dict contains `TMPDIR` with a non-empty value.
   - Acceptance: `tests/test_dispatch.py::test_no_typed_sandbox_field_attempted` asserts `cortex_command/pipeline/dispatch.py` does NOT import `SandboxSettings` or `SandboxFilesystemSettings` from `claude_agent_sdk` (drift detector guards against accidental re-introduction of the broken typed-field path).

9. **Synthetic kernel-EPERM acceptance test (dual-mechanism)**: Two tests in `tests/test_runner_sandbox.py` exercise the same denial assertion against two different enforcement-test mechanisms:
   - **`test_synthetic_kernel_eperm_under_sandbox_exec`** (PRIMARY, blocking): invokes Apple's macOS `sandbox-exec` binary (universally available on Darwin, deprecated since 10.15 but still functional) with a Seatbelt profile equivalent to the cortex-constructed deny-JSON; asserts the child exits non-zero, child stderr contains `"Operation not permitted"`, and the target file does not exist after the run. Validates the Seatbelt kernel layer that Anthropic's sandbox-runtime ultimately invokes on macOS. Must NOT pytest.skip on macOS (skip allowed only on non-macOS platforms which are out of scope per parent epic).
   - **`test_synthetic_kernel_eperm_under_srt`** (SECONDARY, opportunistic): invokes Anthropic's `@anthropic-ai/sandbox-runtime` CLI (`srt`) directly with the cortex-constructed deny-JSON; asserts the same denial. Validates the Anthropic-wrapper layer specifically (catches drift between the wrapper and Seatbelt). Allowed to pytest.skip with informative message (`pytest.skip("sandbox-runtime CLI not installed; sandbox-exec test provides hard coverage")`) when `srt` is not on PATH; the primary `sandbox-exec` test already provides hard kernel-layer coverage.
   - Coverage rationale: `sandbox-exec` catches Seatbelt-layer regressions; `srt` catches Anthropic-wrapper regressions when env permits. Together they bound the test surface across both layers; pre-flight (Req 12) and pin-bump re-run (Req 17) cover the cortex→`--settings`-flag→`claude` path that neither synthetic test exercises.
   - Acceptance (sandbox-exec): `pytest tests/test_runner_sandbox.py::test_synthetic_kernel_eperm_under_sandbox_exec -v` exits 0 (passes, NOT skipped) on the canonical macOS dev environment.
   - Acceptance (srt): `pytest tests/test_runner_sandbox.py::test_synthetic_kernel_eperm_under_srt -v` exits 0 (passes OR skips with the documented "sandbox-runtime CLI not installed" message); does NOT exit with assertion failure.

## Non-Requirements

- Does NOT cover Write-tool calls to home-repo paths including `.git/refs/heads/*` (in-process via SDK's `fs.writeFileSync`, bypasses sandbox per the official docs at https://code.claude.com/docs/en/sandboxing and Anthropic [#26616](https://github.com/anthropics/claude-code/issues/26616); out of scope per parent epic #162 threat-model boundary). **Graduation criterion**: if ≥1 entry in `retros/` cites an Edit-tool-routed or Write-tool-routed `.git/refs/heads/*` rewrite as the failure source, this carve-out is reopened under a successor ticket. ## Edge Cases

- **`CORTEX_SANDBOX_SOFT_FAIL=1` set**: per-spawn JSON sets `failIfUnavailable: false`; sandbox-runtime regression-related crashes downgrade to warning + soft-fail. Two surfacing mechanisms: (a) Req 20's unconditional morning-report header fires at any orchestrator-spawn or per-dispatch where the env var was truthy; (b) the existing stderr-grep for "sandbox" tokens during sandbox-unavailability events surfaces specific runtime errors. (a) catches "user set it and forgot"; (b) catches "sandbox tried to activate and failed."
- **Cross-repo with custom default branch (e.g., `develop`)**: Req 3 enumerates `main` and `master` only. Cross-repos with `develop` or other custom default branches are NOT protected by V1; documented as Non-Requirement. Follow-up triggered by retro signal.
- **Cross-repo dispatched agent loses home-repo cortex project hooks**: pre-Req-6, `_load_project_settings` blob-injected the home-repo's project hooks (skill-edit-advisor, tool-failure-tracker, cortex permissions allowlist) at `--settings` top precedence regardless of dispatch CWD. Post-Req-6, hooks merge naturally via Claude Code's multi-scope merge from project scope — but for cross-repo dispatches under Req 7, CWD is `state.integration_worktrees[repo_key]` (a `git worktree add` directory inside the cross-repo, per `cortex_command/overnight/plan.py:467-482`), so the home cortex repo's `.claude/settings.json` is OUT of project scope. Cross-repo dispatched agents therefore lose all cortex project hooks. **This is intentional**: cortex's project hooks (skill-edit-advisor warns about edits to cortex skills; tool-failure-tracker counts cortex-internal tool failures) are cortex-command-internal observability — they are NOT relevant when the dispatched agent is operating on a different repo's files. The pre-Req-6 force-injection was actually a leak of cortex internals into cross-repo agent contexts; post-Req-6 isolates hooks to the repo they belong to. Documented in `docs/overnight-operations.md`.
- **`state.features` mutation mid-session**: deny-set is recomputed per-dispatch (Req 8); features added during planning phase are correctly included at next dispatch.
- **Tempfile leak on SIGKILL / OOM / kernel panic**: startup-scan in runner-init removes stale `<session_dir>/sandbox-settings/cortex-sandbox-*.json` older than runner-start timestamp (Req 11).
- **Project `settings.json` contains hooks/env/statusLine**: post-Req-6, only sandbox subtree is plumbed through the typed SDK field; other keys merge naturally via Claude Code multi-scope merge from project scope. For HOME-repo dispatches, no regression in hook/env behavior. For CROSS-repo dispatches, see the cross-repo Edge Case above (intentional isolation).
- **`dispatch.py` with no project `settings.json`**: typed SDK field handles this naturally — empty `SandboxSettings` (or omitted `sandbox=`) produces a no-op pass-through; cortex-init's user-scope `allowWrite` is unaffected.
- **`--continue/--resume` flow under `failIfUnavailable=true`**: Anthropic regression #53085 may fire spuriously. cortex's `_spawn_orchestrator` does NOT use `--continue` or `--resume` (verified in research) so the orchestrator path is unaffected. SDK retry-resolve flows MAY use `resume_session` semantics — if regressions fire, user sets `CORTEX_SANDBOX_SOFT_FAIL=1` for the affected session.
- **Pre-existing dispatch flows that wrote to `/tmp/foo` successfully under silent-no-op shape**: post-conversion they hit EPERM unless on the risk-targeted allowWrite list (Req 10). The six documented entries cover enumerated cortex writers (uv cache, $TMPDIR, .claude/sessions, .cache/cortex, .cache/cortex-command, .local/share/overnight-sessions); Req 19 migrates the tool-failure-tracker hook from /tmp/ to $TMPDIR/. Other writers fail loudly during the soak period; surface in morning report; tighten or migrate as observed. The tool-failure-tracker EPERM swallow that previously corrupted morning-report counts is fixed by Req 19.
- **Tempfile path contains `{` or `}`**: SDK heuristic at `claude_agent_sdk/_internal/transport/subprocess_cli.py` distinguishes JSON-string vs filepath via `startswith("{") and endswith("}")`. Cortex's tempfile path is constructed via `mkstemp(prefix="cortex-sandbox-", suffix=".json", dir=<session_dir>/sandbox-settings/)` — none of these path components contain `{` or `}`, so the heuristic correctly routes to filepath form.
- **Two concurrent overnight sessions writing to `<session_dir>/sandbox-settings/`**: each session has its own `<session_dir>` (per `cortex_command/overnight/state.py` session-scoping); no path collision possible.
- **Orchestrator's prompt template includes nested `claude -p` invocations**: verified in research that `cortex_command/overnight/prompts/orchestrator-round.md` contains no nested `claude -p` invocations. Bash subprocesses inherit Seatbelt profile via process-tree inheritance, so any nested command (e.g., `git`, `python`, `pytest`) is automatically constrained at the OS-process layer. No prompt-template edit required for thread-through.
- **Sub-agent settings inheritance at the merge layer**: Task-tool sub-agents and SDK-spawned children that re-invoke `claude -p` would re-enter at the settings-merge layer and inherit nothing from the parent's `--settings` flag. Currently this is not a problem because Req 1 only commits `_spawn_orchestrator` to pass `--settings` and verifications confirmed no nested `claude -p` invocations exist in the orchestrator prompt template. If a future orchestrator template introduces nested `claude -p` invocations, the prompt template MUST also thread through the deny-set (e.g., by exposing the parent's settings-tempfile path and including it in nested-claude argv). This is enforced by Req 17's pre-commit hook that blocks changes to the orchestrator prompt template without a fresh pre-flight.
- **Write-then-Bash-execute vector**: a Claude could use Write to drop a script (`Write /tmp/foo.sh` containing `cd $REPO && git commit`) then invoke Bash (`bash /tmp/foo.sh`). Coverage analysis: the Write step is uncovered (per #26616 carve-out — Write tool bypasses sandbox), but the Bash invocation IS sandboxed (Seatbelt profile inherits process-tree). The executed script's `git commit` attempts to write to `<repo>/.git/refs/heads/main` — Req 3's deny-set covers this path, so the commit fails with EPERM. **This vector is coincidentally covered** because the deny-set targets the FINAL filesystem write, regardless of the call path that initiated it. Documented to remove ambiguity for future spec readers.
- **Linux invocation**: per Req 18, a one-line stderr warning fires; orchestrator continues. Sandbox enforcement under bwrap on Linux may differ from documented Seatbelt behavior. Not a crash; advisory only.

## Technical Constraints

- **macOS-only**: parent epic #162 explicitly drops Linux/WSL2 scope. Active dev environment is Darwin 25.4.0 (Seatbelt). Implementation can assume macOS-Seatbelt without conditional branches; Linux is observable via Req 18 stderr warning.
- **SDK version pinning**: `pyproject.toml:10` pins `claude-agent-sdk>=0.1.46,<0.1.47`. Spec assumes the typed `SandboxSettings`/`SandboxFilesystemSettings` fields exist at this version. Implementation must verify against `claude_agent_sdk/types.py`; if missing, escalate with a wiring ticket rather than working around. Req 15's symbol-import assertion test catches drift on pin bumps.
- **Claude Code CLI version unpinned**: `claude_path = "claude"` (PATH-lookup). Sandbox-runtime version drift is partially covered by the Req 16 precedence-overlap test (which fires if precedence inverts at runtime). Other drift surfaces (e.g., `--settings` flag silently dropped) are uncovered by automated tests; Req 17's pre-flight re-run on dependency-pin updates and source-file changes mitigates by requiring fresh empirical verification at commit time. A `brew upgrade claude` between merge and the next overnight session can still break the mechanism with no detection until the next code change touches the sandbox source files; documented limitation.
- **`--dangerously-skip-permissions` cannot be removed**: `requirements/multi-agent.md:23` mandates `bypassPermissions` for overnight agents. Sandbox is layered UNDER, not in place of, this flag. Verifying the layering invariant at runtime would require Anthropic to ship a layering-invariant test API; outside cortex's scope. The pre-flight gate (Req 12) is the only verification point.
- **Pre-deploy no-active-runner check**: `requirements/multi-agent.md:51` — sandbox-related runner edits must be deployed when no overnight session is active. Implementation PR description must call this out.
- **Atomic writes**: `requirements/pipeline.md:21,126` — tempfile + `os.replace()` pattern. Per-spawn settings tempfile follows this even though it's read-only after write.
- **MUST-escalation policy** (`CLAUDE.md:52-60`): no MUST/CRITICAL/REQUIRED prose escalations added to user-facing rules without evidence-artifact citation. This spec uses positive routing language; `failIfUnavailable: true` is a config flag value, not a prose escalation, so the policy is not implicated.
- **Doc-source convention** (`CLAUDE.md:50`): `docs/overnight-operations.md` owns orchestrator behavior + threat-model boundary; `docs/pipeline.md` owns dispatch internals + sandbox shape; `docs/sdk.md` cross-links rather than duplicates.
- **Dry-run snapshot test**: `requirements/pipeline.md:27` — `cortex overnight start --dry-run` has byte-identical stdout snapshot test. Implementation must verify `tests/fixtures/dry_run_reference.txt` does NOT regress; if sandbox-tempfile-creation log output appears in dry-run, the snapshot must be updated atomically with the source change.
- **`@anthropic-ai/sandbox-runtime` schema** (`research.md` Web Research): only `filesystem.{allowWrite,denyWrite,allowRead,denyRead}` is documented. The granular `write.{allowOnly,denyWithinAllow}` shape is undocumented; cortex's pre-conversion `dispatch.py:546` IS structurally a silent no-op (verified by research; the conversion is the fix).
- **`denyWrite > allowWrite` precedence** (`research.md` Web Research, https://code.claude.com/docs/en/sandboxing): documented verbatim — "denyWrite overrides allowWrite". The orchestrator's deny-set is a small set of specific git-state paths (Req 3); cortex-init's user-scope allowWrite for `<repo>/lifecycle/sessions/` is preserved via multi-scope merge for paths that are NOT in the deny-set. Req 16's synthetic precedence-overlap test detects runtime precedence-flip drift.
