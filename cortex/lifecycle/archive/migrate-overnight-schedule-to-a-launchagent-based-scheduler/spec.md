# Specification: Migrate overnight scheduling and run-now to LaunchAgent

> **Epic reference**: `research/overnight-runner-sandbox-launch/research.md` (epic-level findings on sandbox launch options); `research.md` in this directory (ticket-level decisions DR-1 through DR-10, plus the post-resume re-scoping captured in §Changes to Existing Behavior below).

## Problem Statement

`bin/overnight-schedule` is currently broken — line 44 still calls the retired `overnight-start` binary that epic 113 deleted, so scheduled overnight runs do not work at all today. Even when it worked pre-113, the `caffeinate -i sleep + tmux` mechanism could not survive lid-close (the schedule never fired) and could not survive reboot. Separately, `cortex overnight start` is synchronous in its caller — it blocks the calling process for the runner's full duration (hours), and the MCP `overnight_start_run` tool depends on a 60-second subprocess timeout coincidence rather than a designed async-spawn boundary. This ticket replaces both paths with a unified launchd-based mechanism: scheduling fires at-or-after the requested time with launchd's wake-coalescing semantics, run-now spawns and returns immediately with a confirmed runner-alive handshake, and the runner runs detached under launchd's process management for both. Both paths use launchd specifically (not bare `subprocess.Popen` + `setsid` for run-now) to keep the spawn primitive, launcher script, sandbox treatment, and tmux-retirement story unified across paths — symmetry has operational value here because a single mechanism is simpler to reason about than two. A `Scheduler` abstraction is introduced as a plug-in extensibility point; only `MacOSLaunchAgentBackend` ships, with no second backend committed.

## Requirements

1. **`cortex overnight schedule <target-time>` subcommand exists and bootstraps a LaunchAgent.** Accepts `HH:MM` (24-hour local time) or `YYYY-MM-DDTHH:MM` (ISO 8601). On success, prints the session_id, the LaunchAgent label, and the resolved fire time. Acceptance: `cortex overnight schedule 23:01 --dry-run` prints the would-be label and target ISO time and exits 0; `launchctl print gui/$(id -u)/<label>` returns exit 0 within 1 second of a real schedule call.

2. **`cortex overnight start` is async — caller returns immediately.** Returns within 5 seconds with the session_id and PID written to `runner.pid` in the session directory. Runner runs detached under launchd. Acceptance: `time cortex overnight start --format json` exits in under 5 seconds with stdout containing `"started": true` and a `"session_id"` field; `cortex overnight status` then reports `phase: running` within 30 seconds.

3. **`bin/overnight-schedule` is deleted from the repo, including all parity-check special-cases.** Deletion covers `bin/overnight-schedule` itself, its `justfile` recipe entries (`deploy-bin`, `setup-force`, `check-symlinks`), and the bare-name allowance in `bin/cortex-check-parity` (and its dual-source mirror at `plugins/cortex-interactive/bin/cortex-check-parity`) — the two spots that hard-code `overnight-schedule` as the canonical un-prefixed exception become dead code that must be removed in the same change. Acceptance: `test ! -e bin/overnight-schedule` exits 0; `grep -rn 'overnight-schedule' bin/ skills/ plugins/ docs/` returns no references except in archive paths and historical lifecycle/research artifacts; `bin/cortex-check-parity` (both copies) passes its own self-tests after the special-case removal.

4. **`cortex overnight cancel` handles both active runners and pending scheduled launches.** With no args, lists both and lets the user select; with a session_id, cancels whichever applies to that session. For scheduled launches: `launchctl bootout` + remove plist + remove sidecar entry + clear `scheduled_start` from state file. Acceptance: schedule a fire 5 minutes out, run `cortex overnight cancel <session_id>`, then `launchctl print gui/$(id -u)/<label>` exits 113 (not found).

5. **A `Scheduler` protocol exists in `cortex_command/overnight/scheduler/` with `MacOSLaunchAgentBackend` as the only shipped implementation.** The protocol exposes `schedule()`, `cancel()`, `list_active()`, and `is_supported()`. Backend dispatch selects the macOS backend on `sys.platform == "darwin"`. On other platforms, `is_supported()` returns False and CLI commands exit with `"cortex overnight scheduling requires macOS"`. Acceptance: `python -c "from cortex_command.overnight.scheduler import get_backend; print(type(get_backend()).__name__)"` prints `MacOSLaunchAgentBackend` on macOS; `pytest cortex_command/overnight/tests/test_scheduler_protocol.py` passes (verifies all four protocol methods exist on backend instances).

6. **Plists are written to `$TMPDIR/cortex-overnight-launch/{label}.plist`, never `~/Library/LaunchAgents/`.** Label format: `com.charleshall.cortex-command.overnight-schedule.{session_id}.{epoch_seconds}` (DR-6, never reuse a label). Acceptance: after a schedule call, `ls $TMPDIR/cortex-overnight-launch/com.charleshall.cortex-command.overnight-schedule.*.plist` lists at least one file; `ls ~/Library/LaunchAgents/com.charleshall.cortex-command.*` returns no matches.

7. **Plists are validated with `plistlib` before bootstrap and verified with `launchctl print` after bootstrap.** Wrapper renders plist via Python `plistlib.dumps`, then `plistlib.loads` round-trip as a structural check (DR-10). After `launchctl bootstrap`, run `launchctl print gui/$(id -u)/<label>` and confirm `state = waiting` substring before writing `scheduled_start` to the state file. Acceptance: `pytest cortex_command/overnight/tests/test_plist_validation.py` passes (covers happy round-trip, typo'd key rejection, post-bootstrap `state = waiting` check, and `state = waiting`-absent failure path).

8. **A sidecar index `~/.cache/cortex-command/scheduled-launches.json` tracks active schedules.** Atomic writes (tempfile + `os.replace()`). The schedule subcommand creates `~/.cache/cortex-command/` with `mkdir -p` semantics on first use (handles the first-install case where the parent directory does not yet exist). Schema: list of `{label, session_id, plist_path, scheduled_for_iso, created_at_iso}`. Used by `cortex overnight cancel --list` and `cortex overnight schedule` (collision detection on session_id reuse, GC pass per R19). Acceptance: scheduling adds an entry; canceling removes it; `cortex overnight cancel --list` reads the file and prints rows; on a fresh install with no `~/.cache/cortex-command/` directory, the first `cortex overnight schedule` call succeeds (creates the parent dir) without `FileNotFoundError`.

9. **Launcher script (executed by launchd at fire time) wraps the runner in `/usr/bin/caffeinate -i` and detaches via `setsid nohup` (DR-4, DR-5).** Launcher exits 0 after fork; does NOT call `launchctl bootout` on itself. Removes its own plist file from `$TMPDIR` after fork. Acceptance: when launchd fires the agent, the runner appears in `ps -ef` with PPID 1 (reparented to init), and `launchctl print gui/$(id -u)/<label>` shortly after shows job complete.

10. **Skill and MCP layers reach launchctl via two distinct sandbox paths, not a single per-call flag.** The skill layer (Claude's own Bash tool calls, e.g. `skills/overnight/SKILL.md` Step 7) invokes `cortex overnight schedule` with `dangerouslyDisableSandbox: true` on the Bash tool call — that flag is honored by Claude Code's harness to bypass both the user-local Bash permission system (any personal `Bash(launchctl *)` deny they may have set) and the Bash tool's Seatbelt sandbox. The MCP layer is structurally different: the MCP server (`plugins/cortex-overnight-integration/server.py`) invokes the CLI via stdlib `subprocess.run`, which never traverses Claude Code's Bash permission system at all. The `dangerouslyDisableSandbox` flag does not exist as a parameter on `subprocess.run`; the MCP server's subprocess inherits the MCP server process's own Seatbelt context (which itself is a child of Claude Code), with no per-call gate to bypass. Cortex-command does not ship a launchctl deny in any distributed config (`claude/` directory contains no `settings.json`); any user-local deny in `~/.claude/settings.local.json` only constrains the user's own Bash invocations and is unaffected by either layer. Acceptance: `grep -c 'dangerouslyDisableSandbox' skills/overnight/SKILL.md` returns at least 1 in Step 7; `grep -c 'subprocess\.run' plugins/cortex-overnight-integration/server.py` confirms MCP-layer subprocess invocation (existing pattern); `pytest plugins/cortex-overnight-integration/tests/test_overnight_schedule_run.py` and the corresponding start test pass without any sandbox-flag wiring on the MCP path.

11. **A new MCP tool `overnight_schedule_run` exists in the cortex-overnight-integration plugin.** Input: `target_time: str` (HH:MM or ISO 8601), optional `state_path: str`, `confirm_dangerously_skip_permissions: Literal[True]`. Output: `scheduled: bool`, `session_id: str`, `label: str`, `scheduled_for_iso: str`. Acceptance: `pytest plugins/cortex-overnight-integration/tests/test_overnight_schedule_run.py` passes (covers happy path + collision + invalid target time).

12. **MCP `overnight_start_run` updated for async semantics.** Returns within 5 seconds with `started: true` and `session_id`. Timeout set to 30s (the spawn path includes plist write + bootstrap + `launchctl print` verify + sidecar atomic write + concurrent-runner check; while typical latency is sub-second, the 30s budget preserves headroom for slow-disk and disk-pressure cases without producing spurious MCP failures). Acceptance: `pytest plugins/cortex-overnight-integration/tests/test_overnight_start_run.py` passes with the new timeout; the test fixture includes a slow-disk scenario asserting completion within 30s.

13. **TCC failures surface at fire time via launcher fail-marker, not at schedule time.** A schedule-time pre-flight probe was considered and rejected: the probe runs from the user's Claude Code session (Terminal/Claude Code TCC subject), but at fire time launchd spawns the cortex binary under its own TCC subject (the binary path) — a clean schedule-time probe gives false confidence and does not predict fire-time success. Instead: (a) `docs/overnight-operations.md` documents that Full Disk Access must be granted to the cortex binary (the path printed by `which cortex`), and the user-facing error message at run time names the binary path explicitly; (b) the launcher script (R9) catches EPERM and command-not-found errors during runner spawn, writes a sentinel `<session_dir>/scheduled-fire-failed.json` with `{ts, error_class, error_text, label, session_id}`, and exits non-zero so launchd records the failure in its own logs as well; (c) the morning report scans for these sentinel files and surfaces them loudly. Acceptance: `pytest cortex_command/overnight/tests/test_launcher_fail_marker.py` passes (covers EPERM-on-state-write, command-not-found-on-runner, runner-exit-non-zero-in-first-second); `grep -c 'Full Disk Access' docs/overnight-operations.md` returns at least 1.

14. **Cross-cancel: scheduling while a runner is active warns; running while a schedule is pending prompts via stderr.** `cortex overnight schedule` checks for active runner via `ipc.read_runner_pid` + `ipc.verify_runner_pid`; if found, exits non-zero with `"active runner present (session_id=...); cancel it first"`. `cortex overnight start` checks the sidecar index for pending schedules; if found, exits non-zero with the schedule list and `"cancel pending schedule(s) first or use --force"`. The `--force` flag bypasses the schedule check on start (does not cancel; lets both run; explicit user intent). Acceptance: `pytest cortex_command/overnight/tests/test_cross_cancel.py` passes both directions.

15. **Environment snapshot at schedule time captures the user's interactive PATH plus runner-required vars.** `EnvironmentVariables` in the plist is populated with: `PATH` (verbatim from the schedule-time env), `ANTHROPIC_API_KEY` (if set), `CLAUDE_CODE_OAUTH_TOKEN` (if set), `CORTEX_REPO_ROOT` (if set), `CORTEX_WORKTREE_ROOT` (if set). `HOME`, `USER`, `LOGNAME`, `TMPDIR` are inherited from launchd defaults (correct per research RQ3). Acceptance: `pytest cortex_command/overnight/tests/test_env_snapshot.py` passes.

16. **Feb 29 and other invalid-target-time inputs are loud-rejected.** `cortex overnight schedule 2026-02-29T23:00` exits non-zero with: `"target time invalid: Feb 29 not in 2026"`. Past times rejected with `"target time is in the past"` (existing logic). 7-day ceiling preserved. Acceptance: `pytest cortex_command/overnight/tests/test_target_time_validation.py` passes.

17. **`docs/overnight-operations.md` scheduling section rewritten.** Documents: (a) `cortex overnight schedule <target>` usage, (b) the LaunchAgent mechanism in one paragraph (plist in $TMPDIR, fires via launchd, no tmux), (c) operational caveats: machine must be powered on and not sleeping at fire time (locked is fine), reboot drops pending schedules (re-schedule after reboot), SSH/headless contexts are not supported (requires local logged-in session), (d) cancel/list operations, (e) TCC requirement: Full Disk Access must be granted to the cortex binary (path printed by `which cortex`) — failures surface via fire-time fail markers in the morning report, not at schedule time. Acceptance: `grep -c 'launchd\|LaunchAgent' docs/overnight-operations.md` returns at least 3 in the scheduling section.

18. **Spawn-confirmation handshake — caller waits for runner.pid before returning `started: true`.** The CLI start subcommand (and the launcher invoked at fire time for scheduled runs) writes a per-session "spawn intent" sentinel `<session_dir>/runner.spawn-pending` before invoking `launchctl bootstrap`. The launcher post-fork writes `runner.pid` (existing behavior). The CLI start path polls for `runner.pid` appearance with a 5-second timeout. If `runner.pid` appears within 5s, return `started: true` with the PID. If not, return `started: false` with `error_class: "spawn_timeout"` and clean up the sentinel. For the schedule path, the post-bootstrap `launchctl print` check (R7) confirms registration only — the PID handshake applies at fire time when the launcher runs. Acceptance: `pytest cortex_command/overnight/tests/test_spawn_handshake.py` passes (covers happy path, runner-crash-in-first-second yielding `started: false`, and slow-spawn within timeout). `cortex overnight status` called within the 5-second window after `cortex overnight start` returns `phase: starting` (new phase) instead of returning indeterminate state.

19. **Plist garbage collection at schedule time.** Each invocation of `cortex overnight schedule` performs a GC pass before writing the new plist: enumerate `*.plist` files under `$TMPDIR/cortex-overnight-launch/`; for each, if its label is absent from the sidecar index, OR `launchctl print gui/$(id -u)/<label>` exits 113 (not registered), remove the file. Rationale: macOS does not clear `$TMPDIR` at reboot for per-user `/var/folders/...` paths (correcting an earlier draft assumption); without GC, killed-launcher plists accumulate indefinitely. Acceptance: `pytest cortex_command/overnight/tests/test_plist_gc.py` passes (covers stale-plist removal, in-flight-plist preservation, and graceful handling of corrupt sidecar index).

## Non-Requirements

- **Reboot survival of pending schedules.** $TMPDIR placement is deliberate (DR-3); reboot drops the plist. User re-schedules after reboot.
- **SSH/headless support.** Scheduling assumes a local logged-in macOS session. No detection or fallback to `user/$UID` domain. Document only.
- **Cross-platform implementation.** Only `MacOSLaunchAgentBackend` ships. The `Scheduler` protocol is in place for future Linux/Windows backends but no second backend is built. Non-darwin platforms exit with a clear error.
- **Multiple concurrent schedules.** One pending schedule at a time. Scheduling while one is pending exits non-zero. Future-extensible (sidecar index supports list semantics) but the CLI enforces single-pending today.
- **Migration of `bin/overnight-status` and other overnight bin scripts.** Out of scope; only `bin/overnight-schedule` is deleted.
- **Auto-resume of in-flight runs after Mac sleep.** caffeinate at fire time keeps Mac awake during the run; if Mac sleeps mid-run for other reasons (e.g., low battery), recovery is not in scope.
- **Built-in TCC auto-grant.** Pre-flight detects denial; user must grant manually via System Settings.

## Edge Cases

- **Lid closed at fire time**: launchd queues the fire to next wake. Run starts when the user opens the lid. Document this clearly.
- **Reboot between schedule and fire**: plist in $TMPDIR is gone; schedule does not re-arm; user re-schedules after reboot.
- **Scheduled fire collides with active runner** (e.g., user manually `cortex overnight start`-ed while a schedule was pending and ignored the prompt with `--force`): launchd-spawned runner detects via `_check_concurrent_start` and exits with concurrent-runner error. State file integrity preserved by existing PID lock.
- **Mac asleep across schedule + fire window**: launchd coalesces all missed fires into one event at next wake. Single fire, not multiple.
- **`launchctl bootstrap` fails for a non-obvious reason** (e.g., XML/schema, plist permissions): post-bootstrap `launchctl print` verification catches it; schedule command exits non-zero with the bootstrap exit code and stderr.
- **Sidecar index file corrupted or missing**: schedule subcommand recreates it on next write; cancel-list returns empty (warn but don't crash).
- **Same-second schedule calls** (collision on `{session_id}.{epoch_seconds}` label): epoch is seconds; two schedules within the same second of the same session_id collide. Mitigation: if collision detected on bootstrap, retry with `epoch+1` once. If still collides, exit with `"label collision; retry in 1 second"`.
- **TCC granted but stale binary path**: pre-flight probe detects EPERM regardless of stale grants. User re-grants to the current binary.
- **User runs `cortex overnight schedule` from a path outside the cortex repo**: `_resolve_repo_path()` (existing) handles this; if no repo found, exits with the existing repo-not-found error before any scheduling work.
- **Plist self-delete race**: launcher removes its own plist after fork. If launcher is killed between fork and rm (e.g., user reboots within the ~1ms window), plist persists in $TMPDIR. Note: macOS does NOT clear `$TMPDIR` at reboot for per-user `/var/folders/...` paths — only `/tmp` is subject to boot-time cleanup. Stale plists therefore persist across reboots; the GC pass in R19 handles accumulation by removing plists whose labels are not in the sidecar index or whose launchd registration is no longer present.
- **Launcher fail-marker write itself fails** (e.g., disk full when launcher tries to write `scheduled-fire-failed.json`): launcher exits non-zero so launchd records the failure in its own logs. Morning report has no sentinel to surface, but `launchctl print-disabled` history retains the failure for manual inspection. Documented as a degraded-but-not-silent case.

## Changes to Existing Behavior

- **MODIFIED**: `cortex overnight start` returns within 5 seconds (was: blocks for runner duration). This is a deliberate behavioral contract change — today's blocking-until-completion is the implicit contract for any caller (shell scripts, `&&`-chained commands, CI wrappers); after this change those callers see a fast async return. No `--wait` shim is provided; users who depend on the blocking behavior must migrate to `cortex overnight start && cortex overnight status --wait-until-complete` (a separate ticket if `--wait-until-complete` is needed). Runner detaches under launchd; `started: true` only returns after the spawn-confirmation handshake (R18) verifies the runner wrote `runner.pid`. Caller polls `cortex overnight status` for progress.
- **MODIFIED**: `cortex overnight cancel` handles pending scheduled launches in addition to active runners. Cancel-by-session-id selects the right path automatically.
- **MODIFIED**: MCP `overnight_start_run` becomes a fast async-spawn (timeout reduced to 10s). Comment at server.py:1956-1958 (claiming "runner detached") becomes accurate by design.
- **MODIFIED**: `skills/overnight/SKILL.md` Step 7 schedule branch — command is `cortex overnight schedule <target>` (was `overnight-schedule <target>`), invoked with `dangerouslyDisableSandbox: true`. Confirmation text references launchd label, not tmux session.
- **ADDED**: `cortex overnight schedule` subcommand and `cortex_command/overnight/scheduler/` package (`__init__.py` exposing `Scheduler` protocol + `get_backend()` dispatch, `macos.py` with `MacOSLaunchAgentBackend`, `launcher.sh` shipped via package data).
- **ADDED**: MCP `overnight_schedule_run` tool in `plugins/cortex-overnight-integration/server.py`.
- **ADDED**: Sidecar index file `~/.cache/cortex-command/scheduled-launches.json`.
- **REMOVED**: `bin/overnight-schedule` and its `justfile` recipe entries (`deploy-bin`, `setup-force`, `check-symlinks`).
- **REMOVED**: tmux dependency in the overnight scheduling path (the `excludedCommands` tmux entry was already removed pre-112; this completes the cleanup by retiring tmux-based attach for scheduled runs).

## Technical Constraints

- **macOS-only ships**. `Scheduler.is_supported()` returns False on non-darwin; CLI exits with a clear error. No abstraction is exercised on non-darwin platforms.
- **launchd `gui/$(id -u)` domain required** — assumes a local logged-in Aqua session; SSH/headless contexts unsupported (per non-requirements).
- **`/usr/bin/caffeinate -i`** wraps the runner at fire time (DR-5). Keeps Mac awake during the run; does not rescue pre-fire sleep.
- **Plist label uniqueness**: per-session-id-per-epoch-seconds (DR-6). Label collision triggers epoch+1 retry; second collision exits non-zero.
- **Atomic state writes** for sidecar index — tempfile + `os.replace()` per `requirements/pipeline.md:21`.
- **No new sandbox config additions to anything cortex-command distributes**. Cortex-command does not ship a launchctl deny in any distributed config. Per R10, the skill layer uses `dangerouslyDisableSandbox: true` on Bash tool calls; the MCP layer's launchctl reaches via the MCP server's process Seatbelt context (no per-call flag, no Bash permission gate). Any user-local `Bash(launchctl *)` deny in `~/.claude/settings.local.json` is preserved as the user's own choice and constrains only the user's own Bash invocations — neither layer's behavior depends on it.
- **Plist generation via stdlib `plistlib`** — no PyObjC, no third-party deps. Validation via round-trip `dumps`/`loads`.
- **TCC binary identity** — the launchd-spawned process's TCC subject is the binary path, not the user's Terminal. Pre-flight probe detects denial; user grants Full Disk Access manually.
- **Time precision**: schedule fires at-or-after the requested time (launchd's wake coalescing semantics). Sub-second precision is not guaranteed.
- **Runner concurrent-start guard** (existing `_check_concurrent_start` in `runner.py`) handles the case where a launchd-spawned runner collides with a manually-started one. No change required to the guard.
- **Backwards compatibility**: anyone with `bin/overnight-schedule` in muscle memory or scripts must switch to `cortex overnight schedule`. Migration note in spec; no shim.

## Open Decisions

- **Launcher script language (bash vs. Python).** Defer to plan phase. Bash is simpler for `setsid nohup caffeinate exec`; Python integrates better with the rest of the scheduler code. Decision affects how the launcher is shipped (package data vs. installed script). Cannot be resolved without the implementation context — both are viable; the choice depends on whether the launcher needs to share Python state with the scheduler module.
- **`Scheduler` protocol exact shape.** Defer to plan phase. The four methods (`schedule`, `cancel`, `list_active`, `is_supported`) are agreed; the exact signatures (return types, exception contracts, handle representation) need to be designed during implementation when the macOS backend's actual API surface is concretely available. Pre-designing without an implementation in hand risks a mismatched interface.
