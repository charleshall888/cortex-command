## Web Research

### Anthropic migration guide item 4 — verbatim, unchanged

Source: `https://platform.claude.com/docs/en/about-claude/models/migration-guide`. Item 4 ("Migrating to Claude Opus 4.7" → "Built-in progress updates in agentic traces"):

> "Claude Opus 4.7 provides more regular, higher-quality updates to the user throughout long agentic traces. If you've added scaffolding to force interim status messages ('After every 3 tool calls, summarize progress'), try removing it. If you find that the length or contents of Claude Opus 4.7's user-facing updates are not well-calibrated to your use case, explicitly describe what these updates should look like in the prompt and provide examples."

Text matches what the 2026-04-22 #092 research recorded — the guidance has not changed. The "provide examples" clause is the canonical authority for this ticket's step-3 fallback.

### Corroborating shorter form

`https://platform.claude.com/docs/en/about-claude/models/whats-new-claude-4-7` under "Behavior changes": "More regular progress updates to the user throughout long agentic traces. If you've added scaffolding to force interim status messages, try removing it." This page does NOT include the "if mis-calibrated, provide examples" fallback — the migration guide is the load-bearing source for step 3.

### Anthropic engineering post: durable-artifact guidance is orthogonal

`https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents` (Nov 26, 2025) recommends durable progress artifacts (`claude-progress.txt`, structured JSON status files) consumed by future agent sessions. This is orthogonal to the in-session live narration this ticket addresses — the lifecycle's existing `research.md`/`spec.md`/`plan.md`/`events.log` already serve the durable-artifact role.

### Community signal: zero counter-evidence on narration

HN long-context regression threads (47794961, 47793546) and GitHub issue `claude-code#53459` discuss MRCR retrieval and instruction-following regressions — **none discuss narration cadence**. Independent guides (`claudefa.st`, `keepmyprompts.com`, `mindstudio.ai`) all repeat Anthropic's "remove and re-baseline" recommendation. There is currently no public signal that 4.7 under-narrates or over-narrates in long-running agent traces — the empirical record is empty in both directions.

### Shaped-example pattern: positive-framing rule

No mainstream prior art for the exact `When X, briefly note it; otherwise stay quiet` form. Closest:
- Few-shot guides (PromptHub, IBM, DataCamp) recommend examples that demonstrate when NOT to call a tool — structurally analogous, but use positive+negative pairs rather than a single conditional sentence.
- claudefa.st's 4.7 best-practices guide: "use positive examples of desired voice rather than negative don't-do-this instructions." This argues against the ticket's `... otherwise stay quiet` framing in favor of positive-on-both-branches (e.g., "When X, briefly note it. Otherwise return without comment.").

### Architectural signal (out of scope but flagged)

Claude Code v2.1.98 (April 9, 2026) ships a built-in **Monitor tool** specifically to displace the `sleep N && tail logfile` polling pattern, which Anthropic engineering now formally classifies as an anti-pattern. Per `mindstudio.ai/blog/claude-code-monitor-tool-stop-polling-background-processes`: a 90-second test polled every 10s consumes ~1,800–4,500 tokens of overhead. **This is not what the current ticket asks for**, but if narration calibration repeatedly fails, the upstream fix (replace polling with Monitor) is the correct escalation rather than re-adding scaffolding.

## Tradeoffs & Alternatives

### Approach A (ticket's suggestion): Surgical clause delete + observe + maybe shape

Delete only the post-`and` narration clause at line 119 ("and surface a brief summary of the 5 most recent events to the user"). Keep the `tail -n 5` Bash call. Observe 1–2 daytime dispatch runs. Add a shaped example back at the same site only if 4.7's native narration is mis-calibrated.

- **Pros**: Faithful to Anthropic's published 3-step guidance verbatim. Reversible at every step. Aligns with project's "qualitative regression detection" pattern from #092. Trivial implementation complexity. Cache-friendly (less varying narration text → more stable prompt prefix). The shaped-example fallback uses structural-event triggering, aligning with the project's preferred convention.
- **Cons**: Two-stage rollout requires interactive user observation between stages. Subjective stage-2 judgment with no quantitative threshold. 4.7 long-context regressions mean native narration is weakest precisely in the regime this loop fires. The retained `tail` Bash call may have no consumer once narration is removed (see Adversarial §B).

### Approach B: Delete narration directive AND `tail` Bash call entirely

Remove both. Step (a)'s `kill -0` provides liveness; step (vii)'s `daytime_result_reader` reads `daytime.log`/`events.log` from disk for result surfacing.

- **Pros**: Cheapest per-iteration cost (one fewer Bash call every 120s × 120 iterations). Simplest spec text. No subjective stage-2 judgment. Honest about what 4.7 changes.
- **Cons**: Loses the audit-surface property the ticket's research identified (cached log content in orchestrator transcript on inquiry). Liveness regression scenario: `kill -0` cannot detect a hung process; only event-log progression can. Departs from ticket's stated scope.

### Approach C: Pre-emptive structural-event shaped example (skip observe phase)

Replace line 119 directly with: "When the tail shows a new `task_complete` or `phase_transition` event since the previous poll, briefly note it; otherwise stay quiet." Land in one step.

- **Pros**: Single-stage rollout. Preserves audit surface. Aligns with "structural-event-triggered output floor" convention. Robust to 4.7 long-context regressions.
- **Cons**: Skips Anthropic's "try removing first." If 4.7 self-narrates structural events natively, this is redundant scaffolding. Re-introduces a maintained narration directive without evidence. Re-litigates prior research's deferral.

### Approach D: Move narration directive to `claude/reference/output-floors.md` — UNWORKABLE

`claude/reference/output-floors.md` was deleted on 2026-04-23 (commit `08d1102`). Floor content was distributed into per-skill references. The premise file does not exist; this approach would require resurrecting a recently-deleted central file against a deliberate distribution decision.

### Approach E: Do nothing

Leave line 119 unchanged. `priority: low` makes it eligible for indefinite deferral.

- **Pros**: Zero risk of regression. Free option value if other tickets edit implement.md.
- **Cons**: Does not act on Anthropic's migration guidance for the codebase's one structurally-matching site. Leaves a time-cadence-triggered narration directive in place. Does not change anything (re-affirms prior #092 wontfix posture).

### Recommended approach: A, with adversarial-review caveats applied at Spec time

Approach A is the only approach that executes Anthropic's published 3-step guidance verbatim and avoids re-litigating prior research's deferral. The adversarial review surfaces material refinements that should be settled in Spec — particularly whether the silent-`tail` problem (§B) means Approach A as written should partially adopt Approach B's "delete the entire (b) step" or land with a positive-framing pre-emptive shaped example (a hybrid of A and C). These are Spec-level decisions.

## Adversarial Review

### Failure modes and edge cases

**A. The ticket's "natural pickup" sibling-ticket framing is empirically stale.** The ticket cites "sibling M1 work in #067/#068/#069" as a possible natural-pickup vector — but those tickets have `parent: 66` (epic 66 — output-signal-noise), all `status: complete` since 2026-04-17, and none touched `implement.md`. The actual epic-82 sibling tickets that touched implement.md (#093, #094, #095) are themselves merged. There is no current in-flight work on this file; the alleged opportunism is dead.

**B. The `tail -n 5` Bash call without narration is observable theatre, not a watchdog.** Step (a)'s `kill -0` is the actual liveness check; step (c) is the inter-iteration sleep. With narration removed, step (b)'s `tail` output is read into the orchestrator's context but the orchestrator has no programmatic instruction to do anything with it. There is no comparison-against-prior-tail, no anomaly detection, no event-name discrimination at the prompt level. Approach A as written may be **strictly worse** than Approach B because (b) without narration is pure context contamination at 30+ iterations.

**C. The 3-step workflow has no termination state.** Step 2 ("observe 1–2 daytime dispatch runs") is structurally insufficient sample size — N=2 cannot distinguish "4.7 narrates well" from "this particular feature happened to have narratable progress." There is no decision rule for ambiguous outcomes.

**D. Observation bias is structurally guaranteed.** The user is the same person who applied the change; they know it was applied. There's no way to blind A/B observation in a `git`-driven A→B→maybe-C lifecycle.

**E. The shaped example's event-name pinning is a 4.7 literalism trap.** The ticket's example pins narration to two event names (`task_complete`, `phase_transition`). The codebase emits at minimum `implementation_dispatch`, `dispatch_complete`, `batch_dispatch`, `task_complete`, `phase_transition`, `feature_complete`, `orchestrator_escalate`. 4.7's literalism risk (per epic 82's #100 — verification-mindset literalism under 4.7) means the model is more likely to follow the example exactly and miss other structurally-meaningful events.

**F. Mirror-sync atomicity is enforced but not in this ticket's success criteria.** If the user edits canonical and forgets `just build-plugin`, the pre-commit hook blocks. The interactive lifecycle gives no acceptance criterion that the mirror is in sync — the spec must specify both files.

### Security concerns or anti-patterns

**G. `tail -n 5 events.log` ingesting attacker-controlled JSONL into orchestrator context is a prompt-injection surface.** Once narration is removed, anything appended to `events.log` by a sub-agent (or anything else writing to that path — see #094 in the same epic on worktree atomicity) lands in context with no sanitization. Not new, but the proposed change does not improve it and may make it slightly worse by removing the implicit summarization step.

**H. "Revert via git on loud failure" is not a callable acceptance criterion.** project.md mandates "verifiable by an agent with zero prior context." A future audit agent looking at line 119 a month from now cannot determine whether the change is "still passing" because there is no automated assertion. The `priority: low` label is doing load-bearing work to suppress the verification question.

### Assumptions that may not hold

**I. "Prior research already validated step 3 over pre-emptive shaping" is a misreading.** Prior research listed three options and deferred to Spec without expressing a preference. The current refine cannot rely on a prior preference that wasn't actually expressed.

**J. In-codebase 4.7 evidence points against "trust 4.7 native narration."** Epic 82 contains tickets #100 (verification-mindset literalism under 4.7), #091 (must-tone regression policy), #088 (baseline collection — closed wontfix), #089 (xhigh vs high cost delta). These document 4.7 surprises in this codebase that argue against trusting native narration in long-running daytime traces.

**K. The "user has 3 independent status channels, narration is redundant" framing is wrong-direction for this loop.** Statusline, dashboard, and `bin/overnight-status` are overnight/orchestrator-level. The daytime polling loop §1b is foreground interactive — the user is sitting at the terminal waiting on the orchestrator's main session. The polling-loop narration is the *primary* channel for that flow. Removing it punts the user to `tail -f lifecycle/{feature}/events.log` in another terminal.

### Recommended mitigations (for Spec to consider)

1. **Reframe deliverable as two tickets.** Step 1 (delete the clause) is closeable with a verifiable acceptance criterion. Steps 2–3 (observe + maybe shape) are open-ended human-judgment work that should live as a follow-up tracking item with a defined observation budget (e.g., "after N daytime dispatches OR D days, decide and close"). The current ticket conflates closeable code change with open-ended observation.
2. **Decide upfront what to do with the `tail` Bash call.** Either (a) delete step (b) entirely, (b) keep tail with a behaviorally-positive narration directive ("Read the tail; if it shows a never-before-narrated structural event in this poll loop, briefly note it. Otherwise return without comment."), or (c) replace tail with a structural diff against the previous tail. Choosing (b) up-front avoids the interactive observation step.
3. **Drop the example's event-name pinning.** If a shaped example is added in step 3, abstract over event types ("structurally-significant event (task completion, phase change, dispatch resolution)") to prevent 4.7 from over-literalizing on `task_complete` + `phase_transition`.
4. **Update the ticket's stale "natural pickup" claim** or accept that the priority justification is empirically false — re-justify on standalone grounds at Spec time.
5. **Add an explicit observation budget and decision rule.** Numeric thresholds for "calibrated correctly" rather than qualitative judgment.
6. **Capture an explicit pre-edit baseline.** One pre-edit daytime dispatch transcript provides a comparison anchor against post-edit observation bias.
7. **Treat `events.log` content reaching orchestrator context as untrusted input.** If `tail` is retained, frame the content as data-not-instructions.
