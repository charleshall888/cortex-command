# Specification: Fix runner.pid takeover race in ipc.py:write_runner_pid

## Problem Statement

`cortex_command/overnight/ipc.py:write_runner_pid` has a TOCTOU race in its unlink-and-retry block (lines 192–212): when a stale `runner.pid` claim pre-exists, two concurrent runner starters can each read the stale content, each unlink the file, and each successfully `O_EXCL`-create their own claim — resulting in both believing they own the session lock. A separate but adjacent race exists earlier in the start path: `_check_concurrent_start` (`runner.py:566–605`) calls `clear_runner_pid` before `write_runner_pid`, and that pre-claim sequence is itself unserialized across concurrent starters. The bug surfaces as `tests/test_runner_concurrent_start_race.py::test_two_starters_with_stale_preexisting_lock` flaking ~20% on macOS (currently `@pytest.mark.xfail(strict=False)`). The MCP control plane and the `cortex` pre-install in-flight guard both depend on `verify_runner_pid` reliably distinguishing live from stale claims; silent dual-ownership of `runner.pid` is a load-bearing contract violation.

## Requirements

1. **Acquire-lock helper** — Add `_acquire_takeover_lock(session_dir: Path)` in `cortex_command/overnight/ipc.py` that opens `session_dir / ".runner.pid.takeover.lock"` with `O_RDWR | O_CREAT | 0o600`, then performs a polling `fcntl.flock(fd, LOCK_EX | LOCK_NB)` acquire with a 5-second total budget (sleep 50 ms between retries). On budget exhaustion, raise `ConcurrentRunnerError(existing_session_id="<unknown>", existing_pid=-1)` with an error message that explicitly identifies the timeout case (e.g. `"takeover lock acquire timed out after 5s; another starter holds .runner.pid.takeover.lock"`) so operators can distinguish timeout from a genuine concurrent-runner collision. Return the held fd; the caller is responsible for `fcntl.flock(fd, LOCK_UN)` then `os.close(fd)` in a `finally` block. The 5-second budget applies per call — multiple `_acquire_takeover_lock` invocations across a single runner's lifetime stack additively. Pattern reference: `cortex_command/init/settings_merge.py:_acquire_lock` (sibling-lockfile rationale) and `plugins/cortex-overnight-integration/server.py:_acquire_update_flock` (polling-with-budget shape). Acceptance: `grep -nE 'def _acquire_takeover_lock' cortex_command/overnight/ipc.py` returns exactly one match AND `grep -nE 'fcntl\.flock\(.*LOCK_EX *\| *LOCK_NB' cortex_command/overnight/ipc.py` returns exactly one match AND `grep -nE 'takeover lock acquire timed out' cortex_command/overnight/ipc.py` returns at least one match.

2. **Lock spans `_check_concurrent_start` + `write_runner_pid`** — `_check_concurrent_start` in `cortex_command/overnight/runner.py` acquires the takeover lock before reading `runner.pid`, performs the read-verify-clear-stale sequence inside the lock, calls `write_runner_pid` while still holding the lock, and releases the lock in a `finally` block. `write_runner_pid` accepts an optional `lock_fd: int | None = None` parameter; when `lock_fd is not None`, the function does NOT acquire its own lock (it operates inside the caller's critical section). When `lock_fd is None` (any other call path that may exist now or in the future), `write_runner_pid` acquires its own takeover lock for the duration of its work. Acceptance: `grep -nE '_acquire_takeover_lock' cortex_command/overnight/runner.py` returns at least one match in `_check_concurrent_start` AND `grep -nE 'def write_runner_pid' cortex_command/overnight/ipc.py` shows the signature includes `lock_fd: int \| None = None`.

3. **Cancel acquires the lock; `--force` cancel skips the lock** — `handle_cancel` in `cortex_command/overnight/cli_handler.py` acquires the takeover lock before reading `runner.pid` and holds it for the full read-verify-and-act sequence (verify magic + start_time, decide live-vs-stale, send `killpg(SIGTERM)` to a live runner OR call `clear_runner_pid` for a stale one). The lock is released in a `finally` block before `handle_cancel` returns. The `--force` cancel path explicitly **skips** the lock acquire — `--force` is the operator's escape hatch for the exact failure mode the polling budget exists to handle (a wedged lock-holder), and routing `--force` through the same 5s timeout would collapse the escalation gradient. The `--force` path accepts a brief operator-visible "no active session" race window during a concurrent takeover (which is `<100ms` in normal cases) as the cost of preserving escape-hatch semantics. Acceptance: `grep -nE '_acquire_takeover_lock' cortex_command/overnight/cli_handler.py` returns at least one match in `handle_cancel` AND the implementation contains an explicit conditional like `if not args.force: lock_fd = _acquire_takeover_lock(...)` so the `--force` skip is visible in code (not buried in cleverness).

4. **Retry path under the same lock** — Inside `write_runner_pid`, when the initial `_exclusive_create_runner_pid` raises `FileExistsError`, the read-existing → verify → unlink-stale → re-create sequence (currently lines 186–212) runs under the lock acquired in R2 or R3 (passed-through `lock_fd`) or the lock acquired in `write_runner_pid` itself (if `lock_fd is None`). The verify step under the lock is the load-bearing check that eliminates the takeover-of-live-claim window: any third-party claim that arrived between our first attempt and our lock acquire is detected here. Acceptance: `grep -B2 -A5 -nE '_exclusive_create_runner_pid\(path, payload\)' cortex_command/overnight/ipc.py` shows the retry path is enclosed inside a `try/finally` that holds the lock for the full sequence (no early `return` between `verify_runner_pid` and the second `_exclusive_create_runner_pid`).

5. **Lockfile path and mode** — The lockfile is `{session_dir}/.runner.pid.takeover.lock` with mode `0o600`. The lockfile is never written to (no content), never unlinked by production code, never `durable_fsync`'d (it is kernel coordination state, not file content). Acceptance: `grep -nE '\.runner\.pid\.takeover\.lock' cortex_command/overnight/ipc.py` returns at least one match AND `grep -nE 'write|unlink|fsync|f_fullfsync' cortex_command/overnight/ipc.py | grep -i 'takeover.lock'` returns zero matches.

## Non-Requirements

- **Repo-wide locking-model change**: This ticket does NOT switch all `runner.pid` access to `fcntl.flock`. `_exclusive_create_runner_pid` (single-shot create), the read-only `read_runner_pid`, and `verify_runner_pid` retain their current primitives. The two `clear_runner_pid` call sites that remain unlocked are `cortex_command/overnight/runner.py:540` (`_cleanup` signal-handler shutdown) and `cortex_command/overnight/runner.py:1536` (clean session-exit). Both run intra-process when this process owns the session — there is no concurrent starter to serialize against. The `clear_runner_pid` call at `runner.py:581` (inside `_check_concurrent_start`) IS lock-covered via R2 (the lock is held through `write_runner_pid`), so it does not appear in the unlocked enumeration.
- **Lock acquisition for read-only verifies**: `cortex_command/install_guard.py` reads `runner.pid` directly via `runner_pid_path.read_text()` at `install_guard.py:201` (NOT via `read_runner_pid`); the `verify_runner_pid` call is at `install_guard.py:209`. install_guard does NOT acquire the takeover lock. **Known existing limitation** (pre-existing, not introduced by this fix): during a takeover window, `install_guard.py:201` may transiently see `runner.pid` absent (file briefly gone between unlink and recreate); the `OSError` handler at line 202 yields `pid_data = None`, which routes through the stale-active-session branch at lines 211–217 and `return`s without raising `InstallInFlightError`. This means a concurrent install during a legitimate takeover gets a misleading "stale" warning and proceeds. The fix narrows the takeover window (now serialized) but does not close this install_guard race; closing it requires either lock acquisition in install_guard or bounded read-retry on `None` and is tracked as a separate backlog item.
- **`cli_handler.py:178` pre-flight read**: the JSON-mode collision pre-flight check is unlocked. If it observes `None` during a takeover, execution falls through to `runner_module.run` which calls `_check_concurrent_start` (lock-covered via R2) and produces the correct verdict. Pre-flight is advisory, not authoritative.
- **Cross-platform test harness**: No new platform-specific fixture infrastructure beyond the portable stale-PID fixture in R8. `settings_merge.py` already runs `fcntl.flock` on macOS and Linux without a special harness; the existing test surface plus `pytest-repeat --count=50` is sufficient.
- **NFS / non-local-FS support**: The fix assumes `lifecycle/sessions/` is on a local filesystem. NFS, SMB, and exotic-FS support is explicitly out of scope; `flock` semantics on those filesystems are filesystem-version-dependent and not characterized by this ticket.
- **Schema version bump**: The `runner.pid` payload format is unchanged. `_SCHEMA_VERSION` stays at 1. The lockfile is a sibling artifact, not part of the `runner.pid` IPC contract.
- **Lockfile cleanup helper**: No new code to delete the lockfile. It is created by `O_CREAT` in `_acquire_takeover_lock` and persists for the lifetime of the session directory; under current archival policy session directories accumulate indefinitely (45+ session dirs exist on the maintainer's working tree at the time of writing). This is benign — the file is ~0 bytes and never actively interfered with.
- **CI pytest job**: Adding a `pytest`/`just test` job to `.github/workflows/validate.yml` is out of scope. The recurring detection signal is local-only (developers run `just test` before pushing). A separate backlog item should track adding pytest to CI.

## Edge Cases

- **Lockfile persists across reboots**: After reboot, the kernel's `flock` state is gone but the file `.runner.pid.takeover.lock` still exists as a 0-byte file. The next runner that opens it gets a fresh fd on a fresh inode-with-no-locks and acquires immediately. Benign.
- **Lockfile persists indefinitely**: under current archival policy session directories are not auto-cleaned. The lockfile is a harmless ~0-byte artifact; each new starter `O_CREAT`s it (no-op since file exists) and `flock`s the inode successfully.
- **Two processes racing on `O_CREAT` of the lockfile**: `os.open(path, O_RDWR|O_CREAT, 0o600)` is atomic at the inode level. Both processes end up with FDs on the same inode (the second `open` does not re-create); both then call `flock(LOCK_EX|LOCK_NB)`; one wins, the other polls. Standard mutex serialization.
- **Lockfile open + signal arrival before flock acquire**: `write_runner_pid` runs inside `deferred_signals` (`runner.py:625`), so SIGTERM is stashed during the `os.open` → `flock` window. Both syscalls complete; signals replay after the `with` block exits.
- **Polling-budget timeout (5s) and signal responsiveness**: When the lock-holder is alive but stuck, the polling caller times out after 5s and raises `ConcurrentRunnerError` with the explicit timeout message from R1. SIGTERM during the polling loop is stashed by `deferred_signals` and replayed when the runner exits the `with` block — the operator's `kill -SIGTERM` is honored within ≤5s + `deferred_signals` exit latency. The 50ms sleep cadence is what bounds signal-response latency, not EINTR abort: per PEP 475 (Python 3.5+) `time.sleep` retries to completion through interrupts rather than aborting; the cadence — not interrupt-driven exit — is the signal-response mechanism. Future maintainers must NOT raise the sleep interval under a false belief that signals would shorten it.
- **Cumulative SIGTERM-to-shutdown latency**: each `_acquire_takeover_lock` call independently times out at 5s. If a runner performs multiple `_check_concurrent_start` invocations across its lifetime under sustained contention, worst-case cumulative SIGTERM-to-actual-shutdown latency stacks additively (5s × N acquires). In normal operation a runner acquires the lock once at startup; multiple acquires would only occur during extraordinary recovery loops. If observed cumulative latency becomes a problem, future work can introduce a per-runner-lifetime budget or a single-shot rule.
- **Cancel + non-force takeover overlap**: With R3 in place, non-force cancel acquires the same lock. If a takeover is mid-flight, cancel polls until the lock is free or its own 5s budget expires. The "operator sees 'no active session' for a moment" race window for non-force cancel is closed.
- **Cancel `--force` + takeover overlap**: `--force` cancel skips the lock by design. If a takeover is mid-flight, `--force` may briefly observe `runner.pid` absent (between unlink and recreate) and report "no active session." This is the documented escape-hatch tradeoff: `--force` users are accepting a small race window in exchange for guaranteed bypass. Operator can retry; the race window is `<100ms` in the common case.
- **Cancel timeout error UX**: when non-force cancel exhausts its 5s budget, `ConcurrentRunnerError` carries `existing_session_id="<unknown>"`, `existing_pid=-1`, and the explicit message `"takeover lock acquire timed out after 5s; another starter holds .runner.pid.takeover.lock"` (per R1). Operators retrying a wedged-holder scenario see the same diagnostic each retry; the recommended escalation is `cortex overnight cancel --force` (which skips the lock per R3).
- **Read-without-lock during takeover**: `read_runner_pid` callers without the lock (`cli_handler.py:178` pre-flight) may observe `runner.pid` mid-takeover (file briefly absent between unlink and recreate). The existing `OSError`/JSON-parse fallback in `read_runner_pid` returns `None`; pre-flight handles it as advisory and the authoritative check inside `_check_concurrent_start` resolves correctly. The `install_guard.py` direct `read_text` path has the latent stale-warning behavior described in Non-Requirements; that is a pre-existing gap not closed by this ticket.
- **`_check_concurrent_start` re-entry under lock**: If two starters concurrently invoke `_check_concurrent_start`, both attempt to acquire the lock. Winner runs the read-verify-clear-and-write sequence; loser polls. When loser acquires, runner.pid now has the winner's claim; loser's verify reports live; loser raises `ConcurrentRunnerError(winner_session_id, winner_pid)`. Correct semantics.
- **Glob discipline**: No production code in the current tree globs `session_dir/*.lock` (audited via grep). Note that `cortex_command/overnight/daytime_pipeline.py:152` does `worktree_path.rglob("*.lock")` for SIGKILL recovery, but `worktree_path` operates on per-feature git worktrees (`lifecycle/{feature}/worktree/`), not session_dir (`lifecycle/sessions/{session_id}/`). The two paths do not overlap. R11's documentation update codifies the discipline so future contributors do not redirect a similar glob at session_dir.

## Technical Constraints

- **runner.pid IPC contract preserved**: the JSON payload schema (`schema_version, magic, pid, pgid, start_time, session_id, session_dir, repo_path`) is unchanged; `_SCHEMA_VERSION` stays at 1; `MAX_KNOWN_RUNNER_PID_SCHEMA_VERSION` stays at 1. The MCP control plane (`cortex mcp-server` per `requirements/pipeline.md:153`) reads `runner.pid` directly and is unaffected.
- **`verify_runner_pid` contract preserved**: the magic + schema-version + psutil `create_time` ±2s check (`ipc.py:231–276`) is not modified. The pre-install in-flight guard at `install_guard.py:209` continues to work unchanged.
- **`fcntl.flock` precedent in-tree**: the implementation follows the sibling-lockfile pattern documented in `cortex_command/init/settings_merge.py:_acquire_lock` (lines 8–15 rationale: sibling lockfile inode survives `os.replace()` of the target). The polling-with-budget shape follows `plugins/cortex-overnight-integration/server.py:_acquire_update_flock` (`fcntl.LOCK_EX | fcntl.LOCK_NB` with sleep-and-retry).
- **`deferred_signals` context preserved**: the lock is acquired and released inside the existing `with deferred_signals(coord)` block in `runner.py:625`. PEP 475 (Python 3.5+) makes `time.sleep` retry to completion across signals rather than aborting; this means SIGTERM during a 50ms sleep does not shorten the sleep, but it is stashed by `deferred_signals` and replays when the runner exits the `with` block. The 50ms cadence — not EINTR abort — is the bound on signal-response latency. **Future-maintainer warning**: do not raise the sleep interval under a false belief that signals would interrupt it; the cadence is the signal-response mechanism.
- **macOS + Linux supported**: `fcntl.flock` is BSD `flock(2)` semantics on both platforms — advisory, per-FD, auto-released on FD close or process death. The known macOS-vs-Linux subtleties (NFS, fork-inheritance, `F_OFD_SETLK` Linux-only) do not apply here: the lockfile is on a local FS, the FD is closed in `finally`, and we do not rely on fork-inheritance or OFD locks.
- **No durable_fsync on the lockfile**: kernel coordination state survives only as long as the lock is held in-memory; file content is irrelevant. `durable_fsync` (which is `F_FULLFSYNC` on macOS — expensive full barrier) must NOT be called on the lockfile fd.
- **Atomic-write contract for runner.pid unchanged**: `_atomic_write_json` (tempfile + `durable_fsync` + `os.replace` + `os.chmod`) is still the writer. The lock serializes the *decision* to call `_atomic_write_json`, not the write itself.
- **R6 → R7 → R9 ordering**: R6 (pytest-repeat dep) is a hard precondition. R7 (`--count=50` in `just test`) and R9 (`--count=1000` one-shot) both fail with `unrecognized arguments` until R6 lands. Implementation order: R6 first, R7 immediately after, R9 last.

## Open Decisions

None — all design decisions are resolved by R1–R11 and the answers gathered during the structured interview and the critical review. The polling-budget value (5 s), the `--force` skip-lock policy, the `--count=50` recurring gate, the `lock_fd` parameter API on `write_runner_pid`, the lockfile name (`.runner.pid.takeover.lock`), and the stale-PID fixture approach (spawn-and-kill with NoSuchProcess assertion) are all spec-fixed and not deferred to implementation discretion.
