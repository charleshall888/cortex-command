# Specification: Build cortex CLI skeleton with uv tool install entry point

Epic reference: this skeleton is the foundation for every downstream ticket (115–125) in epic 113. See [`research/overnight-layer-distribution/research.md`](../../research/overnight-layer-distribution/research.md) (DR-1, DR-4, DR-5) for the broader distribution architecture this ticket realizes.

## Problem Statement

Every ticket in epic 113 (115–125) depends on `cortex` being a working Python console entry point — the runner CLI (115), MCP control-plane (116), `setup` subcommand (117), bootstrap installer (118), per-repo scaffolder (119), and the plugin tier (120–122) all need the binary to resolve on PATH. This ticket builds that skeleton: `[build-system]` in `pyproject.toml`, a `cortex` console entry point in a new top-level `cortex_command/` Python package, and `exit 2` stubs for the five subcommands whose real implementations land in later tickets. It also resolves the PyPI `claude` (Anthropic-owned) namespace collision at source by renaming the repo's Python modules from `claude/*` → `cortex_command/*`, keeping the `claude/` directory for non-Python Claude Code config (settings, rules, reference, statusline, bash hooks). After this ticket, `uv tool install -e /path/to/cortex-command` produces a working `cortex --help` on PATH and downstream tickets can proceed.

## Edge Cases

- **Two clones / editable install drift** — user has two repo clones (e.g., main + worktree) and `uv tool install -e .` was run from the wrong one; `cortex --help` silently executes against the wrong source tree. Expected behavior for this ticket: accept the drift; diagnostic `cortex --version` that would surface the mismatch is deferred per user decision.
- **User already has PyPI `claude` installed in a venv that pytest uses** — after the rename, `cortex_command.*` imports resolve cleanly regardless of PyPI `claude` being present in the environment; no collision. Verifiable post-implementation by installing PyPI `claude` in a scratch venv and running `python3 -c "import cortex_command.overnight"`.
- **`uv sync` behavior change** — adding `[build-system]` causes `uv sync` to install the project itself into `.venv` as an **editable install** (`source = { editable = "." }` in `uv.lock`), not as a non-editable build. This is the expected uv behavior for projects with a declared build backend. Because `PYTHONPATH=$REPO_ROOT` and the `.venv` editable install both resolve to the same physical `cortex_command/` source tree at the repo root, module shadowing is harmless — Python's import resolution finds whichever entry comes first in `sys.path`, but both point at the identical files. No mitigation is required; the spec does NOT set `[tool.uv] package = false`. Verification: after `[build-system]` lands, `just overnight-smoke-test` (or equivalent) still passes.
- **Binary name collision** — rare possibility that the user's `$PATH` already contains a `cortex` binary (crypto wallets, NVIDIA tools, other projects). Expected behavior: pre-implementation check with `which cortex` / `brew search cortex` on the development machine before proceeding; if hit, escalate to user before starting the rename.
- **`uv tool update-shell` never run** — first-time install prints a warning and `cortex` is not on PATH; user sees no `cortex` command. Expected behavior: README documents the one-time `uv tool update-shell` requirement; no runtime guard in this ticket.
- **Entry-point reinstall footgun** — any future addition to `[project.scripts]` (say, `cortex-worker` for ticket 115+) requires `uv tool install -e . --force` because uv's editable install live-reloads source but not pyproject entry-points. Expected behavior: document in the README and in the `cortex --help` epilog. No runtime detection.
- **Orchestrator prompts, skill reference docs, and runtime-string references contain `claude.*` strings** — the rename must update text-file references in addition to Python source. Specific confirmed sites from the codebase survey: `justfile:655` (uvicorn colon-syntax), `claude/dashboard/app.py:11` (docstring example), `hooks/cortex-scan-lifecycle.sh:379` (`python3 -m cortex_command.pipeline.metrics`), `claude/settings.json:99` (permission allow rule), `claude/overnight/prompts/orchestrator-round.md` (rendered prompt text; 7 refs), `docs/overnight-operations.md` (7 refs), `skills/overnight/SKILL.md` (12 refs). The implementation must grep exhaustively across `.py`, `.sh`, `.md`, `.json`, `.toml`, `.txt`, `justfile`, and any other extension that might carry dotted-path references.
- **`skills/overnight/SKILL.md:46` paragraph premise is invalidated** beyond mechanical substitution — the paragraph states `claude.overnight.*` modules "are not installed globally," but ticket 114 makes them installable via `uv tool install`. The text needs a semantic rewrite, not just a token-level `claude` → `cortex_command` substitution. Review the paragraph's intent and update appropriately.
- **`unittest.mock.patch("claude.X.Y")` string targets** — 286 occurrences across 28 test files. These are runtime-resolved strings that `unittest.mock` patches at test-execution time; post-rename they must read `patch("cortex_command.X.Y")` or tests fail with `AttributeError` / `ModuleNotFoundError` at `just test` time. Covered by Req 1 AC's dedicated grep check.
- **pytest.ini_options.testpaths are hardcoded to `claude/*/tests/` paths** — the rename must update `[tool.pytest.ini_options].testpaths = ["tests", "cortex_command/dashboard/tests", "cortex_command/pipeline/tests", "cortex_command/overnight/tests"]`.
- **`bin/overnight-*` scripts, `backlog/*.py`, and `hooks/cortex-*.sh` reference `claude.*` modules** — these are outside any `claude/` tree but import / invoke it. Must be updated by the same rename pass.
- **`claude/hooks/cortex-sync-permissions.py`** is a Python hook symlinked to `~/.claude/hooks/` for Claude Code to invoke as a subprocess — it does NOT participate in the `cortex_command.*` import graph (it's invoked directly by Claude Code as a standalone script). Expected behavior: it stays in `claude/hooks/`; its internal imports are standalone (no `from claude.* import`). If it grows imports from `cortex_command.*` in the future, its location will need reconsideration.
- **Backlog items that prescribe future `python3 -m claude.*` invocations** (e.g., `backlog/110-*.md`) must be updated in this pass so that a future agent executing those backlog items does not reintroduce stale module names. Handled by the generic `.md`-include grep in Req 1 AC.
- **In-flight overnight session during the `uv sync` transition** — if an overnight session is running at the moment `pyproject.toml` gains `[build-system]`, the next `uv run` / `uv sync` inside that session will regenerate `uv.lock` and install `cortex_command` editable into the already-activated `.venv`. Already-loaded modules keep their old references; newly-spawned subprocesses see the new layout. Expected behavior: pause overnight sessions before landing this ticket. Not a runtime guard in the spec; an operational note.

## Technical Constraints

- **Build backend**: hatchling `>=1.27` in `[build-system].requires`. No unbounded upper pin. Setuptools is the documented fallback if hatchling surfaces an unexpected issue during implementation; the spec's primary recommendation is hatchling because it's Astral-aligned and handles PEP 621 natively.
- **Wheel declaration**: `[tool.hatch.build.targets.wheel].packages = ["cortex_command"]`. Single-tree wheel (no `claude/*` Python content after the rename). Hatchling's default includes all files under declared packages (verified against pypa/hatch issue #478 and the hatchling source `backend/src/hatchling/builders/config.py` — `default_global_exclude` is `["*.py[cdo]", "/dist"]`; tests/, prompts/, and templates/ ship by default). Do NOT add `[tool.hatch.build.targets.wheel].only-include` or `.exclude` patterns that would strip data files — the wheel must ship `.md` under `prompts/` and templates under `dashboard/templates/`.
- **Do NOT set `[tool.uv] package = false`**. Default `uv sync` behavior installs the project editable into `.venv`; this is harmless because `PYTHONPATH=$REPO_ROOT` and the `.venv` editable install resolve to the same physical source tree. Setting `package = false` would nullify the `[build-system]` work for this ticket's scope.
- **CLI framework**: argparse only. No click, no typer. Matches every existing in-repo CLI (`claude/common.py`, `claude/overnight/batch_runner.py`, all `bin/*` Python tools) and aider's precedent. Revisit in ticket 115 if nested-group ergonomics demand click; migration for 5 flat subcommands is a ~2-hour dispatcher rewrite.
- **Every subparser MUST have `help=` set.** Without `help=`, argparse emits subcommands inline (`{overnight,mcp-server,...}`) rather than per-line, making Req 3's and Req 4's `--help` greps unreliable. `help=` is a hard requirement, not a nice-to-have.
- **Placeholder semantics**: `print("not yet implemented: cortex <subcmd>", file=sys.stderr); sys.exit(2)`. Exit code 2 matches argparse's usage-error convention; stderr keeps pipelines honest. Do NOT `raise NotImplementedError` (traceback is user-hostile) or `exit 0` (lies about success).
- **No subprocess calls inside stubs**: none of the 5 placeholder subcommands may invoke `uv run`, `uv sync`, or any subprocess. Rationale: future subcommands (`cortex setup`, `cortex init`) will invoke `uv`, and PEP 517 build backends are Turing-complete — a skeleton stub accidentally shelling out to `uv sync` inside a stranger's repo is an arbitrary-code-execution vector. Skeleton stubs must `exit 2` without touching subprocess.
- **Sandbox `excludedCommands` untouched**: do not add `uv:*` or `cortex:*` to `claude/settings.json`'s `excludedCommands`. `cortex` may be long-running via later subcommands (115+); `excludedCommands` is reserved for short-lived transactional tools per `claude/rules/sandbox-behaviors.md`.
- **Single-commit / single-PR atomic land**: Requirements 1 through 5 MUST land in a single commit (or single PR merged as one squash commit). Splitting the rename, `[build-system]` addition, and entry-point declaration across commits produces intermediate states where (a) `uv.lock` is out of sync with `pyproject.toml`, (b) `import cortex_command.*` statements reference a non-existent package, (c) `unittest.mock.patch("cortex_command.X")` strings point at modules still on the old path. The `/commit` skill enforces the single-commit discipline for this ticket.
- **`claude/hooks/cortex-sync-permissions.py`** must continue to function as a standalone Claude Code hook after the rename. If it imports from `claude.*`, the rename must update its imports in-place and leave the file location unchanged.
- **Binary name pre-check**: before starting implementation, run `which cortex` and `brew search cortex` on the development machine. If a collision exists on a PATH-earlier entry, pause and escalate to the user before proceeding with the rename. Pre-flight check, not a runtime guard.
- **Atomic `pyproject.toml` edit**: `[build-system]`, `[project.scripts]`, `[tool.hatch.build.targets.wheel]` land in a single `pyproject.toml` edit — not phased across multiple edits during implementation.
- **Pause overnight sessions during the rename commit**: adding `[build-system]` regenerates `uv.lock` and installs `cortex_command` editable into `.venv`; an in-flight overnight session whose `.venv` is already activated will observe the transition in newly-spawned subprocesses. Pause before landing; resume after.
