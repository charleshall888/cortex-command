# Specification: build-shared-synthesizer-for-critical-tier-dual-plan-flow-interactive-overnight

## Problem Statement

Critical-tier features whose plan is generated by `/cortex-interactive:lifecycle plan` §1b (interactive) or `cortex_command/overnight/prompts/orchestrator-round.md` Step 3b (overnight) currently lack a coherent autonomous resolution mechanism. §1b dispatches 2-3 plan-gen sub-agents, presents a comparison table, and asks the operator to pick. Step 3b dispatches a single bare plan-gen sub-agent without consulting criticality, so a critical-tier feature picked up unattended bypasses the dual-plan branch entirely. This ticket builds a shared synthesizer (a prompt fragment dispatched as a Task sub-agent in both surfaces) that takes 2-3 plan-variant artifacts and returns a chosen variant with structured rationale and anti-sway protections, routing low-confidence cases to a defer-to-morning sentinel. It closes the autonomous resolution loop in overnight mode and replaces the operator-pick step in interactive mode with a default-rubber-stamp + override flow, preserving user-pick as a fallback when the synthesizer defers.

## Requirements

1. **Shared synthesizer prompt fragment**: A markdown fragment at `cortex_command/overnight/prompts/plan-synthesizer.md` is created. The fragment is dispatched as the system prompt of a fresh Task sub-agent in both surfaces — interactive (`§1b` skill) and overnight (orchestrator agent's Step 3b). Acceptance criteria: `[ -f cortex_command/overnight/prompts/plan-synthesizer.md ] && grep -q "plan_synthesizer_v1" cortex_command/overnight/prompts/plan-synthesizer.md` exits 0; the fragment contains the verbatim MT-Bench-derived "Avoid any position biases" instruction; the fragment instructs the agent to treat plan-variant content as untrusted data; the fragment specifies a JSON envelope schema (per-criterion scores → verdict → confidence → rationale, in that positional order) for tool-use forcing.

2. **Synthesizer dispatch shape (shared between both surfaces)**: Both interactive §1b and overnight Step 3b dispatch the synthesizer as a fresh Task sub-agent (Opus model, no worktree isolation needed since the synthesizer is read-only). The sub-agent's system prompt is the shared fragment from Requirement 1; its user prompt inlines the variant file paths plus the MT-Bench-style swap-instruction directing the agent to perform position-swap-and-require-agreement internally and emit a JSON envelope via tool-use forcing. The dispatching context (skill, in interactive; orchestrator agent, in overnight) parses the JSON envelope using the LAST-occurrence anchor pattern from `/critical-review` SKILL.md:176-182 and branches on the envelope's `verdict` and `confidence` fields. Acceptance criteria: `grep -q "Task sub-agent" cortex_command/overnight/prompts/orchestrator-round.md` exits 0; `grep -q "Task tool" plugins/cortex-interactive/skills/lifecycle/references/plan.md` exits 0 (both surfaces dispatch via Task); `grep -q "<!--findings-json-->" cortex_command/overnight/prompts/plan-synthesizer.md` exits 0 (envelope delimiter present in prompt fragment for downstream extraction); `grep -q 'LAST.occurrence\|last.occurrence' plugins/cortex-interactive/skills/lifecycle/references/plan.md cortex_command/overnight/prompts/orchestrator-round.md | wc -l` ≥ 1 (envelope-extraction pattern referenced in at least one dispatching context).

3. **Interactive §1b wiring**: `plugins/cortex-interactive/skills/lifecycle/references/plan.md` §1b lines 21-119 are revised so that after variant collection (current line 94), the skill dispatches one fresh Task sub-agent (Opus, no worktree isolation) using the shared prompt fragment. The sub-agent emits the JSON envelope; the skill parses it. On `verdict ∈ {"A","B","C"}` AND `confidence ∈ {"high","medium"}`: the skill presents the chosen variant with rationale; default operator action is rubber-stamp (Enter to accept), override requires the operator to type a different variant label. On `confidence: "low"` OR malformed envelope: the skill displays the legacy comparison table for manual user-pick (preserving §1b's pre-synthesizer behavior); the synthesizer's preliminary rationale is hidden from the comparison table to preserve fresh-judgment freshness. Acceptance criteria: `grep -c "synthesizer" plugins/cortex-interactive/skills/lifecycle/references/plan.md` ≥ 5; `grep -q "rubber-stamp" plugins/cortex-interactive/skills/lifecycle/references/plan.md` exits 0; `grep -q "synthesizer's preliminary rationale is hidden" plugins/cortex-interactive/skills/lifecycle/references/plan.md` exits 0.

4. **Overnight Step 3b wiring (criticality branch in orchestrator agent's turn)**: `cortex_command/overnight/prompts/orchestrator-round.md` Step 3b lines 201-273 are revised to add a criticality branch within the orchestrator agent's existing single-turn flow. When the eligible feature's most-recent `lifecycle_start` or `criticality_override` event in `lifecycle/{feature}/events.log` carries `criticality: "critical"` AND the runtime gate (Requirement 7) is enabled, the orchestrator agent dispatches 2-3 plan-gen sub-agents in parallel (matching §1b's variant count and using the same Sonnet plan-agent prompt at `plan.md:30-92`) writing to `lifecycle/{feature}/plan-variant-A.md`, `plan-variant-B.md`, optionally `plan-variant-C.md`. After all variants return, the orchestrator agent dispatches one fresh synthesizer Task sub-agent (Opus, shared prompt fragment per Requirements 1 and 2). On the synthesizer's `verdict ∈ {"A","B","C"}` AND `confidence ∈ {"high","medium"}`: the orchestrator copies the selected variant's content to `lifecycle/{feature}/plan.md` and appends a v2 `plan_comparison` event to `lifecycle/{feature}/events.log`; the round continues to Step 3c/3d/3e/Step 4 normally. On `confidence: "low"` OR malformed envelope: the orchestrator marks the feature as `deferred` in its envelope output (worker-style exit-report wrapper pattern), and the round-loop in `runner.py` writes the `lifecycle/deferred/{feature}-q{NNN}.md` deferral file via the existing worker-exit-report deferral channel (`cortex_command/overnight/deferral.py:write_deferral`). Non-critical-tier features continue using the existing single-agent path at orchestrator-round.md:236-260. Acceptance criteria: `grep -A 3 "criticality.*critical" cortex_command/overnight/prompts/orchestrator-round.md | grep -q "plan-variant"` exits 0; `grep -q "synthesizer Task sub-agent" cortex_command/overnight/prompts/orchestrator-round.md` exits 0; `python3 -m pytest cortex_command/overnight/tests/test_orchestrator_round.py -k synthesizer` exits 0 (test exercises the dispatch flow with mocked Task sub-agent responses).

Adjacent readers (`cortex_command/dashboard/data.py:323-347 + 459-464`, `claude/statusline.sh:321-322 + 392`, `bin/cortex-archive-sample-select:108-266`, `hooks/cortex-scan-lifecycle.sh:110-117`) use either grep-by-event-name or last-event tail and are field-additive-tolerant; the spec does not require a renderer for v2 fields in this ticket (rationale is preserved in events.log for audit; UI rendering is out of scope, see Non-Requirements).

6. **Unit-test coverage with three calibration probes**: `cortex_command/pipeline/tests/test_plan_synthesizer.py` (or, if the synthesizer ships purely as a prompt fragment with no Python helper, `tests/test_plan_synthesizer_prompt.py` validating the prompt fragment's instructions) covers: (a) **Identical-variants tie test** — present the same plan content as both variants; the synthesizer's response (canned via mocked Task sub-agent SDK call) must include `confidence: "low"` or `verdict: "C"` (tie). (b) **Position-swap consistency** — present a structurally simple variant pair where order-1 SDK response is canned to pick A and order-2 response is canned to pick B; verify the synthesizer's prompt instructs swap-and-require-agreement (test asserts the prompt contains the verbatim "Run the comparison twice with variant order swapped" instruction or equivalent). (c) **Planted-flaw probe** — take a real plan from `lifecycle/install-pre-commit-hook-rejecting-main-commits-during-overnight-sessions/plan.md` (verified to exist on disk), inject a structural defect (a Verification field referencing a file not in any task's Files list) into one variant, present both variants with canned SDK responses scoring the flawed variant lower, verify the synthesizer's envelope-extraction picks the non-flawed variant. Acceptance criteria: `python3 -m pytest cortex_command/pipeline/tests/test_plan_synthesizer.py -v` reports 3+ test cases passing; mocks the Task sub-agent SDK call per the `cortex_command/pipeline/tests/test_repair_agent.py` mocking pattern.

7. **Validation gate (runtime-enforced via lifecycle.config.md flag)**: Item 4 (overnight Step 3b wiring) ships disabled by default. The orchestrator agent's criticality branch consults a runtime gate: a field in `lifecycle.config.md` named `synthesizer_overnight_enabled` with `bool` value (default `false` if absent or set to false). When the flag is `false`, the orchestrator skips the criticality branch and uses the existing single-agent path at orchestrator-round.md:236-260 (preserves status-quo behavior pre-validation). When the flag is `true`, the orchestrator runs the synthesizer-variant path. The flag is flipped to `true` only after the operator has documented two validation events in `lifecycle/archive/build-shared-synthesizer-for-critical-tier-dual-plan-flow-interactive-overnight/events.log`: (a) **Interactive end-to-end**: the synthesizer was exercised against ≥1 real critical-tier `/cortex-interactive:lifecycle plan` dispatch — synthesizer dispatched, JSON envelope parsed, decision rendered to operator (rubber-stamp, override, or fallback to user-pick). Logged as `synthesizer_validated_interactive`. (b) **Overnight smoke test**: `cortex overnight start --dry-run` exercises Step 3b's critical-tier branch with a fixture or a pinned test backlog item; the dry-run mode echoes the synthesizer dispatch, the envelope parse, and the deferral-or-select branch without committing real changes. Logged as `synthesizer_validated_overnight`. After both events are present, the flag is flipped in a follow-up commit. Acceptance criteria: `grep -q "synthesizer_overnight_enabled" cortex_command/overnight/prompts/orchestrator-round.md` exits 0 (orchestrator agent's prompt consults the flag); `grep -q "synthesizer_overnight_enabled" cortex_command/overnight/cli_handler.py` exits 0 OR equivalent runtime-read site (the gate-reading code path); both validation events present in events.log before the flag-flip commit lands.

8. **Defer-count session circuit breaker (halt-session semantics matching multi-agent.md:46)**: When the synthesizer's `synthesizer_deferred` event count reaches 3 in a single overnight session, the round-loop in `runner.py` halts the session — subsequent critical-tier features are marked `paused` and not re-dispatched, matching the existing 3-pause batch circuit breaker behavior at `cortex_command/overnight/orchestrator.py:241,249-253,269-273` (per `requirements/multi-agent.md:46`: *"preventing further dispatches in the session"*). The morning report includes a `synthesizer_circuit_breaker_fired` summary. Acceptance criteria: `python3 -m pytest cortex_command/overnight/tests/test_synthesizer_circuit_breaker.py` exits 0; the test exercises the 3-defer threshold and verifies subsequent features are marked `paused` rather than dispatched.

9. **Anti-sway protections enforced at the prompt-fragment layer**: The shared prompt fragment at `cortex_command/overnight/prompts/plan-synthesizer.md` includes verbatim instructions for: (a) blinded labels (`Variant 1` / `Variant 2`, never `Plan A by Sonnet-Generator-1` or generator IDs); (b) fresh-agent role separation (the fragment is loaded only by sub-agents that did not produce any variant); (c) per-criterion scoring before prose rationale (JSON envelope positional order); (d) explicit instruction to ignore variant order and length (MT-Bench formula); (e) "When uncertain, assign low confidence" reminder; (f) untrusted-variant-data instruction analogous to `/cortex-interactive:research`'s injection-resistance instruction; (g) verbatim MT-Bench swap-and-require-agreement instruction. Fresh-agent role separation is enforced at the dispatch site by both surfaces always dispatching a NEW Task sub-agent (no context reuse from variant generators); this is a property of the dispatch call site, verified by code-review rather than by automated test. Acceptance criteria: `for instruction in "Variant 1" "did not produce any variant" "per-criterion" "ignore.*order" "uncertain.*low confidence" "untrusted" "swap"; do grep -q "$instruction" cortex_command/overnight/prompts/plan-synthesizer.md || exit 1; done` exits 0.

## Edge Cases

- **1 of N plan-gen sub-agents fails to produce a variant**: Synthesizer is dispatched over the surviving variants if ≥2 produce successfully. If only 1 variant is produced, the synthesizer is skipped and the single variant is accepted (matches `plan.md:94` "if 1 succeeds use it sole" precedent).
- **All N plan-gen sub-agents fail**: Falls back to single-plan flow as today (`plan.md:94` "if all fail fall back to single-plan"); synthesizer is not invoked.
- **Synthesizer Task sub-agent fails / agent crashes / SDK error**: Treated as `confidence: "low"` for safety; overnight orchestrator marks feature `deferred` in envelope (runner.py emits deferral file via worker-exit-report channel), interactive falls back to user-pick. Crashes are logged in `events.log` as `synthesizer_error` for post-session review.
- **Synthesizer returns malformed JSON envelope (extraction fails)**: Dispatching context applies LAST-occurrence anchor extraction from `/critical-review` SKILL.md:176-182. On extraction failure, treats as `confidence: "low"` and falls back same as above.
- **Adversarial plan variant containing prompt-injection content**: Mitigated by the untrusted-variant-data instruction in the shared prompt fragment (Requirement 9). Documented attack surface; not an implementation gap.
- **Defer-count circuit breaker fires mid-session**: Subsequent critical-tier features marked `paused`; session halts dispatching critical-tier features for the remainder of the session. The morning report includes the `synthesizer_circuit_breaker_fired` summary so the operator can investigate.
- **Critical-tier feature in events.log has `criticality_override` event**: The most-recent `criticality_override` value takes precedence over the original `lifecycle_start.criticality`, matching `plan.md:14-19` precedent.
- **Operator overrides synthesizer's pick by typing a different label**: The override action is logged in `events.log` as part of the `plan_comparison` event with `disposition: "override"`. The synthesizer's recommendation field is preserved (`selection_rationale`); the operator's choice is stored separately (`operator_choice`).
- **Operator rubber-stamps synthesizer's pick (default action)**: `disposition: "rubber_stamp"` recorded in the event.
- **Synthesizer auto-selects in overnight (no operator)**: `disposition: "auto_select"` recorded; `operator_choice: null`.
- **`synthesizer_overnight_enabled` flag is `false`**: Overnight Step 3b's criticality branch is skipped entirely; critical-tier features use existing single-agent path. No synthesizer dispatch; no `plan_comparison` event v2 emitted; pre-validation status-quo behavior preserved.
- **`lifecycle.config.md` is absent**: Treated as flag = `false` (gate fail-closed). Orchestrator falls back to existing single-agent path.

## Technical Constraints

- **`requirements/multi-agent.md:74` reading**: Resolved per O1 critical-review disposition: permissive reading. The orchestrator (in both senses — the overnight orchestrator agent and the interactive `/cortex-interactive:lifecycle` skill context) is the role authorized to dispatch sub-agents per the rule's plain text *"Parallelism decisions are made by the overnight orchestrator"*. The synthesizer Task sub-agent is a parallelism decision (selecting among parallel-dispatched variants) the orchestrator role makes; this is permissible. The strict-reading-violation concern is resolved as documentation: this spec sets the precedent for the permissive interpretation going forward.
- **Atomic event-log writes**: `plan_comparison`, `synthesizer_*` events use JSONL append semantics. Deferral file writes use the existing `cortex_command/overnight/deferral.py:write_deferral` (tempfile + `os.replace`).
- **Anti-sway role separation**: The synthesizer Task sub-agent is dispatched fresh — neither reuses any plan-gen sub-agent's context, nor shares context with the orchestrator agent or the interactive skill agent. Enforced at the dispatch site (each surface always issues a NEW Task sub-agent).
- **Synthesizer model**: Opus default for both surfaces (per research §Tradeoffs recommendation; bias-resistance reasoning). No Sonnet first-pass; no escalation ladder. Cost: 1 Opus Task sub-agent per dispatch (the sub-agent internally performs swap-and-require-agreement with 2 SDK self-calls) ≈ $10-15 typical, well within `requirements/multi-agent.md:57` $50 Opus budget cap. Plan-gen sub-agents continue to use Sonnet per existing `plan.md:28` precedent.
- **Worktree isolation**: Synthesizer is read-only; no worktree needed.
- **Schema versioning**: Sets the precedent for events.log `schema_version` field, following the convention at `cortex_command/overnight/ipc.py` (runner-pid contract). Future event-schema migrations follow the same pattern.
- **Deferral source channel**: Synthesizer-emitted deferrals flow through the existing worker-style exit-report channel — the orchestrator agent's envelope marks the feature as `deferred`, and `runner.py` writes the deferral file via the existing `write_deferral` path. Avoids `requirements/pipeline.md` amendment.
- **Plan-gen variant prompt**: Step 3b critical-tier branch uses the same plan-agent prompt verbatim as `plan.md:30-92` (§1b interactive). No new variant prompt introduced; reuse via `importlib.resources` or direct file read from the orchestrator template.
- **Bundling rationale**: Per `requirements/project.md:19` ("Complexity must earn its place"), bundling synthesizer fragment + interactive wiring + overnight wiring + schema extension + tests in one ticket is justified because the synthesizer is a Task-sub-agent dispatched identically from both surfaces — splitting interactive/overnight wiring would partition dispatch details inside one feature. The validation gate (Requirement 7) is preserved as the activation mechanism via the runtime flag, with the flag-flip happening in a follow-up commit after both validation events are recorded.
- **Acceptance criteria verification grounding**: All `pytest` and `grep` acceptance commands have been verified to reference files that either exist today or are explicitly added by this ticket's "Changes to Existing Behavior" section.
