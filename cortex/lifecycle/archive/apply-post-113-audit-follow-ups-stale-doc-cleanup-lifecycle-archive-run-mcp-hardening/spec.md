13. **N6.3** — The recipe rewrites lifecycle path references in (a) files inside the moved directory and (b) any remaining `*.md` files under the repo root that cite the moved dir, **excluding `.git/`, `lifecycle/archive/`, `lifecycle/sessions/`, and `retros/`** (retros are immutable historical record). Rewrite covers three citation forms:

### S1 — MCP discovery-cache stale-on-CLI-upgrade

16. **S1.1** — A new exception class `CortexCliMissing(OSError)` is defined in `plugins/cortex-overnight-integration/server.py`. Subclassing `OSError` is load-bearing: existing call-site handlers that catch `OSError` (e.g., `_maybe_check_upstream` at line 466, `_maybe_run_upgrade` at line 974, and similar) continue to swallow it as they swallow other OSError today, preserving the no-op-on-FNF behavior the call sites currently exhibit. New CortexCliMissing-specific handling is opt-in at sites that need cache-clear-and-retry.
    - **Acceptance**: `grep -n 'class CortexCliMissing(OSError)' plugins/cortex-overnight-integration/server.py | wc -l` = 1; tests assert `isinstance(CortexCliMissing(...), OSError)` is True.
17. **S1.2** — Both `_get_cortex_root_payload()` AND `_run_cortex(argv, ...)` (the per-verb dispatch function used by every tool body) wrap their `subprocess.run([cortex_argv...], ...)` calls in a `try/except FileNotFoundError` handler that re-raises as `CortexCliMissing`. This closes the gap where a primed cache + mid-session CLI removal leaves the verb-dispatch path raising bare FNF.
    - **Acceptance**: `grep -n -B2 -A8 'subprocess.run' plugins/cortex-overnight-integration/server.py | grep -c 'CortexCliMissing'` ≥ 2 (one per wrapped subprocess.run for cortex). Tests cover: (a) FNF in `_get_cortex_root_payload` raises CortexCliMissing; (b) FNF in `_run_cortex` raises CortexCliMissing; (c) other OSError subtypes (e.g., PermissionError) do NOT convert.
18. **S1.3** — At verb-dispatch call sites, on `CortexCliMissing`: clear `_CORTEX_ROOT_CACHE`, attempt one retry of the original operation (attempt counter scoped per-tool-call via a local variable in the dispatcher; concurrent dispatch acceptable because retries are idempotent and `_CORTEX_ROOT_CACHE` populates write-once-after-clear). On second `CortexCliMissing`, the verb-dispatch raises the same `CortexCliMissing` exception unhandled — and the MCP framework's exception handler converts it to a JSON-RPC error envelope per S5.2.
    - **Acceptance**: Tests cover: (a) cache-then-FNF-then-recovery succeeds with retry; (b) cache-then-FNF-then-FNF lets the exception propagate to the MCP framework (test asserts the MCP error envelope content). Test (c) asserts that two concurrent tool calls hitting the same FNF window converge to a single re-discovery without doubling the retry budget (acceptance for the asyncio-concurrency edge case).
19. **S1.4** — `_maybe_check_upstream` and `_maybe_run_upgrade` continue to suppress `OSError` (and therefore `CortexCliMissing` as its subclass) at their existing call-site handlers; their behavior is preserved by S1.1's `OSError` parentage. They do NOT add cache-clear-and-retry logic — they retain "swallow and skip" semantics.
    - **Acceptance**: Test asserts `_maybe_check_upstream` returns `None` (or its current sentinel) when CortexCliMissing is raised, matching pre-S1 behavior.

### S5 — MCP graceful-degrade for missing CLI

20. **S5.1** — At server import time (top-level module load in `plugins/cortex-overnight-integration/server.py`), run `shutil.which("cortex")`. If the result is `None`, log a single structured warning to `stderr` containing the install pointer URL. The server still completes initialization and `tools/list` normally; missing-CLI does NOT fail server startup.
    - **Acceptance**: `python -c "import sys; sys.path.insert(0, 'plugins/cortex-overnight-integration'); import server" 2>&1 | grep -F 'cortex CLI not found'` either zero matches (CLI is on PATH) or one match (CLI missing). Test stubs `shutil.which` to assert both branches.
21. **S5.2** — When `CortexCliMissing` propagates out of a tool handler (after S1.3's retry exhausts, or on the very first call if S1.2 wrapping catches it), the tool handler does NOT return a custom-shaped success payload. Instead, it lets the `CortexCliMissing` exception propagate to the MCP framework's exception-to-JSON-RPC-error converter, OR explicitly raises `mcp.types.McpError` (or the equivalent SDK-provided typed error) with `code = -32000` (server error) and `data = {"reason": "cortex_cli_missing", "install_url": "https://github.com/charleshall888/cortex-command#install", "message": "cortex CLI not found on PATH. Install: see https://github.com/charleshall888/cortex-command#install (or `uv tool install -e .` from a local checkout for dev mode)."}`. The result is a JSON-RPC `response.error` envelope visible to the MCP client (Claude Code), NOT a `result` payload that would clash with each tool's declared Pydantic output schema.
    - **Acceptance**: `grep -E '(McpError|mcp\.types\.McpError|raise CortexCliMissing)' plugins/cortex-overnight-integration/server.py | wc -l` ≥ 1; tests stub the MCP framework's error path and assert the JSON-RPC error envelope shape (`code`, `data.reason`, `data.install_url`, `data.message`). Install URL is the anchor-only README form (no versioned doc path that could rot).
22. **S5.3** — A single helper function (e.g., `_raise_cortex_cli_missing()` or a class-level `CortexCliMissing` constructor that already carries the install_url payload) is used at all sites that need to surface this error. Both the S1.2 verb-dispatch retry-exhaust path AND any other site needing to surface the same error use this helper, ensuring identical envelope contents across paths.
    - **Acceptance**: Code review during plan confirms one helper or class constructor used by both call sites; unit test asserts byte-equal envelope contents from both paths.

## Edge Cases

- **N3.1 + dual-source rebuild**: User edits canonical `skills/lifecycle/SKILL.md`. Pre-commit hook regenerates `plugins/cortex-interactive/skills/lifecycle/SKILL.md` and verifies drift. Behavior: edit is accepted, mirror is regenerated automatically, drift check passes. The lifecycle gate's advisory text fires once on the user-initiated edit; auto-regenerated mirror is not a user edit and does not re-fire the gate.
- **N6.3 substring-collision in slugs**: Slugs like `add-foo` vs `add-foo-bar`. The slash-form pattern uses trailing `/` as anchor; the word-boundary form uses `\b` regex anchor; both prevent prefix-substring collisions. Plan must implement the regex with explicit anchors and include a unit test for this collision case.
- **N6.3 nested slug references**: A reference like `lifecycle/add-foo/research.md` where `add-foo` is itself the archived slug. The slash-form pattern `lifecycle/add-foo/` correctly rewrites to `lifecycle/archive/add-foo/research.md`. No special handling needed.
- **N6.3 wikilink with trailing-slash and without**: Wikilinks may appear as `[[lifecycle/foo]]`, `[[lifecycle/foo/]]`, `[[lifecycle/foo|display]]`, or `[[lifecycle/foo/index]]`. The wikilink-form rewrite anchors on `[[lifecycle/{slug}` (after the opening `[[`) so all four variants are caught.
- **N6.3 multiple slugs cited in one file**: Each slug-pattern is applied independently; sed handles multiple substitutions in one file via repeated `-e` expressions or a single combined alternation. Plan choice.
- **N6 active worktree skip**: Recipe's existing `git worktree list` skip continues. New `--dry-run` mode also respects this skip — dry-run output excludes active-worktree dirs.
- **N6 stale symlinks under lifecycle/**: Recipe currently calls `realpath` which can fail on stale symlinks under `set -e`. Plan should add a `[ -e "$dir" ]` guard before `realpath` to skip broken symlinks gracefully.
- **N6 archive ordering**: `feature_complete` event presence is the sole criterion. A directory that has `feature_complete` but is also actively cited by an in-progress lifecycle's research/spec/plan still gets archived; the path-rewrite (N6.3) updates the citing files. Behavior: archived path is canonical; citers continue to work via the rewritten path.
- **N6.5 dirty working tree precheck**: If `git diff --quiet HEAD` returns non-zero on entry, the recipe aborts with a clear error pointing the user to commit or stash before running. This applies to BOTH dry-run and real-run modes (dry-run still requires clean tree because dry-run output may inform the manifest).
- **S1.1 CortexCliMissing as OSError subclass**: Existing `except OSError` handlers (`_maybe_check_upstream`, `_maybe_run_upgrade`) catch CortexCliMissing without code change. Sites that need cache-clear-and-retry add a more specific `except CortexCliMissing` handler upstream of the generic OSError catch. Python's exception ordering guarantees the specific handler fires first.
- **S1.3 retry exhausts on the same call**: The first `CortexCliMissing` triggers cache clear + one retry. If the retry's subprocess.run also raises FNF, the second `CortexCliMissing` propagates unhandled to the MCP framework's exception converter (S5.2). No third attempt.
- **S1.3 concurrent dispatch under asyncio**: Two concurrent tool calls A and B may both observe FNF and clear the cache. A's retry succeeds and primes; B's retry reads the now-primed cache and proceeds without a second subprocess call. The semantics are correct: both calls converge to the recovered state without doubling the retry budget. Tests cover this case.
- **S1.2 FNF from a child subprocess inside cortex** (e.g., cortex invokes git which is missing): This raises FNF from the cortex CLI's child process, not from server.py's invocation of cortex. Handled by cortex's own error reporting → cortex returns non-zero with a descriptive stderr → existing RuntimeError path catches it. The new FNF handler only fires when `subprocess.run(["cortex", ...])` itself raises FNF — i.e., the cortex executable (or its #! interpreter) is missing on PATH.
- **S1.2 shebang-interpreter missing**: If `cortex` is on PATH but its `#!/usr/bin/env python3` interpreter (`python3`) is missing, posix_spawn raises FNF that surfaces to subprocess.run as FileNotFoundError. The CortexCliMissing error message points at install docs, which is somewhat misleading (cortex is installed; python3 is missing) but actionable (install instructions implicitly require a working python3). Plan should consider whether to inspect `errno`/`strerror` to distinguish the two cases; if simpler, accept the conflation as acceptable given low frequency.
- **S5.1 stderr warning visibility**: Logged at module import. In overnight mode, this lands in the orchestrator's MCP debug logs and is included in morning postmortem if the run fails. In interactive mode, it appears in Claude Code's MCP debug surface.
- **S5.2 install_url stability**: README anchor (`#install`) is intentionally stable — anchors don't rot the way versioned doc paths do. If the README's "Install" section is ever renamed, this URL needs an update — but that's a foreseeable churn point.
- **S5.2 JSON-RPC error code choice**: `-32000` is the JSON-RPC "server error" range; the specific code within that range is implementation-defined. Plan may select a different code from the server-error range (`-32000` to `-32099`) if the MCP SDK reserves specific codes for protocol-level errors.

## Technical Constraints

- **Plugin invariant (server.py:1-30)**: `cortex-overnight-integration` must remain pure-subprocess; no `cortex_command.*` imports. New `CortexCliMissing(OSError)` class is local to server.py.
- **Atomic writes**: Manifest append (N6.2) uses tempfile + `os.replace()` (or shell equivalent: write to `.tmp`, `mv` to final). No partial-write corruption.
- **No `check=True` on subprocess**: Existing pattern is no `check=True` + explicit returncode handling. New FNF handlers integrate with this; don't introduce `check=True`.
- **Stderr-only logging from server.py**: stdout is reserved for JSON-RPC. S5.1 startup warning goes to stderr.
- **JSON-RPC error envelope** (S5.2): The error is surfaced via the MCP framework's exception-to-JSON-RPC-error path (e.g., raising `mcp.types.McpError` or letting `CortexCliMissing` propagate to the framework's converter). It is NOT a custom field added to a tool's success-response payload — that would clash with each tool's declared Pydantic output schema (`extra="ignore"` does not satisfy required fields). The error appears in `response.error.code` (server-error range, e.g., -32000) and `response.error.data` (the `{reason, install_url, message}` payload).
- **CortexCliMissing parentage** (S1.1): Subclasses `OSError`. This is load-bearing — existing `except OSError` handlers continue to swallow it, preserving current `_maybe_check_upstream`/`_maybe_run_upgrade` behavior. New cache-clear-and-retry handlers use the more specific `except CortexCliMissing` clause, which fires before the generic OSError clause per Python's exception-ordering rules.
- **Dual-source enforcement**: N3.1 and N4.1 edits to canonical SKILL.md trigger pre-commit `just build-plugin` regeneration of mirrors. This is automatic when `just setup-githooks` was run after clone.
- **Path-rewrite anchoring** (N6.3): Three citation forms (slash, wikilink, word-boundary) require explicit anchored regex. Plan must implement: `\blifecycle/{slug}/` for slash; `\[\[lifecycle/{slug}(/|\]|\|)` for wikilink (catches `]`, `/`, and `|` terminators); `\blifecycle/{slug}\b` for bare word-boundary. Slug values must be regex-escaped before substitution.
- **Clean-tree precheck** (N6.5): Recipe asserts `git diff --quiet HEAD` AND `git diff --quiet --cached HEAD` on entry. Recovery from any abort is `git checkout -- .`. The manifest is audit-only; not used for rollback.
- **MCP framework concurrency** (S1.3): The mcp Python SDK's request handling is async. Per-tool-call retry counters live in the dispatcher's local scope; module-global `_CORTEX_ROOT_CACHE` is single-write-after-clear, which is safe under asyncio because once one coroutine writes, others read the populated value without re-running discovery.
