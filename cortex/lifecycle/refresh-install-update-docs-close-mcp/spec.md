# Specification: refresh-install-update-docs-close-mcp

## Problem Statement

The auto-update mental model implied by cortex-command's README is partially out of sync with what #146's MCP-architecture refactor actually delivers; the Bash-tool subprocess dispatch path in `implement.md §1a` silently fails when the user's install is behind; and wontfix'd lifecycles (#145) keep surfacing in SessionStart's "incomplete lifecycles" list because no convention or detector pathway recognizes terminal wontfix state. This lifecycle bundles three deliverables to close those gaps: a docs refresh that disambiguates the two-layer (marketplace startup + MCP-tool-call CLI) auto-update mechanism and surfaces the `CORTEX_ALLOW_INSTALL_DURING_RUN=1` escape hatch; an inline `command -v cortex-daytime-pipeline` fail-fast preflight on the autonomous-worktree dispatch path; and an end-to-end terminal-state convention for wontfix'd lifecycles spanning skill prose, the phase detector, the statusline mirror, the parity fixture, and the events registry. Items 5 (R8 cwd-vs-installed-wheel divergence) and 6 (CLI_PIN drift lint) are out of scope and filed as separate backlog items for later refine.

## Edge Cases

- **`cortex-daytime-pipeline` is present but broken** (importable / on PATH but exits non-zero on `--help` or subprocess launch): R10's `command -v` preflight only checks presence, not health. A broken-but-installed binary passes the preflight and fails at dispatch time. Stderr lands in `daytime.log`; this is not a regression vs. current behavior. Richer health checks deferred to a `cortex doctor` follow-up.
- **`feature_wontfix` event emitted before `git mv` (workflow order violated)**: Under R12's mandated step order, `git mv` precedes the event emit — this case should not occur in the normal workflow. If it does (e.g., a future operator hand-edits the event without running step (a)), R13's detector patch returns `phase=complete`, which causes SessionStart's incomplete-list filter to drop the lifecycle. The directory remains in `cortex/lifecycle/` (not `archive/`) and is technically misplaced. This is a known cost of choosing the "reorder steps" fix over the "wrapper script" fix: the misplacement is silenced rather than surfaced. Acceptable because (i) the workflow order is documented to prevent this, (ii) the inverted order requires explicit deviation, and (iii) the consequence is silent-correctness rather than active-incorrectness — SessionStart enumeration is still right.
- **`feature_wontfix` event emitted as part of normal workflow** (after `git mv`): event lands under `cortex/lifecycle/archive/{feature}/events.log`. `cortex-scan-lifecycle.sh:227` already skips `archive/` from enumeration. R13's detector patch becomes defense-in-depth for any tool that queries the archived lifecycle's phase directly.
- **Malformed JSON line in events.log.** The existing JSON-parsing loop at `common.py:222–227` already catches `JSONDecodeError` and continues. R13's `elif feature_wontfix` branch inherits this tolerance.
- **Statusline parity test fixture also tested by the SessionStart hook.** `hooks/cortex-scan-lifecycle.sh:252–264` delegates to Python (`python3 -m cortex_command.common detect-phase`), so the hook automatically inherits R13's Python patch. No hook code change needed.
- **#145's events.log has a non-canonical hand-written event row.** The pre-existing row contains `feature_wontfix` but may not match the new schema (missing `reason`, no `ts`, etc.). R13's detector only checks `event_type == "feature_wontfix"`; field-completeness is not a gating concern.
- **A future hypothetical event named `feature_wontfix_aborted`** (or any variant containing `feature_wontfix` as a substring). R13 uses exact-match on `event_type` in the JSON loop, so the new variant would NOT trigger terminal. Forward-safe. The legacy `feature_complete` substring scan at line 249 also stays — adding more substring scans for new variants would not extend it (covered by Non-Requirements).
- **Items 5/6 filing collides with a concurrent backlog mutation.** The `cortex-backlog add` helper is sequential; `just backlog-index` regenerates atomically. R2/R3 require regen to complete; if it fails, surface and resolve. Race risk is low (operator-driven, not automated).
- **`bin/cortex-check-parity` finds drift introduced by Phase 2 docs work.** Phase 2 doesn't touch `cortex_command/common.py` or `skills/` substantively (only docs prose). R11 in Phase 3 explicitly runs the gate before docs and code land together.

- **`bin/cortex-check-parity:431` tracks `command -v cortex-*` pattern.** R10's new `command -v cortex-daytime-pipeline` reference is parity-recognized without an exception.
- **Phase-detector mirror parity is enforced** by `tests/test_lifecycle_phase_parity.py:300–347` (`test_statusline_ladder_matches_canonical`). R13 + R14 + R15 + R16 must land together to keep the test green.
- **Phase 4 six-way coupling: single-commit landing required.** R12 + R13 + R14 + R15 + R16 + R17 form a coupled change set across skill prose, Python, bash mirror, fixture, parity test, and events registry.
