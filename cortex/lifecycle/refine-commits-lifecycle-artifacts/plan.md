# Plan: refine-commits-lifecycle-artifacts

## Overview
Extract a `read_commit_artifacts()` Python helper, route the two live consumers (`plan.md:306`, `complete.md:19`) plus a new `post-refine-commit.md` reference through a shared `cortex-read-commit-artifacts` binstub, delete the dead inline commit at `specify.md:206`, and wire `lifecycle/SKILL.md` Step 3 §4 to invoke `post-refine-commit.md` after the `phase_transition specify→plan` (or `lifecycle_cancelled`) row is logged. Enforcement is prose-only (consistent with the rest of the lifecycle's gate pattern) — a structural exit-code-propagation mechanism is out of scope and is the natural follow-up if the prose gate empirically fails.

## Risks

- **Module docstring contract**: `cortex_command/lifecycle_config.py`'s current docstring claims "exactly one public symbol" — Task 1 expands that to two. If any external consumer or static-analysis check depends on the "single symbol" contract, this would break. Mitigation: the docstring update is part of Task 1; no separate enforcement exists today.
- **`__main__` block convention**: existing `lifecycle_config.py` has no `__main__` block; adding one (Task 2) is a small API expansion. The binstub depends on `python3 -m cortex_command.lifecycle_config` working; if the module is later split into a subpackage, the binstub's invocation path needs updating in lockstep.
- **`specify.md:206` deletion (Task 3)**: research and spec both classify this as dead code on the live invocation path. The risk is a forgotten standalone entry point — if `/cortex-core:specify` is ever invoked directly (not via refine), it will no longer commit. The spec's Non-Requirements explicitly accept this residual ("Standalone `/cortex-core:refine` invocations (not via lifecycle) remain uncommitted by this change").
- **Most-recent-event detection in prose**: Task 4's detection rule for distinguishing approval vs cancel subjects is encoded in skill prose rather than a programmatic helper. The rule is simple (scan events.log bottom-up; first hit wins), but the orchestrator must follow it correctly. If empirically the rule mis-routes, the follow-up would be a `cortex-detect-refine-disposition` binstub — out of scope here.
- **Halt-before-Plan gate is prose-only**: Task 4's gate relies on the orchestrator reading and following the documented procedure. This matches the existing prose-only enforcement style in the codebase (e.g., the kept-pauses inventory is prose-policed by the parity test, but the runtime call sites rely on the model honoring the inventory). If the model bypasses the halt on commit failure, the failure mode is recreating the original defect (bundling refine artifacts under a Plan commit). The MUST language in R4 is the escalation lever; the prose currently uses "MUST" sparingly per the post-Opus-4.7 policy, and the project's MUST-escalation policy requires evidence artifacts (events.log F-row or transcript excerpt + demonstrated effort=high failure) that this PR does not yet provide — escalation to MUST should defer to a follow-up if the soft form empirically fails.
- **No-op short-circuit is silent (observability gap)**: when the stage-first short-circuit fires (`git diff --cached --quiet` reports nothing staged), the orchestrator emits no event, no log line, and no operator-visible artifact. Incorrect short-circuit decisions would be undiagnosable post-hoc. Not mitigated in this PR — adding an `artifacts_no_op` event would require a `bin/.events-registry.md` entry which the spec's Non-Requirements explicitly forbid. If empirically a misfire is observed, a follow-up adds the registered event.
- **Resume-and-edit assumption (operator manual edits between sessions)**: if an operator manually edits `research.md`, `spec.md`, `index.md`, or the backlog file between session end and re-invocation, those edits will be staged and committed under the Refine subject (`Refine {feature}: research and spec`), which may misdescribe the diff content. The orchestrator does not diff-inspect to distinguish operator-authored from orchestrator-authored content. Mitigation: the canonical workflow is to re-invoke `/cortex-core:lifecycle` (which routes through refine) rather than hand-edit refine artifacts. If the assumption empirically breaks, the follow-up is a diff-content reconciliation step in `post-refine-commit.md`.
- **Acceptance-section coverage gap**: the §Acceptance criterion only exercises the happy path. The halt-before-Plan gate has no acceptance check, automated or manual. A complete halt-gate verification requires simulating a commit failure (e.g., introducing a temporary pre-commit hook rejection) and observing the orchestrator halts — interactive and session-dependent, not a CI-runnable step. The Task 6 content greps are the durable proxy for the gate's prose presence; behavioral halt-on-failure remains untested at runtime in this PR.
- **Test file placement**: Task 6 creates `tests/test_post_refine_commit_wired.py`. If the repo uses a `tests/lifecycle/` subdirectory by convention, the file should land there instead — verify via `ls tests/` during implementation.
