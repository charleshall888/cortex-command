# Specification: Merge refine's internal Clarify + Research into a single Investigate reference

## Problem Statement

Refine's internal Step 3 (Clarify) and Step 4 (Research) currently read two distinct reference files (`skills/lifecycle/references/clarify.md` and `skills/lifecycle/references/research.md`) that together carry duplicated requirements-loading prose (~75–100 tokens), redundant framing scaffolding (~30–50 tokens), and a procedurally artificial phase boundary that already chains as a single `/cortex-core:refine` delegation. The duplication is a symptom; the deeper problem is that refine's pre-spec investigation is conceptually one cohesive activity (intent-gating → read-only research → exit gate) but is encoded as two separately-loaded reference files. Merging them into one `investigate.md` reference improves clarity-of-flow for both human authors and LLM agents executing refine, and removes ~100–150 tokens per refine chain. Lifecycle's phase enum, dashboard mappings, parity tests, and overnight tooling remain unchanged; this is a refine-internal refactor.

Those are separate failure modes and warrant their own ticket if/when they reproduce.

- **Archived events.log rewrites**: Historical `events.log` files containing `phase_transition: {from: "clarify", to: "research"}` entries are not modified. Other clarify-critic.md prose (e.g., line 3 `Post-§3 confidence challenge` opening sentence, §-number references) remains valid via R2's section-anchor preservation and is not rewritten.

## Edge Cases

- **Archived events.log with `from: clarify, to: research` entries**: Existing entries remain in append-only logs unchanged. Dashboard parser (`cortex_command/dashboard/data.py:282–331`) reads phase_transition events generically by `from`/`to` fields with no hardcoded "clarify" literal — archived entries continue to render correctly in any historical timeline view. Expected: no regression.
- **Lifecycle session that began under old reference files**: Mid-session resume reads the current investigate.md (not stale clarify.md/research.md); resume logic operates on the per-feature artifact `lifecycle/{slug}/research.md` filename (which is unchanged), not on the reference file. Expected: clean resume.
- **Refine invoked on a feature with pre-existing `research.md` artifact**: refine's Step 2 (Check State) and §1 Sufficiency Check still operate against the artifact filename `lifecycle/{slug}/research.md`. The reference-file rename does not change this. Expected: unchanged.
- **§4 prose locality in clarify-critic.md after §4a/§4b/§4c insertion**: clarify-critic.md describes Ask items as "folded into the §4 question list" — language that presupposes §4 is textually contiguous with §5 (Produce Output). After merge, three research subsections (§4a/§4b/§4c) sit between §4 and §5. An LLM following the §4 pointer reads research procedure before reaching §5's output framing. Mitigation: this is a semantic-neighborhood drift, not an anchor-resolution breakage. Tolerated as known minor; flagged for follow-up if a parity test fails or a real refine run mis-orders §4 → §5 work.
- **A future PR re-introduces a `clarify.md` reference file**: byte-parity test auto-discovers it via glob, but the file would be unused (no skill reads it). Risk is conceptual drift; mitigated by the ADR-style header note (R10) and the merged file being the single canonical pre-spec investigation surface.
- **clarify-critic.md cross-references reach into a deleted file**: clarify-critic.md references `clarify.md §3 / §4` by section number — these continue to resolve semantically via R2's anchor preservation. The one literal proper-name reference `clarify §2` at line 107 is corrected in R14. The opening sentence `Post-§3 confidence challenge` remains valid (§3 still names Confidence Assessment in the merged file).
- **An overnight session runs refine concurrently with this change deployment**: Deployment is git-commit-based; sessions that started under the old skill files complete with the old protocol (their context is already loaded). New sessions read the merged file. No mid-session migration needed.
- **Plugin mirror byte-parity test discovers stale clarify.md or research.md in `plugins/cortex-core/skills/lifecycle/references/`**: rsync `--delete` (via `just build-plugin` in pre-commit) handles cleanup. If a developer commits without running the pre-commit hook, parity test fails — that's the intended drift signal.
- **Parallel #85 audit-and-grandfather lands a MUST in clarify.md before this merge**: R20's pre-flight catches this. Implementation halts; merge decision returns to the user to resolve the policy conflict before proceeding. The merge does not silently strip a grandfathered MUST to satisfy R12.

## Technical Constraints

- **Phase string preservation**: The literal string `"research"` is consumed as a phase value by ≥7 surfaces (`cortex_command/common.py`, `cortex_command/backlog/generate_index.py`, `cortex_command/overnight/backlog.py`, `claude/statusline.sh`, `cortex_command/dashboard/seed.py`, `cortex_command/dashboard/data.py`, and the `lifecycle_phase` frontmatter field). The merged Investigate phase continues to emit `"research"` as its phase string in events.log.
- **Section-numbering preservation handles anchor-based cross-references; proper-name references require targeted edits**: clarify-critic.md (`skills/refine/references/clarify-critic.md`) cross-references clarify.md by `§`-number in most places — R2's anchor preservation handles those without cascade. One proper-name reference at line 107 (`clarify §2`) requires a targeted update (R14); this is the only intentional clarify-critic.md edit. clarify-critic.md's parity tests in `tests/test_clarify_critic_alignment_integration.py` pass unchanged after the line-107 update.
- **§4 prose locality is semantic-neighborhood drift, not anchor breakage**: Inserting §4a/§4b/§4c between §4 (Question Threshold) and §5 (Produce Output) places ~50–150 lines of research procedure between two sections that clarify-critic.md's prose implicitly treats as contiguous. This is tolerated; LLM agents following the §4 pointer encounter the research subsections in order, which is the correct flow. Flagged in Edge Cases for follow-up if it manifests as a real ordering bug.
- **AskUserQuestion is the load-bearing user-question gate**: The tool's blocking semantics enforce the user-question gate. Prose ordering ("Wait for user answers before continuing") is preserved (R4) but is not the load-bearing mechanism — the tool call is.
- **Read-only constraint at §4a/§4b is the only safety rail for dispatched parallel-research subagents**: There is no tool-level enforcement of read-only behavior for subagents launched via the parallel-research template; the constraint is prose-in-prompt only. R18 verifies its preservation in the merged file. Dropping this rail would silently allow subagents to write files.
- **Verbatim/anti-paraphrase rail at §4b governs all parallel sub-agent dispatches**: The researcher prompt template at §4b instructs operators to use the template verbatim. R17 preserves this rail; dropping it would permit silent template drift on subsequent refine runs.
- **Dual-source plugin-mirror discipline**: Canonical sources live in `skills/`; `plugins/cortex-core/skills/` is auto-regenerated by `just build-plugin` (rsync `--delete`) and enforced by `tests/test_dual_source_reference_parity.py`. The test's `_discover_pairs()` helper globs `skills/*/SKILL.md`, `skills/*/references/*.md`, and `skills/*/assets/*.md` — so a single passing run covers both reference-file changes AND SKILL.md mirror parity for `lifecycle/SKILL.md` and `refine/SKILL.md`. No manual edits to the mirror.
- **MUST-escalation policy (CLAUDE.md)**: No new MUST/CRITICAL/REQUIRED language is introduced. R20 pre-flights source files for grandfathered MUSTs before merge begins; if a parallel branch lands one under #85 between spec approval and implementation, the implementer halts and surfaces for resolution rather than silently stripping.
- **Reference-file size cap does not apply**: The 500-line SKILL.md cap (`tests/test_skill_size_budget.py`) does not extend to reference files. The merged investigate.md (~280–330 lines after dedup) is unconstrained.
- **Bin-parity surface unaffected**: No `bin/cortex-*` scripts reference `clarify.md` or the reference file `research.md`. `bin/cortex-complexity-escalator` references the per-feature artifact, which is unchanged.
- **Discovery independence**: `skills/discovery/references/{clarify,research}.md` are separate per-skill copies with discovery-specific protocols. Out of scope for this merge; their continued existence is intentional.
