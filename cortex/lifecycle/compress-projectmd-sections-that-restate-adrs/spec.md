# Specification: compress-projectmd-sections-that-restate-adrs

> Epic context: primary research is the skill-value-scorecard audit (`cortex/research/skill-value-scorecard/report.html`); this spec executes only the eight `project.md` verdicts. See `research.md`.

## Problem Statement

`cortex/requirements/project.md` is always-loaded context on the load-requirements path (lifecycle, refine, discovery). The epic #347 audit verified eight line-ranges as restating content that already lives, sometimes clause-for-clause, in an ADR body (which the ADR README forbids project.md from restating), in always-loaded `CLAUDE.md`, in a test docstring, or in enforcement-site docs. Compressing these — dropping the duplicated prose while preserving the clauses with no other home — shrinks per-invocation context and removes a drift surface, without changing any runtime behavior (no test reads the compressed bodies). The beneficiary is every consumer that loads project.md and the maintainer who otherwise keeps duplicated prose in sync.

## Requirements

**Priority (MoSCoW)**: R1–R8 (the eight audit verdicts) and R9–R12 (the cross-cutting gates) are all **Must** — the ticket's scoped deliverable plus its verification floor. There is no *Should* tier at the requirement level; the one genuine discretionary dimension is the **compression magnitude** per verdict (how aggressively to cut), which is *Should*-grade editorial judgment tuned by the user at §4 approval / PR review. s6's keeps default to **keep-conservative** when magnitude is uncertain. **Won't**: everything in Non-Requirements.

**On verification strength (read before trusting the greps).** The per-requirement greps are a *necessary* floor, not a *sufficient* one. Two limits are load-bearing and were surfaced by critical-review:
- **A presence-grep proves token-survival, not clause-survival.** `grep -Fc 'TOKEN' ≥ 1` catches a *deletion* of the token, not an over-cut that keeps the token while gutting its sentence. So every keep-grep is paired, where the surviving clause is prose, with a named clause the user must eyeball at diff review.
- **The keep side passes on the unedited file.** Because keeps are presence-greps on tokens already in `project.md`, each requirement also carries a **cut-happened** grep — an absence-grep (`= 0`) for a token **that is present in the current file** (a phantom token, absent before any edit, discriminates nothing — R1's original `three-path` was exactly this bug and was removed). Any cut token the plan pins later MUST (a) currently appear in the target range and (b) be distinctive to that range (not survive elsewhere), so it cannot be satisfied by a no-op or a wrong-region deletion.
- **R12 is the master guard.** A whole-file diff-scoping check confines every changed line to the eight verdict ranges and asserts byte-identity of the untouched neighbor bullets — this is what actually protects against neighbor-drift and against "delete instead of compress." Do not treat R1–R8's per-token greps as complete without R12.

Cut-safety (the dropped content lives in its other-home) was verified in `research.md`; the plan re-checks it per section. All greps target `cortex/requirements/project.md` unless noted.

R9–R12 are whole-feature gates: R9 is verified at each phase boundary that touches a lint-adjacent bullet; R9–R12 all finalize at the end of the last phase (Quality Attributes + Optional), which is why they carry that phase tag.

## Edge Cases

- **Contract-lint collision (narrower than it first appears)**: project.md is in `cortex-check-contract` scan scope (`test_check_contract.py:162`), but the two s8 lint tokens R4 keeps (`cortex-check-bare-python-import`, `cortex-check-skill-path`) declare no required flags and no subcommand, so E101/E103 are structurally unreachable for them regardless of prose form — `cortex-check-contract --audit` returns 0 on the current file. The real hazard is only if a compression introduces a *new* bare `cortex-<verb>` mention for a verb that *does* have required args. Expected: keep the existing passing token forms; do not add new bare verb mentions; run `cortex-check-contract` before committing each affected phase (R9).
- **Over-cut of a gate-unverifiable keep**: the must-survive *clauses* (as opposed to the pointer tokens the greps pin) across s4, s6, s7, s9, s10, s11, s8-L44, and s15 have no test backstop — an over-cut that keeps the token while gutting the clause passes the per-token automation. Expected: the added clause-level greps (R3 `never imports`, R6 `NOT exempt`, R7 three-clause greps, R8 bullet-name greps) narrow this, but keep-*sufficiency* is ultimately caught only by the user's diff review at approval/PR (load-bearing, not ceremonial). Note: the earlier framing that enumerated only "s6/s10/s11/s8-L44/s4" understated the surface — s7, s9, and s15 belong on this list too.
- **Neighbor-drift inside a touched H2**: the compressions edit line-ranges *inside* the Philosophy, Architectural Constraints, and Quality Attributes H2s, which also hold untouched siblings (L21 Solution-horizon, L36 TERMINAL_STATUSES, L39 ADR-0002, L40, L54–57/L59). An edit that strays into a neighbor bullet is not caught by any per-verdict grep. Expected: **R12's diff-scoping guard** confines changed hunks to the verdict ranges and asserts neighbor byte-identity — this is the primary defense, not the editor's care.
- **requirements-write token floor**: compression must not drop a required H2 or the Optional convention line to save tokens. Expected: R10 grep guards the H2 count; the convention line is a named keep.
- **s6 medium-confidence**: its keeps (shim policy sentence, `FORCE_SOURCE` remedy) are the weakest-backstopped in the set. Expected: treat conservatively — when in doubt, keep the sentence.
