# Specification: remove-daytime-autonomous-pipeline-and-cancel

## Problem Statement

The daytime autonomous implement-phase pipeline (three modules + `readiness.py` + console-scripts + dashboard surfaces + skill prose + ~30 paths of code-hygiene references) is being retired as the final sweep under epic #237, replaced by the worktree-interactive implement-phase option delivered by siblings #238 (menu update) and #240 (variant-A end-to-end). This PR deletes the daytime code path in full, restores dashboard PR-url rendering for the worktree-interactive replacement (sourcing from `cortex/lifecycle/{slug}/pr.json` artifacts), cancels in-flight ticket #228 with a supersedence record, annotates already-complete #230, and adds a structural contract test for the worktree-interactive dispatch surface to provide forward-going regression coverage for the new path. The contract test is scoped narrower than the deleted `test_daytime_preflight.py` (which covered daytime-specific PID-liveness, polling-fallback, and outcome detection on a surface that no longer exists); it is not a verbatim coverage replacement. Beneficiaries: code maintainability (eliminates dead-after-swap code); operator workflow (cleaner implement-phase preflight); future readers (no orphan references to abandoned code path).

## Technical Constraints

- **Pytest collection-phase safety**: Tests + modules + console-scripts + parity-exception rows + `.gitignore` patterns + audit allowlist entries + justfile recipe must be deleted in a single atomic commit. Multiple couplings force this atomicity: (a) module-import-level references in test files crash pytest collection if modules are deleted in an earlier commit (Adversarial F12); (b) `[project.scripts]` entries pointing at deleted modules would fail at `uv tool install --reinstall` between commits; (c) `bin/.parity-exceptions.md` rows reference console-scripts as `deprecated-pending-removal` — the parity linter flags missing-source-with-no-exception drift if the rows survive past the script removal.
- **Intra-Phase-5 operation order — R21 must complete before R22's `cortex-update-item` invocation**: `cortex_command/backlog/update_item.py:409` reads `TERMINAL_STATUSES` at the moment `cortex-update-item 228 status=superseded` runs and uses it to decide whether to cascade `_remove_uuid_from_blocked_by` and `_check_and_close_parent`. If the `cortex-update-item` invocation runs against a working tree where `common.py:162` has not yet been edited to include `superseded`, the cascade silently does not fire. In practice this is currently harmless (no extant `blocked_by: [228]` chains; #230's frontmatter uses `blocked_by:` with snake-case while the cascade regex matches `blocked-by:` with kebab-case; #237's other children are non-terminal so the parent does not auto-close), but the order dependency is load-bearing for correctness of any future `blocked_by: [228]` chain. Phase 5's commit must apply R21 to the working tree before invoking R22's `cortex-update-item`.
- **Dual-source mirror sync**: `plugins/cortex-core/skills/lifecycle/references/implement.md` auto-regenerates via `.githooks/pre-commit`. No hand-edit needed; verify `just check-parity` (or equivalent) before push.
- **In-PR commit sequence** (bisect-safe): Phase 1 first (tests can land independently because they target HEAD's already-clean `implement.md`); Phase 2 second (single commit: modules + tests + console-scripts + parity-exceptions + .gitignore + audit allowlist + justfile recipe); Phase 3 third (dashboard cleanup + PR-url adaptation); Phase 4 fourth (docs + registries + code-hygiene + requirements catalog + orphan comments + `_DAYTIME_DISPATCH_FIELDS` docstring retitle); Phase 5 fifth (backlog cancellation + CHANGELOG `### Removed` entry + `TERMINAL_STATUSES` extension). Each phase's commit's `just test` passes.
- **`TERMINAL_STATUSES` extension scope**: This PR extends `cortex_command/common.py:162` frozenset only. The second tuple at `cortex_command/overnight/backlog.py:40` flagged at `common.py:159` NOTE is a separate follow-up ticket.
- **`_DAYTIME_DISPATCH_FIELDS` filter retention rationale**: Read-side compatibility shim for archived `pipeline-events.log` rows on operator machines. Removing it would contaminate historical metric aggregation. (Adversarial F6.)
