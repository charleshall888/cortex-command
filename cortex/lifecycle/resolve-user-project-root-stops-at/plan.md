# Plan: resolve-user-project-root-stops-at

## Overview
Add a lock-scoped `_resolve_main_repo_root()` to `interactive_lock.py` (pure-Python, env-first, eager worktree-`.git`-file detection with a `commondir`-aware parse and a `cortex/`-existence guard, falling back to the shared `_resolve_user_project_root()` walk), wire `_lock_path`/`_events_log_path` to it, and prove correctness with a real-worktree-collision regression test — leaving `common.py`'s shared resolvers and the deliberately worktree-local behavior of their ~30 other consumers untouched. Phase 2 broadens edge-case coverage and reconciles the false "resolves up to the main repo" contract claim in the #241 spec and the sandbox-test docstrings.
**Architectural Pattern**: layered

- **What**: Add a new module-private resolver `_resolve_main_repo_root()` that resolves the MAIN repo root regardless of CWD, and rewire `_lock_path` and `_events_log_path` to call it instead of `_resolve_user_project_root()`. Covers Req 1, 2, 3, and makes Req 4's CI test pass with no test change.
- **Depends on**: none
- **Complexity**: complex
- **Context**:
  - New signature: `def _resolve_main_repo_root() -> Path:` (no args, mirrors `_resolve_user_project_root`). Place it in the "Internal helpers" section near `_lock_path` (currently `:72`). Keep the existing `from cortex_command.common import _resolve_user_project_root` import (`:53`) — the new resolver's fallback calls it as a literal `return _resolve_user_project_root()`, which preserves #241 R2's `grep -c '_resolve_user_project_root' ≥ 2` (Req 3).
  - Resolution order (load-bearing, eager worktree-detect — NOT walk-first): **(a)** if `os.environ.get("CORTEX_REPO_ROOT")` is set, `return Path(env).resolve()` verbatim (no `.git` parse, no subprocess); **(b)** else eagerly walk `Path.cwd().resolve()` and ancestors for the first `.git` entry that is a **file**; on it, read the gitfile, strip the literal `gitdir: ` prefix and trailing newline, resolve the pointer relative to the `.git` file's directory to get `$GIT_DIR`; if a sibling `commondir` file exists in `$GIT_DIR`, read it and resolve it relative to `$GIT_DIR` to get `<main>/.git`, then take `.parent` as the candidate main root; if no `commondir`, take `$GIT_DIR.parent` as the candidate (handles the synthetic-direct `gitdir: <main>/.git` shape); **(b-guard)** only `return candidate.resolve()` when `(candidate / "cortex").is_dir()`; **(c)** in every other case (no worktree `.git` file in ancestry, `.git` is a directory, malformed/missing `gitdir:`/`commondir`, or the candidate has no `cortex/`) `return _resolve_user_project_root()`.
  - Pure-Python only — no `subprocess`, no `git rev-parse` (research empirically showed `git rev-parse --git-common-dir` exits 128 on the hand-built fixture and on no-`.git` `cortex/` projects). The `.git`-is-a-directory case must NOT trigger branch (b) — a `.git` directory is neither a terminator nor a match for the (b) scan; it is skipped, and the geometry routes to (c) so #201's anti-leak boundary (an unrelated nested git repo with no `cortex/` still raises via the bounded walk) is preserved. (Restated here, not only at the (a)/(b)/(c) ordering above, so the file-vs-dir predicate is unambiguous.)
  - **Step (c) is best-effort, not universally safe** (critical-review Objection 1): for a well-formed real `interactive/{slug}` worktree, step (b) always succeeds and returns `<main>`, so (c) is reached only for non-worktree projects, `.git`-directory repos, or *corrupt/incomplete* worktree git state. This is no worse than today's behavior and only reachable under git state a real `git worktree add` never produces; it is an accepted, documented limitation (see `## Risks`) and is asserted by a Task 3 edge case rather than left silent. The spec fixes the mechanism (Req 3 mandates the literal `_resolve_user_project_root()` fallback), so this is documented, not redesigned.
  - Edge cases to honor (from spec `## Edge Cases`): real worktree `gitdir: <main>/.git/worktrees/<id>` + `commondir` → `<main>`; synthetic direct `gitdir: <main>/.git` (no `commondir`) → `<main>`; relative `gitdir:`/`commondir` pointers; malformed/missing pointer or a pointer to a root with no `cortex/` → fall to (c); submodule `.git` file → (b-guard) fails → (c); macOS `/tmp`→`/private/tmp` handled by the final `.resolve()`.
  - **Correct the module docstring in THIS task** (critical-review concern R4-1/R4-5): the current "All path resolution flows through `_resolve_user_project_root()`" claim (`:5-7`) becomes actively false the moment the wiring changes — so this task must minimally rewrite it to state the lock's path functions now resolve via `_resolve_main_repo_root()`. The Phase 1 commit must NOT ship a module header that contradicts the code it lands. Task 4 (Req 10) then *expands* this corrected docstring with the fuller guarantee (the why-separate rationale, `scan_live_locks`, and the structural-guard pointer) — it is no longer correcting a false claim, only enriching a true one. `_lock_path`/`_events_log_path`'s own docstrings ("resolved against the main repo root") stay true and need no edit.
  - **Path-hardcoding-gate caution** (critical-review concern R4-3): `.githooks/pre-commit` Phase 1.9 scans `cortex_command/**/*.py` for hardcoded `"<prefix>/"` literals (prefix ∈ {lifecycle, backlog, research, requirements, retros, debug}). Build paths from `Path` joins of separate `"cortex"`/`"lifecycle"`/`feature_slug` segments (as `_lock_path` already does) — do NOT introduce a convenience literal like `f"lifecycle/{slug}/interactive.pid"` in the new resolver or any test helper, or the commit will block.
- **Verification**:

## Risks
- **Step (c) is a best-effort fallback, not a universal main-root guarantee** (critical-review Objection 1): the spec mandates the literal `_resolve_user_project_root()` fallback (Req 3), which from inside a worktree-with-`cortex/` returns the *worktree* root. For a well-formed real `git worktree add`, branch (b) always succeeds and returns `<main>`, so (c) is reached only for non-worktree projects, `.git`-directory repos, or corrupt/incomplete worktree git state. In that last pathological case the resolver degrades to worktree-local resolution (pre-fix behavior, no worse than today). This is an accepted, documented limitation — pinned by `test_resolve_main_repo_root_bfail_inside_worktree_degrades_to_local` (Task 3) so it cannot change silently. Changing it would require touching `common.py` (a Non-Requirement) or a non-literal fallback (would break Req 3's grep ≥2).
- **`.git`-is-a-directory must not trigger branch (b)**: if the implementer lets a `.git` directory enter the gitfile-parse branch, the #201 anti-leak boundary weakens. The plan routes `.git` dirs to step (c) (stated twice in Task 1 Context); the existing nested-repo guardrail (`tests/test_common_utils.py:460`) is re-run by Task 5's full suite.
- **Locus was user-decided; divergence is structurally guarded** (addresses the Approach-D tension surfaced in review): the lock-scoped resolver vs. an explicit `project_root` param was resolved at spec time (`spec.md` Open Decisions: "locus … user-selected"), so the Plan phase implements the chosen locus rather than re-litigating it. The "two resolvers must agree" hazard is enforced structurally, not by prose alone: Task 2's `test_resolve_main_repo_root_worktree_with_cortex_resolves_to_main` (now including a `read_lock`-from-worktree assertion) fails if the lock resolver ever diverges from main, which is the structural enforcement CLAUDE.md prefers.
- **Per-phase commits clear the pre-commit gate chain** (critical-review concerns R4-2/R4-3): each phase commit traverses `.githooks/pre-commit` — the Phase 1.5 parity gate fires on staged `tests/*` (Tasks 2–4) and the Phase 1.9 path-hardcoding gate fires on `cortex_command/**/*.py` (Tasks 1, 4). Both pass for these edits today; the spec's "`cortex-check-parity` does not gate this edit" means only that no `cortex_command/` dual-source *mirror* exists, not that the gate never *fires*. Avoid introducing a hardcoded `lifecycle/{slug}/...` literal (see Task 1's caution).
- **Phase 2 is should-have**: if time-boxed, Phase 1 (Tasks 1–2) is the complete load-bearing fix — and the live `read_lock` divergence is now proven closed by Task 2's worktree `read_lock` assertion (not merely path resolution). Tasks 3–5 add durability/coverage and contract hygiene and can be deferred without leaving the fix incorrect (only less-broadly-tested and less-documented). Note: the CI sandbox test (Task 1) only confirms the Seatbelt write path on vanilla macOS CI — it skips at implement time, so Phase-1's implement-time behavioral proof is Task 2's unit suite.
