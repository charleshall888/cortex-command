# Specification: resolve-user-project-root-stops-at (#271)

## Problem Statement

The interactive lock (`cortex_command/interactive_lock.py`) must write and read `interactive.pid` against the **main repo root** so the lock is visible across sessions (#241's documented contract). It resolves that path via `_resolve_user_project_root()`, whose upward walk checks `(current / "cortex").is_dir()` **before** `(current / ".git").exists()`. This produces two failures: (1) the CI test `tests/test_interactive_lock_sandbox.py::test_lock_write_from_worktree_cwd` fails with `CortexProjectRootError` on vanilla macOS (a synthetic worktree with **no** `cortex/`, so the walk reaches the `.git` *file* and breaks); and (2) the production-reachable bug — because `cortex/` is git-tracked, a real `interactive/{slug}` worktree contains its own `cortex/`, so the walk returns the **worktree** root on **iteration 1** (the `cortex/`-is-dir check fires before the `.git` check ever runs). The lock is *written* to main (acquire runs pre-`EnterWorktree` from the main CWD) but `read_lock` on re-invocation from inside the worktree (`complete.md` Step 3 Variant-A detection) reads the **worktree's** `cortex/` → finds no lock → returns `None` → **false-negative Variant-A detection silently skips the PR-creation wrapper**. The fix gives the lock its own main-repo resolver that **prefers the main repo over a co-located worktree `cortex/`**, leaving the shared `_resolve_user_project_root` (and its `_from_cwd` twin) — and their deliberately worktree-local behavior for ~30 other call sites (#198) — untouched.

## Phases
- **Phase 1: Lock-scoped main-repo resolver** *(must-have)* — add `_resolve_main_repo_root()` with worktree-precedence ordering, wire the lock's path functions to it, and prove correctness with the failing CI test **and** a real-worktree-collision regression test. This phase is the load-bearing fix: it turns CI green and closes the live `read_lock` divergence with no regression to other consumers.
- **Phase 2: Durability & contract reconciliation** *(should-have)* — broaden resolver edge-case coverage (hand-built fixtures, no `git worktree add`), correct the false "resolves up to the main repo" claim in both #241's spec and the test docstring, and document the writer/reader/scanner main-root guarantee.

Priority (MoSCoW): Requirements 1–7 are **must-have** (correct, regression-free fix proven by tests); Requirements 8–11 are **should-have** (broader durability and contract hygiene). The `## Non-Requirements` section enumerates the explicit **won't-do** boundaries.

## Requirements

1. **`_resolve_main_repo_root()` helper in `interactive_lock.py`, with worktree-precedence ordering**: resolves the MAIN repo root regardless of CWD. The order is load-bearing and MUST be implemented as written (eager worktree detection, NOT walk-first): **(a)** if `CORTEX_REPO_ROOT` is set, return `Path(os.environ["CORTEX_REPO_ROOT"]).resolve()` verbatim (env-first, preserved); **(b)** else, eagerly walk `Path.cwd().resolve()` and its ancestors looking for a `.git` entry that is a **file** (worktree gitfile); on the first such `.git` file found, parse its `gitdir:` pointer (resolving a relative pointer against the `.git` file's directory), then read a sibling `commondir` file (resolving relative-to-`$GIT_DIR`) when present to obtain `<main>/.git`, and take its `.parent` as the candidate main root — **this branch takes precedence over any `cortex/` directory found at or below the worktree root**; **(b-guard)** verify `(<candidate> / "cortex").is_dir()` before returning `<candidate>.resolve()`; if the candidate has no `cortex/` (or the pointer/commondir is malformed or missing), do not return it; **(c)** in all other cases — no worktree `.git` *file* in the ancestry, `.git` is a directory, or the (b)/(b-guard) path did not yield a cortex-bearing root — fall back to a literal `return _resolve_user_project_root()` call. Pure-Python (no `git rev-parse` subprocess). Acceptance: `grep -c 'def _resolve_main_repo_root' cortex_command/interactive_lock.py` = 1 (pass if = 1); `grep -nE 'subprocess|rev-parse' cortex_command/interactive_lock.py` shows no git-subprocess call inside the new resolver (manual diff check, pass if absent); the resolver body references `commondir` and returns a `.resolve()`-canonicalized path (manual diff check). **Phase**: Phase 1: Lock-scoped main-repo resolver

## Non-Requirements
- Does **NOT** modify `_resolve_user_project_root` or `_resolve_user_project_root_from_cwd` in `common.py` — the shared walk and its deliberately CWD-following twin (`lifecycle_event.py` event routing) keep their worktree-local behavior; the `.git`-boundary guardrail (`test_common_utils.py:460`) stays green.
- Does **NOT** change `scan_live_locks`'s signature or its reliance on the caller-passed `project_root` (overnight env-pins it to main).
- Does **NOT** reconcile the per-feature `events.log` root divergence into a single file — the lock's events are intentionally main-anchored (consistent with the lock) while `lifecycle_event.py`'s phase events stay worktree-local; see Technical Constraints. Unifying them is out of scope.
- Does **NOT** shell out to `git rev-parse` — pure-Python parse only (per #201's rejection of subprocess in resolvers).
- Does **NOT** add a `git worktree add`-based integration test — hand-built fixtures cover the real-worktree pointer shape (including the `cortex/`-collision case, Requirement 7) without the editable-`.pth` rewrite hazard.
- Does **NOT** modify #241's R2 acceptance grep itself (Requirement 3 keeps `interactive_lock.py`'s reference count ≥ 2 so the existing check still passes).
- Does **NOT** touch the distinct `cortex/lifecycle/sessions/{slug}.interactive.pid` session pidfile (a separate mechanism from the interactive_lock path).
- Does **NOT** extract a shared `pid_lock.py`, address cross-checkout concurrency, sub-agent lock participation, or the `Agent(isolation:"worktree")` silent-failure (all #241 Non-Requirements).

## Edge Cases
- **Real worktree `.git` (`gitdir: <main>/.git/worktrees/<id>`) WITH a co-located `cortex/`**: step (b) follows `gitdir:` → `commondir` → `<main>/.git` → `.parent` = `<main>`; the (b-guard) `cortex/`-existence check passes for `<main>`; resolver returns `<main>` — the worktree-local `cortex/` does **not** win. (This is the production bug's exact shape; Requirement 7 tests it.)
- **Synthetic test `.git` (`gitdir: <main>/.git`, no `commondir`, no worktree-local `cortex/`)**: no `commondir` → `.parent` of the gitdir target = `<main>`; (b-guard) passes; returns `<main>`. Passes the existing CI fixture unchanged.
- **Relative `gitdir:` or relative `commondir` pointer**: resolve `gitdir:` relative to the `.git` file's directory; resolve `commondir` relative to `$GIT_DIR` (the `worktrees/<id>` dir) before normalizing.
- **Malformed/missing `gitdir:` line, missing `commondir`, deleted admin dir, or a worktree pointer whose target has no `cortex/`**: (b-guard) fails → fall back to step (c) `_resolve_user_project_root()` (no crash, no non-cortex path returned).
- **No `.git` anywhere + `cortex/` present (non-git project)**: no worktree `.git` file → step (c) walk resolves the cortex root — do not raise.
- **Unrelated nested git repo with no `cortex/`**: `.git` is a *directory* (not a worktree file) → step (b) does not fire; step (c) walk terminates at the `.git` boundary and raises `CortexProjectRootError`. The #201 anti-leak guarantee holds on the step-(c) path; the (b-guard) `cortex/`-existence check ensures step (b) never returns a non-cortex ancestor either.
- **`CORTEX_REPO_ROOT` set**: returned `.resolve()`-canonicalized; no `.git` parse, no git invocation.
- **Submodule `.git` file (also uses `gitdir:` → `<super>/.git/modules/<name>`)**: step (b) would parse it, but the (b-guard) `cortex/`-existence check on `<super>/.git/modules`'s `.parent` fails (no `cortex/` there) → falls back to step (c). In practice unreachable regardless, because `cortex init` rejects initialization inside a submodule (`init/handler.py`), so no cortex project's `cortex/` ever lives in a submodule. Not special-cased beyond the (b-guard).
- **macOS symlinked tmp (`/tmp` → `/private/tmp`)**: the resolver returns a `.resolve()`-canonicalized path so the lock write path matches the canonical `cortex init` sandbox grant (and the SBPL subpath in the test, which is also `.resolve()`-d).

## Changes to Existing Behavior
- **MODIFIED**: `interactive_lock._lock_path` / `_events_log_path` resolve via `_resolve_main_repo_root()` (was `_resolve_user_project_root()`) → the lock lands at `<main>/cortex/lifecycle/{slug}/interactive.pid` even from a worktree CWD (with a co-located `cortex/`) when `CORTEX_REPO_ROOT` is unset.
- **MODIFIED (behavioral fix)**: `read_lock` invoked from inside a worktree (e.g. `complete.md` Step 3 Variant-A detection on re-invocation) now reads the MAIN-repo lock path — eliminating the false-negative Variant-A detection that silently skipped the PR-creation wrapper.
- **ADDED**: `_resolve_main_repo_root()` helper in `interactive_lock.py`, and a real-worktree-collision regression test (Requirement 7).
- **UNCHANGED**: `common.py`'s two resolvers; `scan_live_locks`; the `.git`-boundary semantics for every other consumer of the shared resolver.

## Technical Constraints
- `cortex_command/common.py` is lifecycle-gated, but this fix does **not** touch it (lives in `interactive_lock.py` + tests + one #241 spec line + one test docstring). No dual-source plugin mirror exists for `cortex_command/`, so `cortex-check-parity` does not gate this edit.
- **Wheel-binstub vs working-tree (scoped)**: the `cortex-interactive-lock` *binstub* executes the installed wheel; tests that shell out to the **binstub** need reinstall / `CORTEX_COMMAND_FORCE_SOURCE` is irrelevant (that env var is honored only by the bash `bin/cortex-*` wrappers, not by console-script entry points). The sandbox test and the new unit tests invoke `python3 -m cortex_command.interactive_lock` / import the module directly, both of which already run working-tree code via the editable install — so no special env is needed for them; `just test` (Requirement 11) likewise runs the working tree.
- **ADR-0003 sandbox grant**: `cortex init` registers the **main** repo's `cortex/` path (canonical/`.resolve()`-d) as writable; the lock must resolve there. Requirement 1's `.resolve()` on the returned path keeps the write inside the grant and avoids a `/tmp` vs `/private/tmp` symlink mismatch on macOS.
- **Pure-Python, env-first, eager-worktree-detect, guarded, walk-fallback**: no subprocess; `CORTEX_REPO_ROOT` first; worktree `.git`-file detection before any `cortex/` acceptance; a `cortex/`-existence guard before returning a parsed root; fall back to `_resolve_user_project_root()` otherwise.
- **events.log root-locality (intentional divergence)**: `interactive_lock`'s events (`interactive_lock_acquired`, `_released`, `_stale_recovered`) resolve to `<main>/cortex/lifecycle/{slug}/events.log` via `_resolve_main_repo_root()`, consistent with the lock being main-anchored; `lifecycle_event.py`'s phase events resolve to the worktree-local `events.log` via `_resolve_user_project_root_from_cwd()` (#198/#260, unchanged). In the live flow `acquire` (the lock's main event emitter) runs pre-`EnterWorktree` from the main CWD, so this is not a behavioral regression for emitted events; the divergence is by design (lock state is cross-session/main; phase events are worktree-local then merged on worktree integration). Unifying the two is a Non-Requirement.
- **Writer/reader/scanner agreement**: writer (acquire) and reader (`read_lock`) both resolve to main via `_resolve_main_repo_root()` regardless of CWD; scanner (`scan_live_locks`) stays main via the overnight orchestrator's `CORTEX_REPO_ROOT` env-pin. All three converge on main.

## Open Decisions
None — locus (lock-scoped resolver, user-selected), mechanism (pure-Python `commondir`-aware parse, resolved by empirical evidence that `git rev-parse --git-common-dir` returns exit 128 against the hand-built fixture), resolver ordering (eager worktree-detect with `cortex/`-existence guard, precedence over local `cortex/`), and `scan_live_locks` scope (unchanged) are all resolved at spec time.
