# Specification: fix-test-cascade-from-252-migration

## Problem Statement

#252's migration of 13 `bin/cortex-*` scripts from canonical-Python to dual-channel bash wrappers left a test cascade — pre-existing tests that loaded those scripts via `importlib.SourceFileLoader`, invoked them via `subprocess.run([sys.executable, str(SCRIPT_PATH), …])` or `subprocess.run([str(SCRIPT_PATH), …])`, parsed the wrapper file as Python source via `ast.parse`, or invoked the wrapper with a stripped environment were not updated. Tasks 11/12/15 patched their own downstream tests; the rest did not. Result: `just test` fails with ~60 failures + 28 errors across at least 11 broken-pattern files plus one pre-existing unrelated failure (`test_log_invocation_perf`). Production behavior is intact on main. This feature repairs the cascade by applying canonical patterns from Tasks 11/12/15 until `just test` reaches its closeout signal (Requirement 4).

## Phases

- **Phase 1: `SourceFileLoader` → direct import** — Rewrite the four test files that load `bin/cortex-*` via `SourceFileLoader` to import from the promoted `cortex_command.*` modules, preserving any module-private symbols the tests consume.
- **Phase 2: subprocess invocation cleanup** — For each subprocess callsite targeting `bin/cortex-*`, apply the appropriate per-callsite handling: default `[sys.executable, "-m", "cortex_command.X.Y", …]`; helper-split for the shared `_invoke` in `test_commit_preflight.py` so `test_shim_records_invocation` keeps wrapper invocation; SCRIPT_PATH re-point for `test_git_env_hardening`'s AST walker (not a subprocess callsite); and committed `-m` conversion for `test_cortex_morning_review_gc_demo_worktrees.py`.
- **Phase 3: `id(tmp_path)` aliasing fix at all 7 parity sites** — Replace `cache_key = (id(tmp_path), case)` with `cache_key = (str(tmp_path), case)` (and update the cache type annotation accordingly) in all seven parity-test files that carry the pattern.
- **Phase 4: Closeout verification** — Run `just test` and confirm exit 0 OR that the only remaining failure is `tests/test_log_invocation_perf.py`.

## Requirements

1. **SourceFileLoader rewrites for the four affected files, preserving consumed symbols**: After Phase 1, `tests/test_resolve_backlog_item.py`, `tests/test_load_parent_epic.py`, `tests/test_superseded_frontmatter_tolerance.py`, and `tests/test_variant_a_writer_sites_baseline.py` import directly from the promoted `cortex_command.*` modules rather than via `importlib.machinery.SourceFileLoader`. For `test_variant_a_writer_sites_baseline.py`, the canonical `escalator_module` fixture pattern from Task 15 (`tests/test_complexity_escalator.py:34,40–43`) is preserved — the import goes via `import cortex_command.lifecycle.complexity_escalator as _escalator_module` and the existing `@pytest.fixture(scope="module") def escalator_module(): return _escalator_module` shape is retained. For the other three files, every module-private symbol the tests currently access (e.g., `_parse_frontmatter`, `_item_title`, `_resolve_*`) remains accessible after the rewrite — either by importing the symbols by name or by binding the imported module to a local alias the tests reference. **Acceptance** (multi-line-safe via Python AST inspection): For each of the four files, the file contains at least one top-level `import cortex_command.…` or `from cortex_command.… import …` statement (verified by `python3 -c "import ast,sys; tree=ast.parse(open(sys.argv[1]).read()); sys.exit(0 if any(isinstance(n, (ast.Import, ast.ImportFrom)) and ((isinstance(n, ast.ImportFrom) and (n.module or '').startswith('cortex_command')) or (isinstance(n, ast.Import) and any(a.name.startswith('cortex_command') for a in n.names))) for n in ast.walk(tree)) else 1)" <file>` exits 0) AND the file contains zero `SourceFileLoader` invocations referencing a `bin/cortex-*` path (verified by `python3 -c "import ast,sys; tree=ast.parse(open(sys.argv[1]).read()); calls=[n for n in ast.walk(tree) if isinstance(n, ast.Call) and ((isinstance(n.func, ast.Attribute) and n.func.attr == 'SourceFileLoader') or (isinstance(n.func, ast.Name) and n.func.id == 'SourceFileLoader'))]; sys.exit(1 if calls else 0)" <file>` exits 0). **Phase**: Phase 1.

2. **Subprocess fixes for the eight affected files**: After Phase 2, every subprocess callsite that previously invoked `bin/cortex-*` via `[sys.executable, str(SCRIPT_PATH), …]` or `[str(SCRIPT_PATH), …]` uses the appropriate handling, documented inline at the callsite or via inline docstring:
   - **2a. For `test_variant_a_writer_sites_baseline.py` specifically, the original callsites use `[str(ESCALATOR_SCRIPT), …]` (wrapper-direct, no `sys.executable`); the conversion drops the wrapper and adds `sys.executable + "-m"`.
   - **2b. Split it into two helpers: `_invoke_module` (default — uses `[sys.executable, "-m", "cortex_command.commit.preflight", …]`) and `_invoke_wrapper` (uses `[str(SCRIPT_PATH), …]` and is invoked only by `test_shim_records_invocation` so the wrapper's `cortex-log-invocation` shim at wrapper lines 12–14 still fires). Update each consuming test function to call the appropriate helper.

Define a local `_PYTHON_SCRIPT_PATH = REPO_ROOT / "cortex_command" / "commit" / "preflight.py"` constant used only by this test, and substitute it for `SCRIPT_PATH` in the `read_text()` + `ast.parse()` calls (and the `filename=` argument). The AST walker's assertion logic continues against the canonical Python source.

   **Acceptance**: `pytest tests/test_resolve_backlog_item.py tests/test_load_parent_epic.py tests/test_check_prescriptive_prose.py tests/test_commit_preflight.py tests/test_superseded_frontmatter_tolerance.py tests/test_variant_a_writer_sites_baseline.py tests/test_clarify_critic_alignment_integration.py tests/test_cortex_morning_review_gc_demo_worktrees.py` exits 0 with no FAILED or ERROR lines in the short test summary. **Phase**: Phase 2.

4. **Closeout signal**: After Phase 4, `just test` exits 0. If `just test` exits non-zero, the failure surface must be exactly `tests/test_log_invocation_perf.py` (the pre-existing unrelated failure the ticket explicitly excludes), demonstrated by running that test in isolation and matching the pre-task failure mode. The two-part check counts collection-time ERROR lines and bare-anchored FAILED lines from direct pytest invocation (bypassing `just test`'s `sed` indent wrapper). **Phase**: Phase 4.

5. **No production-side changes outside test infrastructure**: This feature does not modify files under `cortex_command/` (excluding tests under `cortex_command/dashboard/tests/`), `bin/`, `skills/`, `hooks/`, `claude/`, `pyproject.toml`, or `justfile`. Plugin-tree mirrors under `plugins/*/(skills|hooks|bin)/` are auto-regenerated by the pre-commit hook on canonical-source edits — this exception covers ALL plugin trees (cortex-core, cortex-overnight, cortex-dev-extras, etc.), not just cortex-core, because `just build-plugin`'s write set spans all of them and the pre-commit hook applies uniformly. The `test_git_env_hardening` `SCRIPT_PATH` re-point in Requirement 2c is a test-only change (the test reads the production file but does not modify it). **Acceptance**: `git diff --name-only main...HEAD -- cortex_command/ bin/ skills/ hooks/ claude/ pyproject.toml justfile | grep -vE '^cortex_command/dashboard/tests/' | wc -l` equals 0 AND `git diff --name-only main...HEAD -- plugins/ | grep -vE '^plugins/[^/]+/(skills|hooks|bin)/' | wc -l` equals 0. **Phase**: Phase 1, 2, 3, 4.

## Non-Requirements

- **Do NOT modify the promoted `cortex_command.*` modules.** All 13 modules already expose runnable `__main__` blocks via `pyproject.toml [project.scripts]`; no production-side edits are required to support the `-m` invocation path.
- **Do NOT install or update binstubs as a test-setup precondition.** `uv tool install --reinstall --refresh` is not introduced as a `just test` dependency. The dual-channel wrappers' wheel-import probe is not exercised by these tests; that surface lives in dedicated parity tests under `tests/test_cortex_*_parity.py`.
- **Do NOT rewrite the affected subprocess tests to call promoted-module Python APIs in-process** (e.g., via `monkeypatch.setattr(sys, "argv", …)`). The affected tests specifically verify exit-code surface, env-var handling, stderr formatting, and wrapper-shim side effects under process-boundary conditions; in-process rewrites lose that fidelity.
- **Do NOT auto-fix or update `tests/test_log_invocation_perf.py`.** It is a pre-existing unrelated failure that the ticket explicitly excludes from scope.
- **Do NOT regenerate plugin mirrors as part of this lifecycle.** The pre-existing `test_dual_source_reference_parity.py::test_dual_source_byte_parity[cortex-core::skills/lifecycle/references/complete.md]` failure that Research surfaced was resolved by intervening commits on main (`b081ab7c`, `f86abc36`, `3d494d16`) before Implement began; `just build-plugin` produces an empty diff on the current baseline. Adding a regen step would be either a no-op or would silently expand the diff with unrelated mirror updates. If new mirror drift surfaces during Implement (i.e., a Phase 1/2/3 edit indirectly modifies a canonical source the pre-commit hook regenerates), the regen is handled by the pre-commit hook automatically and falls within Requirement 5's plugin-tree exception.
- **Do NOT auto-expand scope on Phase 4 surprises.** New failures uncovered during Implement that are not `test_log_invocation_perf` are surfaced at Review per Requirement 4 — they do not silently expand the in-scope file set.
- **Structural-prevention status — bound to #248, not shipped here.** Research surfaced that 9 of 25 `bin/cortex-*` scripts remain unpromoted Python and backlog #248 (`status: in_progress`) explicitly proposes a wholesale rewrite of the bin/cortex-* invocation surface. Project.md's Solution-Horizon conditions for the durable fix are MET (named places + planned follow-up). The structural prevention (a lint rule, pre-commit hook, or test-collection guard catching `SourceFileLoader('...', 'bin/cortex-*')` and `subprocess.run(['bin/cortex-*', ...])` patterns in `tests/`) is therefore obligated as a Done-When in #248's body. Adding the prevention to this ticket is rejected as scope expansion that #248 will land more durably alongside its own wholesale rewrite.

## Edge Cases

- **Multi-line callsite formatting**: All `SourceFileLoader` and `subprocess.run([sys.executable, str(SCRIPT_PATH), …])` callsites in the affected files span two or three lines in source — e.g., `subprocess.run(\n        [sys.executable, str(SCRIPT_PATH), *args],` at `tests/test_resolve_backlog_item.py:154`. Acceptance criteria in R1, R2, R3 use multi-line-safe checks (AST inspection, repo-wide canonical-shape regex, positive assertions on the new imports) rather than single-line `grep` regexes that would pass vacuously against the multi-line source.
- **Secondary blast radius from unmasked ImportErrors**: Fixing import-/collection-level failures will unmask additional failing tests previously hidden by `ImportError` at module load. Pytest's short test summary distinguishes `FAILED <path>` (assertion failure) from `ERROR <path>` (collection-time error); Requirement 4's acceptance counts both via direct `pytest` invocation (bypassing `just test`'s `sed` wrapper that prevents bare line-anchor matches). If new failures surface, triage them at Implement: real regressions caused by this work are fixed in-scope; pre-existing unrelated failures are surfaced for follow-up rather than auto-allowlisted.
- **Dashboard tests (`cortex_command/dashboard/tests/test_feature_cards_pr_url.py`, `test_templates.py`)**: Listed in the ticket as affected but containing no `SourceFileLoader` or `bin/cortex-*` subprocess patterns per Research. Expected failure mode is indirect (transitive ImportError from a sibling collection failure). After Phase 2 completes, run `just test` and confirm. If they pass, no Phase 4 action; if they still fail, triage the actual failure at Implement rather than inventing a fix in spec.
- **`test_git_env_hardening` after SCRIPT_PATH re-point**: The test audits whether the script under test passes git-env hardening assertions. Re-pointing the read to `cortex_command/commit/preflight.py` (the canonical Python source) preserves the test's intent — it still audits the right code. If the audit's AST walker references bash-only constructs (unlikely — the canonical was Python before #252), the test fails on the new path and Implement adjusts the walker.
- **`test_shim_records_invocation` after helper-split**: The wrapper's `cortex-log-invocation` shim must remain functional and discoverable on the test host; if `cortex-log-invocation` is itself broken on the fresh checkout, this test fails — but that is a pre-existing condition of the wrapper, not a #261 regression.
- **`test_cortex_morning_review_gc_demo_worktrees.py` env setup**: After conversion to `[sys.executable, "-m", "cortex_command.overnight.gc_demo_worktrees", …]`, the stripped env (`{"TMPDIR": …, "PATH": …}`) is sufficient because `-m` invocation bypasses the wrapper's CLI-discovery cascade entirely. The test's hermeticity intent (isolate from user TMPDIR) is preserved by keeping the explicit `env=` parameter; only the wrapper-discovery failure mode goes away.
- **Working-tree vs wheel-installed divergence**: Tests after this work run against the working-tree module via `python3 -m`. A wheel reinstall is not required between code edits and `just test` invocations.

## Changes to Existing Behavior

- **MODIFIED**: Four test files load promoted-module code via direct Python import rather than `SourceFileLoader` on `bin/cortex-*` (Phase 1).
- **MODIFIED**: Eight test files use the per-callsite patterns documented in Requirement 2 (default `-m`, helper-split for the shared `_invoke` in `test_commit_preflight.py`, SCRIPT_PATH re-point for `test_git_env_hardening`) rather than uniform `[sys.executable, str(SCRIPT_PATH), …]` against `bin/cortex-*` paths (Phase 2).
- **MODIFIED**: Seven parity-test files use `str(tmp_path)` rather than `id(tmp_path)` as a cache key (Phase 3).
- **MODIFIED**: `just test` exits 0 (or fails only on `test_log_invocation_perf`) where it previously failed with ~60 failures + 28 errors.

## Technical Constraints

- **Project.md L35**: Promoted modules expose `[project.scripts]` console-script entries; `python3 -m cortex_command.<skill> <subcommand>` is the documented readable fallback. This is the basis for Phase 2's default pattern.
- **Project.md L38** (Wheel-binstub vs working-tree invocation): Tests run against the working tree via `python3 -m`, not against installed wheel binstubs, because binstubs lag the working tree between edits. `python3 -m cortex_command.X` bypasses the wrapper entirely and is the working-tree-fidelity choice.
- **All affected promoted modules have `__main__` blocks**: Verified at Research and re-confirmed at Plan: `cortex_command.backlog.resolve_item`, `cortex_command.backlog.load_parent_epic`, `cortex_command.lint.prescriptive_prose`, `cortex_command.commit.preflight`, `cortex_command.lifecycle.complexity_escalator`, `cortex_command.overnight.gc_demo_worktrees` (note: under `overnight`, not `morning_review`).
- **`just test` output is `sed`-prefixed**: The `just test` recipe wraps each pytest invocation through `echo "$output" | sed 's/^/       /'` (7-space indent). Closeout-signal greps must NOT use `^FAILED ` line anchors against `just test` output; either grep directly against pytest output (`.venv/bin/pytest tests/`) or use the `sed`-tolerant pattern in Requirement 4.
- **Plugin mirror exception spans all trees**: `just build-plugin` regenerates mirrors under all six `plugins/*/{skills,hooks,bin}/` trees, not just `cortex-core`. Requirement 5's plugin-tree exception is written to match the actual write surface; narrowing it to `cortex-core` only would create a false acceptance failure if pre-commit-hook mirror regen touches other trees during Implement.
- **No environment-state coupling**: Tests must pass on a fresh checkout without `uv tool install --reinstall --refresh` as a precondition (ADR-0002's two-channel distribution does not guarantee binstubs are present).
- **Acceptance criteria must not pass vacuously**: R1, R2, R3 acceptance checks use multi-line-safe tooling (AST inspection, repo-wide canonical-shape regex, positive assertions on the new imports) so a green acceptance signal actually verifies the work happened.
