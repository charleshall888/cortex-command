# Specification: unified-backlog-lifecycle-slug-resolver-extend

## Problem Statement

`cortex-update-item` and `cortex-resolve-backlog-item` resolve backlog inputs through two independent code paths with inverted acceptance — the resolver accepts lifecycle slugs but rejects filename slugs; `cortex-update-item._find_item` does the opposite, plus accepts UUID prefixes. The substring fallback in `update_item.py:142-145` silently picks the first sorted hit when an input matches many items (`"add"` matches 29 items today). This ticket extends `cortex_command/backlog/resolve_item.py` from its current 3-step order (numeric → kebab → title-phrase) to a deterministic 5-step order spanning all four slug forms (UUID prefix → numeric ID → exact filename stem with-or-without `NNN-` prefix → exact `lifecycle_slug` frontmatter → ranked title-substring), and retrofits `cortex_command/backlog/update_item.py:_find_item` to consume it via a new pure library function `resolve()`. The substring-footgun fix and the consumer retrofit ship together because the new contract (ranked candidates / exit-2 ambiguity) requires the consumer-side migration once anyway. Beneficiaries: every skill that mutates backlog state (six skill files) gains explicit ambiguity surfacing and accepts the lifecycle_slug form it currently rejects.

## Phases

> **Shipping unit**: Both phases ship together in the same PR. Phase 1 alone is not a release-worthy state — the resolver advertises a new contract via `cortex-resolve-backlog-item` but the silent-first-match footgun in `update_item._find_item` is only closed in Phase 2. The phase split is a planning decomposition for the implementation plan, not a release-cadence boundary.

- **Phase 1: Resolver extension** — Extend `resolve_item.py` to the 5-step order with a new pure `resolve()` library function returning a tagged `ResolutionResult`. Preserve all current CLI exit codes; add UUID-prefix and lifecycle_slug-frontmatter branches; convert silent substring matching to ranked exit-2 candidates. Capture the frozen 3-step baseline as a JSON snapshot before resolver changes land.
- **Phase 2: Consumer retrofit + skill-prose updates** — Replace `update_item._find_item` body with a library call to `resolve()`. Adopt exit-2 + stderr candidate list in `cortex-update-item`. Promote `CURATED_INPUTS` into a shared corpus fixture (`backlog_resolution_corpus`) consumed by both resolver and update_item test surfaces. Update six skill files (clarify.md, refine SKILL.md, complete.md, wontfix.md, backlog-writeback.md, morning-review SKILL.md) to recognize exit-2 in their failure-handling prose.

## Requirements

1. **Library function `resolve()` exists in `cortex_command/backlog/resolve_item.py`**: After the change, the module exposes a pure function `resolve(input_str: str, backlog_dir: Path) -> ResolutionResult` where `ResolutionResult` is a frozen dataclass with `status: Literal["ok", "ambiguous", "not_found"]`, `item: Path | None`, and `candidates: list[Path]`. IO/parse failures raise a new `ResolutionError` exception caught at the CLI boundary and mapped to exit-70. **Acceptance**: `python3 -c "from cortex_command.backlog.resolve_item import resolve, ResolutionResult, ResolutionError; print('ok')"` exits 0. **Phase**: Phase 1

2. **5-step deterministic resolution order implemented in `resolve()`**: The function applies these steps in order, returning the first step that produces ≥1 match (≥2 matches → `status="ambiguous"` with the full candidates list; 1 match → `status="ok"` with `item` set): (1) UUID prefix, (2) Numeric ID, (3) Exact filename stem with-or-without `NNN-` prefix, (4) Exact `lifecycle_slug` frontmatter match, (5) Ranked title-substring fallback. **Acceptance**: `pytest tests/test_resolve_backlog_item.py -k "test_resolution_order"` exits 0; the test parametrizes one positive case per step against a controlled fixture corpus and asserts which step matched.  **Phase**: Phase 1

3. **UUID-prefix branch requires minimum 8 hex characters**: Inputs that match the regex `^[0-9a-fA-F]+(-[0-9a-fA-F]+)*$` and have a hex-character length (hyphens stripped) of ≥8 are tested against frontmatter `uuid:` values via case-insensitive prefix match. Shorter pure-hex inputs that look UUID-shaped (length 1-7) fall through to subsequent steps rather than producing pathological candidate lists. **Rationale grounded in code**: empirical scan of 230 live UUIDs (`cortex/backlog/*.md` frontmatter) shows zero collisions at length 6+; 8 chars chosen for safety margin per `cortex_command/backlog/resolve_item.py` style of conservative defaults. **Acceptance**: `pytest tests/test_resolve_backlog_item.py -k "test_uuid_prefix_minimum_length"` exits 0; tests cover length 7 (falls through), length 8 (matches), unique 8-char prefix uniquely resolves, ambiguous 8-char prefix returns exit-2 candidate list. **Phase**: Phase 1

4. **`cortex-resolve-backlog-item` CLI preserves its current exit-code contract**: Exit 0 (unambiguous), 2 (ambiguous), 3 (no match), 64 (usage), 70 (IO/parse). New UUID-prefix and lifecycle_slug-frontmatter resolutions feed the same exit-code surface. **Acceptance**: `pytest tests/test_resolve_backlog_item.py` exits 0 with all existing tests passing (no regression on the 30+ `CURATED_INPUTS` corpus). **Phase**: Phase 1

5. **Frozen 3-step baseline snapshot + order-drift regression test (Phase-1-independent of corpus location)**: Before resolver changes land, capture a JSON snapshot at `tests/fixtures/predicate_3step_baseline.json` recording the 3-step resolution outcome (resolved filename, exit code) for every input in the current `CURATED_INPUTS` corpus at `tests/test_resolve_backlog_item.py:77-109`. A parametrized test `test_no_order_drift_against_baseline` then verifies that every snapshot entry's input resolves under the 5-step order to the same outcome. Because the baseline is a frozen JSON file, the test is independent of `CURATED_INPUTS`'s in-file location and survives Phase 2's promotion to the shared fixture. Drift (an input resolving to a different item) is a hard failure. **Acceptance**: (a) `tests/fixtures/predicate_3step_baseline.json` exists and is non-empty; (b) `pytest tests/test_resolve_backlog_item.py -k "test_no_order_drift_against_baseline"` exits 0. **Phase**: Phase 1

6. **`cortex_command/backlog/update_item.py:_find_item` consumes `resolve()` directly**: After the change, `_find_item(slug_or_uuid, backlog_dir)` is implemented as a call to `resolve(slug_or_uuid, backlog_dir)` that returns the matched `Path` on `status="ok"`, returns `None` on `status="not_found"`, and propagates ambiguity to the caller (via a sentinel return or a new internal return type — implementation detail; the CLI handler in `main()` surfaces exit-2). The current inline UUID-prefix branch at `:147-152` and the unranked substring branch at `:142-145` are removed. **Acceptance**: `grep -c "slug_or_uuid in p.stem" cortex_command/backlog/update_item.py` = 0 AND `grep -c "uuid_val.startswith" cortex_command/backlog/update_item.py` = 0. **Phase**: Phase 2

7. **`cortex-update-item` surfaces ambiguity as exit-2 + stderr candidate list**: When the underlying `resolve()` returns `status="ambiguous"`, the CLI exits 2 and writes the candidate list to stderr using the same `_format_candidates` shape as `cortex-resolve-backlog-item` (`ambiguous: N matches\n<filename>\t<title>\n…`, capped at 5 with `… (N more)` overflow). No backlog file is mutated on exit-2. **Acceptance**: `pytest tests/test_update_item_resolution.py -k "test_ambiguous_exits_2_with_candidate_list"` exits 0; the test asserts `exit_code == 2`, asserts the candidate-list format via regex, and asserts no file mutation via mtime comparison. **Phase**: Phase 2

8. **`cortex-update-item` not-found case stays exit-1**: When `resolve()` returns `status="not_found"`, the CLI exits 1 with `Item not found: <input>` on stderr — preserves the current contract at `update_item.py:466-467`. **Acceptance**: `pytest tests/test_update_item_resolution.py -k "test_not_found_exits_1"` exits 0. **Phase**: Phase 2

9. **Shared test corpus fixture (`CURATED_INPUTS` promoted to `backlog_resolution_corpus`)**: `CURATED_INPUTS` and `backlog_resolution_corpus` refer to the same artifact at different phases. Phase 2 extracts the existing `CURATED_INPUTS` constant from `tests/test_resolve_backlog_item.py:77-109` into a shared location — either `tests/conftest.py` (extended) or a new `tests/backlog_resolution_corpus.py` — as a `list[ResolutionCase]` (or equivalent) consumed by both `tests/test_resolve_backlog_item.py` and a new `tests/test_update_item_resolution.py`. The extraction preserves corpus contents verbatim — no case is dropped or rewritten during the move. New corpus cases added by this ticket (after extraction) cover UUID prefix (length 7 falls through; length 8 resolves; ambiguous 8-char prefix), exact `lifecycle_slug` frontmatter match, prefix-stripped stem ambiguity (two items sharing a prefix-stripped stem → exit-2), and the substring-footgun cases that currently resolve silently in `update_item` but will exit-2 under the new contract. **Acceptance**: `grep -c "backlog_resolution_corpus" tests/test_resolve_backlog_item.py tests/test_update_item_resolution.py` = 2 (one citation per file, minimum) AND `pytest tests/test_resolve_backlog_item.py tests/test_update_item_resolution.py` exits 0. **Phase**: Phase 2

10. **Six skill files updated for exit-2 handling**: Each of the six skill files that invoke `cortex-update-item` (`skills/lifecycle/references/clarify.md:112-117`, `skills/refine/SKILL.md:84-87,187-198`, `skills/lifecycle/references/complete.md:203-212`, `skills/lifecycle/references/wontfix.md:44`, `skills/lifecycle/references/backlog-writeback.md:29,80,88`, `skills/morning-review/SKILL.md:99-109`) gains one new sentence (or bullet) in its failure-handling prose specifying that exit-2 indicates ambiguity and listing the user-facing action: "present the candidate list on stderr to the user and ask them to re-invoke with a disambiguated slug." **Acceptance**: `grep -cE "exit[ -]?2|ambiguous|disambiguat" skills/lifecycle/references/clarify.md skills/refine/SKILL.md skills/lifecycle/references/complete.md skills/lifecycle/references/wontfix.md skills/lifecycle/references/backlog-writeback.md skills/morning-review/SKILL.md` returns a count ≥ 6 (one per file). **Phase**: Phase 2

11. **Resolver does NOT verify lifecycle directory existence**: The step-4 lifecycle_slug-frontmatter match reads only frontmatter; it does not check whether `cortex/lifecycle/{slug}/` exists on disk. Callers needing directory verification do it separately. **Acceptance**: `grep -c "cortex/lifecycle" cortex_command/backlog/resolve_item.py` returns the same count post-change as it did pre-change (the change must not introduce a new directory-existence check). **Phase**: Phase 1

## Non-Requirements

- **Cascade-loop refactor is NOT in scope.** `update_item._remove_uuid_from_blocked_by` (L191-245) matches `blocked-by` entries against already-known exact UUIDs/IDs at L225-233 and does not call `_find_item`. Cascade semantics stay byte-identical.
- **`cortex_command/overnight/backlog.py:104-130` (`BacklogItem.resolve_slug`) is untouched.** Confirmed during research as a category mismatch in the ticket's Touch points (slug derivation from already-resolved frontmatter, not fuzzy input resolution). `resolve_item.py:99-109` already mirrors the same fallback chain. This is a scope boundary, not a deliverable.
- **Backlog-dir discovery consolidation is NOT in scope.** `resolve_item._backlog_dir()` walks from `Path.cwd()` and honors `CORTEX_BACKLOG_DIR`; `update_item.main()` uses `_resolve_user_project_root()` and honors `CORTEX_REPO_ROOT`. Consolidating the two walk strategies into a shared `cortex_command.common.resolve_backlog_dir()` helper is a follow-up ticket — flag in the PR description so it can be filed.
- **No new pytest framework dependencies.** The existing manual JSON-fixture snapshot pattern at `tests/test_resolve_backlog_item.py:test_predicate_a_baseline_capture` extends naturally to R5's `predicate_3step_baseline.json`; `pytest-regressions` or `pytest-golden` adoption is out of scope.
- **No new `--json` or `--strict` flag on `cortex-update-item`.** The exit-2 + stderr candidate list is the sole new ambiguity surface; structured-output flags can be added later if a caller needs them.
- **No new bash scripts or `[project.scripts]` entries.** `pyproject.toml` already declares both `cortex-resolve-backlog-item` and `cortex-update-item` (per ticket 252).
- **`tests/test_cortex_resolve_backlog_item_parity.py` is preserved verbatim.** The wheel-vs-working-tree parity test is unrelated to this ticket.
- **Cross-channel rollout coordination between CLI wheel and plugin distribution is NOT in scope.** The user-side update of the CLI wheel (via `uv tool install --reinstall`) and the cortex-core plugin (via `/plugin update`) are asynchronous channels. This ticket relies on the existing CLI/version pin probe at SessionStart (ticket 235, `status: refined`) to coordinate the rollout. The PR description must call out the cross-channel dependency so the merge cadence aligns with a plugin release that ships the updated skill prose.
- **Behavioral testing of skill-prose exit-handling paths is NOT in scope.** Requirement 10's grep-only acceptance verifies that the new prose was added to each of the six skill files; it does not parametrize skills against synthetic CLI outputs to assert their failure-handling routes correctly. This is consistent with existing skill-test practice (no skill in the repo is executed under synthetic CLI fixtures today). A future ticket could build skill-prose behavioral test infrastructure if cross-channel skew incidents accumulate.

## Edge Cases

- **Pure-hex input shorter than 8 chars** (e.g., `a3b9`): falls through the UUID-prefix step rather than triggering exit-64; subsequent steps may still match (e.g., the same string could match a title substring). The minimum-length check is a *skip*, not an error. Historically the inline UUID branch at `update_item.py:147-152` accepted any prefix length; under the new order, short hex strings route to subsequent steps and may resolve to a different item or surface as exit-2. The order-drift baseline (R5) only covers inputs present in `CURATED_INPUTS` today (which contains no short hex prefixes), so this back-compat change is documented behavior, not a regression the snapshot will catch — operators relying on short UUID prefixes must use ≥8 chars going forward.
- **UUID prefix matches two items**: returns `status="ambiguous"` with both candidates; CLI exits 2 with the standard candidate format.
- **Filename stem matches with-and-without `NNN-` prefix simultaneously**: two items `007-foo-bar.md` and `7-foo-bar.md` both share the prefix-stripped stem `foo-bar`. Input `foo-bar` → exit-2 ambiguity (two candidates). This matches the current `_resolve_kebab` behavior; not a new regression.
- **Lifecycle_slug frontmatter present on multiple items**: should not happen by convention (lifecycle slugs are author-unique) but if it does, exit-2 with both candidates.
- **Lifecycle_slug frontmatter empty or absent on all items**: step 4 produces zero matches; falls through to step 5 (title-substring).
- **Input matches a step's predicate but the matched item's frontmatter is malformed**: `_parse_frontmatter` raises `yaml.YAMLError`; the resolver re-raises as `ResolutionError`; the CLI maps to exit-70 (preserves current behavior).
- **Input slugifies to empty (e.g., `!!!`)**: exit-64 (preserves current behavior at `resolve_item.py:286-292`).
- **Backlog directory missing or empty**: exit-70 (preserves current behavior at `resolve_item.py:295-311`).
- **An existing input that currently resolves to one item under the 3-step order resolves to a different item under the 5-step order**: this is the order-drift regression failure mode flagged by adversarial review. Requirement 5's parametrized test against the frozen `predicate_3step_baseline.json` snapshot is the gate — must pass before merge.
- **Skill calls `cortex-update-item` with a single-letter input like `a`**: not pure-digit (numeric step skipped); kebab step (no match); lifecycle_slug step (no match); title-substring step matches many items → exit-2 with candidate list. The skill's failure-handling prose (Requirement 10) now covers this. This is a behavior change from the historical inline-UUID-branch acceptance of single-letter prefixes — operators who relied on single-letter inputs must use a more specific slug.
- **Overnight runner invokes `cortex-update-item` with an ambiguous input**: in practice, overnight callsites pass fully-resolved slugs (filename-slug from `morning-state.json`, numeric ID from `backlog_id` field, or full UUID), making ambiguity-by-fuzzy-input unreachable from autonomous paths. If exit-2 does fire from an overnight context — e.g., a hand-authored skill passes a fuzzy slug — the round halts as a hard failure, matching the current overnight failure-handling for any non-zero `cortex-update-item` exit. The "ask the user" instruction in Requirement 10 is a daytime affordance; overnight failure mode is unchanged (halt, surface in morning report).

## Changes to Existing Behavior

- **MODIFIED**: `cortex-update-item` exit codes. Adds exit-2 (ambiguous) to the existing {0, 1} surface. Six skill files updated to recognize it. The user-facing rollout splits across CLI wheel (`uv tool install --reinstall`) and plugin (`/plugin update`) channels; cross-channel skew is coordinated by ticket 235 (per Non-Requirements).
- **MODIFIED**: `cortex_command/backlog/resolve_item.py` resolution order. From 3 steps (numeric → kebab → title-phrase) to 5 steps (UUID → numeric → kebab+prefixed → lifecycle_slug → title-phrase). Order-drift regression test (R5) against the frozen `predicate_3step_baseline.json` snapshot gates the change.
- **MODIFIED**: `cortex-update-item` ambiguity behavior. Currently picks the first-sorted match silently (`update_item.py:142-145`); after the change, surfaces ambiguity as exit-2 candidate list with no file mutation.
- **MODIFIED**: short UUID-prefix acceptance. The inline UUID branch at `update_item.py:147-152` accepted any prefix length (1+ chars). Under the new order, the UUID-prefix step requires ≥8 hex chars; shorter prefixes skip the step and fall through. Operators relying on short UUID prefixes (1-7 chars) must adopt ≥8-char prefixes.
- **ADDED**: `cortex_command/backlog/resolve_item.py` exposes a new pure library function `resolve()` and a new `ResolutionResult` dataclass + `ResolutionError` exception type. The CLI `main()` is rewritten to call `resolve()`.
- **ADDED**: UUID-prefix branch and exact `lifecycle_slug`-frontmatter branch in the resolver, both reachable via `cortex-resolve-backlog-item` (currently rejected) and `cortex-update-item` (currently rejected for lifecycle_slug; UUID branch unifies the previous inline code).
- **ADDED**: `tests/test_update_item_resolution.py` (new test file), `tests/fixtures/predicate_3step_baseline.json` (new frozen snapshot), and a shared corpus fixture consumed by both resolver and update_item test surfaces.

## Technical Constraints

- **Python 3.9 type-hint compat** (existing module convention): use `List[Path]`, `str | None`. `Literal["ok", ...]` requires `from typing import Literal` (works on 3.9+).
- **Wheel-side import** (project.md Architectural Constraints: "Wheel-binstub vs working-tree invocation"): both `resolve_item.py` and `update_item.py` are in the same package; `from cortex_command.backlog.resolve_item import resolve` resolves identically under wheel-binstub and working-tree (`python3 -m`) modes. No `CORTEX_COMMAND_FORCE_SOURCE` consideration.
- **Two-channel distribution coordination** (project.md / docs/internals/auto-update.md): the CLI wheel and the cortex-core plugin update asynchronously. This spec relies on ticket 235's SessionStart CLI/version pin probe to surface drift; no new coordination mechanism is introduced here. The PR must merge alongside a plugin release that carries the updated skill prose to minimize the skew window.
- **Preserve exit-70 distinction**: the library function raises `ResolutionError` for IO/parse failures; the CLI `main()` catches it and exits 70 (matches current `resolve_item.py:397-399` semantics).
- **Preserve atomic-read-snapshot semantics**: `resolve_item.py` reads frontmatter with no defensive locking (per existing module docstring at lines 9-11 — `atomic_write` callers + `os.replace` provide pre/post snapshot guarantees). The new branches inherit this; no new locking required.
- **Test isolation**: tests use `CORTEX_BACKLOG_DIR` env override (resolver) and direct `backlog_dir` parameter (library function). The shared corpus fixture builds a synthetic backlog via `_make_item` helper. The frozen 3-step baseline (R5) captures the snapshot once against the synthetic corpus and is committed verbatim — no regen-on-test-run.

- Lifecycle_slug frontmatter match form → exact (Decision 5).
- Lifecycle directory existence check → no (Requirement 11).

## Proposed ADR

None considered. The 5-step deterministic resolution order is the canonical pattern endorsed by the discovery research (Q3 in `cortex/research/harness-friction-triage/research.md`) and Decision Record DR4 in that document. The library extraction pattern (pure function returning a tagged Result; CLI shim translates to exit code) is the standard Python CLI design pattern (Click/Typer convention; documented in `cortex/requirements/project.md`'s Skill-helper modules clause). No new architectural decision warrants ADR-level surface.
