**PATH / invocation in CI (verified)** — `bin/` is **not** on PATH in a fresh `actions/checkout@v4` checkout, and the wheel is not installed. So `cortex-check-contract` as a bare command will not resolve; invoke the binstub by **relative path** `bin/cortex-check-contract --audit` (it has `#!/usr/bin/env bash` and is executable). The binstub-with-`FORCE_SOURCE` form is preferable for fidelity (it exercises the actual shipped wrapper); the `python3 -m` form is the simpler fallback.

**No existing CI precedent** — no current workflow runs pytest or any `cortex-check-*` gate (this is the first). `justfile` has `check-contract` → `bin/cortex-check-contract {{args}}` and `check-contract-audit` → `bin/cortex-check-contract --audit`; **neither sets `CORTEX_COMMAND_FORCE_SOURCE`**, so CI must set it explicitly on the audit step for the wheel-immune guarantee.

**Conventions to follow** — append two steps matching existing style; add `pytest` to the `Install dependencies` line; pytest step is blocking (default); audit step gets `continue-on-error: true` + explicit `CORTEX_COMMAND_FORCE_SOURCE=1`.

## Web Research

**`continue-on-error: true` vs `|| true` (GitHub-documented mechanic)** — every step exposes `steps.<id>.outcome` (result *before* `continue-on-error`) and `steps.<id>.conclusion` (result *after*). With `continue-on-error: true`, a failing step has `outcome == failure` but `conclusion == success`, so the **job and run stay green** and the failure remains programmatically inspectable. `|| true` instead forces exit 0, discarding the exit code entirely (failure undetectable even in expressions/logs). **Recommendation: use `continue-on-error: true`** for the non-blocking audit step — it is the canonical documented mechanism and preserves the signal.

**Required-status-checks safety (verified against GitHub docs)** — branch protection treats a check as passing when its conclusion is `success`/`skipped`/`neutral`. A step-level `continue-on-error: true` resolves to `conclusion: success`, so it **does not block required checks**. Apply `continue-on-error` at the **step level**, not job level (job-level keeps the run green but paints the job red in the PR UI — over-signals).

**Visibility caveat** — per community discussion #15452, a step-level `continue-on-error` renders ~green in the UI; the failure is only discoverable by expanding logs. GitHub has not shipped a Travis-style "allow_failures" yellow badge. So decouple "don't fail the build" from "make the signal visible": keep the step exit-neutral with `continue-on-error`, and route findings to a deliberately visible channel — **`$GITHUB_STEP_SUMMARY`** (append GitHub-flavored markdown; renders on the run summary page; summary failures don't affect job status) is the lowest-maintenance option for a small repo. Problem-matchers/annotations are more visible but higher setup cost — overkill here.

**pytest step convention** — after `setup-python`, add `pip install pytest` (or fold into the existing install line) then `pytest tests/test_check_contract.py`. pytest exits non-zero on any failure/collection error → fails the step → fails the job: exactly the blocking behavior wanted. **Do not** add `continue-on-error` to the pytest step.

Key sources: GitHub Actions contexts reference (outcome vs conclusion), workflow-commands ($GITHUB_STEP_SUMMARY), required-status-checks troubleshooting, "Building and testing Python", community discussion #15452.
