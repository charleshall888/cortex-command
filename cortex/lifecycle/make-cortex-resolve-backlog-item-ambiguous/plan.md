# Plan: make-cortex-resolve-backlog-item-ambiguous

## Overview
Replace the two drift-prone byte comparisons in `tests/test_cortex_resolve_backlog_item_parity.py` — the count-pinned `title_phrase_ambiguous` stderr and the title-pinned `numeric_unambiguous` stdout — with local structural assertions that read the resolver's live output and pin only the format contract (header regex, candidate `filename\ttitle` shape, truncation arithmetic, JSON key-presence), deleting the two de-pinned snapshot fixtures. `no_match` stays fully byte-exact. Then de-drift the test docstring and fixture README so a future recapture cannot resurrect the snapshots. This is spec option **(a1) local structural assertions**, confined to one test file plus its fixture README — no `test_parity_contract.py` surgery, no resolver change.

## Outline

### Phase 1: Drift-immune structural assertions (tasks: 1, 2)
**Goal**: Replace the ambiguous-stderr and numeric-stdout byte comparisons with structural assertions against live resolver output, and remove the two snapshot files as drift sources.
**Checkpoint**: `uv run pytest tests/test_cortex_resolve_backlog_item_parity.py -q` passes with all three cases discovered; the two de-pinned fixtures are gone; `just test` is green on the current tree (live ambiguous count 34).

### Phase 2: Fixture-doc hygiene (tasks: 3, 4)
**Goal**: Update the test module docstring and the fixture README so both describe the structural exceptions accurately and the recapture recipe no longer regenerates the de-pinned snapshots.
**Checkpoint**: docstring documents the two structural cases and no longer claims they are byte-compared; README carries no stale count and no recapture command for the de-pinned snapshots.

## Risks
- **Live-output reads are intentionally count-agnostic — with a recurring governance cost.** The ambiguous assertion reads the live `cortex/backlog/` (via the existing `CORTEX_BACKLOG_DIR` determinism override) rather than a frozen corpus. The user chose (a1) over the research-recommended frozen-corpus (b′) in the interview (spec Open Decisions: "None") — settled, not reopened here. The honest trade beyond byte-exactness: a1 defends against snapshot resurrection only with **prose discipline** (Task 3 docstring + Task 4 README de-drift of the recapture recipe), which CLAUDE.md explicitly disfavors ("prefer structural separation over prose-only enforcement"); b′ would have removed the live-backlog coupling structurally and made recapture-driven re-drift impossible. This plan mitigates the prose-only weakness two ways: the committed `test_ambiguous_structure_rejects_malformed` meta-test (Task 1) makes no-op-freedom automatable, and Task 4's verification now gates removal of **both** recapture redirects so a future recapture can't silently resurrect either snapshot.
- **Forking two cases off the shared parity helper** introduces two call-site-local exceptions to the suite's "byte-identical by default; every tolerance named and per-stream" discipline. This leaves the shared `TOLERANCE_CATEGORIES` frozenset (consumed by ~21 siblings) **unchanged** — but the suite's end-state is genuinely heterogeneous: `no_match` byte-exact, `numeric_unambiguous` ID-regex + key-presence, `title_phrase_ambiguous` count-regex + shape. The spec considered and rejected reusing the existing `error-formatter-shape` category (it never inspects bytes → would weaken to a no-op) and rejected adding a sixth category (single-use addition to a closed set). The local exceptions are the accepted cost; the framing here is "no change to the shared frozenset," not "no divergence in discipline."
- **Premise-broke edge cases are accepted, not defended.** If `lifecycle` ever matches ≤1 item (exit ≠ 2) or item 252 is archived/renumbered (numeric exit ≠ 0), the affected case fails by design — the premise would have genuinely broken (spec Edge Cases). Note the numeric helper still pins `^252-` per spec Req 4: it is drift-immune to the *title* edits that drive the recurring failure, but retains a bounded ID pin that breaks only on renumber/archive of one specific completed item. Not mitigated; recorded as bounded residual.

## Acceptance

The drift-immunity is **count-agnosticism**: the ambiguous assertion reads the live resolver output and pins format (header/candidate-shape/truncation-arithmetic), not the count `N` — so it passes for any `N > 1`. Concretely: `just test` exits 0 on the current working tree where the *old* 33-pinned `.stderr` fixture would have failed (live count is 34). Per Spec Req 3 it also exits 0 after `git add` of #277/#278 — but note this is trivially true because the resolver enumerates candidates via a **working-tree glob** (`resolve_item.py:412`), so untracked #277/#278 are *already* in the count of 34 and staging is invisible to the resolver; the staged/unstaged equivalence is a consequence of count-agnosticism, not a meaningful state transition. No-op-freedom (Spec Req 2) is verified by the committed `test_ambiguous_structure_rejects_malformed` meta-test, which fails if the structural helper under-asserts (replacing the un-runnable manual `_format_candidates` mutation as the gate). All three fixture cases remain discovered (`--collect-only` lists `title_phrase_ambiguous`, `numeric_unambiguous`, `no_match`), the two de-pinned snapshot files are gone, and `no_match` stays byte-exact.
