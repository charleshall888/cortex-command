# Specification: Multi-step Complete phase + interactive feature worktree lifecycle

## Problem Statement

Today's lifecycle Complete phase fires before PR merge — it creates the PR and exits. But cleanup of long-lived feature worktrees (the new `interactive/{slug}` prefix introduced by epic #237's daytime-swap) is conceptually a post-merge action: it must wait until the PR is merged on GitHub and the user has no further use for the worktree. The natural model is for Complete to *not finalize* until the merge has happened — Complete becomes a multi-step phase with a mid-phase pause where the user goes to GitHub, merges, and re-invokes `/cortex-core:lifecycle complete <slug>` to finalize. This fixes the cleanup-trigger problem, exposes a latent ordering bug in morning-review's backlog-close path, and aligns the lifecycle's "Done" semantics with the industry convention (GitHub, Linear, Jira, GitLab all treat merge as the terminal event, not PR-open). Beneficiaries: every user of the lifecycle skill (cleaner mental model for "complete"); users of the new interactive worktree dispatch (worktrees are cleaned up at the right time, never destroyed with uncommitted work).

## Phases

- **Phase 1: Foundation** — primitive defects fixed (with all caller updates in lockstep), schemas extended, pre-existing ordering bug corrected, ADR landed. Independently bisectable; tests pass after this commit.
- **Phase 2: Multi-step Complete + interactive worktree lifecycle** — Complete becomes multi-step with merge-wait pause; `pr_opened` event registered with producer + consumers in the same commit; implement.md option 2 creates the worktree; PATH bootstrap migrated; statusline + kept-pauses inventory + tests updated.

11. **complete.md rewritten as multi-step phase**: `skills/lifecycle/references/complete.md` is rewritten with the following ordered step protocol: (1) run tests, (2) commit lifecycle artifacts, (3) push branch + create PR, (4) write `cortex/lifecycle/{slug}/pr.json` atomically (tempfile + `os.replace`), (5) emit `pr_opened` event, (6) exit with handoff message *"PR open at <url>; merge on GitHub, then re-run /cortex-core:lifecycle complete <slug> to finalize"*, (7) on re-invocation, state-aware route per the matrix in R12, (8) worktree cleanup (interactive prefix only, calls `cleanup_worktree(slug, branch=f"interactive/{slug}", force=False)`), (9) backlog write-back, (10) backlog index sync, (11) log `feature_complete` with `merge_anchor: "merge"`, (12) summarize + preserve lifecycle directory. **Acceptance**: `grep -cE "^Step [0-9]+|^## Step [0-9]+" skills/lifecycle/references/complete.md` returns ≥12; `grep -c "pr_opened" skills/lifecycle/references/complete.md` returns ≥1; `grep -c "merge_anchor" skills/lifecycle/references/complete.md` returns ≥1. **Phase**: Phase 2: Multi-step Complete + interactive worktree lifecycle

12. **Complete step 7 is idempotent + state-aware with explicit evaluation order**: each invocation reads `pr.json` (if exists) and queries `gh pr view <number> --json state,mergedAt --repo <pr.json.repo>` (if PR exists). Evaluation order is **strict**: (1) check events.log for `feature_wontfix` → if present, exit "lifecycle was wontfix'd at <ts>; nothing to complete (worktree cleanup skipped)" regardless of pr.json state. (2) Check events.log for `feature_complete` → if present, short-circuit to summary (no duplicate event, no re-cleanup, no second pr.json write). (3) Check `pr.json` existence:
   - **Absent**: probe `gh pr list --head "interactive/{slug}" --state all --json number,state,mergedAt --limit 5` for orphan-PR recovery (the case where step 3 created a PR but step 4 crashed before writing pr.json). If exactly one PR matches, retroactively reconstruct pr.json from the response and proceed to (4). If multiple match (slug-reuse case), surface the candidates and ask the user which to use (interactive); if zero match, run first-run path (steps 1-5).
   - **Present**: proceed to (4).
   (4) Query `gh pr view <number> --json state,mergedAt --repo <repo>`. Exit-code routing:
   - **Auth/network error** (`gh auth status` non-zero, or `gh pr view` exits non-zero with output matching network/auth patterns): exit with "PR state unknown; gh unauthenticated or network error; retry later. (Worktree retained.)"
   - **PR-not-found error** (`gh pr view` exits non-zero with output matching "Could not resolve to a PullRequest" or "GraphQL: not found"): exit with "PR <number> referenced in pr.json was not found on GitHub. The PR may have been deleted. Run `git worktree remove <path>` manually if appropriate, or restore the PR. (Worktree retained.)"
   - **`state=OPEN`**: exit with "PR open at <url>; merge first" message.
   - **`state=MERGED` (`mergedAt != null`) + dirty worktree** (`git status --porcelain` non-empty): exit with "uncommitted changes at <path>; resolve first" message.
   - **`state=MERGED` + clean worktree** + branch is local-ancestor of `origin/main`: continue to steps 8-12.
   - **`state=MERGED` + clean worktree** + branch NOT local-ancestor of `origin/main`: exit with "branch head is not in origin/main (possible squash with non-ancestor commit or fork-merge); refusing cleanup until verified. Run `git worktree remove <path>` manually to override."
   - **`state=CLOSED` (`mergedAt == null`, closed without merge)**: exit with "PR <url> was closed without merging. Either reopen and merge, run `git worktree remove <path>` manually to abandon, or invoke `/cortex-core:lifecycle wontfix <slug>` if appropriate. (Worktree retained.)"
   
   **Acceptance**: `tests/test_lifecycle_complete_state_routing.py` (new) exercises all 9 branches via fixtures (feature_wontfix, feature_complete-already, pr.json-absent-no-orphan, pr.json-absent-one-orphan, pr.json-absent-multi-orphan, OPEN, MERGED+dirty, MERGED+clean+ancestor, MERGED+clean+non-ancestor, CLOSED-unmerged, PR-not-found, auth-error) and asserts the correct exit path for each; `just test` exits 0. **Phase**: Phase 2: Multi-step Complete + interactive worktree lifecycle

- **Concurrency lock semantics** (`interactive.pid` writer, kill -0 liveness check): owned by #241. #239 only agrees on the per-machine `cortex/lifecycle/sessions/{slug}.interactive.pid` location and reads it during cleanup gating.
- **Inverse-direction overnight guard** (overnight preflight scans for live interactive worktrees): owned by #241 or a sibling. Out of scope here.
- **PR-creation hook for the Variant A flow** (#240's territory): #239 ships the `pr.json` write inside Complete step 4. If #240 needs to extend that path or wire alongside, the extension is owned by #240; #239 does not pre-build that extension surface.
- **No new lifecycle phase**: rejected R1 (new Submit phase) explicitly. Complete remains a single phase with multi-step protocol.
- **No automated merge-detection polling**: `hooks/cortex-scan-lifecycle.sh` does NOT add `gh pr view` polling at SessionStart. User's manual re-invocation of `/cortex-core:lifecycle complete <slug>` is the gate. Polling would add network latency, auth dependency, and sandbox noise to every session start.
- **No dashboard rendering changes**: dashboard's awaiting-merge sub-state rendering is deferred to a follow-up ticket (`docs/dashboard.md` is not touched here). Statusline + scan-lifecycle rendering is in scope; the dashboard FastAPI surface is not.
- **No daytime-autonomous removal**: the parallel epic concern (deleting `implement.md` §1a daytime block, removing daytime modules) is #246's territory.
- **No backlog status value added**: no new `pending-merge` or `submitted` status. Existing `in_progress → complete` flow is preserved; `status=complete` writes only at multi-step Complete step 9.
- **No retroactive rewrite of historical `feature_complete` events**: events emitted before this ticket ship without `merge_anchor`; readers default them to `"review"` per backwards-compat.

## Edge Cases

- **User merges PR via GitHub web UI before first /complete invocation**: First /complete runs steps 1-3 (creates PR — wait, step 3 fails because `gh pr create` refuses to create a second PR on the same branch). Recovery: step 7's case (3-Absent) invokes `gh pr list --head interactive/{slug}` orphan-PR probe and discovers the existing PR; if exactly one match (the user's web-merged PR), retroactively writes pr.json and routes to step 7's MERGED+clean path (or MERGED+dirty if worktree is dirty).
- **User invokes /complete twice rapidly with PR still open**: First invocation runs steps 1-6 (creates PR); second invocation reads pr.json + `gh pr view <number>` → `state=OPEN` → exits with same "merge first" message. No duplicate PR creation.
- **User merges, then re-invokes /complete, then a network error during `gh pr view`**: Step 7 catches the auth/network error pattern and exits "PR state unknown; retry later. (Worktree retained.)" Safer-by-default: retain worktree.
- **User merges with `gh pr merge --delete-branch` (deletes remote ref)**: `gh pr view <number> --repo <repo>` still works (PR-number-based, immutable). `state` field returns `MERGED`. Cleanup proceeds.
- **PR is deleted on GitHub after pr.json was written**: Step 7 catches "PR-not-found" error pattern (distinguished from transient network/auth errors by error-output regex) and exits "PR <number> referenced in pr.json was not found on GitHub. The PR may have been deleted. Run `git worktree remove <path>` manually if appropriate. (Worktree retained.)" Does NOT loop in "retry later."
- **PR was closed without merging**: `state=CLOSED` with `mergedAt == null` routes to "PR was closed without merging" message with three offered paths (reopen, manual cleanup, wontfix). Worktree retained.
- **`git merge-base --is-ancestor` fails (branch head not in main)**: indicates squash-merge with non-ancestor commit or fork-merge edge. Exit with "branch head is not in origin/main; refusing cleanup until verified." User can manually `git worktree remove` to override.
- **Recipe invoked from inside the target worktree**: refuse with "cd out of the worktree before running cleanup; current PWD is the worktree being removed." Hard guard via `realpath PWD` comparison; do not auto-cd.
- **`feature_wontfix` event in events.log**: Step 7 evaluation-order-step-1 detects wontfix and exits with "lifecycle was wontfix'd at <ts>; nothing to complete" — takes precedence over all pr.json/PR-state checks, including the pr.json-present case (a wontfix-after-PR-creation lifecycle skips cleanup; worktree is the user's to dispose of manually).
- **`feature_complete` already in events.log**: Step 7 evaluation-order-step-2 short-circuits to summary; no duplicate event, no re-cleanup, no second `pr.json` write.
- **Lifecycle has no `pr.json` but does have a PR (web-UI merge / step-4 crash)**: Step 7 case (3-Absent) probes `gh pr list --head interactive/{slug} --state all --json number,state,mergedAt --limit 5`. Single match → retroactively write pr.json and proceed. Multiple matches (slug-reuse) → surface the candidates to the user. Zero matches → run first-run path.
- **Slug reuse over time produces multi-PR ambiguity**: `gh pr list --head` returns >1 match. Surface the candidates with PR numbers, states, and merged_at timestamps; ask the user which to use (interactive recovery path).
- **`pr.json` references a repo that differs from current `origin`**: `gh pr view --repo <pr.json.repo>` dispatches against the recorded repo, not against `origin`. If the user reset their remote between PR creation and merge, step 7 still queries the correct repository. This is the load-bearing reason for the `repo` field in pr.json (added per critical-review reviewer B).
- **Multi-machine sync: cleanup invoked on machine B for a worktree created on machine A**: `git worktree list --porcelain` shows the worktree's recorded absolute path. If `path does not exist locally`, skip cleanup and warn; do NOT prune (would corrupt machine A's metadata).
- **Option 1 (no worktree) or Option 3 (bare feature branch, no worktree) features**: Step 8 detects no `interactive/`-prefix worktree exists for the feature; skips cleanup silently. Steps 9-12 run normally.
- **Mid-step crash (e.g., commit succeeds, PR creation fails)**: re-invocation reads state: commit happened (visible in `git log`), PR doesn't exist (no `pr.json`). Step 7 case (3-Absent) probes orphan-PR via `gh pr list --head`; if no match, re-runs step 3 (creates PR fresh).
- **Step 1 (tests) fails**: no commit, no PR, no pr.json, no event emitted. Re-invocation hits case (3-Absent) with no orphan PR → first-run path → tests re-run. Step-1 failure is idempotent.

## Technical Constraints

- **Destructive operations preserve uncommitted state** (`cortex/requirements/project.md:42`): Complete step 8 must skip cleanup on dirty worktree; `cleanup_worktree()` primitive must not silently force-remove.
- **Handoff readiness** (`project.md:13`): all step 7 routing decisions are agent-verifiable from `gh pr view` + `git status --porcelain` + `git merge-base` + events.log scanning. No human interpretation required.
- **Atomicity** (`pipeline.md:124-130`): `pr.json` writes use tempfile + `os.replace`.
- **Sandbox-safe worktree path**: `resolve_worktree_root()` defaults to `$TMPDIR/cortex-worktrees/{feature}` outside `.claude/` deny scope. Inherited by `create_worktree()`.
- **Single-chokepoint worktree primitive**: `create_worktree()` and `cleanup_worktree()` remain the single chokepoint; skills do not invoke `git worktree add/remove` directly.
- **Events-registry consumer rule** (`bin/.events-registry.md` audit gate): new `pr_opened` event ships with both statusline renderer and scan-lifecycle phase-label consumer in the same commit (Phase 2). `bin/cortex-check-events-registry --audit` must exit 0 at HEAD.
- **Plugin parity**: canonical edits in `skills/`, `hooks/`, `bin/`, `cortex_command/`; plugin mirrors auto-regenerate via pre-commit hook. New SessionStart hook registered in `plugins/cortex-core/hooks/hooks.json` (not `plugins/cortex-overnight/`) so cortex-core-only installations get the PATH bootstrap.
- **Cortex-shape gate on SessionStart hook**: new PATH bootstrap hook exits 0 silently unless `cortex/lifecycle/` exists in the resolved repo (mirroring `hooks/cortex-scan-lifecycle.sh:29` pattern). Prevents PATH mutation outside cortex contexts.
- **Two-staged commits bisectable**: Phase 1 commit passes `just test` independently before Phase 2 commit lands. R8 (`pr_opened` event registration) and all its consumers ship together in Phase 2, honoring the events-registry consumer rule at commit-grain (not just PR-grain).
- **No mid-session cwd polling**: `hooks/cortex-scan-lifecycle.sh` does NOT invoke `gh pr view` from SessionStart.

## Proposed ADR

### Proposed ADR: NNNN-multi-step-complete-and-interactive-worktree-lifecycle

Cortex's lifecycle Complete phase becomes a single multi-step phase rather than splitting into a new Submit phase plus Complete. The phase progresses through sequential steps including PR creation, a mid-phase merge-wait pause (the user merges on GitHub and re-invokes `/cortex-core:lifecycle complete <slug>` to finalize), and post-merge cleanup. This shape was chosen over four rejected alternatives: R1 (new Submit phase before Complete — earns ~25 files of churn for a named phase boundary that a mid-phase pause already affords); R2 (bimodal Complete — hides the boundary; weak alignment with the project's structural-enforcement direction); R3 (PR creation moved into Review's tail — breaks Review's explicit read-only contract); R4 (status-transitions-only — splits state across phase events.log and backlog frontmatter). The chosen shape also locks the deliberate WorktreeCreate-hook bypass for lifecycle-managed `interactive/{slug}` worktrees (the hook fires only on Claude Code's native `--worktree` flow; lifecycle's manual `git worktree add` from skill bypass is intentional and permanent) and the migration of the PATH bootstrap from the WorktreeCreate hook to a SessionStart hook registered in `plugins/cortex-core/hooks/hooks.json` with a `cortex/lifecycle/` shape gate so PATH is not mutated in non-cortex repos. Three-criteria gate met: hard-to-reverse (changes semantics of when `feature_complete` fires); surprising-without-context (re-runnable single phase is unusual); real-trade-off (gives up phase-boundary observability in exchange for ~25 files less churn).
