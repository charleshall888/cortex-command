# Plan: surface-built-but-merge-blocked-overnight

## Overview
Thread one optional `recoverable_branch` discriminator onto `OvernightFeatureStatus`, re-point only the genuine-merge-conflict terminus (`merge_result.conflict is True`) from `paused` to a recoverable `deferred` disposition carrying that branch, and teach the existing `deferred` consumers (write-back, live status, morning report, `[ZERO PROGRESS]` gate) to check `recoverable_branch` presence first — falling through to the unchanged question-deferral path when absent. The 2-way discriminator is disjoint **only while the sibling-cascade sweep stays deferred** (this MVP adds none): per the spec's "Discriminator scope … not asserted globally" constraint, the only `deferred`+`recoverable_branch=None` features produced here are genuine question-deferrals, so the fall-through is sound. A future task that lands a cascade sweep emitting `deferred`+`None` for never-built dependents (research Adversarial #4) MUST revisit every discriminator site — see Risks. No new status, no new event, no dashboard change.
**Architectural Pattern**: shared-state

## Outline

### Phase 1: Recoverable carrier & write-back foundation (tasks: 1, 2, 3)
**Goal**: Persist the `recoverable_branch` field end-to-end and make the backlog write-back record a recoverable disposition — the foundation every routing/consumer task depends on.
**Checkpoint**: `recoverable_branch` round-trips through `save_state`/`load_state`; `_map_results_to_state` sets it from a `features_deferred` entry; `_write_back_to_backlog(..., recoverable_branch=X)` writes `status: in_progress` + records the branch instead of `status: backlog`. Tasks 1–3 tests green.

### Phase 2: Genuine-conflict routing & live-view discrimination (tasks: 4, 5)
**Goal**: Re-point the conflict terminus to the recoverable disposition (sourcing the suffix-correct branch), preserve the systemic-breaker path for non-conflict failures, stop in-session re-dispatch, and stop the live `cortex-overnight status` view from mislabeling recoverable as failed.
**Checkpoint**: a `conflict=True` merge lands in `features_deferred` with the correct `recoverable_branch`, is not in `features_paused`, is not counted by `_count_pending`, and is not in the live `failed` bucket; a `conflict=False` systemic failure still pauses and still increments `systemic_pauses_in_batch`. Tasks 4–5 tests green.

### Phase 3: Morning-report surfacing & PR gating (tasks: 6, 7, 8, 9)
**Goal**: Stop the from-scratch-rebuild follow-up, positively surface recoverable features in the morning report (excluding them from the deferred-questions count), keep a built-but-merge-blocked session out of the `[ZERO PROGRESS]` label, and widen the canonical `deferred` definition — all while leaving the question-deferral surface byte-identical.
**Checkpoint**: no rebuild follow-up item for a recoverable feature; report body names `pipeline/<feature>` for it; exec-summary deferred count excludes it; `render_deferred_questions` output byte-identical to baseline; `[ZERO PROGRESS]` excludes recoverable home work; `pipeline.md` documents the sub-case. Tasks 6–9 tests green; `just test` green.

## Risks
- **Cross-phase coupling (Tasks 3 → 4)**: the routing change (Task 4) calls the recoverable write-back (Task 3), so Task 3 lands first even though the spec groups R6 under its Phase 2. This keeps every committed state correct (no transient commit where a recoverable feature is written `status: backlog`); the alternative — landing R1 first with the buggy write-back — was rejected to avoid a transient regression on the integration branch.
- **Provisional reuse, not mint**: this MVP reuses `deferred` + a `recoverable_branch` discriminator rather than minting a `merge_blocked` status. Per the spec's Non-Requirements and research, the durable reuse-vs-mint decision is deliberately deferred until genuine-home-merge-block frequency is measured. A future occurrence may justify minting a status (the ADR moment). Revisit if conflicts prove frequent.
- **`render_status()` refactor (Task 5)**: extracting a bucketing helper is a small refactor of print-only code to make R5 testable without stdout-scraping. Low risk, but it touches the operator's live `cortex-overnight status` output — verify the rendered sections are unchanged for non-recoverable features.
- **Per-commit safety is order-independent, not order-dependent (no regression window)**: the overnight scheduler commits per-task by dependency wave, so the plan's three authored phases are NOT commit boundaries — Tasks 5/6/7/8 (each `Depends on [1]`) land in the same wave, and Task 4 (`Depends on [1,2,3]`) lands in the wave after them. But the rollout is safe regardless of interleaving, not merely by that ordering: a feature carries `recoverable_branch` only once BOTH Task 4 (routing emits the entry key) AND Task 2 (carrier persists it) have landed, so until then every consumer's `recoverable_branch is not None` check is a dormant no-op and every conflict still routes to `paused` (old behavior, still tested green). There is therefore no commit where a merge-blocked feature is routed recoverable but still gets a rebuild follow-up or a `[ZERO PROGRESS]` label. If a future edit ever loosens Task 4's `[1,2,3]` edges, preserve this invariant (producer fields populated only after carrier + routing both land).
- **Discriminator is safe only while sibling-cascade stays deferred**: the 2-way `recoverable_branch is not None` check at every consumer (Tasks 5/6/7/8) is disjoint from the question-deferral path only because no `deferred`+`recoverable_branch=None` *never-built dependent* is produced (this MVP adds no sibling-cascade sweep; dependents stay `pending`, resolving as today). A future task that lands `sweep_blocker_blocked_dependents` (research Adversarial #4) would emit exactly that third state, which the current consumers would mis-route to the question-deferral path (render nowhere, mislabel as "questions need answers"). That task MUST upgrade the discriminator to 3-way (built+branch / question+file / blocked+neither) — it cannot be added blind against this plan's 2-way checks.
- **Deferred scope (acknowledged, not bugs)**: sibling-cascade for dependents of a merge-blocked blocker, the web dashboard surface, cross-session resume (`determine_resume_point` treats only `merged` as completed), and the outer ZERO-PROGRESS gate are all explicitly out of scope per the spec, gated on a real observed occurrence.

## Acceptance
An overnight home session where a feature builds but its merge hits a genuine conflict (`merge_result.conflict is True`) results in: the feature persisted `deferred` with `recoverable_branch` set to its suffix-correct `pipeline/<feature>` branch (or `None`, never a bare reconstruction, when unresolvable); excluded from `_count_pending` so it is not re-dispatched in-session; its backlog item written `status: in_progress` recording the recovery branch (not re-queued `backlog`, no rebuild follow-up); the morning report and live `cortex-overnight status` positively listing it as built/merge-blocked/recoverable and excluding it from the deferred-questions count; and the session not labeled `[ZERO PROGRESS]` when integration commits exist — while every genuine question-deferral path (question files, "questions need answers" count, `render_deferred_questions` output, `Retry deferred` follow-up, non-conflict systemic pauses) remains byte-for-byte unchanged. `just test` exits 0.
