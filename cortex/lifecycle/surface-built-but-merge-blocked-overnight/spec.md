# Specification: surface-built-but-merge-blocked-overnight

## Problem Statement

When an overnight home-repo feature builds correctly but its merge hits a genuine conflict that cannot be auto-integrated, the runner routes it to `paused` — which auto-retries every round, churns into a `circuit_breaker (stall)`, writes the feature back to the backlog as `status: backlog` (re-queuing it for a from-scratch rebuild even though its work is intact on `pipeline/<feature>`), and contributes to a `0/N` + `[ZERO PROGRESS]` PR. The operator is told the session made zero progress and the built work is silently stranded. This feature (the split-out Phase 2 of #281, which fixes the common *wrong-tree* cause separately) makes the genuine-conflict case surface as **"built, merge-blocked, recoverable on `pipeline/<feature>`"** — distinct from "never built" and from "awaiting a human answer" — so it stops auto-retrying within the session, is not rebuilt from scratch, and is not labeled zero-progress. Scope is the **minimum-viable** slice chosen after research: it reuses the existing `deferred` status plus a new `recoverable_branch` discriminator (no new status), and defers the sibling-cascade, web-dashboard, cross-session-resume, and ZERO-PROGRESS-outer-gate work until a genuine home merge-block is actually observed (frequency is currently unmeasured).

## Phases
- **Phase 1: Routing, carrier & recoverable state** — re-point the genuine-conflict terminus to a recoverable disposition, persist `recoverable_branch` through the carrier path that actually writes feature status, source the branch correctly, and stop in-session auto-retry without mislabeling the live status view.
- **Phase 2: Write-back & surfacing** — stop the from-scratch-rebuild write-back and the spurious rebuild follow-up for the recoverable sub-case, and positively surface "built, merge-blocked, recoverable on `pipeline/<feature>`" in the morning report without the `[ZERO PROGRESS]` / "questions need answers" mislabels, with question-deferral behavior unchanged.

## Requirements

1. **Re-point the genuine-merge-conflict terminus from `paused` to recoverable.** In `_apply_feature_result` (`cortex_command/overnight/outcome_router.py`), the merge-failure `else` branch (`:666-694`) currently routes every merge failure to `features_paused`. When the failure is a genuine merge conflict (`merge_result.conflict is True`) — the point at which a built home feature's work cannot be integrated — the feature instead routes to a recoverable disposition: it is appended to `features_deferred` with a `recoverable_branch` key (R3) and given the recoverable write-back (R6), NOT `features_paused` + `_write_back_to_backlog(paused)`. Non-conflict merge failures (`merge_result.conflict is False`, including systemic errors) are UNCHANGED: they remain `paused`, still increment `consecutive_pauses`/`systemic_pauses_in_batch` (`:668-671`), and still feed the systemic circuit breaker. The `recover_test_failure` terminus at `:1116-1140` is explicitly OUT of scope — it handles post-merge *test* failures, a different failure mode. Acceptance: a test where a home feature's merge returns `conflict=True` asserts the feature lands in `features_deferred` with `recoverable_branch` non-empty and is NOT in `features_paused`; a companion test asserts a non-conflict merge failure (`conflict=False`, `error ∈ _SYSTEMIC_ERROR_TYPES`) still lands in `features_paused` AND still increments `systemic_pauses_in_batch`. `python3 -m pytest cortex_command/overnight/tests/test_outcome_router.py -k "merge_conflict_recoverable or non_conflict_still_paused"` exits 0 (pass if exit code = 0). File grounding: `outcome_router.py:666-694`, `:1019-1022`. **Phase**: Routing, carrier & recoverable state.

Acceptance: a test drives the R1 routing output through `_map_results_to_state` and asserts the persisted `OvernightFeatureStatus.recoverable_branch` equals the routed branch (non-None); a round-trip test asserts `save_state`→`load_state` preserves it and that a state JSON lacking the key loads with `recoverable_branch is None`. `python3 -m pytest cortex_command/overnight/tests/test_map_results.py cortex_command/overnight/tests/test_state.py -k "recoverable_branch"` exits 0 (pass if exit code = 0). ## Non-Requirements
- Does **not** mint a new feature status. It reuses `deferred` + the `recoverable_branch` discriminator. The durable reuse-vs-mint decision is intentionally **deferred** until the genuine-home-merge-block frequency is measured (research found it currently unmeasured and gated on #281 landing). The #284 ticket's claim that "#281 chose reuse via ADR 0007" is a verified misattribution — that decision is open and this MVP makes a provisional, reversible choice.
- Does **not** touch the `recover_test_failure` terminus (`outcome_router.py:1116-1140`). That path handles post-merge *test* failures, not merge conflicts; re-pointing it (as an earlier draft did) would over-capture six retryable failure types (dirty-base, repo-root, no-worktree, circuit-breaker-no-commits, exhausted, bare-exception) that must stay `paused`/retryable.
- Does **not** change the `batch_plan._map_pipeline_to_overnight` error-substring inference (`batch_plan.py:157-168`). That is the CLI/pipeline-state mapper path; home overnight features route in-process through `_apply_feature_result`, so the substring inference is not on the recoverable path. Out of scope.
- Does **not** add a sibling-cascade sweep for dependents of a merge-blocked blocker, nor a "blocked-but-never-built" surface for those dependents. Reachable only via a multi-feature dependency chain whose head *also* genuinely conflicts (third-order rare). Deferred to a follow-up.
- Does **not** change the web dashboard (`dashboard/alerts.py`, `templates/feature_cards.html`, `templates/session_panel.html`). MVP surfaces via the morning report and the `cortex-overnight status` CLI (R5) only.
- Does **not** fix cross-session resume. `determine_resume_point` (`state.py:713-745`) treats only `merged` as completed, so a recoverable `deferred` feature re-enters the pending set on a resumed session — identical to today's `paused` behavior. Deferred to a follow-up; see Edge Cases.
- Does **not** change the outer ZERO-PROGRESS gate. A session whose *only* outcome is a recoverable feature still produces no PR (its commits live on `pipeline/<feature>`, never the integration branch, so `commit_count == 0`); the operator's surface for that case is the morning report. Deferred.
- Does **not** emit a new event (reuses `feature_deferred` / the persisted field) and does **not** write an ADR (the provisional reuse is not a durable architectural commitment).
- Does **not** change the repair-attempt cap or the circuit-breaker constants.

## Edge Cases
- **`recoverable_branch` unresolvable** (`ctx.worktree_branches` empty/absent at the routing site): persist `None` and surface a degraded "built, merge-blocked (recovery branch unknown)" message — never a bare `pipeline/<name>` reconstruction (R3).
- **Resumed session**: a recoverable `deferred` feature re-enters `pending_features` (same as today's `paused`; `determine_resume_point` treats only `merged` as completed) and `ctx.worktree_branches` is re-resolved on the new round — which can yield a *different* suffix branch. The `recoverable_branch` persisted at first encounter is authoritative for the report; the cross-session re-merge/terminality behavior is the deferred resume limitation, not fixed here.
- **Merge-blocked feature with intra-session dependents**: the dependents stay `pending`. Because the recoverable feature is now terminal-`deferred` (excluded from `_count_pending`), a session whose only remaining non-merged work is the recoverable feature now reaches a clean `pending == 0` exit instead of churning to `circuit_breaker (stall)` — a deliberate improvement (see Changes to Existing Behavior), not behavior-neutral. The dependents themselves get no new sibling sweep (deferred); they resolve exactly as today's features blocked by a non-`merged` blocker.
- **Genuine question-deferral** (`recoverable_branch is None`): every consumer the spec touches checks `recoverable_branch` presence FIRST and falls through to the existing question-deferral path, so question-file rendering, the "questions need answers" count, the `Retry deferred` follow-up, and the `deferred→backlog` write-back are unchanged for question-deferrals.

## Changes to Existing Behavior
- MODIFIED: the genuine-merge-conflict terminus (`_apply_feature_result:666-694`, guarded on `merge_result.conflict is True`) routes `paused` → recoverable `deferred` + `recoverable_branch`. **This is a deliberate behavior change, not behavior-neutral**: the round loop now reaches a clean `pending == 0` exit for a conflict-only session instead of auto-retrying into a `circuit_breaker (stall)`. Non-conflict merge failures are unchanged (still `paused`, still feed the systemic breaker).
- ADDED: `recoverable_branch` optional field on `OvernightFeatureStatus`, threaded through the `features_deferred` carrier dict and `map_results._map_results_to_state`.
- MODIFIED: `status.py` live bucketing — a recoverable feature is no longer shown as `failed`.
- MODIFIED: `_write_back_to_backlog` for a recoverable feature → `status: in_progress` + recovery branch recorded (no from-scratch rebuild).
- MODIFIED: `create_followup_backlog_items` no longer emits a rebuild item for the recoverable sub-case.
- MODIFIED: morning report → positive recoverable surface; executive deferred-questions count excludes recoverable features; `[ZERO PROGRESS]` inner gate excludes sessions with recoverable home work.
- MODIFIED: `cortex/requirements/pipeline.md` `deferred` definition documents the recoverable sub-case.

## Technical Constraints
- **Correct terminus.** The genuine-conflict terminus is `_apply_feature_result:666-694` guarded on `merge_result.conflict` (`merge_feature` detects+classifies+aborts a conflict and returns `conflict=True`, `merge.py:269-285`; the caller diverts conflicts at `outcome_router.py:1019-1022`). The `:1116-1140` `recover_test_failure` terminus is a different failure mode and is out of scope.
- **Correct carrier.** Feature status is written to `OvernightState` by `map_results._map_results_to_state` from the `BatchResult.features_*` dicts — NOT by `update_feature_status` at the routing site. `recoverable_branch` must be threaded through the `features_deferred` dict and that mapper, or it is added to the dataclass but never populated.
- **Correct provenance.** `recoverable_branch` = the in-scope `actual_branch` (`ctx.worktree_branches`), never `merge_start.branch` (log-only, absent from `MergeResult`) nor a bare `f"pipeline/{name}"` reconstruction (drops `-2`/`-N` suffixes).
- **Discriminator scope.** The recoverable path is distinguished by `recoverable_branch is not None` at the consumers this spec touches (R1/R2/R5/R6/R7/R8/R9). The untouched `deferred` producer (`batch_plan` CLI mapper) and consumer (`determine_resume_point`) are out of scope with rationale above — the disjointness guarantee is scoped to question-deferrals on the in-process home path, not asserted globally.
- **Systemic-breaker preserved.** Only `merge_result.conflict is True` becomes recoverable; non-conflict/systemic merge failures stay `paused` and keep feeding `systemic_pauses_in_batch` (`outcome_router.py:668-671`).
- **Atomicity & backward-compat.** `update_feature_status`/`map_results` do the in-memory mutation; persistence stays atomic via `save_state` (`asdict` + tempfile + `os.replace`, `state.py:388-431`). The new optional field defaults to `None`, so older serialized states load unchanged.
- **`deferred` definition honesty.** Reusing `deferred` widens its canonical `pipeline.md` definition to include the recoverable sub-case (R10).
