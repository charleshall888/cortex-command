# Specification: Critical-review sentinel gate excludes async reviewers that drop READ_OK from final message

**Backlog:** #376 · **Complexity:** complex · **Criticality:** high

> Revised after the specify-phase critical review (4 parallel reviewers, 0 A-class findings). The review's central improvement — layering the re-hash in the CLI command wrapper rather than the pure verifier functions — is incorporated below; it keeps the pure text-parsers, their unit tests, and `_VERIFIER_SITE_RE` unchanged, and resolves the telemetry/parity blast-radius concerns.

## Problem Statement

The critical-review drift gate excludes any reviewer whose final message lacks the `READ_OK` sentinel and, when all reviewers are excluded, invalidates the whole pass as "drift or Read failure detected" — even when the artifact provably never changed. Research established this is **systematic, not incidental**: reviewers are dispatched via the Agent/Task tool, which returns only a sub-agent's *final message*, so a sentinel emitted early (as the prompt instructs — "before the first `## ` heading") is structurally discarded when the reviewer concludes in a later turn. Telemetry confirms it: **88 `sentinel_absence` events across 33 lifecycles since the day after the prior #229 fix, on both model tiers, every one `reason: absent` — never a real SHA mismatch**; item 373 alone hit 100% reviewer exclusion on two same-day passes. The gate conflates "the self-report was dropped" (benign, the file is fine) with "the artifact drifted" (real), because it has no way to check the file itself. The fix removes that conflation by measuring drift directly: the sentinel becomes an **advisory** read-attestation, and the gate's own **re-hash of the pinned artifact becomes the authoritative check for the absent case** — while genuine persistent drift still hard-fails.

## Phases

- **Phase 1: Re-hash gate + advisory telemetry** — the CLI gate wrappers re-hash the pinned artifact when the sentinel is absent, passing on a stable file and hard-failing only on confirmed drift; a distinct advisory event keeps the exclusion telemetry meaningful, with the phantom-lifecycle consumer kept in lock-step.
- **Phase 2: Bug-seeding doc/prompt correction + ADR** — delete the language that caused and would re-seed the bug, reframe the sentinel as advisory, and record the contract change as an ADR.

## Requirements

1. **The re-hash disambiguation is layered in the CLI command wrappers, not the pure verifiers.** `_cmd_check_artifact_stable` and `_cmd_check_synth_stable` gain an **optional, backward-compatible** `--artifact-path` argument (the `resolved_path` from `prepare-dispatch`): on an `absent` result from the pure verifier, when `--artifact-path` is provided, re-hash that path (reusing `sha256_of_path`) to decide advisory-pass vs drift-exclusion; when it is **omitted, behavior is exactly today's** (`absent` → exit 3 exclusion). Optional (not required) is deliberate — it keeps every existing CLI caller and test working unmodified, avoids an atomic doc↔argparse flip, and stays fail-closed (a caller that omits the arg degrades to today's safe exclusion, never a false pass); the fail-loud property lives in the re-hash mismatch branch (R3), not in argparse requiredness. The pure `check_artifact_stable`/`check_synth_stable` text-parsers keep their exact return contracts (`("absent", None)` etc.), their three-substring docstring honesty caveats, and stay filesystem-free — so `_VERIFIER_SITE_RE` and their unit tests need no change. Acceptance: `cortex-critical-review check-artifact-stable --help` and `check-synth-stable --help` each list `--artifact-path`; `grep -nE "add_argument.*artifact-path" cortex_command/critical_review/__init__.py` returns ≥ 2 matches; invoking either subcommand **without** `--artifact-path` on a sentinel-free input still yields today's exclusion (exit 3); `check_artifact_stable`'s signature is unchanged (`grep -n "def check_artifact_stable" cortex_command/critical_review/__init__.py` shows no path parameter). **Phase**: Re-hash gate + advisory telemetry. *(File: `cortex_command/critical_review/__init__.py`)*

2. **Absent sentinel + provably-stable artifact → advisory pass (exit 0), not exclusion.** When the pure verifier returns `absent` AND re-hashing `--artifact-path` yields the expected SHA, the wrapper exits **0** (folding into the existing pass exit code, no new exit code) and the reviewer/synth is NOT excluded — the pass proceeds. Acceptance: a test invoking `_cmd_check_artifact_stable` with sentinel-free input and an `--artifact-path` that re-hashes to the expected SHA asserts exit 0 and no exclusion. **Phase**: Re-hash gate + advisory telemetry. *(Files: `cortex_command/critical_review/__init__.py`, `tests/test_critical_review_sentinel_window.py`)*

3. **Absent sentinel + drifted/unreadable artifact → exit 3, unchanged severity.** When the verifier returns `absent` AND re-hashing yields a different SHA (or the file is unreadable/deleted), the wrapper excludes exactly as today (exit 3; total-failure eligible) and emits the `sentinel_absence` event — which now fires only on *confirmed* drift. Real persistent-drift detection is preserved. Acceptance: tests for (a) sentinel-free input + drifted `--artifact-path` → exit 3 + `sentinel_absence`, and (b) sentinel-free input + unreadable `--artifact-path` → exit 3. **Phase**: Re-hash gate + advisory telemetry. *(Files: `cortex_command/critical_review/__init__.py`, `tests/test_critical_review_sentinel_window.py`)*

The three existing pure-verifier unit tests that assert `("absent", None)` (`test_sentinel_absent_returns_absent`, `test_sentinel_in_evidence_quote_past_window_returns_absent`, and the line-51 branch of `test_window_size_default_is_50`) stay green because the pure verifier is unchanged; the new absent→advisory behavior is covered by new wrapper-level tests (R2/R3). Because `--artifact-path` is optional, the existing CLI-invoking tests (`test_critical_review_phantom_guard.py`, `test_variant_a_writer_sites_baseline.py`) also stay green unmodified — they invoke the wrappers without the arg and get today's behavior. ## Non-Requirements

- **No change to `fallback-reviewer-prompt.md` or `competing-plans.md`.** Both are ungated (the Step 2c total-failure fallback proceeds directly to Step 3 with no 2c.5 check per `verification-gates.md:84`; `competing-plans.md`'s sentinel has no gate at all), so neither can produce #376's false-exclusion symptom. Editing their sentinel prose would be a no-op for this bug.
- **No change to the ad-hoc `<path>`-arg standalone path.** That invocation form "skips verification entirely" (`verification-gates.md:5`) — the gate wrappers are never called for it — so the re-hash fix neither runs on nor needs to reason about the ad-hoc snapshot.
- **The fallback's inverse gap is a separate follow-up.** That the total-failure fallback surfaces potentially-drifted output *unchecked* to Step 3 is a real but distinct concern — capture it as a new backlog item, not part of this fix.
- **No Shape B cross-agent re-attestation retry.** A fresh re-attester is also async (final-message-only) and can drop the sentinel again, moving the false-failure one round later and adding retry tax; re-hash achieves the same end deterministically in-process.
- **No Shape C structured-envelope attestation** (SHA in the reviewer's `<!--findings-json-->` envelope). Reviewer-only (fallback/synth have no envelope) and still self-report-dependent; a candidate follow-up, not this fix.
- **The sentinel is not removed, nor relocated to the final message.** It stays as an advisory read-attestation; because re-hash is now authoritative for the absent case, there is no need to fight the async-delivery model by relocating emission (which is counterproductive on the reviewer path anyway — it would push the sentinel past the 50-line window).
- **No new pure-verifier status literal, no new exit code, and no code-fence-aware parsing** (the latter inherited from #229).

## Changes to Existing Behavior

- **MODIFIED:** the CLI gate wrappers (`_cmd_check_artifact_stable`/`_cmd_check_synth_stable`) — bare sentinel-absence was mapped to exit 3 exclusion + `sentinel_absence`; now an `absent` verifier result is re-hashed: stable → exit 0 + `sentinel_advisory`; drifted/unreadable → exit 3 + `sentinel_absence` (unchanged severity).
- **MODIFIED:** the fail-closed total-failure contract — for the absent case, its authority moves from "the reviewer's self-reported sentinel SHA" to "the wrapper's re-hash of the pinned artifact." (ADR-worthy — see Proposed ADR.) Post-fix, `sentinel_absence` means *confirmed* drift, narrowing its historical meaning (registry rebaseline note, R5).
- **ADDED:** an optional, backward-compatible `--artifact-path` CLI arg on both gate subcommands, a re-hash step reusing `sha256_of_path` (fires on `absent` only when the arg is provided), and the `sentinel_advisory` event (added to `_TELEMETRY_ONLY_EVENT_TYPES`).
- **MODIFIED:** `verification-gates.md` / `reviewer-prompt.md` / `synthesizer-prompt.md` — corrected mental model ("final message", not "raw stdout") and sentinel reframed as advisory.
- **UNCHANGED (deliberately):** the pure `check_artifact_stable`/`check_synth_stable` text-parsers, their return contracts, docstring honesty caveats, `_VERIFIER_SITE_RE`, and the three `("absent", None)` unit tests.

## Edge Cases

- **Ad-hoc `<path>`-arg standalone form:** the gate wrappers are not invoked at all ("skips verification entirely", `verification-gates.md:5`), so there is no re-hash to perform and no `source_path`/snapshot reasoning required. The re-hash fix is scoped to the auto-trigger lifecycle flow (`--feature`, live realpath `resolved_path`), which is exactly where the bug fires and where re-hash detects genuine drift.
- **Transient drift, single- and mixed-cohort (change-then-restore within the review window):** if the file goes A→B→A, gate-time re-hash sees the restored (matching) SHA and passes — admitting findings produced against the intermediate version, and (in a fanned-out review) admitting both the A-reading and B-reading cohorts into the single-artifact synthesizer contract with no per-finding provenance. This is the accepted new boundary of the fail-closed guarantee — documented, not engineered against. It is already unprotected today (every failing case is `absent`, so the read-time sentinel guard is not firing), and a deliberate edit-then-revert to a *byte-identical* SHA within one window (format-on-save+lint-revert, autosave, edit-then-undo) still requires the round-trip to land on the exact pinned bytes — a narrow residual, accepted rather than closed.
- **Gate-time vs read-time semantics:** re-hash certifies the file's state *at gate time* (after all reviewers return), not each reviewer's read-time state; it fires only on the `absent` branch, as a tiebreaker between benign omission and drift. A drift that lands after reviewers read but before the gate is treated as drift and hard-fails — consistent with, and no stricter than, today's absent-path behavior (which already hard-fails).
- **Sentinel present + matching SHA (happy path):** unchanged — passes on the sentinel; re-hash never fires.
- **File unreadable/deleted at gate time on absence:** cannot prove stability → treated as drift → exit 3.
- **Adversarial quoted-decoy sentinel:** OK-first-SHA precedence and column-0 anchoring live in the unchanged pure verifier; re-hash fires only when the verifier already returned `absent`, so decoy defenses are untouched.

## Technical Constraints

- **Reuse `sha256_of_path()`** (`cortex_command/critical_review/__init__.py:287`) for the re-hash — do not add a second hashing primitive.
- **Layering:** the re-hash and advisory/drift decision live in the `_cmd_*` wrappers, keeping the pure verifiers filesystem-free and their honesty caveats accurate. This is the design choice that keeps `_VERIFIER_SITE_RE`, the gate-class parity test, and the three `("absent", None)` unit tests green.
- **`resolved_path` for the auto-trigger flow** is the live artifact realpath (drift-detecting); it is bound in Step 2a.5 and must be threaded as `--artifact-path` into the gate calls in `verification-gates.md`.
- **Telemetry consumer coupling:** the new `sentinel_advisory` event MUST be added to `_TELEMETRY_ONLY_EVENT_TYPES` (`common.py:604`); omitting it regresses `is_phantom_lifecycle_dir` (`common.py:672`, subset test) so telemetry-only dirs carrying the advisory event escape phantom cleanup and are treated as real research-phase lifecycles.
- **Events registry is additive-only:** add `sentinel_advisory` plus a dated rebaseline note for `sentinel_absence`'s narrowed meaning in `bin/.events-registry.md`; do not rename `sentinel_absence`, and do not model the advisory outcome as a `reason` value on `sentinel_absence` (that emits a forbidden `sentinel_absence` row).
- **Pinned-literal / fixture surface:** the total-failure literal and per-section exit markers (`test_critical_review_reference_pins.py`) stay byte-identical; only the CLI-contract example gains `--artifact-path`. New fixtures (e.g. "absent sentinel + stable path") are added with a sibling `.meta.json`, not substituted (fixture README: no casual re-baselining).
- **Dual-source:** edit canonical `skills/critical-review/**` and `cortex_command/critical_review/__init__.py` (+ `common.py`) only; `plugins/cortex-core/` mirrors regenerate via the pre-commit hook. `cortex-critical-review` is a `pyproject.toml` console-script, not a `bin/cortex-*` script — no bin-mirror concern.
- **MUST-escalation policy:** the prompt/doc changes REMOVE load-bearing prose; they add no new MUST language, so no `effort=high`-failed evidence artifact is required (and the 88-event backdrop would supply it regardless).

## Open Decisions

None. The three choices the critical review flagged are resolved here: (1) the re-hash lives in the CLI wrappers, so no new pure-verifier status literal is minted (R1/R7); (2) advisory-clean folds into exit 0, no new exit code (R2); (3) the new `--artifact-path` arg is **optional/backward-compatible** (R1) — the plan-phase critical review showed `required` maximizes rather than minimizes blast radius (atomic doc↔argparse flip, a cross-commit crash window, and churn of two unrelated tests), while optional keeps every existing caller working and stays fail-closed. Event-token identity is fixed as `sentinel_advisory` (R5).

## Proposed ADR

### Proposed ADR: 0027-gate-time-rehash-is-authoritative-drift-check

**Context.** The critical-review drift gate relied on each async reviewer emitting a `READ_OK: <path> <sha>` sentinel and treated its absence as drift. Because the Agent/Task tool returns only a sub-agent's final message, early-emitted sentinels were systematically discarded (88 `sentinel_absence` events / 33 lifecycles, all `reason: absent`, zero real mismatches), so clean passes were routinely invalidated. **Decision.** Demote the read-sentinel to an advisory read-attestation and, on the `absent` branch, make the gate wrapper's own re-hash of the pinned artifact the authoritative check: absence with a stable re-hash passes (exit 0, `sentinel_advisory`), absence with a drifted or unreadable re-hash hard-fails (exit 3, `sentinel_absence`), and a surviving sentinel with a mismatched SHA still hard-fails. The re-hash is layered in the CLI wrapper so the pure verifier and its contracts are untouched. **Trade-off.** Drift authority for the absent case moves from agent self-report to direct measurement — deterministic and covering every gated site uniformly — at the cost of a narrow transient-drift blind spot (change-then-restore-to-identical-SHA within one window, including mixed-cohort admission) that is accepted and documented rather than engineered against. This mirrors ADR-0015's split of a benign "could-not-run" outcome from a real failure. Hard-to-reverse (sibling gate sites, telemetry semantics, and downstream consumers depend on "total-failure = confirmed drift"), surprising-without-context (a maintainer would assume the sentinel is the drift authority), and a real trade-off — clearing the ADR three-criteria gate.
