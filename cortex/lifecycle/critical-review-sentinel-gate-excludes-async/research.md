# Research: Critical-review sentinel gate excludes async reviewers that drop READ_OK from final message

**Backlog:** #376 · **Complexity:** complex · **Criticality:** high

**Clarified intent:** Fix the critical-review sentinel gate so a review pass with **zero artifact drift** is not tripped into total-failure merely because async reviewer agents omitted the `READ_OK` sentinel from the *final message* the orchestrator receives.

**Prior art in this exact gate:** This is architecturally **round two** of the same over-fire class. Backlog #229 / lifecycle `critical-review-sentinel-gate-relax-first` (complete) relaxed the *positional* strictness — from "sentinel on literal line 1" to "anywhere in the first 50 lines" (`window_lines=50`) with OK-first-SHA precedence. #376 is a **different, orthogonal** failure at the same gate: the sentinel is *absent entirely* from the captured text, not merely mispositioned. The #229 artifact trail (`cortex/lifecycle/critical-review-sentinel-gate-relax-first/{spec,plan,review}.md`) is a strong structural template for this spec.

---

## Root-Cause / Async-Delivery Analysis

**The failure is SYSTEMATIC, not intermittent to item 373** — settled decisively by telemetry.

**What the orchestrator captures.** Reviewers are dispatched via the interactive `Agent`/Task tool (`docs/internals/sdk.md` "Path A"), whose contract returns **only the sub-agent's single final message** — not a transcript, and not per-turn `AssistantMessage` granularity (that is Path B / SDK streaming, which critical-review does *not* use). `verification-gates.md:27` tells the orchestrator to "Write each reviewer's **raw stdout** to a tempfile" — but for a Task-dispatched agent there is no stdout; "raw stdout" stands in for whatever text the tool returns as the final message. That blob (its first 50 lines) is the entirety of what `check_artifact_stable` ever sees.

**The mechanism.** `reviewer-prompt.md:17` tells each reviewer to emit `READ_OK: <path> <sha>` "before the first `## ` heading — preamble prose before it is fine" — i.e. **early**, assuming one continuous message. When a reviewer emits `READ_OK` in an early turn, then makes further tool calls (grep/read to ground its angle-specific quotes), then concludes in a *later* turn with `## Findings:` + the JSON envelope **without repeating the sentinel**, the early sentinel lives in a non-final turn that is structurally discarded. `check_artifact_stable` finds no sentinel → returns `("absent", None)` → exit 3 → total-failure.

**The decisive evidence.** Filtering `"event":"sentinel_absence"` across every `cortex/lifecycle/*/events.log`:
- **88 events across 33 distinct lifecycles**, spanning **2026-05-18** (the day *after* #229's window fix closed) through 2026-07-13.
- Both model tiers (sonnet and opus).
- **Every one is `reason: "absent"`** — never `sha_mismatch`, never `read_failed`. A clean-read-but-lost-sentinel signature, not drift.
- Item 373's own feature (`build-the-verb-completion-composition-wrapper`) hit **100% reviewer exclusion on two separate passes the same day** (4/4 opus at 19:18Z; 2/2 sonnet at 20:13Z).

Two model tiers, two independent same-day passes, and a two-month/33-feature backdrop rule out "373 was unlucky." This is a systemic property of the async-delivery + early-emission design.

**Undecidable from static artifacts:** whether `READ_OK` was emitted in an earlier discarded turn (mechanism *a*) vs never emitted anywhere (mechanism *c*, instruction-skipping under a heavily loaded prompt). The pure-`absent`/never-`mismatch` signature and the "early within a single message" prompt strongly favour *(a)*. Distinguishing them needs a runtime full-transcript capture of one live dispatch — **but it does not change the recommended fix direction** (a gate-side re-hash is robust to either mechanism).

## Codebase Analysis

**Gate code — `cortex_command/critical_review/__init__.py`:**
- `check_artifact_stable()` (L398–456): scans `"\n".join(output.splitlines()[:window_lines])` (L445, `window_lines=50`) with `_REVIEWER_OK_RE = ^READ_OK: (\S+) ([0-9a-f]{64})\s*$` and `_REVIEWER_FAILED_RE` (L394–395, both `re.MULTILINE`, column-0-anchored). Precedence: first `READ_OK` whose SHA == expected → `("ok", sha)`; else any `READ_FAILED` → `("read_failed", reason)`; else any `READ_OK` (wrong SHA) → `("mismatch", sha)`; else → `("absent", None)`. **The `absent` branch is #376's failure mode: zero information about actual drift, no independent way to learn the artifact was unchanged.**
- `check_synth_stable()` (L350–386): synth sibling, single `_SYNTH_RE` (L347), **searches the whole output — no window**. Shares the final-message-omission risk but *not* the 50-line truncation risk.
- `_cmd_check_artifact_stable` (L677–748) / `_cmd_check_synth_stable` (L613–674): CLI wrappers. Non-`ok` → `record-exclusion` reason, writes `EXCLUDED {reason}`, appends `sentinel_absence` event, exit **3** (or exit **4** when the write-guard skips telemetry — a benign pass). **These functions currently receive only the reviewer *text* (`--input-file`) and `--expected-sha`; they are NOT given the artifact path, so they cannot re-hash today.**

**Existing patterns any fix must preserve:**
- **OK-first-SHA precedence** defends against adversarial quoted-decoy sentinels (`test_multiple_sentinels_first_matching_sha_wins`, fixture `case-adversarial-quoted-sha`).
- **Column-0 anchor** rejects blockquoted `> READ_OK:` (`test_blockquoted_sentinel_is_rejected`).
- **`# gate-class: {security|hygiene|advisory}`** annotation within 3 lines of every return site (closed set; `test_critical_review_gate_class_parity.py` enforces). Any new return status needs one, **and** `_VERIFIER_SITE_RE` updated to match a new status literal.
- **Docstring honesty caveat** (three substrings: `"Does NOT detect"`, `"orchestrator-fabricated input"`, `"engagement"`) on both verifiers — must be preserved.

## Requirements & Constraints

- **Graceful partial failure** (`project.md:55`) is the requirement the fix serves — a false total-failure at zero drift discarding 4 parallel dispatches is exactly this defect. It sits in tension with the gate's deliberate **fail-closed** posture (catch concurrent-write drift); the fix must serve the former without weakening the latter.
- **"Prefer structural separation over prose-only enforcement … prose-only is appropriate only where occasional deviation is cheap"** (CLAUDE.md / `docs/policies.md:11`). Deviation here is *not* cheap (total-failure), so a prose-only prompt reword sits on the wrong side of this rule unless paired with a structural change that makes deviation cheap.
- **MUST-escalation policy** (`docs/policies.md:25–33`): adding new MUST/REQUIRED language to prompts requires an evidence artifact showing `effort=high` failed. The 88 events are that evidence — but they argue *against* leaning on more prose MUST, not for it.
- **ADR-0015** (`review-could-not-run-vs-dispatch-crash-split`) is the closest **structural precedent**: it split a fail-closed `ERROR`/revert into (a) genuine dispatch crash → revert + systemic breaker vs (b) "could-not-run" tooling miss → preserve + degraded marker. Directly analogous to distinguishing sentinel-omission from real drift. Its accepted three-part shape (per-instance graceful path + visible degraded marker + aggregate circuit breaker) is a template if the fix changes *what counts as total-failure*.
- **No ADR** currently governs the sentinel/verification-gate design. If the fix changes the fail-closed contract (not just detection precision) — sentinel demoted to advisory, total-failure redefined — that may clear the ADR three-criteria gate (hard-to-reverse: sibling sites depend on the semantics; surprising: a reader assumes total-exclusion means real drift; real trade-off: per ADR-0015 precedent). **Open Question for spec.**
- **Test/parity surface a fix must reckon with:** `test_critical_review_reference_pins.py` (byte-pins the total-failure literal across `SKILL.md` + `verification-gates.md`, cross-file equality; per-section `EXIT_MARKERS`; step designators), `test_critical_review_gate_class_parity.py` (annotation on every return site), `test_critical_review_sentinel_window.py` (14-case suite; pins the 50-line cliff), `test_critical_review_gate_nonlocal_failsafe.py` (exit-3 vs exit-4 distinctness), `test_plugin_mirror_parity.py` (byte-parity), fixture corpus README (warns against casual re-baseline), `bin/.events-registry.md:103` (`sentinel_absence` schema — field-additive; add a dated rebaseline note, as #229 did). A reasoned re-baseline is expected; it must not be test-chasing.

## Web Research

External literature converges on the same two moves:
1. **Move the required attestation into the part of the exchange that provably survives** — the end of the final message, or (strongest) a **structured/tool-call field** with `required` enforcement (Anthropic structured outputs guarantee presence at generation time). The Claude Code sub-agent docs confirm the architectural constraint driving the bug: *"fresh isolated context in, final message only out"* — the parent never sees intermediate preamble. Instructing a marker "before the first heading" asks for it in a region the transport does not preserve.
2. **Stop conflating "marker missing" with "input changed."** Prior art (Kinney, *Sentinels for Agent Gates*; arXiv *Reason Less, Verify More* deterministic gates) distinguishes **gate-infrastructure faults → fail-open** (a dropped/absent marker is "we don't know," a soft warning) from **policy violations → fail-closed** (an independently-verified hash mismatch is the real hard-fail). Crucially: the fact to verify — *did the pinned bytes change?* — **is checkable independently by hashing the file**, so the orchestrator need not trust the agent's self-report for ground truth at all. Free-text sentinel scanning degrades multiplicatively over multi-step pipelines; the industry is moving to forced structured output.

Sources: [Sentinels for Agent Gates — Kinney](https://stevekinney.com/writing/sentinels-for-agent-gates), [Reason Less, Verify More (arXiv 2607.07405)](https://arxiv.org/html/2607.07405v1), [Structured outputs — Claude Docs](https://platform.claude.com/docs/en/build-with-claude/structured-outputs), [Sub-agents — Claude Code Docs](https://code.claude.com/docs/en/sub-agents).

## Tradeoffs & Alternatives

- **Shape A — reword prompts to bind the sentinel to the final message** (prose-only, all gated sites). *Complexity:* lowest. *Robustness:* **weak, and on the reviewer path counterproductive** — the reviewer prompt ends with findings + JSON envelope, so moving `READ_OK` "to the final message" pushes it *past the 50-line window* whenever findings are long → still `absent` unless the window is also removed. It is the same prose-adherence class that already failed 88 times, against the exact structural constraint (the agent can't reliably reason about which turn is "final"). *Alignment:* violates CLAUDE.md's structural-over-prose rule (deviation is not cheap). **Not viable alone.**
- **Shape B — gate distinguishes omission from drift + bounded re-attestation** (code + orchestration prose). *Robustness:* strong at the fail-closed constraint (`sha_mismatch`/`read_failed` untouched). **But the cross-agent re-attestation re-imports the same discard risk** — the re-attester is *also* async and returns only its final message; a single retry just moves the false-failure one round later unless it is a minimal single-turn attest-only micro-task. And a *fresh* agent re-reading the file tells you nothing about what the *original excluded* reviewer read — it only re-checks file stability, which the orchestrator can do itself. *Cost:* retry tax on the unattended overnight path (auto-triggers before spec **and** plan approval), roughly doubling dispatch when omission is systematic.
- **Shape C — structured attestation (SHA in the reviewer's existing `<!--findings-json-->` envelope).** Best-in-class *for reviewers* (the envelope is empirically the guaranteed-last block; a missing envelope already soft-degrades rather than hard-excludes). **But fallback and synth prompts have no envelope** — Shape C would invent new wire-protocol machinery for them and risk a permanent two-mechanism split. Best scoped as a **reviewers-only follow-up**, and it still depends on agent self-report.
- **Shape D (surfaced by the adversarial pass) — gate-time re-hash; sentinel becomes advisory.** The orchestrator holds the resolved path and expected SHA from `prepare-dispatch`. Thread the path into `check_artifact_stable`/`check_synth_stable`; on `absent`, **re-hash the pinned file directly**: match → the sole rationale for the fail-closed gate (drift) is *disproven by measurement* → pass + log an advisory; mismatch/unreadable → hard-fail exactly as today. *Robustness:* eliminates **both** bug-halves (multi-turn discard AND 50-line truncation) uniformly across reviewer + synth (and would cover fallback/competing-plans for free if ever gated), **without prompt-adherence hope or retry tax**, **without weakening real (persistent) drift detection**. *Alignment:* squarely on the structural-over-prose side; matches the fail-open-on-infrastructure / fail-closed-on-policy pattern from the web literature. *Cost:* converts two currently-pure text-parsers into functions with filesystem access — bounded plumbing plus a reasoned re-baseline of the pinned-literal/window tests. *Security:* opens **no** new hole — the docstrings already concede the gate cannot verify engagement and the orchestrator controls both SHA and file, so the "non-reading reviewer passes" path already exists and is out of scope.

**Contradiction to resolve in spec:** the Tradeoffs agent recommended **Hybrid A+B** (uniform across gated sites); the Adversarial agent argued **Shape D (gate-time re-hash) strictly dominates** — same security profile as B, done in-process and deterministically, covering every site, with no retry tax or prose hope, and it showed Shape A to be counterproductive on the reviewer path. Both agree the fix must span the gated reviewer **and** synth paths and must not weaken persistent-drift detection. See Open Questions.

## Adversarial Review (key challenges folded in)

- **Shape A and the 50-line window fight each other** — late sentinel risks truncation; the two bug-halves are coupled, not independent.
- **Shape B's retry re-hits the systematic loss** and adds an overnight retry tax; a fresh re-attester only re-verifies file stability (which re-hash does deterministically).
- **No new security hole** from distinguishing absent-vs-drift: the engagement-spoof path already exists per the gate's own docstrings.
- **The one narrow shared blind spot** (Shape B *and* re-hash): **transient drift** — the file changes to SHA_x during review then is restored to the pinned SHA by gate time; a gate-time re-read sees "stable" and admits findings produced against SHA_x. The reviewer's own sentinel SHA nominally guards this — but in 100% of failing cases the sentinel is `absent`, so that guard is already not firing. Byte-identical round-trip within one window is not a realistic accidental-write pattern. **Recommendation: document this as the explicit new boundary of the fail-closed guarantee** rather than leave it implicit.
- **Doc language seeds the bug and must be corrected, not merely supplemented:** `verification-gates.md:27` "raw stdout" (there is no stdout — only the final tool-result message) and `reviewer-prompt.md:17` "preamble prose before it is fine" (blesses the early emission that gets discarded).
- **Re-type telemetry:** post-fix, `reason: absent` should mean "self-report missing but file provably stable (advisory)" — distinct from real invalidation. Don't keep inflating the same exclusion counter with advisory absences, or the very signal that measured this bug (`sentinel_absence` events) becomes unreadable.

## Open Questions

1. **Which fix shape? (central design decision — DEFERRED to Spec §4 user approval, with recommendation.)** Research recommendation, weighted toward the adversarial analysis: **Shape D (gate-time re-hash), making the sentinel advisory**, applied uniformly to the reviewer (`check_artifact_stable`) and synth (`check_synth_stable`) gated paths — optionally with a minimal `reviewer-prompt.md`/`verification-gates.md` wording *correction* (delete the "preamble prose before it is fine" blessing and the "raw stdout" mental model) as belt-and-suspenders, **not** the load-bearing mechanism. Shape B's cross-agent re-attestation is not recommended (re-imports the discard risk + retry tax). Shape C is a reviewers-only follow-up. *Deferred rather than resolved here because it is a consequential design/scope choice the spec's §4 approval surface owns; the tradeoff is fully characterized above for that decision.*

2. **Does the fix change the fail-closed *contract* enough to warrant an ADR? (DEFERRED to Spec.)** If the sentinel is demoted to advisory and total-failure is redefined as "persistent drift measured at gate time," that is plausibly ADR-worthy (three-criteria gate, ADR-0015 precedent). Spec decides whether to author one; the #229 spec concluded no ADR was needed because it didn't change the contract — this fix might.

3. **Mechanism (a) discarded-early-turn vs (c) never-emitted. (DEFERRED — does not gate the fix.)** Undecidable from existing artifacts; would need a runtime full-transcript capture of one live dispatch. Explicitly deferred because Shape D (re-hash) is robust to whichever dominates, so resolving it is not a precondition for the recommended fix. Worth a one-off instrumentation only if a prompt-side remedy is chosen after all.

4. **Transient-drift boundary. (RESOLVED — document, don't engineer.)** Change-then-restore-to-identical-SHA within one review window is not a realistic accidental-write pattern and is already unprotected today (failing cases are all `absent`). Resolution: the spec should **document** this as the accepted edge of the fail-closed guarantee, not add machinery for it.

5. **`fallback-reviewer-prompt.md` and `competing-plans.md`. (RESOLVED — out of scope.)** Both are ungated, so neither can reproduce #376's false-exclusion symptom; editing their sentinel prose is a no-op for the reported bug. The fallback's *inverse* latent gap (surfacing potentially-drifted output unchecked at Step 2c→Step 3) is a real but separate concern — flag as a candidate follow-up backlog item, not part of this fix.
