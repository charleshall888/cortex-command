# Specification: make-cortex-update-item-accept-flag

> Epic context: this ticket descends from the [`harness-friction-triage`](../../research/harness-friction-triage/research.md) discovery and scopes to one specific drift signature — `cortex-update-item`'s outlier positional `key=value` convention. Companion tickets 253 (skill-prose contract lint) and 248 (embedded `python3 -m` migration) ship independently.

## Problem Statement

`cortex-update-item` is the only `cortex-*` CLI in the project that uses positional `key=value` arguments — every sibling (`cortex-create-backlog-item`, `cortex-resolve-backlog-item`, `cortex-discovery`, `cortex-critical-review`, `cortex-complexity-escalator`, etc.) uses argparse `--flag value` convention. The inconsistency causes operator muscle-memory failures (`cortex-update-item 257 --status refined` returns `Invalid argument (expected key=value): --status`) and forces skill-prose authoring agents to remember a per-CLI exception. Replacing the positional convention with per-key `--flag value` flags, migrating every in-repo caller atomically, and shipping an argv pre-flight that catches the old form with an actionable migration hint resolves the inconsistency and removes one categorical exception from ticket 253's future contract lint.

## Phases

- **Phase 1: CLI argparse rewrite** — Replace `cortex-update-item`'s manual `sys.argv[2:]` parsing with an `argparse.ArgumentParser` using per-key `--flag value` flags. Include the argv pre-flight migration hint, `allow_abbrev=False`, and `nargs='*'` for list fields. Update the module's docstring. Add a dedicated CLI argparse unit-test file.
- **Phase 2: Caller migration & verification** — In ordered sequence: (a) update all in-repo callers (skill prose ≥10 markdown files, `justfile:131`, `docs/backlog.md` examples); (b) update the test fixture in `tests/test_morning_review_status_close_ordering.py`; (c) run `just build-plugin` to regenerate plugin mirrors; (d) reinstall the wheel via `uv tool install --reinstall --refresh-package cortex-command .`; (e) run `cortex-check-parity --audit` and the full test suite.

## Requirements

1. **Argparse rewrite preserves existing field-set semantics.** The new CLI accepts the same set of scalar fields the current CLI accepts (`status`, `complexity`, `criticality`, `spec`, `lifecycle_slug`, `lifecycle_phase`, `session_id`, `priority`, `parent`, `blocked-by`, `rework_of`) plus the list fields (`areas`, `tags`). All flags are optional; a no-flag invocation is a no-op (matches current behavior). The internal `update_item(item_path, fields_dict, backlog_dir, session_id)` Python API stays unchanged. **Acceptance criteria**: `python3 -m cortex_command.backlog.update_item --help` (working-tree invocation, run from repo root) lists exactly the flags above plus the positional slug argument, and exits 0. `grep -nE "def update_item\(item_path" cortex_command/backlog/update_item.py` returns one line (internal API signature unchanged). **Phase**: Phase 1
2. **`null` / `none` / `""` value coercion preserved for scalar fields.** Current behavior at `cortex_command/backlog/update_item.py:458-461`: scalar values `"null"`, `"none"`, `""` coerce to Python `None` before write. The new argparse layer preserves this for scalar fields only — list-field elements pass through literally (an empty-string list element is not coerced; if the caller wants a cleared list they pass bare `--areas` for empty). **Acceptance criteria**: a unit test asserts that for each scalar flag in {`--status`, `--lifecycle-phase`, `--session-id`}, the values `null`, `none`, `""` each produce Python `None` when fields are passed to `update_item()`. A separate unit test asserts that `--areas '' a b` produces `['', 'a', 'b']` (list elements not coerced). **Phase**: Phase 1
3. **List fields use `nargs='*'` for the single-flag multi-value shape.** `--areas a b c` parses to `['a','b','c']`. Bare `--areas` (no values, followed by another flag or end-of-args) parses to `[]`. `--tags` behaves identically. Duplicate flag occurrences follow argparse's default last-wins (`--areas a b --areas c` produces `['c']`) — this matches the implicit semantic of the legacy `key=value` form and is not specially handled. The existing `_parse_inline_str_list()` utility is retained for the YAML-side parse (loading existing frontmatter from disk) but new CLI input does not depend on it. **Acceptance criteria**: unit tests assert `parse_args(['257', '--areas', 'overnight-runner', 'backlog']).areas == ['overnight-runner', 'backlog']`; `parse_args(['257', '--areas', '--status', 'complete']).areas == []` and `.status == 'complete'`; `parse_args(['257', '--areas', 'a', 'b', '--areas', 'c']).areas == ['c']` (documented last-wins). **Phase**: Phase 1
4. **`allow_abbrev=False` on the parser.** Prevents argparse from silently accepting unambiguous prefixes (`--stat`, `--sess`, `--lif`). Future-proofs against ambiguity from new flags. **Acceptance criteria**: a unit test asserts that `parse_args(['257', '--stat', 'complete'])` raises `SystemExit` (caught via `pytest.raises(SystemExit)`). **Phase**: Phase 1
5. **Argv pre-flight migration hint catches legacy `key=value` form.** Before argparse runs, scan `sys.argv[1:]` (skipping `sys.argv[0]` which is the program name; positional ordering is not assumed since argparse permits flags before positionals) for any token matching `^[a-z_][a-z_]*=` — an unquoted underscore-or-letter prefix immediately followed by `=`. On match, print to stderr: `Detected legacy key=value form '<arg>'. Use --<key> <value> instead. See cortex-update-item --help.` and exit 2. The regex uses `[a-z_]` (no leading `-`) for the first char to avoid false-firing on `--status=complete` (argparse's equals form). It catches the bracket-list legacy form too because the shell strips outer quotes before argv (`"areas=[a,b]"` arrives as `areas=[a,b]` which matches). **Acceptance criteria**: a unit test asserts `_argv_preflight(['cortex-update-item', '257', 'status=complete'])` raises `SystemExit(2)` and the captured stderr contains the literal substring `Detected legacy key=value form 'status=complete'.` A second unit test asserts the same for `['cortex-update-item', '257', 'areas=[a,b]']`. A third unit test asserts `_argv_preflight(['cortex-update-item', '257', '--status=complete'])` does NOT raise (argparse equals form is not legacy). **Phase**: Phase 1
6. **Module docstring reflects the new CLI.** The docstring at `cortex_command/backlog/update_item.py:9-14` shows the `--flag value` form (or is removed in favor of argparse's auto-generated help). The `Usage:` error string in `main()` referencing `key=value` is removed. **Acceptance criteria**: `grep -nE "key=value" cortex_command/backlog/update_item.py` returns zero matches. **Phase**: Phase 1
7. **New CLI argparse unit-test file exists and covers the surface.** New file `tests/test_update_item_cli.py` covers: scalar-flag parsing, list-flag parsing including empty-list and documented last-wins, `null`/`none`/`""` coercion for scalars and list-element non-coercion, `allow_abbrev=False` behavior, and the argv pre-flight migration hint for bare `key=value`, the bracket-list legacy form, and the negative case (`--status=complete` argparse equals form does not false-fire). **Acceptance criteria**: `python3 -m pytest tests/test_update_item_cli.py -v` exits 0 and the test file contains at least 5 named test functions (`grep -c "^def test_" tests/test_update_item_cli.py` ≥ 5). **Phase**: Phase 1
8. **All in-repo executable callers migrated.** Every `cortex-update-item <arg> key=value [key=value ...]` invocation in skill markdown, justfile, docs, and tests becomes `cortex-update-item <arg> --key value [--key value ...]` (or the equivalent list-flag form). The canonical paths to migrate (per research.md call-site inventory): `skills/morning-review/SKILL.md:104`, `skills/morning-review/references/walkthrough.md:537`, `skills/backlog/SKILL.md:79,80`, `skills/lifecycle/references/complete.md:203`, `skills/lifecycle/references/wontfix.md:44`, `skills/lifecycle/references/clarify.md:112`, `skills/lifecycle/references/backlog-writeback.md:29,80,88`, `skills/refine/SKILL.md:84,187,191,194`, `justfile:131`, `docs/backlog.md` (~7 worked examples), `tests/test_morning_review_status_close_ordering.py:22` (`CLOSE_ARG = "status=complete"` → `CLOSE_ARG = "--status complete"`). **Acceptance criteria**: `grep -rnE 'cortex-update-item[^|]*[ "'\''"][a-z_][a-z_-]*=' skills/ docs/ justfile tests/ bin/ hooks/ claude/ cortex_command/ plugins/ 2>/dev/null` returns zero matches. The regex matches any `cortex-update-item` invocation followed by a space-or-quote-prefixed unquoted-key=value token (the `[a-z_]` first-char anchor distinguishes legacy `status=` from the argparse equals form `--status=`). It catches the bracket-list legacy form (`"areas=[a,b]"`) because `"` is in the prefix character class. Hyphenated slugs (`{backlog-slug}`) are tolerated because the `[^|]*` between `cortex-update-item` and the prefix consumes them. **Phase**: Phase 2
9. **Test fixture migrated in same commit as skill prose.** `tests/test_morning_review_status_close_ordering.py:22` `CLOSE_ARG = "status=complete"` becomes `CLOSE_ARG = "--status complete"` (matching the new prose form chosen for `skills/morning-review/SKILL.md:104`). The assertion at line 74 continues to pass against the migrated skill prose. **Acceptance criteria**: `python3 -m pytest tests/test_morning_review_status_close_ordering.py -v` exits 0. **Phase**: Phase 2
10. **Plugin mirrors regenerated as an explicit Phase 2 step.** `just build-plugin` runs cleanly and the mirrored skill prose under `plugins/cortex-core/skills/` and `plugins/cortex-overnight/skills/` reflects the canonical `skills/` changes. The pre-commit dual-source drift check passes — but this requirement encodes `just build-plugin` as an ordered Phase 2 step (per Phases section), not relying solely on the hook trigger. **Acceptance criteria**: after `just build-plugin`, the requirement 8 grep also returns zero matches when run against `plugins/` (covered by the existing scope `plugins/` in that grep). **Phase**: Phase 2
11. **Wheel reinstalled before binstub-tier verification.** As an explicit ordered Phase 2 step (per Phases section), run `uv tool install --reinstall --refresh-package cortex-command .` after the source edits. This ensures the `cortex-update-item` binstub on PATH reflects the new code. All test commands run against the wheel must execute after this step; tests using `python3 -m cortex_command.backlog.update_item` run against the working tree and need no reinstall. **Acceptance criteria**: after reinstall, `cortex-update-item --help` (binstub on PATH) exits 0 and stdout contains the literal substring `--status` (proving the binstub reflects the new flags, not the old positional form). **Phase**: Phase 2
12. **Parity check confirms migration does not regress existing checks.** `cortex-check-parity --audit` (or its `just`-recipe equivalent) runs and exits 0 against the migrated codebase. This requirement confirms the migration does not break `cortex-check-parity`'s existing scope; it does NOT add new convention-enforcement (per Non-Requirements). **Acceptance criteria**: the parity audit recipe exits 0. **Phase**: Phase 2
13. **Automated migration-hint integration test.** A subprocess-based test invokes `python3 -m cortex_command.backlog.update_item 257 status=complete` and asserts exit code 2 and stderr containing `Detected legacy key=value form 'status=complete'.` This closes the integration loop without requiring an interactive operator or a separate wheel-install step (the working-tree invocation uses the same source the wheel ships). **Acceptance criteria**: a named test function in `tests/test_update_item_cli.py` performs the subprocess invocation, asserts the exit code, and asserts the stderr substring; the test exits 0 under `pytest`. **Phase**: Phase 2

## Non-Requirements

- **No deprecation alias.** The old `key=value` positional form is removed entirely; no parallel support, no `DeprecationWarning`-emitting bridge, no follow-up cleanup ticket. Decided during Clarify; reaffirmed by research.
- **No duplicate-flag rejection.** argparse's default last-wins semantics for repeated scalar and list flags is accepted — it matches the implicit behavior of the legacy `key=value` form (which also silently last-wins) and does not introduce a new failure surface. A custom argparse action enforcing rejection was considered and rejected as scope creep beyond the chore framing.
- **No rewrite of historical lifecycle artifact prose.** ~8 in-progress lifecycle artifacts under `cortex/lifecycle/*/{spec,plan,research}.md` contain literal `cortex-update-item key=value` examples in cached prose. These remain unchanged — they are append-only history. The argv pre-flight migration hint (Requirement 5) fires loudly with an actionable message on the first hit, which is sufficient for the human-operator case and for agents whose tool-call wrappers surface stderr; it is not claimed to silently recover for every resume context.
- **No change to the internal `update_item(item_path, fields_dict, backlog_dir, session_id)` Python API.** Only the CLI parsing layer changes. Python callers (`cortex_command/overnight/outcome_router.py`, `tests/test_backlog_worktree_routing.py`) are unaffected.
- **No Click rewrite, no JSON-patch input flag, no `--set key=value` repeating-flag pattern.** These were considered in research and rejected as out-of-scope.
- **No extension of `cortex-check-parity` to enforce the new convention going forward.** That hardening is ticket 253's scope. Requirement 12 only confirms the migration does not regress existing parity checks; it does not add new ones.
- **No coordination with ticket 254 inside this PR.** Both modify `cortex_command/backlog/update_item.py`, but 254's changes are to `_find_item` (resolver internals) and are argparse-layer-agnostic. If 254 lands first, this ticket rebases on the `_find_item` region only — non-overlapping with the `main()` rewrite. Sequencing is a coordination footnote, not enforced.

## Edge Cases

- **`--status ''` (empty string)**: scalar `""` coerces to Python `None`. Writes `status: null` to YAML, NOT the literal empty string.
- **`--areas '' a b` (empty string as list element)**: produces `['', 'a', 'b']`. List elements are not coerced (per Requirement 2). If this surfaces as a real-world issue, address in a follow-up.
- **`--areas` followed by end-of-argv vs. followed by another flag**: both produce `[]`. argparse's `nargs='*'` is well-behaved here (Adversarial-agent verified empirically).
- **`--areas a --areas b` (duplicate list flag)**: documented as argparse-default last-wins → `['b']`. Matches the implicit behavior of the legacy `key=value` form. Not specially handled.
- **`--status complete --status refined` (duplicate scalar flag)**: documented as argparse-default last-wins → `status='refined'`. Same rationale.
- **Slug positional argument missing**: argparse exits 2 with the standard "missing required argument" error.
- **Unknown flag**: argparse exits 2 with "unrecognized arguments" (e.g., `--phasee complete`). With `allow_abbrev=False`, prefix-typos do NOT silently match a longer flag.
- **Bare `cortex-update-item 257 status=complete` (legacy form)**: argv pre-flight fires before argparse, exits 2 with the migration hint message.
- **Bare `cortex-update-item 257 "areas=[a,b]"` (legacy bracket-list form)**: same as above — the shell strips outer quotes, argv arrives as `areas=[a,b]`, which matches the pre-flight regex `^[a-z_][a-z_]*=`.
- **`cortex-update-item --status=complete 257` (argparse equals form, flag before positional)**: argparse handles this natively; pre-flight does NOT false-fire because `[a-z_]` (no leading `-`) is the first char in the regex anchor.
- **Slug containing `=`**: the pre-flight regex requires `[a-z_]` as the first char; UUIDs (hex digits + dashes) and slugs (digits + lowercase + dashes, starting with digits) do not begin with a letter-or-underscore followed by `=`. Pre-flight does not false-fire on real slugs.

## Changes to Existing Behavior

- **MODIFIED**: `cortex-update-item <slug> key=value [key=value ...]` → `cortex-update-item <slug> --key value [--key value ...]`. The positional `key=value` form is removed.
- **MODIFIED**: error message on legacy form. Was `Invalid argument (expected key=value): --status`. Now: `Detected legacy key=value form '<arg>'. Use --<key> <value> instead. See cortex-update-item --help.`
- **MODIFIED**: list-field invocation shape in skill prose. Was `"areas=[area1,area2]"`. Now `--areas area1 area2` (and bare `--areas` for empty).
- **ADDED**: dedicated CLI argparse unit-test file at `tests/test_update_item_cli.py`. No CLI-layer coverage existed before.
- **ADDED**: argv pre-flight migration hint emitted on detection of legacy `key=value` form.
- **REMOVED**: the manual `sys.argv[2:]` `key=value` parsing loop at `cortex_command/backlog/update_item.py:448-461` (replaced by argparse).
- **UNCHANGED**: internal `update_item(item_path, fields_dict, backlog_dir, session_id)` Python API. Python callers see no change.
- **UNCHANGED**: `_telemetry.log_invocation("cortex-update-item")` call at module load — fires before parsing, payload is the CLI name only.
- **UNCHANGED**: backlog file selection logic (`_find_item`). This is ticket 254's territory.

## Technical Constraints

- **Wheel-first distribution (ADR-0002)**: `cortex-update-item` is a `pyproject.toml [project.scripts]` entry. Verification at the binstub level (Requirement 11) must run against the installed wheel after `uv tool install --reinstall ...`. Verification at the source level (Requirement 13) uses `python3 -m cortex_command.backlog.update_item` to invoke the working tree without requiring reinstall.
- **Plugin mirror regeneration**: `just build-plugin` is encoded as an ordered Phase 2 step. The pre-commit dual-source drift check is a defense-in-depth backstop, not the primary gate.
- **`allow_abbrev=False`**: standard argparse pattern, no library dependency.
- **Skill-helper modules pattern (project.md line 35)**: this work preserves the canonical idiom — `cortex_command/backlog/update_item.py` exposing `cortex-update-item` via `[project.scripts]`.
- **No new project dependencies**: argparse is stdlib; nothing else added.
