# Specification: extend-overnight-circuit-breaker-to-halt

## Problem Statement

The overnight orchestrator's session-halt circuit breaker currently fires on only two error types (`budget_exhausted`, `api_rate_limit`). When the execution environment is systemically broken — workers failing to write exit reports for features that produce nothing, repeated `infrastructure_failure` returns from the SDK transport, or other class-recurring failures — the batch continues to dispatch wave after wave of features against the known-broken environment. The existing class-blind `CIRCUIT_BREAKER_THRESHOLD=3` counter is observability-only (logs an event without setting `global_abort_signal`) and resets on any successful feature, so an interleaved success-failure pattern hides the cascade. This feature adds a **session-halting** systemic-cascade detector with two improvements over the class-blind counter: (1) it halts the session by setting `global_abort_signal`, and (2) its counter is total-in-batch (does not reset on intervening successes), so a productive-but-systemically-noisy session still trips the halt. Empirical replay of the historical corpus showed that naive halt-on-first-occurrence of `worker_no_exit_report` would have aborted real successful sessions — the spec gates the silent-worker halt on a zero-commits feature outcome instead.

## Phases

- **Phase 1: Upstream wiring** — make systemic error strings observable on `FeatureResult.error` so the cascade detector can see them. Three sub-changes: (a) restructure exit-report validation in `feature_executor.py` so a feature whose worker emitted `WORKER_NO_EXIT_REPORT` or `WORKER_MALFORMED_EXIT_REPORT` AND produced zero commits is finalized as paused with the matching error string; (b) thread `error_type` through `retry.py`'s `pause_human` branch (parity with `pause_session`); (c) at the brain-triage fallthrough in `feature_executor.py`, when `result.error_type ∈ _SYSTEMIC_ERROR_TYPES`, set `FeatureResult.error = result.error_type` instead of the generic "Task N failed after M attempts" string.
- **Phase 2: Cascade-class detector** — introduce `_SYSTEMIC_ERROR_TYPES`, a `systemic_pauses_in_batch` counter on `CircuitBreakerState` (total-in-batch, no reset on success), a new `pipeline_systemic_failure` event with a list-valued `cause_class` field, and the threshold check that sets `global_abort_signal`.

## Requirements

1. **Slice A re-scoped halt for worker silence**: When a worker has emitted `WORKER_NO_EXIT_REPORT` or `WORKER_MALFORMED_EXIT_REPORT` for at least one task in a feature's plan (`cortex_command/overnight/feature_executor.py:765–779`) AND the feature has produced zero commits across all completed tasks at feature finalization, finalize the feature as `FeatureResult(status="paused", error="worker_no_exit_report")` (or `error="worker_malformed_exit_report"` when the only silent-worker events were malformed reports). When at least one commit was produced, preserve the existing `FeatureResult(status="completed")` behavior — the silent-worker event is treated as bookkeeping omission, not failure. **Acceptance**: `grep -c "WORKER_NO_EXIT_REPORT\|WORKER_MALFORMED_EXIT_REPORT" cortex_command/overnight/feature_executor.py` ≥ 2 (existing emit sites preserved); new test in `tests/test_feature_executor.py` asserts: (a) a feature emitting WORKER_NO_EXIT_REPORT mid-feature with subsequent commit-producing tasks returns `status="completed"`; (b) a feature emitting WORKER_NO_EXIT_REPORT on a task with zero commits at feature finalization returns `status="paused"` with `error="worker_no_exit_report"`; (c) the malformed-report variant behaves identically. **Phase**: Phase 1.

2. **retry.py `pause_human` error_type propagation**: `cortex_command/pipeline/retry.py:370–377` (`pause_human` branch) sets `error_type=error_type` on the returned `RetryResult`, matching the `pause_session` branch at line 398. **Acceptance**: `grep -c "error_type=error_type" cortex_command/pipeline/retry.py` ≥ 2 (presence on both branches); new test asserts that a `pause_human` recovery for `infrastructure_failure` returns `RetryResult.error_type == "infrastructure_failure"`. **Phase**: Phase 1.

3. **Systemic-error-type propagation to `FeatureResult.error`**: In `cortex_command/overnight/feature_executor.py` at the brain-triage fallthrough (currently `return FeatureResult(... error=f"Task {task.number} failed after {result.attempts} attempts")` at lines 707–711), add a guard before the fallthrough: if `result.error_type` is in `_SYSTEMIC_ERROR_TYPES`, return `FeatureResult(status="paused", error=result.error_type)` instead of the generic string. The session-halting tuple (`_SESSION_HALT_ERROR_TYPES`) check at line 688 is unchanged — its members continue to short-circuit. **Acceptance**: new test asserts that a `RetryResult(paused=True, error_type="infrastructure_failure")` propagated through `_run_feature_tasks` yields `FeatureResult(status="paused", error="infrastructure_failure")`, not the generic "Task N failed" string. **Phase**: Phase 1.

4. **`_SYSTEMIC_ERROR_TYPES` tuple**: Define a new module-level tuple in `cortex_command/overnight/feature_executor.py`, sibling to `_SESSION_HALT_ERROR_TYPES`, with initial members `("infrastructure_failure", "worker_no_exit_report", "worker_malformed_exit_report")`. The membership is closed-enum; expansion requires an explicit code edit. **Acceptance**: `grep -n "_SYSTEMIC_ERROR_TYPES" cortex_command/overnight/feature_executor.py` ≥ 1; tuple contents match the three strings above verbatim. **Phase**: Phase 2.

5. **`systemic_pauses_in_batch` counter on `CircuitBreakerState`**: Add a new integer field `systemic_pauses_in_batch: int = 0` to `CircuitBreakerState` (`cortex_command/overnight/types.py:35`). Increment under `ctx.lock` at every pause site in `outcome_router.py` where `result.error ∈ _SYSTEMIC_ERROR_TYPES`. **Do NOT reset** on success or on non-systemic pauses — the counter accumulates across the entire batch and is cleared only by the batch-end construction of a fresh `CircuitBreakerState` in `orchestrator.py:run_batch()`. This is the deliberate design divergence from the class-blind `consecutive_pauses` counter (which resets on success). **Acceptance**: `grep -c "systemic_pauses_in_batch" cortex_command/overnight/types.py` ≥ 1; `grep -c "systemic_pauses_in_batch" cortex_command/overnight/outcome_router.py` ≥ 1 (increment sites; verify against current pause sites at implementation time); `grep -c "systemic_pauses_in_batch = 0" cortex_command/overnight/outcome_router.py` = 0 (no reset sites). **Phase**: Phase 2.

6. **`SYSTEMIC_FAILURE_THRESHOLD` constant**: Add `SYSTEMIC_FAILURE_THRESHOLD = 3` to `cortex_command/overnight/constants.py`. The value matches Celery+SQS broken-worker breaker precedent (`fail_max=3`) and is independent of `CIRCUIT_BREAKER_THRESHOLD` — semantics are total-in-batch vs consecutive. **Acceptance**: `grep -n "SYSTEMIC_FAILURE_THRESHOLD" cortex_command/overnight/constants.py` returns one match with value `3`. **Phase**: Phase 2.

7. **`pipeline_systemic_failure` event with list-valued `cause_class`**: Add a new event constant `PIPELINE_SYSTEMIC_FAILURE = "pipeline_systemic_failure"` to `cortex_command/overnight/events.py` and to `EVENT_TYPES`. When `ctx.cb_state.systemic_pauses_in_batch >= SYSTEMIC_FAILURE_THRESHOLD`, emit the event with `details={"cause_class": [<error_strings of the N most recent systemic pauses, oldest first>], "threshold": SYSTEMIC_FAILURE_THRESHOLD}`. **Acceptance**: `grep -n "PIPELINE_SYSTEMIC_FAILURE" cortex_command/overnight/events.py` returns one match; new test asserts the emitted event's `details.cause_class` is a list of length `SYSTEMIC_FAILURE_THRESHOLD` with the matching error strings in oldest-first order. **Phase**: Phase 2.

8. **`global_abort_signal` set on threshold trip**: When the systemic threshold trips, set `batch_result.global_abort_signal = True` (same mechanism as `_SESSION_HALT_ERROR_TYPES` halt at `orchestrator.py:404–427, 505–519`). The session transitions to `paused` phase via the existing handler. This is the load-bearing behavioral difference from the class-blind `CIRCUIT_BREAKER_THRESHOLD` breaker, which logs an event but does not halt the session. **Acceptance**: integration test asserts that three systemic-class pauses anywhere in a single batch cause `batch_result.global_abort_signal` to be `True` after the third feature returns; session phase is `paused` after `run_batch` returns; integration test also asserts that an `[S, S, success, S]` sequence (where `S` is a systemic pause) trips on the fourth feature (proving total-in-batch semantics — class-blind counter would have reset on the success). **Phase**: Phase 2.

9. **`bin/.events-registry.md` registration**: Add a row for `pipeline_systemic_failure` with target `overnight-events-log`, scan_coverage `manual`, producers citing `cortex_command/overnight/outcome_router.py:<line>` (the threshold-trip emit site), consumer `TBD`, category `live`, registered date today, rationale referencing this lifecycle. **Acceptance**: `grep -c "pipeline_systemic_failure" bin/.events-registry.md` ≥ 1; `just events-registry-audit` (or `cortex-check-events-registry --audit`) passes. **Phase**: Phase 2.

10. **No-op when below threshold**: Until the counter reaches `SYSTEMIC_FAILURE_THRESHOLD`, no event is emitted and `global_abort_signal` is not set. The existing class-blind `CIRCUIT_BREAKER_THRESHOLD` breaker continues to fire independently on its own threshold. **Acceptance**: test asserts that two systemic-class pauses in a batch do NOT emit `pipeline_systemic_failure` and do NOT set `global_abort_signal`. **Phase**: Phase 2.

## Non-Requirements

- **No inline EPERM classifier hoisting**. `report.py:1916`'s closed enum (`plumbing_eperm`, `unclassified_eperm`) is computed post-hoc from `bash.log` and `sandbox-deny-lists/` sidecars and is not threaded through `FeatureResult.error`. This spec does not lift the morning-report classifier inline. A follow-up backlog ticket will be filed for that work, at which point `_SYSTEMIC_ERROR_TYPES` may expand to include EPERM-class strings.
- **No persistence of `systemic_pauses_in_batch` across batch resumes**. Counter is in-memory on `CircuitBreakerState` per existing pattern. A paused-and-resumed session starts the systemic counter at 0 for the new batch.
- **No mid-flight halt check inside `execute_feature`**. Concurrent in-flight features continue to completion even after `global_abort_signal=True`; the pre-dispatch gate (`orchestrator.py:357, 376–384`) blocks only **new** features. Effective halt point is `SYSTEMIC_FAILURE_THRESHOLD + (concurrent_inflight − 1)` features past the third systemic pause — at default tier (`MAX_100`, max_concurrent_workers=2) that is one extra in-flight feature; at top tier (`MAX_200`, max_concurrent_workers=3) that is up to two. Accepted as the cost of preserving the simple pre-dispatch-gate model; mid-flight cancellation is a separate, larger surface area.
- **No `agent_refusal` / `agent_confused` in `_SYSTEMIC_ERROR_TYPES`**. These are agent-side pauses requiring human triage; they are not environment failures. Misclassifying them as systemic would over-halt on legitimate task-specific issues.
- **No `integration_degraded` audit**. The runner's degraded-mode fallback (`state.py:240`, `runner.py:1498, 1549`) may mask systemic cascades upstream of per-feature dispatch. Auditing it is out of scope — file a follow-up.
- **No corpus-replay tooling**. The historical corpus informed Requirement 1's zero-commits gate but is not productionized as a tunable signal. `SYSTEMIC_FAILURE_THRESHOLD` ships at `3` based on industry precedent (Celery+SQS, Resilience4j community).

## Edge Cases

- **Total-in-batch counter, not consecutive**: Distinct from the class-blind `consecutive_pauses` counter (which resets on any success), `systemic_pauses_in_batch` accumulates monotonically until the batch ends. A productive-but-systemically-noisy session — pattern `[S, success, S, success, S]` — trips on the third systemic pause, where the class-blind counter would have reset twice and never tripped. This is the load-bearing reason the new counter exists alongside the class-blind one.
- **Counter cleared only at batch start**: A fresh `CircuitBreakerState()` is constructed inside `orchestrator.py:run_batch()`; that is the only path to a zero counter. Across-resume cascades are not preserved (see Non-Requirement on persistence).
- **Concurrent in-flight overshoot**: If features are dispatched concurrently and several fail with systemic errors near-simultaneously, the counter increments under lock but each was already past the pre-dispatch gate. The session halts after the third pause is processed but the concurrent in-flight features finish their dispatch. Effective halt point is documented in Non-Requirements above.
- **Both counters trip on the same pause**: The class-blind `consecutive_pauses` and the new `systemic_pauses_in_batch` may both cross threshold on the same pause. The class-blind breaker logs `CIRCUIT_BREAKER` (no halt action); the new systemic breaker emits `pipeline_systemic_failure` and sets `global_abort_signal`. Both events are written to the log — the class-blind event remains as ground truth, the systemic event provides the halt-action signal and `cause_class` payload. No suppression logic; morning-report consumers should treat the systemic event as the halt anchor when both are present.
- **Worker emits silent-worker event but feature produces commits**: Treated as bookkeeping omission per Requirement 1's commits-gate. Empirical replay showed sessions with 21 `worker_no_exit_report` events alongside 7 `feature_complete` events on the same features — these would not halt under this spec.
- **Worker emits malformed-exit-report on a feature with zero commits**: Treated identically to `worker_no_exit_report` (Requirement 1 symmetry).
- **First batch trip with cause_class containing duplicate values**: When all N triggering pauses share the same error string (e.g., three `infrastructure_failure` in a row), `cause_class` is `["infrastructure_failure", "infrastructure_failure", "infrastructure_failure"]`. Duplicates are preserved.
- **cause_class ordering on emit**: The list is the most recent N systemic pauses in oldest-first order. The total counter may have incremented past N (e.g., 4 systemic pauses by the time concurrent in-flight features finish), but `cause_class` is capped at N entries to keep the morning-report payload bounded.
- **Empty `_SYSTEMIC_ERROR_TYPES` at runtime** (defensive): If the tuple is misconfigured to empty, the counter never increments and the breaker never fires. No regression on existing batch behavior.

## Technical Constraints

- ADR-0001 (file-based state): `CircuitBreakerState` remains in-memory; the new counter does NOT introduce a new persistent state field. Per-batch semantics is explicit.
- The session-halt check at `feature_executor.py:688` reads `RetryResult.error_type`; the orchestrator check at `orchestrator.py:404–427, 505–519` reads `FeatureResult.error`; the new systemic counter at the outcome-router pause sites reads `FeatureResult.error`. Phase 1 produces `FeatureResult.error = result.error_type` for systemic errors via Requirement 3 (brain-triage fallthrough guard), so the systemic counter observes the same strings the retry pipeline produces. `FeatureResult` is NOT extended with a new `error_type` field — the existing `error` field is the single channel for downstream consumers.
- `bin/.events-registry.md` audit (`just events-registry-audit` or `cortex-check-events-registry --audit`) must pass after the new event row is added.
- The morning report's existing EPERM taxonomy (`report.py:1908–1921`) remains the post-hoc rendering authority. The new event's `cause_class` is a *parallel* taxonomy at the dispatch layer — explicit, documented divergence. A follow-up ticket will reconcile.
- `_SESSION_HALT_ERROR_TYPES` continues to handle `budget_exhausted` and `api_rate_limit` exactly as today. The new systemic-failure path is structurally separate (`global_abort_signal` set at a different call site, distinct event name, distinct counter).
- All new code must compile against the existing wheel-binstub invocation pattern (`bin/cortex-*` reads installed wheel; `python3 -m cortex_command.overnight.*` reads working tree). No new binstubs required.
