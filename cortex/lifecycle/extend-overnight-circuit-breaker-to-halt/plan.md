# Plan: extend-overnight-circuit-breaker-to-halt

## Overview
Add a session-halting systemic-cascade detector to the overnight orchestrator. Phase 1 wires three upstream producers so systemic error strings (`infrastructure_failure`, `worker_no_exit_report`, `worker_malformed_exit_report`) actually land on `FeatureResult.error`. Phase 2 introduces a sibling tuple, a total-in-batch counter on `CircuitBreakerState`, and a new `pipeline_systemic_failure` event that sets `global_abort_signal=True` on the third systemic pause anywhere in a batch.

## Outline

### Phase 1: Upstream wiring (tasks: 1, 2, 3)
**Goal**: Make systemic error strings observable on `FeatureResult.error` so the cascade detector in Phase 2 has signal to read.
**Checkpoint**: `_SYSTEMIC_ERROR_TYPES` tuple exists in `feature_executor.py`; three new tests in `tests/test_feature_executor.py` and `cortex_command/pipeline/tests/test_retry.py` pass; full `just test` green.

### Phase 2: Cascade-class detector (tasks: 4, 5, 6, 7, 8)
**Goal**: Detect three systemic pauses anywhere in a batch and halt the session.
**Checkpoint**: A simulated batch with feature sequence `[paused(infrastructure_failure), paused(worker_no_exit_report), completed(commit), paused(worker_malformed_exit_report)]` causes `run_batch()` to return with `batch_result.global_abort_signal = True`, exactly one `pipeline_systemic_failure` event with `cause_class = ["infrastructure_failure", "worker_no_exit_report", "worker_malformed_exit_report"]` and `threshold = 3`, the pre-dispatch gate suppresses any features queued after the trip, and `just test` plus `just events-registry-audit` both pass.

## Risks

- **Phase 1 cross-coupling**: Task 2 defines `_SYSTEMIC_ERROR_TYPES` before Phase 2 consumes it. The spec lists R4 (tuple definition) under Phase 2 but R3's guard already needs it. Treating the tuple as Phase 1 infrastructure is the natural reading of "Phase 1: Upstream wiring". If the user wants strict phase boundaries (tuple in Phase 2 only), Task 2's guard can be deferred or the tuple can be split across phases — flag if needed.
- **Six pause sites, one practical match**: The `result.error in _SYSTEMIC_ERROR_TYPES` guard at sites 509, 536, 657, 706, and 1098 will rarely or never match — those branches handle merge/recovery/post-completion failures where `result.error` is a wrapped string, not a feature-level error_type. In practice only line 725 (the `else: # paused` branch) increments the systemic counter. The plan applies the guard uniformly per spec R5's literal reading; happy to consolidate to a single increment site if the user prefers, with a comment naming the architectural reasoning.
- **`cause_class` derived rather than tracked**: Deriving `cause_class` from `ctx.batch_result.features_paused` (filter + slice) instead of maintaining a parallel `recent_systemic_errors` field on `CircuitBreakerState` is simpler but ties `cause_class` correctness to `features_paused` being populated at every systemic increment site. The existing pause sites already append to `features_paused` before incrementing the counter, so the invariant holds — but a future contributor adding a new pause site that increments the counter without appending to `features_paused` would silently break `cause_class`. A comment at the threshold-trip block names this invariant.
- **Tier-vs-complexity divergence**: The lifecycle's `lifecycle_start` event recorded `tier: simple` and `criticality: medium`, but the backlog frontmatter declares `complexity: complex` / `criticality: high`. The plan was sized as if the work is structurally complex (multi-phase, 8 tasks). If the user wants the lifecycle to reflect that, emit a `complexity_override` event before Implement begins; otherwise the existing `tier: simple` classification stays in force and tier-gated checks (orchestrator review, critical-review skip, parallel dispatch policy) apply accordingly.
