# Specification: Remove /fresh, /evolve, and /retro skills

## Problem Statement

The session-feedback loop (write a retro at session end → analyze retros for recurring trends → route fixes back into the lifecycle) is no longer used. The three skills that implement it (`/cortex-core:fresh`, `/cortex-core:evolve`, `/cortex-core:retro`) and their supporting infrastructure cost ongoing routing-attention, doc maintenance, and CLAUDE.md policy upkeep without producing actionable improvements. Removing them aligns with the project's documented preference for iteratively trimming skills and workflows that have not earned their place. The downstream effect is a smaller surface for Claude to navigate and one less feedback signal that must be kept honest in policy text.

### Priority classification

All 18 requirements are Must-have. Half-execution of a deletion creates the stale-reference anti-pattern that motivates the cleanup in the first place — every requirement in this spec corresponds to a touchpoint that, if left intact, produces orphaned references to a non-existent skill or hook branch. The Non-Requirements section below is the explicit Won't-do surface.

### Atomicity & Sequencing

All 18 Must-have requirements land in a **single PR**. Within that PR, the following ordering constraints must hold:

- **R3 canonical hook edit → `just build-plugin` → commit**: The pre-commit drift hook will fail if the canonical hook is edited without regenerating `plugins/cortex-overnight/hooks/cortex-scan-lifecycle.sh`. Run `just build-plugin` after R3's canonical edit and stage both the canonical and mirror diffs in the same commit.
- **R6 `git mv` → `.gitkeep` delete → commit**: Move retros/*.md into `retros/archive/` BEFORE deleting `retros/.gitkeep`. The `git mv` makes partial state recoverable via `git checkout` if the move is interrupted; the `.gitkeep` delete is the final step that makes the operation observable as "complete."
- **R13 justfile SKILLS edit AND R18 PLUGINS dict edit must be staged in the same commit.** They are duplicated source-of-truth surfaces; staging them separately is invisible to the parity test until the next dual-source hook fires.
- **R2 `git rm -r plugins/cortex-core/skills/{fresh,evolve,retro}` must run AFTER R1's canonical-source delete**, so the orphan mirrors are removed in the same PR but with the canonical removal as the upstream cause.
- **R17's final sweep is the last gate before merge**, run after all per-requirement checks pass. R17 alone does not catch partial R2 (it excludes plugin trees by design); the per-requirement R2 acceptance is the structural backstop for plugin-mirror cleanup.

If the implementer encounters a mid-PR failure that leaves the working tree in a partially-cleaned state, the recovery path is `git reset --hard` to the last clean commit and re-execute from that point — the spec does NOT support a "partial merge" outcome.

## Non-Requirements

- This work does NOT delete `retros/archive/` or any of its contents. Historical retros are preserved as read-only archive — they retain audit value even though no skill writes new ones.
- This work does NOT introduce a deprecation notice, tombstone SKILL.md, or env-var soft-delete. There is no precedent for these patterns in this repo, and the user has explicitly chosen hard-delete.
- This work does NOT migrate or rewrite content inside archived retros. The `retros/` exclusion in `bin/cortex-archive-rewrite-paths` and `bin/cortex-archive-sample-select` correctly covers `retros/archive/` and continues to do so unchanged.
- This work does NOT close `backlog/170` outright. #170 retains its two unaddressed test failures (`single-incomplete-feature`, `claude-output-format`) as the remaining scope. A successor ticket is not filed.
- This work does NOT modify `requirements/project.md`, `requirements/observability.md`, `requirements/pipeline.md`, `requirements/multi-agent.md`, or `requirements/remote-access.md`. None reference the deleted skills (verified during research).
- This work does NOT add a successor "Manual Backlog Entry" narrative or one-liner to `docs/agentic-layer.md`. The `### 5. Self-Improvement Loop` section is deleted with no replacement.
- This work does NOT touch `harness-review`, `tests/test_archive_rewrite_paths.py`, `skills/refine/references/clarify-critic.md`, `docs/internals/*`, or `requirements/*` — verified to have no live references to the deleted surface.
- This work does NOT introduce a new env var, configuration knob, or feature flag. It removes one (`CLAUDE_AUTOMATED_SESSION`).
- **This work does NOT extend `cortex init --update` to actively prune the orphaned `retros/README.md` template from already-initialized user repos.** `cortex init --update` only adds missing scaffolded files (per `cortex_command/init/scaffold.py` — the `if dest.exists() and not overwrite: continue` branch) and `drift_files()` only iterates current shipped templates. After this work lands, an existing user repo retains its `retros/README.md` and `retros/` directory unchanged. R15's CHANGELOG bullet names the cleanup paths users should manually run; no scaffolder code is modified to prune. This is a deliberate scope choice consistent with R14's CHANGELOG-driven cleanup posture for runtime artifacts.
- This work does NOT specify the cortex CLI release tag this PR ships under. The Technical Constraints section names `docs/release-process.md` L17's SemVer commitment as the binding rule; the actual tag is selected by the maintainer cutting the next release.

## Edge Cases

- **Plugin mirror orphans go unnoticed by drift hook**: Because `tests/test_dual_source_reference_parity.py:_discover_pairs` globs canonical-side `skills/*/SKILL.md` and routes each to a plugin, an orphan mirror dir (canonical missing, mirror still present) generates no parametrized pair and is invisible to the drift test. R2 addresses this directly with explicit `git rm -r`.
- **R17 cannot detect partial R2 cleanup**: R17 excludes `plugins/cortex-core/**` and `plugins/cortex-overnight/**` from the final sweep because mirrors auto-regenerate. If R2 is half-executed (e.g., `plugins/cortex-core/skills/fresh` removed but `plugins/cortex-core/skills/evolve` retained), R17 cannot catch it — only R2's own per-requirement check does. The spec relies on every per-requirement gate firing before declaring complete; R17 is the last sweep, not the only sweep.
- **`tests/test_dual_source_reference_parity.py` PLUGINS dict drift**: This file's `PLUGINS` map duplicates the justfile SKILLS array as a Python source-of-truth. R18 lifts the edit into a numbered requirement; R13 + R18 must land in the same commit per Atomicity & Sequencing.
- **User has stale `retros/.evolve-state.json` or `lifecycle/.fresh-resume` on disk**: After `.gitignore` removal, those paths become git-visible. R14 sweeps this repo defensively. Users updating cortex on existing checkouts may have them; R15's CHANGELOG bullet names the cleanup paths they should manually run.
- **Mirror `plugins/cortex-overnight/hooks/cortex-scan-lifecycle.sh` regenerates from canonical**: After R3's canonical edit, `just build-plugin` regenerates the mirror automatically (cortex-overnight is a `BUILD_OUTPUT_PLUGINS` plugin). The pre-commit drift hook fails if the mirror is not in sync; the Atomicity & Sequencing ordering covers this.
- **`retros/archive/` already exists with content from prior archiving runs**: Verified — currently does not exist in this repo. If it did, `git mv` would refuse to overwrite, requiring manual conflict resolution.
- **CLAUDE.md edit pushes line count near 100**: Verified — current CLAUDE.md is 68 lines and this work removes lines (not adds). The 100-line cap and the `docs/policies.md` extraction trigger are not engaged.
- **Plugin auto-update timing window vs. CLI_PIN bump**: Per `docs/release-process.md` L98-120 ("Tag-before-coupling"), the cortex-overnight plugin's `CLI_PIN` is bumped in a separate commit AFTER the cortex CLI tag is pushed. A user running `/plugin update` between the two commits gets the post-deletion plugin (with regenerated hook) running against the pre-deletion CLI v0.1.0. The plugin behavior is degraded gracefully (the deleted hook code paths are no-ops; the surviving hook still injects active state). R15's CHANGELOG bullet should mention "update plugin and CLI together to avoid mid-bump skew" as user guidance.

## Technical Constraints

- **Dual-source canonical-vs-mirror discipline** (project.md L27, CLAUDE.md L40-50): edits must land in canonical sources only (`skills/`, `hooks/`, `claude/hooks/`, `bin/cortex-*`, `claude/common.py`, `claude/statusline.sh`); plugin mirrors regenerate via `just build-plugin` and the pre-commit drift hook. The exception is the plugin-tree skill *directories* for deleted skills, which require explicit `git rm -r` because the mirror regenerator does not garbage-collect orphan dirs.
- **Maintainability through simplicity** (project.md L33): this work is a direct application of the "iteratively trimming skills and workflows" principle.
- **Destructive operations preserve uncommitted state** (project.md L36): the historical retros are archived (moved to `retros/archive/` via `git mv`), not destroyed. The `.gitkeep` deletion is not destructive of user-visible artifacts (it's a directory-tracking placeholder).
- **CLAUDE.md 100-line cap** (CLAUDE.md L68): this rewrite removes lines, so the cap and the `docs/policies.md` extraction trigger are preserved as not-yet-engaged.
- **No PreUninstall lifecycle hook in Claude Code** (Anthropic issue #11240): all in-repo cleanup must land in the same PR as the skill deletion. There is no upstream-supported "run a script when this skill is uninstalled" mechanism — user-repo cleanup is delegated to CHANGELOG-named manual steps (R15).
- **Pre-commit drift hook is the structural backstop** for canonical/mirror FILE drift; it does NOT catch orphan mirror DIRECTORIES (covered by R2) or partial multi-file requirements (covered by Atomicity & Sequencing).
- **SemVer breaking-change rule** (`docs/release-process.md` L17): removing user-facing slash commands and removing the `CLAUDE_AUTOMATED_SESSION` env var is a breaking change in semver terms — the project's documented patch-release definition is "no behavior changes for users." The next release that ships this work must therefore be at least a minor bump (`0.1.x → 0.2.0`) and the CHANGELOG bullet stacks under `## [Unreleased]` until that release is cut. The spec does not pin the exact tag (that is the maintainer's call at release time), but it does require the next tag NOT be a patch.
