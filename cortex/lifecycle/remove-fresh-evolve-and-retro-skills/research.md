## Web Research

External prior art for this exact scenario is thin. The most actionable findings:

- **Anthropic SessionStart hook deletion** ([code.claude.com/docs/en/hooks](https://code.claude.com/docs/en/hooks)): Hook removal from `settings.json` (or skill frontmatter) is the supported deletion path; `disableAllHooks: true` is a temporary kill switch, not a deletion. Skill-frontmatter hooks auto-cleanup when the skill is deleted.
- **No PreUninstall lifecycle hook exists** ([Claude Code issue #11240](https://github.com/anthropics/claude-code/issues/11240)): all cleanup must be in same commit/PR as the skill deletion — there is no upstream-supported "cleanup script runs on skill removal" mechanism.
- **Plugin uninstall mechanics** ([Claude Code plugins-reference](https://code.claude.com/docs/en/plugins-reference), [issue #52456](https://github.com/anthropics/claude-code/issues/52456)): orphaned plugin-version directories auto-purge after a 7-day grace period.
- **Archive-over-delete for written-data trails** ([Reftab ITAM guide](https://www.reftab.com/blog/when-to-archive-vs-delete-assets-itam-best-practices), [Gatekeeper](https://knowledge.gatekeeperhq.com/en/docs/archive-and-delete-workflows)): both sources converge — archive when an artifact has historical/audit value, delete only for "created in error or for testing." 162 historical retros fall on the archive side; the user's decision matches the consensus pattern. Both also recommend keeping archived items out of active listings/indexes.
- **Treat removal as a project, not a deletion** ([Pensero dead-code 2026](https://pensero.ai/blog/dead-code), [LinearB](https://linearb.io/blog/dead-code)): (a) dependency analysis to find every supporting reference before removal, (b) phased removal, (c) dedicated test pass after deletion. Argues for explicit checklist coverage of CLAUDE.md prose, requirements/, docs/, hooks/, settings, .gitignore, cron entries — not just the three SKILL.md files.
- **Stale-references anti-pattern** ([Rcpp PR #1428](https://github.com/RcppCore/Rcpp/pull/1428)): canonical example of a help page that should have been removed in 2018 when the vignette it referenced was retired. The most-common cleanup mistake is leaving doc that points at the deleted thing.
- **Canonical + auto-generated mirrors** ([sqlite amalg-v-canon](https://sqlite.org/amalg-v-canon.html)): "edit the canonical, regenerate the mirror, never edit the mirror." For deletion specifically: delete from canonical, run regenerator, commit both in the same change so pre-commit drift hook stays green.
- **Telemetry/observability removal mistakes** ([Edge Delta](https://edgedelta.com/company/blog/most-common-observability-mistakes)): confirm nothing downstream silently depends on a removed feedback signal's emission. For this work: confirmed the overnight runner / harness-review do not read retros.

## Tradeoffs & Alternatives

The user pre-selected delete-everything (Option A). Five alternatives evaluated:

- **A. Delete-everything (CHOSEN)** — pros: matches established repo pattern (every prior `lifecycle/archive/remove-*` and `delete-*` is hard-delete with paired cleanup); reaches stable end-state; smallest line count; aligned with `requirements/project.md` L19/L33. Cons: irreversible without revert; ~15 files touched.
- **B. Thin /retro shell skill** — pros: preserves OQ3/OQ6 evidence channel. Cons: argument collapses because the user is *also* rewriting OQ3/OQ6 to remove that channel; leaves 20-line skill nobody wants.
- **C. Keep retro + fresh, drop only evolve** — pros: keeps `/clear`-recovery resume. Cons: half-execution; carries forward the bug from #170; doesn't match user's stated intent.
- **D. Tombstone-instead-of-delete** — pros: discoverable via slash-command. Cons: zero precedent in repo; competes for routing attention; "skill not found" failure is the right signal.
- **E. Env-var soft-delete via CORTEX_NO_FEEDBACK_LOOP** — pros: most reversible. Cons: structurally backwards (the work *removes* an env var; adding one is regression); zero precedent; "git revert" already provides reversibility.

**Precedent**: hard-delete is the only pattern in `lifecycle/archive/` for this class of work (`remove-interpreter-escape-hatch-commands`, `remove-dead-throttled-dispatch-wrapper-keep-concurrencymanager-acquire-release`, `delete-post-shift-orphan-code-scripts-hooks-and-retire-paired-requirements`, `remove-progress-update-scaffolding-from-long-running-prompts-dr-3-wave-1`, `remove-single-agent-worktree-dispatch-and-flip-recommended-default-to-current-branch`). Zero precedent for tombstone or env-var soft-delete.

**Recommended ranking: A > B > C > D > E**. Confirm A.
