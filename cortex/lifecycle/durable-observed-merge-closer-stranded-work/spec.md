# Specification: durable-observed-merge-closer-stranded-work

## Problem Statement

Three tickets have now argued about morning-review's "PR already merged" exit. None of them checked whether that exit can execute. It cannot: `walkthrough.md` L291 runs `gh pr list --head {integration_branch} --json number,url,state,title,isDraft` with no `--state`, and `gh` defaults to `--state open` ‚Äî so `state == "MERGED"` (L295) and `state == "CLOSED"` (L300) are dead branches. Verified empirically: the exact command returns `[]` for a known-merged head branch in this repo, while `--state all` returns `[{"number":25,"state":"MERGED"}]`. A genuinely already-merged PR falls into the **empty-array** branch and is told *"No PR found for `{integration_branch}`. The runner may have failed to create one. Use `/pr` to create it manually."* ‚Äî advice to open a PR for work that already merged. That is the live bug, and it is one missing flag.

Everything built above it was scaffolding on a phantom. #345 researched, specced, and shipped an advisory onto that unreachable branch. #346 proposed a durable auto-closer for it, premised on "the post-merge sync step is skipped on that exit so the local checkout is stale" ‚Äî also false: `SKILL.md` Step 5 runs `git fetch origin` + `git pull --rebase origin main` before ¬ß6 ever checks PR state, and has since `428e54ea` (2026-04-01). A prior draft of this spec then proposed *un-gating* the phantom exit, and a four-angle adversarial review returned 11 fix-invalidating findings against it.

This spec is narrowed to defects that are independently verified against the working tree and hold regardless of any premise:

1. **The merged/closed exits are unreachable** (`--state` missing) ‚Äî the root cause.
2. **`sync-allowlist.conf` is dead** ‚Äî every pattern predates the `cortex/` umbrella relocation (`c8110de5`, 2026-05-12) and matches nothing (verified 0/5), so ¬ß6a's advertised auto-resolution never fires and it aborts on the first conflict of any kind, with zero conflict-path test coverage.
3. **`_behind_count` fails open** ‚Äî any git failure returns 0 ‚Üí "Already up to date" ‚Üí exit 0, a silent false success.
4. **`outcome_router._find_backlog_item_path` step 2 returns `candidates[0]`** from a numeric-prefix glob with no ambiguity check, unlike the canonical resolver beside it. No test covers it.
5. **¬ß6b's ticket closures are never committed or pushed** ‚Äî ¬ß6b ends *"After this section, the review is complete"* and `close_tickets.py` has zero git calls, so a close written to repair a failed overnight write-back never reaches `main`.

Deferred with corrected framing (each needs design the research does not yet have): auto-closing on the merged exit, the orphan reconciler, and re-pick suppression. See Non-Requirements.

## Phases

- **Phase 1: Sync-path repair** ‚Äî make `cortex-git-sync-rebase` do what it claims: live allowlist, no silent false success, conflict-path tests.
- **Phase 2: Closer id-resolution hardening** ‚Äî remove the blind first-match in the write-back's item lookup.
- **Phase 3: Merged-exit reachability** ‚Äî make the merged/closed exits executable and honest. Report only; no closure decision.
- **Phase 4: Durable close** ‚Äî commit and push ¬ß6b's closures via a real push path, or say plainly that it did not.

## Requirements

1. **(Must)** **Repair `sync-allowlist.conf` so its patterns match real repo paths.** Every pattern omits the `cortex/` umbrella prefix (`backlog/[0-9]*-*.md` vs. the real `cortex/backlog/346-*.md`), so `_matches_allowlist` never matches and every conflict falls to `non_allowlist` ‚Üí abort. **Acceptance**: a test asserts each non-comment pattern in `cortex_command/overnight/sync-allowlist.conf` matches at least one representative real path (`cortex/backlog/346-x.md`, `cortex/backlog/index.md`, `cortex/lifecycle/pipeline-events.log`, `cortex/lifecycle/foo/plan.md`, `cortex/lifecycle/sessions/s1/`), and **fails if any pattern matches none** ‚Äî so a future relocation cannot silently re-rot it. **Phase**: Sync-path repair.

2. **(Must)** **Cover `sync_rebase`'s conflict-resolution path with tests.** `tests/test_git_sync_rebase.py` has exactly two tests (noop, clean-rebase) and zero coverage of the allowlist/conflict loop ‚Äî which is how Requirement 1's defect survived two months. **Acceptance**: new tests exercise (i) a conflict in an allowlist file ‚Üí auto-resolved `--theirs`, rebase completes, exit 0; (ii) a conflict in a non-allowlist file ‚Üí rebase aborted, exit 1, no rebase left in progress (`git status --porcelain` shows no `rebase-merge`/`rebase-apply`). `just test` exits 0. **Phase**: Sync-path repair.

3. **(Should)** **Resolve the unreachable `_MAX_NON_ALLOWLIST` threshold.** `sync_rebase.py` L250's `if len(non_allowlist) > _MAX_NON_ALLOWLIST` can never fire ‚Äî L260's `if non_allowlist:` (‚â•1) aborts first, and both return 1. Remove it, or give it a behavior a test can distinguish from the ‚â•1 abort. **Acceptance**: `grep -c "_MAX_NON_ALLOWLIST" cortex_command/git/sync_rebase.py` = 0, OR a test asserts a >3-conflict run produces an outcome distinguishable from a 1-conflict run. **Phase**: Sync-path repair.

4. **(Must)** **Stop `_behind_count` reporting "up to date" on failure.** It returns `0` on any non-zero git rc and on `ValueError` (L136‚Äì141), so a missing `origin/main`, a shallow clone, an auth failure, or network loss all render as "Already up to date with origin/main ‚Äî nothing to rebase" ‚Üí exit 0 ‚Üí a silent false success that pushes nothing while reporting success. Distinguish *git failed* from *legitimately zero*; surface the failure on a **new, unoccupied** exit code (`1` = conflict and `2` = push-failed are taken per the module header). **Acceptance**: a test stubbing a failing `git rev-list` asserts the run exits non-zero, on a code not equal to 1 or 2, with a diagnostic naming the behind-count step; a second test asserts a genuine `behind == 0` still exits 0. **Phase**: Sync-path repair.

5. **(Must)** **Remove the blind first-match from the overnight write-back's item lookup.** `outcome_router.py::_find_backlog_item_path` step 2 (L424‚Äì429) does `sorted(backlog_dir.glob(f"{padded}-*.md"))` then `return candidates[0]` with no ambiguity check ‚Äî weaker than the canonical `resolve_item.resolve()` beside it, which surfaces `ambiguous` rather than guessing. Delete step 2 and route through the canonical resolver, threading `backlog_id` into the step-3 call (L434 currently passes only `feature`). **Acceptance**: `grep -c "candidates\[0\]" cortex_command/overnight/outcome_router.py` = 0; a new test with two backlog files sharing a numeric prefix asserts the lookup returns `None` rather than a silently-chosen file; a second asserts a unique numeric id still resolves; existing `tests/test_outcome_router.py` stays green. **Phase**: Closer id-resolution hardening.

6. **(Should)** **Thread the ticket `uuid` to close time.** `resolve()`'s numeric step matches on filename only (`_resolve_numeric` L156‚Äì170: `re.match(r"^(\d+)-", p.name)`), so Requirement 5 fixes *ambiguity* but not *renumbering* ‚Äî only the uuid path (step 1) is renumber-proof. `BacklogItem` carries `uuid` but `plan.py` (L422‚Äì429) drops it when building `OvernightFeatureStatus`. Add a `uuid` field to `OvernightFeatureStatus`, populate it in `plan.py`, thread it to the write-back, prefer uuid resolution when present. **Should**, not Must: renumbering has occurred once (`94eddf11`, root-caused and fixed by #231), zero duplicate numeric prefixes exist today, and Requirement 5 already removes the silent-wrong-match. **Acceptance**: a test constructs a fixture where a captured `backlog_id` points at a *different* item than the uuid identifies, and asserts the write-back resolves the uuid's item or returns `None` ‚Äî never the numerically-matched wrong item. **Phase**: Closer id-resolution hardening.

7. **(Must)** **Make the merged/closed PR exits reachable.** `walkthrough.md` L291's `gh pr list --head {integration_branch} --json number,url,state,title,isDraft` omits `--state`; `gh` defaults to `open`, so the L295 `MERGED` and L300 `CLOSED` branches are dead and an already-merged PR wrongly reports "No PR found‚Ä¶ Use `/pr` to create it manually." Add an explicit state filter so those branches can execute. **Acceptance**: the `gh pr list --head` invocation in `walkthrough.md` carries an explicit `--state` argument ‚Äî `grep -c 'gh pr list --head[^`]*--state' skills/morning-review/references/walkthrough.md` ‚â• 1; and a test asserts the documented command string in the walkthrough contains `--state`, so the flag cannot be silently dropped again. **Phase**: Merged-exit reachability.

8. **(Must)** **Make the merged/closed exits honest, and report-only.** With Requirement 7 the exits execute for the first time, so their behavior is now live and must be correct rather than inherited. The merged exit reports that the PR merged out-of-band and that this session's completed-feature tickets are worth checking; it **does not** close anything and **does not** decide that this session's work landed. Widening `--state` also admits *prior* PRs bearing a reused head name (`plan.py:391` builds `overnight/{session_id}` from a minute-granular `strftime("overnight-%Y-%m-%d-%H%M")`; the uniquifying loop L363‚Äì370 tests only `os.path.exists(session_dir(session_id))`, a gitignored directory; L391‚Äì407 force-deletes and reuses a colliding local branch while the prior session's merged PR persists on GitHub) ‚Äî so **more than one match is a real case and must not be silently first-matched**. **Acceptance**: the merged-exit span contains no close literal (`grep -cE "update[-_]item|close-tickets"` over that span = 0); a test asserts the walkthrough's PR-query section names a more-than-one-match branch ‚Äî `grep -cE "more than one|multiple PRs|>1"` over the ¬ß6 step-2 span ‚â• 1 ‚Äî and that the span contains no instruction to take the first element (`grep -cE "first match|\[0\]"` = 0); `tests/test_morning_review_status_close_ordering.py` stays green in full. **Phase**: Merged-exit reachability.

9. **(Must)** **Correct the stale advisory text at the merged exit.** L296‚Äì297 tells the operator local `main` "may lag the out-of-band merge ‚Äî `git fetch origin main` (or pull) first". Step 5 already fetched and rebased before ¬ß6, so this is a hedge against a minutes-wide race, not the general case it implies ‚Äî and it was written believing the sync is skipped entirely. State the narrow truth or drop the instruction; do not keep prose that implies a general staleness that does not exist. **Acceptance**: the merged-exit span no longer asserts that Section 6a's sync being skipped leaves local `main` behind as the normal case ‚Äî `grep -c "This skips" ` over that span = 0 ‚Äî and any retained fetch advice is scoped to the race. **Phase**: Merged-exit reachability.

10. **(Must)** **Commit and push ¬ß6b's ticket closures.** ¬ß6b writes closes and the review ends (*"After this section, the review is complete"*); `close_tickets.py` has zero git calls. So when the overnight write-back fails, ¬ß6b's local repair never reaches `main` ‚Äî the "landed but open" gap. Add a commit+push step after ¬ß6b as a CLI verb. It must **not** delegate the push to `cortex-git-sync-rebase`: that binary returns at `if behind == 0: return 0` (L187‚Äì190) more than 100 lines before its `git push` (L292‚Äì294), and Phase 4's situation ‚Äî a fresh local commit, remote `main` unmoved ‚Äî *is* `behind == 0`; its exit 0 is also documented as meaning "nothing to rebase", so its success signal cannot evidence a push. The verb performs its own `git push` and verifies the outcome directly. **Acceptance**: the verb, run after a simulated ¬ß6b close in a fixture repo whose remote has not moved (`behind == 0`), produces a commit containing the changed `cortex/backlog/*.md` **and** leaves `git rev-list origin/main..HEAD --count` = 0 ‚Äî a test that fails if the push is delegated to `sync_rebase`. **Phase**: Durable close.

11. **(Must)** **Never report a close as durable without a verified push.** A local close with no push inverts the bug: the ticket reads `complete` locally while still open on `main` ‚Äî the same gap, hidden behind a false local status instead of a visible open ticket. The verb's `pushed` field must be derived from an observed post-push state, never from a delegated exit code. **Acceptance**: a test simulating a rejected non-fast-forward push asserts the verb returns `pushed: false`, names the un-pushed ticket ids, and that the caller-visible result is not success; a second asserts `pushed: true` is returned only when `git rev-list origin/main..HEAD --count` = 0 is observed after the push. **Phase**: Durable close.

12. **(Must)** **Correct #346's false premise in the ticket body.** The Why asserts the post-merge sync is skipped and the local checkout is stale; Step 5 rebases before ¬ß6, and has since `428e54ea`. Leaving it uncorrected misleads the next reader and re-seeds the same wrong design ‚Äî as it already did twice. **Acceptance**: over `cortex/backlog/346-*.md`, `grep -cE "sync step is skipped|checkout is stale|no durable remote signal"` = 0 (the false claims are gone) AND `grep -cE "\-\-state"` ‚â• 1 (the reachability defect is named) AND `grep -cE "push"` ‚â• 1 (the missing push is named). **Phase**: Merged-exit reachability.

13. **(Must)** **Regenerate the dual-source mirror in the same commit.** Canonical `skills/morning-review/` edits require `plugins/cortex-overnight/skills/morning-review/` regenerated via `just build-plugin` and staged in the same commit. **Acceptance**: `tests/test_dual_source_reference_parity.py` exits 0; the drift pre-commit hook passes. **Phase**: Merged-exit reachability.

## Non-Requirements

- Does **NOT** auto-close tickets on the merged exit. Requires establishing that *this session's work* is on `main` ‚Äî which `gh pr list --head <name>` cannot do (a head-name string match over reusable, minute-granular, gitignore-deduped branch names), and which `state: MERGED` cannot do either (GitHub keeps it permanently after a revert). A sound version needs `git merge-base --is-ancestor <mergeCommit> origin/main` plus a multi-PR disambiguation rule. Deferred to its own ticket, gated on Requirement 7 shipping first ‚Äî the exit must be able to execute before its behavior can be designed.
- Does **NOT** build a remote-authoritative on-main read (`gh api ‚Ä¶/contents?ref=main`). Redundant with Step 5's rebase in the dominant case, and it would fork a second id-resolution mechanism parallel to `resolve_item.resolve()`.
- Does **NOT** build the orphan reconciler (branches/PRs/worktrees from declined/abandoned sessions). The declined path is manual today and the walkthrough says so, but zero orphans exist right now (`git branch -a --list '*integration*' '*pipeline/*'` returns nothing; `git worktree list` shows only `main`), and there is no recorded occurrence ‚Äî so it does not clear `project.md`'s "Complexity must earn its place by solving a real problem now", nor the MUST-escalation evidence bar. Its own ticket, with a census as the trigger.
- Does **NOT** add re-pick suppression for stranded features. The gap is real (`BacklogItem` has no `recoverable_branch` field; `in_progress` is eligible; the merged-branch gate fails open on an unmerged branch) but the obvious design is defective: overloading `recoverable_branch` as a suppression flag makes "reset" and "destroy the recovery pointer" the same keystroke, its clear-to-reset gesture is a no-op through `update_item` (which writes the literal `"null"`, truthy under `backlog.py`'s `fm.get(...) or None` idiom), and excluding a feature from `filter_ready()` removes it from `state.features` ‚Äî the only collection `render_built_merge_blocked` reads ‚Äî so it cannot be surfaced. A sound version needs a dedicated suppression field and a backlog-sourced report section. Its own ticket.
- Does **NOT** fix the unguarded `git worktree remove --force` on the merge-success path (walkthrough L327‚Äì330), a pre-existing hazard shipping today. Its own ticket.
- Does **NOT** reconcile the stale `TERMINAL_STATUSES` divergence note (`project.md:36` and `common.py` L178‚Äì179 claim `overnight/backlog.py` defines its own; it now imports the canonical one). Documentation drift; its own follow-up.
- Does **NOT** align `cortex/backlog/ready.py`'s hand-mirrored `_ELIGIBLE_STATUSES`. Only relevant to the deferred suppression work.
- Does **NOT** backfill `uuid` into the 25 of 374 legacy items lacking one ‚Äî see Edge Cases.

## Edge Cases

- **Widened `--state` returns more than one PR for a head name**: real, because session ids are minute-granular and deduped only against a gitignored directory, and `plan.py` force-deletes and reuses colliding branch names. The walkthrough surfaces the candidates; it never first-matches. (The repo's sibling lifecycle already enumerates this case for slug reuse.)
- **PR merged then reverted on `main`**: `state` stays `MERGED` permanently. Because Requirement 8 is report-only, the exit reports a merge that occurred and asserts nothing about current `main` ‚Äî the operator decides. This is precisely why auto-close is deferred rather than specified.
- **Push fails after a successful close** (rejected non-fast-forward, network, race): the close stays committed locally, the verb returns `pushed: false` naming the un-pushed tickets, and the review does not report success.
- **Remote unmoved at push time (`behind == 0`)**: the dominant Phase 4 case, and the one that defeats a delegated `sync_rebase` push. The verb pushes directly and verifies with `git rev-list origin/main..HEAD --count`.
- **Ticket already `complete` on `main`** (write-back succeeded ‚Äî the common case): ¬ß6b's re-close is event-idempotent (the `status_changed` append is guarded on `new != old`), but not inert ‚Äî `atomic_write`, the `updated:` bump, the terminal cascade, and index regeneration all re-run. The commit step must tolerate an `updated:`-only or empty diff without failing, and must not report a push it did not perform.
- **Ticket has no `uuid` (25 of 374 legacy items)**: the write-back falls back to canonical numeric resolution, which after Requirement 5 refuses on ambiguity rather than guessing ‚Äî no worse than today, strictly better on collision.
- **No completed features this session**: ¬ß6b and the commit+push step are no-ops; no empty commit is created.
- **`gh` unavailable / unauthenticated / offline**: a normal outcome at review time. The PR query reports inconclusive rather than inferring; nothing is closed on an unverifiable state.

## Changes to Existing Behavior

- **MODIFIED**: `gh pr list --head` runs without `--state` (defaulting to `open`, making the MERGED/CLOSED branches dead and reporting "No PR found" for merged PRs) ‚Üí carries an explicit state filter, so those branches execute for the first time.
- **MODIFIED**: the merged exit's advisory implies local `main` is generally stale because ¬ß6a is skipped ‚Üí states the narrow race truth, or drops the claim.
- **MODIFIED**: `sync-allowlist.conf` patterns match nothing (so ¬ß6a aborts on every conflict) ‚Üí match the real `cortex/`-prefixed paths.
- **MODIFIED**: `_behind_count` returns 0 (‚Üí "up to date", exit 0) on git failure ‚Üí surfaces the failure on an unoccupied exit code.
- **MODIFIED**: `outcome_router._find_backlog_item_path` returns `candidates[0]` from a numeric glob ‚Üí routes through the canonical resolver and refuses on ambiguity.
- **ADDED**: a commit+push step after ¬ß6b with a directly-verified push outcome.
- **ADDED**: a rule for a multi-PR result at the head-branch query.
- **REMOVED**: the dead `_MAX_NON_ALLOWLIST` branch (or it gains a real, tested behavior).

## Technical Constraints

- **Verb-first, prose-lean.** Logic lands in `cortex_command/` verbs with the established always-exit-0 JSON-struct contract (`close_tickets.py`, `backlog_backend_cli.py`); `walkthrough.md` gets the minimum hook. Prose costs context on every skill load. **No numeric line cap is set**: the nearest precedent (¬ß6a, L341‚Äì357) is 17 lines for one verb call plus a 3-outcome map, so a small budget would be unmeetable, and the usual gates cannot enforce one anyway ‚Äî `tests/test_skill_size_budget.py` reads only `*/SKILL.md` (never `references/walkthrough.md`) and `tests/test_l1_surface_ratchet.py` measures only frontmatter bytes. This is therefore a **review criterion, not a test-enforced gate** ‚Äî stated plainly rather than dressed as a check that cannot fire. Requirement 8/9's changes are control flow and are necessarily prose; keep them terse, and reclaim lines by deleting advisory text the corrected behavior obsoletes.
- **Acceptance criteria must test the property, not a proxy for it.** The prior draft failed review on exactly this: a `grep` for shell-string git spellings scored 0 against `conflict.py`, which force-removes worktrees via list-form subprocess; a fixture-fed test asserted a code path the live `gh` query never reaches. Every criterion here names an observable the implementation cannot satisfy while the behavior is absent.
- **Ordering guard is load-bearing.** `tests/test_morning_review_status_close_ordering.py` requires the first close literal to appear after the first `gh pr merge` literal, `Section 6b` after the `Section 6` heading, an explicit skip-gate phrase in ¬ß6b, and zero close-pattern occurrences in `SKILL.md`. It is lexical only ‚Äî it cannot see merge identity, so it is not evidence of closure safety. Satisfy it; do not amend it.
- **Dual-source mirror**: `skills/morning-review/` ‚Üí `plugins/cortex-overnight/skills/morning-review/` via `just build-plugin`, same commit.
- **Backend-blind local engine**: `update_item.py`/`resolve_item.py` must not gain backend-routing logic; the three-arm gate stays at the consumer layer. Overnight supports only the `cortex-backlog` backend.
- **MUST-escalation policy**: no MUST/CRITICAL/REQUIRED tokens in new prose without a linked evidence artifact. Soft declarative phrasing only.
- **Atomic state writes**: tempfile + `os.replace()`; no new locking.
- **Test command**: `just test` (per `cortex/lifecycle.config.md`).
