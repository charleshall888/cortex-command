Tier: complex. Criticality: critical.

Local main is **not** stale at that exit in the dominant case. The "durable remote-authoritative on-main primitive" this ticket exists to build is largely **redundant with a rebase that already happens**.

The genuine residue is small and different from what the ticket describes:
1. **No commit/push of §6b's closes** — on *every* path, not just the merged exit. Step 5's `/commit` runs *before* §6b's closures exist, and `close_tickets.py` contains zero git calls. This is #346's premise 2 ("there is no push path on that exit"), which **survives intact** and is the real (a).

3. **A report-only reconciler** for the declined/abandoned path, which is 100% manual today and needs a **requirements amendment before it deletes anything**.

**Recommendation: do not refine against the current Why.** Re-derive the ticket from verified facts, decompose per the ticket's own sanction, and ship in this order: allowlist fix → (c) → (a)-as-push → (b) report-only.

## Web Research

- **Merge durability is a conjunction, never one signal:** PR merged **and** the merge commit is an ancestor of the current default-branch tip (`git merge-base --is-ancestor`) **and** the ticket's state re-derived fresh at action time. GitHub itself ships a per-repo "auto-close issues with merged linked PRs" toggle *because* merged ≠ done; closing keywords are ignored unless the PR targets the default branch. Linear requires an explicit `externally-merged` label rather than inferring. GitLab/Jira gate on default-branch landing too.
- **Remote reads:** `gh api …/contents/{path}?ref={sha}` accepts a merge-commit SHA but takes an **exact path** (no globbing) — so an id→filename resolution step is still required, which would fork a second resolver. `git ls-remote` is the cheapest tip probe. `git archive --remote` is disabled on GitHub. GraphQL (`object(expression: "SHA:path")`) can batch. **Read-replica lag:** 404s for seconds after a merge are widely reported — retry-once-with-backoff; never conclude "not landed" from a fresh 404.
- **GC safety:** two-tier mark-then-sweep is git's own design (`gc.reflogExpire` 90d / `gc.reflogExpireUnreachable` 30d, then `prune --expire 2.weeks`). `git worktree prune --expire` and `git worktree lock` provide grace/protection. `probot/stale` is the canonical mark→wait→delete shape. Kubernetes ownerReferences + foreground deletion: don't delete the owner until blocking dependents are confirmed gone.
- **Backoff:** exponential + **full jitter** + a hard cap (AWS). CrashLoopBackOff: 10s→300s cap, reset only after a **sustained** 10-minute healthy window (not a single success). DLQ/Dead-set for poison pills — bounded, inspectable, manually redrivable, never silently dropped.
- **Level-triggered reconciliation (k8s):** an event is a nudge to re-derive state, never ground truth; idempotent upsert; a restarted controller with zero memory must still converge. Terraform's `-refresh-only` re-queries before deciding drift. Directly applicable: re-derive "is this branch/PR still owned by a live session" *at sweep time*, never trust a mark made minutes earlier.

## On-Main Durability Primitive

- **The exit's only signal is `gh pr list`'s `state`** (walkthrough L291–300). §6a and §6b are both structurally unreachable from it.
- **Two separable facts:** PR MERGED proves (i) code landed; it says nothing about (ii) the ticket's `status: complete` write landing. The write-back is separate and fallible — `outcome_router.py` L465–502 wraps it in a swallow-all `try/except` emitting `BACKLOG_WRITE_FAILED` to a gitignored log.
- **THE INVERSION HAZARD (load-bearing):** adding a local close **without a paired automated push inverts the bug** — the ticket reads `complete` locally while still open on main: the same "landed but open" gap, now *hidden behind a false local status* instead of a visible open ticket. Strictly worse than today's advisory.
- **`_head_has_feature_complete` is not in this code path at all.** It lives in `cortex_command/lifecycle/complete_route.py:276` (the *interactive* lifecycle); `close_tickets.py` imports only `backlog.update_item`, `backlog.resolve_item`, `common`. Any reasoning that routes through it is a fabricated mechanism (#345's research L15 seeded this).
- **Parent-epic cascade hazard — verified real:** `_check_and_close_parent` (`update_item.py` L249–331) globs **local** sibling frontmatter and closes the parent when all look terminal. Its already-terminal short-circuit (L290–292) makes a repeat close safe, but not a stale-sibling close.
- **Re-close is event-idempotent but NOT inert:** the `status_changed` append is properly guarded (L392, `new != old`), but `atomic_write` + the `updated:` bump (L382–385), the terminal cascade (L411, gated only on *new* being terminal), and the index-regen subprocess (L417–421) all re-run unconditionally. #346's Edge "a re-close stays a safe no-op" holds for **correctness**, not for **side effects**.
- **Reliability bar, if a remote check were ever built:** anchor to `mergeCommit.oid`, not the moving `main`; separately read current main to detect revert/reopen; be rename/renumber robust; **fail-closed** (inconclusive → advisory) on missing remote, unauthenticated `gh`, offline, or rate-limit. Offline is a normal outcome at review time, not an exception.

## Stranded-Work Reconciliation & Destructive Safety

- **#281 and #284 are both SHIPPED and `status: complete` — the incident is closed on both fronts.** #281's root cause was home-repo merges resolving to the wrong tree (colliding with the integration worktree that owned `overnight/<session_id>`), fixed by routing merges through the integration worktree. #284 built the surfacing: a genuine merge conflict now routes to `deferred` + `recoverable_branch`, writes back `in_progress` + the branch (not a from-scratch `backlog` re-queue), and surfaces "built, merge-blocked, recoverable on `pipeline/<feature>`" positively, excluded from `[ZERO PROGRESS]`. **Framing #281 as an open problem is stale.**
- **(b)'s actual live gap is the DECLINED/ABANDONED path**, and the walkthrough admits it in its own text at **L314–320**: *"WARNING: closing does NOT delete the integration branch {integration_branch} or its worktree. Run `git push origin --delete {integration_branch}` and check `git worktree list` for orphans manually."* 100% manual today.
- **`gc_demo_worktrees.py` is a disjoint artifact class, not a baseline to extend.** It sweeps `demo-overnight-*` throwaway copies under `$TMPDIR`; it never touches branches or PRs; `git worktree remove` never deletes the checked-out branch, so it *cannot* strand committed work. Its own spec (NR13) explicitly scopes out abandoned-repo cleanup, and item 8 forbids `--force` and any general `--prefix`/`--exclude`. **Reuse the pattern** (session-scoped sweep, active-session exclusion first, per-item skip-and-continue, single trailing prune, named script) — not the code.
- **THE KEY SAFETY FINDING: skip-on-uncommitted-state is NOT sufficient.** #281's stranded work was **12 clean commits** — after a successful build the worktree is clean *by definition*; the risk lives entirely in **committed-but-unmerged** state. A dirtiness check passes trivially on exactly the branch you must not delete. What's needed is a **branch-reachability guard** (`git merge-base --is-ancestor <branch> <target>`), the committed-work analog, which **nothing in the repo provides today**.
- `pipeline/worktree.py::cleanup_worktree` (L332–400) has **no uncommitted check of any kind**; its `git branch -d` (non-force) refuses an unmerged branch, but that protection is incidental to one call site, not a designed invariant. The merge-*success* path already runs an unguarded `git worktree remove --force` (walkthrough L327–330) — a pre-existing hazard shipping today.
- **`resolve_branch_name`'s `-2`/`-3` suffix minting** (`pipeline/worktree.py` L98–116) means a re-picked feature can leave *multiple* `pipeline/<feature>[-N]` branches; only one may hold the real work. A naive age/prefix sweep could delete the wrong one, or all.
- **Recovery has no bounded window.** Deleting a branch ref drops its reflog immediately; `git gc --auto` triggers on object thresholds, not a schedule, so an unreachable commit survives days-to-months with no guarantee. GitHub publishes no restore SLA. **Effectively irreversible** — the only real safety net is pre-deletion proof plus a durable record of the SHA. There is no events-registry entry for branch/worktree deletion today.

## Re-Pick Backoff Policy

- **Selection is a stateless per-session recompute.** `select_overnight_batch()` → `filter_ready()` (L448–632) re-derives eligibility every session from on-disk state; nothing tracks "picked before" or "how many times."
- **DEFINITIVE: `recoverable_branch` does NOT solve cross-session re-pick.** Traced: `status: in_progress` is in `ELIGIBLE_STATUSES` (L42) → passes gate 1; `_is_pipeline_branch_merged` **fails open precisely because the branch is unmerged** → doesn't exclude; and `BacklogItem` (L52–130) has **no `recoverable_branch` field at all** — `parse_backlog_dir`/`load_from_index` never read it. A stranded feature is immediately re-pickable next session and looks like any other `in_progress` item. This is a real, open gap.
- **All existing retry machinery is intra-session:** `recovery_attempts`/`recovery_depth` (`state.py` L105–106), `crash_recovery_attempts` (L265), `SYSTEMIC_FAILURE_THRESHOLD = 3` and the batch circuit breaker. None persist across sessions.
- **Durable state must live in backlog frontmatter** — `cortex/lifecycle/sessions/` is gitignored and `overnight-state.json` is session-scoped scratch the selection pipeline never reads back. Frontmatter is the channel `recoverable_branch` already uses.
- **Minimum shape:** parse `recoverable_branch` into `BacklogItem`; add `repick_attempts` and/or `repick_after`; add a `filter_ready()` gate; surface backed-off items on the morning report with a manual-reset path (extending `render_built_merge_blocked`, `report.py` L1598–1636).
- **Stale doc note:** `project.md:36` and `common.py` L178–179 claim `overnight/backlog.py` defines a divergent `TERMINAL_STATUSES`; it now **imports** the canonical one (L28). The documented divergence appears already fixed.
