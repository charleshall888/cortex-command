# Specification: overnight-watchdogthread-is-a-blind-30

> Epic/background: this corrects the in-process stall watchdog premise that ADR-0011 (#308) and ticket #312 both reference. Full research and the operator's gate decisions: [[research]] (`cortex/lifecycle/overnight-watchdogthread-is-a-blind-30/research.md`). Revised after a 4-angle critical review (1 A-class objection, several framing/B findings applied).

## Problem Statement

The overnight runner's `WatchdogThread` (`cortex_command/overnight/runner_primitives.py`) is documented as a "stall-detection / event-silence" watchdog, but `run()` is a blind fixed-duration timer: it does `elapsed += poll_interval` and calls `_kill_for_stall()` once `elapsed > STALL_TIMEOUT_SECONDS` (1800.0), reading no activity signal and never resetting. Result: every orchestrator and batch_runner subprocess is hard-killed at exactly 30 minutes regardless of how productively it is working (evidence: run-now session `overnight-2026-06-22-1106` killed a healthy `re-tune-late-join-probe-for` worker 18 s after its last progress message), so no feature batch needing >30 min can complete. The beneficiary is any developer running overnight sessions (run-now or scheduled). The fix makes the watchdog reset its timer on a **child-written activity signal** so productive work survives, while genuine *silence* still trips. This is industry-standard (gunicorn's worker/arbiter file-touch timeout; systemd `sd_notify` watchdog): the keepalive comes from the watched child, never the supervisor's own loop.

**Honest scope of the safety property.** The inactivity tier is a *silence* detector, not a *progress* detector: it resets on any child-written byte, so a child that keeps emitting output while making no real progress (a tool-retry / apology loop) is **not** caught by the inactivity tier. For that failure class — and for any site whose child signal turns out weak or absent — the never-reset **absolute ceiling** is the backstop, and the watchdog's effective kill latency for a loud-but-stuck child loosens from today's hard 30 min to that ceiling. That trade is accepted deliberately (a stuck-but-quiet child is the common case and is still caught fast); the ceiling-kill is logged with a distinct reason (Req 7) so a mis-tuned ceiling is visible and tunable.

## Phases

- **Phase 1: Activity-aware watchdog core + batch_runner site** — convert `WatchdogThread` from a blind tick-counter into an activity-aware inactivity timer plus a never-reset absolute ceiling (monotonic clock, injectable seams, kill-reason marker), and wire the **batch_runner** watchdog to the worker-written `pipeline-events.log`. Fixes the demonstrated failure (the evidence kill was a batch_runner). The orchestrator watchdog is left constructed with **no activity probe** in Phase 1, which by Req 2's default behaves exactly as today's blind 1800 s timer — i.e. the orchestrator/planning path is unchanged-from-today after Phase 1, not a new hybrid state.
- **Phase 2: Orchestrator child-progress signal + docs** — give the orchestrator/planning site a child-driven signal, **de-risked first** (Req 12) because the leading mechanism (stream-json stdout growth) is threatened by a known Claude Code file-buffering bug (#25670). Only if the de-risk passes do the flag-flip + parser changes land; otherwise the orchestrator falls back (PTY-wrapped stdout, orchestrator self-heartbeat, or ceiling-only) per an operator decision. Then correct the source-of-truth docs.

**Priority (MoSCoW).** **Phase 1 is must-have** — it fixes the demonstrated (batch_runner) failure and ships independently; note that a known instance of the same blind-timer defect remains on the orchestrator/planning path until Phase 2 (runner.py:411 documents that planning can legitimately run silent >30 min, so the orchestrator is *not* yet fixed — it is merely unchanged). **Phase 2 is must-have per the operator's full-fix decision**, but is genuinely deferrable: if Req 12's de-risk fails, the orchestrator degrades to ceiling-only (a documented, accepted "raised timeout" for that site) pending a follow-up. **All requirements are must-have, including Req 7 (kill-reason marker)** — it is the only feedback loop that surfaces a mis-tuned ceiling, so it is not optional while the ceiling value is an admitted guess. There are no won't-do requirements; intentional exclusions live in `## Non-Requirements`.

## Requirements

**Phase 1**

1. `WatchdogThread.run()` measures inactivity with a **monotonic clock**, not by counting poll ticks. It tracks `last_activity_at` and `started_at` from an injectable time source (default `time.monotonic`); inactivity = `now - last_activity_at`. **Phase**: Activity-aware watchdog core + batch_runner site. *AC*: `grep -c "monotonic" cortex_command/overnight/runner_primitives.py` ≥ 1; `just test` exits 0.

2. `WatchdogThread.__init__` gains two production-defaulted seams: an **activity-probe** (a callable returning a `(size, mtime_ns)` tuple, or `None` when the source is absent/unsupplied) and a **clock**. **Default probe is `None` → the inactivity tier never resets, i.e. the watchdog behaves exactly as today's blind `STALL_TIMEOUT_SECONDS` timer.** This makes the orchestrator spawn site (which is not wired to a probe until Phase 2) unchanged-from-today when Phase 1 ships alone. **Phase**: Activity-aware watchdog core + batch_runner site. *AC*: a test constructs `WatchdogThread` with a fake clock + fake probe and drives reset deterministically — `grep -cE "def test_.*(reset|activity|ceiling)" tests/test_runner_threading.py` ≥ 1; a test asserts that with `probe=None` the watchdog still kills after `timeout_seconds` (blind-timer-equivalent); `just test` exits 0.

3. **Inactivity tier**: when the activity probe advances — `size` grew **OR** `mtime_ns` advanced, or first appearance from a missing/`None` baseline — reset the inactivity timer to the current monotonic instant. Kill via the existing `_kill_for_stall()` only when inactivity exceeds `STALL_TIMEOUT_SECONDS` (kept at 1800). **Phase**: Activity-aware watchdog core + batch_runner site. *AC*: a test where the probe keeps advancing past the timeout shows the child is **not** killed; a test with a non-advancing probe still kills after the timeout. `just test` exits 0.

4. **Absolute-ceiling tier**: a never-reset `ABSOLUTE_CEILING_SECONDS` (set to `14400.0` = 4 h, with a `#:` comment stating it is a deliberate conservative guess to revisit once session-duration data accumulates — and that **ceiling-kills are logged with the Req 7 reason marker, which is the data the revisit consumes**) kills the child once `now - started_at` exceeds it, regardless of activity. **Phase**: Activity-aware watchdog core + batch_runner site. *AC*: `grep -c "ABSOLUTE_CEILING_SECONDS" cortex_command/overnight/runner_primitives.py` ≥ 1; a test with a forever-advancing probe and a tiny injected ceiling shows the child **is** killed by the ceiling.

5. **Robust stat**: a missing file = "no activity yet" (no reset, no crash); reset on **first appearance**; a `size` decrease (rotation/truncation) re-baselines without counting as a stall reset; every stat error is caught (`except OSError`) so a transient blip never silently kills the daemon thread. **Phase**: Activity-aware watchdog core + batch_runner site. *AC*: tests cover missing-then-appears and stat-raises paths; `just test` exits 0.

6. New reset/ceiling state is **thread-confined to `run()`** — no field added to `WatchdogContext`/`RunnerCoordination` for activity bookkeeping (the kill-reason marker of Req 7 is the only permitted new shared field), and the reset path acquires **no locks** (`kill_lock` only on the kill path, `state_lock` untouched). **Phase**: Activity-aware watchdog core + batch_runner site. *AC*: no `kill_lock`/`state_lock` acquisition appears in the reset branch of `run()`; `WatchdogContext` carries only `stall_flag` plus the Req 7 marker.

7. **Kill-reason marker (must-have)**: the implementation records which tier fired — `inactivity` vs `ceiling` — and threads it to the stall-handling log so the morning report can distinguish "the worker stalled" from "hit the ceiling — tune it." This is the **sole feedback loop on the guessed ceiling value** (Req 4); it is not optional. **Phase**: Activity-aware watchdog core + batch_runner site. *AC*: the stall branch carries the reason; `grep -cE "ceiling|inactivity" cortex_command/overnight/runner.py` ≥ 1 in the stall-handling region; a test asserts a ceiling-kill is tagged distinctly from an inactivity-kill.

8. **Shutdown + poll ordering preserved**: the `shutdown_event.wait()` wake-return and the `proc.poll()` early-return run **before** any stat/reset, so a shutdown request always kills promptly, reset never extends a child past a shutdown request, and a dead child's watchdog exits within one poll interval. **Phase**: Activity-aware watchdog core + batch_runner site. *AC*: existing `test_shutdown_event_wakes_watchdog_sleep` and `test_concurrent_cancel_and_stall_dont_double_kill` still pass.

9. **batch_runner watchdog wired to `pipeline-events.log`**: `_spawn_batch_runner` receives the session `pipeline-events.log` path (threaded from the call site, ~runner.py:2841) and builds the watchdog with that activity source. The `_FakeWatchdog` in `tests/test_runner_sandbox.py:98` and both spawn sites are updated for the new signature (the orchestrator site passes no probe — see Req 2). **Phase**: Activity-aware watchdog core + batch_runner site. *AC*: the batch spawn wires `pipeline-events.log`; `just test` exits 0.

10. `STALL_TIMEOUT_SECONDS` stays **1800** so `recovery.py:103`'s `assert WEDGED_STALENESS_SECONDS (2700) > STALL_TIMEOUT_SECONDS (1800)` and `tests/test_recovery_wedged.py` keep passing. **Phase**: Activity-aware watchdog core + batch_runner site. *AC*: `just test` exits 0; `STALL_TIMEOUT_SECONDS` unchanged.

**Phase 2** — *de-risk-gated; Reqs 11/13/14 land only if Req 12 passes.*

11. Orchestrator invoked with **streaming output** (only after Req 12 passes): change `--output-format=json` → `--output-format=stream-json` (adding `--verbose`, which `-p` mode requires) at runner.py:1492. **Ordering gate: Req 13's parser change must already be in place (same commit or earlier) before this flag flips** — flipping the flag while the whole-file `json.loads` parser is still active makes every healthy orchestrator round's telemetry parse raise (swallowed as `dispatch_error` at runner.py:1659-1665). **Phase**: Orchestrator child-progress signal + docs. *AC*: `grep -c "stream-json" cortex_command/overnight/runner.py` ≥ 1 AND the same change set updates the parser (Req 13).

12. **De-risk gate — the FIRST Phase-2 task, gating Reqs 11/13/14**: empirically verify that `--output-format=stream-json` written to a *redirected file* (the runner uses `open(stdout_path,"wb")`, runner.py:1481) advances `mtime`/`size` mid-stream. **This is known-threatened: Claude Code block-buffers stream-json (~4–8 KB) to a non-TTY stdout (anthropics/claude-code Issue #25670), so the naive flag flip is likely to produce no mid-stream growth and an inert orchestrator signal.** If the de-risk fails, do **not** land Reqs 11/13; escalate to the operator with the enumerated fallbacks: (a) **PTY-wrapped stdout** (make the child line-buffer as to a TTY), (b) **orchestrator self-heartbeat** (the child emits a periodic progress line to a dedicated file while `--output-format=json` stays — parser-untouched), or (c) **ceiling-only orchestrator** (accept the absolute ceiling as the orchestrator's sole in-process backstop, a documented "raised timeout" for that site, with a follow-up ticket for a robust signal). **Phase**: Orchestrator child-progress signal + docs. *AC*: Interactive/session-dependent: confirming mid-stream file growth requires spawning a real `claude -p --output-format=stream-json` against a redirected file, which the test suite cannot exercise; the verification outcome and the proceed/fallback decision are recorded in the Phase-2 implementation commit message, and Reqs 11/13/14 land only on confirmed incremental flush.

13. **Orchestrator result parsing handles stream-json NDJSON** (lands with/before Req 11): extract the terminal `type:"result"` object — which is shape-identical to the current `--output-format=json` envelope — instead of `json.loads` of the whole file, in `_emit_orchestrator_round_telemetry` (runner.py:1546) and the call-site read at runner.py:2736. **Scope note: this envelope is consumed only for token/cost telemetry** (the parser reads `usage`/`total_cost_usd`/`duration_ms`/`num_turns`/`model`/`is_error`/`subtype`/`stop_reason`/`effort` — it never reads a `result` field, and no plan is extracted from it; batch plans are written to separate files). The blast radius is telemetry legibility, not plan correctness. **Phase**: Orchestrator child-progress signal + docs. *AC*: a test feeds a stream-json NDJSON fixture and asserts the telemetry extracts the terminal-result envelope and still classifies `is_error`/`error_*` subtypes; `just test` exits 0.

14. **Orchestrator watchdog wired to its chosen child signal** (the stream-json `stdout_path` if Req 12 passed; otherwise the fallback signal), with a code comment at the wiring site recording **why not `overnight-events.log`** (the parent `RunnerHeartbeatThread` advances it regardless of child progress → would make the watchdog never fire; the guardian already owns that signal — and cannot itself catch a hung orchestrator child, so the ceiling is the orchestrator's backstop). **Phase**: Orchestrator child-progress signal + docs. *AC*: `_spawn_orchestrator` builds the watchdog with the chosen probe; the why-not comment is present; `just test` exits 0.

15. **Docs**: update `docs/overnight-operations.md` (the source-of-truth owner per CLAUDE.md) — the stall-watchdog row (~line 43) to describe per-site activity-reset + the absolute ceiling + the per-site asymmetry (batch has a worker signal; orchestrator's is de-risk-gated and may be ceiling-only), and the `WEDGED > STALL` margin prose (~line 301) to state the honest post-change relationship (the in-process watchdog now self-heals on *child silence*, the guardian on *parent-event staleness* — they no longer race on the same stall; the guardian cannot catch an active-but-doomed child, which the ceiling now owns). **Phase**: Orchestrator child-progress signal + docs. *AC*: `grep -cE "activity|ceiling|pipeline-events" docs/overnight-operations.md` ≥ 1 in the watchdog section.

## Non-Requirements

- Do **not** merely raise `STALL_TIMEOUT_SECONDS` (the ticket's hard non-goal; it also breaks the `WEDGED > STALL` assert). The absolute ceiling is a never-reset **secondary** backstop **for the batch_runner site** (which gets a genuine worker-driven activity-reset). **For the orchestrator site, if Req 12's de-risk fails and the ceiling-only fallback is taken, the ceiling becomes that site's *primary* in-process kill — an acknowledged, operator-approved "raised timeout" for the orchestrator, distinct from a blanket cap because (a) the batch site still gets true activity-reset and (b) the ceiling-kill is logged with a tunable reason.** This degradation is named, not hidden.
- Do **not** change the guardian / `RunnerHeartbeatThread` / `recovery.py` wedged-detection (#308 / ADR-0011). The in-process watchdog owns child stalls; the out-of-process guardian owns dead/wedged-host detection.
- Do **not** use any parent-written heartbeat (`overnight-events.log`, `source:"runner"`) as a watchdog reset signal — it reflects parent liveness and would make the watchdog never fire.
- Not the launchd `project_root=/` bug (#311); not a real worker hang (the evidence worker was provably alive).
- No new event type registered — the kill path reuses `ORCHESTRATOR_FAILED` / `BATCH_RUNNER_STALLED` / `STALL_TIMEOUT` (the Req 7 reason is a field on the existing kill events, not a new event type); add a new event only if a real consumer needs it.
- No log-rotation support; document the no-rotation assumption.
- No data-driven ceiling measurement in v1 (operator chose a fixed conservative value to revisit). The Req 7 ceiling-kill telemetry is the fuel for that future revisit.

## Edge Cases

- **Activity file not yet created at watchdog start** → treated as silence (no reset); a child that never produces output and wedges is still killed by the inactivity tier.
- **First output at minute 25** → first-appearance counts as activity (reset), so the child is not killed at 30 unless it then goes silent for the full timeout.
- **Worker stuck in a chatty retry/apology loop emitting output forever** → the inactivity tier is structurally blind to this (output ≠ progress); the **absolute ceiling** kills it. This is the explicit reason the ceiling exists and is logged with the `ceiling` reason (Req 7); kill latency for this class is the ceiling, not 30 min — an accepted trade.
- **mtime granularity** (several appends within one fs-second don't bump mtime) → `size` growth is the primary signal.
- **File rotated/truncated** (size decreases) → re-baseline; do not treat as a stall reset.
- **stream-json block-buffers to a redirected file (#25670)** → Req 12's de-risk gate (the first Phase-2 task) catches it before any parser/flag change; the orchestrator falls back rather than shipping a dead signal.
- **Phase 1 ships alone** → the orchestrator watchdog runs with `probe=None`, i.e. exactly today's blind 1800 s timer (Req 2) — no new behavior, no regression, just unfixed pending Phase 2.
- **Host scheduling delay** (watchdog thread descheduled under load) → the monotonic clock measures real elapsed time, not dropped ticks, so the ceiling stays accurate.

## Changes to Existing Behavior

- **MODIFIED**: `WatchdogThread.run()` from a blind `elapsed += poll_interval` counter → an activity-aware monotonic inactivity timer + absolute ceiling.
- **MODIFIED**: `WatchdogThread.__init__` gains activity-probe + clock seams (probe defaults to `None` = blind-timer-equivalent); both spawn sites and `_FakeWatchdog` updated.
- **ADDED**: `ABSOLUTE_CEILING_SECONDS` constant; a kill-reason marker (inactivity vs ceiling) on the existing kill events.
- **MODIFIED (Phase 2, de-risk-gated)**: orchestrator invocation `--output-format=json` → `--output-format=stream-json --verbose`; orchestrator telemetry parsing reads the NDJSON terminal `result` envelope (telemetry-only).
- **MODIFIED**: `docs/overnight-operations.md` watchdog description (and the kill-path warning strings at runner.py:2757/2858 become accurate without edit).

## Technical Constraints

- Keep `STALL_TIMEOUT_SECONDS = 1800` so `recovery.py:103`'s `assert` and `tests/test_recovery_wedged.py` hold. **Honest note on the invariant**: keeping the *number* preserves the assert, but the `2700 > 1800` margin's original intent — the in-process watchdog gets "first crack" at the *same* stall before the guardian — now applies only loosely, because post-change the watchdog keys on child-written files while the guardian keys on the parent-heartbeat-advanced `overnight-events.log`; the two layers no longer race on a common event. Consequently the guardian's `needs_recovery_wedged` can essentially never fire for an *active-but-doomed* child (the parent heartbeat keeps `overnight-events.log` fresh), so the worst-case time-to-kill for such a child widens from ≤2700 s to the absolute ceiling — which is now the owner of that case. Req 15 updates the docs to state this honestly.
- Watchdog stays **stat-only** (no content parsing) — so it never needs to filter parent-vs-child heartbeats; each site's signal is a file whose growth is child-driven (batch → `pipeline-events.log`; orchestrator → its own `stdout_path` or fallback signal), not a filtered shared log. (Cost of stat-only, accepted: it cannot distinguish progress from churn, which is why the loop class falls to the ceiling — see Problem Statement.)
- Reuse the `cortex_command/common.py:209` `_stat_key` pattern (`exists, st_mtime_ns, st_size`); `size` is the reliable append signal.
- Preserve the no-blocking-sleep loop shape (`coord.shutdown_event.wait(timeout=...)`).
- New activity-tracking state is thread-local; the only new shared field is the Req 7 kill-reason marker.
- `time` is not currently imported in `runner_primitives.py` — add the import.
- `tests/test_runner_heartbeat.py::test_run_is_wired_to_start_the_runner_heartbeat` AST-walks `runner.run` — do not refactor `run()` in a way that breaks that assertion.

## Open Decisions

- **Orchestrator child-signal mechanism** is resolved empirically at Phase 2's de-risk gate (Req 12), not at spec time. Given Claude Code Issue #25670 (stream-json block-buffers to a redirected file), stream-json may prove inert; the fallback choice among PTY-wrapped stdout / orchestrator self-heartbeat / ceiling-only is an operator decision made at that gate with the de-risk result in hand. Reason it cannot be resolved now: it requires running `claude -p --output-format=stream-json` against a redirected file and observing mid-stream growth — implementation-level context unavailable at spec time.

## Proposed ADR

None considered. The supervision division-of-duties (in-process watchdog vs out-of-process guardian) is already governed by ADR-0011; this change brings the implementation into line with that ADR. The per-site signal choice, the absolute ceiling, and the (honestly-disclosed) weakening of the `WEDGED > STALL` first-crack ordering are tactical, reversible choices; the load-bearing "why not the parent heartbeat" rationale is captured in code comments (Req 14), the docs (Req 15), and `research.md`.
