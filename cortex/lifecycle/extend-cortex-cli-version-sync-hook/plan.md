# Plan: extend-cortex-cli-version-sync-hook

## Overview

Five-phase incremental refactor that (a) extracts `CLI_PIN` and the install runner from `plugins/cortex-overnight/server.py` into stdlib-only siblings (`cli_pin.py`, `install_core.py`) so they can be called from both the existing MCP-call path and a new SessionStart-async hook without violating the plugin-imports-zero-cortex-modules contract, (b) adds `hooks/cortex-cli-background-install.sh` registered with `"async": true` and an always-detach `subprocess.Popen(..., start_new_session=True)` so the existing visibility-only sync hook is upgraded to a reinstall-on-drift gate, and (c) reframes the sync hook's `additionalContext` to surface prior-session install state honestly. The refactor preserves existing test monkeypatches via the unqualified `from install_core import _run_install_and_verify` import pattern, and structural guards in `.githooks/pre-commit` enforce stdlib-only siblings and `_enforce_plugin_root` byte-identity.

## Outline

### Phase 1: CLI_PIN extraction (tasks: 1, 2, 3, 4, 5, 6)
**Goal**: relocate `CLI_PIN` from inline `server.py` to a side-effect-free `cli_pin.py` sibling that the hook layer can read without invoking server.py side effects.
**Checkpoint**: `git grep -E "^CLI_PIN\s*[:=]"` returns exactly one match (in `cli_pin.py`); rewriter, workflows, sync hook, and pre-commit guard all read from `cli_pin.py`; `just test` green.

### Phase 2: Factor install_core.py (tasks: 7, 8, 9)
**Goal**: move `_run_install_and_verify` and its install-path helpers from `server.py` into a stdlib-only `install_core.py` sibling, using the unqualified-import pattern so existing `monkeypatch.setattr(server, ...)` test patches continue to bind.
**Checkpoint**: `just test tests/test_mcp_auto_update_real_install.py` passes with no test code changes; `.githooks/pre-commit` rejects stdlib violations in `install_core.py` and `_enforce_plugin_root` drift between server.py and install_core.py.

### Phase 3: Async install hook (tasks: 10, 11)
**Goal**: add `run_install_in_background()` to install_core (with marker, under-lock re-check, sentinel, NDJSON stages, skip predicates), and deploy a bash trampoline hook registered with `"async": true` so SessionStart drift triggers a detached install.
**Checkpoint**: a synthetic SessionStart with a drifted CLI version invokes `uv tool install --reinstall --refresh-package cortex-command git+...@<tag>` once (even with 3 concurrent contenders, under-lock re-check elides 2); hook script exit time < 2 seconds; marker file appears in `${XDG_STATE_HOME}/cortex-command/install.in-progress` during the install window and is removed via try/finally on completion.

### Phase 4: Sync hook enhancements (tasks: 12)
**Goal**: extend `cortex-cli-version-sync.sh` to surface (a) prior-session install in-progress via marker detection, (b) prior-attempt failures via `session-install-failed.*` sentinels with the 30-min retry-aligned window, (c) narrowed dirty-tree skip predicate that only fires inside the cortex-command repo, (d) first-install warn-only branch (no SessionStart auto-trigger of first-install).
**Checkpoint**: hook fixture scenarios produce expected `additionalContext` strings; dirty non-cortex-command repos no longer skip the sync hook.

### Phase 5: Docs + tests (tasks: 13, 14, 15, 16, 17)
**Goal**: revise `docs/internals/auto-update.md` to describe the three-layer architecture and add a Trust Model section (force-push attack + GitHub tag-protection ruleset as immediate mitigation + chicken-and-egg paradox documenting why in-CLI_PIN SHA pinning was deferred); extend the test surface to cover the new async-hook install path, the sync-hook reframing, and the cli_pin.py fixture migration.
**Checkpoint**: `just test` green covering R31's eight async-hook scenarios, R32's five sync-hook scenarios, the R33 invariants migration, and the R34 `install_guard.py` parity preservation; `docs/internals/auto-update.md` describes three layers and the trust model.

## Risks

- **Unqualified-import fragility (R12).** Task 8's design relies on `from install_core import _run_install_and_verify` re-binding the symbol on `server`'s namespace so existing `monkeypatch.setattr(server, ...)` patches continue to bind. If a future refactor silently switches to `import install_core` + `install_core._run_install_and_verify(...)` qualified access, the existing test patches will continue to "pass" but no longer intercept the call — silent regression-coverage gap. Three coupled hazards apply: (a) Task 8's grep guard is symbol-specific — it catches qualified access on `_run_install_and_verify` but silently permits the same antipattern on other factored helpers (`_acquire_install_flock`, `_write_install_failed_sentinel`, `_recent_install_failed_sentinel`); a future refactor using qualified access on any of those would break the patch contract without commit-time detection. (b) A future test author writing `monkeypatch.setattr(install_core, "_run_install_and_verify", spy)` patches a different binding than `monkeypatch.setattr(server, ...)`; the two patches do not interoperate, and the MCP-call path vs. the new async-hook subprocess path resolve to different objects — silent test divergence. (c) The new async-hook subprocess imports `install_core` directly via `sys.path.insert(0, HOOK_PLUGIN_ROOT)`; server.py is never loaded into the subprocess's `sys.modules`, so server-namespace monkeypatches cannot intercept the hook-fired install at all. Task 14's Context (above) documents the subprocess-boundary patch strategy for the hook path so this is mitigated for the V1 test surface; future test authors must read that Context before adding hook-path tests. The reactive `.githooks/pre-commit` qualified-call guard remains a follow-up option if regressions occur.
- **Force-push attack documented but not mitigated in code (R29).** The Trust Model section describes the force-push vector and points operators at GitHub Repository Rulesets as the immediate mitigation — but this is an operator action outside the codebase. If the ruleset is not enabled on the cortex-command repo, the new SessionStart auto-install widens the supply-chain attack surface relative to the pre-Phase-3 state. Sigstore/cosign signing and TOFU SHA recording are deferred to follow-up tickets. Operators dogfooding on dirty cortex-command checkouts are protected by R26's dirty-tree skip; operators in non-cortex repos are not.
- **Flat 30-min retry window (R22).** No exponential backoff on failed installs — if `uv tool install` is failing for a persistent reason (network, GitHub rate limit, malformed tag), the user sees one warning every 30 minutes via the sync hook for as long as the failure persists. Spec defers exponential backoff to follow-up. The 30-min window aligning the retry throttle (R22) and the surfacing window (R25) prevents the "warning fires while retry happens concurrently" UX inconsistency but does not address persistent-failure noise.
- **Async-only under-lock re-check asymmetry (R18).** The under-lock version re-check is scoped to the `session_start_reinstall` stage only — the existing `version_mismatch_reinstall` (MCP-call) stage is unchanged. The asymmetry is deliberate (per spec Non-Requirements: "No modification of MCP-call path's existing version_mismatch_reinstall behavior") but creates a small UX inconsistency: an MCP-call invocation racing with a SessionStart install will not benefit from the under-lock skip and may run a redundant install. Acceptable residual given the regression-coverage-gap argument in the spec critical-review note.
- **Probe-failure mode collapse.** `cortex --print-root --format json` can fail three structurally distinct ways (binary absent, exits non-zero, emits invalid JSON), and Tasks 12(d) and 14 collapse all three under one warn-only prose string (`"cortex CLI is not installed"`) and one path through `run_install_in_background`'s probe-failure silent-skip. Critical review (Reviewer 3, A-class) flagged this as factually-wrong messaging for cases 2 and 3. Decision: keep collapsed. Rationale: the suggested remediation `uv tool install --reinstall --refresh-package cortex-command git+...@<tag>` is the correct action for all three modes (case 2 fixed by `--reinstall` overwriting the corrupt shim; case 3 implies a release-process schema bug the user cannot remediate by reading better prose). The user-visible action does not vary across modes, so per-mode prose buys richer operator diagnostics without changing user recovery. Per-mode NDJSON stages (`session_start_probe_absent` / `_exit_nonzero` / `_invalid_json`) are deferred to a follow-up ticket if real-world reports surface case-2/3 misdiagnoses.

- **Honest-but-bleak sync hook prose (R24).** The sync hook's marker detection is structurally unable to protect the current session's install (it runs concurrently with the async hook, before the marker is written). The user-visible prose is honest about this — `"background install from a prior session is still running"` not `"is currently running"`. The bet is that this honesty is preferable to silently misleading users; the cost is that the marker mechanism is less protective than a naïve reading would suggest.
