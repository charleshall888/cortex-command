# Specification: extend-cortex-cli-version-sync-hook

## Problem Statement

The `cortex-cli-version-sync.sh` SessionStart hook (shipped under #235) is visibility-only — it detects CLI_PIN drift and emits `additionalContext` but does not reinstall. The reinstall path lives in `plugins/cortex-overnight/server.py:_ensure_cortex_installed` and fires only when the cortex-overnight MCP server is invoked. Users who use only daytime workflows (skill prose, Bash `cortex …`, hooks under `claude/hooks/`, plugin-mirrored bin scripts) never trigger the MCP path and stay on a stale CLI indefinitely. Manual remediation (`uv tool install --reinstall --refresh-package cortex-command git+…@<tag>`) is the only recovery path today.

This work extends the SessionStart layer to additionally fire a background `uv tool install --reinstall` on drift, using Claude Code's native `async: true` hook field (added in response to Issue #43123) rather than rolling our own shell-backgrounding. It also closes the force-push supply-chain attack surface that auto-install widens by pinning `CLI_PIN` to a commit SHA in addition to the tag, and re-shapes the install logic so it can be safely invoked from both the MCP-call path and the new SessionStart path without violating the plugin-imports-zero-cortex-modules contract.

## Phases

- **Phase 1: CLI_PIN tag→SHA infrastructure** — Move `CLI_PIN` from inline `server.py` literal to a side-effect-free `plugins/cortex-overnight/cli_pin.py` sibling, expand the tuple shape from `(tag, schema)` to `(tag, sha, schema)`, retarget `bin/cortex-rewrite-cli-pin` to compute and write the SHA, and extend the release.yml CI lint to verify tag↔SHA consistency.
- **Phase 2: Factor `install_core.py`** — Extract `_run_install_and_verify` and its install-path helpers from `server.py` into a stdlib-only plugin-local sibling that the new async hook can also call. Enforce stdlib-only via a pre-commit syntactic check.
- **Phase 3: New async install hook** — Add `hooks/cortex-cli-background-install.sh` registered with `"async": true` in `hooks.json`. It does the drift check, acquires the flock, performs the under-lock re-check, runs the install via `install_core`, writes the install-in-progress marker, and logs to NDJSON via new `_NDJSON_ERROR_STAGES` entries.
- **Phase 4: Sync hook enhancements** — Extend the existing `cortex-cli-version-sync.sh` to (a) detect the install-in-progress marker and adjust its `additionalContext`, (b) surface prior-attempt failures from `session-install-failed.<ts>` sentinels, (c) narrow the dirty-tree skip predicate to only fire when `cwd` is inside the cortex-command repo, (d) drop the first-install branch from SessionStart and warn via `additionalContext` instead.
- **Phase 5: Docs + trust model + tests** — Update `docs/internals/auto-update.md` to describe the three-layer architecture (marketplace fast-forward + MCP-call-gated + SessionStart-async-gated), add the trust model section, document `CORTEX_AUTO_INSTALL=0`, and add test coverage for the new install path, the sync hook's prior-failure surfacing, the under-lock re-check, and the CLI_PIN tuple shape.

## Priority (MoSCoW)

- **Must-have** (V1 ship gate; defines feature completeness): R1–R21, R23–R33. These cover the entire CLI_PIN→SHA transition, the factor, the async install hook with under-lock re-check, NDJSON audit emission, marker file mechanism, sync hook enhancements for safety and visibility, narrowed dirty-tree predicate, first-install warn-only, docs and trust-model updates, and test coverage.
- **Should-have** (V1 ships materially better with these, deferrable if blocked): R22 (uv stdout/stderr captured to `last-install-uv.log`). The NDJSON audit log (R20) already captures structured terminal outcomes; the plain-text log is a debug affordance for cases where uv's progress output is needed to diagnose a failure beyond what the NDJSON record carries. Ship without if the redirect implementation surfaces unexpected issues; file a follow-up.
- **Won't-have** (explicit V1 exclusions, see Non-Requirements section for the full list): SessionStart-triggered first-install, `asyncRewake: true`, PATH refresh after install, exponential backoff on failed installs, separate `cortex-bg-install` launcher script, cross-plugin coupling into `cortex-core`, new env-var kill switches beyond the existing `CORTEX_AUTO_INSTALL=0`, SHA-256 wheel integrity verification.

## Requirements

### Phase 1: CLI_PIN tag→SHA infrastructure

1. **`CLI_PIN` lives in a side-effect-free sibling module.** New file `plugins/cortex-overnight/cli_pin.py` declares `CLI_PIN: tuple[str, str, str]` with shape `(tag, commit_sha, schema_version)`, e.g., `("v2.10.0", "f57ae766c55d3f24f76aa8c7d5c2dad9b4e8a3c1", "2.0")`. The module contains only the `CLI_PIN` declaration plus an optional docstring — no third-party imports, no top-level function calls, no side effects on import. **Acceptance**: `python3 -c "import sys; sys.path.insert(0, 'plugins/cortex-overnight'); from cli_pin import CLI_PIN; print(CLI_PIN)"` from a bare shell (no PEP 723 venv) exits 0 and prints a 3-tuple. **Phase**: Phase 1.

2. **`server.py` re-exports `CLI_PIN` from the sibling.** `plugins/cortex-overnight/server.py:106`'s inline `CLI_PIN = (…)` is replaced by `from cli_pin import CLI_PIN`. `MCP_REQUIRED_CLI_VERSION = CLI_PIN[2]` (was `CLI_PIN[1]`) — the schema index moves from 1 to 2. **Acceptance**: `grep -c "^CLI_PIN = " plugins/cortex-overnight/server.py` = 0; `grep -c "^from cli_pin import CLI_PIN" plugins/cortex-overnight/server.py` = 1; `grep -c "MCP_REQUIRED_CLI_VERSION = CLI_PIN\[2\]" plugins/cortex-overnight/server.py` = 1. **Phase**: Phase 1.

3. **All `CLI_PIN[1]` references retargeted.** Every reference in server.py to `CLI_PIN[1]` (the old schema-version slot) becomes `CLI_PIN[2]`. The install argv at `_run_install_and_verify` retargets `f"@{CLI_PIN[0]}"` to `f"@{CLI_PIN[1]}"` (the SHA), preserving tag context in user-facing remediation messages but pinning the install to the SHA. **Acceptance**: `grep -E "CLI_PIN\[1\]" plugins/cortex-overnight/server.py | grep -v "^#"` returns no non-comment matches that mean "schema version"; the install argv contains `f"git+{REPO_URL}@{CLI_PIN[1]}"` (the SHA, not the tag). **Phase**: Phase 1.

4. **`bin/cortex-rewrite-cli-pin` retargets and computes SHA.** The rewriter's target file moves from `plugins/cortex-overnight/server.py` to `plugins/cortex-overnight/cli_pin.py`. The rewriter accepts a tag argument and computes the SHA via `git rev-list -1 <tag>` before writing the 3-tuple. The script no longer reads or modifies `server.py`. **Acceptance**: `bin/cortex-rewrite-cli-pin v9.9.9-test` (or a dry-run flag if added) modifies only `plugins/cortex-overnight/cli_pin.py` (verify with `git status --porcelain`); the resulting tuple is 3 elements; element 2 matches `git rev-list -1 v9.9.9-test`. **Phase**: Phase 1.

5. **`auto-release.yml` SHA computation.** The auto-release workflow computes the SHA after tagging and before invoking `cortex-rewrite-cli-pin`. The rewritten `cli_pin.py` ships in the same `Release vX.Y.Z` commit that creates the tag. **Acceptance**: `.github/workflows/auto-release.yml` contains a step that runs `git rev-list -1 "$NEXT_TAG"` and passes the result to `cortex-rewrite-cli-pin`; `.github/workflows/release.yml`'s `cli-pin-lint` job verifies both `CLI_PIN[0] == <pushed-tag>` AND `CLI_PIN[1] == git rev-list -1 <pushed-tag>` and hard-fails on mismatch. **Phase**: Phase 1.

6. **Visibility hook reads `CLI_PIN` from `cli_pin.py`.** The hook at `plugins/cortex-overnight/hooks/cortex-cli-version-sync.sh` updates its `parse_cli_pin` helper to read `plugins/cortex-overnight/cli_pin.py` instead of `server.py`, with a regex accepting the new 3-tuple shape: `^CLI_PIN\s*=\s*\(\s*['"]([^'"]+)['"]\s*,\s*['"]([^'"]+)['"]\s*,\s*['"]([^'"]+)['"]\s*,?\s*\)`. The hook reads `CLI_PIN[0]` (tag) for user-facing messages and `CLI_PIN[2]` (schema) for the schema-floor gate. **Acceptance**: `grep -c "cli_pin.py" plugins/cortex-overnight/hooks/cortex-cli-version-sync.sh` ≥ 1; existing hook tests pass against the new tuple shape. **Phase**: Phase 1.

7. **Single-declaration invariant holds across the repo.** `git grep -E "^CLI_PIN\s*[:=]"` returns exactly one match (the new `cli_pin.py`). **Acceptance**: the grep returns exactly 1 line. **Phase**: Phase 1.

8. **`cli_pin.py` parity guard.** Extend `.githooks/pre-commit` to verify `plugins/cortex-overnight/cli_pin.py` contains only the `CLI_PIN` declaration and an optional docstring — no third-party imports, no top-level executable statements beyond the assignment. **Acceptance**: adding `import packaging` to the top of `plugins/cortex-overnight/cli_pin.py` and running `.githooks/pre-commit` exits non-zero with a message naming the offending import. **Phase**: Phase 1.

### Phase 3: New async install hook

13. **New canonical hook source `hooks/cortex-cli-background-install.sh`.** Bash trampoline (`set -euo pipefail`) reads JSON stdin via `jq`, exports `HOOK_CWD`/`HOOK_PLUGIN_ROOT`/`HOOK_STATE_DIR`, applies the PATH bootstrap, and invokes an inline Python heredoc. The Python: imports `install_core` (via sys.path insertion of `HOOK_PLUGIN_ROOT`), probes the installed version via `cortex --print-root --format json`, compares against `CLI_PIN[0]`/`CLI_PIN[1]`/`CLI_PIN[2]`, and on drift runs the install path. The hook itself is invoked by Claude Code in background mode (`async: true` in hooks.json) — it does not need to detach. **Acceptance**: `test -x hooks/cortex-cli-background-install.sh`; `bash -n hooks/cortex-cli-background-install.sh` exits 0; the file contains `python3 - <<'PY'` heredoc; `grep -c "import install_core" hooks/cortex-cli-background-install.sh` ≥ 1. **Phase**: Phase 3.

14. **Async hook registered in `hooks.json`.** `plugins/cortex-overnight/hooks/hooks.json` gets a third SessionStart entry with `"async": true` pointing at `${CLAUDE_PLUGIN_ROOT}/hooks/cortex-cli-background-install.sh`. The two existing entries (`cortex-scan-lifecycle.sh`, `cortex-cli-version-sync.sh`) remain unchanged and remain synchronous. **Acceptance**: `jq '.hooks.SessionStart | length' plugins/cortex-overnight/hooks/hooks.json` = 3; `jq -r '.hooks.SessionStart[].hooks[] | select(.async == true) | .command' plugins/cortex-overnight/hooks/hooks.json` returns exactly the new hook path. **Phase**: Phase 3.

15. **Async hook mirrored by `just build-plugin`.** The `cortex-overnight` plugin manifest in `justfile` (around lines 578-582) adds the new hook to its `HOOKS=(…)` array. **Acceptance**: `just build-plugin cortex-overnight` succeeds; `plugins/cortex-overnight/hooks/cortex-cli-background-install.sh` exists and is byte-identical to the canonical source at `hooks/cortex-cli-background-install.sh`. **Phase**: Phase 3.

16. **Async hook skip-predicate parity.** The async hook honors the same skip predicates as the sync hook: `CORTEX_DEV_MODE=1`, dirty tree (narrowed per Requirement 23), non-`main` branch, and additionally `CORTEX_AUTO_INSTALL=0`. Predicate ordering matches `server.py:_evaluate_skip_predicates` (cheapest first). **Acceptance**: hook invocation with `CORTEX_DEV_MODE=1` exits 0 with no install attempted and no marker file written. **Phase**: Phase 3.

17. **Under-lock version re-check.** After acquiring the install flock, the async hook re-probes `cortex --print-root --format json` and compares the installed version against `CLI_PIN[0]`. If they match (another concurrent install completed first), the hook releases the lock and exits 0 without reinstalling. This applies to both the new `session_start_reinstall` stage and the existing `version_mismatch_reinstall` stage in `_run_install_and_verify`. **Acceptance**: `grep -E "(version_mismatch_reinstall|session_start_reinstall).*re.check" plugins/cortex-overnight/install_core.py` returns at least one match; integration test simulating 3 concurrent installs results in 1 actual `uv tool install` invocation, not 3. **Phase**: Phase 3.

18. **Install-in-progress marker.** Before invoking `uv tool install`, the async hook writes a zero-byte marker at `${XDG_STATE_HOME:-$HOME/.local/state}/cortex-command/install.in-progress`. After the install completes (success or failure), the marker is removed. The marker is written under the flock so concurrent attempts see a consistent state. The hook also tolerates stale markers — a marker older than 600 seconds (2× `_INSTALL_SUBPROCESS_TIMEOUT_SECONDS`) is treated as stale and overwritten without warning. **Acceptance**: during an install, the marker exists; after install completion, the marker is removed; a hook invocation that finds a marker with mtime > 600s ignores it and proceeds. **Phase**: Phase 3.

19. **`session-install-failed.<ts>` sentinel with 30-minute window.** On install failure, the async hook writes a sentinel at `${XDG_STATE_HOME}/cortex-command/session-install-failed.<unix-timestamp>`. Distinct from the existing `install-failed.<ts>` sentinel used by the MCP-call path (which has a 60-second window). Before attempting an install, the hook checks for any `session-install-failed.*` sentinel with mtime within 1800 seconds — if found, skip the install silently. **Acceptance**: a `session-install-failed.*` file with mtime within 1800s causes subsequent async hook invocations to skip the install (no `uv tool install` subprocess fires); the MCP-call path's `install-failed.<ts>` semantics are unchanged (60s window still applies on that path). **Phase**: Phase 3.

20. **NDJSON audit stages added to `_NDJSON_ERROR_STAGES`.** The frozenset gains: `session_start_drift_detected`, `session_start_reinstall`, `session_start_reinstall_parse_failure`, `session_start_reinstall_blocked_by_inflight_session`, `session_start_reinstall_flock_timeout`, `session_start_reinstall_under_lock_skip`. The async hook calls `_append_error_ndjson(stage, error, context)` for each terminal outcome. **Acceptance**: `grep -oE 'session_start_[a-z_]+' plugins/cortex-overnight/install_core.py | sort -u` contains all six new stages; each stage appears in the `_NDJSON_ERROR_STAGES` frozenset at module top level. **Phase**: Phase 3.

21. **`UV_NO_PROGRESS=1` in install environment.** The async hook sets `UV_NO_PROGRESS=1` in the `subprocess.run` `env=` dict for the `uv tool install` invocation. Closes the TTY-coupled-child fd-inheritance risk per §Adversarial F1. **Acceptance**: `grep -E "UV_NO_PROGRESS.*=.*1" plugins/cortex-overnight/install_core.py` returns at least one match within `_run_install_and_verify` or a helper it calls. **Phase**: Phase 3.

22. **uv stdout/stderr captured to `last-install-uv.log`.** The `uv tool install` subprocess's stdout and stderr are redirected to `${XDG_STATE_HOME}/cortex-command/last-install-uv.log`, truncated on each invocation. This is human-readable debug output, distinct from the NDJSON audit log. **Acceptance**: after a background install attempt, the log file exists and contains uv's output (e.g., "Resolved" or "Installed" lines on success, error text on failure). **Phase**: Phase 3.

### Phase 4: Sync hook enhancements

23. **Dirty-tree skip narrowed to cortex-command repo.** The sync hook's dirty-tree predicate (and the corresponding predicate in `server.py:_evaluate_skip_predicates`) fires only when `cwd` resolves into the cortex-command source repo — tested by reading `git rev-parse --show-toplevel` and comparing the remote URL (via `git -C <toplevel> remote get-url origin`) to a known cortex-command remote substring (`cortex-command.git` or `charleshall888/cortex-command`). Dirty trees in unrelated repos no longer skip. **Acceptance**: hook invocation from `~/Workspaces/some-other-dirty-repo` (not cortex-command) does NOT skip on dirty-tree predicate; hook invocation from a dirty cortex-command checkout still skips. **Phase**: Phase 4.

24. **Install-in-progress detection.** The sync hook stats `${XDG_STATE_HOME}/cortex-command/install.in-progress` after determining drift. If the marker exists and mtime is within 600 seconds, the hook's `additionalContext` includes the phrase "background install in progress; bash `cortex …` calls may fail until the install completes" in addition to the standard drift warning. **Acceptance**: hook fixture with a fresh marker file present produces `additionalContext` containing both `"background install in progress"` and the existing drift-warning text. **Phase**: Phase 4.

25. **Prior-failure surfacing.** The sync hook scans `${XDG_STATE_HOME}/cortex-command/session-install-failed.*` for files modified within the last 24 hours. If any are found, the hook's `additionalContext` includes a "Previous background install attempt failed" line citing the timestamp of the most recent failure, the expected and installed versions, and the manual remediation command (`uv tool install --reinstall --refresh-package cortex-command git+…@<tag>`). **Acceptance**: hook fixture with a `session-install-failed.<recent-ts>` file present produces `additionalContext` containing `"Previous background install attempt failed"` and a `uv tool install` remediation line. **Phase**: Phase 4.

26. **First-install case: warn-only (no auto-install at SessionStart).** When the sync hook's probe returns "cortex not installed" (`cortex --print-root` exits non-zero / binary not found), the hook emits `additionalContext` containing "cortex CLI is not installed. Run `uv tool install --reinstall --refresh-package cortex-command git+…@<latest-tag>` to install, or invoke any cortex-overnight MCP tool to auto-install." The hook does NOT trigger a background install in this case. The async hook also detects this state and silent-skips (no install attempted). The MCP-call-gated `_ensure_cortex_installed` first-install path in `server.py` is unchanged — first-install remains MCP-call-gated. **Acceptance**: hook fixture with `cortex` absent from PATH produces `additionalContext` containing `"cortex CLI is not installed"`; the async hook with the same fixture exits 0 without spawning a `uv tool install` subprocess. **Phase**: Phase 4.

## Non-Requirements

- **No factor of `_ensure_cortex_installed`'s high-level orchestrator.** The orchestrator (skip predicates, in-flight session guard, version comparison) stays in `server.py`. Only the install proper (`_run_install_and_verify` and lower-level helpers) factors into `install_core.py`.
- **No `cortex_command/install_core.py` canonical source.** `install_core.py` lives only in the plugin directory; there is no byte-identical mirror in `cortex_command/`. The install runner's natural home is the plugin (it knows `CLI_PIN`), not the CLI.
- **No SessionStart-triggered first-install.** Per Adversarial F7, first-install would auto-trigger in every repo where the plugin is enabled — including non-cortex repos. SessionStart only does reinstall-on-drift; first-install stays MCP-call-gated via `_ensure_cortex_installed`.
- **No `asyncRewake: true`.** The async install hook uses `async: true` (fire-and-forget). Prior-failure surfacing happens on the next session via the sync hook. Per Q3 user input.
- **No PATH refresh after install.** If uv ever relocates the binary, the running session's PATH points at the deleted path. Out of scope; same gap exists in the MCP-call path. File a follow-up if it bites.
- **No exponential backoff on failed installs.** V1 uses a flat 30-min window for `session-install-failed.<ts>` sentinels. File a follow-up if 30-min retries prove too noisy.
- **No `cortex-bg-install` separate launcher script.** The async hook invokes `install_core` directly; no intermediate bin script.
- **No cross-plugin coupling.** The new hook stays in `cortex-overnight`. `cortex-core` is not modified.
- **No env-var kill switch beyond the existing `CORTEX_AUTO_INSTALL=0`.** The existing carve-out is extended to gate the new path; no new variables are introduced.

## Edge Cases

- **Concurrent SessionStarts from 3+ Claude Code instances**: first-to-acquire-lock installs; under-lock re-check (Requirement 17) causes subsequent flock holders to bail without redundant `uv tool install` invocations. Per Adversarial F4.
- **Async hook crashes between writing the install-in-progress marker and starting the install**: marker stays on disk. Mitigation: 600-second mtime staleness check (Requirement 18); after 600s, subsequent hooks treat the marker as stale and overwrite it.
- **Network failure mid-install**: `uv tool install` exits non-zero after `_INSTALL_SUBPROCESS_TIMEOUT_SECONDS=300` (5 min). Hook writes `session-install-failed.<ts>` sentinel; next session's sync hook surfaces the failure via additionalContext (Requirement 25).
- **User on Claude Code < v2.1.139 (no `async:` field honored)**: the hook entry's `"async": true` is silently ignored by older clients, and the hook runs synchronously — which would freeze the launcher per Issue #43123 if a `uv tool install` runs in foreground. **Mitigation**: the hook script itself includes a defensive `subprocess.Popen(..., start_new_session=True, stdin=DEVNULL, stdout=log, stderr=STDOUT)` detach as a belt-and-suspenders for older clients. Newer clients' `async` runner duplicates the work (running the script in background); older clients fall through to the Popen-detach. Both paths must not interact badly — verified by ensuring the hook always exits 0 quickly regardless of whether the install has completed.
- **Force-pushed release tag**: `CLI_PIN[1]` SHA pins the install to the original commit; even if `v2.10.0` is force-pushed to a malicious commit, the install argv resolves to the original SHA. Closes Adversarial S1.
- **uv git-cache staleness**: pinning to SHA (not tag) sidesteps the cache key issue. The `--refresh-package cortex-command` flag is preserved for defense-in-depth on the package cache. Closes Adversarial F3.
- **MCP server process holding `cortex_command.*` modules open during install**: per Adversarial F5, uv's per-file overwrite can corrupt mmap'd venv files. The visibility hook's `additionalContext` (Requirement 24) warns against bash `cortex …` calls during the install window. MCP server processes are not protected by the warning — but they were not protected before either (MCP-call reinstall blocks on its own flock and is internally consistent).
- **`cortex` not installed**: warn-only via additionalContext (Requirement 26). No SessionStart auto-install. MCP-call first-install path unchanged.
- **Dirty cortex-command tree (dogfooder)**: skip predicate fires (Requirement 23 narrowing). No reinstall during dogfooding.
- **Dirty non-cortex-command tree**: skip predicate does NOT fire (Requirement 23 narrowing). Reinstall proceeds normally.
- **`CORTEX_AUTO_INSTALL=0` set**: silent skip on both MCP-call and SessionStart paths (Requirement 29).
- **`CORTEX_DEV_MODE=1` set**: silent skip on all paths (existing behavior, preserved).
- **`cortex --print-root` returns invalid JSON or missing `version`/`schema_version` fields**: hook exits 0 silently; `_append_error_ndjson` records the parse failure with stage `session_start_reinstall_parse_failure`.
- **`cli_pin.py` is malformed**: hook exits 0 silently. The `.githooks/pre-commit` parity guard (Requirement 8) prevents malformed `cli_pin.py` from reaching main.
- **Stale `session-install-failed.<ts>` sentinel from days ago**: the prior-failure surfacing (Requirement 25) limits surfacing to within 24 hours; older sentinels are ignored. (Hook-side 30-min skip window in Requirement 19 is the install-throttle; the 24-hour surfacing window is the warning-fatigue limit.)
- **Multiple `session-install-failed.<ts>` sentinels accumulate over weeks**: not explicitly cleaned by V1. File a follow-up cleanup ticket if accumulation becomes an issue.

## Technical Constraints

- **Plugin-imports-zero-cortex-modules contract** (ADR-0002, `docs/internals/mcp-contract.md:3`): `install_core.py`, `cli_pin.py`, and the new async hook MAY NOT import from `cortex_command.*`. They MUST use subprocess + JSON parsing for any CLI interaction. Enforced by Requirements 1, 9, 12.
- **`cortex --print-root` envelope as forever-public-API** (`docs/internals/mcp-contract.md:22-28`): both hooks and `install_core` rely on `version` (PEP 440 string), `schema_version` (M.m string), and `root` (absolute path) being present and stable.
- **Stdlib-only Python in hooks**: the hook's invocation Python is the system `python3` from PATH; no PEP 723 venv resolution at hook invocation. `install_core.py` and `cli_pin.py` must use stdlib only (no `packaging`, no `mcp`, no `fastmcp`, no `pydantic`). Stdlib version comparison via `version_tuple()`.
- **Defensive `exit 0` on every error path**: hooks must never brick Claude Code launch. The new async hook follows the same defensive pattern as the existing sync hook.
- **`_INSTALL_SUBPROCESS_TIMEOUT_SECONDS = 300.0`** (server.py:296, moves to install_core.py): outer timeout on `uv tool install`. Used as the basis for the 600-second stale-marker cutoff (Requirement 18).
- **Existing `_INSTALL_FLOCK_WAIT_BUDGET_SECONDS = 60.0`** (server.py:287, moves to install_core.py): the new async hook honors this; if the flock cannot be acquired within 60s, hook exits 0 and writes a `session_start_reinstall_flock_timeout` NDJSON record.
- **Existing `check_in_flight_install_core` from `install_guard.py`** is consulted by the async hook before starting an install; if an overnight session is active, the hook skips with stage `session_start_reinstall_blocked_by_inflight_session`. The hook MUST import the vendored `install_guard.py` sibling, not `cortex_command/install_guard.py`.
- **Claude Code `async: true` hook field** is the harness API the async hook depends on. Documented at https://code.claude.com/docs/en/hooks. Available in Claude Code v2.1.139+. Older clients fall through to the defensive Popen-detach (per Edge Cases above).
- **Schema-floor gate parity**: the sync hook's existing `_schema_floor_violated` mirror (lines 213-234) continues to operate against `CLI_PIN[2]` (was `CLI_PIN[1]`) — purely a tuple-index update; no behavior change beyond Requirement 3.
- **`SHA-256` integrity verification of the installed wheel** is out of scope for V1; the SHA pinning closes the force-push attack surface (S1) but does not verify wheel content. File a follow-up if stronger integrity is needed.
