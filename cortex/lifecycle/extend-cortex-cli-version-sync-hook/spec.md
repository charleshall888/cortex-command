# Specification: extend-cortex-cli-version-sync-hook

## Problem Statement

The `cortex-cli-version-sync.sh` SessionStart hook (shipped under #235) is visibility-only — it detects CLI_PIN drift and emits `additionalContext` but does not reinstall. The reinstall path lives in `plugins/cortex-overnight/server.py:_ensure_cortex_installed` and fires only when the cortex-overnight MCP server is invoked. Users who use only daytime workflows (skill prose, Bash `cortex …`, hooks under `claude/hooks/`, plugin-mirrored bin scripts) never trigger the MCP path and stay on a stale CLI indefinitely. Manual remediation (`uv tool install --reinstall --refresh-package cortex-command git+…@<tag>`) is the only recovery path today.

This work extends the SessionStart layer to additionally fire a background `uv tool install --reinstall` on drift via a new async hook (using Claude Code's `async: true` field, added in response to Issue #43123) with a defensive `subprocess.Popen(..., start_new_session=True)` always-detach inside the hook for older clients that don't honor `async: true`. The install logic is factored out of `server.py` into a stdlib-only plugin-local `install_core.py` so both the MCP-call path and the new async hook can call it without violating the plugin-imports-zero-cortex-modules contract. SHA pinning of `CLI_PIN` against force-pushed tags is deferred to a follow-up ticket (chicken-and-egg in the release ritual makes it not straightforwardly implementable); the trust model is documented and the force-push attack is mitigated at the repo-config level via GitHub's tag-protection ruleset.

## Phases

- **Phase 1: CLI_PIN extraction** — Move `CLI_PIN` from inline `server.py` literal to a side-effect-free `plugins/cortex-overnight/cli_pin.py` sibling so it can be read by both the existing hook (via regex parse of a stable, side-effect-free file) and the new async hook (via direct stdlib import). Retain the existing `(tag, schema)` 2-tuple shape. Retarget `bin/cortex-rewrite-cli-pin` and `auto-release.yml`'s `git add` target.
- **Phase 2: Factor `install_core.py`** — Extract `_run_install_and_verify` and its install-path helpers from `server.py` into a stdlib-only plugin-local sibling. Use `from install_core import _run_install_and_verify` (not qualified `install_core._run_install_and_verify` access) so the existing `monkeypatch.setattr(server, …)` in `tests/test_mcp_auto_update_real_install.py` continues to bind the spy through the re-exported name.
- **Phase 3: New async install hook** — Add `hooks/cortex-cli-background-install.sh` registered with `"async": true` in `hooks.json`. The hook always detaches via `subprocess.Popen(..., start_new_session=True, stdin=DEVNULL, stdout=log, stderr=STDOUT)` invoking `install_core.run_install_in_background()`. Always-detach gives uniform behavior across Claude Code versions (newer client: `async: true` runs the hook in background; detach is a no-op-for-correctness; older client: detach prevents launcher freeze per #43123). Background install path acquires the existing flock, performs an under-lock version re-check (async-hook stage only — does not change the existing MCP-call path's behavior), writes the install-in-progress marker INSIDE the flock with try/finally cleanup, and logs NDJSON via existing `_NDJSON_ERROR_STAGES`-allowlisted stages.
- **Phase 4: Sync hook enhancements** — Extend the existing `cortex-cli-version-sync.sh` to (a) detect the install-in-progress marker and adjust its `additionalContext` to indicate a **prior-session install is still running** (not the current session's, which the sync hook necessarily races); (b) surface prior-attempt failures from `session-install-failed.<ts>` sentinels with a window aligned to R19's retry window so warnings only fire while retries are also throttled; (c) narrow the dirty-tree skip predicate to only fire when `cwd` is inside the cortex-command repo; (d) drop the first-install branch from SessionStart and warn via `additionalContext` instead.
- **Phase 5: Docs + trust model + tests** — Update `docs/internals/auto-update.md` to describe the three-layer architecture (marketplace fast-forward + MCP-call-gated + SessionStart-async-gated), add the trust model section documenting force-push attack and the GitHub tag-protection ruleset as the immediate mitigation, document `CORTEX_AUTO_INSTALL=0`, and add test coverage for the new install path, sync hook changes, factor correctness, and the cli_pin.py shape lint.

## Priority (MoSCoW)

- **Must-have** (V1 ship gate; defines feature completeness): R1–R10, R12–R20, R22–R32. Cover the entire CLI_PIN extraction, the install_core factor with corrected symbol list and explicit duplication of `_enforce_plugin_root`, the always-detach async install hook with under-lock re-check, NDJSON audit emission, marker file with try/finally cleanup, sync hook reframing for prior-session install detection, narrowed dirty-tree predicate, first-install warn-only, docs and trust-model updates, and test coverage.
- **Should-have** (V1 ships materially better with these, deferrable if blocked): R11 (timestamped `last-install-uv.<ts>.log` instead of truncate-on-each-invocation — preserves diagnostic data across the 30-min retry boundary but the NDJSON audit log captures structured terminal outcomes); R21 (hook exits in <2s acceptance test — defensive against future Popen-detach regressions but the visible-on-newer-clients behavior already validates).
- **Won't-have** (explicit V1 exclusions): SHA pinning of `CLI_PIN` against force-pushed tags (chicken-and-egg paradox: release commit's tree cannot contain its own SHA — see Trust Model section in `docs/internals/auto-update.md`; mitigation is GitHub tag-protection ruleset at the repo-config level, not in code); SessionStart-triggered first-install in non-cortex repos; `asyncRewake: true` (fire-and-forget `async: true` only); PATH refresh after install; exponential backoff on failed installs; separate `cortex-bg-install` launcher script; cross-plugin coupling into `cortex-core`; new env-var kill switches beyond the existing `CORTEX_AUTO_INSTALL=0`; SHA-256 wheel integrity verification; Sigstore/cosign signing.

### Phase 3: New async install hook

14. **Always-detach via `install_core.run_install_in_background()`.** The hook at `hooks/cortex-cli-background-install.sh` is a bash trampoline (`set -euo pipefail`) that reads JSON stdin via `jq`, exports `HOOK_CWD`/`HOOK_PLUGIN_ROOT`/`HOOK_STATE_DIR`/`CLAUDE_PLUGIN_ROOT`, applies the PATH bootstrap, and invokes an inline Python heredoc. The Python: imports `install_core` (via `sys.path.insert(0, HOOK_PLUGIN_ROOT)`), performs the drift check (reads `CLI_PIN` and probes the installed version), and on drift calls `install_core.run_install_in_background()` which does the install via `subprocess.Popen(["uv", "tool", "install", "--reinstall", "--refresh-package", "cortex-command", f"git+https://github.com/charleshall888/cortex-command.git@{CLI_PIN[0]}"], start_new_session=True, stdin=subprocess.DEVNULL, stdout=open(uv_log, "w"), stderr=subprocess.STDOUT, env={**os.environ, "UV_NO_PROGRESS": "1"})`. The Python heredoc then exits, the bash trampoline exits, and the install proceeds in the detached child. This is uniform behavior across Claude Code versions: on newer clients `async: true` runs the hook in background (and the detach is harmless because the child's parent — the hook script — exits quickly anyway), on older clients the detach prevents launcher freeze per Issue #43123. **Acceptance**: `grep -c "start_new_session=True" plugins/cortex-overnight/install_core.py` ≥ 1 within `run_install_in_background`; `grep -c "stdin=subprocess.DEVNULL" plugins/cortex-overnight/install_core.py` ≥ 1 within `run_install_in_background`; `grep -c "UV_NO_PROGRESS" plugins/cortex-overnight/install_core.py` ≥ 1 within `run_install_in_background`; `bash -n hooks/cortex-cli-background-install.sh` exits 0; the hook script does NOT fork `uv tool install` directly — all install invocation goes through `install_core.run_install_in_background()`. **Phase**: Phase 3.

15. **Async hook registered in `hooks.json`.** `plugins/cortex-overnight/hooks/hooks.json` gets a third SessionStart entry with `"async": true` pointing at `${CLAUDE_PLUGIN_ROOT}/hooks/cortex-cli-background-install.sh`. The two existing entries (`cortex-scan-lifecycle.sh`, `cortex-cli-version-sync.sh`) remain synchronous and unchanged. **Acceptance**: `jq '.hooks.SessionStart | length' plugins/cortex-overnight/hooks/hooks.json` = 3; `jq -r '.hooks.SessionStart[].hooks[] | select(.async == true) | .command' plugins/cortex-overnight/hooks/hooks.json` returns exactly the new hook path. **Phase**: Phase 3.

16. **Async hook mirrored by `just build-plugin`.** The `cortex-overnight` plugin manifest in `justfile` (around lines 578-582) adds the new hook to its `HOOKS=(…)` array. **Acceptance**: `just build-plugin cortex-overnight` succeeds; `plugins/cortex-overnight/hooks/cortex-cli-background-install.sh` exists and is byte-identical to the canonical source at `hooks/cortex-cli-background-install.sh`. **Phase**: Phase 3.

17. **Async hook skip-predicate parity.** The async hook honors the same skip predicates as the sync hook: `CORTEX_DEV_MODE=1`, dirty tree (narrowed per Requirement 25), non-`main` branch, and additionally `CORTEX_AUTO_INSTALL=0`. Predicate ordering matches `server.py:_evaluate_skip_predicates` (cheapest first). **Acceptance**: hook invocation with `CORTEX_DEV_MODE=1` exits 0 with no install attempted and no marker file written. **Phase**: Phase 3.

18. **Under-lock version re-check — scoped to async hook only.** After acquiring the install flock in the `session_start_reinstall` stage path, the install_core function re-probes `cortex --print-root --format json` and compares the installed version against `CLI_PIN[0]`. If they match (another concurrent install completed first), it releases the lock and emits a `session_start_reinstall_under_lock_skip` NDJSON record without reinstalling. **The existing `version_mismatch_reinstall` stage (used by the MCP-call path) is NOT modified.** This preserves the existing MCP-call behavior and avoids the regression-coverage gap. **Acceptance**: `grep -E "(session_start_reinstall).*re.check|re.check.*session_start_reinstall" plugins/cortex-overnight/install_core.py` returns at least one match; `grep -E "version_mismatch_reinstall.*re.check|re.check.*version_mismatch_reinstall" plugins/cortex-overnight/install_core.py` returns NO matches; integration test simulating 3 concurrent async hook installs results in 1 actual `uv tool install` invocation, not 3. **Phase**: Phase 3.

19. **Install-in-progress marker with try/finally cleanup, under flock.** The `run_install_in_background` path writes a zero-byte marker at `${XDG_STATE_HOME:-$HOME/.local/state}/cortex-command/install.in-progress` AFTER acquiring the flock and BEFORE invoking `uv tool install`. The marker write, the subprocess call, and the marker removal all happen inside a single `try/finally` block where the finally clause unlinks the marker and then releases the flock — in that order. This ensures the marker is removed even on SIGKILL of the install subprocess, OOM, exception, or any other failure that interrupts the install — as long as the Python process running install_core survives long enough to execute the finally. The 600-second mtime cutoff (Requirement 20) is the final safety net for catastrophic failure modes (Python process itself SIGKILLed). **Acceptance**: `grep -E "(try:.*marker|finally:.*marker)" plugins/cortex-overnight/install_core.py` returns matches showing the try/finally structure; integration test that raises an exception inside `subprocess.run` (mock uv failure) verifies the marker is removed before the function returns. **Phase**: Phase 3.

20. **600-second stale-marker tolerance.** A marker file older than 600 seconds (2× `_INSTALL_SUBPROCESS_TIMEOUT_SECONDS`) is treated as stale by both the async hook (which overwrites it without warning) and the sync hook (which ignores it when computing `additionalContext`). **Acceptance**: a hook invocation that finds a marker with mtime > 600s does not include the install-in-progress warning in `additionalContext`; the async hook can write a new marker even if a stale one exists. **Phase**: Phase 3.

21. **Hook exits in <2 seconds (defensive acceptance test)** *(should-have)*. The async hook script's wall-clock exit time is bounded at <2 seconds from invocation, regardless of whether the install actually completes. This is enforced by the detach pattern (R14) but validated by a test as defense-in-depth against future regressions. **Acceptance**: `tests/test_cli_background_install_hook.py` (or `.sh`) includes a scenario that times the hook script's exit via `time bash hooks/cortex-cli-background-install.sh < fixture.json` and asserts wall-clock elapsed < 2 seconds on both macOS and Linux CI runners. **Phase**: Phase 3.

22. **`session-install-failed.<ts>` sentinel with 30-minute window.** On install failure, the install_core path writes a sentinel at `${XDG_STATE_HOME}/cortex-command/session-install-failed.<unix-timestamp>`. Distinct from the existing `install-failed.<ts>` sentinel used by the MCP-call path (which has a 60-second window). Before attempting an install via the `session_start_reinstall` stage, the install_core path checks for any `session-install-failed.*` sentinel with mtime within 1800 seconds — if found, skip the install silently. The MCP-call path continues to use only `install-failed.<ts>` with its 60s window; the two namespaces are deliberately decoupled (MCP-call is user-explicit, session-start is automatic — different retry semantics are appropriate). **Acceptance**: a `session-install-failed.*` file with mtime within 1800s causes subsequent async hook invocations to skip the install (no `uv tool install` subprocess fires); the MCP-call path's `install-failed.<ts>` semantics are unchanged (60s window still applies on that path); `grep -E "session-install-failed" plugins/cortex-overnight/install_core.py` returns matches in the `session_start_reinstall` code branch only. **Phase**: Phase 3.

23. **NDJSON audit stages added to `_NDJSON_ERROR_STAGES`.** The frozenset gains: `session_start_drift_detected`, `session_start_reinstall`, `session_start_reinstall_parse_failure`, `session_start_reinstall_blocked_by_inflight_session`, `session_start_reinstall_flock_timeout`, `session_start_reinstall_under_lock_skip`. The async hook calls `_append_error_ndjson(stage, error, context)` for each terminal outcome. **Acceptance**: `grep -oE 'session_start_[a-z_]+' plugins/cortex-overnight/install_core.py | sort -u` contains all six new stages; each stage appears in the `_NDJSON_ERROR_STAGES` frozenset literal at module top level. **Phase**: Phase 3.

### Phase 4: Sync hook enhancements

24. **Install-in-progress detection (reframed for prior-session installs).** The sync hook stats `${XDG_STATE_HOME}/cortex-command/install.in-progress` after determining drift. If the marker exists and mtime is within 600 seconds, the hook's `additionalContext` includes the phrase "background install from a prior session is still running; bash `cortex …` calls may fail until it completes." This protects users who open a new Claude session while a previous session's async install is still in flight. The marker detection is **structurally unable to protect the current session's install** because the sync hook runs concurrently with the async hook (both fire at SessionStart), and the async hook writes the marker only after acquiring the flock — which happens after the sync hook has already composed its `additionalContext`. This is documented honestly in the warning prose. **Acceptance**: hook fixture with a fresh marker file (mtime within 600s) present produces `additionalContext` containing `"prior session"` AND `"bash `cortex"`; hook fixture without a marker file produces `additionalContext` without the install-in-progress phrase. **Phase**: Phase 4.

25. **Prior-failure surfacing aligned with retry window.** The sync hook scans `${XDG_STATE_HOME}/cortex-command/session-install-failed.*` for files modified within the same 1800-second (30-minute) window as the R22 retry-throttle. If any are found, the hook's `additionalContext` includes a "Previous background install attempt failed" line citing the timestamp of the most recent failure, the expected and installed versions, and the manual remediation command (`uv tool install --reinstall --refresh-package cortex-command git+…@<tag>`). The 30-min alignment ensures the warning fires only while retries are also being throttled — eliminating the "warning fires while retry happens concurrently" UX inconsistency surfaced in critical review. **Acceptance**: hook fixture with a `session-install-failed.<recent-ts>` file (mtime within 30 min) produces `additionalContext` containing `"Previous background install attempt failed"` and a `uv tool install` remediation line; hook fixture with the same sentinel but mtime > 30 min ago produces `additionalContext` WITHOUT that line. **Phase**: Phase 4.

26. **Dirty-tree skip narrowed to cortex-command repo.** The sync hook's dirty-tree predicate (and the corresponding predicate in `server.py:_evaluate_skip_predicates`) fires only when `cwd` resolves into the cortex-command source repo — tested by reading `git rev-parse --show-toplevel` and comparing the remote URL (via `git -C <toplevel> remote get-url origin`) to a known cortex-command remote substring (`cortex-command.git` or `charleshall888/cortex-command`). Dirty trees in unrelated repos no longer skip. **Acceptance**: hook invocation from `~/Workspaces/some-other-dirty-repo` (not cortex-command) does NOT skip on dirty-tree predicate; hook invocation from a dirty cortex-command checkout still skips. **Phase**: Phase 4.

27. **First-install case: warn-only (no auto-install at SessionStart).** When the sync hook's probe returns "cortex not installed" (`cortex --print-root` exits non-zero / binary not found), the hook emits `additionalContext` containing "cortex CLI is not installed. Run `uv tool install --reinstall --refresh-package cortex-command git+…@<latest-tag>` to install, or invoke any cortex-overnight MCP tool to auto-install." The hook does NOT trigger a background install in this case. The async hook also detects this state (probe fails) and silent-skips. The MCP-call-gated `_ensure_cortex_installed` first-install path in `server.py` is unchanged — first-install remains MCP-call-gated. **Acceptance**: hook fixture with `cortex` absent from PATH produces `additionalContext` containing `"cortex CLI is not installed"`; the async hook with the same fixture exits 0 without spawning a `uv tool install` subprocess. **Phase**: Phase 4.

### Phase 5: Docs + trust model + tests

## Non-Requirements

- **No SHA pinning of `CLI_PIN`.** The chicken-and-egg paradox (release commit's tree cannot contain its own SHA — predicting a hash of content that includes the prediction is impossible) makes the obvious approach unimplementable. Two-pass tag-move workarounds break the round-trip (installing via SHA pulls stale code). Force-push attack is mitigated at the GitHub-repo-config level via tag-protection rulesets (R29); deeper mitigations (Sigstore signing, TOFU SHA recording) are deferred to a follow-up ticket.
- **No factor of `_ensure_cortex_installed`'s high-level orchestrator.** The orchestrator (skip predicates, in-flight session guard, version comparison) stays in `server.py`. Only the install proper (`_run_install_and_verify` and lower-level helpers) factors into `install_core.py`.
- **No `cortex_command/install_core.py` canonical source.** `install_core.py` lives only in the plugin directory; there is no byte-identical mirror in `cortex_command/`. The install runner's natural home is the plugin (it knows `CLI_PIN`), not the CLI.
- **No SessionStart-triggered first-install.** Per critical review, first-install would auto-trigger in every repo where the plugin is enabled — including non-cortex repos. SessionStart only does reinstall-on-drift; first-install stays MCP-call-gated via `_ensure_cortex_installed`.
- **No `asyncRewake: true`.** The async install hook uses `async: true` (fire-and-forget). Prior-failure surfacing happens on the next session via the sync hook. Per Q3 user input.
- **No PATH refresh after install.** If uv ever relocates the binary, the running session's PATH points at the deleted path. Out of scope; same gap exists in the MCP-call path. File a follow-up if it bites.
- **No exponential backoff on failed installs.** V1 uses a flat 30-min window for `session-install-failed.<ts>` sentinels. File a follow-up if 30-min retries prove too noisy.
- **No `cortex-bg-install` separate launcher script.** The async hook invokes `install_core.run_install_in_background()` directly; no intermediate bin script.
- **No cross-plugin coupling.** The new hook stays in `cortex-overnight`. `cortex-core` is not modified.
- **No env-var kill switch beyond the existing `CORTEX_AUTO_INSTALL=0`.** The existing carve-out is extended to gate the new path; no new variables are introduced.
- **No modification of MCP-call path's existing `version_mismatch_reinstall` behavior.** The under-lock re-check added in R18 is scoped to the `session_start_reinstall` stage only. The existing `version_mismatch_reinstall` semantics in `_run_install_and_verify` are preserved unchanged. Avoids the regression-coverage gap surfaced in critical review.
- **No qualified `install_core._run_install_and_verify(…)` call style.** Per R12, the import style is `from install_core import _run_install_and_verify` so the symbol re-binds on `server`'s namespace and existing test monkeypatches continue to work.

## Edge Cases

- **Concurrent SessionStarts from 3 Claude Code instances**: each instance's async hook spawns one detached `install_core.run_install_in_background()` child. The first to acquire the install flock runs the actual install; the others' under-lock re-check (R18) detects the version is now current and emits `session_start_reinstall_under_lock_skip` NDJSON without invoking `uv tool install`. Always-detach (R14) means exactly 3 contenders compete (one per Claude Code instance), not 6.
- **Marker file leak from a SIGKILL'd install_core process**: the try/finally in R19 cannot run if the Python process itself is killed (kill -9, OOM-kill). The marker stays on disk; next async hook invocation finds mtime > 600s (per R20) and treats it as stale. Sync hook similarly ignores markers older than 600s.
- **Marker file leak from `os.unlink` raising EROFS or ENOSPC mid-finally**: the finally clause logs the failure to `last-error.log` via `_append_error_ndjson(stage="session_start_reinstall", error="marker_cleanup_failed", …)` and the 600s cutoff handles the residual.
- **Network failure mid-install**: `uv tool install` exits non-zero after `_INSTALL_SUBPROCESS_TIMEOUT_SECONDS=300` (5 min). The install_core path writes `session-install-failed.<ts>` sentinel; next session's sync hook surfaces the failure via additionalContext (R25) for the next 30 min.
- **User on Claude Code < v2.1.139 (no `async:` field honored)**: the hook entry's `"async": true` is silently ignored by older clients; the hook runs synchronously. The R14 always-detach pattern means the install subprocess detaches via `start_new_session=True` BEFORE the hook script returns — the hook script's own exit happens within ~50–200ms regardless of install duration. Claude Code's synchronous hook runner sees the script exit cleanly and proceeds. Issue #43123 is avoided because the install subprocess inherits no stdin/stdout/stderr from the hook.
- **Claude Code's async-hook timeout vs. `_INSTALL_SUBPROCESS_TIMEOUT_SECONDS=300`**: irrelevant under always-detach. The hook script exits in <2s (R21). The detached `install_core.run_install_in_background()` child is decoupled from Claude Code's hook lifecycle and survives independently. Claude Code's async-hook timeout (whatever its value) bounds the script's lifetime, not the detached child's.
- **MCP server process holding `cortex_command.*` modules open during install**: uv's per-file overwrite can corrupt mmap'd venv files. The sync hook's `additionalContext` (R24) warns about prior-session installs; users running MCP-tool commands during the install window may see import errors. Acceptable residual — same risk exists in the existing MCP-call reinstall path.
- **`cortex` not installed**: warn-only via additionalContext (R27). No SessionStart auto-install. MCP-call first-install path unchanged.
- **Dirty cortex-command tree (dogfooder)**: skip predicate fires (R26 narrowing). No reinstall during dogfooding.
- **Dirty non-cortex-command tree**: skip predicate does NOT fire (R26 narrowing). Reinstall proceeds normally.
- **`CORTEX_AUTO_INSTALL=0` set**: silent skip on both MCP-call and SessionStart paths (R30).
- **`CORTEX_DEV_MODE=1` set**: silent skip on all paths (existing behavior, preserved).
- **`cortex --print-root` returns invalid JSON or missing `version`/`schema_version` fields**: hook exits 0 silently; `_append_error_ndjson` records the parse failure with stage `session_start_reinstall_parse_failure`.
- **`cli_pin.py` is malformed**: hook exits 0 silently. The `.githooks/pre-commit` parity guard (R8) prevents malformed `cli_pin.py` from reaching main.
- **`CLAUDE_PLUGIN_ROOT` not set in async hook env**: `install_core`'s `_enforce_plugin_root` (R10) calls `sys.exit(1)`. The hook script's bash trampoline exports `CLAUDE_PLUGIN_ROOT` explicitly before invoking Python (R14) to prevent this. If `CLAUDE_PLUGIN_ROOT` is somehow still unset, the install_core import sys.exits — the hook script catches via `set +e` containment around the Python heredoc and exits 0 silently. Defense-in-depth.
- **Force-pushed release tag**: not mitigated by this ticket (SHA pinning deferred). Mitigated at the repo-config layer via GitHub tag-protection ruleset (R29). Documented in trust model section. Residual risk if the ruleset is not enabled.
- **uv git-cache staleness**: `--refresh-package cortex-command` is preserved in the install argv. Per known uv behavior (issue #7866), this doesn't always invalidate the tag→SHA resolution; without SHA pinning, force-pushed tags can pull stale-or-malicious commits. Deferred mitigation — out of scope per Non-Requirements.
- **Multiple `session-install-failed.<ts>` sentinels accumulate over weeks**: not explicitly cleaned by V1. File a follow-up cleanup ticket if accumulation becomes an issue. The existing `install-failed.<ts>` sentinel mechanism has the same property — the follow-up should cover both namespaces.
- **uv stdout/stderr log files (`last-install-uv.<ts>.log`) accumulate**: R11's per-install timestamped filenames are bounded to N=5 most-recent by a single-pass directory scan in install_core at install start.
- **Stale `session-install-failed.<ts>` sentinel from days ago**: the prior-failure surfacing (R25) limits surfacing to within 30 minutes; older sentinels are ignored. (Hook-side 30-min skip window in R22 is the install-throttle; the 30-min surfacing window now matches the throttle window per critical review's UX correction.)
- **Server.py monkeypatch in `test_mcp_auto_update_real_install.py`**: the unqualified `from install_core import _run_install_and_verify` (R12) re-binds the symbol on `server`'s namespace, so existing `monkeypatch.setattr(server, "_run_install_and_verify", _spy_install)` patches continue to intercept. No test code update needed.

## Technical Constraints

- **Plugin-imports-zero-cortex-modules contract** (ADR-0002, `docs/internals/mcp-contract.md:3`): `install_core.py`, `cli_pin.py`, and the new async hook MAY NOT import from `cortex_command.*`. They MUST use subprocess + JSON parsing for any CLI interaction. Enforced by R1, R9, R13.
- **`cortex --print-root` envelope as forever-public-API** (`docs/internals/mcp-contract.md:22-28`): both hooks and `install_core` rely on `version` (PEP 440 string), `schema_version` (M.m string), and `root` (absolute path) being present and stable.
- **Stdlib-only Python in hooks and `install_core`**: the hook's invocation Python is the system `python3` from PATH; no PEP 723 venv resolution at hook invocation. `install_core.py` and `cli_pin.py` must use stdlib only (no `packaging`, no `mcp`, no `fastmcp`, no `pydantic`). Stdlib version comparison via `version_tuple()`.
- **`version_tuple()` scope — bare-semver only.** The stdlib helper handles `vX.Y.Z` and `X.Y.Z` semver tags correctly; non-semver suffixes (`.post`, `.dev`, `+local`, `rcN`) are NOT semantically ordered. The cortex-command release process emits bare semver tags (`vX.Y.Z`); a non-semver suffix is a release-process bug, not a runtime case the helper must handle. If a non-semver tag is ever shipped, `version_tuple` may silently miss or invert drift detection — this is an acceptable residual given the release-process invariant.
- **Defensive `exit 0` on every error path**: hooks must never brick Claude Code launch. The new async hook follows the same defensive pattern as the existing sync hook. The hook's bash trampoline uses `set +e` around the Python heredoc so any unexpected Python failure exits 0.
- **`_INSTALL_SUBPROCESS_TIMEOUT_SECONDS = 300.0`** (server.py:296, moves to install_core.py): outer timeout on `uv tool install`. Used as the basis for the 600-second stale-marker cutoff (R20).
- **Existing `_INSTALL_FLOCK_WAIT_BUDGET_SECONDS = 60.0`** (server.py:287, moves to install_core.py): the new async hook honors this; if the flock cannot be acquired within 60s, the install_core path emits a `session_start_reinstall_flock_timeout` NDJSON record and exits cleanly. Under always-detach (R14) and 3-contender model, the worst case is one install running 300s + two waiters timing out at 60s — both waiters emit benign timeout NDJSON records, which is acceptable noise.
- **Existing `check_in_flight_install_core` from `install_guard.py`** is consulted by both the MCP-call path and the new async hook before starting an install; if an overnight session is active, the hook skips with stage `session_start_reinstall_blocked_by_inflight_session`. The hook imports `install_guard` from the plugin directory (`plugins/cortex-overnight/install_guard.py`), not from `cortex_command/`.
- **Claude Code `async: true` hook field** is the harness API the async hook depends on. Documented at https://code.claude.com/docs/en/hooks. Available in Claude Code v2.1.139+. Older clients silently ignore the field; the always-detach pattern (R14) ensures correct behavior regardless.
- **Schema-floor gate parity**: the sync hook's existing `_schema_floor_violated` mirror continues to operate against `CLI_PIN[1]` (schema, unchanged) — purely a relocation of the source file (cli_pin.py vs server.py); no behavior change.
- **`_NDJSON_ERROR_STAGES` ownership**: the frozenset stays in `install_core.py` per R9 (it's stdlib-only, sits naturally with the install path). Non-install callers in server.py (e.g., `_orchestrate_upgrade`, `_run_verification_probe`) import the allowlist and the emitter from install_core. Acceptable cross-module reference; the surface is small and the alternative (splitting the allowlist into install-specific and upgrade-specific frozensets) adds complexity without clear value.
- **Force-push attack mitigation lives at the GitHub-repo-config layer.** This codebase does not implement in-code mitigations against force-pushed release tags. Operators enable a tag-protection ruleset on the cortex-command repo (R29 documents the operator action). Without that ruleset, the SessionStart auto-install widens the supply-chain attack surface as documented in the Trust Model section.
