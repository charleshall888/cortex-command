# Research: Standalone /refine reconciles lifecycle tier/criticality so §3b critical-review fires

**Topic anchor (clarified intent):** Make standalone `/cortex-core:refine` reconcile the lifecycle `events.log` tier/criticality with the Clarify-determined values, so specify.md §3a/§3b — which read state from `events.log` via `cortex-lifecycle-state`, not backlog frontmatter — observe the correct tier and the §3b critical-review gate fires for Clarify-assessed-complex features, without regressing the `/lifecycle`-delegated refine path that shares refine's code.

**Bug mechanism (confirmed):** refine Step 2 (`skills/refine/SKILL.md:66`) runs `cortex-refine emit-lifecycle-start` → `cortex_command/refine.py:_cmd_emit_lifecycle_start` seeds `lifecycle_start` from backlog frontmatter *at that moment*, BEFORE Clarify. A fresh ticket has no `complexity`/`criticality`, so `_read_backlog_frontmatter` (refine.py:43-44) returns the defaults `("simple","medium")`. Clarify (Step 3) writes the real values to backlog frontmatter only (`cortex-update-item --complexity --criticality`), never to `events.log`. specify.md §3b reads tier/criticality via `cortex-lifecycle-state` (events.log) and sees the stale seed; its run rule is `tier=complex AND criticality∈{medium,high,critical}` (`specify.md:172`), so a stale `simple` seed silently skips the gate.

## Codebase Analysis

**Confirmed change sites / patterns:**

- `cortex_command/refine.py` — exposes exactly one subcommand today (`emit-lifecycle-start`; argparse `sub.required = True`). The seed row is `{schema_version:1, ts, event:"lifecycle_start", feature, tier, criticality, entry_point:"refine"}` (refine.py:125-133). `entry_point` is set **unconditionally** to `"refine"` (line 132). Idempotent via `_lifecycle_start_present` — if any `lifecycle_start` row exists, it exits 0 without appending. The module already has `_now_iso`, `_get_frontmatter_value`, validation constants (`_ALLOWED_COMPLEXITY`/`_ALLOWED_CRITICALITY`), the events.log path convention, and a read-after-write verify idiom — so a sibling reconciliation subcommand is a natural fit (skill-helper-module pattern). **No new console-script entry needed** (`cortex-refine` already maps to `cortex_command.refine:main`; refine deliberately has no `bin/` wrapper).
- `cortex_command/lifecycle/complexity_escalator.py:_emit_event` (lines 189-202) — emits `{ts, event:"complexity_override", feature, from:"simple", to:"complex", gate:<gate>}`. **Hardcoded** simple→complex with a `gate` field tied to the escalator's vocabulary; not directly reusable for a Clarify-driven reconciliation, but its append + read-after-write-verify pattern is the model to mirror locally. No `schema_version`.
- `criticality_override` has **NO Python emitter** anywhere — only readers (`state_cli.py:115`, `common.py:534`) and a prose-emission instruction in `criticality-matrix.md`. A criticality reconciliation requires a net-new emitter.
- **Dual-reader read shapes (critical):** `state_cli.py` (which backs `cortex-lifecycle-state`, the reader §3b uses) reads override value as `record.get("to") or record.get("criticality")` / `or record.get("tier")`. `common.py` `_read_criticality_inner`/`_read_tier_inner` read **`.to` ONLY** (no field fallback). → Any override must set **`to`** so both readers agree.
- Insertion point: `skills/refine/SKILL.md` Step 3, immediately after the `cortex-update-item --complexity --criticality` fenced block. (See Adversarial §6 — placement must also cover resume=spec.)
- Events registry: reusing existing `complexity_override`/`criticality_override` event names means only adding a producer to existing rows — no new registry rows. (See Adversarial §4 for the semantic-overload cost of reusing `complexity_override`.)
- Dual-source mirror: only `skills/` (and `bin/cortex-*`, hooks) mirror to `plugins/cortex-core/`; `cortex_command/` ships via the wheel and is **not** mirrored. `skills/refine/SKILL.md` ↔ its plugin mirror must stay byte-identical (regen via `just build-plugin`).

## Web Research

- **Compensating / override events** are the canonical event-sourcing correction pattern (Microsoft Azure Event Sourcing): never mutate a prior event; append a corrective event. This validates the append-an-override approach over re-seeding `lifecycle_start`. Immutability means a bug that produced a wrong event leaves it in the store — exactly the stale-seed situation.
- **Dual-write problem** (Confluent) is the root anti-pattern: a value written to store A (backlog frontmatter) and never reflected in store B (events.log) silently diverges; the durable fix is to make one store derive from the other (listen-to-yourself), or perform an explicit reconciliation write into the read store. Since events.log is the read path for the gate, reconciling it is the correct local fix.
- **Idempotency via state-based guard, not caller-flag** (Windley guard-rule, Morling idempotency-keys): make the side-effect conditional on current state ("does an authoritative value already exist?"), which is robust to both callers and to retries — a caller-identity flag won't protect against re-runs.
- **Last-writer-wins pitfalls**: rely on deterministic append/line order, not wall-clock timestamps (which are second-granular here and can tie/skew).
- **Circular-logic warning** (Microsoft): appending an event during a phase that re-reads the log can loop — the reconciliation must be conditional so it can't re-fire endlessly.

## Requirements & Constraints

- **Skill-helper-module pattern** (`project.md:35`): collapse SKILL.md dispatch ceremony into an atomic `cortex_command/<skill>.py` subcommand fusing validation+mutation+telemetry, invoked via the `cortex-<skill>` console-script. New events register in `bin/.events-registry.md`. This is the prescribed shape for the fix.
- **Structural over prose** (CLAUDE.md): a gate encoded in skill control flow (a CLI call) is harder to bypass than a prose instruction. The reconciliation must be a Python emitter the skill invokes, not new SKILL.md prose telling the model to emit JSON.
- **No new MUST required**: the structural gate replaces prose enforcement, sidestepping the MUST-escalation evidence burden.
- **No ADR required**: reuses ADR-0001 (file-based state) + existing event semantics; reversible in one PR → fails the three-criteria ADR gate.
- **Bare-Python prohibition (L201)**: SKILL.md must invoke via `cortex-refine`, never an inline `import cortex_command`.
- **R11 regression test** (`tests/test_refine_module.py`, backlog #227) pins the seed-from-backlog behavior — the fix must layer on top of the seed, not replace it.
- **Kept-pauses parity**: the §3b critical-review gate is NOT an `AskUserQuestion` pause and is not in the kept-pauses inventory; making it fire *restores* a protected affordance. But `tests/test_lifecycle_kept_pauses_parity.py` anchors the refine §4 pick-menu at `skills/refine/SKILL.md:166` (±35-line tolerance) — if the edit shifts that line beyond tolerance, update the inventory anchor in lockstep.

## Tradeoffs & Alternatives

**WHAT to reconcile:**
- **A1 — tier-only** (emit `complexity_override` to=complex). Sufficient to FIRE §3b (seed criticality=medium already satisfies the criticality clause). Reuses the existing tier emitter. Leaves criticality stale → downstream overnight model/effort demotion (see Downstream Consumers).
- **A2 — tier + criticality** (also emit a net-new `criticality_override`). Full fidelity; closes the overnight `_MODEL_MATRIX`/`_EFFORT_MATRIX` demotion. More scope (new emitter, dual-reader trap). The Tradeoffs agent recommends A2 on Solution-Horizon grounds (refine's purpose is overnight prep; the criticality demotion is named/concrete). The Adversarial agent recommends scoping to A1 for the §3b fix and treating criticality as a separate ticket on risk grounds. **This is fork A — deferred to Spec (see Open Questions).**

**HOW to reconcile (mechanism):**
- **B1 — new `cortex-refine reconcile-*` subcommand** invoked at Step 3 alongside `cortex-update-item`, mirroring the append+verify pattern locally. **Recommended shape** — cleanest fit with the skill-helper-module pattern and structural-separation; keeps backlog mutation and lifecycle-state mutation as separate single-concern helpers.
- **B2 — single dual-write helper** updating both backlog frontmatter AND events.log. **Rejected** — crosses module-ownership boundaries (`cortex-update-item` owns backlog frontmatter + a separate backlog sidecar event stream + index regen + parent-close cascade); fusing couples unrelated concerns and complicates partial-failure handling.
- **B3 — re-seed `lifecycle_start` post-Clarify**. **Rejected** — violates append-only discipline and the intentional early-anchor ordering of the seed.
- **B4 — import the escalator's private `_emit_event`**. **Rejected** — it hardcodes from/to/gate and is `_`-private; mirror its pattern locally instead.

## Delegation-Interaction Safety

- **`entry_point` CANNOT distinguish standalone vs `/lifecycle`-delegated runs.** `/lifecycle` delegates by reading `skills/refine/SKILL.md` verbatim and executing it (lifecycle/SKILL.md:142), which INCLUDES refine's Step-2 `emit-lifecycle-start` — so the seed is ALWAYS `entry_point:"refine"`. Empirically across the live corpus: 44 `"refine"`, 38 absent (older hand-written), 1 `lifecycle-recovery`, **0 `"lifecycle"`**. There is no in-events.log signal for the caller. **The guard must NOT gate on `entry_point`.**
- **Recommended guard: state-based no-op-when-matches** — read the current reduced tier/criticality via `cortex-lifecycle-state`, and append an override only for a field that DIFFERS from the Clarify value. Entry-point-agnostic, idempotent on resume, harmless on double-fire.
- **No event-count consumer is perturbed** by extra override rows: `detect-phase` counts only `review_verdict`; §2a confidence-check counts only `confidence_check`. No count-based consumer of `lifecycle_start`/`*_override` exists.
- **Delegated-path ordering** (corrected by Adversarial §1): lifecycle logs its OWN post-Clarify `lifecycle_start` AFTER refine's Step 3 completes (lifecycle/SKILL.md:148-153). So the append order is `[seed(simple/medium), refine-override(clarify), lifecycle lifecycle_start(clarify)]`. Under line-order most-recent-wins, lifecycle's trailing `lifecycle_start` SUPERSEDES refine's override — but with the *same correct* Clarify values (lifecycle's Clarify IS refine's delegated Clarify), so it is harmless. **The reconciliation is load-bearing only in the standalone path; tests must not assert the override is authoritative in the delegated path.**

## Downstream Consumers of Tier/Criticality

- **Tier consumers:** §3b gate (specify.md:172) + plan.md gate; research fan-out ROW axis (fanout.md); `requires_review(tier,criticality)` (common.py:647, `complex` alone forces post-merge review); overnight `_MODEL_MATRIX`/`_EFFORT_MATRIX` (`pipeline/dispatch.py` — `complex` is the only tier that can reach Opus/xhigh); lifecycle resume reporting.
- **Criticality consumers:** research fan-out COLUMN axis (medium→6 vs high→8 agents for complex; adversarial angle is high/critical-only); `criticality-matrix.md` interactive model guidance; **overnight `_MODEL_MATRIX` (load-bearing): complex+medium → Sonnet/high vs complex+high → Opus/xhigh**; `requires_review` (but tier=complex already forces review, so medium-vs-high doesn't change the review *decision*, only the review agent's sizing).
- **Consequence of TIER-ONLY reconciliation:** leaving criticality stale at medium is load-bearing in exactly two LATER overnight places — (a) overnight model/effort downgrade (Sonnet/high instead of Opus/xhigh — a silent quality demotion on the high-stakes work, and refine's stated purpose is overnight prep); (b) a later/resumed research run under-sizing its fan-out (6 vs 8 agents, dropping the adversarial angle). **Non-consequences that narrow the blast radius:** refine's OWN research run passes the Clarify criticality directly (unaffected); the §3b gate *decision* is not affected by medium-vs-high.
- **REFUTED ticket premise:** the interactive `critical-review` skill does NOT read criticality at all — fixed 3-4 angles + Opus synthesis. So "stale medium under-powers the §3b critical-review" is **false**. The genuine reason to reconcile criticality is the overnight model/effort matrix + later research fan-out, not critical-review internals.
- **Observability:** `lifecycle_critical_review_skipped` is currently **dead emission** — emitted in `specify.md:177`/`plan.md:279` prose, registry consumer is `(future per-tier compliance audit)` which does not exist; no Python reads it. Extending it to the simple-tier skip would be born-dead unless a consumer (skip-rate report) is built in the same change.

## Adversarial Review

- **Delegated-path ordering:** refine's override is harmlessly SUPERSEDED by lifecycle's trailing correct `lifecycle_start` (not "wins" as the delegation finding implied). Load-bearing only standalone. Don't assert override authority in delegated-path tests.
- **Scope conflation (fork A):** tier-only is provably sufficient to fire §3b; criticality reconciliation is a separate fix for the overnight-matrix demotion. Challenge whether to bundle.
- **Dual-reader split-brain (binding):** if the criticality override is emitted with a bare `criticality` field and no `to`, `state_cli` (§3b) sees the new value but `common.py` (overnight `feature_executor`/`outcome_router`/`runner`) sees the stale seed → §3b and overnight disagree. **Emit `to`-keyed only; pin with a both-readers-agree test.**
- **`complexity_override` semantic overload (mechanism fork B):** the event means "escalator gate fired" and feeds the dashboard's complexity-override badge (`dashboard/data.py:parse_complexity_overrides`, `feature_cards.html`) and the escalator's own already-complex idempotency skip (`complexity_escalator.py:296`). Reusing it for a Clarify determination muddies the dashboard signal. A purpose-named event (e.g. `clarify_complexity_set`) avoids this but requires teaching BOTH readers + a registry row. (The escalator-skip suppression is mostly moot — a Clarify-complex feature would have lifecycle_start=complex anyway, so the escalator was already going to skip.)
- **Torn-line guard (edge case):** `state_cli._reduce_events` returns `None`→`{}` if ANY line is malformed (reduce-to-null), and `common.py` uses a *different* tolerant-skip policy. The no-op-when-matches guard must distinguish "events.log missing" from "reduce returned `{}`/null" — do not treat `{}` as "field unset and safe to emit."
- **resume=spec placement (edge case):** if research.md exists, refine resumes at the Spec phase and SKIPS Clarify (Step 3). A reconciliation placed only in the Step-3 Clarify branch never runs on a direct spec resume. Placement must cover that path (e.g. run at Spec entry / §3b read site, or unconditionally, with the state-based guard making re-runs safe).
- **Read-side alternative (under-weighted by the core wave):** instead of writing a second store, make §3b prefer the Clarify-written backlog frontmatter. Avoids the second writer, the override overload, and the ordering issue — BUT Context B has no backlog frontmatter, and the escalator writes its escalation to events.log (not backlog), so a backlog-preferring read would MISS mid-lifecycle escalations. This is why events.log is the single always-present authoritative store for tier — a legitimate reason to prefer the write-side fix, but it should be argued, not assumed.

## Open Questions

These are consequential design forks deferred to Spec — they will be resolved during the structured interview and the §3b critical-review (which now fires for this feature, since its lifecycle state was reconciled to complex/high). Each is explicitly deferred with rationale.

- **Fork A — scope: tier-only vs tier+criticality.** Tier-only fixes the headline §3b silent-skip and is lower-risk. Tier+criticality additionally closes the overnight `_MODEL_MATRIX` demotion (complex+high features executed on Sonnet instead of Opus) — real because refine prepares features for overnight. *Deferred: resolved in Spec via the §4 complexity/value gate with the user.* Lean: both, because the second consumer is named/concrete (Solution-Horizon durable choice), provided the dual-reader `to`-keyed constraint is honored.
- **Fork B — mechanism: reuse existing override events vs new purpose-named event vs read-side §3b change.** Reuse (simplest, but `complexity_override` semantic overload + dashboard badge); new event `clarify_complexity_set`/`clarify_criticality_set` (clean semantics, costs both-reader + registry changes); read-side §3b-prefers-backlog (no second writer, but breaks on Context B and misses events.log escalations). *Deferred: resolved in Spec, likely surfaced to §3b critical-review given the cross-cutting reader impact.*
- **Binding constraints (NOT forks — settled by research, to be encoded as spec Technical Constraints):** emit `to`-keyed (both readers agree, pin with a test); state-based no-op-when-matches guard (not `entry_point`); never rewrite the seed (append-only); placement must cover resume=spec; torn-line `{}` must not be read as "unset"; structural Python subcommand, not prose; no new MUST, no ADR.
- **Observability (likely Non-Requirement):** extending `lifecycle_critical_review_skipped` to the simple-tier skip is born-dead (no consumer). *Deferred: propose as out-of-scope unless the user wants a skip-rate consumer built in the same change.*
