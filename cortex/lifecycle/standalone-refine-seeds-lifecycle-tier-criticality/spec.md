# Specification: standalone-refine-seeds-lifecycle-tier-criticality

## Problem Statement

When `/cortex-core:refine` is invoked directly on a backlog item, the lifecycle `events.log` is seeded with a stale tier/criticality (`simple/medium` for a fresh ticket) at Step 2, *before* Clarify runs, and is never reconciled with the Clarify assessment. Because the spec-phase gates read tier/criticality from `events.log` via `cortex-lifecycle-state` — not from the backlog frontmatter Clarify writes — a Clarify-assessed-complex feature reads as `simple`. Two gates are affected: **§3a Orchestrator Review** (runs first; its applicability skip-rule keys on tier/criticality) and **§3b Critical Review** (runs only when `tier=complex AND criticality ∈ {medium,high,critical}`). For a Clarify-determined complex feature the stale `simple` read silently skips §3b; for a complex/**low** feature — where §3b never fires by design — §3a is the only adversarial gate and it too reads the stale seed. A second, quieter consequence: the stale criticality later mis-sizes the overnight `_MODEL_MATRIX`/`_EFFORT_MATRIX`, so a `complex+high` feature is executed overnight on Sonnet/high instead of Opus/xhigh. This fixes all of these by reconciling the lifecycle state to the Clarify-determined values before any spec-phase tier/criticality read, so standalone `/refine` runs the same spec-phase gates and produces the same overnight sizing as `/lifecycle`.

## Phases
- **Phase 1: Reconciliation emitter** — a structural `cortex-refine reconcile-clarify` subcommand that appends `to`-keyed `complexity_override`/`criticality_override` rows, with the no-op / no-downgrade / tolerant-read / append-only guards, plus its unit tests.
- **Phase 2: Skill wiring + gates** — invoke the subcommand from `skills/refine/SKILL.md` before the first spec-phase tier/criticality read, update the events-registry producers and the criticality-matrix doc, preserve kept-pauses parity, regenerate the plugin mirror, and add the wiring + regression tests.

## Requirements

1. **Reconciliation subcommand.** Add `cortex-refine reconcile-clarify --lifecycle-slug <slug> [--backlog-slug <slug>] [--complexity simple|complex] [--criticality low|medium|high|critical]` to `cortex_command/refine.py`. It resolves the desired tier/criticality from the explicit flags when passed, else (Context A) from backlog frontmatter via the existing `_read_backlog_frontmatter`, then appends override rows to `cortex/lifecycle/<slug>/events.log` to bring the reduced state into agreement. Acceptance: in-process test — seed `simple/medium`, invoke `reconcile-clarify` with `complex/high`, assert `cortex_command.lifecycle.state_cli._reduce_events(events_log) == {"tier":"complex","criticality":"high"}`. `.venv/bin/pytest tests/test_refine_module.py -q` exits 0. **Phase**: Reconciliation emitter

2. **`to`-keyed emission; both readers agree.** Each emitted override sets `to` (and `from` for audit). Acceptance: test asserts that after reconciliation to `complex/high`, `cortex_command.common.read_tier(...)` and `read_criticality(...)` (which read `.to` only) equal `state_cli._reduce_events(...)["tier"]`/`["criticality"]` (which read `.to or .<field>`) — both return `complex`/`high` (clear the `@lru_cache` via `.__wrapped__.cache_clear()` first). Values are emitted lowercase, matching `requires_review`'s membership tests. **Phase**: Reconciliation emitter

3. **State-based no-op-when-matches, tolerant read.** An override is appended for a field only when the desired value differs from the current reduced value; the guard reads current state via a tolerant reduce that skips malformed lines (consistent with `cortex_command/common.py`'s reader), not the caller identity (`entry_point` cannot distinguish standalone vs `/lifecycle`-delegated). Acceptance: test — invoke `reconcile-clarify` twice with the same values; `grep -c '"event": "complexity_override"' events.log` is identical after the second run and the file size is unchanged (no duplicate row). **Phase**: Reconciliation emitter

4. **Monotonic no-downgrade guard.** Reconciliation never lowers an already-higher reduced value (tier `simple < complex`; criticality `low < medium < high < critical`). This protects a mid-lifecycle escalator `complexity_override` from being clobbered when the backlog frontmatter still reads lower — and is the mechanism that reconciles trusting the backlog as the write-side desired source while the escalator remains the owner of events.log escalations. Acceptance: test — pre-populate `events.log` so the reduced state is `complex/high`, invoke `reconcile-clarify` with `simple/medium`, assert the reduced state remains `complex/high` and no downgrade override was appended. **Phase**: Reconciliation emitter

5. **Graceful behavior on unreadable state; no hard-stop.** Reconciliation must not hard-stop the spec phase because of a pre-existing malformed line elsewhere in `events.log` (per project.md "keep working unless blocked" / "graceful partial failure"). It reads current state tolerantly (skipping malformed lines, per R3) and treats an absent `events.log` as the seed-absent baseline; it appends a well-formed override line and never rewrites existing lines. It does NOT attempt to repair a torn log and does NOT fail loudly on one. Acceptance: test — write a malformed line into `events.log` alongside a valid seed, invoke `reconcile-clarify` with `complex/high`, assert it exits 0 and appends the appropriate override(s) without raising. **Phase**: Reconciliation emitter

6. **Append-only; never re-seed.** Reconciliation only appends; it never rewrites or removes the `lifecycle_start` seed row. Acceptance: test asserts the original `lifecycle_start` line is byte-identical present in `events.log` after reconciliation. **Phase**: Reconciliation emitter

7. **Distinguishable provenance.** Reconciliation overrides carry `gate: "clarify_reconcile"` so they are distinguishable from escalator-gate overrides (`gate: "research_open_questions"` etc.) by any consumer that inspects `gate`. (Verified: the escalator's `read_effective_tier` is gate-agnostic and the dashboard parser never inspects `gate`, so the marker breaks no existing consumer.) Acceptance: `grep -c '"gate": "clarify_reconcile"' events.log` equals the number of fields actually reconciled in the test scenario. **Phase**: Reconciliation emitter

8. **Skill wiring before the FIRST spec-phase tier/criticality read.** `skills/refine/SKILL.md` invokes `cortex-refine reconcile-clarify` at Spec-phase entry, positioned so it runs before **both** §3a Orchestrator Review (whose applicability read consumes tier/criticality) and §3b Critical Review — i.e., before the first tier/criticality read in the spec phase, not merely "before §3b." This covers both the fresh-Clarify path and the `resume=spec` path (where Step-3 Clarify is skipped). On Context A it sources values from the backlog frontmatter Clarify already wrote; on Context B it passes the in-context Clarify values explicitly. Acceptance: static wiring guard test (modeled on `tests/test_refine_lifecycle_start_wiring.py`) asserts the literal `cortex-refine reconcile-clarify` is present in `skills/refine/SKILL.md`; `grep -c 'cortex-refine reconcile-clarify' skills/refine/SKILL.md` >= 1. Manual review confirms the invocation precedes the §3a/§3b reads in the SKILL.md control flow. **Phase**: Skill wiring + gates

9. **Events-registry producers updated (documentation).** Reusing the already-registered `complexity_override` and `criticality_override` events means **no new registry rows** and no new `"event":` literal in any scanned SKILL.md — so the events-registry gate (`cortex-check-events-registry`, which scans skill/orchestrator-prompt `.md` files only) is satisfied without change. The `cortex_command/refine.py` producer is added to both rows' producer columns as documentation (the gate does not scan Python emitters; `complexity_override` is `scan_coverage: manual`, `criticality_override` is `gate-enforced` but unaffected because no scanned literal is added). Acceptance: `.venv/bin/pytest tests/test_check_events_registry.py -q` exits 0; the two rows name `cortex_command/refine.py` as a producer. **Phase**: Skill wiring + gates

10. **Kept-pauses parity preserved.** After editing `skills/refine/SKILL.md`, re-verify the refine §4 pick-menu anchor recorded in `skills/lifecycle/SKILL.md`; update both the inventory anchor and the parity test in lockstep if the line drifts beyond tolerance. Acceptance: `.venv/bin/pytest tests/test_lifecycle_kept_pauses_parity.py -q` exits 0. **Phase**: Skill wiring + gates

11. **Dual-source mirror regenerated.** Run `just build-plugin` and stage the regenerated `plugins/cortex-core/skills/refine/SKILL.md` (note: `cortex_command/` is NOT mirrored — it ships via the wheel). Acceptance: `diff skills/refine/SKILL.md plugins/cortex-core/skills/refine/SKILL.md` is empty (and the dual-source pre-commit drift hook passes). **Phase**: Skill wiring + gates

12. **End-to-end regression + delegated-path no-op test.** A test simulates the headline standalone scenario — fresh ticket → seed `simple/medium` → reconcile to `complex/high` → `cortex-lifecycle-state --feature <slug> --field tier` emits `{"tier":"complex"}`. A second test reproduces the real delegated-path ordering — `[seed(simple/medium), lifecycle_start(complex/high)]` (lifecycle's correct post-Clarify seed, logged before Research) — then invokes `reconcile-clarify` at Spec entry and asserts it **no-ops** (no new override row appended, because the reduced state is already `complex/high`) and the final reduce is `complex/high`. This pins that reconciliation is harmless on the delegated path because lifecycle's correct `lifecycle_start` already moved the state — NOT via supersession, and the test must reflect that reconcile is the later (would-be most-recent) write that the no-op guard suppresses. Acceptance: `.venv/bin/pytest` of the new test file exits 0. **Phase**: Skill wiring + gates

13. **Criticality-matrix doc updated.** Update `skills/lifecycle/references/criticality-matrix.md` so its documented `criticality_override` shape acknowledges the optional `gate` field this fix emits (currently documented as `{ts, event, feature, from, to}`). Acceptance: `grep -c 'gate' skills/lifecycle/references/criticality-matrix.md` >= 1 in the criticality_override shape context. **Phase**: Skill wiring + gates

## Priority (MoSCoW)

- **Must-have** (the fix is incorrect, incomplete, or un-committable without them): R1 (subcommand), R2 (`to`-keyed / both-readers-agree), R3 (no-op-when-matches, tolerant read), R4 (no-downgrade), R5 (graceful on unreadable state), R6 (append-only), R8 (wiring before the first spec-phase read), R9 (registry/gate passes), R10 (kept-pauses parity), R11 (mirror), and the end-to-end regression half of R12.
- **Should-have** (valuable, but the gates fire correctly and the bug is fixed without them): R7 (the `gate: "clarify_reconcile"` provenance marker — only affects consumer distinguishability), R13 (criticality-matrix doc alignment — adjacent documentation contract), and the delegated-path no-op assertion in R12 (defensive).
- **Won't-do**: see Non-Requirements.

## Non-Requirements

- **Does NOT fix the `cortex-lifecycle-state` reduce-to-null divergence.** `state_cli._reduce_events` collapses the entire reduce to null if ANY line in `events.log` is malformed (and `main()` then exits 0 with a non-`complex` result), whereas `common.py`'s reader tolerantly skips malformed lines. On a torn log this makes §3a/§3b read `simple` and silently skip — a pre-existing latent defect in the READ path that this reconciliation fix cannot remedy (the gate reads through the same poisoned reduce regardless of what reconciliation appends). Out of scope; flagged as a follow-up ticket candidate (align `state_cli` torn-line tolerance with `common.py`).
- **Does NOT extend `lifecycle_critical_review_skipped` to the simple-tier skip.** That event is currently dead emission (no consumer). Out of scope unless a skip-rate consumer is built in the same change.
- **Does NOT relabel the dashboard "complexity override" badge.** The `gate: "clarify_reconcile"` marker makes a Clarify reconciliation distinguishable in the data; teaching the dashboard to label it distinctly is a cosmetic follow-up.
- **Does NOT change `/cortex-core:lifecycle`.** Lifecycle already emits a correct post-Clarify `lifecycle_start` before Research; on the delegated path the reconciliation no-ops (the reduced state already matches by Spec entry).
- **Does NOT modify the §3a/§3b read side** (`specify.md`/`orchestrator-review.md` continue to read from `events.log` via `cortex-lifecycle-state`). The read-side alternative was rejected (Context B has no backlog; the escalator writes escalations to `events.log`, so a backlog-preferring read would miss them).
- **Does NOT backfill** historical lifecycles refined before this fix shipped, beyond what the `resume=spec` Spec-entry placement naturally covers.

## Edge Cases

- **`resume=spec` (Clarify skipped):** the Spec-entry placement (R8) runs the reconciliation on this path; Context A sources from backlog frontmatter. This is safe because the Step-3 Clarify write-back (`cortex-update-item`) precedes `research.md` (written in Step 4), and a write-back failure *halts* the session before Step 4 — so any state that has `research.md` (the precondition for `resume=spec`) necessarily has a backlog reflecting a completed Clarify write-back. The reconciliation therefore sources a value that matches the last completed Clarify assessment.
- **`/lifecycle`-delegated run:** lifecycle logs its correct post-Clarify `lifecycle_start` *before Research begins*; by the time refine's reconciliation runs at Spec entry (after Research), the reduced state is already `complex/high`, so the no-op guard (R3) suppresses any override — reconciliation appends nothing on this path. (The R12 delegated test pins exactly this.)
- **Mid-lifecycle escalator override present:** the no-downgrade guard (R4) prevents reconciliation from clobbering an escalator's `complexity_override → complex` when the backlog still reads `simple`.
- **Context B (ad-hoc, no backlog):** reconciliation requires explicit `--complexity`/`--criticality`; `--backlog-slug` is omitted and no backlog read occurs. A Context-B `resume=spec` without in-context values is best-effort (prior session's override persists in `events.log`).
- **Pre-triaged ticket (frontmatter already carries complexity/criticality):** the seed is already correct, so reconciliation no-ops (R3).
- **Pre-existing torn line in `events.log`:** reconciliation reads tolerantly and appends gracefully without a hard-stop (R5), but cannot make §3a/§3b read correctly while the torn line poisons `state_cli`'s reduce (see Non-Requirements — separate latent defect).
- **Double invocation (standalone double-run vs. delegated):** both no-op via R3, but by different mechanisms — on the standalone path the first reconcile appends the override and the second finds the state already matching; on the delegated path lifecycle's interleaved `lifecycle_start` has already moved the state, so reconcile never appends. Both produce no duplicate rows.

## Changes to Existing Behavior

- **ADDED:** `cortex-refine reconcile-clarify` subcommand (new verb on the existing `cortex-refine` console-script; no new `[project.scripts]` entry).
- **ADDED:** `cortex_command/refine.py` becomes a producer of `complexity_override` and `criticality_override` events.
- **MODIFIED:** `skills/refine/SKILL.md` Spec phase now invokes the reconciliation before the first spec-phase tier/criticality read (preceding both §3a and §3b); the §3b adaptation note is updated to record that the lifecycle state is reconciled to the Clarify assessment.
- **MODIFIED:** standalone `/refine` now reads the Clarify-assessed tier/criticality at §3a Orchestrator Review and §3b Critical Review (previously the stale seed, which could silently skip §3b for a complex feature and mis-key §3a's skip rule for a complex/low feature), and produces overnight-correct criticality sizing.

## Technical Constraints

- Emit `to`-keyed overrides (binding: `common.py` reads `.to` only; `state_cli.py` reads `.to or .<field>`); emit values lowercase to match `requires_review` membership tests.
- Guard is state-based (reduce-and-compare via a tolerant reader that skips malformed lines), never `entry_point`-based — `entry_point` is always `"refine"` under both standalone and delegated runs.
- Append-only `events.log`; never rewrite the seed (`lifecycle_start` remains the early anchor).
- Reuse registered event names (`complexity_override`, `criticality_override`) — no new registry rows; add producers only. The events-registry gate scans `.md` files for `"event":` literals, not Python emitters, so structural emission introduces no scanned literal.
- The reconciliation invocation must precede §3a (specify.md:161) and §3b (specify.md:165) in the SKILL.md control flow — both read tier/criticality via `cortex-lifecycle-state`.
- Structural Python subcommand invoked by the skill — not a new SKILL.md prose instruction (per "prefer structural separation over prose-only enforcement"). No bare-Python import in SKILL.md (L201).
- No new MUST/CRITICAL/REQUIRED escalation (the structural gate replaces prose enforcement).
- Preserve the R11 seed-from-backlog regression test (`tests/test_refine_module.py`); the reconciliation layers on top of the seed.
- `cortex-refine` binstub reads the installed wheel; dogfooding/verification of `refine.py` edits needs `CORTEX_COMMAND_FORCE_SOURCE=1` or a wheel reinstall.

## Open Decisions

- **`state_cli` reduce-to-null divergence — follow-up ticket.** The gate's read path (`cortex-lifecycle-state` / `state_cli._reduce_events`) collapses to null on any single malformed line and `main()` exits 0 with a non-`complex` result, diverging from `common.py`'s tolerant skip and silently skipping §3a/§3b on a torn log. This is independent of the reconciliation fix and out of its scope; it warrants a separate ticket to align `state_cli`'s torn-line tolerance with `common.py`. Surfaced for the user to decide whether to file now. Resolution requires read-path changes that affect all `cortex-lifecycle-state` consumers, so it is deliberately not bundled here.
