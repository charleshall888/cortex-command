**Headline bug — pr-review evidence-grounding (exact).** `protocol.md:518-521`: `… | bash "${CLAUDE_SKILL_DIR:-$TMPDIR}/scripts/evidence-ground.sh" 2>/dev/null`. `$CLAUDE_SKILL_DIR` is never set in a shell → `$TMPDIR/scripts/evidence-ground.sh` (nonexistent) → `2>/dev/null` swallows the error → empty stdout. **Precedent — critical-review.** Resolves `${CLAUDE_SKILL_DIR}/references/…` **in its SKILL.md body** (lines 48,54,60,66,74,80,88,94) and reserves `cortex-critical-review` subcommands for SHA validation / realpath / telemetry-gating / atomic sidecar writes — i.e. **body-propagation for content, CLI for state side-effects**. It is *not* a CLI-serves-content precedent, and it is itself a Class-1 instance (the `synthesizer-prompt.md:30` raw token is unfixed). So it argues for body-propagation of content + CLI for executable/state steps.

**Canonical-vs-mirror.** Canonical skills live in top-level `skills/`; `just build-plugin` (`justfile:574,597`, `rsync --delete`) regenerates `plugins/cortex-core/{skills}` (and cortex-overnight). The `.githooks/pre-commit` drift gate (Phase ~2) blocks a commit that edits a canonical source without staging the regenerated mirror — so **critical-review, diagnose, research, refine, lifecycle, discovery fixes land in `skills/` and must regen the mirror in the same commit**. `pr-review` and `android-dev-extras` are hand-maintained directly in `plugins/` (no mirror). Parity also enforced by `bin/cortex-check-parity` (Phase 1.5).

**cortex CLI wiring.** Console scripts declared in `pyproject.toml [project.scripts]` (60+), invoked by bare name in Bash; resolve via PATH from `~/.local/bin`. There is **no `cortex-pr-review` entry and zero pr-review awareness in the wheel** today.

## Web Research

- **`${CLAUDE_SKILL_DIR}` is officially documented** (code.claude.com/docs/en/skills) and explicitly works for **plugin** skills, pointing at the skill's own subdir; intended for "bash injection commands to reference scripts or files bundled with the skill." Shipped in **v2.1.69** ("in SKILL.md content"). **Not** substituted in YAML frontmatter / skill-scoped hooks (community-confirmed on #17741).
- **`${CLAUDE_PLUGIN_ROOT}`** (plugins-reference) *is* exported into hook/MCP subprocesses and valid in frontmatter — unlike `CLAUDE_SKILL_DIR`. Caveat: the previous version's dir lingers ~7 days after update; treat as ephemeral.
- **Reference-file link resolution (OQ3):** docs *demonstrate* bare relative links `[reference.md](reference.md)` as the intended supporting-file pattern and lazy-load, but **never state the resolution rule**. Resolution is model behavior (a Read), not a documented deterministic rewrite — the exact gap #17741 reports.
- **Issue #17741** ("bare relative paths resolved as absolute/guessed, not skill-relative"): **closed `not planned` by stale-bot auto-close, NOT a maintainer decision** (zero maintainer comments). The one human comment notes v2.1.69 added `CLAUDE_SKILL_DIR` "for content, but not frontmatter/hooks." **Sibling issue #11011 is still OPEN**: "plugin scripts fail on first execution with relative path resolution" (`scripts/foo.sh` → not-found on first try, works on retry with absolute path) — directly our scenario. #17564 (closed, stale): skill-scoped hooks run with `cwd=$CLAUDE_PROJECT_DIR` and no skill-dir var.
- **Load-time `` !`cmd` `` injection (OQ4):** documented as official preprocessing ("runs before the skill content is sent to Claude … not something Claude executes"), single-pass, line-start/whitespace trigger, `shell` frontmatter selects bash/powershell, `disableSkillShellExecution` policy switch (bundled/managed skills exempt). **The cwd, `$TMPDIR`, and sandbox posture are NOT documented.**
- **Robust community pattern:** a PreToolUse hook matching the `Skill` tool that injects the resolved skill base dir as `additionalContext` (one report: first-run errors 9/10 → 0/10). Noted as an alternative the prior investigation didn't consider.

## Mechanism Tradeoffs & Alternatives

Re-weighed under the waived contract. Candidates: **(A) body-propagation** (resolve `${CLAUDE_SKILL_DIR}` in the body, propagate the absolute path into shell + inline reference *content* into composed subagent prompts); **(B) cortex CLI subcommands**; **(C) load-time `!`-injection**; **(D) hybrid / per-skill**.

| Dimension | A body-propagation | B cortex CLI | C `!`-injection | 
|---|---|---|---|
| New coupling | none | **new wheel pkg + plugin↔wheel skew (no CLI_PIN for pr-review) + drift surface** | none |
| Off-repo / Dock portability | excellent | conditional (loud `command not found` if absent) | unknown (sandbox) |
| Failure loudness | script step loud; content step prose-gated | loud at process level — **but see swallow below** | **silent unless proven** |
| Token cost (content) | inlines content (~3.7k for pr-review rubric+output-format) | **same bytes land in subagent prompt → zero saving** | same as A |
| Pattern alignment | matches critical-review's content handling | matches console-script promotion (for *state*, not content) | no deployed write-precedent |

**Decisive finding (adversarial, verified):** the pr-review silent-skip is **not** caused by the path mechanism — it is caused by `2>/dev/null` **plus** the protocol treating *empty stdout* as a routable degradation. A missing `cortex-pr-review` binstub writes `command not found` to **stderr** (swallowed) → empty stdout → the **same** announced-degradation fallback. So **CLI does not fix the silence; it relocates the same swallow one layer out.** The load-bearing fix is mechanism-independent: **remove `2>/dev/null` and distinguish "tool absent / crashed" from "ran, zero findings"** so absence halts loudly.

**CLI portability re-weigh (genuine, per operator directive):** the prior bake-off's "CLI portability FAILS / SILENT" verdict was *wrong on two counts*. (1) It conflated the gated bootstrap *hook* with the real exposure — `uv tool install` binstubs land in `~/.local/bin`, **on PATH in any repo independent of the `cortex/lifecycle/` gate** (verified: `cortex-check-contract --help` runs from `/tmp`). (2) Where the CLI is absent the failure is **loud** (`command not found`), not silent. So CLI is genuinely *viable* now. **But viable ≠ better for pr-review:** it adds a net-new `cortex_command/pr_review/` wheel package, an **unprotected** plugin↔wheel version-skew surface (pr-review has no MCP server / no CLI_PIN — verified), and a new dual-source drift surface for `evidence-ground.sh`, in exchange for **zero token saving** and **no reliability gain** (the swallow, not the path, is the bug). For **cortex-internal skills**, CLI is "already paid" (they call `cortex-*` extensively and run only in cortex repos) — but their Class-1 bug is reference *content*, which body-propagation already handles without new coupling.

**Standardize one vs per-skill:** standardize the **invariant**, not the mechanism — *no reference file, composed subagent prompt, or shell command may carry a raw `${CLAUDE_SKILL_DIR}` token or a bare-relative load path; every runtime consumer of the skill dir receives the resolved value explicitly.* The mechanism that satisfies it can vary (body-propagation for content/paths everywhere; CLI for executable/state steps where the skill already depends on the toolchain).

## Empirical Harness Behavior (OQ3 / OQ4)

Claude Code **2.1.168**. `just build-plugin` and the install pipeline **do not rewrite skill links** — the `${CLAUDE_SKILL_DIR}` token and bare-relative paths ship to the cache verbatim (source vs deployed `skills/lifecycle/SKILL.md` are byte-identical), so all resolution is the loader's at runtime. (Confirming datum from *this* session: the harness substituted `${CLAUDE_SKILL_DIR}` → absolute paths when it loaded the lifecycle SKILL.md body into context — body substitution works as documented.)

**OQ3 — bare relative path resolution: RESOLVED EMPIRICALLY = CWD (Claude Code 2.1.168).** A live `oq3probe` skill was installed under `~/.claude/skills/` and invoked from a fresh session with CWD=`~/Workspaces` (≠ skill dir). Result:
- **TEST1 (bare `references/canary.md` in the SKILL.md body)** → resolved against **CWD** → file-not-found (`/Users/charlie.hall/Workspaces/references/canary.md`).
- **TEST2 (bare sibling `canary.md` inside a loaded reference file)** → resolved against **CWD** → file-not-found.
- **CONTROL (absolute `${CLAUDE_SKILL_DIR}/references/canary.md`)** → SUCCESS (`OQ3-CANARY-OK`).

So bare relative paths resolve against the **working directory, not the skill dir**, confirming issue #17741 on 2.1.168. The probe exercised the **Read-instruction** form at both body and reference-file levels; the rendered-markdown-href variant (`[x](references/x.md)`, r8-analyzer/android-cli) was not isolated, but it reduces to the same literal-path Read, so the conservative conclusion is that all bare-relative forms are unreliable off-repo. **Consequence: the Class-2 (bare `../` Read-instructions) and Class-3 (`cat skills/X.sh|bash`) entries are genuinely broken off-repo and warrant the fix — not merely fragile.** Only absolute (`${CLAUDE_SKILL_DIR}`-resolved or propagated) paths are reliable.

**OQ4 — load-time `!`-injection: read-form CONFIRMED deployed** (`android-cli/SKILL.md:8` `` !`command -v android …` ``, byte-identical in cache). **Write-form + `$TMPDIR`-parity remain untested** (zero deployed write-use; load-time sandbox layer may differ from the Bash-tool layer). Test protocol ready (`oq4probe` skill writing a sentinel to `$TMPDIR` at load time, Bash stage reads it back + compares `$TMPDIR`). **The recommended direction abandons injection, so OQ4 does not gate the decision.**

## CLI Portability & Contract

Two independent PATH mechanisms (the prior investigation conflated them): **(1) uv-tool binstubs in `~/.local/bin`** — portable, on PATH in any repo once installed, independent of any cortex dir (the real exposure); **(2) the SessionStart bootstrap hook** — gated on `$CWD/cortex/lifecycle/` (verified `claude/hooks/cortex-session-start-path-bootstrap.sh:24-25`), exists only to repair the macOS Dock/launchd minimal-PATH case. pr-review ships **no** hook and depends on **no** `cortex-*` script today (verified); the "no prerequisites" phrasing exists only in the prior investigation's prose, now waived. Residual CLI gap for pr-review = (Dock-launched ∧ `~/.local/bin` not in `~/.zshenv` ∧ foreign repo) — a one-time documented user fix that fails **loud**. cortex-internal skills already depend on the CLI and run where it's present → CLI is free for them.

## Adversarial Review

Verified challenges that reshaped the recommendation: (1) **CLI doesn't fix the silence** — the `2>/dev/null` + empty-stdout-routing swallow is the real bug (now the centerpiece). (2) **No CLI_PIN for pr-review** → new unprotected version-skew coupling. (3) **New wheel package + drift surface** to host a `cortex-pr-review` subcommand. (4) **critical-review is body-propagation, not a CLI-content precedent**; (6) **CLI content-serving saves zero tokens.** (OQ3) **deferral is scope-determining and the inventory conflates href/Read/shell cases** — must be settled before the Class-2/3 sweep. (Scope) the synthesis trended toward **gold-plating a medium spike**; the minimal harm-closer is one reference file + one body line. (Lint) a naive `${CLAUDE_SKILL_DIR}`-grep **false-positives** on correct body usage and **false-negatives** on token-less Class-2 — it needs two context-aware detectors, the second gated on OQ3.

## Recommended Direction (Synthesis)

**1. Root-cause fix (mechanism-independent, highest value): kill the swallow.** In pr-review `protocol.md` Stage 3.5, remove `2>/dev/null` and split "tool absent / script crashed" (loud halt with a visible message) from "ran, returned zero findings." Without this, *every* mechanism re-buries the failure.

**2. Path resolution: standardize on explicit body-propagation; reserve the CLI for executable/state steps where the skill already depends on the toolchain.**
- **pr-review (Class 1):** drop the raw token; body-propagate the absolute skill dir into `bash "<dir>/scripts/evidence-ground.sh"`; **Read + inline** rubric.md/output-format.md *content* into the Stage-4 subagent prompt (with `{rubric}`/`{output_format}` placeholders). **No new CLI subcommand** — it buys new coupling for zero benefit and doesn't fix the silence.
- **cortex-internal Class 1 (critical-review `synthesizer-prompt.md:30`, diagnose `phase-1-investigation.md:49`):** same body-propagation/inline fix (canonical `skills/`, regen mirror in-commit). Keep their existing CLI calls for state.
- **Class 2/3:** convert to robust forms — `${CLAUDE_SKILL_DIR}/../sibling/…` in bodies, inline/absolute in reference files, body-propagate the absolute path for the `cat …|bash` shell case. These conversions are safe **regardless of OQ3**; OQ3 only decides whether each is *already-broken (must-fix)* or *merely-fragile (deferrable)*.

**3. Anti-recurrence (the actual root cause of the recurrence): re-establish the deleted guardrail, structurally.** Primary = a **lint** scoped correctly (two context-aware detectors: raw-`${CLAUDE_SKILL_DIR}`-in-reference/subagent-prompt, AND bare-relative-load-path — the latter's target set finalized by OQ3); plus a short **positive-routing principle in CLAUDE.md** pointing to the lint; plus **ADR-0009** *if* a standardizing cross-skill mechanism is adopted (plausibly clears the three-criteria gate). Do **not** recreate an orphaned `claude/reference/` doc (that orphaning is what lost it).

**4. Suggested phasing (spec decides breadth — operator routed it there):** Phase 1 = the three Class-1 reference-content fixes incl. the pr-review swallow-kill (same verified-broken bug, no OQ dependency); Phase 2 (gated on OQ3) = Class-2/3 sweep + lint + CLAUDE.md principle + ADR. A scoped Phase 1 is a legitimate lifecycle phase, not a stop-gap (Solution horizon).
