# Plan: restore-worktree-root-env-prefix

## Overview

Move `resolve_worktree_root()`'s same-repo default out of the `.claude/` deny scope to `$TMPDIR/cortex-worktrees/<feature>`, then propagate the change to every dispatch path that produced or consumed the legacy `<repo>/.claude/worktrees/` path. The fix is structural — one resolver chokepoint plus a new `cortex-worktree-resolve` console script that the bash hook shells out to (eliminating duplicated path-computation across Python and shell). Four atomic-commit phases land in order: Phase 1 (resolver) → Phase 2 (init + hook parity) → Phase 3 (verification + atime guard) → Phase 4 (doc sweep + R7 supersession). Phase 1 → Phase 2 ordering is hard (Phase 2's byte-parity test depends on Phase 1's resolver default being live); Phase 3 may overlap with Phase 2.

## Outline

### Phase 1: Resolver + cleanup correctness (tasks: 1, 2, 3, 4)
**Goal**: Land R1+R2+R3+R4+R5 — branch (c) default, `.resolve()` canonicalization, branch (b) sentinel marker, `cleanup_worktree()` fallback routing, and structural regression tests.
**Checkpoint**: `pytest tests/test_worktree.py -q` exits 0 against the new defaults; all four R1–R4 changes land in a single atomic commit so `just test` is never red mid-phase.

### Phase 2: Init handler + hook consumer parity (tasks: 5, 6, 7, 8)
**Goal**: Land R6+R7+R8+R9 — add `cortex-worktree-resolve` console script, remove `cortex init` Step 8, delete the now-invalid init worktree-registration tests, retarget the bash hook through the resolver, and update `tests/test_hooks.sh` assertions.
**Checkpoint**: `just test` and `tests/test_hooks.sh` both exit 0; hook and Python resolver emit byte-identical paths under a controlled `TMPDIR`. All Phase-2 changes land in a single atomic commit.

### Phase 3: Sandbox-aware verification + paused-feature guard (tasks: 9, 10, 11)
**Goal**: Land R10+R11 — atime-touch guard with `CORTEX_SKIP_ATIME_TOUCH=1` opt-out, Seatbelt-active integration test for both dispatch paths, and the `f_row_evidence` event in events.log recording an actual pass under `CLAUDE_CODE_SANDBOX=1`.
**Checkpoint**: `events.log` contains an `f_row_evidence` event with `outcome:"passed"` and `claude_code_sandbox:"1"`. R10 and R11 land in separate commits; R11 may land in parallel with Phase 2.

### Phase 4: Documentation reversal + R7 supersession (tasks: 12, 13, 14, 15, 16)
**Goal**: Land R12+R13+R14 — reverse `multi-agent.md`'s same-repo convention with `.mcp.json`-deny rationale, sweep skills + docs + requirements + bin (~10 files) plus regenerate plugin mirrors, and emit byte-parity-verified R7 supersession callouts plus a `requirement_superseded` event. The R13 sweep is decomposed across Tasks 13, 14, 15 to keep each within the 5-file-per-task cap.
**Checkpoint**: `grep -rln "\.claude/worktrees" skills/ docs/ cortex/requirements/ cortex/lifecycle/ cortex/research/ cortex_command/ tests/ bin/ claude/ plugins/cortex-core/` returns only archive paths and this lifecycle's own / harden lifecycle's own dirs. All Phase-4 changes land in a single atomic commit.

## Risks

- **`Phase 1` is an atomic commit covering four functional changes (R1–R4) plus a rewrite of three existing tests and addition of new ones.** If any single change introduces a regression mid-edit, `just test` can go red before the commit lands. Mitigation: implement task 1 → 2 → 3 → 4 sequentially within the same working tree, run `pytest tests/test_worktree.py -q` after each, and only stage + commit once all four pass.
- **New `cortex-worktree-resolve` console script adds a maintenance surface.** A thin wrapper around `resolve_worktree_root()` is small, but it's a new `[project.scripts]` entry that must remain on PATH for the hook to function. If a user installs cortex-command via a non-standard mechanism that does NOT expose `[project.scripts]` (rare; `uv tool install` and `pip install -e .` both do), the hook will exit non-zero. The alternative — embedding the path logic directly in the hook — was rejected (research §Adversarial FM-3 and spec constraint "single-chokepoint, enforced structurally").
- **The Seatbelt-active R10 integration test (Task 10–11) cannot run in CI** (CI lacks `CLAUDE_CODE_SANDBOX=1`). The `f_row_evidence` event is a manual gate the implementer drives once, recorded durably in `events.log`. Task 11's integrity-binding fields (`pytest_exit_code`, `pytest_summary`, `stdout_sha256`) close the fabricability gap surfaced in critical-review (a fabricated event without a matching pytest stdout to hash is now detectable). Task 10's body-content lint closes the no-op-test-body gap. **Remaining residual**: no automation re-runs the test after lifecycle close — a future regression in `resolve_worktree_root()` would surface only when the next implementer manually re-runs. Filed as follow-up backlog: wire `tests/test_worktree_seatbelt.py` into the overnight smoke test (overnight runs under Claude Code → `CLAUDE_CODE_SANDBOX=1` set by default) so the seatbelt-writability property is re-verified every overnight session. The follow-up is intentionally out-of-scope for this lifecycle (touches the overnight smoke wiring, which is the spec's Non-Requirements boundary).
- **Hook PATH-bootstrap is a pragmatic mitigation, not a guarantee.** Task 7's PATH prepending (`$HOME/.local/bin`, `~/.cargo/bin`, `/opt/homebrew/bin`, `/usr/local/bin`) covers the dominant macOS install layouts and most launchd-launched configurations. Users with non-standard install locations (e.g., `uv tool install --install-dir /custom/path`) or Linux/WSL2 desktop-launched Claude Code with stripped PATH still hit the fail-loud-with-launchd-mentioning-diagnostic exit path. The improved diagnostic names launchd PATH inheritance as a likely cause, but does not auto-recover. A future hardening pass could query `uv tool dir --bin` at hook startup and prepend its result — out-of-scope here to keep the bash hook minimal.
- **Doc sweep (Task 13) touches 10+ files plus plugin mirrors.** `bin/cortex-check-parity --staged` catches literal mirror drift but not semantic doc drift across multiple files. Mitigation: the verification grep over the union of `skills/`, `docs/`, `cortex/requirements/`, `cortex/lifecycle/`, `cortex/research/`, `cortex_command/`, `tests/`, `bin/`, `claude/`, `plugins/cortex-core/` (with archive + this-lifecycle exclusions) returns zero matches as the structural gate.
- **Single-operator migration assumption (spec §Non-Requirements) — atomicity protects in-repo state, NOT operator-machine state.** No automated migration of legacy `<repo>/.claude/worktrees/` settings.local.json entries. The single existing operator (project owner) is expected to remove the entry manually. Two distinct hazards live here: (a) the *resolver-shadowing* hazard (Phase 1 must land R1 + R3 together so the new branch (c) default is not shadowed by a legacy substring-match entry) — this is closed by atomic R1+R3 landing inside Phase 1's single commit; (b) the *operator-machine-state* hazard (the legacy un-suffixed `<repo>/.claude/worktrees/` entry continues to exist in `~/.claude/settings.local.json::sandbox.filesystem.allowWrite` even after Phase 1 lands, granting `.claude/`-scoped write permission that the new resolver chokepoint no longer consumes) — this is **NOT** closed by atomicity, because the operator's machine state lives outside the commit. The fix for (b) is encoded as Task 1's precondition gate (a mechanical pre-edit check that fails if any un-suffixed `worktrees/` entry is present in `allowWrite`), making the spec's Phase-1 prerequisite a hard gate rather than prose. A second user onboarding before manual cleanup would see (i) the resolver behaving correctly (branch (b) ignores the un-suffixed entry per Task 2's sentinel scheme) but (ii) a stale settings entry suggesting `.claude/worktrees/` is still the convention — a discoverability hazard not addressed by any in-repo doc sweep.
