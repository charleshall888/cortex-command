# Specification: reconcile-lifecycle-complete-phase-on-main

## Problem Statement

The lifecycle Complete phase finalization tail (Steps 9–12 in `skills/lifecycle/references/complete.md`) runs the backlog write-back (Step 9), index sync (Step 10), and `feature_complete` append (Step 11), then the Step 12 summary — **none of which commit**. `complete.md` contains exactly one `/cortex-core:commit` (Step 2), and it runs *before* PR creation (Step 3). So whatever the finalization tail produces is left uncommitted on every completion path:

- **Trunk (direct-to-main, no worktree)**: Step 2's on-main short-circuit skips Steps 2–5 and jumps to Steps 9–12, so the lifecycle artifacts (`plan.md`, `review.md`, `index.md`, `events.log` through `feature_complete`) AND the backlog write-back are left uncommitted. Observed in lifecycle #274, where the orchestrator had to commit manually.
- **Worktree-interactive (`interactive/{slug}`) PR flow**: Step 2 committed the lifecycle artifacts on the worktree branch *before* the PR, and they reach `main` via merge — but the **post-merge** Steps 9–11 outputs (backlog status change, regenerated index, `feature_complete` row) are generated *after* merge, on re-invocation, and are never committed.
- **Feature-branch (`feature/{slug}`, no worktree) PR flow**: same post-merge gap as the worktree flow, except the re-invocation session stays on `feature/{slug}` (Step 8 cleanup is skipped for non-`interactive/` branches).

This is the same uncommitted-artifact class on all three paths, even though `commit-artifacts` defaults to `true` and the Refine and Plan phases commit their artifacts unconditionally (flag-gated). The fix adds a single flag-gated, stage-first artifact-commit step at the finalization tail that runs on **all** completion paths, closing the phase-to-phase inconsistency. (Scope was expanded from on-main-only to all-paths at the Plan-phase approval gate, after a branch-disposition analysis showed the on-main step already incidentally covered the dominant worktree-interactive PR flow and only the discouraged feature-branch flow remained uncovered.)

## Phases

- **Phase 1: Finalization-tail artifact commit (all paths)** — add the flag-gated, stage-first commit step (and its false-branch + non-default-branch behavior) to `complete.md`'s finalization tail, delimited by a stable structural anchor, staging the lifecycle dir + backlog write-back. No branch gate — the stage-first guard makes it safe on every path.
- **Phase 2: Test coverage** — add a structural guard test scoped to the anchored commit step, and confirm the dual-source parity and kept-pauses parity tests stay green (after mirror regeneration).

## Requirements

1. **Finalization-tail artifact commit (commit-artifacts=true)**: After Steps 9–11 complete (so the backlog write-back outputs and the `feature_complete` row exist), stage the artifact set (R5) and commit via `/cortex-core:commit`, on **every** completion path (trunk, worktree-interactive post-merge, feature-branch post-merge) — no branch gate. The commit lands the `feature_complete` row (placed after Step 11, before the Step 12 summary — see Edge Cases for the ordering rationale). The new step reads the flag via `cortex-read-commit-artifacts`. Acceptance: `cortex-read-commit-artifacts` is referenced inside the anchored finalization commit step (R11), and `grep -c 'cortex-read-commit-artifacts' skills/lifecycle/references/complete.md` increases from 1 (the current single generic Step 2 flag-read) to 2. **Phase**: Finalization-tail artifact commit
2. **Stage-first idempotent guard**: After staging, run `git diff --cached --quiet`; exit 0 (nothing staged) → skip `/cortex-core:commit` silently; exit 1 → commit. This makes the step safe to re-run AND safe on the worktree-interactive flow (the already-merged lifecycle artifacts stage nothing, so only the new post-merge tail commits — no double-commit). Acceptance: `grep -c 'git diff --cached --quiet' skills/lifecycle/references/complete.md` ≥ 1 (the pattern is new to this file; the precedent lives in `post-refine-commit.md`). **Phase**: Finalization-tail artifact commit
3. **Halt on commit failure**: If `/cortex-core:commit` exits non-zero, surface the error and stop before the Step 12 summary — do not emit a "lifecycle complete" summary implying the artifacts were committed. Phrase as soft positive-routing (no new MUST). Acceptance: the anchored finalization commit step describes the halt-before-summary behavior on commit failure (Interactive/session-dependent: prose-behavioral instruction, verified by reading the anchored Step). **Phase**: Finalization-tail artifact commit
4. **commit-artifacts=false → skip entirely**: When `cortex-read-commit-artifacts` prints `false`, skip the commit entirely (consistent with `post-refine-commit.md` and `plan.md §5`, which skip when false) and note inline that lifecycle artifacts and any uncommitted source are left in the working tree for the operator to commit deliberately. Acceptance: the anchored finalization commit step states the false-branch skips the commit (Interactive/session-dependent: prose-behavioral instruction, verified by reading the anchored Step). **Phase**: Finalization-tail artifact commit
5. **Staging set — explicit paths for the lifecycle dir, directory-scoped add for the backlog write-back**: Stage the lifecycle dir by enumerated paths — `research.md`, `spec.md`, `plan.md`, `review.md`, `index.md`, `events.log` (those present) — NOT `git add cortex/lifecycle/{slug}/`, to avoid sweeping un-gitignored residue (`critical-review-residue.json`, `learnings/*`) into the commit. Stage the backlog write-back with `git add cortex/backlog/` (directory-scoped), which captures the resolved item `.md`, the regenerated `index.json`/`index.md`, AND the sibling/parent `.md` files rewritten by `cortex-update-item`'s terminal-status cascade (`_remove_uuid_from_blocked_by`, `_check_and_close_parent`); the gitignored sidecar `cortex/backlog/*.events.jsonl` is excluded automatically. The backlog dir has no un-gitignored residue, so a directory-scoped add is safe there (unlike the lifecycle dir). Acceptance: the anchored finalization commit step instructs enumerated lifecycle paths (no `cortex/lifecycle/{slug}/` glob) AND a `cortex/backlog/`-scoped add (Interactive/session-dependent: staging-instruction prose, verified by reading the anchored Step). **Phase**: Finalization-tail artifact commit
6. **Commit step never routes into PR creation**: The new step is a *local* commit only — it contains no `git push`/`gh pr create`/PR-creation instruction on any path. The trunk path continues to skip the PR ceremony (Steps 3–6) via Step 2's on-main short-circuit; the PR paths already created their PR upstream. Acceptance: the anchored finalization commit step contains no `git push`/`gh pr create`/PR-creation instruction (scoped to the anchored region per R11, since PR-creation prose legitimately exists elsewhere in Steps 3–5). **Phase**: Finalization-tail artifact commit
7. **No new event type**: Reuse the existing `feature_complete` event (Step 11). Do not introduce `commit_authored`/`artifacts_committed`/`commit_failed`, matching `post-refine-commit.md`'s discipline (commit success observable via `git log`, failure via `/cortex-core:commit`'s error path). Acceptance: `bin/.events-registry.md` gains no new event row (`git diff bin/.events-registry.md` adds no table row). **Phase**: Finalization-tail artifact commit
8. **Structural guard test, scoped to the anchored step**: Add a test (mirroring `tests/test_post_refine_commit_wired.py` style) that extracts the anchored finalization commit step (R11) from `complete.md` and asserts that region mentions `cortex-read-commit-artifacts`, `/cortex-core:commit`, the `git diff --cached --quiet` guard, and the non-default-branch advisory (R13), and contains no PR-creation/`git push` instruction and no `cortex/lifecycle/`-glob `git add`. Scoping to the anchored region is required because those negative-assertion strings appear elsewhere in the file. Acceptance: the new test exists under `tests/` and `just test` exits 0 after the mirror is regenerated (R12) (pass if exit code = 0). **Phase**: Test coverage
9. **Parity tests stay green (post mirror-regen)**: The kept-pauses parity test must remain green — inserting the commit step after Step 11 must not shift any pause anchor beyond its ±35-line tolerance (a tail insertion cannot move the upstream Step 5/Step 6 pause headings, so this holds by construction; verify after editing). Acceptance: `python3 -m pytest tests/test_lifecycle_kept_pauses_parity.py` exits 0 (pass if exit code = 0). **Phase**: Test coverage
10. **MUST-escalation compliance**: The new step uses soft positive-routing phrasing; it introduces no new MUST/CRITICAL/REQUIRED escalation. Acceptance: the diff to `complete.md` adds no new `MUST`/`CRITICAL`/`REQUIRED` token (`git diff skills/lifecycle/references/complete.md | grep -E '^\+' | grep -cE '\b(MUST|CRITICAL|REQUIRED)\b'` = 0). **Phase**: Finalization-tail artifact commit

This is the structural-separation-over-prose choice from the project's authoring guidance. Acceptance: `grep -c 'finalization-commit-step' skills/lifecycle/references/complete.md` ≥ 1 AND the guard test (R8) binds to it. **Phase**: Finalization-tail artifact commit

This covers the discouraged feature-branch (no-worktree) flow, where the post-merge tail lands on `feature/{slug}` rather than `main`. The step does NOT auto-`checkout` the trunk branch (branch normalization stays deferred — see Non-Requirements). Phrase as soft advisory (no new MUST). Acceptance: the anchored finalization commit step contains the non-default-branch advisory referencing `main`/`master` (Interactive/session-dependent: prose-behavioral instruction, verified by reading the anchored Step). **Phase**: Finalization-tail artifact commit

## Non-Requirements

- **Does NOT auto-normalize the branch on the feature-branch flow.** When the finalization commit runs on a `feature/{slug}` branch (no-worktree PR flow, post-merge), it commits the tail *on that branch* and surfaces the R13 advisory rather than running `git checkout main` (which risks conflicts with uncommitted tail changes and entangles with local-merge state). Auto branch-normalization is deferred to the same Step 7 state-aware-routing follow-up that owns the broader re-entry hazards below.
- **Does NOT change Step 7 state-aware routing.** Two distinct re-entry hazards remain deferred to a separate ticket: (a) the contradictory "proceed to Step 7 … treat as first-run path jumping directly to Steps 9–12" prose at `complete.md:21`, and (b) Branch 3's orphan-PR probe firing on a fresh re-entry. Both are PRE-EXISTING. Rationale: the new commit step self-heals or degrades to status quo in every crash window (see Edge Cases), so the routing fix is an improvement to a separate surface, not a prerequisite. (Distinguish from the commit-fail-after-`feature_complete` window, which is NOT a routing bug and is status-quo-equivalent — see Edge Cases.)
- **Does NOT mirror PR-flow Step 2's "commit source when commit-artifacts=false" behavior.** When `false`, the finalization commit skips entirely (R4); it never auto-commits arbitrary uncommitted working-tree source. Source changes are committed per-task during Implement, not by this finalization step.
- **Does NOT address auto-release version bumps on artifact-only commits.** `auto_bump_version.py` classifies by commit message (default `patch`), never by changed files, and `.github/workflows/auto-release.yml` triggers on every push to main. This is a pre-existing project-wide property: the Refine and Plan commits already land artifact-only changes on main and cut patch releases on the trunk flow. This fix adds one more such commit, so a trunk-flow lifecycle cuts up to three artifact-only patch releases (refine + plan + complete) instead of two — consistency-preserving, but a real operator-visible consequence. Out of scope to change here. (On the PR flows the finalization commit lands on `main`/`feature` locally and only triggers a release if/when pushed.)
- **Does NOT touch the overnight runner.** Overnight emits `feature_complete` from Python and commits to the integration branch; it does not run this interactive finalization path.
- **Adds no user pause** (no `AskUserQuestion`, no phase-exit pause).

## Edge Cases

- **Three completion dispositions, one unified step**:
  - *Trunk*: session on `main`; Step 2's on-main short-circuit skipped Steps 2–5, so the finalization commit stages the full set (nothing committed yet) and commits all on `main`.
  - *Worktree-interactive post-merge*: re-invocation runs from the main repo on `main` (the Step 8 hard guard forces the operator to cd out of the worktree first). The lifecycle artifacts are already on `main` via the merge, so the stage-first guard skips them; only the new post-merge tail (`feature_complete` row + backlog write-back) commits — no double-commit.
  - *Feature-branch post-merge*: session on `feature/{slug}` (Step 8 cleanup skipped for non-`interactive/` branches); the finalization commit lands the tail on that branch and surfaces the R13 advisory. Imperfect (artifacts on the merged feature branch, not `main`) but committed-and-reachable rather than left dirty; branch normalization is deferred.
- **Ordering rationale (commit after `feature_complete`)**: The commit is placed after Step 11 so the closing `feature_complete` row lands inside the commit, preserving design history's terminal event. The "commit before the append" alternative is crash-safer but leaves the closing row uncommitted indefinitely on a terminal path; "after" is chosen because its only downside (commit-fail-after-append) is status-quo-equivalent.
- **Re-invocation after a successful commit**: Step 7 Branch 2 (`feature_complete` present) short-circuits to the Step 12 summary; even if the finalization step were reached, the stage-first guard finds nothing staged and no-ops. No duplicate commit.
- **Commit fails after `feature_complete` appended**: Halt before the summary (R3); the operator resolves and re-invokes. On a fresh re-invocation, Branch 2 short-circuits to summary, leaving artifacts uncommitted. This is identical to today's status quo — Step 11 already appends `feature_complete` and nothing commits it today — so the masking-on-recovery is pre-existing, not a regression. The happy path strictly improves on this.
- **Clean tree at commit time** (artifacts already committed, or no changes): the `git diff --cached --quiet` guard skips the commit silently rather than producing an empty-commit error.
- **`commit-artifacts=false`**: the commit is skipped entirely; artifacts and any uncommitted source remain in the working tree (documented inline).
- **Terminal-status cascade touches sibling/parent backlog files**: when the completing item has `blocked-by` dependents or a parent epic whose siblings all become terminal, `cortex-update-item` rewrites those `.md` files; the `cortex/backlog/`-scoped add (R5) captures them so they are not left dirty.
- **Runtime/lock files** (`.session`, `.lock`, `.dispatching`, `agent-activity.jsonl`, `cortex/backlog/*.events.jsonl`): gitignored, so excluded automatically by both the enumerated lifecycle paths and the `cortex/backlog/`-scoped add.

## Changes to Existing Behavior

- **MODIFIED**: `complete.md` finalization tail — currently runs Steps 9–12 with no commit on any path; now, after Steps 9–11, runs a flag-gated, stage-first artifact commit before the Step 12 summary, delimited by a stable anchor, on all completion paths.
- **MODIFIED**: Step 2 on-main short-circuit pointer (line 21) — updated to name the new finalization commit step instead of implying the on-main path commits nothing.
- **ADDED**: a new flag-gated, stage-first commit step on the finalization tail (after Step 11, before Step 12), with a non-default-branch advisory.
- **ADDED**: a structural guard test scoped to the anchored finalization commit step.

## Technical Constraints

- Reuse `post-refine-commit.md`'s stage-first no-op detector (`git diff --cached --quiet`, §"No-Op Short-Circuit") and its HALT-on-commit-failure pattern. The precedent lives in `post-refine-commit.md`, not in `complete.md` (which carries neither pattern today).
- Stage-first, then invoke `/cortex-core:commit`; never raw `git commit`, never `-A`. Enumerated paths for the lifecycle dir; `cortex/backlog/`-scoped add for the write-back (R5).
- `cortex-read-commit-artifacts` defaults to `true` (file/field absent → true).
- Commit subject: imperative, capitalized, ≤72 chars, no trailing period (e.g., `Complete {slug}: lifecycle artifacts and backlog write-back`).
- Soft positive-routing phrasing per the MUST-escalation policy; no new MUST.
- Edit the canonical `skills/lifecycle/references/complete.md` only, then regenerate the `plugins/cortex-core/...` mirror via `just build-plugin` (the pre-commit dual-source hook also regenerates it on commit). `tests/test_dual_source_reference_parity.py` byte-compares the pair at test time, so the mirror must be synced before `just test` is run (R12).
- **Do-not-touch snapshot region**: `tests/test_complete_md_hard_guard_snapshot.py` byte-pins the Step 8 `**Hard guard**:` paragraph against a fixture. The insertion after Step 11 is downstream of that region and must not edit it; if a future revision touches the hard-guard paragraph, update the fixture in the same change.
- Author per "prescribe What and Why, not How"; concreteness is limited to the staging set and the structural anchor (consistent with the existing house style in this commit-staging area and the structural-separation guidance).
- This is a `skills/` edit, so it runs through a lifecycle (the reason #275 is being refined).
