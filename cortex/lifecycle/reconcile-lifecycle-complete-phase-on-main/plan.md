# Plan: reconcile-lifecycle-complete-phase-on-main

## Overview
Insert a single flag-gated, stage-first artifact-commit step at the lifecycle Complete phase's finalization tail (after Step 11 `feature_complete`, before the Step 12 summary), delimited by a stable HTML-comment anchor. The step runs on **all** completion paths (trunk, worktree-interactive post-merge, feature-branch post-merge) — no branch gate. The stage-first `git diff --cached --quiet` guard keeps it safe: on the worktree-interactive flow the already-merged lifecycle artifacts stage nothing, so only the new post-merge tail commits. It reuses `post-refine-commit.md`'s stage-first no-op detector and halt-on-failure discipline; adds a non-default-branch advisory for the discouraged feature-branch flow; introduces no new event type, no new MUST, no new user pause. A scoped guard test plus a cortex-core mirror regeneration round out the change so the dual-source and kept-pauses parity tests stay green.

  - **`false`-branch (R4)**: when the flag prints `false`, skip the commit entirely and note inline that lifecycle artifacts and any uncommitted source are left in the working tree for the operator to commit deliberately (mirrors `post-refine-commit.md` §"Flag Check" and `plan.md §5`).
  - **Staging set (R5)**: stage the lifecycle dir by *enumerated* paths — `research.md`, `spec.md`, `plan.md`, `review.md`, `index.md`, `events.log` (those present) — never `git add cortex/lifecycle/{slug}/` (avoids sweeping un-gitignored residue like `critical-review-residue.json`, `learnings/*`). Never `git commit` directly, never `-A`.
  - **Stage-first idempotent guard (R2)**: after staging, run `git diff --cached --quiet`; exit 0 (nothing staged) → skip `/cortex-core:commit` silently; exit 1 → commit. Pattern precedent is `post-refine-commit.md` §"No-Op Short-Circuit (Stage-First)". This guard is what prevents a double-commit on the worktree-interactive flow.
  - **Commit (R1)**: invoke `/cortex-core:commit` with an imperative ≤72-char subject, e.g. `Complete {slug}: lifecycle artifacts and backlog write-back`.
  - **Halt-on-failure (R3)**: if `/cortex-core:commit` exits non-zero, surface the error and stop before the Step 12 summary — do not emit a "lifecycle complete" summary implying the artifacts were committed (precedent: `post-refine-commit.md` §"Halt-Before-Plan Gate").
  - **Non-default-branch advisory (R13)**: after a successful commit, if the current branch is not `main`/`master`, surface a one-line advisory that the artifacts were committed on `<branch>` rather than the default branch (covers the discouraged feature-branch no-worktree flow, where the tail lands on `feature/{slug}`). Do NOT auto-`checkout` the trunk branch — branch normalization is deferred (Non-Requirements). Soft advisory, no MUST.
  - **No PR routing (R6)**: the step is a local commit only — no `git push`/`gh pr create`/PR-creation instruction, and no `git add cortex/lifecycle/{slug}/` directory-glob.
  - **No new event (R7)**: reuse the existing Step 11 `feature_complete` event; introduce no `commit_authored`/`artifacts_committed`/`commit_failed`. Commit success is observable via `git log`, failure via `/cortex-core:commit`'s error path.
  - Placement-after-Step-11 rationale (Edge Cases): the closing `feature_complete` row lands inside the commit, preserving terminal design history; the commit-fail-after-append window is status-quo-equivalent.
  - Do **not** edit the Step 8 `**Hard guard**:` paragraph — `tests/test_complete_md_hard_guard_snapshot.py` byte-pins it; the insertion is downstream of that region.

## Risks
- **Feature-branch flow commits to the merged feature branch, not `main`.** On the discouraged `feature/{slug}` no-worktree PR flow, the finalization commit lands the post-merge tail on `feature/{slug}` (already merged) rather than `main`. This plan commits it there + surfaces the R13 advisory rather than auto-`checkout main` (which risks conflicts and entangles with local-merge state). Branch normalization is deferred to the Step 7 routing follow-up. The artifacts are committed-and-reachable, not left dirty — better than status quo, imperfect on placement.
- **Three artifact-only patch releases on a trunk completion.** This adds a third artifact-only commit to main (refine + plan + complete), so `auto-release.yml` may cut one more patch release per trunk-flow lifecycle. Spec calls this consistency-preserving and out of scope — flagging it as an operator-visible consequence, not a defect. (On the PR flows the finalization commit is local and only triggers a release if/when pushed.)
- **Recovery-path masking is pre-existing, not introduced.** After a commit failure, Step 7 Branch 2 (`feature_complete` present → summary) short-circuits re-invocations past the new step, so "safe to re-run" holds only on first invocation. This is status-quo-equivalent — Step 11 already appends `feature_complete` uncommitted today and nothing recovers it. The residual recovery hazard lives in deferred Step 7 routing, not in this step.
- **Step 2 short-circuit pointer edit.** Task 1 lightly edits the line-21 prose to name the new step. This is not a routing change (the Non-Requirements defer Step 7 state-aware routing); keep the edit to a pointer reference, not a re-routing.

## Acceptance
An interactive lifecycle completion run with `commit-artifacts=true` lands a single `/cortex-core:commit` ("Complete {slug}: …") containing the enumerated lifecycle artifacts (through the `feature_complete` row) plus the full `cortex/backlog/` write-back — leaving the working tree clean — on the trunk path and on the worktree-interactive post-merge tail (which finalizes on `main`); on the feature-branch post-merge tail it commits the tail on `feature/{slug}` and surfaces the non-default-branch advisory. `commit-artifacts=false` or an already-clean staging area skips the commit silently, and the worktree-interactive flow produces no double-commit (the stage-first guard skips already-merged artifacts). `just test` exits 0 with the new scoped guard test and both the dual-source and kept-pauses parity tests green.
