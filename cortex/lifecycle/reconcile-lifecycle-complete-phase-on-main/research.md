# Research: Reconcile lifecycle complete-phase on-main short-circuit with commit-artifacts flag

> Topic: Make the lifecycle Complete phase's on-main short-circuit (`skills/lifecycle/references/complete.md`) commit lifecycle artifacts (gated on `commit-artifacts=true`), so trunk-based completions preserve design history the same way the PR flow and the earlier Refine/Plan phases already do. The on-main path currently skips Step 2 ("Commit Lifecycle Artifacts") and jumps to Steps 9–12, which never commit the lifecycle dir.

## Codebase Analysis

**The bug** — `skills/lifecycle/references/complete.md:21` (inside Step 2):
> "On-main short-circuit: if the current branch is `main` or `master`, skip Steps 2–5 … and proceed to Step 7 with pr.json absent and no orphan-PR probe needed — treat as first-run path jumping directly to Steps 9–12."

By skipping Steps 2–5 it skips Step 2's artifact commit (lines 17–19). It then runs Steps 9–12 (lines 198–244): Step 9 backlog write-back, Step 10 index sync, Step 11 `feature_complete`, Step 12 summary — **none of which commit**. Result: `plan.md`, `events.log` (through `feature_complete`), `review.md`, `index.md`, and the backlog write-back (`cortex/backlog/index.json` + `index.md` + the item `.md`) are left uncommitted.

**File facts.** `complete.md` is 253 lines (well under the 500-line SKILL cap; the cap applies to SKILL.md, and this is a `references/` file). `plugins/cortex-core/skills/lifecycle/references/complete.md` is an **auto-generated mirror** — edit the canonical source only; the pre-commit dual-source hook regenerates the mirror.

**Insertion point.** Steps 9–10 (the backlog write-back) run *before* any commit, so their outputs (`index.json`, `index.md`, the item `.md`) are available to stage. The ticket suggests committing **after Step 11 (`feature_complete`)**. See Tradeoffs/Adversarial for why the ordering relative to `feature_complete` is the central design question.

**Patterns to reuse (existing precedent):**
- **Stage-first no-op detector** — `post-refine-commit.md:42–53`: after a targeted `git add`, run `git diff --cached --quiet` (exit 0 = nothing staged → skip silently; exit 1 = commit). This is the codebase's canonical idempotency idiom and directly satisfies the ticket's "avoid double-committing" edge.
- **HALT-on-commit-failure** — `post-refine-commit.md:66–70`: if `/cortex-core:commit` exits non-zero, surface the error and stop; do not advance. (Note: that gate uses a *grandfathered* MUST — per the MUST-escalation policy, do not author a *new* MUST without an evidence artifact; use soft positive-routing phrasing.)
- **Subject selection** — path-derived, single unambiguous subject (`post-refine-commit.md:56–64`). Natural subject here: `Complete {slug}: lifecycle artifacts and backlog write-back` (imperative, ≤72 chars, no trailing period).

**Branch-conditioning baseline (corrects the ticket's framing):** `post-refine-commit.md` and `plan.md §5` (line ~306) are **NOT branch-conditioned** — they commit *unconditionally*, gated only on the flag. The PR-flow Step 2 (`complete.md:19`) is also unconditional/flag-gated. The on-main complete path is the **only** place that conditions the artifact commit on branch — that branch-conditional skip *is* the inconsistency the ticket targets.

**Integration points / dependencies:**
- `cortex-read-commit-artifacts` (`bin/` → `cortex_command/lifecycle_config:_main`): prints `true`/`false`; **default `true`** (file/field absent → true). Confirmed `_COMMIT_ARTIFACTS_DEFAULT = True` in `cortex_command/lifecycle_config.py`.
- `/cortex-core:commit`: invocation idiom across all callers is **stage-first, then invoke** (callers `git add <specific paths>` then invoke the skill; they do not pass paths to it). Stage specific files, never `-A`.
- `cortex-update-item <slug> --status complete --session-id null` (Step 9) mutates the backlog item `.md`; index regen (Step 10, `cortex_command/backlog/generate_index.py` or `cortex-generate-backlog-index`) regenerates `cortex/backlog/index.json` + `index.md`. All three are part of the "Complete lifecycle" commit's staging set.

**No new wiring / events.** Reuses `feature_complete` (registry row in `bin/.events-registry.md`, gate-enforced, producer already lists `complete.md`). All scripts used are already wired → no `bin/.parity-exceptions.md` change. Per post-refine-commit precedent, do **not** introduce `commit_authored`/`artifacts_committed`/`commit_failed` events — commit success is observable via `git log`, failure via `/cortex-core:commit`'s own error path.

**This fix is interactive-only.** The overnight runner does **not** run this interactive `complete.md` on-main path — overnight emits `feature_complete` directly from Python (`cortex_command/pipeline/review_dispatch.py`, `merge_anchor: "review"`) and its artifact commits land on the *integration branch* and travel with the PR (`pipeline.md:22–24`); only the morning report goes to local `main`. The interactive `complete.md` is the sole producer of `merge_anchor: "merge"`. Builders commit per-task during implement, so source is usually already committed by the time Complete runs on a trunk-based flow.

**Tests.** No existing test asserts on the on-main short-circuit text (`tests/test_lifecycle_complete_state_routing.py` exercises 12 routing branches, none on-main) — the buggy behavior is **untested**. ## Web Research

- **Idempotent commit guard:** `git diff --cached --quiet` (= `--staged --quiet`) is the canonical, self-healing guard — exit 0 = nothing staged (skip), exit 1 = commit. After a crash where artifacts were already committed, re-running stages nothing new, the guard sees an empty staged diff, the commit is skipped. **No separate dedup token is needed** — git's index-vs-HEAD comparison *is* the idempotency check. Anti-patterns: `git commit || true` (swallows real errors — hook rejection, conflicts), `--allow-empty` (only for intentional marker commits).
- **Trunk-based vs PR finalization:** Committing directly to `main` is a sanctioned trunk-based *style*, not an anti-pattern. The literature frames PR-vs-direct as differing *only* in where human review gates the merge — which supports keeping the two flows **behaviorally consistent in their finalization side effects** (both end with artifacts committed). The current on-main skip is exactly the inconsistency the prior art would flag.
- **Ordering of side effects vs a terminal "done" marker (most decision-relevant):** Distributed-systems guidance (two-phase reservation, transactional outbox) converges on: **record durable work BEFORE flipping the terminal "done" marker.** Mark-then-crash-before-commit silently *loses* work; commit-then-crash-before-mark *self-heals* (re-invocation re-stages nothing, guard skips, marker then written). The practical resolution to the dual-write (commit + event-log append cannot share one atomic transaction) is at-least-once + idempotent = "effectively once" — don't chase atomicity, make the commit re-runnable and write the marker last. **This contradicts the ticket's suggested "commit after `feature_complete`" placement** — see Open Questions.
- **Atomic-commit bundling:** Grouping the regenerated index + status flip + artifacts into one commit is sound — they're one logical change ("finalize this feature"), keeping revert trivial.

Sources: git-scm.com/docs/git-diff; trunkbaseddevelopment.com/styles; microservices.io transactional-outbox; AWS prescriptive-guidance transactional-outbox; dev.to idempotency-in-distributed-systems. WebFetch was available; no unfetchable URLs.

## Requirements & Constraints

- **`project.md` "Multi-step lifecycle phases"**: the Complete phase is the canonical multi-step example; "merge is the terminal event for Done"; "re-invocation routing is state-aware and idempotent." The on-main path is the degenerate (no-PR) variant of the same phase.
- **`project.md` "Solution horizon"**: a consistency fix aligning on-main with the PR flow + Refine/Plan precedent is a *durable* fix, not a stop-gap.
- **`project.md` "Complexity"**: reuse existing `cortex-read-commit-artifacts` + `/cortex-core:commit` rather than new mechanism — simplicity-aligned.
- **ADR-0004 (accepted — MUST honor)**: established the multi-step Complete phase and **rejected R2 "bimodal Complete"** because "the same lifecycle command has different observable semantics is harder to reason about." This *supports* making on-main consistent with PR-flow, against introducing a second divergence. ADR-0004 does not specify on-main artifact-commit behavior, so editing `complete.md` needs **no ADR amendment** (reversible by a one-file edit; fails the three-criteria "hard to reverse" gate).
- **`pipeline.md`**: overnight runner is a separate path (integration-branch commits travel with the PR; pr.json atomicity invariant at `pipeline.md:124–130` is not exercised on-main since pr.json stays absent).
- **`commit-artifacts` semantics**: default `true`; gates staging of `lifecycle/` in commits; each phase reads the flag independently (no centralized resolver). When `false`, existing mid-lifecycle consumers (post-refine-commit, plan §5) **skip the commit entirely**; PR-flow Step 2 still commits *source* and only excludes artifacts. (Terminal-phase behavior is undecided — see Open Questions.)
- **CLAUDE.md authoring constraints**: (a) **MUST-escalation policy** — use soft positive-routing phrasing for the new step; no new MUST without an events.log F-row / transcript evidence artifact + recorded `effort=high` attempt. (b) **"Prescribe What and Why, not How"** — describe the decision/gate/intent; the existing house style for this commit-staging area is moderately concrete (it names the explicit path set to avoid sweeping residue), so concreteness about *what to stage* is consistent. (c) **Kept user pauses** — the on-main commit adds **no** pause and touches no kept pause (on-main already skips Step 6's merge-wait pause). (d) **Structural over prose-only** — the on-main vs PR split is encoded in control flow (Step 2's branch check); keep it structural.
- **Commit conventions** (`skills/commit/`): imperative, capitalized, ≤72-char subject, no trailing period; stage specific files (not `-A`); a PreToolUse hook validates messages — do not bypass.
- **Scope**: editing `skills/lifecycle/references/complete.md` is in-scope AI-workflow orchestration and (per the lifecycle skill's own description) requires the lifecycle gate — which is exactly why #275 is being refined for execution.

## Tradeoffs & Alternatives

Placement of the on-main artifact commit relative to `feature_complete`:

- **Approach A — commit AFTER Step 11 (`feature_complete`)** [ticket's suggestion]: one commit captures the closing event row + all write-back mutations. *Pro:* single commit, matches plan.md §5 / PR-flow Step 2 shape; clean design history. *Con (the crux):* if `feature_complete` is appended and then `/cortex-core:commit` **fails**, the in-tree row is uncommitted, and on a *fresh* re-invocation Step 7 **Branch 2** ("`feature_complete` present → short-circuit to Step 12") fires on the in-tree row and **permanently masks** the stranded commit. A same-run halt gate does **not** close this (verified: Branch 2 evaluates at the top of Step 7 on re-entry, before any commit logic).
- **Approach B — commit BEFORE Step 11**: commit artifacts, then append `feature_complete`. *Pro:* crash-safe and self-healing (commit-then-crash → re-invocation re-stages nothing, guard skips, marker written) — matches the distributed-systems "marker last" principle. *Con:* the closing `feature_complete` row is left uncommitted on the happy path; on a *terminal* path no later phase sweeps it (post-refine's explicit-path staging targets a *specific* feature dir, so a different feature's commit won't pick it up), so design history is missing its final event indefinitely. The "orphaned-row-under-wrong-subject" defect that justified "after" for the Refine→Plan boundary does **not** transfer to a terminal Complete, but the "row never committed" gap is real.
- **Approach C — commit AFTER Step 11 + a cleanliness-aware on-main routing guard**: add a "Branch 0" on-main guard at the top of Step 7 that routes on `feature_complete` presence **and** lifecycle-dir git-cleanliness (present + clean → summary; present + dirty → re-commit). Captures the closing row (A's benefit) *and* self-heals (B's benefit). *Con:* highest complexity — modifies Step 7 routing, needs a new test. But the on-main routing guard is **needed anyway** (see Adversarial #6 — the crashed-on-main routing gap), so the incremental cost over "A + required guard" is just the cleanliness check.
- **Approach D — stage-first no-op guard (`git diff --cached --quiet`)**: cross-cutting; adopt unconditionally regardless of placement. Free given the existing template; satisfies the "avoid double-committing" edge.

**Recommended (for Spec to confirm):** Approach C + D + explicit-path staging + halt-on-failure, IF the routing-guard scope is accepted; otherwise Approach A + D as the minimal fix, accepting the residual masking hazard mitigated by the halt message. The ordering choice (A vs B vs C) is the central decision the Spec must lock.

**Staging — explicit path set, NOT a glob (verified):** `critical-review-residue.json` is NOT gitignored and is tracked in **74** lifecycle dirs (the existing plan.md glob already sweeps it — a defect post-refine-commit deliberately fixed by switching to explicit paths). `learnings/*` (except `recovery-log.md`) is also un-ignored. Stage the explicit set: `research.md spec.md plan.md review.md index.md events.log` + the backlog item `.md` + `cortex/backlog/index.json index.md`. (`.session`/`.lock`/`.dispatching`/`agent-activity.jsonl` are gitignored and excluded automatically, but explicit paths also exclude the un-ignored residue/scratch files.)

## Adversarial Review

- **Branch-2 masking under Approach A is real; the halt gate doesn't fix fresh re-invocation.** Verified `complete.md:103–171`: Branch 2 scans the in-tree `events.log` at the top of Step 7. A new session after a commit-failure has lost the prior halt message. → favors Approach B or C.
- **On-main complete IS single-invocation on the happy path** (Step 6 pause is skipped), so the masking hazard only arises on crash/failure recovery — lowering frequency but not eliminating the risk (a commit-hook rejection or the auto-release hook firing is exactly when clean recovery matters).
- **Glob is provably unsafe** (verified — 74 tracked residue files). Use explicit paths.
- **`commit-artifacts=false` on-main — invert the "mirror PR-flow" recommendation.** Auto-committing arbitrary uncommitted working-tree *source* (scratch edits, debug prints) directly to `main` unprompted, on a terminal path with no PR review gate, is more dangerous than leaving it uncommitted. Stranding source is recoverable (user commits deliberately); auto-committing it to main is not. **Recommend: `commit-artifacts=false` on-main → commit nothing, document inline.**
- **Auto-release patch bump** (verified `auto_bump_version.py`): `classify_messages` classifies by commit *message* only (default `patch`), never by changed files; `no-bump` only on an empty range. So an artifact-only "Complete lifecycle" commit on main contributes a patch bump for zero shipped code. **Important nuance:** this is a *pre-existing project-wide property* — refine/plan commits are already artifact-only and already land on main and already participate in the bump. So it is consistency-preserving, likely **not blocking** for this fix; if "artifact commits shouldn't cut releases" is desired, that's a separate, broader ticket affecting refine/plan too. Surface to the user for a scope call.
- **Crashed-on-main re-invocation routing gap (sharpest, possibly pre-existing).** If an on-main run crashes *before* `feature_complete` is written, re-invocation falls through Branch 1/2 to **Branch 3** (pr.json absent → orphan-PR probe → "zero matches → run first-run path Steps 1–6") — which on main re-runs tests and re-enters Step 2, and a literal agent at Branch 3 runs `gh pr list --head interactive/{slug}` (the PR-creation trap). The "no orphan-PR probe needed" caveat on line 21 only helps an agent threading from Step 2, not one re-entering at Step 7. **Mitigation:** add an on-main guard (Branch 0) at the top of Step 7 that bypasses the orphan-PR probe and routes solely on `feature_complete` presence (+ dir cleanliness for Approach C). This is partly a pre-existing latent bug; whether to fix it here is a scope decision.

## Open Questions

> **Deferred to Spec:** all four items below are design decisions for the Spec phase's structured interview and §4 complexity/value gate — they are not code-resolvable ambiguities. Each carries a research recommendation; the user confirms or redirects during Spec. None blocks the Research→Spec transition.

1. **Commit ordering vs `feature_complete` (PRIMARY — Spec must resolve).** Approach A (after — clean history, residual masking hazard on commit-failure), Approach B (before — crash-safe, closing row left uncommitted indefinitely on a terminal path), or Approach C (after + cleanliness-aware on-main routing guard — both benefits, highest complexity). Research leans **C** (most durable; the routing guard is needed regardless per OQ3) with **A+halt** as the minimal fallback. *Contradiction:* Web/Adversarial agents argue "marker last" (B/C); Codebase/Tradeoffs agents initially favored A.
2. **`commit-artifacts=false` on-main behavior.** Mirror PR-flow (commit source, exclude artifacts) vs commit nothing + document inline. Research (post-adversarial) recommends **commit nothing + document inline** — auto-committing stray source to main unprompted is the larger surprise on a terminal, review-gate-free path. Confirm with user.
3. **Scope: fix the crashed-on-main re-invocation routing gap (Branch 0 guard) in this ticket, or defer?** The Branch-3 PR-creation trap on a crashed on-main re-invocation is partly pre-existing. Including the Branch 0 guard makes the fix durable and is a prerequisite for Approach C; deferring keeps the change minimal but leaves the latent trap (and forces Approach A's residual hazard). This is the §4 complexity/value decision.
4. **Auto-release on artifact-only main commits.** Confirmed pre-existing (refine/plan already do this). Likely out of scope for #275 (broader ticket if undesired), but flag to the user so the consistency consequence is an explicit choice, not a silent side effect.
