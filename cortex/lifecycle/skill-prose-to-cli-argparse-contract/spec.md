# Specification: skill-prose-to-cli-argparse-contract

## Problem Statement

Three skill prose callsites invoke `cortex-create-backlog-item` without the required `--status` and `--type` flags (`skills/morning-review/SKILL.md:91`, `skills/morning-review/references/walkthrough.md:416`, `skills/backlog/SKILL.md:62`, `skills/discovery/SKILL.md:87`). Each invocation fails at runtime; the failures have surfaced across multiple sessions. There is no automated mechanism to catch flag-contract drift between skill prose and the argparse surface of `[project.scripts]`-registered modules — `bin/cortex-check-parity` operates at file-level altitude (does the named script exist?) but does not validate argument shape. This spec defines a pre-commit lint that closes that altitude gap for argparse-shaped modules, blocks the named drift at commit time, and fixes the named drift in the same landing PR.

## Phases

- **Phase 1: Lint core and exception governance** — implement the AST-based parser-surface extractor, the prose-grep scanner, the exception-ledger format, and the JSON/text output surfaces.
- **Phase 2: Pre-commit and corpus integration** — wire the lint into `.githooks/pre-commit` (Phase 1.55), add the `justfile` recipes, scrub the named drift in the same PR, populate the initial exception ledger with categorical entries for non-argparse modules, then flip the lint to hard-block on landing.

## Requirements

1. **AST-based parser-surface extractor**: A `cortex_command/lint/contract.py` module reads each `[project.scripts]`-registered module's source via `ast`, identifies `argparse.ArgumentParser(...)` constructor calls and `.add_argument(...)` / `.add_subparsers(...)` / subparser `.add_parser(...)` calls, and builds a static representation of accepted flags (long-form `--flag` and short-form `-f`), subcommand names, and `required=True` markers. Modules without an extractable parser are categorically exempted via the exception ledger, not silently skipped. Acceptance: `uv run python3 -c "from cortex_command.lint import contract; surface = contract.extract_surface(); assert 'cortex-create-backlog-item' in surface and '--status' in surface['cortex-create-backlog-item']['required_flags']"` exits 0. **Phase**: Phase 1: Lint core and exception governance.

2. **Code-fence-aware prose scanner**: The lint scans `skills/**/*.md`, `hooks/**`, `justfile`, `docs/**/*.md`, `tests/**`, `CLAUDE.md`, and `cortex/requirements/**/*.md`. It uses the fence state machine from `cortex_command/lint/prescriptive_prose.py:113-156` (or a structurally equivalent state machine) to distinguish fenced code blocks from inline-code spans from prose mentions. Only invocations inside fenced code blocks AND inline-code spans are subject to flag validation; narrative prose mentions (e.g., `The cortex-create-backlog-item helper...`) are exempt. Acceptance: `uv run python3 -m cortex_command.lint.contract --self-test` exits 0 — the self-test fixture contains one fenced invocation (must be validated), one inline-code invocation (must be validated), and one prose mention (must be skipped). **Phase**: Phase 1: Lint core and exception governance.

3. **Invocation-to-surface validation**: For each scanned invocation matching the regex `cortex-[a-z][a-z0-9-]*` followed by tokens, the lint tokenizes the tail with `shlex.split(comments=False)`, classifies tokens as subcommands (positionals matching a `_SubParsersAction` choice) or flags (tokens starting with `-`), and validates that (a) every observed flag exists in the parser's accepted set, (b) every `required=True` flag is present on the invocation, (c) the resolved subcommand exists if the parser uses subparsers. Templated placeholders matching `{{...}}`, `{...}`, or `<...>` are treated as opaque values (skip value-shape validation). Acceptance: `tests/fixtures/contract/invalid-missing-required-flag/` is a mini-repo containing a `skills/X/SKILL.md` with `cortex-create-backlog-item --title X --body Y` (missing `--status`/`--type`); `uv run python3 -m cortex_command.lint.contract --root tests/fixtures/contract/invalid-missing-required-flag/` exits 1 and reports `E101 missing required flag --status` and `E101 missing required flag --type`. **Phase**: Phase 1: Lint core and exception governance.

4. **Exception ledger at `bin/.contract-lint-exceptions.md`**: A markdown-table allowlist mirrors the structure of `bin/.parity-exceptions.md`. Columns: `binary | flag-or-subcommand | path-glob | category | rationale | lifecycle_id | added_date`. Categories: `non-argparse-module` (module uses hand-rolled or key=value argv parsing — e.g., `update_item.py`), `migration-note` (prose intentionally references deprecated form for documentation), `template-fragment` (prose is a documentation example showing partial syntax), `intentional-omission` (rare cases where required-flag enforcement is incorrect for a specific callsite). Rationale ≥30 characters, forbidden literals (`internal`, `misc`, `tbd`, `n/a`, `pending`, `temporary`) rejected case-insensitively. Acceptance: `uv run python3 -m cortex_command.lint.contract --validate-exceptions` exits 0 with the seeded initial exception entries (see Requirement 9); rationale validation rejects a fixture entry with rationale `"tbd"`. **Phase**: Phase 1: Lint core and exception governance.

5. **Inline sentinel marker**: The string `<!-- contract-lint:ignore-next -->` (line-level granularity) when present on the immediately preceding non-blank line suppresses validation for the next invocation. Documented in the skill prose of `bin/cortex-check-contract --help` and the justfile recipe description. Acceptance: `tests/fixtures/contract/valid-sentinel-ignored/` contains a SKILL.md with the sentinel above a deliberately-invalid invocation; lint exits 0 against that fixture. **Phase**: Phase 1: Lint core and exception governance.

6. **JSON and text output surfaces**: `--json` emits a JSONL violation array compatible with the `parity_check.Violation.format_text()` shape (`<path>:<line>:<col>: <CODE> <message>`); default text output mirrors `cortex-check-parity` formatting. Exit codes: 0 clean, 1 violations, 2 configuration error. Acceptance: `uv run python3 -m cortex_command.lint.contract --json --root tests/fixtures/contract/invalid-missing-required-flag/` produces parseable JSON with at least one entry whose `code` is `E101`. **Phase**: Phase 1: Lint core and exception governance.

7. **Two-mode `--staged` / `--audit`**: The lint exposes `--staged` (operate on staged diff, mirrors `cortex-check-parity --staged`) and `--audit` (operate on the full repo corpus, invoked via `just check-contract-audit`). Mutually exclusive. `--staged` reads staged blobs from git; `--audit` walks the scan globs. Acceptance: `git stash && touch skills/_test_drift.md && echo 'cortex-create-backlog-item --title X --body Y' > skills/_test_drift.md && git add skills/_test_drift.md && bin/cortex-check-contract --staged; rc=$?; git restore --staged skills/_test_drift.md && rm skills/_test_drift.md && git stash pop; test "$rc" = "1"` succeeds (staged mode detects the drift before commit). **Phase**: Phase 1: Lint core and exception governance.

8. **`bin/cortex-check-contract` dual-channel wrapper**: A bash wrapper at `bin/cortex-check-contract` mirrors `bin/cortex-check-parity:1-35`: try wheel-installed module first, fall back to working-tree `python3 -m cortex_command.lint.contract`. `CORTEX_COMMAND_FORCE_SOURCE=1` forces working-tree. Invokes `cortex-log-invocation` shim for telemetry. Acceptance: `bin/cortex-check-contract --help` exits 0 and prints the argparse-rendered usage. **Phase**: Phase 1: Lint core and exception governance.

9. **Initial exception ledger seed**: `bin/.contract-lint-exceptions.md` is created with categorical entries (`non-argparse-module` category) covering the modules whose argv parsing is not AST-extractable: `cortex-update-item` (key=value form, pending backlog #257), `cortex-lifecycle-state` (hand-rolled argv loop), `cortex-worktree-resolve` (single positional, no argparse), `cortex-overnight-interrupt` (direct `sys.argv` access), `cortex-log-invocation` (direct `sys.argv` access). Each entry cites lifecycle_id `253-skill-prose-to-cli-argparse-contract-lint` and rationale identifying the specific argv-handling style. Acceptance: `grep -c "^| cortex-update-item " bin/.contract-lint-exceptions.md` ≥ 1 and `grep -c "non-argparse-module" bin/.contract-lint-exceptions.md` ≥ 5. **Phase**: Phase 2: Pre-commit and corpus integration.

10. **Pre-commit hook integration as Phase 1.55**: `.githooks/pre-commit` gains a new phase block between Phase 1.5 (parity) and Phase 1.6 (shim presence) labeled `# Phase 1.55 — Skill-prose to CLI argparse contract enforcement`. The phase triggers on staged diffs matching the scan globs (skills/, hooks/, justfile, docs/, tests/, CLAUDE.md, cortex/requirements/) and invokes `just check-contract --staged`. Non-zero exit blocks commit. Acceptance: `grep -n "Phase 1.55" .githooks/pre-commit` returns at least one match and the surrounding block invokes `just check-contract --staged`. **Phase**: Phase 2: Pre-commit and corpus integration.

11. **Justfile recipes**: `justfile` gains `check-contract *args:` (invokes `bin/cortex-check-contract {{args}}`) and `check-contract-audit:` (invokes `bin/cortex-check-contract --audit`). Acceptance: `just --list 2>&1 | grep -E '(check-contract|check-contract-audit)'` returns both recipes. **Phase**: Phase 2: Pre-commit and corpus integration.

12. **In-PR drift fixes**: The same PR that introduces the lint also fixes the four named drift callsites by adding the missing `--status` and `--type` flags. Specifically: `skills/morning-review/SKILL.md:91` (status: `should-have`, type: `bug`), `skills/morning-review/references/walkthrough.md:416` (status: `should-have`, type: `bug`), `skills/backlog/SKILL.md:62` (status: `backlog`, type derived from existing `--type {{type}}` template at line 54 — defer template choice to backlog skill's template inheritance), `skills/discovery/SKILL.md:87` (status: `needs-discovery`, type: `discovery`). Acceptance: `bin/cortex-check-contract --audit` exits 0 against the post-fix tree. **Phase**: Phase 2: Pre-commit and corpus integration.

13. **Test fixture suite**: `tests/test_check_contract.py` parametrizes over `tests/fixtures/contract/valid-*`, `invalid-*`, and `exclude-*` subdirectories mirroring `tests/test_check_parity.py:34-90`. Fixtures cover: (a) valid invocation passes, (b) missing required flag fails with `E101`, (c) unknown flag fails with `E102`, (d) sentinel-suppressed invocation passes, (e) prose-mention-only file passes (no fenced/inline-code occurrence), (f) exception-ledger entry suppresses a flagged invocation, (g) non-argparse module invocation skipped via ledger category. Acceptance: `uv run pytest tests/test_check_contract.py -v` exits 0 with at least seven parametrized cases. **Phase**: Phase 2: Pre-commit and corpus integration.

## Non-Requirements

- **No derived artifact emitted**: 253 does NOT produce a persistent JSON artifact of the canonical CLI surface enumeration. The supposed downstream consumer (`cortex_command/doctor/path_self_test.py` from backlog #252) already shipped reading entry points via `importlib.metadata.entry_points()` directly; there is no actual reader for a persisted artifact. If a future lifecycle needs the enumeration as an artifact, it can either (a) call `cortex_command.lint.contract.extract_surface()` programmatically, or (b) introduce its own emission with explicit consumer.
- **No `python3 -m cortex_command.*` validation**: The bare-python-m audit gate (`bin/.audit-bare-python-m-allowlist.md` + `--audit-bare-python-m-callsites` in `parity_check.py`) is responsible for retiring that syntax. 253 validates `cortex-*` console-script form only. Drift involving `python3 -m cortex_command.discovery` style invocations is the audit gate's responsibility.
- **No `cortex-update-item` validation**: The `key=value` calling convention used by `cortex-update-item` has no argparse parser to validate against. Backlog #257 tracks the syntax refactor; until #257 lands, `cortex-update-item` invocations are categorically exempted via the exception ledger.
- **No mutually-exclusive-group, custom-Action, or exotic-nargs validation**: v1 validates flag-name presence, `required=True` enforcement, and subcommand resolution. `add_mutually_exclusive_group()`, custom `argparse.Action` subclasses, `argparse.SUPPRESS` flags, and exotic `nargs` (`'?'`, `'*'`, `'+'`, `'REMAINDER'`) are out of scope. These produce false-negatives (accept invocations the parser would reject) — the safer-failure direction for initial rollout. A v1.5 follow-up lifecycle owns broadening this surface.
- **No archive corpus scanning**: `cortex/research/archive/**`, `CHANGELOG.md`, `bin/.audit-*-allowlist.md`, `bin/.parity-exceptions.md`, and `cortex/lifecycle/**` are hard-excluded from the scan glob set. These surfaces legitimately reference retired or deprecated invocations as historical record.
- **No baseline-mode "fail only on new violations"**: v1 fails on all drift; the four named callsites are the only known active drift and are fixed in the same PR (Requirement 12). Baseline-mode is unnecessary.
- **No backwards-compatibility shim for the lint's own evolution**: when the lint's violation codes or output format changes in a future lifecycle, callers re-pin to the new schema. No compatibility shims.

## Edge Cases

- **Module import fails during AST extraction**: The lint reads source via `ast.parse(open(path).read())`, not via `importlib.import_module`. Import failures (syntax errors, missing deps) of the target module do not block AST extraction. If a module's source is unparseable, the lint emits `E201 cannot AST-parse module source for cortex-X at path Y` and continues with other modules; this is a configuration error (exit 2) only if every `[project.scripts]` module is unparseable.
- **Multiple `ArgumentParser` instances in a single module**: When AST walking finds more than one `argparse.ArgumentParser()` constructor at module scope, the lint identifies the parser whose subsequent `.add_argument(...)` calls are most numerous (the "main" parser); other instances are ignored. If the heuristic is ambiguous (two parsers with equal add_argument counts), emit `E202 ambiguous main parser for cortex-X` and treat the module as exempt-equivalent until the module is restructured.
- **Subcommand-only invocation in prose**: `cortex-discovery` mentioned without a subcommand inside a fenced code block (e.g., as a usage demonstration). If the binary requires a subcommand (`required=True` on the `_SubParsersAction`), emit `E103 missing required subcommand for cortex-discovery`. Sentinel marker or ledger entry overrides.
- **Invocation split across multiple lines via backslash continuation**: The scanner pre-joins backslash-continued lines within a fenced block before tokenizing. Continuation across a fence boundary (extremely rare) is not supported; the continuation is treated as two separate invocations.
- **Inline-code span containing only the binary name**: `` `cortex-foo` `` inside narrative prose is a name-mention, not an invocation. Heuristic: inline-code is treated as an invocation only if it contains at least one whitespace-separated token after the binary name. A bare-name inline-code is skipped.
- **Lint catches drift in its own skill prose**: First run on the post-landing tree must pass. The lint's own usage examples in `bin/cortex-check-contract --help` output and in any skill prose mentioning it must use canonical syntax; verified by the in-PR drift scrub (Requirement 12).
- **Pre-commit invocation fails on missing project venv**: The lint requires the project's Python environment (`yaml`, `markdown`, etc.). If invoked outside `uv run`, the wrapper falls back to `python3 -m cortex_command.lint.contract` against the installed wheel. If neither is available, exit 2 with a clear configuration-error message identifying how to install.

## Changes to Existing Behavior

- **ADDED**: A new pre-commit gate at Phase 1.55 validates argparse flag-contract drift in skill prose. Commits that introduce drift in the scan corpus are blocked.
- **ADDED**: A new exception-ledger surface at `bin/.contract-lint-exceptions.md` that contributors maintain when adding intentional exemptions (migration notes, template fragments, non-argparse modules).
- **ADDED**: A new sentinel comment marker (`<!-- contract-lint:ignore-next -->`) recognized in skill prose for line-level exemptions.
- **MODIFIED**: Four existing skill callsites (`skills/morning-review/SKILL.md:91`, `skills/morning-review/references/walkthrough.md:416`, `skills/backlog/SKILL.md:62`, `skills/discovery/SKILL.md:87`) gain explicit `--status` and `--type` flags. Runtime behavior of these callsites changes from "fails with argparse error" to "succeeds with the documented status/type pair."
- **ADDED**: A new `cortex-check-contract` console script in `[project.scripts]` and a corresponding `bin/cortex-check-contract` wrapper.
- **ADDED**: Two new justfile recipes `check-contract` and `check-contract-audit`.

## Technical Constraints

- **AST extraction over runtime import**: avoids module-import side effects (`_telemetry.log_invocation`, `claude_agent_sdk` transitive import, `yaml`/`markdown` top-level imports), avoids the ~500ms cold-import penalty across ~33 modules, avoids confused-deputy risk for any future third-party `[project.scripts]` entries, and avoids Python-version coupling to private `_actions`/`_subparsers` API.
- **Pre-commit phase ordering**: Phase 1.55 must run AFTER Phase 1.5 (parity) — the contract lint depends on `[project.scripts]`-resolvability that parity validates. Phase 1.55 must run BEFORE Phase 1.6 (shim presence) and the downstream phases.
- **Scan corpus is explicit, not implicit**: `skills/**/*.md`, `hooks/**`, `justfile`, `docs/**/*.md`, `tests/**`, `CLAUDE.md`, `cortex/requirements/**/*.md`. Hard-excluded: `cortex/research/archive/**`, `CHANGELOG.md`, `bin/.audit-*-allowlist.md`, `bin/.parity-exceptions.md`, `cortex/lifecycle/**`.
- **No new third-party Python dependencies**: stdlib `ast`, `re`, `shlex`, `argparse`, `tomllib` only. `markdown` is already in `pyproject.toml` and may be optionally used for fenced-block extraction if the state-machine port from `prescriptive_prose.py` proves insufficient — fall back is the prescriptive-prose state machine.
- **No event-registry entry**: the lint does not emit lifecycle events at runtime (no `bin/.events-registry.md` row). Pre-commit hook invocation observability lives in the hook's own telemetry surface.
- **Compatibility with the wheel/working-tree distribution**: the lint must function both when invoked via the installed wheel's binstub and via `python3 -m cortex_command.lint.contract` against the working tree. `CORTEX_COMMAND_FORCE_SOURCE=1` honored.
- **Same-PR landing strategy**: lint introduction + drift fixes + exception-ledger seed all land in one PR with hard-block enabled on merge. No warning-mode intermediate state.

## Proposed ADR

None considered. The decisions in this spec (drop unused artifact, AST extraction, narrow v1 scope) are scope decisions rather than architectural-pattern decisions. They do not meet the three-criteria ADR gate (hard to reverse + surprising without context + real trade-off): reversibility is high (any of these decisions can be revised in a follow-up lifecycle without breaking existing callers), and the trade-offs are spec-local rather than project-wide.
