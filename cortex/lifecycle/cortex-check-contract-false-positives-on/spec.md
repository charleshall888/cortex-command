# Specification: cortex-check-contract-false-positives-on

## Problem Statement

`cortex-check-contract` flags 8 false-positive `E101` violations against correct documentation — all `cortex-worktree-create` — because its binary regex (`_BINARY_RE`, `cortex_command/lint/contract.py:451`) matches the token anywhere in a code span with no notion of command position. 7 hits are `.sh`-filename mentions (`…/cortex-worktree-create.sh`); 1 is a `command -v cortex-worktree-create …` PATH probe. The fix makes detection precise — flag a token only when it is a genuine console-script invocation in command position — without regressing real `E101`/`E102` detection, and locks the behavior with regression fixtures.

**Scope honesty (what this fix achieves):** This clears the false positives in the **`--audit`** (whole-repo) view AND fixes the **`--staged`** pre-commit gate's `Path.match("**")` membership drift (R8), so the deep-nested FP files actually reach the commit-time gate. Without R8 the gate is a **false-green**: the bash trigger fires for deep files, `--staged` runs, then the inner `.match` filter silently skips them — dropping 72% of the corpus (129/180 files), including 6 of the 8 FP lines AND the genuine `cortex-worktree-create --feature` invocation at `skills/lifecycle/references/implement.md:119` (so true-positive detection is lost there too). This reproduces on **all supported Python versions (3.12+)** — it is not 3.13-specific. With R8, #279's stated value ("blocks future edits to the affected files") becomes true; today it holds for only 1 of the 8 lines. Two enforcement gaps remain **out of scope as separate follow-up tickets**: (a) CI runs no gates/tests at all (a narrow CI ticket wires `pytest` + `--audit` in source mode); (b) the same `Path.match` drift affects the sibling gates (`check-parity`/`check-prescriptive-prose`/`check-events-registry`) and `bare-python-import`'s `full_match` AttributeErrors on 3.12 (a standalone gate-reliability ticket).

The ticket's second defect (no subcommand-flag modeling, "4 `cortex-discovery` E102s") **does not reproduce on the committed source**: subparser flag resolution already shipped in `5661c467` (2026-05-25). The ticket and the initial triage measured a *stale installed wheel* (`bin/cortex-check-contract` runs the wheel before the working tree). Defect B is therefore scoped as verification + regression-locking, not a build.

## Phases

- **Phase 1: Detection precision + commit-time reach (Defect A)** — gate invocation detection on command position so path-mentions and probe-arguments stop flagging; fix the `--staged` membership drift so deep files reach the commit-time gate; clear the 8 audit false positives; strengthen the `invalid-*` fixture assertion to an exact-set check so true-positive fixtures can catch over-firing; add regression fixtures (false-positive, true-positive incl. the false-negative-trap shapes, and a staged-depth fixture).
- **Phase 2: Subcommand-model verification + documentation** — verify Defect B is clean and lock it with subcommand regression fixtures; confirm no regression to E104/ledger/self-test/two-mode; document the deferred latent items and the stale-wheel verification protocol at their code sites.

## Priority (MoSCoW)

- **Must-have** (the ticket fails without these): R1 (command-position fix), R2 (genuine `E101` preserved, including the positional-only / env-prefix / mixed-span shapes), R3 (false-positive fixtures), R4 (exact-set `invalid-*` assertion — what makes the regression-lock real), R5 (Defect B verified + locked), R6 (no E104/ledger/self-test/two-mode regression), R8 (`--staged` membership reaches deep files — load-bearing for the value claim, coupled to R1).
- **Should-have, low-cost** (documentation, not behavior): R7 (deferred-items + stale-wheel notes at code sites).
- **Won't-do** (this ticket): see Non-Requirements — shell-lexing, ledger suppression, the parent-flag-loss fix (dropped after critical-review showed it is not cheap/safe), latent bugs (b)/(c)/(d), the sibling-gate `Path.match` drift + `bare-python-import` 3.12 portability (separate ticket), and CI wiring (separate ticket).

## Requirements

1. **Command-position guard for invocation detection.** Detection treats a `cortex-*` token as an invocation only when it is in command position within a code span, applying a corrected rule set, evaluated **per regex match** against the immediate left-context (not span-level — a single span may contain both a path-mention and a real run):
   - **Path-component / extension rejection keyed on the TOKEN, not on what follows.** Reject a match when the token itself is part of a filesystem path — it has a path prefix (immediately preceded by `/` or a directory-segment char) or an extension suffix (immediately followed by `.` + an alpha run, e.g. `.sh`/`.py`/`.json`). Do NOT key this on "is the tail a flag" — a command-position token followed only by a bare positional that resembles a path (`cortex-worktree-create feature/foo`) is still an invocation and must still flag a missing required flag.
   - **Command-position acceptance** at line/span start or immediately after a shell separator (`|`, `&&`, `||`, `;`, `$(`, backtick, `(`, `{`, `then`/`do`/`else`), after stripping a leading `$ `/`% ` prompt — evaluated on the immediate left-context of *that* match.
   - **Probe-argument rejection** when the match's command head is a non-executing probe — including but not limited to `command -v`/`-V`, `which`, `type`, `hash` — while keeping bare `command <tok>` (which executes the token). (Additional operand-flag probe variants are recorded as a deferred item per R7, not exhaustively handled here.)
   - **`=`-RHS bare-value rejection** only when the token is the *immediate* RHS of `=` (`VAR=cortex-…` with no intervening space); an env-var prefix (`FOO=1 cortex-worktree-create --base-branch main`, token after a space) is a real run and must NOT be rejected. The `=$(`/backtick command-substitution form is a real run and must NOT be rejected.

   The guard runs inside the scanner (`_scan_file_for_invocations`/`_emit_fenced`/the inline-span loop) before an `Invocation` is constructed (the raw left-context is in scope only there), and the inline bare-no-tail skip is made symmetric with fenced blocks. Acceptance: `CORTEX_COMMAND_FORCE_SOURCE=1 cortex-check-contract --audit --root "$(git rev-parse --show-toplevel)"` exits 0 with empty violations (the 8 `E101` cleared); pass if exit code = 0. (`FORCE_SOURCE` + the bin wrapper runs the working tree and resolves cleanly to the 8 E101 with no E104 — verified; do NOT use bare `python3 -m`, which surfaces 2 unrelated venv-resolution E104s.) **Phase**: Detection precision + commit-time reach (Defect A)

2. **Genuine `E101` detection preserved, including the false-negative-trap shapes.** A real `cortex-worktree-create` missing `--feature` still flags `E101` across: bare/positional tail (`cortex-worktree-create main`, `cortex-worktree-create feature/foo`), piped (`… | cortex-worktree-create …`), `&&`-separated, `$(…)`-substitution head, env-var-prefixed (`FOO=1 cortex-worktree-create --base-branch main`), and `bin/cortex-worktree-create --base-branch main` (path-prefixed *real run* — flags follow). Acceptance: fixtures `tests/fixtures/contract/invalid-missing-feature-baseline/`, `invalid-missing-feature-positional/`, `invalid-missing-feature-piped/`, `invalid-missing-feature-envprefix/`, and `invalid-missing-feature-bin-path/` each exit 1 with exact-set `["E101"]`; pass if `pytest tests/test_check_contract.py -q` exits 0. **Phase**: Detection precision + commit-time reach (Defect A)

3. **False-positive shapes locked as `valid-*` fixtures.** New mini-repo fixtures cover each cleared shape: `.sh` mention inline, `.sh` mention in a markdown table, `.sh` mention fenced, a `command -v` probe, `which`/`type`/`hash` probes, and a **mixed span** (one span containing both a `.sh` path-mention AND a separate real invocation, asserting only the real one's outcome). Acceptance: fixtures `valid-hook-filename-inline/`, `valid-hook-filename-table/`, `valid-hook-filename-fenced/`, `valid-command-v-probe/`, `valid-which-type-hash-probe/` each exit 0 with empty violations, AND `invalid-mixed-span/` (path-mention + a real missing-`--feature` run in one span) exits 1 with exact-set `["E101"]`; pass if `pytest tests/test_check_contract.py -q` exits 0. **Phase**: Detection precision + commit-time reach (Defect A)

4. **Exact-set `invalid-*` assertion.** The harness no longer uses `any(code in expected_codes …)` (`tests/test_check_contract.py:112`); it asserts the multiset of actual violation codes equals the expected set, so a fixture that should emit only `E101` fails if it also spuriously emits `E102` (catches over-firing). Acceptance: observable state — `grep -c "any(code in expected_codes" tests/test_check_contract.py` = 0, AND the harness asserts `sorted(actual_codes) == sorted(expected_codes)` (or equivalent multiset equality); pass if both hold and `pytest tests/test_check_contract.py -q` exits 0. **Phase**: Detection precision + commit-time reach (Defect A)

5. **Defect B verified and regression-locked.** Confirm subparser flag resolution is clean on the committed source and lock it: a valid subcommand-flag invocation passes, and a genuinely unknown flag on a known subcommand still flags `E102`. Acceptance: fixtures `valid-subcommand-flag/` (exit 0, empty) and `invalid-unknown-subcommand-flag/` (exit 1, exact-set `["E102"]`); pass if `pytest tests/test_check_contract.py -q` exits 0. **Phase**: Subcommand-model verification + documentation

6. **No regression to E104 / ledger / self-test / two-mode.** The E104 `extraction_status != "ok"` early-continue and the exception-ledger match-key shape `(binary, flag, path)` are untouched; `--self-test` stays green; `--staged` and `--audit` continue to share the same `validate()` path. Acceptance (all run against the working tree): `pytest tests/test_check_contract.py -q` exits 0 (authoritative, wheel-immune) AND `CORTEX_COMMAND_FORCE_SOURCE=1 cortex-check-contract --self-test` exits 0 AND `CORTEX_COMMAND_FORCE_SOURCE=1 cortex-check-contract --validate-exceptions` exits 0. (Note: `--self-test` exercises inline synthetic fixtures only, not the real corpus — it is supplementary; the corpus-fix verification is R1's `--audit` run plus the R2/R3 fixtures.) **Phase**: Subcommand-model verification + documentation

7. **Deferred latent items documented at their code sites; stale-wheel protocol recorded.** Each deferred latent item is marked as a `# DEFERRED (#279):` comment at its relevant code site in `cortex_command/lint/contract.py`: (i) unknown-subcommand fall-through, (ii) `allow_abbrev` prefix matching — recorded as **exact-match is the intended contract** (modeling abbreviation would reduce `E102` sensitivity; not a bug to fix), (iii) helper-injected-flag AST blindness (`_add_repo_root_arg`), and (iv) parent-flag-loss (top-level flags dropped when a subcommand is matched — needs argparse flag-ordering modeling). A comment at the scan-mode/`main` site records the stale-wheel verification protocol (verify against the working tree via `CORTEX_COMMAND_FORCE_SOURCE=1` or a reinstall; `pytest tests/test_check_contract.py` is the authoritative wheel-immune signal). Acceptance: observable state — `grep -c "DEFERRED (#279)" cortex_command/lint/contract.py` ≥ 4 AND `grep -c "CORTEX_COMMAND_FORCE_SOURCE" cortex_command/lint/contract.py` ≥ 1 (grep targets `contract.py` only, never the lifecycle dir, so the spec/research files cannot vacuously satisfy it). **Phase**: Subcommand-model verification + documentation

8. **`--staged` membership congruent with `--audit` (commit-time gate reaches deep files).** `_scan_staged`'s corpus-membership test no longer uses `rel_path.match(glob)` — which treats `**` as exactly one path segment on all supported Pythons, so it skips depth-1 and depth-≥3 in-scope files (72% of the corpus, including 6 of the 8 FP lines and the genuine `implement.md:119` invocation). It uses a recursive-glob match congruent with `scan_corpus`'s `root.glob`. The replacement must be **Python-3.12-safe**: use `fnmatch.fnmatch(rel, glob)` (works on all supported versions), NOT bare `PurePath.full_match` (added in 3.13, while `requires-python` is `>=3.12`). Acceptance: (a) `pytest tests/test_check_contract.py -q` exits 0, including a new test asserting a depth-≥3 in-scope path (e.g. `skills/lifecycle/references/implement.md` against `skills/**/*.md`) is a member of the `--staged` scan set and that a staged depth-3 file carrying a real violation is flagged; (b) Python-3.12 safety — `grep -c "full_match" cortex_command/lint/contract.py` = 0. **Phase**: Detection precision + commit-time reach (Defect A)

## Non-Requirements

- **No shell-lexing.** The guard stays a heuristic regex/string predicate; it does NOT adopt `shlex`/`bashlex`/`tree-sitter-bash` (rejected approach A2 — new dependency, breaks on markdown placeholders, risks false negatives on un-parseable prose).
- **No exception-ledger suppression of the FPs.** The 8 false positives are fixed by correcting detection, not by adding `bin/.contract-lint-exceptions.md` entries (the anti-pattern the ticket names).
- **The parent-flag-loss bug is NOT fixed** — only documented (R7 item iv). Critical-review showed the naive flag-union "fix" would mask an argparse-illegal post-subcommand placement (a new false negative), and a correct fix requires positional/flag-ordering modeling the static checker lacks. Deferred to a dedicated follow-up.
- **Latent bugs (b) unknown-subcommand fall-through, (c) `allow_abbrev` prefix matching, (d) helper-injected-flag AST blindness are NOT fixed** — only documented (R7). (b) would *reduce* `E102` sensitivity; (c) is by-design (exact-match is the intended contract); (d) is dormant.
- **The sibling gates' identical `Path.match` drift is NOT fixed here** — `check-parity`, `check-prescriptive-prose`, and `check-events-registry` share the same under-scan, and `bare-python-import` already migrated to `full_match` (which AttributeErrors on Python 3.12). These are a **separate gate-reliability follow-up ticket**; this ticket fixes only `cortex-check-contract`'s `_scan_staged` (R8).
- **CI wiring is NOT in this ticket** — wiring `cortex-check-contract --audit` + `pytest tests/test_check_contract.py` into `.github/workflows/validate.yml` is a **separate follow-up ticket** (references #280's "Linux-only non-blocking signal" reopen-condition).
- No change to the `bin/cortex-check-contract` bash wrapper or its `plugins/cortex-core/bin/` mirror (logic change is wheel-only in `contract.py`).

## Edge Cases

- **`worktree_path=$(cortex-worktree-create --feature … --base-branch main)`** (`implement.md:119`): real invocation; `=` precedes `$(` — must remain detected (the `=$(` substitution form is NOT an `=`-RHS bare value). Would flag if `--feature` were missing. (This line lives at depth 3, so R8 is what lets `--staged` see it at all.)
- **`FOO=1 cortex-worktree-create --base-branch main`** (env-var prefix): real run — the token follows a space and is in command position; the `=` belongs to the env assignment, not the token. Must still flag missing `--feature`. (This is the false-negative trap the `=`-RHS rule must avoid.)
- **`cortex-worktree-create feature/foo`** (positional-only tail that resembles a path): real run missing `--feature` → must flag `E101`. The path-component rejection must key on the *token* (`cortex-worktree-create` is not itself a path component here), not on the tail looking path-like.
- **`bin/cortex-worktree-create.sh` vs `bin/cortex-worktree-create --base-branch main`**: the first is a path-mention (token has a `/` prefix and `.sh` suffix) → not flagged; the second is a path-prefixed real run (flags follow, no extension) → flagged.
- **Mixed span**: a single inline/fenced span containing both `…/cortex-worktree-create.sh` (path-mention) and a separate real `cortex-worktree-create …` after a separator — the per-match predicate must classify each match independently; the path-mention is dropped and the real run is still evaluated.
- **`command cortex-worktree-create …`** (bare `command`, no `-v`/`-V`): executes the token → still an invocation; only the probe forms are rejected.
- **Depth-1 staged file** (e.g. `docs/agentic-layer.md`): R8 must scan it too — `Path.match("docs/**/*.md")` is False for depth-1 (zero mid-dirs) as well as depth-≥3, so a correct recursive matcher must accept depth-1, depth-2, and deeper uniformly (congruent with `root.glob`).
- **Orphaned ledger row**: if the fix drops a mention previously covered by an existing ledger entry, that entry becomes dead; re-check `bin/.contract-lint-exceptions.md` after the fix (no test detects orphans).

## Changes to Existing Behavior

- MODIFIED: invocation detection in `_scan_file_for_invocations` / `_emit_fenced` / the inline-span loop now applies a per-match command-position predicate before constructing an `Invocation` → `.sh`/path mentions and `command -v`/`which`/`type`/`hash` operands are no longer treated as invocations.
- MODIFIED: the inline bare-name-no-tail skip (`contract.py:657`) is made symmetric with fenced blocks.
- MODIFIED: `_scan_staged` corpus-membership changes from `rel_path.match(glob)` (non-recursive `**`) to a recursive `fnmatch`-based match congruent with `--audit` → deep-nested files now reach the commit-time gate. (Behavioral effect: editing a deep in-scope file now correctly triggers the gate; before this, the gate silently passed.)
- MODIFIED: `tests/test_check_contract.py` `invalid-*` assertion changes from `any`-membership to exact (multiset) equality.
- ADDED: new `tests/fixtures/contract/` mini-repos (valid-* false-positive shapes incl. mixed-span; invalid-* true-positive shapes incl. positional-only, env-prefix, mixed-span; subcommand FP/TP) and a staged-depth membership test.
- ADDED: `# DEFERRED (#279):` comments at code sites for the four deferred latent items, and a stale-wheel verification-protocol comment.

## Technical Constraints

- The command-position guard must execute where the raw left-context string is in scope (inside the scanner), before `Invocation` construction, and must be evaluated **per regex match** (immediate left-context of each match), not once per span — a span may hold both a path-mention and a real run.
- Must NOT disturb the E104 `extraction_status != "ok"` early-`continue` (`contract.py:1227-1242`) or the ledger match-key `(binary, flag, path)` (the flag token only, never a `sub.flag` composite).
- The path-mention discriminator must be general across extensions and path shapes and keyed on the token's own path-membership; specifically NOT a `.sh`/`.py`-only allowlist, NOT a bare leading-`/` rejection (which would break genuine `bin/cortex-* --flag` runs), and NOT a "is the tail a flag" test (which would drop positional-only missing-flag runs).
- **R8 portability**: the `--staged` membership replacement must run on Python 3.12 (`requires-python >=3.12`). `PurePath.full_match` is 3.13+; use `fnmatch.fnmatch` (or another 3.12-safe recursive matcher). For these anchored `dir/**/*.md` globs `fnmatch` yields the same set as `root.glob` for the corpus shapes. Do not introduce a 3.13 floor.
- Logic lives only in `cortex_command/lint/contract.py` (wheel-only; not mirrored into `plugins/`); `just check-contract` / parity recipes and the two-mode `--staged`/`--audit` split must be preserved.
- **Verification against the working tree, not the installed wheel**: `bin/cortex-check-contract` runs the wheel before the working-tree fallback, so every acceptance command pins `CORTEX_COMMAND_FORCE_SOURCE=1` and `--root "$(git rev-parse --show-toplevel)"` (cwd-independent), and `pytest tests/test_check_contract.py` (which runs `python3 -m` against the repo and is wheel-immune) is the authoritative acceptance signal. Do NOT verify via the bare installed binary (stale wheel) or bare `python3 -m` (surfaces unrelated venv-resolution E104s).
