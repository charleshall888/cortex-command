## Codebase Analysis

**Primary file (only logic file):** `cortex_command/lint/contract.py` (the `cortex-check-contract` console script; `bin/cortex-check-contract` is a thin bash wrapper that dispatches into it).

## Web Research

No widely-used tool does exactly this (auto-detect a console-script token *in command position within prose*, then validate flags against the matched subparser). Established prior art **reduces** the detection problem rather than solving it head-on:
- **markdown-clitest** (tag blocks with `<!-- doctest -->`), **md-cli-test** (restrict to fenced code blocks), **sphinx-argparse** (generate docs *from* the parser — the inverse "can't drift" architecture). Takeaway: "restrict to fenced/inline code spans first, then detect command position within them" is the prior-art-blessed way to cut false positives — which this checker already half-does (it only scans code spans, not bare prose).
- **Shell-aware parsing for command position:** `shlex` (stdlib, tokenizer only — no grammar, can't structurally tell command from argument); `bashlex`/`tree-sitter-bash` (real AST: first WordNode / `command_name` node = command position — structurally distinguishes `command -v X`, `which X`, and `*.sh` paths from a real invocation). **But** a third-party parser is a new dep, hard-fails/silently-drops on un-parseable prose (placeholders `{slug}`/`<name>`, redirections), and over-strict parsing is a confirmed **false-negative** generator (unparsed line → silently unchecked invocation).
- **Anti-patterns (confirmed by real bug reports):** substring/whole-word matching false-positives (oxc #19336 — `alt="helloImage"` matched "image"; fixed via token-boundary matching); matching without lexical context (ansible-lint #470). Direct transfer: a console-script name found by substring false-positives inside `path/cortex-foo.sh` and `cortex-foobar`.
- **argparse subparser introspection** (only relevant if ever switching off AST — not recommended): scan `parser._actions` for the `_SubParsersAction`, read `.choices` (name→subparser, **aliases share a parser**, deprecated subparsers still present), iterate each `_actions[].option_strings`. Pitfalls: `--flag=value` split, `allow_abbrev` prefix matching, parent-vs-subparser scope, nested subparsers. These attrs are private (`_`-prefixed) and not version-stable.

## Tradeoffs & Alternatives

### Defect A — command-position detection
- **A1 — regex/heuristic command-position guard at the match site (RECOMMENDED).** Gate each `_BINARY_RE` match before constructing an `Invocation`. Low complexity (~20-40 lines, no new dep), aligns with the module's existing heuristic regex+state-machine architecture (the module docstring itself calls its parsing "necessarily heuristic"). Maintainability moderate (small hand-maintained separator/probe-verb sets). **See "Command-Position Detection Design" + "Adversarial Review" for the corrected rule set — the naive form proposed in the ticket has holes.**
- **A2 — full shell lexing (shlex/bashlex/tree-sitter).** High complexity, new dependency, breaks on markdown placeholders, and risks false negatives on un-parseable lines. Contradicts the module's zero-dep heuristic posture. **Rejected** as over-engineered for an 8-instance, two-shape defect (no named future requirement demands true shell-command-word resolution).
- **A3 — suppression via exception ledger.** Trivial but the **anti-pattern the ticket names**: conflates "not an invocation" with "documented exemption", scales linearly with every new `.sh`/probe mention, and is position-blind (can't distinguish a real invocation from a path-mention in the same file). Acceptable only as a stopgap if A1 is deferred. **Note (adversarial):** the ledger *does* offer a one-line escape hatch today — so the code fix earns its place over the ledger only if maintainers want path-mentions to **categorically never** be invocations (a corpus-wide property). The spec should state that intent explicitly.

### Defect B — subcommand flag modeling
- **B1 — AST introspection descending into subparsers (RECOMMENDED == already shipped).** Single source of truth derived from argparse source; same `find_spec`+`ast.walk` pattern as the rest of the module. Residual work: regression-lock fixtures + (optionally) latent-bug fixes.
- **B2 — hand-maintained command→subcommand→flags map.** Second source of truth that drifts; makes the checker itself a drift source. **Rejected.**
- **B3 — parse `--help` output.** Reverses the no-import/no-execute design, brittle, slowest. **Rejected.**

## Command-Position Detection Design

**Current logic:** no command-position notion; `_BINARY_RE.finditer` over fenced lines (`:567-580`) and inline spans (`:647-667`); E101 emitted in `validate()` (`:1294-1311`) regardless of position.

**The 8 live false positives (all `cortex-worktree-create`):** 7 `.sh`-filename mentions (`docs/agentic-layer.md:160,210`; `docs/internals/sdk.md:142,157`; `skills/lifecycle/references/implement.md:234`; `skills/lifecycle/references/parallel-execution.md:14,26`) + 1 `command -v` probe (`implement.md:58`). `cortex-worktree-create` is the **only** console-script whose name also exists as a `.sh` hook stem.

**True-positives that MUST keep working** (verified): `worktree_path=$(cortex-worktree-create --feature … --base-branch main)` (`implement.md:119`, clean); piped `… | cortex-critical-review check-synth-stable …` (`verification-gates.md:74`); the canonical negative `cortex-worktree-create --base-branch main` (no `--feature`) → must still E101.

**Recommended rule set (CORRECTED per adversarial review — supersedes the ticket's naive form):** evaluate a command-position predicate against the code-span text at each match `start..end`, *inside* `_emit_fenced`/the inline loop (the raw left-context is only in scope there; the `Invocation` dataclass carries no left-context), before constructing the `Invocation`:

1. **Filename/path-tail rejection (general, not `.sh`/`.py`-only):** reject when the matched token is immediately followed by `.` + an alpha extension run (`.sh`, `.py`, `.json`, `.toml`, …). The `.sh`/`.py` allowlist is too narrow.
2. **Path-mention rejection — NOT a bare "preceded by `/`" rule.** A bare `/`-prefix guard is **over-broad**: it would silently drop genuine relative-path runs like `bin/cortex-lifecycle-state --feature …` (real, in corpus at `tests/fixtures/cortex-lifecycle-state/README.md:146`) and the many `bin/cortex-* --flag` justfile-style invocations. The correct discriminator is: **token is the tail of a path AND is not followed by argv tokens** (extension / `)` / `,` / `.` / EOL / another `/` with no following flags) ⇒ mention; a path-prefixed token **followed by flags/subcommand** ⇒ real run, keep it.
3. **Command-position acceptance:** accept only when the token is at line/span start or immediately after a command separator (`|`, `&&`, `||`, `;`, `$(`, backtick, `(`, `{`, `then`/`do`/`else`), after stripping a leading `$ `/`% ` prompt. Operationally: split preceding text on separators; token is in command position iff the last segment is blank up to `start`. **Required so piped/`&&`/substitution real invocations still flag.**
4. **Probe-argument rejection:** reject when the segment head is `command -v`/`command -V`/`which`/`type`/`hash` (token is the probe's operand). **Keep** bare `command <tok>` (no `-v`/`-V`) — it executes the token. (This branch is currently untestable against real corpus data; treat as speculative-but-cheap.)
5. **`=`-RHS bare value rejection — MUST exempt `$(`/backtick.** `worktree_path=$(cortex-worktree-create …)` has `=` immediately before `$(`; the guard must reject only `=` + bare-word, never `=$(`/`` =` `` (command substitution). Multi-char lookbehind hazard — needs an explicit fixture for the `=$(...)` shape, not just the positive case.

Also: make the **bare-no-tail skip symmetric** across fenced and inline (currently inline-only at `:657`).

## Subcommand / argparse-Model Design

**Already works** (verified): `validate()` (`:1256-1291`) resolves the matched subcommand and validates against it; the extractor populates per-subcommand flags. `cortex-discovery emit-research-sizing --topic … --complexity … --criticality …` validates clean; an unknown flag on a known subcommand correctly fires E102.

**E104/ledger seam (must not change):** E104 logic is upstream of and independent from subcommand resolution (early `continue` at `:1227-1242`); match-key must stay `(binary, flag, path)` with the flag token only.

**`cortex-discovery` structure** (`discovery.py:1203-1360`): top-level has zero `--flags` and a `required=True` subparser (`dest="command"`); subcommands `emit-research-sizing` (req `--topic/--complexity/--criticality`), `read-research-sizing` (req `--topic`), plus `resolve-events-log-path`, `emit-checkpoint-response`, `generate-brief`, `score-corpus`.

**Verified latent gaps (candidate scope — see adversarial dispositions):**
- (a) **parent-flag-loss**: matching a subcommand switches `resolved_surface` wholesale to the subparser (`:1265`), discarding parent/top-level optional flags → false E102. Reproduced: `cortex-critical-review check-synth-stable --lifecycle-root /tmp`. One-line fix = union `parent.flags | subparser.flags`. A real binary already has the at-risk shape; dormant in the corpus today.
- (b) **unknown-subcommand fall-through**: an unrecognized subcommand token leaves `resolved_surface` at top-level → false E102/E103 on trailing flags.
- (c) **`allow_abbrev` prefix matching not modeled**: `--top` for `--topic` → false E102+E101.
- (d) **helper-injected flags invisible to the AST walker**: `--repo-root` (added via `_add_repo_root_arg(p)`) is absent from every discovery subcommand surface; a future doc using `--repo-root` would false-positive.

**Other console-scripts with subparsers the model must keep handling** (blast-radius): `cortex-critical-review` (4 subs), `cortex-discovery` (6), `cortex-interactive-lock` (4), `cortex-lifecycle-event` (1), `cortex-refine` (1). (`cortex` umbrella and `install_guard` are out — not `cortex-`-prefixed console scripts.)

## Adversarial Review

- **Command-position rule holes (folded into the corrected rule set above):** rule (2) "preceded by `/`" is over-broad (kills genuine `bin/cortex-foo --flag`); the `.sh`/`.py` allowlist misses other extensions; rule (5) collides with `=$(…)`; rule (4) must read raw text *left* of the match (not `tail_tokens`) and must run before `Invocation` construction. New false-negative risk: a guard that requires command-position could drop a real invocation written mid-prose — fixtures must cover piped / `&&` / substitution / `bin/`-path-with-flags positives.
- **Highest-risk hazard — stale wheel vs working tree:** `bin/cortex-check-contract` runs the **installed wheel before the working-tree fallback**. So `just check-contract` reflects the implementer's `contract.py` edits **only if wheel == source**. If the overnight runner's installed wheel lags the working tree, manual `just check-contract` runs stale code and the implementer may thrash. **Spec MUST mandate:** (1) verify wheel/source identity (or use `CORTEX_COMMAND_FORCE_SOURCE=1` / reinstall) for all manual verification; (2) treat `pytest tests/test_check_contract.py` (wheel-immune) as the authoritative acceptance signal; (3) record installed-wheel vs working-tree state at task start.
- **Staged/audit `**` asymmetry (Python ≥3.13):** `scan_corpus` uses `root.glob(...)` (recursive) but `_scan_staged` uses `Path(rel).match("skills/**/*.md")`, which returns **False** for deep-nested files on 3.13+. Verified: of the 4 FP-bearing files, **only `docs/internals/sdk.md` is reachable by the staged pre-commit gate** — the other three are audit-only. So the ticket's "these FPs block the gate" premise is mostly false in the staged path. This is itself a latent bug (and Python-version-dependent — runner interpreter unverified). Spec should decide whether to fix the staged/audit membership drift.
- **E104/ledger/placeholder/two-mode:** a scanner-side guard does not touch E104 (upstream early-continue) or ledger ordering, but dropping a mention that an existing ledger row covered would **orphan that row** (no test detects orphans) — re-check ledger entries post-fix. Tokenizer splits a shell word at placeholder boundaries (`interactive-{slug}` → `['interactive-','{slug}']`) — harmless for the 8 FPs; note for awareness.
- **Regression-detection gap:** the `invalid-*` assertion is `any(code in expected)` (`test_check_contract.py:112`) — it **cannot detect over-firing** (a fixture that should emit only E101 still passes if it also spuriously emits E102). Mitigation: assert **exact** violation sets (count+codes) for the true-positive fixtures (likely needs harness extension — in-scope test-infra work). `valid-*` fixtures (assert empty) are the strong guard for new FPs. **Highest-value guard given no-CI:** a test that runs `--audit` against the real repo root and asserts the expected violation set. Strongly consider wiring `just check-contract --audit` + `pytest tests/test_check_contract.py` into `validate.yml` (the durable "solution-horizon" fix — without CI the FP fix will rot).
- **Scope dispositions for latent bugs:** (a) parent-flag-loss = defensible **in-scope** (real at-risk binary exists, one-line union fix, cheap insurance) — require a `valid-parent-flag-after-subcommand` fixture. (b) unknown-subcommand, (c) allow_abbrev, (d) helper-injected flags = **DEFER** (no corpus case; (b)/(c) would *reduce* E102 sensitivity, opposing the ticket's precision goal). Record (d) in the ticket so a future `--repo-root` doc isn't mysteriously flagged.
