# Plan: cortex-check-contract-false-positives-on

## Overview
Make `cortex-check-contract` precise by adding a per-match command-position predicate to the prose scanner (so `.sh`/path-mentions, `command -v`/`which`/`type`/`hash` probe-operands, and `=`-RHS bare values stop being treated as invocations) while preserving genuine `E101`/`E102` detection, and fix the `--staged` membership drift so deep-nested files actually reach the commit-time gate. All logic lives in `cortex_command/lint/contract.py` (wheel-only — not mirrored to `plugins/`); the rest is regression-locking fixtures, a stronger exact-set test assertion, and documentation of deferred latent items. Defect B (subcommand-flag modeling) already shipped in `5661c467`, so it is verify-and-lock, not a build.

### Task 1: Command-position guard for invocation detection (R1)
- **Files**: `cortex_command/lint/contract.py`
- **What**: Add a per-match command-position predicate and apply it inside `_emit_fenced` and the inline-span loop **before** each `Invocation` is constructed, so path-mentions / probe-operands / `=`-RHS bare values stop flagging while genuine command-position runs still flag. Also make the inline bare-name-no-tail skip symmetric with fenced blocks.
- **Depends on**: none
- **Complexity**: complex
- **Context**: `_BINARY_RE` at `contract.py:451`. Two call sites: `_emit_fenced` (`:565-580`) where the left-context is `combined[:m.start()]` and the raw tail is `combined[m.end():]`; the inline loop (`:647-667`, bare-no-tail skip at `:657`) where the left-context is `span_content[:m_bin.start()]` and the tail is `span_content[m_bin.end():]`. The predicate must be evaluated **per regex match** against the immediate left-context (a single span may hold both a path-mention and a real run). Corrected rule set per spec R1 and research §"Command-Position Detection Design":
   - **(1) Path-mention rejection — keyed on the TOKEN, and (for the path-prefix case) conditioned on the tail.** Reject when the token is immediately followed by `.`+alpha-run (`.sh`/`.py`/`.json`… — an extension suffix). For the path-*prefix* case (token immediately preceded by `/` or a directory-segment char): reject **only when the token is NOT followed by argv tokens** (flags or a subcommand) — i.e. it is the bare tail of a path. A path-prefixed token **followed by flags is a real run and must be KEPT** (`bin/cortex-worktree-create --base-branch main` → flag). This is the explicit correction the spec Technical Constraints and research rule 2 demand: a bare leading-`/` rejection is forbidden because it drops genuine `bin/cortex-* --flag` runs (and is exactly the contradiction with Task 7's `invalid-missing-feature-bin-path` fixture). Do NOT key the rejection on "is the tail a flag" for the *positional* case — `cortex-worktree-create feature/foo` (token not itself path-prefixed; path-like positional tail) is still a real run that must flag.
   - **(2) Command-position acceptance** at line/span start or immediately after a shell separator (`|`, `&&`, `||`, `;`, `$(`, backtick, `(`, `{`, `then`/`do`/`else`), after stripping a leading `$ `/`% ` prompt.
   - **(3) Probe-operand rejection** when the segment head is `command -v`/`-V`/`which`/`type`/`hash` (keep bare `command <tok>`, which executes).
   - **(4) `=`-RHS bare-value rejection** only when the token is the immediate RHS of `=` with no intervening space — exempt `=$(`/`` =` `` substitution and the space-separated env-prefix form `FOO=1 cortex-… `.
   - **Rule precedence (load-bearing): rejection rules (1)/(3)/(4) take precedence over command-position acceptance (2).** Two of the eight FPs (`cortex-worktree-create.sh` at `agentic-layer.md:210` and `sdk.md:157`) sit at span start (rule 2 would accept) yet carry a `.sh` extension (rule 1 rejects) — extension-rejection must win, or those 2 FPs stay uncleared.

   Do NOT touch `validate()`/E101 emit (`:1305-1311`), the E104 early-`continue` (`:1227-1242`), or the ledger match-key `(binary, flag, path)`. Heuristic regex/string predicate only — no `shlex`/`bashlex`/`tree-sitter`. Wheel-only; no `plugins/` mirror obligation.
- **Verification**: run the canonical working-tree `AUDIT` command from the Verification-channel note — `PYTHONPATH="$(git rev-parse --show-toplevel)" .venv/bin/python -m cortex_command.lint.contract --audit --root "$(git rev-parse --show-toplevel)"` — and assert **`grep -c E101` of the output = 0** (the 8 `E101` cleared; down from 8 at baseline). Via `.venv` the run is clean so it also exits 0; do NOT gate on exit code alone (a `yaml`-less interpreter would add 2 out-of-scope `E104`). `.venv/bin/pytest tests/test_check_contract.py -q` exits 0 is the authoritative wheel-immune corroborating signal.
- **Status**: [x] done

## Risks
- **R8 matcher subtlety (highest)**: the bug is not just depth — it is glob *shape* × version-dependent `**` semantics. A naive `fnmatch.fnmatch(rel, glob)` swap misses depth-1 `.md` files (verified: `docs/agentic-layer.md` vs `docs/**/*.md` → `False`); and `root.glob` is itself version-dependent for the extensionless `hooks/**` (files on 3.13+, directories-only on the 3.12 floor), so the matcher must NOT be pinned to `root.glob` (doing so would regress depth-1 hook coverage the current code provides). Task 3 specifies the membership set directly (`**`=zero-or-more-segments) and its test must cover depth-1 `.md`, depth-≥3 `.md`, a `hooks/**` file, and an exact-name — not just the depth-3 `.md` case the spec's R8 acceptance names.
- **Verification-channel / E104 contamination (load-bearing)**: `cortex-check-contract` on `PATH` is the uv-tool console-script — it runs the **installed wheel and ignores `CORTEX_COMMAND_FORCE_SOURCE`**, so it is blind to working-tree edits; only the repo-relative `./bin/cortex-check-contract` (or `python3 -m`) exercises the working tree. On the working-tree path under a `yaml`-less interpreter, `find_spec` raises for `cortex-auth`/`cortex-report` and the audit emits 2 pre-existing `E104` (out of scope — Task 1 does not touch the E104 path). Therefore the audit acceptance asserts **zero `E101`**, never "exit 0 / empty array" (the exit code is E104-contaminated), and `.venv/bin/pytest tests/test_check_contract.py` (wheel-immune, per-fixture, never hits cortex-auth/cortex-report) is the authoritative signal. The spec's R1 phrasing ("resolves cleanly … no E104 — verified") holds only in a yaml-equipped interpreter; this plan supersedes it with the environment-robust zero-`E101` criterion.
- **Exact-set assertion surfacing existing over-fire**: tightening the `invalid-*` assertion (Task 2) could fail an existing fixture if it emits more codes than its `expected.json` lists. Task 2 includes updating those `expected.json` files; the three current fixtures are single-invocation and verified to emit exactly one code each (`E101`/`E104`/`E102`).
- **Heuristic predicate edge cases**: the command-position guard is a heuristic (no shell lexing, by design). The corrected rule set closes the known holes from adversarial review, but novel prose shapes could surface new edges; the paired valid/invalid fixtures are the regression guard. New false-negative risk (a real invocation written mid-prose being dropped) is covered by the piped / `&&` / `$()` / env-prefix / `bin/`-with-flags true-positive fixtures.
- **Scope discipline**: parent-flag-loss and latent bugs (b)/(c)/(d), the sibling-gate `Path.match` drift, and CI wiring are explicitly out of scope (separate tickets #281–#283 already filed) — documented, not fixed, per the spec's Non-Requirements.
