# Specification: requirements-skill-v2

## Problem Statement

The cortex-command requirements system (v1, shipped via epic #009 in 2026-04) underperforms in measurable ways uncovered by research: only 1 of 7 consumer skills uses the parent doc's structured tag-based Conditional Loading protocol (the rest fall back to heuristic name-matching); the drift-detection mechanism from ticket #013 fires reliably (88.8%) but 8/26 detected-drift artifacts breach the "include Suggested Requirements Update" protocol with no enforcement; the parent `project.md` consumes ~1,785 always-loaded tokens with no explicit prunable surface; and the `/requirements` skill is 348 lines of procedural narration that violates cortex's own *"prescribe What and Why, not How"* principle. This v2 lifecycle rebuilds the three layers in five sequenced phases: consumer-loading consistency, drift-mechanism tightening, parent-doc trim with progressive disclosure, area-doc accuracy audit, and a skill rewrite split into composable `gather` + `write` sub-skills. Each phase ships independently, with measurable acceptance, so blast radius stays contained on this trunk-critical infrastructure consumed by lifecycle, refine, discovery, and critical-review.

## Phases

- **Phase 1: Consumer navigability** — Extract the tag-based Conditional Loading protocol from `review.md` into a shared reference and apply it across the six non-review consumers (lifecycle clarify/specify, discovery clarify/research, refine). Backfill `tags:` into the 10 existing lifecycle index.md files that lack the field so the loader does not regress for them. Pure additive change to consumer skill prose; no changes to requirements docs.
- **Phase 2: Drift tightening** — Enforce the `## Suggested Requirements Update` section when state=detected via post-dispatch re-dispatch with max-retry=2 and morning-report breach surfacing. Run a one-shot remediation script against the 8 historical detected-drift-without-suggestion artifacts (no permanent runner.py modification). Add an informational parity-audit script.
- **Phase 3: Parent trim + Optional partition** — Trim `cortex/requirements/project.md` to ≤1,200 tokens (verified via tiktoken) and introduce an `## Optional` section (llms.txt pattern) that consumers under token pressure can explicitly prune.
- **Phase 4: Area-doc audit + patch** — Audit each of the four area docs (`multi-agent.md`, `observability.md`, `pipeline.md`, `remote-access.md`) against current code; patch drift in place rather than rewriting. Resolve the discovery/backlog area-doc question by clarifying `project.md` Project Boundaries (no new area docs created in v2).
- **Phase 5: Skill split + brevity** — Split `/requirements` into `/requirements-gather` (interview) and `/requirements-write` (synthesize); keep `/requirements` as a thin orchestrator. Adopt mattpocock-style patterns (recommend-before-asking, codebase-trumps-interview, lazy artifact creation). Hit the size budget (gather ≤80 lines, write ≤50 lines, `/requirements` ≤30 lines) and verify end-to-end routing.

Requirements R9, R21 are **Should-have** — R9 (parity-audit script) is informational tooling that surfaces drift gaps but does not change runtime behavior; R21 (parity hygiene) is automated by the existing pre-commit dual-source hook and only fails if that hook is broken. **Could-have** requirements were not identified. **Won't-have** boundaries are captured in the Non-Requirements section. ## Non-Requirements

- Does NOT replace ticket #013's drift-detection mechanism. Phase 2 tightens the existing mechanism (enforce Suggested Update section, remediate historical breaches one-shot, add parity audit) but preserves the `requirements_drift: none | detected` field and the auto-apply cascade as the load-bearing primitives.
- Does NOT add a permanent auto-apply backfill pass to `cortex_command/overnight/runner.py`. The historical 8-artifact gap is remediated via the R8 one-shot script; future sessions are protected by R7's post-dispatch enforcement, not a recurring backfill. Solution Horizon compliant — no speculation that future sessions will recreate the historical gap.
- Does NOT rewrite the four area docs from scratch. Phase 4 audits in place and patches drift; total replacement is out of scope.
- Does NOT import the OpenAI Model Spec's Platform/Developer/User/Guideline authority tagging. Cortex's single-user, single-repo scale does not justify the overhead; revisit if multi-user collaboration emerges.
- Does NOT create new area docs (`cortex/requirements/discovery.md`, `cortex/requirements/backlog.md`) in v2. Per R14, discovery/backlog scope is clarified as inline-documented in Project Boundaries. If either subsystem later warrants a dedicated doc, that is a separate ticket.
- Does NOT change the `cortex/requirements/` directory location, naming convention, or YAML frontmatter schema.
- Does NOT modify the Requirements vs CLAUDE.md content separation. Operational content remains in CLAUDE.md; strategic content remains in `cortex/requirements/`. R10's trim is a length reduction within the strategic-content surface, not a content-category re-shuffling.
- Does NOT add a description-triggered semantic-match loader (Cursor's `description:` mode). Tag-based matching is the chosen mechanism; description-mode is left for a future iteration if tag-based proves insufficient.
- Does NOT add a hook or pre-commit gate that fails when parent project.md exceeds the 1,200-token cap. R10's check is acceptance-time, not enforcement-time; cap drift is surfaced by future audits, not blocked at commit.
- Does NOT modify the existing `cortex-update-item`, `cortex-resolve-backlog-item`, or `cortex-lifecycle-state` CLIs.

## Edge Cases

- **Lifecycle index.md without `tags:` field**: R6 backfills all 10 existing affected files in Phase 1, eliminating this case for existing lifecycles. For *new* lifecycles created after Phase 1 ships, the lifecycle skill's backlog-writeback path already inlines `tags:` from the backlog item; R1's fallback only fires for the truly tag-less edge case (lifecycle with no parent backlog and no tags assigned).
- **Empty `tags: []` array**: R1's fallback applies — load `project.md` only, no error. R6 explicitly permits `[]` for lifecycles with `parent_backlog_id: null`.
- **Tag word matches no Conditional Loading phrase**: Per R1, that tag is dropped; remaining tags still match where applicable. If no tags match, `project.md`-only is loaded.
- **Reviewer refuses to add Suggested Update section in re-dispatch (pathological)**: R7's max-retry=2 escape hatch logs `drift_protocol_breach` and proceeds. The morning report surfaces the breach so it is visible, not silent.
- **R8 one-shot script fails on a specific historical review**: The script logs the failure and continues to the next artifact. Acceptance allows ≤1 unfixed (≥7/8 fixed); a single intractable case does not block Phase 2.
- **Parity-audit script (R9) finds drift on a long-archived lifecycle**: The script reports the gap but does not auto-fix; remediation is the operator's call. The report is informational, not blocking.
- **Layer A ships and a consumer has `tags: []`**: Per R1 fallback, loads `project.md` only — same as the v1 heuristic would have loaded for a topically-irrelevant area. R6's backfill minimizes this case for existing lifecycles.
- **Parent doc trim (R10) removes a section another consumer prose references by name**: R10 must preserve sections that are referenced by name in any skill or hook. Audit before trimming; if a section is referenced and cannot be moved to `## Optional` cleanly, exempt it from the trim and document the exemption inline in the spec amendments.
- **`/requirements-gather` is invoked with no existing doc**: R15's lazy creation pattern handles this — no file is written until synthesis runs.
- **End-to-end test (R20) hits a network or model rate limit**: Test mocks the user-interview surface; if any external dispatch is required (e.g., the orchestrator skill calls a real Claude API), the test must mock that dispatch deterministically. The test is hermetic by design.
- **Layer order is changed after spec approval**: A→E→B→D→C is the recommended order but acceptance is independently checkable per phase. If layer order shifts during implementation, the gather/write skills (Phase 5) remain agnostic to parent doc's exact size — they consume the format template, not absolute thresholds.

## Technical Constraints

- **Soft positive-routing only**: Per `CLAUDE.md` MUST-escalation policy, no new MUST/CRITICAL/REQUIRED escalations may be added without an F-row evidence artifact + effort=high attempt. R7's "must produce Suggested Update section" enforcement is structural (post-dispatch validation), not prose MUST — the soft-form prompt is repeated, then the gate re-dispatches. No new MUST language enters the prose surfaces.
- **What/Why, not How**: New skill prose (R15-R17) must describe decision criteria, gates, output shapes, and intent — not step-by-step procedure. Honor cortex's design principle stated in CLAUDE.md `## Design principle: prescribe What and Why, not How`. The brevity targets (≤80, ≤50, ≤30 lines) are the enforcement mechanism.
- **Dual-source mirror enforcement**: Skill changes land in canonical `skills/` paths; the pre-commit hook regenerates `plugins/cortex-core/skills/` mirrors. New skills (`requirements-gather`, `requirements-write`) trigger fresh mirror entries.
- **SKILL.md size budget (≤500 lines per `cortex/requirements/project.md:32`)**: All new SKILL.md files are well under this cap. No size-budget-exception markers required.
- **Bin-script parity gate**: R9's new bin script (`cortex-requirements-parity-audit`) must register an in-scope reference per `bin/cortex-check-parity` (SKILL.md, requirements, docs, hooks, justfile, or tests). The `just requirements-parity-audit` recipe satisfies this; no allowlist exception needed.
- **Events-registry**: R7's new `drift_protocol_breach` event type registers in `bin/.events-registry.md`. Existing `requirements_updated` event continues to be used by the auto-apply cascade.
