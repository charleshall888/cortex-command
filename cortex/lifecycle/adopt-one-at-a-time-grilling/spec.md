# Specification: adopt-one-at-a-time-grilling

This spec implements the cadence-refresh child (ticket 222) of epic 221 (Adopt grill-with-docs progressive-disclosure system). See `cortex/research/grill-me-with-docs-learnings/research.md` for the epic discovery and `cortex/lifecycle/adopt-one-at-a-time-grilling/research.md` for the ticket-specific research.

## Problem Statement

The two requirements-interview surfaces in this repo — `skills/requirements-gather/SKILL.md` and `skills/lifecycle/references/specify.md` §2 — currently allow batched questioning, which research shows produces partial answers, decision-tree drift, and respondent fatigue. The repo already encodes Pocock's `recommend-before-asking` pattern but does not encode his one-at-a-time cadence. This work tightens both interview surfaces with explicit cadence prose, ties file-path citation to the existing `codebase-trumps-interview` decision gate, and adds a judgmental edge-case invention guideline to `specify.md` §2. All behaviors are passive-precondition prose; no MUST/CRITICAL/REQUIRED escalations, no new files, no Q&A schema change. The spec ships prose emission; behavioral validation (whether the new prose measurably changes interview cadence in real sessions) is governed by the CLAUDE.md MUST-escalation policy and is a future evidence-collection cycle, not this cycle.

## Non-Requirements

- Does NOT add new files. Approach B (shared reference file at `skills/lifecycle/references/interview-cadence.md`) is the durable form for cross-surface parity (matching the structural pattern used elsewhere — see `tests/test_lifecycle_kept_pauses_parity.py`). This work defers B and ships A + pointer-note as a scope-bounded simpler form. Pointers are weak human-readable cross-references, not enforced parity; they aid a reader comparing the two surfaces but provide no automated drift detection. If post-landing observation shows the two cadence blocks drift in substance, revisit B as a follow-up.
- Does NOT add new Q&A block fields. The 4-key schema (Q / Recommended answer / User answer / Code evidence) is preserved verbatim per R11.
- Does NOT reposition the §2b Verification check sub-block to during-interview. The category-error argument prevailed; instead R7 adds a separate soft §2 guideline.
- Does NOT change AskUserQuestion call shape. Cadence is encoded as prose; no new call sites land in `specify.md` §2, so the kept-user-pauses inventory at `skills/lifecycle/SKILL.md:189–200` does not gain new entries.
- Does NOT add new MUST/CRITICAL/REQUIRED escalations (R12). Per CLAUDE.md MUST-escalation policy, escalation requires effort=high dispatch evidence + events.log F-row, which this work does not collect.
- Does NOT instrument behavioral validation. Whether the prose-only cadence change measurably alters real interview cadence is a future evidence-collection cycle. This spec ships prose emission, verified by grep-based acceptance criteria. Transcript inspection, events.log F-row capture, and effort=high dispatch attempts are out of scope here; they are the gating prerequisites for any future MUST-escalation if the soft-positive form is later observed to fail.
- Does NOT add a cross-surface cadence-parity test (analogous to `tests/test_lifecycle_kept_pauses_parity.py`). Pointer notes are the chosen mechanism per Approach A; structural parity enforcement is Approach B's contribution and is deferred per the Non-Requirements note above.
- Does NOT implement interrupt-driven behaviors (active conflict-flagging, fuzzy-language sharpening, real-time code contradiction surfacing). Held per epic 221 Out of Scope pending effort=high evidence.
- Does NOT touch glossary or ADR mechanisms — those are sibling children of epic 221.
- Does NOT lower `LINE_TOLERANCE` from 35 to 20. The drift fix updates the SKILL.md prose to match the code constant; lowering the constant risks unrelated parity failures.
- Does NOT prescribe categorical per-requirement edge-case invention. Judgmental application only (R8); Pocock's actual usage is judgmental, and categorical invention compounds latency and amplifies hallucination on already-clear criteria.
- Does NOT prescribe universal file-path citation. Grounded-only (R3, R6); preserves existing `Code evidence: omit otherwise` semantics and avoids prompt-injection-via-codebase incentive.

## Edge Cases

- **Intent-only question with no code grounding**: the `Code evidence` field is omitted, matching existing semantics. The agent does not fabricate a citation or write `N/A`. Expected behavior under R3/R6 grounded-only rule.
- **Acceptance criteria already tight on first pass**: judgmental edge-case invention skips invention; no extra turn is consumed. Expected behavior under R8.
- **§2 prose later copied verbatim into a third interview surface** (hypothetical new skill): the cross-reference pointer notes (R2, R5) make the duplication self-documenting and act as a trigger for revisiting Approach B. The pointer is documentation, not load-bearing — it serves a reader comparing surfaces but does not prevent drift.
- **Author edits one cadence surface without the other**: pointer notes (R2, R5) act as a reader-facing cross-reference, not as an enforced gate. If both authors miss the cue, drift accumulates silently — the spec accepts this as the known cost of choosing A over B. Future drift observation is the trigger to revisit B as a follow-up.
- **`specify.md` §2 anchor renamed**: pointer in `requirements-gather/SKILL.md` becomes stale. Acceptable — pointer is a hint, not a contract. The kept-user-pauses parity test catches AskUserQuestion site drift; this work adds no new call sites so the test is unaffected.
- **Pre-existing parity drift (±20 prose vs LINE_TOLERANCE=35) compounds with a future tolerance change**: R10 fixes the drift toward the code constant. A future author tightening `LINE_TOLERANCE` must update both surfaces.
- **R7 baseline already contains the verb `verify`** (e.g., references to the §2b Verification check sub-block): R7's AC is delta-based — it requires the count of `\bverify\b`-bearing lines in §2 to grow by at least 1 from baseline, isolating newly-added verification-guideline prose from incumbent text.

## Technical Constraints

- **SKILL.md 500-line size cap** (`tests/test_skill_size_budget.py`) — both edited files have ample headroom; size-cap exception not required.
- **Soft-positive routing only** per `CLAUDE.md` MUST-escalation policy — R12 enforces zero MUST/CRITICAL/REQUIRED tokens in the final committed state of both edited files (content-based, not diff-based, so phase merge timing does not affect verification). Evidence-artifact requirement under the post-Opus-4.7 escalation policy is not satisfied for any escalation in this work, so escalations are disallowed.
- **Passive-precondition encoding only** per ticket 222 Edges — cadence, recommend-before-asking, grounded file-path citation, and judgmental edge-case invention are all satisfiable as preconditions the agent passes through. Interrupt-driven mid-turn injections (active conflict-flagging, fuzzy-language sharpening, real-time code contradiction surfacing) are explicitly out of scope.
- **Q&A block contract preservation** — the schema at `skills/requirements-gather/SKILL.md:44–57` is immutable in this work (R11). New behaviors land as prose within the existing four fields.
- **No new AskUserQuestion call sites in `specify.md` §2** — cadence is encoded as prose, not as call-shape constraints. The kept-user-pauses inventory at `skills/lifecycle/SKILL.md:189–200` does not gain new entries.
- **Design principle "Prescribe What and Why, not How"** per `CLAUDE.md` — the cadence prose describes the decision rule and intent (one question is the gate to the next; grounded citation when code informs the recommendation), not procedural narration of method.
- **Acceptance-criteria verify emission, not behavior.** The grep-based ACs (R1, R3, R4, R6–R11) verify that the prose was added in the right place, not that the prose changes observable interview cadence. This is the structural limit of prose-only enforcement under CLAUDE.md's soft-positive routing policy, which presumes capable models interpret soft-positive prose probabilistically. Behavioral validation depends on a separate evidence-collection cycle (see Non-Requirements).
