# Research: observed-merge remediation for morning-review's "PR already merged" exit

**Scope anchor (clarified intent):** On morning-review's "PR already merged — main is up to date" exit (`skills/morning-review/references/walkthrough.md` §6 step 2), remediate the residual gap where a completed-feature ticket lands on main but stays open — the rare `BACKLOG_WRITE_FAILED` × out-of-band-merge intersection that #342 left covered only by a non-silent verify-closure advisory. Evaluate three options: **(A)** full observed-merge auto-close; **(B)** enhanced advisory that emits the exact ready-to-run close command(s); **(C)** status-quo/wontfix. **Out of scope:** stranded-merged-work reconciliation (orphaned integration branches/PRs/worktrees + runner re-pick backoff) — to be split into its own ticket.

**Headline outcome:** The evidence moved the recommendation *away* from the ticket's premise. A cheap, durable "is this ticket complete on main" check does **not** exist among the signals morning-review reads at this exit (§6a sync is skipped ⇒ local disk is stale), and this exit has **no push machinery** to fix main anyway. The load-bearing primitive a correct closer needs is **unbuilt** and is **shared** with the split reconciliation work. Recommendation: **do not bolt an auto-close or a local-filtered advisory onto this exit now** — keep it an advisory (at most a filter-free, fetch-first correctness tweak) and defer the real closer to the reconciliation ticket.

## Codebase Analysis

- **Primary edit site:** `skills/morning-review/references/walkthrough.md` §6 step 2 (the `state == "MERGED"` early-exit, ~L461–465) — currently #342's Req-7 soft advisory ("PR already merged — main is up to date. Then stop. Verify this session's completed-feature tickets are `complete`…"). §6b (the sole post-merge closer, L540–604) is **skipped** on this exit by its own guard (L543–545).
- **Reuse, don't reinvent:** §6b already carries the `cortex-read-backlog-backend` three-arm gate (`cortex-backlog` / `none` / external, L547–560) and the `cortex-update-item {backlog_id} --status complete` pattern (L565–582). Any remediation should reuse these.
- **`cortex-update-item` mechanics** (`cortex_command/backlog/update_item.py`): single-repo scope (no `--repo`/`--backlog-dir`; resolves via `_resolve_user_project_root()/cortex/backlog`, L540). Exit 0 success / 1 not-found (silent) / 2 ambiguous (candidate list). Terminal cascade: strips closed UUID from siblings' `blocked-by`, auto-closes parent epic if all siblings terminal (`_check_and_close_parent`, no-op if parent already terminal), regenerates `index.md`. Idempotent on the *event* (`status_changed` appended only when `new != old`, L385) — but still `atomic_write`s the file (bumps `updated:`) and re-runs cascade/index every call.
- **`overnight-state.json`** (`cortex_command/overnight/state.py`): reader fields `integration_branch`, `worktree_path`, per-feature `backlog_id` (zero-padded), `repo_path` (None for home), `status ∈ {pending,running,merged,paused,failed,deferred}`. File is **gitignored** (`cortex/lifecycle/sessions/`), never reaches main; per-feature `status: merged` means "merged into the **integration branch**," not "merged to main."
- **Durability primitive exists** (`cortex_command/lifecycle/complete_route.py::_head_has_feature_complete`, ~L256–288): `git show HEAD:cortex/lifecycle/{feature}/events.log` grep `feature_complete` — working-tree presence is **not** durable (ADR-0004 #339). But this reads **local HEAD**, which is stale at this exit (see On-Main-Signal).
- **Dual-source mirror:** canonical `skills/morning-review/` → `plugins/cortex-overnight/skills/morning-review/` via `just build-plugin` (`rsync -a --delete` over the whole tree; justfile ~L601–617). Editing walkthrough.md/SKILL.md requires a **same-commit** mirror regen (drift pre-commit hook + `tests/test_dual_source_reference_parity.py`).
- **Guard test:** `tests/test_morning_review_status_close_ordering.py` (spelling-agnostic `CLOSE_PATTERN`, merge-gate + SKILL single-source assertions). Any change must keep it green.
- No new ADR expected (the closure-on-merge-only stance applies existing ADR-0004) unless Option A's inferred-state auto-close is treated as a new hard-to-reverse decision.

## Web Research

- **GitHub** gates closing-keywords on **two** independent conditions — the PR must target the **default branch** *and* the repo's auto-close policy toggle must be on; "merged" alone never closes an issue. Conflating merge with closure is a documented anti-pattern (community discussion #193412).
- **Linear** requires an explicit `externally-merged` label **before** closing an out-of-band merge — it does **not** infer completion after the fact. Precedent: treat "automation didn't witness the merge" as needing an explicit confirming signal, not silent auto-close.
- **`git status`** prints copy-pasteable commands it never runs; the `--dry-run` family (rsync/terraform/kubectl) generalizes "describe the effect, don't apply it." Dominant idiom for handing a state-mutating decision back to a human.
- **Kubernetes reconciliation** is level-triggered: an event is a *nudge to re-derive current state*, never ground truth; idempotency means a re-run is a safe no-op. Maps onto ADR-0004: re-derive "is this durably on main" at action time, never trust a cached `MERGED` flag.
- **HITL/SRE guidance:** "risk and reversibility, not feature completeness, determine automation priority." A false auto-close that hides an undetected real gap is worse than a visible open ticket.
- **Cross-repo:** repo-context resolution from cwd/remote is a known ecosystem footgun, but only reachable with a bare numeric ID + ambiguous cwd — **latent, not live** for a repo-pinned/repo-local command.

## On-Main Completion Signal: Detectability, Durability & Trigger Predicate

- **Decisive fact:** §6a (post-merge sync) is **structurally skipped** on the already-merged exit (its own guard at ~L518–520; §6 step 2 says "Then stop"). So local main has **not** pulled the out-of-band merge — every local-disk signal (`events.log` HEAD row, backlog frontmatter `status`, `overnight-state.json`) can be **stale** here. A "still shows open" local read may just mean "local hasn't pulled," not "the write failed."
- **`overnight-state.json`** never reaches main (gitignored) and encodes integration-branch merge, not main merge — cannot serve as on-main proof.
- **Durability:** PR `state == MERGED` proves the **code** landed but says nothing about whether the *separate, fallible* `status: complete` write-back succeeded (`outcome_router.py::_write_back_to_backlog` L439–502 wraps the write in a swallow-all `try/except` emitting `BACKLOG_WRITE_FAILED`, surfaced nowhere and stored in an ephemeral gitignored log).
- **A cheap remote-authoritative check *does* exist but is unimplemented:** `gh api repos/{owner}/{repo}/contents/cortex/backlog/{id}-*.md?ref=main` (or `ref={mergeCommit.oid}`) reads the ticket's on-main frontmatter directly, consistent with the `gh pr list` call this exit already makes.
- **Trigger predicate:** "still open" is a sound **necessary** gate (skip if already complete — cheap, safe) but **not sufficient**. Over-fire risks: (1) local staleness (dominant), (2) manual reopen between overnight and review, (3) post-merge revert.
- **Idempotence:** `cortex-update-item` is well-behaved for repeated closes (event-guarded; parent-close no-op if terminal; safe exit codes).

## Tradeoffs & Alternatives

- **A — full auto-close:** closes the gap with no operator action, *but* reintroduces the blind-close class #342 just removed, closes on "a merge this review did not perform" (#342's explicit Non-Requirement), and its parent-epic cascade would compute from **stale local** sibling statuses. Requires machinery disproportionate to a rare×rare, already-non-silent fault.
- **B — enhanced advisory (print exact commands):** non-destructive, human-in-the-loop, matches #342's Req-6/7 soft-note precedent. *But* (per Adversarial) selecting "which tickets are still open" from a **stale local read** over-fires in the common write-back-succeeded case — so a *trustworthy* B needs the same unbuilt remote check as A, erasing its cost advantage.
- **C — status-quo/wontfix:** zero new surface/risk; aligned with #340's efficiency discipline and project.md simplicity; matches sibling #344's wontfix precedent. Cost: leaves the operator to identify the ticket and recall the command on the rare occasion this fires (a small, filter-free wording tweak removes most of that without new machinery).
- **Cross-repo hazard reachability — NOT reachable** via morning-review: `repo:` frontmatter routes *code execution* (integration worktree/branch), never ticket **storage**; the ticket file always lives in the home `cortex/backlog/`; the runner write-back and morning-review's `cortex-update-item` both target the home backlog. Dropping a cross-repo *disambiguation guard* is correct.

## Adversarial Review

- **B's cost advantage is overstated.** The write-back targets the integration worktree's backlog (→ integration branch → remote main on merge), and **never touches the home checkout**; with §6a skipped, the home backlog file is stale. A local status read **cannot distinguish** "already complete on main (write-back succeeded — common)" from "open on main (write-back failed — rare)". A local-filtered B **over-fires in the common case** (suggests closing a ticket already complete on main). Trustworthy B needs the unbuilt remote check ⇒ no cost edge over A.
- **Neither A nor B durably fixes the gap here.** `cortex-update-item` writes only the local home file; there is **no push** on this exit (§6a, which pushes via `cortex-git-sync-rebase`, is skipped). A "successful" close sits unpushed — it does not fix "ticket open *on main*", which is the gap #342 names. The durable fix belongs in a post-pull re-run / dedicated reconciliation flow, not a bolt-on at this early-exit.
- **A-only cascade hazard:** A's close can trigger the parent-epic auto-close (`update_item.py` `_check_and_close_parent`) computed from **stale local** sibling statuses — closing a parent whose other children were reverted on main. B (operator-gated) avoids this.
- **Idempotence doesn't neutralize the over-fire cost** at this exit: since local is stale-*open*, the close *will* mutate the local file and diverge it from unpulled main.
- **Cross-repo confirmed unreachable — but a real §6b-wide concern survives, mis-labeled:** bare-ID resolution trusts that `overnight-state`'s `backlog_id` matches the home backlog's *current numbering*; `CORTEX_REPO_ROOT` is honored verbatim (`common.py` ~L84–86), so a wrong root or a renumbered backlog mis-resolves — this corrupts §6b's **existing** closer too. The split ticket should capture this as a *general §6b* concern, not lose it under "cross-repo → drop."
- **Known-rework coupling:** B-here and the general reconciliation need the *same* durable on-main primitive; shipping B-local now would create two divergent status-reading mechanisms (Solution-horizon violation). Split is safe only if B-local is **not** shipped now.
- **Adversarial recommendation:** **C now** (the exit is already a non-silent advisory; the fault is rare×rare; no events.log F-row / transcript evidence shows operators get stuck), with at most an **honest, filter-free, fetch-and-push-aware wording tweak** (name this session's completed-feature IDs; tell the operator to `git fetch origin main` first because local is stale here; print `cortex-update-item {id} --status complete` + a push reminder to run only if still open post-fetch — **no local status filter, no new machinery**). Build the durable remote check once, inside the split reconciliation ticket.

## Open Questions

- **Scope decision (for Spec / user):** Given the evidence, choose among **C-pure wontfix**, **C-honest-advisory** (filter-free fetch-first tweak that also corrects a latent staleness wrongness in #342's current advisory), or **defer entirely** into the reconciliation ticket. Research recommends **C-honest-advisory** as the proportionate answer; a genuine auto-close/filtered-advisory (B/A) is **not** advisable at this exit without the durable remote check + push, which belong in the reconciliation ticket.
- **RESOLVED — local vs remote signal (Codebase vs On-Main-Signal contradiction):** the durability primitive `_head_has_feature_complete` reads *local* HEAD, which is stale on this exit because §6a is skipped; a correct check must be **remote** (`gh api …?ref=main`) and is currently **unbuilt**. This resolves in favor of On-Main-Signal.
- **Evidence gate (for any escalation beyond the current advisory):** no `events.log` F-row or transcript evidence was found that operators actually get stuck at this exit. Per the MUST-escalation / evidence discipline, that absence argues against building new mechanism now.
- **Split-ticket contents (for Complete):** the reconciliation follow-up should carry (a) observed-merge close done right — durable remote on-main check + push, (b) stranded-merged-work reconciliation (orphaned integration branches/PRs/worktrees + runner re-pick backoff), and (c) the **general §6b bare-ID/backlog-numbering-trust** concern (re-scoped from the unreachable "cross-repo" framing).
