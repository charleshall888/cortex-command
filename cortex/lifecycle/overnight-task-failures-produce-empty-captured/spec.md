# Specification: overnight-task-failures-produce-empty-captured

## Problem Statement

When an overnight worker subprocess fails, the runner already captures the child's stderr tail and exit code in `cortex_command/pipeline/dispatch.py` — but **discards them from the result carrier** (`DispatchResult` has only `success/output/error_type/error_detail/cost_usd`). The only failure-evidence that reaches the brain (the LLM that makes the retry/defer/pause decision) and the morning report is `output` — the joined assistant text, which is **empty** when a worker crashes before emitting text. The result is an opaque `ProcessError: exit code 1` with no diagnostic content (incident `overnight-2026-06-17-1821`, feature `codify-sim-render-boundary` Task 5, failed 4× across two rounds).

This feature threads the already-captured diagnostics (stderr tail, exit code, cwd) onto the path the brain and the morning report consume. **Honest scope of the win:** for failure classes whose command *emits diagnostic output* (test/build defects, pre-commit-gate rejections, crashes that print), the brain and report gain real, localized evidence. For the *silent-crash* class that motivated the ticket — where stderr is empty and the exit code is a generic `1` or unknown — the feature converts a fully-opaque failure into a **labeled** one (exit code + cwd + an explicit "stderr was empty" marker), which is a meaningful improvement over today's blank `**Last worker output**` but is **not a causal diagnosis**; diagnosing *why* a worker died silently is #308's supervision/liveness territory, explicitly out of scope here. Because surfacing arbitrary subprocess stderr into the morning report (which is committed to local `main`) widens a credential-leak surface that today scrubs only `sk-ant-` keys, the same change hardens the capture path — as a defense-in-depth reduction, not a guarantee of completeness.

## Phases

- **Phase 1: Harden the capture path** — broaden `_on_stderr` secret redaction (cue-anchored, not prefixless-blob) and add a byte cap so every downstream consumer reads an already-scrubbed, bounded stderr tail; stop tracking new `pipeline-events.log` in git.
- **Phase 2: Thread diagnostics to the brain** — carry a single diagnostics bundle through `DispatchResult` → `RetryResult` → `BrainContext` and into the `batch-brain.md` prompt, so the brain's retry/defer decision sees stderr/exit_code/cwd.
- **Phase 3: Surface diagnostics in the morning report** — extend the `task_output` event with the diagnostics and render them in the failed-feature section, replacing the empty `**Last worker output**` line on crash failures; verify end-to-end fidelity.

## Requirements

1. **Broaden secret redaction in `_on_stderr` (cue-anchored, no prefixless blobs)**: The stderr-capture callback in `cortex_command/pipeline/dispatch.py` scrubs additional credential shapes **value-level and cue-anchored** — each pattern is anchored on a distinguishing prefix or keyword so it cannot match arbitrary high-entropy text: GitHub tokens (`ghp_`/`gho_`/`ghs_`/`ghr_`), Slack (`xoxb-`/`xoxp-`), `Bearer <token>`, the value after `password=`/`token=` delimiters, and URL userinfo (`https://user:pass@host`). It **must NOT** introduce a prefixless fixed-length blob matcher (e.g. a bare "40-char base64 AWS-key shape"), because that is regex-indistinguishable from git SHAs, UUIDs, content hashes, and base64 test fixtures — the diagnostic content the feature exists to preserve. Additionally, when a multi-line-secret **cue** appears on a line (e.g. a PEM `-----BEGIN … PRIVATE KEY-----` marker), mask conservatively at line level for the cued block, since a per-line value regex cannot span the secret (see Technical Constraints). Acceptance: a new test in `cortex_command/pipeline/tests/test_dispatch.py` (a) feeds each cue-anchored shape through the capture path and asserts the secret value is replaced by a redaction marker while surrounding context survives, AND (b) feeds a benign 40-char hex git SHA, a UUID, and a base64 fixture and asserts they are **NOT** redacted (over-redaction guard); `just test` exits 0. **Phase**: Phase 1: Harden the capture path

2. **Add a byte cap to `_on_stderr` capture**: Captured stderr is bounded by total bytes in addition to the existing `_MAX_STDERR_LINES=100` line cap, so a single pathological multi-megabyte line cannot reach the event log, brain prompt, or committed report. The byte cap is applied **without bisecting a cue-anchored token** where avoidable (truncate at the tail boundary, not mid-token of a matched secret). Acceptance: a test asserts that capturing a single >cap-byte stderr line yields a stored tail ≤ the byte cap; `just test` exits 0. **Phase**: Phase 1: Harden the capture path

3. **Stop tracking new `pipeline-events.log` (scoped)**: `cortex/.gitignore` ignores `pipeline-events.log` under `cortex/lifecycle/` (matching the existing `lifecycle/overnight-events*.log` precedent at `cortex/.gitignore:28`), scoped so it does not over-broadly ignore the file at unintended repo paths. Acceptance: `git check-ignore cortex/lifecycle/pipeline-events.log` exits 0 AND `git check-ignore cortex_command/pipeline/tests/fixtures/pipeline_logs/pipeline-events.log` exits 1 (pass if the lifecycle path is ignored and the test-fixture path is not). **Phase**: Phase 1: Harden the capture path

4. **Add a diagnostics bundle to `DispatchResult`, populated from the real capture locals**: `DispatchResult` (`cortex_command/pipeline/dispatch.py`) gains one optional field carrying the failing dispatch's diagnostics — `child_stderr: Optional[str]`, `exit_code: Optional[int]`, `cwd: Optional[str]` — grouped as a single value object (e.g. `DispatchDiagnostics`), following the `cost_usd: Optional[float] = None` optional-field template. Populated at the two error return sites from the already-computed locals (`_stderr_lines`, `exit_code`, `worktree_path`); left `None` on the success and budget-exhausted paths. Acceptance (behavioral, not a token grep): a test constructs a `ProcessError` dispatch with a known stderr/exit_code and asserts the returned `DispatchResult`'s diagnostics field is populated with **those captured values** (and is `None` on the success path); `just test` exits 0. **Phase**: Phase 2: Thread diagnostics to the brain

5. **Thread the diagnostics bundle through `RetryResult` from the same final attempt**: `RetryResult` (`cortex_command/pipeline/retry.py`) carries the same diagnostics bundle as **one** optional field, set once from the final attempt's `DispatchResult` at every failure-path construction site (the ~7 sites incl. circuit-breaker, `pause_human`, `pause_session`, escalation-exhausted, all-retries-exhausted). The bundle and `final_output` must originate from the **same** final `result` binding (same-attempt provenance — see Technical Constraints invariant). Acceptance: a test asserts that **each** failure-path `RetryResult` exit carries the diagnostics from its final `result` (guards the silent-drop risk of a missed site) AND that the bundle and `final_output` on a given `RetryResult` describe the same attempt (e.g. by constructing a multi-attempt retry where the final attempt's stderr differs from earlier attempts' and asserting the final one surfaces); `just test` exits 0. **Phase**: Phase 2: Thread diagnostics to the brain

6. **Feed diagnostics into the brain context and prompt (values, not just a heading)**: `BrainContext` (`cortex_command/overnight/brain.py`) gains a diagnostics field, populated in `feature_executor.py:_handle_failed_task` from `retry_result` via the existing defensive-`getattr` idiom; `batch-brain.md` gains a `## Final Attempt Diagnostics` section (placeholder rendered via `_render_template`) showing exit_code, cwd, and the stderr tail, **framed with their limits**: exit 1 is generic; exit_code may be unknown on timeout/connection failures; an empty stderr is a "silent failure" marker, not a localized diagnosis — and the learnings file remains the primary cross-attempt evidence. Acceptance (renders values + framing, not just heading presence): a `test_brain.py` test renders the prompt from a `BrainContext` carrying a known exit_code/cwd/stderr and asserts the rendered output **contains those values** AND the limits-framing text — not merely that the heading string exists; `just test` exits 0. **Phase**: Phase 2: Thread diagnostics to the brain

7. **Carry diagnostics on the `task_output` event (emitted for every task)**: The `task_output` event written **unconditionally for every task** (success and failure) in `feature_executor.py` (the emit at ~line 704, before the success/failure branch; carries `task_number`) gains the diagnostics as **distinct** fields (NOT folded into `output`, which is empty on crash), sourced from the same `RetryResult` bundle the brain reads. Acceptance: a test asserts a failing task's `task_output` event includes the stderr/exit_code/cwd fields sourced from the bundle, and that a successful task's `task_output` omits/null-defaults them; `just test` exits 0. **Phase**: Phase 3: Surface diagnostics in the morning report

8. **Render diagnostics in the morning report failed-feature section**: `render_failed_features` (`cortex_command/overnight/report.py`) shows exit_code, cwd, and a stderr tail (with an explicit `(empty)` marker when absent, and `unknown` for a `None` exit_code) for a failed feature, read via `_read_last_task_output` (or a sibling reader) from the `task_output` event. The stderr tail uses a **deliberate tail-anchored truncation budget larger than the current 500-char `output` cap** (`report.py:_read_last_task_output`) so the diagnostically-valuable end of the stderr is not clipped. Acceptance: a `test_report.py` test asserts the failed-feature block renders the exit_code and stderr lines (and the `(empty)` marker on empty stderr, `unknown` on `None` exit_code); `just test` exits 0. **Phase**: Phase 3: Surface diagnostics in the morning report

9. **Register event field-additive extensions (row-scoped verification)**: `bin/.events-registry.md` documents the new optional fields under the existing `dispatch_error` event (add `cwd`) and `task_output` event (add the diagnostics fields), as field-additive schema extensions (emitters omit when `None`, consumers tolerate absence) — **no new event-name row**. Acceptance (row-scoped, not a whole-file grep): the events-registry audit recipe (`just`-invoked) passes with the new fields recorded under their owning event rows; a check confirms the `cwd`/diagnostics tokens appear **within** the `dispatch_error`/`task_output` row blocks specifically (e.g. an awk/section-scoped check), not merely somewhere in the file; `just test` exits 0. **Phase**: Phase 3: Surface diagnostics in the morning report

10. **End-to-end fidelity: real captured stderr reaches the surfaces**: An integration test simulates a failing dispatch whose captured `_stderr_lines` carry a known sentinel string and asserts that **that same sentinel** (post-redaction) reaches BOTH the brain prompt (via the carrier) AND the morning report (via the `task_output` event) — closing the gap where a hardcoded stub at any single layer would pass the per-layer tests (R4–R8) undetected. Acceptance: the integration test passes asserting the sentinel propagates source→brain and source→report; `just test` exits 0. **Phase**: Phase 3: Surface diagnostics in the morning report

11. **Preserve metrics pairing and report fixtures**: Adding fields to `dispatch_error`/`task_output` does not break `pair_dispatch_events` (`cortex_command/pipeline/metrics.py`, which keys on event type, not field presence) or the daytime-skip field-presence heuristic; failure-path report fixtures (`tests/test_morning_report.py`, `test_runner_morning_report_commit.py`) are regoldened. Acceptance: a `test_metrics.py` guard asserts `dispatch_error` with the new fields still pairs; full `just test` exits 0. **Phase**: Phase 3: Surface diagnostics in the morning report

> **Note on acceptance criteria**: The `just test` exits-0 conjunct on each requirement is a non-regression guard, not the proof of new behavior — the requirement-specific behavioral test named first in each criterion is the load-bearing check.

## Priority (MoSCoW)

- **Must-have**: All 11 requirements, for the agreed scope. The diagnosability core (R4–R8, R10) is the feature's reason to exist. The hardening (R1–R3) is must-have **because the user explicitly scoped it into #309** — surfacing subprocess stderr into the committed morning report is what makes broadened redaction load-bearing. Governance/safety (R9 event registry, R11 metrics/fixtures) are must-have to land without breaking the audit trail or the suite.
- **Should-have**: none. The natural should-have candidates (byte cap, redaction breadth) were deliberately elevated to must by the user's hardening-scope decision; leaving them out would ship a known credential-leak widening.
- **Won't-do (this ticket)**: enumerated concretely in Non-Requirements — raw stdout capture, literal inner-command capture, the `worker_no_exit_report` class, #308's supervision/circuit-breaker/liveness work, the upstream wild-light gate, git-history rewrite of existing tracked logs, any change to `error`/`error_type` halt-token semantics, and a complete (vs defense-in-depth) redaction guarantee.

## Non-Requirements

- Does **not** capture raw child **stdout** — SDK v0.1.46 (`claude_agent_sdk`) exposes no stdout callback; child stdout is the internal JSON message stream. The ticket's "stdout/stderr tail" reduces to the stderr tail.
- Does **not** capture the literal inner failing shell command — the CLI argv is built then discarded by the SDK and is unrecoverable from `ProcessError`. The ticket's "exact command" is reinterpreted as `cwd` plus the already-present invocation identity (`feature`/`skill`/`attempt` on the `dispatch_start` event). No synthesized "command" string is surfaced to the brain (it would mislead the brain into blaming the SDK invocation rather than the inner failure).
- Does **not** cover the `worker_no_exit_report` failure class — that branch has **no `DispatchResult`** (the dispatch succeeded; the failure is a missing exit-report file), so there is no subprocess stderr to capture. That class is #308's, not #309's.
- Does **not** fix the orchestrator supervision / circuit-breaker / out-of-process-liveness gap (companion ticket #308), the upstream wild-light pre-commit gate, or the *causal* diagnosis of a silent worker crash (the motivating incident gets a labeled failure, not a root cause).
- Does **not** modify `error`/`error_type` string values or `_SESSION_HALT_ERROR_TYPES` / `_SYSTEMIC_ERROR_TYPES` membership — those are the closed-vocabulary halt-decision channel that #308/#262 consume; #309's diagnostics ride in parallel structured fields.
- Does **not** retroactively untrack or rewrite git history for already-committed `pipeline-events.log` copies — the gitignore rule applies to new commits only.
- Does **not** add a `dispatch_error` re-correlation reader for the report — `dispatch_error` lacks a `task_number`; the report uses the `task_output` event (which has one) instead.
- Does **not** claim complete redaction — an enumerated cue-anchored allowlist is defense-in-depth against an arbitrary subprocess, not a guarantee that no secret can ever reach a sink.

## Edge Cases

- **`exit_code` is `None`** (`CLIConnectionError`/`asyncio.TimeoutError` — common overnight modes): the diagnostics bundle still carries `cwd` and the stderr tail; the brain prompt and report render exit_code as `unknown` rather than omitting the line.
- **stderr is empty** (worker crashed before writing stderr — the motivating incident's shape): render an explicit `(empty)` marker, framed as a "silent failure" signal (distinguishes "crashed silently" from "ran but said nothing"), **not** a localized diagnosis. No requirement asserts the brain's decision *branches* on this marker — it is improved evidence presentation, not a new control path.
- **A credential is split across two `_on_stderr` callback lines, or spans multiple lines (PEM block)**: a per-line value regex cannot reassemble it; R1's cue-anchored line-level masking (on a `BEGIN PRIVATE KEY`-class cue) is the mitigation, and the residual limit (a token bisected mid-stream by an arbitrary buffer flush can leak a fragment) is an accepted defense-in-depth gap, documented, not silently assumed away.
- **A benign high-entropy token appears in stderr** (git SHA, UUID, base64 fixture, stack-trace hex): must **survive** redaction — R1's over-redaction guard test asserts this, because destroying it would defeat the feature's purpose.
- **stderr contains a single multi-megabyte line**: the new byte cap (R2) truncates it tail-anchored before any sink; the 100-line cap alone does not catch this.
- **A feature has multiple failing tasks**: the `task_output` event carries `task_number`, so the report surfaces the diagnostics for the specific failed task rather than a feature-level smear.
- **A `RetryResult` construction site is missed during threading**: guarded by R5's per-site test asserting every failure-path exit propagates the bundle — a missed site is a test failure, not a silent empty.
- **Multiple retry attempts**: the surfaced diagnostics reflect the **final** attempt; R5's provenance assertion guards that the bundle and `final_output` describe the same attempt.
- **`worker_no_exit_report` failure** (dispatch succeeded, no exit-report file): there is no `DispatchResult`, so no diagnostics bundle exists — this feature surfaces nothing here (see Non-Requirements; it is #308's class).

## Changes to Existing Behavior

- MODIFIED: `_on_stderr` (`dispatch.py`) — `sk-ant-`-only redaction → cue-anchored value-level redaction of multiple credential shapes (explicitly excluding prefixless blob matching); line-cap-only → line cap + byte cap.
- MODIFIED: `DispatchResult` / `RetryResult` / `BrainContext` — gain an optional diagnostics bundle field.
- MODIFIED: `batch-brain.md` — gains a `## Final Attempt Diagnostics` section (values + limits framing).
- MODIFIED: `dispatch_error` event gains `cwd`; `task_output` event gains diagnostics fields (both field-additive).
- MODIFIED: `render_failed_features` (`report.py`) — the failed-feature block shows exit_code/cwd/stderr-tail lines.
- ADDED: `cortex/.gitignore` ignores `cortex/lifecycle/**/pipeline-events.log`.

## Technical Constraints

- **SDK boundary (`claude_agent_sdk` v0.1.46, pinned `>=0.1.46,<0.1.47`)**: only a `stderr` line-callback exists; `ProcessError.stderr` is a hardcoded placeholder (use `_stderr_lines`); `.exit_code` is real but `None` for `CLIConnectionError`/`asyncio.TimeoutError`. Diagnostics must tolerate `exit_code = None` (render "unknown").
- **Redaction is per-line and cue-anchored, with a stated residual limit**: `_on_stderr(line: str)` is invoked once per stderr line. A value-level regex therefore (a) cannot match a token split across two callback invocations, and (b) cannot match a multi-line secret (PEM block). The mitigation is cue-anchored line-level masking for multi-line-secret markers; the residual gap (a token bisected by an arbitrary stream-buffer flush) is an accepted defense-in-depth limit, not a guarantee. Redaction must be cue-anchored — never a prefixless fixed-length blob matcher — to avoid destroying benign high-entropy diagnostics. Scrub at source so every downstream sink reads pre-scrubbed text.
- **Same-attempt provenance invariant (dual-plumbing safety)**: the brain (carrier → `BrainContext`) and the report (`task_output` event) both read the diagnostics from the **same in-memory `RetryResult` instance** of the final attempt. They cannot diverge by construction — *not* because of write/read ordering (the `task_output` write is mid-run, before the brain is assembled; only the report's *read* is post-run). The non-race safety is the shared object. R5 must preserve this: the bundle and `final_output` on a `RetryResult` must come from the same final `result` binding. A future refactor that re-sources either sink from the event log would break the invariant.
- **Event extensions are field-additive**: no new event names; emitters omit `None`; consumers tolerate absence (`bin/.events-registry.md` field-additive rule).
- **No plugin mirror / no `build-plugin`**: `cortex_command/` is not mirrored into any plugin. Verify against the working tree (`just test` runs the editable install; `CORTEX_COMMAND_FORCE_SOURCE=1` forces source) — prefer sequential dispatch over a worktree for these package edits.
- **Coordination with #308**: land #309 first (additive, low blast radius); #309's structured fields become available inputs for #308's halt/classification logic. The bundle is an orthogonal sibling field — verified not to collide with the closed `error_type` halt vocabulary. The shared hot lines (`dispatch.py` error branches; `feature_executor.py` no-exit/zero-commits gate) should be coordinated if the two land near-simultaneously.

## Open Decisions

None — the technical design is resolved from research (single diagnostics bundle; `task_output`-based report surfacing; defer `command`; drop `stdout`), the hardening scope (broaden redaction + byte cap + gitignore, all in #309) was resolved with the user, and the critical-review findings (cue-anchored redaction, end-to-end fidelity test, provenance invariant, row-scoped registry check) are folded into the requirements above. Exact truncation byte budgets and the precise cue-anchored regexes are implementation details for the plan, constrained by R1's no-prefixless-blob rule and the over-redaction guard test.
