# Plan: overnight-task-failures-produce-empty-captured

## Overview
Thread the already-captured failure diagnostics (stderr tail, exit code, cwd) — currently discarded from `DispatchResult` — through a single `DispatchDiagnostics` value object that rides the result carriers (`DispatchResult` → `RetryResult` → `BrainContext`) to the brain, and onto the `task_output` event the morning report already reads, after first hardening the capture path (cue-anchored redaction + byte cap) because this change makes subprocess stderr human-facing and committed to local `main`.
**Architectural Pattern**: pipeline
<!-- The diagnostics bundle flows through a pipeline of result carriers (dispatch → retry → brain) plus a field-additive event extension for the report consumer; the carrier-threading is the dominant data-flow shape. -->

## Outline

### Phase 1: Harden the capture path (tasks: 1, 2, 3)
**Goal**: Every downstream sink reads an already-scrubbed, byte-bounded stderr tail, and new `pipeline-events.log` copies stop entering git history.
**Checkpoint**: `_on_stderr` redacts cue-anchored secret shapes (without over-redacting benign `token=RPAREN`-style diagnostics) and enforces a byte cap; `git check-ignore --no-index cortex/lifecycle/pipeline-events.log` exits 0 (rule matches) while the test-fixture copy is unaffected and a new untracked nested path is ignored; `just test` exits 0.

### Phase 2: Thread diagnostics to the brain (tasks: 4, 5, 6)
**Goal**: Carry one diagnostics bundle from the final failed attempt through the result carriers into the brain prompt, so the retry/defer decision sees exit_code/cwd/stderr framed with their limits — for the failure classes that produce a bundle (`ProcessError`-class via the dispatch exception path); session-halting classes routed to `pause_session` carry `None` by construction (rendered as `(empty)`/`unknown`).
**Checkpoint**: A `ProcessError` dispatch's captured stderr/exit_code/cwd reaches the rendered `batch-brain.md` prompt via the carriers (not the event log); `just test` exits 0.

### Phase 3: Surface diagnostics in the morning report (tasks: 7, 8, 9, 10, 11)
**Goal**: Render the same diagnostic fields (exit_code/cwd/stderr tail — same source, subject to the report's larger display cap) in the failed-feature report section via the `task_output` event, register the field-additive schema extensions, and prove end-to-end fidelity without breaking metrics pairing or fixtures.
**Checkpoint**: The failed-feature block renders exit_code/cwd/stderr-tail (with `(empty)`/`unknown` markers); an integration test asserts a sentinel propagates source→brain AND source→report; the registry audit passes; `just test` exits 0.

### Task 1: Broaden `_on_stderr` redaction to cue-anchored value-level scrubbing (R1)

- **What**: Replace the single `sk-ant-` substitution in `_on_stderr` (dispatch.py:645–648) with cue-anchored value-level redaction of multiple credential shapes, explicitly excluding any prefixless fixed-length blob matcher, and add conservative line-level masking when a multi-line-secret cue (PEM `-----BEGIN … PRIVATE KEY-----`) appears.
- **Depends on**: none
- **Complexity**: simple
- **Context**: Edit the redaction step inside `_on_stderr(line: str)` (dispatch.py:645). Keep the existing `sk-ant-[a-zA-Z0-9_-]+` rule and add two kinds of cue-anchored pattern. **(i) Prefix-cued shapes** (unambiguous distinguishing prefix → near-zero over-redaction risk): GitHub tokens (`ghp_`/`gho_`/`ghs_`/`ghr_`), Slack (`xoxb-`/`xoxp-`), AWS access keys (`AKIA`/`ASIA` + the 16-char tail), and URL userinfo (`https://user:pass@host`). **(ii) Keyword-delimiter shapes** (`Bearer <token>`, the value after `password=`/`token=` delimiters): these keywords ALSO appear in benign diagnostic stderr (`token=RPAREN` from a parser, `expected token='EOF'` from a test, `?token=abc` in a URL, `Bearer` as an English word or a `BearerTokenFilter` class name in a stack frame). The exact regex is the implementer's call within this constraint and the over-redaction guard test below. For a cued PEM line, mask at line level. Factor the patterns into a module-level helper (a `_redact(line)` function or a list of `(pattern, replacement)` pairs) near `_MAX_STDERR_LINES` (dispatch.py:356); mirror the redact-at-source convention noted at `overnight/auth.py:54-59`. MUST NOT add a bare fixed-length base64/hex blob matcher — it is regex-indistinguishable from git SHAs, UUIDs, content hashes, and base64 test fixtures (the diagnostic content the feature preserves). This enumerated allowlist is defense-in-depth, not complete: other stably-cued families (Google `AIza`, GitLab `glpat-`) and all prefixless secrets are out of scope and may still reach the committed report — disclosed in Risks, not silently assumed covered. Add tests to `test_dispatch.py` near the existing redaction test class (test_dispatch.py:624 `Tests: dispatch_task _on_stderr redaction`): (a) feed each prefix-cued shape AND a secret-shaped `token="<≥16-char secret>"` / `password=<long-secret>` through the capture path and assert the secret value is replaced by a redaction marker while surrounding context survives; (b) the over-redaction guard — assert NOT redacted for: a benign 40-char hex git SHA, a UUID, a base64 fixture (prefixless blobs), AND the cued-keyword-in-benign-context cases `unexpected token=RPAREN`, `expected token='EOF'`, `Bearer of bad news`, and a benign short `password=changeme` parser line. The guard must cover the keyword-delimiter patterns being added, not only prefixless blobs.

## Risks
- **`git check-ignore` on a tracked file (Task 3)**: `cortex/lifecycle/pipeline-events.log` is currently git-TRACKED, and plain `git check-ignore` suppresses any tracked path regardless of a correct matching rule (verified: tracked → exit 1; `--no-index` → exit 0; an untracked nested path → exit 0). The root cause is index/tracking suppression, NOT `**/` glob depth — the `lifecycle/**/pipeline-events.log` rule is confirmed to match all relevant depths. Task 3's verification therefore uses `--no-index` (plus an untracked probe). Existing tracked copies are deliberately NOT untracked (Non-Requirement); do not `git rm --cached` to make the check pass.
- **Redaction breadth vs. over-redaction (Task 1)**: two-sided risk.
