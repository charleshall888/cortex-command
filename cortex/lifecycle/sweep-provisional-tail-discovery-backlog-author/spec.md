# Specification: Sweep provisional tail — discovery + backlog-author clusters (#359)

## Problem Statement
The discovery and backlog-author skill clusters carry 43 provisional (adversarially-unverified) trim candidates from the #357 skill-value audit — ~7.0k weighted tokens of prose the audit flagged as possibly low-value. This child verifies each candidate, applies the surviving trims to the canonical `skills/` sources (regenerating the `plugins/cortex-core` mirrors), and records the applied/refuted outcome of every candidate so the deferred reconciliation can update the ledger and a future sweep does not re-propose the refuted ones. The benefit is lower context cost on these hot-path skills without silently removing load-bearing routing/interview/gate text; the ledger status write-back is deferred to a single #357 reconciliation commit so the four sibling children never contend on `master_candidates.json`.

## Phases
- **Phase 1: Preflight & independent trims** — re-derive the in-scope set at execution time, then apply the independent, well-pinned candidates per file-family behind the strengthened acceptance gate.
- **Phase 2: Interacting & high-risk candidates** — apply the collision clusters as atomic joint edits with post-apply joint re-reads, and adversarially re-check the zero-pin / frontmatter-field / block-moving candidates.
- **Phase 3: Refutation recording & parity backstop** — record every candidate's applied/refuted outcome for the deferred #357 reconciliation, realign the CI mirror-parity tuple, and pass the commit-time lint and full-suite gates.

Acceptance: the recomputed set has `count==43` AND `weighted_cost` sum `==6987` AND the 7-file histogram equals {backlog-author/SKILL.md:9, body-template.md:6, discovery/SKILL.md:7, clarify.md:6, decompose.md:6, orchestrator-review.md:2, research.md:7}; on any mismatch, halt and surface the drift. 3. **Independent candidates verified and applied per file-family behind the strengthened gate.** For the independent (non-interacting) subset — the candidates whose safety argument does not depend on another candidate in this batch, where "another candidate" spans *any file*, including a shared test fixture both candidates must mutate (e.g. the `RULE_CARRIERS` tuple; see Requirement 4) — apply extractive-only trims (delete spans; never paraphrase surviving text), honoring each row's embedded keep-list (`claim` field) plus every token named in its `pins`/`mech_pins`. Pin-hit is triage, not proof. Acceptance is **two-part, both required** (the structural tests are necessary but not sufficient — they assert heading/marker/count presence and cannot see prose deleted *between* two pinned anchors): **(a)** after each file-family's trims, the affected test suite is green — at minimum `tests/test_backlog_author.py`, `tests/test_discovery_gate_presentation.py`, `tests/test_discovery_gate_brief.py`, `tests/test_decompose_rules.py`, `tests/test_discovery_module.py`, `tests/test_load_requirements_protocol.py`, and `tests/test_l1_surface_ratchet.py` each exit 0 (`python3 -m pytest <files> -q` exit code 0); **and (b)** a recorded per-trim keep-list confirmation — for each applied trim, the implementer records (in the commit body or a per-file note) that the deleted span is confined to what the candidate's `claim` authorized and that every keep-list token named in the `claim`/`pins`/`mech_pins` is still present in the post-trim file (greppable). A trim whose deletion exceeds its `claim` scope, or that drops a keep-list token, is refuted, not applied. **Phase**: Preflight & independent trims

9. **Every candidate's outcome recorded for the deferred reconciliation.** The reconciliation needs a deterministic full-roster record over all 43 candidates, so this child records **both** outcomes in `cortex/lifecycle/sweep-provisional-tail-discovery-backlog-author/outcomes.md` — not the refuted subset alone. Each line uses a fixed, greppable marker so the format is unambiguous within this child and uniform for the reconciliation to parse:

   - Applied: `- APPLIED <id> <file> — <one-line action> (commit: <subject>)` → transcribes to the survives/applied ledger state.

   Applied trims additionally use self-documenting commit messages naming the candidate `id`, file, and action. Acceptance: `outcomes.md` exists with exactly one `REFUTED`/`APPLIED` line per in-scope candidate; `grep -c '^- \(REFUTED\|APPLIED\) ' outcomes.md` equals the in-scope count reconciled in the completion summary (43 minus any candidate the preflight legitimately dropped). **Phase**: Refutation recording & parity backstop

Both are recorded here as reconciliation-owner responsibilities, not this child's.*

## Non-Requirements
- Does NOT write candidate status back into `master_candidates.json`/`dup_groups.json` — that is the deferred, shared #357 reconciliation commit's job (Requirement 2).
- Does NOT paraphrase, reword, or restructure surviving skill prose — edits are extractive (span deletions) only, so the diff stays an auditable subset.
- Does NOT touch the cross-cluster dedup group `discovery/references/clarify.md ↔ refine/references/clarify.md` beyond whatever falls inside in-scope candidate `s7`; the `refine/` side is owned by #361 and this group is opportunistic-only — left alone unless explicitly coordinating.
- Does NOT rewrite existing `MUST`/`CRITICAL`/`REQUIRED` language into soft phrasing (or vice versa) as a side effect of trimming — that is a separately-evidenced audit action, out of scope here.
- Does NOT process any candidate flagged `overlaps_ticket` or `reproposal_of` (the 7 already excluded).
- Does NOT define or enforce a cross-child `outcomes.md` schema or a reconciliation completion handshake — those are the parent #357 reconciliation's responsibility (noted under Requirement 9); this child only fixes its own file's format.

## Edge Cases
- **In-scope set drifted since research**: the preflight count/sum/histogram no longer matches (research snapshot was stale, or out-of-band corruption — not a sibling write, which Requirement 2 forbids) → Requirement 1 halts and re-scopes rather than trimming a now-out-of-scope span.
- **A candidate's "no test greps this" claim is false**: the affected test fails after the trim → the candidate is refuted (recorded per Requirement 9), the span is restored, and the trim is not applied.
- **A trim's deletion exceeds its `claim` scope, or drops a keep-list token**: caught by Requirement 3(b)'s per-trim keep-list confirmation even when the structural tests stay green → refuted, not applied.
- **Stale line anchors**: `start_line`/`end_line` in the ledger predate recent commits → locate every span by heading text + pinned token, never by line number.
- **Interacting candidates applied out of order**: independent passes would double-delete (`s3`↔`s5`) or misplace interview content (`s6`/`s7`/`s8`) → Requirement 4's atomic-joint-edit rule prevents this.
- **Mirror left stale**: a canonical trim staged without its regenerated mirror → the pre-commit drift hook (`git diff --quiet plugins/`) fails the commit; Requirement 8 forbids `--no-verify`.
- **All candidates in a file survive / none survive**: both are valid outcomes — surviving-all means no trim for that file; refuting-all still records the refutations and leaves the file unchanged.

## Technical Constraints
- Verification bar (user-approved): **pin-hit triage → apply per file-family → the affected test suite passes**, with interacting candidates as atomic joint edits + a post-apply joint re-read. Pins/mech_pins are advisory starting points, not acceptance proof — the corpus's own adversarial pass refuted 5/101 pin-clean candidates on provenance/consumer/runtime facts a grep cannot see.
- Edit canonical `skills/` only; mirrors regenerate via `just build-plugin` (run by `.githooks/pre-commit` when `just setup-githooks` is active) and must be staged in the same commit.
- Commit via `/cortex-core:commit` only; imperative, capitalized, ≤72-char subject, no trailing period.
- The L1 surface ratchet is a non-issue: no in-scope candidate changes `description`/`when_to_use`, so no budget-row edit is needed; the `s1` risk is the dropped declared fields vs the contract/skill-path lint, not the ratchet.
- Extractive-only edits keep the diff a literal subset of the original (LLMLingua-2 faithfulness discipline); short-skill research (arXiv 2606.09421) warns that rare-path operational anchors can look dead under normal testing — bias toward keeping when a span guards a recovery/edge path.
