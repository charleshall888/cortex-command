# Plan: sweep-provisional-tail-discovery-backlog-author

## Overview
Verify and apply the 43 provisional trim candidates across the discovery + backlog-author skill clusters as an extractive-only (span-deletion) sweep: a preflight re-derivation guard, then per-file-family independent trims, then the seven cross-candidate collision clusters as atomic joint edits, closing with the parity-tuple realignment, the full-roster outcomes record, and the final suite/lint gate. Every canonical `skills/` edit regenerates and stages its `plugins/cortex-core` mirror in the same commit; the `master_candidates.json`/`dup_groups.json` ledger stays byte-unchanged (status write-back is the deferred #357 reconciliation's job). Edit tasks form a single deliberately-linear dependency chain (each depends on the prior) so no two file-editing tasks ever co-schedule into one shared overnight worktree — see Risks for the rationale and the worktree-per-task alternative.

## Outline

### Phase 1: Preflight & independent trims (tasks: 1, 2, 3, 4, 5, 6)
**Goal**: Confirm the in-scope set has not drifted (down to per-id set identity), then apply every independent (non-interacting) candidate per file-family behind the strengthened gate: affected suite green PLUS the per-trim keep-list gate.
**Checkpoint**: Preflight prints count==43 / weighted==6987 / histogram match AND the per-file id sets match; each independent file-family committed with its mirror; `master_candidates.json`/`dup_groups.json` still byte-unchanged.

### Phase 2: Interacting & high-risk candidates (tasks: 7, 8, 9, 10, 11, 12)
**Goal**: Apply the seven collision clusters as single atomic edits with post-apply joint re-reads, preserving section-boundary invariants and adjusting the shared `RULE_CARRIERS` fixture without letting that adjustment become the sole guard.
**Checkpoint**: Each cluster is one commit whose post-trim combined text still carries the load-bearing content each member claimed lived "elsewhere"; `test_backlog_author`, `test_discovery_gate_presentation`, `test_load_requirements_protocol` green; the keep-list gate passes for every applied trim.

### Phase 3: Refutation recording & parity backstop (tasks: 13, 14, 15)
**Goal**: Realign the CI mirror-parity tuple with the justfile, record every candidate's applied/refuted outcome for the deferred reconciliation, and pass the commit-time lints (working-tree/audit mode) plus the full suite.
**Checkpoint**: `outcomes.md` has one APPLIED/REFUTED line per in-scope candidate and its id set equals the re-derived ledger id set; `test_dual_source_reference_parity` green with four names added; `just test` and the four commit-time lints (audit-mode) exit 0.

## Risks
- **Edit tasks form a single deliberately-linear dependency chain (1→2→…→15), not a parallel fan-out.** `just build-plugin` regenerates ALL `plugins/cortex-core` mirrors globally and every edit task stages + commits, so co-scheduling two file-editing tasks into one shared overnight worktree (the batch dispatch mode) would race on mirror staging and the git index and could silently ship mirror drift or drop a trim (last-writer-wins). Each edit task therefore `Depends on` its predecessor — including the cluster tasks (7–12) whose real data-dependency on an earlier same-file task is thereby also satisfied transitively (Task 9 after Task 4, Task 10 after Task 5, Task 11 after Task 6). **Alternative for throughput:** the Phase-1 independent tasks (2–6) touch strictly disjoint canonical files, so an operator wanting speed can run them worktree-*isolated* per task — each gets its own index and worktree, `build-plugin` regenerates deterministically, and each isolated commit still satisfies the per-commit dual-source parity invariant (Requirement 8) and merges back cleanly. That restores the spec's Phase-1 parallelism with no parity loss; it is a throughput/complexity tradeoff, not a safety concession. The linear chain is the conservative default.
- **Pin-hit is triage, not proof.** 11 of 43 candidates carry zero curated pins and `mech_pins` are noise-dominated; the corpus's own adversarial pass refuted ~5% of already-verified candidates on provenance/consumer/runtime facts a grep cannot see. Expect a non-trivial refuted subset — refuting is a valid, recorded outcome, not a failure. The keep-list gate (Requirement 3b, now in every edit-task Verification) is what stops an over-deletion between two pinned anchors from shipping green.
- **Extractive-only + rare-path anchors.** Short-skill research warns operational anchors that guard recovery/edge paths can look dead under normal-case testing. Bias toward keeping when a span guards a recovery/edge path; never paraphrase surviving prose.
- **Section-boundary and cross-file test coupling** (`test_backlog_author` section split, `RULE_CARRIERS` in `test_load_requirements_protocol`) breaks invisibly to grep — the atomic-joint-edit + joint-re-read discipline in Tasks 8/11/12 is the guard, and Task 11's fixture edit is corroborated by an independent grep of the untouched canonical carrier so the loosened assertion is never the sole evidence.
- **Scope boundary held:** the `clarify.md ↔ refine/references/clarify.md` cross-cluster dedup group is left to #361 (opportunistic-only); the ledger JSONs stay byte-unchanged (deferred #357 reconciliation owns write-back).
