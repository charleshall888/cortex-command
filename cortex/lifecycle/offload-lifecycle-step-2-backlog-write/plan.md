# Plan: offload-lifecycle-step-2-backlog-write

> Epic #336 (last child). Spec-approved 2026-06-30; scope narrowed after critical-review (close path stays prose). Two new `cortex-lifecycle-*` verbs replace the hand-narrated index.md creation template and the lifecycle-start `cortex-update-item` write-backs.

## Overview

Two single-purpose, console-script-only verbs (Approach B from research): `cortex-lifecycle-create-index` owns the byte-faithful index.md creation template; `cortex-lifecycle-start-sync` owns the lifecycle-start `in_progress` / `lifecycle-slug` write-backs with caller-passed `--backend` routing (applying ADR-0019's structural-guard principle — with a scope-extension flagged in Risks). Each verb clones `cortex_command/lifecycle/stage_artifacts.py`'s module shape. The two reference files are thinned to single full-flag invocations; canonical blocks and the Close/Continue kept-pause survive untouched. Sequential trunk dispatch (the `cortex_command` package is editable-installed, so `just test` exercises the real code — a worktree would test stale code).
**Architectural Pattern**: layered — deterministic templating/routing moves into a CLI command layer beneath the skill-prose orchestration layer; no event flow or shared state changes.

## Risks

- **`--backlog-file` is the resolver's `filename` basename — NOT a path (critical-review A-fix).** Step 1's resolver (`resolve_item._build_json`) emits only `filename` (basename) and `backlog_filename_slug` (stem); it has **no path field**, and the spec's "Step-1-resolved path" phrasing was counterfactual. Both verbs consume that basename token: `create-index` locates the file via the canonical backlog dir (`root / cortex/backlog / <basename>`) — an earlier draft opened `root / backlog_file`, which would have missed the file and raised an uncaught `FileNotFoundError` (the A-class defect, now fixed and regression-tested in Task 1 case (e)); `start-sync` takes `Path(...).stem` for the `cortex-update-item` positional, preserving the historic `cortex-update-item <stem> …` invocation byte-for-byte (Req 7 argv test). One token, two non-conflicting consumers; no SKILL.md change needed (the verb normalizes a basename).
- **ADR-0019 scope-extension — open question for approval (critical-review C-finding).** `start-sync` applies ADR-0019's structural-guard principle but extends it past ADR-0019's stated scope ("skill-helper verbs operating over the local events.log"): it writes backlog *frontmatter* via `cortex-update-item` and gates its *entire* primary action on `--backend` (vs the `emit-lifecycle-start` shape, where the events.log write always runs and only a slug param is guarded). The spec deliberately filed **no new ADR** (three-criteria gate — "no decision meets Hard-to-reverse + Surprising + Real-trade-off beyond what ADR-0019 records"), but ADR-0019's own Trade-off section names "extension to another verb" as the hard-to-reverse cost. Surfaced as the single Ask at approval: add a brief ADR addendum (or a one-line scope note amending ADR-0019), or rely on the spec's recorded reasoning. Either way the citation in this plan is now qualified, not presented as unconditional authority.
- **`start-sync` does not clone the `try/except CortexProjectRootError` arm** of the `stage_artifacts` skeleton, because it resolves no project root (it only shells `cortex-update-item`, which resolves its own). Flagged for honesty against the spec's "clone the module shape" constraint — the divergence is structural, not a gap.
- **Stale binstub for manual runs.** The new console-scripts won't be on `PATH` until the uv-tool/editable install is refreshed. This does not affect any gate (`cortex-check-contract` AST-parses via `find_spec`; tests call `main()` directly), but a manual shell invocation of either verb before reinstall would hit the old binstub set.
