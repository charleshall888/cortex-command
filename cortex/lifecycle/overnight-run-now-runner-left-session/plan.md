# Plan: overnight-run-now-runner-left-session

## Overview
Introduce an out-of-process supervision model for the run-now cortex-CLI overnight runner: a new `recovery.py` module holding a false-positive-free pid-death predicate and a writer-authorized recovery core (transitionâ†’`paused` â†’ partial morning report â†’ env-match orphan reap â†’ clear stale `runner.pid`), surfaced through a manual `cortex overnight recover` verb and a single persistent launchd guardian, with a crash-loop resume guard. Phase 2 extends detection to the alive-but-wedged case via a runner-level heartbeat and a staleness threshold strictly above the in-process watchdog's `1800s`. The recovery core re-implements pause/report from the pure `state`/`ipc`/`report` primitives â€” it never touches the runner's in-process-lock helpers.

**Recoveryâ†”resume ordering (load-bearing).** The takeover lock does not serialize `save_state` cross-process, and `cortex overnight start`'s resume path mutates state via `interrupt.handle_interrupted_features` (`runner.py:994`) *before* it acquires that lock (`_check_concurrent_start`â†’`_acquire_takeover_lock`, `runner.py:944/1032`). So a concurrent resume can clobber a recovery in flight. The plan resolves this two ways: (a) the authoritative completion marker is a standalone atomic `recovery-complete.json` sidecar (a separate file the resume `save_state` cannot overwrite), not a `paused_reason` flip inside `overnight-state.json`; and (b) the resume-side crash-loop guard (Task 11) acquires the takeover lock and reads the sidecar + `crash_recovery_attempts` *before* `handle_interrupted_features` runs.

**Automatic-detection scope (honest framing).** On macOS, `caffeinate -i -w <runner_pid>` dies with the runner, so a hard-dead runner lets the host idle-sleep and the `StartInterval` guardian fires only on next wake. The guardian therefore delivers prompt, human-free detection for the **alive-but-wedged** case (Phase 2; caffeinate still up) and post-wake detection for the **hard-dead** case (Phase 1's primary pid-death signal); the manual `cortex overnight recover` verb is the immediate path for hard-dead. Both are strictly better than the status-quo "stuck forever," but Phase 1's automatic trigger is wake-bounded for exactly the mode its predicate targets.

## Risks
- **New long-lived process surface.** The persistent guardian is a net-new daemon (who-watches-the-watchman). Mitigated by: a single host-level agent (not per-session), the manual `recover` verb as a backstop, and a false-positive-free primary signal (pid-death). The operator may prefer to ship Phase 1's manual verb first and defer the guardian â€” the phasing supports that (Phase 2 Task 10 is the only guardian-install dependency).
- **macOS-only, sleep-bounded.** `StartInterval` does not fire while the Mac sleeps, and `caffeinate` dies with the runner, so hard-dead detection is bounded to next wake (documented, Task 16). Acceptable: still strictly better than "stuck forever," and the wedged case (the likely #308 mode) is caught promptly.
- **Reaper depends on env-marker propagation.** `CORTEX_RUNNER_CHILD` is the load-bearing discriminator (set only at the two runner spawn sites) and is the safety conjunct; `LIFECYCLE_SESSION_ID` alone is insufficient because interactive sessions carry it too. The orchestrator + SDK-leaf workers are confirmed marked; the brain-path `claude -p` *internal* subagents are runtime-unconfirmed (spec Open Decision, Task 3). If a class lacks the markers it is surfaced as un-reaped in the report, never broad-matched â€” so the failure mode is a visible un-reaped process, not a killed operator session.
- **Recoveryâ†”resume cross-process race (mitigated).** The takeover lock does not serialize `save_state` cross-process and resume mutates state before acquiring it. Resolved by a standalone atomic `recovery-complete.json` sidecar (Task 5) as the race-authoritative marker plus a resume-side guard that acquires the lock before `handle_interrupted_features` (Task 11). The wedged path additionally SIGKILLs the live runner before transitioning (Task 15).
- **Phase 2 staleness threshold is the fraught signal.** A too-aggressive threshold false-positives under load/GC pauses. Mitigated by the pinned `WEDGED_STALENESS_SECONDS = 2700` (strictly > 1800s, 900s margin) and by requiring the planning-phase heartbeat (Task 14) first.
- **report.py render change.** Task 6's load-bearing change is a new `orchestrator_crash` branch in `render_executive_summary` (setting `paused_reason` alone renders no banner). `generate_and_write_report` has no live callers; keep the branch additive so existing `budget_exhausted`/`api_rate_limit` report output is unchanged.
- **#262 breaker deliberately untouched.** Per the approved spec's Non-Requirements, no breaker change is in scope â€” the #308 non-fire was a dead-consumer problem no counting change fixes.
