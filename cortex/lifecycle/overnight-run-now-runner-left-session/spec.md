# Specification: overnight-run-now-runner-left-session

## Problem Statement

The run-now cortex-CLI overnight runner (`cortex overnight start`) can die or hang silently mid-session, leaving the session stuck in `phase: executing` with no live runner, no morning report, and orphaned worker agents still running — the third recurrence of the "left executing / silent stop" failure class (#039/#278), now on the cortex-CLI path that prior fixes (#001/#039 to the deleted `runner.sh`; #277 to the scheduled path) never covered. Every recovery primitive (`WatchdogThread`, `_cleanup`, the SIGTERM descendant-tree reaper) lives **inside** the runner process and fires only from its control flow or a delivered signal, so when the runner itself dies hard (SIGKILL/OOM) or its event loop wedges, nothing out-of-process detects it and nothing recovers. The ticket's named requirement is an **out-of-process liveness check so orchestrator death is caught without a human** — because the failure window is overnight, while the operator sleeps. The benefit: a session that loses its runner is automatically transitioned to `paused`, reaps its orphans, and produces a partial morning report — so the operator wakes to a diagnosable handoff instead of a silently-hung machine.

## Phases

- **Phase 1: Automatic out-of-process recovery via the safe death signal** (**must-have**) — a persistent host-level guardian (single launchd agent) that polls all `executing` sessions and, on the **false-positive-free runner-pid-death signal**, runs a writer-authorized recovery core (transition→`paused` → partial morning report → reap session-matched orphans → clear stale pid), plus a manual `cortex overnight recover` verb as a complementary on-demand surface and a crash-loop resume guard. This delivers the ticket's core ask — automatic detection without a human — for the case the in-process watchdog structurally cannot handle (a dead runner).
- **Phase 2: Wedged-runner (alive-but-hung) detection** (**should-have**) — extend detection to the harder alive-but-wedged case via event/heartbeat staleness, after first making that signal safe: emit a runner-level heartbeat that covers every `executing` sub-phase (closing the planning-phase blind window) and use a staleness threshold strictly greater than the in-process watchdog's `1800s` so the runner's own watchdog gets first crack. Lower priority because it is the fraught signal (false-positive risk) and the pid-death signal already covers the dominant dead-runner mode.

Priority: every Phase 1 requirement is **must-have**; every Phase 2 requirement is **should-have**. Won't-do items are in `## Non-Requirements`. The most-likely #308 mode (wedged) is addressed in Phase 2; Phase 1 alone closes the hard-dead mode and is safe to ship independently.

## Requirements

### Phase 1 — Automatic recovery via the safe death signal

1. **Runner-pid-death detection predicate (false-positive-free)**: A pure, read-only predicate flags a session for recovery when `overnight-state.json.phase == "executing"` AND `ipc.verify_runner_pid(...)` returns False (the recorded session-leader pid is gone — `verify_runner_pid` already matches `psutil.create_time` ±2s, so PID reuse is defended). This signal cannot false-positive on healthy work: a live runner's pid is always alive. It does NOT key on event-log staleness (staleness is Phase 2, after the planning-phase blind window is closed). Mirror the existing predicate shape in `fail_markers.py` (`_advisory_is_stale`/`scan_advisory_dirs`) but for the mid-round-loop `executing` case. **Acceptance**: a unit test constructs (a) executing + dead-pid → needs-recovery True, (b) executing + live-pid → False, (c) paused/complete → False; `just test` exits 0. **Phase**: Automatic recovery via the safe death signal

2. **Recovery core sequence**: A single function performs, idempotently and in order: (1) re-confirm the predicate under the lock; (2) `state.transition(state, "paused")` + set `state.paused_reason = "orchestrator_crash"` + atomic `save_state`; (3) `ipc.update_active_session_phase(session_id, "paused")` (retain the active-session pointer, do not clear it); (4) write a partial morning report (req 4); (5) reap session-matched orphans (req 5); (6) clear the stale `runner.pid` via the existing CAS `clear_runner_pid`. It re-implements pause/report from the pure primitives (`state.transition`, `save_state`, `ipc.*`, `report.generate_and_write_report`) — it must NOT call the runner's `_transition_paused`/`_generate_morning_report`, which require an in-process `RunnerCoordination` (threading locks). For the pid-death case (Phase 1) no live runner exists, so no SIGKILL-before-transition is needed; that guard belongs to the Phase 2 wedged case (req 10). **Acceptance**: an integration test drives a synthesized stuck-`executing` + dead-pid session dir through the core and asserts final `phase == "paused"`, `paused_reason == "orchestrator_crash"`, a morning report file exists, and `runner.pid` is cleared; `just test` exits 0. **Phase**: Automatic recovery via the safe death signal

3. **Idempotency + concurrency safety**: The whole detect→recover sequence runs under the existing `ipc._acquire_takeover_lock(session_dir)` flock; a second invocation on an already-recovered session is a no-op (guarded by `phase in ("paused","complete")` — `state.transition` raises on illegal re-pause — and a `recovery-complete` marker: `paused_reason == "orchestrator_crash_recovered"` or a `recovery-complete.json` sidecar). Every state write is atomic (tempfile + `os.replace`). **Acceptance**: a test invokes recovery twice on the same session and asserts the second call makes no further state change and raises no exception; `just test` exits 0. **Phase**: Automatic recovery via the safe death signal

4. **Partial morning report on the recovery path**: Recovery calls `report.generate_and_write_report(...)` in `--interrupted` mode (already external-callable and partial-safe — all inputs are optional disk paths). The report conveys: the death timestamp (last `overnight-events.log` event ts; note the gap vs last `pipeline-events.log` ts when available), the count of features left non-terminal (`running`/`pending`), and the orphan-reap outcome. Both report paths are written (`cortex/lifecycle/sessions/{id}/morning-report.md` + `cortex/lifecycle/morning-report.md`). **Acceptance**: a test runs recovery on a session with ≥1 non-terminal feature and asserts both report files exist and contain the interrupted-session banner and a non-terminal-feature count; `just test` exits 0. **Phase**: Automatic recovery via the safe death signal

5. **Orphan reaping by session-marker enumeration**: Because the runner is dead, its descendants have reparented to launchd (PID 1), so neither the in-process `_terminate_descendant_tree` walk (it walks the *caller's* children) nor a recorded-pgid `killpg` works — the codebase already rejects recorded child-PGIDs (`runner.py` `_terminate_descendant_tree` docstring: "the runner does not know in advance which PGIDs batch_runner workers will use"). The external reaper therefore enumerates candidate processes via `psutil` and selects only those matching this session — `CORTEX_RUNNER_CHILD=1` AND `LIFECYCLE_SESSION_ID == session_id` (env/cmdline inspection) — then SIGTERM → grace → SIGKILL per matched process. It MUST NOT broad-match all `claude` processes (would kill the operator's interactive sessions). Identity safety: because env-matched pids have no recorded `start_time`, the reaper re-reads each pid's `create_time` immediately before signalling and skips if the process vanished/changed between enumeration and kill (this guards the enumerate→kill TOCTOU window; it does not assert prior session membership beyond the env match, which is the identity anchor). Confirm during implementation that `CORTEX_RUNNER_CHILD`/`LIFECYCLE_SESSION_ID` propagate to BOTH leaf SDK workers (they inherit the batch_runner env via the SDK's `{**os.environ}` merge) AND the brain-path `claude -p` subagents; any class that does not carry the markers is reaped only if it is a descendant reachable another way, else surfaced in the report as un-reaped. **Acceptance**: unit tests assert (a) only env-matched (`CORTEX_RUNNER_CHILD=1` + matching `LIFECYCLE_SESSION_ID`) pids are selected and (b) a `claude` process lacking the session marker is never selected; `just test` exits 0. Reaping a live process tree is interactive/session-dependent and verified by a fixture-driven selection test, not a real kill. **Phase**: Automatic recovery via the safe death signal

6. **Persistent guardian (the automatic trigger)**: A single host-level launchd guardian (one LaunchAgent on a `StartInterval` cadence, NOT per-session) scans all sessions whose state/pointer is `executing`, applies the req-1 pid-death predicate, and invokes the req-2 recovery core for each session that needs it. One persistent agent (not per-session) avoids the install/GC-per-session problem; its install/remove verbs and its own who-watches-the-watchman posture (e.g. `KeepAlive` on the guardian) are documented. Reuse the `scheduler/macos.py` launchd machinery. **Acceptance**: a test invokes the guardian's scan entrypoint against a fixtures dir with one stuck-`executing`+dead-pid session and one healthy session and asserts only the stuck one is recovered; install/remove commands exit 0; `just test` exits 0. **Phase**: Automatic recovery via the safe death signal

7. **Manual recovery verb (complementary surface)**: `cortex overnight recover [--session <id>]` invokes the recovery core from a writer-authorized surface (NOT folded into `cortex overnight status`, which stays a read-only snapshot per `observability.md`). It is the on-demand path for when the guardian is not installed or for immediate operator-driven recovery; it self-heals or reports cleanly when there is nothing to recover. **Acceptance**: `cortex overnight recover --help` exits 0 and lists `--session`; running it against a synthesized stuck-`executing`+dead-pid fixture transitions it to `paused` (assert via state read); `just test` exits 0. **Phase**: Automatic recovery via the safe death signal

8. **runner.pid clear-on-crash**: Ensure a non-signaled runner exit does not leave a `runner.pid` whose process is gone while state stays `executing` undetected — recovery clears it (req 2 step 6), and (best-effort) broaden the runner's exit path so a non-signaled exit also attempts the clear. The standard write path already records the live session-leader pid (`runner.py:_start_session`, `os.getpid()`); no change to which pid is recorded. **Acceptance**: a test asserts that after recovery the session's `runner.pid` is cleared and `verify_runner_pid` on the prior data returns False; `just test` exits 0. **Phase**: Automatic recovery via the safe death signal

9. **Crash-loop resume guard**: A session recovered with `paused_reason="orchestrator_crash"` carries a `crash_recovery_attempts` counter on `OvernightState`; `cortex overnight start` resume refuses to auto-resume such a session once the counter exceeds a small bound (default 1) unless `--force` is passed, because the #308 trigger class (e.g. a pre-commit gate blocking every commit) is deterministic and would otherwise crash-loop. A clean pause (`budget_exhausted`/`signal`) is unaffected. **Acceptance**: a test sets `crash_recovery_attempts` above the bound and asserts resume declines without `--force` and proceeds with `--force`; `just test` exits 0. **Phase**: Automatic recovery via the safe death signal

10. **Event registration + docs**: Any new event names emitted by recovery (e.g. `orchestrator_crash_recovered`) are registered in `bin/.events-registry.md`; `docs/overnight-operations.md` (owner of the round loop) documents the out-of-process supervision model, the recovery sequence, the manual verb, and the guardian, including the **caffeinate-sleep limitation** (req 13); `docs/internals/pipeline.md` links to it. **Acceptance**: the events-registry gate passes; `grep -c "recover" docs/overnight-operations.md` ≥ 1 and `grep -c "caffeinate" docs/overnight-operations.md` ≥ 1; pre-commit exits 0. **Phase**: Automatic recovery via the safe death signal

### Phase 2 — Wedged-runner (alive-but-hung) detection

11. **Close the planning-phase heartbeat blind window**: Today `HEARTBEAT` is emitted only by `_heartbeat_loop` inside `run_batch` (the batch_runner subprocess); during the planning sub-phase (between `ROUND_START` and the batch_runner spawn) the runner only polls via `_poll_subprocess` (`time.sleep(0.1)`, no events), so `overnight-events.log` can be silent >30 min during healthy plan generation. Add a runner-level heartbeat that is written on a fixed cadence across ALL `executing` sub-phases (planning + batch), so event-log staleness becomes a valid liveness signal in every phase. **Acceptance**: a test simulating a planning-phase window with no batch heartbeat asserts the runner-level heartbeat still advances the last-event timestamp within the cadence; `just test` exits 0. **Phase**: Wedged-runner detection

12. **Staleness predicate for the wedged case (safe threshold + SIGKILL-before-transition)**: Extend detection so that `phase == "executing"` AND `verify_runner_pid` reads **alive** AND the last heartbeat (req 11) is older than a threshold **strictly greater** than the in-process watchdog's `STALL_TIMEOUT_SECONDS` (`1800s`, `runner.py:87`) flags a wedged runner — the strict-greater margin lets the in-process watchdog attempt its own graceful self-heal first, avoiding the race. Because the wedged runner is still alive and could overwrite `paused`→`executing`, recovery for this case MUST `os.kill(pid, SIGKILL)` the create_time-verified runner before transitioning. **Acceptance**: a test with executing + alive-pid + heartbeat older than the threshold asserts needs-recovery True, and with heartbeat younger than the threshold asserts False; a test asserts the wedged-recovery path SIGKILLs the recorded pid before the `transition` call; `just test` exits 0. **Phase**: Wedged-runner detection

13. **Document the caffeinate-sleep limitation**: `caffeinate -i -w <runner_pid>` (`runner.py:139`) exits when the runner dies, so a hard-dead runner lets the Mac idle-sleep and the `StartInterval` guardian fires only on wake; the guardian reliably catches the alive-but-wedged case (caffeinate still up) promptly and catches the hard-dead case on next wake (still strictly better than the status-quo "stuck forever"). Document this bound and the manual verb as the immediate path. Optionally (should-have within Phase 2) evaluate a guardian-maintained `caffeinate` while any session is `executing` to shrink the hard-dead detection latency, weighed against keeping the host awake. **Acceptance**: `grep -c "caffeinate" docs/overnight-operations.md` ≥ 1 and the limitation section names the guardian, the wake bound, and the manual verb; pre-commit exits 0. **Phase**: Wedged-runner detection

## Non-Requirements

- **No change to the #262 systemic circuit breaker.** Research's claimed "field disconnect" was incorrect: the no-exit-report finalization path already sets `FeatureResult.error` and `outcome_router.py:880` already counts it (it is in `_SYSTEMIC_ERROR_TYPES`) for the `total_commits==0` case. The `total_commits>0` "bookkeeping-only" behavior is a deliberate #262 design (commit `ca89f643`) — making it count would require converting `completed→paused`, regressing #262 and abandoning legitimately-completed features. And the #308 breaker non-fire was a dead-consumer problem (the orchestrator stopped before consuming round-2 results), which no counting change fixes. Any genuine breaker hardening (e.g. the events-vs-feature-results granularity / threshold question) is orthogonal to #308 and belongs in a separate ticket. (Scope change from the Clarify/Research full-scope choice — see `## Changes to Existing Behavior` and the approval note; raised because critical review found the premise factually wrong.)
- **Not fixing the wild-light pre-commit gate** that blocked all worker commits — the upstream trigger, in the wild-light backlog.
- **Not the opaque-failed-task-output capture** (companion ticket #309).
- **Not changing which pid `runner.pid` records** — it already records the live session-leader pid.
- **Not guaranteeing in-flight `git commit` completion during reaping** — see the Edge Case; the rule honored is "do not discard *committed* worktree state," not "never interrupt a commit."
- **Not migrating off file-based state** — recovery is grounded in existing file artifacts (ADR-0001).

## Edge Cases

- **Hard-dead runner (SIGKILL/OOM)**: `verify_runner_pid` reads dead → req-1 predicate fires → recovery runs. No false-positive risk (a live runner reads alive).
- **Alive-but-wedged runner (likely #308 mode)**: pid reads alive; caught only by the Phase 2 staleness predicate (req 12) after the planning-phase blind window is closed (req 11) and with a threshold strictly > 1800s; recovery SIGKILLs the wedged runner before transitioning so it cannot overwrite `paused`→`executing`.
- **Healthy long planning round**: must NOT be recovered. Phase 1 (pid-death) is immune (pid alive). Phase 2 staleness is made immune by req 11 (runner-level heartbeat in the planning phase) + req 12 (threshold > the in-process watchdog timeout).
- **Reaping vs in-flight git commit**: `killpg`/SIGTERM to a matched process group also signals an in-flight `git`/`gpg` child, so a grace window reduces but does not guarantee commit completion (with `commit.gpgsign=true` a commit may block on gpg-agent past the grace). The honored invariant is preserving *committed* worktree state; reaped features are routed through `handle_interrupted_features` on resume (preserves worktree commits + per-feature recovery counters). Truncating an uncommitted in-flight commit is accepted and surfaced in the report.
- **Concurrent recovery vs a runner coming up / being cancelled**: the takeover lock serializes; recovery re-confirms the predicate under the lock before any destructive step.
- **PID reuse in the env-match reaper**: each kill re-reads `create_time` immediately before signalling to guard the enumerate→kill window; the env markers are the session-identity anchor.
- **Guardian fires while the Mac is asleep**: it does not (StartInterval); hard-dead recovery is bounded to next wake (req 13). The manual verb is the immediate path.
- **Deterministic crash trigger**: bounded by the crash-loop guard (req 9) rather than auto-resuming into the same crash.
- **Brain-path `claude -p` subagents lacking the session markers**: if a worker class does not carry `CORTEX_RUNNER_CHILD`/`LIFECYCLE_SESSION_ID`, it is not env-matched; the reaper surfaces un-reaped processes in the report rather than broad-matching.

## Changes to Existing Behavior

- **ADDED**: a single persistent launchd guardian + its install/remove verbs (the automatic out-of-process detector).
- **ADDED**: `cortex overnight recover [--session <id>]` manual writer verb.
- **ADDED** (Phase 2): a runner-level heartbeat covering all `executing` sub-phases.
- **MODIFIED**: `cortex overnight start` resume refuses to auto-resume an `orchestrator_crash`-paused session past the crash-loop bound without `--force`.
- **MODIFIED**: a non-signaled runner exit now best-effort clears `runner.pid` (was cleared only on clean shutdown / signal).
- **MODIFIED**: `OvernightState` gains `crash_recovery_attempts` and the `orchestrator_crash`(`_recovered`) `paused_reason` value(s).
- **REMOVED (from prior spec draft, not from the system)**: the #262 breaker-wiring change — dropped after critical review found its premise factually wrong (see Non-Requirements). No behavioral change to the breaker.

## Technical Constraints

- Session phases are `planning|executing|complete|paused|starting` (`state.py:40`) — no `failed` session phase; the only legal escape from `executing` is `→ paused`.
- Observability surfaces (`status`, dashboard, MCP) are read-only w.r.t. session state (`observability.md:93/99`); recovery writes come only from the guardian / manual verb.
- The runner's `_transition_paused`/`_generate_morning_report` take an in-process `RunnerCoordination` (threading locks) and are NOT externally reusable; recovery uses the pure primitives (`state.transition`, `save_state`, `ipc.*`, `report.generate_and_write_report`).
- Recorded child-PGIDs are an explicitly rejected approach in this codebase (`runner.py` `_terminate_descendant_tree` docstring); the external reaper uses session-marker (`CORTEX_RUNNER_CHILD` + `LIFECYCLE_SESSION_ID`) enumeration.
- `verify_runner_pid` (`ipc.py:392`) is the false-positive-free liveness primitive (magic + schema bound + `create_time` ±2s); it is the Phase 1 detection anchor.
- `_heartbeat_loop` (`orchestrator.py:462`) runs only inside `run_batch`/batch_runner — staleness is NOT a valid liveness signal until req 11 adds a runner-level heartbeat.
- The in-process watchdog `STALL_TIMEOUT_SECONDS=1800` (`runner.py:87`) is a wall-clock timer; any staleness recovery threshold must be strictly greater to avoid racing its self-heal.
- All state writes atomic (tempfile + `os.replace`); readers lock-free, safe via forward-only/idempotent transitions.
- New event names register in `bin/.events-registry.md` (`cortex-check-events-registry` gate); a `runner.pid` schema change would require bumping `schema_version` + `MAX_KNOWN_RUNNER_PID_SCHEMA_VERSION` together — avoided here (no runner.pid schema change; the guardian/reaper read existing fields).

## Open Decisions

- **Brain-path `claude -p` subagent markers**: confirm at implementation whether the orchestrator's Task-tool subagents carry `CORTEX_RUNNER_CHILD`/`LIFECYCLE_SESSION_ID` so the env-match reaper reaches them. Deferred because it requires observing the actual env propagation of SDK-spawned subagents at implementation time; the spec's fallback (surface un-reaped processes in the report) bounds the risk either way.

## Proposed ADR

### Proposed ADR: 0011-out-of-process-overnight-runner-supervision

**Context**: Every overnight-runner liveness/recovery primitive lives inside the runner process (in-process `WatchdogThread`, signal-driven `_cleanup`, descendant-tree reaper), so a runner that dies hard or wedges leaves the session stuck in `executing` with orphaned workers and no report — a recurring failure no in-process mechanism can fix, because a supervisor cannot detect its own host's death. **Decision**: introduce an out-of-process supervision model — a single persistent launchd guardian (plus a manual `cortex overnight recover` verb) that detects orchestrator death and runs a writer-authorized recovery core (transition→`paused` → partial report → reap session-matched orphans → clear pid). Detection is keyed primarily on the false-positive-free **runner-pid-death** signal (`verify_runner_pid`); event/heartbeat **staleness** for the alive-but-wedged case is a deliberately later, lower-priority signal because it requires first closing the planning-phase heartbeat blind window and using a threshold strictly above the in-process watchdog's timeout to avoid false positives and self-heal races. Recovery is re-implemented from the pure state/ipc/report primitives (the runner's helpers require in-process locks), and orphan reaping uses session-marker enumeration (the codebase explicitly rejects recorded child-PGIDs). **Trade-off**: this adds a long-lived guardian process and a second recovery code path (re-implemented, not shared); on macOS the hard-dead case is bounded to machine-wake latency because `caffeinate` dies with the runner. Accepted because in-process supervision is structurally incapable of detecting its host's death, the guardian is bounded (single host-level agent, manual-verb backstop, false-positive-free primary signal), and it stays within file-based state (ADR-0001 is "no database," not "no daemon"). Hard to reverse (process-model + persistent agent + new writer surface), surprising without context (why pid-death not staleness; why re-implement not reuse; why env-match not killpg), and a real trade-off (automatic detection vs. simplicity and the caffeinate-sleep ceiling) — meeting the three-criteria ADR gate.
