## Web Research

**Bundled `/code-review` current behavior** (source: `code.claude.com/docs/en/code-review`; strongly version-pinned — renamed from `/simplify` at v2.1.147): the *local* command defaults to terminal output, posts only with `--comment`, applies fixes with `--fix`; effort levels low→max (omitted = session effort); targets accept a file/PR-number/branch/ref-range; `ultra` runs cloud multi-agent. **`REVIEW.md` is documented as tuning the managed/cloud Code Review service** (severity redefinition, nit caps, skip rules, verification bar, re-review convergence, summary tally) — whether the *local* command reads it is **not stated** (open question). `/security-review` is a separate terminal command (HIGH-confidence newly-added vulns; warns it is not prompt-injection-hardened).

**Conventional Comments** (`conventionalcomments.org`): format `<label> [decorations]: <subject>`. Labels: praise, nitpick, suggestion, issue, todo, question, thought, chore, note (+ typo/polish/quibble). Decorations: `(non-blocking)`, `(blocking)`, `(if-minor)`. Clean drop-in vocabulary; reconcile with bundled Code Review's coarser 🔴 Important / 🟡 Nit / 🟣 Pre-existing tiers.

**Single-pass terminal-review prior art:** CodeRabbit CLI (`--plain`/`--prompt-only` terminal modes, no autopost), GitHub Copilot CLI `/review` (terminal, no post), Qodo PR-Agent (single LLM call + PR-compression for any size). **Terminal-first / no-autopost is now the industry norm** for CLI review tools → no-autopost is parity, not a differentiator. Single-LLM-call review is validated prior art.

**Deterministic-linter grounding** (the OUT-OF-SCOPE fast-follow): `tag1consulting/claude-comprehensive-review` runs semgrep/shellcheck/etc. opportunistically in parallel, `[tool]`-attributed as ground truth, silently skipping absent binaries (the gap the footer could fill). SGCR (arXiv 2512.17540) and Semgrep's own guidance both endorse blending deterministic rules + AI. Confirmed sound as a later additive capability.

**Observability footer:** no tool ships a named "dropped-findings footer" — **this differentiator appears genuinely novel.** Nearest analogs: reviewdog `-filter-mode=nofilter` (surface-don't-drop), bundled Code Review's "Additional findings" + independent severity table (don't lose findings when the inline surface rejects them).

## Engine Architecture: Wrap /code-review vs Bespoke Single-Reviewer

**What "System B" (the winning baseline) actually was — DECISIVE: a bespoke full-context agent, not `/code-review`.** measurement.md:13 and the column headers define B as "a single high-effort single-pass reviewer"; its recorded behaviors are a full-context investigating agent's (measurement.md:37 ran `uv run pytest` and `--help`; measurement-large-prs.md:71 "cloned the repo, enumerated all 217 call sites, simulated the bracket logic, and read acorn.js"). The ticket itself (line 25) says the reviewer gathers context "the way the winning baseline runs did." The recommendation must anchor here.

**Option A (wrap `/code-review`) — feasibility blocker:** there is no `SlashCommand` tool; the Skill tool launches *skills* by name, and `/code-review` is a built-in *command*, so it is not Skill-tool-invocable with structured output capture. A command runs inline at the session model/effort and returns prose, not a parseable tool result. `REVIEW.md` tunes only the cloud service (not the local command) — so the ticket's fallback phrase "wrap /code-review plus a tuned REVIEW.md" rests on a **false premise**. The only on-disk wrappable artifact is the auto-posting fan-out fork-parent — wrapping it re-adopts the deleted fan-out AND inherits unconditional autopost. *(Adversarial nuance: a `code-review` skill IS available in this session, so "wrap" is not categorically impossible — but it is not output-capturable the way a bespoke Agent dispatch is, which is what the footer + fail-loud contract + model-reporting all need.)*

**Option B (bespoke single-reviewer Agent dispatch) — full control:** a Task/Agent dispatch takes an explicit `model:` (satisfies the de-pin and lets the agent report the model that ran), returns output verbatim (footer becomes trivial post-processing), and lets the skill own the finding schema + fail-loud contract + terminal/markdown mode. This mirrors the in-repo `critical-review` dispatch+synthesis pattern (proven shape). Cost: the skill owns and maintains one reviewer prompt (see Adversarial E).

**Recommendation: Option B, with the bespoke prompt kept thin enough that swapping it for a `/code-review` invocation later is a small change, not a rewrite.** This anchors on the measured winner, delivers the three differentiators that require flow control, and keeps the pure-buy retreat reversible. **But see Adversarial F** — once no-autopost is conceded as table-stakes and cross-PR is descoped, the owned-harness case narrows, and pure-buy-plus-footer deserves explicit consideration as the *primary* design, not just the fallback. This is the central spec decision.

## Grounder Decision: In-Context vs Fixed evidence-ground.sh

**What the script does / its defects (file:line):** per-finding, verifies `quoted_text` appears on the correct diff side. Defects: exact-substring match `if q in ct` (337) → silent near-miss drops of real findings, and `evidence-not-found` is excluded from the footer (rubric.md:64); `slack=10` (216) applied as a hard window filter (249-252) — too lenient and too strict at once; multi-line consecutive-hunk machinery defeated by its own any-single-line fallback (356-361); matched_side mis-attribution via substring overlap; fatal-error branches emit success-shaped JSON with a populated `.grounded` key then `exit 1` (75-116) → setup error can masquerade as a clean empty review; fixed-name temp files → `$TMPDIR` cross-run collision. 553 lines, three languages, zero behavioral tests.

**What the measurements proved:** non-deterministic drops of correctly-quoted findings (dropped_count 5/5/2 on identical input, measurement.md:41); `$TMPDIR` collision ingesting another PR's findings (measurement.md:41); **silent erasure of the single best merge-blocking finding in 2/3 runs on cli #13459 via line-number drift** (measurement-large-prs.md:43,90). The component meant to improve grounding measurably manufactured the inconsistency.

**In-context grounding as design:** the reviewer holds the full diff; grounding becomes a finding-schema requirement + prompt decision criterion — *"confirm each finding's quoted text appears on the added side; if you cannot locate it, mark it evidence-weak and surface it in the footer with the reason, never silently drop it."* Gains: no silent near-miss drops, no `$TMPDIR` collision, no post-image-coordinate ambiguity, no fatal-branch ambiguity, −551 lines. **Cost: loses determinism** (run-to-run variance in which findings are flagged).

**Recommendation (from the angle agent): delete-and-ground-in-context.** **STRONGLY CONTESTED by the adversarial pass (challenge A) — this is over-reach relative to the ticket's "probation + calibration" framing and the evidence.** See Open Questions. If the script is kept, the prerequisite fixes are: visible evidence-weak drops, `slack` removal + `line_range` as tie-breaker not hard filter, fatal branches emit `{"status":"error"}` with no `.grounded` key, fix the self-defeating fallback + matched_side, and add the golden-fixture test suite. **Scope guard:** this decision is only about the substring matcher; deterministic *linter* grounding is a separate out-of-scope fast-follow.

**Cross-PR lookup — what it must recover:** prev-PR-comments was "the only critic angle that produced repeatable, answer-key-rewarded true positives B could not reach," on 2 of 3 large PRs (measurement-large-prs.md:69-71): cli #13459's external corroboration via follow-up PR #13568, cargo #17083's design-direction thread (#17012) and maintainer "easy to misuse" discussion (#17092).

**Feed shape (load-bearing correction):** the fetched material must enter as a **context block the reviewer reads BEFORE forming findings**, NOT as a separate post-grounded critic. The old defect: the standing prev-comments critic's findings quote code *not in the diff*, so Stage-3.5 grounding dropped them as `evidence-not-found` (report.md:47, measurement-large-prs.md:44). Feeding it as prior context is what turns B's all-3-runs dismissal of cli #13459 as "benign" into a block. Out-of-diff findings survive via an explicit grounding class (`matched_side=external` / `category:cross-pr`) verified against the *fetched material* (does the cited PR/URL exist in what we fetched?), never routed to the silent `evidence-not-found` bucket. Adversarial recommends **flag-only first, auto-trigger deferred to a measured fast-follow.** See Open Questions.

## Correctness Contract & Output Format

**Fail-loud contract — verdict set: `APPROVE` / `REQUEST_CHANGES` / `REVIEW_INCONCLUSIVE`** (recommend NO `COMMENT` verdict — a severity-keyed model carries non-blocking nuance in labels). `REVIEW_INCONCLUSIVE` fires on **zero surfaced findings AND any degradation signal**. Degradation signals in the single-reviewer world: (1) reviewer agent errored/timed-out/no parseable output; (2) diff missing or empty; (3) grounding could not complete (errored/absent/malformed/timeout); (4) metadata fetch failed. **Decisive distinction:** "grounding ran and returned zero findings" is a legitimate APPROVE; "grounding could not run" is INCONCLUSIVE. Structural derivation (control-flow, not string-match): blocking finding ⇒ REQUEST_CHANGES (dominates); else zero findings + any signal ⇒ REVIEW_INCONCLUSIVE; else APPROVE.

**Verdict-vs-label fix:** collapse the 3-axis rubric to **one binary verdict-driving severity (`blocking` / not)** plus a hard grounding gate (subsumes the old `solidness` axis; the `signal` axis — the primary wobble source — is dropped). The `(blocking)` decoration is rendered *from* the severity field, so label and verdict cannot diverge. Remove `suggestion (blocking)` from the vocabulary (exactly one blocking label form: `issue (blocking):`); delete per-label caps and the alphabetical tie-break (they suppress real signal); the unreachable-praise bug dissolves.

**Output format:** terminal-first plain text by default (no `<details>`/HTML/markdown table), GitHub-markdown only when posting; findings sorted **blocking-first** then file/line; footer fields: `model` (the model that actually ran, captured at dispatch — kills the lying footer), `findings_surfaced`, `findings_dropped_total` (**includes evidence-weak drops that vanish silently today** — report.md's #1 fix), `drops_by_reason` (evidence-weak drops VISIBLE with a "possible real finding lost" note), and whether the cross-PR lookup ran. One canonical label↔decoration↔severity table owned by `output-format.md`, referenced (not re-specified) by protocol/SKILL/rubric.

## Adversarial Review

- **(A) Grounder delete-now discards independence and is unmeasured.** The script's one virtue is verifying by a *different mechanism than the one that authored the finding*; in-context grounding folds verifier into generator (self-grading conflict). The measurements scored recall vs an answer key, **never quote-fidelity**, so "in-context grounding preserves hallucination resistance" is an untested hypothesis presented as result. Real risk: the rewrite ships a hallucinated finding the model is confident in — it won't even show as evidence-weak. **Mitigation: keep the ticket's probation framing; the gating calibration must measure false-attestation rate (quote-fidelity), not recall; require a golden-fixture test for whatever grounding survives.**
- **(B) Evidence is thinner than a wholesale rewrite of a high-criticality skill warrants.** The most serious gap is single-agent *emulation* of the pipeline (measurement.md:13, measurement-large-prs.md:17) — several of A's failures ($TMPDIR collision, grounding variance) are partly emulation artifacts. Steelman to consciously reject: on cli #13459, A caught the highest-value merge-blocker (1/3) that **B dismissed as benign in all 3 runs** — depth-over-breadth's clearest counterexample; the injected bug never tested hard-bug recall. **Mitigation: record the cli #13459 false-negative as an accepted cost; add a natural-bug regression fixture before declaring victory.**
- **(C) REVIEW_INCONCLUSIVE breaks zero consumers — VERIFIED.** The overnight pipeline uses `APPROVED`/`CHANGES_REQUESTED`/`REJECTED`/`ERROR` (heavily machine-consumed: `review_dispatch.py`, `outcome_router.py`, `common.py:342-349`); `/pr-review`'s `APPROVE`/`REQUEST_CHANGES` are consumed by nothing. **But the two near-identical vocabularies are a future foot-gun** — the spec should note `/pr-review` verdicts are terminal-human-only and deliberately distinct from the overnight JSON contract.
- **(D) Cross-PR auto-trigger is the over-engineered part of a deletion-thesis rewrite.** Unmeasured thresholds; `≥2 authors` makes "auto" fire near-universally; new `matched_side=external` machinery on 2 intermittent hits / 3 PRs. **Mitigation: ship flag-only first; defer the auto-trigger heuristic to a measured fast-follow (same discipline as the linter deferral).**
- **(E) Option B re-incurs (a smaller version of) the fork-maintenance liability the audit warned about** — you own a reviewer prompt that must track model behavior (cf. epic #82 harness adaptation). Smaller than the status quo decisively; does NOT cleanly beat pure-buy where Anthropic maintains the prompt for free. **Mitigation: state the maintenance bet explicitly; keep the prompt thin enough to swap for `/code-review`.**
- **(F) Internal contradiction in the three-differentiators justification.** No-autopost is table-stakes (parity); cross-PR (per D) is unmeasured machinery possibly descoped; that leaves the **footer alone**. The owned harness then reduces to "/code-review + footer" — exactly the audit's keep-and-improve fatal weakness (report.md:141). **Mitigation: the spec must pick a lane — either commit to pure-buy-plus-footer as the primary design, or state the explicit reason Option B (output-capture / grounding control) beats footer-on-/code-review.**
- **Other:** the footer's integrity is partly hollowed under in-context grounding (self-reported drops vs mechanical drops); zero existing test coverage makes the grounding-delete decision untestable without a new golden-fixture suite; if the rewrite inlines protocol content into SKILL.md it must stay ≤500 lines.

## Open Questions

These are genuine design-shape decisions the adversarial pass surfaced. **All are deferred to the Spec phase** (structured interview + the complex/high critical-review gate that fires before spec approval) — this is written rationale per the Research Exit Gate, not an unaddressed gap. Each reshapes the spec's core and warrants a user decision:

1. **Grounder: delete-now vs probation-with-calibration?** The grounder angle recommends outright deletion; the adversarial pass argues this over-reaches the evidence (self-grading conflict; quote-fidelity never measured) and the ticket's own framing is probation. *Deferred to Spec:* decide whether the first cut (a) deletes `evidence-ground.sh` and grounds in-context with a golden-fixture hallucination-resistance test as the safety net, or (b) retains it dormant pending a quote-fidelity calibration. Practically both ship in-context grounding for the first cut; the question is whether the file is deleted now and what test pins hallucination resistance.

2. **Cross-PR lookup: flag-only first vs auto-trigger now?** The adversarial pass makes a strong case that the auto-trigger heuristic is unmeasured over-engineering in a deletion-thesis rewrite and that `≥2 authors` would fire near-universally. *Deferred to Spec:* decide whether the first cut ships flag-only (auto-trigger deferred to a measured fast-follow) or includes the auto-trigger with the proposed thresholds.

3. **Engine: Option B (bespoke owned harness) vs pure-buy-plus-footer as the PRIMARY design?** Challenge F shows that once no-autopost is table-stakes and cross-PR is potentially descoped, the owned-harness case narrows to "footer alone." *Deferred to Spec:* decide whether to build the bespoke single-reviewer harness (Option B) or make pure-buy-plus-footer (wrap output + append footer) the primary, stating the explicit differentiator that justifies the chosen lane.

4. **Does the local `/code-review` command read `REVIEW.md`?** Documented only for the cloud service. Bears on both the fallback design and any pure-buy lane. *Deferred to Spec:* resolve by checking current behavior, or scope the design to not depend on it.

Non-question corrections the spec must absorb (not open — settled by research): the ticket's "canonical-plus-mirror dual-source discipline" line is factually wrong (hand-maintained, no mirror); the 500-line size cap applies but L1-ratchet/parity do not; add the two-verdict-vocabulary guard note.
