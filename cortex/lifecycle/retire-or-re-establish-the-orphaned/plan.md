# Plan: retire-or-re-establish-the-orphaned

## Overview
Hard-delete the two fully-orphaned discovery events (`architecture_section_written` + its dead `has_why_n_justification` field/flag, and `prescriptive_check_run`) from the single-source code, registry, tests, and one stale fixture, while leaving the live `approval_checkpoint_responded` event fully intact and repointing its dangling consumer reference. The work splits along the spec's two phases: Phase 1 removes the producing code and registry rows (grep/gate-verified, because the `just test` import coupling makes the suite red until Phase 2 lands); Phase 2 reconciles the test module, fixture, and changelog (the suite is asserted green only here).

## Outline

### Phase 1: Retire orphaned event code & registry rows (tasks: 1, 2)
**Goal**: Remove both orphaned emit paths (and their now-dead private helpers) from `cortex_command/discovery.py`, and hard-delete their two `bin/.events-registry.md` rows while repointing the surviving live row's consumer cell.
**Checkpoint**: `grep` for the retired tokens (including `_STATUS_VALUES`) returns 0 in `discovery.py`; both registry gates (`just check-events-registry`, `just check-events-registry-audit`) exit 0. `just test` is **expected red in the working tree** in the interval between Task 1's deletions and Task 3's reconciliation — the test module imports the just-deleted functions at module scope, so collection fails until Task 3 lands. This is the documented import-coupling interval (spec Edge Cases / Technical Constraints), not a regression. To keep every *committed* revision green (the project's "Tests pass" quality bar), the recommended commit boundary is to land Task 1 and Task 3 in a single commit — they form the atomic import-consistent unit (Task 3 `Depends on: [1]`). See Risks for why the red interval is harmless even if they are committed separately.

### Phase 2: Reconcile tests, fixtures & changelog (tasks: 3, 4, 5)
**Goal**: Reconcile `tests/test_discovery_module.py` to the surviving single emitter, conform the one stale fixture, and record the retirement in `CHANGELOG.md`.
**Checkpoint**: `just test` exits 0 (the import coupling is resolved); no retired token survives in the test module; the stale `### Why N pieces` heading is gone from all fixtures; the `## [Unreleased]` changelog entry is present.

## Risks

- **Dead-residue scope extension beyond the spec's literal grep criteria.** The spec's Req 1 added a `_coerce_bool_namespace` grep precisely because that helper's name contains none of the event tokens, so a token-grep alone would miss leftover dead code. Three more symbols are the *identical* residue class — `_validate_architecture_payload`, `_validate_prescriptive_payload` (each with exactly one caller, the emit function being deleted), and the `_STATUS_VALUES` frozenset (consumed only by the deleted architecture validator and subparser; verified to have no other reference repo-wide, and not read by the surviving `_validate_checkpoint_payload`). None of their names contains a retired event token, so the spec's acceptance greps cannot catch them. Task 1 removes all three and verifies their absence (`_STATUS_VALUES` added to the Task 1 residue grep), treating them the same way the spec treats `_coerce_bool_namespace`. This is a deliberate, durable-version completion of "fully remove the function" consistent with the ticket's anti-residue intent — but it is broader than the spec's enumerated acceptance greps. **Operator: ratify or veto this extension at approval.** (Veto would leave two dead private validators and one orphaned constant in `discovery.py`.)
- **Knowingly-red `just test` in the working tree between Task 1 and Task 3.** Because the test module imports the deleted functions at module scope, the suite fails at collection from the moment Task 1's deletions land until Task 3 reconciles it. This is sanctioned by the spec (Edge Cases / Technical Constraints). It is harmless to *commit* through because (a) pre-commit runs gate recipes only, never the suite, so no commit is blocked, and (b) the overnight runner's `test_command` gate fires at feature-merge granularity — after Task 3 — so no merge-time gate ever observes the red interval. To keep every *committed* revision green per the project's "Tests pass" quality bar, the **recommended** boundary is to land Task 1 + Task 3 as one commit (they are the atomic import-consistent unit). Committing them separately is safe per (a)/(b) but leaves up to four working-then-committed revisions (Batch-0 Tasks 1, 2, 4, 5 dispatch concurrently) on a red-suite tree until Task 3 lands.
- **Intentional retirement of cross-emitter parity coverage only.** Task 3 retires the multi-emitter path-routing test's *parity* property (three emitters resolve the same path), which is unprovable with one surviving emitter. The test's *single-emitter* override-delegation property (that `emit_checkpoint_response` routes to the lifecycle `events.log` under an active `LIFECYCLE_SESSION_ID` rather than hardcoding the research path) is NOT dropped — Task 3 mandates a dedicated replacement test for it, because the existing checkpoint tests run with the override unset and do not cover it. The net coverage reduction is exactly the parity assertion, which is logically forced; the override property is preserved by construction, not by an optional fold.
