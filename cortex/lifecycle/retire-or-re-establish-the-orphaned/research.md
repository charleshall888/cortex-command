## Web Research

The decision maps onto three converging bodies of practice — YAGNI/speculative-generality, feature-flag-debt cleanup, and telemetry/event-schema deprecation — all of which default to **retire unused-but-unwired infrastructure**.

- **Fowler, "Yagni"** (https://martinfowler.com/bliki/Yagni.html): presumptive features carry build/delay/carry/repair costs; the "cost of carry" makes code harder to modify. The load-bearing carve-out: *"Yagni only applies to capabilities built into the software to support a presumptive feature; it does not apply to effort to make the software easier to modify."* So keeping the event is justified **only if** it is a genuine low-carry-cost malleability seam — **not** if it is dormant capability. A no-emitter event with a never-wired field is dormant capability.
- **Fowler, "Tolerant Reader"** + Postel's Law (https://martinfowler.com/bliki/TolerantReader.html): removing an event type or field is backward-compatible for append-only logs as long as readers ignore unknowns / default missing fields. **Neutralizes the archive-history objection** — the fix is reader tolerance, not schema preservation.
- **LaunchDarkly flag-debt** (https://launchdarkly.com/docs/guides/flags/technical-debt): unremoved unused flags become debt and risk "evaluating to an invalid/undesired state." Keep only *permanent* flags that provide ongoing operational control; remove *temporary/dormant* ones.
- **Schema-versioning practice** (Confluent, OpenTelemetry, Akka event-sourcing): **field removal is the cheap, low-risk in-place change**; **event-type removal is the heavier breaking-change-like move**, reserved for zero-usage + zero-credible-future. This argues field-removal (the Why-N field) is unambiguous regardless of option; whole-event removal needs confirmation of zero usage (which the codebase supplies).
- **Speculative Generality** smell (https://luzkan.github.io/smells/speculative-generality/): capability provisioned ahead of an actual consumer; "the imagined future feature rarely matches the one you actually need," so the preserved seam is usually the wrong one.

**Prior-art lean**: delete the speculative parts (definitely the unwired field; the event too unless it has a live, named purpose), rely on Tolerant Reader for log compatibility, trust that malleable code makes re-adding cheap.

## Tradeoffs & Alternatives

- **Approach A (RETIRE-EVENT)** — best maintainability, strongest philosophy fit. Removes a dead schema surface, a false producer attribution, and a dangling-ref row in one stroke. Con: delete-and-re-add cost if an audit is genuinely wanted later — but re-add is ~30 lines of a known pattern, and the current schema still encodes the retired Why-N concept, so a future re-add would want a fresh schema anyway (so "keep it to save re-add" buys a discount on work you wouldn't redo identically).
- **Approach B (RETIRE-FIELD-ONLY)** — worst maintainability. Preserves an event no live code emits whose only remaining stated purpose (R12 `re_walk_attempt`) is a closed one-shot; the producers column would have to read "no live producer," i.e. a permanent orphan blessed as intentional — textbook speculative generality. The honest case for B: `audit-affordance` is a recognized repo pattern (the sibling rows carry it). But no audit consumer has ever materialized for any of the three siblings.
- **Approach C (RE-ESTABLISH)** — highest complexity; reverses #269/#268/ADR-0007; no demonstrated demand; a product call, not a cleanup call. Correct only if a human decides the Why-N gate has genuine structural review value.
- **Dangling-ref fix scope**: **family-wide** for the surviving rows. The fix is identical (repoint `consumers` from the nonexistent `tests/test_discovery_events.py` to `tests/test_discovery_module.py (tests-only)`), the places are nameable now, and the durable-version test ("applies in multiple known places you can name") favors fixing all surviving rows rather than leaving siblings lying about their consumer.
- **Recommended approach: (a) RETIRE-EVENT** (confidence high that a > b; a vs c conditional on a product call the evidence doesn't support), **with the scope and mechanism corrections from the Adversarial review** (expand to the second orphan, hard-delete, family-wide ref fix).

## Adversarial Review

- **FM-1 (decisive): two orphans, not one.** `prescriptive_check_run` has no live emitter either — the prescriptive check is performed by `cortex-check-prescriptive-prose` (LEX-1 scanner) wired as a hard pre-commit gate (`.githooks/pre-commit:219`), not by the Python `emit-prescriptive-check` event. Its registry `producers` cell (`skills/discovery/references/decompose.md`) names the scanner, not the event literal — false, exactly like row 115. Retiring only `architecture_section_written` leaves an identically-orphaned sibling → the next #270.
- **FM-2: the MUST-escalation evidence-pipeline counter-argument collapses for the orphans.** The Why-N soft gate is gone from all discovery prose (template at `research.md:120-123` has only `### Pieces` + `### How they connect`); an event cannot supply skip-rate evidence for a gate that no longer exists. `prescriptive_check_run`'s check is now a hard hook (no "skip rate" possible). The argument holds only for `approval_checkpoint_responded`, whose gates still exist in prose and are emitted live — which is precisely why it must be **kept**.
- **FM-3: shared-test blast radius.** `test_emit_subcommands_honor_resolve_events_log_path` (`:359-413`) co-asserts all three events (`len==3`, `names == {…}`). A two-event retirement must drop both orphan emit-call sites, set `len==1`, reduce the name-set to `{"approval_checkpoint_responded"}`, and remove top-level imports at `:25,:27` (else all 16 tests fail at collection). Risk: an off-by-one `len==2` would pass while asserting an impossible state — mitigated by running `just test` (green baseline: 16 passed).
- **Assumptions verified**: no Python reader parses any of these events (grep exit 1 across `cortex_command/` minus `discovery.py`) → hard row-deletion breaks no reader. On-disk historical rows exist (`cortex/research/interactive-overnight-mode/events.log:2-3`, `cortex/lifecycle/reframe-.../events.log:11`) but are Tolerant-Reader-safe; line 11 even carries `"status": "re_walk"`, not in the current `_STATUS_VALUES` — proving they are non-reproducible historical artifacts, not live output (Fowler's "dormant capability," not a malleability seam).
- **Tombstone precedent inapplicable** → hard-delete is *more* correct here: the `decompose_flag` 30-day grandfather exists because in-flight prose sessions could still emit a just-removed event; neither orphan was ever emitted by live prose, so there are no in-flight emitters to protect.
- **Net verdict**: option (a) survives and is strengthened, with three mandatory adjustments — (1) **expand scope to both orphans** (`architecture_section_written` + `prescriptive_check_run`), keeping `approval_checkpoint_responded` fully intact; (2) **hard-delete, not tombstone**; (3) **family-wide dangling-ref fix** including repairing the surviving live sibling's consumer reference. Option (b) is dominated (would bless two permanent orphans). Option (c) is contradicted by the Why-N gate already being fully removed from prose.
