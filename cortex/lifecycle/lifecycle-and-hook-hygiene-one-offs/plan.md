# Plan: lifecycle-and-hook-hygiene-one-offs

## Overview

Three surviving sub-items from #193 ship in one PR with one commit per sub-item: (Sub-item 4) replace skill-edit-advisor's full `just test-skills` invocation with the two SKILL.md-load-bearing sub-suites and cap stdout to ≤500 chars (suffix-inclusive); (Sub-item 1) collapse lifecycle SKILL.md Step 2's four backlog-glob blocks to a single `bin/cortex-resolve-backlog-item` invocation with documented exit-code handling and prose-driven attention discipline through the four Step-2 sub-sections; (Sub-item 3) DELETE the discovery auto-scan no-topic branch — including its reference file, SKILL.md trigger phrases, fixture entry, and orphan doc reference — gated by a `deprecated-auto-scan-2026-05-11` git-tag recovery affordance (pushed to origin) and a CHANGELOG `### Removed` entry naming `/cortex-core:dev` as the replacement. Per the spec's Rollback Playbook, commits land in order **4 → 1 → 3** so the narrower advisor scope is active when Sub-item 1's lifecycle SKILL.md edits trigger the PostToolUse hook, and the recovery tag is created (and pushed) before the deletion commit lands. A one-line prose-collision backtick-wrap in a closed feature's `review.md` (covered by the `clarify_critic_amendment` waiver event already in `events.log`) is bundled into the Sub-item 4 commit as a strict prerequisite for the scoped advisor to run cleanly. Critical-review feedback applied: plugin-mirror regeneration and staging are explicit substeps in Tasks 4 and 6 (the pre-commit drift hook blocks rather than auto-stages); test↔hook contract is pinned to a fixture-shaped invocation; verification regexes are tightened to detect the absence of re-scan instructions rather than the presence of literal tokens; tag push is an explicit substep of Task 5; merge-policy guidance is recorded in Veto Surface.

## Veto Surface

- **Sub-item 1's prose-discipline contract is honor-system on both halves**, not only the frontmatter-collapse half the prior Veto Surface conceded. The CLI-collapse half is structurally enforced (the resolver is a real tool boundary) but no automatic AC verifies that the four Step-2 sub-sections actually stop instructing re-scans. The strengthened ACs (`grep -c 'do not re-scan' ≥ 4`, single-occurrence glob token check) raise the bar but remain literal-token checks — a sufficiently determined paraphrase could still satisfy them while keeping re-scan behavior in the sub-sections. If the operator wants structural enforcement, the escalation is Alt C from research: a new `bin/cortex-resolve-backlog-for-feature` CLI returning JSON for all four sub-sections, with the four sub-section blocks rewritten to consume the CLI's structured output. That exceeds the spec's "no resolver restructuring" out-of-scope clause and requires explicit scope expansion.
- **Sub-item 3's deletion is unilateral** for the no-topic invocation path; recovery via `deprecated-auto-scan-2026-05-11` is now a fetch-tag + checkout two-step rather than fully blocked. If the operator wants an easier rollback path, OPTIMIZE could be reinstated instead.
- **Sub-item 4's commit-order ordering (4 → 1 → 3) is load-bearing**. The narrower advisor scope must be in effect when Task 4's lifecycle SKILL.md edits trigger the PostToolUse hook; otherwise Task 4's edits will surface ~10s of test-skills latency + ~60-200 tokens of output per edit.
