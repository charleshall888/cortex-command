# Specification: lifecycle-implement-auto-enter-worktree-via

## Problem Statement

The C lifecycle (`lifecycle-implement-auto-enter-worktree-drop`) shipped the per-repo `branch-mode: worktree-interactive` default, deliberately deferring Approach A — mid-session auto-enter of the `interactive/{slug}` worktree via the platform `EnterWorktree(path=...)` tool — to this ticket. Today, after `create_worktree(...)` materializes the worktree, the implement phase's §1a step v shells out a `cd $(cortex-worktree-resolve interactive/{slug})` handoff. Auto-enter removes that handoff for the implement → review window, decomposing Approach A's three deferred design questions and wiring the call through. The motivating value: a user with `branch-mode: worktree-interactive` (the shipped Approach C default) gets a worktree-isolated implement-phase session with no manual `cd` step; the Complete phase remains the explicit user-exit boundary.

## Phases

- **Phase 1: Foundations** — promote ADR-0004 from `proposed` to `accepted`; land new ADR-0006 extending `cortex init`'s consumer-write surface; fix the documented `cortex-worktrees/` → `.claude/worktrees/` path drift in `implement.md` and `complete.md`.
- **Phase 2: Authorization plumbing** — extend `cortex init` to additively append a cortex-managed authorization clause to consumer `CLAUDE.md`; add `cortex init --revoke-worktree-auth` (rollback) and `cortex init --verify-worktree-auth` (precondition probe); cover with init-handler tests.
- **Phase 3: Auto-enter wiring** — add `EnterWorktree(path=...)` to `skills/lifecycle/references/implement.md` §1a after `create_worktree(...)` returns; re-order `_origin_pwd` capture and `interactive_worktree_entered` event emission; re-author §1a step vi narrative with session-exit safety guidance; add already-in-worktree precondition probe with graceful fallback; call `cortex init --verify-worktree-auth` before the `EnterWorktree` call.
- **Phase 4: Tests and observability** — parity test enumerating EnterWorktree call sites under `skills/lifecycle/`; picker-label parity test; CLAUDE.md authorization round-trip test (init writes, `--revoke` removes, `--verify` returns canonical exit codes); WorktreeCreate-bypass implementation test; manual gate-firing verification checklist; SKILL.md kept-pauses inventory unchanged.

Acceptance: `just test tests/test_init_claude_md_authorization.py` exits 0 and covers (a) absent → append, (b) present-and-current → no-op, (c) present-and-stale → replace, (d) user-edited-in-fence → replaced (not preserved). **Phase**: Authorization plumbing.

## Non-Requirements

- **`ExitWorktree` skill-side orchestration**: the skill MUST NOT call `ExitWorktree(action="keep")` or `ExitWorktree(action="remove")` proactively. The live tool schema forbids this ("Do NOT call this proactively — only when the user asks"). Decision (1)'s 1C hybrid considered in research is explicitly out of scope.
- **Same-session Complete UX stutter as a regression**: the same-session Complete re-invocation flow under auto-enter is a 3-step stutter (hard-guard fires → user runs `ExitWorktree action="keep"` → re-invoke Complete) versus the status-quo 2-step (`cd ..` → re-invoke). This is a known long-tail UX edge of the Decision (1) tradeoff — the modal Complete flow is cross-session per research finding F4. R13's hard-guard prose update teaches the user the safe exit path; the residual stutter is acceptable.
- **Overnight-runner branch**: no special overnight-context guard. Verified: `cortex_command/overnight/` does NOT read `branch-mode` from `lifecycle.config.md`, so the suppressed-picker `§1a` auto-enter path is not reachable from the overnight runner today. If overnight is later extended to honor `branch-mode`, that's the moment to revisit (per Solution-horizon principle — anchor on current knowledge).
- **`isatty`/CLAUDE_CODE_ENTRYPOINT defensive guard**: not added. The §1a path already requires lifecycle-skill invocation in interactive context; auto-enter inherits that gating.
- **Variant B (`claude --worktree=<path>` fresh-session launch)**: removed from the implement.md narrative. If users have a stated need for the fresh-session variant, a separate ticket can re-introduce it.
- **Glossary entries** for `EnterWorktree`, `ExitWorktree`, `interactive worktree`, `auto-enter`, `hard guard`: not added. `cortex/requirements/glossary.md` does not exist (open requirements gap noted but not in scope here).
- **`worktree.bgIsolation` / background-session Claude Code setting**: not configured. The lifecycle skill does not run in background-session mode today.
- **`docs/internals/sdk.md` update** stating `EnterWorktree` is now used (currently says "not used; `isolation: \"worktree\"` on Agent is the safe path in sandbox"): out of scope. A small follow-up cleanup commit is acceptable; not gated by this spec.
- **Auto-update flow integration**: the cortex-command auto-update flow (`docs/internals/auto-update.md`) is NOT extended to re-run `cortex init` post-update. The `--verify-worktree-auth` probe per R20 catches stale clauses on every §1a entry, which is the simpler durable fix than wiring upgrade-time auto-init.

## Edge Cases

- **Already-in-worktree at §1a entry** (e.g., user launched Claude Code with `--worktree=<path>`): probe via `git rev-parse --show-toplevel` vs `git rev-parse --git-common-dir`; skip `EnterWorktree`, surface a single-line diagnostic naming the detected worktree, proceed with the existing `cd`-shim handoff. R9 covers this.
- **`EnterWorktree` rejects the path** (path not in `git worktree list`, schema rejection): R10 fallback — `cd`-shim handoff with diagnostic.
- **CLAUDE.md clause absent or stale** (silent non-invocation case): R20's `--verify-worktree-auth` probe returns exit 1 or 2; R10's fallback fires with diagnostic pointing to `cortex init`.
- **User edits the in-fence clause body** (e.g., changes the path scope, deletes the word "worktree"): on next `cortex init` run, R5's "stale" predicate (`fence-version < canonical`) does NOT detect in-body edits — but the canonical fence body is regenerated every time `cortex init` is invoked with a fence-version bump. Between cortex init runs, R20's probe may pass (version matches) while the actual gate fails (body mangled). This is the "in-fence edit during a single cortex-version" gap; documented as a known limitation. Mitigation: R21 manual gate-firing test runs once per spec close; bad-body cases surface there.
- **Multi-operator coordination (two operators, two cortex versions, one repo)**: R5's version-attribute fence + "latest writer wins" policy (declared in ADR-0006 per R2(c)) resolves this — operator-B's newer `cortex init` overwrites operator-A's older fence iff version is higher. Locally-meaningful in-fence edits by operator-A are NOT preserved by design.
- **Upgrade-time clause drift (single operator)**: user runs `cortex init` on v2.9.14, upgrades to v2.10.0 with newer canonical fence body, does NOT re-run `cortex init`. R20's probe fires on every §1a entry; on stale-version (exit 2), R10's fallback fires with diagnostic pointing to `cortex init` to refresh.
- **Mid-lifecycle revocation**: user runs `cortex init --revoke-worktree-auth` while a lifecycle is in implement phase. R7's amendment refuses with exit 2 + diagnostic listing live `*.interactive.pid` files unless `--force` is passed; with `--force`, the next §1a entry's R20 probe catches the missing clause and R10's fallback fires.
- **Uninstall (`uv tool uninstall cortex-command`)**: the cortex-managed CLAUDE.md fence remains stranded. ADR-0006 section (c) per R2(c) documents this: the user is recommended to run `cortex init --revoke-worktree-auth` pre-uninstall; if not, the dead clause is harmless prose (no active skill to act on it).
- **Cross-session Complete re-invocation from inside the worktree**: existing Step 8 hard guard fires per R13 (updated to teach both exit paths). User runs `ExitWorktree action="keep"` (preferred) or `cd $(git rev-parse --show-toplevel)` and re-invokes.
- **Same-session Complete re-invocation from inside the auto-entered worktree**: same hard-guard path as cross-session (R13's update covers both); the same-session case benefits from `ExitWorktree action="keep"` clearing the keep/remove prompt at session end.
- **Concurrent sessions, different slugs**: per-feature `interactive.pid` liveness file (`cortex/lifecycle/sessions/{slug}.interactive.pid`) gates picker behavior. EnterWorktree's "Must not already be in a worktree" precondition is session-scoped; sessions remain isolated.
- **Session exit with uncommitted changes**: per the live `EnterWorktree` schema's session-exit-prompt clause (quoted in Technical Constraints), the harness prompts "keep or remove" when a session exits while still in an EnterWorktree-tracked worktree. A misclick on "remove" discards uncommitted work. R12's narrative paragraph warns the user about this at handoff time; R22 verifies the prompt is live in the deployed Claude Code version.
- **`cortex init` run on a consumer repo with existing CLAUDE.md** that has prior cortex-managed content: `ensure_claude_md_authorization` is additive-idempotent per R5; existing fenced blocks are replaced (when stale by version) or left untouched (when current); non-fenced content is never touched.
- **WorktreeCreate hook is later added against `EnterWorktree` event**: not the actual invariant — see R18's updated test which verifies `create_worktree`'s implementation rather than hook registration. If a hook IS added that fires on the lifecycle's `git worktree add` invocation, R18 catches it.

## Proposed ADR

### Proposed ADR: 0006-cortex-init-consumer-claude-md-authorization-surface

**Context**: ADR-0003 declares `~/.claude/settings.local.json::sandbox.filesystem.allowWrite` as "the only write cortex-command makes outside its own tree." Approach A's auto-enter wiring requires `EnterWorktree(path=...)` authorization in the picker-suppressed path (per-repo `branch-mode: worktree-interactive` default). The live `EnterWorktree` schema gates the tool on *"explicit instruction [...] either by the user directly, or by project instructions (CLAUDE.md / memory)."* — the picker-suppressed path has no per-invocation user instruction, so authorization must come from a CLAUDE.md or memory file. A `.claude/cortex-authorizations.md` sibling file (considered) is neither CLAUDE.md nor user-scope memory and would silently fail the gate (Adversarial F5 in `research.md`).

**Decision**: `cortex init` additively appends a fenced cortex-managed authorization clause to consumer `CLAUDE.md`. The fence carries a `version=N` attribute; replacement is gated on `fence-version < canonical-version` ("latest writer wins, no in-fence user edits respected"). The clause names the lifecycle skill, the `EnterWorktree` tool, and the `interactive/{slug}` path scope. Three subcommands manage the clause lifetime: `cortex init` (write/replace per stale predicate), `cortex init --revoke-worktree-auth` (remove; refuses on live session without `--force`), `cortex init --verify-worktree-auth` (probe with exit 0 OK / 1 absent / 2 stale). The `ensure_claude_md_authorization()` function mirrors `ensure_gitignore()`'s additive-idempotent shape.

**Lifecycle of the clause**:
- **Write**: `cortex init` writes the canonical fence (with `version=N`) when absent, or replaces when present-and-stale.
- **Verify**: lifecycle skill's §1a calls `cortex init --verify-worktree-auth` before `EnterWorktree`; non-zero exit routes to cd-shim fallback.
- **Revoke**: `cortex init --revoke-worktree-auth` removes the fence; refuses on live `*.interactive.pid` without `--force`.
- **Uninstall**: `uv tool uninstall cortex-command` leaves the fence stranded; ADR text recommends running `--revoke-worktree-auth` pre-uninstall, but documents that the post-uninstall dead clause is harmless prose (no active skill to act on it). This explicit lifecycle-of-the-clause coverage is required by R2(c).

**Three-criteria gate clearance**:
- **Hard to reverse**: existing installs that have run the new `cortex init` need backfill via `--revoke-worktree-auth` + re-init.
- **Surprising without context**: contributors expect `cortex init` to touch only `~/.claude/settings.local.json` per ADR-0003; extending to consumer `CLAUDE.md` is a real surface change.
- **Real trade-off**: the picker-fires-only alternative was considered and rejected because it degrades the Approach C convergence target (the suppressed-picker default loses the high-value auto-enter benefit); a sibling-file alternative was considered and rejected because the live schema names only `CLAUDE.md / memory`.

**Status**: `accepted` (lands in Phase 1 alongside ADR-0004 promotion).
