# Specification: Restore subscription auth for autonomous-worktree subprocess

## Problem Statement

Subscription-only users (no Anthropic Console API key) cannot launch `cortex-daytime-pipeline` from inside Claude Code's autonomous worktree dispatch path: the subprocess exits with `auth probe failed: vector=none, keychain=absent`. Commit 122037d0 replaced the bash `runner.sh â†’ claude -p` chain (where the CLI subprocess resolved auth on its own, including the Keychain subscription fallback) with the Python `claude_agent_sdk` path, which requires an explicit env-resolved auth vector. The consumer side of the existing `oauth_file` vector (`auth.py:_read_oauth_file` reads `~/.claude/personal-oauth-token` and exports it as `CLAUDE_CODE_OAUTH_TOKEN`) is fully wired â€” the gap is purely producer-side: nothing walks the user through Anthropic's currently-documented "mint a long-lived token" workflow. Add a `cortex auth` subcommand namespace that runs `claude setup-token` (Anthropic's currently-documented mint command for scripts and non-interactive subprocess use), captures the one-year OAuth token, and writes it to `~/.claude/personal-oauth-token`. Update the probe-absent stderr message to point users at the new command and ship the integration smoke test alongside, so a single Phase 1 release closes the ticket's reproducer end-to-end.

## Phases

- **Phase 1: Bootstrap + status + probe-message + integration test** â€” Implement `cortex auth bootstrap` (mint + atomic write + lock + format validation + stdin-TTY guard + heartbeat UX) and `cortex auth status` (resolved-vector diagnostic that flags shadowed vectors). Update `auth.py`'s probe-absent stderr message to reference `cortex auth bootstrap`. Add unit tests for each surface AND an integration smoke test that exercises the bootstrap â†’ resolved-vector chain end-to-end. After Phase 1 ships, the originating ticket reproducer's failure path points the user at a working remediation.
- **Phase 2: Documentation and ergonomics** â€” Update `docs/setup.md` with the subscription-auth setup section. Update `docs/overnight-operations.md` lines 667â€“697 cross-reference. Phase 2 is documentation polish; the ticket is functionally closed at Phase 1.

## Requirements

1. **New `cortex auth` subparser registered in `cortex_command/cli.py`**: The top-level `cortex` CLI gains an `auth` subparser with two verbs: `bootstrap` and `status`. Help text describes each verb in one line. **Acceptance**: `cortex auth --help` exits 0 and the output contains both `bootstrap` and `status` verb names; `grep -cE "^[[:space:]]+(bootstrap|status)" <(cortex auth --help)` â‰Ą 2. **Phase**: Phase 1.

2. **`cortex auth bootstrap` invokes `claude setup-token` and captures the token to `~/.claude/personal-oauth-token`**: Subprocess invocation uses `subprocess.Popen` or `subprocess.run` with `stdout=subprocess.PIPE, stderr=None, text=True, check=False` so the child's stderr (where `claude setup-token` prints the OAuth URL) inherits the parent's stderr and the user sees the URL directly; stdout is captured for token parsing. `capture_output=True` is explicitly NOT used (incompatible with `stderr=None`). On clean exit (returncode 0), the captured stdout is scanned line-by-line for the first line matching the token regex. **Acceptance**: with `claude setup-token` mocked to print a valid token and exit 0, running `cortex auth bootstrap` results in `~/.claude/personal-oauth-token` existing, mode 0600, containing exactly the matched token followed by a single trailing newline (`stat -f "%Lp" ~/.claude/personal-oauth-token` returns `600` on macOS). **Phase**: Phase 1.

3. **Token-format validation tolerant of Anthropic prefix-version bumps**: The captured token must match the regex `^sk-ant-oat[0-9]+-[A-Za-z0-9_-]{20,}$` â€” tolerates both Anthropic suffix-length changes AND prefix-version bumps (e.g., `oat01` â†’ `oat02`, mirroring Anthropic's documented `api01/02/03` versioning pattern). Bootstrap scans all non-blank lines of `claude setup-token` stdout for the first line matching this regex (not "last non-blank line" â€” that heuristic is defeated by trailing upgrade banners). If zero lines match, or if multiple non-adjacent lines match (suggesting unexpected output structure), exits non-zero with `error: claude setup-token output did not contain a recognizable OAuth token`. **Acceptance**: (a) with mock printing `sk-ant-oat02-XXXXXXXXXXXXXXXXXXXX` followed by a banner line `A new version of claude is available...`, bootstrap captures the token line (not the banner), writes it, exits 0. (b) with mock printing only a banner and no token line, bootstrap exits non-zero and `~/.claude/personal-oauth-token` is NOT created or modified. **Phase**: Phase 1.

4. **Atomic file write**: The token file is written via tempfile-in-same-directory + `os.replace` to guarantee no partial-write torn state. Mode is set to `0o600` on the tempfile before rename. **Acceptance**: a unit test that simulates a crash between tempfile write and replace confirms the canonical path `~/.claude/personal-oauth-token` is either unchanged or contains the complete new token â€” never a partial. **Phase**: Phase 1.

5. **Idempotence â€” always overwrite**: Re-running `cortex auth bootstrap` always re-runs `claude setup-token` and overwrites the file. No mtime check, no `--force` flag, no prompt. **Acceptance**: with a pre-existing token file containing `sk-ant-oat01-OLD-...`, running `cortex auth bootstrap` (mocked `claude setup-token` printing `sk-ant-oat01-NEW-...`) results in the file containing `sk-ant-oat01-NEW-...`; `grep -c OLD ~/.claude/personal-oauth-token` = 0. **Phase**: Phase 1.

6. **Concurrency lock with explicit cleanup**: `cortex auth bootstrap` acquires an `fcntl.LOCK_EX` on a sibling lockfile at `~/.claude/.personal-oauth-token.lock` (mode 0o600) for the duration of the mint + write, mirroring the pattern in `cortex_command/init/settings_merge.py:60â€“66`. Lock acquisition is wrapped in a `try/finally` (or context manager) that calls `fcntl.LOCK_UN` and closes the fd on every exit path â€” normal return, exception, KeyboardInterrupt. The lockfile sibling is left in place (kernel releases the advisory lock on fd close or process death; the 0-byte sibling file lingering is harmless). **Acceptance**: (a) two concurrent `cortex auth bootstrap` invocations produce a final file containing exactly one of the two minted tokens; the second observes the lock blocking it until the first completes. (b) a unit test that simulates KeyboardInterrupt during the mint phase verifies the lock fd is released (no lingering `fcntl.LOCK_EX` on the sibling file) before the exception propagates. **Phase**: Phase 1.

7. **`claude` CLI on PATH AND `setup-token` verb support check**: Before invoking `claude setup-token` for real, `cortex auth bootstrap` checks that the `claude` binary is resolvable via `shutil.which("claude")` AND probes the verb's existence by running `claude setup-token --help` with a 5s timeout, checking returncode. If `shutil.which` returns None, exits with code 2 and prints `error: 'claude' CLI not found on PATH. Install Claude Code from https://code.claude.com and retry.` to stderr. If the verb-probe fails (returncode != 0 or timeout), exits with code 2 and prints `error: 'claude setup-token' verb not supported by your claude version (got <returncode>). Upgrade Claude Code and retry.` **Acceptance**: with `PATH=` (cleared), `cortex auth bootstrap` exits 2 and stderr contains `'claude' CLI not found on PATH`. With a fixture `claude` binary that exits 2 on `setup-token --help`, bootstrap exits 2 and stderr contains `'claude setup-token' verb not supported`. **Phase**: Phase 1.

8. **`cortex auth status` reports the resolved auth vector AND flags shadowed vectors**: Invokes `auth.ensure_sdk_auth(event_log_path=None)` and prints to stdout the resolved `vector` field, the source ("environment", "settings.json apiKeyHelper", "personal-oauth-token file", or "none"), and a one-line remediation when vector is `none` (pointing at `cortex auth bootstrap`). When a non-`none` vector resolves AND lower-precedence vectors are also present (e.g., `personal-oauth-token` file exists but is being shadowed by a `CLAUDE_CODE_OAUTH_TOKEN` env var or by `apiKeyHelper`), prints an additional `shadowed: <list of also-present-but-not-resolved vectors>` line with a one-sentence "unset / remove to enable the lower-precedence vector" hint. Does NOT print the actual token, key, or any secret material. **Acceptance**: (a) with only `~/.claude/personal-oauth-token` present, status prints `vector: oauth_file` and `source: ~/.claude/personal-oauth-token`, no `shadowed:` line. (b) with `CLAUDE_CODE_OAUTH_TOKEN` exported in env AND a `personal-oauth-token` file present, status prints `vector: env_preexisting` AND a `shadowed: personal-oauth-token file` line. (c) with nothing set, prints `vector: none` followed by a remediation line containing `cortex auth bootstrap`. (d) Output never contains `sk-ant-` substring (`! cortex auth status | grep -q sk-ant`). **Phase**: Phase 1.

9. **Auth precedence enforced â€” apiKeyHelper overrides `oauth_file`**: The existing `auth.py:ensure_sdk_auth` 5-vector precedence chain (`ANTHROPIC_AUTH_TOKEN` > `ANTHROPIC_API_KEY` > `apiKeyHelper` > `CLAUDE_CODE_OAUTH_TOKEN` env > `personal-oauth-token` file) MUST continue to enforce that a configured-and-working `apiKeyHelper` resolves first, allowing per-repo API-key use via global `~/.claude/settings.{json,local.json}` to override a global subscription token file. No changes to the chain itself; the spec adds an explicit test asserting the precedence. **Acceptance**: with both `~/.claude/settings.local.json` containing a valid `apiKeyHelper` script that prints an API key AND `~/.claude/personal-oauth-token` containing a valid OAuth token, `auth.ensure_sdk_auth()` returns `{"vector": "api_key_helper", ...}` and sets `ANTHROPIC_API_KEY` (not `CLAUDE_CODE_OAUTH_TOKEN`). Test name: `test_apikeyhelper_overrides_oauth_file`. **Phase**: Phase 1.

10. **Probe-absent stderr message points at `cortex auth bootstrap`**: Update `auth.py:381` (the `probe_message` for `vector="none" + keychain="absent"`) to include a one-line remediation pointing at `cortex auth bootstrap`. Current text: `"auth_probe: vector=none, keychain=absent â€” Keychain entry not found; startup will fail"`. New text appends: ` â€” run 'cortex auth bootstrap' to mint a subscription OAuth token.` **Acceptance**: `grep -F "run 'cortex auth bootstrap'" cortex_command/overnight/auth.py` returns 1 match; existing `tests/test_runner_auth.py` and `cortex_command/overnight/tests/test_daytime_auth.py` assertions on the probe-absent stderr message are updated to match. **Phase**: Phase 1.

11. **`docs/setup.md` subscription-auth section**: Add a section titled `## Subscription Auth Setup` describing the `cortex auth bootstrap` workflow in 5â€“10 lines: when to use it (subscription user without an API key), the one-shot command, what it does, the yearly-renewal cadence, and a note that `ANTHROPIC_API_KEY` or `apiKeyHelper` take precedence when configured. **Acceptance**: `grep -F "cortex auth bootstrap" docs/setup.md` returns â‰Ą 1 match; the section explicitly notes the precedence relationship with `ANTHROPIC_API_KEY` and `apiKeyHelper`. **Phase**: Phase 2.

12. **`docs/overnight-operations.md` cross-reference**: Add a one-line link from the existing auth-resolution section (lines 667â€“697) to the new `docs/setup.md#subscription-auth-setup` anchor. No structural changes to the existing auth-resolution prose. **Acceptance**: `grep -F "subscription-auth-setup" docs/overnight-operations.md` returns â‰Ą 1 match. **Phase**: Phase 2.

13. **Integration smoke test**: A serial test (in the existing `tests/` tree) that:
    1. Sets up a clean temp `~/.claude/` via `pathlib.Path.home` patching.
    2. Mocks `claude` and `claude setup-token` via a fixture script in `tests/fixtures/` that prints a valid `sk-ant-oat01-*` token and (when invoked with `setup-token --help`) returns exit 0.
    3. Invokes `cortex auth bootstrap` end-to-end.
    4. Asserts the token file is written with correct mode and content.
    5. Invokes `auth.ensure_sdk_auth()` and asserts the resolved vector is `oauth_file`.
    6. Invokes `cortex auth status` and asserts the printed vector matches and that the absence of shadowing is correctly reported.

    Marked `@pytest.mark.serial` since it spawns real subprocesses. **Acceptance**: `pytest tests/test_auth_bootstrap_integration.py -v` exits 0. **Phase**: Phase 1.

14. **Heartbeat UX during browser flow**: Before invoking `claude setup-token`, `cortex auth bootstrap` prints to stderr: `Running 'claude setup-token' â€” complete the browser OAuth flow when it opens. (Press Ctrl-C to abort.)` so the user has a clear signal what is happening during the multi-minute block. The child's stderr (where the OAuth URL appears) inherits parent stderr per Requirement 2. **Acceptance**: bootstrap's own stderr (separate from the child's stderr) contains the literal heartbeat line before the subprocess is invoked. **Phase**: Phase 1.

15. **Stdin must be a TTY**: Before invoking `claude setup-token`, `cortex auth bootstrap` checks `sys.stdin.isatty()`. If False (e.g., piped, cron, `< /dev/null`), exits with code 2 and prints `error: 'cortex auth bootstrap' requires an interactive terminal (stdin is not a TTY). Run from an interactive shell.` to stderr. **Acceptance**: `cortex auth bootstrap < /dev/null` exits 2 with the named stderr message; no subprocess is invoked. **Phase**: Phase 1.

16. **Bootstrap surfaces stale-env shadowing post-write**: After writing the token file successfully, `cortex auth bootstrap` re-runs the auth resolution chain (via `auth.ensure_sdk_auth(event_log_path=None)` with the env state at write time) and inspects the resolved vector. If the resolved vector is NOT `oauth_file` (i.e., a higher-precedence vector wins despite the freshly-minted file), prints a warning to stderr: `warning: token file written, but resolved vector is <X> â€” your fresh subscription token will be shadowed by <X>. Run 'cortex auth status' to investigate.` Bootstrap still exits 0 (file write succeeded). **Acceptance**: with `CLAUDE_CODE_OAUTH_TOKEN=stale...` exported in env, running bootstrap (mocked to print a new token) writes the file successfully, exits 0, AND prints the named warning to stderr. **Phase**: Phase 1.

## Non-Requirements

- **No changes to `cortex_command/overnight/auth.py`'s 5-vector `ensure_sdk_auth` resolution chain.** The chain is already correct; the spec adds a producer for an existing consumer.
- **No changes to `resolve_and_probe` R3 policy.** With the bootstrap subcommand available, the probe-absent failure mode becomes a clear remediation path, not a behavioral change.
- **No changes to `daytime_pipeline.py`, `runner.py`, `readiness.py`, or `pipeline/dispatch.py`.** The bootstrap subcommand operates entirely upstream of these.
- **No new auth vector added to the precedence chain.** The `oauth_file` vector already exists; the spec only adds a way to populate it.
- **No automatic token refresh** (e.g., via direct `POST claude.ai/v1/oauth/token`). Tokens are one-year; user reruns `cortex auth bootstrap` when expired. Anthropic enforced against third-party direct-OAuth-refresh in Feb 2026; this is explicitly out of bounds.
- **No CI / headless bootstrap.** `claude setup-token` requires a browser AND interactive stdin (Requirement 15 enforces the TTY check). CI users either copy a manually-minted file from a browser-capable machine OR use `ANTHROPIC_API_KEY` (the existing precedence chain handles both). Document this in `docs/setup.md`.
- **No `cortex auth login/logout/refresh/clear` subcommands.** Only `bootstrap` and `status`. Future verbs can be added later if needed.
- **No `cortex init` integration.** Bootstrap is a separate explicit step. Folding into init is a future option if the standalone subcommand proves clunky.
- **No probe-detection fix** (`probe_keychain_presence` missing `-a "$USER"`). Filed as a separate ticket; with the bootstrap path, the happy path resolves via `oauth_file` and the probe doesn't fire.
- **No new env vars introduced.** Reuses the existing `CLAUDE_CODE_OAUTH_TOKEN` env var (set by `_read_oauth_file`).
- **No GUI prompt or interactive menu beyond what `claude setup-token` itself prints.**
- **No repo-local `.claude/settings.json` apiKeyHelper override.** `auth.py:_read_api_key_helper` reads only `~/.claude/settings.json` and `~/.claude/settings.local.json`. Per-repo API-key vs subscription-token override is achievable via per-repo `.envrc` (or equivalent) that exports `ANTHROPIC_API_KEY`, which wins via step 2 of the precedence chain. Adding repo-local `apiKeyHelper` resolution is a separate ticket if needed.
- **No real-`claude` CLI contract test in CI.** Tests mock `claude setup-token` via fixtures. Upstream contract drift (verb rename, output shape change, prefix-version bump beyond `oat[0-9]+`) is monitored manually via cortex-command's own ongoing usage; canary tests against the real Anthropic CLI are out of scope.

## Edge Cases

- **`claude setup-token` exits non-zero**: bootstrap exits with the same non-zero code, prints `error: 'claude setup-token' exited with code <N>; no token written.` to stderr, and does not create or modify the token file.
- **`claude setup-token` prints a banner line AFTER the token line** (e.g., upgrade-available notice): bootstrap's all-lines regex-scan (Requirement 3) captures the token line, ignores the banner. The "last non-blank line" heuristic is NOT used.
- **`claude setup-token` prints zero token-matching lines** (verb deprecated, output format changed): bootstrap exits non-zero, file NOT written, error message names the unexpected output.
- **`claude setup-token` prints multiple token-matching lines** (unexpected output structure): bootstrap exits non-zero with `error: claude setup-token output contained multiple OAuth-token candidate lines; refusing to guess`. Defends against future ambiguity.
- **Anthropic bumps the OAuth-token prefix to `oat02-...`**: Requirement 3's regex `^sk-ant-oat[0-9]+-[A-Za-z0-9_-]{20,}$` tolerates the bump; bootstrap accepts the new prefix and writes the file. No spec change required.
- **`claude setup-token` hangs (user never completes browser flow)**: bootstrap does NOT impose its own timeout â€” the browser flow legitimately takes several minutes. The user can Ctrl-C; SIGINT reaches both the parent Python process AND the `claude` subprocess because both share the parent's process group (default `subprocess.run` behavior; `start_new_session=False` is the default and is required for this propagation). Requirement 6's `try/finally` releases the lock cleanly on Ctrl-C.
- **stdin is not a TTY** (`cortex auth bootstrap < /dev/null`, cron, non-interactive harness): Requirement 15's `sys.stdin.isatty()` check exits 2 with a clear error before invoking the subprocess.
- **`~/.claude/` does not exist**: bootstrap creates `~/.claude/` with mode 0o755 before acquiring the lock. Matches `cortex_command/init/settings_merge.py`'s pattern.
- **`~/.claude/personal-oauth-token` exists as a directory** (corrupt state): bootstrap fails with a clear error directing the user to remove the directory manually. Does not auto-delete (`shutil.rmtree`) â€” destructive operation requires explicit user action.
- **`~/.claude/` not writable** (read-only home, permission issue): bootstrap fails on lock acquisition or atomic write with the OSError message surfaced. No retry, no fallback location.
- **Concurrent `cortex auth bootstrap` invocations**: the `fcntl.LOCK_EX` on the sibling lockfile serializes them. Both produce a valid token in the final file (the latter wins); no torn write possible.
- **Prior `cortex auth bootstrap` was SIGKILL'd mid-flow**: the kernel releases the `fcntl` advisory lock automatically on process death; the lockfile sibling persists as a 0-byte file (harmless â€” recreated/reused by the next invocation). Atomic write protects the canonical token file from torn state.
- **Stale `CLAUDE_CODE_OAUTH_TOKEN` env var present at bootstrap time**: Requirement 16 detects the shadowing post-write and warns the user via stderr; bootstrap still exits 0 because the file write succeeded. User runs `cortex auth status` for full diagnosis.
- **Token file pre-existing with a stale/expired token**: bootstrap doesn't validate the existing token before mint â€” it always re-mints. The user explicitly asked for a fresh token by running the command.
- **`cortex auth status` invoked with a malformed apiKeyHelper script** (raises `_HelperInternalError`): status exits non-zero with the error surfaced, mirroring `ensure_sdk_auth`'s existing behavior.
- **`cortex auth status` invoked with `ANTHROPIC_API_KEY` exported in env AND a `personal-oauth-token` file present**: status prints `vector: env_preexisting`, source `ANTHROPIC_API_KEY environment variable`, AND `shadowed: personal-oauth-token file` with a hint to `unset ANTHROPIC_API_KEY` to fall back to the subscription token. Does not print the key value.
- **`cortex auth bootstrap` invoked under Claude Code (`CLAUDECODE=1` set)**: the `claude setup-token` subprocess inherits the user's terminal env, including `CLAUDECODE`. Anthropic's `claude` CLI handles this case (some commands warn about nested sessions); no special handling needed in cortex.
- **`claude setup-token` verb not supported by the installed `claude` version** (older or post-deprecation): Requirement 7's pre-flight verb-probe (`claude setup-token --help`) detects this and exits 2 with the named error message before the real invocation.

## Technical Constraints

- **`auth.py` stdlib-only convention**: `cortex_command/overnight/auth.py` remains stdlib-only. The new bootstrap module may import `cortex_command.overnight.auth.ensure_sdk_auth` for the status subcommand and the post-write shadowing check (Requirement 16) but must not depend on `claude_agent_sdk` at module level.
- **One converged auth chain (122037d0 intent)**: The bootstrap subcommand operates *upstream* of `resolve_and_probe` â€” it writes the file that the existing `oauth_file` vector reads. No second execution path or branching in the dispatch hot path.
- **5-vector resolution chain** (per `auth.py:ensure_sdk_auth`): `ANTHROPIC_AUTH_TOKEN` (env) â†’ `ANTHROPIC_API_KEY` (env) â†’ `apiKeyHelper` (settings.json) â†’ `CLAUDE_CODE_OAUTH_TOKEN` (env) â†’ `~/.claude/personal-oauth-token` (file). The spec preserves this ordering unchanged; Requirement 8's shadowing diagnostic and Requirement 16's post-write warning surface the precedence to the user instead of changing it.
- **Atomic write pattern**: tempfile in the same directory + `os.replace()`. Same convention as `cortex_command/init/settings_merge.py:atomic_write` (see `cortex_command/init/settings_merge.py:108â€“125` for the canonical pattern).
- **Lockfile convention**: sibling-of-target naming (`.personal-oauth-token.lock` not `personal-oauth-token.lock`) so the lock survives `os.replace` inode swap. Mirrors `cortex_command/init/settings_merge.py:_lockfile_path`. **Lock cleanup**: `try/finally` (or context manager) wraps acquisition; `fcntl.LOCK_UN` + `os.close(fd)` runs on every exit path (normal return, exception, KeyboardInterrupt). Kernel-level release on process death is the SIGKILL safety net.
- **Token regex**: `^sk-ant-oat[0-9]+-[A-Za-z0-9_-]{20,}$`. Tolerates prefix-version bumps (`oat01`, `oat02`, ...) AND suffix-length changes. Tighten only if Anthropic publishes a stricter canonical shape.
- **Token parsing**: scan all non-blank lines of `claude setup-token` stdout for the first line matching the regex. Reject (non-zero exit) if zero lines match OR if multiple lines match. Do NOT use the "last non-blank line" heuristic â€” it is defeated by trailing upgrade banners.
- **Subprocess invocation for `claude setup-token`**: `subprocess.run(["claude", "setup-token"], stdout=subprocess.PIPE, stderr=None, text=True, check=False, start_new_session=False)`. `stderr=None` inherits the parent's stderr so the user sees the OAuth URL the CLI prints. `start_new_session=False` (the default) is required for Ctrl-C propagation â€” the `claude` subprocess shares the parent's process group, so terminal SIGINT reaches both. **Do NOT use `capture_output=True`** â€” it sets `stderr=subprocess.PIPE` and is mutually exclusive with the explicit `stderr=None` argument (Python raises `ValueError: stdout and stderr arguments may not be used with capture_output`).
- **No background or detached invocation** â€” `cortex auth bootstrap` is interactive (the user must complete the browser flow before the command exits).
- **Verb-probe subprocess invocation for the pre-flight check** (Requirement 7): `subprocess.run(["claude", "setup-token", "--help"], stdout=subprocess.DEVNULL, stderr=subprocess.DEVNULL, timeout=5, check=False)`. Returncode 0 = verb supported; anything else (including TimeoutExpired) = verb unsupported.
- **Test harness pattern**: Mock `claude setup-token` via a fixture script (e.g., `tests/fixtures/fake_claude_setup_token.sh`) that prints a canned token AND handles the `--help` verb probe (exit 0 with empty output). Patch PATH in the test to find the fixture first. This mirrors existing `tests/test_runner_auth.py` mock patterns for apiKeyHelper.
- **No new events in `bin/.events-registry.md`**: Bootstrap is a one-shot CLI action, not an overnight pipeline event source. The existing `auth_bootstrap` and `auth_probe` events fire normally on the next pipeline invocation after the bootstrap creates the file.
