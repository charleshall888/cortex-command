# Plan: restore-subscription-auth-for-autonomous-worktree

## Overview

Add a `cortex auth` subparser with `bootstrap` and `status` verbs in a new `cortex_command/auth/` package, wrap `claude setup-token` to mint a one-year OAuth token and write it atomically (with concurrency lock) to `~/.claude/personal-oauth-token`, and update the existing probe-absent stderr message in `cortex_command/overnight/auth.py` to point users at the bootstrap command. The 5-vector resolution chain in `auth.py:ensure_sdk_auth` is left unchanged — this work adds a *producer* for the existing `oauth_file` *consumer*. Phase 1 (Tasks 1–11) closes the originating ticket reproducer end-to-end; Phase 2 (Tasks 12–13) is documentation polish.

## Outline

### Phase 1: Module scaffold + CLI wiring (tasks: 1)
**Goal**: Create the `cortex_command/auth/` package and register the `cortex auth {bootstrap, status}` subparser in `cortex_command/cli.py` so the surface exists end-to-end (placeholder handlers exit non-zero with a "not implemented" notice).
**Checkpoint**: `cortex auth --help` exits 0 and lists both `bootstrap` and `status`; `cortex auth bootstrap` exits non-zero with a placeholder error (not a Python traceback).

### Phase 2: Bootstrap implementation (tasks: 2, 3, 4)
**Goal**: Implement `cortex auth bootstrap` end-to-end — pre-flight checks (PATH/verb-probe/TTY), heartbeat banner, `claude setup-token` subprocess invocation, regex-based token parsing, atomic write under sibling lockfile, and post-write shadowing diagnostic.
**Checkpoint**: With the fixture script masquerading as `claude` on PATH, `cortex auth bootstrap` produces `~/.claude/personal-oauth-token` mode 0600 containing a valid token followed by a single newline.

**Checkpoint**: Manual run with only the OAuth file present prints `vector: oauth_file` and no `shadowed:` line; with `CLAUDE_CODE_OAUTH_TOKEN` exported and the file present, prints `shadowed: personal-oauth-token file`.

### Task 3: Implement bootstrap mint + atomic write under sibling lockfile
- **Files**: `cortex_command/auth/bootstrap.py`
- **What**: Replace the post-pre-flight placeholder with the real mint + write flow. Acquire an `fcntl.LOCK_EX` on `~/.claude/.personal-oauth-token.lock` (mode 0o600, created via `os.open(..., O_RDWR | O_CREAT | O_CLOEXEC, 0o600)`) inside a `try/finally` that calls `fcntl.LOCK_UN` and `os.close(fd)` on every exit path. Inside the lock: invoke `subprocess.run(["claude", "setup-token"], stdout=subprocess.PIPE, stderr=None, text=True, check=False, start_new_session=False)`. On non-zero returncode → exit with the same code, print `error: 'claude setup-token' exited with code <N>; no token written. Common causes: the verb may have been renamed or its output format changed in your installed Claude Code version; run 'claude --help' to verify and consider upgrading.`, write nothing. On zero returncode: scan stdout line-by-line; collect every line matching `^sk-ant-oat[0-9]+-[A-Za-z0-9_-]{20,}$`. Zero matches → exit non-zero with `error: claude setup-token output did not contain a recognizable OAuth token (output format may have changed)`, write nothing. Multiple matches → exit non-zero with `error: claude setup-token output contained multiple OAuth-token candidate lines; refusing to guess`, write nothing. Exactly one match → write `<token>\n` to `~/.claude/personal-oauth-token` via tempfile-in-same-directory + `os.replace` with mode 0o600 set on the tempfile before rename. Ensure `~/.claude/` exists (mode 0o755) before locking. Reject if `~/.claude/personal-oauth-token` exists as a directory: exit non-zero with a clear error directing the user to remove the directory manually (no `shutil.rmtree`).
- **Depends on**: [2]
- **Complexity**: complex
- **Context**: Sibling-lockfile lock pattern at `cortex_command/init/settings_merge.py:60–82` is the canonical reference (`_lockfile_path`, `os.open` with `O_RDWR | O_CREAT | O_CLOEXEC`, `fcntl.flock(lock_fd, fcntl.LOCK_EX)`, `try/finally` releases via `fcntl.LOCK_UN` + `os.close`). Atomic-write helper: `cortex_command.common.atomic_write(path, content)` writes via tempfile + `os.replace` with `durable_fsync`. The token regex MUST be a module-level `re.compile` constant `_TOKEN_RE = re.compile(r"^sk-ant-oat[0-9]+-[A-Za-z0-9_-]{20,}$")` and MUST carry an inline `# CONTRACT:` comment noting that the regex pins the assumed output shape of Anthropic's `claude setup-token` CLI — drift in token-prefix format, banner placement, or non-token line shapes is the most likely cause of bootstrap failures. `start_new_session=False` is the default but specify explicitly so the SIGINT-propagation behavior is documented in code. `stderr=None` is mutually exclusive with `capture_output=True` — do NOT use `capture_output`.

## Risks

- **Lock-cleanup test for KeyboardInterrupt (Task 9 sub-test j)**: `KeyboardInterrupt` propagation from a patched subprocess call is straightforward, but verifying "the lock fd is closed" relies on a second `fcntl.flock(LOCK_EX | LOCK_NB)` succeeding from a freshly-opened fd in the same test process. On filesystems where `fcntl` semantics are non-standard (CI sandboxes, network mounts), this assertion may produce false positives. The plan defaults to the `LOCK_NB` reacquire approach; an alternative is asserting `gc.collect()` reaped the file object via `tracemalloc` snapshots (more brittle).
- **Test-vs-real-`claude` divergence**: All Task 9 tests exercise the fixture scripts, not the real `claude setup-token`. If Anthropic ships a verb rename, output-format change, or prefix-version bump beyond `oat[0-9]+`, the unit-test suite stays green but the production path breaks. Mitigations baked into the plan: (a) Task 3 attaches a `# CONTRACT:` comment beside the regex flagging the assumed output shape; (b) Task 3's mint-failure error string names verb-contract drift as a common cause and points users at `claude --help`; (c) Task 12's docs section names the same diagnostic path. The spec's Non-Requirement #11 ("No real-`claude` CLI contract test in CI") accepts the underlying trade-off — the residual risk after these mitigations is bounded to "user gets a clear error message naming the most likely cause" rather than silent failure.
- **Probe message change (Task 6) is observable behavior**: any external consumer that greps stderr for the prior probe-absent message text will break on the new suffix. The spec authorized this change as part of the user-facing remediation surface; flagging here so reviewers can sanity-check that no skill, hook, or doc outside the spec's listed test files pins the prior text.
- **Verb-probe gate's narrow scope**: Task 2's verb-probe checks `claude setup-token --help` exit-code zero only — it does not validate the mint-mode contract. A future rename-with-stub (verb renamed but `--help` shim retained) or arg-shape change passes the gate; the failure surfaces downstream in Task 3's mint subprocess. The plan accepts this scope (the spec's Requirement 7 mandates the specific mechanism) and pairs the gate with an honest error string and Task 3's "Common causes: verb-contract drift" hint so the failure mode is at least diagnosable. A stricter gate (e.g., `claude --help | grep -q setup-token`) would require a spec amendment.
