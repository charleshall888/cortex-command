# Specification: improve-discovery-gate-presentation

## Problem Statement

`/cortex-core:discovery`'s Research→Decompose gate presents only the Architecture section (Pieces / Integration shape / Seam-level edges) — engineering scaffolding that buries the topic's actual verdict and reasoning. On a real run (`cortex/research/artifact-format-evaluation/`) the user reacted *"that architecture thing doesn't say much to me — summarize your findings"* and selected `drop`. The friction quote directly supports a gate-presentation reorder. The `drop` selection's "failure semantics" complaint is a vocabulary/perception issue that resolves once the gate stops frustrating the user (drop would only fire for genuine abandonment), with a small prose tweak on `drop`'s gate-option description providing additional belt-and-suspenders coverage. An earlier draft of this spec proposed a new `finalize-findings` gate option to handle the "research is good, no tickets needed" case; that direction was rejected as duplicative — `drop` already produces the same downstream behavior (no `decomposed.md` written, `research.md` as durable audit trail), no consumer reads the proposed `findings-only` verdict distinction, and the semantic gap is resolvable in prose. Per `CLAUDE.md` *Solution horizon* reconciliation with simplicity: the simpler fix is correct when no current consumer needs the distinction.

## Phases

- **Phase 1: Headline-finding template + gate-presentation reorder + drop-prose tweak** — discovery research.md template gains a `## Headline Finding` slot with content-population obligation; R4 gate prose leads with this slot before Architecture sub-sections; `drop`'s gate-option description rewords to remove the failure-only connotation. Structural tests verify prose conformance; runtime validation is the next applicable posture-check discovery run (subjective re-run signal, post-merge).

## Requirements

1. **`## Headline Finding` slot with content-population obligation**: The discovery research.md template at `skills/discovery/references/research.md` gains a `## Headline Finding` heading above `## Architecture`. The template's authoring guidance under the slot instructs the agent to populate it with one paragraph stating the verdict + the one or two key findings underlying it. Acceptance: (a) `grep -c '^## Headline Finding$' skills/discovery/references/research.md` ≥ 1; (b) a structural test asserts `## Headline Finding` appears above `## Architecture` in the template; (c) the template's authoring guidance under `## Headline Finding` contains a stable marker phrase the structural test can grep for, anchoring the content-population directive (e.g., a verbatim sentence the template carries instructing the agent to populate the slot with non-empty content). **Phase**: Phase 1: Headline-finding template + gate-presentation reorder + drop-prose tweak

2. **R4 gate-presentation reorder**: `skills/discovery/SKILL.md` R4 gate prose states (as a decision criterion, not a procedural sequence) that the gate's first content section is `## Headline Finding` from research.md, followed by `## Architecture` sub-sections. Acceptance: (a) `grep -c 'Headline Finding' skills/discovery/SKILL.md` ≥ 1; (b) a structural test asserts the R4 gate-prose section names `## Headline Finding` before `## Architecture` in its description. Additionally, an empty-body detector at gate-prompt-construction time inspects the slot: if the body under `## Headline Finding` is empty or whitespace-only, the agent surfaces a warning and falls back to Architecture-first presentation. Runtime validation of the soft-form prose's routing under a cold agent is deferred to the next applicable posture-check discovery run post-merge (subjective re-run signal per the user's clarify §4 answer). **Phase**: Phase 1: Headline-finding template + gate-presentation reorder + drop-prose tweak

3. **`drop`'s gate-option description rewords to neutral terminus**: `skills/discovery/SKILL.md` R4 gate enumeration changes `drop`'s description so it reads as a neutral close, not a failure. Specifically, the new description names both legitimate uses (close because research is sufficient and no tickets are warranted; close because abandoning) without privileging either, so posture-check topics can select `drop` without the "failure semantics" connotation. Acceptance: (a) `grep` confirms `drop`'s description contains a stable marker phrase indicating dual-use (e.g., explicitly names both "research sufficient, no tickets warranted" and "abandon" as legitimate uses) — the marker phrase is part of the implementation, not pre-specified here; (b) the description does not contain the words `abandon`/`failure`/`fail`/`discard` as the only descriptors of the action. **Phase**: Phase 1: Headline-finding template + gate-presentation reorder + drop-prose tweak

## Non-Requirements

- Does NOT undo #195's Architecture→Decompose pipeline; Architecture stays as decompose-phase input (the decompose-phase zero-piece `## Verdict` branch and uniform Role/Integration/Edges/Touch-points ticket template are preserved).
- Does NOT introduce topic-shape branching (Option C from research); no gate-time classification of posture-check vs ticket-decomposition vs audit.
- Does NOT remove `## Architecture` from research.md or relocate it elsewhere — Architecture remains in research.md, just no longer the gate's first content section.
- Does NOT alter `_CHECKPOINT_VALUES` — `research-decompose` remains the sole R4 checkpoint.
- Does NOT split the gate's `revise` scoping — `revise` stays scoped to Architecture only; revising the headline finding is a research-revision action (out of scope here).
- Does NOT introduce a `finalize-findings` (or any other new) closed-set response value. Earlier draft proposed it as a structural exit for "research good, no tickets needed"; that direction was rejected as duplicative with `drop` once `drop`'s prose is reworded to support neutral-terminus semantics. No new `_RESPONSE_VALUES` entry, no new helper subcommand, no new `decomposition_verdict` value, no new event registration or extension.
- Does NOT introduce a discovery-side kept-pauses inventory or parity test. Since the gate-option set is unchanged (still four options: approve / revise / drop / promote-sub-topic), the inventory-drift backstop is not load-bearing in this lifecycle. The parity-test idea is filed as a follow-up if a future lifecycle adds gate options.
- Does NOT add new event types or extend the `approval_checkpoint_responded` response enum.
- Does NOT change R13 re-run slug-collision semantics.
- Does NOT change the decompose-phase zero-piece `## Verdict` branch (existing path stays intact for topics that approve→decompose→produce zero pieces).
- Does NOT introduce a separate Recommend/Conclude phase (Option D from research) — heavier-scope alternative ruled out.
- Does NOT run a runtime dispatch-verification test before merge. An earlier draft included a content-gated transcript requirement; that direction was rejected after user pushback — the structural prose tests in Task 3 plus the next applicable posture-check discovery run (subjective re-run signal, post-merge) are the accepted validation chain. Per Solution horizon's reconciliation with simplicity, this is not a stop-gap; it is the validation depth the change earns.

## Edge Cases

- **Headline Finding section missing in research.md (heading absent)**: The gate's first content section falls back to `## Architecture` (current behavior). Agent surfaces a warning: "research.md has no `## Headline Finding` section — gate falls back to Architecture-first presentation."
- **Headline Finding section present but empty/whitespace-only**: A structural check at gate-prompt-construction time inspects the slot's content. If the body under `## Headline Finding` is empty or whitespace-only (no non-blank lines), the agent surfaces a warning: "research.md has empty `## Headline Finding` section — the slot exists but no content was authored. Gate falls back to Architecture-first presentation and recommends regenerating research." This closes the structural gap between R1's grep-only template check and the runtime gate emission.
- **`drop` semantics post-reword**: `drop` covers both "research sufficient, no tickets warranted, close discovery" and "abandon outright." Both produce the same downstream behavior (no `decomposed.md` written; research.md is the durable audit trail). The reworded prose makes both uses legitimate; the user selects `drop` whenever they want to exit without filing tickets, regardless of motive.
- **Existing decompose-phase zero-piece `## Verdict` path preserved**: A topic that goes approve → decompose → zero pieces still produces `decomposed.md` with `decomposition_verdict: zero-piece` and a `## Verdict` section per `skills/discovery/references/decompose.md:77-89`. This path is reached when the agent's Architecture genuinely names zero pieces (e.g., "fold-into-#N" or "no-tickets verdict"); it is structurally distinct from `drop` and is unchanged by this lifecycle.

## Changes to Existing Behavior

- **ADDED**: `## Headline Finding` section in `skills/discovery/references/research.md` template (one paragraph, structural slot above `## Architecture`, with explicit content-population guidance).
- **MODIFIED**: R4 gate prose in `skills/discovery/SKILL.md` — gate's first content section is `## Headline Finding`, followed by `## Architecture` sub-sections.
- **MODIFIED**: R4 gate-option description for `drop` in `skills/discovery/SKILL.md` — rewords to neutral-terminus dual-use semantics (research-sufficient close OR abandon).
- **ADDED**: gate-prompt-time empty-slot detector that warns on `## Headline Finding` present but empty.

## Technical Constraints

- **CLAUDE.md "What and Why, not How" principle**: Gate prose uses decision-criterion encoding rather than procedural narration. Reasoning: aligned with the principle in `CLAUDE.md:64-70`; less brittle against model evolution.
- **CLAUDE.md MUST-escalation policy**: Default to soft positive-routing for all new gate prose. No dispatch verification gates merge in this lifecycle; if the next applicable posture-check discovery run shows the soft form does not route correctly, escalation to MUST is filed as a follow-up with the failing run's events.log F-row citation per `CLAUDE.md:72-76`.
- **CLAUDE.md Solution horizon (reconciled with simplicity)**: Per the project's Philosophy of Work, "If no, **Complexity** above still applies — the simpler fix is correct, and speculating about future redo is itself over-engineering." The earlier-draft `finalize-findings` option was speculative complexity — no current consumer reads the proposed `findings-only` distinction, and `drop` with reworded prose covers the user-stated friction at lower cost. This spec applies that reconciliation.
- **No new event types or response values**: This lifecycle introduces zero new entries in `_RESPONSE_VALUES`, zero new `decomposition_verdict` values, and zero new `bin/.events-registry.md` rows. Existing events-registry parity check (`bin/cortex-check-events-registry --staged`) must continue to pass post-change.
- **SKILL.md size cap**: Discovery SKILL.md currently 107 lines; additions stay under the 500-line cap.
- **Evidence-thin acceptance**: The friction-run is a single instance with direct evidence for the gate-presentation reorder (the friction quote). Structural prose tests gate merge; runtime validation is the next applicable posture-check discovery run (subjective re-run signal per the user's clarify §4 answer). The dispatch-verification path was considered + dropped — see Non-Requirements.

## Open Decisions

None at spec time. The earlier draft's open questions about a new gate option, helper subcommand surface, verdict-value semantics, and `pieces_declined` field naming are all moot under this trimmed scope — the `finalize-findings` direction has been dropped.
