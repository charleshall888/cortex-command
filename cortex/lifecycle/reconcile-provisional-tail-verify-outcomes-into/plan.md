# Plan: reconcile-provisional-tail-verify-outcomes-into

## Overview
Build a single one-off scratch reconciler (`scratchpad/reconcile_363.py`, uncommitted per R10) that parses the two remaining #357 children (#359 `outcomes.md`, #360 `verdicts.md`), reconstructs/resolves their `(file,id)` keys and commit provenance, validates coverage and consistency against the ledger, and upserts 69 verdicts into `master_candidates.json` under an optimistic compare-and-swap write. The script defaults to `--dry-run` (parse + validate + report, no write) so every build task verifies before the single `--apply` mutation; Phase 2 then verifies the folded ledger, files a residual-debt successor ticket, and re-scopes + closes #363.
**Architectural Pattern**: pipeline

## Outline

### Phase 1: Fold (tasks: 1, 2, 3, 4, 5, 6)
**Goal**: Parse both children, validate keys/coverage/provenance against the live ledger, and atomically upsert the 69 verdicts.
**Checkpoint**: `master_candidates.json` shows 21 `unverified` / 221 `verified_survives` / 23 `verified_refuted`; byte-exact serialization holds; only the 69 target rows changed.

### Phase 2: Verify & discharge (tasks: 7, 8, 9)
**Goal**: Confirm post-fold integrity, track the residual debt, and discharge #363.
**Checkpoint**: Integrity check prints PASS; a new residual-debt backlog ticket exists; #363 frontmatter is `status: complete` and its re-scoped Done-when + successor id are committed.

## Risks
- **Concurrent mutation of the ledger** between now and Task 6 could shift counts or pre-fold some targets. This is expected and handled by the corrected concurrency design: the CAS retry (R9) wraps the whole `fold_once` pipeline so every attempt re-runs validation + the double-fold guard (R6) against the current ledger; the pre-flight (R7) runs first and short-circuits to a verified-consistent exit-0 when all targets are already folded; and the symmetric-diff compares against *target membership* (unverified ∪ agreeing-already-folded), so a concurrent fold converges rather than raising. Verification is target-keyed and recomputed from the child artifacts (Task 6 asserts the 69 specific `(file,id)` carry their mapped status), never a hardcoded whole-ledger aggregate — snapshot absolutes (21/221/23) are indicative only.
- **Scratch-script location.** The reconciler and integrity checker live in `scratchpad/` (uncommitted) to match #358/#361 precedent and satisfy R10. `scratchpad/` is not gitignored, so Task 9 stages explicit paths only and asserts no `scratchpad/` file entered the commit. A committed CLI verb was rejected as over-built (research Tradeoffs B).
- **`corrected_in_commit` is a first-of-its-kind key** (R5 option (b), resolved at approval). It preserves the correction row's edit provenance without a new status enum value. No code reads the ledger, so runtime is safe today; the *future-audit* risk raised by Adversarial #8 (a `verified_refuted` row's stale trim being re-proposed against corrected text) is mitigated by a code-comment convention plus the Task 8 successor ticket's residual (c), which directs future re-proposal scans to treat `corrected_in_commit`-bearing rows as already-actioned.
