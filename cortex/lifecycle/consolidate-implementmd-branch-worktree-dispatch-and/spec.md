# Specification: consolidate-implementmd-branch-worktree-dispatch-and

## Problem Statement

`skills/lifecycle/references/implement.md` is the largest lifecycle reference file (~25KB / 329 lines), and its §1/§1a branch/worktree pre-flight re-narrates a decision tree the CLI verbs already resolve, then embeds a 33-line `python3 - <<'EOF'` heredoc — a script masquerading as a reference — to check "is the worktree path inside the repo." This pollutes runtime context every implement phase and keeps deterministic logic in prose where the model must hand-execute it. Folding the heredoc's check into `cortex-worktree-create` (the epic 336 thesis: remove prose for deterministic outcomes) and trimming the redundant narration removes that prose, while the consolidation also fixes a live `interactive/{slug}` (slash) path-resolution bug in §1a step v. Beneficiary: every lifecycle implement run (interactive and overnight) gets a smaller, correct, harder-to-misexecute reference.

## Phases

- **Phase 1: CLI verb side** — extract the inside-repo check as a reusable `worktree.py` helper (canonicalizing both sides), fold it into `create_worktree` so it gates **every** return path, and add the discriminating exit-contract tests. Must land before Phase 2.
- **Phase 2: implement.md consolidation** — delete the heredoc, trim the redundant §1/§1a narration to route on verb outputs, fix the step-v dash/slash bug, correct the stale guardrail, update kept-pauses, regenerate the mirror.

## Requirements

**Priority (MoSCoW).** This is a correctness-and-guardrail-dominated refactor, so most items are genuinely Must-have. **Must** = R1, R2, R3, R5, R7, R8, R9, R10, R11, R12 — any miss is a regression, a broken bug-fix, a violated lock, or a containment hole. **Should** = R4 (the emit-gating guard — valuable defense-in-depth, but the gating is already enforced by R2/R3's exit-contract tests) and R6's narration-trim *depth beyond the mandatory heredoc removal* (R5 removes the heredoc unconditionally; how far the surrounding narration is consolidated scales the value but a modest trim with all locks intact is an acceptable floor). **Won't-do** = the Non-Requirements section.

1. **Inside-repo helper canonicalizes both sides.** A new pure helper in `cortex_command/pipeline/worktree.py` performs the containment check on the **resolved** worktree path AND the **resolved** repo root. Today `_repo_root()` (worktree.py:56) returns an unresolved `Path(result.stdout.strip())` while `resolve_worktree_root` (:167) returns a `.resolve()`d path — the asymmetry must be closed by resolving the repo side in the helper. Use `Path.resolve()` on both, or `os.path.commonpath` over resolved operands — never `startswith`/`commonprefix`. Acceptance: a unit test with a symlinked repo root (or a `/tmp`→`/private/tmp`-style symlink) asserts the legitimate same-repo case is NOT flagged as an escape (`just test` exits 0 with the new test present). **Phase**: CLI verb side
2. **Containment check folded into `create_worktree`, gating EVERY return path.** The check runs **after `worktree_path` and `repo` are computed and resolved (worktree.py ~:199–201) and BEFORE the idempotent `worktree_path.exists()` branch (:204) and the `git worktree add` branch (:260)** — so it gates the two idempotent early-returns (:231, :244) AND the fresh-create return, not just fresh create. (A check placed only after creation is bypassed by the idempotent returns — the exact path a stale `CORTEX_WORKTREE_ROOT` worktree escapes through.) It is conditioned on the same-repo case (the `cross_repo`/`$TMPDIR` branch is exempt). On escape it raises a **containment-specific** error (distinct message/exception, e.g. `worktree_escapes_repo`, NOT the generic `worktree_creation_failed`) so the CLI wrapper returns exit 1. Because the check precedes creation, a failure leaves nothing on disk. Acceptance: the unit tests in Requirement 3. **Phase**: CLI verb side
3. **Discriminating exit-contract tests with negative controls and the idempotent path.** The inside-repo contract has NO test today (it lived only in the heredoc), so add a parametrized test covering: (a) fresh same-repo create → success/exit 0; (b) escape via `CORTEX_WORKTREE_ROOT` on a **fresh** feature → failure asserting the **containment-specific** error message (NOT merely exit 1 — a generic `git worktree add` failure at :266 also exits 1, so the assertion must distinguish them); (c) escape via a **pre-existing/registered out-of-repo worktree** (the idempotent path: pre-create + `git worktree add` registration, then call create) → failure, proving the check fires before the idempotent early-return; (d) resolver-unavailable → failure; (e) `cross_repo`/`$TMPDIR` → success (negative control, not blocked). Acceptance: `just test` exits 0 and the new test asserts all five cases including (c) the idempotent-path block and (b) the message-discriminated escape. **Phase**: CLI verb side

Acceptance: (Phase 1) Requirement 3's exit-1-on-escape test proves create halts before the session is rooted, so the downstream emit cannot fire on an escape; (Phase 2) a prose/structural check that §1a's emit-event operation still sits on the success branch only (after the precondition/`EnterWorktree` success, not before). ## Non-Requirements

- **Does NOT ship the `cortex-lifecycle-picker-options` verb.** The option-array string assembly looks offloadable, but its value-dense part is two probes (runtime `command -v` + overnight-rejection sidecar) that gate on what the agent's own Bash tool sees (PATH/sandbox/harness-execution failure) — signals a child-process verb cannot faithfully reproduce. Out of scope; a separate ticket if ever pursued.
- **Does NOT migrate the #330-owned event sites** — `batch_dispatch` and the two `phase_transition` rows (§2/§3/§4) are disjoint from 332's §1/§1a region and belong to #330. Referenced by section, not line number, because the §1/§1a trim shifts their line positions (see Technical Constraints).
- **Does NOT alter the `interactive_worktree_entered` emission** — the emit invocation is kept verbatim and emitted from inside the worktree CWD (load-bearing for `_resolve_user_project_root_from_cwd()`). 332 changes only *when* the session is rooted (the dash re-resolve), not the emitted row's bytes.
- **Does NOT flatten the `selected`/`suppressed` structural authorization gate** to prose (ADR-0008) — the trim removes decision *narration*, not the control-flow branch.
- **Does NOT add new MUST/CRITICAL/REQUIRED language** to the trimmed prose without the CLAUDE.md evidence-artifact policy; default to soft positive-routing phrasing.

## Edge Cases

- **Symlinked repo root** (macOS `/var/folders`, `/tmp`→`/private/tmp`, symlinked parent): the helper resolves both sides, so the legitimate same-repo case is not falsely flagged as an escape (Req 1).
- **Stale out-of-repo worktree from a prior bad `CORTEX_WORKTREE_ROOT` (idempotent path)**: on a later run `worktree_path.exists()` is True and `create_worktree` would take an idempotent early-return — so the containment check MUST sit before that branch (Req 2) to catch it. This is the most dangerous manifestation of the override escape and the one a post-creation check would miss.
- **`CORTEX_WORKTREE_ROOT` override pointing outside the repo (fresh feature)**: the check fires before creation → containment-specific exit 1, distinguishable from a generic create failure (Req 2, Req 3b).
- **Cross-repo / `$TMPDIR` overnight worktree**: legitimately outside `.claude/worktrees`; the same-repo conditioning means the check does not block it (Req 2, Req 3e).
- **Exit-code collapse 0/2 → 0/1**: a path-escape that previously surfaced exit 2 from the heredoc now surfaces exit 1 from create. Behaviorally identical — the skill already treats step-iii (create, exit 1) and old step-iv (heredoc, exit 2) failures the same way (surface stderr, halt §1a), and no consumer branches on the specific code (confirmed in research).
- **Containment check runs before creation**: unlike the deleted heredoc (which ran after create), a containment failure now leaves NO worktree on disk — strictly better than the heredoc's dangling-worktree behavior.

## Changes to Existing Behavior

- **MODIFIED**: `cortex-worktree-create` gains an inside-repo containment check (folded in, before the idempotent/create branch split) that raises a containment-specific error → exit 1 on a same-repo escape. Behaviorally identical halt + stderr surfacing to the old heredoc, but now covering the idempotent return paths the heredoc-after-create model would have missed.
- **MODIFIED**: §1a step v re-resolves the worktree with the dash `interactive-{slug}` form instead of the buggy slash `interactive/{slug}` form — fixes the phantom-nested-path bug. (Re-resolve, not variable propagation: `$worktree_path` does not cross Bash tool-call boundaries.)
- **REMOVED**: the 33-line step-iv `python3` heredoc in implement.md (its check now lives in `create_worktree`).
- **MODIFIED/TRIMMED**: redundant §1/§1a branch/worktree decision narration consolidated to route on verb outputs.
- **ADDED**: `create_worktree`'s inside-repo containment check and its first-ever automated, message-discriminated exit-contract tests (including the idempotent path).

## Technical Constraints

- Containment check must canonicalize both operands (resolve/commonpath) — `startswith`/`commonprefix` is a known CVE class (sibling-prefix false positives).
- **Phase 1 must land before Phase 2, and R5 (heredoc removal) is gated on R2/R3 passing** — Phase 2's heredoc deletion must not create a window where no inside-repo check exists anywhere. Prefer landing the containment check (Phase 1) and the heredoc deletion such that no commit between them has the heredoc gone without the check present.
- The step-v fix is a **re-resolve**, not variable propagation: each §1a operation is a separate Bash tool call, so a shell variable captured in step iii is empty in step v. Re-resolve consistently with the dash `interactive-{slug}` form.
- Editing the `cortex_command` package: use **sequential dispatch (not worktree)** for implementation — `just test` runs the editable install, so a worktree would verify stale code; this also means no wheel reinstall is needed between Phase 1 and Phase 2 (the editable `.pth` makes Phase 1's `worktree.py` edits live for Phase 2's test imports), provided the editable install is intact.
- Mirror regen via `just build-plugin`; commit canonical + mirror together (drift pre-commit hook). The drift hook validates canonical==mirror only per-commit-per-branch — so coordinate with #330 (same byte-identical-mirrored file): whichever of {#330, #332} lands second must **rebase onto the other before running `build-plugin`** and re-derive its line anchors, or a merge can combine canonical regions while dropping one side's mirror edits without the hook firing.
- Keep green: `cortex-check-parity` (new helper/verb wires through an in-scope reference + test), `cortex-check-skill-path` (SP001/SP002 — preserve the `${CLAUDE_SKILL_DIR}/` sidecar form), `cortex-check-bare-python-import` (L201 — do NOT introduce `import cortex_command` into implement.md), and `cortex-check-contract` (E101/E103 — any verb mention in prose carries argv/flags or a `{placeholder}`).
- Update the implement.md section references in `worktree_precondition.py` / `worktree_create_cli.py` docstrings if §1a sub-steps renumber.
