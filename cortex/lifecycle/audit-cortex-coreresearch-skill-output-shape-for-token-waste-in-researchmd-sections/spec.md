# Specification: Audit /cortex-core:research output shape and reconcile schema sources

## Problem Statement

Backlog item #185 set out to audit the drift between `skills/lifecycle/references/research.md`'s documented research-artifact schema and the section shape `/cortex-core:research` actually emits, and to update both surfaces so future emissions match a canonical schema without per-agent invention. The motivating evidence (from `research/epic-172-audit/research.md`) was a ~3,000-token-per-feature waste estimate driven by allegedly-invented sections in real `lifecycle/<feature>/research.md` artifacts.

Research and spec-phase verification produced a different picture. `skills/research/SKILL.md` Step 4 already prescribes the six sections that 18-of-19 sampled real artifacts emit. The "drift" exists in one place: `skills/lifecycle/references/research.md` documents a phantom four-section schema (`Codebase Analysis | Web Research | Dependency Verification | Open Questions`) with a `## Dependency Verification` section that appears in zero real artifacts. That phantom is also referenced by R5 in `skills/lifecycle/references/orchestrator-review.md:137`, which evaluates the Post-Research checklist against artifacts produced by `/cortex-core:research` — meaning R5 is structurally impossible to pass for the standard flow.

Spec-phase verification further confirmed that `skills/lifecycle/references/research.md` is **never loaded by any code path**. Lifecycle SKILL.md (lines 225–274) delegates the Clarify, Research, and Spec phases entirely to `/cortex-core:refine`, which in turn delegates research to `/cortex-core:research`. The only phase-reference files lifecycle SKILL.md loads at runtime are `plan.md`, `implement.md`, `review.md`, and `complete.md` (lines 269–274). The exhaustive load-status recheck during spec covered `skills/`, `docs/`, `tests/`, `cortex_command/`, `bin/`, `hooks/`, `claude/hooks/`, `.githooks/`, `justfile`, `plugins/`, `requirements/`, and overnight prompts — zero loaders in any surface. The lifecycle research template is dead documentation that survives only because nothing currently audits its load-status.

A consequence: `skills/lifecycle/references/research.md:187` contains the only canonical line ("read and follow `references/orchestrator-review.md` for the `research` phase") that would have wired research-phase artifacts to orchestrator-review's Post-Research Checklist (R1–R5). Because that container is unloaded, the entire Post-Research Checklist is already structurally orphaned today — not just R5. The fix therefore extends to hard-deleting the whole checklist, not only the R5 row.

The fix is narrower in surface than the ticket assumed (no schema change to `/cortex-core:research`) but slightly wider in cleanup scope than initially scoped (delete the orphaned Post-Research Checklist alongside the orphan template). Leave `/cortex-core:research`'s schema unchanged, add a small in-skill clarification that Step 4 is the canonical schema source, and surface the verification-criteria reframe back to the backlog item itself.

## Non-Requirements

- **Do not change the section schema prescribed in `skills/research/SKILL.md` Step 3 or Step 4.** The six-section emission (Codebase Analysis, Web Research, Requirements & Constraints, Tradeoffs & Alternatives, Adversarial Review, Open Questions) plus conditional Considerations Addressed remains exactly as today. No agent is removed; the agent-count matrix is unchanged.
- **Do not modify `skills/discovery/references/research.md`, `skills/discovery/SKILL.md`, or `skills/discovery/references/orchestrator-review.md`.** Discovery's research path is a separate ecosystem with a programmatic citation-format contract parsed by `skills/discovery/references/decompose.md`; it is explicitly out of scope for this audit. Discovery's own R5 keys off `## Web & Documentation Research` (which IS emitted), so it is not structurally orphaned and stays in place.
- **Do not add a pre-commit schema-validator, Stop-hook validator, post-synthesis validator in SKILL.md, or any new `bin/cortex-validate-*` script.** Current 18-of-19 post-prescription convergence with prose-only enforcement does not justify additional enforcement infrastructure per the "complexity earns its place" requirement.
- **Do not prose-condense existing emission patterns** (bullets-only-with-citations instead of narrative paragraphs). This was raised as a potential ~18–30% token-reduction lever in research but is deferred to a separate ticket if pursued — it is a different decision from the schema audit and would alter the voice/density of emitted artifacts beyond the current scope.
- **Do not add WHAT/WHY/HOW (or DITA task/concept/reference) classification** to prescribed sections. Adds documentation surface without a load-bearing consumer at present; deferred unless future audit needs it.
- **Do not migrate existing archived `lifecycle/<feature>/research.md` files.** Only `## Open Questions` is programmatically parsed and it is preserved across this change. (Live-tree dangling citations into the deleted lifecycle template are handled by Requirement 9, distinct from archived-artifact migration.)
- **Do not satisfy the original ticket's ≥40% per-feature research.md token reduction verification gate as originally written.** That gate was set against a misdiagnosed premise (alleged sub-agent invention) and is incompatible with keeping the load-bearing sections that research-phase evidence shows are not waste. This spec explicitly replaces the gate via the backlog-body edit in Requirement 3 (visible to humans) and the no-regression criterion in Requirement 7 (auditable for automation).
- **Do not address backlog items #184 (merge clarify+research into "investigate") or #179 (extract conditional content blocks to references) within this ticket.** Both reference `skills/lifecycle/references/research.md` and become trivially moot once it is deleted; updating them is downstream work.
- **Do not modify `bin/cortex-complexity-escalator` or its tests.** It parses only `## Open Questions`, which is preserved verbatim in `/cortex-core:research`'s emission.
- **Do not preserve R1–R4 of the Post-Research Checklist as "still valid documentation."** R1–R4 are unreachable from any code path today (the only invocation line lives inside the deleted lifecycle template) and become formally orphaned after Requirement 1. Per the project's hard-deletion preference (`requirements/project.md:23`), they are deleted alongside R5.

## Edge Cases

- **Plugin-mirror regeneration fails after deletion**: `just build-plugin` removes `plugins/cortex-core/skills/lifecycle/references/research.md` via `rsync -a --delete` per `justfile:514`. If rsync is invoked without `--delete` (or if a future justfile edit drops the flag), the mirror persists and pre-commit drift detection fires. Expected behavior: the build script's `--delete` flag is load-bearing for this change; do not invoke `just build-plugin` with a `--no-delete` variant.
- **Pre-commit hook semantics**: `.githooks/pre-commit:187` runs `git diff --quiet -- plugins/$p/`, comparing working tree to index for plugin paths. `rsync --delete` writes deletions to the working tree only; the index reflects the deletion only after a staging step. If an implementer runs `just build-plugin` followed immediately by `.githooks/pre-commit` without staging the mirror deletion, the hook will detect index-vs-working-tree drift and exit 1. Requirement 5 routes the realistic commit through `/cortex-core:commit`, which stages files via `git add` before running `git commit` (and thus before the hook fires). If for some reason the implementer invokes the hook directly outside `git commit`, they must first run `git add -u plugins/cortex-core/` to stage the rsync-driven deletions.
- **An overnight runner has a feature mid-flight whose `lifecycle/<feature>/research.md` was produced under the old phantom-template assumption**: In-flight research.md files were produced by `/cortex-core:research`, not by the lifecycle template, so they carry the actual six-section schema. Deletion of the template does not affect them. Verified by spot-checking the 19-artifact inventory: all 19 follow the SKILL.md schema, including artifacts created during prior lifecycle runs.
- **R5 (and R1–R4) operational state today**: Per `orchestrator-review.md:137`'s "If external dependencies exist" gating, R5 was valid-N/A for deps-free features and structurally unsatisfiable for deps-touching features. Independently, all R-rows in the Post-Research Checklist were unreachable from `/cortex-core:refine`-mediated lifecycles because the only invocation line lives in the deleted lifecycle template. Deletion removes both the dead-letter rule for deps-touching features and the unreachable rules for deps-free features.
- **A future ticket re-introduces external-dependency verification**: If the project later wants an enforced "external API endpoints exist and are not deprecated" check, the right place is a new check wired through a live invocation path (e.g., a new section in `skills/research/SKILL.md` Step 4 plus an updated `skills/lifecycle/references/orchestrator-review.md` checklist that is actually invoked by `/cortex-core:research`'s Step 5 or `/cortex-core:refine`'s Step 4). This spec does not block that future work — it just removes the current orphan rules.
- **Test runner discovery**: The project uses both `pytest` (Python tests in `tests/`) and shell tests via `just test`. Acceptance Requirement 6 cites both. If `just test` does not include `pytest` in its invocation chain, both must be run.
- **`tests/test_plugin_mirror_parity.py` per-phase commit risk**: The test parametrizes byte-parity over `orchestrator-review.md` for the lifecycle mirror. If an implementer commits Phase 1 (canonical edits) without first running `just build-plugin` to regenerate the mirror, the test fails between commits. Mitigation: Requirement 5 and 6 explicitly require Phase 2 to include `just build-plugin` before the commit, and the lifecycle's commit step is single-pass via `/cortex-core:commit`. Atomic per-phase commits should not split canonical edits from mirror regeneration.

## Technical Constraints

1. **Dual-source canonical-only edits** (CLAUDE.md, `requirements/project.md`): Edits go to `skills/*` only. `plugins/cortex-core/skills/*` is regenerated by `just build-plugin` (`justfile:491-523`). The dual-source pre-commit drift hook entrypoint is `.githooks/pre-commit:178-194`; `just setup-githooks` (`justfile:526-528`) wires it via `git config core.hooksPath .githooks`.
2. **SKILL.md 500-line cap** (`requirements/project.md:30`): `skills/research/SKILL.md` is 254 lines. Adding ~3 lines for the Schema authority note keeps it well under cap; no extraction to `references/` required and no size-budget-exception marker needed.
3. **MUST-escalation policy** (CLAUDE.md): The Schema authority note must use positive-routing phrasing ("Step 4 is the canonical schema source"; "downstream consumers read research.md whole-cloth and do not parse by section name except for `## Open Questions`"). No new MUST/CRITICAL/REQUIRED phrasing is introduced. The note is descriptive, not prescriptive-prohibition.
4. **Hard-deletion preference** (`requirements/project.md:23`): "Workflows that have not earned their place are removed wholesale rather than deprecated in stages. Hard-deletion is preferred over deprecation notices, tombstone skills, or env-var soft-deletes when the surface has zero downstream consumers (verified per-PR)." Applies directly to: (a) the lifecycle research template (zero loaders), and (b) the Post-Research Checklist R1–R5 (zero invocation paths). Hard-delete both.
5. **Lifecycle requirement for skills/ edits** (CLAUDE.md): All edits to `skills/research/SKILL.md`, `skills/lifecycle/references/research.md`, and `skills/lifecycle/references/orchestrator-review.md` are routed through this `/cortex-core:lifecycle` invocation. No standalone edits.
6. **No new programmatic consumers introduced**: The only programmatic parser of research.md remains `bin/cortex-complexity-escalator`, which reads only `## Open Questions`. This change does not add Stop-hooks, pre-commit validators, or test harnesses against research.md structure.
7. **Backlog body edit must preserve the inventory criterion's traceability**: Requirement 3's edit to `backlog/185-*.md`'s `## Verification` section retains the inventory criterion (the audit's deliverable) and adds a pointer to its location: `lifecycle/<slug>/research.md`'s `### Empirical inventory — 19 recent lifecycle/*/research.md artifacts` block (research.md:55–99) satisfies "≥10 research.md samples with section presence/absence and per-section token estimates."

## Acceptance

The whole-feature acceptance criterion has two parts:

(a) **Structural correctness**: a fresh `/cortex-core:research` dispatch (any topic, any tier, any criticality) produces a `research.md` whose section schema matches `skills/research/SKILL.md` Step 4's `### Output structure` block. `skills/lifecycle/references/research.md` does not exist. `skills/lifecycle/references/orchestrator-review.md` contains no `### Post-Research Checklist` heading and no R-rows in the Post-Research range. `just test` passes.

(b) **Value-claim accounting**: the change introduces no regression in the per-feature `lifecycle/<feature>/research.md` section schema — every section present in `skills/research/SKILL.md` Step 4 before this change is present after this change, and no new sections are added (Requirement 7's diff returns empty). The original ticket's ≥40% reduction premise is reframed in the backlog body itself (Requirement 3) so any human re-read of `backlog/185-*.md`'s `## Verification` section sees the supersession pointer, not the obsolete gate.

Verified by reading the resulting `research.md` headers (`grep -E '^## ' lifecycle/<test-feature>/research.md`) and confirming they match the canonical six-section list (plus conditional sections per their existing trigger rules), and by running the Requirement 7 diff command.
