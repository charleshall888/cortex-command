# Research: Offload complete.md PR-state routing + shared stage-artifacts to CLI verbs (#331)

Offload the lifecycle Complete phase's Step 7 PR-state router and the finalization staging shared by `complete.md` Step 11a and `post-refine-commit.md` into two CLI verbs — a `cortex-lifecycle-complete-route <slug>` classifier and a `cortex-lifecycle-stage-artifacts --phase {complete|refine}` verb — while keeping genuine-judgment steps as prose. Complex tier, high criticality; depends on #330. The single most consequential research output: the "byte-identical events.log rows" invariant the ticket pins is **partly mis-stated** and must be reframed before building (see Adversarial Review and Open Questions).

## Web Research

(Internal-tooling refactor — transferable prior art is uneven; flagged where thin.)

- **State machine as a CLI classifier**: the git/docker-style `argparse.add_subparsers()` dispatcher is the canonical Python idiom. Keep the decision function pure (no I/O) so it is unit-testable without the CLI; the subcommand is a thin shell that serializes the verdict. Own the exact `json.dumps` serialization rather than reaching for a framework (`jsonargparse` is overkill here).
- **Graceful degradation when `gh` is absent/offline** (best-supported angle): detect with `shutil.which("gh")` rather than relying on the call to fail (documented GUI/non-login-shell PATH footgun — relevant to launchd/overnight). Handle three failure bands distinctly, all collapsing to a degraded `unknown` state, not a crash: (a) binary missing → `FileNotFoundError`; (b) present, non-zero exit → `CalledProcessError`; (c) present, exit 0, but no network/auth → inspect output, not just returncode. Return an explicit `unavailable`/`unknown` flag (not null-that-looks-like-data). AWS Well-Architected REL05-BP01: "transform hard dependencies into soft dependencies."
- **Byte-stable serialization**: do NOT rely on `json.dumps` defaults. `separators` defaults to `(', ', ': ')` (spaced) when `indent is None`; `sort_keys=False`; `ensure_ascii=True`. For stable rows, set `separators` explicitly and decide `sort_keys` deliberately — do **not** introduce `sort_keys=True` if the current emitter uses insertion order. Match the current emitter exactly (separator, trailing newline, ASCII escaping).
- **Golden/characterization testing** (Feathers): capture current output as the oracle before refactoring, assert the new code reproduces it. Tooling: `syrupy` (note: serializes its own representation — prefer a checked-in raw fixture for literal bytes), `pytest-regressions` (`file_regression`/`data_regression`). Round-trip the actual stdout bytes, not a re-parsed object (parsing-then-comparing hides separator/whitespace/order drift — the exact thing to prevent).
- **Pitfalls extracting prose→code** (principle, thin prior art): prose branches rely on the model silently handling underspecified cases — enumerate the implicit "else" the prose never named, via characterization tests over real historical inputs. Determinism preserves behavior *including bugs* → capture current behavior, and make any correctness fix a separate, reviewed change. Don't lose the human-redirect affordance.
- On the backend-routing consideration: web's only transferable principle is **dependency-injection of already-resolved state** — pass the resolved backend *into* the staging verb rather than re-resolving inside it (re-deriving in a second place is a classic divergence source).

## Tradeoffs & Alternatives

- **Fork 1 — event-emission ownership → recommend C (emission stays a thin prose `cortex-lifecycle-event log` call), with caveats.** `complete-route` is read-only and the events fire at different steps (pr_opened Step 5, feature_complete Step 11) than routing (Step 7), so embedding emission in the classifier breaks both purity and timing. (But see Adversarial #11/#13 and OQ2: emission-as-prose contradicts the tickets' "into the verbs" directive, and no verb naturally owns the steps.) `complete-route` legitimately writes `pr.json` in Branch-3 single-match recovery — that is a recovery side-effect distinct from event emission; keep it.
- **Fork 2 — one `--phase` verb vs two → recommend one `--phase` verb over a shared staging engine**, because the genuinely shared discipline (stage-first guard + explicit-add "never directory-glob" rule + warning text) is exactly what drifts between two copies, and one verb makes the staged-set parity test a single harness. Auto-detect refine's approval/cancel via the events.log scan. (Adversarial #14 warns this "engine" is more than a path-set selector — it parses `review.md`, shells the resolver, scans events, and reconciles two divergent backlog-staging strategies.)
- **Fork 3 — byte-identical rows → recommend a shared `build_row` realized through #330's `--field` CLI.** (Adversarial #1/#2/#3/#12 substantially refute this: the corpus is already mixed, the load-bearing invariant is field-set+values not bytes, two `feature_complete` Python emitters already exist, and the ordering is circular. See Open Questions.)
- **Fork 4 — where emission lives → mechanism in `lifecycle_event.py` under #330; call sites in #331.** (Tension with Adversarial #11/#12 unresolved.)
- **Fork 5 — output transport → recommend richer `{route, terminal, message, pr_state, pr_url, pr_number, head_branch}` with the verb owning the ~10 terminal message strings** (the biggest prose reduction; testable; byte-stable). ## Adversarial Review

Verified against the canonical code; several prior findings refuted or reframed.

1. **"Byte-identity" is partly a fiction.** Two Python emitters already ship different bytes into events.log: `lifecycle_event.py:174` (compact, microseconds+offset) and `pipeline/state.py:304` (spaced default, microseconds+offset). The hand-written `Z` second-precision is a **template artifact** — *no* Python emitter produces it. Real files mix forms (e.g. `add-bidirectional-concurrency-guards.../events.log` = 11 spaced + 6 compact); `pr_opened` already exists in two byte-shapes in production. Matching the hand-written `Z` form would require a bespoke timestamp formatter that exists nowhere, diverging from both emitters — code added purely to match a cosmetic format the corpus doesn't honor and consumers ignore. The "ts frozen in the golden test" approach hides this (test passes; production output matches no historical row).
2. **The real invariant is field-set + values**, especially `merge_anchor`: `metrics.py:226-230,989-992,1034-1040` uses `merge_anchor` ("merge"=interactive post-merge vs "review"/absent=legacy overnight) to select the metrics regime and fork phase-duration segmentation. A `build_row` that defaults/drops `merge_anchor` silently reclassifies an interactive feature and corrupts metrics. Cosmetic separators/timestamps are tolerated by every consumer (`json.loads`/substring/regex).
3. **Two `feature_complete` Python emitters already exist** — `pipeline/review_dispatch.py:323-327` (overnight, `merge_anchor:"review"`, no counts) via `pipeline/state.py`. With `schema_version` non-uniform across events (`pr_opened`=1, `feature_complete`=none, `clarify-critic`=3), a "shared `build_row`" is a per-event schema registry that largely duplicates #330's `--field`. Redundant scope.
4. **on-main short-circuit collides with `complete-route` Branch 3.** On main, `complete.md:21` jumps past Step 7 with pr.json absent; Branch 3 fires *because* pr.json is absent and runs `gh pr list`. A naive call triggers a spurious orphan probe + mis-route. The branch check must live somewhere explicit (internal `on_main` route, or prose-gated).
5. **Worktree-CWD path resolution.** `lifecycle_event` resolves from physical CWD, ignoring `CORTEX_REPO_ROOT` (deliberate — lands rows in the worktree). `complete-route` invoked from inside `interactive/{slug}` reads the worktree's events.log/pr.json, which may diverge from main. Needs an explicit root-resolution contract + worktree-CWD test.
6. **Step 11/11a ordering hazard.** `feature_complete` is emitted at Step 11 *before* the 11a commit. If 11a's commit fails (halt), `feature_complete` is already logged; on re-invocation Branch 2 short-circuits to Step 12 — permanently skipping the commit retry. "Preserve exactly" immortalizes this latent unrecoverability. Fix option: emit `feature_complete` only after a successful 11a commit, or have Branch 2 verify the commit landed.
7. **Test migrations are larger than "migrate when prose collapses"**; the negative-token controls guard against real past defects and must port 1:1 with dirty-unrelated-file fixtures, or a directory-glob regression goes uncaught.
8. **A new literal `AskUserQuestion` token would break parity silently.** complete.md has no such literal today (Branch-3 says "ask the user" in prose); introducing the token fails `test_lifecycle_kept_pauses_parity.py` until a `kept-pauses.md` entry is added. (Correction: the parity test checks the *anchor line within ±35 of a step heading*, not message-text proximity; the merge-wait anchor is `complete.md:73`, Step 6 heading at 68 — stable unless Step 2/3 prose reflows.)
9. **"read-only classifier that writes pr.json + does network I/O" is a mislabel** that invites speculative callers (dashboard/statusline) into mutation + network. It is idempotent only across *sequential* re-invocations. Mitigation: name it a stateful recovery+routing verb, document the single write + network calls, restrict invocation to the complete orchestration path.
10. **Backend-conditioning is a behavior change, not byte-identity.** `git add -u cortex/backlog/` stages *every* tracked-modified file under the dir. On none/external + an unrelated dirty tracked `cortex/backlog/*.md`, today stages it; a backend-conditioned verb would not. Even on `cortex-backlog`, `-u` sweeps unrelated dirty backlog files — staging "only the resolved file" diverges there too. Honest framing: a deliberate, defensible bug-fix scoped as an *explicit* behavior change with a test, NOT under the byte-identity PIN.
11. **Emission-as-prose contradicts both tickets.** #330 and #331 both say the events move *into* #331's verbs, "not pre-migrated by #330 … to avoid touching the same lines twice." The prior design (emission as thin prose through #330's verb) reintroduces that double-touch + a three-way coupling. The divergence is unacknowledged and must be reconciled.
12. **The `build_row` de-risk is circular.** Epic mandates "#330 lands first." "#331 introduces `build_row`, #330 adopts it" is impossible under that order. A shared `build_row` must be introduced by #330 (or a pre-#330 shared commit).
13. **No natural verb home for the events** — they fire at Step 5 and Step 11; `complete-route` is Step 7 (and can't emit `feature_complete` idempotently); `stage-artifacts` is Step 11a (after `feature_complete` is written). The structural awkwardness is the real reason emission was punted to prose; state it.
14. **`stage-artifacts` is not a "thin path-set selector"** — it parses `review.md`'s dynamic `**File**:` path, shells `cortex-resolve-backlog-item`, scans events.log bottom-up for approval/cancel, and reconciles two divergent backlog-staging strategies.

Recommended mitigations: reframe the invariant to field-set+values; simplest viable design emits both events as thin prose through #330's `cortex-lifecycle-event log --field` (the compact format the verb already produces) and drops `build_row` + the "shared engine" abstraction — but reconcile the tickets' "into the verbs" directive first; scope backend-conditioning as an explicit behavior change with a test; rename `complete-route` + gate on not-on-main + pin root-resolution; surface (or fix) the Step 11/11a ordering hazard; port every negative-token control; preempt the kept-pauses entry.
