# Plan: fix-feature-complete-emission-ordering-strand

## Overview
Narrow `complete_route.classify()`'s Branch-2 `already_complete` short-circuit so a *retryable interactive finalization* (a `merge`-anchor `feature_complete` row present in the working tree, absent from HEAD, with `commit-artifacts:true` and a committable finalization set) falls through to a recovery route instead of dead-ending at `step12`; make `complete.md` Step 11's emission idempotent; and record the new "committed-iff-complete" refinement in ADR-0004. The fall-through is **constrained to the two routes the spec intends**: the direct-to-main path (`on_main`/step9 — re-runs the finalization tail and *auto-commits*, the #339 strand's primary case) and the PR-bearing path (`merged_dirty` — terminal "resolve first"; the operator commits, then a re-invocation sees `H` and routes `already_complete`). An off-main, pr.json-absent state is **not** routed for retry (it would land on `first_run`/step1 = lifecycle restart) — it stays `already_complete`, matching prior behavior. Safety rests on the `committable` signal degrading toward `already_complete` on any git failure (today's behavior); `H` is read against the correct git-top-relative path so it is never *spuriously* False on a working repo (a spuriously-False `H` is the unsafe direction — toward retry). No new route is added (count stays 12); every existing `already_complete` fixture stays green via the not-committable carve-out.
**Architectural Pattern**: shared-state — the fix is concentrated in the events.log/HEAD/git-status read model that `classify()` derives its verdict from; routing structure and the route enumeration are unchanged.

## Outline

### Phase 1: Committed-aware Branch 2 + idempotent emission (tasks: 1, 2, 3, 4)
**Goal**: `classify()` distinguishes a genuinely-done completion from a commit-failed retryable one and falls through to the constrained recovery routes; Step 11's emit carries a prose idempotent guard. The **routing** fix is covered by discriminating real-repo tests (including a nested-cortex-root case for the `H` read); R4's emit guard is prose-only per spec mandate and is not unit-testable via the read-only `classify()` (see Task 4).

**Checkpoint**: `just test` green including the updated `test_feature_complete_precedes_pr_state`; ADR-0004 carries the committed-iff-complete amendment; the `nothing_staged` note is confirmed or corrected; mirror byte-identical.

## Risks
- **Committability probe scope (resolved Open Decision; corrected after critical-review).** Scope is `cortex/lifecycle/{slug}/` + `cortex/backlog/index.md` + the exact review.md-recorded drift path(s) — **not** the whole `cortex/backlog/` directory. The directory form over-detects unrelated dirty sibling tickets; combined with a gitignored-`cortex/lifecycle/` repo (where `H` is structurally False), that pins `committable=True` on every re-invocation and the feature never converges to `already_complete` — strictly worse than the bug being fixed. `index.md` is the faithful co-dirty proxy for the backlog write-back (Step 10 regenerates it whenever Step 9 changes ticket status). Residual behavior, stated accurately: a spurious over-detection on the **on_main** path does not strand — it retries, and because Step 9 re-bumps the ticket `updated:` date the retry produces a (benign, idempotent) churn commit rather than a clean `nothing_staged` no-op, then converges once nothing is left to commit; on a PR-bearing path the retryable state routes to the terminal `merged_dirty` ("resolve first"), which does not auto-commit. A transient `git status` failure degrades `committable→False`→`already_complete` (no worse than today's unconditional strand). The narrow true-miss window (Step 9 ran but the Step-10 index regen was skipped, leaving the ticket dirty but `index.md` clean) self-heals on the next retry's 9/10 re-sync.
- **R4 emission guard is prose-only by spec mandate.** Spec R4 mandates the Step-11 dedup as prose-only soft routing (the `cortex-lifecycle-event` verb appends unconditionally; no structural `--if-absent` flag). This sits in tension with CLAUDE.md's "prefer structural separation over prose-only enforcement for sequential gates," and an un-skipped duplicate `merge`-anchor row would be a soft corruption of the working-tree-based metrics/scanner surfaces. Accepted here because (a) the approved spec scoped it prose-only and a structural verb flag is out of this ticket's scope, (b) routing still converges even with a duplicate (the "∃ a merge-anchor row" predicate and final `H=True` both tolerate it), and (c) the emit is reached only on a human-initiated Complete re-invocation, where the operator reads the guard prose.
