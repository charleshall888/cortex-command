# Specification: fix-feature-complete-emission-ordering-strand

> Epic context: parent #336 (offload deterministic lifecycle mechanics). This ticket fixes the offloaded `complete-route` verb; see `research.md` for the epic reference and the full angle findings. The design here is the post-adversarial, post-critical-review correction of the design the core research converged on.

## Problem Statement

The interactive lifecycle Complete phase emits `feature_complete` (Step 11) *before* the Step-11a finalization commit, and `events.log` is itself in Step 11a's staged set. If that commit fails (pre-commit hook rejection, index lock, conflict), the row is left in the working tree but not in HEAD — and `complete-route` Branch 2 short-circuits on the working-tree row straight to `step12`, so the commit is never retried: artifacts strand and the operator must commit by hand. This benefits anyone completing a feature interactively; it matters because the canonical multi-step Complete contract should make a failed finalization *recoverable on re-invocation*, not a silent strand. The fix must also not regress the success path (#331's prescribed "emit after commit" was itself buggy — it would leave the terminal row permanently uncommitted) and must not duplicate `feature_complete` rows.

## Phases

- **Phase 1: Committed-aware Branch 2 + idempotent emission** — narrow `classify()`'s `already_complete` short-circuit to genuinely-done states and let the commit-failed state fall through to the existing recovery routes; make Step 11's emission idempotent; cover with discriminating real-repo tests.
- **Phase 2: Contract & documentation coherence** — record the new "committed-iff-complete" invariant in ADR-0004, finalize the `complete.md` prose (+ mirror), and confirm the golden-route-table and `nothing_staged` note stay coherent.

## Requirements

> **Priority (MoSCoW):** R1–R5 and R7 are **must-have** — the routing correctness fix, its read-only/graceful invariants, the idempotent emission, the discriminating round-trip test, and golden-table coherence are the deliverable. R6 (ADR-0004 amendment) is **should-have** — it gives the new invariant its durable home per the no-content-duplication rule, but the code fix is correct and shippable without it, so it may be split to a fast-follow if needed. There is no won't-do requirement here; intentional exclusions live in `## Non-Requirements`.

### Definitions (shared by R1–R3)

These four read-only signals are computed by `classify()` against a single `root = _resolve_user_project_root_from_cwd()`, all via the existing graceful `_git_out`/`_run` helpers, all degrading toward **"done"** on any git failure (the safe direction — a git failure must never trigger a spurious retry or commit):

- **W** — a `feature_complete` row is present in the working-tree `events.log` (today's `complete_seen`).
- **H** — a `feature_complete` row is present in **HEAD's** `events.log`, read via `git show HEAD:cortex/lifecycle/{slug}/events.log` (cwd = `root`, `root`-relative path). Exit 128 / no-commits / git-absent / `None` → **H = false**.
- **anchor** — the `merge_anchor` of the working-tree `feature_complete` row(s), parsed by the same line-wise `json.loads` (skip-on-`JSONDecodeError`) that `classify()` already uses — **not** a substring scan. A row whose `merge_anchor` field is **absent defaults to `"review"`** (the canonical `event.get("merge_anchor", "review")` convention, `bin/.events-registry.md` + `metrics.py`), so a legacy/omitting row is **not** a retry trigger.
- **committable** — running `git status --porcelain` (read-only, untracked-inclusive, ignored-exclusive) over the **finalization stage-set paths** reports a non-empty result. The stage-set is what `cortex-lifecycle-stage-artifacts` would stage — `cortex/lifecycle/{slug}/` (incl. `events.log`), the resolved backlog ticket file + `cortex/backlog/index.md`, and any review-drift requirements file recorded in `review.md`. Exit 128 / git-absent / `None` → **committable = false**. The probe MUST NOT be scoped to `events.log` alone (that strands the committable backlog write-back under a gitignored `cortex/lifecycle/`) and MUST NOT be a bare pathspec-less `git status --porcelain` (that over-detects unrelated dirty files); it MUST NOT mutate the index (no `git add` — Req 1b stays satisfied).

1. **Narrow Branch 2 to genuinely-done states.** `classify()` (`cortex_command/lifecycle/complete_route.py`) triggers Branch 2 on `(W ∨ H)` (so a committed-then-reverted tree still short-circuits) and returns `already_complete`/`continue_to:"step12"` **unless** a *retryable interactive finalization* is detected, in which case it does **not** short-circuit and falls through to the existing branch logic. A **retryable interactive finalization** := `¬H` **AND** there exists a working-tree `feature_complete` row whose `anchor == "merge"` **AND** `commit-artifacts` is `true` (via `read_commit_artifacts(root)`) **AND** `committable`. Everything else is done: HEAD-present, review-anchor-only, missing-anchor (→`review`), `commit-artifacts:false`, or not-committable. *Acceptance*: new parametrized cases in `tests/test_complete_route.py` (all using a real repo via `_init_repo`) assert — committed-in-HEAD → `already_complete`; uncommitted+`merge_anchor:review` → `already_complete`; uncommitted+**no `merge_anchor` field** → `already_complete` (regression guard); uncommitted+`commit-artifacts:false` → `already_complete`; uncommitted+`merge_anchor:merge`+`commit-artifacts:true`+committable → **not** `already_complete` (falls through); in-HEAD-but-absent-from-working-tree → `already_complete`. `just test` exits 0. **Phase**: Committed-aware Branch 2 + idempotent emission

2. **Commit-failed retry routes via existing routes (no new route).** With Branch 2 narrowed, a commit-failed re-invocation on `main`/`master` falls through to the existing `on_main` route (`continue_to:"step9"`, re-runs the finalization tail); a commit-failed re-invocation from inside a dirty `interactive/{slug}` worktree with a present MERGED `pr.json` falls through to the existing `merged_dirty` (4d, "resolve first") route. No `finalization_retry` route is added; the route enumeration stays at 12. *Acceptance*: `test_twelve_routing_branches_covered` (in `tests/test_lifecycle_complete_state_routing.py`) still asserts exactly 12 and passes; a test drives a commit-failed-on-main state (real repo, `merge_anchor:merge`, uncommitted, committable) and asserts route `on_main`/`continue_to:"step9"`. `just test` exits 0. **Phase**: Committed-aware Branch 2 + idempotent emission

3. **Read-only, graceful, same-root, same-detection signal reads.** All new git reads in `classify()` are read-only (Req 1b preserved: zero `events.log` writes on every route) and degrade per the Definitions block (H → false, committable → false, both pushing toward `already_complete` = today's behavior — never toward a spurious retry). The HEAD read uses the existing `_git_out`/`_run` helper with `cwd=str(root)`; the committability read is `git status --porcelain` over the finalization stage-set paths (NOT `git diff HEAD`, which is blind to untracked files; NOT events.log-only; NOT pathspec-less); the `commit-artifacts` flag is read via `read_commit_artifacts(root)` (`cortex_command/lifecycle_config.py:157`) — **not** by importing `stage_artifacts.collect_paths` (which globs+YAML-parses the backlog). `root` is the single `_resolve_user_project_root_from_cwd()` value used for W, H, anchor, and committable. `anchor` and the Step-11 idempotency guard (R4) MUST use the **same** detection method — line-wise `json.loads` with skip-on-decode-error — so a torn final row is classified identically by both (never "absent for classify, present for the guard"). *Acceptance*: `grep -n 'git diff HEAD' cortex_command/lifecycle/complete_route.py` returns nothing for the new code; `grep -n 'collect_paths' cortex_command/lifecycle/complete_route.py` returns nothing; the Req-1b "no events.log writes on any route" assertion in `tests/test_complete_route.py` still passes; a test with a fresh repo (no commits) and a stale `interactive/{slug}` worktree asserts no traceback and that W/H/committable resolve against the same `root`. `just test` exits 0. **Phase**: Committed-aware Branch 2 + idempotent emission

4. **Idempotent Step-11 emission.** `complete.md` Step 11 emits `feature_complete` only when a `feature_complete` row is **absent** from the working-tree `events.log` (detected by the same JSON-line parse as R3, not a substring grep); on the on_main retry path (which re-runs Steps 9–11a) this guards against a duplicate row. The guard is prose-only soft routing (the `cortex-lifecycle-event` verb itself appends unconditionally — no structural dedup), authored without new MUST/CRITICAL language. The emitted row keeps its exact 6-key shape `{ts, event, feature, tasks_total, rework_cycles, merge_anchor}` and is emitted only through the registered `cortex-lifecycle-event --event feature_complete` verb (no new raw `"event":"feature_complete"` literal). *Acceptance*: the Step 11 region of `skills/lifecycle/references/complete.md` contains an idempotent-skip guard clause keyed on JSON-parsed presence; `cortex-check-events-registry` (and its `just`-wrapped audit) pass; the canonical + mirror (`plugins/cortex-core/skills/lifecycle/references/complete.md`) are byte-identical after `just build-plugin` (drift hook green). **Phase**: Committed-aware Branch 2 + idempotent emission

5. **Discriminating commit-failure-recovery round-trip test.** A behavioral test drives a real failed-Step-11a-then-reinvoke cycle (a failing commit, then a succeeding one) and asserts the lifecycle ends committed with exactly one `feature_complete` row — not a self-sealing prose-only grep of `complete.md`. *Acceptance*: a test in `tests/` (real repo) simulates commit-failure → re-invoke routes to retry (`on_main`/step9) → commit succeeds → `feature_complete` in HEAD and `grep -c '"event": "feature_complete"'` on the committed `events.log` = 1. `just test` exits 0. **Phase**: Committed-aware Branch 2 + idempotent emission

6. **Record the committed-iff-complete invariant in ADR-0004.** The fix introduces a new operative semantic — on re-invocation, an uncommitted working-tree `feature_complete` no longer counts as done when it is a retryable interactive finalization (a `merge`-anchor, committable, `commit-artifacts:true` row absent from HEAD). This **reinforces** ADR-0004's existing "merge is the terminal event / Done means merged" decision (it does not contradict it); it adds the committed-state refinement and its carve-outs (HEAD-present / review-anchor / missing-anchor→review / `commit-artifacts:false` / not-committable) plus the rejected alternatives (two-commit, emit-then-rollback) with their #331-documented weaknesses. Add it as a concise amendment subsection to `cortex/adr/0004-multi-step-complete-and-interactive-worktree-lifecycle.md` (the file already amends ADR-0004 for Complete-phase continuations; an amendment, not a new/superseding ADR number, is the correct form since ADR-0004's core decision stands). Do **not** restate the rule in `complete.md` or `project.md` (no-content-duplication). The amendment text must describe ADR-0004's *actual* prior position accurately (merge-is-terminal; events.log is the canonical runtime state store) — it must not claim ADR-0004 held an "emission-is-terminal" or "working-tree-is-canonical" position, which it never did. *Acceptance*: the new amendment subsection is present in `cortex/adr/0004-*.md` and states the committed-iff-complete refinement; no duplicate normative statement of the invariant appears in `complete.md`/`project.md`; the amendment makes no false attribution to ADR-0004's prior text. **Phase**: Contract & documentation coherence

7. **Golden-route-table and `nothing_staged` note coherence (incl. the precedence test).** Both golden tables (`tests/test_complete_route.py::_GOLDEN` and `tests/test_lifecycle_complete_state_routing.py::_ROUTE_CASES`), the `test_twelve_routing_branches_covered` count assertion, **and the standalone precedence test `test_lifecycle_complete_state_routing.py::test_feature_complete_precedes_pr_state`** remain coherent. The precedence test must be updated so its asserted invariant matches the narrowed behavior: a *committed* (real-repo) `feature_complete` short-circuits before pr-state (route `already_complete`), and the test no longer relies on its no-repo fixture accidentally satisfying the carve-out — its discriminating intent is restored, not silently degraded. Re-validate the `complete.md:176` "`nothing_staged` … common on the worktree path post-merge" note for staleness under the narrowed Branch 2 and correct it if it no longer describes reality. *Acceptance*: `just test` exits 0 with all three tests green; `test_feature_complete_precedes_pr_state` uses `_init_repo` + a committed row (or an equivalent real-repo construction) for its short-circuit assertion; the `nothing_staged` note is either confirmed accurate (no change) or corrected with a one-line rationale in the commit body. **Phase**: Contract & documentation coherence

## Non-Requirements

- **No new `finalization_retry` route, no new `continue_to` value.** The commit-failed state reuses `on_main` (trunk) and `merged_dirty` (worktree) — both ticket-endorsed recovery routes (Edges).
- **No attempt-count marker / bounded-retry terminal route.** Termination is operator-driven: re-invocation of the Complete phase is human-initiated (a person re-types `/cortex-core:lifecycle complete <slug>` — this is the load-bearing fact for both the `on_main` and worktree paths, not the merge-wait gate, which does not apply on `main`). No automated caller drives interactive Step-11a today: the overnight path emits `feature_complete` via `review_dispatch.py` (`merge_anchor:"review"`) at review time and never runs `complete-route`; the guardian/recovery layer recovers the runner process, not the Complete skill; and `complete.md` halts loudly per invocation on commit failure. A persistently-failing commit therefore halts loudly each time, as today, now retryable instead of stranded. **This property is scope-contingent, not durable** — it holds only while Complete's *invocation* is human-initiated; if epic #336's offload trajectory ever automates the Complete invocation, a bounded-retry/attempt-count guard would need revisiting (out of scope here).
- **No change to the `feature_complete` row shape, the events-registry entry, or the metrics/dashboard/scanner consumers.** They stay working-tree-based (read-only observability surfaces).
- **No fix to the working-tree-based phase readers** (`common.py:detect_lifecycle_phase`, `metrics.py`, `scan_lifecycle.py`, `_pipeline_state.py`). Their transient "shows complete during the failed-commit window" divergence is out of the ticket's touch-points and self-heals on retry (documented as a known edge).
- **No change to the overnight emission path** (`review_dispatch.py`, `merge_anchor:"review"`).

## Edge Cases

- **On-main first-commit fails → artifacts untracked.** `git status --porcelain` reports untracked (`??`) → committable=true → (with `merge_anchor:merge`) falls through → `on_main`/step9 retry. (`git diff HEAD` would have read empty and reproduced the strand — hence porcelain.)
- **Committable backlog write-back under gitignored `cortex/lifecycle/`.** Step 9/10 write `status:complete` + regenerate `cortex/backlog/index.md` (committable, outside the gitignored tree) even though `events.log` is gitignored. The committability probe spans the **full** finalization set, so it reports the backlog delta → committable=true → retry → the backlog write-back is committed on retry (events.log stays gitignored). An `events.log`-only probe would read empty and strand the backlog write-back — which is why the probe is set-scoped, not events.log-scoped.
- **`commit-artifacts:false`.** Row deliberately uncommitted by design → not a retryable interactive finalization → `already_complete`. Never routed to retry.
- **Legacy / missing-`merge_anchor` row.** No `merge_anchor` field → defaults to `"review"` → not a `merge`-anchor row → not retryable → `already_complete` (matches today's behavior; the success-path regression guard).
- **Overnight-completed feature re-invoked interactively.** Working-tree `feature_complete` has `merge_anchor:"review"`, uncommitted → not a `merge`-anchor row → `already_complete`. Not re-finalized.
- **Gitignored `cortex/lifecycle/` with no committable backlog delta.** `feature_complete` not committable and nothing else pending → committability probe empty → not retryable → `already_complete`. Degrades to today's working-tree behavior for such repos. (If `events.log` was force-added (`git add -f`) into a gitignored tree it is tracked and *does* show in `git status --porcelain` → committable → retry — a correct route; the probe makes no blanket "gitignored ⟹ never committable" assumption.)
- **Multiple `feature_complete` rows** (e.g. an earlier `merge`-anchor row whose commit failed, plus a later `review`-anchor row): the retry trigger is "**∃** a working-tree `merge`-anchor row (¬H, committable)", not "the last row's anchor" — so a stranded `merge`-anchor finalization is retried even when a later `review`-anchor row exists (last-wins would mask it).
- **Committed-then-reverted working tree** (`feature_complete` in HEAD, absent from working tree): Branch 2 triggers on `(W ∨ H)`; H present → `already_complete`.
- **Torn final `events.log` row** (crash mid-write; `flock`+`O_APPEND` makes appends atomic but a partial write is possible): both `classify()`'s detection and the Step-11 guard use the same line-wise `json.loads` skip-on-error, so a torn row is treated identically by both — never "absent for routing, present for the emit-guard."
- **Botched/conflicted merge with a dirty tree.** Unreachable with `feature_complete` present in normal flow (a conflict halts at 4d *before* Step 11 emits). If reached from inside a dirty worktree, falls through to `merged_dirty` (4d, "resolve first") — not a blind re-commit; and Step 11a's staging is path-scoped (no `-A`/`-u` sweep), so foreign dirty files are never committed.

- **MODIFIED:** `complete-route` Branch 2 fires `already_complete` on any working-tree `feature_complete` row → now fires `already_complete` only when the completion is genuinely done (committed, or a review-anchor/missing-anchor/flag-false/not-committable carve-out); a retryable interactive finalization (`merge`-anchor, `¬H`, committable, `commit-artifacts:true`) falls through to `on_main`/`merged_dirty`.
- **MODIFIED (test):** `test_feature_complete_precedes_pr_state` updated to assert the precedence invariant for a *committed* row (real repo), restoring its discriminating intent under the narrowed logic.
- **ADDED:** `complete.md` Step 11 gains an idempotent-emission guard (skip the emit if `feature_complete` already present in the working-tree `events.log`).
- **MODIFIED (doc):** ADR-0004 amended with the committed-iff-complete refinement and its carve-outs.

## Technical Constraints

- `classify()` must stay read-only (Req 1b: zero `events.log` writes on every route); module contract: non-zero exit only on usage error / unresolvable root — every git failure degrades, never a traceback (`complete_route.py:30-33`).
- The committability probe replicates `stage_artifacts`' *intent* (is the finalization set committable) read-only, but must **not** import `stage_artifacts.collect_paths` (a backlog-glob + YAML-parse weight regression on the routing path, per research F7) and must **not** run `git add`. The Plan chooses the exact read-only path enumeration — coarse pathspecs (`cortex/lifecycle/{slug}/ cortex/backlog/`, cheap; may miss a drift-requirements-only delta) or the precise recorded set — but never events.log-alone and never pathspec-less.
- Dual-source mirror: edit canonical `skills/lifecycle/references/complete.md`, run `just build-plugin`, commit canonical + `plugins/cortex-core/...` mirror together (drift pre-commit hook).
- The `<!-- finalization-commit-step -->` region token guard (`tests/test_complete_md_finalization_commit.py`) must stay satisfied after any Step-11a prose edit (positive tokens present, no directory-glob/`-u` staging).
- MUST-escalation policy: author the idempotent guard and any halt framing as soft positive routing, not new MUST/CRITICAL imperatives.
- Wheel-binstub vs working-tree: edits to `complete_route.py` are exercised by `just test` via the editable install — use sequential (non-worktree) dispatch for implementation so tests see the working tree.
- **Golden-fixture invariant (load-bearing):** every existing `already_complete` fixture uses `_make_root` (no git repo), so it stays green via the not-committable carve-out, *not* via a preserved "feature_complete ⟹ short-circuit" contract. Discriminating tests for the committed (2a) and retry (fall-through) states MUST construct a real repo (`_init_repo`); "fixing" an existing no-repo fixture into a committed repo without updating its expectation would flip it to `pr_open`/`on_main` and is the trap to avoid.

## Open Decisions

- **`finalization_retry` re-entry vs fall-through** — *Resolved at spec time*: fall-through to existing `on_main`/`merged_dirty` routes (no new route), grounded in the ticket's Edges and the verified idempotency of the finalization tail (Steps 9/10 idempotent — modulo a benign `updated:` date re-bump in `cortex-update-item`; Step 11 made idempotent; Step 11a stage-first guard). Recorded here only to note the research-converged "add a new route" alternative was deliberately rejected as unneeded complexity.
- **Exact committability path enumeration** — *Deferred to Plan*: coarse pathspecs vs the precise recorded stage-set; distinguishable only against the live `stage_artifacts` path-assembly code, so it belongs in implementation. The Definitions/Constraints bounds (set-scoped, read-only, no import, no `git add`) hold either way.

## Proposed ADR

### Proposed ADR: 0004 amendment — committed-iff-complete refinement (not a new ADR number)

ADR-0004 already decides that **merge** is the terminal event ("Done means merged, not PR-opened") and that the events log is the canonical runtime state store. This fix **refines** that decision for the re-invocation path: an uncommitted working-tree `feature_complete` that is a *retryable interactive finalization* (a `merge`-anchor, committable, `commit-artifacts:true` row absent from HEAD) is treated as commit-failed and retried, while every other shape (HEAD-present, review-anchor, missing-anchor→review, `commit-artifacts:false`, not-committable) remains done. This reinforces "Done means merged" rather than contradicting it; it does not change the terminal event or the row shape. Trade-off: rejects the two-commit alternative (inter-commit residual window; still loops under `commit-artifacts:false`) and emit-then-rollback (mutates an append-only log; own crash window), both documented in #331's plan critical-review. Recorded as a concise amendment to ADR-0004 (its core decision stands → amendment, not supersession), and it must state ADR-0004's actual prior position accurately (merge-is-terminal; events.log canonical) — making no claim that ADR-0004 previously held an "emission-is-terminal" or "working-tree-is-canonical" position.
