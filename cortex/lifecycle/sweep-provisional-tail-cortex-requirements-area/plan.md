# Plan: sweep-provisional-tail-cortex-requirements-area

## Overview
Verify the 32 provisional-tail editorial trim candidates under `cortex/requirements/` file-by-file (whole-file, recall-first), capturing each `(file,id)` outcome to a structured `verify-outcomes.md`, then apply the surviving trims **one file per commit** — coordinated same-text clusters as single edits — with two gating checks per commit: **G1** (a fully mechanical heading-set diff over `##`+`###` headings) and **G2** (normative-line survival, made evidence-backed rather than self-attested — each removed normative line must have a grep-provable surviving token or a quoted informative-only span), plus a scoped cross-file citer refresh. Follows the spec's Specify-approved OQ1/OQ2 resolutions (refresh `cortex_command/**`+`docs/**` citers; pull observability s14). **OQ3 (defer vs. direct ledger write) is NOT resolved by the spec** — it is the one genuinely-open policy decision and is resolved at this plan's §4 approval surface (see Risks); until then the plan carries the deferral path as its provisional default.
**Architectural Pattern**: pipeline

## Outline

### Phase 1: Verify & capture (tasks: 1, 2a, 2b, 2c, 2d, 2e, 3)
**Goal**: Confirm the candidate set is exactly the 32 filtered rows, then verify each candidate whole-file against its inline keep-guidance and pins — one verification sub-task per file writing a disjoint fragment — and assemble the fragments into a single `(file,id)`-keyed `verify-outcomes.md`.
**Checkpoint**: `verify-outcomes.md` exists with 32 `(file,id)`-keyed rows — each survivor carrying a Req 2 signal (a/b/c) and a Req 3 heading+token anchor; observability s14 recorded `deferred: out-of-editorial-scope`; the two drifted backlog claims (s8, s14) corrected in-record. No requirements file edited yet.

### Phase 2: Apply, refresh & gate (tasks: 4, 5, 6, 7, 8, 9, 10, 11)
**Goal**: Enumerate live line-anchor citers (pre-batch), then apply surviving trims per file in a serial chain (one commit each) with G1+G2 gates and same-commit citer refresh, then re-verify citers, backfill commit hashes, and file the two follow-up backlog items.
**Checkpoint**: All 5 `cortex/requirements/*.md` trimmed and committed (one commit each); a post-batch citer re-enumeration shows zero live citer in `cortex_command/**`+`docs/**` pointing at a changed line; `applied_in_commit` backfilled; observability-s14 relocation and reconciliation-debt follow-up backlog items filed.

- **OQ3 policy is the one genuinely-open decision (defer vs. direct ledger write) — resolved at plan approval, not by the plan.** The spec resolved OQ1/OQ2 but explicitly left OQ3 open; this plan carries **defer** only as a provisional default (record a `(file,id)` delta + file a reconciliation follow-up in Task 11). The evidence (the never-run reconciliation, `applied_in_commit` on only 8/96 rows) actively pushes toward direct-write. **The lifecycle's §4 plan-approval step must present the OQ3 choice and persist it as an `oq3_decision` event** (`defer` | `direct-write`); Task 11 reads that recorded field and does the exact swap for `direct-write`. This keeps the decision the operator's and keeps overnight execution zero-context (the choice is a logged value, not an approval-conversation memory).
- **Serial per-file apply chain (Tasks 5→6→7→8→9) trades parallelism for safety.** Per-file tasks can touch overlapping citer files in `cortex_command/**`+`docs/**` (a doc citing two target files) and would race in a shared overnight worktree; the `Depends on` chain serializes them so each commits one file (#353 precedent). This rests on the overnight runner honoring a linear `Depends on` as start-after-completion — the documented dispatch semantics, but if it degraded to a mere scheduling hint the race could resurface. **Correctness does not depend solely on that contract:** each apply task locates its citations by content token (not a stored citing-side line), so even a citing-side line shift from an earlier commit does not misdirect a later one, and Task 10's broad-baseline post-check catches any residual stale citer. Parallelism cost is small (5 fast editorial edits). Phase 1 verification (Tasks 2a–2e) *is* parallel-safe — disjoint fragment files, read-only inputs.
- **No test backstop.** `tests/test_requirements_skill_e2e.py` validates the writer in `tmp_path`, never the committed area docs — so G1 (H2+H3 heading-set diff), G2 (per-removed-line grep-proven survival or quoted informative-only), and the Req 7 broad-baseline citer re-check are the only regression guards. G1 is fully mechanical; G2's "preserved-elsewhere" half is mechanized to a per-line token grep, and only the "is this line normative / informative-only" classification remains editorial judgment — an irreducible residue of editorial mode, but one that must now be shown (quoted span), not merely asserted. A MERGE_DEDUP that empties an H2 or H3 now fails G1; one that drops the last copy of a normative line now fails G2's grep.
- **Pulling observability s14 leaves a real cross-file relocation unaddressed.** Mitigated by the Task 11 follow-up item; recorded as `deferred` (not refuted) so the work is not lost.
- **Editorial judgment risk on the backlog s3/s5/s6/s7 cluster.** The sharp case: s6's enum (FR L52) and s7's decisions (FR L60 / OQ L129) do NOT live in the AC sink, so a naive "fold all four into AC" drops them. Tasks 2a and 5 call this out explicitly and G2 catches a dropped normative line.
