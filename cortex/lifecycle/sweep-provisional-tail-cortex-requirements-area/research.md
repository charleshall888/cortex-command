## Codebase Analysis

This is the mechanical basis for editorial mode: a trim's realized cost = trim size × how often features carry a matching tag, not raw bytes.

## Web Research

Transferable prior art for "confirm a span is non-load-bearing before removing it" in agent-facing prose:
- **Recall-first, then precision** (Anthropic context-engineering): don't cut in the same pass you're trying to preserve everything in. "Smallest high-signal set" ≠ shortest.
- **Chesterton's Fence**: remove a rule only after identifying *why* it was added and confirming that reason is now obsolete — not because its purpose is illegible.
- **Structural-substitution test**: a prose instruction is redundant only if the constraint it encodes is already enforced by a hook/test/gate elsewhere (aligns with this repo's "structural gates > prose enforcement" principle). This is the cleanest positive test for non-load-bearingness here.
- **Normative vs. informative**: keep `shall/must`/behavior-determining content; rationale/example prose is the condensable part. Test each bullet, not the paragraph as an atom.
- **Instruction Bleed** (arXiv 2606.26356): isolated review of a single trimmed span is *insufficient* — a span can be load-bearing for a different section's interpretation. Argues for reviewing the whole file, not the span alone.
- **Context-rot / ablation evidence**: keeping genuinely non-load-bearing prose has a real, measured cost (justifies the sweep) — but verbose/example-heavy spans are often exactly the load-bearing ones (an ablation dropped reliability to 0%). Length is not a proxy for redundancy.
- Behavioral-equivalence testing is the gold-standard verification primitive, but there is no eval harness over these files (see Adversarial §3), so verification here is editorial judgment plus the mechanical gates proposed below.

## Tradeoffs & Alternatives

**Decision A — execution/verification ordering.** Recommend **A3 (per-file grouping)**: hold one file in context, verify + resolve all its candidates together (handling intra-file MERGE_DEDUP interactions), apply, commit — ~5 commits, one per file, each body listing which candidate `(file,id)`s applied/refuted. Read the ticket's "single-pass" as ruling out A2's redundant separate verify-then-apply round-trip over the same corpus, **not** as mandating A1's per-candidate isolation (which is unsafe for MERGE_DEDUP candidates that move content across sections). A3 matches both #351 (whole-file editorial pass) and #353 (one commit per file, locate by heading/token).

**Decision B — where per-candidate verify outcomes are recorded** (given the ledger write is deferred). Recommend **B1-structured** — a small per-candidate outcomes record at a fixed relative path in each child's lifecycle dir (e.g. `cortex/lifecycle/{slug}/verify-outcomes.{md,json}`, keyed by `(file, id)` → applied/refuted/commit/reason), so the eventual reconciler has 4 known locations to glob — paired with **B2** commit-body notes (matches #353's existing habit at zero extra cost). Reject B3 (net-new scratch-delta convention).

**Live finding: the promised reconciliation has already failed once.** #353's applied candidates still show `status: verified_survives` with **no** `applied_in_commit`; `applied_in_commit` is populated on only **8 of 96** `verified_survives` rows (a single old #351 batch). The "single reconciliation pass" #353/#357 promise has not occurred. **Refuted-recording only actually prevents re-proposal if it eventually lands as a `status` field in `master_candidates.json`** — the only artifact a future audit consults; a verdict recorded solely in a lifecycle artifact or commit is undiscoverable by construction. Because `(file, id)` is unique and the 5 children touch disjoint files, each child editing only its own rows (atomic read-modify-write) does not logically contend — the "concurrent write" risk the ticket cites is a whole-file last-write-wins race, resolvable with a light write-once-near-end discipline or a lock. This reopens the deferral-vs-direct-write decision (OQ3).

## Adversarial Review

1. **Cross-file line-anchor staleness — the batch's biggest hazard, missed by the other angles.** The inventory flagged one stale citation; in reality these 5 files are hard-cited by `file.md:N` in **code, living docs, ADRs, and active (non-archived) lifecycle specs**, and most trims sit near the top of each file so they shift every line below. Confirmed live citers inside/above trim ranges: `cortex_command/lifecycle_event.py:15-16` (a **code docstring**) cites `pipeline.md L143/146/151` (pipeline s13 trims L149-155); `docs/overnight-operations.md:655` cites `multi-agent.md:77` (s9 trims L73-79); `docs/internals/pipeline.md:178` + `ADR-0015:60` cite pipeline.md's "Post-Merge Review" section by name as "the contract" (s7 compresses it). The "locate by heading, not line" convention protects the *trimmer*, not these external consumers. **Nuance: refresh code + `docs/` + active-lifecycle citers; leave `cortex/lifecycle/archive/**` frozen** (historical records).
2. **`observability.md s14` is not an editorial trim** — it is a LAZY_REF that relocates an entire H2 section *out of the 5-file scope* into `docs/internals/auto-update.md`, contradicting the batch's "no out-of-area change" framing, and it targets a section the Requirements angle listed as **must-keep**. Treat as a category error: pull from the editorial batch (route to a real lifecycle change) or downgrade to an in-file compress.
3. **No test backstop — confirmed and stronger than assumed.** `tests/test_requirements_skill_e2e.py` operates on the *writer's* output in `tmp_path`, never on the committed `cortex/requirements/*.md`. A MERGE_DEDUP that empties an H2 in these files passes every test silently.
4. **The "fixed 7-H2 schema" is already violated** — `observability.md` has **8 H2s** (`## Install-mutation invocations` between Edge Cases and Open Questions). So the schema is a guideline, not an enforced invariant, which makes an H2-thinning MERGE_DEDUP a real silent-regression mode. Two concrete hazards: observability s9↔s13 both touch the Edge Cases array-clobber bullet; backlog s14 could gut the Open Questions H2.
5. **Candidate `id` is not globally unique** (`s3` appears 31×, `s4` 26× across files). `(file, id)` **is** unique (verified, zero collisions). Any outcome-recording or reconciliation keyed on `id` alone patches the wrong rows — key on `(file, id)`.
6. **backlog.md has a hidden third coordinated cluster**: s3/s5/s6/s7 all fold FR content into the **same Architectural Constraints sink (L100-107)**. Applied independently, later merges' "AC already states X" assumptions break and FR content risks being dropped. Freeze AC L100-107 and apply s3/s5/s6/s7 as one edit.
7. **The two "minor drift" candidates are not safe as-written.** backlog s14 applied verbatim would write a decision record for a question the code never settled (a factual defect injected into a requirements doc — worse than uncut prose); backlog s8 collapsing on a weaker-than-verbatim third match risks erasing a distinct NFR nuance. Correct both claims to reality *before* trimming.
