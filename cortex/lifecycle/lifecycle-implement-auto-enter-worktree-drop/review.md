## Stage 2: Code Quality

- **Naming conventions**: Consistent. Python files use `snake_case` for functions (`read_branch_mode`, `should_fire_picker`, `_extract_frontmatter_text`), `UPPER_SNAKE` for module constants (`REASONS`, `_VALID_BRANCH_MODES`, `_CONFIG_RELPATH`), kebab-case for config field (`branch-mode`) and closed-set values (`worktree-interactive`, `feature-branch`). Underscore-prefixed names (`_pathlib`, `_yaml`, `_VALID_BRANCH_MODES`) explicitly mark the closed public API surface — exactly two public symbols per module (`read_branch_mode` in lifecycle_config; `should_fire_picker` + `REASONS` in lifecycle_implement). The frozenset `REASONS` documents the closed return-value set without exposing internal state.

- **Error handling**: Appropriate for the context. `read_branch_mode` returns `None` on missing file, missing cortex dir, OSError, unclosed frontmatter, missing field, null value, malformed YAML; only the malformed-YAML case emits a stderr warning (naming both the file and the parse error). `should_fire_picker`'s `_is_dirty_tree` treats subprocess errors and non-zero git exit as "not dirty" (matches implement.md's existing fallback for `uncommitted-changes guard skipped: git status failed`). `_has_live_interactive_session` returns `False` on missing file, empty content, unparseable PID, or `os.kill(pid, 0)` raising — graceful degradation in every branch. No try/except over-broad swallowing; only specific exceptions caught (`FileNotFoundError`, `OSError`, `yaml.YAMLError`, `ValueError`, `subprocess.SubprocessError`).

- **Test coverage**: 27 tests total across the three new test files, all passing:
  - `tests/test_lifecycle_config.py`: 16 tests covering R2's 8 acceptance cases plus edge cases (missing cortex dir, unclosed frontmatter, empty frontmatter, explicit null, typo variants).
  - `tests/test_lifecycle_implement_branch_mode.py`: 9 tests — 5 primitive (a1)–(a5) for `should_fire_picker` exercising real `git init`'d repos, plus 4 structural-doc tests verifying implement.md wiring (helpers named, open-paren invocation present, invocation precedes §1 AskUserQuestion, all four closed-set values within ±10 lines of `should_fire_picker` mention). The (a3) live-PID test correctly uses `.git/info/exclude` to ignore the `cortex/` dir so the sessions PID file doesn't dirty the tree (which would short-circuit at dirty_tree before reaching the live-PID carve-out).
  - `tests/test_lifecycle_kept_pauses_parity.py`: 2 tests covering both directions (inventory → call sites, call sites → inventory) plus the conditional-pause sentinel branch.

- **Pattern consistency**: Follows existing project conventions. The frontmatter parsing pattern (read text, split on `---` delimiters, `yaml.safe_load`) matches the inline pattern at `cortex_command/overnight/cli_handler.py:58` and `skills/critical-review/SKILL.md:38`. The inline-Python pattern in `implement.md` (`python3 -c "..."`) matches the existing §1 runtime-probe pattern at lines 56–64. The deliberate non-migration of the five existing ad-hoc parsers (spec Non-Requirements clause) is honored — `lifecycle_config.py` exposes only `read_branch_mode` and does not pre-commit to a multi-field API. The `REASONS` frozenset pattern echoes `cortex_command/common.py:TERMINAL_STATUSES` (Architectural Constraints, line 36). The "exactly two public symbols" approach and underscore-prefixing the rest is a clean signal of the intentionally-narrow scope per spec R2 ("primitive is intentionally narrow"). The closed-set check is correctly placed at the caller (`should_fire_picker`) rather than the primitive, matching spec R3's "caller-side closed-set check" requirement.
