# Plan: Lifecycle implement — config-defaulted branch mode (Approach C)

## Overview
Introduce a per-repo `branch-mode` config field in `lifecycle.config.md` that, when set, short-circuits the implement-phase picker on a clean working tree with no concurrent live interactive worktree. Two minimal Python primitives (`read_branch_mode()` and `should_fire_picker()`) carry the structural enforcement; the parity-test sentinel extension keeps the kept-pauses inventory honest. Phase 4 records the C/A split framing in ADR-0004 and files the follow-up ticket capturing Approach A's deferred design surface.

## Risks

- **Skill control flow change at Task 5**: `skills/lifecycle/references/implement.md` §1 is on the daytime lifecycle hot path. The §1 preflight is additive (defensive redundancy preserves the §1a:78-82 check) but the new inline `python3 -c` calls add subprocess invocations to every implement-phase entry. Back-of-envelope latency cost: ~200-400ms per invocation on a warm Python interpreter. Acceptable for interactive flow; flag for measurement if it becomes noticeable.
- **Integration-boundary regression class is structurally uncovered**: The Task 5 multi-anchor V and Task 6's `TestImplementMdWiring` structural-doc class together close the mention-vs-wire ambiguity and the wrong-routing-presence ambiguity. They do NOT cover **inverted-boolean** routing (an implementer who routes `(False, "suppressed")` to the picker instead of to the configured branch). Catching this would require behavioral testing of the skill prose's runtime interpretation — the markdown is interpreted by the model at runtime, so static analysis cannot fully simulate it. Accepted limitation: the documented `## Acceptance` paragraph is the human-review gate for this class. If autonomous overnight execution ever runs this lifecycle, the integration-test gap must be closed first (e.g., by an end-to-end harness that drives `/cortex-core:lifecycle implement` and asserts on the path taken).
- **Python primitive scope is closed by structural V, not prose**: Tasks 2 and 4 each have negative-scope clauses naming exactly which public symbols the module exposes, backed by `dir()`-based public-symbol-count verification steps. Task 4's `REASONS` frozenset closes the reason-string set the same way. These are structural gates — an implementer who adds helpers (e.g., `_normalize_value`, `read_review_criteria`, `check_overnight_active`, additional reason strings) fails V. Explicitly out of scope: helpers for other §1a guards (overnight detection, sandbox preflight, settings drift), additional reason variants, other config fields, normalization layers in `read_branch_mode`.
- **Parity-test sentinel (Task 7) is documentation parity, not runtime enforcement**: The `conditional pause` rationale tag's structural marker regex (`\bread_branch_mode\b|\blifecycle_config\b`) is a literal-string proximity check. R5 of the spec frames this honestly as documentation parity; the runtime gate is at R3 (Task 5's `read_branch_mode` invocation) + R4 (Task 6's test coverage). The sentinel's job is to keep the inventory rationale's tag honest with the existence of the structural marker — together with Task 5's tightened multi-anchor V, this prevents a coordinated tag+wiring removal from silently passing.
- **`should_fire_picker()` semantics overlap with existing §1a:78-82 check**: Both check the same PID file.
