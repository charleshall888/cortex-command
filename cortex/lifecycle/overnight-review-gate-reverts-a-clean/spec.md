# Specification: overnight-review-gate-reverts-a-clean

## Problem Statement

The overnight review gate reverts a correctly-built, already-merged feature when the review agent completes cleanly (`stop_reason=end_turn`) but writes no parseable `review.md`. `parse_verdict()` collapses "no usable verdict" into a synthetic `ERROR`, and `outcome_router` treats that `ERROR` as `review_dispatch_crashed`/`could_not_run`, reverts the merge, and feeds the systemic circuit breaker — discarding verified, merged work over a tooling/adherence miss and mislabeling it a "crash." This fix distinguishes a review that **could not produce a usable verdict** (agent completed, no artifact) from a **genuine dispatch crash** (agent failed): the former preserves the merge and flags it for human re-review *on the surfaces a human actually acts on*; the latter keeps today's revert-and-trip behavior. A systemic no-artifact pattern still pauses the batch (via the breaker, tagged distinctly), so a broken review model cannot silently ship many unreviewed merges. Beneficiary: the overnight operator, who no longer loses verified work to a review-tooling miss and is not misled into merging unreviewed code.

## Phases
- **Phase 1: Distinguish & preserve safely** — discriminate could-not-run from crash by threading the review dispatch result, add the orthogonal `ReviewResult.could_not_run` signal, guard the upstream revert at all three gate sites, fix the latent stale-verdict bug, render preserved merges correctly in the morning report, AND flag the integration PR so the human's merge-decision surface is not misleading.
- **Phase 2: Signal hygiene, systemic detection & docs** — split the misleading `review_dispatch_crashed` flag, feed the breaker for could-not-run under a distinct cause-class (wired end-to-end through `_SYSTEMIC_ERROR_TYPES`), register the signals, amend the documented contract, and add the ADR.

## Requirements

1. **Thread the review dispatch result into the verdict decision (fixes the latent stale-verdict bug).** `dispatch_review()` (`cortex_command/pipeline/review_dispatch.py`) currently captures the review agent's `DispatchResult` (~L264) and discards it before resolving the verdict at ~L277–278; its `.success` is read **nowhere** (the existing `result.success` reads at ~L415/470 are the *fix-agent* and *re-merge* sub-dispatch results, not the review result — this is net-new plumbing, not a mirror of an existing idiom). Thread the review `DispatchResult` to the verdict-resolution point so that when `result.success == False`, the failure/crash path is forced and any on-disk `review.md` is ignored (today a stale prior-cycle `APPROVED` can wrongly approve a feature whose fresh dispatch failed). Acceptance: a new `test_review_dispatch.py` case asserts that `success=False` + a stale `APPROVED` `review.md` on disk yields `approved == False`; `just test` exits 0. **Phase**: Distinguish & preserve safely

2. **Add the orthogonal `ReviewResult.could_not_run` boolean and set it at every ERROR-producing site.** Add `could_not_run: bool = False` to `ReviewResult` (~L29). Set it `True` only when `result.success == True` AND the resolved verdict is the synthetic `ERROR` sentinel (file missing, no JSON block, unparseable, missing/non-canonical `verdict` key) — i.e. the agent completed but produced no usable verdict. This must cover **both** the cycle-1 ERROR site (~L314) **and** the cycle-2 rework ERROR site (resolved ~L530, returned via the `ReviewResult` at ~L575–582). The `_write_review_deferral()` deferral file written inside `dispatch_review` for these paths must carry the same could-not-run signal so the on-disk deferral and the `FEATURE_DEFERRED` event cannot disagree. Do not introduce a tri-state verdict; the verdict-string vocabulary is unchanged. Acceptance: `grep -n 'could_not_run' cortex_command/pipeline/review_dispatch.py` shows the dataclass field and the set sites; tests assert `could_not_run=True` for success-true + no-artifact at **both** cycle-1 and cycle-2, `could_not_run=False` for `APPROVED`/`REJECTED`/`CHANGES_REQUESTED`, and `could_not_run=False` for the `success=False` crash cell. **Phase**: Distinguish & preserve safely

3. **Preserve the merge for could-not-run by guarding the UPSTREAM revert at all three gate sites.** The merge revert does **not** live in the `if rr.verdict == "ERROR":` detail-tag branches (those only tag `deferred_details` and feed the breaker). It fires earlier, gated on `rr.deferred`, via path-divergent mechanics: `revert_merge(live_merge_sha)` on the recovery (~L1087) and primary (~L1728) paths, and `_reset_ff_merge(repo, pre_ff_base_sha)` (a `git reset --hard`) on the repair (~L1457) path. Guard each of these upstream reverts so a could-not-run feature (`rr.could_not_run == True`) is **not** reverted — the merge is preserved. A shared helper sets the deferral-detail flags (`could_not_run`, `merge_reverted`) coherently and exception-safely across the three sites, but the revert-skip is a per-site guard on each site's own revert call (the helper cannot abstract `revert_merge` vs `_reset_ff_merge`). `could_not_run == True` is the **positive** discriminator the report keys on — never inferred from `merge_reverted == False` (which a genuine crash with a failed revert, or a repair-path `_reset_ff_merge` returning `False`, also produces). Acceptance: `test_outcome_router.py` asserts, for each of the three paths, that `revert_merge`/`_reset_ff_merge` is NOT called on the could-not-run path and IS called on the crash path; the `feature_deferred` event carries `could_not_run=True` with `merge_reverted=False` for the preserved case. **Phase**: Distinguish & preserve safely

4. **Render preserved-unreviewed merges distinctly in the morning report — both report blocks.** In `report.py`, the preserved could-not-run feature never emits `feature_merged` and has `merge_reverted=False`, so today it falls through to the generic "Answer this question and re-run the feature" text (instructing a re-run of already-merged code). (a) In `render_deferred_questions()` (~L1244–1272), add a branch keyed on `could_not_run == True` (read with `.get(..., False)` for archived-log tolerance), placed **before** the generic re-run fall-through, emitting an explicit "unreviewed merge preserved — review could not run; kept on the integration branch for human re-review (intentional, not an error)" annotation, plus a could-not-run sub-count in the executive summary. (b) The second `merged_to_integration` / "Do NOT re-run the feature — it is already on the integration branch" block (~L1412–1463) must not mis-render a preserved could-not-run feature; confirm by test that a preserved feature is excluded from or correctly annotated by that block. Acceptance: `test_report.py` asserts a preserved could-not-run feature renders the new annotation, NOT "do NOT re-run" and NOT the generic re-run text, from both rendering paths; `just test` exits 0. **Phase**: Distinguish & preserve safely

5. **Flag the integration PR when the batch contains a preserved unreviewed merge.** The overnight runner auto-creates a **non-draft** integration PR titled "N features merged" whenever the batch has ≥1 merged feature (`runner.py` ~L2103/2118); its branch tip includes the preserved unreviewed merge, and the PR-body warning is gated on `integration_degraded`, which the could-not-run path never sets — so a human can merge unreviewed code to `main` without ever opening the morning report. When the batch contains ≥1 preserved could-not-run merge, the integration PR must carry a visible signal at the merge-decision surface: set the existing `integration_degraded` signal (and/or add a PR-body warning) that names the unreviewed feature(s) and states the merge is unreviewed-pending-re-review. Acceptance: a test asserts that a batch with a preserved could-not-run feature produces an integration PR/body carrying the degraded/unreviewed marker naming the feature (e.g. the PR-body builder includes the warning when `integration_degraded` or an equivalent could-not-run set is non-empty); `just test` exits 0. **Phase**: Distinguish & preserve safely

6. **Split `review_dispatch_crashed` so it is accurate.** The could-not-run case must no longer set `review_dispatch_crashed` (today both flags are co-set at ~L1144–1146 and the two other sites); `review_dispatch_crashed` denotes only genuine dispatch crashes (except-block / `success=False`). Update the existing `test_outcome_router.py::test_error_verdict_event_carries_could_not_run_marker` (~L1698), which currently asserts `review_dispatch_crashed is True` on the ERROR path — that assertion inverts under this requirement. Acceptance: `test_outcome_router.py` asserts the could-not-run path sets `could_not_run=True` but NOT `review_dispatch_crashed`, and the crash path sets `review_dispatch_crashed=True`; the previously-passing L1698 assertion is updated, not left to break. **Phase**: Signal hygiene, systemic detection & docs

7. **Feed the systemic breaker for could-not-run under a distinct cause-class, wired end-to-end.** Add `REVIEW_NO_ARTIFACT = "review_no_artifact"` (`cortex_command/overnight/constants.py`) **and add it to `_SYSTEMIC_ERROR_TYPES`** (alongside `REVIEW_DISPATCH_CRASH`), because the `cause_class` derivation (~L973–979) filters the combined list against that tuple. Parameterize `_record_review_crash_systemic` to accept a cause-class argument (or add a sibling helper) — it currently hardcodes appending `REVIEW_DISPATCH_CRASH` (~L961) and accepts only `(name, ctx)`. The could-not-run path records the systemic event tagged `review_no_artifact`. The threshold is on the **aggregate** count of systemic review failures of either class (a mixed crash + no-artifact batch trips at `SYSTEMIC_FAILURE_THRESHOLD` — this is intended: any systemic review-tooling failure should pause the batch); the cause-class label distinguishes the kinds **in the emitted event for diagnosis**, not in the counter. Acceptance: `grep -n 'REVIEW_NO_ARTIFACT' cortex_command/overnight/constants.py` shows the constant AND its `_SYSTEMIC_ERROR_TYPES` membership; `test_outcome_router.py` asserts (i) 3 could-not-run features in one batch emit `PIPELINE_SYSTEMIC_FAILURE` with `cause_class` containing `review_no_artifact` while their merges are preserved, and (ii) a mixed 2-crash + 1-no-artifact batch also trips at threshold with both labels present. **Phase**: Signal hygiene, systemic detection & docs

8. **Register the signals and amend the documented contract.** Document the changed `feature_deferred` detail keys and the new `review_no_artifact` cause-class value in `bin/.events-registry.md` (field-additive schema-extension note). Re-pin **all three** `pipeline_systemic_failure` producer line numbers (`outcome_router.py:922,980,2166`) that Phase-2 edits will shift. Amend `cortex/requirements/pipeline.md` (~L86) and `docs/internals/pipeline.md` to state the crash-vs-no-artifact split: a genuine crash reverts + feeds the breaker as `review_dispatch_crash`; a could-not-run review preserves the merge, flags it on the report AND the integration PR for human re-review, and feeds the breaker as `review_no_artifact`. Acceptance: the events-registry audit / `just test` passes; `grep -n 'review_no_artifact' cortex/requirements/pipeline.md bin/.events-registry.md` returns matches. **Phase**: Signal hygiene, systemic detection & docs

9. **Record the decision as an ADR.** Add `cortex/adr/0014-review-could-not-run-vs-dispatch-crash-split.md` capturing the split and honestly stating the safety relocation (see Proposed ADR), and back-point `pipeline.md` to it. Acceptance: the ADR file exists and `pipeline.md` references it. **Phase**: Signal hygiene, systemic detection & docs

## Non-Requirements

- **No retry or escalation.** No re-dispatch of the review, no Sonnet→Opus escalation, no N-retry loop. (User-chosen Clarify policy.)
- **No auto-re-review on resume.** Preserved could-not-run features carry no new status vocabulary and are not automatically re-reviewed when a session resumes; the operator handles them manually via the morning report + the flagged integration PR + the deferral file.
- **No tri-state verdict enum.** The verdict-string vocabulary (`APPROVED`/`CHANGES_REQUESTED`/`REJECTED`) is unchanged; the could-not-run signal is an orthogonal boolean.
- **Fix #5 (dispatch complexity mislabel) dropped.** Research confirmed review dispatch already uses the correct feature tier (`read_tier`); the observed `complexity="complex"` was the intentional repair-agent path (`integration_recovery.py`), not a review-dispatch bug.
- **No change to `SYSTEMIC_FAILURE_THRESHOLD`** numeric value (stays 3). Note: its *population* now includes preserved (not just reverted) failures — see Edge Cases.
- **No redesign of batch-abort / not-yet-run-feature accounting.** Global-abort semantics for not-yet-run features are unchanged from today's crash-triggered abort; this fix only adds that some already-merged features may be preserved rather than reverted when the abort fires (see Edge Cases).
- **No merge-before-gate vs gate-before-merge restructuring** — out of proportion to this fix.

## Edge Cases

- **`success=False` + stale parseable `review.md` on disk** → treated as a genuine crash (revert + `review_dispatch_crash`); the stale verdict is ignored, not used to approve. (Latent bug fixed by Req 1.)
- **Non-canonical verdict string** (e.g. `verdict: "BLOCKED"`) with `success=True`, file present → resolves to the `ERROR` sentinel → `could_not_run=True` (preserve + flag), NOT a crash. Named `could_not_run`, not `no_artifact`, because a file may exist.
- **Cycle-2 rework produces no parseable verdict** (`success=True`, ERROR at ~L530) → `could_not_run=True` on the cycle-2 `ReviewResult` and on its deferral file; same preserve + flag behavior as cycle-1.
- **Repair-path preserve vs failed reset** → both yield `merge_reverted=False`, so the report/PR surfaces key on the **positive** `could_not_run` flag (set only for the no-artifact case, never for a crash after Req 6) to disambiguate a deliberate preserve from a failed `_reset_ff_merge`.
- **Revert failure (exception) on a genuine crash** → `merge_reverted=False` with `could_not_run=False`; the legacy "do NOT re-run" annotation is retained (it is a crash, not could-not-run).
- **`budget_exhausted`** (`success=False`, normal-completion route) → genuine crash path (revert + crash breaker), not could-not-run.
- **All three review paths** (primary integration, test-recovery, repair ff-merge) → revert-guard applied at each site's own revert call; flag-setting shared.
- **Systemic no-artifact in a batch** → 3 could-not-run features trip the breaker (tagged `review_no_artifact`) and pause the batch via `global_abort_signal`, yet each individual merge remains preserved. The batch then holds a mix of preserved-unreviewed merges and not-yet-run features under a global abort; not-yet-run features are abandoned per existing global-abort semantics (unchanged), and the preserved merges remain on the integration branch surfaced via Req 4 + Req 5.
- **Mixed systemic batch** (e.g. 2 crashes + 1 no-artifact) → trips at threshold 3; `cause_class` carries both `review_dispatch_crash` and `review_no_artifact` labels for diagnosis.

## Changes to Existing Behavior

- **MODIFIED**: review no-artifact case (agent completed, no usable verdict) → was: revert merge + `review_dispatch_crashed` + feed breaker as a crash. Now: preserve merge + `could_not_run` (not "crashed") + feed breaker as `review_no_artifact` + flag the integration PR.
- **MODIFIED**: `dispatch_review()` now threads the review `DispatchResult.success` into the verdict decision — a failed dispatch no longer trusts a stale on-disk `review.md` (cycle-1 and cycle-2).
- **MODIFIED**: morning report renders preserved could-not-run merges with an "unreviewed merge preserved — needs re-review" annotation + exec-summary sub-count (was: legacy "do NOT re-run" / generic re-run text), in both report blocks.
- **MODIFIED**: the integration PR carries an unreviewed/degraded marker when a preserved could-not-run merge is present (was: unflagged non-draft PR).
- **MODIFIED**: `_record_review_crash_systemic` is parameterized by cause-class; `_SYSTEMIC_ERROR_TYPES` gains `review_no_artifact`.
- **MODIFIED**: existing test `test_error_verdict_event_carries_could_not_run_marker` (~L1698) updated for the `review_dispatch_crashed` split.
- **MODIFIED**: `cortex/requirements/pipeline.md` (~L86) and `docs/internals/pipeline.md` contract amended.
- **ADDED**: `ReviewResult.could_not_run` field; `REVIEW_NO_ARTIFACT` cause-class constant; ADR 0014.

## Technical Constraints

- **Discriminator is `DispatchResult.success`, threaded net-new** — `stop_reason` is logged to events.log (`dispatch.py` ~L865/880) but is NOT a field on the `DispatchResult` dataclass; do not thread it. The review result's `.success` is currently read nowhere (the ~L415/470 reads are the fix/remerge sub-dispatch results) — Req 1 adds that wiring.
- **The revert is upstream of the named ERROR blocks**, gated on `rr.deferred` (~L1087/1457/1728), and uses path-divergent mechanics (`revert_merge` with `.aborted`/`.success` vs `_reset_ff_merge` returning a bare bool). The flag-helper unifies detail-flag setting; the revert-skip is a per-site guard.
- **`cause_class` is filtered against `_SYSTEMIC_ERROR_TYPES`** (~L973–979), and `_record_review_crash_systemic` hardcodes the appended class (~L961) — a new cause-class requires both a constant + tuple membership AND a helper signature change, not just a new constant.
- **Three report surfaces and three registry pins.** `report.py` has two "do NOT re-run" blocks (~L1244–1272 and ~L1412–1463); the `pipeline_systemic_failure` registry row pins three producer lines (`922,980,2166`).
- **Events-registry gate + tolerant readers.** Changed `feature_deferred` detail keys are field-additive; consumers read with `.get(key, default)` for archived-log tolerance; register in `bin/.events-registry.md` in the same commit.
- **Doc ownership.** `cortex/requirements/pipeline.md` + `docs/internals/pipeline.md` own this behavior; update the owning docs and cross-link.
- **Backward compatibility.** `could_not_run` defaults `False`; report and metrics readers tolerate its absence in archived events.

## Proposed ADR

### Proposed ADR: 0014-review-could-not-run-vs-dispatch-crash-split

**Context**: A prior lifecycle (`overnight-review-gate-crashes-to-cycle`) deliberately chose to revert any unreviewed merge — including a "could-not-run/crashed review with verdict ERROR" — so no unreviewed code remains on the integration branch, and to feed all such cases into the systemic circuit breaker. The integration branch is auto-published as a non-draft "N features merged" PR, so that revert was load-bearing: it kept the PR trustworthy. Operationally the revert also discards verified, already-merged work whenever the review *agent* completes but fails to write a parseable verdict (a tooling/adherence miss), and mislabels it a "crash."

**Decision**: Split the single `ERROR` outcome into two. A **genuine dispatch crash** (`DispatchResult.success == False`) retains the revert + systemic-breaker behavior under the `review_dispatch_crash` cause-class. A **could-not-run review** (`success == True`, no usable verdict) **preserves** the merge, surfaces an "unreviewed merge preserved — needs human re-review" deferral on the morning report **and a degraded/unreviewed marker on the integration PR**, and feeds the systemic breaker under a distinct `review_no_artifact` cause-class (aggregate threshold; label for diagnosis) so a systemic adherence failure still pauses the batch without discarding individual work.

**Trade-off**: The prior "nothing unreviewed survives" guarantee was enforced in code (SHA-anchored revert under `ctx.lock`). This decision relocates that boundary: unreviewed code may remain on the overnight integration branch in the could-not-run case, and safety now rests on three surfaces — the morning-report annotation, the **integration-PR degraded marker** (so the merge-decision surface itself warns the operator, not just the report), and the systemic breaker that halts a batch-wide review-tooling failure. The morning-merge gate is operator-discipline rather than a code-enforced revert; the PR marker is what keeps that discipline honest. We accept this in exchange for not discarding verified, hook-passing, merged work over a tooling miss, and because the integration branch is not `main`. Supersedes the relevant clause of the prior revert-all-unreviewed posture in `pipeline.md`.
