# Plan: overnight-review-gate-reverts-a-clean

## Overview
Thread the review agent's `DispatchResult.success` into the verdict decision and add an orthogonal `ReviewResult.could_not_run` boolean, so the gate distinguishes a *could-not-run review* (agent completed, no usable verdict → preserve the merge + flag for human re-review) from a *genuine dispatch crash* (agent failed → keep today's revert-and-trip). The signal is set once in `dispatch_review` (pipeline layer) and consumed in three `outcome_router` gate sites, `report.py`, and `runner.py`; a distinct `review_no_artifact` systemic cause-class still pauses the batch on a systemic adherence failure without discarding individual work.
**Architectural Pattern**: pipeline
<!-- A new orthogonal signal threaded through the dispatch → outcome-router → report/runner stages; no new state machine or event bus. -->

## Outline

### Phase 1: Distinguish & preserve safely (tasks: 1, 2, 3, 4, 5)
**Goal**: Discriminate could-not-run from crash, preserve the merge at all three gate sites, fix the latent stale-verdict bug, and surface the preserved merge correctly on both the morning report and the integration PR.
**Checkpoint**: A could-not-run review (success=True, no parseable `review.md`) preserves the already-merged feature (no revert), the `feature_deferred` event carries `could_not_run=True`/`merge_reverted=False`, the morning report renders the "unreviewed merge preserved" annotation, the integration PR carries an unreviewed marker, and a genuine crash (success=False) still reverts. `just test` exits 0.

### Phase 2: Signal hygiene, systemic detection & docs (tasks: 6, 7, 8, 9)
**Goal**: Stop mislabeling could-not-run as a crash, feed the systemic breaker under a distinct `review_no_artifact` cause-class wired end-to-end, amend the documented contract + registry, and record the ADR.
**Checkpoint**: The could-not-run path sets `could_not_run` but not `review_dispatch_crashed`; 3 could-not-run features (or a mixed crash+no-artifact batch) trip the breaker tagged `review_no_artifact` while their merges stay preserved; the events-registry, `pipeline.md`, and ADR 0014 reflect the split. `just test` exits 0.

> **Phases 1 and 2 land in ONE PR — the split is implementation/review sequencing, not independent shippability.** Phase 1 alone is behaviorally safe (the merge is preserved and the report/PR consumers key on the POSITIVE `could_not_run` flag, which is correct in both states; verified that NO code in `cortex_command/` reads `review_dispatch_crashed`, so the Phase-1 co-set of both flags is event-log-cosmetic, not a code-consumer hazard). But shipping Phase 1 without Phase 2 would leave the per-feature `review_dispatch_crashed` mislabel and the systemic `cause_class` still reading `review_dispatch_crash` for a no-artifact batch. Both phases ship together so the final state is fully correct.

## Risks

- **Req 5 (integration-PR flag) is the least-specified change** — `integration_degraded` has no runtime producer today (fixture-only) and `warning_file` is read but never written, so Task 5 must build both the aggregate-detection and the producer wiring from scratch. The exact site where the batch's preserved-could-not-run set is available at PR-creation time needs implementer investigation; this is the most likely source of in-flight scope discovery.
- **The shared flag-helper + per-site revert guard (Task 3) is the highest-risk change** — three sites with divergent revert mechanics (`revert_merge` with `.aborted`/`.success` vs `_reset_ff_merge` returning a bare bool) must set `could_not_run`/`merge_reverted` coherently and exception-safely. **The subtlest failure mode (per critical review): the in-band preserve block is nested inside the crash-`try`, so the crash-`except` handler — not just the revert-failure handler — must be made could-not-run-aware, or an exception while preserving reverts-as-crash.** A bug here either discards verified work (revert fires on could-not-run) or ships unreviewed code mislabeled as reverted.
- **The PR marker is advisory on the success path, by design** — the spec/ADR deliberately chose an integration-PR text marker over drafting the whole PR (drafting a healthy batch would block its co-resident *reviewed* merges, gating N good features on one unreviewed one). Critical review pushed for a structural draft-block; we keep advisory on success but added a **draft-on-failure** degraded fallback (Task 5) so a marker-write failure cannot silently publish an unmarked non-draft PR. If the operator wants a stronger structural gate (e.g. always-draft when any could-not-run merge is present), that is a spec-level change to revisit.
- **`max_tokens`/context-window truncation classifies as could-not-run (preserve), diverging from `budget_exhausted` (revert)** — a review truncated mid-run returns `success=True` (only `budget_exhausted` sets `success=False`), so a truncated no-verdict review is preserved + flagged, while its sibling resource-exhaustion case reverts. This is consistent with the success-based discriminator's intent (agent ran, produced no usable verdict → could-not-run) and is covered by the success-true+no-artifact test cell, but it is an asymmetry the operator should know exists; `stop_reason` is deliberately not threaded (research decision).
- **Phase split is sequencing, not separate shippability** — both phases land in one PR (see the Outline note). The Phase-1-only doubled-signal (`review_dispatch_crashed` + `could_not_run` co-set) is event-log-cosmetic: verified that NO `cortex_command/` code reads `review_dispatch_crashed`, so no consumer mislabels.
- **Dispatch mode**: this edits the `cortex_command` package, so Implement should use sequential trunk dispatch, not worktree — `just test` runs the editable install and a worktree would verify stale code (per the established package-editing constraint). The Depends-on chain is largely linear, so sequential ordering is natural.
- **Concurrent live lifecycle on main** (`overnight-dispatch-sends-opus-only-xhigh`, also Plan-paused) — commits must use explicit pathspec so the other session's staged files don't leak in.

## Acceptance
An overnight batch in which the review agent completes (`success=True`) but writes no parseable `review.md` leaves the already-merged feature **on** the integration branch (no revert), records a `feature_deferred` event with `could_not_run=True`/`merge_reverted=False`/no `review_dispatch_crashed`, renders the "unreviewed merge preserved — needs re-review" annotation on the morning report (not "do NOT re-run" or the generic re-run text), flags the integration PR with an unreviewed marker naming the feature, and feeds the systemic breaker under `cause_class` `review_no_artifact` (3 such features, or a mixed crash+no-artifact batch, trip at `SYSTEMIC_FAILURE_THRESHOLD` and pause the batch while each merge stays preserved) — while a genuine dispatch crash (`success=False`) still reverts and trips as `review_dispatch_crash`. `just test` exits 0; the events-registry audit passes.
