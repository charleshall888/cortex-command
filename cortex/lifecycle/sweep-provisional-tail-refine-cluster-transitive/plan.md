# Plan: sweep-provisional-tail-refine-cluster-transitive

## Overview

Verify-then-apply-or-refute the 42 filtered provisional-tail candidates (~7.9k weighted tokens) across a 12-file set, one file per task: each task performs a single coordinated edit of its file, regenerates the plugin mirror where one exists, runs the file's pinned test(s), and commits via `/cortex-core:commit` as **exactly one commit** carrying both the edit and its verdict record. Every applied trim cites one independent checkable signal (Req 3 a/b/c); every candidate's verdict is emitted as a fenced `verify-outcomes` block **in that file's commit body**, so Phase 3 assembles `verify-outcomes.md` deterministically by extracting those blocks from the **raw** commit body (`git show -s --format=%B <hash>`, not indented `git log`) and attaching each block's commit hash. The `applied_in_commit` hash exists only post-commit, so trim tasks record it in the commit body itself and Phase 3 backfills nothing — it only relocates and **cross-checks** each hash (the cited commit's diff must actually touch the row's `file`; see Task 12).

**Disjointness is on declared source edits, not the write set.** Each trim task's *declared* `Files` are disjoint from every other's, but `just build-plugin` regenerates the whole `plugins/` mirror tree, so the tasks are **not** disjoint on their actual write set — which is exactly why they cannot run concurrently. Trim tasks are therefore serialized by a linear `Depends on` chain (1→11); the lifecycle Implement phase batches on `Depends on` and dispatches each batch's tasks concurrently, so a linear chain structurally forces batch-size-1. This is interactive daytime work on `main` (no worktree, per spec Non-Requirements).

**Architectural Pattern**: pipeline

## Risks

- **Serialization via `Depends on` chain (Tasks 1→11) is deliberate, not accidental.** The trim tasks are disjoint on their *declared source edits* but NOT on their write set — `just build-plugin` regenerates the whole `plugins/` mirror tree and the drift pre-commit hook fails on a split mirror, so two concurrent trims would race the mirror tree even though their declared `Files` look disjoint (the Implement co-scheduling disjointness check keys on declared `Files` and cannot see this). The chain is the structural fix (linear `Depends on` → batch-size-1 → strictly sequential); it encodes the spec's interactive-daytime-on-`main`, no-worktree stance (Non-Requirements). Parallelism would require worktree isolation, which the spec rejected as riskier than the mechanical gain warrants.
- **Commit-body `verify-outcomes` block as the trim→Phase-3 interface — a self-report the gate must audit, not trust.** Chosen over trim tasks writing `verify-outcomes.md` directly, because `applied_in_commit` needs a post-commit hash and direct writes would make trim tasks share a file. The block is a self-report by the trimming agent, so Task 12 does not merely check pair membership (which is blind to a present-but-wrong row): it provenance-checks each hash (`git show --name-only` must list the row's file), verifies signal-(a)/(c) cited locations exist, and runs a normative-token scan on every signal-(b) quoted span. Two structural assumptions the interface still rests on and that the plan now pins: (i) each trim task is **exactly one commit** carrying both the edit and its block (Overview), so the hash unambiguously contains the span; (ii) Phase 3 reads the **raw** commit body (`git show -s --format=%B`), not indented `git log`.
- **`applied_in_commit` durability under a squash merge.** Recorded hashes are valid on the working branch. If this feature is later PR-flowed and **squash-merged**, the 11 trim commits collapse to one hash and every recorded `applied_in_commit` becomes a dangling ref on `main`. Mitigation: the spec's stance is direct-on-`main` (no branch), and Task 13's follow-up already instructs the #357 reconciler to re-locate rows by `(file,id)` + heading+token — so `applied_in_commit` degrades to advisory-provenance rather than a hard key if history is rewritten. Surfaced as an FYI on the approval surface; if a branch/PR mode is chosen at §4, prefer a merge-commit or rebase-merge over squash to keep the hashes live.
- **`adr/README.md` (Task 5) is the least-gated, highest-fan-out file** — no mirror, no prose test. Its automated safety net is now the *conjunctive* three-name grep (`MUST automatic`/`MUST NOT automatic`/`SHOULD surface` each checked separately) plus the signal-correctness scan, not a `MUST`-substring floor; residual correctness still rests on the elevated per-candidate signal discipline. s3 is refuted-by-default to protect live #304's citation.
- **Anchor bump (Task 1) edits a lifecycle-domain file (`kept-pauses.md`) owned by none of the four #357 children.** Authorized by Req 5/Req 10 as the 13th file, but it is the one cross-domain edit — if the net line shift turns out ≤28, the bump is unnecessary and should be skipped (fallback path).
- **Ledger is not written (Req 12).** Refuted verdicts produce no diff and will evaporate unless Task 13's follow-up item is actually filed and later executed — the discharge mechanism is the only thing preventing re-proposal during the deferral window.
