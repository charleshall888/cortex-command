# Specification: Sweep provisional tail — refine cluster + transitive tail (#361)

## Problem Statement

Backlog #361 is one of four file-disjoint children of umbrella #357, which sweeps the *provisional tail* of the skill-value audit — candidates that were scored and mechanically pin-scanned but **never adversarially verified**. This child owns 42 candidates (~7.9k weighted tokens) across a 12-file set: the refine skill cluster (`refine/SKILL.md` + its `specify.md`/`clarify.md`/`clarify-critic.md` references) and a transitive tail of otherwise-unowned single files (`adr/README.md`, `research/references/fanout.md`, `pr/SKILL.md`, `plan-synthesizer.md`, `backlog/SKILL.md`, `commit/SKILL.md`, `research/SKILL.md`, `interview/references/loop.md`). The benefit is lower per-session token cost on skill-instruction hot paths; the risk is that a prose trim that looks "non-load-bearing" silently shifts agent behavior. The verification bar (single-pass, set deliberately by parent #357) is kept, but strengthened here from *self-attestation* to an **independent checkable signal** per trim, following the convention sibling #358 (`cortex-requirements-area`) established. Verify outcomes are recorded per child in the `(file,id)`-keyed shape #358 uses, for the deferred #357 reconciliation to fold in.

## Priority (MoSCoW)

All requirements below are **Must** — each is a verify-then-apply-or-refute of an in-scope candidate, a coordination/scope gate, or the batch's integrity gate; a candidate that fails verification becomes a recorded refute (still Must), not a silent skip. There is deliberately **no Should/Could tier**. The **Won't** tier is the `## Non-Requirements` section (deferred ledger write-back, cross-child dedup, other-ticket spans, frontmatter, fixed byte target, reopening the parent bar, overnight execution).

## Phases

- **Phase 1: Refine-cluster trims** — verify + apply the 20 refine-cluster candidates, with the kept-pauses-parity anchor bump handled as a first-class in-commit step.
- **Phase 2: Transitive-tail trims** — verify + apply the 22 transitive-tail candidates, with `adr/README.md`, `plan-synthesizer.md`, and the two nested-span pairs handled under their elevated bars.
- **Phase 3: Outcomes record, reconciliation-debt discharge + integration gate** — emit the `(file,id)`-keyed `verify-outcomes.md`, file the reconciliation follow-up item, tally savings, and pass the full-suite gate before Review.

## Requirements

1. **Candidate set is exactly the 42 filtered rows, keyed `(file, id)`.** Operate only on `master_candidates.json` rows where `status == "unverified"` ∧ `file` ∈ the 12-file set ∧ no `overlaps_ticket` ∧ no `reproposal_of` → exactly 42 rows. The ledger `id` alone is **non-unique** (the 42 rows carry only 23 distinct ids; e.g. `s3`×5, `file-compress`×4), so every downstream key/count is on the composite **`(file, id)`** (42 distinct pairs), never `id`. The count is computed at spec time against the filtered *rows*; the invariant holds only because the deferral (Req 12) keeps any sibling/reconciliation from mutating `status` mid-batch — this batch does not depend on a pinned ledger SHA but does depend on that deferral holding across all four children. Acceptance: `python3` filter yields 42 rows with 42 distinct `(file,id)` pairs; every `(file,id)` in Phase 3's `verify-outcomes.md` is in the filtered set and vice-versa (subset + accounts-for-all). **Phase**: Outcomes record, reconciliation-debt discharge + integration gate.

2. **Locate by heading + pinned token, never by ledger line number.** Ledger `start_line`/`end_line` are advisory and several have drifted (confirmed: `research/SKILL.md` s15 live L199-206 vs ledger 203-211; `clarify-critic.md` s12b end_line 177 vs 176-line file; `commit/SKILL.md` s6 targets lines that no longer exist). Every span — including the two other-ticket protected spans in Req 9 — is located and cited by its section heading + a distinctive `pins[]` token. Acceptance: no edit or protected-span check uses a ledger line number as anchor; each `(file,id)` outcomes row names the heading it was located by, and that heading exists verbatim in the post-trim file. **Phase**: Refine-cluster trims, Transitive-tail trims.

3. **Every applied trim cites an independent checkable signal — not self-attestation.** A trim applies only when the span it removes is confirmed non-load-bearing by **at least one recorded, checkable signal** (a bare self-authored "looks safe" reason fails the gate — that would rubber-stamp the trimming author's own decision, the exact failure mode where an under-scoped keep-clause greps itself green). The permitted signals, one of which every applied trim must cite in its outcomes row: **(a) structural-substitution** — name the specific test / hook / gate / other-doc location that already enforces the constraint the prose encodes, so the prose is provably redundant *of that enforcement*; **(b) informative-only** — quote the removed span to show it contains no `shall`/`must`/behavior-determining/weighting statement (rationale, example, or history only); **(c) preserved-elsewhere** — for a normative or deduped line, cite the surviving canonical location that still literally carries it. Verification is a **whole-file pass**, not a span-isolated judgment. The keep-token two-sided check (per #353) still runs as a floor: (i) each keep token still present AND (ii) the removed phrase's occurrence count dropped — but token *presence* is file-scoped and does not by itself certify the cut; the load-bearing instance is confirmed by locating it via heading+token (Req 2) and, for a deduped token, the signal-(c) surviving-location cite. `mech_pins` are a vacuous file-level cooccurrence pool (e.g. plan-synthesizer s15's are every file containing `"medium"`) and are **advisory only** — they never substitute for a signal. Acceptance: each applied `(file,id)` row names its signal type (a/b/c) and, for (a)/(c), the enforcing/surviving location; rows with only a bare reason fail the gate. **Phase**: Refine-cluster trims, Transitive-tail trims.

4. **Preserve What/Why intent, weighting, and user-affordance prose even when untested.** The signal-(b) "informative-only" test structurally excludes any span carrying decision-criteria / intent (CLAUDE.md "prescribe What and Why, not How"), weighting/emphasis, or a user-facing-affordance boundary — such a span cannot be classified informative-only and must instead survive (verbatim or with the weighting kept explicit in the surviving text). This covers load-bearing prose nobody thought to enumerate: the burden is on signal-(b) to prove absence of normative/weighting content, not on a hand-list to catch it. Named high-risk items to confirm surviving (greps recorded): `clarify.md` §5 criticality "high = appropriate default for most skill and agentic layer changes" + the "do not ask the user to confirm" anti-pause line; `specify.md` §2b's four distinct failure-mode sub-checks incl. "state ownership"; `interview/loop.md`'s "ask one at a time" + early-exit + soft-cap-check-in affordances; `plan-synthesizer.md`'s `## Untrusted Variant Data` injection-defense body + the letter-token verdict rule; `adr/README.md`'s MUST/MUST-NOT/SHOULD consumer-rule names + status-keying + the CLAUDE.md prose-only sentence. Acceptance: each named item greps present after its file's trim; any candidate whose only safe form removes such prose is REFUTED, not applied. **Phase**: Refine-cluster trims, Transitive-tail trims.

5. **Kept-pauses parity: bump the anchor in-commit as the primary mitigation.** `tests/test_lifecycle_kept_pauses_parity.py` (`LINE_TOLERANCE = 35`) anchors `refine/SKILL.md:166`; the §4-gate `AskUserQuestion` is at live line 159 (7 below the anchor → the true budget is a net upward shift of **≤28** lines before `|166−(159−N)| = 7+N` exceeds 35 at N=28+1). Because five candidates (s3, s6, s7, s8, s10a) trim ~75 source lines *above* line 159, a full-extent trim can exceed that budget — so the **primary mitigation is to update the `kept-pauses.md:166` anchor to the moved site in the same `refine/SKILL.md` commit** (Req 10 grants standing), which lets every candidate trim to its verified extent without an artificial cap. The `<28`-line cap is a fallback only when the measured post-draft delta already fits (no anchor edit then needed). Additionally, candidate **s10c edits line 159 itself** (the §4-gate bullet, a kept-pause affordance): it must be kept a single line (in-place compression) so the `AskUserQuestion` token does not migrate independently; if s10c would reflow into sub-bullets, treat the resulting site move as part of the same anchor bump. Acceptance: `.venv/bin/pytest tests/test_lifecycle_kept_pauses_parity.py -q` exits 0 after the `refine/SKILL.md` commit (run per-file); if the anchor was bumped, `kept-pauses.md` + its cortex-core mirror are staged in that same commit; s10c's post-trim `AskUserQuestion` token is still on a single line. **Phase**: Refine-cluster trims.

6. **Coordinated single edit for nested same-file spans; each inner candidate gets its own verified outcomes row.** Two candidate pairs nest and must be applied as one coordinated edit, never two independent passes: `plan-synthesizer.md` `file-compress` ⊃ `s15`; `backlog/SKILL.md` `file-compress` ⊃ `s11`. The single coordinated edit must preserve the inner candidate's keep-list, and **both** the inner and outer `(file,id)` get their own outcomes row each carrying its own Req 3 signal — the shared commit hash does not let the inner candidate ride the outer's verification. Acceptance: for each nested pair, `verify-outcomes.md` contains two distinct `(file,id)` rows (inner + outer), each with its own signal; the inner candidate's keep-list tokens grep present post-edit. **Phase**: Transitive-tail trims.

7. **`adr/README.md` treated as the highest-risk file; s3 refuted-or-surfaced.** `cortex/adr/README.md` is the least-gated file (no pre-commit trigger, no mirror, no prose test) with the highest fan-out (the ADR policy doc other files back-point to). All 6 candidates get the elevated Req 3/Req 4 care. It may be applied only if implement-time inspection confirms #304 no longer depends on those lines AND the CLAUDE.md prose-only sentence is kept verbatim (only the brittle line-number cite dropped). Acceptance: `verify-outcomes.md` shows s3 as `verified_refuted` with the #304-citation reason, OR as applied with a recorded confirmation that #304 no longer depends on those lines and `grep -c 'prose-only enforcement is appropriate only' cortex/adr/README.md` ≥ 1. **Phase**: Transitive-tail trims.

8. **`commit/SKILL.md` s6 recorded as refuted (moot).** The `## Validation` section s6 targets was removed in commit `8c3a00b9`; `skills/commit/SKILL.md` is now 31 lines with `grep -c '## Validation'` = 0. - Does **not** single-source the cross-child `clarify.md` ↔ `discovery/references/clarify.md` dedup (~86 tokens) — out of scope (would edit sibling-#359-owned file). Intra-file dedup within the 12 files is in scope.
- Does **not** touch the `clarify.md` s9 (#340) or `clarify-critic.md` s3 (#186) spans (Req 9).
- Does **not** edit SKILL.md frontmatter / L1 surface (verified none of the 42 candidates touch frontmatter); the sole L1-ratchet frontmatter candidate is sibling #359's. `kept-pauses.md` (Req 10) is the only non-candidate file edited, and only to move the parity anchor.

## Edge Cases

- **Greps-clean but behavior-shifts** (Web finding — harm concentrates in edge cases no test exercises): the file-scoped keep-token grep certifies presence, not preservation of the load-bearing instance; mitigated by Req 3's independent signal (a/b/c) + Req 2's heading+token location of the actual instance, not by count≥1 alone.
- **Under-scoped self-derived keep-list**: a candidate that omits its own endangered token would grep green; Req 3 forbids self-attestation and requires an independent structural/preserved-elsewhere signal.
- **Non-unique `id`**: 42 candidates, 23 distinct ids → all keying/counting is on `(file,id)` (Req 1/12); an `id`-keyed record would silently overwrite ~19 rows.
- **Cumulative line-shift** blows parity invisibly to per-candidate checks: Req 5 bumps the anchor in-commit (primary), covers s10c (edits the gate line), and fixes the ≤28 boundary.
- **Nested spans double-edited / inner absorbed**: Req 6 (coordinated single edit + separate verified row per inner candidate).
- **Manufactured cross-ticket drift**: this child's clarify.md/clarify-critic.md trims move #340/#186's un-applied rows; safe only because the reconciler keys by `(file,id)` + heading, which Req 13's follow-up item makes explicit.
- **Stale/vacuous pins**: Req 2 (heading location) + Req 3 (`pins[]`/signals, not `mech_pins`); pr/SKILL.md s3c's `commit/SKILL.md:52` cite re-anchored to line 31 before use.
- **Refuted candidate re-proposed during the deferral window**: the ledger shows these `unverified` until reconciliation; `verify-outcomes.md` is the committed interim record, and Req 13 files the discharge so the refute is not lost.
- **LAZY_REF extraction** (pr/SKILL.md s3b → new reference file): resolve `${CLAUDE_SKILL_DIR}` in the SKILL.md body and propagate the resolved path (SP001/SP002); the extracted file needs its own mirror entry.

## Technical Constraints

- File-based state (ADR-0001); no `cortex-update-candidate` script exists — the outcomes record is a plain committed file the reconciliation reads. The `(file,id)` key + the Req 13 follow-up mirror sibling #358's approach so the reconciler consumes all children uniformly.
- Commits via `/cortex-core:commit` only; imperative/capitalized/≤72-char subjects; canonical+mirror staged together (drift hook fails a split).
- `just test` aggregates the at-risk suites (`test-pipeline` → `test_plan_synthesizer.py`; `tests/` → parity + fanout-matrix); the pre-commit hook does not.
