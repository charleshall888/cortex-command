# Research: Harden the overnight pipeline against an out-of-vocabulary complexity value crashing the session

> **Re-scoped per Clarify.** The original ticket (#278) blamed a "criticality `medium` leaking into the complexity slot via a backlog-resolution failure." That diagnosis is **disproven by the code** (see Codebase Analysis §1 and Adversarial §3). This research targets the *real* defect: an out-of-vocabulary (OOV) complexity value authored into a feature's `plan.md`, and the cascade behavior that turned one bad feature into a 0/N session.

## Codebase Analysis

### Target 1 — Where an OOV complexity value enters `task.complexity`

The dispatch-bound complexity has **exactly one source**: the per-task `**Complexity**` field in each feature's `cortex/lifecycle/{feature}/plan.md`.

- `cortex_command/overnight/feature_executor.py:532` — `feature_plan = parse_feature_plan(plan_path)` (path is `cortex/lifecycle/{feature}/plan.md`).
- `cortex_command/pipeline/parser.py:317` — `complexity = _parse_field_string(task_body, "Complexity") or "simple"`. The `or "simple"` fires **only when the field is absent**; a *present* value passes through verbatim.
- `cortex_command/pipeline/parser.py:377-383` — `_parse_field_string` returns `match.group(1).strip()` with **no vocabulary validation**. Any string survives: `medium`, `moderate`, even the literal template placeholder `trivial|simple|complex`.
- `cortex_command/pipeline/parser.py:325` — written into `FeatureTask(complexity=…)` (dataclass field at `parser.py:56`, default `"simple"`).
- `cortex_command/overnight/feature_executor.py:623` — `complexity=task.complexity` passed to `retry_task`.

**The generator (prime suspect).** The `**Complexity**` field is authored by the plan-phase LLM agent, instructed by `skills/lifecycle/references/plan.md:170,179` (template literally shows `- **Complexity**: trivial|simple|complex`) and the classification table at `plan.md:209-221`. The overnight plan-gen path that writes these files is `cortex_command/overnight/prompts/orchestrator-round.md:412`. Nothing constrains the model from emitting a criticality-style word (`medium`) into a Complexity field — and two complexity vocabularies coexist in the harness (discovery/lifecycle sizing uses `{simple, complex}`; dispatch uses `{trivial, simple, complex}`; criticality uses `{low, medium, high, critical}`), making `medium` a foreseeable authoring slip, not a one-off.

**Batch-plan derivation is NOT a dispatch vector.** `cortex_command/overnight/batch_plan.py:62-69` derives a feature-level row complexity as the **max tier** across tasks; its `else` branch silently coerces any OOV value (incl. `medium`) to `trivial`. This explains the ticket's observation that `batch-plan-round-1.md` showed `Complexity: complex` while a task still carried `medium`: the `complex` task wins the max, masking the bad per-task value. But `parse_master_plan` (`orchestrator.py:250-251`) consumes only `f.name` — the batch-plan complexity column is **never** passed to dispatch. Only the per-task `plan.md` value reaches it.

### Target 2 — Fail-soft at the dispatch boundary (per-feature isolation ALREADY works)

An OOV complexity hits **two** raise sites, the earlier one first:
- `cortex_command/pipeline/retry.py:215` — `resolve_model(complexity, _criticality_val)` at `retry_task` entry, **before** the retry loop. `resolve_model` raises `ValueError` at `dispatch.py:231-235`. **This is the first raise.**
- `cortex_command/pipeline/dispatch.py:519-524` — the `tier is None` guard in `dispatch_task` (a backstop only reached if the first were bypassed).

The `ValueError` propagates out of `retry_task` (no try/except at `retry.py:215` or `:256`), out of `_run_task` (`feature_executor.py:558`), and is **caught** by `asyncio.gather(..., return_exceptions=True)` at `feature_executor.py:678-689`, which returns `FeatureResult(status="failed", error="Unexpected error: Unknown complexity tier 'medium'…")`. The `"Unexpected error:"` prefix (`feature_executor.py:688`) matches the ticket's reported string verbatim. A **second** outer boundary exists at `orchestrator.py:485-501` (`gather(return_exceptions=True)` → `FeatureResult(status="failed")` → `apply_feature_result`).

**Conclusion: in the current shipped code, the crash is already isolated to a single feature** — it does not propagate fatally. The reusable fail-soft machinery is this `gather(return_exceptions=True)` pattern; typed per-feature failures are already surfaced as `FeatureResult` (`feature_executor.py:533-539` parse error, `:547-552` dependency error).

### Target 3 — Runtime backlog resolver behind `backlog_write_failed`

Not in `common.py` and not the CLI resolver. It lives in `cortex_command/overnight/outcome_router.py`:
- `events.py:50` — `BACKLOG_WRITE_FAILED = "backlog_write_failed"`.
- `outcome_router.py:421-428` — `except Exception` in `_write_back_to_backlog` logs the event and **swallows** (best-effort; never aborts the session — docstring `outcome_router.py:388-393`).
- `outcome_router.py:402-405` — "Backlog item not found" raised when the resolver returns `None`.
- `outcome_router.py:351-378` — `_find_backlog_item_path(feature, backlog_id)`, three strategies: (1) exact slug match — strip `NNN-` from each `cortex/backlog/[0-9]*-*.md` stem and compare `slug_part == feature`; (2) match by zero-padded `backlog_id` glob; (3) substring fallback via `_backlog_find_item`.

**Why slug ≠ filename-stem breaks it:** strategy 1 requires `feature` to equal the kebab slug embedded in the filename. The overnight `feature` key derives from `lifecycle_slug` (`backlog.py:104-120`, priority `lifecycle_slug` → dirname → `slugify(title)`); when that diverges from the filename's kebab portion, strategy 1 misses, strategy 2 fires only if `backlog_id` was threaded (`orchestrator.py:278` from `fs.backlog_id`, `None` when absent), and strategy 3 can mismatch.

**Durable fix available:** the canonical robust resolver `cortex_command/backlog/resolve_item.py:resolve` already implements a 5-step chain (UUID-prefix → numeric → kebab with symmetric `NNN-` strip → `lifecycle_slug` frontmatter equality → title-phrase). Frontmatter fields `uuid`, `backlog_id`, `lifecycle_slug` are parsed (`backlog.py:335`) but unused by the runtime resolver. Route `_find_backlog_item_path` through `resolve_item.py` instead of extending ad-hoc heuristics. **Note:** this resolver feeds only status *write-back*, NOT any dispatch argument — it is independent of the complexity value (Adversarial §8).

### Target 4 — Plan-gen commit lands on home `main` instead of `overnight/{session_id}`

- The commit is performed by the **orchestrator-round agent** (a `claude -p` subprocess), instructed by `orchestrator-round.md:439` ("commit them using `/commit`"), not by Python.
- `_spawn_orchestrator` (`runner.py:1036-1055`) is spawned with **no `cwd=`** → inherits the runner's cwd = the home repo (on `main`, per `runner.py:993-995`).
- The integration branch is a **separate worktree** (`outcome_router.py:182` `git worktree add`), so the home working tree stays on `main`.
- `git` is **excluded from the orchestrator sandbox** (`runner.py:1025` `excluded_commands=["git:*"]`), so the agent's `git` runs unsandboxed in the home cwd; the path-level main-ref deny (`sandbox_settings.py:55-60`) is bypassed because git's own process does the ref update.
- **Correct pattern already in tree:** `_commit_followup_in_worktree` (`runner.py:436-527`) runs `git` with explicit `cwd=str(worktree_path)` precisely "so the follow-ups land on the integration branch, not the home repo" (`runner.py:444-447`). The fix: pin the plan-gen commit's cwd to the integration worktree (available from `state.integration_worktrees`).

### Conventions to follow
- **Fail-soft isolation:** `asyncio.gather(..., return_exceptions=True)` + per-item `isinstance(..., BaseException)` → `FeatureResult(status="failed")` (`feature_executor.py:678-689`, `orchestrator.py:485-535`).
- **Best-effort write-back:** swallow + log a structured event, never abort (`outcome_router.py:421-428`).
- **Commit-in-worktree:** explicit `cwd=worktree_path` (`runner.py:436-527`).
- **Resolver reuse:** prefer routing through `backlog/resolve_item.py:resolve` over new filename heuristics.
- **Gate:** edits to `cortex_command/overnight/` and `cortex_command/pipeline/` do NOT require the lifecycle skill; edits to `cortex_command/common.py`, `skills/`, `hooks/` DO (per CLAUDE.md).

## Web Research

- **Python `enum._missing_(cls, value)`** ([docs.python.org/3/library/enum.html](https://docs.python.org/3/library/enum.html)) is the canonical normalize-at-the-boundary hook: returning a member maps an unknown value onto a known one; returning `None` lets the normal machinery raise `ValueError`. Co-locates the normalize-vs-reject policy in one place.
- **"Parse, don't validate"** (Alexis King; [bitecode.dev](https://www.bitecode.dev/p/what-parse-dont-validate-means-in), [deviq.com](https://deviq.com/practices/parse-dont-validate/)) — parse external data into a precise type at the entry boundary so downstream code can trust it. The parser *is* the validator. Python lacks Result/Option, so exceptions are the mechanism — convert the raw string to an enumerated type at the edge.
- **Anti-robustness principle** ([Beeminder](https://blog.beeminder.com/postel/), [Robustness principle — Wikipedia](https://en.wikipedia.org/wiki/Robustness_principle)) — the strongest case *against* silent normalization: leniency breeds delayed/mysterious failures and an irreversibility trap. Crucial: **"strict-first is reversible, lenient-first is not"** — you can always loosen later, but once callers depend on lenient behavior you can't retrofit strictness. If you normalize, it **must be loud** (a logged WARNING naming the bad value + substitution); a silent default is the worst of both worlds.
- **Error-aggregation / dead-letter patterns** ([KDnuggets](https://www.kdnuggets.com/5-error-handling-patterns-in-python-beyond-try-except), [Redpanda DLQ](https://www.redpanda.com/blog/reliable-message-processing-with-dead-letter-queue)) — the canonical fail-soft batch shape: detect the poison value early, isolate the failing unit, log with triage metadata, keep the healthy stream moving.
- **Boundary validation vs defense-in-depth** ([Security Innovation](https://blog.securityinnovation.com/blog/2011/09/validating-trust-boundaries.html), Bloomberg P2053) — a single chokepoint at the trust boundary is "easier to implement correctly and much easier to maintain." Defense-in-depth adds redundant *checks* (internal-invariant assertions) for resilience, but **two places *normalizing* the same value is the anti-pattern** — it splits the policy. One place parses; downstream sites only assert the invariant.
- **Fail-fast vs fail-safe** ([DZone](https://dzone.com/articles/fail-fast-principle-in-software-development), [Google SRE — cascading failures](https://sre.google/sre-book/addressing-cascading-failures/)) — compatible at different scopes: **fail-fast *within* a unit, fail-safe *across* units.** Synthesis for this bug: raise a precise typed error at the smallest unit (the tier boundary), catch it at the per-feature orchestration boundary, record + skip — don't normalize into a silent default, don't let it reach the whole-session frame.

## Requirements & Constraints

- **Fail-forward is a hard requirement.** `pipeline.md` (Feature Execution acceptance criteria): *"One feature's failure does not block other features in the same round (fail-forward model)."* `multi-agent.md`: *"One feature's failure does not abort other in-flight features."* `project.md` Quality Attributes: *"Graceful partial failure: Tasks may fail. The system retries… fails gracefully — completing the rest."* The **only sanctioned whole-session pauses** are budget-exhaustion and rate-limits (`pipeline.md` NFR "Graceful degradation"; `multi-agent.md:64`). An OOV-complexity crash zeroing the session is **out of the sanctioned set → a requirements violation.**
- **Integration-branch commit requirement (explicit).** `pipeline.md` Session Orchestration: *"Artifact commits (lifecycle files, backlog status updates, session data) land on the integration branch, not local `main` — they travel with the PR."* Documented sole exception: the tracked `cortex/lifecycle/morning-report.md` copy is committed to `main` by the runner. Plan-gen `plan.md` files are lifecycle artifacts → must land on `overnight/{session_id}`.
- **Feature status vocabulary.** `pending → merged` (success); `running → paused` (recoverable, **auto-retries** on resume — would re-hit the same bad value); `running → deferred` (human decision, sourced from `action:"question"`/CI block/review verdict); `running → failed` (**unrecoverable**). The spec does **not** explicitly map a config/validation failure to a status; by elimination an invalid complexity best fits `failed` (paused would loop, deferred is for ambiguous intent). `observability.md` requires `failed`/`deferred` be surfaced with error messages.
- **Enum vocabularies live in code, not requirements.** `TIER_CONFIG` (`dispatch.py:127`) = `{trivial, simple, complex}`; `_VALID_CRITICALITY` (`dispatch.py:191`) = `{low, medium, high, critical}`. Requirements enumerate them only descriptively (`multi-agent.md:57`). Precedent for dual-location enum maintenance: backlog `TERMINAL_STATUSES` (`common.py` + `overnight/plan.py:_TERMINAL`, with an "update both" rule) — though complexity currently lives in a single location.
- **No requirements constraint** governs `_resolve_user_project_root` / CWD-independence / lifecycle-slug→backlog mapping — these are code-level. Nearest precedent: the worktree-root chokepoint (`resolve_worktree_root()` in `pipeline/worktree.py`), a model for routing path resolution through one canonical function.
- **Philosophy bearing on scope.** `project.md`: *"Complexity must earn its place… When in doubt, simpler wins"*; *"Solution horizon… if the patch applies in multiple known places, propose the durable version."* The complexity enum is read at multiple sites (`dispatch.py:231`, `:521`; `metrics.py` ×4) and the OOV value flows through one construction site (`parser.py:317`) — so a single parser-boundary chokepoint meets the "durable" test.
- `glossary.md` is listed in `project.md` `## Global Context` but **does not exist on disk** (loaded-files note: `cortex/requirements/glossary.md` (skipped: file absent)).

## Tradeoffs & Alternatives

### A. Fail-soft policy for the OOV value
- **A1 — Normalize to a default at dispatch (`dispatch.py:519`).** ~5 lines (`.get(complexity, DEFAULT)` + warning). **Rejected as primary:** normalizes at the *deepest* layer, where all context to pick a sensible default is lost; masks the authoring bug indefinitely; functionally equivalent to the EDGE-forbidden "add `medium` to TIER_CONFIG," one layer up.
- **A2 — Fail only that feature (catch in caller).** **Already implemented** (`feature_executor.py:684-689`). Honest improvement: give it a typed `error_type` (e.g. `invalid_complexity`) instead of the opaque `"Unexpected error: …"` so morning-review/brain-triage route it as a config fault. **But A2 alone does not change the reported severity** — the feature already fails cleanly today and the session still died via the cascade (see Adversarial).
- **A3 — Validate at parse time (`parser.py:317`).** Two variants: **(A3a) reject** — raise a descriptive `ValueError` that the existing `batch_plan.py:76-80` exclude-with-reason path converts to an `excluded` entry; **(A3b) normalize-at-parse** — coerce unknown → within-vocab with a logged WARNING, at the layer where it's still a known parse artifact. Best maintainability ("parse, don't validate"); surfaces the error pre-dispatch where it's actionable. **Caveat (Adversarial §1): A3a does NOT prevent the 0/N outcome for a feature with dependents.**

### B. Validation placement
- **Single chokepoint at `parser.py:317`** is genuinely sufficient — it is the *sole* construction site for the dispatch-bound `FeatureTask.complexity` (`batch_plan.py` complexity never reaches dispatch). The dispatch-layer guard (`dispatch.py:519`, `resolve_model:231`) **already exists** as a backstop. So the pragmatic answer is naturally **2-layer**: parser-boundary parse (primary) + keep the existing dispatch raise as a cheap invariant assertion. Do **not** add a third plan-gen-output validator (simpler wins; the parser sees 100% of plan output). Also fix `batch_plan.py:62-69`'s silent `else`-coercion so the batch layer doesn't hide the same class of error.

### C. Backlog resolver robustness
Independent of the complexity value (feeds only status write-back). Durable fix = route `_find_backlog_item_path` through `backlog/resolve_item.py:resolve` + add `lifecycle_slug`-frontmatter matching, rather than a 4th heuristic. Shares the CWD/root-resolution theme with D and #277.

### D. Plan-commit-target
Pin the plan-gen commit cwd to the integration worktree, mirroring `_commit_followup_in_worktree`'s `cwd=worktree_path`. Shares the **same CWD/root-resolution root cause** as the relative-path reads (`read_criticality`, the `plan.md` read) — see Adversarial §8.

### Recommended shape (synthesized)
Primary fix = **parser-boundary handling of OOV complexity** (A3) + **typed error** (A2) + **keep dispatch backstop** (B) + **fix `batch_plan.py` silent coercion**. The Tradeoffs agent recommended *splitting* C/D into a separate ticket; the Adversarial agent **partially refuted** that (D + the criticality/plan reads are one CWD/root-resolution bug → keep together; only C-the-resolver is cleanly separable). The user's Clarify decision was to **keep all four in scope**. **Net recommendation for Spec: keep all four, organized as two sub-themes — (i) OOV-complexity vocabulary boundary, (ii) CWD/root-resolution hardening (D + the relative-path reads, with C routed through the canonical resolver).** The harder, unresolved question is whether the **cascade** itself is in scope (below).

## Adversarial Review

- **Parser-reject does NOT fix the headline 0/N (verified).** `intra_session_blocked_by` (`orchestrator-round.md:188-196`) leaves a dependent `pending` until its blocker is `merged`. Excluding the OOV feature at parse time means it never reaches `merged`, so dependents (#026/#027 `blocked-by: 025`) **starve every round** → `merged_delta <= 0` for two rounds → zero-progress breaker (`runner.py:2474-2487`). Same 0/N outcome. **A3a fixes the crash symptom, not the severity.**
- **A second, faster breaker.** Three same-round features failing → `consecutive_pauses += 1` ×3 → `>= CIRCUIT_BREAKER_THRESHOLD` (=3, `constants.py:7`) → batch breaker `circuit_breaker_fired = True` (`outcome_router.py:750-751`) **intra-round**, before the zero-progress path. Parser-reject (excluded ≠ pause) avoids *this* breaker but not the starvation one.
- **Normalize-vs-reject is a genuine fork with opposite cascade behavior.** Reject → dependents starve (cascade persists, honors "don't mask"). Normalize-and-run → the feature can reach `merged` and unblock its chain (cascade avoided) but masks the authoring bug and may under-resource a `complex` task run as `simple` (wrong model/budget/turns per `TIER_CONFIG`). **Neither pure approach prevents 0/N for a feature with dependents** — only the feature *succeeding* or **breaking the blocked-by starvation loop** does. A third, unscoped piece may be required: "blocker failed/excluded → deterministically fail or release its dependents instead of leaving them `pending` forever."
- **Typed error_type is cosmetic for breaker math** unless also added to a suppression set — `outcome_router.py:712-715` treats an error specially only if `result.error in _SYSTEMIC_ERROR_TYPES` (`constants.py:14-18`). A bare new `error_type` still feeds the 3-pause breaker.
- **Per-feature isolation works, but "isolated" ≠ "session survives."** The double `gather` boundary contains the ValueError, but the contained failure still feeds the breakers. The synthesis conflated the two.
- **C/D coupling (refines the split).** `_find_backlog_item_path` (C) feeds only write-back → independent of dispatch. BUT three things share the **same CWD/relative-root footgun**: `read_criticality(feature)` uses relative default `Path("cortex/lifecycle")` (`common.py:544`) → wrong CWD yields the `medium` default; `plan_path = Path(f"cortex/lifecycle/{feature}/plan.md")` (`feature_executor.py:529`) is relative → the dispatch-vector read itself is CWD-fragile; `_spawn_orchestrator` passes no `cwd=` (`runner.py:1036`) → defect D. **A complexity fix that doesn't also pin the root could read the wrong/no plan.md.** Keep D with the complexity fix; pin the root once.
- **The specified regression test tests the wrong thing.** Complexity comes from `plan.md` (`parser.py:317`), never the backlog — so "assert dispatch receives the backlog's `complex`" tests a non-existent path and would pass while the bug persists. **False-confidence test already in tree:** `test_parser.py:142-159` (`test_mixed_separators_all_parsed`) feeds `Complexity="moderate"` (OOV) and asserts only counts — its current green status is *evidence the parser swallows OOV today*. A parser-reject fix must update it.
- **Correct regression test:** (a) `parse_feature_plan` on a plan with `**Complexity**: medium` rejects loudly or normalizes-with-warning; AND (b) end-to-end, one OOV feature degrades without tripping the session breaker or starving siblings — the test must cover the **cascade**, not just the dispatch call.
- **Caveat on inference:** the actual session log `overnight-2026-06-01-1518` is not in this tree (consuming-repo). "plan.md contained `**Complexity**: medium`" is the only code-consistent explanation but remains inferential; every *code* claim above is line-verified.

## Open Questions

The factual questions are resolved inline above. These remaining items are **design/requirements decisions deferred to the Spec interview (§4)** — they cannot be resolved by reading code; they are scope and policy choices for the user:

1. **Is the `blocked-by` cascade in scope for #278?** *Deferred to Spec.* This is the crux: per Adversarial §1–§3, neither normalizing nor rejecting the OOV complexity prevents the 0/N session outcome for a feature with dependents — only the feature reaching `merged`, or a change to dependent-starvation behavior (deterministically fail/release dependents when a blocker fails/excludes), delivers the headline value. The spec must decide: (a) scope #278 to the OOV-value handling only and accept that a *dependency-rooted* failure can still zero the session (documenting the cascade as a separate concern), or (b) include the cascade/dependent-release fix so a single non-systemic feature failure cannot zero its dependents.

2. **Normalize-and-run vs reject-and-exclude for the OOV value?** *Deferred to Spec.* The EDGE forbids masking; the fail-forward requirement favors keeping the chain alive. If normalize-and-run is chosen, what is the safe default for an OOV complexity word — `complex` (over-resource, safe) or `simple` (under-resource a genuinely complex task)? Resolution interacts with Q1 (normalize avoids the cascade; reject does not).

3. **Should the typed `invalid_complexity` error be suppressed from the consecutive-pause breaker count?** *Deferred to Spec.* Per Adversarial §6, typing the error is cosmetic for breaker math unless wired into a suppression set. Decide whether a config-validation failure should count toward the 3-pause batch breaker.

4. **Scope confirmation: keep all four targets, or split C/D?** *Deferred to Spec §4 complexity/value gate.* User's Clarify decision was to keep all four; Adversarial supports keeping D with the complexity fix (shared CWD root cause) and notes C is cleanly separable. The Spec's complexity/value gate should confirm the final boundary.
