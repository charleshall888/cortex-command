# Specification: overnight-feature-crashes-with-unknown-complexity

## Problem Statement

During an overnight session, a feature crashed with `Unknown complexity tier 'medium'` and the session ended 0/N merged. The original ticket blamed a "criticality value leaking into the complexity slot via a backlog-resolution failure" — but that diagnosis is disproven by the code: complexity (`task.complexity`, parsed from `plan.md`) and criticality (`read_criticality`) are sourced independently (`feature_executor.py:623-624`), and the per-feature crash is already isolated by `asyncio.gather(return_exceptions=True)` (`feature_executor.py:678-689`). The real defects are: (1) an out-of-vocabulary (OOV) complexity value authored into a feature's `plan.md` passes through the parser unvalidated (`parser.py:317`) and reaches the dispatch enum lookup; (2) when a feature does reach terminal `failed`, its `blocked-by` dependents are left `pending` and starve every round — the session churns wasted rounds until a circuit breaker stops it, and the morning report never explains why; and (3) three adjacent root-resolution defects (criticality/plan reads relative to CWD, a backlog resolver that misses when lifecycle-slug ≠ filename-stem, and a plan-gen commit landing on home `main`). This feature hardens all three so a malformed plan degrades gracefully, a failed blocker's dependents resolve deterministically with an accurate report, and the dispatch path resolves lifecycle files correctly regardless of CWD.

## Phases
- **Phase 1: OOV complexity fail-soft** — normalize an unknown complexity value to `complex` at the parser boundary so the feature still runs, and surface the normalization as a structured, report-visible event.
- **Phase 2: Cascade containment** — when a feature reaches terminal `failed`, an end-of-round sweep deterministically fails its in-session dependents so the loop terminates cleanly and the report explains the cascade.
- **Phase 3: Root-resolution hardening** — resolve the dispatch path's lifecycle reads/writes against the project root, route the runtime backlog resolver through the canonical resolver, and land the plan-gen commit on the integration branch.

## Requirements

_Priority (MoSCoW): R1, R2, R3, R4, R5, R6, R8 are **must-have** (the crash fix, its report-visible signal without which the fix silently masks the bug, the backstop integrity, the cascade containment, and the dispatch-critical plan/criticality reads). R7, R9, R10 are **should-have** (de-emphasis of cascade casualties in the report, and the two non-fatal-today adjacent defects — `backlog_write_failed` is swallowed and the stray plan commit crashed nothing). Won't-do items are enumerated in Non-Requirements._

1. **Normalize OOV complexity at the parser boundary, and expose which tasks were normalized.** When `parse_feature_plan` reads a per-task `**Complexity**` value that is present but not in `{trivial, simple, complex}`, it coerces the constructed `FeatureTask.complexity` to `complex` (safe over-provision) and records the normalization (original value + task identity) in a form the caller can read (e.g. a field on the returned plan/task, or a returned list) — the parser itself does no logging or event emission (it has no session-events-log handle). Acceptance: a unit test in `cortex_command/pipeline/tests/test_parser.py` feeds a `plan.md` task with `**Complexity**: medium` and asserts (a) the constructed `FeatureTask.complexity == "complex"` and (b) the returned plan records the original `"medium"` against that task. **Phase**: OOV complexity fail-soft

2. **Surface every OOV normalization as a structured, report-visible event.** The dispatch caller that holds the session events-log path (`feature_executor.py` around the `parse_feature_plan` call, and the pre-flight `batch_plan.py` path) emits a structured event — registered in `bin/.events-registry.md` — for each normalized task, carrying the feature, task, and original OOV value; the morning-report scanner surfaces it. This is the loud signal that satisfies the EDGE/Non-Requirement "do not mask the authoring bug" — a bare log line would be invisible to the report. Acceptance: (a) `grep -c "<new-event-name>" bin/.events-registry.md` ≥ 1 (pass if ≥ 1); (b) a `cortex_command/overnight/tests/test_report.py` test asserts the morning report names the OOV feature and its original value. **Phase**: OOV complexity fail-soft

3. **Keep the dispatch enum guard as a backstop, not the primary surface.** The `resolve_model` raise (`dispatch.py:231-235`) and `dispatch_task` `tier is None` raise (`dispatch.py:519-524`) remain unchanged as invariant assertions; after R1 no OOV value reaches them on the normal path. Acceptance: a test asserts (a) the dispatch raises still fire for a directly-passed unknown tier, and (b) an end-to-end normalized plan never triggers either raise. Grounded in `cortex_command/pipeline/dispatch.py:231-235,519-524`. **Phase**: OOV complexity fail-soft

4. **Repair every parser test that asserts an OOV complexity survives.** In `cortex_command/pipeline/tests/test_parser.py`, the tests asserting `complexity == "moderate"` (`test_mixed_separators_all_parsed` ~line 146, and the assertions at lines 426, 472, 526) must be updated: those expecting the value to pass through now expect `"complex"`. The truncation discriminator at line 526 (which used `"moderate"` as a sentinel to prove an H2 did not leak a later `critical` value into the task) must switch to an **in-vocabulary** sentinel pair (e.g. `trivial` vs `complex`) so normalization does not erase its diagnostic power. Acceptance: `python -m pytest cortex_command/pipeline/tests/test_parser.py` exits 0 (pass if exit code = 0), and no remaining assertion in that file expects a post-parse complexity outside `{trivial, simple, complex}`. **Phase**: OOV complexity fail-soft

5. **Deterministically fail in-session dependents of a failed blocker via an end-of-round sweep.** A sweep over the canonical `OvernightState` — running in the runner round loop after batch results are mapped into state (`map_results._map_results_to_state`) and before the round's pending/merged checks — transitions every not-yet-terminal feature whose `intra_session_blocked_by` contains a now-`failed` feature to `failed` with reason `blocker_failed`. The sweep re-applies to a fixpoint each round (so transitive `A→B→C` chains resolve) and fires only on terminal `failed`, never on `paused` (recoverable; auto-retries) or `deferred`. It is extracted as a pure, unit-testable helper over a state object. Acceptance: a unit test of the helper asserts that given `A=failed` and `B` (`intra_session_blocked_by=["A"]`), `B` becomes `failed` with reason `blocker_failed`; and a transitive test (`A→B→C`) asserts `C` also becomes `failed`. Grounded in `cortex_command/overnight/state.py:554-608` (`update_feature_status`), `cortex_command/overnight/map_results.py:75-128`, and the round loop in `cortex_command/overnight/runner.py`. **Phase**: Cascade containment

6. **The sweep lets the loop terminate cleanly instead of churning to a circuit breaker.** Because the sweep drives `intra_session_blocked_by` dependents from `pending` to terminal `failed`, `_count_pending` reaches 0 and the round loop takes the clean `pending == 0` exit (`runner.py:2107-2113`) instead of churning wasted rounds until the `merged_delta`-based stall breaker (`runner.py:2398-2400,2474-2487`). Note: the cascade fix does **not** increase merges — a genuinely failed blocker still yields zero merges for its subtree; the value is deterministic termination plus an accurate report, not a higher merge count. Acceptance: a unit/integration test asserts that after the sweep, `_count_pending(state) == 0` for a state whose only non-terminal features were dependents of a failed blocker (pass if count = 0). Grounded in `cortex_command/overnight/runner.py:2095-2113`. **Phase**: Cascade containment

7. **Surface cascade casualties distinctly so they do not read as real failures.** Auto-failed dependents appear in the morning report tagged with reason `blocker_failed`, and are distinguishable from features that failed on their own work (so one OOV blocker with N dependents does not read as N+1 independent failures). Acceptance: a `cortex_command/overnight/tests/test_report.py` test asserts the report renders an auto-failed dependent with the `blocker_failed` reason and separates/labels it from primary failures. Grounded in `cortex_command/overnight/report.py` (`render_failed_features`), `cortex/requirements/observability.md`. **Phase**: Cascade containment

8. **Resolve the dispatch path's lifecycle reads/writes against the project root.** The per-feature dispatch path in `feature_executor.py` currently uses ~12 CWD-relative `Path(f"cortex/lifecycle/{feature}/…")` literals (plan.md read, `mark_task_done_in_plan` writes, exit-reports, progress/learnings, orchestrator-note, activity log) plus the CWD-relative default base of `read_criticality`/`read_tier`. The dispatch-critical reads — the feature `plan.md` (must read to dispatch at all) and the criticality/tier reads (drive model selection and review-gating) — must resolve against `_resolve_user_project_root()` so a non-home CWD/worktree does not yield a missing-plan parse error or the `medium` criticality default. The remaining sibling reads/writes in the same function should be resolved consistently or explicitly enumerated as a follow-up; silent partial fixes are not acceptable. Prefer supplying a root-resolved base to `read_criticality`/`read_tier` over changing their global default (which is widely called). Acceptance: a test invoking the plan read and the criticality/tier read from a non-root CWD returns the backlog-declared values (not the relative-path default), pass if the returned criticality == the declared value and the plan parses. Grounded in `cortex_command/common.py:542-565,615-636` and `cortex_command/overnight/feature_executor.py` (the 12 lifecycle-path literals). **Phase**: Root-resolution hardening

9. **Close the runtime backlog-resolver gap for slug ≠ filename-stem.** `_find_backlog_item_path` (`outcome_router.py:351-378`) already resolves the project root (`_resolve_user_project_root()`) and tries an exact stem match plus a substring fallback; the residual gap is a feature whose lifecycle-slug differs from its `NNN-` stem AND is not uniquely recovered by the substring fallback. Route that case through the canonical `backlog/resolve_item.py:resolve` (which matches on `uuid`/`backlog_id`/`lifecycle_slug` frontmatter) so it resolves deterministically. Acceptance: a test with a feature slug ≠ filename stem (e.g. `build-the-grinder-agnostic-knowledge-layer` → `025-grinder-agnostic-knowledge-layer.md`) resolves to the correct file and emits no `backlog_write_failed`. Grounded in `cortex_command/overnight/outcome_router.py:351-378`, `cortex_command/backlog/resolve_item.py:resolve`. **Phase**: Root-resolution hardening

10. **Land the plan-gen commit on the integration branch via a Python helper, not agent prose.** Today the round `plan.md` commit is performed by the spawned orchestrator agent following the prose instruction at `orchestrator-round.md:439`, in the home-repo CWD → it lands on `main`. The durable fix is a Python step that commits the generated plan files inside the integration worktree (mirroring `_commit_followup_in_worktree`'s `cwd=worktree_path` pattern, `runner.py:436-527`) rather than cwd-pinning the entire orchestrator agent (which would relocate all of the agent's lifecycle reads/writes). Acceptance: an integration-level assertion (not a pure unit test — the current commit is agent-driven) confirms the plan commit lands on `overnight/{session_id}` and not `main`; if implemented as a Python helper, a unit test asserts it runs git with `cwd` = the integration worktree. Grounded in `cortex_command/overnight/runner.py:436-527,1036-1055`, `cortex_command/overnight/prompts/orchestrator-round.md:439`. **Phase**: Root-resolution hardening

## Non-Requirements
- Does NOT add `medium` (or any criticality word) to `TIER_CONFIG`; the complexity vocabulary stays `{trivial, simple, complex}` (ticket EDGE).
- Does NOT "fix" a criticality→complexity conflation — no such code path exists (disproven; `feature_executor.py:623-624` sources the two independently).
- Does NOT make the cascade fix increase merges or "save" a session whose blocker genuinely failed — the subtree still yields zero merges. The fix delivers deterministic termination and an accurate report, not a higher merge count.
- Does NOT redesign circuit-breaker thresholds or the round/blocked-by assignment logic beyond the dependent-failure sweep of R5.
- Does NOT inspect or depend on the consuming-repo session artifacts (`overnight-2026-06-01-1518`, its `plan.md`, commit `6b8cb06`) — code-only, black-box symptom.
- Does NOT cwd-pin the entire orchestrator agent (R10 uses a scoped Python commit helper instead).
- Does NOT address the #277 observability overlap (a circuit-broken session showing `phase: executing` with no live runner) — that belongs to #277.
- Does NOT add a normalization layer at plan-gen output or batch_plan; normalization is a single parser chokepoint (`batch_plan.py` inherits clean values because it calls `parse_feature_plan`).

## Edge Cases
- **Transitive dependents**: `A` fails → `B` (blocked-by `A`) fails → `C` (blocked-by `B`) must also fail. The end-of-round sweep re-applies to a fixpoint so transitive chains fully resolve within the round.
- **Multi-blocker**: a feature blocked-by `[A, B]` where `A` failed and `B` merged still fails (`A` will not merge this session). Rule: ANY blocker at terminal `failed` → the dependent fails with `blocker_failed`.
- **Blocker `paused`, not `failed`**: dependents are NOT failed — `paused` is recoverable and auto-retries on resume; dependents remain `pending`.
- **Resume timing**: a blocker that pauses in round 1 then fails in round 3 triggers the sweep in round 3 (the sweep runs every round over the full `OvernightState`), so its dependents fail then — no perpetual-pending regression.
- **Absent vs present complexity**: an absent `**Complexity**` field keeps the existing `or "simple"` default (`parser.py:317`); only a *present* out-of-vocab value normalizes to `complex`.
- **Typo'd-trivial over-provision**: normalizing a mistyped trivial task to `complex` runs it at the Opus/higher-budget tier (`TIER_CONFIG`, `dispatch.py:127-131`). This is the accepted cost of the "safe direction" (over- not under-provision); the structured event (R2) makes the mis-authoring visible so it can be corrected at source rather than silently recurring.
- **Broadly-malformed plan**: an OOV complexity may co-occur with other plan corruption. Normalization does not suppress other parse errors — `parse_feature_plan` still raises on genuinely unparseable bodies (existing `ValueError` → `FeatureResult(parse_error=True)` path), so a truly broken plan still fails its own feature cleanly.
- **Resolver no-match / ambiguity**: the canonical resolver's existing disambiguation applies; on a genuine no-match the existing best-effort swallow remains (`backlog_write_failed` logged, session not aborted — `outcome_router.py:421-428`).
- **Integration worktree absent/torn-down at plan-commit time**: the plan-commit helper must fail safe (no crash; surfaced), not emit the ticket's `followup_commit_failed` (rc=128) against a removed worktree.

## Changes to Existing Behavior
- MODIFIED: `parse_feature_plan` normalizes a present OOV complexity → `complex` and records the original value → (was: passed verbatim → crashed at the dispatch enum lookup).
- ADDED: a structured, report-surfaced event for each OOV normalization (emitted by the dispatch caller).
- MODIFIED: an end-of-round sweep fails the `blocked-by` dependents of a terminally-failed feature (was: dependents left `pending`, churning rounds until a circuit breaker).
- MODIFIED: the dispatch path's plan and criticality/tier reads resolve against the project root (was: CWD-relative → `medium` default and missing-plan reads under a non-home CWD).
- MODIFIED: the runtime backlog resolver falls back to the canonical frontmatter-based resolver for slug ≠ stem (was: exact-stem + substring fallback that could miss).
- MODIFIED: the plan-gen commit targets the integration worktree via a Python helper (was: agent-prose `/commit` on home `main`).

## Technical Constraints
- New events must register in `bin/.events-registry.md` (project Architectural Constraint); the events-registry pre-commit gate applies.
- Editing `cortex_command/common.py` is gated to the lifecycle skill (this work is in a lifecycle, so permitted). R8 should prefer passing a root-resolved base to `read_criticality`/`read_tier` over changing their global default, to avoid regressing the many other callers and their tests.
- There are two state-write paths: the in-process `outcome_router.apply_feature_result` (batch_runner) and the live runner round loop's `map_results._map_results_to_state`. The cascade sweep (R5) must operate on the canonical `OvernightState` in the live round loop — that is the path overnight executes.
- The R5 sweep transitions use the existing atomic state-write path (tempfile + `os.replace()`; pipeline.md NFR "Atomicity").
- The dispatch enum guard and the `{trivial, simple, complex}` vocabulary are unchanged (R3).
- R10 cannot be a pure unit test of the existing agent-driven commit; it requires either a Python commit helper (unit-testable) or an integration assertion that spawns the agent. Prefer the helper (CLAUDE.md: "structural separation over prose-only enforcement").
- `cortex/requirements/glossary.md` is referenced by `project.md` Global Context but is absent on disk — not introduced or relied upon by this feature.
