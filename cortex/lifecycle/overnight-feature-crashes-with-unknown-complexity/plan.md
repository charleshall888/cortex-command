# Plan: overnight-feature-crashes-with-unknown-complexity

## Overview
Harden the overnight pipeline against an out-of-vocabulary (OOV) per-task complexity value so a single malformed `plan.md` degrades gracefully instead of zeroing the session. Three independent sub-themes: (1) normalize an OOV complexity to `complex` at the sole parser chokepoint and emit a loud, report-visible event; (2) add an end-of-round sweep that deterministically fails the in-session dependents of a terminally-failed blocker so the failed subtree drains to terminal and the round loop can reach its clean `pending == 0` exit (with the existing stall breaker as the backstop for independent churn) with an accurate report; (3) resolve the dispatch path's lifecycle reads against the project root, route the runtime backlog resolver through the canonical resolver, and land the plan-gen commit on the integration branch via a Python helper. The dispatch enum guard stays as an invariant backstop; the `{trivial, simple, complex}` vocabulary is unchanged.

## Outline

### Phase 1: OOV complexity fail-soft (tasks: 1, 2, 3)
**Goal**: A present-but-OOV `**Complexity**` value normalizes to `complex` at the parser boundary, the normalization is surfaced as a structured report-visible event, and the dispatch enum guard remains a proven backstop.
**Checkpoint**: `test_parser.py` green with no assertion expecting a post-parse OOV value; an end-to-end OOV plan dispatches without raising; the morning report names the OOV feature and its original value.

### Phase 2: Cascade containment (tasks: 4, 5, 6)
**Goal**: When a blocker reaches terminal `failed`, its non-terminal in-session dependents deterministically transition to `failed` (`blocker_failed`) via an end-of-round sweep that runs every round; the failed subtree thereby drains to terminal so that, when it is the only remaining pending work, the round loop reaches the clean `pending == 0` exit instead of churning to the stall breaker; and the report distinguishes cascade casualties from primary failures.
**Checkpoint**: The pure sweep helper resolves direct and transitive chains to a fixpoint over `pending`/`running`/`paused` dependents; `_count_pending` reaches 0 after the sweep for a dependents-only state; the report renders `blocker_failed` casualties distinctly.

### Phase 3: Root-resolution hardening (tasks: 7, 8, 9)
**Goal**: The dispatch path's plan read and `read_criticality` read (plus the sibling lifecycle-path literals in the dispatch path) resolve against `_resolve_user_project_root()`; the runtime backlog resolver falls back to the canonical frontmatter resolver for slug ≠ filename-stem; the plan-gen commit lands on `overnight/{session_id}` via a Python helper, not agent prose on home `main`.
**Checkpoint**: With `CORTEX_REPO_ROOT` set to a fixture root and CWD elsewhere, reads return the backlog-declared criticality and parse the plan; no CWD-relative `cortex/lifecycle/{feature}/…` literal remains reachable from the dispatch path; a slug ≠ stem feature resolves with no `backlog_write_failed`; the plan commit is asserted on the integration worktree, not `main`.

- **Files**: `cortex_command/overnight/state.py`, `cortex_command/overnight/tests/test_state.py`
- **What**: Add a pure, unit-testable helper over `OvernightState` that transitions every not-yet-terminal feature whose `intra_session_blocked_by` contains a now-`failed` feature to `failed` with reason `blocker_failed`, re-applying to a fixpoint so transitive chains resolve.
- **Depends on**: none
- **Complexity**: complex
- **Context**: Operate on the canonical `OvernightState` (`state.py`), reusing `update_feature_status(state, dep, "failed", error="blocker_failed")` (`state.py:554-613`) — `error="blocker_failed"` is the channel the report (Task 6) reads to distinguish casualties. `intra_session_blocked_by` is a field on `OvernightFeatureStatus` (`state.py:103`). **Two distinct status sets — do not conflate them:** (a) the *blocker* trigger fires **only** when the blocker is at terminal `failed`, never on a `paused` blocker (a paused blocker may still merge, so it must not cascade — Edge: "Blocker paused, not failed") nor `deferred`; (b) the *dependent* eligibility is **every non-terminal dependent** — `pending`, `running`, AND `paused`. The helper takes and returns the state object (pure over its argument); persistence is the caller's responsibility (Task 5). Add unit tests: (i) `A=failed`, `B(intra_session_blocked_by=["A"])` with `B` pending → `B` becomes `failed` with `error == "blocker_failed"`; (ii) transitive `A→B→C` → `C` also becomes `failed`; (iii) a `paused` *blocker* leaves its dependent untouched; (iv) a `paused` *dependent* of a `failed` blocker IS swept to `failed` (the recoverability-vs-termination distinction).

## Risks
- **Task 9 worktree-resolution + file-presence (highest design risk; reshaped by critical review).** A git worktree has its own working directory and `state.integration_worktrees` excludes the home repo (`plan.py:429-431`) — so the helper must resolve the worktree per feature (`state.worktree_path` for home features, `state.integration_worktrees[repo_path]` for cross-repo) or it no-ops on the common home-only session. The orchestrator writes `plan.md` into the home tree, so the helper copies it into the worktree before committing and leaves the home copy in place (Task 7's dispatch read consumes it). Flagged as the place where "commit inside the worktree" meets two non-obvious mechanics (home-key absence + cross-tree file presence).
- **Task 1 over-provision cost (accepted).** Normalizing a mistyped `trivial` task to `complex` runs it at the higher Opus/budget tier (`TIER_CONFIG`, `dispatch.py:127-131`). This is the deliberate "safe direction" (over- not under-provision); the Task 2 event makes the mis-authoring visible so it is corrected at source. Not a bug to fix here.
- **Task 5 sweep placement is load-bearing.** The sweep must run on the canonical `OvernightState` in the *live runner round loop* (the path overnight executes), at indent 12 where both the batch-plan and no-batch-plan branches re-converge, so it fires every round — not nested inside the batch-plan `else`, and not on the in-process `outcome_router.apply_feature_result` path (only the round loop holds the full dependency graph). Technical Constraint, Open Decisions: resolved in favor of the sweep.
- **Task 4 paused-dependent semantics (critical-review correction).** The "paused is recoverable" principle is *blocker-side*: a paused **dependent** of a `failed` blocker is not recoverable (its blocker will never merge, and `_count_pending` counts `paused`), so the sweep correctly fails it.
