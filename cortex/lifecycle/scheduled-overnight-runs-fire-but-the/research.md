# Research: Backlog #277 — scheduled overnight runs fire but the runner is silently killed (no setsid on macOS) + launch-flow observability gaps

> Scope (per Clarify): full bundle in one feature — P0 detach fix, P0b fire-time runner-liveness + loud diagnostics, P1 schedule-verifier false-failure + bookkeeping skip, P1b `session_start` timing, P2 scheduled-dormant status state, P2-doc (stale `$CORTEX_COMMAND_ROOT`, planning-helper CLI surface, missing `overnight-state.json` symlink). Complexity: complex. Criticality: high.

> **Root-cause reframing (load-bearing).** Adversarial research verified that the scheduled-fire path fails at the **CLI-contract level before the setsid detach issue can bite**: the launcher invokes `cortex overnight start --launchd --session-id <id>`, but `start` has no `--session-id` flag → argparse exit 2. Because the launcher backgrounds the call (`nohup … &`), exit 2 does not propagate; the launcher exits 0 — which exactly reconciles the ticket's observed `last exit=0` + empty runner logs + no live runner. The setsid/detach defect is **real but latent** (it bites only once the flag contract is fixed). The spec must treat this as a **chain of defects in series**, not a single setsid bug.

## Web Research

Authoritative sources (all primary / Apple):

1. **launchd reaping (root cause confirmed) — Apple TN2083 (Daemons and Agents).** When a launchd job process exits, launchd sends `SIGTERM` to the **whole job process group**, then SIGKILL after `ExitTimeOut`. A launcher that backgrounds a child and exits 0 leaves the child in that group → reaped. TN2083 gives three sanctioned fixes, in priority order: (1) **run the child as its own launchd job** (most robust); (2) **`setsid()` the child** (fork → parent `_exit()` → child `setsid()`, so the child is not a group leader and the call succeeds); (3) **`AbandonProcessGroup = true`** in the parent plist (disables launchd's group GC).
2. **`setsid` on macOS**: the `setsid(1)` **binary** is util-linux and is **absent on stock macOS**, but the `setsid(2)` **syscall and `os.setsid()` are present**. So the fix must call the syscall (Python), not shell out. Canonical thread-safe idiom: **`subprocess.Popen(..., start_new_session=True)`** (the Python docs say this "should take the place of … `preexec_fn` to call `os.setsid()`").
3. **`nohup … & disown` is insufficient** — `nohup` ignores SIGHUP, `disown` removes the job from the shell's table; neither creates a new session, so the child stays in the original group and is still reaped. Confirmed anti-pattern.
4. **`launchctl print` is explicitly "NOT API"** (Apple's own warning, quoted: "Do NOT rely on the structure or information … It may change from release to release without warning"). `state = not running` / `state = waiting` is the **normal** reading for an **armed, dormant** `StartCalendarInterval` agent that hasn't fired. Polling the `state` string is unreliable across Darwin versions. Durable armed-check = domain-presence (`launchctl print` exit 0 / appears in `launchctl list`) + enabled (`launchctl print-disabled`) + the registered calendar block.
5. **`StartCalendarInterval` has no `Year` key** (launchd.plist(5): keys are Minute/Hour/Day/Weekday/Month, "missing arguments … wildcard … similar to crontab"). A "full date" spec therefore **recurs annually**; launchd has **no self-expiry**. To make a fire effectively one-shot you must tear the job down after it runs (`launchctl bootout gui/<uid>/<label>`).
6. **Sleep / wake**: asleep at fire → job runs **on next wake** (missed intervals coalesced into one run); machine off → **skipped** until next scheduled time. **launchd does not wake the Mac.** `caffeinate -i` only prevents **idle** sleep *during* a run (holds the assertion only while its child runs) and **never wakes** a sleeping machine — orthogonal to firing.
7. **Liveness pattern**: because the launcher's exit code is meaningless once it backgrounds, verification must be an out-of-band **pidfile-appearance poll with timeout**, and the pidfile must be written by the **post-detach survivor** so the signal reflects the real runner.

Sources: Apple TN2083; Apple "Scheduling Timed Jobs"; launchd.plist(5) (keith.github.io xcode-man-pages); launchd-dev setsid/setpgrp thread; ersatz-setsid (github jerrykuch); Python subprocess docs; bugs.python.org/issue37586; gist on "launchctl print … NOT API"; Apple Developer Forums thread 4921; caffeinate(8) man page; launchd.info.

## Requirements & Constraints

- **`runner.pid` IPC contract** (pipeline.md:151, verbatim): JSON `{schema_version, magic, pid, pgid, start_time, session_id, session_dir, repo_path}`, mode 0o600, atomic write; cancel verifies `magic + start_time (±2s via psutil)` before signalling; `1 ≤ schema_version ≤ MAX_KNOWN_RUNNER_PID_SCHEMA_VERSION` (pipeline.md:28). The schema **already carries `pgid`** — designed for the session/group fix. P0b's liveness signal must produce a `runner.pid` conforming to this exact schema + verification rule. **Stable contract for the MCP control plane** — do not break it.
- **Persisted phase enum is CLOSED**: `phase: "planning|executing|paused|complete"` (pipeline.md:152, shared by `active-session.json` and the state file). `phase: "starting"` is documented as observable-via-status-but-**never persisted** (pipeline.md:152; observability.md:68 binds it to the spawn-pending handshake window). This is the precedent: a scheduled-dormant display state must be **inferred at read time, not added to the persisted enum**.
- **Status path is read-only** (observability.md:9,93,99). The dormant render and any P1b elapsed/watchdog re-anchoring must be inferred at read time — the status path must not write session state. Existing precedent: the "Stale PID → 'dead (stale PID)'" edge (observability.md:117) already derives a display state from `runner.pid` liveness.
- **`scheduled-launches.json` sidecar + `.lock`** (pipeline.md:156-157): the documented home for schedule bookkeeping. Fields are `label, session_id, plist_path, launcher_path, scheduled_for_iso, created_at_iso`. The `.lock` (`fcntl.LOCK_EX`) is held across GC + plist install + bootstrap + verify + sidecar-write. P1's fix must complete the sidecar write inside this locked critical section regardless of the liveness probe. **Naming note**: the ticket's `scheduled_start` is the *state-file* field (state.py:198); the sidecar's field is `scheduled_for_iso` — reconcile naming in the spec.
- **`runner-bootstrap.log`** (pipeline.md:155): existing diagnostic channel for pre-`events.log`-init failures; the std-stream-redirect Edges item aligns with extending this.
- **CLI surface + macOS-only `schedule`** (pipeline.md:28): documented enum `cortex overnight {start|status|cancel|logs|schedule|list-sessions}`. A new `prepare`/`launch` verb extends this enum — the requirement listing it must be amended. `schedule` is macOS-only.
- **Skill-helper-module pattern** (project.md:35) + **bare-Python prohibition L201** (project.md:45): skill files must not contain bare-Python `cortex_command` imports (enforced by `cortex-check-bare-python-import`). This is the canonical justification for promoting the planning helpers to a `cortex overnight launch`/`prepare` CLI surface (P2-b).
- **ADRs** (all accepted → MUST-automatic compliance): ADR-0001 (file-based state — new liveness/dormant signals stay plain files), ADR-0002 (CLI wheel + plugin distribution — new verbs ship through the two-channel contract; no symlink-deploy), ADR-0003 (per-repo sandbox registration — relevant to the `$CORTEX_COMMAND_ROOT`/`_resolve_user_project_root` footgun), ADR-0006 (peripheral). **Determination — scheduled-dormant status is DISPLAY-ONLY; do NOT amend the persisted enum.** Reason: the persisted enum is a versioned IPC contract (MCP/dashboard/statusline); the established precedent (`phase:starting`, and the stale-PID edge) is to infer transient pre-run states at status read time. The requirements **docs** should be amended to *document* the new observable-but-unpersisted display state (mirroring how `starting` is documented) — a doc amendment owned by the lifecycle review phase / `requirements-write`, not a change to the persisted phase field.

## Tradeoffs & Alternatives

**Decision 1 — detach mechanism.**
- (a) Launcher invokes `cortex overnight start` **without** `--launchd` (so it hits `_spawn_runner_async`'s `Popen(start_new_session=True)`). Reuses the one proven detach path (single source of truth); the runner becomes a real session leader (`os.getsid(pid)==pid`) that survives launchd's group SIGTERM. No `setsid`-binary dependency.
- (b) `python -c 'import os; os.setsid(); os.execvp(...)'` shim in the launcher. Portable, but introduces a *third* daemonization codepath that must re-implement std-stream redirect, caffeinate wrap, pid-wait, and fail-marker — does not reuse the liveness handshake.
- **Recommended: (a)**, with the caveats below (caffeinate placement, `--state`, cross-cancel guard, fail-marker) made explicit — because the adversarial review shows (a) is **not** a literal drop-in.

**Decision 2 — schedule verifier.**
- (a) Accept `not running` AND `waiting` as armed — one-line, immediate unblock, but keeps coupling to the volatile (non-API) `state` string.
- (b) Verify the durable structural fact (registered calendar block + enabled state), not the `state` word — robust across Darwin versions, more parsing.
- **Recommended: (b)** as the durable fix (Solution-horizon: we can already name why (a) will need redoing — Apple has changed the string before and says it's not API), with (a) acceptable as an interim hotfix. **Higher-value, orthogonal half of P1: decouple bookkeeping** — make the liveness probe **advisory (warn, never abort)** so the sidecar + `scheduled_start` writes always complete.

**Decision 3 — session_start timing.**
- (a) Don't pre-log `session_start` on the schedule path; let the runner log it at fire — deletes the defect class (no double `session_start`, no `session_id:"manual"` orphan).
- (b) Keep pre-logging but anchor elapsed/watchdog to `scheduled_start` — patches the display but leaves two malformed events for every downstream consumer to dedupe.
- **Recommended: (a)** (note: adversarial review rates P1b as cosmetic, not load-bearing — a missing `session_start` only degrades `time_limit/max_rounds` display, does not crash).

**Decision 4 — scheduled-dormant status.**
- (a) Inferred display-only state (no persisted phase). Mirrors `starting`; keys off `scheduled_start` (already persisted + read) + `runner.pid` liveness; zero schema/transition/migration change.
- (b) Amend the closed phase enum — contradicts the documented "never persisted" invariant; requires new `_FORWARD_TRANSITIONS` edges, state-file migration, and updates to every phase-validating callsite.
- **Recommended: (a).** Predicate must be **conjunctive** (see Adversarial F5).

**Decision 5 — P2-b CLI surface.**
- (a) Two verbs `prepare` + `bootstrap` (two mutating verbs; more drift surface).
- (b) One umbrella mutating `cortex overnight launch` (fuses select→plan→bootstrap→telemetry into one atomic verb, matching the existing convention), optionally paired with a **read-only** `cortex overnight prepare` that renders the plan JSON for the human Approve gate before mutating.
- **Recommended: (b)** — one mutating `launch` + optional read-only `prepare`. Avoid two mutating verbs.

**Coupling P2-b ↔ P0/P0b: INDEPENDENT.** P0b is entirely server-side (`scheduler/launcher.sh` + `cli_handler`/`macos.py`); the skill's launch/schedule step already calls the CLI. P2-b is a DX cleanup of the *prep* path (steps 7.1–7.5). They can land in any order. (User chose everything-in-one-feature, so both ship together — this only informs **phase ordering within the spec**: reliability core first, P2-b as a later phase.)

**Verification/testability — recommended.** A platform-agnostic detach unit test asserting the runner is a **session leader** (`os.getsid(pid) == pid`, distinct from the parent's session) exercises the *real* production path (`Popen(start_new_session=True)`) and **runs on Linux CI** — "process is backgrounded" is insufficient because `nohup … & disown` backgrounds without a new session. Keep the darwin-gated integration test for launchd-specific seams (caffeinate, plist bootstrap, fail-marker). **Add a test that renders the launcher template and asserts the emitted cortex argv parses cleanly** — this is the test that would have caught F1.

## Adversarial Review

Findings verified against code (file:line):

- **F1 (VERIFIED, new, P0): the launcher passes `--session-id`, which `start` rejects → argparse exit 2.** `launcher.sh:146,153` invoke `cortex overnight start --launchd --session-id "$SESSION_ID"`; `start` defines no such flag (cli.py start subparser: `--state/--time-limit/--max-rounds/--tier/--dry-run/--format/--force/--launchd`). Confirmed by execution: `unrecognized arguments: --session-id foo`. Because the call is backgrounded, exit 2 doesn't propagate → launcher exits 0 (reconciles `last exit=0`). **This fires before the setsid bug.** The spec must fix the launcher↔CLI flag contract first.
- **F2 (VERIFIED, P0-adjacent): under launchd, cwd is `/`; the runner can't find its state.** The launcher passes neither `--state` nor a working directory; `_resolve_repo_path` falls back to `Path.cwd()` (`cli_handler.py:124-139`) = `/` under launchd → auto-discovery scans `/cortex/lifecycle/sessions/` → exit 1. The plist passes `repo_root` as `ProgramArguments[1]` but the launcher never consumes `$1`/`$2`. **Fix (1a) must pass an absolute `--state <path>`; auto-discovery cannot be relied on.**
- **F3 (VERIFIED, contradicts the "free lunch" framing): under (1a), `caffeinate -i cortex overnight start` wraps the wrong process.** Default `start` async-spawns the runner (`start_new_session=True`) and **returns within ~5s**; `caffeinate` (its parent) then exits, dropping the idle-sleep assertion while the multi-hour runner runs **unprotected**. Fix: wrap `caffeinate -i` around the **long-lived runner inside `_spawn_runner_async`** (server-side, benefits run-now too), not around the 5s shim in the launcher.
- **F4 (VERIFIED, nuance): the session-leader survival works; the 5s budget is the risk.** The runner does become a session leader and survives launchd's group SIGTERM (so the `os.getsid` test is valid). `started:true` is **not** premature (envelope returns only after `wait_for_pid_file` confirms `os.kill(pid,0)`). BUT if the runner takes >5s to write `runner.pid`, `start` returns `spawn_timeout` and **kills the orphan group** (`_terminate_orphan_child` `killpg`), killing a slow-but-healthy runner. Re-examine `_SPAWN_HANDSHAKE_TIMEOUT_SECONDS` for the constrained fire context.
- **F5 (VERIFIED, P0b is not free): the fail-marker is only written by the launcher's shell trap.** A `started:false` envelope returns exit 1, which the launcher maps to `error_class="EPERM"` (launcher.sh:100), **discarding the real `error_class`** (`spawn_timeout`/`spawn_died`) and never reaching the morning scanner with accurate data. The launcher must **parse stdout JSON** and write `scheduled-fire-failed.json` with the envelope's real `error_class`.
- **F6 (VERIFIED, correctness/safety): annual refire — nothing boots out a spent job.** `StartCalendarInterval` `Year` key is inert (Web Research); the only `bootout` is user-initiated `cancel()` (macos.py:355-365). The runner never boots out its own job; GC keeps the plist while the job is still registered. `cleanup_self` (launcher.sh:88-93) removes the plist *file* but a registered launchd job refires without it. **A spent scheduled run silently refires ~1 year later, possibly against stale state / wrong branch.** Mitigation: launcher (or runner-on-fire) `launchctl bootout`s its own label after fire, OR `AbandonProcessGroup`, OR GC treats past-`scheduled_for_iso` sidecar entries as spent regardless of registration state.
- **F5/P2 dormant predicate (VERIFIED): `scheduled_start` is cleared only by `cancel`.** Set at cli_handler.py:1628; never cleared on fire. So after a fire it remains set for live/completed/crashed sessions → a naive "scheduled_start set ⇒ dormant" rule misclassifies all three. The dormant predicate must be **conjunctive**: `scheduled_start` in the **future** AND no live `runner.pid` (`verify_runner_pid`) AND phase ∈ {planning / not-started}. Watch clock skew near fire time. The runner must **clear `scheduled_start` on fire**.
- **Unfounded worries to DROP from the spec** (verified): (i) **No `schema_version` bump needed** — `verify_runner_pid` (ipc.py:392-437) checks only magic, schema_version∈[1,1], pid type, create_time ±2s; it does **not** check `pgid`, so `start_new_session` making `pid==pgid==sid` changes nothing. (ii) **The in-flight install guard does not read `session_start`** — it reads `active-session.json` phase + `runner.pid` liveness (`install_guard.py:192-234`); P1b is safe. (iii) **`--force` does not defeat live-runner collision** — the JSON `concurrent_runner` refusal (cli_handler.py:578-597) is **`--format json`-gated, not `--force`-gated**; only the cross-cancel *pending-schedule* guard is `--force`-gated (527-531). So passing `--force --format json` from the launcher bypasses only the launcher's own pending-sidecar guard, **not** live-runner protection. **Caveat: the launcher MUST use `--format json`** or the run-now collision check at 578 is skipped.
