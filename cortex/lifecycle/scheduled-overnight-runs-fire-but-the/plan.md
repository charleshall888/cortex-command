# Plan: scheduled-overnight-runs-fire-but-the

## Overview
Fix the chain of defects that makes scheduled overnight runs fire-but-do-nothing on stock macOS, and make any genuine fire failure loud. The load-bearing move (ADR-0007): the launchd launcher stops re-implementing detach in bash and instead invokes `cortex overnight start` **without** `--launchd`, routing through the proven run-now `subprocess.Popen(start_new_session=True)` path so the **runner** is the surviving session leader; `caffeinate` becomes a runner-lifetime-bound child. The remaining phases add a live-vs-dead spawn discriminator with loud diagnostics, fix the Darwin-25 verifier false-failure and its bookkeeping skip, correct `session_start` timing and add a display-only scheduled-dormant status, tear down spent one-shot jobs, and clean up the launch-flow DX footguns.

**Commit isolation under per-task commits.** The lifecycle implement phase commits **per task** (one sub-agent commit per task; the checkpoint verifies each task produced a commit), not per-phase. The cut-line is therefore enforced by *change-surface ordering*, not a per-phase squash: every reliability-core task (Phases 1–3) modifies **only `cortex_command/`**, so its commit never touches the SKILL.md↔CLI-contract, mirror-drift, or parity gates — those gates fire only on the skill-editing tasks in Phases 4 and 6. A later skill-phase gate failure strands only its own task's commit; the already-landed reliability-core commits are untouched. Tasks are ordered (via `Depends on`) so the core lands first.

## Outline

### Phase 1: Detach & launcher↔CLI contract (tasks: 1, 2, 3)
**Goal**: A scheduled fire launches a surviving session-leader runner via the real launcher→`start` linkage; false detach claims corrected.
**Checkpoint**: Launcher template renders a valid `start` argv (no `--launchd`/`--session-id`); a routing test proves the no-`--launchd` argv dispatches to the async-spawn (session-leader) path; the session-leader guard is green under `just test`; commits touch only `cortex_command/`.

### Phase 2: Fire-time liveness & loud diagnostics (tasks: 4, 5, 6)
**Goal**: A genuinely-dead fire is surfaced loudly with its real failure class; a slow-but-live fire is left running and surfaced as a non-failure advisory; a stale unconfirmed fire escalates to a failure rather than staying advisory forever.
**Checkpoint**: Spawn envelope carries a live-vs-dead discriminator emitted in a launcher-robust form; the timeout branch never kills a live child; the marker decision is driven by the discriminator (not the process exit code); status + morning report render failure-vs-advisory distinctly and escalate stale advisories; tests green.

### Phase 3: Schedule verifier & bookkeeping (tasks: 7, 8)
**Goal**: `cortex overnight schedule` stops false-failing on Darwin 25+ and always records `scheduled_start` + the sidecar entry even when the liveness probe is inconclusive.
**Checkpoint**: e2e variant with `state = not running` verifies (exit 0); a probe-failure test shows `scheduled_start` non-null and a sidecar entry present. End of the must-have reliability core.

### Phase 4: session_start timing & scheduled-dormant status (tasks: 9, 10, 11)
**Goal**: Exactly one fire-time `session_start` (authored by the runner); a display-only scheduled-dormant status state inferred at read time.
**Checkpoint**: Single-`session_start` runner test passes; status renders "Scheduled (dormant)" and suppresses the watchdog block under the conjunctive predicate; `PHASES` tuple unchanged; mirror staged for the skill-flow edit.

### Phase 5: Spent-job teardown & GC (tasks: 12, 13, 14)
**Goal**: A spent one-shot fire cannot refire ~a year later; stale schedules are reaped by timestamp; sleep/wake reality is documented.
**Checkpoint**: Launcher boots out its own label post-fire (darwin e2e); GC reaps past-`scheduled_for_iso` entries + their plist/launcher; docs note the inert `Year` key and caffeinate-no-wake.

### Phase 6: DX footguns & CLI surface (tasks: 15, 16, 17, 18)
**Goal**: Remove the stale `$CORTEX_COMMAND_ROOT`, promote planning helpers to a CLI surface, create the top-level state symlink, and keep the dual-source mirror + contract/parity gates green.
**Checkpoint**: `cortex overnight launch/prepare --help` exit 0; bare-Python-import + skill-prose↔CLI contract (`cortex-check-contract`) + parity + events-registry + mirror-drift gates all pass.

## Risks
- **Fire-path budget + discriminator ownership (Task 4).** Task 4 owns the `--scheduled` flag, the `_FIRE_HANDSHAKE_TIMEOUT_SECONDS` constant, AND the launcher-robust discriminator emission (the `spawn-outcome` token file); Task 5 only reads them. This keeps the selector from dangling across tasks. If the reviewer prefers no new flag, a plist `EnvironmentVariables` signal read by `_spawn_runner_async` is the fallback — but it is less testable.
- **Marker write stays launcher-side, but the discriminator is a token file, not a bash JSON parse (Tasks 4/5).** The spec's R6 assigns marker-writing to the launcher (which natively holds `@@LABEL@@`). We keep that, but resolve the "launcher parses JSON under launchd's degraded env" hazard by having `start --scheduled` emit a single-token outcome file the launcher reads via shell builtins — no interpreter re-invocation, no PATH dependency. This is a method choice within the spec's What; flagged so it can be vetoed at approval.
- **caffeinate placement (Task 2) changes run-now behavior and has a residual sub-second gap.** Binding caffeinate to the runner's lifetime newly adds idle-sleep prevention to the interactive run-now path (the spec accepts this as intentional, resolving research OQ2). Spawning it as the runner's first action shrinks the unasserted window to the `Popen`→entry gap (sub-second, far below idle-sleep timers); the residual gap is accepted, not engineered away.
- **Non-killing timeout + wedged-runner reaping (Tasks 4/6).** Not killing a live-but-slow child is deliberate (killing it was the original bug). The new risk — a runner wedged *before* its round loop (no internal watchdog yet) left alive — is mitigated at the observability layer by Task 6's stale-advisory→failure escalation (it is surfaced loudly, not silently advisory forever). Reaping the wedged process itself relies on the existing lock-takeover / `concurrent_runner` machinery on the next `start`; a dedicated pre-loop runner watchdog is out of scope (flagged, not built).
- **Verifier durable-parse vs allowlist-widen (Task 7).** Parsing the calendar block + enabled state is the solution-horizon-correct fix (Apple says `state` is not API and has changed it before); widening the allowlist to accept `not running` is a smaller interim. The plan recommends the durable parse but notes the interim is acceptable — a scope call the reviewer may want to make.
- **Test reachability — load-bearing verifications are darwin-gated and unenforced in CI.** The headline defect is macOS-specific. The platform-agnostic guards that run in `just test` anywhere are the argv-render + routing + session-leader tests (Task 3), the spawn-discriminator/non-kill test (Task 4), the launcher token-read test (Task 5), the marker-render/escalation tests (Task 6), the bookkeeping-completes test (Task 8, via probe mock), the GC test (Task 13), and the runner/state tests (Tasks 2, 9, 10, 11, 17). But the true end-to-end fire behaviors — the verifier e2e (Task 7), live `cortex overnight schedule` exit-0 (Task 8), bootout-on-fire (Task 12) — are **darwin-gated** (`test_scheduler_e2e.py:46` module-level `skipif`), and `validate.yml` runs **no** Python tests at all (only skill + callgraph validators). So even the platform-agnostic guards run only when a developer runs `just test` locally; the darwin-gated ones run only on a Mac. Wiring the Python suite into CI is an explicit out-of-scope follow-up. Net: the regression guards exist but enforcement is a local/manual gate, not cloud CI.
- **Foreground-wait detach is not a reap race.** The launcher now blocks on `start` for up to the fire budget while still a launchd-job member, but `Popen(start_new_session=True)` performs `setsid` synchronously at spawn (posix_spawn), so the runner is already in its own session before the wait begins and is immune to any SIGTERM to the launcher's process group. (This is the dismissed framing concern; recorded here for the implementer.)
