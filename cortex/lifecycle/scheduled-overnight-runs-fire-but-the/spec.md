# Specification: scheduled-overnight-runs-fire-but-the

## Problem Statement

Scheduled overnight runs fire but do no work and leave no error: on stock macOS the LaunchAgent launcher fails to launch a surviving runner, and a chain of launch-flow defects hides the failure. This makes the core "schedule an overnight run" capability silently unreliable. Verified root cause is a **chain**: (1) the launcher invokes `cortex overnight start --launchd --session-id <id>`, but `start` has no `--session-id` flag → argparse exit 2 — and because the call is backgrounded (`nohup … &`), the launcher exits 0, so launchd records `runs=1, last exit=0` with empty runner logs; (2) even with the flag fixed, the launcher's bash `setsid`/`nohup` detach does not create a new session on a Mac without the `setsid` binary, so launchd reaps the runner's process group when the launcher exits; (3) under launchd the working directory is `/`, so state auto-discovery fails. Adjacent defects (schedule-verifier false-failure that also skips bookkeeping, premature `session_start` logging, a status renderer that never shows a dormant state, and several DX footguns) hid and compounded the failure. This feature makes scheduled runs actually run, fail **loudly** when they can't, and removes the launch-flow rough edges. Beneficiary: anyone scheduling an overnight run on macOS (the primary supported platform).

## Phases
- **Phase 1: Detach & launcher↔CLI contract** — make the scheduled fire launch a surviving session-leader runner using the proven run-now detach path.
- **Phase 2: Fire-time liveness & loud diagnostics** — detect and surface a genuinely-dead fire, while NOT mislabeling a slow-but-healthy runner as failed.
- **Phase 3: Schedule verifier & bookkeeping** — stop false-failing on Darwin 25+, and never skip schedule bookkeeping because of the liveness probe.
- **Phase 4: session_start timing & scheduled-dormant status** — one fire-time `session_start`; an inferred display-only dormant state.
- **Phase 5: Spent-job teardown & GC** — prevent annual refire; reap spent schedules; document sleep/wake reality.
- **Phase 6: DX footguns & CLI surface** — fix the stale root, promote planning helpers to a CLI, create the top-level state symlink.

> **Commit isolation (cross-phase coupling guard).** Phases 1–3 (the must-have reliability core) MUST be landable independently of Phase 6. Phase 6 edits canonical skill files (`skills/overnight/`) and triggers the SKILL.md↔CLI parity gate and the dual-source mirror `git diff --quiet` gate; a Phase-6 gate failure must not block the commit carrying Phases 1–3. The plan phase commits per-phase (each phase is its own checkpoint/commit) so a later-phase failure strands only itself. This replaces any notion of a human "deciding to split" — there is no human in the overnight loop; the cut-line is enforced by per-phase commits, not by a runtime decision.

## Requirements

### Phase 1 — Detach & launcher↔CLI contract

1. **Launcher emits a valid, complete `start` argv.** The rendered launcher template invokes `cortex overnight start` with only flags `start` accepts, and with the full flag set required for correct behavior. Acceptance (all binary-checkable against the rendered template, not raw comments): a test renders the template via `_install_launcher_script` and asserts the emitted cortex argv (a) parses under the `start` subparser with no `SystemExit`, (b) contains `--state <abs>`, `--format json`, and `--force`, and (c) does NOT contain `--session-id` or `--launchd`. The argv assertions are made against the parsed/tokenized invocation line, not a `grep` that a comment could satisfy. **Phase**: Detach & launcher↔CLI contract.
2. **Scheduled runner detaches as a session leader, via the real launcher→start linkage.** The launcher invokes `cortex overnight start` WITHOUT `--launchd`, so the runner is spawned through `_spawn_runner_async`'s `subprocess.Popen(..., start_new_session=True)` (cli_handler.py:382) and becomes a session leader that survives launchd's process-group SIGTERM. Acceptance (two complementary checks, because `os.getsid` alone is a POSIX-invariant proxy that would pass even on the buggy `--launchd` path): (a) a platform-agnostic test spawns the runner through the production detach path and asserts `os.getsid(pid) == pid` AND `os.getsid(pid) != os.getsid(0)`; (b) the R1 launcher-argv test asserts the launcher routes through the no-`--launchd` path — these two together tie the session-leader property to the actual launcher invocation. **Note on where this runs**: this test lives in the standard Python suite (`just test`) and is NOT `skipUnless(darwin)`, so it runs anywhere the suite runs (local, pre-commit if wired, the overnight test-gate). GitHub Actions `validate.yml` does NOT currently invoke pytest/`just test` (it runs only skill + callgraph validators) — wiring the Python suite into GitHub Actions CI is out of scope for this ticket (follow-up). Do not claim this guard runs in GitHub Actions today. **Phase**: Detach & launcher↔CLI contract.
3. **State is resolved explicitly, not via cwd.** The launcher passes an absolute `--state <path>` (cwd is `/` under launchd, so auto-discovery via `_resolve_repo_path` cli_handler.py:124-139 cannot be relied on). Acceptance: the R1 argv test asserts `--state` is present with an absolute-path value; a darwin-gated integration test asserts the runner loads the correct session state when cwd is not the repo. **Phase**: Detach & launcher↔CLI contract.
4. **caffeinate's idle-sleep assertion is bound to the runner's lifetime, with the runner remaining the session leader.** Process-tree contract (resolves the R2↔R4 tension): the **runner** is the `start_new_session=True` session leader (R2); the `caffeinate -i` assertion is held for the runner's full lifetime by making caffeinate a **child of the runner bound to the runner's pid** — e.g. the runner spawns `caffeinate -w <runner_pid>` (or `-i` held by a child it owns) at startup so the assertion drops only when the runner exits. caffeinate is NOT the parent/`Popen` target (that would make caffeinate the session leader and break R2) and is NOT spawned by the ~5s `_spawn_runner_async` shim (that would drop the assertion at handshake return — the F3 bug). This applies to BOTH the scheduled and run-now paths uniformly (the run-now path gains lifetime-bound caffeinate — intentional; see Changes to Existing Behavior). Acceptance (behavioral, not a bare token grep): a test asserts the caffeinate assertion is held by a process bound to the runner pid and outlives the `_SPAWN_HANDSHAKE_TIMEOUT_SECONDS` window (e.g. the caffeinate child is alive after the spawn shim returns and exits when the runner exits). **Phase**: Detach & launcher↔CLI contract.
5. **Correct the false detach claims.** Acceptance: `grep -c "reparents to PID 1" cortex_command/overnight/scheduler/launcher.sh` = 0 and the `docs/overnight-operations.md` line claiming the launcher "detaches … into its own process group" is rewritten to describe the real `start_new_session` detach. **Phase**: Detach & launcher↔CLI contract.

### Phase 2 — Fire-time liveness & loud diagnostics

6. **The launcher parses the JSON envelope and records the failure class, distinguishing a dead fire from a slow-but-live one.** The launcher invokes `start` with `--format json`, reads the envelope, and branches on a **live-child vs dead-child discriminator** (the envelope must carry one — e.g. a `child_alive` boolean or distinct `error_class` values `spawn_died` vs `spawn_unconfirmed`; today `spawn_timeout` is emitted identically for both at cli_handler.py:451 and MUST be split). On a dead child it writes `scheduled-fire-failed.json` with the real failure `error_class`; on a slow-but-live child it writes a **distinct advisory marker** (a severity/kind field marking it "started, not yet confirmed"), NOT a failure marker. Acceptance: a test injects a dead-child envelope and asserts a failure marker with the real `error_class`; a separate test injects a slow-but-live envelope and asserts an advisory (non-failure) marker; the launcher's cortex invocation includes `--format json` (verified via the R1 argv test, not a comment grep). **Phase**: Fire-time liveness & loud diagnostics.
7. **A genuinely-dead fire is surfaced loudly; a slow-but-live fire is surfaced as advisory, not failure.** When the runner child dies before claiming `runner.pid` (`spawn_died`), the failure marker is surfaced by BOTH `cortex overnight status` and the morning report (`report.render_scheduled_fire_failures`, report.py:1614). When the child is alive but slow (advisory marker), status/report render it as a distinct "scheduled fire started — awaiting confirmation" state, NOT as a failed fire. Acceptance: a test producing `spawn_died` asserts the failure is scanned (`fail_markers.scan_session_dirs`) and rendered as a failure in both surfaces; a test producing the slow-but-live advisory asserts it renders as a non-failure advisory in both surfaces. **Phase**: Fire-time liveness & loud diagnostics.
8. **Slow-start neither kills nor mislabels a healthy runner.** On the fire path, a handshake timeout where the child process is still alive does NOT `killpg` the runner (today cli_handler.py:451 unconditionally calls `_terminate_orphan_child` on `spawn_timeout` — this MUST become conditional on the child being dead); the runner is left running and an advisory (per R6) is recorded. The fire-path handshake budget is a named constant set higher than the run-now `_SPAWN_HANDSHAKE_TIMEOUT_SECONDS = 5.0` (the runner does more work at fire: state load, lock acquisition, post-sleep cold caches). Acceptance: a test simulates a live-child timeout and asserts (a) no `killpg`/`SIGKILL` is sent, (b) an advisory (non-failure) marker is written, and (c) the runner process is left alive; the fire-path budget constant exists and is > 5.0. **Phase**: Fire-time liveness & loud diagnostics.

### Phase 3 — Schedule verifier & bookkeeping

9. **Verifier no longer false-fails on Darwin 25+.** `_bootstrap_and_verify` (macos.py:569) verifies the durable fact that the job is armed (registered calendar block + enabled state, and/or accepting `state = not running` and `state = waiting`) instead of requiring the volatile `state = waiting` substring. Acceptance: a `test_scheduler_e2e` variant where `launchctl print` returns `state = not running` for a correctly-armed agent verifies successfully (exit 0). **Phase**: Schedule verifier & bookkeeping.
10. **Bookkeeping completes regardless of the liveness probe.** The verify/liveness probe is advisory; after `cortex overnight schedule`, the `scheduled_start` state-file write (cli_handler.py:1628) AND the `scheduled-launches.json` sidecar entry (`_write_sidecar_entry`, inside the `fcntl.LOCK_EX` critical section) both complete even when the probe fails or times out. Acceptance: a test simulates a probe failure and asserts `scheduled_start` is non-null and a sidecar entry exists; `cortex overnight schedule` exits 0 (or with a non-fatal warning) when the job is armed but the probe is inconclusive. **Phase**: Schedule verifier & bookkeeping.

### Phase 4 — session_start timing & scheduled-dormant status

11. **`session_start` is logged once, at fire, by the runner.** Two checkable halves of the de-dup: (a) the schedule branch of the skill flow does not invoke the prep-time `session_start` log — verified by an assertion that `skills/overnight/references/new-session-flow.md`'s step-5 `session_start` log is gated to the run-now branch only (the schedule branch reaches the launch without it); (b) a runner-level test asserts that after a (simulated) fire the session's `events.log` contains exactly one `session_start` whose `session_id` is the real session id, not `manual`. Acceptance: both (a) the skill-flow gating check and (b) the single-`session_start` runner test pass. **Phase**: session_start timing & scheduled-dormant status.
12. **Status renders a scheduled-dormant state (display-only, inferred at read time).** When `scheduled_start` is in the future AND no live `runner.pid` (`verify_runner_pid`) AND the session is not executing/complete, `cortex overnight status` renders a "Scheduled (dormant) — fires at {scheduled_start}" line and suppresses the executing-watchdog block. The render reads the **explicit per-session state path** (status already resolves the session), NOT the no-arg top-level `load_state()` — so R12 does not depend on R19. No value is added to the persisted `PHASES` tuple (which is already 5-valued: `planning, executing, complete, paused, starting` — `starting` itself is the precedent for an observable-but-unpersisted display state; the on-disk `phase` field value set is the real invariant). Acceptance: a test with a future `scheduled_start` and no live `runner.pid` asserts the dormant render and the absence of the elapsed/watchdog block; a test asserts `PHASES` is unchanged — `len(PHASES) == 5` AND `"scheduled_dormant" not in PHASES` AND `"scheduled" not in PHASES` (NOT a `grep -c "executing"` count, which is invariant to appending and cannot detect the mutation). **Phase**: session_start timing & scheduled-dormant status.
13. **`scheduled_start` is cleared when the runner fires.** The runner clears `scheduled_start` on start (a plain field write via the atomic `save_state` tempfile+`os.replace`, not a phase transition) so the dormant predicate cannot false-positive on a live, completed, or crashed run. The clear reads/writes the explicit per-session state path (not no-arg `load_state`), so R13 does not depend on R19. Acceptance: a test asserts `scheduled_start` is `None` in the session state file after the runner reaches its session-start. **Phase**: session_start timing & scheduled-dormant status.

### Phase 5 — Spent-job teardown & GC

14. **Spent one-shot jobs are torn down.** Because `StartCalendarInterval` has no `Year` key and recurs annually, the runner (or launcher post-fire) issues `launchctl bootout gui/$(id -u)/<label>` for its own label after a successful fire, so the schedule cannot refire ~a year later. Acceptance: a darwin-gated integration test schedules a near-future fire, lets it fire, and asserts the label is no longer registered (`launchctl print` exits non-zero for it). **Phase**: Spent-job teardown & GC.
15. **GC reaps spent schedules by timestamp.** The `schedule()` GC pass reaps `scheduled-launches.json` entries whose `scheduled_for_iso` is in the past — and the associated plist + launcher under `$TMPDIR/cortex-overnight-launch/` — regardless of `launchctl print` registration state (so a missed/failed bootout is still cleaned up). Acceptance: a test seeds a past-`scheduled_for_iso` sidecar entry plus its plist/launcher files and asserts the next `schedule()` removes all three and the sidecar entry. **Phase**: Spent-job teardown & GC.
16. **Document inert `Year` key and caffeinate-no-wake.** `docs/overnight-operations.md` notes that the StartCalendarInterval `Year` key is inert (one-shot semantics are enforced by bootout + GC, not by the date), and that `caffeinate -i` only prevents idle sleep during a run and does NOT wake a sleeping Mac (asleep at fire → runs on next wake; off → skipped). Acceptance: both notes are present in `docs/overnight-operations.md`. **Phase**: Spent-job teardown & GC.

### Phase 6 — DX footguns & CLI surface

17. **Replace the stale `$CORTEX_COMMAND_ROOT`.** `skills/overnight/SKILL.md` and `references/new-session-flow.md` resolve session paths via `cortex --print-root`'s `root` field (or `bootstrap_session`'s returned `state_dir`), not `$CORTEX_COMMAND_ROOT`. Acceptance: `grep -c "CORTEX_COMMAND_ROOT" skills/overnight/SKILL.md skills/overnight/references/new-session-flow.md` = 0 for session-path usage. **Phase**: DX footguns & CLI surface.
18. **Planning helpers exposed via CLI.** A mutating `cortex overnight launch` (select → plan → bootstrap → telemetry, returning a structured envelope) and a read-only `cortex overnight prepare` (renders the selected plan as JSON for the human Approve gate) are wired in cli.py; the skill flow calls these verbs instead of shelling into internal Python APIs. Acceptance: `cortex overnight launch --help` and `cortex overnight prepare --help` exit 0; `cortex-check-bare-python-import` passes for `skills/overnight/` (no bare-Python `cortex_command` imports); the new verbs are reflected in skill prose (SKILL.md↔CLI parity gate passes). **Phase**: DX footguns & CLI surface.
19. **`bootstrap_session` creates the top-level state symlink (for no-arg `load_state` callers).** `bootstrap_session` (plan.py:551) creates the top-level `cortex/lifecycle/overnight-state.json` symlink pointing at the per-session state file, so `load_state()` with no args does not raise `FileNotFoundError` (state.py:331). Contract: the symlink points at the most-recent bootstrapped session and is overwritten on each bootstrap (single-active-session pointer, consistent with the existing host-global active-session model); concurrent multi-session bootstrap is not a supported overnight mode. This requirement serves no-arg `load_state` consumers only — R12/R13 use explicit per-session paths and do NOT depend on it, so deferring R19 cannot break Phase 4. Acceptance: a test asserts the symlink exists after `bootstrap_session` and that `load_state()` (no args) returns the session state. **Phase**: DX footguns & CLI surface.
20. **Registry, parity, and mirror discipline — advisory diagnostic is a marker file, not an event.** The fire-time advisory/failure diagnostics (R6/R8) are **marker files** (`scheduled-fire-failed.json` plus the advisory marker, carrying a severity/kind field), NOT new `events.log` event types — so no `EVENT_TYPES` (events.py:91) addition and no `bin/.events-registry.md` registration are required by this work (see Non-Requirements). The dual-source plugin mirror is rebuilt via `just build-plugin` and staged, and any skill-prose changes pass the parity gate. Acceptance: `just check-events-registry` passes (trivially, since no new event literal is introduced); `git diff --quiet plugins/cortex-overnight/skills/overnight/` after `just build-plugin`; `cortex-check-parity` passes. **Phase**: DX footguns & CLI surface.

## Priority (MoSCoW)

Not everything here is equal weight — the ticket's own P0/P1/P2 labels and the research map onto must/should.

- **Must-have (without these, scheduled runs stay broken or fail silently):** Phase 1 (R1–R5, detach + launcher↔CLI contract — the runner must actually launch and survive), Phase 2 (R6–R8, fire-time liveness + loud diagnostics that distinguish dead from slow-but-live — failures must be visible without false alarms), Phase 3 (R9–R10, verifier false-failure + bookkeeping — a correctly-armed schedule must be recorded and not report failure). These are the P0/P0b/P1 reliability core.
- **Should-have (the capability works without these, but they remove real footguns/safety gaps):** Phase 5 (R14–R16, spent-job teardown + GC — prevents an unintended annual refire; high-priority should because it is a correctness/safety gap, but it does not block the first correct run), Phase 4's R12–R13 (scheduled-dormant status + clear-on-fire — observability accuracy; both read explicit per-session state, independent of R19).
- **Could-have (polish; genuinely low risk if deferred):** Phase 4's R11 (session_start de-duplication — research rated it cosmetic; degrades only `time_limit`/`max_rounds` display), Phase 6 (R17–R20, DX footguns + CLI surface — R19's symlink serves only no-arg `load_state` callers and is decoupled from R12/R13, so deferring it does not break a higher-priority requirement).
- **Won't-do:** see Non-Requirements.

Per the user's explicit "everything in one feature" scope decision, all Must/Should/Could items are in scope for this lifecycle. The classification drives **phase ordering and per-phase commit isolation** (see the Commit isolation note under `## Phases`): Phases 1–3 land independently of Phase 6 so a could-have gate failure cannot strand the must-have core.

## Non-Requirements
- Do NOT add a new value to the persisted `PHASES` tuple or write a new value to the on-disk `phase` field; the scheduled-dormant state is display-only and inferred at status read time (mirrors the `starting` precedent — `starting` is in the `PHASES` tuple for validation/display but is never written to the on-disk `phase` field).
- Do NOT introduce a new `events.log` event type for the fire diagnostics; they are marker files (so no `EVENT_TYPES`/`bin/.events-registry.md` change). If a future implementation chooses an event channel instead, it MUST register it — but this spec commits to the marker-file channel to keep R20's gate unconditional.
- Do NOT bump the `runner.pid` `schema_version`: `verify_runner_pid` (ipc.py:392-437) checks magic + schema_version∈[1,1] + create_time ±2s and does NOT inspect `pgid`, so `start_new_session` making `pid==pgid==sid` requires no IPC change.
- Do NOT add a wake-the-Mac mechanism (`pmset schedule`); the caffeinate-no-wake limitation is documented, not engineered around.
- Do NOT use `AbandonProcessGroup` as the detach mechanism (the chosen `start_new_session` reuse supersedes it).
- Do NOT wire the Python test suite into GitHub Actions CI as part of this ticket (the regression guard runs via `just test`; CI-wiring is a separate follow-up — flagged, not done here).
- Out of scope (consuming-repo / environment, per the ticket): the gaggimate-barista damage-control hook substring false-positives and signed-commit-under-sandbox friction.

## Changes to Existing Behavior
- MODIFIED: the LaunchAgent launcher invokes `cortex overnight start --state <abs> --format json --force` (no `--launchd`, no `--session-id`) and parses the JSON envelope → was `setsid nohup caffeinate -i cortex overnight start --launchd --session-id <id> &`.
- MODIFIED: `caffeinate -i` becomes a runner-lifetime-bound child of the runner (in `_spawn_runner_async`/the runner), so BOTH the scheduled and run-now paths hold the idle-sleep assertion for the runner's lifetime → was launcher-only, dropped at handshake return. **This newly adds idle-sleep prevention to the interactive run-now path** — intentional and desirable (an active overnight run should not be interrupted by idle sleep regardless of how it was launched); this resolves research OQ2 (run-now-acceptability) in favor of the change rather than leaving it open.
- MODIFIED: `_spawn_runner_async`'s timeout path no longer unconditionally `killpg`s on timeout — it kills only a dead child; a live-but-slow child is left running with an advisory (cli_handler.py:451).
- MODIFIED: `cortex overnight schedule` completes `scheduled_start` + sidecar bookkeeping even when the launchd liveness probe is inconclusive → was aborted-on-probe-failure.
- MODIFIED: the schedule path stops pre-logging `session_start`; the runner is the sole author at fire → was double-logged (prep + fire).
- ADDED: `cortex overnight status` scheduled-dormant render and slow-but-live advisory render; `cortex overnight launch` and `cortex overnight prepare` verbs; runner self-`bootout` on fire; GC-by-past-`scheduled_for_iso`; top-level `overnight-state.json` symlink at bootstrap; a live-child-vs-dead-child discriminator in the spawn envelope.
- REMOVED: reliance on the `setsid` binary; reliance on the volatile `state = waiting` substring; reliance on cwd for state discovery.

## Edge Cases
- **Mac asleep at fire**: job runs on next wake (missed intervals coalesced); machine off → skipped until next scheduled time. Documented, not fixed.
- **Run-now holds the lock when a scheduled fire arrives**: the launcher's `--format json` invocation hits the `concurrent_runner` refusal (cli_handler.py:578, json-gated) and does NOT clobber the live runner; `--force` only bypasses the launcher's own pending-schedule guard (cli_handler.py:520-542), not live-runner protection. The launcher MUST pass `--format json` (enforced by R1's argv test).
- **Slow runner startup at fire** (post-sleep wake, cold caches): handled by R6/R8 — a live-but-slow runner is left running and surfaced as a non-failure advisory, never killed and never reported as a failed fire; the fire-path handshake budget is raised above 5s.
- **Clock skew near fire time**: the dormant predicate (`scheduled_start` in the future) may flip around the fire boundary; the conjunctive predicate (also requires no live `runner.pid`) prevents misclassifying a just-started run as dormant; R13 clears `scheduled_start` on fire so a fired run is never shown dormant.
- **Stale `scheduled_start` from a prior spent fire**: cleared on fire (R13) and reaped by GC (R15).
- **Concurrent multi-session bootstrap vs the top-level symlink (R19)**: the symlink is a single-active-session pointer overwritten per bootstrap; concurrent multi-session bootstrap is not a supported overnight mode, and per-session consumers use explicit `--state` paths (R3) that bypass the symlink.
- **`setsid` binary present** (e.g. `brew install util-linux`): the fix no longer branches on it; behavior is identical with or without it.

- Status/morning-report surfaces are read-only (observability.md:93) — the dormant state, the slow-but-live advisory render, and any re-anchoring are inferred at read time; the advisory **marker** and the `scheduled_start` clear are written from the runner/launcher path, never from the status renderer.

The platform-agnostic `os.getsid` session-leader test (R2) is the only detach guard that runs on non-macOS; it runs via `just test` (not GitHub Actions today — CI-wiring is a flagged follow-up, not in scope).
- Dual-source mirror: edit `skills/overnight/` canonical only, rebuild `plugins/cortex-overnight/` via `just build-plugin`; `cortex_command/` is not mirrored. Phases 1–3 (which touch only `cortex_command/`) are commit-isolated from Phase 6 (which touches the mirrored skill files) per the Commit isolation note.
- The fire diagnostics are marker files, not events — no `EVENT_TYPES`/`bin/.events-registry.md` change.
- Authoring: prescribe What/Why not How; soft positive-routing (no new MUST without the evidence-artifact policy).

## Proposed ADR

### Proposed ADR: 0007-scheduled-fire-detach-via-start-new-session
Context: the scheduled LaunchAgent fire path detached the runner with a bash `setsid`/`nohup` fallback that does not create a new session on stock macOS (no `setsid` binary), so launchd reaped the runner's process group. Decision: the launcher reuses the proven run-now detach — `cortex overnight start` (without `--launchd`) spawning the runner via `subprocess.Popen(start_new_session=True)`, with the **runner** as the session leader and `caffeinate -i` held by a runner-lifetime-bound child (not the parent), folding the fire-time liveness handshake into the same path. Trade-off: this couples three follow-on fixes (caffeinate bound to the runner's lifetime as a child; the launcher passes an absolute `--state`; the launcher parses the JSON envelope for diagnostics) and changes the run-now path (it gains lifetime-bound caffeinate — judged desirable), versus a self-contained `python os.setsid()` shim that would add a third daemonization codepath, or `AbandonProcessGroup=true` that patches reaping but leaves the bash detach and observability gaps. Clears the three-criteria gate: hard-to-reverse (it is the spawn contract), surprising-without-context (the prior "reparents to PID 1" comment was wrong), real trade-off (three named alternatives). Industry-validated by Apple TN2083 and the Python `start_new_session` idiom.
