# Specification: swap-implement-phase-preflight-option-2

## Problem Statement

The lifecycle implement-phase preflight menu in `skills/lifecycle/references/implement.md` §1 currently offers `"Implement in autonomous worktree"` as option 2, which dispatches to the daytime SDK pipeline (§1a, ~107 lines). Parent epic #237 is replacing this autonomous-handoff pathway with an interactive worktree flow that runs in the active Claude Code session and ends with a PR. This ticket performs the menu-shape half of that swap: rename option 2 to `"Implement on feature branch with worktree"`, rewrite its description, and rewire its dispatch routing. The body of the new dispatch path is built by sibling tickets #239 (worktree primitive) and #240 (Variant A end-to-end); the daytime path itself is removed by #246. #238 is therefore a coordinated kickoff — it lands together with #239 and #240, before #246's deletion sweep.

The benefit: users see one fewer mode-conflated option ("autonomous worktree" conflated path-isolation with operator-stance) and one more semantically clean one (named output artifact + isolation mechanism, matching the industry convention surfaced in research). The daytime escape hatch out of daytime-as-collaboration goes away in #246; this ticket lays the structural foundation.

## Edge Cases

- **Implementer naively edits the plugin mirror**: the pre-commit hook regenerates the mirror from canonical via `just build-plugin` (Justfile L640–L672), so the manual edit is overwritten. Edit canonical only — verify with `git config core.hooksPath` returning `.githooks` before committing.
- **Implementer commits with `--no-verify`**: bypasses the pre-commit hook entirely; the mirror does not regenerate. Mitigation: implementer should never use `--no-verify` for this ticket. Reviewer must visually confirm mirror parity at PR time.
- **Pre-commit hook not active** (`git config core.hooksPath` is unset or not `.githooks`): implementer runs `just setup-githooks` and re-commits. The hook is opt-in but the project README warns about this.
- **Implementer renames line 56 (§1a body) by accident**: Requirement 6 explicitly disallows this. Verified by Acceptance: post-Phase-1, exactly one `Implement in autonomous worktree` literal must remain in `implement.md` (the line-56 reference).
- **#240 spec-phase reverses to Variant B** (fresh `claude --worktree` session) after #238's description copy is written: Requirement 2 caps the "When to pick" wording to be variant-agnostic ("medium-task features where you want feature-branch isolation and a PR-based handoff, with live steering throughout" — true under both A and B). No rework needed.
- **#239 or #240 ships with a different section name than "§1a slot via renumber-on-deletion"**: the prose anchor in Requirement 5 (`proceed to the worktree-interactive dispatch path (defined by ticket #240)`) is resilient — it points to the flow by name, not by section number. A later edit can pin the actual §N anchor as a follow-up if desired.
- **Overnight runner picks up #238 before #239+#240 are ready**: Requirement 8 writes `blocked-by: ["239", "240"]` to backlog frontmatter, which the runner respects. Defense-in-depth alongside the spec's documented landing gate.

## Technical Constraints

- **Sandbox-safe path**: `cortex/requirements/multi-agent.md:73–78` mandates worktrees for the default repo live at `$TMPDIR/cortex-worktrees/{feature}/` to avoid Seatbelt's `.mcp.json` deny scope. The new option-2 description must not mention a path under `.claude/`.
- **Day/night split**: `cortex/requirements/project.md:11` — daytime is iterative collaboration. The rename preserves this; #246 reinforces it by removing the daytime-autonomous escape hatch.
- **Dual-source enforcement**: `CLAUDE.md` Conventions and `cortex/requirements/project.md:29`. Edit `skills/lifecycle/references/implement.md` canonical only; pre-commit hook regenerates the plugin mirror.
- **Kept user pause**: `skills/lifecycle/SKILL.md:199` names this AskUserQuestion site. The parity test (`tests/test_lifecycle_kept_pauses_parity.py`) enforces file:line presence with ±35-line tolerance. Rename does not move the call site; parity passes.
- **Daytime preflight contract test**: `tests/test_daytime_preflight.py:316–407` asserts §1a content invariants (`cortex-daytime-pipeline --feature`, `"mode": "daytime"` exactly once, `cortex-daytime-result-reader`, `cortex-daytime-dispatch-writer` ≥ 2). Holds because Requirement 6 explicitly skips §1a body.
- **What and Why, not How** (CLAUDE.md Design principle): the new description states intent and "When to pick" — not procedure.
- **MUST-escalation policy** (CLAUDE.md): label rename is not a behavior-correctness change; no events.log evidence link required. Description avoids `MUST`/`REQUIRED`/`CRITICAL`.
- **Probe coarseness limitation**: `find_spec('cortex_command')` is too coarse for the worktree-interactive path (does not check cortex-init state, sandbox allowWrite, or `$TMPDIR/cortex-worktrees` writability). Out of scope for #238; candidate follow-up ticket (see Open Decisions §1).
- **Landing gate**: #238 must not merge before #239 and #240 reach merge-ready. Enforced structurally via Requirement 8's backlog frontmatter write; documented in prose throughout this spec.

## Open Decisions

1. **Follow-up backlog ticket for richer worktree-interactive probe semantics**: the adversarial review identified that `find_spec('cortex_command')` is too coarse — a user with cortex_command installed but no `cortex init` (no sandbox allowWrite registration) sees option 2 in the menu and gets a Seatbelt-deny error after selection. Filing a backlog item is implementation-level context (whether the richer probe lives in skill prose, a new bin script, or as part of cortex init itself) that benefits from being decided alongside the worktree-interactive dispatch path's implementation. **Reason for deferral**: the right probe design depends on whether #239 builds a readiness helper or relies on direct shell checks; that decision is internal to #240's implementation. Recommend filing the follow-up at review-phase commit of this lifecycle or as part of #239's spec.
