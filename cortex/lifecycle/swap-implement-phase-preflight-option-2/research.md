## Web Research

### Label-naming convention

The proposed label aligns with industry convention. Across blog-level and tool-level prior art (incident.io, MindStudio, Nx, Anthropic's skill spec, GitHub's Agent HQ taxonomy), the dominant pattern is **name the output artifact, mention the mechanism as a modifier**. "Implement on feature branch with worktree" names the feature branch (artifact) and treats worktree as the isolation mechanism. The current label "Implement in autonomous worktree" conflates the mechanism (worktree) with the autonomy mode (autonomous) — exactly the conflation industry copy avoids. Cf. GitHub Agent HQ's "Local for fast, interactive assistance" / "Cloud for autonomous tasks" — the dominant axis is *operator stance and location*, not *git mechanism*.

### Differentiator readability

The Adversarial review flagged a real concern: option 3 is bare "Create feature branch" and option 2 (new) is "Implement on feature branch with worktree." Both lead with the same artifact ("feature branch"). Casual scanners may not parse the "with worktree" tail as the differentiator. Alternatives surfaced: "Implement in feature worktree (PR-based)" leads with the differentiator; "Implement in feature worktree" is the shortest. The discovery research and parent epic #237 bake "Implement on feature branch with worktree" — spec-phase reconsideration is possible but not required.

### "When to pick" copy

Convention across the wider AI-coding ecosystem (Cursor, GitHub Agent HQ, Swarmia's autonomy-axis article): ~one short sentence, lead with operator stance, pair with the affordance unlocked. Examples worth borrowing: "for fast, interactive assistance"; "You watch it work"; "Each agent gets its own isolated directory, its own dependencies, its own context." Avoid MUST-style framing — soft positive routing is the post-4.7 default.

### Probe-based menu gating

Niche pattern; canonical prior art is exit-code-driven dispatch (Codex Agent Skills docs, opencode skill-creator guide), `importlib.util.find_spec` as the side-effect-free probe. The codebase's current pattern (probe + route by exit code, silent hide on exit 1, fail-open on other non-zero exits) is slightly ahead of published practice. Adversarial concern: `find_spec('cortex_command')` is too coarse for the worktree-interactive path — it returns "yes" for a freshly-installed CLI where `cortex init` has not registered the sandbox allowWrite path, so `git worktree add` to `$TMPDIR/cortex-worktrees/` would fail at runtime. Out of scope for #238 (which intentionally preserves the probe mechanism) but worth a follow-up backlog item.

### Multi-ticket discriminator-rename hazard

The label string functions as a dispatch discriminator: it appears at the option-declaration site (L19) and at the routing branch (L46) within the same file, plus the probe-routing prose (L39, L40). Industry guidance on this hazard is thin — closest hit is "Skill Renamer" tools that do mechanical find/replace without semantic awareness. The project-level mitigation (grep the file for stragglers before merge, land all rename edits together) is more rigorous than published practice.

## Tradeoffs & Alternatives

### Axis 1 — Routing destination

- **(A)** Stub §1b in this ticket. Bloats #238 beyond its stated `implement.md §1` touchpoint and starts authoring content #240 owns.
- **(B)** Forward-reference `proceed to §1b` and let #240 write the body. Was the initial recommendation, but the adversarial review found this is wrong: §1b will not exist after #246 deletes §1a and the new worktree-interactive path inherits the §1a slot via renumber-on-deletion. So `§1b` is a dangling reference both pre- and post-sweep.
- **(C)** Route the new option to §1a (existing daytime) under the new label. Wrong: creates a label/mechanism mismatch.

**Recommended: write the dispatch line as `proceed to the worktree-interactive dispatch path` (prose anchor, not a §N citation)**, with a parenthetical note that #240 supplies the destination. After #240+#246 land, a follow-up edit (in #240's scope or as a small final touch in #246) can pin the eventual §N anchor. This avoids both the dangling-§1b problem and the pre-#246 §1a daytime-mismatch problem.

### Axis 2 — Probe semantics

- **(A)** Keep `find_spec('cortex_command')` probe unchanged; update only the label string in the routing prose at L39 and L40.
- **(B)** Replace with a worktree-specific probe (e.g., `command -v cortex-worktree-resolve`). More brittle (PATH-dependent), no semantic gain — `cortex_command` is the load-bearing package for both daytime and worktree-interactive.
- **(C)** Remove the probe. Loses degrade path.

**Recommended: (A).** The adversarial caveat — that the probe is too coarse for the worktree-interactive path's setup state — is real but out of scope for #238. File a follow-up backlog item for richer probe semantics (post-cortex-init state check, `$TMPDIR/cortex-worktrees` writability).

### Axis 3 — Atomic-rename strategy

- **(A)** Rename lines 19, 39, 40, 46 only. Leave line 56 alone — that line is in §1a body which #246 deletes wholesale; #238's edit would be wasted and would create a textual merge conflict with #246's sweep.
- **(B)** Add the new option alongside the old, remove the old in a follow-up. Foreclosed by the user's landing-gate constraint.

**Recommended: (A) scoped to lines 19, 39, 40, 46.** Plan body should document the deliberate skip of line 56 so the implementer doesn't see it as a missed occurrence.

### Axis 4 — Description body authoring

- **(A)** Author full description (one sentence + "When to pick" + routing) in #238 matching options 1 and 3's shape.
- **(B)** Defer all description copy to #240. Ships a stub the user actually sees.
- **(C)** Author placeholder description that names the dependency explicitly ("interactive worktree flow — see #240 for the locked interaction model").
