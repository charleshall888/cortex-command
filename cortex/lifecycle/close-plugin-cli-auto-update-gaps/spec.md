# Specification: close-plugin-cli-auto-update-gaps

> See `cortex/lifecycle/refresh-install-update-docs-close-mcp/research.md` and `spec.md` (epic predecessor #210) for the broader auto-update epic context. This spec is scoped to ticket #213's gap-closure work only.

## Problem Statement

The cortex-overnight plugin and cortex CLI are designed to stay version-coupled through a documented two-layer auto-update flow — Claude Code's marketplace fast-forwards the plugin clone at session start; on each MCP tool call, the plugin's MCP server detects when the installed CLI version differs from its pinned `CLI_PIN[0]` and reinstalls via `uv tool install --reinstall`. Empirically, three structural gaps prevent this flow from delivering its user-visible promise under wheel install: (A) `CLI_PIN` literal at `plugins/cortex-overnight/server.py:105` did not propagate from git tag to plugin source across `v0.1.0 → v1.0.0 → v1.0.1 → v1.0.2` (3-for-3 manual-step failure); (B) `pyproject.toml` `version` stayed frozen at `"0.1.0"` across all four tags and `cortex_command/cli.py:226` hardcoded the JSON envelope's `"version": "1.1"` independently of the package version, so `cortex --print-root` reported a version unrelated to the installed tag; (C) `_ensure_cortex_installed` at `plugins/cortex-overnight/server.py:443-470` short-circuited on `shutil.which("cortex") is not None` with no version comparison, so the dominant runtime case (cortex on PATH, version stale) never triggered a reinstall. This spec closes those three gaps end-to-end via a structural envelope migration (major-bump 1.x → 2.0 per `docs/internals/mcp-contract.md:19`'s forever-public-API contract), authoritative-source-of-truth docs, an agent-verifiable integration test, a custom auto-release GitHub Actions workflow (push-to-main triggers auto-bump of `CLI_PIN[0]` + tag creation, using a Personal Access Token to bypass `GITHUB_TOKEN`'s no-retrigger rule so `release.yml` fires normally on the new tag), a CI lint that hard-fails the release on `CLI_PIN[0]` drift as defense-in-depth (subsuming #212's lint goal), and the close-out of superseded #211/#212. A future migration to branch-protected PR flow + release-please is noted as a longer-term option in Open Decisions §1 but is explicitly out of scope.

## Phases

- **Phase 1: Versioning baseline (CLI side, MAJOR-bump migration)** — Adopt hatch-vcs for the CLI wheel; repurpose the `version` field in every JSON envelope emitted by `cortex_command/` to mean package version (sourced from `importlib.metadata.version("cortex-command")`); introduce a new `schema_version` field carrying the M.m schema-floor semantic that `version` previously held; bump the schema major to `"2.0"` (per `docs/internals/mcp-contract.md:19`'s forever-public-API rule that repurposing an existing field requires a major bump); update `_JSON_SCHEMA_VERSION` and all 17 `_emit_json` call sites in `cortex_command/overnight/cli_handler.py`; update `docs/internals/mcp-contract.md` to reflect the new envelope shape; update all test mocks across the 7+ test files that currently assert `"version": "1.x"`. Required substrate for Phase 2.
- **Phase 2: Plugin auto-update wiring** — Wire the version-comparison branch into `_ensure_cortex_installed` (reads `payload["version"]`, which is now the package version, compared to `CLI_PIN[0]`); update the schema-floor check `_check_version` to read `payload["schema_version"]` (which is now the M.m schema-floor); bump `CLI_PIN[1]` and `MCP_REQUIRED_CLI_VERSION` to `"2.0"` in lockstep with Phase 1's schema major; refactor `cortex_command/install_guard.py` to expose a stdlib-only core function and **vendor that core as a sibling file** `plugins/cortex-overnight/install_guard.py` enforced via `.githooks/pre-commit` dual-source parity (analogous to the existing `BUILD_OUTPUT_PLUGINS` mirroring); add the inline guard to the new R4 branch; replace the R13 silent short-circuit with a stderr remediation surface; replace the brittle `_parse_major_minor` with `packaging.version.Version`-based comparison; add `packaging` to the plugin's PEP 723 dependency declaration; pin post-reinstall verification to the absolute path returned by `uv tool list --show-paths` (closes the PATH-poisoning surface).
- **Phase 3: Auto-release on push to main + CI lint safety net** — New `.github/workflows/auto-release.yml` runs on `push: branches: [main]`: checks out main with a PAT (stored as `AUTO_RELEASE_PAT` secret) and `fetch-depth: 0` for hatch-vcs; runs validate.yml's checks inline; invokes `bin/cortex-auto-bump-version` to determine the next tag (default patch bump; `[major]` / `[minor]` markers in any commit subject since the last tag override; `[skip release]` marker bails out cleanly); rewrites the `CLI_PIN` literal at `plugins/cortex-overnight/server.py:105` to the new tag; commits with the message `Release vX.Y.Z` (passes `/cortex-core:commit`'s validation regex naturally — imperative, capitalized, no period, <72 chars); creates and pushes the tag using the PAT so `release.yml` fires normally on the tag push. A CI lint in `release.yml` hard-fails the release on `CLI_PIN[0] != tag` as defense-in-depth (catches drift even if the auto-release workflow has a bug, e.g., the PAT was revoked and a maintainer manually pushed a tag without bumping). Subsumes #212's drift lint. Also updates `release.yml` to add `fetch-depth: 0` + `fetch-tags: true` for hatch-vcs compatibility.
- **Phase 4: Tests, docs, and cleanup** — `tests/test_mcp_auto_update_real_install.py` (real `uv tool install` end-to-end in `tmp_path`, with explicit per-branch assertion contracts: each enumerated test path emits a distinct counter / NDJSON stage record so acceptance can verify all branches fired, NOT just exit-0); `tests/test_release_artifact_invariants.py` (scoped to tags ≥ first post-fix tag — accepts that v0.1.0–v1.0.2 violated the invariant by definition, and the spec's primary deliverable is preventing future violation); `tests/test_install_guard_parity.py` (asserts the stdlib-only core in cortex_command/install_guard.py and its vendored sibling in plugins/cortex-overnight/install_guard.py produce identical decisions across a shared fixture matrix); `docs/internals/auto-update.md` (with content-quality acceptance — the audit table has named rows for every load-bearing component; each `intent vs currently-wired` cell has explicit text, not TODO); update `docs/internals/mcp-contract.md` to reflect the major-2.0 envelope; rewrites of `docs/setup.md`, `README.md`, `skills/lifecycle/references/implement.md` §1a, `install.sh`; supersede #211 and #212; file the follow-up release-please ticket.

### Phase 3: Auto-release on push to main + CI lint safety net

18. **CI lint hard-fails the release-tag push on CLI_PIN drift (defense-in-depth)**. `.github/workflows/release.yml` (or a new sibling workflow `cli-pin-lint.yml`) on `push: tags: v*.*.*` runs a step that compares the `CLI_PIN[0]` literal at `plugins/cortex-overnight/server.py:105` to the pushed tag string (extracted from `${{ github.ref_name }}` or `$GITHUB_REF_NAME`). Mismatch causes the job to exit non-zero, blocking the release. Subsumes #212's lint goal. The lint is a redundant safety net even after the auto-release workflow (R19) is wired — it catches drift when the maintainer pushes a tag manually (PAT revoked, workflow disabled, emergency release, etc.). **Acceptance**: `grep -E 'CLI_PIN\[0\]|cli_pin\[0\]' .github/workflows/*.yml` returns at least one line; `tests/test_release_artifact_invariants.py` (Requirement 23) also asserts the same invariant pre-CI. **Phase**: Auto-release on push to main + CI lint safety net

19. **Auto-release workflow exists and bumps CLI_PIN + tags on push to main**. `.github/workflows/auto-release.yml` declares `on: push: branches: [main]`. Permissions block grants `contents: write`. Concurrency group is `auto-release` with `cancel-in-progress: false`. The job: (a) checks out main with `token: ${{ secrets.AUTO_RELEASE_PAT }}` and `fetch-depth: 0`, `fetch-tags: true`; (b) runs validate.yml's checks inline (or via `workflow_call` if validate.yml is later restructured as reusable — out of scope for this ticket); (c) invokes `bin/cortex-auto-bump-version` to determine the next tag; (d) exits cleanly (no-op) when the helper indicates "no bump" (no commits since last tag, OR `[skip release]` marker present in any commit subject since last tag); (e) rewrites `CLI_PIN` literal at `plugins/cortex-overnight/server.py:105` to `("<new_tag>", "2.0")` (the `"2.0"` schema-major is held constant per Requirement 17 unless a future change bumps it); (f) commits with message exactly `Release vX.Y.Z` (substituting the new tag); (g) creates the new tag pointing at the bump commit; (h) pushes branch + tag using the PAT, so the tag-push event retriggers `release.yml`. **Acceptance**: `test -f .github/workflows/auto-release.yml`; `grep -cE 'branches:\s*\[?main\]?' .github/workflows/auto-release.yml` returns at least 1; `grep -E 'AUTO_RELEASE_PAT' .github/workflows/auto-release.yml` returns at least one match; `grep -E 'contents:\s*write' .github/workflows/auto-release.yml` returns one match; `grep -E 'concurrency:' .github/workflows/auto-release.yml` returns one match. **Phase**: Auto-release on push to main + CI lint safety net

20. **`bin/cortex-auto-bump-version` helper exists with deterministic version-selection logic**. The helper takes no required args; supports `--dry-run` for tests. Logic: (i) read latest tag via `git describe --tags --abbrev=0`; if no tags exist, default to `v0.1.0`; (ii) read commit subjects since the latest tag via `git log <latest-tag>..HEAD --format=%s`; (iii) if any subject contains `[skip release]` (case-insensitive), emit `no-bump\n` to stdout and exit 0; (iv) if any subject contains `[major]`, return major-bump tag (e.g., `v1.0.2` → `v2.0.0`); (v) else if any subject contains `[minor]`, return minor-bump tag (e.g., `v1.0.2` → `v1.1.0`); (vi) else return patch-bump tag (`v1.0.2` → `v1.0.3`); (vii) if HEAD already equals latest tag (no new commits since tag), emit `no-bump\n` and exit 0. The helper writes the next tag string to stdout (single line, `vX.Y.Z` form). Unit-tested in `tests/test_cortex_auto_bump_version.py` with at least 6 cases: no tags, no commits since tag, pure-patch, `[minor]` in commit, `[major]` in commit, `[skip release]` in commit, mixed-markers (highest wins). **Acceptance**: `test -x bin/cortex-auto-bump-version`; `just test -- tests/test_cortex_auto_bump_version.py -v` exits 0 and output lists at least 6 PASSED lines. **Phase**: Auto-release on push to main + CI lint safety net

21. **PAT setup documented in release-process.md**. `docs/release-process.md` is updated with a section "Auto-release PAT setup (one-time)" that describes: (a) creating a fine-grained PAT scoped to the cortex-command repo; (b) required permissions (`contents: write`); (c) storing as repo secret `AUTO_RELEASE_PAT`; (d) rotation cadence (the maintainer rotates the PAT every 90 days or when GitHub notifies of expiry); (e) emergency manual override (if the PAT is revoked / workflow disabled, the maintainer can push a tag manually and the CI lint at R18 catches CLI_PIN drift). **Acceptance**: `grep -E '^## Auto-release PAT setup' docs/release-process.md` returns one match; `grep -E 'AUTO_RELEASE_PAT' docs/release-process.md` returns at least two matches (in the section body). The PAT itself is a **one-time interactive setup the maintainer performs manually** — outside the scope of agent-verifiable acceptance. The spec acknowledges this as Interactive/session-dependent: an agent cannot verify the PAT exists or is valid in the GitHub UI; only the workflow's first run after PAT setup confirms it operationally. **Phase**: Auto-release on push to main + CI lint safety net

### Phase 4: Tests, docs, and cleanup

22. **Real-install integration test with per-branch assertion contracts**. `tests/test_mcp_auto_update_real_install.py` exists and exercises (within a `tmp_path`-isolated env with `UV_TOOL_DIR`/`UV_TOOL_BIN_DIR`/`UV_CACHE_DIR` redirected) six explicitly-asserted test paths:

## Non-Requirements

- **No back-port compatibility shim**: pre-v2.0.0 plugin/CLI combinations are not supported. Per the user's decision at spec-phase critical-review, they will manually reinstall both halves on a coordinated upgrade. This is a one-time breaking change.
- **No editable-install support**: spec scopes to non-editable wheel install only (per `project.md:55`). Editable contributor flow uses `uv build` + `uv tool install --force` manually.
- **No mocking of Claude Code marketplace fast-forward**: real-install test (R19) covers cortex-command-side mismatch detection only. The Anthropic-owned marketplace fast-forward layer (force-pushed-tag handling, session-restart timing, tag-not-yet-on-origin race) is documented residual risk in `docs/internals/auto-update.md`.
- **No release-please / branch-protected PR flow adoption**: Phase 3 ships a custom auto-release GitHub Actions workflow + PAT (Requirements 19-21) — simpler than release-please for cortex-command's current single-maintainer shape. Adoption of release-please + conventional-commits convention + branch-protected PR flow is a longer-term option noted in Open Decisions §1; not deferred to a specific follow-up ticket because it depends on project shape evolving (multi-contributor, CHANGELOG-as-API).
- **No expansion of the `~/.claude/` write surface**: per `project.md:28`, the only `~/.claude/` write is via `cortex init`. The integration test's tmpdir stays in `$TMPDIR` (in global sandbox allowWrite).
- **No PEP 440 local-version emission**: `[tool.hatch.version.raw-options] local_scheme = "no-local-version"` suppresses `+gabc1234` suffixes on dirty checkouts.
- **No retention of `_parse_major_minor`**: the brittle parser is removed entirely (Requirement 14); `packaging.version.Version` is the universal comparison API in the plugin.
- **No removal of the existing R4 first-install branch**: the new version-comparison branch is additive; the `shutil.which → None` first-install path is preserved as-is.
- **No support for non-deterministic tag-build order**: `release.yml` must check out the tag commit's tree, build the wheel from that exact tree state, and the `package_version` it derives must equal the tag (with `v` stripped). The auto-release follow-up will need to preserve this property.

## Edge Cases

- **First push to main after this ticket lands**: the implement-PR's merge to main triggers the auto-release workflow's first run. The workflow detects commits since `v1.0.2` (the previous tag) including `[major]` markers (the implement-PR's schema-major-bump commit subjects MUST include `[major]`), bumps to `v2.0.0`, rewrites `CLI_PIN`, commits, tags, pushes. The PAT must already be configured (R21) before merge; if not, the workflow fails and the maintainer either configures the PAT and re-triggers via `workflow_dispatch`, or pushes the `v2.0.0` tag manually (CI lint R18 verifies CLI_PIN matches; release-artifact invariant test R23 skips historical v0.1.0–v1.0.2 tags via date scoping).

- **Maintainer forgets `[major]` marker on a breaking change**: the workflow defaults to patch bump; a breaking change ships as a patch. The CI lint catches CLI_PIN drift but NOT semantic-versioning violations. Mitigation: `/cortex-core:commit` skill prose adds a reminder to include `[major]`/`[minor]`/`[skip release]` markers when applicable. The maintainer can manually push a corrected tag if the workflow under-bumps; the CI lint at R18 will then enforce CLI_PIN consistency with the manual tag.
- **Pre-v2.0.0 plugin paired with v2.0.0+ CLI**: hard-fail by design. Old plugin's `_parse_major_minor` reads the new envelope's `version` field (now a PEP 440 string like `"2.0.0"`), splits on `.` `maxsplit=1`, calls `int("0.0")` → `ValueError`. User must manually `uv tool install --reinstall cortex-command` to get the new CLI, and Claude Code marketplace will fast-forward the plugin within one session-start cycle. Documented in `docs/internals/auto-update.md` as a one-time migration step.
- **Force-pushed release tag**: documented residual risk; uv tool cache may serve the stale wheel. The CI lint (R18) cannot catch this; the recommended remediation is `uv cache clean cortex-command` before reinstall on force-push (documented in `docs/internals/auto-update.md`).
- **`packaging.version.Version` raises `InvalidVersion`**: the version-compare branch (R9) catches the exception and emits NDJSON with `stage="version_mismatch_reinstall_parse_failure"` — explicit disambiguation from the legitimate-mismatch stage so the integration test (R22 phase c) can verify which branch fired.
- **`importlib.metadata.version("cortex-command")` raises `PackageNotFoundError`**: the print-root path falls back to `"0.0.0+source"` sentinel. The plugin's R9 branch sees this PEP 440 string and triggers reinstall (legitimate path; the `+source` local-version is parseable by `packaging.version.Version`).
- **Concurrent overnight session + R4 version-mismatch detected**: R12's in-flight guard fires; R4 aborts the reinstall with an NDJSON record (`stage="version_mismatch_blocked_by_inflight_session"`) and a stderr message. The session completes; the next MCP tool call after the session exits retries the reinstall.
- **macOS without `/proc`**: the vendored `install_guard.py` core falls back to `os.kill(pid, 0)` for liveness AND a `ps -p <pid> -o comm=` parse to verify the pid is actually a cortex process (closes the recycled-pid false-positive surface). The parity test (R24) verifies both implementations make identical decisions on macOS-style fixtures.
- **Test runner is sandboxed without network egress**: R19's real-install test calls `pytest.fail("uv tool unavailable; integration test cannot run as agent-verifiable acceptance")` — NOT `pytest.skip` — so the acceptance gate explicitly fails rather than silently passing. Per project.md:13 Handoff readiness.
- **Auto-release workflow concurrent with manual tag push**: if the maintainer manually pushes a tag while the workflow is mid-bump, the workflow's `git push --tags` may collide. Concurrency control with `cancel-in-progress: false` queues runs but does not prevent manual-tag collisions. Recommended convention: maintainer disables the workflow (`gh workflow disable auto-release.yml`) before any emergency manual tag push.

## Technical Constraints

- **Plugin's interpreter ≠ CLI's interpreter under `uv tool install`**: confirmed by research and adversarial review. Plugin runs in an ephemeral PEP 723 venv with only `mcp`, `pydantic`, and (post-spec) `packaging`. `importlib.metadata.version("cortex-command")` from the plugin raises `PackageNotFoundError`. The plugin learns the installed CLI's version exclusively by subprocess-shelling `cortex --print-root --format json`.
- **`packaging.version.Version` is mandatory for plugin-side comparison**: post-Phase-1, the envelope's `version` field carries PEP 440 strings (e.g., `"2.0.0"`, possibly `"2.0.0+source"`). `_parse_major_minor` cannot parse these.
- **hatch-vcs requires full git history**: `fetch-depth: 0` + `fetch-tags: true` mandatory in any workflow that builds the wheel.
- **In-flight install guard semantics matched by structural enforcement, not prose**: per CLAUDE.md "Prefer structural separation over prose-only enforcement," parity between `cortex_command/install_guard.py:check_in_flight_install_core` and `plugins/cortex-overnight/install_guard.py:check_in_flight_install_core` is enforced by (a) `.githooks/pre-commit` dual-source byte-identity check, (b) `tests/test_install_guard_parity.py` source-identity assertion + decision-matrix parity, (c) both implementations consume the same `_ACTIVE_SESSION_PATH` constant via duplication-by-design (mirroring the existing `cortex_command/install_guard.py` comment that justifies the path duplication relative to `ipc`).
- **Schema-major bump is one-way; pre-v2.0.0 pairings hard-fail by design**: spec accepts the one-time breaking change per the user's decision; documented in CHANGELOG (R29) and `docs/internals/mcp-contract.md` Schema-evolution-log subsection.
- **MCP-contract forever-public-API rule**: per `docs/internals/mcp-contract.md:19`, "existing fields never change semantics or types without a major bump." Repurposing `version` IS a major bump (1.x → 2.0); this spec executes that major bump rather than evading it via an additive-only path.
- **`uv tool install --reinstall` cache reuse on force-pushed tags**: documented residual risk in `docs/internals/auto-update.md`. Recommended remediation: `uv cache clean cortex-command` before reinstall on suspected force-push. Out of scope for #213.
- **`fetch-depth: 0` cost**: acceptable for a single-maintainer project at the current repo size; documented as accepted cost.
- **The auto-release workflow (R19) handles routine bumps; the CI lint (R18) is the defense-in-depth for manual-tag emergencies**: maintainer doesn't bump `CLI_PIN` manually on the happy path — the workflow does it on every push to main. The CI lint catches drift only when the workflow is disabled or PAT-revoked and the maintainer pushes a tag manually.

## Open Decisions

- **§1. Long-term evolution to branch-protected PR flow + release-please**: this ticket ships a custom auto-release workflow + PAT (Requirements 19-21) — simpler than release-please for cortex-command's current single-maintainer trunk-based shape. The longer-term recommendation, once cortex-command grows multi-contributor and CHANGELOG-as-API matters, is to migrate to: (a) branch protection on main (require PR + status checks); (b) `/cortex-core:commit` rewrite to commit on a feature branch + open PR; (c) release-please (Google's industry-standard tool, used by Anthropic's `anthropic-sdk-python` and other open-source SDK repos); (d) conventional-commits convention adoption to drive release-please's version bumps; (e) GitHub App authentication replacing the PAT. This migration is intentionally NOT scoped to a specific follow-up ticket because the trigger is project-shape evolution (number of contributors, CHANGELOG-as-API need), not a fixed date. One-sentence reason: filing this as a follow-up ticket today would commit to an architecture migration whose trigger is operational (multi-contributor adoption), and the trigger isn't yet present.
- **§2. `_emit_json` consumer enumeration timing**: the spec lists 17 call sites by line number to verify at implement time, but the exact list may shift if the file changes between spec approval and implement start. Implement-phase tasks must regenerate the list via `grep -nE '_emit_json\(' cortex_command/overnight/cli_handler.py` and reconcile with the spec's enumeration before claiming Phase 1 complete. One-sentence reason: this is an implementation-time freshness check, not a spec-time decision — the spec's enumeration captures the snapshot at spec-write time.
