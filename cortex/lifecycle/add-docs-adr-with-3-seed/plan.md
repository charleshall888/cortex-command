# Plan: add-docs-adr-with-3-seed

## Overview

Two-phase implementation: Phase 1 (Mechanism) creates `cortex/adr/` with a 5-section policy doc and three seed ADRs (file-based-state, CLI-wheel-plus-plugins, sandbox-registration), then replaces four source prose locations in `cortex/requirements/project.md` (Overview + three Architectural Constraint bullets) with `→ ADR-NNNN` back-pointers. Phase 2 (Integration) modifies `skills/lifecycle/references/specify.md` to add the §2 in-the-moment ADR-proposal posture, the always-rendered `## Proposed ADR` section in §3's spec template, and the `Proposed ADRs` consent bullet in §4 — together making a silently-skipped emission gate observable at the approval surface.

## Outline

### Phase 1: Mechanism (tasks: 1, 2, 3, 4, 5)
**Goal**: Ship `cortex/adr/` as a self-contained, browsable decision log with three seeds and project.md back-pointers — satisfying the no-duplication discipline at introduction.
**Checkpoint**: `cortex/adr/` contains `README.md` + `0001-*.md` + `0002-*.md` + `0003-*.md`; `cortex/requirements/project.md` contains four `→ ADR-NNNN` back-pointers (Overview line ~7 → ADR-0002, Architectural Constraint line ~27 → ADR-0001, line ~28 → ADR-0003, line ~32 → ADR-0002) and the corresponding source prose has been moved entirely to the ADRs.

### Phase 2: Integration (task: 6)
**Goal**: Wire the §2/§3/§4 always-render observability anchor into specify.md so spec authors see an explicit `Proposed ADRs: <list or None>` line at every approval surface.
**Checkpoint** (plan-verifiable on the working tree): `skills/lifecycle/references/specify.md` contains the §2 posture line, `## Proposed ADR` template section with `None considered.` default, and `Proposed ADRs` consent bullet inside §4 — verifiable via the Task 6 grep clauses.
**Post-deploy integration check** (not a plan-acceptance gate): once the cortex-core plugin is re-published and `/plugin update` runs, a fresh `/cortex-core:lifecycle` invocation on a one-line test feature produces a spec.md whose `## Proposed ADR` section reads `None considered.` and whose §4 approval surface output contains `Proposed ADRs: None`. This is a future-state assertion, not checkable from the merge candidate.

## Risks

- **The §2 in-the-moment posture is prose-only enforcement.** The structural anchor is the §4 `Proposed ADRs: None` line, which surfaces a skipped gate but does not enforce that the gate's criteria were correctly applied. Acknowledged in spec Edge Cases ("Author skipped the three-criteria gate during §2") and is the deliberate v1 posture per the CLAUDE.md:58 carve-out defended in the README — but if events.log F-row evidence accumulates that authors are skirting the gate, the deferred upgrade is a pre-commit check on new ADRs (per spec Non-Requirements).
- **Back-pointer staleness after ADR rename via supersede** is offloaded to PR review (spec Edge Cases). No CI grep ships with this plan. If a seed gets superseded post-merge, the back-pointer in project.md will silently go stale.
- **The CLAUDE.md MUST-escalation policy section** (originally proposed as ADR-0003) stays in CLAUDE.md unchanged. If a future operator looks at CLAUDE.md and asks "shouldn't this be an ADR?", the answer is "no — it's a policy specification, not a decision record" (spec Non-Requirements). The risk is the question recurs; the mitigation is the README explaining the distinction.
- **Sequential-numbering merge race** is acknowledged in spec Edge Cases — the second to merge bumps the number. Because the spec's `Proposed ADRs` bullet shows the slug, the user approved a specific number, but a post-merge bump is treated as cosmetic (the content was approved, not the number). No code change needed; PR reviewer responsibility.
