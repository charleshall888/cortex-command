# Research: Harden the refine→research considerations hand-off (drop the arg-escaping caveat) — #337

**Clarified intent**: Replace the fragile `research-considerations="..."` argument channel between `/cortex-core:refine` and `/cortex-core:research` — whose value rides inside research's model-parsed `$ARGUMENTS` string and therefore cannot contain `=` or `"`, forcing both skills to carry character-stripping prose — with a channel that carries arbitrary multi-line text without escaping, so neither skill needs the caveat. Decide between an implicit slug-derived file and an explicit file-path argument, and resolve the absence/stale-file semantics.

**Tier/criticality**: complex / high. **Mode**: lifecycle. **research-considerations injection**: none (clarify-critic produced no `origin: alignment` findings, so nothing was propagated).

## Headline finding (read first)

Research weighs the two candidate designs as a **clear lean toward Approach B (explicit path argument)**, reversing the mid-research tentative lean toward Approach A. The reversal is grounded in file-verified evidence, not preference:

1. **The original #322 research already decided this** and chose explicit-path. `cortex/lifecycle/offload-refine-step-2-state-checking/research.md:124` states verbatim: *"Implicit-by-slug (e) is worse than explicit (a) — a hidden channel that exists only in lifecycle mode and is invisible to research's standalone path… prefer explicit `--research-considerations-file`; inject file content, not the path."* #337 was created to **re-decide** this with its own spec/critical-review, so the prior decision is not binding — but reversing it requires new evidence, and the new evidence (below) *reinforces* B, it does not overturn it.
2. **A re-introduces the very prose discipline #337 exists to remove.** Today's channel "fires only when at least one Apply'd alignment finding exists… omit the argument entirely" (`skills/refine/SKILL.md:129`) — absence is structural (no arg). Approach A makes research *always* derive-and-read the path (no arg to omit), so "no findings" must be actively materialized as a cleared/empty file. That mandatory-clear-each-run rule is load-bearing prose; if it lives only as a SKILL instruction, #337 has **swapped an escaping caveat for a clear-each-run caveat**. Approach B keeps today's conditional-omit unchanged and needs no clear discipline at all.
3. The "stale-file risk is symmetric across A and B" claim (the Tradeoffs angle's central pro-A argument) is **provably false** — verified against the source it cited. The asymmetry is the whole point.

This is the spec's central decision and is restated as the lead item under `## Open Questions`. The remaining findings hold regardless of which design wins, except where noted.

## Web Research

File-based hand-off is the **established orchestration norm**, not a one-off. Airflow XCom docs + Astronomer call passing large/unstructured data through the inline channel "an anti-pattern" and prescribe writing the payload to file storage and passing only the reference — exactly #337's shape. Cloud Build and Google ADK/LangChain "artifacts" reinforce reference-not-inline for sizeable/unstructured content. Microsoft's command-line-quoting writeups document the precise failure mode #337 hits (values with quotes/`=`/spaces break the parse; escaping differs across parsers) and conclude the durable fix is to stop encoding arbitrary text in a parsed string.

The load-bearing risks of a file channel (independent of A/B):
- **Stale sentinel** — a file left from a prior run read as if it were this run's signal ("a circuit breaker that never resets"). Mitigation seen in the wild: producer rewrites/clears at the start of each run (Gradle/Shake "clean-before-produce"), or embeds a run-id/TTL.
- **Present-empty vs. absent vs. present-with-value are three distinct states** and must be checked distinctly (Baeldung empty-vs-null guidance) — name the check, don't conflate.
- **Atomic write** (temp-in-same-dir → rename) avoids torn reads; but for #337's **sequential, same-process, single-filesystem** handoff (producer fully completes before consumer starts) the concurrent-read race is largely moot — atomic rename is cheap insurance, not strictly required. The genuinely load-bearing risks are prior-run staleness and the empty-vs-absent ambiguity.

## Requirements & Constraints

**SP001/SP002 + ADR-0009 is the only constraint the ticket itself flags**, and it is the central one. The invariant (project.md:46, ADR-0009): `${CLAUDE_SKILL_DIR}` resolves only in a SKILL.md body; bare relative paths resolve against CWD off-repo; composed subagent prompts must receive the **resolved absolute path or inlined content**, never a raw token / bare consult-ref / bare-relative path. The compliant shape for this change is: research's **body** reads the file and injects the **content** into the core-angle prompts (it already does exactly this for the arg value via `{research_considerations_bullets}`).

**Gates that run on the edits but pass unchanged for a body-only / no-new-CLI / no-new-event design:**
- **SKILL↔bin parity** (`cortex-check-parity`): script-centric, not prose-centric. Only bites if a new `cortex-*` helper is introduced (then it needs a wiring reference or a `.parity-exceptions.md` row).
- **Mirror canonical-edit rule**: `cortex-core` is a build-output plugin; both refine and research are in its SKILLS array; `just build-plugin` rsyncs wholesale; `test_dual_source_reference_parity.py` byte-checks it and the pre-commit drift gate blocks a stale mirror.
- **L1 surface ratchet**: bounds only `description`+`when_to_use` bytes. refine = 624B, research = 379B; **both are cluster skills** (`dev/lifecycle/refine/research/discovery/critical-review`). Body-only edits don't touch frontmatter → stays green. Raising a row needs documented rationale + lifecycle-id, but no raise is needed.
- **Size cap** (500 lines): refine = 200, research = 244 — ample headroom.
- **Events-registry / backlog-grep**: the channel emits no event today and introduces none → no registration needed.

**Structural-over-prose + What/Why-not-How (CLAUDE.md:58–59, 65–75)** are the principles motivating the change: the existing prose caveats "exist only because the channel is fragile." This cuts *toward Approach B* on the absence-semantics axis — B preserves a structural absence signal (arg omitted) whereas A's mandatory-clear is itself a prose discipline unless made structural by other means.

## Tradeoffs & Alternatives

**Approach A — implicit slug-derived file.** Refine writes `cortex/lifecycle/{slug}/research-considerations.md`; research derives the path from the slug it holds and reads it. *Pros*: no new arg; lowest interface novelty (reuses the existing slug→path derivation); standalone-mode reads nothing by construction; SP-compliant by construction. *Cons*: research **always** reads in lifecycle mode, so "absence = no findings" depends entirely on a mandatory clear-each-run discipline (the highest stale-file reliance of any option — see Stale-File and Adversarial sections); inherits the lifecycle-dir git-disposition problem; weaker test-token (filename only).

**Approach B — explicit path argument.** Refine writes the file and passes `research-considerations-file=<path>`, fired only when findings exist this run. *Pros*: **drops the escaping caveat for the considerations text** (a path almost never contains `=`/`"`); **keeps today's structural absence semantics** (arg omitted = no read = no clear discipline needed); safe on `resume=research` and cross-invocation re-runs by construction; clean `research-considerations-file` token for the handoff-schema test; lets the file live outside the lifecycle dir to sidestep the git-disposition problem entirely. *Cons*: retains a model-parsed arg (the fragility *class*, narrowed to a path that rarely needs escaping); still requires the "inject content, not the path" discipline (un-linted — see below); one new supported key in research Step 1.

**Approach C — third options, all rejected/dominated**: appending considerations into a file research already reads collapses into A (no pre-seeded input file exists in lifecycle mode); a subdir `_inputs/considerations.md` is strictly worse than a flat file (more novelty); reusing `index.md` frontmatter or `events.log` re-introduces escaping/pollutes machine-parsed channels; a **new `cortex-refine write-considerations` CLI verb** was explicitly rejected by #322 as "gratuitous ceremony" (research.md:102) — but see the Stale-File section for the counter-argument that the unconditional-clear *is* deterministic logic worth a verb if Approach A is chosen.

## Stale-File & Absence Semantics

**Considerations derive only from Clarify-stage clarify-critic dispositions, not from interview answers.** Therefore, within a single refine invocation, they are **invariant across a specify.md §2a loop-back** (verified: `specify.md:71–81` and refine Step 5 loop back into the *Research phase*, not Clarify, so dispositions can't change mid-invocation). This produces the sharp design fact: in the same directory, **`research.md` IS invalidated on a §2a loop-back** (it depends on the interview that changed) **while the considerations are NOT** — opposite loop-back semantics for two co-located artifacts.

**But the §2a-invariance argument does NOT generalize** (Adversarial verification): the `resume=research` branch (`skills/refine/SKILL.md:48`) explicitly **skips Clarify**, so clarify-critic never runs and the propagation block has no fresh source — yet under Approach A a leftover `research-considerations.md` from a prior full run persists and is read → **stale considerations injected**. Cross-invocation re-runs where the user dismisses previously-Apply'd findings are similarly exposed. Approach B is safe on both by construction (no Clarify → no findings → no arg → no read).

**Reader contract** (define explicitly, for whichever design): `absent == present-but-empty == whitespace-only == empty-bullet-only` ⇒ **no injection**, mirroring today's `empty/absent → no considerations injection`. This decouples the reader from the writer's delete-vs-truncate choice and must reject the "halt on missing file" treatment that `research.md` gets at `refine SKILL.md:116`.

**Write-ordering hazard** (confirmed against the file): refine's propagation block (118–131) is authored *after* "Research Execution" (104–116). For an arg this is harmless (the model assembles+dispatches atomically). For a file, a faithful read of the prose writes the file *after* the dispatch — a **read-before-write defect** injecting nothing on the first run. The spec must sequence the write/clear *before* the dispatch (or, under B, fold path-write into the same step that builds the dispatch).

**Structural-vs-prose ledger**: the escaping-caveat removal, standalone-reads-nothing, absent==empty, and §2a-invariance are all structurally enforceable. The **clear-each-run / no-leak-on-re-run** requirement is the one that becomes prose-only under Approach A (research always reads) — making A's central edge a relocated caveat unless backed by a deterministic write-or-truncate verb. Under Approach B it stays structural (arg omitted), needing no clear at all.

## Path-Resolution & Subagent Propagation

Today's channel is **already "content into subagent prompts"** (the arg value is bullet text, injected via `{research_considerations_bullets}` into the three core-angle prompts built in research's body) — confirmed.

**Neither A nor B is mechanically lint-gated for a `cortex/lifecycle/...` path.** Verified in `cortex_command/lint/skill_path.py`: D2's `_D2_PATH_BODY` (line 139) matches only `references/`, `../`, `skills/` roots — a `cortex/lifecycle/...` path matches none. D1 fires only inside `<!-- BEGIN/END SUBAGENT PROMPT -->` fences or `*-prompt.md` files; research/SKILL.md uses plain code fences and has neither, so D1 is **dormant** there.

**Consequence (a new, unguarded failure mode):** the arg channel was self-contained by construction (the arg value *was* the content). A file channel forces research to read content and inject the **content**; if a naive implementation instead injects the **path** (so subagents read the file themselves), **nothing catches it** — D1 dormant, D2 regex doesn't match the path shape. "Inject content, not the path" is therefore a *design-discipline* requirement with **zero regression guard**, applicable to both A and B. It must be specified as a hard requirement and covered by a test (a negative assertion that the path string does not appear inside the core-angle prompt blocks).

## Test & Verification Design

The channel is purely SKILL.md prose with no code surface (`cortex_command/refine.py` has zero considerations logic) and **no existing test**. Realistic verification surface:

1. **Prose-contract tests on canonical refine + research**, modeled on `tests/test_refine_skill.py` (anchored block slicing via `_slice_section_*`, presence/proximity/ordering, and the **negative-regression guard** pattern `assert "<old prose>" not in block`). Each test pairs a positive assertion with a **non-vacuity anchor check** (assert the section heading still exists) so a section rename can't make an absence-assertion pass falsely. Discriminating tests to author: (a) escaping caveat gone from both skills [+ anchor exists]; (b) file-channel described AND old `research-considerations="` / supported-keys entry removed; (c) absence=no-findings preserved AND no halt-on-missing; (d) standalone reads nothing AND the read is lifecycle-gated; (e) `resume=research` / re-run path re-derives-or-clears (not "read existing as-is").
2. **Handoff-schema registration**: add the channel as a `handoff_field` in `tests/fixtures/skill_handoff_schema.yaml` so `tests/test_skill_handoff.py` enforces producer+consumer prose presence, with its existing `pytest.raises(AssertionError)` fixture as a built-in negative control. **This favors B**: the schema keys on a literal field-name token — B gives a clean compound token `research-considerations-file`; A has only the filename `research-considerations.md`, which is ambiguous against prose nouns.
3. **Inject-content-not-path** negative test (closes the un-linted gap from the SP section).
4. **Mirror parity**: `tests/test_dual_source_reference_parity.py` is THE gate (routes both refine & research to the cortex-core mirror, byte-compares); `test_plugin_mirror_parity.py` does NOT cover these files. Pre-commit Phases 2–4 block a stale mirror.
5. **Structural unit test only if** a path-deriving helper / CLI verb is added (a parametrized slug→path test + a `valid-*`/`invalid-*` contract fixture). Absent a deliberate decision to add code, that is over-engineering a prose change.

Runner: `just test` (`.venv/bin/pytest tests/ -q`); new `tests/test_*.py` files are auto-collected.

## Adversarial Review

(Findings verified against source; folded into the sections above and the Open Questions below. Recorded here as the explicit critique of record.)

- **Untracked is not a neutral non-decision — it breaks three dirty-tree gates** for *any* file placed in `cortex/lifecycle/{slug}/`: `lifecycle_implement.py:48–65` `_is_dirty_tree` (spurious `dirty_tree` picker), `complete_route.py:449` `_branch4` (routes to `merged_dirty`, blocking auto-completion), `complete_route.py:333` `_finalization_committable` (directory pathspec makes finalization look committable). **Gitignore is forced, not optional.**
- **Committing the file via `collect_paths` is also broken**: truncate-clear → an empty file committed every refine run (churn + a committed blank artifact); delete-clear → the deletion is filtered out of the add-list (candidates are `present-on-disk`, `stage_artifacts.py:265`) → unstaged tracked deletion → dirty tree persists after `nothing_staged`. So **gitignore is the only workable disposition** for an in-lifecycle-dir file.
- **Existing-repo propagation tail**: the `.gitignore` fix lives in `cortex_command/init/templates/cortex/.gitignore` (new repos only). Existing repos won't gain the rule without re-syncing — there the file is untracked and the dirty-tree breakage applies. `test_cortex_gitignore_template.py` pins template content but doesn't exercise the propagation gap. (This is the clearest argument for **Approach B with the file living OUTSIDE the repo tree** — e.g. a scratch path — which sidesteps git disposition entirely.)
- **`gc_demo_worktrees.py:90`** uses `git status --porcelain --ignored=traditional` (counts ignored as dirty) — but any lifecycle-ran demo worktree is already dirty-under-ignored from `.session`/`agent-activity.jsonl`/`metrics.json`, so net-new impact is zero. The blanket "every consumer is SAFE" was imprecise; this is the one consumer that surfaces ignored files, already saturated.
- **No cleaner reclaims the file** (`clean.py` only sweeps `_adhoc/` + walks events.log); minor per-feature disk cruft. If `complete`/archival moves a lifecycle dir file-by-file (`git mv`) rather than whole-dir rename, a gitignored file is orphaned in the pre-archive path — pin which archival mode is used.
- **Slug-collision inheritance**: lifecycle slug is a lossy truncated prefix of the backlog filename (`stage_artifacts.py:210`); two items truncating to the same slug share the dir and the considerations file — a pre-existing hazard (research.md/spec.md already collide) that the new file inherits.

## Open Questions

1. **[CENTRAL — spec + critical-review decision] Approach A (implicit slug-derived file) vs. Approach B (explicit path argument).** Research leans **B**, but flags this as a genuine contradiction across the angles that must be resolved deliberately, not defaulted:
   - Pro-B (the weight of evidence): #322's prior decision; B preserves today's structural absence-semantics (no clear-discipline prose → doesn't relocate a caveat); B is safe on `resume=research` + cross-invocation re-runs by construction; B is more testable (clean handoff token); B can place the file outside the lifecycle dir to sidestep the git-disposition breakage entirely.
   - Pro-A (the dissent): no new arg key; lowest interface novelty; standalone-reads-nothing and SP-compliance by construction. The Tradeoffs angle preferred A — but its "symmetry" premise was refuted, so A's case rests only on interface-minimalism.
   - The decision pivots on: does the spec value "zero new interface surface" (A) over "preserved structural absence-semantics + no caveat relocation" (B)? Structural-over-prose argues B.
2. **If Approach A is chosen**: must the unconditional write-or-truncate-each-run be (a) SKILL prose, or (b) a deterministic `cortex-refine write-considerations --lifecycle-slug X [--clear]` verb? The verb makes "clears each run" unskippable (structural) and carries genuine deterministic logic — but contradicts #322's "gratuitous ceremony" rejection and adds CLI surface (parity/contract gates then engage). Prose-only risks the caveat-relocation that defeats #337's goal. Resolve only if A wins.
3. **Where does the file live?** If in `cortex/lifecycle/{slug}/`: gitignore is **mandatory** (template + `test_cortex_gitignore_template.py` case, e.g. `lifecycle/**/research-considerations.md`), and the existing-repo propagation gap must be acknowledged. If Approach B with a scratch/out-of-repo path: git disposition disappears, but the path must be deterministically derivable by research and the cleanup/lifetime stated. Decide placement alongside #1.
4. **Write-ordering**: the spec must sequence the file write/clear *before* the research dispatch (the propagation block currently sits after Research Execution — a naive port injects nothing on the first run).
5. **"Inject content, not the path" must be a hard, tested requirement** — the SP lint cannot enforce it for a `cortex/lifecycle/...` path (D1 dormant, D2 regex non-matching). Add a negative regression test asserting the path string never appears inside the core-angle prompt blocks.
6. **Reader contract precision**: define no-injection as `absent OR empty OR whitespace-only OR empty-bullet-only`, and confirm the `## Considerations Addressed` trigger (today "research-considerations non-empty AND lifecycle mode") still fires correctly when the source is filesystem state — restate it in file terms and drop the now-redundant mode clause if A is chosen.
7. **Archival mode for `complete`**: confirm whole-dir rename (moves a gitignored file) vs. per-file `git mv` (orphans it); minor, but pin it so a gitignored considerations file isn't left behind.
