# Specification: fix-validate-brief-substring-anchors-that

## Problem Statement

The discovery brief-gen gate is dead by default. Across the seven observed `gate_brief_generated` events in `cortex/research/*/events.log` and `cortex/lifecycle/*/events.log`, zero produce a passing brief — but the failure modes split into three independent classes that this spec must keep distinct:

- **6/7 events**: sub-agent emits empty output (`brief_word_count: 0`). These are upstream SDK / agent failures the gate correctly catches; this spec does not change that behavior. They are not paraphrase-rejection cases.
- **1/7 events**: the 439-word brief from `cortex/research/harness-friction-triage/events.log` blows the word cap. This is the only event from which we can recover any prose at all, and even that recovery requires re-running the sub-agent — the brief text itself is not preserved on failure (Req 2 fixes the observability gap).
- **Independent structural bug** (not in the event corpus): `cortex_command/discovery.py:285-322` (`GATE_BRIEF_RUBRIC`) is a plain triple-quoted string, not an f-string. Line 305 reads `Word target: write no more than GATE_BRIEF_WORD_CAP words.` — the literal token reaches the sub-agent rather than the numeric `250`. This is the more parsimonious explanation for the 439-word over-cap brief; the paraphrase-vocabulary issue may or may not be coincident.
- **Independent structural contradiction** (orthogonal to the event corpus): the rubric instructs the sub-agent to "Use ordinary words" and offers `settled on` as its own example verb, both of which the validator's narrow substring anchors (`decided|decide`, `alternative|options`, `tradeoff|cost`) reject. The rubric's own example fails its own validator — a load-bearing inconsistency regardless of empirical pass-through rate.

Result: the human gate-reader sees the dense Architecture section when they should be seeing the compressed brief, and the structural contradiction in the rubric/validator pair prevents the brief.md path from ever firing reliably even if the SDK and word-cap issues are fixed independently.

This spec broadens the three anchor sets to natural English paraphrases (ticket families + MADR canonical vocabulary), fixes the rubric f-string bug, adds observability on validation failure, and adds a canonical-floor-pinned parity test so the specific drift direction identified (validator anchors narrower than rubric's instructed vocabulary) cannot recur. The spec does NOT claim to prevent all forms of rubric/validator drift — only the direction that produced the bug.

## Phases

- **Phase 1: Foundational bug fixes** — convert the rubric to an f-string so the word cap actually substitutes (Req 1), and preserve brief text on validation failure so future corpus analysis is not blind (Req 2). These ship as their own atomic edits.
- **Phase 2: Anchor broadening + rubric/validator alignment + structural parity** — broaden the three anchor sets, rewrite the rubric example verbs to use the shared example-token constant, rewrite the retry feedback against the same canonical token source, add the canonical-floor-pinned parity tests. **Phase 2 has a cross-cutting acceptance gate (Req 11) that requires Phase 1's acceptance commands to also pass — enforcing the phase ordering structurally rather than in prose.**

## Requirements

1. **Rubric f-string substitution**: `cortex_command/discovery.py:285-322` (`GATE_BRIEF_RUBRIC`) must render with the numeric word-cap value substituted from `GATE_BRIEF_WORD_CAP`, not the literal token. Acceptance: `python3 -c "from cortex_command.discovery import GATE_BRIEF_RUBRIC; assert '250' in GATE_BRIEF_RUBRIC and 'GATE_BRIEF_WORD_CAP' not in GATE_BRIEF_RUBRIC"` exits 0. **Phase**: Phase 1: Foundational bug fixes

2. **Brief-text observability on validation failure**: `gate_brief_generated` events emitted with `status: "validation_failed"` must include a `brief_excerpt` field containing the first 200 characters of the rejected brief (or fewer, if the brief is shorter). Successful events do not require this field. Acceptance: `grep -c '"event": "gate_brief_generated"' bin/.events-registry.md` ≥ 1 (entry already exists) AND a unit test in `tests/test_discovery_gate_brief.py` (e.g., `test_validation_failed_event_includes_brief_excerpt`) asserts that a `validation_failed` event payload contains a `brief_excerpt` key with a non-empty string value when the brief was non-empty, verified by `python3 -m pytest tests/test_discovery_gate_brief.py::test_validation_failed_event_includes_brief_excerpt -v` exiting 0. **Phase**: Phase 1: Foundational bug fixes

3. **Broadened anchor set — decision**: `validate_brief()` must accept any of these tokens (case-insensitive, word-boundary matched) in the decision anchor: `decide`, `decided`, `decision`, `decisions`, `chose`, `chosen`, `concluded`, `settled`, `selected`, `picked`, `opted`, `agreed`. Acceptance: `python3 -m pytest tests/test_discovery_gate_brief.py::test_validate_brief_decision_anchor_paraphrases -v` exits 0; the test is parametrized over the 12 tokens and asserts `validate_brief(brief)` returns `(True, "")` for a minimal brief containing each token (with valid alternatives + tradeoff anchors) and `(False, reason_containing_decision)` for a brief missing all decision tokens. **Phase**: Phase 2: Anchor broadening + rubric/validator alignment + structural parity

4. **Broadened anchor set — alternatives**: `validate_brief()` must accept any of these tokens (case-insensitive, word-boundary matched) in the alternatives anchor: `alternative`, `alternatives`, `option`, `options`, `considered`, `considerations`, `weighed`, `evaluated`, `rejected`. (Note: `considered` and `considerations` are listed separately — `\bconsidered\b` does NOT match `considerations` under word-boundary regex, so both forms are explicitly enumerated.) Acceptance: `python3 -m pytest tests/test_discovery_gate_brief.py::test_validate_brief_alternatives_anchor_paraphrases -v` exits 0; parametrized as in Req 3 over the 9 alternatives tokens. **Phase**: Phase 2: Anchor broadening + rubric/validator alignment + structural parity

5. **Broadened anchor set — tradeoff**: `validate_brief()` must accept any of these tokens (case-insensitive, word-boundary matched) in the tradeoff anchor: `tradeoff`, `trade-off`, `cost`, `drawback`, `downside`, `sacrifice`, `consequence`, `compromise`, `risk`. Acceptance: `python3 -m pytest tests/test_discovery_gate_brief.py::test_validate_brief_tradeoff_anchor_paraphrases -v` exits 0; parametrized as in Req 3 over the 9 tradeoff tokens. **Phase**: Phase 2: Anchor broadening + rubric/validator alignment + structural parity

6. **Word-boundary matching prevents false positives**: anchor matching must use word-boundary regex (`re.search(r"\bWORD\b", text, re.IGNORECASE)`) rather than naked substring. Acceptance: `python3 -m pytest tests/test_discovery_gate_brief.py::test_validate_brief_word_boundary_false_positives -v` exits 0; the test is parametrized over the substring-FP token list `["optional", "optionality", "committee", "unsettled", "disagreed", "costume", "pickup"]` and asserts that briefs containing only an FP token (and valid prose for the other two anchors) return `(False, reason_naming_the_unmet_anchor)`. **Phase**: Phase 2: Anchor broadening + rubric/validator alignment + structural parity

7. **Shared example-tokens constant**: introduce a module-level constant `_GATE_BRIEF_EXAMPLE_TOKENS: dict[str, tuple[str, ...]]` in `cortex_command/discovery.py` mapping each anchor name (`"decision"`, `"alternatives"`, `"tradeoff"`) to a tuple of example tokens used in the rubric body. `GATE_BRIEF_RUBRIC` must reference these tokens via f-string interpolation (e.g., `f"... use {', '.join(_GATE_BRIEF_EXAMPLE_TOKENS['decision'])} ..."` or equivalent). Acceptance: `python3 -c "from cortex_command.discovery import GATE_BRIEF_RUBRIC, _GATE_BRIEF_EXAMPLE_TOKENS; assert all(tok in GATE_BRIEF_RUBRIC for tokens in _GATE_BRIEF_EXAMPLE_TOKENS.values() for tok in tokens)"` exits 0. **Phase**: Phase 2: Anchor broadening + rubric/validator alignment + structural parity

8. **Canonical-floor-pinned parity test (validator side)**: `tests/test_discovery_gate_brief.py` must include a no-auth unit test (e.g., `test_validate_brief_canonical_floor`) that asserts the validator's accepted vocabulary is a superset of a **frozen literal canonical token set** declared inside the test file itself (not imported from `cortex_command/discovery.py`). The canonical floor is the full enumerated set from Reqs 3-5: 12 decision tokens + 9 alternatives tokens + 9 tradeoff tokens = 30 tokens (frozen at this spec's approval). For every token in the frozen floor, the test constructs a minimal brief containing only that token plus one canonical token from each other anchor and asserts `validate_brief(brief)` returns `(True, "")`. This test fails if any future commit removes a token from the validator's anchor set — closing the regression mode the previous spec draft missed (where shrinking both `_GATE_BRIEF_EXAMPLE_TOKENS` and the validator anchor set in lockstep passed the test green). Acceptance: `python3 -m pytest tests/test_discovery_gate_brief.py::test_validate_brief_canonical_floor -v` exits 0. **Phase**: Phase 2: Anchor broadening + rubric/validator alignment + structural parity

9. **Retry-feedback as bound module constant**: the retry-feedback prose must be extracted from the inline construction at `cortex_command/discovery.py:783-810` into a module-level f-string constant (e.g., `_GATE_BRIEF_RETRY_TEMPLATE`) that interpolates from `_GATE_BRIEF_EXAMPLE_TOKENS` rather than enumerating tokens verbatim in source. The retry-feedback must include every token from `_GATE_BRIEF_EXAMPLE_TOKENS` (each anchor's full example set) and instruct the model to include at least one token from each anchor. Acceptance: `python3 -m pytest tests/test_discovery_gate_brief.py::test_retry_feedback_covers_example_tokens -v` exits 0; the test imports the rendered retry-feedback constant and asserts every token in `_GATE_BRIEF_EXAMPLE_TOKENS` appears in it. (Per Adversarial Review §7: retry feedback is recovery, not teaching — verbatim "use one of these tokens" maximizes recovery rate, and binding it to the shared constant prevents the third-vertex drift critical-review flagged.) **Phase**: Phase 2: Anchor broadening + rubric/validator alignment + structural parity

10. **Validate_brief error messages name the broadened anchors**: when `validate_brief()` returns `(False, reason)` for an anchor-missing failure, the `reason` string must enumerate at least three representative tokens per failed anchor (e.g., `"brief is missing decision anchor (one of: decided, chose, settled, ...)"`). Acceptance: `python3 -m pytest tests/test_discovery_gate_brief.py::test_validate_brief_error_messages_name_broadened_anchors -v` exits 0; the test asserts that for a brief containing only valid alternatives + tradeoff anchors (and no decision token), `validate_brief(brief)[1]` contains the substrings `"chose"` and `"settled"`. (The early-exit empty-brief reason at `discovery.py:553` is preserved unchanged — this requirement applies only to the three anchor-missing reasons.) **Phase**: Phase 2: Anchor broadening + rubric/validator alignment + structural parity

11. **Cross-cutting Phase 2 acceptance gate**: at the close of Phase 2, all Phase 1 acceptance commands (Reqs 1, 2) must still pass — enforcing structural phase ordering. Acceptance: a single `just`-recipe-or-equivalent that runs `python3 -c "from cortex_command.discovery import GATE_BRIEF_RUBRIC; assert '250' in GATE_BRIEF_RUBRIC and 'GATE_BRIEF_WORD_CAP' not in GATE_BRIEF_RUBRIC"` AND `python3 -m pytest tests/test_discovery_gate_brief.py::test_validation_failed_event_includes_brief_excerpt -v` AND every Phase 2 acceptance command (Reqs 3-10) — and exits 0 only if all of them exit 0. Practically: add a pytest marker `--co -q` collection check + a combined invocation `python3 -m pytest tests/test_discovery_gate_brief.py -v -k "validate_brief or retry_feedback or validation_failed or canonical_floor"` exits 0 AND the Req 1 python3 -c check exits 0 — both run together. **Phase**: Phase 2: Anchor broadening + rubric/validator alignment + structural parity

## MoSCoW Classification

- **Must have** (load-bearing for gate to function correctly, not speculative): Reqs 1 (rubric f-string — the literal token reaches the sub-agent, structurally broken), 3-5 (broadened anchor sets — the rubric's own example verb fails the validator without them), 6 (word-boundary FP prevention — without it, broadening introduces noise), 7 (shared example-tokens constant — required mechanism for Req 8's parity test).
- **Should have** (durability scaffolding — predicted-drift prevention, useful but not load-bearing for the immediate failure): Reqs 8 (canonical-floor parity test), 9 (retry-feedback bound to constant), 11 (cross-cutting acceptance gate). These prevent re-regression but the immediate failure is closed without them. Critical-review noted scope-vs-evidence tension on these; classifying them honestly as Should rather than Must.
- **Should have** (operability win): Req 2 (brief-text observability) — future corpus analysis stays blind without it but the gate works either way. Req 10 (error messages name broadened anchors) — operator polish.
- **Won't do** (this ticket): semantic-gate replacement (deferred to #255); dropping/softening the gate; modifying `_run_brief_query`'s permission mode; lemma-stem broadening (FP risk); unskipping auth-gated tests; covering FN paraphrases beyond MADR canonicals (deferred to #255). See Non-Requirements for full enumeration.

## Non-Requirements

- **Semantic / LLM-as-judge gate replacement**: deferred to backlog ticket #255 (gate-policy taxonomy + critical-review gate fixes). #255's spec should reference #256's post-merge corpus to inform whether semantic-gate work is empirically justified.
- **Dropping the gate or softening it to warn-only**: rejected per Tradeoffs §E–F. The six empty-brief failures in the current corpus are real sub-agent failures the gate should still catch.
- **Modifying `_run_brief_query`'s `permission_mode="bypassPermissions"` and `max_turns=3`**: latent surface noted in Adversarial §8; separate hardening ticket.
- **Lemmatized regex word-stems (Alternative C with `\w*` suffixes)**: rejected per Adversarial §3 — false-positive risk on `pickup`, `committee`, `optional` is unacceptable.
- **Re-running the seven existing failed corpus events post-fix to validate pass-through**: useful, but out of scope. The post-merge experiment that would distinguish "Req 1 alone closed the rate" from "Reqs 3-5 were necessary" is explicitly deferred — operators may run it ad-hoc.
- **Unskipping auth-gated tests** (`test_brief_passes_all_fixtures`, `test_brief_failure_falls_back_to_architecture`): separate CI infrastructure question. This spec instead requires that all new tests run without auth.
- **Adding paraphrase tokens beyond MADR canonicals**: paraphrases like `landed on`, `went with`, `on the table`, `accepted` still fail. The FN boundary remains; permanent removal is deferred-semantic-gate work in #255.
- **Perfect rubric example-verb exclusivity**: Req 7 verifies presence (every `_GATE_BRIEF_EXAMPLE_TOKENS` token appears in the rubric) but not exclusivity (the rubric could in principle retain hardcoded narrow example verbs alongside the f-string interpolation). The cost of perfect exclusivity (parse the rubric AST and assert no decision/alternative/tradeoff verbs outside the interpolation point) is judged high relative to the residual risk after the canonical-floor parity test (Req 8) is in place. Documented limitation; revisit if the failure mode is observed.

## Edge Cases

- **`The optional flag was set`**: must NOT satisfy the alternatives anchor (no word-boundary match on `option` inside `optional`). The full brief still fails validation (decision + tradeoff missing). Covered by Req 6.
- **`The committee decided to use X. Alternatives were considered. Tradeoffs are X.`**: must satisfy validation. `decided` matches decision anchor; `Alternatives` matches alternatives anchor (case-insensitive); `Tradeoffs` matches tradeoff anchor. `committee` must NOT be misread as satisfying any anchor via a partial-word match.
- **`We settled on X. Two options were weighed. The compromise was Y.`**: must satisfy validation. All three anchors covered by MADR canonical vocabulary (`settled`, `options`, `weighed`, `compromise`).
- **`Trade-off is severe`** (hyphenated): must satisfy tradeoff anchor. Word-boundary regex `\btrade-off\b` matches because `-` is `\W` in Python's `\b` definition.
- **Empty brief**: validator exits early with `(False, "brief is empty")` per existing behavior at `discovery.py:552-553`. Unchanged.
- **Brief over word cap**: validator returns `(False, "brief word count N exceeds cap K ...")` per existing behavior at `discovery.py:572-578`. Unchanged.
- **`considerations` vs `considered` word-boundary behavior**: confirmed by inspection — `re.search(r'\bconsidered\b', 'considerations are real')` returns `None`. The word boundary at position 9 fails because both characters are `\w`. Resolved by listing both `considered` AND `considerations` separately in Req 4 (and in the test file's canonical-floor frozen set).
- **Plural blind spot (acknowledged in scope)**: tokens like `risks`, `consequences`, `drawbacks`, `costs`, `sacrifices`, `compromises` are NOT in the broadened set. A brief writing "the risks are X" or "the consequences include Y" will fail the tradeoff anchor. This is a known FN boundary. Resolution: deferred to #255 (semantic gate makes inflection irrelevant) OR a follow-up ticket if the empirical post-merge corpus shows this is the dominant failure mode. The pre-existing alternatives anchor's `alternative/alternatives` and `option/options` pairing is preserved (Req 4); the same lemma-pairing discipline is not extended to tradeoff and decision anchors in this ticket because (a) the empirical case for it is the same n=0 corpus, and (b) the resulting set would be 50+ tokens with diminishing FP discipline.

## Changes to Existing Behavior

- **MODIFIED**: `validate_brief()` anchor-rejection rate. Pre-fix: 1/1 paraphrase rejection in the only observable non-empty brief (the 439-word over-cap from harness-friction-triage). Post-fix: paraphrase-rejection should drop for natural-English briefs containing the broadened vocabulary; the 6 empty-brief sub-agent failures remain (correctly) rejected; the word-cap miscommunication is resolved by Req 1 independently of the anchor work.
- **MODIFIED**: `GATE_BRIEF_RUBRIC` is now an f-string (or built via `.format()`) with the numeric word cap substituted at module load. The rubric body itself uses tokens drawn from `_GATE_BRIEF_EXAMPLE_TOKENS` (Req 7).
- **MODIFIED**: `gate_brief_generated` event schema. Failed events now carry an additional `brief_excerpt: str` field (≤200 chars). Successful events unchanged. Readers MUST tolerate absence of this field on legacy events.
- **ADDED**: module-level constants `_GATE_BRIEF_EXAMPLE_TOKENS` and `_GATE_BRIEF_RETRY_TEMPLATE` in `cortex_command/discovery.py`. Source of truth for rubric body and retry-feedback prose.
- **ADDED**: parametrized paraphrase + false-positive unit tests, retry-feedback coverage test, and the canonical-floor-pinned parity test in `tests/test_discovery_gate_brief.py`. All new tests run without auth.

## Technical Constraints

- **Word-boundary regex**: implementation must use `re.search(r"\bTOKEN\b", text, re.IGNORECASE)` per anchor.
- **Hyphen handling**: tokens with `-` (e.g., `trade-off`) match correctly under Python's `\b`.
- **F-string conversion ordering**: `GATE_BRIEF_RUBRIC` must be defined after `GATE_BRIEF_WORD_CAP` and after `_GATE_BRIEF_EXAMPLE_TOKENS` so all referenced names exist at f-string evaluation. `_GATE_BRIEF_RETRY_TEMPLATE` must be defined after `_GATE_BRIEF_EXAMPLE_TOKENS` for the same reason.
- **Event schema backward compatibility**: the new `brief_excerpt` field is optional. Readers MUST tolerate absence on legacy events — same as the `parent_epic_loaded` precedent in clarify-critic events.
- **Parity-test asymmetry made explicit**: Req 7's acceptance is presence-only ("every token in `_GATE_BRIEF_EXAMPLE_TOKENS` appears in the rubric"); Req 8's parity test pins the canonical floor on the **validator side** (the validator must accept every token in the frozen literal canonical set declared in the test file). Together these mean: (a) the rubric examples cannot drift narrower than `_GATE_BRIEF_EXAMPLE_TOKENS` (Req 7); (b) the validator cannot drift narrower than the frozen canonical floor (Req 8). The remaining drift direction — `_GATE_BRIEF_EXAMPLE_TOKENS` itself shrinking — is bounded by the canonical floor's literal-set fix in the test file and would still cause Req 8 to fail if the validator shrank with it. Acknowledged limitation: if both `_GATE_BRIEF_EXAMPLE_TOKENS` and the validator's anchor sets shrink while the frozen canonical floor in the test file stays at 30 tokens, Req 8 fails and the regression is caught. If a future PR ALSO modifies the canonical floor in the test file (atomic three-way shrinkage), the test passes — but that is now a reviewable structural change to the test file itself, not silent drift.
- **Test-side constraints**: new tests must not import auth-gated fixtures or call live Sonnet dispatches. They construct minimal brief strings inline and run `validate_brief()` directly.
- **CI gate**: all new tests must run in `just test` without requiring `_REQUIRES_AUTH`.
- **No third-vertex drift in retry feedback**: Req 9 binds retry feedback to `_GATE_BRIEF_EXAMPLE_TOKENS` via interpolation, closing the inline-OR-constant hedge the previous draft permitted.

## Proposed ADR

None considered. The three-criteria ADR gate from `cortex/adr/README.md` requires Hard-to-reverse + Surprising-without-context + Real-trade-off (all three). This work fails the gate on at least two criteria:
- **Not hard to reverse**: anchor sets are pure data; tightening them later is a one-line edit.
- **Not surprising without context**: the structural fix (validator-side canonical-floor parity test, shared example-tokens constant) follows the existing `prefer structural separation over prose-only enforcement` principle in `cortex/requirements/project.md`.

The real-trade-off dimension is genuinely present (broaden-now vs. defer-to-semantic-gate-in-#255), but with two of three criteria failing, the gate's strict-AND rule routes the decision to ticket-body documentation rather than an ADR. The deferral rationale is captured in Non-Requirements above.
