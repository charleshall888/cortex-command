# Plan: fix-validate-brief-substring-anchors-that

## Overview

Two-phase implementation that ships the foundational fixes (rubric f-string substitution; brief-text observability on validation failure) atomically, then layers the anchor broadening, the shared-constant refactor, the retry-feedback rebinding, and the structural parity test on top. Phase ordering is structurally enforced via dependency annotations (file-conflict-aware): Phase 1 tasks serialize on `cortex_command/discovery.py`; Phase 2 anchor-and-rubric tasks chain explicitly through Task 4. Task 8's cross-cutting `just` recipe (Req 11) re-runs every Phase 1 and Phase 2 acceptance command in a single invocation; it is an audit/regression gate, not a sequencer — the dependency graph does the sequencing.

`_GATE_BRIEF_EXAMPLE_TOKENS` (Task 3) is the single source of truth for **what the sub-agent sees** (rubric body + retry feedback), not for the validator's accepted vocabulary. The validator's accepted vocabulary is independently pinned by Task 7's frozen literal canonical floor — this asymmetry is deliberate per spec Req 8 and spec §Technical Constraints' "Acknowledged limitation," and is the regression guard against lockstep shrinkage of `_GATE_BRIEF_EXAMPLE_TOKENS` and the validator anchor sets.

## Outline

### Phase 1: Foundational bug fixes (tasks: 1, 2)
**Goal**: Fix the two pre-existing structural bugs that are orthogonal to anchor breadth but materially distort the gate's observable behavior: the rubric's literal `GATE_BRIEF_WORD_CAP` token never substitutes (Req 1), and `gate_brief_generated` failure events drop the brief text needed for any future corpus analysis (Req 2).
**Checkpoint**: `GATE_BRIEF_RUBRIC` renders with the numeric `250` substituted; failure events carry `brief_excerpt`; both Phase 1 acceptance commands exit 0.

### Phase 2: Anchor broadening + rubric/validator alignment + structural parity (tasks: 3, 4, 5, 6, 7, 8, 9)
**Goal**: Broaden the three validator anchors to the canonical floor with word-boundary regex (Reqs 3–6), introduce `_GATE_BRIEF_EXAMPLE_TOKENS` as the single source of truth for **agent-facing prose** (rubric examples + retry feedback) per Reqs 7 and 9, update error messages to enumerate broadened anchors (Req 10), and pin the validator's accepted vocabulary independently against drift via the frozen literal floor in the test file (Req 8). The phase closes behind the cross-cutting acceptance gate (Req 11).
**Checkpoint**: All Phase 1 acceptance commands AND every Phase 2 acceptance command exit 0 under a single invocation.

## Risks

- **Word-boundary regex semantics on hyphenated tokens**: Python's `re.search(r"\btrade-off\b", ...)` matches because `-` is `\W`, but the regex anchor at the leading `\b` is at position 0 (where the leading `t` of `trade` borders a word boundary). The edge-case test for `Trade-off is severe` is in spec Req 6's FP test set. Implementer verifies during Task 4 that both `tradeoff` and `trade-off` match correctly.
- **`considered` vs `considerations` word-boundary asymmetry**: Spec Req 4 lists both separately because `\bconsidered\b` does NOT match `considerations` (the boundary at position 9 fails — both chars are `\w`). The canonical floor in Task 7 must include BOTH; missing either causes Task 7's parity test to fail on the missing token.
- **Retry-feedback template form (constant vs. function)**: Task 6 allows either a `.format()`-template constant or a small renderer function. Both forms satisfy Req 9 in spirit. Risk if the constant form is chosen: the `{reason}` placeholder must NOT be interpolated at module load (since `reason` is dynamic per dispatch). The correct pattern is `template = "...{reason}..."; rendered = template.format(reason=...)` — NOT `f"...{reason}..."` at module scope.
- **Line-number drift across tasks editing `discovery.py`**: Task 4 cites line numbers from the current `cortex_command/discovery.py`; Task 5, 6, 9 cite "after Task N lands" semantic anchors. Implementers must grep on the semantic anchor (function name, distinctive string literal, code pattern) rather than relying on the cited line number when the source file has already been edited by an upstream task. The pre-edit line numbers in this plan are starting hints, not contracts.
- **Agent-facing / validator-accepted vocabulary asymmetry**: `_GATE_BRIEF_EXAMPLE_TOKENS` exposes a representative subset to the rubric and retry feedback; the validator accepts the full 30-token canonical floor. This is the spec's intentional design (Req 8 + §Technical Constraints' "Acknowledged limitation") — the regression guard against lockstep shrinkage is precisely Task 7's frozen literal in the test file, which is independent of `_GATE_BRIEF_EXAMPLE_TOKENS`. Implementers must not "fix" this by binding the constant to the validator's full canonical floor — that would defeat the regression guard.
- **Test coverage gap acknowledged**: Auth-gated fixture tests (`test_brief_passes_all_fixtures`, `test_brief_failure_falls_back_to_architecture`) remain skipped in CI per spec Non-Requirements. Live SDK pass-through verification is out of scope for this lifecycle.
