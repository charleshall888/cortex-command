# Specification: gate-worktree-option-on-console-script

## Problem Statement

The lifecycle implement-phase "Implement on feature branch with worktree" menu option silently disappears in consumer repos installed via the documented `uv tool install` path, because the current runtime probe gates on bare-Python `importlib.util.find_spec('cortex_command')` — which fails outside the source tree even when the `cortex-*` console-scripts on the user's PATH work fine. Replace the proxy signal with a console-script reachability check, add a new `cortex-worktree-create` binstub that the gated downstream call invokes (so gate and gated path agree on what "available" means), and add a class-level lint rule preventing bare-Python `cortex_command` imports in skill files so this regression class is structurally prevented going forward.

## Phases

- **Phase 1: Gate fix** — Replace the bare-Python probe with a console-script reachability check, add `cortex-worktree-create` as the new gate-binary and §1a step iii's invocation surface, extend the structural-marker contract test to verify the gate↔gated-path pairing.
- **Phase 2: Lint rule** — Add a class-level lint module that catches bare-Python `cortex_command` imports in skill files, wire it into pre-commit alongside the existing lint chain.

## Requirements

1. **New `cortex-worktree-create` console-script entry with explicit IO and idempotency contract**: `pyproject.toml`'s `[project.scripts]` table includes a `cortex-worktree-create` entry pointing at a new wrapper module (`cortex_command/pipeline/worktree_create_cli.py`). The wrapper accepts `--feature <name>` and `--base-branch <branch>` arguments, calls `create_worktree(feature=<name>, base_branch=<branch>)`, and observes the following IO contract:
   - **stdout**: exactly the absolute worktree path followed by a single `\n`, with nothing else. The wrapper redirects `create_worktree`'s subprocess stdout (e.g., `git worktree add` chatter) to stderr to prevent leakage.
   - **stderr**: all informational output (git subprocess output, `.venv symlinked`, exception traces).
   - **Idempotency**: when the worktree already exists and is valid, `create_worktree` returns the existing `WorktreeInfo`; the wrapper prints its path to stdout, writes `worktree already exists` to stderr, and exits 0. Re-entry is success, not error.
   - **Exception handling**: on any exception raised by `create_worktree` (`ValueError`, `subprocess.CalledProcessError`, `OSError`, etc.), the wrapper writes `repr(exc)` (preserving exception type name and message) to stderr and exits 1. The wrapper does NOT differentiate exception classes via exit code — the stderr trace is the differentiator. This preserves R3's "surface the error to the user" prose from the prior fenced-Python form.

   **Acceptance**: `grep -c '^cortex-worktree-create = ' pyproject.toml` = 1; `command -v cortex-worktree-create` exits 0 after a fresh `uv tool install --reinstall`; `cortex-worktree-create --feature test-fixture --base-branch main` materializes a worktree at `<repo>/.claude/worktrees/test-fixture/`, prints exactly the path to stdout, exits 0; a second invocation with the same arguments prints the same path to stdout, prints `worktree already exists` to stderr, exits 0; invoking with a non-existent base-branch produces a `repr(exc)`-shaped error on stderr and exit 1. **Phase**: Phase 1.

2. **Gate probe swaps from bare-Python import to console-script reachability with honest disposition contract**: `skills/lifecycle/references/implement.md` §1 (currently lines 55–66) replaces the `python3 -c "...importlib.util.find_spec('cortex_command')..."` block with a single Bash call: `command -v cortex-worktree-create >/dev/null 2>&1`. The three-disposition contract is preserved but expressed honestly against the new primitive:
   - **exit 0** → the binary is reachable on PATH → all three menu options remain: `Implement on current branch`, `Implement on feature branch with worktree`, `Create feature branch`.
   - **exit 1** → the binary is not on PATH → remove `Implement on feature branch with worktree` from the options array; silent hide, no diagnostic.
   - **Bash tool execution failure (sandbox rejection, missing /bin/sh, shell unavailable) OR `command -v` exit code other than 0 or 1** → fail open: all three options remain, surface the literal diagnostic `runtime probe skipped: console-script probe failed`. This branch's reachability depends on the Bash tool's own failure modes rather than `command -v`'s exit code — preserving the precedent's fail-open posture without misrepresenting the primitive.

   **Acceptance**: `grep -c 'importlib.util.find_spec' skills/lifecycle/references/implement.md` = 0; `grep -c 'command -v cortex-worktree-create' skills/lifecycle/references/implement.md` ≥ 1; the §1 routing prose enumerates all three dispositions verbatim (exit 0, exit 1, tool failure / other-exit). **Phase**: Phase 1.

3. **§1a step iii invokes `cortex-worktree-create` rather than fenced Python**: `skills/lifecycle/references/implement.md` §1a step iii (currently lines 123–128) replaces the fenced ```python block (`from cortex_command.pipeline.worktree import create_worktree` followed by the `create_worktree(...)` call) with a single Bash call to `cortex-worktree-create --feature interactive-{slug} --base-branch main`. The surrounding prose preserves the existing semantics (interactive- prefix → branch `interactive/{slug}`, worktree at `<repo>/.claude/worktrees/interactive-{slug}/`, `.claude/settings.local.json` copy, `.venv` symlink — all delegated to `create_worktree`'s implementation) and the existing failure-mode prose ("If creation fails: surface the error to the user and exit §1a") — under the new IO contract, the error surfaced is the wrapper's stderr (which contains `repr(exc)`) plus exit code. The skill captures stdout as the resolved worktree path via standard `$(...)` substitution per R1's stdout contract. **Acceptance**: `grep -c 'from cortex_command.pipeline.worktree import create_worktree' skills/lifecycle/references/implement.md` = 0; `grep -c 'cortex-worktree-create --feature' skills/lifecycle/references/implement.md` ≥ 1; the §1a step iii prose explicitly notes that the wrapper's stderr (`repr(exc)`) is the surfaced error. **Phase**: Phase 1.

4. **Structural-marker contract test verifies gate↔gated-path binary pairing, not just gate presence**: `tests/test_implement_worktree_interactive_contract.py` adds a new test function `test_gate_and_gated_path_use_same_binary` that:
   - extracts the binary name from §1's `command -v <name>` directive (regex `command -v (\S+)`),
   - extracts the binary name from §1a step iii's `<name> --feature ...` Bash directive (regex `^(\S+)\s+--feature\s+interactive-` in §1a step iii context),
   - asserts both regexes match at least once each AND the two captured binary names are identical.

   This is the class-level structural guard against the regression mode the spec is fixing (literal-substring drift breaking gate↔gated agreement) — equivalent to R5's class-level lint for the sibling problem. **Acceptance**: `grep -c "test_gate_and_gated_path_use_same_binary" tests/test_implement_worktree_interactive_contract.py` ≥ 1; `pytest tests/test_implement_worktree_interactive_contract.py::test_gate_and_gated_path_use_same_binary -v` exits 0; a fixture mutation that renames §1a step iii's binary while leaving §1 untouched causes the test to fail with a clear identity-mismatch message. **Phase**: Phase 1.

5. **New lint module `cortex_command/lint/bare_python_import.py` with explicit python-source-region definition**: A new sibling lint module that scans `skills/**/*.md` for bare-Python `cortex_command` imports. The module exports a `main()` entry point (registered as `cortex-check-bare-python-import` in `[project.scripts]`) supporting `--staged` (git's staged blob view) and `--audit` (working-tree-wide), mirroring the two-mode pattern of `cortex-check-contract` and `cortex-check-events-registry`.

   **Python-source-region definition** (replaces the prior under-specified "python-body region" framing — the precedent `contract.py` fence machine is language-agnostic and does not categorize fences by info-string; this definition makes the scope explicit):
   - **Rule 1 (labeled fences)**: The body of any fenced code block whose info-string matches the regex `^(python|python3|py)$` case-insensitively is a python-source region.
   - **Rule 2 (`python3 -c` invocations)**: The single-line or multi-line quoted argument body of any `python3 -c "<body>"` (or `python -c`) invocation, appearing in ANY fenced code block (regardless of info-string) OR in inline prose, is a python-source region. The scanner tokenizes shell-quoted strings and scans the unquoted body.
   - **Rule 3 (heredocs)**: The body of any `python3 <<MARKER ... MARKER` or `python3 - <<MARKER ... MARKER` heredoc, in ANY containing context, is a python-source region.
   - **Rule 4 (unlabeled fences)**: Unlabeled fences are NOT themselves python-source regions, but Rule 2 and Rule 3 invocations appearing inside them ARE scanned.

   The scanner extracts python-source regions per these four rules and applies the regex `\b(?:import\s+cortex_command|from\s+cortex_command(?:\.[a-z_][a-z0-9_]*)*\s+import)\b` to those regions only.

   **Negative cases (must NOT flag)**: commentary mentioning `cortex_command` in prose; stdlib-only `python3 -c` (e.g., `python3 -c "import json,sys; ..."` in `_interactive_overnight_check.sh` — the regex's `cortex_command`-scoping handles this); ```python fences in `## Touch points` sections that quote bare-Python code for documentation purposes are flagged unless the sentinel below precedes them (intentional design: Touch points authors must explicitly mark illustrative quotes; the project's convention against bare-Python skill invocations is uniform).

   **Sentinel suppression**: A line `<!-- bare-python-lint:ignore-next -->` immediately preceding (no blank line between) the python-source region suppresses the next finding. To accommodate standard markdown formatters (markdownlint MD012/MD031) that enforce a blank line between HTML comments and fenced blocks, the scanner uses the `prev_nonblank` pattern from `cortex_command/lint/contract.py:553` — sentinels are recognized when they are the most recent non-blank line preceding the region (i.e., intervening blank lines do not defeat the sentinel). One sentinel suppresses one finding; stacked sentinels suppress multiple.

   **Acceptance**: `test -f cortex_command/lint/bare_python_import.py`; `grep -c '^cortex-check-bare-python-import = ' pyproject.toml` = 1; the test suite (R7) covers each rule (1–4) and each negative case with positive and negative fixtures; running against the live `skills/` tree exits 0 after Phase 1's §1a:126 fix lands. **Phase**: Phase 2.

6. **Pre-commit wiring**: `.githooks/pre-commit` adds a new stanza after the existing `prescriptive_prose_triggered` block that triggers on the same file patterns (`skills/*|cortex/backlog/*.md|bin/cortex-check-bare-python-import`) and invokes `just check-bare-python-import` against the staged blob view. Failure prints the diagnostic `pre-commit: bare-python cortex_command import check failed — convert to a cortex-* console-script call, or add the <!-- bare-python-lint:ignore-next --> sentinel if the bare-Python form is intentional.` and exits 1. The `justfile` gains a `check-bare-python-import` recipe wrapping `cortex-check-bare-python-import --staged` (and a paired `check-bare-python-import-audit` recipe for the `--audit` mode, per the two-mode gate pattern in project.md). **Acceptance**: `grep -c 'check-bare-python-import' .githooks/pre-commit` ≥ 1; `grep -c '^check-bare-python-import' justfile` ≥ 1; staging a test fixture containing `python3 -c "import cortex_command"` and running `.githooks/pre-commit` exits non-zero with the expected diagnostic. **Phase**: Phase 2.

7. **Lint test coverage covering each definitional rule and negative case**: A new test file `tests/test_bare_python_import_lint.py` exercises the lint against positive fixtures covering each detection rule (R5's Rules 1–4: labeled fences, `python3 -c` invocations, heredocs, Rule 2/3 inside unlabeled fences) and negative fixtures (stdlib-only `python3 -c`, prose commentary mentioning `cortex_command`, sentinel-suppressed, sentinel-with-intervening-blank-line per R5's `prev_nonblank` accommodation). Fixtures live under `tests/fixtures/bare_python_import/`. **Acceptance**: `test -f tests/test_bare_python_import_lint.py`; `pytest tests/test_bare_python_import_lint.py -v` exits 0; the test asserts ≥4 positive cases pass (one per Rule 1–4) and ≥4 negative cases pass (stdlib-only, prose-only, sentinel-immediate, sentinel-with-blank-line). **Phase**: Phase 2.

8. **Events-registry / contract-lint registration**: `bin/.events-registry.md` is not affected (the new console-scripts do not emit events). If `cortex_command/lint/bare_python_import.py`'s `main()` uses standard `argparse.ArgumentParser` + `.add_argument` calls, no `bin/.contract-lint-exceptions.md` entry is required; otherwise, an entry is added under category `non-argparse-module` with a one-sentence rationale. The new `cortex-worktree-create` wrapper similarly follows standard argparse and requires no exception. **Acceptance**: `cortex-check-contract --audit` exits 0 after Phase 1+2 land. **Phase**: Phase 2.

## Non-Requirements

- **Not relaxing the gate to always-on.** The "no runner installed" failure mode remains a real degrade-target. `command -v cortex-worktree-create` returning non-zero must still hide the worktree option.
- **Not changing the project's install topology.** This work does not touch the non-editable wheel + uv-tool isolation stance, plugin distribution, or any ADR-0002 surface.
- **Not flagging `python3 -m cortex_command.<module>` invocations in skill prose.** Those work under `uv tool install` and do not reproduce the disappearing-option failure. Project.md retains `python3 -m cortex_command.<module>` as "a readable fallback for ad-hoc invocation."
- **Not retiring `cortex-worktree-resolve`.** The two binaries wrap distinct primitives (`resolve_worktree_root` is read-only path computation; `create_worktree` is side-effecting materialization). 10+ production callsites consume `cortex-worktree-resolve` for its read-only role.
- **No telemetry on the gate probe.** Preserving the archived spec's stance — no `runtime_probe` event, no metrics, no log line per probe. The fail-open diagnostic and the wrapper's `repr(exc)` stderr trace are the only user-facing signals.
- **No `--check` flag on `cortex-worktree-create`.** Gate uses `command -v` alone; the residual false-positive class (wrapper on PATH but venv broken) is accepted per archived precedent.
- **No transition shim for users upgrading the plugin before the wheel.** The transition cohort (recent-but-pre-#266 wheel + new plugin) sees silent hide. Accepted per upgrade-velocity assumption (small consumer base, plugin and wheel typically upgrade in the same maintenance pass).
- **Not bringing back removed autonomous-daytime pipeline worktree mode.** The option this ticket gates is the **interactive** Variant A flow — runs in the user's foreground Claude Code session. The overnight orchestrator calls `create_worktree()` as a Python function (`cortex_command/overnight/orchestrator.py:340`), not through any CLI wrapper; adding `cortex-worktree-create` does not widen the overnight surface.

## Edge Cases

- **Wrapper on PATH but venv broken (R1 idempotency unaffected)**: `command -v` exits 0 → menu offers the option → user selects → `cortex-worktree-create` crashes inside `create_worktree`. R1's exception-handling contract captures `repr(exc)` to stderr and exits 1. §1a step iii's preserved prose surfaces the stderr trace per R3. No mid-creation partial-state masking is added — accepted residual class per archived precedent.
- **Transition cohort (recent wheel without `cortex-worktree-create` entry)**: Older `cortex-command` wheel installed via `uv tool install` may not have the new entry. `command -v` exits non-zero → silent hide → same behavior as the "module absent" branch the current probe produces. Users on stale wheels see no worktree option but no diagnostic; running `uv tool install --reinstall` restores it. Per the upgrade-velocity assumption above, no shim is shipped.
- **Wrapper re-entered with same `--feature` value**: Per R1's idempotency contract, the wrapper prints the existing worktree path to stdout, prints `worktree already exists` to stderr, and exits 0. The §1a step iii caller is unaware whether the worktree was newly-created or already-existed; both are success cases for the lifecycle.
- **Sentinel-with-intervening-blank-line (R5 markdownlint accommodation)**: Markdown formatters (prettier, markdownlint with MD012/MD031) may insert a blank line between `<!-- bare-python-lint:ignore-next -->` and the following fenced block. Per R5's `prev_nonblank` rule, the sentinel is still recognized as long as no intervening non-blank content appears between sentinel and region. This diverges from a strict "immediately preceding" rule and aligns with `contract.py:553`'s established precedent.
- **Multi-line `python3 -c` with leading whitespace**: A heredoc-fed or quoted multi-line `python3 -c "\n    import cortex_command\n..."` is matched correctly because the regex anchors on word boundaries (`\b`), not on line-start. Indentation does not defeat detection.
- **Unlabeled fence containing `python3 -c` invocation**: Per R5's Rule 4, the unlabeled fence is NOT itself a python-source region, but the `python3 -c` invocation inside it IS scanned per Rule 2. Existing `implement.md:57-66` (the current probe in an unlabeled fence) and `:117-119` (bash invocation with sidecar) are handled correctly: after Phase 1 lands, the bare-Python `cortex_command` import is gone, so the lint sees no violation; before Phase 1, the lint would correctly flag `find_spec('cortex_command')` if it were on a `cortex_command`-import line (it is not — `find_spec('cortex_command')` is a function CALL with a string literal, not an `import` statement, so the regex correctly does not flag it).
- **Touch-points illustration containing bare-Python import**: A skill author quoting `from cortex_command.X import Y` inside `## Touch points` for documentation must precede the python-source region with the sentinel. The lint flags otherwise, by design — the project's convention against bare-Python skill invocations is enforced uniformly.

## Technical Constraints

- **Wheel-binstub vs working-tree invocation** (`cortex/requirements/project.md`): The gate runs at lifecycle-skill invocation time and reads the binstub from PATH — executing against the installed wheel's `site-packages/`. Dogfooders iterating on working-tree code without `--reinstall` should export `CORTEX_COMMAND_FORCE_SOURCE=1` (the dual-channel wrapper convention) so the new `cortex-worktree-create` shim runs the working-tree module. The new wrapper follows the same dual-channel pattern as `bin/cortex-check-contract` and `bin/cortex-check-prescriptive-prose`.
- **SKILL.md-to-bin parity enforcement** (`cortex/requirements/project.md`): The new `cortex-worktree-create` and `cortex-check-bare-python-import` binstubs wire through an in-scope SKILL.md / requirements / docs / hooks / justfile / tests reference. `cortex-worktree-create` is referenced by `implement.md` §1 and §1a step iii. `cortex-check-bare-python-import` is referenced by `.githooks/pre-commit` and `justfile`.
- **Two-mode gate pattern** (`cortex/requirements/project.md` Optional): The lint exposes both `--staged` (pre-commit) and `--audit` (time/repo-wide via `just check-bare-python-import-audit`), matching `bin/cortex-check-events-registry`.
- **Defense-in-depth for permissions** (`cortex/requirements/project.md`): `command -v` is a POSIX-mandated shell builtin; no new external process is invoked at gate-probe time beyond the shell's own dispatch. No new sandbox / settings.json work required.
- **CLI/plugin version contract** (ADR-0002): The new console-scripts ship inside the non-editable wheel. Consumer repos pick them up on the next `uv tool install --reinstall`.
- **Existing parity-test scope** (`tests/test_implement_worktree_interactive_contract.py`): The three pre-existing tests cover (a) menu-label presence, (b) `cortex-interactive-lock acquire` substring, (c) `_interactive_overnight_check.sh` reference count ≥2. None test the gate-probe shape or pairing. R4 adds the pairing test as a net-new structural-marker assertion; the existing tests continue to pass trivially since none assert anything about the probe text.
