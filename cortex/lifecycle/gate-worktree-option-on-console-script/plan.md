# Plan: gate-worktree-option-on-console-script

## Overview

Replace the bare-Python `find_spec('cortex_command')` probe at `implement.md` §1 with a `command -v cortex-worktree-create` reachability check, and rewrite §1a step iii to invoke the same `cortex-worktree-create` console-script — so gate and gated path resolve via the same surface under the documented `uv tool install` topology. The new check is PATH-reachability, not invocation-viability — residual false-positives (broken venv, missing base-branch, partial install) still surface as `repr(exc)` on stderr at §1a step iii rather than being caught at gate-time. Deploy a thin `cortex_command/pipeline/worktree_create_cli.py` wrapper that observes a strict stdout-only / stderr-error / repr(exc)-on-failure IO contract and preserves `create_worktree`'s native idempotency. Add a lint (`cortex_command/lint/bare_python_import.py`) that scans `skills/**/*.md` for the documented bare-Python `cortex_command` invocation shapes (labeled `python|python3|py` fences, `python3 -c` invocations, `python3 <<MARKER` heredocs, Rule 2/3 inside unlabeled fences) covering both static `import`/`from import` and the dynamic-import family (`importlib.util.find_spec`, `importlib.import_module`, `__import__`) — the shape this lifecycle is removing. Out-of-scope coverage (alternate python binaries `python`/`python3.N`/`uv run python`, `python3 -m cortex_command` module dispatch, command-substitution-wrapped heredocs, 4-space indent code blocks, bash-wrapper invocations) is enumerated as a documented known-coverage gap rather than silently uncovered. Sentinel `<!-- bare-python-lint:ignore-next -->` follows the `prev_nonblank` single-sentinel-per-region rule from `contract.py:553`. Wire the lint into pre-commit alongside `cortex-check-prescriptive-prose`.

- **`cortex-worktree-resolve` precedent omits telemetry** — Task 1 may inherit that omission for `cortex-worktree-create`, which leaves the new wrapper off the invocation-report dashboard. Trade-off: matching the resolve_cli precedent is structurally consistent, but explicit telemetry is the wider repo convention (per R22 from the precedent lifecycle). Recommended: add the telemetry call to be consistent with the wider convention, even though the sibling resolve_cli omits it. If reviewer disagrees, drop in favor of sibling-parity.
- **Subprocess stdout leakage in `create_worktree`** — if `git worktree add` writes to the wrapper's stdout (rather than being captured by `subprocess.run(capture_output=True)`), the spec R1 stdout contract is violated without an explicit `contextlib.redirect_stdout` wrapper. The plan assumes `create_worktree` uses `capture_output=True` throughout but does not pin verification of this until Task 1's verification cycle. Mitigation: Task 1's verification command set includes "prints exactly the absolute path followed by `\n` to stdout" — this catches leakage immediately.
- **Documented known coverage gaps in the lint** — the four R5 rules cover `python3` literal invocation and the `import`/`from import` plus dynamic-import (`find_spec`, `import_module`, `__import__`) regex shapes. Out of scope without enumeration in this lifecycle: alternate python binaries (`python`, `python3.11`, `uv run python -c`, `.venv/bin/python`), `python3 -m cortex_command` module-dispatch, command-substitution-wrapped heredocs (`$(python3 <<EOF...)`), env-preamble heredocs (`PYTHONPATH=. python3 <<EOF...`), 4-space-indent code blocks, bash-wrapper invocations (`bash -c "python3 -c '...'"`). These shapes are deliberately deferred — the regression mode this lifecycle is preventing is `find_spec`/`import`/`from import` against `cortex_command` inside `python3 -c`/heredoc/```python fences. A future regression in an out-of-scope shape would require extending the rule set; treat as a known coverage limit rather than a regression of this lint.
- **`tests/fixtures/bare_python_import/` self-trigger risk** — the lint scans `tests/**/*.md` by default. Without an explicit hard exclusion for `tests/fixtures/bare_python_import/`, the positive fixtures would self-flag during live-tree audit. Plan adds the exclusion; verify the exclusion does not also accidentally exclude `tests/test_bare_python_import_lint.py` (which only mentions `cortex_command` in import statements for the test code itself — those are Python files, not `.md`, so the `**/*.md` glob excludes them anyway).
- **Plugin mirror regeneration drift** — Task 1 edits `skills/lifecycle/references/implement.md`. The four-phase pre-commit hook runs `just build-plugin` and refuses divergence. If the build is non-deterministic (extremely unlikely but worth flagging), Task 1's commit may fail repeatedly. Mitigation: if drift surfaces, halt Task 1 and investigate the build target rather than `--no-verify`-ing past it.
- **Wheel-reinstall ordering** — Task 1 adds the `[project.scripts]` entry but the editable install picks up new console-scripts only after `uv tool install -e . --reinstall`. If the reinstall step is forgotten between Task 1 and Task 2's verification, the gate↔gated pairing test still passes (it only reads `implement.md` text — no subprocess invocation), but the actual gate evaluation against PATH fails because `command -v cortex-worktree-create` returns non-zero. This is an *operational* risk, not a test-suite-visible failure. Mitigation: Task 1's verification explicitly includes the wheel-reinstall step plus a `command -v` check.

## Acceptance

The "Implement on feature branch with worktree" option appears in the §1 implement-phase menu under the documented `uv tool install` consumer topology (where `command -v cortex-worktree-create` exits 0), and is silently hidden when the wrapper is not on PATH (e.g., transition cohort with a stale wheel). Selecting the option dispatches `cortex-worktree-create --feature interactive-{slug} --base-branch main` and proceeds through §1a as before. Residual false-positives — wrapper on PATH but invocation fails (broken venv, non-`main` default branch, generic `git worktree add` failure) — surface as `repr(exc)` on stderr at §1a step iii rather than being caught at gate-time; this is by design (PATH-reachability is the gate's scope, not invocation-viability). The lint catches the in-scope bare-Python `cortex_command` import shapes in `skills/**/*.md` — labeled `python|python3|py` fences, `python3 -c` invocations, `python3 <<MARKER` heredocs, Rule 2/3 inside unlabeled fences — covering both static `import`/`from import` AND the dynamic-import family (`importlib.util.find_spec`, `importlib.import_module`, `__import__`). A verbatim regression of the §1 probe being removed (`importlib.util.find_spec('cortex_command')`) is structurally prevented by `test_find_spec_regression_caught`. Known coverage gaps (alt python binaries, `python3 -m` dispatch, command-substitution-wrapped heredocs, 4-space-indent code blocks, bash-wrapper invocations) are enumerated in the Risks section as deliberate scope decisions, not silent gaps. The sentinel `<!-- bare-python-lint:ignore-next -->` follows single-sentinel-per-region semantics (N sentinels suppress N consecutive regions in order; one sentinel suppresses exactly one region). All four pre-existing structural-contract tests plus the new pairing test plus the bare-python-import lint suite pass under `pytest`.
