# Specification: wire-daytime-dispatch-through-cortex-cli

## Problem Statement

Daytime dispatches initiated from inside a Claude session fail with `EPERM: mkdir ~/.claude/session-env/<child-uuid>/` because the spawned child Claude SDK process inherits the calling session's Seatbelt sandbox. Empirical smoke tests (research.md Empirical Smoke Tests §1, §2) confirmed that `subprocess.Popen(start_new_session=True)` alone does NOT escape Seatbelt inheritance. Overnight works from Claude sessions today because of two harness behaviors orthogonal to process detachment: (a) the `/overnight` skill invokes Bash with `dangerouslyDisableSandbox: true`, and (b) MCP servers run unsandboxed at hook trust level. Daytime lacks both surfaces today — it has zero MCP tools and no Claude-session-friendly CLI verb. Symptom evidence: `cortex/lifecycle/lead-refine-4-complexity-value-gate/exit-reports/{1,2,3,4}.json`. This work adds the missing surfaces — a `cortex daytime start` CLI verb and a separate `cortex-daytime` MCP plugin exposing 4 tools — so daytime dispatches reach the same harness-escape paths that overnight already uses. The direct `cortex-daytime-pipeline` console-script entry point is preserved as the fresh-terminal regression guard.

This spec delivers Alternative B (Popen + start_new_session + documented harness dependencies). The ticket's "with launchd detachment" framing was research-falsified — overnight's run-now path does not use launchctl. Real launchctl bootstrap (Alternative A) remains the durable upgrade path if Anthropic restricts either harness behavior; it is documented as such in R14 but not delivered here.

## Phases

- **Phase 1: Foundation hardening** — PID schema upgrade, install-guard awareness, input validation. Prerequisites that stand alone as security/correctness fixes. **Scope question for §4 approval**: whether to split Phase 1 into a separate preceding ticket.
- **Phase 2: CLI + MCP surfaces** — `cortex daytime` CLI verb and `plugins/cortex-daytime/` MCP plugin. The new entry points.
- **Phase 3: Documentation, tests, validation** — `docs/daytime-operations.md`, integration tests, from-Claude-session smoke test (release-gate required, not OR), MCP contract updates.

## Requirements

### Phase 1: Foundation hardening

1. **`daytime.pid` schema upgrade in `daytime_pipeline.py`**: The file at `cortex/lifecycle/<feature>/daytime.pid` becomes JSON-shaped `{"schema_version": 1, "magic": "cortex-daytime-v1", "pid": <int>, "pgid": <int>, "start_time": <float, from psutil.Process(os.getpid()).create_time()>, "session_id": "<str>", "feature": "<str>", "repo_path": "<str>"}`, mode 0o600, atomic O_EXCL claim. **`start_time` is captured via `psutil.Process(os.getpid()).create_time()` — matching the verification-time metric in `cortex_command/overnight/ipc.py:392-403`**, so the ±2s comparison in R2 is between two psutil readings of the same kernel-provided value (not a disk-pressure-sensitive skew). The change touches `_write_pid` (`daytime_pipeline.py:107-113`), `_read_pid`, `_is_alive` (`:86-104`), and `_recover_stale` (`:125-168`). **Clean-shutdown clears the file**: `run_daytime`'s normal exit path and `_orphan_guard` both call `_clear_pid` (new) before returning, leaving stale files only on hard-crash paths. **Verification**: `python3 -c "import json, pathlib; data = json.loads(pathlib.Path('cortex/lifecycle/<test-feature>/daytime.pid').read_text()); assert data['magic'] == 'cortex-daytime-v1' and 'pgid' in data and 'start_time' in data and data['schema_version'] == 1"`. **Phase**: Phase 1: Foundation hardening

2. **Liveness verification uses `magic` + ±2s `start_time` window** (`±2s = _START_TIME_TOLERANCE_SECONDS` matching overnight `ipc.py:85`): Concurrent-dispatch detection consults the JSON `daytime.pid`; alive-PID acceptance requires (a) `magic == "cortex-daytime-v1"`, (b) `psutil.Process(pid).create_time()` within ±2 seconds of stored `start_time`. The ±2s tolerance is a floating-point precision allowance between two `psutil.Process.create_time()` readings of the same kernel value — NOT a disk-pressure budget. PID-recycling false-positives are bounded by the ±2s window's narrowness AND by R1's clean-shutdown-clears-file behavior. **Additional check (defense-in-depth)**: when verifying liveness, also confirm `os.readlink(f"/proc/{pid}/exe")` resolves to a Python interpreter on Linux, or `psutil.Process(pid).cmdline()` contains `daytime_pipeline` on macOS — a process running an unrelated program with a recycled PID is rejected even within the ±2s window. **Verification**: `pytest tests/test_daytime_concurrent_dispatch.py -v` exits 0 with cases covering (a) valid alive PID, (b) PID recycled by foreign process within ±2s (rejected via cmdline mismatch), (c) PID recycled by foreign Python process within ±2s (rejected via cmdline mismatch since cmdline lacks `daytime_pipeline`), (d) stale PID file (cleared on read). **Phase**: Phase 1: Foundation hardening

3. **`install_guard` scans daytime PID files for liveness**: `cortex_command/install_guard.py` `check_in_flight_install_core` (lines 71-73, 288-294) extended to glob `cortex/lifecycle/*/daytime.pid` and trip on any file whose `magic` + `start_time` + cmdline-check (R2's full verification, not just liveness) confirms alive AND matches an in-flight daytime dispatch. Same `CORTEX_ALLOW_INSTALL_DURING_RUN=1` bypass and same carve-out list. When `install_guard` encounters a stale file (PID dead OR cmdline mismatch), it logs a warning event and clears the file before returning safe — preventing the "false-positive-tripwire forever" failure mode. **Verification**: `pytest tests/test_install_guard_daytime_awareness.py -v` exits 0 with cases (a) valid alive dispatch trips guard, (b) stale file cleared and guard passes, (c) PID recycled to foreign process recognized and cleared. **Phase**: Phase 1: Foundation hardening

4. **Feature-slug validator applied at four layers (regex + path containment)**: Tightened regex `^(?!\.{1,2}$)[a-zA-Z0-9_][a-zA-Z0-9._-]{0,127}$` (rejects `.`, `..`, and any leading `.` or `-`) enforced at (a) Pydantic input model `Field(pattern=...)` on every daytime MCP tool, (b) argparse type-converter on `--feature` in `cortex daytime` CLI subparser. Plus (c) **path containment check**: after constructing `Path(f"cortex/lifecycle/{feature}").resolve()`, assert the resolved path is a child of `Path("cortex/lifecycle").resolve()` (mirrors `assert_path_contained` semantics that overnight's `session_validation.py` pairs with its regex). The containment check fires at three layers: (i) MCP delegate `_delegate_daytime_start_run` after Pydantic validation, (ii) CLI handler `_dispatch_daytime_start` after argparse parsing, (iii) `daytime_pipeline.run_daytime` at the top of the function, before any path construction. **The Non-Requirement allowing `run_daytime` to host a runtime guard is documented in this spec's Non-Requirements section** — see carve-out below. **Verification**: `pytest tests/test_daytime_feature_slug_validation.py -v` exits 0 with cases for (a) valid slug, (b) `feature=".."` rejected, (c) `feature="."` rejected, (d) `feature="-foo"` rejected, (e) `feature="../etc/passwd"` rejected (containment check), (f) over-length, (g) special chars. **Phase**: Phase 1: Foundation hardening

### Phase 2: CLI + MCP surfaces

5. **`cortex daytime start --feature <slug> --format json` CLI verb**: New subparser in `cortex_command/cli.py` mirrors the overnight pattern at `cli.py:378-456` (subparser block + `_dispatch_daytime_start` lazy importer). Handler lives in a new `cortex_command/overnight/daytime_cli_handler.py` symmetric to `cli_handler.py`. **The parent CLI handler does the O_EXCL atomic claim on `cortex/lifecycle/<feature>/daytime.pid` BEFORE Popen** (writing a sentinel `{"schema_version": 1, "magic": "cortex-daytime-v1", "pid": null, "pgid": null, "start_time": null, "feature": "...", ...}` with the wrapper's intent; the spawned child then atomically updates it with real values). This puts the concurrent-dispatch refusal contract on the parent's path, so two racing parents deterministically resolve via O_EXCL's `FileExistsError` rather than via post-spawn handshake timing. The verb spawns the pipeline via `subprocess.Popen(start_new_session=True)` matching `cli_handler._spawn_runner_async` at lines 377-384 (DEVNULL stdin, log-file stdout/stderr, `close_fds=True`). On successful spawn returns the JSON envelope `{"schema_version": "2.0", "started": true, "pid": <int>, "feature": "<slug>", "started_at": "<ISO 8601>"}` on stdout, exit 0. Hidden `--launchd` flag (`argparse.SUPPRESS`) signals the child "you ARE the runner, skip the fork" — same pattern as overnight at `cli.py:450-455` and `cli_handler.py:603-611`. **Verification**: `cortex daytime start --feature smoke --format json` from a fresh terminal exits 0 with a JSON envelope whose `pid` field references a process where `ps -o ppid= -p <pid>` returns 1 (orphaned to init). **Phase**: Phase 2: CLI + MCP surfaces

6. **Concurrent-dispatch refusal envelope (parent-side O_EXCL contract)**: When the parent CLI's O_EXCL claim raises `FileExistsError`, the verb verifies whether the existing file is alive (R2's full check). If alive, exits non-zero with `{"schema_version": "2.0", "started": false, "reason": "concurrent_dispatch", "feature": "<slug>", "existing_pid": <int>}` on stdout. If stale (cmdline mismatch or dead PID), the verb clears the file and retries the O_EXCL claim once before Popen. Matches the overnight `concurrent_runner` refusal contract documented at `docs/internals/mcp-contract.md:70-90`. **The contract is deterministic across parent races**: O_EXCL is atomic at the kernel level; one parent wins, the other observes `FileExistsError` and emits `concurrent_dispatch` (NOT `spawn_timeout`). **Verification**: `pytest tests/test_daytime_cli_concurrent_refusal.py -v` exits 0 with cases (a) sequential dispatches refused cleanly, (b) two-parent race deterministically resolves one win + one `concurrent_dispatch` envelope, (c) stale-file path clears and proceeds. **Phase**: Phase 2: CLI + MCP surfaces

7. **Spawn handshake matches overnight's 5-second budget for child process initialization**: After parent's O_EXCL claim and Popen, the CLI polls for the spawned child to update `daytime.pid`'s `pid`/`pgid`/`start_time` fields from `null` (parent's sentinel write) to real values, using a 5-second timeout. This budget covers Python startup + filesystem bookkeeping under disk pressure (Time Machine, Spotlight) — matching overnight's `_SPAWN_HANDSHAKE_TIMEOUT_SECONDS`. If the child doesn't update the sentinel within 5s, the CLI clears its sentinel file and exits non-zero with `{"started": false, "reason": "spawn_timeout"}`. **Verification**: `pytest tests/test_daytime_spawn_handshake.py -v` exits 0 with cases for (a) normal sub-second handshake, (b) injected disk pressure delay <5s succeeds, (c) injected delay >5s fails cleanly with sentinel cleared. **Phase**: Phase 2: CLI + MCP surfaces

8. **`DAYTIME_DISPATCH_ID` hybrid auto-mint with strict disk verification**: The CLI verb's handler resolves the dispatch ID in this order with **fail-closed semantics on branch (b)**: (a) `--dispatch-id <hex>` flag value if provided (validated against `^[a-f0-9]{32}$` at the wrapper layer before any disk write), (b) `DAYTIME_DISPATCH_ID` env-var if set AND a matching `cortex/lifecycle/<feature>/daytime-dispatch.json` exists on disk whose `id` field equals the env-var value (validated against same regex). **If env-var is set but no matching file exists, the wrapper errors with `{"started": false, "reason": "stale_dispatch_id_env"}` — does NOT silently propagate the dead ID**, (c) mint a fresh uuid4 hex AND write `cortex/lifecycle/<feature>/daytime-dispatch.json` ONLY if the file does not already exist. The wrapper never clobbers an existing `daytime-dispatch.json`. An `events.log` entry `daytime_dispatch_id_minted_by_wrapper` is emitted when path (c) fires, for diagnosability. **Verification**: `pytest tests/test_daytime_dispatch_id_resolution.py -v` exits 0 with cases (a) flag-supplied valid id, (b) flag-supplied invalid id rejected, (c) env-var matches existing file (accepted), (d) env-var set with no matching file (`stale_dispatch_id_env` error), (e) env-var matches existing file with mismatched id field (rejected), (f) mint-with-no-existing-json (success), (g) mint-with-existing-json (no clobber, file read). **Phase**: Phase 2: CLI + MCP surfaces

9. **New plugin `plugins/cortex-daytime/`** exposes 4 MCP tools (NOT embedded in `cortex-overnight`): `daytime_start_run`, `daytime_status`, `daytime_logs`, `daytime_cancel`. Plugin files: `server.py` (mirrors `plugins/cortex-overnight/server.py` patterns), `.mcp.json` (registers the server with `command: "uv", args: ["run", "${CLAUDE_PLUGIN_ROOT}/server.py"]`), `.claude-plugin/plugin.json` (metadata: name `cortex-daytime`, description, author). The plugin has its own `CLI_PIN = ("<tag>", "2.0")` and `MCP_REQUIRED_CLI_VERSION` constants matching the cortex CLI schema floor. R1 architectural invariant preserved: zero `cortex_command.*` imports in `server.py`. **Verification**: `python3 -c "from pathlib import Path; assert (Path('plugins/cortex-daytime/server.py').exists() and Path('plugins/cortex-daytime/.mcp.json').exists() and Path('plugins/cortex-daytime/.claude-plugin/plugin.json').exists())"`; `grep -c "^from cortex_command\|^import cortex_command" plugins/cortex-daytime/server.py` = 0. **Phase**: Phase 2: CLI + MCP surfaces

10. **Both `daytime_start_run` AND `daytime_cancel` Pydantic input models include `confirm_dangerously_skip_permissions: Literal[True]`**: The gate matches overnight's `StartRunInput` at `plugins/cortex-overnight/server.py:2049-2053`. 12. **`daytime_cancel` uses `os.killpg(pgid, SIGTERM)` from the upgraded PID schema**: signals the process group, not just the leader PID, so SDK children are terminated cleanly. ## Non-Requirements

- **No `daytime schedule` verb**: daytime is per-feature and short-running; no future-scheduled launch semantics. Confirmed by user in Clarify Q1.
- **No `daytime list-sessions` verb / `daytime_list_sessions` MCP tool**: daytime dispatches are feature-scoped, not session-scoped; the natural query is `daytime_status --feature <slug>`. Confirmed by user in Clarify Q1.
- **No real `launchctl bootstrap`-based detachment (Alternative A)**: explicitly deferred. Documented as the durable upgrade path in `docs/daytime-operations.md` (R14) if Anthropic restricts `dangerouslyDisableSandbox: true` or sandboxes MCP servers upstream.
- **No internals refactor of `cortex_command/overnight/daytime_pipeline.py` beyond two explicit carve-outs**: (a) the minimal PID-write/read/recover contract upgrade (R1, R2), and (b) a single-statement feature-slug containment check at the top of `run_daytime` (R4 layer c, R18). `_check_cwd`, `_check_dispatch_id`, `_spawn_pipeline_worktree`, `_orphan_guard`, and the worktree-creation/teardown flow are untouched. The earlier "untouched" claim about `run_daytime` is hereby narrowed: `run_daytime`'s body is unchanged except for the one-statement input-validation guard at its top, which is required to make R18's regression-guard path safe against path-traversal.
- **No replacement of the `no_commit_guard` pause behavior**: tracked elsewhere as a separate concern.
- **No migration of dispatched-session auth from OAuth to API key**: `--bare` mode was a tangent during research; not a blocker.
- **No `active-session.json` participation**: daytime registers awareness via R3 (install_guard scanning daytime PID files), not by writing to the overnight-namespaced host-global pointer.
- **No semantic feature-slug existence check**: `daytime_start_run` does NOT verify that the feature slug corresponds to an existing backlog entry or non-empty lifecycle directory. Regex-valid-but-novel slugs will create new lifecycle directories. **Rationale**: the lifecycle skill is the legitimate creator of lifecycle directories before dispatch; adding a semantic check at the MCP layer would block the lifecycle skill's own usage pattern. Prompt-injection that creates novel lifecycle directories with regex-valid names is a concern flagged for follow-up but not addressed in this lifecycle.
- **Scope question (surfaced for §4 approval): whether Phase 1 ships as part of this lifecycle or as a preceding ticket**. By the spec's own framing, Phase 1 is "prerequisites that stand alone as security/correctness fixes." Critical review surfaced this as scope creep beyond ticket #228. The §4 approval surface presents three alternatives (drop entirely, bugs-only/Phase-1-only, minimum viable).

## Edge Cases

- **Disk pressure during spawn handshake**: 5s `wait_for_pid_file` budget in R7 matches overnight's `_SPAWN_HANDSHAKE_TIMEOUT_SECONDS`. The ±2s window in R2 is unrelated to disk pressure — it's a floating-point precision tolerance between two psutil readings of the same kernel-provided `create_time`, NOT a disk-pressure budget.
- **PID recycling within ±2s**: defended by R1's clean-shutdown-clears-file behavior AND R2's cmdline-mismatch check. A foreign Python process recycling the PID still fails the cmdline check unless it happens to be running `daytime_pipeline` — a vanishingly narrow window.
- **TOCTOU concurrent dispatch**: O_EXCL atomic claim is in the **parent CLI** (R5), not the spawned child. Two racing parents deterministically resolve via `FileExistsError`; the loser emits `concurrent_dispatch` (R6), not `spawn_timeout` (R7).
- **Parent-CLI O_EXCL claim with crashed-parent stale file**: if a parent CLI dies between O_EXCL claim and Popen, the sentinel file (with null pid/pgid) is left. R7's 5s handshake will time out waiting for the child update, AND the wrapper's startup logic checks for sentinel-with-null-fields-older-than-handshake-timeout and clears them on next dispatch.
- **Crashed child without `daytime.pid` cleanup**: `_recover_stale` (R1) consults the full magic + start_time + cmdline verification before forcibly removing the worktree; install_guard (R3) likewise clears stale files on encounter rather than tripping indefinitely.
- **`daytime-dispatch.json` already present when wrapper auto-mints**: per R8, the wrapper never clobbers an existing file. The mint path is gated on file absence.
- **Stale `DAYTIME_DISPATCH_ID` env-var without matching file**: per R8 branch (b)'s fail-closed semantics, the wrapper errors with `stale_dispatch_id_env` rather than silently propagating a dead ID.
- **Non-darwin invocation (Linux/CI)**: the CLI verb works (Popen + start_new_session is portable). The harness-dependency narrative does not apply (no Seatbelt). The EPERM hazard is macOS-specific; the CLI verb falls through to the same Popen detachment for behavioral consistency.
- **Plugin install ordering**: `cortex-daytime` and `cortex-overnight` install independently. Both `CLI_PIN` constants target the same cortex CLI schema floor (`"2.0"`).
- **MCP tool name length**: with `mcp__plugin_cortex-daytime_cortex-daytime__` (40 chars) prefix, longest unprefixed tool name `daytime_start_run` (17 chars) totals 57 chars — under 64-char limit.
- **`pgid` missing from corrupted PID file**: R12 distinguishes JSON-parse failure (error envelope, no signal sent) from missing-field-in-well-formed-JSON (fall back to `os.kill(pid, SIGTERM)`).
- **Hardcoded `.vscode`/`.idea` Claude-binary denies**: per `pipeline.md:167`; unaffected by daytime's surface.

## Technical Constraints

- **Stateless MCP server**: `plugins/cortex-daytime/server.py` follows the `pipeline.md:153` requirement.
- **R1 architectural invariant**: zero `cortex_command.*` imports in `plugins/cortex-daytime/server.py`.
- **`confirm_dangerously_skip_permissions: Literal[True]` on both spawn AND destroy MCP tools**: required on `daytime_start_run` (R10) AND `daytime_cancel` (R10). Status and logs are reads and do not require the gate. Asymmetric gating between spawn and destroy is explicitly rejected as a security posture.
- **Feature-slug regex `^(?!\.{1,2}$)[a-zA-Z0-9_][a-zA-Z0-9._-]{0,127}$` PAIRED with path-containment check**: regex alone is insufficient; `assert_path_contained` is the load-bearing check. Both enforced at 4 layers (R4).
- **Spawn handshake 5s, not 1s**: matches overnight's proven budget under disk pressure.
- **Subprocess timeout 30s for `daytime_start_run`**: matches `_START_RUN_TOOL_TIMEOUT`.
- **No `check=True` on subprocess.run inside the MCP server**: explicit returncode inspection.
- **Separate plugin, not embedded in `cortex-overnight`**: MCP tool name budget; conceptual separation; independent versioning.
- **JSON envelope schema_version "2.0"**: all CLI verb JSON output prepended with `"schema_version": "2.0"` via the same `_emit_json` pattern as overnight.
- **Harness dependencies are detected via R7's spawn handshake**: framing is honest — Bash-path requires `dangerouslyDisableSandbox: true` (cortex cannot enforce); MCP-path requires MCP servers being unsandboxed at hook trust level (cortex cannot enforce). However, R7's 5s `wait_for_pid_file` handshake DOES detect EPERM-induced spawn failures and surfaces them as `spawn_timeout` rather than silent failure. Future Anthropic changes that restrict either harness behavior will be surfaced via this detection path before causing silent regressions.
- **macOS-only EPERM hazard**: the Seatbelt-inheritance failure mode is macOS-specific. The CLI verb is portable but provides no EPERM-relief on non-darwin (where the bug doesn't exist).
- **`daytime-result.json` schema_version stays at 1**: hard-equality check at `daytime_result_reader.py:163-167` is unchanged.
- **CLI_PIN coupling**: adding new MCP tools to the cortex schema floor is additive. The new `cortex-daytime` plugin gets its own `CLI_PIN = ("<tag>", "2.0")` tuple tracking the cortex CLI version.
- **Parent-side concurrency contract**: the `daytime.pid` O_EXCL claim is on the parent CLI handler, not the child pipeline. This makes the `concurrent_dispatch` refusal envelope deterministic under parent races and decouples it from R7's spawn handshake.
