# Research: Fix selected-path interactive-worktree orphan-lock (#355) — reorder vs release-on-reject

**Decision under research:** On the picker-`selected` entry mode of the interactive-worktree implement flow (`skills/lifecycle/references/implement.md`), §1 **Step B** runs `cortex-interactive-lock acquire {slug}` **before** §1a.i's second overnight guard. If overnight goes live in that window, §1a.i rejects (exit 1) and exits §1a while the Step-B lock is never released → **orphaned lock**, self-blocking the session on retry. The `suppressed` entry mode is already safe (its acquire lives in §1a.ii, *after* §1a.i — ticket #348's fix). Two candidate fixes: **(A) REORDER** the selected-path acquire into §1a.ii so acquire always follows the overnight guard, vs **(B) RELEASE-ON-REJECT** — keep the early acquire and call `cortex-interactive-lock release {slug}` on the reject exit.

**Recommendation: Option A (REORDER).** All five research angles — codebase, external prior art (CWE-367), project conventions, tradeoff analysis, and adversarial review — converge on Option A for #355's named bug, with two load-bearing refinements from the adversarial pass (verification must gain a negative assertion; the edit is a multi-site prose scrub, not a one-liner) and one open scope decision for Spec (§1a.iii-failure orphan: defer vs include).

## Web Research

External prior art converges — from three independent directions — on structural reordering over manual per-path release:
- **CWE-367 (TOCTOU)** ranks its mitigations: "implement locking *before* the check rather than after" and atomicity are ranked **above** window-shrinking patches. Release-on-reject is a window-shrinking patch → ranked weaker by the authoritative taxonomy.
- **Acquire-late / minimize-critical-section** (OS/concurrency practice): validate preconditions *before* acquiring, so a rejected operation never holds the resource.
- **RAII / `with` / try-with-resources**: bind release to scope so it is exit-path-independent — precisely to remove "human forgets the `finally`."
- The **"forgot to add cleanup on a new early-exit path"** failure mode is documented as endemic to manual cleanup (SEI CERT **MEM12-C**, Go `defer` leak literature, single-exit-point retrospectives). Manual per-path release (goto-chains, `defer`, try/finally) is uniformly presented as the *fallback* only when scope-bound/precondition-ordered acquisition is unavailable — never as preferable when reordering is available. This is exactly Option B's fragility and Option A's advantage.

## Requirements & Constraints

- **CLAUDE.md — "Prefer structural separation over prose-only enforcement for sequential gates. A gate encoded in skill control flow is harder to accidentally bypass than one that relies on the model reading and following a prose instruction. Prose-only enforcement is appropriate only for guidelines where the cost of occasional deviation is low."** (RC2) This bears **directly and asymmetrically**: Option A is a gate encoded in control flow (if §1a.i rejects, execution never reaches the acquire). Option B is prose-only enforcement at each exit; and its exception clause ("cost of occasional deviation is low") does **not** apply — an orphaned per-feature lock blocks all future same-slug interactive sessions until stale recovery. "Prescribe What and Why, not How" independently reinforces A (Option B pushes procedural "remember to release here" narration into the skill body).
- **#348's `release_lock` decision** (RC1). #348 spec.md line 42: *"Does **not** add lock release/cleanup on session end (`release_lock` remains unused in the skill flow; a held lock is released implicitly when the owner process dies → stale recovery — the correct behavior for a single interactive session)."* Verdict: this was scoped to **normal session-end**, so Option B (release on an *error-abort* exit where the process keeps running) does not *literally* contradict it. But #348 fixed the symmetric suppressed-path orphan by **REORDER** ("positioned after the §1a-ii overnight guard passes — so a rejecting overnight guard cannot orphan the lock") and never adopted release-on-abort anywhere. #348 sets a **directly transferable structural precedent for Option A**.
- **Solution horizon** (project.md): #348 named this exact selected-path instance and carved it to #355 by name — "a follow-up already planned" for the *same hazard shape* → the horizon test's trigger for the durable version (reorder, matching #348).
- **Scope/discipline**: `implement.md` is a reference file, **exempt** from the SKILL.md L1 byte-budget ratchet — Option B's added prose has no budget cost, and Option A's deletion no budget benefit. `cortex_command/*.py` is **not** mirrored. **#353 sequencing**: #348 flagged that any other `implement.md`-editing ticket must run **sequential-after** (not concurrent) to avoid a mirror-parity collision — commit #355 explicit-pathspec.

## Tradeoffs & Alternatives

| Dimension | Option A (REORDER) | Option B (RELEASE-ON-REJECT) |
|---|---|---|
| Implementation | Net-negative diff (deletes Step B + collapses §1a.ii branch), but multi-site prose scrub (A6) | Smallest single-site diff *if* scoped to §1a.i only; grows if it must cover §1a.iii too |
| Maintainability | **Best** — any future early-exit inserted before the relocated acquire is orphan-safe by construction | **Weakest** — "remember to release on every exit"; every future §1a exit needs a paired release nobody forces |
| Correctness completeness | Closes §1a.i orphan for both modes uniformly; **cannot** touch §1a.iii-failure orphan (acquire must precede create) | If fully scoped, could also close §1a.iii; as ticket-scoped ("arguably"), closes §1a.i only |
| Pattern alignment | Extends #348's shipped guard-then-acquire pattern; removes the entry-mode asymmetry at the acquire | Introduces net-new release-on-failure surface to the skill |
| Verification | Testable with genuine discriminating power (order + absence + negative selected-arm assertion) | Position-check possible, but static tests can't certify a *future* exit got its release — verification-vacuity risk |

**Recommended: Option A**, paired with a discriminating structural test (not the existing presence-only test). Rationale centers on maintainability dimension 2: Option B's correctness degrades every time someone adds a new §1a exit and doesn't wire a release — a failure mode with no test that can catch it in advance.

## Adversarial Review

- **A1 — mutual exclusion preserved; lock is best-effort regardless.** Relocating the acquire does not break mutual exclusion (both modes gate on the single §1a.ii acquire; the loser is rejected). Separately (and pre-existing, unchanged by #355): `acquire_lock` uses read→verify→`os.replace`, **not** `O_CREAT|O_EXCL` — under true simultaneity two arrivals can both read-None and both write. **Out of #355 scope**; noted so #355 is not sold as providing hard mutual exclusion.
- **A2 — the §1a.iii-failure orphan is the weakest link in the scope decision.** §1a.iii (`cortex-worktree-create` fails → "exit §1a" with the §1a.ii lock held) is the **same failure class** and can be **named today** (solution-horizon). Crucially, it is **structurally unfixable by reorder** (the acquire must precede worktree creation), so it needs a *different* mechanism (release-on-abort) — Option A cannot cover it. After Option A there is exactly **one** residual post-acquire hard-exit: §1a.iii. Adversarial's call: do release-on-abort at §1a.iii **in the same edit**, not deferred. (Counter-weighing in Open Questions.)
- **A3 — "unifies both entry modes" is overstated.** Option A unifies the **acquire position** only. `selected` still runs Step A *and* §1a.i (two overnight checks); `suppressed` runs only §1a.i. Removing Step A would widen scope **and break** `test_sidecar_invocation_form_bash_s_count` (exactly 2) + `test_overnight_guard_sidecar_called_at_least_twice` (≥2) — the double-check is a **tested contract**. **Keep Step A.**
- **A4 — delayed acquire widens the picker-suppression under-detection window.** Crash-before-acquire: Option A is *cleaner* (no stale lock file). Concurrent slip-in: a later same-slug session can win §1a.ii before an earlier session that already got the human's picker click → the earlier one is bounced. Fairness/UX regression only (still single-winner), **low severity** (two concurrent same-slug interactive implements is already operator error). Overnight Phase-2 `scan_live_locks` can't see the in-flight selected session for longer — probably benign (§1a.i re-guards overnight before commit); one confirming look at the orchestrator is out-of-file.
- **A5 — the proposed test is SELF-SEALING against the likeliest mis-edit (must fix).** §1a.ii's *suppressed* arm already has an acquire after §1a.i. If Step B is deleted but the `selected` arm ("already acquired… do not acquire again", line 104) is **not** flipped, the selected path acquires **nothing** and builds a worktree with **no lock** — and the proposed "acquire index after overnight index" test **still passes** (the suppressed acquire satisfies it). The test **must** add a **negative assertion** that §1a.ii's `selected` arm no longer carries the skip-acquire escape (or that the acquire is not guarded by an entry-mode conditional excluding `selected`), plus: measure both indices in one coordinate space, pin to the *earliest* acquire, and keep the §1-has-no-acquire clause.
- **A6 — coupled prose scrub + edit hygiene.** Step B is referenced at lines **19, 63, 73, 104** (+ the 75–81 block) — all must be scrubbed or the result reads self-contradictory (e.g., Step A "proceed to Step B" pointing at a deleted block); a command-string grep test catches none of these. **Preserve the `selected`/`suppressed` marker** — Step v branches on it (cd-shim vs `EnterWorktree`; ADR-0008 ties EnterWorktree authorization to the `selected` marker, not the acquire). De-conditionalize the §1a.ii *acquire*, but keep the marker recording. Preserve the `**iii.`/`**iv.` list markers (`test_gate_and_gated_path_use_same_binary` extracts via them).

## Open Questions

Adversarial argues include-now (solution-horizon; unfixable-by-reorder so deferral yields no better mechanism).
