# Plan: restructure-commit-skill-lazy-load-release

## Overview
Extract the rarely-needed release-type version-bump mechanics out of the always-resident `skills/commit/SKILL.md` into a new lazily-loaded `skills/commit/references/release-type.md`, keeping the *decision trigger* (marker tokens + own-line rule + one-line `BREAKING:` backstop) resident so the silent under-versioning hazard is still guarded; then compress the Workflow/Format/Command narration and fold Validation in place. Under the Execution Mode below, all content edits accumulate in a single shared working tree and land as **one commit** (canonical SKILL.md + new ref + regenerated cortex-core mirror), staged explicitly and gated by the fail-closed drift + skill-path pre-commit hooks.

## Execution Mode (hard constraint — precondition for correctness)
This plan MUST run **trunk-sequential in a single shared working tree**, with all commits deferred to Task 6. This is not a preference — it is the precondition the design's correctness rests on, and it is stated here as a gate rather than as buried prose intent:

- **No worktree/parallel dispatch of Tasks 1–5.** Tasks 2–5 edit the same file (`skills/commit/SKILL.md`) and do not commit; their `Depends on` edges fix *edit order*, not shared-file safety. Under `Agent(isolation:worktree)` each successor is checked out fresh from main and cannot see a predecessor's *uncommitted* edits, so parallel/worktree dispatch would sequence last-writer-wins clobbers rather than accumulate edits. Run them as direct orchestrator edits (or strictly sequential same-worktree dispatch).
- **No per-task commits for Tasks 1–5.** The single-commit + shared-file + fail-closed-drift design requires the canonical edits and the regenerated mirror to land in one commit (mirror regen cannot be stranded after an already-committed skill edit). An intermediate commit of Task 2 would also land the `${CLAUDE_SKILL_DIR}` pointer while its target ref is still untracked (`git commit --only`/pathspec drop untracked files) — a pointer-without-target broken state.
- **Forced-fallback escape (only if single-worktree sequential is impossible):** land multiple commits, but then EACH `skills/`-touching commit must independently run `just build-plugin`, `git add` the (possibly untracked) canonical + mirror paths it touches, and pass the drift gate in that same commit. This is the degraded path the spec's Phases note allows; it is not the intended path.

## Risks
- **Dispatch mode is a correctness precondition, not a preference** (see Execution Mode): the single-commit + shared-file + fail-closed-drift design is correct only under trunk-sequential single-worktree deferred-commit execution. Parallel/worktree dispatch would clobber the shared `SKILL.md` edits (uncommitted edits are invisible across worktrees); per-task commits would strand mirror regen or land a pointer whose target ref is still untracked. The forced-fallback escape (multi-commit, each with its own `build-plugin` + `git add`) is the documented degraded path, not the intended one.
- **Decision-trigger placement** (Req 2): a grep cannot assert the trigger sits at the message-composition step, so it is guarded structurally by the Task 2 Placement gate (`tok < ptr` — the first marker token precedes the ref pointer), re-checked at the Task 6 end-state gate. The gate is necessary-not-sufficient (it kills the inert-footnote layout but does not fully prove "at the composition step"), so final placement is confirmed at implementation review. The earlier framing that removal-greps + the body-size gate "prove" the trigger placement was a non-sequitur — those establish only compactness and mechanics-absence.
- **Spec acceptance greps carry a backslash defect** (corrected in this plan): the spec's Req 1/Req 3 use `grep -F '(?im)^\s*[release-type:'`, which returns 0 against the verbatim `\[release-type:` content — so as written the Req 1 keep would fail a correct implementation and the Req 3 removal would pass vacuously. This plan's Task 1/Task 2 greps use the byte-correct `\[release-type:`. The spec files still hold the un-escaped form; flag for correction on any spec touch.
- **s8 examples folded into the ref, not dropped** (resolved in spec): zero hot-path cost, defense-in-depth for a silent/irreversible failure mode. Dropping was defensible; fold was chosen.
- **No single-sourcing of the regex** (deliberate Non-Requirement): the regex now lives in three places (ref + `auto_bump_version.py` + `docs/release-process.md:124`). "Reduces duplication" is explicitly not a claimed benefit; no byte-identity test is added.
