# Research: Profile and root-cause the 50-70ms Python boot overhead in cortex-log-invocation

## Codebase Analysis

### Files in scope

- **`bin/cortex-log-invocation`** (31 lines) — bash dual-channel wrapper. Force-source / wheel-probe / working-tree-fallback / exit-2 branch logic.
- **`cortex_command/log_invocation.py`** (157 lines) — Python module implementing the fail-open JSONL logger. Stdlib-only (no third-party imports).
- **`tests/test_log_invocation_perf.py`** (186 lines) — currently `@pytest.mark.skip`'d perf-budget tests:
  - `test_log_invocation_fast_path_budget` (p50 ≤ 15ms / mean ≤ 18ms / p95 ≤ 25ms) — budgets from the bash-shim era, unachievable post-rewrite.
  - `test_log_invocation_fast_path_faster_than_slow` (delta ≥ 2ms) — premise is broken; both paths now converge to Python boot cost.
- **`tests/test_cortex_log_invocation_parity.py`** — byte-equivalence parity suite (must continue passing after any change).
- **`plugins/cortex-core/bin/cortex-log-invocation`** — plugin mirror; auto-regenerated by dual-source pre-commit hook.

### Critical structural finding: double Python boot in the bash wrapper

The bash wrapper at `bin/cortex-log-invocation:17-20` runs `python3 -c "import cortex_command.log_invocation"` as an **importability probe**, then `exec python3 -m cortex_command.log_invocation` for the actual work. That is **two Python boots back-to-back** on every happy-path invocation:

```bash
# (b) Wheel-import probe — fires ~20ms subprocess to TEST importability
if python3 -c "import cortex_command.log_invocation" 2>/dev/null; then
    exec python3 -m cortex_command.log_invocation "$@"  # ANOTHER ~30-40ms boot
fi
```

**~20ms of pure interpreter startup is paid twice.** This is the single largest source of slack and is the dominant addressable cost.

### Measured baseline (this hardware, Apple Silicon, Python 3.14)

| Measurement | Wall time |
|---|---|
| Bare `python3 -c "pass"` (interpreter floor) | ~20ms |
| `python3 -m cortex_command.log_invocation` (Python only, happy path) | ~30-40ms |
| `bash bin/cortex-log-invocation ...` (full wrapper, happy path) | ~70ms p50 |

The ~70ms decomposes roughly as: ~20ms wheel-probe Python boot + ~30-40ms actual Python module exec + ~10ms bash + I/O.

### Python module import profile (`-X importtime`, top costs)

| Import | Cumulative time |
|---|---|
| `subprocess` | ~6-7ms |
| `pathlib` | ~2-3ms |
| `json` | ~1-2ms |
| `datetime` | ~3-5ms |
| `typing` | ~1-2ms |
| `site` + `sitecustomize` | ~6ms (paid regardless) |

The module's top-level imports total ~11ms. `subprocess` is the heaviest single import — and it's **only used on the fallback path** (when `CORTEX_REPO_ROOT` is unset and we shell out to `git rev-parse`).

### `main()` work breakdown (happy path with `LIFECYCLE_SESSION_ID` + `CORTEX_REPO_ROOT` set)

1. Check `LIFECYCLE_SESSION_ID` (free)
2. `_resolve_repo_root()` — validates `CORTEX_REPO_ROOT` is a directory with a `.git` marker; only spawns `subprocess.run(["git", "rev-parse"])` when validation fails (~2-3ms when env-var path succeeds)
3. `session_dir.mkdir(...)` (~1-2ms when dir already exists)
4. JSONL write (~3-5ms)
5. Returns exit 0 unconditionally (fail-open contract)

### Prior-art lifecycle: #198 `trim-cortex-log-invocation-shim-cost-per-call-21ms`

The prior work optimized the **bash shim** from ~21ms → ~9ms by introducing `CORTEX_REPO_ROOT` env-var fast-path (skipping `git rev-parse` fork when a cooperating parent sets it). Three producers were wired to export `CORTEX_REPO_ROOT`:

1. Overnight runner (`cortex_command/overnight/runner.py:2014`)
2. SessionStart hook (`hooks/cortex-scan-lifecycle.sh:9-10`)
3. Dispatch orchestrator (`cortex_command/pipeline/dispatch.py:541-554`)

Then commit **`7c05529c`** (dated 2026-05-25) replaced the bash shim entirely with the dual-channel Python wrapper to satisfy the wheel + working-tree distribution contract. The ~9ms → ~70ms regression was introduced at that point and is the reason for the current ticket.

### Callers (per-session frequency)

- ~10+ leaf scripts invoke directly (`cortex-lifecycle-state`, `cortex-lifecycle-counters`, `cortex-jcc`, etc.)
- 13+ scripts use a Python one-liner subprocess invocation pattern (`cortex-commit-preflight`, `cortex-check-parity`)
- No hot loops identified, but bulk dashboard queries could plausibly fire 50+ invocations
- At 70ms/call: 10 calls = 700ms session overhead; 50 calls = 3.5s session overhead

- `cortex_command/interactive_lock.py:141` — lazy `import psutil`
- `cortex_command/cli.py:714` — lazy lifecycle-subcommand imports

These are the patterns to align with.

## Web Research

### Irreducible Python interpreter boot floor

- Apple Silicon Python 3.11+: estimated ~8-12ms floor for `python3 -c "pass"` (~5-7ms with `-S`)
- Frozen modules (Python 3.11 default) gave ~10-15% startup reduction over 3.10
- The 70ms observed in the wrapper is well above the floor — dominated by **wrapper-level work and imports**, not interpreter init

### Techniques with strongest real-world evidence

| Technique | Reported savings | Notes |
|---|---|---|
| **Function-scoped (lazy) imports** | 40-86% total reduction across several real CLIs (Posting, pypistats, pipenv) | Single biggest lever |
| **Avoid `pkg_resources`** | ~100ms saved on pipenv | Not currently used in cortex-log-invocation — non-issue |
| **`-S` flag (skip site.py)** | ~5-6ms | Risk: breaks venv path setup, can't be used with wheel-import branch |
| **Frozen modules** (Python 3.11+) | 10-15% | Already default; no action needed |
| **Bytecode `.pyc` priming** | 10-50ms when missing | `__pycache__` is present; no work needed |

### Heavy stdlib modules (community-validated list)

- `inspect` — ~8ms on M3 (cpython#117865), worst offender
- `dataclasses` (pulls in `inspect`)
- `pkg_resources` / `setuptools` — universally flagged
- `subprocess` — meaningful cost (~6ms confirmed here)
- `urllib`, `email`, `http.client` — community consensus: avoid at top-level
- `argparse` — moderate (~3-10ms); not currently used in this module

### PEP 810 (explicit lazy imports) — Accepted Nov 2025, targets Python 3.15

Direct fit for this use case once 3.15 ships. Pre-3.15: function-scoped imports are the canonical workaround.

## Requirements & Constraints

### Dual-channel contract is load-bearing (project.md:38, ADR-0002)

> **Wheel-binstub vs working-tree invocation**: `cortex-<skill>` binstubs execute against the installed wheel's `site-packages/`; `python3 -m cortex_command.<skill>` runs against the working tree. ... Setting `CORTEX_COMMAND_FORCE_SOURCE=1` in the environment makes the dual-channel wrappers ... skip the wheel-import branch and execute the working-tree module directly via `python3 -m cortex_command.<module>`, regardless of whether a wheel is installed.

**Implication**: The three operative branches (force-source / wheel / working-tree) cannot be removed. They may be **reordered** and the wheel-import probe may be **cached**, as long as `CORTEX_COMMAND_FORCE_SOURCE=1` semantics are preserved.

### Quality bar (project.md:23)

> Tests pass; the feature works as specced.

Currently both perf tests are skipped; this work must un-skip them with achievable budgets.

### Solution horizon (project.md:21)

User has directed root-cause-first investigation with budget-raise as last resort. The investigation has surfaced a concrete root cause (double Python boot in the wrapper), validating the user's framing.

### Backlog `grep -c` resolution (project.md:42)

Any Done-When/acceptance check using `grep -c "<token>"` must reference tokens in `bin/.events-registry.md` or literal strings under `cortex_command/`.

### Sandbox defense-in-depth (project.md:49)

`git` is sandbox-excluded; allow/deny is sole enforcement. If approach B1 (pure-Python `.git` walk) is taken, the subprocess call to `git` is eliminated — net positive for sandbox surface.

### Adjacent ADRs

- **ADR-0001** (file-based state): JSONL write to `cortex/lifecycle/sessions/<id>/bin-invocations.jsonl` must remain file-based.
- **ADR-0002** (wheel + plugin distribution): the dual-channel wrapper expresses this ADR's commitment to two independent channels.

No new MUST escalations expected — perf work is naturally suited to soft positive routing.

## Tradeoffs & Alternatives

### Approach A — Lazy imports + deferred work (in Python module)

**Description**: Push `subprocess`, `datetime`, `json`, `pathlib` out of module top-level and into the functions that need them. Reorder `main()` to check `LIFECYCLE_SESSION_ID` first, before any heavy import.

**Pros**: Keeps dual-channel contract intact. Aligns with existing `interactive_lock.py:141` and `cli.py:714` patterns. Low risk.

**Cons**: Slight readability cost. Subtle import hazard with `datetime`: must import once at function entry, not on every call.

**Expected savings**: ~5-8ms on no-session-id breadcrumb path; ~2-3ms on full write path. The biggest practical win is the no-session-id path, hit any time a `cortex-*` binstub runs outside a session.

**Alignment**: Strong.

### Approach B — Subprocess-avoidance restructure (in Python module)

**Description**: Replace `subprocess.run(["git", "rev-parse"])` fallback in `_resolve_repo_root()` with a pure-Python walk over `Path.parents` checking for `.git`. Combined with approach A, this removes the `subprocess` import entirely from the happy path.

**Pros**: Eliminates ~6ms `subprocess` import + ~10-20ms `git` fork on fallback path. Existing code already handles `.git` as both directory and file (worktree case).

**Cons**: Edge cases (symlinks, bare repos) need verification — `git rev-parse` handles natively, pure-Python may differ.

**Expected savings**: ~5-10ms import + ~10-20ms on `no CORTEX_REPO_ROOT` fallback.

**Alignment**: Strong — direct continuation of #198's env-var fast-path philosophy.

### Approach C — Replace argparse (not applicable)

**The module doesn't use `argparse`** — it parses `sys.argv` directly. Already done. Zero savings available.

### Approach D — Bash wrapper probe restructure (in bash wrapper) — **CRITICAL**

**Description**: The bash wrapper's wheel-import probe (`python3 -c "import ..."`) is the single biggest source of overhead — it costs a full ~20ms Python boot just to test importability. Two sub-options:

- **D1 — Cache the probe result**: write `~/.cache/cortex/wheel-probe-success` (with version stamp) and `stat` it. Skip the probe when the marker is fresh. Invalidation tied to `uv tool install --reinstall`.
- **D2 — Reorder branches**: put the cheap `grep pyproject.toml` working-tree check BEFORE the expensive `python3 -c` probe. Dogfooders skip the probe entirely. Wheel-only users still pay the probe, but they typically lack a `pyproject.toml` so branch (c) no-ops cheaply.

**Pros**: Largest single addressable win — ~15-20ms on dogfooder path, ~5ms (with cache) on wheel-only path. Preserves all three branches and `CORTEX_COMMAND_FORCE_SOURCE` semantics.

**Cons**: D1 adds cache-invalidation surface. D2 may subtly change which branch fires for which user — needs careful testing across all four parity-test scenarios.

**Expected savings**: ~15-20ms on dogfooder path; ~5ms on wheel-only path (with D1 cache hit).

**Alignment**: Strong — preserves the ADR-0002 dual-channel contract while addressing the actual cost source.

### Out-of-scope approaches (noted, not pursued)

- **E. Bash fast-path** — user-excluded.
- **G. Daemon / persistent helper** — multi-week project; not warranted.
- **H. Rewrite in Rust/Go** — same.

### Fallback F — Raise the perf budget

Triggers after approaches A + B + D are measured. New budget will be set at the post-optimization floor with headroom for CI noise.

### Recommended sequence

1. **Profile first** (data already collected). Top three sources: bash wrapper's wheel-probe (~20ms, addressable), `subprocess`/`pathlib` eager imports (~9ms combined, addressable), Python interpreter boot (~20ms, intrinsic floor).
2. **First attempt — A + B1**: lazy-import `subprocess`/`pathlib`/`json`/`datetime` inside `main`; replace `subprocess.run(["git", "rev-parse"])` with pure-Python parent walk; reorder `main()` to check `LIFECYCLE_SESSION_ID` first. Expected: ~8-12ms saved on no-session path, ~5-8ms on full path.
3. **Second attempt — D1 or D2**: cache wheel-probe success OR reorder branches in bash wrapper. Expected: ~15-20ms saved on dogfooder path.
4. **Measure new floor; raise budget; un-skip tests** (approach F). Post-optimization estimate: ~25-35ms p50. Budget at ~50ms p50 / ~60ms mean / ~80ms p95 (with headroom for CI noise).
5. **Rethink the delta test**: `test_log_invocation_fast_path_faster_than_slow` premise (two structurally different paths) no longer holds. Either replace with a new invariant (e.g., "no probe invoked twice") or remove.

## Open Questions

The following questions are deferred to the Spec phase and will be resolved with the user there:

- **Deferred: which D variant — cache (D1) vs reorder (D2)?** Will be resolved in Spec by asking the user. D2 has lower complexity (no cache invalidation) but may shift which branch fires for some users; D1 has higher complexity but preserves branch ordering exactly. The user's "profile-and-iterate, bounded" guidance suggests trying the simpler D2 first and falling back to D1 if D2 doesn't deliver.
- **Deferred: replace the fast-vs-slow delta test, or remove it?** Will be resolved in Spec by asking the user. The original invariant (catching silent fall-through between bash branches) is structurally gone. Possible new invariant: "no Python boot occurs twice per invocation" (asserted via `time python3 -c "pass"` × 2 ≤ wrapper wall time). Or just delete the test.
- **Deferred: verify pure-Python `.git` walk parity** with `git rev-parse --show-toplevel`. Will be resolved during Implementation by running existing parity tests under both implementations. Edge cases to verify: worktrees (handled), bare repos, symlinked CWD, `.git` files pointing to pruned gitdirs. Existing parity test (`tests/test_cortex_log_invocation_parity.py`) should catch behavioral drift.
- **Deferred: budget numerics**. Will be resolved in Spec by asking the user after measuring the post-optimization floor. Headroom recommendation: 50ms p50 / 60ms mean / 80ms p95. Subject to CI hardware variance — may need to be loosened if CI runners are slower than the dev hardware.
