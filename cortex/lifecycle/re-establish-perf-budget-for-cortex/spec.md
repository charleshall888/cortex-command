# Specification: re-establish-perf-budget-for-cortex

## Problem Statement

Commit `7c05529c` replaced `bin/cortex-log-invocation`'s bash shim with a dual-channel Python wrapper, regressing per-invocation cost from ~9ms (cooperating-parent fast path) to ~70ms p50. The two perf tests at `tests/test_log_invocation_perf.py` were `@pytest.mark.skip`'d because their budgets (p50 ≤ 15ms) and delta invariant are unachievable against the new wrapper. Local developer perf-regression detection is gone. Research surfaced that the dominant addressable cost is **not** the Python module (which is ~30ms) but the **bash wrapper's wheel-import probe**, which spawns a full Python interpreter just to test importability — producing ~20ms of double Python boot on every invocation. This work restores local perf-regression detection (developer `just test` cadence) for this wrapper by optimizing the Python module (lazy imports + subprocess-avoidance), reordering bash wrapper branches so dogfooders skip the probe, and re-establishing a perf budget that reflects the new measured floor. CI does not currently run pytest; wiring this test into CI is explicitly out of scope and is a follow-on concern.

## Phases

- **Phase 1: Python module optimization** — defer imports of `datetime`, `json`, `pathlib`; remove `subprocess` import by replacing `git rev-parse` with a pure-Python repo-root resolution; reorder `main()` so the `LIFECYCLE_SESSION_ID` early-return runs before any heavy import.
- **Phase 2: Bash wrapper branch reorder** — move the working-tree `pyproject.toml` cortex-command predicate check before the wheel-import probe; preserve `CORTEX_COMMAND_FORCE_SOURCE=1` semantics and all three operative branches; add a structural ordering test.
- **Phase 3: Perf-test re-establishment** — un-skip `test_log_invocation_fast_path_budget` with budgets reflecting the measured post-optimization floor plus dev-hardware variance headroom; delete `test_log_invocation_fast_path_faster_than_slow` (its premise is structurally gone); extend parity-test fixtures to bind the invariants the deleted test was implicitly defending; fix the `_p95` percentile-rank computation.

## Requirements

1. **Defer top-level Python imports in `cortex_command/log_invocation.py` and remove `subprocess`**: `datetime`, `json`, `pathlib` become function-scoped in the paths that need them. `subprocess` is no longer imported anywhere in the module (Requirement 2 removes its only call site). Top-of-module imports reduce to what every code path uses (`os`, `sys`). Acceptance: (a) `python3 -c "import cortex_command.log_invocation; import sys; assert 'subprocess' not in sys.modules"` exits 0, and (b) `python3 -X importtime -m cortex_command.log_invocation 2>&1 | grep -E '^import time:.*(datetime|pathlib|json)' | head -1` — match indicates an import occurred (location-of-import is verified by code review against Requirement 3). **Phase**: Phase 1: Python module optimization

2. **Replace `git rev-parse` subprocess with pure-Python repo-root resolution in `_resolve_repo_root()`**: When `CORTEX_REPO_ROOT` is unset or invalid, walk `[Path.cwd(), *Path.cwd().parents]` searching for a `.git` directory or file (worktree case). Note: the walk explicitly includes `Path.cwd()` itself as the first candidate. The `subprocess` module call is removed. Existing parity test `tests/test_cortex_log_invocation_parity.py` continues passing after the fixture extensions in Requirement 11. Acceptance: `just test -- tests/test_cortex_log_invocation_parity.py tests/test_log_invocation_perf.py` exits 0. **Phase**: Phase 1: Python module optimization

3. **Reorder `main()` so `LIFECYCLE_SESSION_ID` check runs before heavy imports**: The `no_session_id` breadcrumb early-return path must not pay for any deferred import that the breadcrumb itself does not use. Acceptance: read `cortex_command/log_invocation.py:main()` and confirm the `LIFECYCLE_SESSION_ID` env-var check is the first executable statement after function entry, before any conditional `import` statement. (Form (b): observable state, specific file path, named pattern.) **Phase**: Phase 1: Python module optimization

4. **Reorder `bin/cortex-log-invocation` branches**: Branch order becomes (a) `CORTEX_COMMAND_FORCE_SOURCE=1` → working-tree exec; (b) `grep -q '^name = "cortex-command"' "${_root}/pyproject.toml"` succeeds → working-tree exec; (c) wheel-import probe (`python3 -c "import cortex_command.log_invocation"`) succeeds → wheel exec; (d) exit-2 remediation. The branch (b) predicate is the existing cortex-command name-match, not a file-presence check; reordering must not weaken it. The cheap `grep` check fires before the expensive `python3 -c` probe. Acceptance: `bash -n bin/cortex-log-invocation` exits 0 (syntax valid) — the durable structural assertion is Requirement 12. **Phase**: Phase 2: Bash wrapper branch reorder

5. **Preserve `CORTEX_COMMAND_FORCE_SOURCE=1` semantics across reorder**: When `CORTEX_COMMAND_FORCE_SOURCE=1` is set, the wrapper execs `python3 -m cortex_command.log_invocation` against the working-tree module — same behavior as today, regardless of wheel presence or working-tree pyproject match. Acceptance: a parity-test fixture exercises this path (covered by the existing `force-source` parity branch in `tests/test_cortex_log_invocation_parity.py`); `just test -- tests/test_cortex_log_invocation_parity.py` exits 0. **Phase**: Phase 2: Bash wrapper branch reorder

6. **Regenerate plugin mirror**: `plugins/cortex-core/bin/cortex-log-invocation` is auto-regenerated by the dual-source pre-commit hook and matches `bin/cortex-log-invocation` byte-for-byte. Acceptance: `diff bin/cortex-log-invocation plugins/cortex-core/bin/cortex-log-invocation` exits 0. **Phase**: Phase 2: Bash wrapper branch reorder

7. **Un-skip `test_log_invocation_fast_path_budget` with new budgets calibrated to dev hardware**: Remove the `@pytest.mark.skip` decorator. Set new thresholds reflecting the post-optimization measured floor plus dev-hardware run-to-run variance headroom. The numerics are set during Phase 3 implementation against actual measurement; the recommended starting point is **p50 ≤ 50ms, mean ≤ 60ms, p95 ≤ 80ms**. The test gates local developer `just test` runs; CI does not invoke pytest today (see Non-Requirements). Budgets must leave headroom for natural dev-hardware run-to-run variance but must be tight enough to detect a ~20ms regression (the cost of a single accidental Python boot). Acceptance: `just test -- tests/test_log_invocation_perf.py::test_log_invocation_fast_path_budget` exits 0 on dev hardware (Apple Silicon). **Phase**: Phase 3: Perf-test re-establishment

8. **Delete `test_log_invocation_fast_path_faster_than_slow`**: Its premise — that fast and slow paths are structurally distinct in the bash shim — is gone after the dual-channel wrapper rewrite. Delete the test function. The invariants it implicitly defended (env-var honored, branches don't silently converge) are covered by Requirements 11 and 12. Acceptance: `grep -c "test_log_invocation_fast_path_faster_than_slow" tests/test_log_invocation_perf.py` = 0. **Phase**: Phase 3: Perf-test re-establishment

9. **Parity test continues passing under extended fixtures**: After Requirement 11 extends the fixture set, `tests/test_cortex_log_invocation_parity.py` must pass. Acceptance: `just test -- tests/test_cortex_log_invocation_parity.py` exits 0. **Phase**: Phase 3: Perf-test re-establishment

10. **Document the bash-wrapper branch reorder in the file's header comment**: Add a comment block at the top of `bin/cortex-log-invocation` explaining why the working-tree check precedes the wheel probe and warning future readers not to re-order it back. Acceptance: read `bin/cortex-log-invocation` and confirm a comment block of ≥3 lines at the top documents the reorder rationale (must mention both "wheel-probe cost" and "double Python boot" as literal substrings; verifiable via `grep -c "wheel-probe cost" bin/cortex-log-invocation` ≥ 1 and `grep -c "double Python boot" bin/cortex-log-invocation` ≥ 1). **Phase**: Phase 2: Bash wrapper branch reorder

11. **Extend parity-test fixtures to bind invariants the deleted delta test was implicitly defending**: Add three fixture variants to `tests/test_cortex_log_invocation_parity.py`: (a) `.git` constructed as a *file* (worktree case) — pointing to a synthetic gitdir, asserts the parent walk accepts both forms; (b) `CORTEX_REPO_ROOT` set to a path containing a real `.git`, with CWD at a *different* path that ALSO contains a `.git` — asserts the env-var is honored (output's repo_root matches `CORTEX_REPO_ROOT`, not CWD); (c) `CORTEX_REPO_ROOT` unset, CWD at a path whose subtree contains no `.git` and no parent contains `.git` — asserts the `no_repo_root` breadcrumb fires. Acceptance: `grep -c 'def test_.*_worktree_file' tests/test_cortex_log_invocation_parity.py` ≥ 1, `grep -c 'def test_.*_env_var_honored' tests/test_cortex_log_invocation_parity.py` ≥ 1, `grep -c 'def test_.*_no_repo_root_breadcrumb' tests/test_cortex_log_invocation_parity.py` ≥ 1, and `just test -- tests/test_cortex_log_invocation_parity.py` exits 0. **Phase**: Phase 3: Perf-test re-establishment

12. **Add structural ordering test for bash wrapper branches**: New test in `tests/test_log_invocation_perf.py` (or a new `tests/test_log_invocation_branch_order.py` file) reads `bin/cortex-log-invocation` and asserts the line number of the working-tree `grep -q '^name = "cortex-command"'` predicate is less than the line number of the `python3 -c "import cortex_command.log_invocation"` wheel-probe predicate. This is a structural file-content test, not a timing test. Acceptance: `grep -c 'def test_bash_branch_order' tests/test_log_invocation_perf.py tests/test_log_invocation_branch_order.py 2>/dev/null` ≥ 1, and `just test` exits 0. **Phase**: Phase 2: Bash wrapper branch reorder

13. **Fix `_p95` percentile-rank computation in `tests/test_log_invocation_perf.py`**: The current implementation computes `int(0.95 * n) - 1` which yields the ~92nd percentile for n=25. Replace with a correct nearest-rank percentile (e.g., `sorted_samples[min(n - 1, int(math.ceil(0.95 * n)) - 1)]`) or equivalent. Acceptance: read `tests/test_log_invocation_perf.py` and confirm `_p95` uses `math.ceil` (or equivalent correct rank), verifiable via `grep -c 'math.ceil' tests/test_log_invocation_perf.py` ≥ 1. **Phase**: Phase 3: Perf-test re-establishment

## Non-Requirements

- **Do not reintroduce a bash fast-path**. The bash wrapper must still always exec into Python on every invocation. Probe-cache approaches (`~/.cache/cortex/wheel-probe-success`) and bash-only telemetry writes are explicitly excluded.
- **Do not pursue Python 3.15 PEP 810 lazy-import syntax**. Targets a Python version this project does not yet require.
- **Do not introduce a daemon, persistent helper, or Unix-socket aggregator**. Out of scope.
- **Do not rewrite in a compiled language** (Rust/Go). Out of scope.
- **Do not modify Branch (d) remediation path**. Per ticket #264 §Edges, that path's perf is not the concern.
- **Do not change the JSONL output schema**, file path, or fail-open contract. The byte-equivalence parity test must continue passing.
- **Do not skip `site.py`** (Python `-S` flag). Risks venv path-setup breakage; not warranted for the marginal savings.
- **Do not replace `argparse`**. The module already uses hand-rolled `sys.argv` parsing — there is no `argparse` to remove.
- **Do not wire the perf test into CI in this work**. CI (`.github/workflows/validate.yml`) does not currently invoke pytest. Adding CI wiring is a follow-on concern, scoped separately. This work restores local-developer perf-regression detection only.
- **Do not implement honoring of `GIT_DIR`, `core.worktree`, or other git config redirections in the pure-Python repo-root resolution**. These are accepted divergences from `git rev-parse --show-toplevel` and are documented in Changes to Existing Behavior. If a user invokes `cortex-log-invocation` from a context where these are set (e.g., inside a git hook with `GIT_DIR` exported), telemetry will emit a `no_repo_root` breadcrumb instead of resolving via git config.
- **Do not bound or "loosen" budgets repeatedly when CI/portability flakes occur**. If the budgets cannot pass on dev hardware after Phase 1+2, do not loosen them iteratively until they stop detecting regressions; instead, re-skip the budget test with a comment pointing to a follow-up backlog item. The budget exists to detect regressions; a budget too loose to detect a ~20ms regression provides no value.

## Edge Cases

- **CWD is the repo root** (the common `cd $(git rev-parse --show-toplevel)` pattern): The walk inspects `Path.cwd()` first (per Requirement 2's explicit `[Path.cwd(), *Path.cwd().parents]` walk), so this case resolves correctly. The previous draft of this spec listed `Path.cwd().parents` which silently excluded CWD itself; the correct walk includes CWD.
- **Worktree case** (`.git` is a file pointing to the gitdir): The pure-Python walk accepts both `.git`-as-directory and `.git`-as-file. The parity test at Requirement 11(a) covers this.
- **`CORTEX_REPO_ROOT` set, points elsewhere than CWD**: Honored — the walk is not invoked. Covered by Requirement 11(b).
- **`CORTEX_REPO_ROOT` unset, no `.git` in CWD or any parent**: `_resolve_repo_root()` returns None; `no_repo_root` breadcrumb fires; fail-open contract honored. Covered by Requirement 11(c).
- **`GIT_DIR` set by git during hook execution**: The pure-Python walk does not honor `GIT_DIR`. If the parent walk finds a `.git` anyway (typical case — hook runs inside the repo working tree), behavior is unchanged. If the walk fails (e.g., CWD is outside the working tree but `GIT_DIR` points back), the `no_repo_root` breadcrumb fires instead of resolving via `GIT_DIR`. This is an accepted divergence from `git rev-parse --show-toplevel`.
- **`core.worktree` config redirecting the working tree**: Same as `GIT_DIR` — not honored, accepted divergence. Bare-repo-plus-detached-worktree setups will breadcrumb `no_repo_root`.
- **Bare repository as CWD**: Walk returns None; `no_repo_root` breadcrumb fires. This is intentional — `cortex-log-invocation`'s telemetry contract requires a working tree, not just a git database. The breadcrumb is the documented graceful-failure mode.
- **macOS CWD under `/var/...` or `/tmp/...`**: `Path.cwd()` returns the physical path (`/private/var/...`) on macOS via `os.getcwd()`. This was previously mis-described in the spec as "preserving logical semantics"; it does not. Accepted: telemetry paths and breadcrumbs may emit `/private/var/...` instead of `/var/...` when CWD is in the macOS `$TMPDIR`. Functionally equivalent for filesystem operations; minor cosmetic difference in emitted paths.
- **CI hardware running this test**: Currently not applicable — CI does not invoke pytest. If a follow-on ticket wires CI execution, budgets calibrated to dev hardware (Apple Silicon) will likely fail on standard `ubuntu-latest` runners (Python 3.12 cold-start is ~60-120ms vs. ~20ms on Apple Silicon). The follow-on ticket must address calibration or environment-conditional budgets; this spec does not.
- **`CORTEX_COMMAND_ROOT` env-var explicit override**: Users who have explicitly set this variable to point at a working tree they want as the source-of-truth will see their override's precedence flip — the wrapper now finds the working-tree pyproject.toml before probing for a wheel. This is intentional alignment with `CORTEX_COMMAND_FORCE_SOURCE`'s spirit (working-tree dogfooding). Documented in Changes to Existing Behavior.
- **Editable install** (`uv pip install -e .` or `pip install -e .`): Behaviorally equivalent to a working-tree dogfooder. The pyproject-match branch fires first; the resulting Python module is the same one resolved by the wheel branch (since editable installs symlink site-packages back to the working tree), so observable behavior is unchanged. The resolution *path* differs (PYTHONPATH injection vs. site-packages); documented in Changes to Existing Behavior for completeness.

## Changes to Existing Behavior

- **MODIFIED**: `bin/cortex-log-invocation` branch evaluation order — working-tree `pyproject.toml` cortex-command predicate check now fires before the wheel-import probe. User-visible effect: when a user is inside a cortex-command clone (the script's `_root` resolves via `dirname "$0"/..` or explicit `CORTEX_COMMAND_ROOT` to a directory whose `pyproject.toml` contains `^name = "cortex-command"`) AND has the wheel installed AND has NOT exported `CORTEX_COMMAND_FORCE_SOURCE`, they now get the working-tree module instead of the wheel. Editable installs follow the working-tree path (was: wheel) — functionally equivalent module, different resolution mechanism.
- **MODIFIED**: `CORTEX_COMMAND_ROOT` precedence — when set explicitly, it now triggers the working-tree branch before the wheel probe (was: it could be set without triggering working-tree because the wheel probe ran first). Aligns with `CORTEX_COMMAND_FORCE_SOURCE`'s working-tree-dogfooder spirit.
- **MODIFIED**: `cortex_command/log_invocation.py` no longer calls `git rev-parse --show-toplevel`. The `subprocess` import is removed entirely from the module. The fallback path now uses a pure-Python `[Path.cwd(), *Path.cwd().parents]` walk for `.git`.
- **MODIFIED**: `_resolve_repo_root()` no longer honors `GIT_DIR` or `core.worktree` git config. Where `git rev-parse --show-toplevel` would have returned a non-CWD-derived path under those settings, the pure-Python walk now returns the directory containing `.git` (or `None`, falling through to `no_repo_root` breadcrumb).
- **MODIFIED**: On macOS, telemetry repo-root and breadcrumb paths may emit physical (`/private/var/...`) rather than logical (`/var/...`) paths when CWD is in a symlinked filesystem (notably `$TMPDIR`). Functionally equivalent for downstream consumers.
- **MODIFIED**: Bare repositories no longer resolve to a repo root via `_resolve_repo_root()`. Previously, `git rev-parse --show-toplevel` would have succeeded on a bare repo (though returning the gitdir itself); the parent walk does not. `no_repo_root` breadcrumb fires.
- **REMOVED**: `test_log_invocation_fast_path_faster_than_slow` test function. Replaced by Requirement 11's positive env-var-honored fixture and Requirement 12's structural branch-order assertion.
- **ADDED**: Header comment block at the top of `bin/cortex-log-invocation` documenting why the working-tree check precedes the wheel probe (Requirement 10).
- **ADDED**: Three new parity-test fixtures binding the invariants previously left to the delta test (Requirement 11).
- **ADDED**: Structural branch-order test for the bash wrapper (Requirement 12).

## Technical Constraints

- **Dual-channel contract is load-bearing** (`cortex/requirements/project.md:38`, ADR-0002). All three operative branches (force-source, working-tree, wheel) must remain; only reorder is permitted. The branch (b) predicate must remain the cortex-command name match (`^name = "cortex-command"`), not be weakened to file-presence.
- **`CORTEX_COMMAND_FORCE_SOURCE=1` semantics must be preserved verbatim**. When set, the wrapper bypasses both the working-tree pyproject check and the wheel probe and execs the working-tree module directly. Acceptance for Requirement 5 verifies this.
- **File-based state** (ADR-0001). JSONL writes to `cortex/lifecycle/sessions/<id>/bin-invocations.jsonl` remain file-based.
- **Fail-open contract**. Every code path returns exit 0 even on internal error; failures emit breadcrumb categories (`no_session_id`, `no_repo_root`, `session_dir_missing`, `write_denied`, `other`) to `~/.cache/cortex/log-invocation-errors.log`.
- **Byte-equivalence parity test must hold**. `tests/test_cortex_log_invocation_parity.py` exercises the JSONL field-order contract; passes after Requirement 11's fixture extensions.
- **Sandbox defense-in-depth**. Removing the `subprocess` call to `git` is a net positive for sandbox surface. No new sandbox-excluded commands introduced.
- **Dual-source enforcement**. `plugins/cortex-core/bin/cortex-log-invocation` is auto-regenerated by the pre-commit hook; do not hand-edit.
- **Backlog grep -c resolution** (`cortex/requirements/project.md:42`). All `grep -c` acceptance criteria in this spec reference literal strings present in the codebase or this spec — no hallucinated tokens.
- **Structural separation over prose-only enforcement** (`CLAUDE.md` skill authoring guidelines). Requirement 4's branch ordering is a sequential gate; Requirement 12 provides the structural assertion. Requirement 1's subprocess-absence claim is bound to a positive `sys.modules` check, not just prose.
- **Local-only regression detection**. CI does not invoke pytest today (`.github/workflows/validate.yml`); this work restores local developer `just test` regression detection only. Wiring CI execution of this test is explicitly deferred to a follow-on backlog item.

## Open Decisions

None. The two consequential design choices raised in research (bash-wrapper fix variant; fast-vs-slow delta test fate) were resolved during the interview. The critical-review-surfaced issues were resolved during spec revision: parity defects of the pure-Python walk are documented as accepted divergences in Changes to Existing Behavior; the bash branch ordering gets a structural test (Req 12); the budget is calibrated to dev hardware with explicit acknowledgment that CI wiring is out of scope; `_p95` correctness is fixed (Req 13).

## Proposed ADR

None considered. The dual-channel contract is already governed by ADR-0002; the branch reorder is an implementation detail consistent with that ADR's intent. No new architectural decision is introduced.
