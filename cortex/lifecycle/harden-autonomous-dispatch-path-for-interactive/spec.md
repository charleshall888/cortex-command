# Specification: harden-autonomous-dispatch-path-for-interactive

## Problem Statement

`/cortex-core:lifecycle implement` → "Implement in autonomous worktree" launched from an interactive Claude Code session fails three times in succession before reaching the first task dispatch. Each failure is a different symptom of the same root cause: `skills/lifecycle/references/implement.md` §1a hardcodes a runtime contract that the launchd-fired runner satisfies but the Bash-tool–invoked dev path does not — auth (vector-resolution policy diverges between `runner.py` and `daytime_pipeline.py`), sandbox (the worktree-root escape hatch exists but is never wired into the dispatch skill), and Python env (~13+ modules invoked as bare `python3 -m cortex_command.<x>` rely on the `uv tool install` venv being on path, which the dev-clone Bash-tool path doesn't satisfy). This lifecycle hardens the autonomous-dispatch path so both invocation contexts reach the first task dispatch reliably, replaces the in-flight soft-fall-through patches with probe-based solutions, and adds a parity test surface (both stub-SDK CI regression freeze and opt-in real-launchd matrix) that prevents the launchd-only test bed from masking Bash-tool regressions in the future. Without this work, interactive autonomous dispatch is blocked: the operator either disables sandbox manually (eliminating the per-spawn defense-in-depth security surface for autonomous execution) or falls back to scheduled launchd dispatches, losing the daytime collaborative iteration loop entirely.

## Phases

- **Phase 1: Auth resolution + probe** — extend `ensure_sdk_auth`, add stdlib-only Keychain presence probe, emit `auth_probe` events, converge `runner.py` and `daytime_pipeline.py` on identical resolution + fail policy.
- **Phase 2: Sandbox-friendly worktree wiring** — single `resolve_worktree_root()` resolver consulted by both `cortex init` registration and dispatch-time creation; `probe_worktree_writable()` failing fast with diagnostic for sandbox-blocked roots and hardcoded-deny cases.
- **Phase 3: Console-script sweep (F3)** — audit every `python3 -m cortex_command.*` callsite, promote each to `pyproject.toml` `[project.scripts]`, update skill/hook/doc/justfile/test callsites, document the dev-clone install path.
- **Phase 4: Dispatch preflight + parity test** — fuse the three probes into `verify_dispatch_readiness()` called from `run_daytime` Phase A; add `tests/test_dispatch_parity.py` (stub-SDK CI) and `just test-dispatch-parity-launchd-real` (opt-in real-environment); produce the sandbox preflight artifact.

## Non-Requirements

- **No standalone `cortex doctor` CLI subcommand.** The readiness fuse lives only inside `daytime_pipeline.run_daytime` Phase A (Requirement 18). A standalone operator-facing `cortex doctor --dispatch-readiness` is filed separately if operators want explicit pre-flight invocation outside the dispatch context.
- **No `dangerouslyDisableSandbox` runtime toggle exposed by cortex-command.** The escape hatch remains operator-supplied in their own settings. Documenting it as a workaround for the consumer-repo `.vscode/`/`.idea/` constraint (Requirement 11) is documentation only — cortex-command does not wire any toggle for it.
- **No per-batch-dispatch auth re-probing.** Multi-hour drift (Keychain re-lock, OAuth token expiry mid-session) surfaces as ordinary SDK errors in the running session. The probe is startup-only. If multi-hour drift becomes a recurring failure mode, per-batch re-probing is filed as a follow-up.
- **No PyPI publication.** Per `cortex/requirements/project.md` scope; this lifecycle does not change distribution model.
- **No fix of consumer-repo `.vscode/`/`.idea/` hardcoded-deny failure in code.** Documented as a constraint (Requirement 11) and surfaced via `probe_worktree_writable()` diagnostic (Requirement 8). Consumer-repo durability for this failure class is out-of-scope; if a consumer hits it, the documented workarounds apply.
- **No change to per-spawn sandbox-settings tempfile contract or `cortex_command/pipeline/dispatch.py` sandbox-dict shape.** F2 is filesystem-side (worktree root + `allowWrite` registration), strictly complementary to the sandbox-side enforcement.
- **No `--during-dev` escape hatch on the in-flight install guard.** Dev iteration loop is `uv pip install -e . --no-deps` against the active `.venv` (Requirement 16). The existing carve-outs (pytest, `CORTEX_RUNNER_CHILD=1`, dashboard, cancel-force) are unchanged.
- **No probe-result caching or memoization.** Probes run on every dispatch; staleness is preferable to silent reliance on a cached "yes" from a previous environment.

## Edge Cases

- **First-after-reboot Keychain unlock (or launchd pre-login-unlock)**: `probe_keychain_presence()` (no `-w`) does NOT prompt for ACL on a locked keychain. If the search list is unavailable (Darwin-but-locked context), the probe returns `"unavailable"` and the policy (R3) treats this as continue-with-soft-warning, not startup failure. The SDK call within the dispatched session surfaces ordinary auth errors if unlock-required state actually blocks credential retrieval.
- **macOS Keychain v2.0.14 service-name aliasing**: Probe checks only the canonical `"Claude Code-credentials"` name. A user authenticated under a buggy version may have a stale entry there with the live entry under the alias `"Claude Code"` (or vice versa). The probe cannot disambiguate this — the SDK reads from a specific name and may get stale data; surfaces as an ordinary SDK auth error in the running session, not at probe time. The probe's bounded purpose (entry-absent detection) is not extended to alias-validity inference. Affected users update Claude Code to v2.0.15+ (which fixes the write-side bug) and re-auth to converge entries.
- **ACL trust mismatch after Claude Code upgrade**: After a Claude Code binary upgrade, the keychain entry's ACL trust list may no longer include the new binary's code signature. `probe_keychain_presence()` still returns `"present"` (the entry exists), but the SDK call inside the dispatched session fails with `errSecAuthFailed`/`errSecInteractionNotAllowed`. The probe is informational (R3) — this case surfaces downstream as an ordinary SDK auth error, not at startup. Documented as a known limit.
- **Auth state expires mid-session**: Not detected by startup probe; surfaces as ordinary SDK error. Per Non-Requirements.
- **Cross-repo dispatch**: `resolve_worktree_root()` returns `$TMPDIR/overnight-worktrees/{session_id}/{feature}/` per existing convention. `cortex init` registration skipped (cross-repo paths are already sandbox-writable).
- **Operator runs `cortex init` with non-default `CORTEX_WORKTREE_ROOT` set**: Resolver consults env first; the env-set path is registered. Documented in `cortex init` UX text.
- **Operator unsets `CORTEX_WORKTREE_ROOT` after registering one**: Resolver returns default; registration is additive so previous path remains in `allowWrite`. Not removed automatically — operators clean up manually if desired (matches existing `cortex init` registration semantics).
- **Console-script name collision**: An entry like `cortex-daytime-pipeline` collides with a pre-existing binary on PATH. Impl-time check: each chosen name verified via `command -v` against a clean PATH before commit; collisions block commit. Documented in the audit log (Requirement 12).

The diagnostic appears in `daytime.log`'s `cat` (the skill captures launch-line output to stderr). CHANGELOG.md (Requirement 17) documents the install-time requirement. `verify_dispatch_readiness()` does NOT check console-script PATH presence (rationale in R18) — the launch-time failure surface is the correct enforcement point.
- **In-flight install guard blocks `--reinstall` during partial active session**: Dev workflow uses `uv pip install -e . --no-deps` against `.venv` (Requirement 16); installed-tool refresh happens between sessions or with the existing `CORTEX_ALLOW_INSTALL_DURING_RUN=1` inline bypass.
- **Tracked `.vscode/` or `.idea/` in consumer repo**: `probe_worktree_writable()` (Requirement 8) catches the `git worktree add` failure; stderr names the likely cause and the documented workarounds (Requirement 11).
- **Audit-list grows post-merge**: A new module added to `cortex_command/` that uses `python3 -m` invocation re-introduces the F3 failure mode. Mitigation: the recurring `bin/cortex-check-parity --audit-bare-python-m-callsites` gate (wired by Requirement 12) re-runs the F14 grep over the full path set. Morning-review or a manual `just check-bare-python-callsites` invocation surfaces violations against `bin/.audit-bare-python-m-allowlist.md` (closed-enum + ≥30-char rationale per entry). Drift cannot reach an operator's dispatch silently.
- **Stub-SDK test catches only named regressions**: The acceptance-bar honesty docstring (Requirement 21) makes this explicit. Future Claude Code updates that change the sandbox profile or Bash-tool env-passing pass the CI test but would fail `just test-dispatch-parity-launchd-real`.
- **launchd job execution after this lifecycle**: The launchd-fired runner continues to satisfy the runtime contract; the parity test (Requirement 19) ensures the Bash-tool path matches. No launchd-side behavioral changes.
- **Worktree-root registration overlaps with existing manual `allowWrite` entries**: `settings_merge.register_path` is additive and idempotent; existing manual entries are preserved.

## Technical Constraints

- **Stdlib-only constraint on `cortex_command/overnight/auth.py`** (line 2 docstring): pre-venv invocability is load-bearing (the module runs as `python3 -m cortex_command.overnight.auth --shell` before `.venv` is on path during runner startup). All new auth code — including `probe_keychain_presence()` — MUST be stdlib-only. No new dependencies.
- **Sandbox preflight gate** (`bin/cortex-check-parity`) trips on F1+F2+F3-touched sandbox-source files (`cortex_command/overnight/sandbox_settings.py`, `cortex_command/pipeline/dispatch.py`, `cortex_command/overnight/runner.py`, `pyproject.toml`). `preflight.md` (Requirement 22) is required to commit.
- **SKILL.md-to-bin parity gate** governs the new entry-point names. Each name needs either wiring references (callsite edits in Requirement 14 provide these) or `.parity-exceptions.md` rows with closed-enum category + ≥30-char rationale.
- **Two-mode gate pattern** (`--staged` / `--audit`) is the model for any new gate added by this work. If a future audit-mode check is wired for the audit-list (per Edge Cases), it follows this pattern.
- **Dual-source plugin mirror**: edits land in `skills/`, `hooks/`, `bin/` only. Pre-commit hook auto-regenerates `plugins/cortex-core/{skills,hooks,bin}/`. `just setup-githooks` enables this.
- **MUST-escalation policy** (`CLAUDE.md`): default to soft positive-routing for any new SKILL.md prose authored by this lifecycle. Adding MUST/CRITICAL/REQUIRED phrasing requires F-row evidence + `effort=high` attempt + linked commit-body/PR artifact.
- **Distribution model is `uv tool install git+<url>@<tag>` non-editable wheel.** F3's expansion to first-class dev-clone support is a deliberate scope expansion (per `cortex/requirements/project.md`'s "secondary path" wording). This lifecycle does not change the primary install command; it makes the dev-clone path explicit in skill prose and adds the `uv pip install -e . --no-deps` dev-loop documentation.
- **Per-spawn sandbox enforcement contract** is load-bearing for autonomous-execution security. F2 is filesystem-side (worktree root + `allowWrite`); MUST NOT bypass the per-spawn sandbox-settings tempfile contract in `cortex_command/pipeline/dispatch.py`.
- **In-flight install guard carve-outs** unchanged: pytest, `CORTEX_RUNNER_CHILD=1`, dashboard, cancel-force, `CORTEX_ALLOW_INSTALL_DURING_RUN=1` inline. F3 does not add new carve-outs; the dev-loop uses `uv pip install -e . --no-deps` against the existing `.venv` instead.
- **Atomic state writes** (tempfile + `os.replace()`) apply to any new state files this work introduces. `~/.claude/settings.local.json` registration goes through existing `settings_merge` machinery (already serialized via `fcntl.flock`).
- **Workflow-trimming convention**: removing the uncommitted `daytime_pipeline.py` soft-fall-through patch is hard-deletion with a `CHANGELOG.md` entry, not a deprecation notice.
- **Probe latency budget**: `verify_dispatch_readiness()` must complete in <500ms on a warm machine and <2s on first-after-reboot. Keychain presence probe (no `-w`, exit-code-only) is sub-100ms typical. Worktree-writable probe (no-op file create + `git worktree add` to `$TMPDIR`) is ~200-500ms. SDK importability check is <50ms. Budget is structural, not measured — if it exceeds 2s in dispatch logs over a session, file a follow-up.
