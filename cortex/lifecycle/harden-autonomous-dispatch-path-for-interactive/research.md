## Web Research

### F1 — Claude CLI auth resolution + Keychain probe semantics

The `claude` CLI's documented auth-resolution order (https://code.claude.com/docs/en/authentication):

1. Cloud provider env vars (`CLAUDE_CODE_USE_BEDROCK` / `_VERTEX` / `_FOUNDRY`)
2. `ANTHROPIC_AUTH_TOKEN` (sent as `Authorization: Bearer`)
3. `ANTHROPIC_API_KEY` (sent as `X-Api-Key`) — in `-p` non-interactive mode, used unconditionally when present
4. `apiKeyHelper` settings-defined script output
5. `CLAUDE_CODE_OAUTH_TOKEN` env var (long-lived, generated by `claude setup-token`, ~1 year)
6. Subscription OAuth from `/login` (Keychain on macOS under service name **`"Claude Code-credentials"`**; `~/.claude/.credentials.json` on Linux/Windows)

Key takeaways:
- **Pre-resolution in parent + env export is the robust pattern.** Generate a `CLAUDE_CODE_OAUTH_TOKEN` once via `claude setup-token`, then export it to the dispatched subprocess env. This sidesteps the Keychain GUI-context dependency that breaks under SSH, launchd, and (likely) Claude Code's Bash-tool subprocess.
- **Subprocess Keychain failure pattern**: over SSH or launchd-spawned subprocesses, `security` returns `errSecInteractionNotAllowed` (exit 36) silently (https://github.com/anthropics/claude-code/issues/9403, #44089). The bash wrapper `security unlock-keychain ~/Library/Keychains/login.keychain-db` works in-session but is a foot-gun.
- **Known v2.0.14 bug**: Claude itself reads `"Claude Code"` (no suffix) while storing as `"Claude Code-credentials"` (https://github.com/anthropics/claude-code/issues/9403). Probe must handle both names.
- `--bare` mode does NOT read `CLAUDE_CODE_OAUTH_TOKEN` — relevant if dispatch uses bare mode.
- `claude auth status` and `/status` introspection are referenced in support docs but have known flakiness (issue #8002).

**Anti-pattern**: relying on Keychain probe from a subprocess is fragile across SSH, launchd, and Bash-tool contexts. Pre-resolve in parent.

### F2 — Seatbelt sandbox + worktree write paths

`sandbox.filesystem.allowWrite` semantics (https://code.claude.com/docs/en/sandboxing):
- Shape: `{"sandbox": {"enabled": true, "filesystem": {"allowWrite": ["~/.kube", "/tmp/build"]}}}`
- Path prefixes: `/foo` absolute, `~/foo` HOME-relative, `./foo` project-relative (or `~/.claude` for user settings).
- **OS-level enforcement via Seatbelt on macOS** applies to ALL subprocesses spawned inside the sandbox, including `git`, `npm`, `kubectl`.
- **Settings-scope merging is additive** across managed / user / project / local — paths combine, not replace. This is exactly the property cortex-command's existing `cortex init` registration relies on.

**Seatbelt subprocess inheritance** (https://igorstechnoclub.com/sandbox-exec/, https://newosxbook.com/files/HITSB.pdf): profiles inherit to every child and **cannot be removed from inside**. So a subprocess launched from the Bash tool inherits the parent Claude session's profile — meaning the worktree path must be in the *parent's* effective `allowWrite` list.

**Critical prior art for the F2 failure**:
- https://github.com/anthropics/claude-code/issues/51303 — `.vscode/` and `.idea/` are **hardcoded denies in the Claude Code binary's `_SBX` module**, NOT configurable via settings. User-set `allowWrite` does not override. The half-completed state leaves `.git/worktrees/` and the branch behind; manual `git worktree prune` + `git branch -D` required.
- https://github.com/anthropics/claude-code/issues/39886 — `isolation: "worktree"` silently falls back to running in the main repo. Detection markers: `worktreePath: "done"` (a status string, not a path), `worktreeBranch: undefined`. The parity test should detect this by asserting `pwd != main_repo_path` and `worktreePath` is an absolute path.

### F3 — Python module dispatch + entry points + dev clone vs installed

`uv tool install` semantics (https://docs.astral.sh/uv/concepts/tools/, https://docs.astral.sh/uv/concepts/projects/config/):
- Supports `git+https://…` URLs (cortex-command's existing install pattern).
- `--editable` (`-e`) works with `uv tool install -e .` for local-path editable installs.
- Executables are symlinked into `~/.local/bin` on Unix.
- **Critical gotcha**: changes to `[project.scripts]` in `pyproject.toml` do not take effect until uninstall + reinstall. Existing `v0.1.0` users won't see new entry points without `uv tool install --reinstall`.

**Dual entry-point pattern (idiomatic)** (https://peps.python.org/pep-0338/, https://packaging.python.org/specifications/entry-points/):
- `main()` in a single module (e.g., `package/cli.py`)
- `package/__main__.py` calls `main()` — enables `python -m package`
- `[project.scripts]` entry `package = "package.cli:main"` — enables `package` console script
- Both invocations call the same `main()` — no duplication.

**Detect-and-fallback pattern** (PyTorch ExecuTorch precedent):
```python
if shutil.which("cortex-overnight"):
    cmd = ["cortex-overnight", ...]
else:
    cmd = [sys.executable, "-m", "cortex_command.overnight.daytime_pipeline", ...]
```
Handles both "installed via uv tool install" and "running from dev clone with a venv."

**Anti-pattern**: hardcoding `python3 -m cortex_command.overnight.X` assumes dev-clone layout. Hardcoding the console-script name breaks dev-clone users without the tool installed. Detect-and-fallback is the durable form.

### Parity testing patterns

- `pytest-venv` / `pytest-virtualenv` (https://pypi.org/project/pytest-venv/, https://pypi.org/project/pytest-virtualenv/) — fresh-venv subprocess fixtures.
- **No out-of-the-box pytest isolation from parent env vars** (https://github.com/pytest-dev/pytest/issues/4226). Tests must explicitly scrub `PATH`, `VIRTUAL_ENV`, `CLAUDE_CODE_OAUTH_TOKEN`, etc. via `monkeypatch.delenv` or `subprocess.run(env={...})`.
- **launchd parity**: no off-the-shelf framework. launchctl-spawned processes don't load `~/.zshrc` / `~/.bash_profile`; `PATH` differs from interactive shells. Tests should subprocess with `env -i` or a minimal explicit env dict to simulate the launchd-bare environment.
- **Silent-worktree-fallback assertions** (from #39886): `pwd != main_repo_path`, `git worktree list | wc -l > 1`, `git branch --show-current != main`, dispatch metadata `worktreePath` is an absolute path not `"done"`.

### F1 — auth probe shape

| Option | Probe target | Pros | Cons |
|---|---|---|---|
| **A1 (Keychain `security find-generic-password`)** | macOS Keychain entry for `Claude Code-credentials` | Cheap (~10ms), catches exact failure mode | Darwin-only; couples to implementation detail of where `claude` stores creds; **lies in launchd/SSH context** (Adversarial A2 — returns success in Bash-tool context, fails in launchd, so it hides the divergence we're trying to detect); may prompt for ACL the first time. |
| **A2 (Spawn `claude -p` with timeout)** | The actual binary that will run | Portable; tests the real contract | ~200–500ms; brittle to CLI output changes; **2s timeout is too tight** for first-after-reboot or cold model warmup (Adversarial A1); **may spend real tokens** if vector is `ANTHROPIC_API_KEY`. |
| **A3 (No-op SDK API call with timeout)** | End-to-end auth-to-API | Highest fidelity | Requires network (~200ms+); costs token quota; introduces network dependency in startup path. Unacceptable for fail-fast startup. |
| **A4 (Soft-fall-through + dispatch-time assertion)** | None — just record vector | Cheapest | Doesn't satisfy "real probes" acceptance bar. |

Adversarial review surfaced a stronger alternative: **don't probe at all at dispatch time; pre-resolve auth in the parent and export `CLAUDE_CODE_OAUTH_TOKEN` to the child env.** This sidesteps Keychain-in-subprocess entirely (Agent 2's "robust pattern"). Variant: extend `ensure_sdk_auth`'s resolution chain to include `ANTHROPIC_AUTH_TOKEN` and a stdlib Keychain entry-presence check (`security find-generic-password -s … 2>/dev/null` without `-w` — checks existence without retrieving the secret, avoids ACL prompts), then on success export `CLAUDE_CODE_OAUTH_TOKEN` (or `ANTHROPIC_API_KEY`) to subprocess env.

**Open question — see Open Questions below**: probe vs. pre-resolve-and-export, and whether/how the probe spawns `claude`.

### F2 — sandbox-friendly worktree path

| Option | Mechanism | Pros | Cons |
|---|---|---|---|
| **B1 (Skill sets `CORTEX_WORKTREE_ROOT` in implement.md)** | Prose-only env-prefix | Smallest blast radius | Prose-only enforcement; next skill that calls `daytime_pipeline` silently re-introduces the bug. Violates "structural over prose" guideline. |

| **B4 (Extend `cortex init` to register worktree root in `allowWrite`)** | Reuse `fcntl.flock`-serialized additive `settings_merge.register_path` | Aligns 1:1 with existing per-repo sandbox-registration pattern; structural; preserves default `.claude/worktrees/<feature>` | Requires `cortex init` re-run on existing checkouts; cross-repo work needs per-repo registration. **Critical caveat from Adversarial A4**: install-time-registered path may not match dispatch-time `CORTEX_WORKTREE_ROOT` — drift risk unless a single resolver owns the choice. |

### Parity test surface

**Location**: `tests/test_dispatch_parity.py` (next to `test_lifecycle_phase_parity.py`, `test_lifecycle_kept_pauses_parity.py`).

**Approach**: Subprocess `run_daytime` (or a slice — auth + worktree phases only) under two distinct synthetic env dicts:
1. **Launchd-shaped env**: keys from `cortex_command/overnight/scheduler/macos.py:_OPTIONAL_ENV_KEYS` + `PATH`, mirroring what `launcher.sh` produces.
2. **Bash-tool-shaped env**: capture what Claude Code's Bash tool actually passes (sandbox-clean PATH, no `CORTEX_*` snapshot, no preset `ANTHROPIC_API_KEY`).

**Invariant**: both invocations reach Phase B (post-auth-probe, post-worktree-create) without `startup_failure` against a fixture plan with one trivial task.

**Hermetic via SDK stub**: `cortex_command/tests/_stubs._install_sdk_stub` already exists; reuse so CI doesn't need an API key.

**Optional real-launchd recipe**: `just test-dispatch-parity-launchd-real` for opt-in macOS-local verification via actual `launchctl bootstrap`. Not in CI default.

**Adversarial caveat (A8)**: a stub-SDK synthetic-env test catches *named* regressions (the F1/F2/F3 failure modes) but **freezes the current env shape**. If Claude Code ships an update that changes the sandbox profile or Bash-tool env-passing, the stubbed test still passes. The parity test as scoped is "regression freeze of known failures," not "real dispatch-contract verification" — that's the honest acceptance-bar wording.

## Adversarial Review

The adversarial agent challenged the synthesis. Key findings — most are flagged into Open Questions for Spec to resolve.

### Failure modes and edge cases

- **A1. The `claude -p` 2s timeout is a guess against an unstated contract.** First-after-reboot Keychain unlock can block longer than 2s; cold model warmup is 200ms–3s; sending an empty prompt may spend real tokens; the binary's exit code behavior is patch-release-unstable. Reframed: don't spawn `claude` at all if pre-resolve-and-export works.
- **A2. The Darwin Keychain fast-path lies about the launchd/Bash-tool divergence.** `security find-generic-password` from launchd/SSH context returns `errSecInteractionNotAllowed`; from interactive Terminal.app it succeeds. So a Keychain probe would **agree across the two paths in the wrong way** — would call them "parity'd" when they actually diverge at SDK call time. Mitigation: presence-only check (no value retrieval, no ACL prompt) or pre-resolve in parent and export, sidestepping the subprocess Keychain question.
- **A3. `auth.py`'s priority order does not match the SDK's actual resolution chain.** Missing `ANTHROPIC_AUTH_TOKEN`; misleading "Keychain will save you" message with no Keychain probe. The probe codifies a different question than the SDK answers with. Fix: extend chain to include `ANTHROPIC_AUTH_TOKEN`; probe Keychain presence or remove misleading messaging.
- **A4. F2 install-time registration may not match dispatch-time `CORTEX_WORKTREE_ROOT`.** `cortex init` registers a path; `worktree.py` reads a possibly-different env var at dispatch. Silent denial recurs. Fix: single resolver consulted by both call sites.
- **A5. `.vscode`/`.idea` hardcoded denies block F2 in consumer repos.** This repo has neither, but the autonomous-dispatch path is supposed to work for any repo that runs `cortex init`. Tracked `.vscode/settings.json` (common in Node/Python/Go) breaks `git worktree add` under sandbox regardless of `CORTEX_WORKTREE_ROOT`. Mitigation: add `git worktree add` probe with actionable diagnostic; document constraint.
- **A6. In-flight install guard turns `uv tool install --reinstall` into a hostile dev workflow.** Developers iterating on dispatch fixes typically have a partial active-session.json — every reinstall aborts. Combined with F3's requirement to reinstall for `[project.scripts]` propagation, the F3 dev experience pushes contributors back to bare `python3 -m`. Mitigation: document `uv pip install -e . --no-deps` against the live `.venv` as the dev loop; or add a `--during-dev` escape hatch.
- **A7. F3 entry-point promotion lands in parity-gate scope as expected** (`gather_entry_point_names`), but each of the 13 new names needs wiring-side mentions or `.parity-exceptions.md` rows. Each callsite-rewrite *is* a wiring signal, so this composes — but worth a one-line impl-time sanity check.
- **A8. The parity test as proposed is decorative.** Stub-SDK + synthetic env → catches named regressions only; freezes the current env shape. If Claude Code ships an update changing the sandbox profile or Bash-tool env-passing, the stubbed test still passes. The "Hardened path with parity" acceptance bar should be wording-honest: "regression freeze of named failures + opt-in real-environment verification."
- **A9. One-shot dispatch-time auth probe doesn't survive long-running sessions.** Multi-hour overnight sessions: screen-lock invalidates Keychain unlock state for subsequent probes; OAuth tokens can expire mid-session. Mitigation: probe per-batch-dispatch OR document the probe as startup-only and surface mid-session auth-failure as ordinary SDK error.

### Security concerns / anti-patterns

- **S1. `auth.py` writes credentials into `os.environ`** — inherited by every subprocess. MCP servers spawned by `claude_agent_sdk` inherit the full env including credentials. Pre-existing; the ticket is a reasonable point to audit whether the SDK transport zeros credential vars before launching third-party MCP servers (verify rather than assume).
- **S2. `_invoke_api_key_helper` runs arbitrary shell from `~/.claude/settings.json`** — pre-existing risk; widens if F2 grows what writes to `settings.local.json`. Path-confused write attacks become more relevant.
- **S3. Keychain "fast-path" via `security … -g` may prompt for ACL or fail in launchd context** — probe presence only (no `-w`), and document the one-time per-binary prompt expectation explicitly.

### Assumptions that may not hold

- **U1. Solution-horizon applied selectively.** Ticket lists 4 suspect skills but punts them. The durable F3 is "promote every `python3 -m cortex_command.<x>` callsite," not "13 named ones." Acceptance bar should either extend F3 scope to cover all callsites or explicitly justify the bound with a *why this round and not the others* statement.
- **U2. "Hardened path with parity" treats parity as the test surface, not the runtime invariant.** A test catches regressions in CI; preflight catches user-environment drift. The ticket's punt of `cortex doctor` may be miscaled — runtime preflight is the durable mitigation for the class of failure the ticket targets.
- **U3. `CORTEX_WORKTREE_ROOT` is read at dispatch but set per-invocation by the Bash-tool launch line.** The chosen path needs to be (a) within `allowWrite`, (b) consistent across dispatches so cleanup finds the right place, (c) shared across sessions or per-session-namespaced. Lock the resolved root choice somewhere persistent — likely via the single resolver above.
- **U4. The two-mode-gate precedent may not generalize to dispatch parity.** Phase-parity (code-path-equivalence) ≠ dispatch parity (environment-equivalence). Different invariant shape → different test architecture.

### Mitigations the adversarial review recommends

- Drop the spawning probe; pre-resolve auth in parent + export to child env.
- Single resolver for worktree root (F2 B4 + B3 only sound if drift is eliminated).
- Audit all `python3 -m cortex_command.*` callsites in one sweep — don't carve out arbitrary subsets.
- Wording-honest acceptance bar for the parity test.
- Reconsider `cortex doctor` scope — preflight is the durable mitigation, not just the test.
- Add `--during-dev` escape for the install guard, or document the dev loop pattern.
- Log every probe outcome to `events.log` as `auth_probe` so MUST-escalation F-row evidence accumulates organically.
