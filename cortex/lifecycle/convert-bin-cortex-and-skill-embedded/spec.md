# Specification: convert-bin-cortex-and-skill-embedded

## Problem Statement

Under `uv tool install` distribution, four callsites that invoke `python3 -c "import cortex_command..."` fail because the system `python3` cannot import the per-tool venv's `cortex_command` package. Two bin/* wrappers (`cortex-backlog-ready`, `cortex-morning-review-complete-session`) fall through to a misdirecting "CLI not found" error even when the CLI is correctly installed; two skill files contain four embedded `python3 -c` snippets (`residue-write.md` two snippets, `implement.md` two snippets) that crash with raw `ModuleNotFoundError` inside autonomous Claude execution paths where no human is at the failure surface. This work converts all four skill snippets to dedicated `[project.scripts]` console-scripts, retires the two redundant bash wrappers, extends the telemetry gate to cover the new entry-point modules, and updates the bash consumer-side parsers in skill prose where output formats change.

## Phases

- **Phase 1: Retire bin/* wrappers** — delete the two bash wrappers and their plugin mirrors; address byte-equivalence regression for the two affected `main()` functions.
- **Phase 2: Author and register four skill-helper console-scripts** — create four dedicated modules under `cortex_command/lifecycle/` and `cortex_command/critical_review/` with `main()` entry points; register as `[project.scripts]`. Bootstrap the `cortex_command/critical_review/` package directory.
- **Phase 3: Rewrite skill prose to invoke console-scripts** — update `implement.md` and `residue-write.md`; migrate the bash-side parser where output format changes; update `implement.md:22` self-documenting prose to reference the new structural marker.
- **Phase 4: Extend telemetry gate + parity-test regex** — restructure `bin/cortex-invocation-report --check-entry-points` to iterate (package, module) tuples; extend `tests/test_lifecycle_kept_pauses_parity.py:42`'s marker regex; address `_telemetry` cross-package import.
- **Phase 5: Documentation** — CHANGELOG entry with the `uv tool install --reinstall` remediation note and the `argv_count` record-shape change for the two retired wrappers.

## Non-Requirements

- **No `cortex <group> <verb>` subcommand wiring added to `cortex_command/cli.py`.** Rejected per the symmetric-defense re-evaluation in research: the four skill helpers are not a coherent noun-group like `git remote *`; they are sequenced inline-flow predicates tightly coupled to one preflight block per skill, with no expansion pathway. Subcommand grouping would oversell their coherence.
- **No consolidation of the four new console-scripts into two umbrella scripts.** Earlier draft consolidated into `cortex-lifecycle-config` and `cortex-critical-review-residue`. Rejected after critical-review surfaced (Critic 4) that consolidation IS noun-grouping at one level shallower, and `cortex-lifecycle-config` is a misleading umbrella for `picker-decision` (which is a state inspector, not a config operation); `cortex-critical-review-residue resolve-feature` is generic session lookup mis-housed under a residue-named script.
- **No cross-lifecycle SHOULD-sequencing constraint against `skill-prose-to-cli-argparse-contract`.** Earlier draft mandated this lifecycle ship AFTER the sibling lifecycle's lint v1. Rejected after critical-review (Critic 3) surfaced that the parent epic (#251) explicitly states children ship in any order with no inter-dependence for correctness, and the SHOULD-sequencing is unenforceable without a CI gate. Both lifecycles ship independently. R14's sibling-lint-compatible-argparse requirement is the substantive coordination point; the sibling lint can validate the new modules whenever it ships.
- **No probe-substitution fix retained for skill snippets.** Option (d) replacing `python3 -c "import cortex_command..."` with `command -v cortex` doesn't fix the snippet body itself, which still does `python3 -c "import cortex_command..."`. The minimum-viable (d) for snippets requires resurrecting the `--print-python` flag the predecessor lifecycle explicitly retired (`cortex/lifecycle/resolve-cortex-interpreter-via-cli/research.md:71-79`).
- **No shebang-resolve mechanism (option a).** No precedent in major OSS; fragile under shebang rewrites and kernel BINPRM_BUF_SIZE limits.
- **No structural test enforcing that ALL `[project.scripts]` entry-point `main()`s call `_telemetry.log_invocation()`.** The extension to six modules is bounded; a full-fleet scan covering all 35+ entries is a follow-up.
- **No SessionStart hook refactor.** Covered by the completed `resolve-cortex-interpreter-via-cli` lifecycle.
- **No bash-test scaffolding rewrite (backlog #170).** Separate concern.
- **No retirement of any other bin/cortex-* script beyond the two named here.**
- **No shared schema registry for the picker-decision JSON output.** The closed-set `reason` token vocabulary lives in `cortex_command.lifecycle_implement.REASONS`; no cross-lifecycle handshake. The consumer (bash skill prose) parses via `jq` and relies on the documented closed set; future contract enforcement is out of scope.

## Edge Cases

- **User has stale venv (pre-PR install without reinstall)**: the bash wrappers are gone; the console-scripts are already at their installed location. Recovery: `uv tool install --reinstall --refresh git+https://github.com/charleshall888/cortex-command.git@<tag>`. Covered in CHANGELOG per R24.
- **`cortex-lifecycle-picker-decision` called with a slug containing `..` or `/`**: argparse accepts the slug at the type-level (str); `should_fire_picker` receives it. The function computes a decision and does not write to disk, so path traversal is not a vector here. Behavior: as-today; if the slug doesn't correspond to a real lifecycle, the function's existing not-found handling applies.
- **`cortex-critical-review-write-residue --feature ../../etc`**: rejected with exit 2 + stderr "invalid --feature: ..." per R11's regex validation. No filesystem write attempted.
- **`cortex-critical-review-write-residue` with empty stdin**: exit 2 + stderr "no payload on stdin" (preserves the existing snippet's heredoc-fed contract); do not produce an empty residue file.
- **Concurrent `cortex-critical-review-write-residue` for the same `--feature`**: tempfile + os.replace gives last-writer-wins (same as today's inline snippet). No advisory lock added.
- **`cortex-lifecycle-branch-mode` invoked outside a repo**: argparse passes the path; `read_branch_mode` returns None; stdout is empty string; exit 0.
- **`bin/cortex-invocation-report --check-entry-points` invoked when one of the 10 modules (4 backlog + 2 existing-promoted + 4 new) is missing**: prints `<path>: file missing` to stderr, increments missing-count, exits non-zero. Existing behavior; works correctly after R21's restructure since each iteration unit is a (package, module) tuple.
- **`_telemetry` import error inside `main()` (e.g., R13 option-b chose explicit `from cortex_command.backlog import _telemetry` but the backlog package fails to import)**: `_telemetry.log_invocation` is fail-open by design (per `cortex_command/backlog/_telemetry.py:5-9`) — but if the IMPORT itself fails, `main()` crashes before doing useful work. Acceptable as today's behavior (the bash wrapper has the same coupling — if `cortex-log-invocation` itself crashes, the wrapper continues per `|| echo` tolerance, but the in-process call would crash). The CHANGELOG note should advise users seeing import errors to run `uv tool install --reinstall`.
- **Sibling lint v1 ships before this work and fails E202 ambiguous-parser on the four new modules**: R14's "single ArgumentParser at module scope or inside main()" pattern is the mitigation. If the sibling lint catches it after this work merges, the fix is a small refactor inside each new module; no spec change needed.
- **Sibling lint v1 ships after this work**: the four new modules are validated on first sibling-lint run; if they fail, address via a follow-up. This work does not block on sibling-lint validation.
- **`_telemetry` is moved to top-level `cortex_command/_telemetry.py` (R13 option-a)**: all existing importers under `cortex_command.backlog.*` need their import statement updated. This is a separate cross-cutting edit that the implementer must enumerate in the PR description. The byte-equivalence test (R23) catches any breakage in `cortex_command.backlog.*` callers.
- **Parity test passes locally but fails in CI**: investigate via `pytest tests/test_lifecycle_kept_pauses_parity.py -v` — most likely cause is the regex extension (R19) didn't land or the new CLI invocation is outside the ±35-line tolerance window around the inventory anchor.

## Technical Constraints

- **`[project.scripts]` registration requires reinstall**: per `pyproject.toml` comment lines 25-27: "Adding or changing `[project.scripts]` entries requires reinstalling the tool with `uv tool install -e . --force` so the console scripts are regenerated." Implementation must include the reinstall step in any local-verification ritual.
- **Dual-source mirror pre-commit hook auto-syncs `bin/` deletions**: `.githooks/pre-commit` Phase 2 runs `just build-plugin` on bin/ changes; the build copies bin/ to plugins/cortex-core/bin/ and removes mirror entries for deleted canonical files. No manual deletion of `plugins/cortex-core/bin/cortex-backlog-ready` etc. is needed.
- **JSON output for `picker-decision`**: closed-set `reason` token vocabulary preserved verbatim from `cortex_command.lifecycle_implement.REASONS`. New module wraps the function output as `{"fire": fire, "reason": reason}` via `json.dumps`. The skill bash consumer parses via `jq -r '.fire'` and `jq -r '.reason'`.
- **Slug validation regex for `--feature`**: `^[a-z0-9][a-z0-9-]*$` matches the canonical slug shape from `cortex_command/common.py:slugify`. First character restricted to alphanumeric to reject pathological inputs like `--evil`.
- **Telemetry call placement**: per `bin/cortex-invocation-report --check-entry-points` awk pattern (`_telemetry.log_invocation\(*`), the FIRST non-trivial statement of `def main()` must match. New modules must follow this placement; the two pre-existing modules (`backlog.ready`, `complete_morning_review_session`) must have the call inserted at the same position.
- **`_telemetry` cross-package import**: the helper lives at `cortex_command.backlog._telemetry` today. Five of six modules in scope live outside `cortex_command/backlog/`. Implementer choice per R13: move to top-level `cortex_command/_telemetry.py` (preferred long-term — eliminates the backlog-package private smell) OR use explicit `from cortex_command.backlog import _telemetry` cross-package import (acceptable short-term — minimal-diff path). Either choice satisfies R22's awk gate.
- **Sibling lint argparse compatibility**: the sibling lifecycle `skill-prose-to-cli-argparse-contract` builds an AST walker that reads each `[project.scripts]`-registered module's source for `argparse.ArgumentParser(...)` constructor calls and `.add_argument(...)` / `.add_subparsers(...)` / `.add_parser(...)` chains.
