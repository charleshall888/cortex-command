# Specification: Extract conditional content blocks to references/ (Path 1 — in-place trim of §1a + bundled rubric trigger tests)

> **Scope revision from the backlog item body.** The original backlog item (#179) scoped two wholesale `references/*.md` extractions. Research (`research.md`) and the adversarial review found that (a) Extract B's existing deterministic test gives false confidence (only Trigger 1 is exercised — Triggers 2/3/4 are unverified), (b) the worked-examples extraction would establish a first-of-its-kind sub-agent-prompt-template anti-pattern in the codebase, (c) the §1a wholesale relocation breaks `test_skill_contracts`'s regex anchor at `tests/test_daytime_preflight.py:339`, and (d) ~50–75% of §1a is unique main-session orchestration that #177 explicitly preserved. ## Problem Statement

Epic #172's discovery audit framed hot-path skill context (`implement.md` 302 lines; `critical-review/SKILL.md` 369 lines) as a token-efficiency concern. Two surface-level remedies — extracting `implement.md §1a` Daytime Dispatch and the A→B downgrade rubric's 8 worked examples to dedicated `references/*.md` files — were initially scoped. Research surfaced structural risks that invalidate the original remedy. The replacement work — in-place trim of §1a's helper-module-contract recipe lines AND four new live-LLM tests covering Triggers 2/3/4 of the rubric — captures the achievable hot-path savings and replaces the verification debt left by Extract B's Trigger-1-only deterministic test with held-out fixture coverage. The four new tests are not "deterministic" in the binary-check sense of the existing Trigger 1 test (which checks a single field's presence); they are single-rubric-application live-model tests with built-in stability tolerance (2-of-3 retry) and rationale-anchored assertions that verify the intended trigger fired rather than just that some downgrade fired. The bundled test cost (worst-case ~50 minutes of `claude -p --model opus` subprocess time on the critical path) is acknowledged as a deliberate trade for retiring the verification debt within this ticket rather than deferring it.

## Edge Cases

- **Trim accidentally drops a §1a contract marker below its grep-gate count**: `test_skill_contracts` or a structural backstop gate fails. The remediation is to **revert the over-aggressive trim and preserve the surrounding behavioral prose** — NOT to add a one-sentence stub containing the marker. A stub satisfies the structural backstop but would fail the behavior-content gates (which anchor on phrases like `kill -0`, `active-session.json`, `plan.md not found`, `pause polling`, `120 iterations` that survive only inside the preserved behavioral description). If both backstop and behavior-content gates fire on the same trim, the trim is over-aggressive at the level the gates were designed to catch; back off the trim, do not paper over it.
- **Fixture text inadvertently overlaps with a worked example**: When authoring the Trigger 2/3/4 fixtures, run `grep -F "<fixture-string>" skills/critical-review/SKILL.md` for each candidate `fix_invalidation_argument` value before committing it. If any match is returned, the fixture is non-held-out — pick different content. This is the same defense the spec uses for the structural backstops: the fixture-authorship gate prevents the prompt-recall failure mode at the source.
- **Trigger fixture passes via cross-trigger misclassification**: Each test asserts both a downgrade-fired regex AND a per-trigger rationale-anchored regex (Reqs 2/3/5). If the model fires the wrong trigger, the rationale regex fails to match and the test fails — closing the gap that a trigger-agnostic assertion would leave open. The Trigger 3-with-straddle test (Req 4) is the asymmetric exception: it uses a positive ratification regex instead of a negative-substring assertion, so meta-reasoning traces ("I considered downgrading but ratified because...") don't flip the test on false-positive grounds.
- **Stochastic decoding causes a single-run flake**: The four new tests use the file-level `_run_n_times` infrastructure (lines 99–140 of `test_critical_review_classifier.py`) with 2-of-3 pass tolerance (max_attempts=4). This is the same pattern `test_weak_argument_downgrade` uses for the rubric on the reviewer side. The existing Trigger 1 test (`test_synthesizer_rubric_deterministic`) stays single-invocation because its check is binary-field-absence (low decoding variance). The new tests are semantic-judgment cases (higher decoding variance) and get the retry budget.
- **Trigger 3 pair is confounded by varying surrounding content**: Mitigated by the Req 4 fixture requirement to use SAME finding/evidence_quote/fix_invalidation_argument content as Req 3, with `straddle_rationale` as the only difference. The pair isolates the straddle-exemption variable; non-load-bearing content is held constant.
- **Synthesizer subprocess latency variance**: `claude -p --model opus` is a subprocess invocation. The existing Trigger 1 test uses `timeout=600`; new tests inherit the same timeout. With 2-of-3 tolerance at max_attempts=4, worst-case per-test latency is 4 × 600s = 40 minutes; worst-case across all four new tests is 160 minutes (sequential). In practice 2-of-3 typically converges in 2–3 attempts so realistic latency is much lower. Don't silently raise the per-attempt timeout; if a single attempt hits 600s, investigate prompt size growth.
- **Pre-commit hook regenerates mirror but the trimmed file is not staged**: If the implementer trims `implement.md` and stages it, the pre-commit hook runs `just build-plugin`, which updates `plugins/cortex-core/skills/lifecycle/references/implement.md`. The mirror must be `git add`-ed in the same commit; otherwise Phase 4 drift check fails. Implementer must run `git add plugins/` after the trim or rely on the hook's stage-regenerated-files behavior (verify the hook does this).
- **`@pytest.mark.slow` mark is not run in the default test invocation**: All four new tests inherit the `@pytest.mark.slow` mark from the existing Trigger 1 pattern. Run via `pytest -m slow` or an explicit `pytest tests/test_critical_review_classifier.py`. The plan-time verification must invoke the slow-marked tests explicitly.

## Technical Constraints

- The trim must preserve every item on #177's contract-preservation list (per `lifecycle/trim-verbose-lifecycle-skill-content-implementmd-1a-planmd-1bb-skillmd-gate-compression/spec.md` Item 1): dispatch_id semantics rationale, outcome-map schema, all four guards, polling-loop user-pause behavior at iteration 30, `dispatch_complete` event with outcome field.
- The trim must NOT modify `tests/test_daytime_preflight.py::test_skill_contracts` (regex anchor, grep counts, file path all unchanged).
- New tests must follow the existing `test_synthesizer_rubric_deterministic` pattern for prompt assembly (`_extract_synthesizer_template()` → regex-substitute placeholders → `subprocess.run(["claude", "-p", assembled, "--model", "opus"], timeout=600)`). Each new test implements an **inline 2-of-3 retry loop**: up to 3 sequential subprocess invocations; record `(passed: bool, reason: str)` per attempt; then call `_apply_pass_criterion(results, "2-of-3")` (defined at `tests/test_critical_review_classifier.py` lines 128–138) to derive the overall pass/fail boolean and assert on its first element. The shared `_run_n_times` helper at lines 99–125 is **not used** for these new tests because its dispatch hardcodes `_run_critical_review(fixture_path)` — which fires the full `/cortex-core:critical-review` slash command — and is incompatible with the synthesizer-only direct-prompt pattern these tests require. A prior revision of this spec cited `_run_n_times` lines 99–140 with `max_attempts=4` as the retry harness; `_run_n_times` actually ends at line 125, its `max_attempts` argument is declared but never read (the body hardcodes `n = 3`), and its dispatch path is the slash command, not the synthesizer-only subprocess.
- Fixture text for the `fix_invalidation_argument` field of each new test must be **novel held-out content** that does not match any line in `critical-review/SKILL.md` lines 229–277 (the 8 worked examples). Verify by `grep -F` during authorship.
- Trigger 3 fixture pair (Req 3 vs Req 4) must hold finding/evidence_quote/fix_invalidation_argument content CONSTANT; the only load-bearing difference is `straddle_rationale` presence/value.
- Each test's assertion must include a per-trigger rationale anchor (Reqs 2/3/5) or a positive ratification anchor (Req 4), in addition to or in place of the trigger-agnostic downgrade-fired regex. Single-anchor (downgrade-only) assertions are insufficient because they admit cross-trigger misclassification.
- Pointer sentences for trimmed recipe lines must use cortex's existing precedent form: imperative ("Invoke `daytime_dispatch_writer.main` in init mode for the canonical atomic-write contract"), not conditional ("if X, see Y"). The existing post-#177 §1a Step 2/Step 4 invocations of `daytime_dispatch_writer` are the in-tree precedent.
- Plugin mirroring is owned by `just build-plugin` (which uses `rsync -a --delete`) and the pre-commit hook; the implementation must not hand-edit mirror files under `plugins/cortex-core/skills/`.
- The four new tests inherit `@pytest.mark.slow` from the existing pattern; they are run only when the slow marker is enabled.
