# Specification: overnight-launchd-scheduled-runner-operates-from

> Backlog #311 · tier=complex · criticality=high. See `cortex/lifecycle/overnight-launchd-scheduled-runner-operates-from/research.md` for the full call-chain map, fix-option tradeoffs, and the adversarial research pass. This spec incorporates a critical-review pass that corrected the tz fix mechanism, hardened the validity guard, fixed the regression-test namespace, and added a wheel/launchd verification gate.

## Problem Statement

Scheduled (launchd-fired) overnight runs silently operate from filesystem root `/` instead of the user's repo, failing every feature (the reported session failed 0/4 in 23 min). The launchd job starts with CWD=`/` and a bare environment, and the overnight CLI's single repo-root resolver (`_resolve_repo_path`) trusts `git rev-parse`→`Path.cwd()` and ignores both the `CORTEX_REPO_ROOT` the scheduler can supply and the correct `project_root` already stored in `overnight-state.json`. The identical defect silently disables the #308 guardian scan under launchd. Separately, `cortex overnight status` crashes (`can't compare offset-naive and offset-aware datetimes`) because it compares a naive-local `scheduled_start` against a UTC-aware "now", making the command unusable. This fix restores scheduled overnight execution and guardian supervision, and makes `status` robust — the day/night handoff that is the project's north star is currently broken on scheduled runs.

## Phases

- **Phase 1: Root resolution** — make the overnight CLI's single resolution site (`_resolve_repo_path`) resolve the correct repo root under launchd, fixing scheduled `start`, the re-resolving async-spawn child, and the guardian scan; verify end-to-end in the installed-wheel/launchd path.
- **Phase 2: Status tz crash** — make `cortex overnight status` tz-safe without crashing or misreporting fire times, and emit aware `scheduled_start` going forward.

## Requirements

**Priority (MoSCoW):** **Must** — R1, R2, R3, R4, R5, R6, R8, R9 (the root-resolution fix, the status-crash fix, their un-masked regression tests, the read-only constraint, and the end-to-end wheel/launchd verification). **Should** — R7 (writer-side aware `scheduled_start`: a durability improvement; the R6 reader normalization already resolves the crash, so R7 can slip to a follow-up without leaving a bug). **Won't-do** — see Non-Requirements (run-now path, planning orchestrator, plist `WorkingDirectory`, three-resolver convergence).

### Phase 1 — Root resolution

1. **Single-funnel precedence with a marker-based validity guard.** `_resolve_repo_path` (`cortex_command/overnight/cli_handler.py`) gains an optional `state_project_root: Path | None = None` parameter and resolves by precedence: **valid `state_project_root` → valid `CORTEX_REPO_ROOT` → `git rev-parse --show-toplevel` → `Path.cwd()`**. A candidate (whether from `state_project_root` OR `CORTEX_REPO_ROOT`) is **valid** only if, after `.resolve()` canonicalization, it: is non-null, exists, is a directory, is not `/`, **and contains a repo marker — a `.git` entry (file or dir) or a `cortex/` directory** (matching the house idioms in `log_invocation.py:_resolve_repo_root` which requires `.git`, and `common._resolve_user_project_root` which walks for `cortex/`). The validity predicate is applied **uniformly to both the state and env candidates** so a wrong-but-existent non-repo directory (`$HOME`, a parent dir) or a poisoned `CORTEX_REPO_ROOT=/` is rejected and resolution falls through. `git rev-parse` and `cwd` remain the unguarded existing fallback. All resolution stays inside this one function (R20). **Acceptance**: a unit test exercises every precedence branch and the marker-based guard — with cwd a non-git dir and `CORTEX_REPO_ROOT` unset: a non-existent / `/` / existent-but-marker-less `state_project_root` (e.g. a bare tmp dir) falls through to git→cwd; a marker-bearing `state_project_root` wins; with `state_project_root` invalid and a marker-bearing `CORTEX_REPO_ROOT` set, the env value is returned; with `CORTEX_REPO_ROOT=/` (or a marker-less dir), it is rejected and resolution falls through. `just test` exits 0 (pass if exit code = 0). **Phase**: Root resolution

2. **`handle_start` threads `state.project_root` into resolution; the regression test targets the `--launchd` inline path where `runner.run` is actually called.** `handle_start` loads the `--state` file and passes its `project_root` to `_resolve_repo_path(state_project_root=…)` before threading `repo_path` into `runner.run`. The launchd fire is **two `handle_start` entries**: the parent (`start --state <abs> --scheduled`, no `--launchd`) takes the async-spawn path (`_spawn_runner_async` → `subprocess.Popen` of a child, no in-process `runner.run`); the child is re-invoked `start --launchd --state <abs>` and takes the **inline** path (`_run_runner_inline` → `runner.run`). Both entries call `handle_start`→`_resolve_repo_path`, so the single-funnel fix covers both. **Acceptance**: a regression test builds a session whose `overnight-state.json` has a correct (marker-bearing) `project_root`, sets cwd to `/` (or a marker-less tmp dir) and clears `CORTEX_REPO_ROOT`, calls `handle_start` with the **`--launchd` (inline) namespace** (where `runner.run` runs in-process), spies on `runner.run`'s `repo_path` kwarg, and asserts it equals `state.project_root` (not `/`). The test must **not** monkeypatch `_resolve_repo_path` (that masking is why the bug shipped). **Phase**: Root resolution

3. **Guardian scan resolves correctly under launchd.** `_dispatch_overnight_guardian_scan` (`cortex_command/cli.py`) is the only host-wide verb fired under launchd (`build_guardian_plist_dict` writes a plist whose `ProgramArguments` is `overnight guardian scan` and whose `EnvironmentVariables` sets `CORTEX_REPO_ROOT` unconditionally, `macos.py`). It passes no state, so it relies on the new env precedence in R1. After the fix, `_resolve_repo_path()` (no state arg) with a marker-bearing `CORTEX_REPO_ROOT` set and cwd=`/` returns the env value, not `/`. (`_dispatch_overnight_recover` is **operator-invoked only — no plist invokes it**; it is not a launchd path, though it benefits from the same env precedence if run from a non-repo cwd.) **Acceptance**: a unit test asserts `_resolve_repo_path()` (no state arg) honors a marker-bearing `CORTEX_REPO_ROOT` over a `/` cwd — the env branch of R1's test. `just test` exits 0. **Phase**: Root resolution

4. **Per-feature `repo: null` resolves to the session root.** With `repo_path` corrected at the CLI, a feature whose `repo` is null in state operates from the session's `project_root` (not `/`). The mechanism: `handle_start` always threads a non-null resolved `repo_path` into `runner.run`, which exports `os.environ["CORTEX_REPO_ROOT"]` (`runner.py:2503`) for the per-feature executor, so `feature_executor`'s `Path.cwd()` fallback for `repo_path is None` is never reached with `/`. **Acceptance**: the R2 regression test additionally asserts the exported/threaded root reaching the per-feature path is the session root, not `/`. **Phase**: Root resolution

5. **Existing `_resolve_repo_path` monkeypatch sites updated WITHOUT becoming the only coverage.** The ~30+ tests that monkeypatch `_resolve_repo_path` (e.g. `test_cli_overnight_format_json.py`, `test_guardian_install.py`, `test_scheduler_e2e.py`) must accept the new optional parameter (e.g. `lambda *a, **k: tmp_path`) so they do not raise on the new signature. These remain monkeypatched (legitimately — they only need *a* root). The protection against recurrence is **not** these sites but R1 (un-patched precedence unit test) and R2 (un-patched fire-path test), which exercise the real resolver. **Acceptance**: `just test` exits 0 with no signature-mismatch errors, AND R1+R2 exist and do not monkeypatch `_resolve_repo_path` (`grep -L "_resolve_repo_path" <R1/R2 test files>` confirms the resolver is not patched in those two files). **Phase**: Root resolution

### Phase 2 — Status tz crash

6. **`status` no longer crashes on naive/aware compare, and does not misreport fire times.** Make `_parse_iso` (`status.py`) normalize every parsed timestamp to UTC-aware via `datetime.fromisoformat(ts).astimezone(timezone.utc)`: a **naive** value is interpreted as **system-local** (Python's `.astimezone()` attaches the offset for that value's own wall-clock date — DST-correct — then converts to UTC); an **aware** value is converted to UTC. This is correct because the only producer of naive timestamps is the scheduler, which writes `scheduled_start` in **local** wall-clock (`scheduler/macos.py:parse_target_time`); `started_at`/event-`ts` are written UTC-aware in production and pass through converted, not reinterpreted. This is a uniform, field-agnostic change implementable in the single `_parse_iso` chokepoint — **not** a per-field rule, and **not** the `_is_spent` `replace(tzinfo=now.tzinfo)` idiom (which, against the UTC-aware `status._now()`, would wrongly stamp local wall-clock as UTC). All compare sites (`_is_scheduled_dormant`, `render_status` elapsed/last-event/per-feature) then compare UTC-aware vs UTC-aware. **Acceptance**: a regression test calls `render_status` (or `handle_status`) with `phase: "executing"` and a **naive-local** `scheduled_start` near a fire boundary under a non-UTC `TZ` (e.g. `America/New_York`), asserting (a) no exception, (b) output lacks `Error reading status:`, (c) the dormant/"fires at" decision matches the local-wall-clock interpretation (not skewed by the UTC offset). Cases cover naive-local `scheduled_start`, aware `started_at`, and a mix. `just test` exits 0. **Phase**: Status tz crash

7. **Writer emits aware `scheduled_start` going forward, without breaking GC reaping.** The scheduler writes `scheduled_start` tz-aware with a local offset (e.g. `resolved_target.astimezone().isoformat()`), so new sessions are unambiguous; the R6 reader normalization remains the backstop for already-scheduled (legacy naive) sessions. Note `state.scheduled_start` and the sidecar `scheduled_for_iso` are the **same source string** (`cli_handler.py:1784` assigns `st.scheduled_start = handle.scheduled_for_iso`), so this is one writer change. Because the spent-fire GC loop calls `_is_spent` with a **naive** `now = datetime.now()` (`macos.py:860`), the change must confirm `_is_spent` still reaps correctly when `scheduled_for` is now aware (its existing `now.tzinfo is None` branch must normalize `now` to the aware value's offset). **Acceptance**: a test asserts a newly written `scheduled_start` parses to a tz-aware datetime (`.tzinfo is not None`); the R6 reader test still passes for legacy naive input; an `_is_spent` test confirms a spent aware `scheduled_for` is still reaped against a naive `now`. `just test` exits 0. **Phase**: Status tz crash

8. **`status` stays read-only.** The tz fix normalizes in-memory only and must not write normalized timestamps back to disk (ADR-0011: status is a read-only observability surface). **Acceptance**: Interactive/review-dependent (a negative property — absence of writes — is not reliably assertable by one runnable command without false positives): the `status.py` diff for R6 introduces no `os.replace`, `write_text`, `open(..., "w")`, or state-mutation call; `git diff` of `status.py` shows additions confined to read/normalize logic. **Phase**: Status tz crash

### Phase 1 — Verification (end-to-end, installed-wheel/launchd path)

9. **The fix is verified in the path that actually broke — the installed wheel under launchd — not only the working-tree test suite.** Because the launchd-fired `cortex` runs the **installed wheel** (not the working tree `just test` exercises), green unit tests alone do not prove #311 is fixed; the rollout requires `uv tool install …` reinstall (or `CORTEX_COMMAND_FORCE_SOURCE=1`). This requirement converts that from a footnote into a gating step. **Acceptance**: Interactive/session-dependent (requires a real launchd fire or a faithful simulation of it; a pure unit command cannot exercise the installed-wheel launchd path): after reinstalling the wheel, either (a) schedule a near-term fire (`cortex overnight schedule` for the next 1–2 min boundary), let launchd fire it, and confirm `cortex/lifecycle/sessions/<id>/active-session.json` `repo_path` is the real project root (not `/`) and `overnight-events.log` shows no `morning_report_commit_failed` with `details.project_root: "/"`; OR (b) a documented test that invokes the installed console-script entrypoint with `cwd=/` + a bare env + a state file and asserts the resolved root. The completion summary must explicitly state the wheel was reinstalled and which verification (a or b) was performed. **Phase**: Root resolution

## Non-Requirements

- **The run-now (interactive) path** — already works; the fix must not regress it (R2 pins this: `state.project_root` is authored from the correct cwd at session create, so preferring it changes nothing for run-now, which resolves the same root from cwd).
- **The planning orchestrator** — already succeeds; out of scope.
- **Setting plist `WorkingDirectory`** (option D1) — rejected as the mechanism: it re-trusts launchd's CWD, contradicting "never trust launchd CWD/env" and R20, and would require rescheduling every pending job. Optional belt-and-suspenders only; not implemented here.
- **Converging the three resolver families** (`_resolve_repo_path`, `common._resolve_user_project_root`, init's resolver — option D2) — the right long-term end-state but over-scoped for a high-priority bugfix; `common.py` keeps init's resolver separate by spec. Deferred to a follow-up (solution-horizon: a deliberately-scoped phase, not a stop-gap). This spec aligns the *guard idiom* with the siblings (R1's marker check) without merging the functions.
- **Fixing the other 6 interactive `_resolve_repo_path` call sites** (`status`, `cancel`, `logs`, `list-sessions`, `schedule`, `prepare`, `launch`) — these are operator/MCP-invoked with cwd inside the repo, so `git rev-parse` already resolves correctly; they are not launchd-reachable. They benefit transitively from R1 (more robust resolution) but need no dedicated change. The launchd-reachable resolution set is exactly **{`start`, `guardian scan`}**.

## Edge Cases

- **`state.project_root` wrong-but-existent (`$HOME`, sibling dir, relocated checkout)**: the marker-based guard (R1) rejects any candidate lacking a `.git`/`cortex/` marker, so a non-repo dir falls through to `CORTEX_REPO_ROOT`→git→cwd rather than overriding a correct env value. A `state.project_root` that points at a *different valid repo* is the session's intended repo and is honored (correct).
- **`state.project_root` null / `/` / missing**: rejected by the guard → falls through.
- **`CORTEX_REPO_ROOT=/` or marker-less** (e.g. poisoned by `runner.py:2503` before the fix): rejected by the same uniform guard → falls through. This closes the env-poisoning vector.
- **Symlinked / `/private/var`-vs-`/var` (macOS) roots**: candidates are `.resolve()`-canonicalized before the guard and before being returned, matching sibling resolvers and keeping downstream string comparisons (`report.py`) consistent.
- **Async-spawn child re-resolution**: child re-runs `handle_start` from inherited cwd=`/`; routing through `handle_start` (not the spawn argv) covers it (R2). The child's argv carries no repo-root, so it re-resolves from `state` independently.
- **Legacy already-scheduled session with naive `scheduled_start`**: R6's `.astimezone()` normalization interprets it as local (correct); R7's writer fix only affects new sessions.
- **DST boundary**: `.astimezone()` on a naive value attaches the offset for that value's own wall-clock date, so a `scheduled_start` written in winter and read in summer is normalized on its own date's offset — no fixed-offset skew. (This is why R6 mandates `.astimezone()`, not a single `replace(tzinfo=…)`.)
- **Aware `scheduled_for` in the GC `_is_spent`**: after R7, the spent-fire GC loop (`macos.py:860`, naive `now`) must still reap; R7's acceptance pins this.
- **Same-user LaunchAgent** (`launchctl bootstrap gui/$(id -u)`): git dubious-ownership is unlikely; the real failure is plain `git rev-parse` from cwd=`/`, which the state/env precedence sidesteps.

## Changes to Existing Behavior

- **MODIFIED**: `_resolve_repo_path()` → `_resolve_repo_path(state_project_root: Path | None = None)` with precedence `valid-state → valid-env → git → cwd`, a marker-based validity guard (`.git`/`cortex/`, `.resolve()`-canonicalized, not `/`) applied uniformly to the state and env candidates.
- **MODIFIED**: `handle_start` reads the `--state` file's `project_root` and passes it into resolution.
- **MODIFIED**: `status.py:_parse_iso` normalizes all timestamps to UTC-aware via `.astimezone(timezone.utc)` (naive interpreted as system-local).
- **MODIFIED**: scheduler emits `scheduled_start`/`scheduled_for_iso` tz-aware (was naive-local).
- **FIXED (behavioral)**: `cortex overnight guardian scan` now finds sessions under launchd instead of resolving to `/` and recovering nothing (#308 supervision restored under launchd).

## Technical Constraints

- **R20** (`cli_handler.py` docstring; `runner.py` "never derives user-repo paths"): resolution stays at the single CLI site (`_resolve_repo_path`); downstream modules must not re-derive. The single-funnel design with the `state_project_root` parameter preserves this literally (one function, one precedence; callers supply inputs, do not resolve).
- **ADR-0011**: `status` is read-only; no write-back of normalized timestamps.
- **House idiom for repo-root validity**: marker check (`.git` or `cortex/`), per `log_invocation.py:_resolve_repo_root` and `common._resolve_user_project_root` — R1's guard matches this rather than bare existence.
- **Wheel-vs-working-tree**: the launchd-fired `cortex` runs the installed wheel — the fix only reaches scheduled runs after `uv tool install` reinstall (or `CORTEX_COMMAND_FORCE_SOURCE=1`). R9 gates on this; it is not merely a note.
- **Launchd-reachable resolution set**: exactly `start` (via the launcher → async-spawn parent + `--launchd` child) and `guardian scan`. The other 6 `_resolve_repo_path` sites are interactive (cwd=repo). `recover` has no plist.
- **Test discipline**: the fire-path regression (R2) must not monkeypatch `_resolve_repo_path`; the ~30+ existing monkeypatch sites only widen their lambda signature (R5).
- **Events registry**: no new event literals are expected; if any are added, register in `bin/.events-registry.md`.

## Open Decisions

None — both scope decisions surfaced in research were resolved at spec time (guardian fix included per operator; writer-side aware `scheduled_start` included per solution-horizon, ranked Should). Critical-review findings were all applied (validity-guard hardening, tz mechanism correction, test-namespace fix, wheel verification gate).

## Proposed ADR

### Proposed ADR: 0013-overnight-cli-repo-root-resolution-precedence

**Context**: The overnight CLI's single repo-root resolver (`_resolve_repo_path`, the R20 resolution site) trusted `git rev-parse`→`cwd`, which silently resolves to `/` under launchd (CWD=`/`, bare env), breaking every scheduled run and the guardian scan — while the correct root sat unused in both `overnight-state.json:project_root` and the `CORTEX_REPO_ROOT` the plists can carry. **Decision**: the single resolution site resolves by explicit precedence — valid `state.project_root` → valid `CORTEX_REPO_ROOT` → `git rev-parse` → `cwd` — where "valid" means a `.resolve()`-canonicalized, existing, non-`/` directory bearing a repo marker (`.git` or `cortex/`), applied uniformly to both the state and env candidates. `handle_start` supplies `state.project_root` (from the always-absolute `--state`); host-wide verbs (guardian scan) supply `None` and rely on the plist-set env var. **Trade-off**: prefers persisted/explicit sources over ambient runtime context (the opposite of the original CWD-trusting design), accepting that resolution now depends on a correct, marker-validated state file or env var; in exchange, scheduled runs and supervision become robust to launchd's CWD/env without scattering resolution (R20 preserved) and without plist regeneration. This meets the ADR three-criteria gate (hard to reverse once load-bearing; surprising without context — "why does `start` read root from the state file?"; real trade-off — explicit-source vs ambient-resolution).
