# Plan: overnight-launchd-scheduled-runner-operates-from

## Overview
Fix the overnight CLI's single repo-root resolver (`cli_handler._resolve_repo_path`) to resolve the correct root under launchd via a marker-validated precedence (state→env→git→cwd), with `handle_start` supplying `state.project_root`; independently fix `status._parse_iso` to normalize naive (local) timestamps to UTC-aware. The two bugs are orthogonal and land in parallel (different files: `cli_handler.py` vs `status.py`).
**Architectural Pattern**: layered
<!-- Resolution stays at the single CLI layer (R20); callers supply inputs, the resolver owns precedence. -->

## Outline

### Phase 1: Root resolution (tasks: 1, 2, 3, 4, 5, 6, 7)
**Goal**: scheduled `start`, the re-resolving async-spawn child, and the guardian scan resolve the real repo root under launchd, verified in the installed-wheel path. (Mechanism notes: the guardian scan is fixed by T1's **env branch** — its plist already sets `CORTEX_REPO_ROOT`, `macos.py:1175` — not by state-threading, which is why `cli.py` needs no edit; the morning-report `project_root` is fixed **transitively** — it is NOT independently corrected: `report.py`'s `_resolve_user_project_root()` is env-first, and the env it reads is only correct once the corrected `repo_path` is re-exported as `CORTEX_REPO_ROOT` at `runner.py:2503`, which in turn requires the T3 thread to have reached `runner.run` with a non-`/` `repo_path`. The morning-report fix is therefore downstream of T3, not a separate fix.)
**Checkpoint**: `_resolve_repo_path` precedence unit test + the `--launchd` fire-path regression test pass un-patched (`just test` green); ADR-0013 written; R9 end-to-end verification performed.

### Phase 2: Status tz crash (tasks: 8, 9, 10)
**Goal**: `cortex overnight status` renders without crashing and without skewing fire-time math; new sessions write aware `scheduled_start`.
**Checkpoint**: status tz regression test (naive-local `scheduled_start` under a non-UTC `TZ`) passes; writer emits aware `scheduled_start` and GC `_is_spent` still reaps.

## Tasks

### Task 1: Add marker-validated precedence to `_resolve_repo_path`
- **Files**: cortex_command/overnight/cli_handler.py
- **What**: Give `_resolve_repo_path` an optional `state_project_root` parameter and resolve by precedence valid-state → valid-env (`CORTEX_REPO_ROOT`) → `git rev-parse` → `cwd`, with a marker-based validity guard applied uniformly to the state and env candidates.
- **Depends on**: none
- **Complexity**: complex
- **Context**: Current body at `cli_handler.py:141-156` (`def _resolve_repo_path() -> Path:` — `git rev-parse --show-toplevel` then `Path.cwd()`). New signature: `def _resolve_repo_path(state_project_root: Path | None = None) -> Path:`. Add a module-level helper, e.g. `_is_valid_repo_root(candidate: Path | None) -> bool`, returning True only when, after `Path(candidate).resolve()`, the path exists, is a directory, is not `Path("/")`, AND contains a repo marker — `(p / ".git").exists()` or `(p / "cortex").is_dir()`. **Borrow only the marker *predicate* from the house idioms — NOT an upward walk.** `_is_valid_repo_root` validates the *single supplied candidate in place* and returns False on miss (the caller falls through to the next precedence tier); it must NOT walk up toward a marker the way `common._resolve_user_project_root` does. The non-walking design is load-bearing: a walking validator applied to a stale/wrong `state_project_root` or `CORTEX_REPO_ROOT` could resolve *up* to a marker-bearing ancestor (a monorepo root, `$HOME/.git`) and silently select the wrong repo. Reference idioms for the marker check only: `log_invocation.py:_resolve_repo_root` (requires `.git`); `common._resolve_user_project_root` (its `cortex/` marker test — ignore its walk). Apply the guard to both `state_project_root` and `os.environ.get("CORTEX_REPO_ROOT")`; on a valid candidate return its `.resolve()`d Path; otherwise fall through. Keep `git rev-parse`→`cwd` as the unguarded tail. R20: all resolution stays in this one function.
- **Verification**: `grep -nE "def _resolve_repo_path\(.*state_project_root" cortex_command/overnight/cli_handler.py` returns 1 match (pass if ≥1); full precedence behavior verified by Task 2.
- **Status**: [x] complete (commit 12a7a174)

### Task 8: Fix `status._parse_iso` to normalize naive→local→UTC
- **Files**: cortex_command/overnight/status.py
- **What**: Make `_parse_iso` return a UTC-aware datetime for every input — naive values interpreted as system-local (DST-correct), aware values converted — so all compare sites (`_is_scheduled_dormant`, `render_status` elapsed/last-event/per-feature) compare aware-vs-aware (R6, R8). This also de-risks the cross-file consumer `recovery.py:149` (`datetime.now(timezone.utc) - last_ts`, where `last_ts` flows from `status._read_last_event_ts` → `_parse_iso`), a guardian/wedge path that runs under launchd and shares the same parser.
- **Depends on**: none
- **Complexity**: simple
- **Context**: `_parse_iso` at `status.py:98-100` (bare `datetime.fromisoformat(ts)`); `_now()` at `:94-95` (`datetime.now(timezone.utc)`). Change to `datetime.fromisoformat(ts).astimezone(timezone.utc)` — `.astimezone()` on a naive value presumes system-local and attaches that value's own wall-clock-date offset (DST-correct), then converts to UTC; an aware value is converted. This is field-agnostic and correct because **all naive values that reach `_parse_iso` originate from the scheduler and are local**: of the scheduler's bare-`datetime.now()` writes (`scheduled_for_iso`/`scheduled_start` AND `created_at_iso`, `macos.py:398-399`; `scheduled_for_iso` also at `cli_handler.py:1744`), only `scheduled_start` flows into `_parse_iso` — `created_at_iso` is a `ScheduledHandle` sidecar field that `_parse_iso` never reads. Every other `_parse_iso` input (`started_at` via `state.py:_now_iso`, event/last-event `ts` via `events.py:_now_iso`) is written aware-UTC (`datetime.now(timezone.utc).isoformat()`) and is *converted*, not reinterpreted. Audit confirms zero `datetime.utcnow()` (naive-UTC) producers anywhere in `cortex_command/`, so no value is silently misread by the assume-local rule. Do NOT use the `_is_spent` `replace(tzinfo=now.tzinfo)` idiom (would stamp UTC against the UTC `_now()`). In-memory only — no write-back (R8, ADR-0011). Preserve the existing `except ValueError` handling around malformed strings.
- **Verification**: `grep -c "astimezone" cortex_command/overnight/status.py` ≥ 1 (pass if ≥1); behavior verified by Task 9. R8 read-only check (Interactive/review-dependent — a negative property is not reliably assertable by one command): `git diff cortex_command/overnight/status.py` shows the edit confined to `_parse_iso`'s return expression with NO `os.replace`, `write_text`, `open(..., "w")`, or state-mutation added.
