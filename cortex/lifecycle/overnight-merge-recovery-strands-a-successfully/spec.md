# Specification: overnight-merge-recovery-strands-a-successfully

## Problem Statement

In overnight session `overnight-2026-06-02-1312`, feature 025 built its full spec correctly (12 commits on `pipeline/build-the-grinder-agnostic-knowledge-layer`) but never merged; the session reported `0/3` and opened a `[ZERO PROGRESS]` PR. **Root cause (verified against the session's `pipeline-events.log`/`overnight-events.log`): no revert ever ran.** Home-repo features carry `repo_path=None`; `_effective_merge_repo_path(None, …)` returns `None` (`outcome_router.py:157-158`), so `merge_feature` falls back to the home working tree (`merge.py:198`) and runs `git checkout overnight/<session_id>` (`merge.py:244`) there — colliding with the long-lived integration worktree that already owns that branch (`fatal: '…' is already used by worktree at …`). The checkout failure is mis-routed into test-failure recovery, whose dirty-base precheck reads the *wrongly-targeted* home tree — dirty from the runner's intentionally-uncommitted round `plan.md` copies — and pauses at `attempts=0` with the misnomer `error="dirty base branch after revert"`. Round-2 re-dispatch repeats the collision → circuit-breaker stall → `0/3`. Correcting the merge-target resolution makes built home features actually merge, which closes the observed bug.

**Scope (post-critical-review):** This lifecycle delivers the **root-cause merge-resolution fix** only. The separable second concern — surfacing a *genuinely* merge-blocked-but-built feature (a real conflict where repair is exhausted) as "recoverable on `pipeline/<feature>`" rather than failed/zero-progress — is split to **follow-up ticket #284**. Critical review showed that surfacing needs its own design (it fractures against three `deferred` consumers) and is gated on a genuine home-feature merge-conflict the incident never exhibited; #284 carries the recorded design constraints.

## Phases
- **Phase A: Route home-repo merges to the integration worktree** — make home-repo merge and recovery operations target the session's integration worktree instead of the home working tree, so the integration-branch checkout no longer collides.
- **Phase B: Correct the dirty-base precheck** — once merges target the integration worktree, scope the precheck to it and replace the misleading error string with an accurate one.

## Requirements

1. **Home-repo merge operations target the integration worktree, not the home working tree.** For a home-repo feature (`repo_path=None`), the merge-target resolution must resolve to the session's home integration worktree (`state.worktree_path`), the same way `_resolve_feature_integration_worktree` (`runner.py:594-595`) does (`if fs.repo_path is None: return Path(state.worktree_path) if state.worktree_path else None`), so `merge_feature`'s `git checkout overnight/<session_id>` (`merge.py:244`) becomes a same-worktree no-op instead of colliding. The same resolution must apply to the recovery call (`outcome_router.py:1071`) and the review-dispatch call (`:879`) that share the resolver. Cross-repo resolution (`repo_path != None`) must be unchanged. Acceptance: a new test that **creates a real second git worktree (via a `tmp_path` fixture) checked out to `overnight/<id>`** and then drives a home-repo merge through the merge path asserts the merge succeeds with **no `"already used by worktree"` error** — the test must reproduce the on-disk collision, not stub the resolver's return — and a companion assertion confirms cross-repo resolution is byte-for-byte unchanged. `python3 -m pytest cortex_command/overnight/tests/test_outcome_router.py -k "home_repo_merge_worktree or cross_repo_resolution_unchanged"` exits 0 (pass if exit code = 0). File grounding: `cortex_command/overnight/outcome_router.py:116-158` (`_effective_merge_repo_path`), `:565/:845/:1071/:879` (call sites), `cortex_command/pipeline/merge.py:198,244`, `cortex_command/overnight/runner.py:578-602`. **Phase**: Route home-repo merges to the integration worktree.

2. **The repair-completed re-merge targets the integration worktree, not `Path.cwd()`.** The hard-coded `repo = Path.cwd()` checkout in the `repair_completed` branch (`outcome_router.py:462-466`) — which also feeds the ff-merge (`:471`) and branch-delete (`:504`) in that block — must resolve to the integration worktree the same way as Requirement 1. Acceptance: reading the `repair_completed` block shows no `Path.cwd()` used as the git `repo`/`cwd` for the integration-branch checkout (`grep -n "Path.cwd()" cortex_command/overnight/outcome_router.py` shows no occurrence inside the `repair_completed` block), and a test exercising the repair-completed re-merge for a home feature against a real second worktree (per Requirement 1's fixture) asserts the merge succeeds with no `"already used by worktree"` error. **Phase**: Route home-repo merges to the integration worktree.

3. **The dirty-base precheck runs against the integration worktree.** Once Requirement 1 lands, `recover_test_failure` (`merge_recovery.py:187`) receives the integration-worktree `repo_path`, so its `git rev-parse --show-toplevel` (L228, `cwd=repo_path`) and `git status --porcelain --untracked-files=no` (L246) read the integration worktree, not the home tree. Acceptance: the existing `test_dirty_base_check_ignores_untracked_files` and `test_dirty_base_check_blocked_by_tracked_changes` in `cortex_command/pipeline/tests/test_merge_recovery.py` are updated to assert the precheck's `git` commands run with `cwd` equal to the **passed `repo_path`** (the integration worktree), and `python3 -m pytest cortex_command/pipeline/tests/test_merge_recovery.py` exits 0. **Phase**: Correct the dirty-base precheck.

4. **The misleading error string is corrected and pinned.** The hardcoded `error="dirty base branch after revert"` (`merge_recovery.py:259`) is replaced with a string that accurately describes the condition (a dirty integration worktree) and makes **no claim that a revert occurred**. The replacement must not contain the word "revert". When the (correctly-scoped) precheck finds the integration worktree genuinely dirty, it returns with this accurate error via the existing pause path — no destructive restore is performed, and the improved "recoverable on `pipeline/<feature>`" surfacing of that genuine merge-block is #284's responsibility. Acceptance: `grep -c "dirty base branch after revert" cortex_command/pipeline/merge_recovery.py cortex_command/pipeline/tests/test_merge_recovery.py` totals 0 (the misnomer is gone from both source and its test assertion at `test_merge_recovery.py:551`); the updated test asserts the exact new error string, pinning the replacement. **Phase**: Correct the dirty-base precheck.

## Non-Requirements
- Does **not** implement the built-but-merge-blocked recoverable surfacing — the `deferred` + `recoverable_branch` state, morning-report recoverable rendering, dashboard handling, `[ZERO PROGRESS]` PR suppression, write-back-target change, and sibling-cascade handling are **split to follow-up ticket #284** and are out of scope here.
- Does **not** perform any destructive operation. The earlier draft's deterministic `reset --hard`/clean of a dirty integration worktree is removed: a successful revert leaves a clean tree, so a genuinely-dirty integration worktree only arises from a *conflicted* revert (a genuine merge-block), which is #284's surfacing concern — not something Phase B should silently restore.
- Does **not** change the repair-attempt cap (max 2 for test failures; single Sonnet→Opus for conflicts) or the `before_sha == after_sha` circuit breaker — both fixed architectural constraints.
- Does **not** delete, move, or reassign the integration branch `overnight/<session_id>` — it persists by design.
- Does **not** change the worker's dispatch/allowlist resolution. `feature_executor.py` resolves a home feature's `integration_base_path`/`integration_worktree_path` to `Path.cwd()`; that is **correct and unchanged** — it is only the *worker's* allowlist base (the worker builds in its own `pipeline/<feature>` worktree), not the merge target. The Requirement-1 change is confined to the merge/recovery/review *target* resolution; `tests/test_feature_executor.py::test_same_repo_uses_cwd` is **not** modified.
- Does **not** introduce a new feature status.

## Edge Cases
- **`state.worktree_path` falsy/missing** (resumed session, TMPDIR cleared mid-session): the resolution returns `None` and the caller skips gracefully — exactly as `_resolve_feature_integration_worktree` (`runner.py:595`) already does when `worktree_path` is falsy. A test covers `worktree_path=None → None` (no crash).
- **Cross-repo features**: already resolve correctly via `integration_worktrees` (`repo_path != None`); the home fix is additive and must not regress them (asserted by Requirement 1's cross-repo companion check).
- **`plan.py` wild-light interaction** (`plan.py:508-511`): the conditional backfill of `integration_worktrees[project_root]` fires **only `if not has_home_features`** and is therefore mutually exclusive with the home-feature (`repo_path=None`) resolution path this fix targets — so the two cannot both apply to the same feature and do not double-resolve. The implementation must preserve that mutual exclusion (resolving home features via `state.worktree_path`, not via a wild-light-populated `integration_worktrees[project_root]`).
- **Precondition the no-op claim depends on**: the home integration worktree is created already checked out to `overnight/<session_id>` (`plan.py:406-410`); this is what makes the home-tree-relocated `git checkout overnight/<session_id>` a same-worktree no-op. If that precondition ever changes, Requirement 1's no-op assumption must be revisited.
- **Genuinely-dirty integration worktree** (a prior conflicted revert): the correctly-scoped precheck returns the accurate (non-"revert") error via the existing pause path; recoverable surfacing of that genuine merge-block is deferred to #284.

## Changes to Existing Behavior
- MODIFIED: home-repo merge / recovery / review-dispatch operations run in the home working tree → run in the home integration worktree (`state.worktree_path`); cross-repo unchanged.
- MODIFIED: the `repair_completed` re-merge runs against `Path.cwd()` → against the integration worktree.
- MODIFIED: the dirty-base precheck targets the home tree → the integration worktree; the misleading `"dirty base branch after revert"` error string is replaced with an accurate, revert-free description.

## Technical Constraints
- `_effective_merge_repo_path(repo_path, integration_worktrees, integration_branches, session_id)` has **no access to `state`/`state.worktree_path`** (verified at `outcome_router.py:116-121`). Requirement 1 therefore requires either threading the home worktree path into the resolver (signature change) or routing the merge/recovery/review call sites through `_resolve_feature_integration_worktree(state, fs)` (which already encodes the correct home-vs-cross fork). The choice is an implementation-pattern decision (see Open Decisions). The naive "just populate `integration_worktrees[project_root]` at plan bootstrap" is a **no-op** because the `repo_path is None` early-return fires before the dict is consulted.
- Fixed architectural constraints preserved: repair-attempt cap; integration-branch persistence; atomic state writes (tempfile + `os.replace()`); `before_sha == after_sha` circuit breaker.
- Acceptance tests must reproduce the **on-disk** worktree collision (a real second worktree owning `overnight/<id>`), not mock the resolver — the existing dirty-base tests mock `subprocess.run`, which would let a mock-only test pass while the real collision ships.
- This fix is confined to `cortex_command/` pipeline/overnight modules; the SKILL.md-to-bin parity gate is not expected to fire. No new events are emitted (no events-registry change).

## Open Decisions
- **Requirement-1 implementation pattern** — signature change to `_effective_merge_repo_path` (threading the home worktree path) vs routing the merge/recovery/review call sites through `_resolve_feature_integration_worktree(state, fs)`. Deferred because the cleaner integration — preserving the resolver's existing cross-repo lazy-creation and "already checked out" handling while adding the home case — is only distinguishable once the call-site wiring is in front of the implementer.
