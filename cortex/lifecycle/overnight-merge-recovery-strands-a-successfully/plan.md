# Plan: overnight-merge-recovery-strands-a-successfully

## Overview
Home-repo features (`repo_path=None`) resolve their merge/recovery/review target to `None`, so `merge_feature` falls back to the home working tree and its `git checkout overnight/<session_id>` collides with the long-lived integration worktree that already owns that branch. The fix threads the home integration worktree path (`overnight_state.worktree_path`) into `OutcomeContext` and routes the merge-execution sites through a single `_merge_target_repo_path(ctx, name)` helper that returns the home integration worktree for home features and delegates byte-for-byte to the existing `_effective_merge_repo_path` for cross-repo features. A home feature whose worktree path is unresolved (`None`) is **surfaced as paused, never merged against the home tree** — the degraded case must not silently re-create the collision. Once the recovery call site passes the integration worktree, the dirty-base precheck reads the correct tree; Phase B then relabels the misnamed error string (the `attempts=0` pause itself is intentionally retained for genuinely-dirty trees — its recoverable surfacing already shipped in #284).
**Architectural Pattern**: layered
<!-- The fix inserts a single resolution layer (the helper) between the merge-execution call sites and the two underlying resolvers, without changing the event/dispatch flow. -->

## Outline

### Phase A: Route home-repo merge/recovery/review to the integration worktree (tasks: 1, 2, 3, 4)
**Goal**: Home-repo merge, repair-completed re-merge, review-dispatch, and recovery operations target the session's home integration worktree instead of the home working tree, eliminating the `already used by worktree` collision; an unresolved (`None`) home worktree surfaces as paused rather than falling back to the home tree; cross-repo resolution unchanged.
**Checkpoint**: A test that creates a real second git worktree checked out to `overnight/<id>` and drives a home-repo merge through the **un-mocked `merge_feature`** asserts the merge succeeds with no `"already used by worktree"` error and an observable post-merge git state change; `test_outcome_router.py` and `test_lead_unit.py` green.

### Phase B: Correct the dirty-base precheck error string (tasks: 5)
**Goal**: With the recovery call site now passing the integration worktree, the precheck's `git` commands run against it; replace the misleading `"dirty base branch after revert"` error (which falsely claims a revert occurred) with an accurate, revert-free description and pin it — and the precheck's `cwd` — in tests using a non-`None` `repo_path` fixture. The `attempts=0`/`paused=True` return is deliberately retained (its recoverable surfacing is #284's shipped concern).
**Checkpoint**: `grep -c "dirty base branch after revert"` across source + test totals 0; `test_merge_recovery.py` green with the precheck's `cwd` asserted equal to a **non-`None`** passed `repo_path`.

## Risks
- **Implementation-pattern choice (spec Open Decision).** Resolved by adding a defaulted `home_worktree_path` field to `OutcomeContext` + a `_merge_target_repo_path(ctx, name)` helper, rather than changing `_effective_merge_repo_path`'s signature. The field is appended last with `= None` so the two test construction sites (`test_outcome_router.py:51`, `test_lead_unit.py:69`) keep working unmodified.
- **`None`/degraded home worktree.** `state.worktree_path` is `Optional[str] = None` (`state.py:241`), so the helper can return `None` for a home feature. The fix surfaces that as a paused feature (never a home-tree merge); in normal sessions `worktree_path` is always set (`plan.py:406-410`), so this guards the resumed-session / TMPDIR-wiped degraded path. The wild-light branch (`plan.py:508-513`) reassigns `worktree_path` to a cross-repo path only when `not has_home_features`, so it is mutually exclusive with the home-feature path and cannot mis-resolve a home feature.
- **Read-only home-tree resolvers remain (out of scope).** `_get_changed_files` (`outcome_router.py:254`) and `_classify_no_commit` (`:275`) run `git diff`/`git rev-list` against `config.base_branch` from `Path.cwd()` (the home tree). These are read-only (no checkout/merge → no collision) and pre-date this fix; they are NOT rerouted here. Flagged as a known adjacent gap — the merge-decision inputs read from the home tree while merge execution moves to the worktree. Deferred unless a correctness defect is observed; not expanded into this collision-scoped lifecycle.
- **Phase B retains the `attempts=0` pause.** Phase B relabels only — the `paused=True, attempts=0` return is intentionally kept. Phase A's re-scoping removes the *spurious* pause (the wrongly-targeted home tree dirty from uncommitted round `plan.md` copies); the *genuine*-dirty case (conflicted prior revert) still pauses, and its recoverable surfacing shipped in #284. The contract between this plan's new error string and #284's surfacing is assumed compatible (#284 keys on the `paused` state, not the error text); verified by the full suite.
- **Shared-file churn with #284.** `outcome_router.py`, `runner.py`, `report.py` were modified by #284 within the last day; line numbers re-grounded against current `HEAD`. #284 added no merge-target resolution site (its commits are write-back / counter / display only), so `_merge_target_repo_path` does not conflict with #284's routing. The full-suite regression in Acceptance is the backstop.
