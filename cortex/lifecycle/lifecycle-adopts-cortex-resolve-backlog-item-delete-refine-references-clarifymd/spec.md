# Specification: Lifecycle adopts cortex-resolve-backlog-item, delete refine clarify.md, simplify helper to slugify-only with pre/post baseline survey

## Problem Statement

Stream A of epic 172 calls for cross-skill deduplication: lifecycle's `skills/lifecycle/references/clarify.md` and refine's `skills/refine/references/clarify.md` are ~95% byte-identical (124 vs 130 lines, differing only in §1 Resolve Input). #176 collapses **the lifecycle/refine pair** to one canonical file in lifecycle, retargeting refine via a bare relative path; `skills/discovery/references/clarify.md` is a separate semantic protocol (~65 lines, pre-research ideation gate) and remains unchanged — post-#176 the repo has two `clarify.md` files (lifecycle + discovery), not one.

The ticket also bundles a coupled helper simplification that no sibling ticket covers: `bin/cortex-resolve-backlog-item`'s title-phrase predicate uses a Predicate-A ∪ Predicate-B union that web research confirms is not a recognized public idiom, and that the helper's own test author at `tests/test_resolve_backlog_item.py:300-398` worked through ~80 lines of candidate constructions but ultimately settled for a passing case proving "Predicate A is part of the union" rather than "Predicate-A-only is constructible." That weaker prior-art result suggests but does not prove Predicate-A-only matches are rare in the live backlog. Because Predicate-A removal is a **routing/correctness change** (not a refactor) and OQ3 evidence policy applies, #176 also adds a two-step pre/post baseline survey that records union-predicate outcomes against the live `backlog/` set BEFORE removing Predicate A, then asserts post-removal outcomes either match the baseline or appear in a per-case-judgment allow-list. Bundling the simplification into #176 is justified by causal entanglement (lifecycle's clarify.md §1 prose documents the helper's predicate semantics; rewriting that prose while leaving Predicate A intact means writing prose that documents a predicate we expect to remove) — not by implementer convenience.

After #176: the helper is slugify-only (Predicate B), `skills/lifecycle/references/clarify.md` is the canonical clarify protocol home, `skills/refine/references/clarify.md` is deleted, refine's `SKILL.md` retargets three call sites via bare relative path `../lifecycle/references/clarify.md`, and the plugin mirror is auto-pruned by `just build-plugin`.

## Requirements

1. **Lifecycle clarify.md §1 invokes the helper.** `skills/lifecycle/references/clarify.md` §1 contains the five-exit-code branch table refine currently has (exits 0/2/3/64/70), the JSON output schema (`filename`, `backlog_filename_slug`, `title`, `lifecycle_slug`), and a title-phrase predicate explainer that documents the post-#176 slugify-only predicate (Requirement 4). §2–§7 are unchanged.
   - Acceptance: `grep -c 'cortex-resolve-backlog-item' skills/lifecycle/references/clarify.md` ≥ 1; `grep -c 'Exit 0' skills/lifecycle/references/clarify.md` ≥ 1; `grep -c 'Exit 2' skills/lifecycle/references/clarify.md` ≥ 1; `grep -c 'Exit 3' skills/lifecycle/references/clarify.md` ≥ 1; `grep -c 'Exit 64' skills/lifecycle/references/clarify.md` ≥ 1; `grep -c 'Exit 70' skills/lifecycle/references/clarify.md` ≥ 1.

2. **Refine clarify.md is deleted.**
   - Acceptance: `test ! -f skills/refine/references/clarify.md` (pass if exit 0).

3. **Refine SKILL.md retargets all three call sites to bare relative paths.** Lines 38, 65, and 86 of `skills/refine/SKILL.md` reference `../lifecycle/references/clarify.md`. The bare-relative form was selected by the user during the spec interview; no `${CLAUDE_SKILL_DIR}/..` substitution is introduced. The bare-relative form has no precedent for cross-skill `..` traversal in the repo — the path-resolution semantics are asserted (anchor relative to the SKILL.md file's directory) and verified by Requirement 8's invocation-layout matrix, not just by literal-string presence.
   - Acceptance: `grep -c '\.\./lifecycle/references/clarify\.md' skills/refine/SKILL.md` = 3; `grep -cE '(^|[^/.])references/clarify\.md' skills/refine/SKILL.md` = 0.

4. **Helper Predicate A is removed; slugify-only Predicate B remains. This is a routing/correctness change, not a refactor.** `bin/cortex-resolve-backlog-item._resolve_title_phrase` retains only the slugified-substring predicate. The case-folded raw-substring predicate (`predicate_a`) is removed along with its `lower_input` derivation and the union-comment block. Function docstring is updated to drop Predicate-A language. The script's title-phrase predicate explainer at the file-header comment level (if any) is updated to match.
   - **Behavior change surface**: inputs where `lower(input) ⊆ lower(title)` but `slugify(input) ⊄ slugify(title)` may resolve differently post-#176. The pre/post baseline survey (Requirement 5) detects every such case in the live `backlog/` set; per-case judgment from the survey lands in this spec's Edge Cases before any code merges.
   - Acceptance: `grep -c 'predicate_a' bin/cortex-resolve-backlog-item` = 0; `grep -c 'lower_input' bin/cortex-resolve-backlog-item` = 0; `grep -c 'slug_input' bin/cortex-resolve-backlog-item` ≥ 1 (predicate B still present); `grep -ciE 'Predicate A' bin/cortex-resolve-backlog-item` = 0.

5. **Pre/post baseline divergence-survey (the OQ3 evidence record).** The survey is a two-step contract test, NOT a single post-removal pin. The two steps land in separate commits within the same PR so the baseline can be regenerated and re-reviewed if needed.

   **Step 5a (pre-removal baseline, lands first)**: extend `tests/test_resolve_backlog_item.py` with a baseline-capture test `test_predicate_a_baseline_capture` that runs the curated input set against the **current** helper (Predicate-A ∪ Predicate-B) over the live `backlog/[0-9]*-*.md` items and writes the resulting (input, exit-code, resolved-filename-or-None) tuples to a fixture file `tests/fixtures/predicate_a_baseline.json`. The fixture is checked into the repo as the frozen pre-removal baseline. Curated inputs MUST include all categories enumerated in the original ticket (numeric IDs including zero-padding, kebab slugs, title fuzzy matches, uppercase inputs, inputs with punctuation, ambiguous-multi inputs, no-match inputs) PLUS at least three candidate Predicate-A-only constructions reverse-engineered by inspecting current backlog titles for shapes where slugify strips characters (backticks, parentheses, slashes, dots in version numbers, periods, internal multi-spaces) and where `lower(input) ⊆ lower(title)` but `slugify(input) ⊄ slugify(title)`. If any of those reverse-engineered inputs resolves differently against the live helper than against the slugify-only helper, those rows are documented divergences.
   - Acceptance for 5a: `pytest tests/test_resolve_backlog_item.py -v -k baseline_capture` exits 0 (pass if exit code = 0); `tests/fixtures/predicate_a_baseline.json` exists; `python3 -c "import json; assert len(json.load(open('tests/fixtures/predicate_a_baseline.json'))) >= 10"` (≥10 input cases).

   **Step 5b (post-removal divergence assertion, lands with Requirement 4)**: extend `tests/test_resolve_backlog_item.py` with a divergence assertion test `test_predicate_a_divergences_match_judgment` that re-runs the same curated inputs against the post-removal slugify-only helper and asserts each (input → outcome) tuple either (i) matches the baseline tuple, or (ii) appears in an explicit `documented_divergences` list inside the test file with a per-case judgment (`bug-shaped` or `legitimate-feature`). Any unexpected divergence (a tuple that differs from baseline AND is not in `documented_divergences`) fails the test.
   - Acceptance for 5b: `pytest tests/test_resolve_backlog_item.py -v -k divergences_match_judgment` exits 0; `grep -c 'documented_divergences' tests/test_resolve_backlog_item.py` ≥ 1.

   - **Per-case judgment policy**: any `documented_divergences` row classified `legitimate-feature` blocks the merge until either Predicate A is restored AND OQ3 evidence (events.log F-row + dispatch transcript) is recorded, OR the user explicitly accepts the divergence on the record (event logged in `lifecycle/{feature}/events.log` as `divergence_accepted` with rationale). Bug-shaped rows merge as-is.

6. **Plugin mirror is auto-pruned.** `plugins/cortex-core/skills/refine/references/clarify.md` does not exist after `just build-plugin` runs.
   - Acceptance: `test ! -f plugins/cortex-core/skills/refine/references/clarify.md` (pass if exit 0).

7. **Dual-source parity test passes with one fewer pair.** `tests/test_dual_source_reference_parity.py` exits 0; the count of collected pairs decrements by 1 (from the pre-#176 baseline).
   - Acceptance: `pytest tests/test_dual_source_reference_parity.py` exits 0.

8. **End-to-end resolution works under both invocation layouts (smoke-test matrix).** Before merge, the implementer manually exercises the bare-relative cross-skill path under both invocation layouts and pins the results in a one-line note in the PR description:
   - **Layout 1**: `/cortex-core:refine <known-id>` direct invocation. Verifies refine reads `../lifecycle/references/clarify.md` correctly when refine is the active skill and the file containing the path (`skills/refine/SKILL.md`) is the same as the active skill's SKILL.md.
   - **Layout 2**: `/cortex-core:lifecycle <known-id>` delegating into `/cortex-core:refine`. Verifies the bare-relative path resolves correctly when the active skill is lifecycle and the file containing the path was loaded by lifecycle's "Read `skills/refine/SKILL.md` verbatim" directive at `lifecycle/SKILL.md:220` — the runtime layout most likely to expose CWD-vs-file-relative resolution differences.
   - Acceptance: Interactive/session-dependent — refine and lifecycle skill invocations are interactive Claude Code skills and cannot be executed from a shell command. Verification is a manual two-shot smoke test (one invocation per layout) with a one-line PR note recording the (layout, backlog-id, exit-status) for each. If either layout fails, the bare-relative path is invalidated and the spec must be revised to either (a) use the substituted form `${CLAUDE_SKILL_DIR}/../lifecycle/references/clarify.md` or (b) re-architect (back-reference open question).

## Edge Cases

- **`CORTEX_BACKLOG_DIR` env var override.** The helper honors this env var (`bin/cortex-resolve-backlog-item` L212) for tests; lifecycle's prose-based scanning did not. Post-adoption, lifecycle's resolution context becomes env-var-aware (matches refine's current behavior). No additional handling required.
- **`cortex-log-invocation` side effect.** `bin/cortex-resolve-backlog-item:16` logs every invocation to `lifecycle/sessions/{id}/bin-invocations.jsonl`. Adoption increases that log's size by one entry per lifecycle clarify call. Acceptable side effect; no requirement.
- **Exit 70 (IO/parse failure) handling.** Lifecycle's clarify.md §1 surfaces the stderr diagnostic and halts without falling through to disambiguation, matching refine.
- **Exit 64 (usage error) handling.** Same — surface stderr, halt.
- **Predicate-A-only legitimate-feature divergence revealed by Step 5b.** Post-removal helper assertion fails for an input where `lower(input) ⊆ lower(title)` but `slugify(input) ⊄ slugify(title)`, AND per-case judgment classifies the prior union behavior as legitimate (the prose predicate clearly intended that match). Action: restore Predicate A, record an OQ3 events.log F-row + dispatch transcript, OR (with explicit user override on the record) accept the divergence and document the new resolution in the spec. The Step 5b assertion fails until one path is chosen.
- **Predicate-A-only bug-shaped divergence revealed by Step 5b.** Same shape as above but per-case judgment classifies the prior union behavior as a bug. Action: list the input in `documented_divergences` with `bug-shaped` rationale; merge proceeds.
- **Pre-commit drift hook firing.** If `just build-plugin` is not run before the commit lands, the dual-source drift pre-commit hook blocks the commit. Expected behavior; the implementer must run `just build-plugin` after the canonical edits and before staging the deletion.
- **Concurrent edits with #175.** Sections are disjoint (§1 here, §3a in 175). Either ticket can land first without conflict; the second rebase is a clean apply.
- **Cross-skill load cascade with §3a Critic Review.** Lifecycle's canonical clarify.md L49 (post-rewrite preserved from current §3a) instructs `Read references/clarify-critic.md`. When refine reads lifecycle's clarify.md via the bare-relative path (Layout 1 or Layout 2 of Requirement 8), Claude must resolve the inner `references/clarify-critic.md` reference. Resolution semantics are unverified: if anchored to the file being read (lifecycle's clarify.md), it correctly hits `skills/lifecycle/references/clarify-critic.md`; if anchored to the active skill, it points at `skills/refine/references/clarify-critic.md` (still exists today; #175 will eventually delete it). Requirement 8's smoke-test matrix MUST exercise the §3a Critic Review path under both invocation layouts to confirm anchor semantics; if cross-load anchor is active-skill-relative, the inner reference must also be retargeted before #175 deletes refine's clarify-critic.md.
- **#184 lands first.** If 184 relocates clarify.md content into `investigate.md`, #176 becomes a textual edit on a different file. Spec is unaffected; the file path moves and the requirements still describe the same content.
- **A backlog item with `status: archived`** sitting in the live `backlog/` glob during the survey. The helper does not filter by status; the survey pins whatever resolution the helper produces.
- **A backlog item with malformed YAML frontmatter** during the survey. The helper returns exit 70; the survey pins (input, 70, None) for that input.

## Technical Constraints

- **Stream A canonical direction (epic 172) — coordination preference, not architectural rule.** Lifecycle is the canonical home for the clarify protocol because (a) lifecycle is the broader skill (12+ reference files vs refine's 1 functional + 1 critic), (b) lifecycle's SKILL.md already delegates Clarify/Research/Spec to refine, so the live trigger path runs through refine which already invokes the helper — making lifecycle's clarify.md a documentation-target rather than a runtime-target, and (c) sibling tickets 174 and 175 also collapse to lifecycle, so consistent direction simplifies reviewer comprehension. Reversing direction (refine canonical) is mechanically possible but would invert the broader-vs-narrower ownership pattern and break sibling consistency.
- **`just build-plugin` uses `rsync -a --delete`** at `justfile:509`. Mirror auto-pruning works for canonical-deletion → mirror-deletion; no manual mirror cleanup required.
- **Pre-commit dual-source drift hook.** Enforces canonical-vs-mirror parity. Canonical-file edits and deletions both require `just build-plugin` to land cleanly.
- **OQ3 escalation policy applies to behavior-correctness changes** (CLAUDE.md). Predicate-A removal IS a routing/correctness change (per Requirement 4 framing). The pre/post baseline survey (Requirement 5) is the evidence record: Step 5a freezes the union-predicate's behavior on the curated set; Step 5b asserts the post-removal helper either matches the baseline or has an explicit per-case judgment row. Any `legitimate-feature` divergence triggers the OQ3 events.log F-row + dispatch transcript requirement before merge.
- **Cortex's instruction-budget surface is already 43% over Horthy's threshold** (`research/vertical-planning/research.md`). #176 is net line-reducing (deletes 130-line refine clarify.md, adds ~10 lines to lifecycle's §1, removes ~25 lines of Predicate-A logic from the helper).
- **No new sharing primitive in the substitution layer.** Bare relative paths require no new substitution mechanism beyond Claude's existing relative-file-read resolution. The path-syntax form (`\.\./` cross-skill traversal) is new in the repo, but the resolution mechanism is asserted to be Claude's standard relative-read behavior anchored to the SKILL.md file's directory. The smoke-test matrix (Requirement 8) verifies this assertion under both critical invocation layouts.
- **No MUST/CRITICAL/REQUIRED language** in lifecycle's clarify.md §1 prose. Soft-positive-routing form is used per CLAUDE.md post-Opus 4.7 escalation policy default.
- **Test author's prior result is "A-in-union, not A-only-impossible."** `tests/test_resolve_backlog_item.py:300-398` ends with a passing test using input `4.7` against title `Upgrade to 4.7 stable` — Predicate B fires for this case too, so the test proves "A is part of the union" rather than "A-only constructions are impossible." This weakens the prior expectation that Predicate-A-only matches are absent in the live backlog; Requirement 5's reverse-engineered candidates (titles with stripped-by-slugify characters) backstop the empirical check.
