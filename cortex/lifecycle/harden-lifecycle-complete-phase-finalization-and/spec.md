# Specification: harden-lifecycle-complete-phase-finalization-and

## Problem Statement

Running `/cortex-core:lifecycle` Complete end-to-end on a trunk checkout (feature #291) surfaced four latent finalization-tooling defects plus, on investigation, a fifth `rework_cycles` writer. None blocked completion — each had an inline workaround — but each recurs on the next trunk-completed or concurrently-edited lifecycle. This feature hardens the Complete-phase finalization tail (`complete.md` Steps 10 & 11a) and reconciles the `rework_cycles` metric so a clean first-pass approval reports `0`, and the counter, the dashboard, and the morning report agree on the same definition for interactive features going forward.

## Phases

- **Phase 1: Finalization-tail skill fixes** — fix the dead index-sync fallback (defect 1), the trunk-path drift-drop (defect 2), and the unrelated-ticket sweep (defect 3), all in `complete.md`, with the structural test updated to the new staging contract.
- **Phase 2: rework_cycles counter reconciliation** — re-source `count_rework_cycles` to count rework iterations from `events.log` (defect 4), fix the fifth writer (`walkthrough.md` crash-recovery), and update the fixtures/parity tests/unit coverage and stale docstrings that the re-source touches.

## Requirements

1. **Defect 1 — working index-sync invocation.** In `complete.md` Step 10, the first index-regeneration fallback uses `python3 -m cortex_command.backlog.generate_index` instead of the bare `python3 cortex_command/backlog/generate_index.py` (which raises `ModuleNotFoundError`). The `test -f cortex_command/backlog/generate_index.py` source-tree guard is retained as the availability probe before the `-m` run. **Acceptance**: `grep -c 'python3 cortex_command/backlog/generate_index.py' skills/lifecycle/references/complete.md` = 0 AND `grep -c 'python3 -m cortex_command.backlog.generate_index' skills/lifecycle/references/complete.md` ≥ 1. **Phase**: Finalization-tail skill fixes.

2. **Defect 2 — stage the specific review-phase requirements-drift file on the trunk path.** `complete.md` Step 11a stages the auto-applied requirements-drift edit by the **exact path** the review phase recorded — read the `## Suggested Requirements Update` → `**File**:` value from `cortex/lifecycle/{slug}/review.md` and `git add --` that path (skip silently when no such section exists, i.e. no drift was applied). It does NOT use a directory-scoped `git add -u cortex/requirements/`, which would sweep unrelated tracked-modified requirements edits (e.g. an in-flight `project.md` change) into the finalization commit on the guard-less trunk path. **Acceptance**: the `<!-- finalization-commit-step -->` region references reading review.md's requirements-update `File` path and staging it, and contains NO `git add -u cortex/requirements/` and NO bare `git add cortex/requirements/`; verified by the updated `tests/test_complete_md_finalization_commit.py` (Requirement 8) — `just test` exits 0. **Phase**: Finalization-tail skill fixes.

3. **Defect 3 — scoped backlog staging that excludes unrelated untracked tickets.** `complete.md` Step 11a replaces the directory-scoped `git add cortex/backlog/` with: (a) `git add -u cortex/backlog/` (tracked-modified only — captures the `cortex-update-item` terminal-status cascade rewrites, all of which modify pre-tracked sibling/parent files, while skipping untracked unrelated tickets), plus (b) an explicit add of the resolved backlog item by slug-derived glob `git add -- cortex/backlog/*-{slug}.md` (covers the case where the resolved item itself was created-but-not-committed, i.e. untracked; the slug is already known to the skill, so the path is obtainable without the numeric ID). The enumerated lifecycle-artifact adds and the stage-first `git diff --cached --quiet` guard are preserved. **Acceptance**: the `<!-- finalization-commit-step -->` region contains `git add -u cortex/backlog/` AND a slug-globbed `cortex/backlog/*-` add, and does NOT contain the bare line `git add cortex/backlog/` (unscoped) nor `git add cortex/lifecycle/`; `just test` exits 0. **Phase**: Finalization-tail skill fixes.

4. **Defect 4 — rework_cycles counts rework iterations from events.log.** `cortex_command/lifecycle/counters.py:count_rework_cycles` is re-sourced to count `review_verdict` events with `verdict == "CHANGES_REQUESTED"` in the per-feature `events.log`, replacing the `review.md` verdict-block regex. **Signature change** (explicit): the function's parameter changes from `review_path: Path` to the events.log path (or `feature_dir`); `main()` (counters.py ~line 99-102) is updated to pass `feature_dir / "events.log"`; it no longer reads `review.md` for `rework_cycles`. **Malformed-line tolerance** (explicit): each line is parsed defensively — non-JSON / malformed lines are skipped, not raised on (consistent with the tolerant-reader convention for `events.log`). Missing/empty events.log → `0`. The module docstring (counters.py lines ~6-16, currently "count of `\"verdict\"` matches in review.md, pinned to common.py:182-183") is rewritten to describe the events.log CHANGES_REQUESTED source. **Acceptance** (via the unit test, Requirement 9): no events.log → 0; first-pass APPROVED (one APPROVED review_verdict, cycle 1) → 0; one CHANGES_REQUESTED then APPROVED → 1; two CHANGES_REQUESTED then APPROVED → 2; a stream with a malformed line plus one CHANGES_REQUESTED → 1 (malformed line skipped); `grep -c 'review.md' cortex_command/lifecycle/counters.py` docstring no longer claims review.md is the rework source. `just test` exits 0. **Phase**: rework_cycles counter reconciliation.

5. **Fifth writer — fix walkthrough.md crash-recovery.** `skills/morning-review/references/walkthrough.md` step 6b (crash-recovery `feature_complete` emission, ~lines 280-283) writes `rework_cycles` using the corrected semantics (count of `CHANGES_REQUESTED` `review_verdict` events / `0` for a clean approval), not the raw `cycle` number `C`. The step-5 synthetic path (~line 260) emits an `APPROVED` `review_verdict` with `cycle: 0`; it is left unchanged and is correctly counted as `0` rework by the new counter (it is not `CHANGES_REQUESTED`) — the spec notes this so the synthetic path is confirmed consistent, not overlooked. **Acceptance**: `grep -c 'cycle number from the last' skills/morning-review/references/walkthrough.md` = 0 AND `grep -c 'CHANGES_REQUESTED' skills/morning-review/references/walkthrough.md` ≥ 1. **Phase**: rework_cycles counter reconciliation.

6. **Document the distinct sites; do not touch the already-correct one.** `cortex_command/dashboard/data.py`'s independent phase-transition computation is KEPT and its docstring is left UNCHANGED — verification confirmed it already documents the review-predecessor guard ("transitions that follow a `to == \"review\"` transition", data.py ~lines 298-301), so no docstring edit is needed. `cortex_command/common.py`'s `review_verdict_count` / `cycle` number (~lines 257, 296) is a deliberately distinct cycle-number and is left unchanged in logic, with a one-line code comment added noting it is NOT `rework_cycles`. `cortex_command/pipeline/metrics.py:236` continues to read `rework_cycles` verbatim from the event (no change). **Acceptance**: `git diff cortex_command/dashboard/data.py` shows no docstring/logic change to the rework_cycles computation; `grep -c 'rework_cycles' cortex_command/common.py` ≥ 1 (the new clarifying comment; every occurrence a comment, no new computation); `just test` exits 0. **Phase**: rework_cycles counter reconciliation.

## Non-Requirements

- Does NOT migrate or rewrite historical `feature_complete` events. The append-only ledger convention holds; the `avg_rework_cycles` aggregate (which averages over all completed features, archived included, with no anchor partition) therefore retains historical over-counted interactive values indefinitely — see Technical Constraints for the accurate backward-compat statement.
- Does NOT change `common.py`'s `cycle` number semantics (a distinct cycle-number, not a rework count) — only adds a clarifying comment.
- Does NOT change the `feature_complete` event name, schema_version, `merge_anchor`, or any events-registry entry (a value-semantics change to an existing field is neither a new event nor a new field).
- Does NOT change `dashboard/seed.py`'s demo fixtures or `dashboard/templates/feature_cards.html` — both are value-agnostic renderers that display whatever `rework_cycles` value flows through; they were reviewed and require no change under the corrected semantics.
- Does NOT alter `tests/failure_matrix.py`'s bucketing logic — it is a descriptive one-shot report (not an alert) reading historical events; its exact-value buckets are left as-is per the no-migration decision.
- Does NOT touch lifecycle phase boundaries, the Step-6 user pause, or the kept-pauses inventory.
- Does NOT add a `bin/` staging helper or a hermetic git-staging dry-run harness for defects 2/3 (they remain skill-prose, covered by the strengthened structural token test). Behavioral staging coverage is explicitly deferred as out of scope.
- Does NOT edit `bin/cortex-lifecycle-counters` (a dual-channel wrapper that execs `python3 -m cortex_command.lifecycle.counters`; changing the module changes both channels).
- Does NOT modify `cortex_command/backlog/update_item.py` to report its cascade writes (the chosen skill-only `git add -u` captures the pre-tracked cascade without it).

## Edge Cases

- **Simple-tier feature (no review):** events.log has no `review_verdict` events → `count_rework_cycles` returns 0.
- **REJECTED-then-escalate:** REJECTED is not `CHANGES_REQUESTED` and routes to user escalation without re-implementation → counts as 0 rework, correctly.
- **Malformed/legacy line in events.log:** skipped per-line; does not raise (Requirement 4 tolerance).
- **review.md overwritten per round:** irrelevant under the new source — events.log is append-only and retains every `review_verdict`.
- **Resolved backlog item is untracked (created-but-not-committed):** captured by the slug-globbed explicit add `git add -- cortex/backlog/*-{slug}.md` (Requirement 3b), not missed.
- **Concurrent tracked-modified UNRELATED backlog ticket on the trunk path:** `git add -u cortex/backlog/` would still stage it (accepted residual — far rarer than the original untracked-ticket sweep; the guard-less trunk path cannot distinguish it without per-file enumeration the skill cannot perform).
- **Brand-new UNRELATED untracked ticket authored mid-lifecycle (a follow-up, not the resolved item):** `git add -u` skips it and the slug-glob does not match it → not staged (accepted residual).
- **Requirements drift over-stage:** eliminated — Requirement 2 stages only the exact `File` path review.md recorded, not all dirty `cortex/requirements/` files, so an unrelated in-flight `project.md` edit on the trunk path is NOT swept in.
- **index.json / index.md:** gitignored — never staged (correctly; the old prose claim that the dir-add captured them was misleading).

## Technical Constraints

- **rework_cycles canonical definition** = number of `CHANGES_REQUESTED` `review_verdict` events in the per-feature events.log. This matches `dashboard/data.py`'s review→implement loop-back count on the corpus's event-emitting features; the two are not definitionally identical (a hypothetical 2nd-cycle `CHANGES_REQUESTED` would be counted by counters.py but emit a `review → escalated` transition that data.py does not count), but the capped 2-cycle review loop does not produce that stream in practice, so they agree empirically. `common.py`'s `cycle` is a separate, intentionally-distinct number.
- **events.log availability:** the interactive review phase emits `review_verdict` events to `cortex/lifecycle/{feature}/events.log` (`review.md:161-164`), so the source is present on the interactive complete path — the only path that writes `rework_cycles` via counters.py (the overnight path's `feature_complete` omits the field; `metrics.py` reads `None` there).
- **Backward-compat (accurate):** historical interactive `feature_complete` events keep their old inflated `rework_cycles`; there is no migration. `metrics.py:236` reads the field verbatim and `compute_aggregates` averages it across ALL completed features (archived included) with no `merge_anchor` partition, so `avg_rework_cycles` permanently blends old-inflated and new-correct interactive values — it does not converge to truth, it dilutes toward it as new features accumulate. This is descriptive-only (the morning-report calibration line); no alert keys off `avg_rework_cycles`. The dashboard live card (data.py) and the morning-report aggregate (event-stored) can therefore disagree on a historical interactive feature; this is accepted, not fixed.
- **Test gates:** the strengthened `tests/test_complete_md_finalization_commit.py` must bind `-u` to the backlog path so a scoping regression fails. SKILL.md-to-bin parity, size budget, kept-pauses parity, and events-registry gates are not newly tripped.
- **Sequencing:** implement as two commits — Commit A (Phase 1: complete.md + finalization-commit test + mirror); Commit B (Phase 2: counters.py + signature/docstring + fixtures + unit test + both parity tests + walkthrough.md + common.py comment + mirror). The two-commit order is safe: complete.md Step 11's "no review → rework_cycles 0" prose holds under both old and new counters.py, and Phase 1 shipping before Phase 2 leaves no inconsistent window (the only behavioral change is fixing the over-count, not a contract the other phase depends on).
