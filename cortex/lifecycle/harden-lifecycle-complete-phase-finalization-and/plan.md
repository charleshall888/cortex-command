## Risks
- **A1 deviation from approved spec (surface at approval).** Spec Requirement 3b literally specifies `git add -- cortex/backlog/*-{slug}.md`. That glob matches zero files because the lifecycle slug is a truncated prefix of the backlog filename slug (verified). Task 1 substitutes resolver-based staging (`cortex-resolve-backlog-item {slug}` → exact filename), which preserves Requirement 3's *intent*. This is a correctness fix to the spec's mechanism, made at Plan time because the high-criticality critical-review surfaced it; the user can object at plan approval.
- **A2 commit model is harness-native, not the spec's two-commit deferral.** The Implement builder template commits every dispatched task and marks no-commit tasks as failed, and worktree dispatch isolates each task's edits until merge-back — so "do not commit / defer to a commit task" cannot work. Each task here self-commits; `build-plugin` is folded into the canonical-edit tasks so canonical + mirror co-commit. This yields four commits, not two; the grouping is logical (finalization-tail; counter-core; walkthrough; common-comment), each independently green at its commit. This is a deliberate, documented deviation from the spec's two-commit idealization.
- **Atomic 7-file counter task.** Task 2 exceeds the 1–5 file guideline by necessity: the counter semantics change and its fixtures/parity/unit tests must commit together to keep `just test` green. Structural atomicity is preferred over a prose-coordinated split that would guarantee a red window.
- **Behavioral staging coverage deferred (per spec).** Defects 2/3 ship as `complete.md` prose verified by the structural-token test, which checks that the *correct mechanism is referenced* (resolver invocation, `git add -u`, absence of the broken glob) but does NOT execute git staging. A token-satisfying-but-wrong prose edit is not caught at runtime; the spec explicitly defers a behavioral staging harness as out of scope.
- **counters.py ↔ data.py cap-boundary divergence (pre-existing, accepted).** A cycle-2 `CHANGES_REQUESTED` → escalated → user-directed-completion produces counters.py = 2 vs data.py = 1 (data.py never counts `review → escalated`). This is the concrete instance of the divergence the spec's Technical Constraints already declares "accepted, not fixed"; data.py is intentionally left unchanged (Requirement 6). The `0`-on-absent-verdicts ambiguity in the new counter is a pre-existing exposure (the old review.md counter shared it) on a path other than the interactive complete path that invokes counters.py.
- **Path-scoped staging is mandatory under sequential dispatch.** The main checkout carries unrelated in-flight changes (`pipeline/parser.py`, backlog #293); every commit stages explicit paths, never `git add -A`.
