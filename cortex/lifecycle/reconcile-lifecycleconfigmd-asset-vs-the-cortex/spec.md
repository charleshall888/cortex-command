# Specification: Reconcile lifecycle.config.md asset vs the cortex-init template (drift, no sync gate)

## Problem Statement

The lifecycle config schema lives in two independently-maintained files in different distribution channels: the **skills asset** `skills/lifecycle/assets/lifecycle.config.md` (ships in the `cortex-core` plugin; what docs tell users to copy and what plugin-only users read as the schema reference) and the **init template** `cortex_command/init/templates/cortex/lifecycle.config.md` (what `cortex init` drops verbatim into a repo). They drifted: the init template carries the #317 `backlog:` backend block; the asset does not. Nothing links the two, so the omission was silent, and the docs compound it by citing the stale asset as the "source of truth." A user copying the asset gets a config missing the backend section. This reconciles the asset up to the template, adds a parity test that **fails `just test`/CI if the two frontmatters diverge** (catching the next such omission before merge, though not at commit time ŌĆö see Non-Requirements), and (per the operator's broad-scope choice) corrects the docs that enumerate the schema.

## Phases

- **Phase 1: Reconcile + Parity Gate** ŌĆö add the `backlog:` block to the asset, add a frontmatter parity test (true byte compare + a positive content assertion) that fails CI on future assetŌåötemplate divergence, and record the design decision as an ADR.
- **Phase 2: Docs Consolidation** ŌĆö repoint and correct the docs that enumerate the config schema so they cite one consistent authority and accurately reflect the scaffolded field set, with consumed-but-unscaffolded fields flagged as explicit exceptions.

## Non-Requirements

- Does **not** generate the asset from the init template (no codegen, no "GENERATED ŌĆö DO NOT EDIT" header). Rejected: incoherent on a file users are told to copy and edit, and inverts the CLIŌåÆplugin channel dependency. (Adversarial #4ŌĆō5.)
- Does **not** collapse the asset to a thin pointer or delete it. Plugin-only users (installed `cortex-core` via `/plugin install`, never `uv tool install`ed the CLI) have neither the init template nor `cortex init` on disk; the asset is their only schema reference.
- Does **not** edit the init template's content. Reconcile is **up** (asset ŌåÉ template's block), never down. Editing the template changes the init-artifacts hash and fires a one-time `.cortex-init` drift report into every initialized repo; reconciling the asset avoids that, and a collapse-down would silently regress #317 (no test catches it ŌĆö `test_init_backlog_scaffold.py` only asserts the default). (Init-scaffold + Adversarial #1.)
- Does **not** scaffold `branch-mode` (or any new field) into the templates. `branch-mode` is consumed but absent from both scaffolded templates today. Adding it would require editing the asset **and** the init template together (R3 parity binds them) ŌĆö i.e. it pulls in the init-template edit this ticket scopes out. This ticket *documents* `branch-mode` as a consumed-but-unscaffolded exception (R8); *scaffolding* it into the templates is a separate follow-up ticket.
- Does **not** add a **consumer-side** no-regression default assertion for a future non-default-safe field (e.g. a field where absence is not safely equivalent to a default). This is distinct from file-presence parity, which R3 already delivers: R3 fails on any field present in the template and absent from the asset, default-safe or not. The deferred concern is the consumer behavior when such a field is *absent from a user's config*, which is out of scope while `backlog.backend` is default-safe.
- Does **not** add a pre-commit Phase or `just check-*` recipe for the assetŌåötemplate gate. The assetŌåömirror pair is already pre-commit-blocking; the assetŌåötemplate pair is caught at `just test`/CI before merge. This is a deliberate friction trade-off (CI-time, not commit-time enforcement ŌĆö see Problem Statement); the pytest is sufficient given CI gates merges.
- Does **not** add a parity gate over the asset/template **body** prose (the `# Lifecycle Configuration` intro, `## Review Criteria` section). The gate covers the frontmatter (the field list) only; the body is lower-stakes and intentionally divergent (the "Copy this fileŌĆ”" sentence). Body documentation drift is an accepted, unguarded surface ŌĆö not claimed as gate-protected.
- Does **not** fix the repo's own project-root `cortex/lifecycle.config.md`. It is a live per-repo config expected to diverge from the template after init.

## Edge Cases

- **Comment drift inside the `backlog:` block ŌĆö single-file** (e.g., the `github-issues` example dropped from one file only): caught by R3's two-file compare (the regions diverge). Unlike a `yaml.safe_load` parsed-dict compare, the byte-region compare sees comment lines.
- **Comment drift ŌĆö convergent** (the same option line deleted from *both* files at once): the two-file compare alone would stay green (the regions remain equal). Caught instead by R4's **positive content assertion** (the required option lines must be present in the asset). This is the residual reason the gate is not a pure two-file diff.
- **The intended body sentence** ("Copy this file to your project rootŌĆ”"): tolerated for free ŌĆö it is in the body, outside the frontmatter region the gate compares. No allowlist entry needed.
- **A future *intended* frontmatter difference between the two files** (e.g. an init-only key): would fail R3. There is no drop-in tolerance ŌĆö `test_parity_contract.py`'s named-tolerance set is JSON-stream-specific and does not transfer to a markdown byte-slice. Resolution is a deliberate gate redesign (compare a named sub-region, fail-loud) recorded at that time, not a quiet allowlist add. Today no such difference exists; the gate stays whole-frontmatter.
- **Direct edit of the plugin mirror**: already caught by the pre-commit dual-source hook + `tests/test_dual_source_reference_parity.py` glob. Unchanged.

## Technical Constraints

- Reconcile direction is **up** (asset ŌåÉ init-template block); never strip the template.
- The gate is a **true byte compare** of the raw frontmatter region (bytes between the `---` delimiters), read via `Path.read_bytes()` + slice ŌĆö NOT via `_extract_frontmatter_text` (whose `splitlines()`+`"\n".join()` normalizes line endings and would mask CRLF/trailing-newline drift). It additionally asserts the asset frontmatter contains the load-bearing option lines (R4).
- After the asset reconcile, the asset frontmatter == init-template frontmatter byte-for-byte (verified: lines 1ŌĆō17 are already identical; only the block differs). The gate guarantees parity on the **frontmatter region only** ŌĆö the bodies deliberately differ and are not gated.
- Edit the canonical asset only; regenerate the `plugins/cortex-core` mirror with `just build-plugin` and stage it in the same commit (pre-commit dual-source drift hook blocks otherwise).
- New test at `tests/test_*_parity.py`; runs under `just test` (`.venv/bin/pytest tests/ -q`). Enforcement is CI-time (gates merge), not commit-time.
- Editing `skills/` is lifecycle-gated; commit via `/cortex-core:commit`.

## Proposed ADR

### Proposed ADR: 0017-reconcile-and-gate-lifecycle-config-sources

**Context.** The lifecycle config schema exists as two near-identical, independently-maintained files in different distribution channels ŌĆö the `cortex-core` plugin asset (`skills/lifecycle/assets/lifecycle.config.md`, what docs cite and plugin-only users read) and the CLI init template (`cortex_command/init/templates/cortex/lifecycle.config.md`, dropped verbatim by `cortex init`). They share all frontmatter except a `backlog:` block, plus one intended body difference (the asset's "Copy this fileŌĆ”" sentence). No tooling linked them, so they drifted silently.

**Decision.** Keep both files hand-maintained and reconcile the asset **up** to the template, guarded by a parity test that byte-compares the **frontmatter region** (tolerating the body sentence for free, plus a positive content assertion catching convergent comment loss). Keep the asset as the canonical schema referent in docs **for the scaffolded field set**; the parity gate is what makes that citation trustworthy *on the frontmatter / field list* (the asset's frontmatter is provably in sync with the operational template). The asset's body prose and any consumed-but-unscaffolded field (e.g. `branch-mode`) are explicitly outside the gate's guarantee and are handled as documented exceptions.

**Trade-off.** Accepts ongoing dual-maintenance of the shared frontmatter in exchange for avoiding: (a) a cross-channel generator dependency inversion (CLI package driving plugin source) and a self-contradictory "DO NOT EDIT" header on a copy-target file (rejected: generate-from-source); (b) blindness to comment loss (rejected: `yaml.safe_load` parsed-dict gate); (c) stranding plugin-only users who lack the CLI (rejected: thin-pointer/delete); and (d) leaving recurrence unguarded while the schema is still moving under #318 (rejected: bare reconcile with no gate). Enforcement is CI-time, not commit-time ŌĆö a deliberate friction trade-off. The maintenance *mechanism* (hand-maintain + gate) is reversible to a generator later, but the content reconcile is effectively a one-way ratchet once plugin-only users have copied the completed asset, and reversing the channel arrangement would require re-architecting the two-channel distribution (ADR-0002), not merely a cheaper generator.
