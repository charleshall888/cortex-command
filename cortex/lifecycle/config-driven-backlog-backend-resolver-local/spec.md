# Specification: Config-driven backlog backend (resolver, local/none routing, overnight safety) — #317

> Epic context: parent [[315-optional-backlog-plugin-configurable-backend]]; authoritative area spec `cortex/requirements/backlog.md`; epic analysis `cortex/research/backlog-optional-plugin/research.md`. This ticket delivers the config-respecting, opt-out-able, overnight-safe backlog that works **without any external tracker**.
>
> **Sequencing**: #316 (plugin extraction + the `/cortex-core:backlog`→`/cortex-backlog:backlog` rename) lands first (in implementation now). #317 is planned/implemented on top of the renamed state, so the consumer files #317 edits will already carry the new namespace; #317 adds routing guards, it does not rename. #318 (external best-effort arm) is deferred behind a real external-tracker user.
>
> **Owner's bar (load-bearing acceptance lens for this spec):** building the configurable backend must **(1) never change the behavior of a normal `cortex-backlog` (local) repo**, and **(2) add no meaningful token cost to the normal path.** Every requirement below is designed so the routing mechanic *fails toward today's behavior* and keeps its non-local prose off the hot path. The "Local-path invariance" section states the guarantee; the per-requirement acceptance criteria enforce it.

## Problem Statement

Every backlog consumer hard-codes the local backlog and the `cortex init` config has no switch to say "this repo uses a different tracker" or "don't manage tickets here." Multiple real users on external trackers need this. This ticket introduces the configurable-backend seam: a config-authoritative resolver (default `cortex-backlog`), a `backlog:` config block, backend routing in the interactive consumers for the local and `none` arms, and a structural overnight guard.

**Where the overnight guard sits (precise framing):** the only code path that selects/bootstraps an overnight batch is the (attended) prep step — `handle_prepare`/`handle_launch`, invoked from the interactive `/overnight` skill where a human is at the Approve gate. The unattended launchd fire (`overnight start --scheduled` → `handle_start`) consumes an *already-bootstrapped* state file and never re-resolves the backend or re-selects. So gating the prep verbs means a non-local session can never be bootstrapped, and the unattended runner therefore only ever runs sessions already guarded under `cortex-backlog`.

## Local-path invariance (the owner's bar, made structural)

These two properties are guarantees, not aspirations, and each is tied to a requirement:

- **Fail-toward-normal.** There is no config or environment input that makes a `cortex-backlog` repo behave differently from today. The resolver returns `cortex-backlog` for every degenerate input (absent file/block, scalar `backlog:`, null/empty `backend`, malformed YAML — R1); the reader binstub prints `cortex-backlog` and exits 0 even when the wheel can't be resolved (R2, graceful console-script shape); and each consumer's local arm is the *default first branch*, so even a misread guard proceeds normally (R7). Behavior diverges only when the operator explicitly sets `backend: none` or a non-local value. The one place the fail direction is deliberately *opposite* — the overnight guard fails closed — is in-process code with its own test (R5/R6), and the two readers must not be DRY-merged (Technical Constraints).
- **Off-hot-path cost.** On a normal run, each backlog-touching consumer reads **one guard line** and performs **one cheap config read** (the same mechanism already used by `cortex-read-commit-artifacts`/`cortex-read-branch-mode`). All none/external routing prose is reached only when the resolved backend ≠ `cortex-backlog` — never, for a local repo. No new logged events; no per-consumer fallback prose (the resolver owns the default). The refine local seed/reconcile invocations stay byte-identical with no Python change (R8).

## Phases
- **Phase 1: Resolver + config seam + ADR** — `resolve_backlog_backend`, its CLI module + graceful binstub, the scaffold `backlog:` block, and ADR-0016.
- **Phase 2: Overnight safety guard** — a structural, in-process, fail-closed first-check that refuses non-local backends in `handle_prepare`/`handle_launch` via a JSON error envelope.
- **Phase 3: Consumer routing + degrade + criticality-feed decoupling** — inline ~1-line backend guards (local/none) in the interactive consumers, structural-consumer degrade, `none` discovery behavior, and the Design-A criticality-feed fix.

## Requirements

1. **Backend resolver (config-authoritative, fail-toward-default).** `resolve_backlog_backend(repo_root) -> str` in `cortex_command/lifecycle_config.py`, reusing `_extract_frontmatter_text` (`:29-46`) and mirroring `read_branch_mode` (`:49-88`). Descends the **nested** `backlog:` mapping to read `backend`, with an explicit `isinstance(backlog_block, dict)` guard before `.get("backend")`. Returns `"cortex-backlog"` for every degenerate case (absent file/block, scalar block, null/empty `backend`, malformed YAML); returns the raw string for an explicit value. Must NOT introspect installed plugins. **Acceptance**: pytest covering absent-file, absent-block, scalar block, null/empty backend, a valid value, and malformed YAML — each resolves without raising; all degenerate cases return `"cortex-backlog"`; `just test` exits 0. **Phase**: Resolver + config seam + ADR.

2. **Resolver CLI module + graceful binstub.** New `cortex_command/lifecycle/backlog_backend_cli.py` mirroring `cortex_command/lifecycle/branch_mode_cli.py` (argparse, `_telemetry.log_invocation`, stdout = value + newline, return 0); `pyproject.toml [project.scripts]` entry `cortex-read-backlog-backend`; binstub `bin/cortex-read-backlog-backend` (`chmod +x`) + auto-swept mirror `plugins/cortex-core/bin/cortex-read-backlog-backend`. **Failure contract — graceful, not exit-2**: when the wheel cannot be resolved, the binstub prints `cortex-backlog` and exits 0 (the `branch_mode_cli` graceful shape), NOT the exit-2 shape of `cortex-read-commit-artifacts`. This makes the interactive fail-safe live in the reader, so no consumer carries fallback prose, and removes the `set -euo pipefail` abort risk for an unconfigured repo. **Acceptance**: a pytest invokes the CLI module and asserts stdout = resolved backend + newline for a fixture config, and = `cortex-backlog` + exit 0 on an unresolvable invocation; `grep -c "cortex-read-backlog-backend" pyproject.toml` ≥ 1; the dual-source drift gate passes with canonical + mirror co-landed; `just test` exits 0. **Phase**: Resolver + config seam + ADR.

3. **Config scaffold `backlog:` block.** Add a `backlog:` block to `cortex_command/init/templates/cortex/lifecycle.config.md`: `backend: cortex-backlog` (default) + commented `# backend: github-issues|jira|none` alternatives + an optional freeform `instructions:` string with a documented `github-issues` example, and a comment noting external backends are best-effort and land in #318. (Documenting the external example is mandated by `backlog.md:35`.) **Acceptance**: a pytest reads the scaffolded template **through `resolve_backlog_backend`** (R1) and asserts it returns `"cortex-backlog"` — proving the block is nested at the path the resolver descends, not merely that the string `backlog:` is present; `tests/test_init_artifacts_hash_inputs.py` passes; `just test` exits 0. **Phase**: Resolver + config seam + ADR.

4. **ADR-0016.** `cortex/adr/0016-configurable-backlog-backend-and-llm-as-adapter.md` (0015 is already taken — re-confirm the next free number at write time), `status: proposed`, with context / decision / rejected-alternatives (per-tool code adapters; plugin-introspection) / consequences, clearing the three-criteria gate. Back-pointer added from `cortex/requirements/backlog.md`. **Acceptance**: the ADR file exists with the four sections; `grep -rc "0016" cortex/requirements/backlog.md` ≥ 1. **Phase**: Resolver + config seam + ADR.

5. **Overnight refusal guard (structural, in-process, fail-closed).** A shared helper invoked at the top of `handle_prepare` and `handle_launch` — before `select_overnight_batch` (`cli_handler.py:2013`/`:2087`) and before `bootstrap_session` — resolves the backend **in-process via `resolve_backlog_backend(repo_path)`** (reading config directly; it must NOT shell the graceful binstub, which would inherit the fail-*open* default) into a variable named `backlog_backend` (avoiding the scheduler `backend` at `:1125`/`:1756`), and refuses unless the value is exactly `cortex-backlog`. Refusal is a versioned JSON error envelope `{"version": "1.0", "error": "backend_not_supported", "message": ...}` naming the configured backend, plus non-zero exit — matching the `selection_failed` envelope at `:2020`/`:2094`. The guard stays at the handler layer, NOT inside `select_overnight_batch` (which has no `repo_root`, returns a `SelectionResult` not an envelope, and is called by unit tests with no config). **Acceptance**: pytest asserts that with an explicit non-`cortex-backlog` backend, `handle_prepare`/`handle_launch` each return the `backend_not_supported` envelope + non-zero exit and `select_overnight_batch` is not reached (patched to raise); and that an absent `backlog:` block proceeds normally; `just test` exits 0. **Phase**: Overnight safety guard.

6. **Guard short-circuits before any selection or write (positive assertion).** **Acceptance**: pytest asserts (positively) that on a non-local backend the guard's refusal return is the first observable effect — `select_overnight_batch` and `bootstrap_session` (both patched to raise if reached) are never called; `just test` exits 0. **Phase**: Overnight safety guard.

7. **Interactive consumer backend routing (inline, local-default).** Each backlog-touching consumer — lifecycle write-back (`skills/lifecycle/references/backlog-writeback.md`), discovery decompose (`skills/discovery/references/decompose.md` + `SKILL.md`), refine (`skills/refine/SKILL.md`), dev (`skills/dev/SKILL.md`), morning-review (`skills/morning-review/references/walkthrough.md`) — carries a short **inline** guard (no shared cross-plugin reference; see Technical Constraints for why): resolve the backend once via `cortex-read-backlog-backend`; if `cortex-backlog` (the default arm), proceed exactly as today; if `none`, skip with a one-line advisory; any other value → a one-line advisory that external support is configured but not yet available (#318), surfacing output rather than erroring. Consumers carry **no** fail-safe/fallback prose (R2's graceful reader owns that). The `cortex-*` backlog CLIs gain NO backend awareness. **Acceptance**: (structural) each edited consumer contains a reachable guard whose first/default arm is the unchanged local behavior and that includes a `none` arm — verified by grep for the reader invocation and the `none` branch; (cost) the local (default) arm requires no additional reference load; (lint) no new MUST/REQUIRED (escalation lints pass); `tests/test_l1_surface_ratchet.py` passes. Routing behavior in pure-prose consumers is model-executed and verified structurally + by interactive confirmation; the code-seam consumers are covered by R9. `just test` exits 0. **Phase**: Consumer routing + degrade + criticality-feed decoupling.

8. **Criticality-feed decoupling (Design A, local byte-identical).** Lifecycle state (local `events.log`, reduced by `common.reduce_lifecycle_state`) is the authoritative tier/criticality feed — it already is the read source for both the critical-review gate and overnight. **Local path is byte-identical**: under `cortex-backlog` the refine seed/reconcile invocations are unchanged (`emit-lifecycle-start --backlog-slug … --lifecycle-slug …`; `reconcile-clarify --lifecycle-slug … --backlog-slug …`). **Only the non-local branch changes**: keyed on the resolved backend, both invocations OMIT `--backlog-slug` (the existing Context-B form), so `_read_backlog_frontmatter(None)` returns defaults without reading/validating any local file (structurally avoiding the `exit-64` at `refine.py:157`/`:231` on a stale/invalid file — config-authoritative: a non-local backend ignores local files even if present), and Clarify's computed values pass as explicit `--complexity/--criticality` flags (precedence at `refine.py:158-160`; monotonic ratchet). The `cortex-update-item --complexity/--criticality` write-backs (Clarify §7 at `skills/lifecycle/references/clarify.md`; refine write-back) are gated on `backend == cortex-backlog`. **No change to `cortex_command/refine.py`, `common.py`, or `state_cli.py`.** **Acceptance**: (a) a regression pytest asserts `reconcile-clarify` honors explicit `--complexity complex --criticality high` (events.log reduces via `cortex-lifecycle-state --field tier` to `complex`); (b) (structural) grep asserts the refine skill's non-local branch omits `--backlog-slug` from both invocations and passes explicit flags, while the local branch is unchanged; `just test` exits 0. **Phase**: Consumer routing + degrade + criticality-feed decoupling.

9. **Structural-consumer degrade (ordering-assertable).** Under a non-local backend, dev's epic-map/triage (`skills/dev/SKILL.md`, reads `cortex/backlog/index.{md,json}` directly) skips triage with an advisory routing to `/cortex-core:lifecycle`/`/cortex-core:discovery`; refine's parent-epic alignment (`skills/refine/references/clarify-critic.md`, via `cortex-load-parent-epic`) guards off. Where a CLI seam exists (`cortex-build-epic-map`/`cortex-load-parent-epic`), the backend read gates the call so the structural local-index read is not reached under a non-local backend (the same shape as R5). **Acceptance**: the backend-gated advisory/guard is present (grep); where a CLI seam exists, a check asserts the structural read is gated; end-to-end is interactive. `just test` exits 0. **Phase**: Consumer routing + degrade + criticality-feed decoupling.

10. **`none`-backend discovery surfaces composed bodies.** Under `backend: none`, discovery surfaces composed ticket bodies inline (in `decomposed.md` / its research artifact) rather than failing. **Acceptance**: (structural) discovery decompose prose contains a reachable `none`-branch that surfaces bodies inline rather than calling the create CLI (grep); end-to-end is interactive. **Phase**: Consumer routing + degrade + criticality-feed decoupling.

11. **Two readers, opposite fail directions — kept distinct (anti-DRY).** The interactive fail-safe (empty/unresolvable → `cortex-backlog`) lives in R2's graceful binstub/resolver default; the overnight fail-closed (unresolvable/non-local → refuse) lives in R5's in-process strict check. These must remain two distinct code points — a future "simplification" that routes the overnight guard through the graceful reader would silently make it fail-open (a safety regression). **Acceptance**: R5/R6's tests assert the overnight refuse-direction; a comment/Technical-Constraint documents the asymmetry so it is not DRY-merged; `just test` exits 0. **Phase**: Consumer routing + degrade + criticality-feed decoupling.

## Non-Requirements
- **No external-tracker code path** (LLM best-effort create/round-trip via `gh`) — that is #318. #317's external arm is a one-line "configured but not yet available (#318)" advisory.
- **No slash-rename in #317** — the `/cortex-core:backlog`→`/cortex-backlog:backlog` rename is owned by #316 (landing first). #317 adds routing to the already-renamed consumers; it does not rename.
- **No shared cross-plugin routing reference** — the routing guard is inlined per consumer. A single shared reference is infeasible: the five consumers span two plugins (morning-review builds into cortex-overnight, the others into cortex-core), and `${CLAUDE_SKILL_DIR}/../sibling` resolves only intra-plugin while ADR-0009 forbids the bare-relative escape hatch. The guard is short enough that inlining is cheaper than a dual-source-mirrored reference.
- **No removal of the backlog engine from the wheel** — `cortex_command/backlog/*` stays as a shared library.
- **No backend awareness in the `cortex-*` backlog CLIs** — they remain the cortex-backlog-only local engine; routing lives in the consumer/skill layer. (If a future need to drive normal-path cost to literally zero arises, pushing local/none routing into the tools is the alternative — explicitly out of scope here.)
- **The directly-invoked `/cortex-backlog:backlog` management command is exempt from backend routing** — it *is* the local-engine surface; explicit user invocation is a deliberate local action, distinct from the consumers that auto-touch tickets.
- **No structural read-back backstop before the critical-review gate** — by decision, the harness trusts the criticality wiring rather than adding a defensive re-check.
- **No new logged event** — the overnight refusal uses a JSON error envelope (not an events-registry event): the refusal is synchronous and attended (the operator reads stdout at the Approve gate), so the envelope is the observable surface and a logged event would add no observability while incurring the events-registry dual-gate.
- **No fix to the pre-existing independent-axis behaviors** — task-complexity (plan.md) vs feature-tier (events.log), discovery research-sizing as its own stream, and the dashboard `dispatch_start` complexity field are intentional/independent and out of scope.
- **No interactive backend prompt in `cortex init`** — scaffold-with-default + commented alternatives only.
- **No re-resolution inside the unattended runner / no backend gate on the runner's finalize follow-up writer** — those writes are local `cortex/backlog/` writes, never external, and unreachable under a non-local backend because such a session can never be bootstrapped past the R5 prep guard.

## Edge Cases
- **Absent `backlog:` block** → `cortex-backlog`; zero behavior change (implicit migration).
- **Scalar `backlog:` value / null-empty `backend` / malformed YAML** → `cortex-backlog` (the `isinstance` guard prevents `AttributeError`); local proceeds, overnight proceeds (safe — local, no external write).
- **Unrecognized/typo backend value** (e.g. `cortx-backlog`) → treated as the external bucket: interactive shows the "configured, not yet available (#318)" advisory; overnight refuses. Fail-safe; no typo detection added.
- **Binstub unresolvable** (wheel off PATH) → prints `cortex-backlog` + exit 0 (R2 graceful); interactive proceeds normally. The overnight guard resolves in-process (not via the binstub), so it is unaffected by binstub PATH issues.
- **Stale local backlog files while `backend: non-local`** → config-authoritative; the refine non-local branch omits `--backlog-slug` and never reads/validates the stale file (no `exit-64`); tier/criticality flow from Clarify.
- **Overnight invoked under non-local backend** → R5 refusal envelope naming the configured backend; selection/bootstrap not reached.

## Changes to Existing Behavior
- **ADDED**: `backlog:` config block; `resolve_backlog_backend` + graceful `cortex-read-backlog-backend` reader; inline backend guards in five interactive consumers; the in-process overnight refusal guard.
- **MODIFIED (non-local only)**: the Clarify §7 and refine `cortex-update-item --complexity/--criticality` write-backs become backend-gated (skip under non-local); under a non-local backend the refine seed/reconcile omit `--backlog-slug` and pass Clarify's explicit values. **Under `cortex-backlog`, behavior is byte-identical to today (output and refine control flow).**
- **No REMOVED behavior** for existing local-backlog repos. (The `/cortex-core:backlog`→`/cortex-backlog:backlog` rename + new-plugin install requirement are #316's changes, not #317's.)

## Technical Constraints
- Backend resolution is config-authoritative; never introspects installed plugins (`backlog.md:45`).
- The `cortex-*` backlog CLIs stay cortex-backlog-only (`backlog.md:100`).
- Routing prose is soft positive-routing (no MUST/REQUIRED); the overnight guard is structural control flow (a code gate — preferred per CLAUDE.md "structural separation over prose"); the R9 structural-degrade gates are verified by ordering (the R5 shape), not string presence alone.
- **Reader failure contract is graceful** (print `cortex-backlog`, exit 0 on unresolvable) — the `branch_mode_cli` console-script shape, not the exit-2 `cortex-read-commit-artifacts` shape — so the interactive fail-safe lives in the reader and no consumer carries fallback prose.
- **The overnight guard resolves in-process** (`resolve_backlog_backend`), strict-compares to `cortex-backlog`, and must NOT be routed through the graceful binstub. The interactive (fail-open) and overnight (fail-closed) readers are deliberately two distinct code points — do not DRY-merge them (R11).
- The overnight guard stays at the handler layer (a shared helper in `handle_prepare`/`handle_launch`), NOT inside `select_overnight_batch` (no `repo_root` access; returns `SelectionResult` not an envelope; called config-less by unit tests).
- The overnight refusal is a versioned JSON error envelope (matching `:2020`/`:2094`), not a logged event.
- Resolved-backend variable is named `backlog_backend` (avoids the scheduler `backend` collision at `cli_handler.py:1125`/`:1756`).
- `events.log` is authoritative for tier/criticality; the backlog write-back is a backend-gated mirror. No Python change to `refine.py`/`common.py`/`state_cli.py` — the decoupling is the skill omitting `--backlog-slug` under non-local and passing Clarify's explicit flags (R8).
- **Cross-plugin reference infeasibility** — the routing guard is inlined per consumer (not a shared reference) because the consumers span cortex-core and cortex-overnight and `${CLAUDE_SKILL_DIR}/../sibling` resolves only intra-plugin (ADR-0009).
- Dual-source: the binstub canonical + `plugins/cortex-core/bin/` mirror co-land in the same commit (drift gate); morning-review edits land in the `plugins/cortex-overnight/` tree, not cortex-core.
- L1 surface ratchet is unaffected (routing lives in SKILL/reference bodies, not frontmatter); SKILL 500-line cap applies.
- Terminology: the local backend is named `cortex-backlog` consistently (not `local`).
- **#317/#318 handoff**: #318 must revisit the five R7 consumer guards to expand the external arm from a one-line advisory into real `gh`-best-effort routing. Noted so the handoff is explicit.
- **Sequencing**: #316 lands first (in implementation now); #317's consumer edits go on top of #316's renamed files. Commit #317 artifacts with an explicit pathspec to avoid contaminating the concurrent #316 session's index.

## Open Decisions
None — the binstub uses the graceful console-script shape, routing is inlined, the overnight guard is in-process at the handler layer, and the criticality feed keeps the local path byte-identical. All forks resolved above.

## Proposed ADR

### Proposed ADR: 0016-configurable-backlog-backend-and-llm-as-adapter
**Context**: cortex's backlog is hard-wired to local files and the management surface ships in cortex-core, blocking adoption by teams (multiple real users) on GitHub Issues/Jira. **Decision**: the active backend is declared in `cortex/lifecycle.config.md` (`backlog.backend`, default `cortex-backlog`); consumers route on the resolved value at the skill layer; external backends are driven by the LLM plus a freeform `instructions` hint (no per-tool code adapter); the unattended overnight runner structurally refuses any non-local backend (gated in-process at the prep step, the only path to bootstrapping a session). The routing is designed to fail toward today's local behavior so a normal repo is never affected. **Trade-off / rejected alternatives**: per-tool code adapters (rejected for O(N) per-tracker maintenance — the core motivation is zero per-tool maintenance); plugin-install introspection (rejected — the wheel cannot reliably introspect Claude Code plugins, and config must be the source of truth). **Consequences**: extensibility without per-tool code; external backends are honest best-effort (GitHub well-supported, Jira unverified); consumers back-point to this ADR rather than restating rationale. (Three-criteria gate: hard-to-reverse — config schema + per-consumer routing; surprising — why LLM-as-adapter over typed clients; real trade-off — per-tool adapters considered and rejected.)
