## Web Research

### Anthropic skill-authoring guidance

- **`platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices`** — "Pattern 1: High-level guide with references" endorses pointer pattern. 500-line SKILL.md cap is the documented threshold.
- **Anthropic "Examples pattern":** worked examples ARE load-bearing when "output quality depends on seeing examples" (format-alignment work). Removing them when this is the case will degrade quality. **Per-trim test required.**
- **One-level-deep reference rule:** Anthropic warns against nested references — Claude may partial-read with `head` if pointer chains through another reference. Pointer must be direct.
- **Test-with-all-target-models clause:** the best-practice checklist requires testing against Haiku/Sonnet/Opus. The trim's "capable models can derive HOW from a pointer" claim must be tested on the **weakest** model the skill targets.

### Empirical evidence on inline-vs-pointer

- **arXiv 2506.14641 — "Revisiting Chain-of-Thought Prompting":** zero-shot ≥ few-shot for capable modern reasoners (DeepSeek-R1-class). Few-shot can degrade reasoning. **Caveat: smaller/older models still benefit from worked examples.** Per `requirements/multi-agent.md` and `dispatch.py:135–148`, lifecycle skills run under multiple models (Haiku/Sonnet/Opus); the caveat is operative.
- **Microsoft LLMLingua, LongLLMLingua (arXiv 2310.06839):** prompt compression with minimal performance loss. LLM performance drops 15–47% as context length increases ("lost in the middle"). Empirical justification for trimming overall.

### Process gate removal patterns

- **Stage-Gate / Phase-Gate literature** ([Wikipedia](https://en.wikipedia.org/wiki/Phase-gate_process), [Cerri](https://cerri.com/mastering-the-stage-gate-process-a-comprehensive-guide-from-a-to-z/)): modifying the gate model is sanctioned, but **mainstream guidance is to *reduce gate formality* rather than fully delete**. If deleting, a downstream check should absorb the responsibility.
- **Toyota Production System / Jidoka (TPS):** "quality at source" — replace redundant downstream inspection by building quality into the upstream step. **Pure removal without verifying upstream covers the same signal is the documented anti-pattern.**

### PR sizing literature

- [Swarmia](https://www.swarmia.com/blog/why-small-pull-requests-are-better/), [BSSW](https://bssw.io/items/pull-request-size-matters), [Codacy](https://blog.codacy.com/small-pull-requests), [Pete Hodgson](https://blog.thepete.net/blog/2019/05/10/6-practices-for-effective-pull-requests/): 200 LOC target, 400 LOC ceiling. Review effectiveness drops exponentially past 400. Hodgson explicitly warns: **"separate refactorings from feature work" and don't mix unrelated changes.**
- arXiv 2509.14745: 40% of agent-PRs combined multiple tasks; "too large" was a top-3 rejection reason.
- [Aviator stacked PRs](https://www.aviator.co/blog/rethinking-code-reviews-with-stacked-prs/), [Greiler stacked PRs](https://www.michaelagreiler.com/stacked-pull-requests/): documented mitigation for grouped-but-unbundled changes.

### Net web takeaway

- Items 1 and 3 have stronger external support and clearer compensating-control patterns (Jidoka, Stage-Gate "reduce formality").
- Item 2 has the highest empirical-validation gap. Anthropic's own best-practices preserve worked examples for format-alignment cases. A behavioral-diff test on the synthesizer is the correct gating criterion.
- Bundling all three is acceptable only if total diff stays under 400 LOC and the items share a single review angle.
