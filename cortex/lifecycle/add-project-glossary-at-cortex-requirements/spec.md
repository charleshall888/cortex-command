# Specification: add-project-glossary-at-cortex-requirements

> **Epic Reference**: Tier-1 of [[221-adopt-grill-with-docs-progressive-disclosure-system]]. Discovery context lives at `cortex/research/grill-me-with-docs-learnings/research.md`; ticket-specific implementation research at `cortex/lifecycle/add-project-glossary-at-cortex-requirements/research.md`.

## Problem Statement

Cortex skills and subagents re-learn project-specific vocabulary every session because no canonical-terms artifact exists. The cost shows up as cold-load overhead (each new session paraphrases terms instead of using them) and as drift across skill prose (the same concept gets different names in different files — `sentinel` already means subtly different things across three skills). This feature adds `cortex/requirements/glossary.md` — a small, opinionated, project-specific glossary written inline at the requirements interview and read by every skill that loads requirements via the tag-based protocol, including critical-review's Project Context block (narrowly scoped to the Language section only). The producer is single-surface (requirements-gather); feature-spec interviews do not produce glossary writes nor parking artifacts — when feature-spec readers encounter a missing term, the consumer-rule prose surfaces it to the user as a signal to raise in the next requirements interview, human-mediated. The loading mechanism is a new `## Global Context` schema section in project.md. The classifier is Pocock's binary rule (project-specific vs general programming) at write time. The four-bucket Cortex synthesis is documented in this lifecycle's research.md and intentionally *not* built in v1 — it has no defined consumer (no audit surface invokes it), and the discipline-as-audit framing it supports is an aspiration rather than a workflow. If a glossary-audit workflow earns its place later, the advisory doc is a separate ticket.

## Phases

- **Phase 1: Loading infrastructure** — extend `project.md` schema with a `## Global Context` section; extend `load-requirements.md` to honor it (with Matching Semantics counterpart and skipped-entry recording); update template and tests. After Phase 1, `glossary.md` can be loaded by every consumer when (eventually) it exists.
- **Phase 2: Producer wiring** — narrow `requirements-gather`'s no-filesystem contract to D2's explicit positive grant; add inline-write rule, binary classifier, term-already-exists probe, user-confirmation gate, and mid-interview abandonment semantics. Cascade-update `requirements-write/SKILL.md:4` and the `requirements` orchestrator's passive-artifact framing.
- **Phase 3: Consumer surfaces** — add Language-section-only glossary read to `critical-review` Step 2a (with Language-content constraint); narrow the exemption rationale at `SKILL.md:41` to "priorities and area-specific decisions, not vocabulary"; add consumer-rule prose to non-exempt consumers (the only surfacing path for absent terms — there is no spec-side parking artifact).

## Non-Requirements

- **No deferred-write Q&A path for glossary**: glossary entries are written inline by `/requirements-gather` only; they do NOT flow through `/requirements-write`'s synthesis step. (Re-affirms DR-2's chosen direction within the single-producer scope.)
- **No pre-commit hook scanning for new terms in v1**: drift detection is purely behavioral (challenge-against-glossary in the next interview). No automated freshness checks, no commit-hook lint.
- **No glossary writes from `lifecycle/specify.md §2`**: feature-spec interviews do not write glossary entries. They also do NOT park "Candidate Glossary Term" items in spec artifacts — the prior version of this spec proposed that mechanism; it was dropped because the consumer-side scan rule was unspecified and lifecycle archival would have made parked candidates structurally unreachable. The only surfacing path for an absent term is the consumer-rule prose (Req 14), which is human-mediated: the consumer skill nudges the user to raise the term in the next requirements interview at the user's discretion. No parking, no automated promotion.
- **No four-bucket classifier at inline-write time**: the producer applies Pocock's binary rule (project-specific vs general programming) only. The four-bucket Cortex synthesis (Contract / Compressing reference / Author scaffolding / Genuinely-domain term) is documented in this lifecycle's research.md but NOT built as an advisory doc in v1 — it has no defined consumer (no audit skill, no source-skill-author workflow that loads it). Promoting it to a real doc is a follow-up ticket if and when an audit workflow earns its place.
- **No automatic structural gate (`bin/cortex-check-glossary-entry`) in v1**: shape validation (sentence count, `_Avoid_:` line presence) is a possible later add-on; not in v1. The producer's interview discipline IS the gate.
- **No new events-registry entry in v1**: glossary writes do not emit events. Post-hoc audit relies on `git log -- cortex/requirements/glossary.md` and the producer's interview cadence (the glossary append is part of the same interview turn that drafted it).
- **No always-load semantic in `load-requirements.md`**: the file existence check happens at Global Context resolution time, not via a new "load-if-exists" protocol step.
- **No exemption removal for critical-review**: the deliberate-exemption notice stays; only the rationale narrows to "not vocabulary." The tag-based protocol still does NOT apply to critical-review.
- **No glossary-audit skill in v1**: there is no scheduled audit pass on the glossary. Audit (re-classifying entries, cutting author-scaffolding) would be an ad-hoc activity. The four-bucket Cortex synthesis that would frame such an audit is documented in this lifecycle's research.md; if the audit becomes routinely necessary, a separate ticket defines both the audit skill and its advisory framework.

## Edge Cases

- **Glossary file does not exist (lazy-creation steady state)**: every consumer must proceed silently. `load-requirements.md` records the absent Global Context entry as skipped per Req 3, not as a warning. `critical-review` Step 2a skips the Language-section read. Consumer-rule prose remains a no-op for readers.
- **Term resolved by a Recommended-answer line, not user-confirmed**: producer does NOT write. The interview proceeds; if the user later confirms or names the term explicitly, the write fires then.
- **Same term resolved differently across two requirements interviews**: producer reads the existing entry; surfaces the conflict via AskUserQuestion ("the glossary defines X as Y; this interview suggests Z — keep / replace / surface as Flagged Ambiguity?"). No silent overwrite.
- **General-programming concept proposed for glossary entry**: producer rejects via the binary classifier ("timeouts" is general; "phase_transition" is project-specific). The producer's interview turn explains the rejection and proceeds; the term is not written.
- **Feature spec interview references a term not yet in the glossary**: consumer-rule prose triggers — the skill surfaces the absence as a signal to the user, recommending they raise the term in the next requirements interview. The spec interview does not write inline, does not park the term, and does not block on the absence; it proceeds with the best paraphrase available.
- **User abandons requirements interview mid-stream after some terms have been appended**: appended entries persist (per-term durable append per Req 6). The Q&A block held in conversation context is lost. Partial glossary growth is the documented design — abandonment leaves the glossary in a coherent state because each entry was a complete unit when written.
- **Parallel-overnight requirements interviews running concurrently**: not a v1 concern — `/requirements-gather` is invoked by the user interactively (project-level interviews are not overnight surfaces). If this changes, the events-registry + atomic-rename helper becomes a separate ticket.
- **Glossary entry references a term that gets later renamed**: glossary edits flow through the next interview (interview surfaces the rename; producer updates the existing entry verbatim per the term-already-exists path). No special "rename" subcommand.
- **`tests/test_load_requirements_protocol.py:84` exemption anchor accidentally edited**: parity test fails the PR. The anchor phrase "Requirements loading: deliberately exempt" stays load-bearing; only the surrounding rationale prose is in scope to narrow.
- **Producer writes a classification-shaped Language entry** (e.g. `phase_transition — genuinely-domain term`): violates the Language-content constraint added by Req 12. The next critical-review pass will surface the leak via its Step 2a context read; the user can then edit the entry definitionally. No automated guard in v1, no audit skill in v1.

## Technical Constraints

- **SKILL.md size cap (500 lines)**: requirements-gather is 72 lines today; adding inline-write rule + binary classifier + term-already-exists probe + user-confirmation gate + mid-interview-abandonment + Language-content constraint likely lands between +50 and +75 lines. Well within cap. critical-review (115) and specify.md (191) have ample headroom for their additions.
- **Parity-pinned exemption anchor at `critical-review/SKILL.md:41`**: literal substring "Requirements loading: deliberately exempt" must remain present after edits (test_load_requirements_protocol.py:84).
- **Lazy file creation contract**: `cortex/requirements/glossary.md` must NOT be created by Phase 1 (loading infrastructure). It is created only by Phase 2's producer code path when the first term resolves. Tests must verify that absent-file behavior is silent.
- **MUST-escalation policy default**: all new authoring prose for the producer body, classifier, and consumer-rule citations must use soft positive-routing. No MUST/CRITICAL/REQUIRED escalations without effort=high evidence (none expected for v1).
- **`bin/.events-registry.md`**: no new event names introduced in v1. The producer's inline write does NOT emit a registry event. Post-hoc audit relies on git history.
- **`cortex/` umbrella sandbox write**: already covered by per-repo `cortex init` registration; no new sandbox permission needed for glossary.md.
- **Two parity-test cascade points beyond the four named tests**: (a) `tests/test_skill_callgraph.py` — verify no new skill-to-skill calls introduced. (b) `tests/test_check_events_registry.py` — verify no new `"event":` literals in skill prose.
- **CLAUDE.md "Prescribe What and Why, not How"**: the new prose surfaces must describe decisions and intent, not step-by-step procedure.
- **Solution horizon**: the `## Global Context` schema extension is the durable shape for any future always-load file (project lexicon, ADR README). Adding a second entry there in a later ticket should not require re-architecting the protocol. The section-scoped awk pattern in Req 2's acceptance criterion is also the durable shape — it remains correct as the section grows.
- **Phase 1 observable behavior on day one**: Phase 1 ships schema + load-rule + section entry pointing at an absent file. No glossary writes occur until Phase 2. The Phase 1 acceptance criteria are presence-grep based; runtime behavior is end-to-end testable only after Phase 2's producer ships. This is deliberate — Phase 1 lays the consumer-side path so Phase 2's producer can write into a system that already knows how to read.
