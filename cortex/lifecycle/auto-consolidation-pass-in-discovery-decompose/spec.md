# Specification: auto-consolidation-pass-in-discovery-decompose (#268)

> **Re-scope note.** #268 was authored as "a detector that scans drafted ticket bodies before the R15 gate." Research established that the ticket *count* is decided upstream — in `/discovery` decompose, each research-phase `### Pieces` bullet maps 1:1 to a ticket (§2), and §4 "Determine Grouping" only branches on the count, emitting one child per piece. So the count is locked before any body is drafted, and #268's framing (≡ the Alternative A that #247 already rejected) aims at the wrong altitude. This spec implements the user-selected mechanism instead: **make decompose §4 genuinely group tightly-coupled pieces into a single ticket (M pieces → 1 ticket) using research-phase coupling signals, before bodies are drafted, with a new `split-piece` re-derivation at the existing R15 gate.** Background: `…/research.md`; prior decision: `cortex/lifecycle/offer-consolidation-clusters-before-r15-gate/` (#247, shipped the manual `consolidate-pieces` fallback this spec retains).

## Problem Statement

`/discovery` decompose maps research pieces to tickets 1:1, so when the research-phase Architecture over-splits, the operator receives many small tickets (e.g. ~10 where ~3 larger ones were the right cut) — a recurring friction the operator reports, grounded in the structural fact that decompose has no grouping step. The cost is not reading: it is that each ticket then drags through full `/cortex-core:lifecycle` ceremony, so 10 tickets is 10× the overhead for work a capable model handles in 3 larger units. The only lever on ticket count today is the manual `consolidate-pieces` action at the final R15 gate, which requires the operator to spot the false boundaries unaided. This feature makes decompose land fewer, larger tickets by default — grouping the *grossly* over-split, architecture-visible cases at §4 — while keeping the operator able to break a wrong grouping back apart. (Corpus note: looped R15 gates are rare in committed history; this feature targets the operator's reported recurring case, not a high-frequency corpus signal, and is scoped accordingly — see Non-Requirements and the §4-yield constraint.)

## Phases

- **Phase 1: §4 grouping** — decompose §4 groups tightly-coupled pieces into single tickets before bodies are drafted; reconcile the §2/§3/Constraints "one ticket per piece / frozen piece-set" wording; preserve intra-group ordering; record groupings in `decomposed.md`.
- **Phase 2: R15 split re-derivation** — add a `split-piece <N>` affordance at the R15 gate to re-derive a wrongly-grouped ticket back into its constituent pieces from the retained Architecture source.

**Priority (MoSCoW)**: All eight requirements are **must-have**, and the feature is **not shippable until both phases land**. The operator-selected design is "auto-group **and** split at R15"; auto-grouping without the split re-derivation is the design the operator explicitly rejected (anchoring with no escape) and the ADR's named risk-reducer would be absent. Phase 1 may be *built and reviewed* first, but it must **not ship to users without Phase 2** — there is no independently-shippable Phase 1. There is no should-have/could-have tier by design — the scope is already the minimum that delivers "fewer, larger tickets" with an escape hatch. Won't-do items are in `## Non-Requirements`.

## Requirements

1. **§4 groups coupled pieces into tickets.** `skills/discovery/references/decompose.md` §4 "Determine Grouping" is rewritten so that, for `piece_count ≥ 2`, pieces exhibiting tight coupling are grouped into a single child ticket (M pieces → 1 ticket) instead of the current strict one-child-per-piece. Grouping is expressed as decision-criteria + output-shape (the coupling signals to weigh and the grouped result), not a token-matching algorithm. Acceptance: observable state — `grep -nE "group" skills/discovery/references/decompose.md` returns ≥1 hit inside §4 naming the grouping criteria; and `tests/test_decompose_rules.py` gains a test asserting §4 documents piece-grouping criteria — `just test` exits 0 (pass if exit code = 0). **Phase**: §4 grouping

2. **Coupling signals are inferred from the Architecture content decompose actually receives.** §4 grouping reasons over the approved Architecture sub-sections as *actually emitted by the research template* — `### Pieces` (one role bullet per piece) and `### How they connect` (the connection/boundary prose) — plus dependency relationships. From these the agent *infers* the coupling indicators: pieces that share a connection seam, sit in one integration cluster, carry near-identical roles, or only deliver value when shipped together. "Value when shipped together" is an inferred judgment, not a literal emitted field. The spec must not key §4 on `### Integration shape`/`### Seam-level edges` headings, which `decompose.md`/`SKILL.md` name but the research template does not emit (pre-existing drift — see Technical Constraints); §4 reads whatever connection content the Architecture carries and degrades gracefully if a heading is absent. Acceptance: observable state — `grep -nE "How they connect|infer|coupl|value.*together" skills/discovery/references/decompose.md` shows the inferred-signal framing in §4, and `grep -c "Seam-level edges" ` of the new §4 grouping prose = 0 (it does not depend on the non-emitted headings). **Phase**: §4 grouping

3. **Grouped child-ticket bodies are coherent merged bodies; the per-piece Architecture source is retained.** When §4 groups pieces, §5 authors one body per grouped ticket whose `## Why`/`## Role`/`## Integration` are prose-merged and whose `## Edges`/`## Touch points` bullets are unioned and deduplicated — mirroring the existing R15 `consolidate-pieces` body-merge convention (decompose.md:109). The research-phase Architecture `### Pieces` source (in `research.md`) is **not** modified by grouping and remains the authoritative per-piece record that `split-piece` re-derives from. Acceptance: observable state — decompose.md §4/§5 prose references the existing merge convention AND states the Architecture source is retained unchanged; `just test` exits 0. **Phase**: §4 grouping

4. **§2/§3/Constraints reconciled — grouping is an explicit packaging decision, not silent piece-set mutation.** decompose.md §2 ("each bullet … becomes one ticket candidate"), §3 ("Consolidation Review" return-to-research stance), and the Constraints "Architecture-section-driven" bullet ("do not silently mutate the set at decompose time") are updated so that grouping pieces into ticket units is (a) explicit and surfaced (never silent), and (b) distinguished from re-deriving or mutating the analytical piece-set, which remains forbidden and research-owned (the `### Pieces` set is unchanged; only ticket packaging coarsens). §3's "return to research" stance is updated to cover the case it still owns (a piece-set that is *wrong*, needing re-research) versus the new §4 case (a piece-set that is *right but over-split for ticketing*). Acceptance: observable state — `grep` finds the reconciliation wording in §2, §3, and Constraints; `just test` exits 0. **Phase**: §4 grouping

5. **Groupings and surviving intra-group ordering are recorded in `decomposed.md`.** §6 "Write Decomposition Record" records, under a dedicated `## Grouping Notes` heading (parallel to `## Consolidation Notes` / `## Dropped Items`): which pieces were grouped into which ticket, a one-sentence rationale per group, AND any surviving intra-group ordering — when grouped pieces had `blocked-by` relationships among themselves, the ordering is preserved as an explicit intra-ticket sequence note (mirroring the corpus precedent's "internal phase boundary" at `swap-daytime-autonomous-for-worktree-interactive/decomposed.md`), not dropped. Acceptance: observable state — `grep -nE "Grouping Notes" skills/discovery/references/decompose.md` returns ≥1 hit in §6, and the §6/§4 prose names intra-group ordering preservation; `just test` exits 0. **Phase**: §4 grouping

6. **`split-piece` is a valid R15 response value.** `split-piece` is added to `cortex_command/discovery.py` `_RESPONSE_VALUES` (currently the frozenset at lines 403–412). The `--response` argparse choices and the validation at lines 458–460 auto-derive from `sorted(_RESPONSE_VALUES)`, so no separate edit is required there. Acceptance: observable state — `grep -c '"split-piece"' cortex_command/discovery.py` = 1; and command check — invoking the helper's checkpoint-response emission with `--response split-piece --checkpoint decompose-commit` exits 0. `just test` exits 0. **Phase**: R15 split re-derivation

7. **R15 gate documents `split-piece <N>` as re-derivation from the retained Architecture source.** The R15 gate prose in `decompose.md` and the gate-option inventory in `skills/discovery/SKILL.md` document `split-piece <N>`: it re-derives a previously-grouped ticket N back into its constituent pieces **by re-authoring each piece's body from the retained Architecture `### Pieces` source** (the authoritative per-piece record, unchanged by grouping — per R3), restores any intra-group ordering recorded per R5, and re-presents the FULL renumbered batch. The prose states plainly that this is re-derivation from the Architecture source (the same source the no-grouping path would have used), not reconstruction from the merged body; it does not delete committed tickets (nothing is committed until `approve-all`). A single-index/non-grouped target re-prompts naturally rather than erroring (mirroring `consolidate-pieces`). Acceptance: observable state — `grep -nE "split-piece" skills/discovery/references/decompose.md` and `grep -nE "split-piece" skills/discovery/SKILL.md` each return ≥1 hit; `tests/test_decompose_rules.py`'s R15-options test includes `split-piece` — `just test` exits 0. **Phase**: R15 split re-derivation

8. **R15 responses (incl. `split-piece`) emit the existing event; no new event type.** Each R15 response emits one `approval_checkpoint_responded` event with `checkpoint: decompose-commit` and the chosen `response` (decompose.md:115), reusing the existing event — no new event name is registered. Acceptance: observable state — no new `"event":` literal is added to discovery skill prose; `just test` exits 0 and the events-registry check exits 0. **Phase**: R15 split re-derivation

## Non-Requirements

- Does **not** add a detector that scans drafted ticket *bodies* after the fact (that is #268-as-written ≡ #247's rejected Alternative A; explicitly out).
- Does **not** remove or replace the manual R15 `consolidate-pieces` affordance (#247, shipped) — it stays as the post-draft fallback for the subtle, body-level couplings §4 cannot see pre-draft.
- Does **not** change how the research phase identifies pieces; the analytical `### Pieces` set remains research-owned and unmodified. Only ticket *packaging* coarsens at §4.
- Does **not** fix the pre-existing research-template/decompose heading drift (`### How they connect` emitted vs `### Integration shape`/`### Seam-level edges` referenced). §4 is written to read the emitted content robustly; reconciling the upstream template vocabulary is a separate pre-existing cleanup (candidate follow-up ticket).
- Does **not** add a pre-R15 user pause — grouping results surface at the existing user-blocking R15 gate.
- Does **not** add a config flag to enable/disable grouping. The `split-piece` re-derivation handles wrong groupings by rebuilding from the retained Architecture source; an on/off knob would not earn its complexity.
- Does **not** recalibrate the research-phase decomposition granularity or the "Why N pieces" gate (the alternative the operator did not select).
- Does **not** target subtle, body-only couplings — §4 groups only the gross, architecture-visible over-splitting; finer couplings remain the manual `consolidate-pieces` fallback's domain (see §4-yield constraint).
- Does **not** add discovery kept-pauses parity-test coverage (a pre-existing gap; this change adds no new pause, so it is not on the critical path — candidate follow-up).

## Edge Cases

- **`piece_count` ∈ {0, 1}**: unchanged — the zero-piece (fold-into / verdict) and single-piece branches run as today; grouping is inert.
- **All N pieces group into one ticket**: collapses to single-ticket output (one ticket, no epic), reusing the single-piece branch semantics; the `## Grouping Notes` record (R5) makes the N-piece→1-ticket collapse auditable and `split-piece` can still re-derive the N pieces from the retained Architecture source. (`discovery_source` continues to point at the topic `research.md`, as for any single ticket.)
- **No coupling inferred**: §4 falls back to the current 1:1 one-child-per-piece mapping. Grouping is opportunistic, never forced.
- **`split-piece <N>` on a ticket that wrapped only one piece (never grouped)**: re-prompt naturally ("piece N wasn't grouped — nothing to split"), no error — mirrors `consolidate-pieces` single-index behavior.
- **Intra-group dependencies**: when coupled pieces with `blocked-by` relationships among themselves are grouped into one ticket, the cross-ticket dependency becomes an **intra-ticket ordering constraint that is preserved** as an explicit sequence note in the grouped body and recorded in `## Grouping Notes` (per R5) — it is not dropped. Dependencies from *outside* the group retarget the surviving ticket. On `split-piece`, the recorded intra-group ordering is restored to the re-derived pieces.
- **`split-piece` then `consolidate-pieces` (or re-group) in the same R15 loop**: split re-derives constituent pieces from the Architecture source; the batch renumbers contiguously from 1 and re-presents, after which the operator may consolidate differently — mirrors the existing renumber bookkeeping.

## Technical Constraints

- **Prescribe What/Why, not How**: grouping criteria are the coupling indicators to infer + the grouped-ticket output shape, never a token-overlap algorithm. Research (web agent) found brittle heuristic detectors produce false positives/negatives and that capable models judge a handful of pieces holistically more reliably.
- **§4-yield scope**: §4 sees only pre-draft Architecture content (`### Pieces` roles + `### How they connect`), so it can reliably catch only *gross* over-splitting (pieces that are obviously one unit by role/connection — the operator's stated pain). Subtle couplings that surface only once bodies are drafted are explicitly the domain of the retained manual `consolidate-pieces` fallback (decompose.md:109). The feature does not claim to catch all couplings.
- **Pre-existing heading drift (dependency)**: `decompose.md:9/:44` and `SKILL.md:82,85` reference `### Integration shape`/`### Seam-level edges` as Architecture sub-sections, but the research template (`research.md:120,123`) emits `### Pieces` + `### How they connect`. §4 grouping must read the emitted content and not hard-depend on the non-emitted headings. This drift pre-dates #268; reconciling it is out of scope (candidate follow-up).
- **MUST-escalation policy**: soft positive-routing phrasing for the new §4 grouping prose; no new MUST/CRITICAL without an events.log F-row or transcript artifact.
- **Event reuse**: R15 responses reuse `approval_checkpoint_responded` (`checkpoint: decompose-commit`) via the `response` field; the events-registry row does not enumerate response values, so `split-piece` needs no registry change and no new event type.
- **Frozenset surface**: `_RESPONSE_VALUES` (cortex_command/discovery.py:403–412) is the single source of truth; `--response` choices (line 1324) and validation (lines 458–460) derive from it. Adding `split-piece` is one line + tests.
- **Size/parity**: `decompose.md` is a `references/*.md` file (exempt from the 500-line SKILL.md cap); `SKILL.md` has headroom for one new R15 option line. No new `bin/cortex-*` script → `bin/cortex-check-parity` unaffected.
- **Tests**: `tests/test_decompose_rules.py` (section placement + R15 options) is the primary test surface; `tests/test_discovery_events.py` / `tests/test_discovery_module.py` if helper behavior is touched.

## Proposed ADR

### Proposed ADR: 0007-decompose-groups-pieces-into-tickets

**Context**: `/discovery` decompose has implicitly mapped one research-phase piece to one backlog ticket; ticket #247 (shipped) deliberately kept consolidation *user-driven* (the manual R15 `consolidate-pieces` action, no detector) after rejecting an automatic detector as unjustified on a thin corpus. **Decision**: decompose §4 now *auto-groups* the grossly over-split, architecture-visible cases into single tickets before bodies are drafted, using inferred coupling signals from the emitted Architecture content, with a `split-piece` re-derivation at R15 (rebuilding constituent pieces from the retained, unmodified Architecture source). **Trade-off**: this reverses the 1:1 contract and part of #247's user-driven posture, accepting a *residual* anchoring risk (a wrong auto-grouping the operator might rubber-stamp at R15). The risk is **reduced, not eliminated**: grouping fires only on clear architecture-level coupling, each group's rationale is surfaced at the R15 gate so the operator sees *why* a merge was made, and `split-piece` lets the operator rebuild a wrong grouping from the authoritative Architecture source — but a tired operator who does not notice a bad grouping is not fully protected, which is the accepted residual. Chosen over (a) an after-the-fact body-scanning detector before R15 (#268-as-written / #247's rejected A — wrong altitude, anchoring, contradicts the frozen-piece-set invariant) and (b) recalibrating research-phase granularity (a more diffuse change the operator did not select). Qualifies for an ADR: hard-ish to reverse (changes the decompose mental model), surprising without context (reverses a documented prior decision), and a real trade-off (ceremony vs. residual anchoring).
