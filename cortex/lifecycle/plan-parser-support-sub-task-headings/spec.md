# Specification: plan-parser-support-sub-task-headings

## Problem Statement

Plan authors organically decompose a task into ordered sub-units with `### Task Na` headings (`### Task 3a`, `### Task 3b`) — 12 plan files in the repo already do. The parser's task-heading regex captures integers only (`^###\s+Task\s+(\d+)`), so today (since commit 87239c4b, under #293) a letter-suffixed heading raises `ValueError`/`parse_error` — the parser refuses to dispatch the feature at all. This work makes `### Task Na` headings parse into first-class, independently-dispatchable units whose intra-parent ordering and cross-task dependencies are preserved, so an author can decompose a task without losing dispatch fidelity or hitting the fail-loud wall — while integer-only plans (the overwhelming majority) parse and dispatch byte-identically.

## Phases

> **Critical sequencing note (from critical review):** the identity migration is **atomic** — the heading regex, the `FeatureTask` identity fields, the `depends_on` collapse-removal, the dependency-batcher key swap, the checkoff regex, and the executor identity sites are mutually coupled. A commit that lands the parser side (string `depends_on`) without the consumer side (batcher keyed on `task_id`) makes `compute_dependency_batches` compare `"2" in {1,2}` → always False → spurious cycle error, **even on integer-only plans**. Therefore Phase 1 below is a single atomic landing; the plan phase MUST NOT split its coupled files across separately-committed tasks (see Technical Constraints).

- **Phase 1: Atomic identity migration** — recognize `### Task Na`, give `FeatureTask` a `task_id`-based identity, stop the `depends_on` collapse, key dependency batching + checkoff + the executor identity sites on `task_id`, and fix the pre-existing checkoff bleed — all landing together so no intermediate commit is broken. (Reqs 1, 2, 3, 4, 6, 7, 11.)
- **Phase 2: Validation + display** — additive guards and cosmetics that depend on Phase 1 but cannot break its intermediate: loud failure on dangling/self-referential dependencies, fail-loud on malformed suffixes, suffix-aware dashboard sort. (Reqs 5, 8, 9.)
- **Phase 3: Requirements + docs** — amend the `pipeline.md` must-have criterion and document the sub-task syntax. (Reqs 10, 12.)

## Requirements

> **Priority**: Reqs 1–8 and 10–11 are **must-have** — the feature does not work, or silently regresses backward-compat, without them. Reqs 9 (dashboard sort) and 12 (docs) are **should-have** — cosmetic display correctness and authoring guidance that can land in the same change but do not gate the core capability.

1. **Suffix-aware identity on `FeatureTask`**: add `suffix: str = ""`, a derived `task_id` property = `f"{number}{suffix}"`, and a derived `sort_key` property = `(number, suffix)`. `.number` stays `int`. **Phase**: Atomic identity migration. **AC**: a unit test asserts `FeatureTask(number=3, suffix="a").task_id == "3a"` and `.sort_key == (3, "a")`; an integer-only task has `task_id == "3"`, `suffix == ""`, `sort_key == (3, "")`; `python3 -m pytest cortex_command/pipeline/tests/test_parser.py` exits 0.

2. **`### Task Na` heading parse**: the task-heading regex recognizes an optional single lowercase letter suffix (`^###\s+Task\s+(\d+)([a-z]?)\s*[:—–-]\s*(.+)$`), integer part accepts `0`. The 87239c4b fail-loud guard (`parser.py:319-336`) is removed. **Phase**: Atomic identity migration. **AC**: parsing a plan containing `### Task 3a: …` yields a task with `task_id == "3a"` (no raise); `TestSubTaskHeadingFailLoud` is replaced with positive-parse cases asserting this; pytest on `test_parser.py` exits 0.

3. **Stop the `depends_on` collapse — preserve task_id references**: `depends_on` becomes `list[str]` of task_ids; `_parse_field_depends_on` returns `["1", "3a", "3b"]` verbatim (no longer collapses `3a`→`3`). **Phase**: Atomic identity migration. **AC**: `test_subtask_letter_suffix_tolerated` is inverted — `- **Depends on**: [1, 3a, 3b]` now parses to `["1", "3a", "3b"]` (was `[1, 3, 3]`); integer-only `[1, 2, 3]` parses to `["1", "2", "3"]`; pytest exits 0.

4. **Dependency batching keys on `task_id`**: `compute_dependency_batches` keys `done`/`assigned` sets on `.task_id` (`set[str]`) — **all four `.number` references** (`common.py:692, 699, 705, 706`) move together; ordering is driven solely by declared `Depends on` edges (no implicit `a<b` edge). **Phase**: Atomic identity migration. **AC**: a `test_common.py` case asserts (a) `3a→3b` (3b deps `[3a]`) yields ordered batches `[…3a…], […3b…]`; (b) `13a/13b/13c` (all dep `[10]`) co-schedule in one batch after `[10]`; (c) **the merge guard** — a plan where `3a` is `status:done` and `3b` is pending does NOT drop `3b` from scheduling (asserts `3b` still appears in a batch, proving the `done`/`assigned` sets key on `task_id` not the shared `.number`); all existing integer-only batch tests stay green; `just test` exits 0.

5. **Loud, clear failure on dangling and self-referential dependencies**: an unresolvable `depends_on` task_id (e.g. `[3]` when only `3a`/`3b` exist) or a self-dependency surfaces as a loud feature failure with a message naming the offending task_id. **This preserves — does not change — the existing failure classification**: `compute_dependency_batches` already raises a hard `ValueError` for unsatisfiable dependencies today (routed to a counted `status:failed`, NOT `parse_error`); this requirement keeps that classification and circuit-breaker accounting, adding only message clarity and explicit self-loop coverage. **Phase**: Validation + display. **AC**: a test asserts a plan whose task references `[3]` when only `3a`/`3b` exist fails loudly (raises, or `FeatureResult.status == "failed"`) with the unresolvable id in the message; a self-referential `depends_on` fails loudly; the failure does NOT set `parse_error=True` (preserving the existing counted-toward-circuit-breaker semantics — verify against `outcome_router.py:856`); pytest exits 0.

6. **`mark_task_done_in_plan` matches `task_id` and no longer bleeds across headings**: the checkoff regex matches the full `task_id` (`### Task 3a:`) and is bounded so its `.*?` cannot cross a subsequent `^###\s` heading (tempered-dot under `re.MULTILINE`), fixing the pre-existing cross-task checkbox bleed. **Phase**: Atomic identity migration. **AC**: a regression test asserts that marking an already-`[x]` parent task does NOT flip the next task's `[ ]`, and that marking `3a` checks `3a`'s status (not `3b`'s); `python3 -m pytest cortex_command/tests/test_common.py cortex_command/tests/test_common_utils.py` exits 0.

7. **Identity-bearing executor sites switch to `task_id` atomically — read AND write**: exit-report filename **write** (`{task_id}.json`, `feature_executor.py:829`) AND its symmetric **read** (`_read_exit_report`, `feature_executor.py:169`, called at ~784, which independently builds `{task_number}.json`), the idempotency token input, and the `has_dependents` membership check (`feature_executor.py:247`) all use `task_id`, not `.number`. **Phase**: Atomic identity migration. **AC**: a test asserts (a) `3a` and `3b` produce distinct exit-report filenames (`3a.json` ≠ `3b.json`) and distinct idempotency tokens; (b) a **read/write round-trip** — an exit report written for `3a` is read back by the orchestrator for `3a` (not `3`/absent), proving write and `_read_exit_report` switched together; (c) `has_dependents` is `True` for a task referenced as `3a` in another task's `depends_on` (guards the silent-False membership path); (d) for an integer-only task the idempotency token is byte-identical to the pre-change value (token shape `{feature}:{task_number}:{plan_hash}` and stringification preserved).

8. **Reject malformed suffixes with explicit fail-loud**: multi-letter (`3ab`), space-separated (`3 a`), and uppercase (`3A`) suffix headings raise rather than silently absorb (no regression to the pre-87239c4b silent-drop). **Phase**: Validation + display. **AC**: tests assert `### Task 3ab:`, `### Task 3 a:`, and `### Task 3A:` each raise `ValueError`; pytest exits 0.

9. **Dashboard exit-report sort handles suffixed stems** *(should-have)*: `dashboard/data.py` sorts exit-report stems by `(numeric-prefix, suffix)` so `3a.json` sorts after `3.json` and before `4.json` (display-only). **Phase**: Validation + display. **AC**: a unit test on the sort key asserts `["1","2","3","3a","3b","10"]` ordering from stems `["10","3b","1","3","2","3a"]`.

10. **Amend `pipeline.md` line 42 must-have criterion**: update the mechanism clause to reflect task_id-keyed batching and narrow the "unparseable … fails loud" clause so it no longer implies `### Task Na` headings are unparseable (the fail-loud net still covers genuinely-unparseable metadata). **Phase**: Requirements + docs. **AC**: `grep -c "fails the feature loudly" cortex/requirements/pipeline.md` ≥ 1 against the amended text; the criterion no longer asserts letter-suffixed headings are unsupported (Interactive/session-dependent: requirements prose, semantic narrowing has no command oracle).

11. **Backward compatibility — integer-only plans byte-identical**: integer-only plans parse, batch, checkoff, and dispatch identically; JSON `task_number` payloads remain integer-typed (`.number` is unchanged). Because Phase 1 lands atomically, there is no intermediate where integer-only `depends_on` strings meet an int-keyed batcher. **Phase**: Atomic identity migration. **AC**: the full pre-existing integer-based assertions in `test_parser.py` and `test_common.py` pass unmodified except for the `depends_on` string-typing migration; `just test` exits 0.

12. **Document the sub-task syntax** *(should-have)*: `skills/lifecycle/references/plan.md` and `implement.md` §2 document `### Task Na` as supported, including the same-batch disjoint-`Files` authoring guidance (see Non-Requirements / Open follow-up). **Phase**: Requirements + docs. **AC**: `grep -c "Task Na\|sub-task" skills/lifecycle/references/plan.md` ≥ 1; mirror regeneration (`plugins/cortex-core/skills/`) is current (pre-commit drift gate passes).

## Non-Requirements

- **No group-reference fan-out**: a bare `[3]` dependency means the literal task `3` only — it is NOT auto-expanded to `{3, 3a, 3b}`. Authors enumerate sub-task dependencies explicitly (the corpus universally does this for *dependency references*); group sugar is not built.
- **No sub-task concurrency guard (OQ-1 decision) — with an explicit, evidenced caveat**: per the OQ-1 decision, no implicit serialization edge between same-parent sub-tasks and no `Files`-overlap validator is built. **Caveat surfaced by critical review:** the disjoint-`Files` assumption is *not* reliably honored by real authors — the corpus's own `13a/13b/13c` all declare `Files: tests/test_cli_auto_update.py` (one batch) and `2a/2b` both declare `docs/setup.md`. The same-batch shared-worktree write race is therefore **newly reachable** for sibling sub-tasks (the current parser fail-loud-rejects these headings, so they have never run as concurrent writers). We accept this risk for now — those plans are COMPLETE and will not re-dispatch, and the race is a property of the existing `asyncio.gather`-into-one-worktree dispatch model, not introduced code. **Explicit follow-up trigger**: file a `Files`-overlap guard ticket (validate disjoint `Files` for same-batch tasks; serialize or fail-loud on overlap) the first time an overnight session exhibits a sub-task overlap-race. The concurrent-write runtime path is intentionally not covered by an AC.
- **No multi-letter / uppercase / spaced suffix support**: only single lowercase `a`–`z` after the integer. Malformed forms fail loud (Req 8), they are not supported.
- **No migration of historical plans**: the 12 existing suffixed plans are COMPLETE history and serve as parse fixtures only — they are not rewritten or re-dispatched.
- **No new circuit-breaker reclassification for dangling refs**: Req 5 deliberately keeps the existing counted-`status:failed` classification (not `parse_error`) so overnight circuit-breaker accounting is unchanged.
- **No `task_id` on JSON telemetry payloads**: the event-log `task_number` fields may stay on integer `.number` for telemetry continuity; switching them to `task_id` is not required (only identity-bearing sites in Req 7 must switch).

## Edge Cases

- **Orphan suffix (`8b` with no `8a`)**: parses normally — identity is `(number, suffix)`; a sub-task may start at any letter.
- **Parent + sibling coexist (`### Task 8:` AND `### Task 8b:`, no `8a`)**: both parse as distinct units (`task_id` `"8"` and `"8b"`); `8b`'s `depends_on [8]` resolves to the real `8` (no self-loop, no collision — the old collapse caused both).
- **Zero-prefixed (`0a`/`0b`/`0c`)**: integer part `0` is accepted.
- **Em-dash separator (`### Task 0a — Q1:`)**: the identifier delimiter is the first em-dash/colon after `0a`; title becomes `Q1: …` (the second colon stays in the description). Greedy title capture handles this without special-casing.
- **Same-parent sub-task already `[x]` in the batcher**: a `status:done` `3a` must NOT cause pending `3b` to be skipped — the `done`/`assigned` sets key on `task_id`, so `"3a" ∈ assigned` does not imply `"3b" ∈ assigned` (Req 4 AC (c) guards this). This is the silent-merge path the `.number`-as-group-ordinal design must not reopen.
- **Same-batch sub-tasks touching overlapping `Files`**: shared-worktree race, newly reachable for sibling sub-tasks (see Non-Requirements). Not guarded this cycle; authoring guidance recommends disjoint `Files`; follow-up trigger documented.
- **Dangling reference (`[3]` when only `3a`/`3b` exist)**: loud feature failure naming the unresolvable id. (Today the integer equivalent already fails loudly at batch time via `compute_dependency_batches`' unsatisfiable-deps `ValueError` — this is not a previously-silent stall; Req 5 improves the message and adds self-loop coverage while keeping the same failure classification.)
- **Integer-only resume across the change**: completion/idempotency tokens recompute identically (int `3` and str `"3"` hash to the same token), so an in-flight integer plan resuming under the new code re-skips done tasks correctly.

## Changes to Existing Behavior

- **MODIFIED**: `### Task Na` headings now parse into first-class units → previously raised `ValueError`/`parse_error` (87239c4b guard removed).
- **MODIFIED**: `depends_on` is now `list[str]` of task_ids, resolved verbatim → previously `list[int]`, collapsing `3a`→`3`.
- **MODIFIED**: `compute_dependency_batches` keys on `task_id` (`set[str]`) → previously `.number` (`set[int]`); all four `.number` references migrate together.
- **MODIFIED / FIXED**: `mark_task_done_in_plan` matches the full `task_id` and is bounded to its own task body → previously matched an integer and could bleed its `[ ]` flip into the next task (pre-existing bug).
- **MODIFIED**: exit-report filename **read and write** and idempotency tokens key on `task_id` → previously `.number` (integer-only values unchanged).
- **MODIFIED**: `pipeline.md` line 42 must-have criterion text (mechanism + fail-loud carve-out).
- **ADDED**: `FeatureTask.suffix`, `.task_id`, `.sort_key`.
- **UNCHANGED (deliberate)**: dangling-ref / unsatisfiable-dependency failure classification stays `status:failed` (counted toward the circuit breaker), not `parse_error` — Req 5 adds clarity, not a reclassification.

## Technical Constraints

- **Atomic identity landing (from critical review)**: Reqs 1, 2, 3, 4, 6, 7, 11 are one atomic unit. The plan phase **MUST NOT** split the coupled files (`parser.py` FeatureTask + heading regex + `_parse_field_depends_on`; `common.py` `compute_dependency_batches` + `mark_task_done_in_plan`; `feature_executor.py` identity sites at lines 169/247/637/784/829) across separately-committed tasks — every per-task commit must keep `just test` green, and a parser-side-only commit raises a spurious cycle error even on integer-only plans (`"2" in {1,2}` is False). If the plan must split them, it must introduce a transitional shim (batcher accepting both key types) so each intermediate commit stays green.
- **`parser.py` is stdlib-only** — hand-roll the `(number, suffix)` tuple `sort_key`; do NOT import `packaging`/PEP 440 machinery (the composite-tuple pattern is the idiom, not the library).
- **`task_id` is the sole identity key for the enumerated identity-bearing sites** (batching, checkoff, exit-report read/write, idempotency token, `has_dependents`); `.number` remains present as a non-unique `int` "group ordinal" used only by telemetry JSON. This is a **residual documented hazard, not a structural guarantee** (ADR-0010): a future consumer that keys identity on the still-present `.number` would silently merge `3a`/`3b`. It is contained — not eliminated — by (a) migrating every enumerated site in Req 4/7 together, and (b) the merge-guard tests (Req 4 AC (c), Req 7 AC (b)/(c)) that catch the silent-False / silent-merge paths, which fail by returning `False` rather than raising.
- **Idempotency token invariant**: preserve the key shape `{feature}:{task_number}:{plan_hash}` and the `str()` stringification exactly, or all prior tokens invalidate and resume re-runs everything.
- **Suffix grammar parity**: the heading suffix grammar (`[a-z]`) and the `depends_on` token grammar must agree on case (the depends_on matcher is currently `re.IGNORECASE` — case-fold `task_id` on both sides or restrict both to lowercase) to avoid a silent `"3A"` vs `"3a"` dangling mismatch.
- **Dev loop**: editing `cortex_command/common.py` + `skills/` runs under this lifecycle; use sequential dispatch (not worktree) for implementation — `just test` runs the editable install, so worktree verification tests stale code.

## Open Decisions

None — all design questions were resolved during the interview (OQ-1, OQ-5), from research evidence (fan-out, ordering, identity representation, suffix grammar), or from critical review (phase atomicity, dangling-ref classification, the disjoint-Files caveat).

## Proposed ADR

### Proposed ADR: 0010-task-id-is-task-identity-not-number

**Context**: Making `### Task Na` sub-tasks first-class requires an identity that orders `3 < 3a < 3b < 4` and is distinct per sub-task. `FeatureTask.number` was the de facto identity, consumed as an `int` in ~19 places (exit-report read/write filenames, idempotency tokens, dependency batching, plan checkoff, `has_dependents`). **Decision**: introduce a canonical `task_id` string (`f"{number}{suffix}"`) as the *sole* identity key for batching, checkoff, exit-report read/write, idempotency tokens, and `has_dependents`; keep `.number` as an `int` "group ordinal" used only for telemetry. **Trade-off**: the alternative of retyping `.number` itself to a richer type minimizes the number of identity concepts but forces every integer wire-format (JSON payloads, int-equality tests, `set[int]`) to change, making the "integer-only plans dispatch identically" guarantee a careful audit rather than a structural property. Keeping `.number` int + `task_id` str makes integer-only plans byte-identical by construction (in the *final* state — the migration itself must land atomically, since a partial commit breaks even integer-only plans), at the cost of a documented hazard: a future consumer that keys on the non-unique `.number` would silently merge `3a`/`3b`, contained by the merge-guard tests rather than structurally. **Why ADR-worthy**: hard-to-reverse (touches the identity contract across the dispatch path), surprising-without-context (a maintainer would assume `.number` is the identity), and a real trade-off (structural backward-compat vs. single-identity-concept).
