# Plan: plan-parser-support-sub-task-headings

## Overview
Make `### Task Na` sub-task headings first-class by introducing a canonical `task_id` string (`f"{number}{suffix}"`) as the sole identity key across parsing, dependency batching, plan checkoff, exit-report read/write, idempotency tokens, and `has_dependents` ‚Äî keeping `.number` as an `int` telemetry-only "group ordinal" so integer-only plans stay byte-identical by construction. The coupled identity sites land in one atomic commit (a partial landing breaks even integer-only plans); validation guards, display sort, requirements text, docs, and the ADR follow as additive, independent tasks.
**Architectural Pattern**: layered
<!-- A data-model identity type threaded through the existing parser ‚Üí batcher ‚Üí executor layers; no new dispatch architecture, just a richer identity key replacing a non-unique int across the layer boundaries. -->

## Outline

### Phase 1: Atomic identity migration (tasks: 1)
**Goal**: Recognize `### Task Na`, give `FeatureTask` a `task_id`-based identity, stop the `depends_on`‚Üíint collapse, key dependency batching + checkoff + the executor identity sites on `task_id`, and fix the pre-existing checkoff cross-task bleed ‚Äî all in one commit so no intermediate is broken.
**Checkpoint**: `just test` green; a suffixed plan parses to distinct `3a`/`3b` units with distinct batches, exit-report filenames, and idempotency tokens; integer-only plans parse/batch/checkoff/dispatch byte-identically.

### Phase 2: Validation + display (tasks: 2, 3, 4)
**Goal**: Additive guards and cosmetics that depend on Phase 1 but cannot break its intermediate ‚Äî loud failure on dangling/self-referential deps, explicit fail-loud on malformed suffixes, suffix-aware dashboard sort.
**Checkpoint**: dangling `[3]`-when-only-`3a/3b` and self-deps fail loudly naming the offending id; `3ab`/`3 a`/`3A` raise; dashboard sorts `3a` after `3` before `4`; `just test` green.

### Phase 3: Requirements + docs (tasks: 5, 6, 7)
**Goal**: Amend the `pipeline.md` must-have criterion, document the sub-task syntax in the lifecycle skill references, and record the identity decision + residual hazard as ADR-0010.
**Checkpoint**: `pipeline.md` line 42 reflects task_id-keyed batching and no longer reads as forbidding `### Task Na`; `plan.md`/`implement.md` document the syntax (mirrors regenerated); `cortex/adr/0010-*.md` exists.

## Risks

- **Task 1 is intentionally large (8 files, ~15 test edits).** The spec's Technical Constraints forbid splitting the coupled identity files across separately-committed tasks ‚Äî a partial landing breaks even integer-only plans (`"2" in {1,2}` is `False`). The documented fallback if smaller commits are required is a **transitional dual-key shim** (batcher accepting both `int` and `str` keys) so each intermediate stays green; this adds throwaway code and is not recommended for a once-only migration. Flagging because Task 1's size raises a turn-budget concern in the Implement phase ‚Äî if it exhausts mid-task, the commit is left red. The complex tier gets the largest turn budget; consider this the main scheduling risk.
- **One-batch-boundary silent-drop window for malformed suffixes.** Task 1 removes the broad 87239c4b guard; the narrowed malformed-suffix fail-loud lands in Task 3. Between those commits a malformed heading (`3A`, `3ab`, `3 a`) would silently absorb into the prior task rather than failing loud. Scope of the window, stated precisely (correcting an earlier "sequential dispatch" framing): Tasks 2‚Äì7 all declare `Depends on: [1]`, so they form one batch (Batch 1) dispatched **concurrently** ‚Äî Task 3 is not a deterministic "immediately after" commit but one of six parallel sub-agents, and the guard is absent for the span from Task 1's commit until Task 3's sub-agent commits within Batch 1. The window is nonetheless bounded to **one batch-boundary and closes before the feature is ever released**, on *both* execution surfaces: this lifecycle implements the plan in its interactive Implement phase, and even under the overnight feature-executor a feature's batches all run inside one `execute_feature` call (no cross-round split). The corpus contains zero malformed headings and none are dispatched, so no real plan is exposed. This follows the spec's deliberate Phase 1/Phase 2 split. **Zero-window alternative if the operator prefers it**: fold Req 8's narrow malformed-suffix guard into Task 1 (it is a small addition to the parser change Task 1 already makes), eliminating the window entirely at the cost of a slightly larger atomic task.
- **Same-batch sub-task shared-worktree write race (accepted, not guarded).** Per the spec Non-Requirements (OQ-1), no implicit serialization edge or `Files`-overlap validator is built. Sibling sub-tasks with overlapping `Files` co-scheduled in one batch race in the overnight shared worktree. Accepted for now (the 12 existing suffixed plans are COMPLETE and will not re-dispatch). **Caveat on the follow-up trigger**: the spec's trigger ("file a `Files`-overlap guard ticket the first time an overnight session exhibits the race") is weak because the failure mode is *silent* last-writer-wins corruption ‚Äî there is no loud, attributable signal an operator would recognize as "the race," so the trigger may not reliably fire. The primary interim mitigation is therefore **preventive, not reactive**: Task 6's disjoint-`Files` authoring guidance (verified present by Task 6's check 2). If a structural guard is wanted before a race is ever observed, the durable option is a same-batch `Files`-overlap validator (fail-loud on overlap) ‚Äî out of scope this cycle per OQ-1, but the more reliable mechanism than waiting for a silent corruption to surface.
- **Dev loop is sequential, not worktree.** Per spec Technical Constraints and project memory, editing `cortex_command/` + `skills/` under this lifecycle uses sequential dispatch ‚Äî `just test` runs the editable install, so worktree verification would test stale code.

## Acceptance
A feature plan containing `### Task 3a:`/`### Task 3b:` headings parses into distinct, independently-dispatchable units whose declared `Depends on` ordering is preserved through `compute_dependency_batches`, whose checkoff/exit-report/idempotency-token operations key on the full `task_id` (no `3a`/`3b` collision), and whose dangling/malformed references fail loud with a clear message ‚Äî while every integer-only plan in the existing test corpus parses, batches, checks off, and dispatches byte-identically (`just test` exits 0).
