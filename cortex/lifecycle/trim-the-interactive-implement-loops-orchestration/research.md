## Web Research

**Claude Code dispatch semantics (version-dependent, upstream-owned).** Official docs: as of v2.1.198 subagents run in the **background by default**; as of **v2.1.211** the completion notification reliably carries the subagent's final report (before that, stale/incomplete results were a real race). `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1` forces synchronous; there is **no flag forcing background**. Agent Teams (experimental, off by default) has native task-dependency unblocking but is token-expensive and carries a known bug: teammate→lead SendMessage is **silently dropped** (#43706) — the worker→coordinator direction is exactly the unreliable one. SubagentStop hooks receive `last_assistant_message` but **cannot rewrite it**; blocking-reliability issues were reported (#33049, #20221 — may be fixed since).

**Live observation (this orchestrating session, 2026-07-18)**: every Agent dispatch ran in the background and every completion notification carried the agent's full final report — zero fetch round-trips across 8 dispatches. On this runtime, #353's two-fetch round-trip does not reproduce.

**Anthropic's own posture**: their production multi-agent research system executes subagents **synchronously (BSP) by design**, and they state coding tasks have fewer truly parallelizable units than research and that real-time LLM-to-LLM coordination isn't yet reliable — arguing against teammate-style dispatch for implement/review.

**Prior art for pipelined dispatch**: LLMCompiler (arXiv:2312.04511) — Planner builds a DAG, a Task Fetching Unit dispatches each task the moment its own deps are satisfied (claimed 3.7x latency, 6x cost gains). A cohesion-aware partitioning study (arXiv:2606.00953) recommends profiling dependency density first and treating >20–30% of wall-time in synchronization as the signal to act. Prompt-template prior art converges on four explicit parts per dispatched task: objective, output format, tool/source guidance, boundaries — and reviewers seeing only the diff/output, never the builder's reasoning.

## Tradeoffs & Alternatives

- **Dispatch mode**: (A) accept synchronous as permanent — matches `docs/internals/sdk.md`'s documented intent ("`run_in_background` unused: interactive skill dispatch is synchronously coupled to the batch verify-and-merge loop") but can't explain #353; (B) adopt background/teammate — matches #353 and this session's runtime, unlocks pipelining, but reports depend on notification behavior that changed twice in 13 patch releases. The in-repo docs and the live runtime **contradict each other**; the mode is upstream-owned and version-dependent, not a stable project fact.
- **Item 1**: (A) dual-path mandate (SendMessage or report file) — carries a permanent equivalence burden between two channels; (B) structured commit-message trailers read by the existing §2d `git log` checkpoint — single channel, free read, but fights the commit-message validation hook's prose conventions.
- **Item 2**: (A) prose-driven readiness re-evaluation per loop turn — exactly the live-state bookkeeping the repo elsewhere moved into verbs; (B) **plan-time straggler isolation** (split a topological batch when a `complex` task is co-batched with fast siblings) — works under both modes, zero new mechanism, no-regrets; (C) a verb-owned readiness resolver (`cortex-lifecycle-next-ready` reading plan.md + events.log) — matches verb-first, but see Adversarial: it needs a per-task *dispatch* event nobody scoped.
- **Item 3**: (A) `--complexity` axis on the verb — anticipated by the ticket but forks the verb's fail-loud contract and creates meaningless non-builder cells; (B) **downgrade-only wrapper** over the existing resolution ("trivial/simple → down one tier, floor sonnet") — makes fail-safe-up structurally visible; (C) prose table — rejected (ADR-0023).
- **Item 4**: (A) `task_complete` emitted orchestrator-side at the §2d checkpoint (tied to a verified commit, matching the rule that builders never call event verbs); (B) status quo reconstruction — see Events & Measurement for what it actually supports.
- **Item 5**: (A) prose single-writer + tallies convention; (B) structural restriction on verifier dispatches. CLAUDE.md's exception reads both ways: deviation was recoverable (§4 escape exists) but the ticket's own evidence says convergence was luck.
- **Item 6**: 6(a) hub-file seam — authoring bullet **and** a mechanical checklist row (flag any file in ≥3 tasks' `Files` lists without an early registration seam); 6(b) dress-rehearsal — judgment call, stays prose.

## Model-Tiering Lattice

**Two lattices, genuinely shape-incompatible (recorded decision).** `cortex_command/lifecycle/resolve_model_cli.py` owns the interactive lattice, keyed (role, criticality) — `builder: {low: sonnet, medium: sonnet, high: opus, critical: opus}`, never haiku. `cortex_command/pipeline/dispatch.py:136–149` owns overnight's `_MODEL_MATRIX`, keyed (complexity, criticality), no role axis, paired with `_EFFORT_MATRIX` and `TIER_CONFIG` (turns/budget). The separation is deliberate and documented in both `resolve_model_cli.py:7–10` ("structurally different lattice: `orchestrator-fix`'s `sonnet|sonnet|sonnet|opus` row is unrepresentable in the complexity×criticality pipeline matrix, so the two stay separate") and `docs/internals/sdk.md:142–145` ("intentionally separate … structurally unmergeable").

**Overnight's Complexity chain**: `parser.py:396–401` parses per-task Complexity — **absent → default `"simple"`** (a fail-safe-DOWN if imported into interactive: simple→sonnet at every criticality would downgrade a high-criticality opus builder); present-but-invalid → normalized **up** to `complex` and surfaced loudly via a `COMPLEXITY_NORMALIZED` event. The pattern (never silently absorb) transfers; the absent-value default does not.

**Interactive today**: `implement.md` resolves `cortex-resolve-model --role builder --criticality …` once per **batch** (inside "Per batch, in order", value-constant per feature); per-task Complexity is pasted verbatim into builder prompts and never consulted by the dispatcher. The Agent tool exposes `model` only — no turn/budget/effort knobs, so overnight's `TIER_CONFIG` semantics cannot carry over; a downgraded model with an undowngraded ~40-turn cap is an unvalidated configuration, not "the overnight mapping".

**Adversarial corrections that reshape item 3**:
- `plan.md:74,78` defines `trivial` as "no commit" and mandates "`simple` minimum — never `trivial`" for committing tasks; the interactive loop requires every task to commit (implement.md step 4, §2d fails zero-commit tasks). **A trivial task in an interactive plan is a plan-authoring defect**, and #353's motivating "comment-only trivial task" was one. No current P-row checks trivial/commit consistency.
- The only legitimate interactive tiering signal is **simple vs complex**, and a downgrade-only wrapper has **zero effect at low/medium criticality** (builder is already sonnet) — the machinery fires only for high/critical features.
- Interactive fail-loud on a malformed Complexity would halt the whole implement loop — an availability regression vs today's ignore; overnight's normalize-up event has no interactive surface.
- "Lighter verification posture" belongs to wild-light #373 per the ticket's own Role split (this repo owns what the builder is *told*, at most a "don't over-verify beyond the Verification field" template line).

## Events & Measurement

**How `batch_dispatch` works**: emitted by `implement_transition.py` batch mode (`{ts, event, feature, batch: N, tasks: [ids]}`), idempotent on `{"batch": N}`, invoked through the **gated** `cortex-lifecycle-advance implement-transition --mode batch` path, which routes through `advance()`'s from-state gate via `transition_table.py`'s `implement.dispatched` self-loop row (`EDGE_KIND_IN_STATE_ACTION`). Live consumer: `cortex_command/pipeline/metrics.py:302–307` (`batch_count`/`batch_sizes`) — pipelining that keeps "batch" as a fictional recording unit silently drifts those metrics.

**What `task_complete` would require**: a new `--mode task-complete --task <id> --commit <sha>` arm (a different lifecycle moment than either existing mode — today's §2d checkpoint calls **no verb at all**), argparse + dispatch wiring in `advance.py` (both enumerate modes explicitly), a `transition_table.py` self-loop row, a registry row (`scan_coverage: manual`), and **no PROTOCOL_VERSION bump** (additive; a new mode old prose never calls — matching the ticket's Edge and every additive event since the handshake was introduced). Adversarial hardening: idempotency must qualify on **(task, commit)**, not task alone, or the cycle-2 completion after rework is false-skipped; and the arm needs `implement-rework` from-state coverage or every rework-cycle emission is **refused** by the gate — pushing the orchestrator at the very `--from-state` escape item 7 worries about.

**The reconstruction was actually run** (lifecycle 378: 5 batches, 11 tasks): `batch_dispatch` ts + `[x] done (<sha>)` plan annotations + `git log` yields complete per-task dispatch→complete durations (1m17s–31m40s), and empirically confirms the barrier — each batch dispatches 1–2 min after the prior batch's *slowest* task lands. **Limits**: the sha-annotation habit is undocumented (zero instruction anywhere in `skills/`), adopted in ~14 of 56 plan.md files (374 uses bare `[x] complete` — nothing per-task reconstructible); the sha lives in prose, not machine-queryable rows; and under sync dispatch a §2d-checkpoint `task_complete` timestamp would measure the barrier, not the task (all tasks in a batch checkpoint together) — commit timestamps are strictly better data in that world. The sha-in-checkbox annotation records completion **regardless of dispatch mode**; only dispatch-*time* reconstruction is barrier-dependent.

**Item 7 (`--from-state`)**: the flag substitutes the caller's value as `effective_from` (advance.py:947) — `--from-state <detected>` makes the gate pass for **any** mismatch cause, including genuine corruption; advertising it in the refusal trains the orchestrator to defeat the gate mechanically. The in-file precedent cuts the other way: `_pause_refusal()`'s `typed_resume` field recommends the *typed* surface over the hand-append. The served envelope already carries the sanctioned value (`advance_contract.expected_from_state`, honored at advance.py:905–906), so the correct remedy is **re-sync** ("re-run `cortex-lifecycle-next`, thread its advance_contract") — never echo-back-the-detected-phase. Second, independent gap: neither `plan.md` §4 nor `review.md` §5 routing tables carries a `refused` row at all — operators currently have zero scripted guidance.

## Enforcement Architecture

**Decision rule** (CLAUDE.md + `docs/policies.md` + `project.md:41`): scheduling/judgment = lean control-flow prose; recording/integrity = verb; prose-only is fine where deviation is cheap; where the ticket names evidence deviation was NOT cheap, prefer the cheapest structural lever that closes the observed gap — new machinery still clears Deletion bias.

**The in-repo structural precedent for item 5** is `skills/critical-review/`: parallel reviewers return `<!--findings-json-->` message envelopes, only the synthesizer writes durable artifacts (via a dedicated console-script), and SHA-256 stability gates (`check-artifact-stable`/`check-synth-stable`) catch drift structurally. Ranked options for review.md: (1) message-envelope contract for spawned verifiers — cheapest, closes the observed mode; (2) tool-capability restriction on the verifier dispatch — but the Agent tool exposes only `model`/`subagent_type`, so naming a restricted type (e.g. Explore-shaped) is still prose-compliance at dispatch time; (3) sentinel+rehash gate — disproportionate to one near-miss; (4) write-guard verb — no precedent for free-form markdown. **Wording caveat (adversarial)**: "only the agent dispatched via §2's template writes review.md" as drafted forbids review.md's own §4 missing-drift re-dispatch *and* §4a drift-apply re-dispatch — the rule must be scoped to the reviewer **role including its retries**, constraining who writes, not call-count. Share-the-tallies: run the suite once orchestrator-side in §1 and inject **summary + log path** (not the ~11-minute transcript) into the reviewer template, with a staleness rule.

**Item 6**: named checklist rows have the strongest evidence in the ticket (the existing generic P4 pass ran on #353's plan and missed all three rig defects), but the checklist header requires binary-checkable criteria and P4 itself is proof judgment rows miss — P11 (hub-file: file in ≥3 tasks' `Files` without an early seam task) is mechanical; P12 (dress-rehearsal) is a judgment classification and **collides with P7** (self-sealing verification: a rig validating its own discarded sample is literally the artifact-the-task-created verifying that task) — needs an explicit carve-out or stays an authoring bullet.

## Adversarial Review

- **The dispatch-mode decision is asymmetric and mostly not harness-ownable**: only synchronous is pinnable (`CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1` — which also kills background Bash/Monitor for the whole session); background is an undocumented upstream default that changed twice in 13 patch releases. This repo ships to arbitrary runtimes — any mode-dependent prose is guaranteed wrong for part of the installed base; mode-*adaptive* prose (two protocols in one always-loaded file) is the worst outcome. The ticket's "cheap probe" is a single sample of one runtime version.
- **Pipelining trades tokens for wall-clock**: ~25 per-task completion turns + per-task checkpoints replace 9 batch returns against a saving bounded by one ~30-min straggler; each extra orchestrator turn re-reads a growing context (turns × context). Hidden dependencies: option C needs an unscoped per-task *dispatch* event; `metrics.py`'s `batch_count`/`batch_sizes` drift silently; §2d's `git log -N` attribution assumes batch-contiguous commits; same-tree parallel commits race on `.git/index.lock` and the dual-source pre-commit hook lets one in-flight builder block every other builder's commit — pipelining widens that window from bursty to continuous; the `implement-batch-failure` pause mid-stream with completions still arriving is runtime-undefined; §3 rework has no batch-numbering story.
- **Item 1's channels are individually broken**: the template already mandates a report (step 5); no SubagentStop hook exists in this repo (the "stop hook eats the report" cause is operator-environment config prose can't fix); SendMessage child→parent is the documented-unreliable direction; a worktree builder's report file lands in the *worktree's* `cortex/` and reaches the orchestrator only at merge-back (too late), possibly sandbox-prompting (worktree paths aren't the registered umbrella); cycle-1 reports are stale in rework cycle 2; a failed builder that committed its report file defeats §2d's zero-commits-means-failed signal.
- **Attestation hazard**: flipping `[x]` or emitting `task_complete` from builder self-report without §2d git verification re-opens the fabricated-attestation hole the merge-consent cross-check exists to close; durable report files also make injected content (a builder that read hostile data) committed and re-readable.
- **Item 7 as proposed is a universal gate bypass** (see Events & Measurement — remedy is re-sync, not `--from-state <detected>`).
- **MUST-escalation policy** applies to items 1/5 and nobody in the ticket cited it.
- **Reference growth lands silently**: items 1/2/5/6 plausibly add 25–40% to implement.md against the leanification floors; no test sees it.

**Ranked mitigations**: (1) write mode-agnostic prose — report in the final message, checkpointing derives from git state, never from return shape; treat background-dependent pipelining as blocked on upstream stability. (2) Reproduce the round-trip on the current runtime before building item 1; if it doesn't reproduce, the fix is one sentence tightening template step 5's report format. (3) Take plan-time straggler isolation now; gate runtime pipelining on a measured case that survives token accounting. (4) Add the trivial⇒no-commit P-row; tiering, if built, is a downgrade-only wrapper, simple-vs-complex, high/critical-only, absent/malformed→inherit. (5) Mandate sha-in-checkbox now; build `task_complete` only with (task, commit) idempotency and rework from-state coverage, after item 2's fate is known. (6) Role-scoped single-writer wording covering §4/§4a retries; tallies as summary + log path. (7) Refusal gains a typed re-sync recommendation plus `refused` routing rows in both tables.
