# Specification: trim-the-interactive-implement-loops-orchestration

## Problem Statement

A 25-task interactive implement (wild-light #353) paid a measurable slice of wall-clock and tokens to the orchestration layer this repo owns: per-dispatch report round-trips, batch barriers idling ready work, one model tier for every task, unobservable completions, and a review phase that duplicated its most expensive step. Research resolved the ticket's gating question — the dispatch mode (synchronous vs background) is upstream-owned, version-dependent, and not harness-ownable (only synchronous is pinnable, with session-wide collateral; background behavior changed twice in recent patch releases; the #353 round-trip does not reproduce on the current runtime). This spec therefore lands the durable, mode-agnostic subset: tighten the report contract where it already lives (the builder's final message), make per-task completion measurable with existing tools, fix the plan-authoring defects that caused the observed stragglers and rig failures, close the review phase's single-writer and duplicate-suite gaps, and give gate-mismatch refusals a sanctioned re-sync remedy. Runtime pipelining and a `task_complete` event are explicitly deferred with recorded rationale. Beneficiaries: every interactive lifecycle run; unbuilt, each future implement re-pays the same overhead.

## Phases

- **Phase 1: Implement-loop prose** — mode-agnostic report contract, commit-sha completion annotation, scoped verification.
- **Phase 2: Plan & review authoring discipline** — straggler isolation, hub-file seam, dress-rehearsal, trivial-consistency checks; review single-writer and shared test baseline.
- **Phase 3: Verb layer** — gate-mismatch re-sync remedy + `refused` routing rows; per-task builder tiering in `cortex-resolve-model`.

## Non-Requirements

- **No runtime dependency-pipelined dispatch** — deferred: token cost (extra orchestrator turns × growing context) is unpriced against a saving bounded by one straggler; it requires an unscoped per-task dispatch event, a `metrics.py` batch-semantics decision, a commit-serialization story (index.lock races; the dual-source pre-commit hook lets one in-flight builder block all others), and it interacts undefined with the `implement-batch-failure` pause. Blocked on upstream dispatch-mode stability. "Batch N+1 waits for batch N" stands.
- **No `task_complete` event, no new events-registry rows, no `implement_transition.py` changes, no PROTOCOL_VERSION bump** — under the batch barrier a checkpoint-time row measures the barrier, not the task; commit timestamps (requirement 3) are strictly better data. Revisit only alongside pipelining, with (task, commit)-qualified idempotency and `implement-rework` from-state coverage.
- **No report file, no SendMessage channel, no SubagentStop hook work** — each channel is individually broken (worktree-invisible files, documented-unreliable worker→coordinator messaging, operator-environment hook config this repo can't fix).
- **No Agent Teams adoption; no `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS` mandate** — the prose is mode-agnostic instead.
- **No kept-pause marker, kind, or `kept-pauses-data.toml` change** (parity test untouched).
- **No prose model literals and no reuse of the pipeline `_MODEL_MATRIX`** — ADR-0023 single-matrix-owner; the recorded lattice separation stands (overnight's absent→`simple` default is fail-safe-down and must not be imported).
- **No verification-execution tooling** (probe dedupe, green-stamp, suite-to-area mapping) — wild-light #373 owns that layer.
- **No `--from-state <detected>` advertisement** — it is a universal gate bypass (reverses the ticket's Integration suggestion; flagged at approval).

## Edge Cases

- **Mixed-complexity batch**: model resolves per task; tasks in one batch may run different models — dispatch semantics otherwise unchanged.
- **Malformed/absent task Complexity**: absent → feature-level cell (today's behavior); present-but-invalid → stderr warning + feature-level cell. The implement loop never halts on a Complexity typo (availability preserved; fail-safe up).
- **Low/medium-criticality features**: the tiering rule is a no-op (builder cell is already sonnet) — expected, not a defect.
- **Rework cycles (§3)**: a reworked task's annotation is updated to the newest verifying commit sha + timestamp at its re-checkpoint. Rework re-dispatches emit no `batch_dispatch` row, so rework *durations* are undefined — only completion instants are recorded (accepted; revisit with any future task_complete event).
- **Worktree dispatch**: §2d annotates the sha verified on the worktree branch; merge-back's five cases plain-merge and never rewrite task commits, so the sha stays valid there. The real durability risk is downstream — a squash-merge at Complete's PR flow plus branch deletion can make per-task shas unreachable/gc-prunable — which is why the annotation embeds the committer timestamp: the completion instant survives even when the sha no longer resolves.
- **Re-synced rework transition provenance**: when R11's re-sync clears an `implement-rework` gate-mismatch, the emitted `phase_transition` row's `from` field stays the table-canonical `implement` (hard-coded in `_emission_plan`); true provenance is recoverable from the adjacent `review_verdict` and `review→implement-rework` rows — accepted trade-off, not silently wrong data.
- **Baseline staleness**: implementation commits after the §1 test baseline → orchestrator re-runs once and replaces the baseline; reviewers never self-serve a re-run.
- **`refused` persists after re-sync**: escalate to the operator with both the detected phase and the expected from_state — do not retry with the detected value.
- **Reviewer runs out of context mid-write**: §4's missing-drift re-dispatch and §4a's cap-2 retries remain authorized writers under the single-writer rule (role-scoped, not call-count-scoped).

- **ADR-0023**: model names live only in the verb; the golden-anchor test (`tests/test_resolve_model.py`) updates in the same change; `_LIFECYCLE_MATRIX`'s recorded separation from `pipeline/dispatch.py` is preserved (docstring updated to note the task-complexity extension).
- **Verb house style** for the advance.py change: never-crash `{state,...}` envelope, additive field only, `PROTOCOL_VERSION` untouched.
- **MUST-escalation policy** (`docs/policies.md`): the policy governs MUST/CRITICAL/REQUIRED escalations, each needing a linked evidence artifact — prose additions here use imperative or SHOULD phrasing and add no such escalation. (Separately, as a spec-local choice, the single-writer rule is phrased as role authorization rather than a bare prohibition; ordinary scoping words like requirement 4's "only that" are not policy-restricted.)
- **Prose budget**: net addition across the three reference files stays lean (target ≤ ~25 net lines total) — reference growth is invisible to tests and counts against the leanification floors.
- **Test command**: `just test` (from `cortex/lifecycle.config.md`) is the review baseline command.

## Proposed ADR

### Proposed ADR: 0030-mode-agnostic-interactive-dispatch

Interactive lifecycle dispatch prose is written mode-agnostic: the builder's exit report lives in its final message, and task completion is derived exclusively from the git checkpoint — never from return-delivery shape. Rationale: the dispatch mode (synchronous vs background) is owned by the Claude Code runtime, changed twice in recent patch releases (background-by-default ≥v2.1.198; report-bearing completion notifications ≥v2.1.211), and only the synchronous direction is pinnable — via a session-wide env var with collateral damage. Mode-dependent or mode-adaptive prose would be wrong for part of the installed base at any given time. Consequence: dependency-pipelined dispatch and a `task_complete` completion event are deferred until the upstream substrate is stable and a measured case survives token accounting (turns × context); the batch barrier stays. Trade-off accepted: bounded wall-clock idling behind stragglers (mitigated at plan time by straggler isolation) in exchange for prose that is correct on every runtime version and zero new coordination machinery. This reverses #401's "decide the dispatch mode first" framing: the durable decision is to not couple the harness to either mode.
