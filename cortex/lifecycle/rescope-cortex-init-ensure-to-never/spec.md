# Specification: rescope-cortex-init-ensure-to-never

## Problem Statement

`cortex init --ensure` — the in-session entry point invoked by `cortex-lifecycle-init-ensure` before each `/lifecycle` phase — currently writes `~/.claude/settings.local.json` (three calls in `_run_ensure`: `validate_settings`, `register`, `unregister_matching_in_place` at `cortex_command/init/handler.py:163,233,238`). Inside a Claude Code session the sandbox denies writes to `~/.claude/`, so the calling sub-agent must retry with `dangerouslyDisableSandbox: true`. For a new adopter, that visible permission-bypass on first contact is a deal-breaker — users distrust a tool that asks to bypass the security model it was just installed under, and the README/landing-page guidance leads adopters straight to `/plugin install` + `/lifecycle` (the modal first-contact path) without ever running terminal `cortex init`. This work restricts `--ensure` to repo-scope writes only: it never touches `~/.claude/`. The `cortex/`-refresh path (marker present) keeps working under the sandbox because those writes are inside the repo working directory. The clean-repo bootstrap path is removed — `--ensure` refuses with exit 2 and a directive pointing the user to terminal `cortex init`, which (unchanged) carries the user's explicit consent to mutate `~/.claude/`. The flag encodes the consent boundary: a write the user typed in their terminal may mutate `~/.claude/`; a write an in-session AI helper performs may not.

## Phases
- **Phase 1: Rescope `--ensure`** — remove the three `~/.claude/` writes from `_run_ensure`; convert the marker-absent clean-repo case from bootstrap to refuse-with-directive (exit 2); preserve the R19 foreign-content refusal and terminal `cortex init`.
- **Phase 2: Reconcile dependents** — amend the completed `auto-apply-cortex-init-at-lifecycle` spec inline; update the affected tests; correct README + landing-page guidance so the modal first-contact path runs terminal `cortex init` before `/lifecycle`.

## Requirements

1. **In-session `--ensure` performs no `~/.claude/` write.** Remove the three `~/.claude/`-touching calls from `_run_ensure` (`cortex_command/init/handler.py`), identified by call rather than by line (line numbers given are current and will drift as this function is edited): `settings_merge.validate_settings` (currently :163), `settings_merge.register` (currently :233), `settings_merge.unregister_matching_in_place` (currently :238). No replacement `~/.claude/` write is added anywhere in the `--ensure` path. Acceptance: a unit test spies on `cortex_command.init.settings_merge` and asserts that across the cases that previously reached the post-dispatch block — case (ii) marker-present+hash-mismatch, case (iii) marker-absent+clean, and case (v) R8 recovery — `register`, `validate_settings`, and `unregister_matching_in_place` are each called zero times (cases (ii) and (v) must be exercised so the assertion is non-trivial, not only the early-return case (i)); and after `cortex init --ensure` runs in a temp HOME, `~/.claude/settings.local.json` is byte-identical to its pre-invocation state (or still absent). **Phase**: Phase 1.

2. **Marker-present refresh succeeds under the sandbox after the change.** Cases (ii) marker-present + hash-mismatch and (v) R8 recovery scaffold `cortex/` (repo-scope) and refresh the marker, with no `~/.claude/` access. **Baseline correction**: today these cases DO touch `~/.claude/` — the post-dispatch `register`/`unregister` block (currently handler.py:229-238) runs on cases (ii)/(iii)/(v), so the refresh path currently trips the sandbox in-session. R1's removal is therefore load-bearing for this requirement too, not only for the clean-repo case; "succeeds under the sandbox" is the post-change state, not a preserved current behavior. Case (i) marker-present + hash-match remains an early `return 0` no-op. Acceptance: in a temp repo with a drifted marker-present `cortex/`, `cortex init --ensure` exits 0, `cortex/` content is refreshed (a `cortex/.cortex-init` marker mtime/content reflects the refresh), and no `~/.claude/` file is created or modified. **Phase**: Phase 1.

3. **Marker-absent clean repo refuses with exit 2 + a directive.** Case (iii) (marker absent AND `cortex/` absent-or-empty) no longer bootstraps. It refuses via the handler's user-correctable-refusal path (`ScaffoldError`/`SettingsMergeError` → exit 2, per the taxonomy in `handler.main()`, currently ~:536-578), emitting a single-line stderr directive that names terminal `cortex init` as the corrective action and is textually distinct from the R19 foreign-content message (`scaffold.check_content_decline`, which itself contains the string "cortex init"). To make distinctness binary-checkable, the clean-repo directive must contain a unique marker substring that does NOT appear in the R19 message — e.g. it must reference that the repo is not yet initialized / has no `cortex/` (the R19 message instead references "pre-existing content"). No `cortex/`, `CLAUDE.md`, `.gitignore`, or `~/.claude/` writes occur. Acceptance: `cortex init --ensure` in a clean scratch repo (no `cortex/`) exits 2; its stderr contains the chosen clean-repo marker substring AND that substring is absent from the R19 `check_content_decline` message (asserted against both messages in one test, so reusing the R19 text verbatim fails the gate); `git status --porcelain` shows no new/modified files and no `~/.claude/` file is created. **Phase**: Phase 1.

4. **Marker-absent + `cortex/`-has-content (R19) is unchanged.** Case (iv) still fires `scaffold.check_content_decline(repo_root)` → exit 2 with the existing R19 foreign-content message. Acceptance: in a scratch repo where `cortex/SOMETHING.md` exists but no `.cortex-init` marker, `cortex init --ensure` exits 2 with the R19 decline message (unchanged from current behavior); the message differs from the R3 clean-repo directive. **Phase**: Phase 1.

5. **Terminal `cortex init` is unchanged.** The standard `_run` path (currently `handler.py:438-535`) still writes the `~/.claude/` grant at Step 7 (`settings_merge.register`) and runs the Step 7b `cortex-worktrees` migration on `--update` (the migration is gated behind `if args.update:`, so a fresh `cortex init` writes the grant but does not run the migration — unchanged from today). `--update`/`--force`/`--unregister` are untouched. Acceptance: `cortex init` (no `--ensure`) in a clean scratch repo with a temp HOME writes `~/.claude/settings.local.json` containing the repo's `cortex/` entry under `sandbox.filesystem.allowWrite` and exits 0; the existing terminal-init test suite passes unchanged (`just test` exits 0). **Phase**: Phase 1.

6. **`CORTEX_AUTO_ENSURE=0` and worktree-attached guards are unchanged.** The `(a)` opt-out (`handler.py:143`) and `(b)` worktree-attached refusal (`handler.py:150`), plus the `cortex-lifecycle-init-ensure` worktree guard, behave exactly as today. Acceptance: `CORTEX_AUTO_ENSURE=0 cortex init --ensure` exits 0 writing nothing; the existing worktree-attached refusal test still asserts exit 2. **Phase**: Phase 1.

## Non-Requirements

- **No change to terminal `cortex init`, `--update`, `--force`, `--unregister`, `--revoke-worktree-auth`, `--verify-worktree-auth`.** Only the in-session `--ensure` path changes.
- **No relocation of the sandbox grant** to repo-local `.claude/settings.local.json` or committed `.claude/settings.json`. The grant continues to live in `~/.claude/settings.local.json`, written by terminal `cortex init`. (Relocation was an earlier mis-framing; it is explicitly out of scope.)
- **No ADR-0003 supersession or amendment.** ADR-0003's decision — terminal `cortex init` registers the `cortex/` grant in `~/.claude/settings.local.json` — still holds, because terminal init is unchanged. (ADR-0003 has no "Revisit when" clause; none is added.)
- **No deletion of a user's pre-existing `~/.claude/settings.local.json` entries.** `--ensure` simply stops adding new ones. Manual cleanup, if any, is the user's choice; document it only if the docs work (R9) has a natural place.
- **The `cortex-worktrees` legacy-entry migration is dropped from `--ensure` — by necessity, not oversight.** `unregister_matching_in_place("cortex-worktrees")` is a cortex-authored cleanup that mutates `~/.claude/settings.local.json`; it therefore cannot run in-session under the sandbox and must be removed alongside the other two `~/.claude/` calls. Consequence (acknowledged, not fixed here): a user whose only invocation path is in-session `/lifecycle` and who never runs terminal `cortex init --update` will not have stale `cortex-worktrees` entries expunged. This is acceptable — the entries are inert (the worktree base they named is obsolete), and terminal `cortex init --update` still performs the migration. Not re-homing the migration to a repo-scope location is in scope as a deliberate non-goal.
- **No lifecycle SKILL.md control-flow change.** `skills/lifecycle/SKILL.md:128` already halts on non-zero exit from `cortex-lifecycle-init-ensure`, so the new exit-2 routes through the existing gate. An optional one-line clarifying note is permitted but not required; if `skills/lifecycle/SKILL.md` is edited, the `plugins/cortex-core/` mirror regenerates via the dual-source pre-commit hook (edit canonical only).
- **No new event registration.** `--ensure` is silent operational like `--update`/`--force` (per the auto-apply spec's "No new event registration").
- **No `CLAUDE.md` rewrite required.** `CLAUDE.md:5` describes terminal `cortex init`'s `~/.claude/` registration, which remains accurate. An optional clarifying note that in-session `--ensure` does not perform that write is permitted, not required.
- **No overnight change.** Overnight is independent of the persistent grant (per-spawn `--settings` tempfiles in `cortex_command/overnight/sandbox_settings.py` are authoritative).

## Edge Cases

- **First contact on a fresh repo**: `/plugin install cortex-core` → `/lifecycle` → `cortex-lifecycle-init-ensure` → `--ensure` hits case (iii) → exit 2 → lifecycle halts at SKILL.md:128. The R3 directive must be actionable enough that the user runs terminal `cortex init` and re-invokes `/lifecycle` without reading source. This is the load-bearing UX surface.
- **Grant-shape drift in-session**: if a future cortex version changes the `allowWrite` entry shape, in-session `--ensure` refreshes `cortex/` content but does not (and cannot) update the `~/.claude/` grant; only terminal `cortex init --update` applies the new grant shape. Acceptable and consistent with the consent boundary.
- **CI / read-only `~/.claude`**: `--ensure` no longer touches `~/.claude/`, so a read-only or absent `~/.claude/` is no longer a `--ensure` failure surface (net simplification — graceful degradation is automatic).
- **Malformed `~/.claude/settings.local.json` during `--ensure`**: removing `validate_settings` means `--ensure` no longer surfaces a malformed-settings `SettingsMergeError` (exit 2). This is acceptable and intended — `--ensure` no longer reads or writes that file, so validating it is moot; terminal `cortex init` retains its own `validate_settings` pre-flight. A malformed user settings file is surfaced the next time the user runs a terminal `cortex init` verb, not by `--ensure`.
- **Halted lifecycle re-entry after the clean-repo refuse**: after `--ensure` refuses on a clean repo (R3) and the user runs terminal `cortex init`, the `cortex/.cortex-init` marker now exists, so the next `/lifecycle` invocation drives `--ensure` into case (i) (hash match → no-op) or case (ii) (refresh) and proceeds normally. No partial/in-between lifecycle state is created on the refuse path, because the refuse occurs at the `cortex-lifecycle-init-ensure` gate (SKILL.md:128) before any phase dispatch; re-invocation is a clean restart, not a resume of corrupted state.
- **Exit-2 frequency shift (acknowledged)**: this change moves `--ensure`'s exit-2 from rare paths (worktree-attached, foreign-content) onto the common clean-repo first-run path. The lifecycle SKILL.md halt-on-non-zero gate (SKILL.md:128) handles it unchanged, but the halt now fires routinely for new adopters — which is why R3's directive quality and R9's docs correction (run terminal `cortex init` first) are load-bearing rather than cosmetic.
- **Marker present but `cortex/` partially populated**: handled by the existing case (i)/(ii) hash dispatch; unchanged.
- **Two repos on one machine**: each repo's `--ensure` only refreshes its own `cortex/`; neither touches `~/.claude/` nor the other repo. No cross-repo interference from `--ensure`.

## Changes to Existing Behavior

- **MODIFIED**: `--ensure` case (iii) marker-absent + clean `cortex/`: bootstrap + exit 0 → refuse + exit 2 with a directive to run terminal `cortex init`.
- **REMOVED**: `--ensure`'s three `~/.claude/` touches — `validate_settings`, `register`, and the `cortex-worktrees` `unregister_matching_in_place` migration (all in `_run_ensure`'s pre-flight + post-dispatch block; current lines :163/:233/:238, anchored by call identity since Phase 1 edits this function). The migration is a cortex-authored cleanup whose removal from `--ensure` is mandated by the no-`~/.claude/`-write goal (see Non-Requirements).
- **MODIFIED**: `auto-apply-cortex-init-at-lifecycle/spec.md` — R4 dispatch-table case-(iii) row, R4 acceptance #1, and the clean-repo-bootstrap Problem-Statement clause, plus a top-of-file superseding pointer (deliberate contract revision of a completed spec, user-approved inline; `plan.md`/`review.md` left as historical records — see R7).
- **MODIFIED**: `README.md` `cortex init` OPTIONAL framing; `docs/index.html` cortex-core start-here path.
- **UNCHANGED**: terminal `cortex init` (all verbs), the R19 foreign-content refusal, overnight, ADR-0003, lifecycle SKILL.md control flow.

## Technical Constraints

- **Exit-code contract** (`handler.main()`, currently ~:536-578): 0 = success/no-op, 2 = user-correctable refusal (raised `ScaffoldError`/`SettingsMergeError`), 1 = unexpected failure. The R3 clean-repo refusal must surface as exit 2 via the user-correctable-refusal path (the same taxonomy that already maps R19/worktree-attached refusals to 2 by raising `ScaffoldError`) — not as a generic error (which would map to 1).
- **`cortex-lifecycle-init-ensure` passes exit codes through** (`cortex_command/lifecycle/init_ensure.py`): it delegates to `handler.main()` and returns its exit code, so no change is needed there for the new exit-2 to reach SKILL.md:128's halt.
- **Dual-source mirror**: canonical skill source is `skills/lifecycle/SKILL.md`; `plugins/cortex-core/skills/lifecycle/SKILL.md` is auto-generated. Only relevant if SKILL.md is edited (default: not edited).
- **The auto-apply spec is `status: complete`**: amending it (R7) is a sanctioned contract revision, not editing in-flight work.
- **Repo-scope writes are sandbox-allowed**: scaffolding under `<repo>/cortex/`, `<repo>/CLAUDE.md`, `<repo>/.gitignore` is inside the session CWD and needs no grant — this is why the marker-present refresh path (R2) works in-session without `~/.claude/` access.
