# Specification: re-validate-test-worktree-seatbeltpy-on

## Problem Statement

`tests/test_worktree_seatbelt.py` exists to detect regressions in `cortex_command/pipeline/worktree.py:resolve_worktree_root()` — the single chokepoint cited by `cortex/requirements/multi-agent.md:77` that protects Seatbelt-writability for every overnight per-feature dispatch worktree. Today the test runs at most once per implementer's manual invocation; its skipif gate (`os.environ.get("CLAUDE_CODE_SANDBOX") != "1"`) selects on an env var not documented as set by Claude Code on macOS Seatbelt, and the lone existing `f_row_evidence` event in the parent lifecycle was implementer-attested without raw stdout preserved. The combination silently fails open: a regression that breaks Seatbelt-writability would only surface when the next implementer happened to re-run the test. This feature replaces the test's gate with a kernel-level capability probe anchored on a path the project actually denies, fixes a latent NameError in the smoke runner, and runs the test once per overnight cycle at session start via a thin probe module — accumulating durable evidence under a tracked log without coupling smoke or the per-feature dispatch path to a new failure surface.

## Phases

- **Phase 1: Test correctness foundation** — replace the env-var skipif in `tests/test_worktree_seatbelt.py` with a kernel-probe fixture using `<repo>/.git/HEAD` (which `build_orchestrator_deny_paths()` already denies); fix the latent `os` import bug in `cortex_command/overnight/smoke_test.py`; amend the parent lifecycle's stale `CLAUDE_CODE_SANDBOX=1` acceptance command reference; register the `seatbelt_probe` event in `bin/.events-registry.md`.
- **Phase 2: Recurring probe at overnight session-start** — add `cortex_command/overnight/seatbelt_probe.py` providing `run_probe()`, which spawns `claude -p` to invoke pytest under sandbox-active settings and reads result files from `$TMPDIR/` after the spawn returns (bypassing model paraphrase). Wire the probe into the overnight runner's session-start sequence; emit a `seatbelt_probe` JSONL event to the session's `overnight-events.log` and the top-level `cortex/lifecycle/seatbelt-probe.log`. Failure is surfaced in the morning report; nothing hard-blocks. Document.

## Non-Requirements

- **Hard-blocking the overnight session on probe failure.** The runner logs the event and continues to round 1. Rationale: the probe spawn has a real failure surface (claude API rate-limit, network blip, agent paraphrase) and hard-blocking creates a "one bad probe kills the overnight" hazard inconsistent with the project's "surface failures in the morning report; keep working unless blocked" principle.
- **Integrating the probe into `verify_dispatch_readiness`.** That function runs once per feature dispatch; wiring the probe there would multiply spawn count by `N_features` per night. The probe lives in the runner's session-start sequence (called once per overnight cycle).
- **A new `cortex-seatbelt-probe` console script.** The probe is a Python module imported and called by the runner; there is no operator-facing command. If operator-runnable invocation is wanted later, a justfile recipe can wrap `python3 -m cortex_command.overnight.seatbelt_probe` — out of scope for this lifecycle.
- **Stream-json envelope parsing.** The probe uses `--output-format=json` for the spawn-completion signal only; pytest's literal output and exit code are read from `$TMPDIR/`-resident files the agent's Bash command writes via `tee` and `printf`. This bypasses the model-paraphrase failure mode without coupling the probe to `stream-json` event-stream parsing.
- **Setting `"allowUnsandboxedCommands": false` separately for the probe spawn.** `build_sandbox_settings_dict` already sets this to `false` at `sandbox_settings.py:196` for every spawn that uses it — no probe-specific override needed.
- **Per-feature `events.log` emission of `seatbelt_probe`.** The probe is session-scoped; per-feature emission is unnecessary and would pollute per-feature event logs.
- **Retroactive annotation of the parent lifecycle's existing `f_row_evidence` event** in `cortex/lifecycle/restore-worktree-root-env-prefix/events.log:14`. That event is treated as historic-evidence-of-unknown-quality; the first session-start probe under this lifecycle produces a kernel-probe-gated `seatbelt_probe` event that supersedes it.
- **Renaming `f_row_evidence`.** The new event uses the name `seatbelt_probe`; the parent lifecycle's existing event name is preserved historically.
- **Linux/bwrap support for the kernel-probe gate.** Per `docs/overnight-operations.md` ("Sandbox enforcement is macOS-Seatbelt-only ... behavior under Linux/bwrap is undefined"), the probe is macOS-only. Linux/bwrap behavior remains undefined; the test continues to skip if the kernel-probe write succeeds on Linux.
- **Modifying the existing test functions' bodies.** R1 only modifies the skipif decorators (replacing them with the fixture); the existing assertions in both test functions are preserved verbatim.
- **A justfile recipe for the probe.** Operator-runnable invocation is deferred; the probe runs only when the overnight runner invokes it.

## Edge Cases

- **`claude` binary missing or not in `PATH`**: `run_probe` catches the `FileNotFoundError`, sets `result="failed"` with `cause="claude binary not found"`, `pytest_exit_code=null`, `stdout_path=null`. The morning-report banner surfaces this; the runner continues.
- **`claude -p` returns non-zero exit code** (auth failure, rate limit, agent refusal, max-turns hit before the Bash tool fires): probe sets `result="failed"` with `cause` quoting the non-zero exit and a short tail of stderr. Result and output files may not exist; `stdout_path=null`, `stdout_sha256=null`.
- **The agent paraphrases or skips the Bash invocation**: `<result_path>` is not created. Probe detects this via `result_path.exists()` and sets `result="failed"` with `cause="result file not written; agent likely paraphrased instead of executing the bash command"`. This is the robust path — the probe never trusts model text for pass/fail.
- **Pytest summary line format change** (pytest upgrade changes summary token shape): R4's regex matches `(\d+) passed`, `(\d+) failed`, `(\d+) skipped`, `(\d+) error` tokens individually within `<output_path>`'s bytes. Adding context framing (`===`) does not break individual-token matching. If a future pytest version changes the per-result-type tokens entirely, the probe emits `result=failed` with `cause="unparseable pytest summary"`; the morning-report banner surfaces this and the operator updates the regex.
- **Test count drift** (a contributor adds a third test function): probe's assertion is `passed >= 2 AND failed == 0 AND skipped == 0 AND error == 0` — drift-tolerant by design. The parent lifecycle's literal `passed=2,failed=0,skipped=0` criterion is out of scope (Non-Requirements); R7's amendment removes the prefix only, not the literal count.
- **`$TMPDIR` resolves to a path Seatbelt denies in the spawn-active settings**: `build_orchestrator_deny_paths` does not deny `$TMPDIR/`, so the agent's Bash writes succeed. The probe's settings dict explicitly passes `allow_paths=[resolved-$TMPDIR]` to make the allow-listing exact. If a future deny-path change collides with `$TMPDIR/`, the agent's Bash write fails, result file is empty, probe sets `result="failed"` with `cause="result file empty"`.
- **`runner.py`-context probe failure**: if `run_probe` itself raises an unhandled exception (e.g., session_dir not writable, tempfile creation fails), the runner catches and translates to a `result=failed` event with `cause=<exception text>` rather than letting the exception abort session startup. The runner logs the event and continues to round 1.
- **Concurrent probe invocations**: the probe's tempfile paths use a UUID-randomized basename so concurrent invocations do not collide on the result/output files in `$TMPDIR/`.
- **`CORTEX_SANDBOX_SOFT_FAIL=1` set**: the probe reads `read_soft_fail_env()` and passes `soft_fail=True` into `build_sandbox_settings_dict`, which downgrades `failIfUnavailable` to `false`. The `seatbelt_probe` event records `softfail_active=true`. The morning report's existing `CORTEX_SANDBOX_SOFT_FAIL activation` banner is independent and continues to fire if soft-fail was active anywhere in the session.
- **Sentinel path `.git/HEAD` is missing** (the test runs in a non-git checkout): the `os.open` call raises `FileNotFoundError`, not `PermissionError`. The fixture treats `FileNotFoundError` as "kernel probe inconclusive" and calls `pytest.fail("kernel probe sentinel .git/HEAD missing; run from a git checkout")` rather than skipping (silent skip would be the failure mode the lifecycle exists to close).
- **Existing test file modified by a future feature dispatch**: out of scope here. A follow-up could add a content-hash assertion to the probe; not in scope.

## Technical Constraints

- **Per-spawn settings file required** (per `cortex/requirements/multi-agent.md:23`): every `claude -p` orchestrator-style spawn passes `--settings <tempfile>` with the `sandbox.filesystem.{denyWrite,allowWrite}` dict. The probe complies via the existing `sandbox_settings.write_settings_tempfile` helper.
- **`"allowUnsandboxedCommands": false` already on**: per `cortex_command/overnight/sandbox_settings.py:196`, `build_sandbox_settings_dict` sets this for every spawn. The probe inherits this default — no explicit override is needed in the spec or the implementation.
- **Sandbox-settings tempfile location** (per `cortex/requirements/pipeline.md:158`): `cortex/lifecycle/sessions/<session_id>/sandbox-settings/cortex-sandbox-*.json`, mode `0o600`, atomic write. The probe writes its settings tempfile through the existing helper, which already enforces this convention.
- **Sentinel path is anchored on `build_orchestrator_deny_paths`'s output**: `<repo>/.git/HEAD` is one of `GIT_DENY_SUFFIXES` at `sandbox_settings.py:55-60`. Any test or probe that needs a kernel-probe sentinel anchored on Cortex's documented deny set MUST use one of these four suffixes. The user's interactive Claude Code session is expected to deny these same paths in `~/.claude/settings.local.json`; the implementation verification step in R1 acceptance is the gate that catches a missing deny.
- **`os.O_WRONLY` open without `O_TRUNC` is safe on `.git/HEAD`**: opening a file for write without truncating does not modify the file content. The FD is closed immediately. No write, no truncation, no corruption of the user's git state under the non-sandbox path.
- **Probe uses orchestrator-style settings, NOT dispatch-style**: `build_dispatch_allow_paths` puts the per-feature worktree plus `OUT_OF_WORKTREE_ALLOW_WRITERS` (including `$TMPDIR/`) into `allowWrite`. The probe is not a per-feature dispatch — it deliberately uses `build_orchestrator_deny_paths` (the git-internals deny set) with an explicit minimal `allow_paths=[resolved-$TMPDIR]` for its own tee/printf targets. This keeps the probe surface tight without piggybacking on the dispatch allow-list semantics.
- **Append-only JSONL** for `cortex/lifecycle/seatbelt-probe.log` and `overnight-events.log` (per `cortex/requirements/pipeline.md:129`). Atomic appends via `cortex_command.common.atomic_write`.
- **macOS-Seatbelt-only enforcement scope** (per `docs/overnight-operations.md`): probe is intended for macOS; Linux/bwrap behavior is best-effort and not blocking.
- **No prompt-corpus emission**: the `seatbelt_probe` event literal lives only in `cortex_command/overnight/seatbelt_probe.py` and `cortex_command/overnight/runner.py` (Python source), not in `cortex_command/overnight/prompts/*.md` or `skills/**/*.md`. `scan_coverage: manual` in the events-registry row is correct.
- **No new spawn site beyond the probe's**: the probe is the only new `claude -p` invocation introduced. The per-feature dispatch and orchestrator-round spawns are unchanged.
- **No hard-block contract on `verify_dispatch_readiness`**: this spec does NOT add a probe step to `verify_dispatch_readiness`. That function's three-check structure (auth, worktree, ...) is preserved as-is.
