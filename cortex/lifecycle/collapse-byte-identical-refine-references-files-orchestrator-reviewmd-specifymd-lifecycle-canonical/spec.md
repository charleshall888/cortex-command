# Specification: Collapse byte-identical refine/references files (orchestrator-review.md + specify.md → lifecycle canonical)

## Problem Statement

`skills/refine/references/orchestrator-review.md` (183 lines) and `skills/refine/references/specify.md` (163 lines) are near-byte-identical to their `skills/lifecycle/references/` siblings — refine's copies differ only in deliberately-omitted plan-phase content (the P8 row in orchestrator-review; the trailing `### 5. Transition` block + `## Hard Gate` table in specify). Maintaining two canonical copies has empirically drifted (parent epic 172 audit found 1-line drift) and burns ~346 lines of static-corpus cost. This is the largest single deduplication move identified by the audit and the prerequisite for stream F (vertical-planning adoption, ticket 182). Eliminating the duplication consolidates refine's spec-phase contract into one canonical source while preserving refine's documented adaptations.

## Edge Cases

- **End-user plugin-cache install**: refine ships under `~/.claude/plugins/cache/<marketplace>/cortex-core/<version>/skills/refine/`. The R3 edit uses `${CLAUDE_SKILL_DIR}/../lifecycle/references/specify.md` — `${CLAUDE_SKILL_DIR}` substitutes at SKILL.md invocation time to refine's directory regardless of CWD, and `..` walks up to the plugin's `skills/` parent. This resolves correctly under both repo-development and plugin-cache layouts.
- **Lifecycle's own use of bare `references/orchestrator-review.md` (line 145) is unchanged**: lifecycle SKILL.md invokes specify.md, Claude reads specify.md as raw text, sees `Read references/orchestrator-review.md`, and resolves it relative to the file containing the reference (lifecycle/references/) — the same heuristic that makes refine's existing bare-relative reads work. No `${CLAUDE_SKILL_DIR}` substitution is introduced into lifecycle's specify.md because the substitution is documented to apply only to the invoked skill's SKILL.md at invocation time, not to files read via the Read tool. When refine reads lifecycle's specify.md mid-flight (after R3), the same heuristic Claude uses for refine's own intra-skill reads applies to lifecycle's transitive intra-directory read — both resolve relative to the file containing the reference.
- **Smoke test fixture impossibility under refine standalone**: a fully synthetic fixture cannot reliably exercise the orchestrator-review hop because (a) `complexity` and `criticality` are *outputs* of refine's Clarify phase, not preconditions an operator can pre-set; (b) refine standalone never emits `lifecycle_start`, and lifecycle/orchestrator-review.md's Applicability check defaults `criticality: medium` if absent but does not specify a tier default — leaving the skip-rule evaluation undefined under refine. R8b therefore defers the runtime smoke to the *next* genuine refine invocation post-merge, which has all the operator-driven inputs the synthetic path lacks.
- **Spurious `phase_transition` event emission**: lifecycle/specify.md §5 contains a copy-paste-ready event JSON template plus a `/cortex-core:commit` invocation. R4's tightened §5 adaptation explicitly forbids both, defending against post-Opus-4.7 literal prompt-following.
- **Hard Gate inheritance**: lifecycle/specify.md ends with a `## Hard Gate` table (lines 177–186) refine inherits unannotated today. R5 adds an explicit "Applies" acknowledgment so the adaptation list is exhaustive over the file's top-level sections.
- **Ticket 173 lands first and shifts line numbers**: ticket 173 deletes ~40 lines at refine SKILL.md:117–157, shifting the current line 177 upward. R3's content-addressed edit (against the literal `Read \`references/specify.md\`` string) absorbs the shift without manual reconciliation.
- **Forward-coupling cost when stream F (ticket 182) adds sections to lifecycle/specify.md**: This is acknowledged as a cost of approach A. The mitigation is documented (refine SKILL.md adaptation list becomes the contract); no automated re-audit owner is created in this ticket. Ticket 182 inherits responsibility for re-auditing refine's adaptation list when adding new sections.
- **Sibling 182 contains a forward-ordering note that mentions the deleted path**: The path is also mentioned in `backlog/182-*.md:105` and in `research/vertical-planning/research.md` (this ticket's discovery source). R10's pathspec excludes `backlog/`, `research/`, and other historical-record dirs precisely because such mentions are non-live record-keeping prose. R10 does not require these be scrubbed.

## Technical Constraints

- **Dual-source parity model**: `tests/test_dual_source_reference_parity.py` discovers canonical–mirror pairs via `skills/*/references/*.md` glob; deleted canonical files automatically drop from the parametrized count. No test changes needed.
- **Build-plugin rsync semantics**: `justfile` `build-plugin` recipe runs `rsync -a --delete skills/$s/ plugins/$p/skills/$s/` per skill; `--delete` prunes orphaned mirror copies. After this ticket: `just build-plugin` is required before commit, and the resulting plugin-tree changes must be staged or the pre-commit Phase 4 drift hook will block the commit.
- **Pre-commit drift hook**: `.githooks/pre-commit` Phase 4 runs `git diff --quiet plugins/$p/`; unstaged mirror changes block the commit with "dual-source drift detected." Re-stage after `just build-plugin`.
- **`${CLAUDE_SKILL_DIR}` semantics (corrected per critical review)**: per Anthropic's Claude Code skills doc, `${CLAUDE_SKILL_DIR}` is a string substitution applied to the *invoked* skill's SKILL.md content at skill-invocation time. The substitution resolves to the directory containing the invoked skill's SKILL.md and has one binding per invocation. The substitution is **not** re-evaluated when Claude reads other files via the Read tool — file content comes back as raw bytes, and any `${CLAUDE_SKILL_DIR}` token inside such content is a literal string in Claude's context. This means: (a) R3 is correctly placed in refine's SKILL.md, where the substitution is guaranteed to resolve to refine's directory at invocation time; (b) the originally-proposed M1 edit to lifecycle/specify.md:145 was dropped because `${CLAUDE_SKILL_DIR}` in raw read-text would resolve to the invoking skill's directory (refine's), not the file's containing directory — the opposite of the desired behavior.
- **Anthropic "one level deep" guidance trade-off**: pointing refine at lifecycle's specify.md (which itself references orchestrator-review.md) is a documented two-hop pattern. Accepted as a deliberate cost; R8b's manual smoke covers the runtime path. If `head -100` truncation behavior is observed on lifecycle/orchestrator-review.md (lines 100+ contain S1–S6 Post-Specify Checklist), the mitigation is to add a TOC at the top of that file in a follow-up ticket — not in scope here.
- **Refine never reaches plan phase**: refine SKILL.md Steps 1–6 stop at user approval of the spec. The lifecycle copy of orchestrator-review.md contains a P8 plan-phase row that is harmless when reached because refine never invokes plan-phase orchestrator-review. This invariant is not programmatically enforced.
- **Refine standalone does not emit `lifecycle_start`**: only `/cortex-core:lifecycle` writes that event. lifecycle/orchestrator-review.md's Applicability check (lines 6–9) reads tier and criticality from the most-recent `lifecycle_start` or `criticality_override` event and defaults criticality to `medium` when absent — but does not specify a tier default. Under standalone refine, the skip-rule evaluation is undefined when no `lifecycle_start` exists. R8b acknowledges this and routes runtime verification to the next genuine refine invocation (which inherits its event-log context from `/cortex-core:lifecycle` or from the operator's manual seeding) rather than a synthetic fixture.
