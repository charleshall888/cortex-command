The research settled on a phased rollout: land the small point fixes now, build a set of composed wrapper verbs and generated pause data as a self-contained middle layer next, and treat the full CLI-served next/advance loop with its protocol handshake as a separate decision to be made only after living with that middle layer's evidence.

The main alternative on the table was funding the full loop immediately, including the protocol handshake, fragment-distribution flavor, and pause-runtime enforcement — that was dropped because a full sweep of incident history found zero real cases of the model actually misrouting the lifecycle sequence, so the failure the loop guards against isn't actually happening yet; the wrapper-verb layer already fixes the failures that do show up in the record, like prose drifting from code and event-emission ordering mistakes, at roughly half to a third of the cost. Delivery mechanisms other than a plain CLI were also weighed and dropped: an MCP server and a hooks-only approach both scored lower because background workers can't reach plugin MCP servers or hooks at all, so CLI is the only surface every execution mode shares. Whether fragments get served by the wheel or just selected from the plugin was left open, deferred to whenever the full loop actually gets funded.

The tradeoff being accepted now is that the sequencing guarantee everyone actually wants — a machine that structurally prevents wrong-phase routing — isn't being built yet. What ships is drift-proofing and event-ordering fixes, but the model still navigates phase-to-phase by convention rather than by an enforced state machine, so the specific failure mode motivating this whole effort remains structurally unguarded until a second decision is made later, based on whether the simpler layer turns out to be insufficient in practice.
