# Gap 1: Plugin-install feasibility

## Defense of current wheel-owns-writes architecture

ADR-0002 (`cortex/adr/0002-cli-wheel-plus-plugin-distribution.md:7`) is explicit about *why* there are two channels but silent about *which* channel owns which write. Its stated rationale is independent cadence: "the CLI and plugins evolve at independent cadences (plugin marketplace updates land on different schedules than wheel releases), and a narrow versioned envelope lets each side ship without lockstep coordination." It rejected the prior symlink-into-`~/.claude/` deployment for tying installs to working-tree checkouts and conflicting with Claude Code's plugin model. It does **not** state that writes must originate from the wheel; that allocation is downstream of two technical facts that show up only in ADR-0003 and in the plugin-system docs themselves.

**Technical fact 1: install-time write capability lives in the wheel, not the plugin.** Claude Code's plugin system documents no install-time hook. [`code.claude.com/docs/en/plugins`](https://code.claude.com/docs/en/plugins) (fetched 2026-05-29) lists eight component directories and zero install-time execution surfaces. The `Setup` hook event in plugins-reference fires **"when you start Claude Code with `--init-only`, or with `--init` or `--maintenance` in `-p` mode. For one-time preparation in CI or scripts"** ([plugins-reference, fetched 2026-05-29](https://code.claude.com/docs/en/plugins-reference)) — a runtime mode, not a post-`/plugin install` callback. The `/plugin install` command, per [`code.claude.com/docs/en/discover-plugins`](https://code.claude.com/docs/en/discover-plugins) (fetched 2026-05-29), only clones to `~/.claude/plugins/cache/` and writes a reference into the chosen scope's settings file under `enabledPlugins`. The "Will install" section lists **"the plugin's commands, agents, skills, hooks, and MCP and LSP servers"** before confirming — no settings.local.json modifications, no allowWrite mutations, no arbitrary scripts.

**Technical fact 2: the wheel is the only surface with shell access at first contact.** A first-time adopter following `docs/setup.md:29` runs `uv tool install git+...@<tag>` in a shell. That shell is not sandboxed and has the user's full filesystem authority. Per [`code.claude.com/docs/en/sandboxing`](https://code.claude.com/docs/en/sandboxing) (fetched 2026-05-29), in-session Bash is sandboxed and **"the sandbox automatically denies write access to Claude Code's settings.json files at every scope and to the managed settings directory, so a sandboxed command cannot modify its own policy."** A wheel-resident `cortex init` invoked from the terminal sidesteps that gate entirely because it runs in the user's shell, not Claude's Bash tool.

**What the split protects.** (a) *Bootstrap-from-cold-start*: a user with no Claude Code session running can install the wheel and run `cortex init` before opening Claude Code; the plugin route requires Claude Code already running. (b) *Per-repo additivity*: ADR-0003 frames the carve-out as additively registering the current repo's umbrella path with `fcntl.flock` serialization (`cortex_command/init/settings_merge.py:136-205`). The granularity is the repo; `/plugin install` is per-user/project/local-scope, not per-repo across many checkouts. (c) *Auditability of the actor*: a user grep'ing `~/.claude/settings.local.json` can trace `"/<repo>/cortex/"` to a discrete `cortex init` in shell history; a plugin SessionStart hook firing on every launch obscures that.

## Plugin-install alternative: technical analysis

### End-to-end flow

If all four host-scope writes (settings.local.json allowWrite entry, `.gitignore` lines, CLAUDE.md fence, `cortex/` scaffold + `.cortex-init` marker) moved to the plugin: user opens Claude Code, runs `/plugin install cortex-core@<marketplace>`, Claude Code clones to `~/.claude/plugins/cache/` and writes the reference to `enabledPlugins` in the chosen scope. **No further writes occur at install.** The plugin's `SessionStart` hook (`plugins/cortex-core/hooks/hooks.json:3-12`) fires on the next session and would need to: (a) detect whether cwd is a cortex-shaped repo or candidate, (b) check whether settings.local.json already has the umbrella entry, (c) if not, write the four artifacts.

### API surface

Plugin SessionStart hooks **can** write to settings.local.json in principle: the sandbox restriction in [`code.claude.com/docs/en/sandboxing`](https://code.claude.com/docs/en/sandboxing) applies to Bash *tool* subprocesses; plugin hooks run with user permissions per [`code.claude.com/docs/en/hooks`](https://code.claude.com/docs/en/hooks) — see also `plugins/cortex-core/hooks/cortex-session-start-path-bootstrap.sh:33` which already writes to `$CLAUDE_ENV_FILE` from a SessionStart context. Superpowers' SessionStart hook (`obra/superpowers/hooks/session-start`, fetched 2026-05-29) demonstrates the shell-script-with-user-permissions pattern but writes nothing outside stdout. The **mechanism is available** even if no surveyed plugin uses it for host-state writes.

### "Will install" preview coverage

NOT_FOUND(query="plugin install preview displays settings.local.json modifications", scope=`code.claude.com/docs/en/discover-plugins`+`code.claude.com/docs/en/plugin-marketplaces`). The Discover-tab "Will install" preview explicitly enumerates **"commands, agents, skills, hooks, and MCP and LSP servers"** (per discover-plugins, v2.1.145+). It does not advertise settings.local.json modifications, allowWrite mutations, `.gitignore` appends, or CLAUDE.md edits because plugins are not, in the documented install path, supposed to perform those writes. A plugin SessionStart hook that performed them would technically be visible in the "hooks" line of the preview (the hook command path), but the *consequences* would not be disclosed at install time. This is a meaningful UX gap: install-time consent would cover "this plugin defines a SessionStart hook" without telling the user the hook will mutate `~/.claude/settings.local.json` on first run.

### Per-repo dimension

`/plugin install` registers a plugin once at user/project/local scope. The cortex allowWrite entry is keyed on **repo path**, and a user with five cortex-enabled checkouts needs five distinct entries that must coexist. The current wheel approach runs `cortex init` once per repo, each invocation appending one umbrella entry. A SessionStart hook could replicate per-repo behavior by detecting cwd on each session — the existing cortex-shape gate at `cortex-session-start-path-bootstrap.sh:22-25` shows the pattern. But that hook gates on `cortex/lifecycle/` existing, which won't be true in a fresh-clone scenario where init hasn't run. The hook would need to (a) gate differently for first-touch repos, (b) probe settings.local.json for the entry, (c) acquire the same `fcntl.flock` discipline `settings_merge.register` uses (`settings_merge.py:181-205`) — either reimplemented in shell or shelled out to a wheel-provided console script. The latter keeps the wheel; the former duplicates locking logic in plugin shell.

### Per-repo writes (CLAUDE.md fence, `.gitignore`, `cortex/` scaffold, `.cortex-init` marker)

These repo-local writes depend on repo identity (per-repo by construction) and happen *inside* the working tree, where the sandbox permits writes (cwd default-allow). A SessionStart hook detecting "new cortex-bootstrap candidate" could perform them — but must decide *when* it's appropriate. The wheel's `handler.py:498-501` uses `check_content_decline` (R19) to refuse adoption of a non-empty repo where the user might not want cortex artifacts. A SessionStart hook bootstrapping on every fresh repo without explicit `/lifecycle` invocation would silently mutate working trees opened for unrelated reasons. The cleaner shape is invoking these writes from the lifecycle skill's first call (what backlog #267's `cortex-lifecycle-init-ensure` was designed for, see `cortex_command/lifecycle/init_ensure.py`). The bottleneck the original research surfaced is that helper today *also* attempts host-scope writes, creating the EPERM + `dangerouslyDisableSandbox` retry loop. Splitting host-scope off (leaving repo-scope in the helper) is what Approach D proposed (`research.md:121`).

### Marker file, `.gitignore`, CLAUDE.md fence at /lifecycle invocation time

The strongest part of the plugin-only alternative: repo-local artifacts could be written by a plugin-internal helper invoked from the lifecycle skill's first call, no wheel involvement, because the cortex/ tree is inside the sandbox's default-allow cwd region. Edit-tool writes to repo-root files are not sandbox-blocked.

What this alternative **cannot do** without wheel collaboration is the settings.local.json allowWrite entry — outside both the sandbox-allowed region and the repo. Two options:

- **(a) Plugin SessionStart hook writes unsandboxed.** Hooks run with user permissions. Mechanically possible. Cost: opacity — the "Will install" preview does not advertise the write; the user sees the modification only by post-hoc inspection.
- **(b) Plugin shells out to wheel-provided `cortex` console script.** This is what `cortex-worktree-create.sh:34-37` already does for `cortex-worktree-resolve`. Converges architecture back to "wheel writes host scope," just from a different entry point. Depends on wheel being installed; `docs/setup.md:40` already states the cortex-overnight MCP auto-installs the CLI on first tool call, so wheel-presence is already assumed.

Option (b) preserves the write surface but shifts the trigger to first session in a cortex-marked repo. It does not eliminate the wheel-as-write-actor; it changes who *calls* it.

### Verified vs NOT_FOUND

- **Verified**: Plugin install writes only to `enabledPlugins` in the chosen scope's settings file [discover-plugins].
- **Verified**: SessionStart hooks run unsandboxed with user permissions [hooks, sandboxing].
- **Verified**: Plugin SessionStart hooks can run arbitrary shell commands referenced via `${CLAUDE_PLUGIN_ROOT}` [plugins-reference, demonstrated in `plugins/cortex-core/hooks/hooks.json:7-9`].
- **NOT_FOUND**(query="documented mechanism for plugin install-time settings.local.json modification with user consent preview", scope=`code.claude.com/docs/en/discover-plugins`).
- **NOT_FOUND**(query="plugin marketplace entry declares install-time file writes", scope=`code.claude.com/docs/en/plugin-marketplaces` persisted output).
- **Unclear**: whether a SessionStart hook's settings.local.json write would race with Claude Code's own settings-reload-on-file-change behavior described in `research.md:76`. The wheel uses `fcntl.flock` for serialization (`settings_merge.py:182-205`); a shell hook would need to do likewise or shell out to a wheel-provided locked writer.

## Comparator: how comparable plugins handle host-state

### Surveyed plugins

- **Superpowers** (`obra/superpowers`, plugin.json v5.1.0, fetched 2026-05-29): SessionStart hook present at `obra/superpowers/hooks/session-start`. The hook reads the plugin's `using-superpowers/SKILL.md` and emits it as `additionalContext` via stdout JSON. **It writes nothing to `~/.claude/settings.local.json` or any user-scope file.** The only side-channel is a warning emitted via `additionalContext` if a legacy `~/.config/superpowers/skills` directory exists. Pattern: context injection only.
- **cortex-core** (this repo, `plugins/cortex-core/hooks/cortex-session-start-path-bootstrap.sh`): SessionStart hook gates on `cortex/lifecycle/` existing in cwd, then emits a PATH augmentation via `$CLAUDE_ENV_FILE`. **It writes nothing outside `$CLAUDE_ENV_FILE` (the session-scoped env file Claude Code itself provides).** Pattern: env-var augmentation only.
- **Community marketplace plugins** ([`anthropics/claude-plugins-community/marketplace.json`](https://github.com/anthropics/claude-plugins-community/blob/main/.claude-plugin/marketplace.json), surveyed via WebFetch 2026-05-29): plugins needing per-project state (alfred, anson, ai-coding-ok, agentic-swe, aldc) **all use command-driven initialization** (`/alfred:bootstrap`, `/install`, `/aldc:al-initialize`). The user explicitly runs a slash command after install to bootstrap project state. None modify user-scope settings from install hooks.
- **ccstatusline** (`sirmalloc/ccstatusline`, surveyed via web 2026-05-29): npm-installed CLI, not a plugin. Configures itself by writing `statusLine` to `.claude/settings.json` via a TUI invoked by the user. **Does not modify `sandbox.filesystem.allowWrite`.** Pattern: out-of-band CLI + interactive TUI prompt, similar in shape to the current `cortex init`.
- **`claude mcp add`** (first-party, [`code.claude.com/docs/en/mcp`](https://code.claude.com/docs/en/mcp), cited from `comparator.md:15`): writes user/project/local-scope MCP server config when the user explicitly invokes it with `--scope`. Pattern: explicit user verb + scope selection.

### Patterns identified

- **Pattern A (dominant in plugins): host-state stays read-only at install time.** The plugin defines hooks and skills; users invoke slash commands to perform any project-state setup. Settings modifications are user-initiated through the `/plugin` UI or `claude mcp add`, not plugin-initiated.
- **Pattern B (out-of-band CLI + interactive prompt): ccstatusline.** The user installs an npm CLI, runs the TUI, the TUI prompts and confirms the settings write. This is structurally what current `cortex init` already does, minus the prompt.
- **Pattern C (companion CLI for elevated writes): cortex-overnight + cortex wheel.** The plugin (`plugins/cortex-overnight/server.py`) shells out to a wheel-provided `cortex` console script that performs operations the plugin itself cannot. This is the hybrid the prior comparator survey (`comparator.md:33-37`) identified as cortex's current de facto pattern.

The clean rule that emerges across surveyed instances: **`/plugin install` owns plugin-cache + `enabledPlugins` writes; everything else is either a separate user-invoked verb (CLI, slash command, TUI) or doesn't happen.** No surveyed plugin uses install-time or first-SessionStart hooks to mutate `~/.claude/settings.local.json`'s sandbox arrays.

The hybrid plugin + companion CLI pattern (Pattern C) is already validated in production by cortex itself (overnight MCP shells out to wheel) and by precedent (Superpowers ships as plugin only; ccstatusline ships as CLI only). Cortex's two-channel design with the wheel doing host writes is an instance of Pattern C, not a deviation from convention.

## Synthesis: feasibility verdict

**Technical possibility per write:**

| Write | Wheel-only (today) | Plugin-only | Hybrid (plugin SessionStart calls wheel) |
|---|---|---|---|
| `~/.claude/settings.local.json` allowWrite entry | Verified working (`settings_merge.py:136-205`) | Mechanically possible via SessionStart hook + shell-implemented flock; **unclear-need-Anthropic-input** on race with Claude Code's settings-reload-on-change | Verified working pattern (existing `cortex-worktree-create.sh:34-37`) |
| Repo `.gitignore` lines | Verified working (`scaffold.py:71-75`) | Technically possible from plugin SessionStart hook or from lifecycle-skill Edit-tool call (cwd-relative, sandbox-allowed) | Verified working |
| Repo CLAUDE.md fence | Verified working (`scaffold.py:678-757`) | Same as `.gitignore` | Verified working |
| `cortex/` scaffold + marker | Verified working (`scaffold.py:312-365`) | Same as above | Verified working |

**Verdict by question:**

- *"Could the writes move to the plugin's install handler?"* — **Technically impossible as stated.** Plugin install does not have a handler (no `postInstall`, no install-time code-execution surface per [`code.claude.com/docs/en/plugins`](https://code.claude.com/docs/en/plugins) and [`code.claude.com/docs/en/plugins-reference`](https://code.claude.com/docs/en/plugins-reference)). The closest surrogate is a SessionStart hook that fires on the next session start after install.
- *"Could the writes move to a plugin SessionStart hook?"* — **Technically possible but architecturally regressive on three axes:** (a) **consent UX**: the "Will install" preview does not advertise settings.local.json modifications, so the actual write surface is hidden behind an opaque "hooks: SessionStart" line in the preview, which is *worse* opacity than the prior research's already-flagged complaint about cortex init being silent; (b) **per-repo timing**: the SessionStart hook would need to decide on every session start in every repo whether to write, and to gate on "is this a cortex-candidate repo" without prematurely adopting random working trees; (c) **bootstrap precondition**: the user must already have Claude Code running and have run `/plugin install`, which means cold-start automation (CI, scripted installs that today rely on `uv tool install` + `cortex init`) loses its non-interactive path.
- *"Could the writes move to the lifecycle skill's first invocation, via a plugin-internal helper?"* — **Partially.** Repo-local writes (`.gitignore`, CLAUDE.md fence, `cortex/` scaffold, `.cortex-init` marker) can move to the lifecycle skill's first call without wheel involvement, because they fall inside the sandbox's default-allow cwd region. The `~/.claude/settings.local.json` allowWrite entry cannot, because (i) the file is outside the sandbox-allowed region and (ii) the sandbox docs are explicit that "sandboxed command cannot modify its own policy." This is the same finding the parent research already documented at `research.md:73-74,162`; this gap analysis confirms that finding holds independently for the plugin route.
- *"Does this eliminate the need for the wheel's `cortex init`?"* — **No, not for the host-scope write.** The wheel's `settings_merge.register` (with its `fcntl.flock` discipline and JSON shape validation at `settings_merge.py:182-205`) is the load-bearing piece. The wheel could be invoked from somewhere other than terminal-init (e.g., from a plugin hook shelling out, as `cortex-worktree-create.sh:34-37` already does for the worktree-resolve helper), but the wheel-as-author of the host write remains structurally necessary unless Anthropic ships an install-time consent surface that handles allowWrite array entries on the plugin's behalf.

**Where this leaves the original framing.** The critical review's premise — "route host-scope writes through `/plugin install`" — does not survive contact with the plugin-system documentation. `/plugin install` is a clone-and-register operation, not a write-to-host-scope operation; there is no install handler to route writes through. What *is* possible is routing host-scope writes through a plugin SessionStart hook, which is a different proposal with different tradeoffs (opaque to install-time consent, fires repeatedly, must shell out for sandbox-clean settings writes). The cleanest version of that alternative converges to the existing Pattern C (plugin shells out to companion CLI) — which is approximately what cortex already is.

The original research's Approach D bootstrap-flaw critique stands in its narrow form (an in-session ensure path *also* attempting host-scope writes contradicts the sandbox model). The plugin-install rerouting alternative does not dissolve that critique; it relocates the same write to a different unsandboxed entry point (a hook) with weaker consent surfacing than terminal init.

## Open questions

1. **Could Anthropic add an install-time consent surface for arbitrary settings modifications?** The "Will install" preview currently enumerates plugin components but not their side effects. If marketplace.json could declare `"installEffects": {"sandbox.filesystem.allowWrite": ["<repo>/cortex/"]}` and have the install UI display + consent that, the plugin route would become viable. This is a feature request, not an architectural option available today. NOT_FOUND(query="plugin marketplace install effects declaration", scope=`code.claude.com/docs/en/plugin-marketplaces`).

2. **What's the race condition between a SessionStart hook writing to settings.local.json and Claude Code's settings-reload behavior?** `research.md:76` notes Claude Code reloads on settings file change. If a SessionStart hook writes during session bootstrap, does Claude Code see the new settings for that session or only the next one? The shell-based hook can't easily replicate `fcntl.flock` semantics across concurrent Claude Code sessions in different repos.

3. **Would CI / scripted-install flows tolerate the plugin-only path?** Today users can run `uv tool install ... && cortex init <repo>` in non-interactive scripts. A plugin-only path requires interactive Claude Code at least once. NOT_FOUND(query="claude code plugin install headless CI", scope=web).

4. **What is the user-visible difference if a plugin SessionStart hook writes settings.local.json silently vs `cortex init` writing it silently?** Both are opaque writes. The prior research framed install-time opacity as the real UX failure; moving the write to a different opaque entry point doesn't fix it. Empirical test would be required.

5. **Is the "Will install" preview text actually enforceable as install-time consent for downstream side effects?** Per the docs the preview lists components, not effects. If a plugin's SessionStart hook writes to settings.local.json, the user has not explicitly consented to that write at install time. This is a policy question for Anthropic; NOT_FOUND(query="claude code plugin install transitive consent", scope=`code.claude.com`).

6. **Could the wheel ship inside the plugin as a vendored binary?** `bin/` is a documented plugin directory whose contents are added to the Bash tool's `PATH` while the plugin is enabled, per [`code.claude.com/docs/en/plugins`](https://code.claude.com/docs/en/plugins). In principle the `cortex` console script could be vendored into `plugins/cortex-core/bin/`. The wheel's actual Python package would still need to live somewhere importable; bundling a uv-managed venv inside a plugin cache directory has no documented support. NOT_FOUND(query="plugin ship python package vendored runtime", scope=`code.claude.com`).
