## Headline Finding

**Anthropic's June 15, 2026 policy split is real and documented**: Agent SDK (Python + TS) and `claude -p` usage move to a separate, capped monthly credit pool ($20 Pro / $100 Max 5x / $200 Max 20x) billed at full API rates; interactive Claude Code CLI usage stays on subscription quota unchanged ([source](https://code.claude.com/docs/en/agent-sdk/overview), [Help Center](https://support.claude.com/en/articles/15036540-use-the-claude-agent-sdk-with-your-claude-plan)). The just-landed `#214` work delivers OAuth-token auth for the SDK (`CLAUDE_CODE_OAUTH_TOKEN`) but does **not** close the gap — per Anthropic, OAuth-authed SDK calls go to the new SDK credit pool, not subscription quota. **Recommended approach**: build the interactive overnight mode as **Architecture B (external orchestrator + in-session worker pool)** — keep the existing Python runner (which preserves the orchestrator-layer primitives: state, `runner.pid` IPC + cancel, atomic writes, dashboard read, LaunchAgent scheduling, pre-install in-flight guard), and replace only the two SDK transport surfaces (`[cortex_command/pipeline/dispatch.py:22-37]` and `[cortex_command/overnight/runner.py:1028-1049]`) with a bridge that routes agent calls onto an in-session worker pool dispatching via the Task tool (the only in-session primitive plausibly subscription-billed; see Q1 caveat). **B does NOT preserve sandbox-settings-per-spawn**: Seatbelt profiles attach at process exec time, the in-session worker is post-exec, so per-feature sandbox isolation is weakened to per-session for any bridge-routed dispatch (see Q6). **B does NOT extend process-group watchdog protection to the bridge worker** — the worker is in-session, not a Python subprocess, so it requires new bridge-layer liveness primitives (idempotency-keyed dispatch ledger, heartbeat, orchestrator-side timeout, auto-respawn — see piece 1 below and Q7). **Recommendation is conditioned on two binary invalidators**: (i) Q1 — Task-tool dispatches must empirically draw from subscription quota post-June-15; piece 1 ships a small empirical probe as its first acceptance criterion before significant build (see DR-6); (ii) Q2 — compaction-survival must be achievable without the unmerged `PreCompact` hook via a stateless-restartable worker design (see DR-5). SDK overnight stays as the unattended-host default; interactive overnight is the operator-attached, subscription-billed, opt-in mode. Architecture A (in-session orchestrator as a Skill loop) is rejected — breaks 7+ `pipeline.md` primitives at the orchestrator layer and concentrates the whole batch into one fault domain. Architecture C (hybrid routing — Opus / high-effort dispatches through bridge, cheap dispatches on SDK) is the recommended **rollout strategy within B**: ship the bridge serving the most expensive dispatches first, then expand. The sharpest unresolved risk is auto-compaction firing during an in-flight bridge dispatch; the proposed mitigation (stateless-restartable worker + idempotency key + orchestrator-side timeout/respawn) is feasible without `PreCompact` but is decompose-time-sensitive — see DR-5.

## Research Questions

1. **What is Anthropic's pricing change — date, scope, mechanism, tier impact?** → **Confirmed**. Effective **June 15, 2026** ([source](https://code.claude.com/docs/en/agent-sdk/overview)). Reclassified: `claude-agent-sdk` (Python + TS), `claude -p` CLI subprocess, Claude Code GitHub Actions, third-party agents using Agent SDK auth. Unchanged: interactive Claude Code REPL (subscription quota), direct `anthropic` SDK with API keys (already pay-as-you-go). Mechanism: **auth-token-derived** — `sk-ant-oat01-` OAuth tokens (subscription accounts) draw from a new separate monthly Agent SDK credit pool; `sk-ant-api03-` API keys continue pay-as-you-go ([Help Center](https://support.claude.com/en/articles/15036540-use-the-claude-agent-sdk-with-your-claude-plan), [DEV walkthrough](https://dev.to/aviv_shaked/how-to-use-your-claude-promax-subscription-with-the-agent-sdk-python-typescript-4emi)). Tier credits: Pro $20/mo, Max 5x $100/mo, Max 20x $200/mo, Team Premium $100/seat/mo, Enterprise Premium $200/mo (non-rolling).

2. **Which overnight call sites generate the bulk of token spend?** → **Structurally known, quantitatively unmeasured.** Two SDK call surfaces: the `claude_agent_sdk` import at `[cortex_command/pipeline/dispatch.py:22-37]` (wrapped by `dispatch_task`, called from conflict/retry/review/merge-recovery/integration-recovery/brain) and the `claude -p` Popen at `[cortex_command/overnight/runner.py:1028-1049]` (orchestrator-round driver; spawns an Opus plan-synthesizer Task subagent per critical feature). Per-dispatch cost is captured (`cost_usd` from `ResultMessage.total_cost_usd` at `[dispatch.py:750-757]`) but only persisted to per-feature `pipeline-events.log` files which are gitignored (`NOT_FOUND(query="cost_usd", scope="cortex/lifecycle/sessions/*")`). Orchestrator-round cost is in `stdout.json` envelopes but `[overnight/report.py:_aggregate_feature_cost]` never parses them. Structural priority order: orchestrator-round, per-feature implement on Opus tier (`complex/high`, `complex/critical` via the dispatch matrix at `[dispatch.py:127-148]`), review + review-fix on Opus, integration-recovery (hard-coded Opus + `effort=max` at `[integration_recovery.py:215-226]`), conflict-repair / merge-test-repair Opus-escalation arms.

3. **Among architecture candidates A/B/C, what's the recommended v1?** → **B, with C as the rollout strategy.** **A** (long-lived in-session orchestrator as Skill loop) breaks `pipeline.md` primitives broadly: process-group watchdog kills `[runner.py:271, 705-708]` lose their PID target; per-spawn sandbox-settings tempfiles `[dispatch.py:573-602]` can't attach to Task-tool dispatches (the `settings=` filepath is a `ClaudeAgentOptions` field, not a Task parameter); `runner.pid` IPC + cancel-via-signal `[ipc.py:243-353]` has no PID; pre-install in-flight guard breaks; `cortex overnight cancel --force` (SIGKILL the pgid) is meaningless. **B** (external Python orchestrator + in-session worker pool reached via bridge) preserves every `pipeline.md` primitive because the orchestrator stays put; only the two SDK transport surfaces change. **C** (hybrid: route only Opus / high-effort dispatches through the bridge, keep cheap on SDK) is structurally a routing branch inside `[dispatch.py:644-656]` that depends on the bridge from B existing; it's a rollout strategy within B, not a separate architecture — ship the bridge serving Opus / `effort=xhigh` dispatches first (the biggest spend), expand to all dispatches once stable.

4. **Session-keepalive mechanics under stated operator setup?** → **Viable but with three sharp risks.** `caffeinate -d -i -s -u` stacks four IOPMAssertions (display/idle/system/user-active); `-s` (system sleep) silently degrades on battery (no-op when not on AC). Lock-screen (Cmd+Ctrl+Q) does not stop the CPU scheduler — terminal processes keep running. **tmux is required** to survive accidental terminal close (Terminal.app/iTerm/Ghostty SIGHUP); standard pattern is `tmux new -d -s cortex` + detach. Caveats: Claude Code has open tmux issues — bracketed-paste corruption with `extended-keys-format=csi-u` ([#43169](https://github.com/anthropics/claude-code/issues/43169)), bracketed-paste mode not disabled on exit ([#3134](https://github.com/anthropics/claude-code/issues/3134)), ANSI escape leakage ([#32632](https://github.com/anthropics/claude-code/issues/32632)) — mostly harmless for hands-off batch but can leave a corrupted terminal state for the operator's resume session. Power-source change: `pmset -g pslog` streams transitions; no first-class "you got unplugged" callback — poll `pmset -g batt` every N seconds inside the orchestrator. Linux equivalent: `systemd-inhibit --what=idle:sleep --mode=block`, `xset s off`, `upower --monitor` for AC/battery.

5. **Inside a Claude Code interactive session, which primitives actually stay on subscription?** → **Task tool (subagent dispatch) and Skill tool are the strongest candidates** but the billing boundary is `[premise-unverified: not-searched-deeply]` (Anthropic docs do not enumerate per-tool billing; the policy split is documented at the SDK/CLI level and the in-session tools are not in scope). The inference is one-way: the policy *names* SDK/CLI as reclassified and is *silent* on in-session tools, which is consistent with Task-tool dispatches staying on subscription but does not prove it. The recommended architecture's load-bearing premise is "Task-tool dispatches are subscription-billed"; **this is binary and unverifiable until post-June-15** — DR-6 records the resulting probe-first sequencing. By contrast: **Bash tool spawning `claude -p` — LIKELY SDK BUDGET** (subprocess re-auths fresh and is exactly the type of headless usage the new policy reclassifies — same logic should make us cautious about Task tool's transport, see DR-3 caveat); **MCP server calls with internal SDK invocations — SDK BUDGET** (MCP server subprocess auths independently). Session duration: no documented hard cap; auto-compaction at ~83.5% context fill is the binding constraint and the documented multi-agent failure mode is real ([#11280](https://github.com/anthropics/claude-code/issues/11280), [#25999](https://github.com/anthropics/claude-code/issues/25999), [#43733](https://github.com/anthropics/claude-code/issues/43733) — `PreCompact` hook is still an open feature request; DR-5 records the stateless-restartable fallback). Quotas: Pro/Max have rolling 5-hour window + weekly cap; no pre-flight "tokens-left" API.

6. **Which `pipeline.md` acceptance criteria break under the recommended architecture?** → **Architecture B preserves the orchestrator-layer primitives; it weakens or replaces the agent-execution-layer primitives.** Orchestrator-layer (preserved unchanged because the Python runner stays put): atomic state writes, JSONL events.log, `runner.pid` IPC + cancel-via-signal, active-session pointer, pre-install in-flight guard, dashboard read, LaunchAgent scheduling, the recovery/retry control-flow loops. Agent-execution-layer (weakened or replaced): **process-group watchdog protection** — preserved at the orchestrator level but NOT extended to the in-session worker (the worker is not a Python subprocess; watchdog cannot see it; bridge-layer liveness primitives required, see Q7); **sandbox-settings-per-spawn** — weakened to per-session (Task tool has no `settings=` parameter; the in-session worker shares its parent session's Seatbelt profile across all bridge-routed dispatches; cross-feature isolation downgraded, see Q6); **`DispatchResult.error_type` granularity** — depends on the bridge worker marshaling Task-tool free-form output back into the structured shape via sentinel + JSON envelope discipline (see piece 1 IPC contract). **Architecture A breaks more aggressively** (and at the orchestrator layer, which is why B is preferred): watchdog kills (no PID), runner.pid IPC, cancel-via-signal, pre-install in-flight guard, `cortex overnight cancel --force`, plus the same sandbox + watchdog-of-work weakening B has. **Compaction-survival is a NEW constraint for B's worker host** (not in `pipeline.md` today): the in-session worker must be designed stateless-restartable — dispatches carry idempotency keys, the worker's only persistent state is the in-flight dispatch ledger written before each Task spawn — so a post-compaction or session-restart resume reattaches to (or no-ops on) in-flight work without #11280-style duplicate spawns. See DR-5.

7. **What gap remains after #208 + #214?** → **The transport-layer gap remains.** #208 made the autonomous worktree launchable from a Claude Code Bash tool (verify_dispatch_readiness fuse, auth-vector convergence with Keychain probe, sandbox-friendly worktree-root resolver registered via `cortex init`, console-script promotion of ~13 modules). #214 added `cortex auth bootstrap` (runs `claude setup-token`, writes `~/.claude/personal-oauth-token`) so subscription-only users can authenticate the SDK. **But per Anthropic's policy, OAuth-token-authed SDK calls draw from the new SDK credit pool, not subscription quota.** #208 + #214 together solved "the autonomous dispatch path works for subscription users" — they did not solve "the agent calls land on subscription quota." That gap requires re-hosting agent invocation onto in-session primitives (Task tool), which is what Architecture B (or A) would do. The work in those tickets is reusable: the OAuth-token-aware auth chain still gates whether the SDK path works at all, and the daytime-pipeline entrypoint is the existing fall-through if the bridge fails or the operator selects SDK mode.

## Codebase Analysis

**SDK transport surfaces** (exhaustive — only two structural call sites in production code):

- `[cortex_command/pipeline/dispatch.py:22-37]` — sole `claude_agent_sdk` import (`query`, `ClaudeAgentOptions`, message types, error classes). Every `dispatch_task` call routes here. Indirect callers: `[cortex_command/pipeline/conflict.py:349]`, `[cortex_command/pipeline/retry.py:256]`, `[cortex_command/pipeline/review_dispatch.py:253, 385, 500]`, `[cortex_command/pipeline/merge_recovery.py:332]`, `[cortex_command/overnight/integration_recovery.py:216]`, `[cortex_command/overnight/brain.py:225]`.
- `[cortex_command/overnight/runner.py:1028-1049]` — `subprocess.Popen(["claude", "-p", filled_prompt, "--settings", ..., "--dangerously-skip-permissions", "--max-turns", ..., "--output-format=json"])`. Per-round orchestrator driver. NOT `claude_agent_sdk` — it's a direct `claude -p` subprocess. Spawns an Opus plan-synthesizer Task subagent per critical feature per `[overnight/prompts/orchestrator-round.md:300]`.

**Model & effort routing** lives at `[cortex_command/pipeline/dispatch.py:127-148]` (the `(complexity, criticality)` matrix). Opus + `effort=xhigh` cells (complex/high, complex/critical) plus the `effort=max` overrides for review-fix `[review_dispatch.py:385-396]` and integration-recovery `[integration_recovery.py:215-226]` are the per-call cost peaks.

**Cost emission** at `[cortex_command/pipeline/dispatch.py:750-757]` writes `cost_usd` from `ResultMessage.total_cost_usd` to per-feature `pipeline-events.log` (gitignored, transient). `NOT_FOUND(query="cost_usd|token_usage|input_tokens|output_tokens", scope="cortex/lifecycle/sessions/*")` — session dirs hold only `bin-invocations.jsonl` and `sandbox-deny-lists/`. The single surviving `pipeline-events.log` in archive shows 3 `dispatch_complete` events totalling $12.52 ($4.17 mean, $5.17 max, $2.71 min) for Sonnet/complex/medium. Orchestrator-round `claude -p` cost in `stdout.json` envelopes is never parsed (`[overnight/report.py:_aggregate_feature_cost]` only reads `dispatch_complete` events from pipeline-events.log).

**MCP control plane** at `[plugins/cortex-overnight/server.py:2861-2960]` exposes five stateless tools (`overnight_start_run`, `overnight_status`, `overnight_logs`, `overnight_cancel`, `overnight_list_sessions`) that `subprocess.run` `cortex <verb> --format json` per the R1 invariant (zero `cortex_command.*` imports). Sufficient to launch/observe an external runner from in-session, **does NOT exchange agent traffic** — no MCP tool exists today that "accepts a prompt + options and returns model output." A bridge would need a sixth tool plus a side-band IPC (Unix socket / named pipe / file queue) for a long-lived bidirectional worker channel. The R17 confused-deputy guard at `[server.py:51-83]` (`CLAUDE_PLUGIN_ROOT` containment) is fine for control-plane delegation; a bridge worker would need its own startup integrity check. `NOT_FOUND(query="mcp tool that accepts agent prompt", scope=plugins/cortex-overnight/server.py)`.

**Task-tool dispatch patterns in existing skills** (three idiomatic shapes worth reusing for the in-session worker):

- Parallel-then-synthesize: `[skills/critical-review/SKILL.md:55-77]` — 3-4 reviewer agents in one batch + Opus synthesizer. Sentinel-first verification (`READ_OK: <path> <sha>` first line), last-occurrence JSON envelope extraction. Same shape: `[skills/research/SKILL.md:59-73]`, `[skills/lifecycle/references/plan.md:32-88]`.
- Worktree-isolated workers: `[skills/lifecycle/references/implement.md:188]` — `Agent(isolation: "worktree")` with `git log HEAD..worktree/{task-name} --oneline` as success check.
- Model-tier routing: `[skills/lifecycle/assets/model-selection.md:11]`, `[skills/lifecycle/references/orchestrator-review.md:55]` — orchestrator picks model per dispatch.

Aggregation in current skills is free-form (orchestrator agent parses sub-agent output strings). The bridge's worker host would need to marshal results back into the `DispatchResult` shape `[dispatch.py]` expects (including `error_type` classification — a structured surface that the current Task-tool patterns lack).

**#208 + #214 delivered vs. remaining gap**:
- #208 (spec at `cortex/lifecycle/harden-autonomous-dispatch-path-for-interactive/spec.md`): R18 single `verify_dispatch_readiness()` fuse; R1-R3 auth-vector convergence + `probe_keychain_presence()`; R6-R10 sandbox-friendly worktree-root resolver + `cortex init` registration; R12-R14 console-script promotion (~13 modules previously bare `python3 -m cortex_command.<x>`); R19-R21 parity-test matrix. Effect: launching `python3 -m cortex_command.overnight.daytime_pipeline` from inside a Claude Code session works.
- #214 (spec at `cortex/lifecycle/restore-subscription-auth-for-autonomous-worktree/spec.md`): new `cortex auth bootstrap` subcommand `[cortex_command/auth/bootstrap.py]` that runs `claude setup-token` and writes `~/.claude/personal-oauth-token`; `cortex auth status` diagnostic.
- **Gap**: the subprocess #208 + #214 enables is still `python3 -m cortex_command.overnight.daytime_pipeline`, which still calls `dispatch_task` → `claude_agent_sdk.query()`. The SDK forwards `CLAUDE_CODE_OAUTH_TOKEN` per `[dispatch.py:547-548]`, but per Anthropic, OAuth-authed SDK traffic goes to the new credit pool — **the transport-side enforcement is the policy, not the auth method**. The gap requires re-hosting agent invocation onto in-session Task-tool primitives.

## Architecture

**Topic shape**: Constructive architecture for the new interactive overnight mode, additive to the existing SDK overnight (which stays as the default unattended-host path). Four pieces.

### Pieces

1. **Subscription-routed agent transport** — external Python orchestrator's handoff API + in-session Skill worker host + IPC channel between them. **Worker design is stateless-restartable** (each dispatch carries an idempotency key generated orchestrator-side; the worker's only persistent state is an in-flight dispatch ledger written before each Task spawn) so a post-compaction or session-restart resume reattaches to (or no-ops on) in-flight work without #11280-style duplicate spawns — see DR-5. **IPC contract includes**: (a) idempotency key per dispatch; (b) worker-side heartbeat at fixed interval; (c) orchestrator-side timeout that triggers auto-respawn-and-resume (the orchestrator re-spawns the worker via the operator-presence script in piece 2, then re-issues the dispatch with the same idempotency key so the worker either returns the cached result or re-runs idempotently — see Q7); (d) sentinel-+-JSON-envelope marshaling so Task-tool free-form output round-trips into the `DispatchResult` shape with `error_type` classification preserved (pattern from `[skills/critical-review/SKILL.md:55-77]`). **First acceptance criterion (gated by DR-6)**: a Q1 empirical probe — a single Task-tool dispatch through the worker host under a metered subscription account, observed against Anthropic's documented credit-pool accounting post-June-15 — confirms subscription billing before significant code commits. Replaces the SDK transport at `[cortex_command/pipeline/dispatch.py:22-37]` and the `claude -p` Popen at `[cortex_command/overnight/runner.py:1028-1049]` when interactive mode is selected.
2. **Operator-presence operations** — `bin/cortex-interactive-overnight-start` (or similar) that wraps `tmux new -d -s cortex 'caffeinate -dimsu claude ...'`, plus a power-source guard inside the orchestrator (poll `pmset -g batt`, pause on AC→Battery), plus a cancel handoff path that lets `cortex overnight cancel` reach an in-session worker (not just SIGKILL the pgid as today).
3. **Mode selection & routing** — operator-facing config (`overnight.mode = sdk | interactive | hybrid`) recorded in `overnight-state.json`, plus a per-dispatch routing branch inside `[cortex_command/pipeline/dispatch.py:644-656]` for hybrid mode that sends Opus/`effort=xhigh` dispatches through the bridge and keeps cheap dispatches on SDK. Hybrid is the recommended rollout path (ship bridge for the most expensive dispatches first, expand).
4. **`pipeline.md` amendments + parity tests** — codify which acceptance criteria apply in interactive mode. Explicit mode-conditional clauses for: (a) per-spawn sandbox-settings → per-session (operator-accepted relaxation; cross-feature isolation downgraded — see Q6); (b) watchdog-of-the-agent-work replaced by bridge-layer heartbeat/timeout/respawn (orchestrator-layer watchdog still active — see piece 1 IPC contract + Q7); (c) JSONL events.log append-discipline → flock-locked (orchestrator still owns the writer; lock guards against bridge-side concurrent writes); (d) `DispatchResult.error_type` granularity → preserved via sentinel + JSON envelope marshaling in the worker. Add `tests/test_interactive_mode_pipeline_parity.py` that walks the mode-conditional clauses and verifies the implementation matches each one.

### Integration shape

Piece 1 is the load-bearing transport — pieces 2 and 3 only have meaning if the transport exists. Piece 3 (mode selection) is the operator entry point that selects which transport applies per-session; piece 2 (operator ops) is what the operator runs once `mode = interactive` is selected.

Piece 4 documents the resulting two-mode system; it is the contract surface between the runtime and the operator's understanding of what each mode guarantees, and the parity tests are the drift guard that keeps the two from diverging silently.

Rollout sequencing (within piece 3's hybrid routing): the bridge ships first wired only to the Opus / `effort=xhigh` dispatch cells in the `[dispatch.py:127-148]` matrix (`complex/high`, `complex/critical`, `integration_recovery`, review-fix on Opus, conflict/merge-repair Opus escalations). Cheap dispatches keep going through SDK. Once stable, the operator-facing config grows the `mode = interactive` value that routes everything through the bridge.

Named contract surfaces between pieces:
- (1) ⇄ (3): the routing branch at `[dispatch.py:644-656]` calls into the bridge handoff API for any dispatch the routing rule selects.
- (1) ⇄ (2): the cancel handoff path (in (2)) talks to the in-session worker (in (1)) via the same IPC channel the bridge uses for dispatch.
- (1) ⇄ (4): the relaxed `pipeline.md` acceptance criteria for the bridge worker's sandbox-settings + events.log behavior.
- (3) ⇄ (4): mode-conditional `pipeline.md` clauses describe what changes when `overnight.mode = interactive` (or when hybrid routing fires for a given dispatch).

### Seam-level edges

- Piece 1 ⇄ `[plugins/cortex-overnight/server.py:2861-2960]` (new MCP tool: `agent_dispatch` accepting prompt + options + idempotency key, returning structured result) + new file `plugins/cortex-overnight/agent-bridge.py` (Unix-socket worker-side listener) + new skill `skills/overnight-worker/SKILL.md` (in-session worker host that long-polls the bridge and Task-dispatches per request) + new file `cortex/lifecycle/sessions/{sid}/dispatch-ledger.jsonl` (append-only in-flight dispatch ledger keyed by idempotency key; the worker's only persistent state per DR-5). IPC contract details: (i) dispatch envelope includes idempotency-key (orchestrator-generated UUID), prompt, options, deadline-timestamp; (ii) worker emits a `dispatch_started` ledger entry before the Task spawn, a `dispatch_completed` entry on success with the marshaled `DispatchResult`, a `dispatch_failed` entry on error with `error_type` from the JSON envelope; (iii) worker emits a heartbeat every N seconds (default 30) via a separate IPC channel; (iv) orchestrator-side timeout = heartbeat-interval × 3 (configurable); on timeout the orchestrator triggers worker auto-respawn (calls into piece 2's start script with a respawn flag), then re-issues the dispatch with the same idempotency key — the respawned worker reads the ledger and either returns the cached completion or re-runs from scratch (stateless-restartable per DR-5). Routing edge: `[dispatch.py:22-37]` SDK import becomes conditional on a `transport=` argument; new transport function `_dispatch_via_bridge()`.
- Piece 2 ⇄ new file `plugins/cortex-overnight/bin/cortex-interactive-overnight-start` (tmux + caffeinate wrapper) + extend `[cortex_command/overnight/ipc.py]` with a cancel-via-IPC channel that doesn't require a PID + power-source poll loop attached to runner.py main loop. New env var `CORTEX_OVERNIGHT_PRESENCE_POLL_SECONDS`.
- Piece 3 ⇄ `cortex/lifecycle/overnight-state.json` (schema add: `mode: "sdk"|"interactive"|"hybrid"`) + `[dispatch.py:644-656]` (routing branch keyed on `mode` + `(model, effort)` for hybrid) + `cortex overnight start --mode <m>` CLI flag.
- Piece 4 ⇄ `cortex/requirements/pipeline.md` (add a "Mode-conditional clauses" section that calls out which acceptance criteria are mode-specific) + new test file `tests/test_interactive_mode_pipeline_parity.py` that walks the mode-conditional clauses and verifies the implementation matches.

Piece count is 4. Falsification gate is conditional on piece_count > 5 per the research-phase template; gate does not fire.

## Decision Records

### DR-1: Build without pre-measurement (per operator direction)

- **Context**: Original Architecture proposed a measure-first piece that would instrument the existing `cost_usd` emission, parse orchestrator-round `stdout.json` cost, and produce a written decision artifact comparing per-month autonomous spend against tier credits for SDK budget. Operator declined this sequencing — preference is to build the interactive overnight mode directly.
- **Options considered**: (a) measure first, gate the bridge on the data; (b) build the bridge directly per operator direction; (c) bundle a minimal cost telemetry into piece 4's parity tests as ongoing visibility (no gate).
- **Recommendation**: (b). Operator has the clearest view of acceptable engineering investment for this benefit; explicit direction supersedes the measure-first default. Cost instrumentation is a legitimate follow-up if visibility matters post-rollout, but not a gate on this work.
- **Trade-offs**: We may build the bridge and later discover that pay-the-SDK-credit would have been the cheaper rational answer. Acceptance criterion: operator accepts this risk explicitly. If post-rollout the bridge proves not worth the maintenance, the SDK overnight default makes interactive mode opt-in only — operators can simply not select it.

### DR-2: Prefer Architecture B over A (with scoped parity claim)

- **Context**: A (long-lived in-session Skill loop) is conceptually the cleanest "everything on subscription" path. B keeps the Python runner and bridges only the agent transport.
- **Options considered**: A, B, C, plus the alternatives D (pay), E (direct API), F (wait), G (scheduled daytime).
- **Recommendation**: B. **B's preservation claim is scoped to the orchestrator layer**: process-group watchdog of the orchestrator, `runner.pid` IPC + cancel-via-signal, in-flight install guard, dashboard reading state, LaunchAgent scheduling — all preserved because the Python runner stays put. A breaks every one of these. At the agent-execution layer B and A both weaken: sandbox-settings-per-spawn → per-session (Task tool has no per-call `--settings`; in-session worker is post-exec for Seatbelt), watchdog-of-the-agent-work absent (worker is not a Python subprocess the process-group watchdog can see; replaced by bridge-layer heartbeat/timeout/respawn in piece 1). B's advantage over A is large at the orchestrator layer; small but non-zero at the agent-execution layer (B's bridge worker is a smaller fault-domain than A's whole-batch in-session orchestrator). The bridge is the hard part of B; once it exists, C (hybrid routing) is a one-conditional rollout strategy that lets gradual cutover by tier or by model.
- **Trade-offs**: B requires new IPC + a new MCP tool + a new Skill (the in-session worker host) + an idempotency-keyed dispatch ledger + a heartbeat-timeout-respawn liveness layer + sentinel/JSON-envelope marshaling for `DispatchResult.error_type`. This is real engineering — not "bounded" in the sense of "small," but bounded in the sense of "the contract surfaces are enumerable and the piece-1 IPC contract names each of them." **Reversibility**: if Q1 fails post-June-15 (DR-6 probe shows Task dispatches go to SDK budget) the bridge is dead code; the operator-facing affordance is "don't select `--mode interactive`," and the new MCP tool / IPC files / worker skill remain on disk but unused — they do not poison the SDK overnight path because piece 3's routing branch defaults to SDK. The maintenance cost of unused dead code is non-zero; DR-6 minimizes that exposure by gating significant build on the probe. A would be a rewrite of `runner.py` from Python to a Skill, with permanent loss of OS-process-based guarantees and no comparable rollback affordance.

### DR-3: Don't use Bash + `claude -p` from inside a session as the bridge

- **Context**: The lowest-effort "in-session" mechanism would be to have the in-session worker shell out to `claude -p` via the Bash tool — preserving the existing `claude -p` invocation patterns at `[runner.py:1028]`.
- **Options considered**: Bash + `claude -p`; Task tool (subagent dispatch); Skill tool invocation.
- **Recommendation**: Task tool. The `claude -p` subprocess re-auths fresh from the inherited environment; it is explicitly named in Anthropic's June 15 policy as a reclassified workload. Even though the OAuth token would be inherited from the parent session, the policy is transport-side (the SDK / `claude -p` traffic is the categorization, not the auth method). Task tool dispatches are in-session by construction and are the documented mechanism for subagent work inside Claude Code.
- **Trade-offs**: Task tool's free-form output requires the bridge to add structured-result marshaling (the worker prompts must include a sentinel-+-JSON-envelope pattern like `/critical-review` already uses at `[skills/critical-review/SKILL.md:55-77]`), whereas `claude_agent_sdk.query()` returns typed `ResultMessage` objects directly. The marshaling overhead is small; the policy-conformance risk of using `claude -p` is large. **Asymmetry caveat**: the transport-side-policy argument that disqualifies `claude -p` (the policy reclassifies workloads by SDK/CLI transport, not by auth method) should make us cautious about Task tool too — if Task tool's underlying transport shares the SDK categorization, the same disqualification would apply. The artifact treats this asymmetry as a probe target (Q1 + DR-6), not a settled fact. If the post-June-15 probe shows Task dispatches go to SDK budget, the chosen mechanism in this DR loses its advantage over Bash + `claude -p` and the recommendation collapses to "pay the SDK credit pool" (option D in Feasibility).

### DR-4: Keep SDK overnight as the default; interactive is the operator-opt-in mode

- **Context**: User explicitly wants both modes.
- **Options considered**: Default to interactive; default to SDK; let mode follow auth (subscription-only users → interactive, API-key users → SDK).
- **Recommendation**: Keep SDK as default. SDK overnight is the proven autonomous path (launchd-fired, watchdog-restartable, runs unattended on any host including remote). Interactive overnight requires an operator-attached session (tmux + caffeinate + plugged in + locked) and is fragile-er. Default should be the mode that works without the operator being at their machine.
- **Trade-offs**: Subscription-only users who don't want to pay the SDK credit are opted into the fragile-er mode. The decision artifact (DR-1) should explicitly note this — if those users' workload is small enough to fit in the SDK credit, they're better off paying and using SDK overnight.

### DR-5: Stateless-restartable worker design (compaction-survival fallback for missing `PreCompact` hook)

- **Context**: Piece 1's worker host runs inside a Claude Code session, which auto-compacts at ~83.5% context fill. The documented multi-agent failure mode ([#11280](https://github.com/anthropics/claude-code/issues/11280)) is "after compaction the orchestrator loses awareness of live agents and can spawn duplicates." The `PreCompact` hook ([#43733](https://github.com/anthropics/claude-code/issues/43733)) that would allow in-process checkpointing is an open feature request — not available. Q2 asks: is compaction-survival achievable without `PreCompact`?
- **Options considered**: (a) wait for `PreCompact` to land before building piece 1 (open-ended dependency on Anthropic); (b) checkpoint at every dispatch boundary, accept that mid-dispatch compaction loses the live dispatch's record; (c) **stateless-restartable worker** — the worker holds no in-RAM state beyond what is reconstructible from disk, every dispatch carries an idempotency key, the only persistent state is an append-only dispatch ledger written before each Task spawn; on post-compaction or session-restart resume the worker reads the ledger and either returns the cached completion or re-runs the dispatch idempotently.
- **Recommendation**: (c). The orchestrator side enforces the contract: dispatches always carry a UUID idempotency key; the orchestrator-side timeout-and-respawn (piece 1 IPC contract) re-issues the dispatch with the same key after worker death from any cause, including compaction; the worker reads the ledger before each Task spawn to detect duplicates. The #11280 duplicate-spawn failure mode is neutralized because the second spawn-attempt finds the ledger entry from the first and either returns the result or no-ops on a still-in-flight dispatch.
- **Trade-offs**: Requires dispatch payloads to be fully idempotent at the prompt level (the Task agent must not have side effects that change behavior on re-run — for read-only or git-worktree-isolated work this is given; for state-mutating prompts the orchestrator may need to wrap with a "check-then-write" pattern). Adds the dispatch-ledger file as a new operational artifact. Cannot recover *partial* in-flight dispatch state — if a Task agent was halfway through a long edit when compaction fired, the rerun starts from scratch (with whatever git state the prior partial run committed). For the existing overnight workload this is acceptable because every per-feature dispatch already runs in a fresh git-worktree-isolated agent context — partial progress that wasn't committed is by-construction discardable.

### DR-6: Post-June-15 Q1 empirical probe gates piece 1 significant code commit

- **Context**: Architecture B's entire benefit (agent traffic on subscription quota) rests on Q1 — "Task-tool dispatches actually stay on subscription quota in practice." This is `[premise-unverified: not-searched-deeply]` per RQ5. The policy split takes effect June 15, 2026; before then there is no observable credit-pool accounting to test against. The operator declined a full cost-instrumentation pre-gate (DR-1), but a single Task-tool dispatch through the worker host under a metered subscription account, observed against Anthropic's documented credit-pool accounting, is a different (much cheaper) experiment that resolves Q1 binarily.
- **Options considered**: (a) skip the probe, commit fully to the bridge build, accept worst-case sunk cost; (b) the operator's declined "measure-first" piece (full cost-telemetry decision artifact); (c) a minimal Q1 probe wired as the first acceptance criterion of piece 1 — the operator runs one Task-tool dispatch through the prototype worker host post-June-15 and confirms the spend lands on subscription, not SDK budget. Piece 1's design work (IPC contract, MCP tool stub, worker skill skeleton) can land pre-probe; *significant* build (the marshaling logic, the dispatch ledger, the heartbeat/timeout/respawn machinery) gates on probe pass.
- **Recommendation**: (c). The probe is one dispatch; the verification is reading a number from Anthropic's account UI; it does not require the operator to instrument anything. It is not "measure-first" in the sense the operator rejected — that was a months-long cost-telemetry workstream. This is a one-Task-call sanity check that prevents a worst-case where the bridge is built and immediately invalidated.
- **Trade-offs**: Adds a sequence point between piece-1 design and piece-1 implementation. Worst case: probe fails, design work is sunk (cheap; days, not weeks); recommendation collapses to D (pay the SDK credit pool) and operator proceeds with SDK overnight. Best case: probe passes, build proceeds with confidence. This DR is *additive* to the operator's declined measure-first work, not a re-statement of it; if the operator declines the probe too, this DR collapses to (a) and DR-1's risk acceptance becomes load-bearing.

## Open Questions

- **Q1**: Confirm Task-tool subagent dispatches actually stay on subscription quota in practice. `[premise-unverified: not-searched-deeply]` — Anthropic docs do not enumerate per-tool billing, and the policy split is documented at SDK/CLI level. The strong inference is that in-session Task dispatches are subscription, but the only way to be sure is empirical testing once June 15 lands. **Architecture B is invalidated if Task dispatches turn out to be SDK-budgeted.** Gated by DR-6 (the post-June-15 probe is the first acceptance criterion of piece 1).
- **Q2**: Compaction-survival mechanism for the in-session worker host. The `PreCompact` hook is an [open feature request](https://github.com/anthropics/claude-code/issues/43733). Without it, the worker can checkpoint state on every dispatch boundary, but a compaction firing mid-dispatch can still lose the live dispatch's record. **Recommended answer (DR-5): stateless-restartable worker design** — dispatches carry idempotency keys, the worker's only persistent state is the append-only dispatch ledger, post-compaction resume reads the ledger to detect duplicates. Open sub-question: are there workload classes within cortex-overnight that are *not* idempotency-safe at the prompt level (i.e., dispatches that mutate non-git-isolated state)? Audit at decompose time.
- **Q3**: Cancel handoff mechanism. `cortex overnight cancel --force` today is SIGKILL the pgid; for an in-session worker there is no pgid. Decompose-time question: does cancel become "write a cancel-sentinel file and have the worker poll" (additive, simple) or "issue an MCP tool call that the worker is waiting on" (cleaner, more invasive)?
- **Q4**: Power-source guard granularity. Polling `pmset -g batt` every N seconds works; what's the right N (responsive vs. CPU overhead) and what's the right pause-vs-stop behavior on AC→Battery (pause and resume on re-plug, or hard-stop and require operator-restart)?
- **Q5**: Hybrid routing v1 cutover scope — at decompose time, decide exactly which `(model, effort)` cells in the `[dispatch.py:127-148]` matrix the v1 bridge serves. Conservative starting set is `(opus, *)` + `effort=xhigh` + the hard-coded Opus sites at `[integration_recovery.py:215]` and review-fix on Opus `[review_dispatch.py:385-396, 500]`. Whether to include `complex/medium` Sonnet dispatches in v1 is the open call.
- **Q6**: Sandbox-isolation blast radius when bridge workers share parent session's Seatbelt profile. The bridge worker runs all in-flight dispatches inside one Claude Code session and shares its Seatbelt deny/allow set across them. The current SDK path's per-feature sandbox tempfiles `[dispatch.py:573-602]` are designed to constrain each dispatch to its own worktree + a documented deny list; under the bridge, all dispatches see the union (operator's session) sandbox. Decompose-time questions: (a) what specific paths become reachable to dispatch N that were denied to dispatch M under per-spawn? (b) is the operator's interactive-overnight sandbox profile strict enough that the union is acceptable (e.g., does the operator's `~/.claude/settings.local.json` already deny enough cortex-sensitive paths)? (c) is there a way to *tighten* the parent session's sandbox specifically when `mode = interactive` is selected, so per-session approximates per-spawn for the duration of the batch?
- **Q7**: Worker-liveness contract — concrete values. The IPC contract in piece 1 specifies heartbeat interval + orchestrator-side timeout + auto-respawn behavior; the values are placeholder (30s heartbeat, 3× timeout). Decompose-time question: what's the actual right heartbeat cadence given Task-tool dispatch latency distributions (Opus-on-complex can run minutes), and how many consecutive missed heartbeats before respawn? Also: what's the operator-visible failure surface when respawn itself fails three times in a row (escalate to morning report? hard-pause the session?)?
