The research settled on extracting only the interactive backlog management skill into an optional `cortex-backlog` plugin, while leaving the local engine, the `backlog-author` body composer, and the overnight selection machinery in the core wheel; a new `resolve_backlog_backend` config reader becomes the routing spine, defaulting to `cortex-backlog` when the key is absent.

Three alternatives were weighed. Moving `backlog-author` alongside the management skill was dropped because the external-backend create path in discovery and morning-review depends on that composer even when no local plugin is installed — pulling it out would gut the feature it exists to enable. Building typed per-tracker code adapters for GitHub and Jira was rejected in favor of using the LLM itself as the integration layer via a freeform `instructions` field, which matches how Claude Code already drives `gh` and avoids owning brittle auth and API surface for every tracker variant; Jira stays prose-only and unverified because no single canonical Jira CLI exists. A persisted external-ID map for round-trip tracking was held as a phase-2 trigger; the chosen approach embeds the cortex slug in the issue body and re-resolves by listing and disambiguating, which survives human title edits at lower cost.

The accepted cost is that the external path is genuinely best-effort and loses fidelity in several concrete ways: GitHub search is fuzzy and eventually consistent so create-then-search can race and duplicate on retry; the `dev` epic-map read and `refine` parent-epic alignment have no local index under an external or none backend and degrade to an advisory rather than functional guidance; and fields like `complexity` and `criticality` are absent from `index.json`, so any external read-back is structurally lossy from the start.
