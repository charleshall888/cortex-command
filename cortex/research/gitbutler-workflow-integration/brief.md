The team settled on doing nothing: keep git worktrees for parallel isolation, skip GitButler as a dependency, and defer stacked PRs entirely. No GitButler-derived concept clears the bar for building right now.

Several alternatives were considered. Replacing worktrees with GitButler virtual branches was rejected because virtual branches share one working tree with no filesystem isolation — all concurrent agents operate in the same directory and overlapping file edits race, a fact confirmed by both GitButler's own documentation and a production user who fell back to worktrees for exactly that reason. Adopting the `but` CLI as a runtime dependency was rejected on three independent grounds: its FSL-1.1-MIT license carries a non-compete clause, the tool is pre-1.0 and not a pip wheel, and its internal state model conflicts with the file-based architecture. Stacked PRs were the most seriously weighed option and initially looked high-value, but a frequency measurement killed the case: only one of 260 backlog items carries any dependency chain at all, and the naive implementation — delegating re-targeting to GitHub's auto-delete — does not land a chain cleanly when the bottom PR is squash-merged or revised, which is the common case. An oplog-style snapshot recovery model was considered but deferred because it overlaps existing commit history and has no second consumer to justify the machinery.

The accepted tradeoff is that the status quo keeps a forward-only, lossy error-recovery ladder and cannot author a dependency chain in a single session — but those are real costs against a near-empty demand signal today.
