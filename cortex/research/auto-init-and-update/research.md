# Research: auto-init-and-update

Investigation of whether and how to auto-apply `cortex init` (first-time per-repo scaffolding) and `cortex init --update` (drift apply after CLI bumps) so consumers no longer need to manually invoke them. Triggered by the user's observation that the existing 3-layer CLI/plugin auto-update story (Layers 1–3 in `docs/internals/auto-update.md`) leaves per-repo init as the last manual verb a user must know.

**Direction adopted**: narrow skill-entry, wired into `/lifecycle` only (the user's original framing). A new `cortex init --ensure` CLI verb computes the installed CLI's `init_artifacts_hash`, compares it against the per-repo hash in `.cortex-init`, and dispatches to `cortex init` (missing marker) or `cortex init --update` (hash mismatch) synchronously — or no-ops on hash match. The lifecycle SKILL.md adds a single entry-time wiring line: "Before phase 1, run `cortex init --ensure`." Other cortex skills (discovery, refine, backlog, requirements-gather/write, critical-review, research) are NOT wired. The user-visible drift symptoms most worth catching (CLAUDE.md fence drift, stale allowWrite path, missing gitignore entries) are concentrated at the lifecycle entry point — the moment of "I'm committing to multi-hour work on this repo." Other skills tolerate slightly-stale READMEs; the existing `cortex init --update` verb remains the user-facing manual recourse for the narrower drift cases this design does not cover.

## Research Questions

1. **Init-version tracking surface.** What signals exist today and what cheap probe can detect "auto-update needed"?
   → **Answer**: Today there is no single "init schema version" signal. Three surfaces exist that each version a sub-piece: the `.cortex-init` JSON marker stores `cortex_version` (PEP 440 pkg string) and `initialized_at` but is currently used only as a presence check, with content never read [`cortex_command/init/scaffold.py:402-425`, `cortex_command/init/handler.py:244`]; the CLAUDE.md auth fence carries `version=N` but scoped narrowly to the EnterWorktree clause only [`cortex_command/init/scaffold.py:92`, value currently `1`]; `drift_files()` byte-compares ~4 template files against shipped versions but skips *missing* files entirely [`cortex_command/init/scaffold.py:367-399`, `scaffold.py:392-394`]. The `cortex --print-root` envelope already carries `version` and `schema_version` and is cleanly extensible [`cortex_command/cli.py:195-273`]. **Cheap probe (mechanical, no human discipline)**: derive a content hash over the shipped template inventory (`cortex_command/init/templates/cortex/**` + the literal `_GITIGNORE_TARGETS` set + the literal `_CLAUDE_MD_AUTH_VERSION` integer + the literal allowWrite-entry shape). Surface the hash in `cortex --print-root --format json` as `init_artifacts_hash`. Store the last-applied hash in `<repo>/cortex/.cortex-init` as `init_artifacts_hash`. Probe = single 16-byte read + string compare. The hash fires on *every* substantive change including additive template files, gitignore-target additions, and fence-version bumps — without requiring any developer to remember to bump a constant. This replaces the earlier `_INIT_SCHEMA_VERSION` integer proposal because that signal coupled detection reliability to author discipline (forgotten bump → silent rot → drifted users stay drifted). The hash signal is mechanically derived from file state and cannot silently fail to fire.

2. **Trigger surface comparison.** SessionStart-async vs skill-entry vs hybrid — which balances "users don't need to know commands" against "don't write to repos the user doesn't want cortex on"?
   → **Answer**: Narrow skill-entry — wired into `/lifecycle` only — is the recommended surface. The decision walks through eliminations: (a) Silent SessionStart-write was ruled out by the failure-mode review's five concrete failure modes (random-repo contamination, ADR-0003 violation, CLAUDE.md no-flock race amplification, force-push amplification, SessionStart docs warn off filesystem mutation). (b) SessionStart-emits-additionalContext was ruled out because it depends on unmeasured Claude additionalContext routing reliability AND on directive-routing-prose escalation that the CLAUDE.md MUST-escalation policy blocks without effort=high empirical evidence this discovery did not produce. (c) Hybrid (SessionStart probe + skill-entry backstop) was structurally redundant once skill-entry does the correctness work. (d) Broad skill-entry (all 8 mutating skills + parity test) was further narrowed: the user-visible drift symptoms most worth catching (CLAUDE.md fence drift, stale allowWrite path, missing gitignore entries) are concentrated at the lifecycle entry point — the moment of "I'm committing to multi-hour work on this repo." Other mutating skills (discovery, refine, backlog, etc.) tolerate the failure mode of slightly-stale READMEs in `cortex/`, and the existing `cortex init --update` verb remains the manual recourse. The remaining surface — `/lifecycle` only — has reliable apply (synchronous foreground), matches prior-art Pattern A (piggyback on the user's deliberate engagement with cortex), sidesteps the additionalContext-routing-reliability question entirely, and adds minimal complexity (one new CLI flag, one skill-prose line, ~70 lines of new code).

3. **Trust posture & opt-out.** Is opt-out (`CORTEX_AUTO_INIT=0`) sufficient, or is opt-in required?
   → **Answer**: The industry consensus is opt-in via piggyback on a user-initiated command. direnv's `allow` per-file-hash, husky/lefthook/pre-commit/git-lfs/dotnet-tool-restore all require explicit install verbs; mise/asdf/rustup permit auto-on-entry only for READ-style operations (PATH shim, env-var load), never for writes. An opt-OUT env var is the wrong inversion — every other declarative tool defaults to "I do nothing unless you've explicitly opted this repo in." Cortex's natural opt-in event is the first invocation of any cortex skill in this repo, which is also the natural place for first-time init.

4. **Reconciliation with existing machinery.** Layer 3 hook (`cortex-cli-background-install.sh`), visibility hook (`cortex-cli-version-sync.sh`), `--verify-worktree-auth` §1a probe — ordering, redundancy, dependencies?
   → **Answer**: Critical ordering: any auto-apply path MUST run AFTER Layer 3 reinstall completes, otherwise auto-init runs against the OLD CLI's templates and writes the wrong schema version. Layer 3 uses `start_new_session=True` detach into the background; both new and old SessionStart hooks fire concurrently per `hooks.json` array order which does NOT guarantee execution sequencing. Two viable mechanisms: (a) auto-init hook polls for the existing `${XDG_STATE_HOME}/cortex-command/install-in-progress` marker that Layer 3 writes [`docs/internals/auto-update.md:41`] and skips if Layer 3 is mid-flight; (b) fold auto-init into Layer 3 as a post-reinstall continuation. The `--verify-worktree-auth` §1a probe is narrowly scoped to the EnterWorktree fence — keep it as-is (a fail-fast diagnostic for hook regressions); auto-init handles a strictly broader surface.

5. **Async-apply race window & CLAUDE.md write safety.** What's the partial-state failure mode?
   → **Answer**: `ensure_claude_md_authorization` does `read → splice → atomic_write` with **NO flock** [`cortex_command/init/scaffold.py:554-633`]. The sibling `settings_merge.register` has a flock on `~/.claude/settings.local.json` [`cortex_command/init/settings_merge.py:63-88`], but CLAUDE.md does not. Atomic_write prevents torn writes only; it does NOT prevent the read-stale-then-write-clobber pattern. The exploit (two parallel sessions read-stale → write-clobber) requires concurrent invocation, which is rare under today's manual-`cortex init` usage — no user-hit incident is documented. The bug therefore exists as a **latent correctness issue** today and would be **amplified by any auto-apply path** (concurrent SessionStarts in multiple Claude windows; mid-session re-apply on CLI bump). The fix (sibling-lockfile flock matching `settings_merge.py:63-88`) is desirable independently of auto-init but does not have a today-incident driving urgency; it is a co-traveler with auto-init rather than a strict gate. See Decision 4 for the framing.

6. **Prior art.** How do uv, asdf, mise, direnv, brew, husky, lefthook, pre-commit, git-lfs, devcontainers, rustup, dotnet handle this?
   → **Answer**: Universal pattern: explicit opt-in command per repo, often piggybacked on a package-manager-level command the user already runs (husky/lefthook ride the npm `prepare` script; pre-commit needs `pre-commit install`; git-lfs needs `git lfs install`; direnv needs `direnv allow`; brew bundle needs `brew bundle`; dotnet needs `dotnet tool restore`). The closest analog to "auto-apply on update" is `pre-commit autoupdate`, which the maintainers deliberately kept manual because auto-mutating pinned config silently breaks things; the community then built pre-commit.ci to gate it behind PR review [pre-commit.com, pre-commit/pre-commit#2785]. Claude Code SessionStart hooks themselves are documented as a context-injection surface, not a filesystem-mutation surface — "keep these hooks fast" appears explicitly in the hooks docs [code.claude.com/docs/en/hooks].

7. **First-time blanket-init harm model.** Concrete worst-case for auto-init on a non-cortex repo?
   → **Answer**: A security researcher clones a third-party repo to inspect for vulnerabilities, opens Claude Code to ask questions, and unwittingly creates `cortex/`, modifies `CLAUDE.md`, appends to `.gitignore`, and stages those changes for accidental commit/push back to upstream — leaking that they use cortex-command and polluting the upstream repo. The current `cortex init` design has an R19 "content-aware decline" gate [`cortex_command/init/scaffold.py:138-156`] but `--update` bypasses R19 [`cortex_command/init/handler.py:264-269`]. Auto-fired `--update` therefore has no consent gate at all. Undo cost: removing `cortex/`, reverting CLAUDE.md/`.gitignore`, running `cortex init --revoke-worktree-auth` + `--unregister`, manually editing `~/.claude/settings.local.json` to remove the allowWrite entry.

## Codebase Analysis

### What `cortex init` writes today (the auto-apply target surface)

`cortex_command/init/handler.py` orchestrates writes to four locations in this order [`handler.py:228-298`]:

1. **`<repo>/cortex/**` umbrella scaffold** — 4 markdown templates from `cortex_command/init/templates/cortex/`: `backlog/README.md`, `lifecycle.config.md`, `lifecycle/README.md`, `requirements/project.md`. Last meaningfully changed Nov 2024 with `c8110de5` (#202 umbrella relocation). Written via `scaffold.scaffold()` with additive-no-overwrite semantics on `--update` [`scaffold.py:312-313`].
2. **`<repo>/.gitignore`** — `ensure_gitignore` appends `_GITIGNORE_TARGETS` block [`scaffold.py:70-74, 428-497`]. Idempotent. Has orphan-prefix repair logic because prior truncated writes were observed in the wild — adding concurrent writers expands the surface for that bug class.
3. **`<repo>/CLAUDE.md`** — `ensure_claude_md_authorization` splices the cortex-managed fence (currently `_CLAUDE_MD_AUTH_VERSION = 1`) [`scaffold.py:92, 554-633`]. Versioned-replace policy: "latest writer wins, no in-fence user edits respected" per ADR-0006. **No flock** — read-stale-then-write-clobber race exists today.
4. **`~/.claude/settings.local.json`** — `settings_merge.register` adds `<repo>/cortex/` to `sandbox.filesystem.allowWrite[]` via sibling-lockfile flock [`settings_merge.py:63-88, 136-205`]. Idempotent exact-string append.

Marker: `<repo>/cortex/.cortex-init` JSON storing `cortex_version` + `initialized_at` [`scaffold.py:402-425`]. Currently presence-check only — its content is never read.

### What state-mutating cortex skills exist (the auto-apply trigger surface, if skill-entry-driven)

Eight skills write under `cortex/` [agent findings from `/Users/charlie.hall/.claude/plugins/cache/cortex-command/cortex-core/e8f94aa00756/skills/`]:

- `backlog` (add/new/archive/reindex) — writes `cortex/backlog/`
- `lifecycle` — writes `cortex/lifecycle/{slug}/`
- `refine` — writes `cortex/lifecycle/{slug}/{research,spec}.md` + backlog frontmatter
- `discovery` — writes `cortex/research/{topic}/`, `cortex/backlog/NNN-*.md`
- `research` (lifecycle mode) — writes `cortex/lifecycle/{slug}/research.md`
- `requirements-gather` — writes `cortex/requirements/glossary.md` (per-term append)
- `requirements-write` — writes `cortex/requirements/{project|area}.md`
- `critical-review` — writes `cortex/lifecycle/{feature}/critical-review-residue.json` (optional)

Read-only consumers: `dev`, `pr`, `commit`, `diagnose`, `backlog-author`.

### The existing `--verify-worktree-auth §1a` precedent

`skills/lifecycle/references/implement.md:181` shows the closest existing pattern: skill-entry probe-and-warn (NOT probe-and-apply). On stale/absent, the skill falls back to `cd`-shim and emits a single-line diagnostic pointing the user to `cortex init`. The auto-init surface would generalize this from probe-only to probe-and-apply at one of the trigger surfaces.

### The existing 3-layer CLI/plugin auto-update architecture

`docs/internals/auto-update.md` documents the Layer 1 (Claude Code marketplace fast-forward of `CLI_PIN`), Layer 2 (MCP-tool-call-gated CLI reinstall via `_ensure_cortex_installed` at `plugins/cortex-overnight/server.py:775`), and Layer 3 (SessionStart-async-gated CLI reinstall via `cortex-cli-background-install.sh` calling `install_core.run_install_in_background()`). Layer 3 explicitly widens the supply-chain attack surface relative to Layers 1–2 and offers `CORTEX_AUTO_INSTALL=0` as the per-user opt-out [`docs/internals/auto-update.md:88`]. Auto-init compounds this widening — a force-pushed malicious release would not only reinstall the CLI but also auto-mutate every consumer repo's CLAUDE.md/.gitignore on next SessionStart.

### ADR conflict surface

`cortex/adr/0003-per-repo-sandbox-registration.md:7` declares `~/.claude/settings.local.json::sandbox.filesystem.allowWrite` as *"the only write cortex-command makes outside its own tree."* `cortex/adr/0006-cortex-init-consumer-claude-md-authorization-surface.md` already widened this narrowly to add the CLAUDE.md fence write, calling out the contradiction explicitly. Auto-firing `cortex init` on every SessionStart further detaches the CLAUDE.md write from ADR-0006's narrow EnterWorktree-auth justification — converting it into a blanket-mutation-on-session-open surface that neither ADR currently authorizes.

### Backlog #145 prior wontfix

Closed 2026-04-25 with explicit "Out of scope" language [`cortex/backlog/145-...md:71-72`]:

> *Auto-applying `cortex init --update` for project scaffolding. Orthogonal; infrequent; user-triggered is fine.*

The "infrequent" premise is invalidated by data: 32 commits to `cortex_command/init/` over the past ~16 months, of which 6–8 were schema-level changes that would have required users to re-run `--update` (the CLAUDE.md fence introduction, #202 umbrella relocation, allowWrite scope widenings, worktree path migrations). That's roughly one re-run-worthy change every 2 months — re-examination is justified. The closure itself documented that the broader CLI auto-apply rejection was about the wrong APPLY VENUE, not about the probe/apply DECOUPLING PATTERN being unsound — so #145's Shape 3 hybrid pattern remains viable when re-appropriated for per-repo scaffolding.

## Web & Documentation Research

### Claude Code SessionStart hook docs

[code.claude.com/docs/en/hooks] describes SessionStart hooks as a context-injection surface: *"Useful for loading development context… or setting up environment variables"* with explicit *"keep these hooks fast"* guidance. Only `command` and `mcp_tool` hook types are allowed. Real-world failure modes already filed against SessionStart: [anthropics/claude-code#55427] EEXIST on Windows; [#19491] hook fires before plugins ready; [#10373] hook not firing for new conversations. The docs do not endorse filesystem-mutation use cases.

### pre-commit autoupdate as cautionary tale

[pre-commit.com] and [pre-commit/pre-commit#2785] document that `pre-commit autoupdate` is deliberately a manual command because mutating pinned config silently can break CI. The community wrapper [pre-commit.ci] gates it behind PR review for controlled cadence. Direct analog to "auto-apply `cortex init --update` on CLI bump."

### npm postinstall as anti-pattern

[news.ycombinator.com/item?id=45040282], [pnpm.io/supply-chain-security] — the npm `postinstall` design is widely cited as a major supply-chain attack vector. pnpm v10 disabled it by default; common advice is `npm config set ignore-scripts true`. Reinforces the prior-art recommendation against silent auto-mutation triggered by package-level events.

### Kiro IDE auto-writes `.vscode/settings.json`

[kirodotdev/Kiro#3628], [microsoft/vscode#206802] — exact behavioral analog: an IDE auto-creates a per-workspace config file on activation. Users filed issues asking for opt-out. Establishes the surface as user-disliked when applied silently.

## Domain & Prior Art

### Pattern A — Explicit per-repo opt-in (dominant)

Examples: direnv `allow` (TOFU per file-hash), husky `install` (npm prepare script piggyback), lefthook `install`, pre-commit `install`, git-lfs `install`, brew bundle, dotnet tool restore. Why dominant: a per-repo config from a stranger is arbitrary code or arbitrary configuration applied to your machine. One extra command at clone time is universally judged less bad than silent execution. Pattern is typically paired with package-manager piggybacking — the user already ran something explicit, and the framework recycles that consent.

### Pattern B — Auto-on-entry (narrow, READ-only)

Examples: mise/asdf (only after explicit `mise install` or shim invocation; bare `cd` is silent), rustup (only on first `cargo`/`rustc` if `rust-toolchain.toml` present), Claude Code SessionStart hooks (designed for context injection). Pattern only works when: (a) the action is idempotent and reversible (env-var load, PATH shim swap), (b) the user has globally opted in once (installed the shell hook), (c) the per-repo file is READ, not written. mise explicitly refused "install on cd" because *"a cd into a project directory shouldn't silently block while downloading and installing tools"* [jdx/mise#8014].

### Pattern C — Hybrid: detect + prompt (rare, requires UI)

Examples: VS Code devcontainers ("Reopen in Container?"), Cursor MCP prompt-on-detect. Rare because CLI-only tools have no good interactive-prompt surface. Well-regarded examples are all editor-integrated.

### Concrete recommendation from prior art

**Pattern A with package-manager piggyback** maps cleanly to cortex: `/plugin install cortex-core` per Claude Code install + `cortex init` per repo are the existing user-initiated commands. The per-repo bootstrap rides on the first of those, not on SessionStart. For the update-on-CLI-bump case, model on `pre-commit autoupdate` — explicit command (`cortex sync` or use existing `cortex init --update`), optionally CI-gated. SessionStart's correct role is to *detect* and *surface* drift via `additionalContext` (in-policy: context injection), letting Claude run the apply in the user's foreground on the next turn (consent gate: Claude visibly invokes the verb).

### Survey-canvas caveat

The 12-tool survey above is asymmetric: it canvassed tools that follow Pattern A and Pattern B's narrow-and-READ-only form, with one negative Pattern-B-write example (Kiro IDE). It did NOT canvass known silent-auto-mutation precedents that are nevertheless well-regarded: VS Code language extensions auto-writing `.vscode/settings.json` on activation; gh CLI per-repo state files; JetBrains `.idea/` directory autocreation; GitHub Codespaces auto-config. Those examples may indicate Pattern B's write-on-entry posture is more shippable than the canvas implies — though they also have filed-issue equivalents and the survey-side asymmetry should be acknowledged rather than presented as universal industry rejection. The hybrid recommendation below stands on independent structural grounds (consent semantics, mid-session-bump handling, ADR-0003 narrowness) and does not rest solely on the prior-art canvas.

## Architecture

### Pieces

- **Content-hash signal (`init_artifacts_hash`)** — mechanically derived hash over the shipped template inventory (`cortex_command/init/templates/cortex/**`) + the literal `_GITIGNORE_TARGETS` set + the literal `_CLAUDE_MD_AUTH_VERSION` integer + the literal allowWrite-entry shape. No human bump discipline; fires on every byte-level change including additive templates. The authoritative "what version is installed."
- **`.cortex-init` marker `init_artifacts_hash` field** — per-repo record of "what hash was last applied here." Per-machine state (correctly does not travel via git). Read by `cortex init --ensure` to compare against the installed hash.
- **`cortex init --ensure` CLI flag** — new flag on the existing `cortex init` verb. Computes the installed `init_artifacts_hash` (in-process — no subprocess), reads `<cwd>/cortex/.cortex-init`'s hash, string-compares. (1) On hash match: silent no-op exit 0. (2) On missing marker: runs the existing `cortex init` first-time logic. (3) On hash mismatch: runs the existing `cortex init --update` logic. Either branch writes the new marker with the current installed hash. This is the **sole** apply surface.
- **Lifecycle SKILL.md wiring directive** — `/cortex-core:lifecycle` adds a single line near phase 1 entry: "Before phase 1, run `cortex init --ensure`." Cost: one Bash tool call from Claude (~30-50ms cold; the in-process hash compare is microseconds).
- **`--print-root` envelope extension** (optional) — `init_artifacts_hash` field added to the envelope. Not strictly required by `--ensure` (the CLI computes it in-process), but cheap to add for future consumers (a `cortex doctor` drift check, a status-line indicator, a CI-friendly probe).
- **CLAUDE.md flock primitive** — sibling-lockfile flock around the `ensure_claude_md_authorization` read-splice-write sequence, matching `settings_merge.py:63-88`. Fixes a latent correctness issue; desirable independently. Becomes more pressing once concurrent-apply pressure exists (two parallel Claude Code windows in the same repo each triggering `--ensure`) but does not gate shipping the flag.

### How they connect

The lifecycle skill's phase-1 prose tells Claude to invoke `cortex init --ensure` as a Bash tool call before doing any lifecycle work. The CLI computes its own installed hash in-process, reads the repo's `.cortex-init`, and either no-ops (hash match) or runs the appropriate init verb synchronously. The user sees the Bash invocation in the transcript and, on actual drift, sees the init/update output. On hash match (the common case), the entire interaction is a single CLI invocation that completes in ~30-50ms with no visible effect. The trigger surface is exactly `/lifecycle` invocation — the unambiguous opt-in signal that the user is committing to multi-hour work on this repo and wants cortex artifacts in good shape. Repos where the user invokes only `/discovery`, `/backlog`, or other non-lifecycle skills don't get auto-bootstrapped; they're expected to function on whatever cortex state is present (existing skills already cope with partial cortex/ state). First-time-uninitialized repos and drift-update repos use the same `--ensure` code path. The CLAUDE.md flock fix is independent — it remains a latent correctness issue worth fixing on its own merits.

Skills other than `/lifecycle` and bare-shell `cortex` invocations are out of coverage by design. The user's manual recourse for those cases is the existing `cortex init --update` verb, documented in `docs/setup.md`. The narrow scope reflects that the most user-visible drift symptoms (CLAUDE.md fence drift caught by `--verify-worktree-auth §1a`, stale allowWrite causing sandbox prompts, missing gitignore entries causing silent leak risk) are either already detected by other mechanisms or concentrate at lifecycle entry.

## Decision Records

**Decision 1: No SessionStart surface — apply happens at skill-entry only.** There is no SessionStart hook for auto-init/auto-update. The skill-entry helper is the sole apply surface. This keeps the SessionStart hook list focused on the existing CLI-reinstall + visibility surfaces (Layer 3) and avoids the random-repo-contamination, ADR-0003-violation, and unmeasured-additionalContext-routing failure modes. *Alternative considered*: silent SessionStart-async write. *Rejected because*: the failure-mode reviewer surfaced five concrete failure modes (random-repo contamination on every git repo opened in Claude Code, ADR-0003 violation, CLAUDE.md no-flock race amplification, force-push-attack amplification, SessionStart docs warn off filesystem mutation). The prior-art canvas was asymmetric (didn't include silent-auto-mutation precedents like VS Code language extensions, gh CLI, JetBrains `.idea/`, Codespaces) but Pattern A dominance among declarative tools is real and the user-facing risk of writing to third-party clones is concrete. *Alternative considered*: SessionStart probe emits `additionalContext` only, Claude is expected to invoke apply. *Rejected because*: depends on unmeasured additionalContext routing reliability, and the CLAUDE.md MUST-escalation policy defaults to soft routing and blocks directive escalation without effort=high evidence this discovery did not produce — if the additionalContext doesn't reliably route, the original problem (users not knowing the verb) is unsolved. *Alternative considered*: SessionStart probe + skill-entry backstop (hybrid). *Rejected because*: skill-entry alone handles the correctness work; SessionStart adds only an early-visibility UX nudge with no compensating reliability gain. Project philosophy favors simpler — Complexity must earn its place by solving a real problem now.

**Decision 2: The apply path is a new `cortex init --ensure` CLI flag, wired into `/lifecycle` only.** The flag computes the installed hash in-process, reads the per-repo `.cortex-init` hash, and dispatches `cortex init` (missing marker), `cortex init --update` (mismatch), or no-op (match). The lifecycle SKILL.md adds one line: "Before phase 1, run `cortex init --ensure`." *Alternative considered*: a separate stdlib helper module (`cortex-ensure-init`) with a lazy session cache, wired into all 8 mutating skills. *Rejected because*: too much surface area for the value delivered — the most user-visible drift symptoms concentrate at lifecycle entry, other skills tolerate slightly-stale READMEs, and the existing `cortex init --update` verb remains the manual recourse. The narrower scope is ~70 lines + 1 prose line vs the broader scope's ~90 lines + 8 prose lines + parity test. *Alternative considered*: extend the existing `--verify-worktree-auth §1a` probe to apply rather than warn. *Rejected because*: §1a is narrowly scoped to the EnterWorktree fence; broadening it would conflate two concerns and the new flag is a cleaner separation.

**Decision 3: `--ensure` handles both first-time auto-init and drift-update through the same code path.** Missing `.cortex-init` is treated as a hash mismatch against the installed hash, branching to the `cortex init` first-time logic rather than `cortex init --update`. The user invoking `/lifecycle` on a previously-uninitialized repo gets bootstrapped automatically. Other skills (discovery, backlog, etc.) on uninitialized repos continue working with whatever cortex/ state is present (existing skills already cope with partial state). *Alternative considered*: separate flags for first-init (`--ensure-init`) vs update (`--ensure-update`). *Rejected because*: the user invoking `/lifecycle` doesn't know or care which case applies — one verb that handles both is cleaner.

**Decision 4: CLAUDE.md flock fix is a co-traveler, not a strict prerequisite.** The existing no-flock read-splice-write race is a latent correctness issue that exists today but has no documented user-hit incident — manual `cortex init` is single-writer-per-invocation. The race becomes more pressing under any auto-apply path (concurrent SessionStarts, mid-session re-apply on CLI bump), so the fix is desirable independently of this discovery's outcome. Bundling it as a "strict prerequisite" to auto-init would inflate this discovery's prerequisite chain; instead, ship the flock fix as an independently-justified correctness improvement and let it ride to production alongside (or ahead of) the auto-init work. *Alternative considered*: declare the fix a strict gate. *Rejected because*: the today-state has no incident driving urgency; framing it as a gate misrepresents the dependency.

**Decision 5: Init-version signal is a mechanically-derived content hash, not a manually-bumped integer.** The signal is `init_artifacts_hash` — a hash over the shipped template inventory (`cortex_command/init/templates/cortex/**`) + the literal `_GITIGNORE_TARGETS` set + the literal `_CLAUDE_MD_AUTH_VERSION` integer + the literal allowWrite-entry shape. Surfaced via `cortex --print-root` envelope; persisted in `<repo>/cortex/.cortex-init` as `init_artifacts_hash` string. The probe and skill-entry helper both read the two hashes and string-compare. *Alternative considered*: a manually-bumped `_INIT_SCHEMA_VERSION` integer constant. *Rejected because*: the bump is gated on developer discipline with no CI gate, release-blocking test, or pre-commit hook documented; a release shipping a template change without the bump produces silent rot (no probe fires, no additionalContext emitted, drifted users stay drifted indefinitely). The artifact's own historical data (32 commits to `init/` in 16 months, 6–8 schema-bumps) shows the classification is fuzzy enough that present-day reviewers cannot retroactively classify to within ±2; future committers face the same fuzziness. The content-hash signal is mechanically derived from file state and cannot silently fail to fire — it fires on *every* byte-level change including additive template files and gitignore-target additions. *Alternative considered*: piggyback on the CLAUDE.md fence's `version=N`. *Rejected because*: that fence is narrowly scoped to the EnterWorktree auth clause per ADR-0006; broadening it conflates two concerns. *Alternative considered*: `~/.local/state/cortex-command/repo-init-versions.json`. *Rejected because*: marker is the natural home and avoids a second per-repo write surface. *Alternative considered*: extend `drift_files()` to include missing-file enumeration. *Considered but more complex than the hash approach*: the hash absorbs missing-file detection mechanically without requiring `drift_files()` to walk both directories on every probe.

## Open Questions

These are genuine unknowns requiring user input for decomposition choices — not deferred research.

- **Q1. `--ensure` failure visibility.** If `cortex init --ensure` fails (e.g., a malformed `.cortex-init`, a permission error writing CLAUDE.md), how should the lifecycle skill respond? Options: (a) fail-fast — block lifecycle phase 1 with a diagnostic naming the error; (b) warn-and-proceed — emit a one-line warning, continue into phase 1 against potentially-stale artifacts; (c) silent-skip — Layer 3's pattern. Phase 1 of lifecycle is the "clarify" step which doesn't yet write to `cortex/lifecycle/`, so there's some grace for stale state, but a hard failure in `--ensure` may indicate a deeper config problem worth surfacing. **Decomposition impact**: determines the exit-code handling in the lifecycle SKILL.md prose and whether failures emit NDJSON to an audit log.

- **Q2. Interactive confirmation on first-time init.** When the user invokes `/lifecycle` on a previously-uninitialized repo, `--ensure` would run the first-time `cortex init` flow — creating `cortex/`, modifying `CLAUDE.md`/`.gitignore`, and adding the allowWrite entry. Should this happen silently (consistent with "users don't need to know commands") or should `--ensure` prompt for confirmation on first-time-init only (preserving the explicit consent semantics that other init tools require)? Update-on-drift is less load-bearing — the user has previously initialized this repo, so they've already opted in. **Decomposition impact**: determines whether `--ensure` has an `--interactive` mode, what the lifecycle skill prose says about confirmation, and whether the AskUserQuestion is fired by the skill or by the CLI.

- **Q3. CI integration.** Should this design also ship a CI-friendly `cortex init --check` verb (exit non-zero on hash mismatch, no writes) so teams can fail CI on per-repo init drift, matching the `pre-commit autoupdate` + dependabot/renovate pattern? Could be a flag on `--ensure` (e.g., `--ensure --check` = exit non-zero, never write). **Decomposition impact**: separate ticket or in-scope for this discovery.

- **Q4. ADR-0003 amendment scope.** The narrow design's writes remain triggered by explicit cortex-skill invocation (`/lifecycle` → Bash call to `cortex init --ensure`). Structurally identical to a user typing `cortex init --update`. ADR-0003 governs write surfaces, not invocation frequencies. The Reviewer 4 reading is that no amendment is needed. **Decomposition impact**: defaults to no ADR amendment unless a specific surface-shape change is identified.

- **Q5. Empirical bump cadence projection.** The "32 commits, 6–8 schema-bumps in 16 months" data is historical and includes a heavy stretch of `cortex_command/init/` churn during the umbrella relocation epic. What's the projected steady-state bump cadence post-stabilization? If it falls to <1/year, the value proposition weakens; if it stays >1/quarter, the value proposition is firm. The user explicitly raised this question in the clarify phase. **Decomposition impact**: determines priority — high-priority work if steady-state is >1/quarter; medium priority if rare. Note that with the content-hash signal (Decision 5), ANY change to templates/gitignore/fence/settings shape fires `--ensure` — even small additions that wouldn't have triggered a manual bump under the prior `_INIT_SCHEMA_VERSION` design — so the effective drift-rate the user experiences may be higher than the "schema-level bumps" count suggests.

- **Q6. Non-lifecycle drift exposure.** The narrow design intentionally does NOT auto-fire on `/discovery`, `/backlog`, `/refine`, etc. invocations. Users who never invoke `/lifecycle` on a repo stay drifted indefinitely (within the bounds of "drift is mostly cosmetic for these skills"). Is this the right scope, or should at least `/refine` and `/discovery` also wire `--ensure`? Both write to `cortex/` substantially. **Decomposition impact**: determines whether the wiring expands beyond `/lifecycle` in a follow-up ticket.
