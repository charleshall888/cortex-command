# Research: overnight-plan-building

## Research Questions

1. **How does the orchestrator dispatch plan-gen sub-agents, and what's the context cost?**
   → The orchestrator (a Claude CLI session running `orchestrator-round.md`) dispatches plan-gen sub-agents via the Agent tool at Step 3b. Each sub-agent is an independent Claude session that reads the spec from disk, writes `plan.md`, and returns a one-line JSON status. The sub-agent *content* doesn't pollute orchestrator context — but the dispatch/collection cycle consumes orchestrator turns. The orchestrator's 50-turn budget is shared between plan-gen management (Steps 3a-3e) and its actual duties (escalation resolution, batch plan generation). **In practice, plan-gen has not been observed to trigger in production sessions** — every feature entering the overnight pipeline already has `plan_path` pointing to an existing file, because `/refine` and `/lifecycle plan` produce plans before sessions start. When plans exist, Steps 3a-3e cost 1-2 turns (2-4% of budget).

2. **Could plan generation move entirely out of the orchestrator?**
   → **Partially, but with structural costs.** The mechanical parts of Steps 3a-3e (file checks, state updates, git commits) could move to a pre-round Python module. However, the orchestrator's Step 0 (escalation resolution) can change which features need plans — it resolves ambiguities that caused prior-round deferrals and sets features back to `pending`. Plan generation must run *after* escalation resolution to avoid permanently deferring features the orchestrator could rescue. This temporal coupling means a pre-round module would introduce a state ordering bug unless it also handles escalation resolution, which requires LLM judgment.

3. **What orchestrator decisions are coupled to plan-gen?**
   → Two categories: (a) **Mechanical decisions** — spec file exists? plan file exists? mark failed/skip. These are pure `Path.exists()` checks. (b) **LLM judgment** — "is this spec too ambiguous for a concrete plan?" This judgment lives inside the plan-gen sub-agents, NOT in the orchestrator itself. The orchestrator just reads the sub-agent's `{"status": "deferred"}` response and writes a deferral file. However, there is a third, implicit coupling: **(c) Temporal dependency** — the orchestrator's Step 0 escalation resolution can change feature statuses from `deferred` to `pending`, making them eligible for plan generation. Steps 0 and 3 must execute in sequence within the same decision context.

4. **How do implementor agents receive their plans?**
   → **Pure disk reads, zero context leakage.** The implement template (`claude/pipeline/prompts/implement.md`) gives task agents: (a) single task description string, (b) task metadata (files, depends_on, complexity), (c) `spec_path` as a file path reference (not content), (d) learnings from prior failed attempts only. Full `plan.md` is never embedded. Implementor agents have no awareness that plans were generated by sub-agents.

5. **What's the token/turn overhead of the current approach?**
   → The orchestrator prompt includes ~46 lines of plan-gen instructions (Steps 3a-3e, 14.4% of the 320-line prompt). These are input tokens processed once at negligible marginal cost — not turns. When plans exist (the observed steady state), the orchestrator verifies file existence in 1-2 turns (2-4% of 50-turn budget). When plans would need generation (not yet observed in production), each sub-agent dispatch costs 1+ turns for dispatch and 1+ for result collection, plus 1 turn for git commit. The hypothetical worst case (5-feature round, 3 missing plans) would be ~8-10 turns (16-20%), but this scenario has not been observed.

## Codebase Analysis

### Current Architecture

```
runner.sh (main loop)
│
├─ Pre-round: interrupt recovery, worktree validation, pending count
│
├─ Orchestrator agent (claude -p, 50-turn budget)
│  ├─ Step 0: Resolve worker escalations (LLM judgment)
│  ├─ Step 1: Read state, filter features (mechanical)
│  ├─ Step 1a: Load session strategy (mechanical)
│  ├─ Step 1b: Conflict recovery awareness (state analysis)
│  ├─ Step 2: Read session plan (mechanical)
│  ├─ Step 2a: Dependency gate (mechanical)
│  ├─ Step 3a: Hard-fail missing specs (mechanical)          ← plan-gen block
│  ├─ Step 3b: Dispatch plan-gen sub-agents (delegation)     ← plan-gen block
│  ├─ Step 3c: Handle sub-agent results (mechanical)         ← plan-gen block
│  ├─ Step 3d: Commit generated plans (mechanical)           ← plan-gen block
│  ├─ Step 3e: Final plan validation (mechanical)            ← plan-gen block
│  ├─ Step 4: Generate batch master plan (Python call)
│  └─ Step 8: Exit
│
├─ batch_runner.py (Python process)
│  ├─ Parse batch-plan-round-N.md
│  ├─ Create per-feature worktrees
│  └─ Dispatch implementor agents (Agent SDK, parallel)
│     └─ Each receives: task description + metadata + spec path + learnings
│
└─ Loop: increment round
```

### Context Isolation Assessment

| Role | What enters context | Assessment |
|------|-------------------|------------|
| **Orchestrator** | Plan-gen sub-agent dispatch/results, escalation context, state JSON, session strategy | Partially involved in plan-gen management, but sub-agent content stays isolated. Steady-state cost is 1-2 turns for file-existence checks. |
| **Plan-gen sub-agent** | Spec content, research, recovery history → writes plan.md | Clean — isolated session, reads from disk, exits |
| **Implementor** | Single task description, metadata, spec path, learnings | Clean — no plan-gen artifacts, no spec content embedded |

### What the Orchestrator Does vs. What Requires LLM Judgment

| Step | Work | Requires LLM? | Could be Python? |
|------|------|---------------|-----------------|
| 3a: Hard-fail missing spec | `Path(spec_path).exists()` | No | Yes |
| 3b: Dispatch plan-gen | Spawn sub-agents, pass prompt | No (judgment is in sub-agents) | Yes — Agent SDK (with caveats, see Approach B risks) |
| 3c: Handle results | Parse JSON, write deferrals, update state | No | Yes |
| 3d: Commit plans | `git add` + `git commit` | No | Yes — subprocess |
| 3e: Final validation | `Path(plan_path).exists()` | No | Yes |
| **Sub-agent interior** | Read spec, judge ambiguity, design implementation approach, decompose into tasks | **Yes** | No |

**Finding**: The orchestrator's plan-gen *management* is mechanical. The LLM judgment in the pipeline lives inside the sub-agents. However, there is a temporal dependency: the orchestrator's Step 0 (escalation resolution) can change which features need plans, so Steps 0 and 3 must execute in the same decision context.

### Observed Production Behavior

Plan-gen Steps 3a-3e have not triggered sub-agent dispatch in observed overnight sessions. The `/refine` skill and `/lifecycle plan` phase produce plans before sessions start. The plan-gen mechanism in the orchestrator prompt is a fallback for features that enter sessions without pre-existing plans — a safety net, not a hot path.

## Feasibility Assessment

### Approach A: Status Quo (Recommended)

Keep plan-gen dispatch inside the orchestrator. Sub-agents are already separate sessions.

| Dimension | Assessment |
|-----------|-----------|
| Effort | None |
| Risks | If plan-gen frequency increases significantly, the orchestrator's turn budget could erode. Currently not observed. |
| Prerequisites | None |
| Context isolation | Plan-gen sub-agents are already isolated sessions. Orchestrator's steady-state overhead is 1-2 turns for file-existence checks. Implementors are fully clean. |

**Why this is the right default**: The current architecture already achieves the three-way isolation goal. The orchestrator dispatches plan-gen sub-agents as separate Claude sessions (Agent tool), so plan content never enters orchestrator context. Implementors receive only task descriptions and file paths. The orchestrator's plan-gen management overhead in the steady state (plans already exist) is 1-2 turns — negligible against a 50-turn budget. The temporal coupling between Step 0 (escalation resolution) and Step 3 (plan generation) is correctly handled by their co-location in the same orchestrator session.

### Approach B: Pre-Round Python Plan Generator (Conditional — only if plan-gen frequency increases)

New Python module (`plan_generator.py`) runs before orchestrator. Dispatches plan-gen agents via Agent SDK. Orchestrator Steps 3a-3e deleted.

```
runner.sh (main loop)
│
├─ Pre-round: interrupt recovery, worktree validation
│
├─ NEW: python3 -m cortex_command.overnight.plan_generator
│  ├─ Load state, filter features for this round
│  ├─ For each feature with missing plan_path:
│  │   ├─ Validate spec_path exists (hard-fail if not)
│  │   └─ Dispatch plan-gen agent via Agent SDK
│  ├─ Collect results: ok → proceed, deferred → write deferral + update state
│  ├─ Commit generated plans
│  └─ Save updated state
│
├─ Orchestrator agent (slimmer prompt, full 50-turn budget)
│  ├─ Step 0: Resolve worker escalations
│  ├─ Step 1: Read state, filter features
│  ├─ Step 2: Dependency gate
│  ├─ Step 3: Generate batch master plan (was Step 4)
│  └─ Exit
│
├─ batch_runner.py → implementor agents
└─ Loop
```

| Dimension | Assessment |
|-----------|-----------|
| Effort | M-L — new module, prompt edit, runner.sh integration, new dispatch infrastructure |
| Context isolation | Orchestrator fully clean; implementors clean; plan-gen in isolated SDK sessions |

**Structural risks**:

1. **State mutation ordering**: `plan_generator.py` runs before the orchestrator. A feature deferred in round N (spec ambiguity) could be rescued by the orchestrator's Step 0 escalation resolution — but by then plan generation has already passed. The feature wastes an entire round. Fixing this requires either: (a) moving escalation resolution into the pre-round module (requires LLM judgment, undermining the "pure Python" benefit), or (b) running plan generation *after* the orchestrator but *before* batch_runner (requiring the orchestrator to not exit after batch plan generation, complicating the clean exit model).

2. **SDK dispatch protocol mismatch**: `batch_runner.py` uses file-based exit reports (`lifecycle/{feature}/exit-reports/{task_number}.json`) for result collection. Plan-gen uses last-line JSON in agent text output (`{"status": "ok"}` or `{"status": "deferred"}`). `dispatch_task()` returns concatenated `TextBlock` contents — extracting last-line JSON from multi-message streaming output is fragile. A new result-parsing layer would be needed.

3. **Working directory**: Plan-gen agents write `plan.md` to `lifecycle/{slug}/plan.md` relative to the main repo, not a per-feature worktree. `dispatch_task()` is oriented around `worktree_path` — a new `cwd` mode for "shared repo root" would be needed.

4. **CLAUDE.md context divergence**: Agent tool sub-agents inherit the orchestrator's loaded CLAUDE.md instructions and conditional-load reference files. SDK-dispatched agents receive only `system_prompt`. The plan-gen prompt references "the lifecycle plan phase protocol" and "the standard plan.md format" which may be defined in reference files the SDK agent won't have.

5. **Two sequential state writers**: `plan_generator.py` and the orchestrator both write to `overnight-state.json`. A crash between state write and deferral file write in the pre-round module could leave features permanently stuck as deferred with no deferral file.

**Prerequisites** (not "None" — actual list):
- New `cwd` mode in `dispatch_task()` for shared-repo-root dispatch
- Response-parsing layer for last-line JSON protocol
- Sandbox write-allowlist entries for `lifecycle/` paths
- Resolution of the escalation-before-plan-gen ordering constraint

### Approach C: Dedicated Plan-Gen Claude Session

runner.sh spawns a separate `claude -p` session with a plan-gen-only prompt before the orchestrator.

| Dimension | Assessment |
|-----------|-----------|
| Effort | S — new prompt template, runner.sh integration |
| Risks | Full Claude session for mechanical work is wasteful; still burns LLM tokens on file-existence checks. Same state ordering problem as Approach B (runs before escalation resolution). |
| Prerequisites | None |
| Context isolation | Orchestrator clean; but burns a full Claude session on mostly mechanical work |

## Decision Records

### DR-1: Where should plan-gen dispatch live?

- **Context**: The orchestrator currently manages plan-gen dispatch (Steps 3a-3e) alongside its core duties (escalation resolution, batch plan generation). This includes ~46 lines of prompt instructions. In the observed steady state, plan-gen sub-agents have not been dispatched — plans already exist when sessions start.
- **Options considered**:
  - **(A) Status quo** — keep in orchestrator. Already works, sub-agents are isolated.
  - **(B) Pre-round Python module** — dispatch plan-gen agents from Python before orchestrator runs.
  - **(C) Dedicated Claude session** — separate `claude -p` session for plan-gen.
- **Recommendation**: **(A) Status quo**
  - Current architecture already achieves three-way context isolation (orchestrator, plan-gen sub-agents, implementors)
  - Steady-state overhead is 1-2 turns (2-4% of budget) — not a bottleneck
  - Step 0 escalation resolution and Step 3 plan generation are temporally coupled — they must execute in sequence within the same decision context. Approaches B and C break this ordering.
  - Plan-gen is a safety net, not a hot path. Building extraction infrastructure for a code path that has not triggered in production is premature.
- **Contingency**: If plan-gen frequency increases (e.g., sessions that skip `/refine` or accept features without pre-built plans), revisit Approach B with the state ordering constraint solved first.
- **Trade-offs**: The orchestrator prompt retains ~46 lines of plan-gen instructions even when they don't trigger. This is input token overhead (processed once, negligible cost) not turn overhead.

### DR-2: Should the plan-gen sub-agent prompt be a shared template?

- **Context**: Currently the plan-gen prompt lives inline in `orchestrator-round.md` (Step 3b, lines 238-265). If plan-gen is ever extracted to Python, duplication risks drift.
- **Recommendation**: **Defer.** With the status quo recommendation, there is only one consumer of this prompt. Extract to a shared template only if/when Approach B is pursued.

### DR-3: Model selection for plan-gen sub-agents

- **Context**: Plan-gen reads a spec, designs an implementation approach, decomposes it into tasks with correct dependency ordering, assigns file scopes and complexity tiers, and judges whether the spec is too ambiguous to plan. This is architectural reasoning — not structured translation.
- **Options**:
  - **(A) Inherit orchestrator model** (currently Opus)
  - **(B) Default to Sonnet**
  - **(C) Tier-based** — simple specs get Sonnet, complex get Opus
- **Recommendation**: **(A) Inherit orchestrator model** for now. Plan quality directly determines implementation success — wrong dependency ordering, incorrect file scopes, or missed complexity classification all produce `{"status": "ok"}` (bypassing the deferral escape hatch) and lead to faithful execution of a broken plan. Recovery is capped at 1 retry per feature. A bad plan burns two full implementation cycles before deferring. Since this code path has zero production invocations, optimizing its model tier is premature.
- **Contingency**: If plan-gen becomes a hot path and cost pressure increases, evaluate Sonnet with empirical A/B comparison of plan quality on actual specs before committing.

## Open Questions

- **When would plan-gen actually trigger?** The current pipeline always has plans pre-built via `/refine` and `/lifecycle plan`. If the goal is to support sessions where features skip these steps (e.g., rapid-fire backlog items), the plan-gen mechanism becomes load-bearing and its design deserves more attention. Clarify the intended usage pattern.
- **Prompt trimming**: Even with the status quo, the 46 lines of plan-gen instructions could be conditionally included in the orchestrator prompt only when features with missing plans are detected (runner.sh could check before filling the prompt template). This would eliminate the input token overhead without any architectural change.
