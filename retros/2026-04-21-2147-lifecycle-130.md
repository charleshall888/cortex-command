# Session Retro: 2026-04-21 21:47

## Problems

**Problem**: Initial spec's Non-Requirements section made a factually wrong claim about cross-repo behavior — asserted that for cross-repo sessions, followups "continue to land in the home-repo worktree" and that "today's broken path" writes untracked to the home repo. In fact, per `plan.py:489-494`, pure wild-light sessions have `state.worktree_path` pointing to the cross-repo worktree, and the pre-fix cwd-based path already writes there, not to home repo. **Consequence**: Would have shipped a false-premise scope boundary; required a full rewrite of the Non-Requirements' cross-repo carve-out mid-critical-review to accurately describe behavior across three session classes (home-repo-only, mixed, pure wild-light). Only caught by the cross-repo adversarial reviewer reading `plan.py` directly.

**Problem**: Initial Requirement 6 specified "re-raises the exception so the orchestrator does not continue with a wrong backlog directory" based on an unverified assumption about control flow. Re-raise from `orchestrator.py` actually unwinds through `asyncio.run` and `runner.sh:752-755` catches the non-zero exit as a warning and continues the round loop — so the spec's "fail-loud" was cosmetic, not functional. The spec even listed "transitions the session to paused" as an "equivalent" variant, but pausing requires a working `save_state` that may itself be broken when state.json is corrupt. **Consequence**: A requirement marketed as eliminating silent misdirection would have implemented something that STILL silently misdirects but now also emits `orchestrator_failed` events in a potential infinite loop. User had to be re-consulted to downgrade R6 to log-only; added a new "session-pause primitive" follow-up ticket to the scope-out list.

**Problem**: Initial R3/R8 design had cascade helpers (`_remove_uuid_from_blocked_by`, `_check_and_close_parent`) falling back to `Path.cwd() / "backlog"` when `backlog_dir` argument is absent or None. An internal caller inside `update_item()` that forgot to thread the argument through would silently resolve to process cwd — the exact bug the ticket exists to fix. R3's grep-based acceptance ("no module-level BACKLOG_DIR uses") would pass while the cascade drift survived in internal thread-throughs. **Consequence**: The fix would have re-introduced its own bug as a silent fallback; caught by the interactive-CLI adversarial reviewer. Had to invert R3's default: internal API now raises on missing `backlog_dir`, cwd-fallback isolated to CLI `main()` entry points only.

**Problem**: Initial R4 did not address the SIGINT trap path. The spec's Edge Cases said "the followups are discarded with the worktree on the cleanup path — whichever the existing trap logic does" — which is deferring a real decision inside a spec that claimed to fix followup persistence. The trap at `runner.sh:507-521` calls `create_followup_backlog_items` inline, writes a report, then `exit 130`s — never reaching the new second-commit block placed after `runner.sh:1207`. **Consequence**: Requirement 1 required the trap caller to pass `backlog_dir`, but the followups would be written to the worktree and then discarded on cleanup — the requirement delivered nothing for the trap path. Caught by the second-commit adversarial reviewer; had to add a second-commit block inside the trap before `exit 130`.

**Problem**: Initial R4 had no success guard on `generate_and_write_report`. The Python call's non-zero exit is already swallowed by `|| echo "Warning: morning report generation failed"` at `runner.sh:1198` and `:1214`, so the second-commit block would fire unconditionally — committing partial followup state under a misleading "record followup backlog items" message after report-gen crashed mid-write. **Consequence**: A data-integrity regression masquerading as a fix; caught by adversarial review. Had to add an exit-code check and a `followup_commit_skipped` event for observability.

**Problem**: Research Agent 1 (Codebase) claimed `runner.sh:986`'s cp runs "before the round loop" as a "pre-stage cp." Corrected by adversarial review's direct reading, but the incorrect framing had to be unwound during research synthesis.

**Problem**: Research Agent 4 (Tradeoffs) claimed "runner does `cd $REPO_ROOT` at line 1222 before report generation." The fix's "what was wrong in the nominal path" framing needed to be narrowed to the SIGINT trap path only during research synthesis.
