# Pipeline Execution Module

**Internal reference — not a user-facing skill.**

The `cortex_command/pipeline/` Python module is the async execution backend for overnight mode.
It is invoked automatically by `cortex_command/overnight/batch_runner.py` — do not invoke it directly.
To run multi-feature autonomous sessions, use [`/overnight`](overnight.md).

---

> For how `dispatch.py` interacts with the Claude Code SDK — model selection matrix, error classification, and design trade-offs — see [SDK Integration](sdk.md).

## Module Contents

The `cortex_command/pipeline/` module contains the execution machinery used by the overnight runner:

| File | Role |
|------|------|
| `state.py` | `pipeline-state.json` read/write, phase transitions, event logging |
| `dispatch.py` | Agent SDK wrapper: model/budget tier selection, streaming progress events |
| `merge.py` | Feature branch merge with CI gate, post-merge tests, auto-revert on failure |
| `retry.py` | Retry loop with model escalation (Haiku → Sonnet → Opus) |
| `worktree.py` | Git worktree create/cleanup for feature isolation |
| `parser.py` | Parse `master-plan.md` and per-feature `plan.md` into task objects |
| `report.py` | Generate execution summary report |
| `metrics.py` | Cost and timing metrics collection |
| `conflict.py` | Merge conflict classification and repair agent dispatch; inspects unmerged files, aborts in-progress merges, provides `dispatch_repair_agent()` |
| `merge_recovery.py` | Post-merge test-failure recovery loop; orchestrates flaky guard followed by up to two code-repair attempts with model escalation (sonnet → opus) |

## State Files Reference

| File | Description |
|------|-------------|
| `lifecycle/pipeline-state.json` | Live execution state during overnight runs. Written atomically by `cortex_command/pipeline/state.py`. Records execution phase, per-feature statuses, and timestamps. |
| `lifecycle/master-plan.md` | The batch session plan (features, round assignments, execution strategy). |
| `lifecycle/pipeline-events.log` | Append-only JSONL event log. Each line is a timestamped event (dispatch_start, dispatch_complete, merge_start, merge_success, etc.). |
| `lifecycle/{feature}/plan.md` | Per-feature task plan. Generated by the orchestrator before dispatch if missing. |

### Plan Structure

The pipeline uses a two-level plan hierarchy:

- **`lifecycle/master-plan.md`** — The top-level batch plan produced by `/overnight`. It
  covers all features for the session: feature slugs, round assignments, and execution
  strategy. `batch_runner.py` reads this once at session start.
- **`lifecycle/{slug}/plan.md`** — The per-feature task plan. Contains the task breakdown
  for a single feature (tasks, dependencies, complexity estimates). Generated by the
  orchestrator before dispatch if missing; read by `dispatch.py` when building the
  per-feature agent prompt.

`parser.py` reads both levels and converts them into task objects consumed by the runner.

### pipeline-state.json Schema

The state file is written by `cortex_command/pipeline/state.py` (`PipelineState` dataclass):

```json
{
  "phase": "executing",
  "features": {
    "my-feature": {
      "status": "pending",
      "retries": 0,
      "last_error": null,
      "started_at": null,
      "completed_at": null
    }
  },
  "started_at": "2026-03-11T17:00:00+00:00",
  "updated_at": "2026-03-11T17:05:00+00:00",
  "paused_from": null
}
```

**Valid execution phases**: `planning`, `executing`, `merging`, `integration-review`,
`complete`, `paused`.

**Valid feature statuses**: `pending`, `executing`, `reviewing`, `merging`, `merged`,
`paused`, `failed`.

**Phase transitions** (forward only):
`planning → executing → merging → integration-review → complete`

Any non-complete phase can transition to `paused`. A paused session resumes only to the
phase it was paused from (`paused_from` field records this).

Writes are atomic: `state.py` writes to a temp file in the same directory, then uses
`os.replace()` to swap it in. Shell scripts and the statusline hook can read the file
without observing a partial write.

## Integration Branch

Feature branches are named `pipeline/{feature}`. After `batch_runner.py` merges a feature branch,
the source worktree at `.claude/worktrees/{feature}/` is cleaned up automatically by
`worktree.py`.

For overnight sessions, merges target an integration branch `overnight/{session_id}`.

## Keeping This Document Current

`cortex_command/pipeline/` is the authoritative source for state file paths and schema. When the
`PipelineState` dataclass in `state.py` changes (new fields, new phases), update the
schema section above. When the `cortex_command/pipeline/` module adds or removes files, update
the module table.
