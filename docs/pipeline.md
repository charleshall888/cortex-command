# Pipeline Execution Module

**Internal reference — not a user-facing skill.**

The `claude/pipeline/` Python module is the async execution backend for overnight mode.
It is invoked automatically by `claude/overnight/batch_runner.py` — do not invoke it directly.
To run multi-feature autonomous sessions, use [`/overnight`](overnight.md).

---

## Module Contents

The `claude/pipeline/` module contains the execution machinery used by the overnight runner:

| File | Role |
|------|------|
| `state.py` | `pipeline-state.json` read/write, phase transitions, event logging |
| `dispatch.py` | Agent SDK wrapper: model/budget tier selection, streaming progress events |
| `merge.py` | Feature branch merge with CI gate, post-merge tests, auto-revert on failure |
| `retry.py` | Retry loop with model escalation (Haiku → Sonnet → Opus) |
| `worktree.py` | Git worktree create/cleanup for feature isolation |
| `parser.py` | Parse `master-plan.md` and per-feature `plan.md` into task objects |
| `report.py` | Generate execution summary report |
| `metrics.py` | Cost and timing metrics collection |

---

## State Files Reference

| File | Description |
|------|-------------|
| `lifecycle/pipeline-state.json` | Live execution state during overnight runs. Written atomically by `claude/pipeline/state.py`. Records execution phase, per-feature statuses, and timestamps. |
| `lifecycle/master-plan.md` | The batch session plan (features, round assignments, execution strategy). |
| `lifecycle/pipeline-events.log` | Append-only JSONL event log. Each line is a timestamped event (dispatch_start, dispatch_complete, merge_start, merge_success, etc.). |
| `lifecycle/{feature}/plan.md` | Per-feature task plan. Generated by the orchestrator before dispatch if missing. |

### pipeline-state.json Schema

The state file is written by `claude/pipeline/state.py` (`PipelineState` dataclass):

```json
{
  "phase": "executing",
  "features": {
    "my-feature": {
      "status": "pending",
      "retries": 0,
      "last_error": null,
      "started_at": null,
      "completed_at": null
    }
  },
  "started_at": "2026-03-11T17:00:00+00:00",
  "updated_at": "2026-03-11T17:05:00+00:00",
  "paused_from": null
}
```

**Valid execution phases**: `planning`, `executing`, `merging`, `integration-review`,
`complete`, `paused`.

**Valid feature statuses**: `pending`, `executing`, `reviewing`, `merging`, `merged`,
`paused`, `failed`.

**Phase transitions** (forward only):
`planning → executing → merging → integration-review → complete`

Any non-complete phase can transition to `paused`. A paused session resumes only to the
phase it was paused from (`paused_from` field records this).

Writes are atomic: `state.py` writes to a temp file in the same directory, then uses
`os.replace()` to swap it in. Shell scripts and the statusline hook can read the file
without observing a partial write.

---

## Integration Branch

Feature branches are named `pipeline/{feature}`. After `batch_runner.py` merges a feature branch,
the source worktree at `.claude/worktrees/{feature}/` is cleaned up automatically by
`worktree.py`.

For overnight sessions, merges target an integration branch `overnight/{session_id}`.

## Recovery Procedures

### Inspecting State Manually

```bash
# What phase is the execution in?
python3 -c "
import json
state = json.load(open('lifecycle/pipeline-state.json'))
print('phase:', state['phase'])
for name, f in state['features'].items():
    print(f['status'].ljust(12), name)
"

# What events have fired?
python3 -c "
import json
for line in open('lifecycle/pipeline-events.log'):
    e = json.loads(line)
    print(e['ts'][:19], e['event'])
"
```

### Recovery: Feature Stuck in `executing`

If the overnight runner process was killed hard (OOM, power loss) rather than receiving a
signal, a feature may be stuck in `executing` with no live process. To reset manually:

```python
from pathlib import Path
from claude.pipeline.state import load_state, save_state

p = Path("lifecycle/pipeline-state.json")
state = load_state(p)
for name, f in state.features.items():
    if f.status == "executing":
        print("resetting:", name)
        f.status = "pending"
        f.last_error = "reset from stuck-executing after crash"
save_state(state, p)
```

Then restart the overnight runner via `/overnight resume`.

### Recovery: Merge Conflict on Feature Branch

If a feature fails with a merge conflict (visible in the morning report as `failed` with
a merge error), the integration branch is left at the last clean merge point — `merge.py`
aborts a conflicted merge automatically. Options:

1. **Resolve and retry**: Checkout the feature branch, fix the conflict, commit, then
   restart the overnight runner (the feature will be retried from `pending`).
2. **Skip the feature**: Leave it as `failed`. It will appear in the morning report as
   carried over. Open a manual PR for it later.
3. **If the integration branch has a partial state**: Verify with `git log --oneline main | head -5`
   that no half-merged commit exists. If a revert is needed, `merge.py`'s `revert_merge()`
   handles it — run it via `python3 -c "from claude.pipeline.merge import revert_merge; revert_merge('my-feature')"`.

## Keeping This Document Current

`claude/pipeline/` is the authoritative source for state file paths and schema. When the
`PipelineState` dataclass in `state.py` changes (new fields, new phases), update the
schema section above. When the `claude/pipeline/` module adds or removes files, update
the module table.
