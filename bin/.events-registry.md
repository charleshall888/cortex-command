# Events registry

Source of truth for the `bin/cortex-check-events-registry` static gate (spec R5/R7 of the events.log discipline ticket). Every `"event": "<name>"` literal that appears in skill prompts or orchestrator-round-style prompt corpus (gate-enforced scan surface) must be declared here. Python emission sites are out-of-scan (`scan_coverage: manual`) and listed here for human reference only.

The schema is documented in `docs/internals/events-registry.md`. Required column order matches spec R5: `event_name | target | scan_coverage | producers | consumers | category | added_date | deprecation_date | rationale | owner`.

Adding an entry is an architectural commitment: each row promises a documented consumer. `deprecated-pending-removal` rows are pruned by a follow-up cleanup PR after the 30-day grace window (the `--audit` recipe surfaces stale rows; `owner` identifies bump authority).

## Field-additive schema extensions

Field-additive extensions extend an event's schema without minting a new event-name row. Optional fields are declared here under their owning event; emitters omit the field from the JSON when its value is `None`. Consumers tolerate the absence and treat the field as `None`/missing.

### `sentinel_absence` (Req 7/8, 2026-05-20)

- `source_path:` (string, optional) â€” original caller-supplied path when the artifact was ad-hoc-snapshotted via `validate_artifact_path(..., allow_adhoc=True)`. Preserved verbatim post-NUL/surrogate validation; ASCII control characters (including newlines) are legal and are JSON-escaped on write by `json.dumps` in `append_event`. Omitted from the emitted JSON when the artifact resolved under `cortex/lifecycle/` or `cortex/research/` (non-adhoc path).
- `snapshot_sha:` (string, optional) â€” full hex SHA-256 of the ad-hoc snapshot content. Consumed by `cortex clean --adhoc` (Requirement 9) as the retention-pin token: any snapshot whose computed SHA appears in any `events.log` `snapshot_sha:` value is retained regardless of mtime. Omitted from the emitted JSON when the artifact resolved under `cortex/lifecycle/` or `cortex/research/` (non-adhoc path).

### `dispatch_error` (#309 diagnostics, 2026-06-17)

- `cwd:` (string, optional) â€” the dispatch worktree working directory (`str(worktree_path)`) at the point of failure, added so a failed dispatch's location is diagnosable from the event log alongside the pre-existing `child_stderr`/`exit_code` fields. Emitted only from the two exception error branches in `dispatch_task` (`cortex_command/pipeline/dispatch.py`); omitted on the success and budget-exhausted returns.

### `task_output` (#309 diagnostics, 2026-06-17)

Field-additive extension carrying the final failed dispatch attempt's diagnostics as **distinct** keys (NOT folded into `output`, which is empty on a worker crash), sourced from `RetryResult.last_dispatch_diagnostics`. Emitters omit each field when the bundle is `None` (the success path); the consumer (`render_failed_features` via `_read_last_task_diagnostics`, `cortex_command/overnight/report.py`) tolerates absence.

- `child_stderr:` (string, optional) â€” redacted, byte-capped stderr tail of the final failed attempt. Omitted on success.
- `exit_code:` (int, optional) â€” exit code of the final failed attempt; absent for `CLIConnectionError`/timeout (rendered `unknown` by consumers) and on success.
- `cwd:` (string, optional) â€” dispatch worktree working directory of the final failed attempt. Omitted on success.

### `feature_deferred` (overnight-review-gate-reverts-a-clean Phase 2, 2026-06-23)

Field-additive extension to the review-failure deferral details emitted by `outcome_router.py`. These keys split a *could-not-run review* (the review agent completed, `success=True`, but produced no parseable verdict â€” the merge is **preserved** on the integration branch for human re-review) from a *genuine dispatch crash* (`success=False` / a raised exception â€” the merge is **reverted** and the systemic breaker is tripped as a crash). All keys are optional; consumers (`render_deferred_questions` / `render_merged_to_integration` in `cortex_command/overnight/report.py`) read each with `.get(key, default)` for archived-log tolerance.

- `could_not_run:` (bool, optional) â€” `True` ONLY when the review agent completed but produced no usable verdict (resolved verdict `ERROR` sentinel); the merge is preserved (`merge_reverted=False`). This is the **positive** discriminator the report and integration-PR surfaces key on â€” never inferred from `merge_reverted=False`, which a genuine crash with a failed revert (or a repair-path `_reset_ff_merge` returning `False`) also produces. Defaults to `False` / absent for `APPROVED`/`REJECTED`/`CHANGES_REQUESTED` and for the genuine-crash path.
- `merge_reverted:` (bool, optional) â€” whether the live merge commit was reverted. `False` on the preserved could-not-run path; the authoritative revert outcome (`revert_merge(...).success` or `_reset_ff_merge(...)`) on the crash path. A could-not-run preserve and a crash with a failed revert both record `False`, which is why the report disambiguates on `could_not_run`, not on this field.
- `review_dispatch_crashed:` (bool, optional) â€” set `True` ONLY on a genuine dispatch crash (`success=False` / raised exception). The could-not-run path no longer co-sets this flag (it was previously co-set with the ERROR verdict); `review_dispatch_crashed` now denotes a crash exclusively (overnight-review-gate-reverts-a-clean R6).

### `pipeline_systemic_failure` (overnight-review-gate-reverts-a-clean Phase 2, 2026-06-23)

- `cause_class:` (list[str]) â€” the trailing-window systemic-failure class labels at the trip. Now admits the new `review_no_artifact` value (could-not-run review) alongside the existing `review_dispatch_crash` (genuine crash). A mixed crash + no-artifact batch surfaces BOTH labels for diagnosis; the threshold counts the aggregate (`SYSTEMIC_FAILURE_THRESHOLD = 3`), the labels distinguish the kinds. Both values are members of `_SYSTEMIC_ERROR_TYPES` so the threshold derivation recognizes them.

### `spec_approved` (epic 371 Phase B, 2026-07-10)

- `decision:` (enum `approved`, optional) â€” durable operator-consent marker recorded at the specifyâ†’plan approval gate, emitted via the typed `spec-approved --decision approved` subcommand of `cortex-lifecycle-event`. Enum-typed (single value `approved` this wave) so future consent values are additive; enum/identifier-only by policy because `events.log` has no redaction path (consent-field precedent: `plan_approved`'s optional `dispatch_choice`). Emitters omit the key from the JSON when the flag is absent, keeping the bare `{ts, event, feature}` form valid; consumers tolerate the absence and treat it as `None`. The field makes `spec_approved` self-describing as a consent record without minting a new event name.

### `drift_protocol_breach` (epic 371 Phase B follow-up, 2026-07-11)

- `cycle:` (int, optional) â€” the review cycle the breach fired in, emitted via the typed `drift-protocol-breach --cycle N` subcommand. The review-verdict verb (`cortex_command/lifecycle/review_verdict.py`) presence-checks the breach row on `event`+`cycle` (mirroring `review_verdict`), so a genuine second breach in a later review cycle is no longer suppressed by the cycle-1 row â€” while crash-recovery re-runs within a cycle stay idempotent. Emitters omit the key when absent (legacy/unqualified emissions), and consumers tolerate the absence.

### `feature_paused` (374 Phase 1, 2026-07-11)

Field-additive per-pause accountability (374 R5) so a `feature_paused` row records WHICH kept-pause it represents and what resume authority that pause needs. Both keys emit via the typed `feature-paused --slug <slug> --kind <kind>` subcommand of `cortex-lifecycle-event`; emitters omit each key from the JSON when its flag is absent, so a legacy bare `{ts, event, feature}` row stays valid.

- `slug:` (string, optional) â€” the kept-pause marker slug this pause corresponds to (the discriminant in `skills/lifecycle/references/kept-pauses-data.toml`; e.g. `plan-approval`). Consumers tolerate the absence and treat it as `None`.
- `kind:` (enum `question | phase-exit-wait | config-conditional | relayed-consent`, optional) â€” the pause's resume-authority kind. The `reduce_lifecycle_events` reducer (`cortex_command/common.py`) reports this as `state["pause_kind"]`; a kind-absent OR out-of-vocab (legacy) row fails closed to the most-restrictive `relayed-consent` (operator-resume-only) so an under-specified pause never silently auto-resumes (hazard 3). Pause-enforcement consumers land in Phase 3.

### advance-emitted `invocation_id` (374 R12/R14, 2026-07-11)

Field-additive extension carried by the dual-emitted **legacy-vocabulary** transition rows that the `advance` write-side verb (`cortex_command/lifecycle/advance.py`, 374 Task 14) appends alongside its machine rows â€” `phase_transition`, `review_verdict`, `spec_approved`, `plan_approved`, and `feature_complete`. The key does NOT extend the independent typed-subcommand emitters of those same events (`cortex_command/lifecycle/plan_decision.py`, `review_verdict.py`, `spec_approve.py`, `implement_transition.py`, `finalize.py`), which omit it. It is core (not field-additive) on the `advance_started`/`advance_committed` rows themselves; this subsection covers only its field-additive appearance on the legacy-vocabulary rows.

- `invocation_id:` (string, optional) â€” the deterministic claim/commit correlation token (the same value persisted in the paired `advance_started`/`advance_committed` rows; business-derived, generated once, reused across retries). Present only on advance-emitted legacy-vocabulary rows, where it distinguishes an advance-authored transition from an independent legacy emission of the same event during the dual-emission coexistence window (R12/R17, ADR-0024). Emitters omit the key from the JSON when the row was not authored by `advance`; consumers tolerate the absence and treat it as `None`. Canonical serialization is unchanged (`json.dumps(row)+"\n"`, default spacing) so both shapes parse identically to the typed-subcommand rows. Semantics: ADR-0020 (374 in-file amendment).

### advance-emitted `consent_utterance` on `plan_approved` / `spec_approved` (374 Task 14b, 2026-07-11)

Field-additive quoted-utterance payload (spec R12, hazard 5 â€” the fabricated-attestation defense) carried on the consent-bearing legacy-vocabulary rows the `advance` write-side verb (`cortex_command/lifecycle/advance.py`, 374 Task 14b) emits: `plan_approved` (the plan gate) and `spec_approved` (the spec gate). When a human consent underlies the transition, `advance` lands the operator's VERBATIM quoted consent text on the row so a fabricated attestation is a specific, falsifiable lie rather than a bare flag. The key does NOT extend the independent typed-subcommand emitters of those events (`cortex_command/lifecycle/plan_decision.py`, `spec_approve.py`), nor any other legacy-vocabulary row (`phase_transition`/`feature_paused`/`review_verdict`/`lifecycle_cancelled`/`batch_dispatch`) â€” only `plan_approved`/`spec_approved`.

- `consent_utterance:` (string, optional) â€” the operator's verbatim quoted consent text underlying the approval, threaded via `advance --consent-utterance` and appended field-additively to the arm's own fields (after `dispatch_choice`/`decision`, before the additive `invocation_id`). Enum/identifier-only policy does NOT apply (this is quoted free text by design); as with every field-additive extension, `events.log` has no redaction path, so callers pass consent text that is safe to persist durably. Emitters omit the key from the JSON when no utterance was supplied, keeping the bare pre-14b row shape valid; consumers tolerate the absence and treat it as `None`. Pairs with the merge-consent gh PR-state cross-check (same hazard-5 defense; that check refuses `review.approved` when a recorded PR is not actually merged) but is itself a pure additive payload, never a runtime refusal.
